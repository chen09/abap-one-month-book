Chapter 01 - SAP 和 ABAP 基础

3. ABAP Platform 平台概念

  • Chapter 01
  • Section 03
  • self-contained study note

3.1 一句话结论

ABAP Platform 是 SAP 用来承载 ABAP-based solutions 的技术平台，不等于 SAP S/4HANA 这个业务产品，也不等于 ADT 这个开发工具。学它的目的，是把前两节的 SAP system landscape 系统环境分层 和 Application Server ABAP 运行时基础 放进更大的“产品 / 平台 / 运行时 / 开发模型”地图里。

3.2 到底学什么

先抓一个边界：ABAP Platform 是“技术基盘”，不是“业务套件”。SAP Help 对 ABAP Platform 的基本定位是：它是 ABAP-based SAP solutions 的 foundation；在本书面向的 SAP S/4HANA / ABAP Cloud 语境下，它建立在 SAP HANA 的 database and data management platform 之上。也就是说，ABAP Platform 提供 ABAP 语言、repository、runtime、Dictionary、SQL/CDS、RAP、OData、authorization、jobs、logs、workflow 等技术能力；SAP S/4HANA 则是在这些平台能力之上交付业务流程、主数据、应用和行业功能的产品。

这和 Java/Python 背景里的“语言 + runtime + framework + app”很像，但 SAP 的命名更容易混。Java 世界里，Java 语言、JVM、Spring、某个 ERP 应用通常不会被叫成同一个词；SAP 项目里，大家却经常把 SAP、S/4、ABAP、NetWeaver、ABAP Platform、ADT、ABAP Cloud 混在一句话里说。这里要特别记住：SAP NetWeaver AS ABAP 是旧项目资料里常见的说法，ABAP Platform 可以理解为较新命名体系中对 AS ABAP 技术平台的延续和现代化表述；两者不是两个完全分离的架构层。初学者如果不先分层，很容易以为“学 ABAP Platform”就是“学 SAP S/4HANA 业务”，或者以为“装了 ADT”就等于拥有平台。

本节先学五个词的相对位置。第一，SAP S/4HANA 是业务产品 / business suite。第二，ABAP Platform 是技术平台 / foundation。第三，AS ABAP 是运行时和 application server layer。第四，ADT 是开发工具。第五，ABAP Cloud 是现代开发模型和规则集合，强调 released APIs、Clean Core、restricted language scope 和 ATC checks。它们有关联，但不能互相替代。

还要学一个项目判断：ABAP Platform 能力和具体系统 release 有关。项目里不能只说“我们是 ABAP”，还要看实际 SAP S/4HANA release、ABAP Platform release、support package、software component 和系统部署形态。对开发者来说，ABAP Platform 通常是作为 SAP S/4HANA / SAP S/4HANA Foundation 等产品基础的一部分出现，不是自己单独下载、单独安装来承载生产业务的独立产品。某些语法、RAP 能力、released API、ADT 检查、cloud restriction，在不同 release 和产品形态下会不一样。

对初学者来说，最重要的不是背产品名，而是避免把问题问错。客户说“在 S/4HANA 上加一个按钮”，这句话属于业务产品语境；开发者说“用 RAP 暴露 OData V4 service”，这属于 ABAP Platform 和开发模型语境；Basis 说“work process 满了”，这属于运行时和系统运维语境；架构师说“只能使用 released APIs”，这属于 Clean Core / ABAP Cloud 规则语境；开发者说“用 ADT 创建 behavior definition”，这属于工具和 repository 对象语境。五层分开，沟通才不会互相误解。

本节可以作为后续所有章节的“坐标轴”。学 DDIC、CDS、RAP、OData、authorization、background job、application log 时，都要问：这是业务产品能力、平台技术能力、运行时行为、开发模型约束，还是工具操作？同一个词放错层，结论就会变形。

3.3 为什么重要

如果不理解 ABAP Platform，后面的学习会在技术选型上失焦。看到一个需求时，初学者可能马上问“要不要写 ABAP class”，但真实项目会先问：标准功能能不能覆盖，是否有 released API 或 extension point，是否允许自定义 repository object，是否符合 Clean Core，目标系统的 release 是否支持对应能力。平台概念的价值，就是把“能写代码吗”提升成“在当前产品、平台、运行时和开发模型下，哪种实现方式可交付”。

对项目交付来说，平台边界决定技术选型。一个需求能不能直接改表、能不能用 classic enhancement、能不能调用某个类、能不能建 RAP business object、能不能发布 OData V4 service、能不能进入 Public Edition，都不是单看 ABAP 语法能决定的。它取决于平台能力、开发模型、release、产品形态和 Clean Core 约束。平台概念学清楚，后面学 CDS view entity 视图实体、RAP business object 业务对象、Service binding 服务绑定、Released API dependency check 时才不会漂。

对日本 SAP 项目来说，这一层也很实用。客户或上级可能问：S/4HANA の標準機能ですか、それとも ABAP 側の拡張ですか、ABAP Cloud で実装できますか、リリース済み API はありますか。如果只会说“我写 ABAP”，就无法回答它属于标准业务产品、平台技术、扩展模型还是交付治理。

五层图还能帮助需求评审。评审一个需求时，可以按顺序问：

Product:
  Is this covered by SAP S/4HANA standard configuration or process?

Platform:
  Which ABAP Platform capabilities are needed: CDS, RAP, OData, jobs, logs, authorization?

Runtime:
  What runs synchronously, asynchronously, in dialog work process, background job, or service call?

Development model:
  Is classic ABAP allowed, or must it follow ABAP Cloud / Clean Core / released API rules?

Tooling:
  Which tools and checks prove it: ADT, ATC, transport, service preview, API test?

这不是形式主义。很多项目失败不是因为不会写代码，而是需求一开始没有分清“标准功能能不能配置”“是否允许扩展”“扩展在哪个系统里做”“调用哪个 released API”“谁来运行后台作业”。ABAP Platform 的学习价值，就是让这些问题在 Day 01 就有语言。

3.4 Key mechanism / 核心机制

可以用一张五层图记住这节，后面的例子只是在把同一张图套到不同场景：

Business product
  SAP S/4HANA, SAP BW/4HANA, industry solutions

Technology platform
  ABAP Platform / ABAP Cloud development model / released APIs

Runtime
  Application Server ABAP, dispatcher, work processes, database interface

Development model
  Classic ABAP, RAP, ABAP Cloud, Clean Core rules

Tooling
  ADT, SAP GUI transactions, ATC, transport tools

第一层机制是 product decision vs platform capability。一个需求说“在 S/4HANA 里做”，下一句不应该立刻变成“写哪个程序”，而是先判断标准业务能力、配置、released API、extension point 和自定义对象边界。平台决定可用技术能力，产品决定业务语义和标准流程，两者合在一起才形成技术方案。

第二层机制是 platform vs runtime。前一节 AS ABAP 讲的是程序运行时如何处理请求；ABAP Platform 的范围更广，还包括开发对象模型、Dictionary、ABAP SQL/CDS、RAP/OData、authorization、job/log/test/quality 相关基础能力。可以粗略理解为：AS ABAP 是平台中的运行时核心，ABAP Platform 是围绕 ABAP 开发和运行的一整套技术基础。

第三层机制是 platform vs development model。ABAP Cloud 不是“另一个 ABAP Platform”，也不是“只在云上才需要关心”的口号，而是在 ABAP Platform 能力之上规定“应该怎样写可升级、云就绪、Clean Core 的 ABAP”。它可以出现在 SAP BTP ABAP environment、SAP S/4HANA Cloud Public Edition 中的 ABAP environment，也可以在较新的 SAP S/4HANA on-premise / SAP S/4HANA Cloud, private edition 系统中作为 Clean Core 导向的开发选项使用；具体可用性必须看 release。SAP Learning 对 released APIs 的说明强调，客户新开发和扩展开发应使用 public-released interfaces；在 ADT 中，ABAP Cloud development model 会通过 syntax checks 和 ATC checks 执行这些限制，例如 non-released APIs、SAP GUI-dependent 语句或不适合 cloud-ready extension 的对象会被限制或标红。

第四层机制是 release dependency。平台能力不是抽象口号，而是版本相关事实。项目里要学会查看 System -> Status、software component、SAP_BASIS / SAP_ABA / S4CORE 之类组件和 release 信息；读 SAP Help 时也要确认文档版本。不要把网上某个 ABAP 语法或 RAP 功能直接套到任何系统。

第五层机制是 deployment caveat。在 SAP S/4HANA Cloud Public Edition 中，底层 ABAP Platform 由 SAP 托管，开发者不会像 on-premise / SAP S/4HANA Cloud, private edition 项目那样接触全部 AS ABAP 管理层。Public Edition 的扩展更强调 key user extensibility、developer extensibility、released APIs 和 ABAP Cloud development model；side-by-side 场景则常用 SAP BTP ABAP environment。Day 01 只要记住：同一个“ABAP Platform”词，在不同部署形态下给开发者开放的能力和边界不同。

可以用一个对照表记住部署差异：

部署/开发场景开发者通常关心什么常见误解
SAP S/4HANA on-premiseclassic ABAP、RAP、transport、system release、Basis 协作以为任何旧式增强都适合未来升级
SAP S/4HANA Cloud, private edition类似 on-premise，但更强调升级治理和 Clean Core以为“私有云”就没有约束
SAP S/4HANA Cloud Public Editionkey user / developer extensibility、released APIs、ABAP Cloud以为能像 on-premise 一样改所有对象
SAP BTP ABAP environmentside-by-side ABAP、ABAP Cloud、released APIs、service integration以为它等同于客户 S/4HANA 后端
学习/POC sandbox练习语法和对象关系以为 POC 可用能力等于生产可交付能力

这张表只给方向，不替代 SAP Help 和项目架构约束。真正项目里，要看系统 landscape、contract、edition、release、extension policy 和客户治理。

第六层机制是 evidence。平台判断不能只靠口头说法。设计书里应能指出：系统版本在哪里确认、目标 edition 是什么、使用的 API 是否 released、ATC/ABAP Cloud 检查是否通过、服务是否能 preview/test、transport 到 QAS 后是否仍可运行。没有这些证据，“平台支持”就只是猜测。

3.5 日本项目现场说法

  • 常见日语：ABAPプラットフォーム、技術基盤、SAP S/4HANA基盤、アプリケーションサーバ、開発モデル、クリーンコア開発モデル、バージョン依存、リリースバージョン依存、リリース済みAPI、クリーンコア、拡張性。
  • 说明句：SAP S/4HANA は業務アプリケーションであり、ABAP Platform はその ABAP 開発・実行を支える技術基盤です。
  • 说明句：この要件は ABAP Cloud の開発モデルで実装できるか、リリース済み API の有無を確認する必要があります。
  • 使用场景：基本设计、技术方案说明、Clean Core review、升级影响调查、开发规范说明、客户/上级技术边界说明。
  • 现场提醒：基盤 在日本项目里可能指基础设施、技术平台或 Basis 负责区域，语境很重要。说明时最好补一句是 ABAP 開発・実行の技術基盤，避免被理解成服务器/网络层。不要写 基盤プラットフォーム 这种重复表达。

3.7 Example / 学习例子

假设项目要做一个新的审批应用。业务上，它运行在 SAP S/4HANA 的采购或销售流程附近；技术上，可能需要使用 ABAP Platform 提供的 repository、CDS、RAP、authorization、OData service、application log、background job 等能力；运行时，它会在 AS ABAP 的 work process 中被执行；开发时，开发者用 ADT 创建对象；架构约束上，项目可能要求 ABAP Cloud 和 released APIs。把这几层拆开，技术方案才不会混乱。

再看一个典型误会。有人说“这个是 S/4HANA 项目，所以一定可以用最新 RAP 功能”。这句话不一定成立。是否能用某个 RAP feature，要看实际 release、support package、开发模型、系统形态和已释放 API，而不是只看产品名。另一个人说“我装了 ADT，所以可以做 ABAP Cloud”。这也不成立：ADT 是工具，ABAP Cloud 是开发模型，系统必须支持对应的检查、released objects 和 runtime 能力。

最小练习：写五行对照表：SAP S/4HANA = business product，ABAP Platform = technology platform，AS ABAP = runtime，ADT = development tool，ABAP Cloud = development model。再把一个你见过的 POC 对象放进这五层里。

AC0567C 风格的批量上传 POC 可以这样放进五层：

Business product:
  It supports an SAP S/4HANA business process, such as upload/validate/execute around business data.

Technology platform:
  It may use ABAP repository objects, DDIC, CDS/RAP, OData service binding, authorization, application log, background job.

Runtime:
  API request runs in AS ABAP service processing; long-running execution may move to background job.

Development model:
  New code should prefer Clean Core, released APIs, ABAP Cloud-compatible patterns where applicable.

Tooling:
  ADT for repository objects, ATC for checks, service preview/API client for OData, transport tools for DEV -> QAS -> PRD.

这能帮助读者把“一个上传按钮”拆成项目语言。它不是单纯“写一个 ABAP 程序”，也不是单纯“做一个 OData API”；它跨越产品语义、平台能力、运行时、开发模型和交付工具。

3.8 Counterexample / Failure mode

  • 把 SAP S/4HANA、ABAP Platform、AS ABAP、ADT、ABAP Cloud 都叫成“SAP 平台”，会导致设计说明和 review 无法定位问题。
  • 以为 ABAP Platform 等于 SAP S/4HANA，会把业务产品能力和技术平台能力混在一起。
  • 以为 ABAP Cloud 等于云环境，会忽略它也可以作为开发模型出现在不同 ABAP 环境中。
  • 以为 ADT 是平台，会误判工具、repository、runtime 和 release 的关系。
  • 只按网上示例写代码，不检查系统 release 和 released API，可能导致语法不可用、ATC 不通过、Clean Core review 失败。
  • 修正动作：回到 MOC SAP System Foundations，把一个需求同时标注为 product / platform / runtime / tool / development model 五层。
  • 把 Public Edition、private cloud、on-premise、BTP ABAP environment 的开发边界混为一谈，会导致方案在目标系统不可实施。
  • 把 POC sandbox 里能跑的做法当成生产设计，会忽略 transport、authorization、ATC、released API、performance 和 operations。
  • 把 基盤 只理解成服务器/网络，会错过 ABAP Platform 作为开发与运行技术基础的含义。

3.9 思考问题

  • SAP S/4HANA 和 ABAP Platform 的区别是什么？
  • AS ABAP 和 ABAP Platform 的关系是什么？为什么不能完全等同？
  • ABAP Cloud 为什么更像 development model，而不是单独的产品名？
  • 为什么项目里要确认 release / software component，而不是只说“我们用 ABAP”？
  • 如果客户问 リリース済み API はありますか，这属于 product、platform、runtime、tool、development model 中哪一层的问题？
