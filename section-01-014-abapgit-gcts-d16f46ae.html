Chapter 01 - SAP 和 ABAP 基础

14. abapGit 与 gCTS 认知

  • Chapter 01
  • Section 14
  • self-contained study note

1) 一句话结论

abapGit 与 gCTS 这一节，要让 ABAP 初学者知道 SAP 世界也会使用 Git 风格工具，但这些工具不能替代 Repository object 仓库对象、ABAP package 包与子包、Activation 激活机制、Transport request 传输请求 和客户正式移送治理；本节要学的是：看到 Git 不等于可以绕过 SAP transport。

2) 到底学什么

前面已经学过：ABAP 对象不是普通文件夹里的 .py 或 .java 文件，而是 SAP system 中的 repository object；修改后要保存、激活、记录到 transport request，再通过受控的 transport / software logistics 流程进入 QAS/PRD。到这里再看 abapGit 和 gCTS，才不会把它们误解成“ABAP 终于和普通 Git 项目一样了”。

abapGit 是 ABAP 世界里最常见的 Git 入口之一。它可以理解为运行在 ABAP 侧的 Git client / import-export 工具，用来把 ABAP development objects 和 Git repository 之间做同步、导入、导出。SAP Help 对 SAP 版本的描述是：abapGit 是用于 ABAP server 的 Git client，可以在 ABAP systems 之间 import/export ABAP objects。社区版 abapGit 文档也强调：它是用 ABAP 写成的 Git client，适合代码共享、开源 ABAP 包、POC、迁移、cloud/on-premise 之间的代码转移、离线 ZIP 交换等场景。

但是 abapGit 不是“生产部署按钮”。它能把 Git repository 中的对象导入 SAP system，也可能创建/修改/删除 repository objects；导入后对象通常还要激活、处理 transport request、检查 activation errors、遵守命名空间、package、released API、Clean Core 和客户安全要求。尤其在日本项目里，直接对 PRD 做 abapGit pull 几乎一定会撞上审计、审批、责任分界和恢复方案问题。学习者要先形成这个判断：abapGit 更像开发协作和代码转移工具，不是替代 STMS / cTMS / Change Request / 承認フロー 的上线治理。

gCTS 的全称是 Git-enabled Change and Transport System。它是 SAP 把 Git 引入 ABAP change and transport management 的官方方向之一。和 abapGit 的“开发者导入导出工具”感受不同，gCTS 属于 SAP transport/change management 体系：它让 ABAP change and transport processes 使用 Git 作为 version management system，但仍然连接 transport request、object lock、software component、import queue、approval 和系统 landscape。

在 SAP BTP ABAP Environment 里，software component 是理解 gCTS 的关键入口。SAP Learning 说明：software component 可以和一个 Git repository、一个 transport layer 建立 1:1 关系；开发者仍然在 ADT 中开发，完成后 release transport request，变更会推送到关联 Git repository。这里有 SAP-managed gCTS 和 Bring Your Own Git 两种方向：前者由 SAP 管理底层 Git repository，开发者不直接接触；后者允许连接自己的 Git provider，例如 GitHub、GitLab 或 Bitbucket，但仍要遵守 ABAP Cloud 开发规则和 SAP consistency checks。

本节只要求 awareness，不要求会配置 gCTS。初学者要能问出五个问题：本项目使用 classical CTS、gCTS/software component、abapGit、cTMS/Cloud ALM/ChaRM，还是混合流程？代码从 DEV 到 QAS/PRD 的正式路径是什么？谁 release task / request？谁批准 QAS/PRD import？Git repository 里的历史记录是否能对应到 SAP transport log 和项目承認フロー？问得出这些问题，就已经避免了“我会 Git，所以我会 SAP 移送”的大坑。

3) 为什么重要

对 Java/Python 背景学习者来说，Git 是默认心智模型：branch、commit、pull request、merge、CI/CD、deploy pipeline。但 SAP 的核心交付对象不是单纯文本文件，而是 repository object + package + activation + transport + landscape。即使用了 Git，SAP 系统里仍然有 active/inactive version、object lock、transport request、import log、client、authorization 和 Basis/治理流程。

abapGit 很有价值，因为它把 ABAP object 变成可读的文本表示，让团队更容易做代码分享、开源库引入、POC 迁移、版本比较和跨系统转移。它也让 ABAP 开发者看到熟悉的 Git vocabulary。但越是熟悉 Git，越要小心：pull 一个 repository 不是普通 git pull 那么轻。它可能在 SAP system 里创建对象、覆盖对象、删除对象、触发 transport request prompt、产生 inactive objects 和 activation errors。项目是否允许、谁负责检查、能否进入客户代码库，不能由个人开发者临时决定。

gCTS 也很重要，因为它说明 SAP 并不是排斥 Git，而是在把 Git 纳入 ABAP change and transport governance。它可以连接 DevOps、CI/CD、software component、branch/tag、pipeline 和 transport management。但它不是“Git 分支魔法”。SAP 官方资料特别提醒：即使用 gCTS，ABAP Workbench 的 object locking 仍然存在，不能简单假设两个开发者可以像普通 Java 项目那样在同一个 ABAP system、同一个对象或同一个 package 上随意并行开发。并行开发、hotfix codeline、五系统 landscape 等设计，属于更后面的 DevOps / project governance 话题。

日本 SAP 项目通常重视证据链：移送依頼番号、承認フロー、移送ログ、インポートキュー、本番反映手順。如果学习者只说“代码在 Git 里”，但说不清 SAP 里哪个 package、哪个 transport request、是否有 activation error、是否通过 QAS test sign-off、谁批准 PRD import，就无法参与上线会、缺陷分析和审计说明。

4) Key mechanism / 核心机制

第一层机制是 abapGit 与 package。abapGit repository 会关联到目标 SAP system 中的 top/root package。这个 Git repository 本身不保存目标系统的 package 名；创建 online/offline repo 时，要指定或创建 SAP package。package 是否是 local package、是否有 transport layer、CTS 是否配置，会影响 pull / uninstall 时是否需要 local 或 transportable workbench request。换句话说，abapGit 不是绕过 package，而是必须落到 package。

第二层机制是 abapGit 与 transport。abapGit pull/import 可能在 SAP system 中创建、修改、删除 objects；这些动作若落在 transportable package 中，仍可能触发 transport request 选择。导入后的对象可能处于 inactive 状态，需要 activation；如果对象不支持、依赖缺失、命名空间冲突或源代码不符合 ABAP language version / released API 规则，就会产生错误。因此从 abapGit 导入之前，要先审查 remote repository、object list、package、namespace、license/security、Clean Core 影响和客户是否允许。

第三层机制是 abapGit 的适用边界。它适合开源 ABAP library、内部公共工具、学习 POC、系统间代码转移、on-premise 到 SAP BTP ABAP Environment 的迁移准备、离线 ZIP 交换、代码备份与 review 辅助。它不适合作为未经审批的 PRD deployment 机制。更安全的现场表达是：可以在 DEV / sandbox / dedicated development system 中使用 abapGit 辅助开发，然后按客户正式 transport process 推进到 QAS/PRD。

第四层机制是 gCTS 与 SAP transport governance。gCTS 把 ABAP change and transport management 与 Git repository 连接起来，但开发者仍然使用 ABAP tools 进行开发，仍然会遇到 object lock、request/task、activation 和 release。gCTS 的重点不是让 ABAP 摆脱 transport，而是把 Git 的 version management 能力纳入 transport governance。是否启用 gCTS，取决于系统版本、Basis 配置、landscape、客户 DevOps 策略和运维能力。

第五层机制是 software component。尤其在 SAP BTP ABAP Environment 中，software component 是开发和 transport 的组织层。一个 software component 可以对应 Git repository 和 transport layer；它决定包结构、依赖、API release 和跨系统传递路径。学习者不需要 Day 02 就会配置 Manage Software Components，但要知道：在 ABAP Cloud / BTP ABAP Environment 中，“这个 package 属于哪个 software component”往往比“这个目录在哪个 Git branch”更接近 SAP 项目现实。

第六层机制是 classical CTS 仍会存在。即使项目使用 gCTS，也可能有 classical CTS / Transport Organizer / STMS / ChaRM / Cloud ALM / cTMS 的混合流程。例如 SAP Note correction transport、Basis 维护、legacy on-premise 变更、customizing、客户规定的上线审批，都可能继续走 classical 或客户既定流程。不要在项目里说“有 gCTS 就不需要 CTS”；更准确的说法是“本项目的 transport governance 使用哪些工具组合，需要按客户手顺确认”。

第七层机制是 abapGit 与 gCTS 的边界。两者都和 Git 有关，但层级不同：abapGit 偏 ABAP object import/export 与代码交换；gCTS 偏 SAP change/transport lifecycle 与 Git-backed version management。在 SAP BTP ABAP Environment 中，gCTS 通常是 software component lifecycle 的推荐 transport mechanism；abapGit 仍可用于迁移、代码共享、POC 等特定场景。但不要在同一个 package 上混用 abapGit 与 gCTS，除非项目治理明确允许并有清楚的风险控制；初学者应默认避免。

5) 日本项目现场说法

  • 常见日语：移送管理、変更管理、移送手順、移送プロセス、承認フロー、Git連携、gCTS連携、abapGit連携、ソフトウェアコンポーネント、リポジトリ、ブランチ、コミット、プル、プッシュ。
  • 说明句：Git 管理を利用している場合でも、本番反映は正式な移送手順と承認フローに従う必要があります。
  • 说明句：このパッケージは abapGit 管理なのか、gCTS / ソフトウェアコンポーネント管理なのかを先に確認してください。
  • 使用场景：项目 onboarding、开发环境说明、开源代码引入评审、移送前确认、DevOps 方案讨论、Cloud / on-premise 混合 landscape 说明。

6) English Terms

  • Official / search term: abapGit, Git-enabled Change and Transport System, gCTS, Change and Transport System
  • BTP / Cloud terms: software component, Manage Software Components, SAP-managed gCTS, Bring Your Own Git, SAP Cloud Transport Management
  • Governance terms: transport request, transport layer, import queue, change request, approval flow, audit trail
  • Recommended search pattern: abapGit SAP Help, Git-enabled Change and Transport System SAP Help, SAP BTP ABAP Environment gCTS software component
  • 读官方资料时优先保留 gCTS、software component、transport request、Git repository、transport layer 等英文术语；这些词在 SAP Help / SAP Learning / ADT / Fiori apps 中会直接出现。

7) Example / 学习例子

  • 看一个开源 ABAP library 的 README，先不要导入系统，只回答：它要求哪个 ABAP release？支持哪些 object types？推荐放在哪个 package？是否需要 transport request？是否有 license/security 风险？
  • 问项目负责人：本项目的自定义 ABAP 代码是 classical CTS 管理、gCTS/software component 管理、abapGit 辅助管理，还是混合方式？上线时谁负责 import / 承認フロー？
  • 找一个 package，确认它是否是 $TMP / local package，是否有 transport layer，是否属于某个 software component。思考：如果从 abapGit pull 到这个 package，会不会要求 transport request？
  • 对比一次 Git commit 和一次 transport request release：Git commit 记录文本/对象序列化历史；transport request release 记录 SAP system 中的对象变更并进入移送路径。写出二者为什么不能互相替代。
  • Week 3/4 后，可以把一个 POC object 映射到：repository object -> package -> activation -> transport request -> Git/abapGit/gCTS 是否参与 -> QAS/PRD import evidence。

8) Counterexample / Failure mode

  • 误区一：以为“有 Git 就可以直接部署到 QAS/PRD”。结果缺少 SAP transport log、import log、approval evidence 和回退说明。
  • 误区二：把 abapGit 当成生产环境部署工具，在 PRD 上直接 pull。即使技术上可能，项目审计、稳定性、恢复性和客户运维责任通常都不允许。
  • 误区三：从 Git 导入代码前不审查 object list、namespace、package、license、released API、Clean Core 影响和 activation errors。
  • 误区四：以为 gCTS 等于普通 Git flow，忽略 ABAP object lock、transport request、software component 和系统 landscape。
  • 误区五：在同一个 package 上混用 abapGit 和 gCTS，造成对象来源、变更记录、transport 路径和责任边界不清。
  • 误区六：看到 BTP ABAP Environment / Public Edition 就套用 on-premise gCTS 配置教程。cloud 场景下 transport mechanism、Fiori apps、software component 和 provider-managed Git 细节可能完全不同。

9) 思考问题

  • abapGit 解决的是“代码交换/版本可见性/迁移辅助”问题，还是“正式上线审批/生产部署”问题？
  • gCTS 和 classical CTS 是替代关系、增强关系，还是项目治理里的组合关系？
  • 一个 ABAP package 如果属于 transportable layer，从 abapGit pull 对象时为什么仍可能需要 transport request？
  • 如果项目使用 gCTS，ABAP object lock 是否消失？为什么不能简单承诺“像普通 Git 一样随便并行开发”？
  • 在日本项目里，如何用一句日语说明“Git 管理不等于绕过移送承認”？
  • 哪些场景应该先问 Basis / project lead / security team，而不是自己导入 Git repository？
