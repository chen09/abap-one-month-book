29. Validation 校验

  • Chapter 03
  • Section 29
  • self-contained study note

29.1 一句话结论

Validation 校验 是 RAP business object 在保存前执行的业务一致性检查，用来判断一个 BO instance 是否允许进入持久化阶段；它不负责派生字段、不负责执行业务动作，也不负责最终保存，而是通过 failed / reported 把“哪里不合法、为什么不合法、消息应该显示在哪个字段上”交回 RAP runtime、OData consumer 和 Fiori UI。

一句话记忆：validation 是 RAP 的“保存闸门”。它不是前端输入框的简单必填提示，而是 BO 层对业务数据边界的正式检查。

29.2 概念说明

在 RAP 中，validation 是 behavior definition 中声明、behavior pool 的 handler class 中实现的一类检查逻辑。常见写法是在 BDEF 中声明：

define behavior for ZI_Travel alias Travel
{
  validation validateDates on save { field BeginDate, EndDate; }
}

然后在 behavior implementation 的 local handler class 中实现对应方法：

CLASS lhc_travel DEFINITION INHERITING FROM cl_abap_behavior_handler.
  PRIVATE SECTION.
    METHODS validate_dates FOR VALIDATE ON SAVE
      IMPORTING keys FOR Travel~validateDates.
ENDCLASS.

这段代码表达的是：当 Travel 进入保存流程时，RAP runtime 需要针对相关实例执行 validateDates 检查。检查失败时，不是抛出普通异常，也不是直接写数据库标记错误，而是把失败实例放进 failed，把面向用户或 consumer 的消息放进 reported。

初学者容易把 validation 理解成“字段必填”。这只是很小一部分。真实项目里的 validation 通常包括：

  • 日期区间是否合法，例如开始日期不能晚于结束日期。
  • 金额、数量、货币、单位是否成组有效。
  • 状态迁移是否合法，例如已取消订单不能再次提交。
  • 外键或主数据是否存在，例如客户、供应商、物料是否可用于当前公司代码。
  • 多字段组合是否满足业务规则，例如审批原因在特定状态下必须填写。
  • 当前 BO 与关联对象之间的一致性，例如 item 总额、header 状态、关联对象可用性。

Validation 的目标不是让 UI 看起来更友好，而是守住 BO 的业务边界。即使 consumer 不是 Fiori 页面，而是 OData API、批处理或 EML Entity Manipulation Language，同一套 validation 仍应工作。

29.3 为什么重要

没有 validation，RAP 服务很容易变成“只要字段能传进来就保存”。这在 demo 阶段看起来能跑，但进入项目后会迅速暴露问题：脏数据被写入业务表，Fiori 页面只显示笼统错误，OData consumer 不知道哪一行失败，测试无法稳定复现，后续报表和接口还要补救这些错误数据。

Validation 把业务规则放在 BO 层，而不是散落在前端、接口调用方、数据库触发器或 batch 程序里。这样同一个 Travel、Sales Order 或自建业务对象，不管从 UI、OData、EML 还是测试入口进入，都共享同一套保存前检查。

对项目交付来说，validation 直接影响缺陷归因。结合测试中常见的缺陷描述是：“保存时报错，但不知道哪一行哪一个字段错了。”写得好的 validation 会通过 reported 返回带字段定位的消息，例如把日期错误标到 BeginDate 或 EndDate，让 Fiori Elements 能把错误显示在正确位置；写得差的 validation 只返回一个笼统异常，用户和测试人员只能猜。

对 Java/Python 背景学习者，可以把 validation 类比成 domain model 的 invariant check 或 application service 的 save-time business rule。但 RAP 的差异是：validation 是 BDEF 契约的一部分，runtime 会把它纳入 save sequence、draft prepare、OData 错误响应、Fiori 消息显示和 EML COMMIT ENTITIES 的结果处理中。

在日本项目现场，validation 往往不是“开发者觉得应该检查一下”的附加逻辑，而是设计书、単体テスト、結合テスト、障害調査都会追问的交付对象。规则没有写在哪里、消息如何出、错误时是否阻止保存，都会影响验收。

29.4 边界

第一条边界：validation 不等于 Action 业务动作。Action 表达用户或 consumer 主动触发的业务动词，例如 approve、reject、cancel、submit；validation 表达保存前必须满足的规则。Action 内部可以让数据进入需要 validation 的状态，但不要把“保存时必须检查日期”包装成一个 action。

第二条边界：validation 不等于 Determination 派生逻辑。Determination 负责自动补全、派生或重新计算，例如根据 item 更新 header total；validation 负责判断数据是否允许保存。能自动修正的数据，优先考虑 determination；必须由用户或调用方改正的数据，通常应通过 validation 报错。

第三条边界：validation 不等于 authorization。Authorization 关注“谁可以做”，validation 关注“数据是否成立”。例如用户是否有权审批是 authorization；审批金额是否超过剩余额度、审批原因是否填写，是 validation 或 action 内的业务检查。不要用 validation 替代安全控制。

第四条边界：validation 不等于 feature control。RAP feature control 可以让某些字段、action 或 association 在当前状态下不可编辑、不可执行；validation 仍要在保存时兜底检查。前端按钮 disabled 不是完整保证，API consumer 也可能绕过 UI。

第五条边界：validation 不等于 precheck。RAP 里 precheck 更偏向在 modify 请求进入 transactional buffer 前做即时检查，常用于早期拒绝明显非法的 create/update/delete/action 输入；validation on save 更偏向保存前的 BO 一致性检查。项目里可以两者都用：precheck 提前挡住明显错误，validation 作为保存闸门保证最终一致性。

第六条边界：validation handler 不应直接 COMMIT WORK、直接持久化数据库或发送不可回滚外部请求。FOR VALIDATE ON SAVE 是 handler method，但它在 save sequence 的 check-before-save 相关阶段被触发；它的职责是检查和返回消息，不是承担 Saver class 保存器类 的最终持久化职责。

第七条边界：validation 不应只停留在 UI 必填。Fiori Elements 可以根据 metadata 和 annotation 提供字段必填、格式、value help 等前端体验，但保存时 business rule 仍应由 BO validation 兜底。外部 OData consumer、batch、EML 单元测试或未来新 UI 可能不经过同一个画面；如果规则只放在页面上，接口路径仍可能写入脏数据。

第八条边界：validation 的质量取决于消息质量。一个好的 validation 不只是“阻止保存”，还要让 consumer 知道哪一个实例、哪一个字段、哪条业务规则失败。能定位到字段的错误尽量绑定字段；跨字段或对象级规则则要明确业务含义。否则 Fiori 只能显示笼统错误，API 调用方只能收到难以处理的 generic failure，缺陷单也会变成“保存失败，原因不明”。

29.5 Key mechanism / 核心机制

第一层机制是 BDEF 声明。Validation 通常写在 define behavior for ... 内，并通过 on save 指明触发时机。大多数教学和项目场景先从 validation <name> on save { field ...; } 开始理解。field 列表不是消息字段位置本身，而是告诉 runtime：哪些字段变化与这个 validation 有关，便于触发和优化。

第二层机制是 handler method。BDEF 中的 validateDates 会在 behavior pool 里对应一个 FOR VALIDATE ON SAVE 方法。方法通常接收 keys，开发者再用 local EML 读取需要检查的字段：

METHOD validate_dates.
  READ ENTITIES OF ZI_Travel IN LOCAL MODE
    ENTITY Travel
    FIELDS ( BeginDate EndDate )
    WITH CORRESPONDING #( keys )
    RESULT DATA(travels).

  LOOP AT travels INTO DATA(travel).
    IF travel-BeginDate > travel-EndDate.
      APPEND VALUE #( %tky = travel-%tky ) TO failed-travel.
      APPEND VALUE #(
        %tky = travel-%tky
        %msg = new_message(
                 id       = 'ZTRAVEL'
                 number   = '001'
                 severity = if_abap_behv_message=>severity-error )
        %element-BeginDate = if_abap_behv=>mk-on
        %element-EndDate   = if_abap_behv=>mk-on
      ) TO reported-travel.
    ENDIF.
  ENDLOOP.
ENDMETHOD.

这段示例的重点不是消息类名称，而是三件事：

  • 用 IN LOCAL MODE 读取 RAP transactional buffer 中的当前状态，而不是绕过 RAP 直接查旧数据库状态。
  • 用 failed-<entity> 告诉 runtime 哪些实例不能保存。
  • 用 reported-<entity> 返回可展示消息，并用 %element-<field> 帮 UI 或 consumer 定位到字段。

第三层机制是 state message。SAP Help 对 RAP validation 的说明强调，validation 常与 state messages 配合。State message 表达“这个实例当前处于不合法状态”，在 draft 和 Fiori 场景尤其重要：用户可能先保存草稿，系统保留错误消息，之后用户修正字段再清除或替换消息。项目里要避免只返回一次性、无实例定位的 transient message，尤其是会影响后续修改体验的业务规则。

第四层机制是 save sequence。FOR VALIDATE ON SAVE 不在普通 action handler 那种即时 interaction 语境里完成最终业务提交，而是在保存流程中由 RAP runtime 触发。OData UI 的保存、EML 的 COMMIT ENTITIES、draft 的 Prepare / Activate 流程，都可能让 validation 被执行。理解这一点，才能避免在 validation 里做最终保存、发外部请求或依赖前端按钮顺序。

第五层机制是 draft。对于 RAP draft handling，validation 要考虑 active instance 与 draft instance 的体验差异。许多校验会在 draft Prepare 时提前执行，让用户在激活前看到错误；激活时仍要确保最终数据合法。初学者不要把 draft 当成“可以跳过 validation 的临时保存”，更准确的说法是：draft 允许用户分阶段编辑，但最终 active 数据仍必须过业务闸门。

第六层机制是测试入口。Validation 不只在 UI 点保存时测试。可以用 EML 准备非法数据，然后 COMMIT ENTITIES，检查 FAILED / REPORTED 是否包含预期实例、消息和字段定位。这样才能证明规则属于 BO，而不是只靠某个 Fiori 控件或前端校验。

29.Validation design review checklist

评审一条 RAP validation 时，可以先问“这到底是不是 validation”。如果规则是在保存前判断业务数据是否成立，例如日期区间、金额范围、主数据有效性、状态迁移一致性，它通常适合 validation。如果规则是在用户触发某个业务动词时改变状态，要先看 action；如果规则可以由系统自动补齐，要先看 determination；如果规则判断当前用户是否有权操作，要先看 authorization；如果规则只是为了在 modify 进入 buffer 前尽早拒绝明显非法输入，则可能更接近 precheck。

第二，确认触发条件是否清楚。BDEF 中的 validation ... on save { field ...; } 应该让读者看出这条规则和哪些字段变化有关。字段列表不是“错误字段清单”，但如果列表过宽，validation 会显得粗糙；如果列表漏掉关键字段，规则可能在某些修改路径下不容易被理解或维护。

第三，确认读取的是 RAP transactional buffer 中的当前状态。Validation 常需要用 READ ENTITIES ... IN LOCAL MODE 读取本次请求中的已修改数据。直接 SELECT 数据库表可能读到保存前的旧状态，特别是在同一请求中已经有 create/update、determination 或 action 修改过实例时，会导致“明明前端改了，validation 却按旧值判断”。

第四，确认失败表达完整。只有 failed 没有 reported，用户不知道为什么失败；只有 message 没有正确实例 key，consumer 很难定位哪一行错；字段级规则没有 %element-<field>，Fiori 或 API 调用方只能收到笼统错误。项目上应要求 validation 至少说明：哪一个 entity、哪一个实例、哪条规则、哪个字段或对象级位置失败。

第五，确认 message strategy 稳定。面向 UI 的文本可以友好，但 API consumer 还需要稳定的消息类、编号、severity、业务含义和可翻译文本。不要在 validation 中拼接不可治理的自由文本，也不要把多个业务错误压成一个 generic error。日本项目里这会直接影响障害調査、测试证据和多语言交付。

第六，确认 validation 没有副作用。Validation 可以读取、判断、返回 failed / reported，但不应提交事务、直接更新数据库、调用不可回滚外部接口、发送邮件或写入不可恢复日志。需要记录证据时，应区分“保存失败时也要保留的 application log”与“随事务一起回滚的业务数据”，并在 save sequence / logging 设计里说明。

第七，确认有跨入口测试。至少用一个 UI 保存或 service preview 场景证明用户能看到错误，再用 EML 或 API 测试证明非 UI consumer 也会被同一条 validation 拦住。只有画面测试通过，不代表 BO validation 已经可靠；只有 ABAP Unit/EML 测试通过，也要确认 OData/Fiori message mapping 对读者可解释。

29.6 日本项目现场说法

常见日语：

  • 検証：validation / check 的一般说法。
  • 保存時チェック：on-save validation，保存时检查。
  • 業務チェック：业务规则检查。
  • 入力チェック：输入检查，但要注意它可能被误解成前端输入框校验。
  • エラーメッセージ / 項目単位のメッセージ：错误消息 / 字段级消息。
  • 下書き保存、有効化、Prepare：draft 场景下的保存、激活和准备动作。

说明句：

このチェックは画面側の入力チェックではなく、RAP BO の保存時 validation として実装します。

意思是：这个检查不是画面侧输入检查，而是作为 RAP BO 的保存时 validation 实现。

エラー時は failed / reported に対象インスタンスと項目単位のメッセージを返す方針です。

意思是：发生错误时，按方针把目标实例和字段级消息返回到 failed / reported。

设计评审时可以这样问：

この業務ルールは action 内のチェックですか、それとも保存時 validation として共通化しますか。

意思是：这个业务规则是 action 内的检查，还是要作为保存时 validation 共通化？

结合测试缺陷分析时常见表达：

保存時にどの validation が発火して、どの項目にメッセージが紐付いているかを確認してください。

意思是：请确认保存时触发了哪个 validation，以及消息绑定到了哪个字段。

29.7 English Terms

  • Official / search term: RAP validation
  • ABAP syntax terms: validation ... on save, FOR VALIDATE ON SAVE, failed, reported, %msg, %element-<field>, state message
  • Recommended search pattern: ABAP RAP validation on save failed reported
  • 读官方资料时优先保留英文对象名、关键字、类名、annotation 名称。

29.8 Example / 学习例子

假设有一个 Travel BO，字段包括 BeginDate、EndDate、CustomerID、BookingFee、TotalPrice。业务规则是：

  • BeginDate 不能晚于 EndDate。
  • CustomerID 必须存在并且可用。
  • BookingFee 不能为负数。
  • 当 Travel 状态为 Submitted 时，关键字段不允许再被改成不一致状态。

较好的 RAP 设计通常是：

  1. 在 BDEF 中声明多个小 validation，例如 validateDates、validateCustomer、validateAmount。
  2. 在 handler class 中用 local EML 读取 transactional buffer 中的当前数据。
  3. 对每条非法实例 append 到 failed-travel。
  4. 对每个错误 append 到 reported-travel，消息尽量带 %element-<field>。
  5. 在 EML 或 OData 测试中验证非法数据无法 COMMIT ENTITIES，并且消息能定位到字段。

一个简化的思考过程：

用户把 BeginDate 改成 2026-07-10，EndDate 仍是 2026-07-01。
RAP transactional buffer 中先有了这个修改。
保存时 validateDates 被触发。
handler 读取当前 buffer，发现 BeginDate > EndDate。
handler 把该 Travel 放入 failed-travel，并在 reported-travel 中返回错误消息。
runtime 阻止该实例保存，Fiori/OData consumer 显示错误。

这就是 validation 的本质：它不一定阻止用户“输入”，但会阻止非法业务状态“保存为正式结果”。

在 AC0567C POC 或类似批量上传场景中，可以把 validation 思维迁移到上传 API：文件行解析成功不代表业务数据可入库。行级错误应清楚说明哪一行、哪个字段、哪条规则失败；这和 RAP failed / reported 的思想一致。Week 4 复盘 POC 时，可以检查批量导入是否也有类似“结构错误、主数据错误、业务状态错误、保存失败消息”的分层。

29.9 Counterexample / Failure mode

反例一：只在 Fiori 前端做必填或日期检查。这样页面上看似没问题，但 OData API、batch、EML 测试仍可能写入非法数据。修正方式是：前端可以做即时提示，但 BO 级 validation 必须兜底。

反例二：在 validation 中直接 UPDATE ztable 或 COMMIT WORK。这会绕开 RAP transaction，导致 runtime 认为只是检查，实际数据库已经被改。修正方式是：validation 只做读取、判断、返回 failed / reported；最终持久化交给 managed runtime 或 saver。

反例三：把所有业务规则都写成一个巨大 validate_all。短期省事，长期难测试、难定位、难复用，消息也容易笼统。修正方式是按业务规则拆成小 validation，命名表达规则，例如 validateDates、validateCustomer、validateAmount。

反例四：只返回 failed，不返回 reported。保存确实失败了，但用户不知道为什么失败。修正方式是失败实例和消息成对设计，至少给出可理解的错误文本；字段相关错误尽量使用 %element-<field>。

反例五：把 authorization 错误写成 validation。结果是无权限和数据不合法混在一起，审计和安全 review 都不好判断。修正方式是授权走 RAP authorization，业务数据规则走 validation。

反例六：draft 场景中只在 activate 时检查，导致用户编辑很久之后才看到大量错误。修正方式是结合 Prepare / state message，让用户在草稿阶段尽早看到可修正的问题，但最终激活仍执行必要校验。

29.10 思考问题

  • 这条规则应该是 validation、action 内部检查、determination、precheck，还是 authorization？判断依据是什么？
  • 错误消息应该绑定到实例、字段，还是只作为全局消息返回？
  • 如果同一 BO 同时被 Fiori、OData API 和 EML 使用，validation 如何保证三者一致？
  • Draft 场景下，这条规则应在 Prepare 阶段提示，还是只在 Activate / final save 阶段阻止？
  • 在日本项目里，设计书应写成 画面入力チェック、業務チェック 还是 保存時 validation？
  • 用 EML 写一个非法输入测试时，预期 FAILED 和 REPORTED 应该长什么样？
