Chapter 02 - Core ABAP Programming

30. Interface 接口定义

  • Chapter 02
  • Section 30
  • self-contained study note

30.1 一句话结论

Interface 接口定义是 OO ABAP 中用来表达“这个对象承诺提供哪些能力”的抽象契约。它不关心具体 class 如何实现，只规定调用者可以依赖哪些 method、type、constant 或 event。class 用 INTERFACES 声明自己实现某个 interface；调用者可以用 TYPE REF TO if_xxx 持有 interface reference，只依赖 contract，不依赖具体实现类。

初学者最重要的判断是：interface 不是“把 class 再包一层”，而是为了把稳定契约和可替换实现分开。它特别适合 constructor injection、ABAP Unit test double、外部依赖隔离、Clean Core 依赖边界和 framework contract。接口设计得好，class 可以换实现，测试可以换替身，调用方仍然只看同一份 contract。

30.2 到底学什么

上一节 Constructor 构造方法 讲了 constructor injection：把依赖通过 constructor 传给对象。本节讲的是这个依赖最好长什么样。答案通常不是具体 class，而是 interface。

本节要掌握这些边界：

  • INTERFACE ... ENDINTERFACE 定义一个 interface contract。interface 没有 DEFINITION addition，也没有 implementation part。
  • INTERFACE ... ENDINTERFACE 是声明层面的结构，不能写在 class、method、form、event block 这类局部处理块里面。
  • interface 里没有 PUBLIC SECTION / PRIVATE SECTION / PROTECTED SECTION；interface components 对实现方和调用方来说都是 public contract。
  • interface 可以声明 method、type、constant、attribute、event，甚至嵌套 interface；初学阶段先把 method contract 学稳。
  • INTERFACES if_xxx. 表示 class 实现这个 interface，通常写在 implementing class 的 PUBLIC SECTION。
  • interface method 只声明 signature；实现 class 必须实现 interface 中声明的所有 methods，否则语法检查不会通过。
  • 实现 interface method 时写完整限定名：METHOD if_xxx~method_name. ... ENDMETHOD.。这里的 ~ 不是装饰符，而是区分“interface component”和“class 自己的同名 component”的必要写法。
  • TYPE REF TO if_xxx 表示 interface reference，调用者只知道“这个对象能做什么”，不知道也不需要知道“它具体是哪一个 class”。
  • 一个 class 可以实现多个 interfaces；ABAP class inheritance 是单继承，interface 常用于补足“多个能力 contract”的表达。
  • interface 适合表达 capability，不适合塞进所有方法变成“大而全契约”。
  • PUBLIC interface 或 global interface 不自动等于 SAP released API；Clean Core / ABAP Cloud 仍要看 release contract 和 package/use access。

最小例子：

INTERFACE lif_discount_reader.
  METHODS get_rate
    IMPORTING iv_customer_id TYPE string
    RETURNING VALUE(rv_rate) TYPE decfloat34.
ENDINTERFACE.

CLASS lcl_fixed_discount_reader DEFINITION FINAL.
  PUBLIC SECTION.
    INTERFACES lif_discount_reader.
ENDCLASS.

CLASS lcl_fixed_discount_reader IMPLEMENTATION.
  METHOD lif_discount_reader~get_rate.
    rv_rate = '0.10'.
  ENDMETHOD.
ENDCLASS.

lif_discount_reader 说的是“我能根据 customer id 返回 discount rate”。lcl_fixed_discount_reader 是一种实现。以后可以有 database reader、customizing reader、test double reader，只要它们都实现同一个 interface，调用者就不必改 contract。

30.3 为什么重要

interface 是 ABAP 项目里把代码从“能跑”推进到“能替换、能测试、能长期维护”的关键工具。

没有 interface 时，业务 service 可能直接依赖一个具体 class：

DATA(lo_reader) = NEW lcl_db_discount_reader( ).

这会让调用者知道太多实现细节：依赖来自数据库、具体类名是什么、测试时也必须处理同一个 class 的副作用。后续想换成 customizing reader、mock reader、remote API reader，就会牵动很多调用方。

有 interface 后，调用者只依赖能力：

DATA lo_reader TYPE REF TO lif_discount_reader.
lo_reader = NEW lcl_fixed_discount_reader( ).
DATA(lv_rate) = lo_reader->get_rate( iv_customer_id ).

真实项目里，interface 带来四个价值：

  • 稳定 contract：调用者只依赖 method signature，不依赖具体实现。
  • 可测试：ABAP Unit 可以用 local test double 实现同一个 interface。
  • 可替换：生产实现可以从 DB reader 换成 API reader，调用方不需要知道。
  • 可治理：Clean Core / package boundary / released API 可以围绕稳定 interface 设计，而不是让调用方直接依赖内部 class。

Java/Python 背景学习者可以把它类比为 Java interface、Python protocol / duck typing 的显式版本。但 ABAP 是静态类型语言，interface reference、method signature、activation、where-used、transport、package dependency 都是显式存在的项目资产。

30.4 Key mechanism / 核心机制

30.4.1 interface defines contract

interface 先定义“调用者可以期待什么”：

INTERFACE lif_tax_calculator.
  METHODS calculate_tax
    IMPORTING iv_net_amount    TYPE decfloat34
              iv_tax_code      TYPE string
    RETURNING VALUE(rv_tax)    TYPE decfloat34.
ENDINTERFACE.

这段代码没有说税率来自哪里，也没有说如何四舍五入。它只说：任何实现 lif_tax_calculator 的对象，都必须提供 calculate_tax 这个能力。

interface 定义很像 class definition 的 public contract 部分，但它不是 CLASS ... DEFINITION 的缩小版。INTERFACE 语句本身不写 DEFINITION，也没有单独的 implementation part；method body 一定出现在 implementing class 的 CLASS ... IMPLEMENTATION 里。interface 也不写 visibility sections，因为 interface components 本质上就是公开 contract。

在语法位置上，INTERFACE ... ENDINTERFACE 是 program global context 的定义，不能嵌套在 class、method、form、function module 或其他 processing block 里面。局部 interface 通常放在 global class 的 local types area 或 report/program 的全局定义区；global interface 则是独立 repository object。

interface 里可以声明 method，也可以声明 type、constant、attribute、event、nested interface 等 public contract。初学阶段先把 method contract 学稳：输入、输出、异常、命名和边界要清楚。

30.4.2 class implements interface

class 用 INTERFACES 声明实现：

CLASS lcl_simple_tax_calculator DEFINITION FINAL.
  PUBLIC SECTION.
    INTERFACES lif_tax_calculator.
ENDCLASS.

CLASS lcl_simple_tax_calculator IMPLEMENTATION.
  METHOD lif_tax_calculator~calculate_tax.
    rv_tax = iv_net_amount * '0.10'.
  ENDMETHOD.
ENDCLASS.

lif_tax_calculator~calculate_tax 中的 ~ 是 interface component selector。它告诉读者：这个 method 是 interface contract 的实现，而不是 class 自己随便定义的普通 method。

这里有两个硬规则要记住。第一，INTERFACES lif_tax_calculator. 应放在 class 的 PUBLIC SECTION，因为 interface component 的可见性不能被实现类改成 private 或 protected。第二，只要 class 声明实现 interface，就要实现 interface 中声明的所有 methods，也就是 must implement all interface methods；少一个 method implementation，activation 前的语法检查就会报错。

如果调用者持有 interface reference，可以这样调用：

DATA(lo_tax_calculator) = NEW lcl_simple_tax_calculator( ).
DATA(li_tax_calculator) = CAST lif_tax_calculator( lo_tax_calculator ).

DATA(lv_tax) = li_tax_calculator->calculate_tax(
  iv_net_amount = '100.00'
  iv_tax_code   = 'A1' ).

在很多场景里，变量一开始就声明成 interface reference 更清楚：

DATA li_tax_calculator TYPE REF TO lif_tax_calculator.
li_tax_calculator = NEW lcl_simple_tax_calculator( ).

把 implementing class reference 放进 interface reference，是安全的 upcast：lcl_simple_tax_calculator 实现了 lif_tax_calculator，所以它一定能被当作“会算税的对象”使用。反过来，从 interface reference 转回具体 class reference 是 downcast，只有当运行时对象真的属于目标 class 时才成立。需要访问具体 class 的额外能力时，才考虑：

IF li_tax_calculator IS INSTANCE OF lcl_simple_tax_calculator.
  DATA(lo_simple) = CAST lcl_simple_tax_calculator( li_tax_calculator ).
ENDIF.

如果代码经常需要 downcast，通常说明 interface 设计得不够完整，或者调用方其实不应该只依赖这个 interface。interface reference 的好处正在于：它故意只暴露 interface 上声明的 components，隐藏具体 class 的其他 public methods。

30.4.3 interface reference decouples caller

调用方使用 interface reference 时，只能看到 interface 上声明的能力：

METHODS constructor
  IMPORTING io_tax_calculator TYPE REF TO lif_tax_calculator.

这比传入 TYPE REF TO lcl_simple_tax_calculator 更稳定。前者说“我需要一个会算税的东西”；后者说“我必须使用这个具体 class”。在项目 review 里，前者通常更利于测试和替换。

30.4.4 constructor injection uses interface

把 interface 和 Constructor 构造方法 连起来，就得到常见的 constructor injection：

CLASS lcl_price_service DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS constructor
      IMPORTING io_tax_calculator TYPE REF TO lif_tax_calculator.

    METHODS calculate_gross
      IMPORTING iv_net_amount       TYPE decfloat34
                iv_tax_code         TYPE string
      RETURNING VALUE(rv_gross)     TYPE decfloat34.

  PRIVATE SECTION.
    DATA mo_tax_calculator TYPE REF TO lif_tax_calculator.
ENDCLASS.

CLASS lcl_price_service IMPLEMENTATION.
  METHOD constructor.
    mo_tax_calculator = io_tax_calculator.
  ENDMETHOD.

  METHOD calculate_gross.
    DATA(lv_tax) = mo_tax_calculator->calculate_tax(
      iv_net_amount = iv_net_amount
      iv_tax_code   = iv_tax_code ).
    rv_gross = iv_net_amount + lv_tax.
  ENDMETHOD.
ENDCLASS.

lcl_price_service 不知道税额来自数据库、配置表、固定规则还是 test double。它只知道：mo_tax_calculator 遵守 lif_tax_calculator contract。

30.4.5 test double implements same interface

ABAP Unit 中可以写一个 local test double：

CLASS lcl_tax_calculator_double DEFINITION FINAL.
  PUBLIC SECTION.
    INTERFACES lif_tax_calculator.
ENDCLASS.

CLASS lcl_tax_calculator_double IMPLEMENTATION.
  METHOD lif_tax_calculator~calculate_tax.
    rv_tax = '5.00'.
  ENDMETHOD.
ENDCLASS.

测试里把这个 double 传给 service：

DATA(lo_service) = NEW lcl_price_service(
  io_tax_calculator = NEW lcl_tax_calculator_double( ) ).

这样测试不需要真实数据库，也不需要复杂 customizing。interface 让“生产实现”和“测试实现”共享同一份 contract。

30.4.6 interface is not inheritance

interface 和 Inheritance 继承 都能带来多态，但语义不同：

  • interface：表达 capability contract，重点是“谁能做这件事”。
  • inheritance：表达 is-a / specialization 关系，重点是父类与子类之间的扩展关系。

ABAP class inheritance 是单继承：一个 class 不能同时继承多个 superclass。但一个 class 可以实现多个 interfaces，例如同一个 service 同时承诺 lif_tax_calculator 和 lif_audit_writer 两种能力。这是 interface 在项目设计里非常实用的原因。

初学阶段更保守的建议是：如果只是为了让调用方能替换实现、隔离依赖、写 test double，优先考虑 interface；不要为了复用几行代码就引入 inheritance。代码复用可以用小 class、小 method、composition；抽象契约才是 interface 的主要价值。

30.4.7 global and local interfaces

接口可以是 global repository object，例如项目里的 ZIF_TAX_CALCULATOR，也可以是 local interface，例如本节示例里的 lif_tax_calculator。

常见命名：

  • Global interface：ZIF_...、/ABC/IF_... 或项目命名规范要求的前缀。
  • Local interface：LIF_...。

global interface 是独立 ABAP repository object，可以在 ADT 或 SE24 中管理，属于 package，随 transport request 迁移，也能被 where-used、package dependency、release contract、Clean Core 检查追踪。ZIF_...、/ABC/IF_...、IF_... 这类命名是项目或 SAP 命名约定，不是语法要求，但对团队阅读非常重要。

local interface 不是独立 repository object。它定义在某个 program、global class 的 local types area 或 class pool 的局部上下文中，只服务当前上下文，并随包含它的对象一起激活和传输。不要把 demo 用的 local interface 直接等同于项目对外稳定 API。

30.4.8 interface and Clean Core

Clean Core 语境下，interface 只是表达 contract 的语言工具，不自动代表“已释放可用”。一个 SAP standard interface 或项目 global interface 是否能被依赖，要看 released API、package interface、use access、language version 和项目治理。

所以 review 时要区分两句话：

  • “这个 class 实现了某个 interface，所以语法上可以通过 interface reference 调用。”
  • “这个 interface 是 released / allowed / stable API，所以项目上可以长期依赖。”

第一句是 OO ABAP 语法问题，第二句是 Clean Core / 架构治理问题。

30.4.9 interface design checklist

在 code review 里，interface 设计可以用一张表快速收口。重点不是“有没有 interface”，而是“这个 interface 是否表达了稳定、可替换、可测试的能力”。

QuestionGood signalRisk signal
What capability does it express?名字像 lif_tax_calculator、lif_discount_reader，能力清楚名字像 lif_service、lif_processor，什么都能塞
Is the contract small?少量 method，围绕同一能力giant interface，几十个 method 混在一起
Are types stable?参数类型具体、语义清楚、可被调用方长期理解全部 string / ANY，或暴露临时内部结构
Can implementations vary?DB reader、API reader、test double 都能实现同一 contract只有一个实现，且 interface 完全复制 class public methods
Can tests replace it?ABAP Unit 能写 local test double调用方仍然 new 具体 class 或 static call
Is it allowed as dependency?global interface 有 package / release / Clean Core 证据误以为 PUBLIC 或 ZIF_ 就自动 released

一个实用判断是：如果 interface 只是把某个 class 的所有 public methods 原样搬出来，它通常没有提供真正抽象；如果 interface 让调用方从“依赖具体实现”变成“依赖一项清楚能力”，它才开始发挥价值。接口越靠近项目边界、包边界、released API 边界，越要控制大小和稳定性；接口越只是 local test seam，越要保持简单，不要过度设计。

30.5 日本项目现场说法

  • 常见日语：インターフェース、抽象契約、実装クラス、依存性注入、テストダブル、差し替え可能性、疎結合。
  • 设计说明句：呼び出し側は具象クラスではなく、インターフェース型に依存する設計にしています。
  • Review 指摘句：このサービスは具象クラスを直接生成しているため、テストダブルに差し替えにくくなっています。
  • Clean Core 指摘句：このインターフェースがリリース済み API かどうかを確認してから依存関係として利用します。
  • 使用场景：class 设计、constructor injection 设计、ABAP Unit、test double、package dependency review、Clean Core/released API review。

日本项目里 インターフェース 也可能指外部系统接口、IDoc/RFC/OData 等 integration interface。本节讲的是 OO ABAP interface。现场沟通时要根据上下文说清楚：OOのインターフェース、外部IF、システム間インターフェース 不是同一个层次。

30.6 English Terms

  • Official / search terms: INTERFACE, INTERFACES, ABAP interface, interface reference, interface component, interface implementation, polymorphism, test double
  • Related syntax: INTERFACE lif_xxx, ENDINTERFACE, INTERFACES lif_xxx, METHOD lif_xxx~method, TYPE REF TO lif_xxx, CAST lif_xxx( ... )
  • Review terms: contract, abstraction, implementation class, interface-typed dependency, dependency inversion, constructor injection, released interface, upcast, downcast
  • 读官方资料时注意：interface 在 SAP 文档里也可能表示 integration/API/interface technology；检索 OO ABAP 语法时加上 ABAP Objects 或 INTERFACES statement。

30.7 Example / 学习例子

30.7.1 例 1：业务能力接口
INTERFACE lif_payment_authorizer.
  METHODS is_allowed
    IMPORTING iv_user_id        TYPE string
              iv_amount         TYPE decfloat34
    RETURNING VALUE(rv_allowed) TYPE abap_bool.
ENDINTERFACE.

这个 interface 只说“能判断某用户是否允许支付某金额”。它不说权限来自哪里。

30.7.2 例 2：生产实现
CLASS lcl_role_authorizer DEFINITION FINAL.
  PUBLIC SECTION.
    INTERFACES lif_payment_authorizer.
ENDCLASS.

CLASS lcl_role_authorizer IMPLEMENTATION.
  METHOD lif_payment_authorizer~is_allowed.
    rv_allowed = xsdbool( iv_amount <= 1000 ).
  ENDMETHOD.
ENDCLASS.

真实项目里可能会结合 Authorization check 权限检查、角色、DCL 或 RAP authorization；本例只演示 interface 结构。

30.7.3 例 3：测试替身
CLASS lcl_authorizer_double DEFINITION FINAL.
  PUBLIC SECTION.
    INTERFACES lif_payment_authorizer.
ENDCLASS.

CLASS lcl_authorizer_double IMPLEMENTATION.
  METHOD lif_payment_authorizer~is_allowed.
    rv_allowed = abap_true.
  ENDMETHOD.
ENDCLASS.

这个 double 可以让 ABAP Unit 专注测试业务计算，不依赖真实权限配置。

30.7.4 例 4：服务依赖接口而不是具体类
CLASS lcl_payment_service DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS constructor
      IMPORTING io_authorizer TYPE REF TO lif_payment_authorizer.

    METHODS can_pay
      IMPORTING iv_user_id        TYPE string
                iv_amount         TYPE decfloat34
      RETURNING VALUE(rv_allowed) TYPE abap_bool.

  PRIVATE SECTION.
    DATA mo_authorizer TYPE REF TO lif_payment_authorizer.
ENDCLASS.

CLASS lcl_payment_service IMPLEMENTATION.
  METHOD constructor.
    mo_authorizer = io_authorizer.
  ENDMETHOD.

  METHOD can_pay.
    rv_allowed = mo_authorizer->is_allowed(
      iv_user_id = iv_user_id
      iv_amount  = iv_amount ).
  ENDMETHOD.
ENDCLASS.

lcl_payment_service 只依赖 lif_payment_authorizer。生产代码传 lcl_role_authorizer，测试代码传 lcl_authorizer_double。

30.8 Counterexample / Failure mode

30.8.1 误区 1：给每个 class 都建一个 interface

interface 有成本。没有替换需求、没有测试替身、没有稳定边界，只是机械创建 zif_xxx，会增加文件和认知负担。

修正动作：有多实现、可替换依赖、跨 package contract、ABAP Unit 替身或 Clean Core 边界时优先考虑 interface。

30.8.2 误区 2：接口太大

一个 interface 里塞几十个 method，会让所有实现类都被迫承诺太多能力。

修正动作：按 capability 拆小接口，例如 reader、writer、validator、formatter，而不是一个万能 service interface。

30.8.3 误区 3：接口暴露不稳定内部类型

interface method 参数如果直接使用临时结构、具体实现类、内部缓存类型，调用方就被内部实现绑住。

修正动作：interface signature 使用稳定 DDIC type、public type、released API type 或项目稳定 value object。

30.8.4 误区 4：有 interface 但仍直接 new 具体类

service 内部直接 NEW lcl_db_reader( )，即使这个 class 实现了 interface，也没有真正解耦。

修正动作：把 interface reference 通过 constructor、factory 或 framework context 传入，调用方依赖 interface，创建逻辑集中管理。

30.8.5 误区 5：把 OO interface 当外部接口

OO interface 是代码 contract；OData/RFC/IDoc/API 是系统间接口或 integration interface。二者会有关联，但不是同一个概念。

修正动作：文档里明确说 OO ABAP interface、OData API、外部IF，避免把抽象层级混在一起。

30.8.6 误区 6：把 public interface 当 released API

能看到一个 public global interface，不代表它是 ABAP Cloud / Clean Core 允许依赖的 released API。

修正动作：查 released object 状态、package/use access、SAP Help、SAP API Business Hub 或项目架构标准。

30.8.7 误区 7：通过 interface reference 后又立刻 downcast

如果调用方拿到 TYPE REF TO lif_xxx 以后马上 CAST zcl_concrete_xxx( ... )，说明调用方并没有真正依赖抽象契约，而是在绕回具体实现。

修正动作：确认缺少的方法是否应该进入 interface contract；如果只是某个实现类的额外细节，就不要让通用调用方依赖它。

30.9 思考问题

  • 这个 interface 表达的是一个清楚 capability，还是只是某个 class 的影子？
  • 调用方是否真的只需要 interface contract，而不需要知道具体实现类？
  • 这个 interface 是否能让 ABAP Unit 使用 test double？
  • 如果明天从 DB reader 换成 API reader，调用方是否需要改？
  • interface method signature 是否稳定，是否暴露了内部临时类型？
  • 这个 interface 是 local demo contract、项目内部 contract，还是 released API？
  • 在日本项目里，应说 OOのインターフェース，还是 外部IF？
  • 哪一个 SAP Help 页面可以证明 INTERFACE / INTERFACES 的语法？
