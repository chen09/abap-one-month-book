Chapter 02 - Core ABAP Programming

2. SELECT statement 查询语句

  • Chapter 02
  • Section 02
  • self-contained study note

2.1 一句话结论

SELECT 是 ABAP SQL 里读取数据的核心语句。学它不是背一条语法模板，而是学会把“从哪里读、读哪些列、按什么条件读、结果放到哪里、顺序是否稳定、没有结果怎么办、权限和事务在哪里处理”一次性想清楚。项目里很多 bug 并不是因为开发者不会写 SELECT，而是因为写出了“能跑、但业务语义不稳定”的 SELECT。

2.2 到底学什么

本节专讲读取语句本身。上一节 ABAP SQL overview 已经建立了 ABAP SQL 的大图；本节开始进入最常见的读数动作。后续 Host variables 宿主变量 会专讲 @，INTO vs INTO TABLE 结果接收 会专讲结果接收，ORDER BY 排序确定性 会专讲排序。本节先把一条 SELECT 的骨架讲完整。

现代 ABAP SQL 里，推荐读者先熟悉这种结构：

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status,
              created_at
       WHERE customer_id = @lv_customer_id
         AND status      = @lc_status_open
       ORDER BY created_at DESCENDING,
                sales_order_id
       INTO TABLE @DATA(lt_orders).

这条语句可以按五段读：

  • FROM zsales_order：从哪个 ABAP SQL data source 读取。
  • FIELDS ...：读取哪些列，形成什么样的结果行。
  • WHERE ...：用什么业务条件过滤。
  • ORDER BY ...：如果结果顺序有业务意义，明确排序。
  • INTO TABLE @DATA(lt_orders)：把多行结果接收到 ABAP 内表。

老 ABAP 程序里还会看到 SELECT field1 field2 FROM ... INTO ... WHERE ... 这样的排列方式。初学者要能读懂旧代码，但写新代码时优先采用现代、清晰、ADT 友好的写法，尤其在 ABAP Cloud 或严格语法检查语境里更要遵守现代 ABAP SQL 习惯。

2.3 为什么重要

SELECT 是业务逻辑的入口。销售订单、客户、物料、库存、会计凭证、配置表、状态表，最终都要被某种读取逻辑取出来。只要读取条件错了，后面的判断、消息、更新、接口响应、测试结果都会跟着错。

它的重要性体现在四个方面。

第一是业务正确性。WHERE customer_id = @lv_customer_id 和没有 WHERE 完全不是一个风险等级；SELECT SINGLE 用完整主键读取，和用非唯一条件随便取一行，也不是同一件事。业务上说“取该客户最新一张未完成订单”，技术上就必须出现客户条件、状态条件和稳定排序。

第二是性能。FIELDS *、弱 WHERE、循环里反复 SELECT、无意义排序、把大量数据读到应用服务器再筛，都可能在开发机上没感觉，在 QAS 或 PRD 数据量上爆炸。性能不是后期才考虑的“调优”，而是每条读取语句的基本素养。

第三是权限和合规。SELECT 本身不是授权设计。能写 SQL 不代表用户能看业务数据；能查到表不代表这个表适合在当前 Clean Core / ABAP Cloud 语境下直接访问。读取逻辑要和 CDS Access Control DCL 行级访问控制、RAP global authorization 全局授权、AUTHORITY-CHECK 经典授权检查 分工理解。

第四是可审查性。日本 SAP 项目 review 中，SELECT 经常被问得非常具体：検索条件は十分ですか、ソート順は保証されていますか、全件検索になっていませんか、権限チェックはどこですか、本番データ量で問題ありませんか。如果这些问题回答不上来，代码即使能激活也很难说“可交付”。

2.4 Key mechanism / 核心机制

SELECT 的核心机制可以拆成六个判断点。

第一，数据源。FROM 后面可以是 ABAP SQL 支持的数据源，例如 DDIC database table、classic view、CDS entity，或者特定语法下的 internal table。初学者先不要把它理解成“物理数据库表名”，而要理解成 ABAP repository 和数据模型世界里的对象。特别是在 S/4HANA、RAP、Clean Core 场景中，优先确认是否有 released CDS view、released API、RAP BO 或项目自有持久化对象。

第二，结果列。FIELDS 决定结果集的结构。FIELDS * 在 demo、临时调查、表结构很小且上下文明确时可能方便，但业务代码和服务代码应优先选取需要的列。这样更容易 review，也减少数据库传输、ABAP 内存占用和后续字段误用。

第三，过滤条件。WHERE 是正确性和性能的共同入口。条件里的 ABAP 变量需要用 host variable 写法，例如 @lv_customer_id。这样能清楚区分 ABAP 变量和 SQL 标识符，也符合现代 ABAP SQL 语法。

第四，结果接收。单行结果可以进入 work area、结构或 inline data object；多行结果通常进入 internal table。INTO TABLE @DATA(lt_orders) 是初学者最常用、最容易读的多行接收写法。注意：inline 内表默认是 standard table 且 empty key，后续如果要频繁按 key 读取，需要再设计表类型。

第五，结果顺序。数据库不会因为你“上次看到是这个顺序”就保证下一次还是这个顺序。没有 ORDER BY 的多行结果，不要拿来做“第一条”“最后一条”“最新一条”的业务判断。需要顺序，就写出排序字段；需要唯一 tie-breaker，就把主键或业务唯一字段补进排序。

第六，系统字段和异常路径。SELECT 会设置 sy-subrc 和 sy-dbcnt，但具体含义要看语句形态。典型多行 INTO TABLE 后，sy-dbcnt 可用于确认读取行数；单行读取后，sy-subrc = 0 通常表示读到行，sy-subrc <> 0 表示没有读到。学习时要连接 ABAP system fields sy fields，不要把 sy-subrc 检查写散、写晚，或被后续语句覆盖。

2.5 日本项目现场说法

常见日语：

  • SELECT文
  • 検索条件
  • WHERE条件
  • 取得項目
  • 取得対象
  • 検索結果
  • 全件検索
  • 単一行取得
  • 複数行取得
  • ソート順
  • 取得件数
  • 存在チェック
  • 主キー検索
  • 権限チェック

现场说明句：

  • この SELECT文 は、主キー検索なので SELECT SINGLE で問題ありません。
  • この条件では一意にならないため、SELECT SINGLE ではなく、業務上の ソート順 を明示した取得にしてください。
  • 本番データ量を考えると、この WHERE条件 では絞り込みが弱すぎます。
  • 取得項目は必要な列だけに絞り、FIELDS * は避けます。
  • この SELECT文 はデータ取得のみであり、権限チェックは別レイヤーで確認します。
  • 存在チェックだけであれば、全項目を取得する必要はありません。

日本项目里，“全件検索” 是一个很敏感的词。它不一定永远错误，但如果 PRD 数据量大、没有分页、没有最大件数、没有业务过滤条件，就会被当成性能和稳定性风险。

2.7 Example / 学习例子

多行读取

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status,
              created_at
       WHERE customer_id = @lv_customer_id
         AND status      = @lc_status_open
       ORDER BY created_at DESCENDING,
                sales_order_id
       INTO TABLE @DATA(lt_orders).

IF sy-subrc <> 0.
  " No open orders found for this customer.
  RETURN.
ENDIF.

这类写法适合“取一组业务记录”。重点不是 SELECT 能不能跑，而是条件、列、排序和空结果路径是否表达了业务意思。

完整主键单行读取

SELECT SINGLE FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status
       WHERE sales_order_id = @lv_sales_order_id
       INTO @DATA(ls_order).

IF sy-subrc <> 0.
  " The requested order does not exist.
  RETURN.
ENDIF.

如果 sales_order_id 是主键或业务上唯一的键，这种 SELECT SINGLE 很自然。review 时要能说明“为什么唯一”。

按业务排序取一条

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status,
              created_at
       WHERE customer_id = @lv_customer_id
         AND status      = @lc_status_open
       ORDER BY created_at DESCENDING,
                sales_order_id DESCENDING
       INTO TABLE @DATA(lt_latest_orders)
       UP TO 1 ROWS.

如果业务语义是“最新一条”，不要用非唯一条件下的 SELECT SINGLE 假装它会知道“最新”。要把 ORDER BY 写出来，再限制一行。SAP Samples 的 ABAP SQL cheat sheet 也提示：当 INTO 放在最后时，UP TO n ROWS 放在 INTO TABLE ... 后面。

只做存在性检查

SELECT SINGLE @abap_true
       FROM zsales_order
       WHERE customer_id = @lv_customer_id
       INTO @DATA(lv_exists).

IF sy-subrc = 0.
  " At least one row exists.
ENDIF.

如果只是问“有没有”，就不需要把整行或大量列读出来。存在性检查要尽量小、清楚、可解释。

2.8 Counterexample / Failure mode

失败模式一：没有条件或条件太弱。

SELECT FROM zsales_order
       FIELDS *
       INTO TABLE @DATA(lt_orders).

这可能在开发机只读几十行，但生产里读几十万行。除非这是明确的后台统计、迁移、分页或受控批处理逻辑，否则 review 会追问为什么全件读取。

失败模式二：非唯一条件下滥用 SELECT SINGLE。

SELECT SINGLE FROM zsales_order
       FIELDS sales_order_id,
              created_at
       WHERE customer_id = @lv_customer_id
       INTO @DATA(ls_order).

如果一个客户有多张订单，这段代码没有说明取哪张。数据库返回的某一行不是业务规则。要么补完整唯一条件，要么用 ORDER BY ... UP TO 1 ROWS 表达业务优先级。

失败模式三：只看 sy-subrc，不看业务路径。没有读到数据可能是正常情况，也可能是配置缺失、权限过滤、数据同步延迟、输入错误。好的代码会把“没有结果”的业务含义写清楚，而不是统一吞掉。

失败模式四：把 SELECT 当授权。SELECT 成功不等于用户有权。特别是在直接读表、后台程序、API、RAP unmanaged 逻辑里，权限检查要有明确位置。

失败模式五：为了省事 FIELDS *。字段多、包含大字段、后续结构变化或服务输出变化时，* 会让代码更难维护。初学阶段可以在 demo 里看懂 *，但项目代码要养成“取需要的列”的习惯。

2.9 思考问题

  • 这条 SELECT 的数据源是什么对象？DDIC table、CDS entity、view、internal table，还是不该直接访问的底层表？
  • FIELDS 里的每个字段后续都真的会用到吗？
  • WHERE 是否表达了业务范围？是否存在全件读取风险？
  • ABAP 变量是否都用了 @ 作为 host variable？
  • 如果只读取一行，唯一性从哪里来？
  • 如果读取多行，是否有稳定 ORDER BY？
  • 读取为空时，业务含义是什么？
  • 权限检查在哪里？是不是误以为 SELECT 自动处理了业务权限？
  • 如果后续要更新数据，锁和事务边界在哪里？
  • 在 ABAP Cloud / Clean Core 语境下，这个数据源是否允许直接访问？

2.12 学习任务

  • 写一个多行 SELECT FROM ... FIELDS ... WHERE ... ORDER BY ... INTO TABLE ...，并解释每一段的作用。
  • 写一个完整主键 SELECT SINGLE，说明为什么它是唯一读取。
  • 把一个非唯一条件的 SELECT SINGLE 改成 ORDER BY ... INTO TABLE ... UP TO 1 ROWS。
  • 找一段 FIELDS *，判断它是 demo、调查代码，还是应该改成明确字段列表。
  • 写一句日语 review 说明：この SELECT文 は検索条件が弱く、全件検索になる可能性があります。
  • 用 SAP Help 的 SELECT, SINGLE 和 SELECT, UP TO, OFFSET 页面确认“取一行”的两种不同语义。
