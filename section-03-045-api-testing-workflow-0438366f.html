45. API testing workflow 接口测试流

  • Chapter 03
  • Section 45
  • self-contained study note

45.1 一句话结论

API testing workflow 接口测试流 是把 RAP/OData 服务从“preview 能打开”推进到“可以交付”的固定验证流程。它要求把 metadata、成功请求、失败请求、权限、并发、重复请求、错误 payload、目标系统验证和证据归档都做成可重复动作，而不是临时点几下页面。

一句话记忆：API 测试不是证明“能通一次”，而是证明“按契约成功、按契约失败、换用户换系统仍可解释”。

在一本实战书里，API testing workflow 要回答的不是“用 Postman 还是 curl”，而是“怎样让接口交付可被信任”。一个 RAP/OData 服务在 ADT 里 preview 成功，只说明服务入口大概率能打开；它没有证明业务契约正确、权限正确、错误可被消费、重复请求安全、移送后配置一致，也没有证明出现障害时团队能复现问题。

因此，本节把接口测试看成一条证据链：从 $metadata 到请求集合，从 happy path 到 negative path，从开发者用户到业务用户，从 DEV 到 QAS，从 response body 到 backend log。读者学完以后，应该能拿任何一个 OData/RAP API，写出一张可执行的测试矩阵，而不是只说“我点过，能通”。

45.2 到底学什么

前一节 Service preview 服务预览 解决“服务是否活着”。本节解决“服务是否可交付”。在真实项目中，接口测试不能只靠 Fiori preview 或一次 GET 成功。你需要固定记录：调用者是谁、调用哪个系统、URL 是什么、metadata 是否正确、payload 是什么、返回码是什么、错误长什么样、重复调用会怎样、权限不足会怎样、QAS/PRD 是否一致。

API testing workflow 不是某个工具名。Postman、Bruno、curl、HTTPie、SAP Gateway Client、ADT preview、自动化测试都可以成为工具。关键是流程：先确认 contract，再跑 happy path，再跑 negative path，再跑权限和并发，再保存证据。

本节的目标，是让初学者以后看到任何 OData/RAP 服务，都能按同一套清单验证，而不是只凭经验临时测试。

API 测试至少要覆盖三种层次。第一层是技术连通性：URL、authentication、CSRF token、headers、content type、metadata、service binding 或 Gateway endpoint 是否可用。第二层是业务契约：哪些 entity/action/function 可以调用，字段含义、key、status code、message、side effect 是否符合设计。第三层是项目可交付性：谁测的、在哪个系统测的、用什么用户、什么数据、什么时间、证据放在哪里，移送后如何复验。

本节也要建立一个边界：API testing workflow 不等于完整自动化测试平台。它可以用手工工具开始，也可以逐步沉淀成 collection、script、CI smoke test 或验收证据。初学阶段先把测试矩阵和证据格式做对，再考虑工具自动化。否则很容易出现“工具很专业，但测试内容只覆盖一次 GET”的假进步。

一套最小可交付 API 测试矩阵通常包括：

类别要证明什么常见证据
Contract$metadata、entity、key、action、property 与设计一致metadata 片段、OpenAPI/接口说明、版本
Connectivity目标系统 endpoint 可访问，认证方式正确URL、status code、headers
Happy path正常读写/action 返回正确业务结果request/response、DB 或 status 后验
Negative path非法输入、状态不允许、对象不存在时可解释失败OData error payload、business message
Authorization不同用户/角色/technical user 的结果符合授权用户、role、SU53/trace 或拒绝响应
ConcurrencyETag、If-Match、锁、并发更新语义正确412/409 等冲突证据、前后状态
Idempotency重试不导致重复副作用重复请求、日志/状态/业务表后验
EnvironmentDEV/QAS/PRD 配置差异被复验系统、client、publish/role/destination 记录

书稿里建议把每个关键 API 用例都沉淀成一个最小 evidence packet，而不是只保存截图。最小 evidence packet 至少包含：系统/client、用户或 technical client、service binding / endpoint、HTTP method、URL、关键 headers、request payload、status code、response body、业务后验、backend log 或 trace 入口、测试时间、测试人、未覆盖项。这样 defect、review、受入测试和障害調査可以复用同一份材料。

还要提醒读者：HTTP status code 不能孤立解释。比如 403 可能是业务角色缺失，也可能是 communication arrangement、OAuth scope、DCL/RAP authorization、CSRF token 或 endpoint 暴露问题；412 可能是 ETag/If-Match 并发控制；400 可能是 payload 格式，也可能是 RAP validation message。好的 API 测试不是只写“返回 400”，而是把 response payload、业务语义和后端证据连起来。

45.3 为什么重要

API 测试流程重要，是因为接口问题通常不是“完全不能用”，而是“某个边界不可解释”。开发者自己用户能调用，业务用户没权限；GET 能成功，POST 因 CSRF/ETag/authorization 失败；正常数据能保存，validation failure 的 error payload 前端解析不了；DEV 能通，QAS publish 或 role 缺失；第一次调用成功，重复请求生成重复数据。

对项目交付来说，接口测试证据是设计、开发、联调、验收和障害調査之间的共同语言。没有证据，问题会变成“我这边没问题”“你那边再试试”。有证据，就能定位到 request、response、用户、系统、时间、错误码、backend log 和业务规则。

对 Java/Python 背景学习者，API testing workflow 类似 contract test + integration test + negative test 的组合。但 SAP OData/RAP 还要特别关注 $metadata、entity set、action/function、authorization、DCL/RAP auth、CSRF/ETag、draft、save sequence、Gateway/RAP runtime log 和 transport 后配置。

这也是为什么 API 测试是 Week 3 到 Week 4 的桥。Week 3 学 RAP/OData 的建模和暴露，Week 4 学 authorization、transaction、log、test、performance、Clean Core。真正的接口验收会把这些全部连起来：一个 POST 失败，可能是 validation；一个 PATCH 被拒绝，可能是 ETag；一个 GET 少数据，可能是 DCL；一个 action 重复执行，可能是 idempotency；一个 QAS 不通，可能是 service binding、role、communication arrangement 或 transport 后缓存问题。

对团队交付来说，好的 API 测试证据能减少“口头排查”。例如缺陷单里写“API 失败”价值很低；写“QAS client 100，user Z_API_TEST_02，PATCH /Travel('4711')，header If-Match 使用过期 ETag，返回 412，response payload 如附件，Application Log 编号 BAL-xxx，无 DB 更新”，团队立刻知道该看并发控制而不是泛泛检查网络。

45.4 Key mechanism / 核心机制

第一步是 contract check。打开 $metadata，确认 entity set、property、key、navigation、action/function 是否符合设计。若 metadata 已经不对，后面的调用成功也只是偶然。

第二步是 authentication / identity check。确认调用者是谁：开发者用户、业务用户、技术用户、communication user、OAuth client、principal propagation。不同身份看到的数据和权限可能完全不同。

第三步是 happy path。至少测试读取列表、读取单条、创建/更新/action 中的核心成功路径。每个请求都记录 method、URL、headers、payload、status code、response body、业务解释。

第四步是 negative path。故意构造非法字段、缺少必填、状态不允许、主数据不存在、权限不足、对象不存在、并发冲突、重复请求。好的 API 不只会成功，也要可解释地失败。

第五步是 transaction / idempotency check。对于 RAP action、save sequence、外部副作用、日志和 outbox，要测试重复点击、重试、部分失败、rollback 后是否残留数据。尤其是前面学过的 COMMIT WORK restriction in RAP 和 Direct INSERT 反模式，都要通过 API 测试暴露出来。

第六步是 environment check。DEV、QAS、PRD 的 binding publish、role、communication arrangement、destination、ICF/OData runtime、metadata cache 可能不同。至少在移送后做一次目标系统 smoke + contract + 权限验证。

第七步是 evidence archive。把测试结果保存成可复用证据：请求集合、截图、response JSON、metadata 片段、错误日志编号、测试用户、系统、时间、结论和未覆盖风险。

可以把 workflow 固定成下面的顺序：

1. Identify service and scope
2. Capture contract
3. Prepare identities and test data
4. Execute happy path
5. Execute negative path
6. Execute authorization patterns
7. Execute concurrency / retry / idempotency checks
8. Cross-check backend evidence
9. Repeat target-system smoke test after transport
10. Archive evidence and open defects for gaps

每一步都要有输入和输出。比如 contract check 的输入是 service binding URL 和设计说明，输出是 $metadata 片段和接口字段确认；authorization check 的输入是用户/role 矩阵，输出是不同行为的 response 和后端授权证据；idempotency check 的输入是同一业务键或 idempotency key，输出是重复执行后的状态、日志和副作用记录。

45.5 日本项目现场说法

  • APIテスト：API testing。
  • 正常系：happy path / normal case。
  • 異常系：negative/error case。
  • 権限パターン：authorization pattern。
  • テスト証跡：test evidence。
  • リクエスト / レスポンス：request / response。
  • エラーペイロード：error payload。
  • 再実行 / リトライ：rerun / retry。

プレビューだけではなく、正常系、異常系、権限パターンの API テスト証跡を残してください。

意思是：不要只做 preview，请留下正常路径、异常路径、权限模式的 API 测试证据。

障害調査のため、リクエスト URL、ペイロード、レスポンス、ユーザ、実行時刻を記録してください。

意思是：为了故障调查，请记录 request URL、payload、response、用户和执行时间。

同じリクエストを再実行した場合に二重登録されないか確認してください。

意思是：请确认同一个请求重复执行时是否不会重复登记。

更像项目邮件或测试说明的写法：

$metadata、正常系、異常系、権限パターン、再実行時の結果を同一証跡セットとして保管します。

含义：把 metadata、正常路径、异常路径、权限模式和重复执行结果作为同一套证据保存。

QAS 移送後に、サービスバインディング、ロール、通信設定、メタデータキャッシュを含めて疎通確認を実施します。

含义：QAS 移送后，连同 service binding、role、communication setting、metadata cache 一起做连通确认。

異常系では HTTP ステータスだけでなく、OData エラーペイロードと業務メッセージも確認対象にしてください。

含义：异常路径不仅确认 HTTP status，也要确认 OData error payload 和业务消息。

リトライ時に外部副作用やアプリケーションログが重複しないことを後続確認します。

含义：重试时要后验确认外部副作用和 application log 不重复。

现场表达要尽量避免“APIテスト完了です”这种空泛结论。更好的结论是“正常系 3 件、異常系 4 件、権限 2 パターン、再実行 1 パターンを確認済み。未確認範囲は draft 編集競合と外部連携タイムアウトです”。这能让读者养成报告覆盖范围和残余风险的习惯。

45.7 Example / 学习例子

一个最小 API 测试表可以这样写：

Service: ZUI_TRAVEL_O4
System: DEV
User: TRAVEL_TEST_01
Metadata: /.../$metadata captured

TC01 Read list
  GET /Travel?$top=5
  Expected: 200, list returned

TC02 Read single
  GET /Travel('<key>')
  Expected: 200, key fields match

TC03 Execute action
  POST /Travel('<key>')/AcceptTravel
  Expected: 200/204, status changes or action result returned

TC04 Validation failure
  PATCH /Travel('<key>') with invalid BeginDate
  Expected: 4xx or OData error payload with field/business message

TC05 Authorization failure
  Same request with unauthorized user
  Expected: denied, no database change

TC06 Retry / duplicate request
  Repeat action request
  Expected: no duplicate log, no duplicate external side effect

测试后要保存 request/response，而不是只写“OK”。如果错误发生，要记录 backend evidence：RAP message、Gateway log、dump、authorization trace、application log 或业务表状态。

对 AC0567C POC，可以把 /validate、/execute、/status 做成三组测试：validate 只校验不落库；execute 成功后状态可查询；重复 execute 不应制造重复结果；status 对不存在 job id 要返回可解释错误。

把 AC0567C 写成矩阵会更清楚：

Test IDAPICaseExpected
AC-API-01/validate合法 payload返回可执行判断，不创建最终业务结果
AC-API-02/validate缺少必填字段返回业务错误 payload，错误字段可定位
AC-API-03/execute合法 payload创建 job/result，返回可追踪 id
AC-API-04/execute重复业务键或重复 idempotency key不产生重复副作用，返回既有状态或可解释拒绝
AC-API-05/status已存在 job id返回当前状态、消息和可追踪时间
AC-API-06/status不存在 job id返回可解释错误，不 dump，不泄漏内部表名
AC-API-07/execute无权限用户拒绝执行，无 DB 更新，无外部副作用
AC-API-08/execute部分数据失败明确成功/失败边界，日志能追踪每条记录

一个测试证据可以这样写：

TC: AC-API-04 Retry execute
System: QAS 100
User: Z_API_TECH_BATCH
Request 1:
  POST /execute
  Payload businessKey = BATCH-2026-0001
  Header Idempotency-Key = 8b7...
  Expected: 201/202, jobId returned

Request 2:
  POST /execute
  Same businessKey and same Idempotency-Key
  Expected: no duplicate job/result; response returns existing jobId or duplicate-safe message

Backend evidence:
  Status table has one active job for BATCH-2026-0001.
  Application Log has one business execution record plus optional retry audit record.
  No second external side effect was triggered.

这类例子让读者看到 API testing workflow 不是“点接口”，而是把接口行为、事务边界、幂等性、日志和业务后验串成一套检查。

45.8 Counterexample / Failure mode

反例一：只测 GET。读取成功，但创建、更新、action、错误路径全没测。修正方式是覆盖 read + write/action + negative。

反例二：只用开发者用户。开发者能通，业务角色不通。修正方式是测试业务用户和技术用户。

反例三：只测正常数据。非法日期、缺少主数据、状态不允许、重复请求都没测。修正方式是把 negative path 写成固定用例。

反例四：不保存 response。联调出错时没人知道当时返回了什么。修正方式是归档 response body、status code、headers。

反例五：DEV 通过就结束。QAS 缺 publish/role/communication arrangement。修正方式是移送后目标系统复验。

反例六：不测幂等。用户或外部系统重试后重复插日志、重复调用外部接口。修正方式是设计 retry test 和数据后验检查。

反例七：只看 HTTP 200。某些业务错误可能包装在 response body 里，也可能返回 4xx/5xx；只看 status code 不看 OData error payload、business message 和后端状态，会错过真正的契约问题。

反例八：忽略 headers。写操作常涉及 CSRF token、ETag、If-Match、content type、language、correlation id。headers 不记录，复现时很容易“同一个 URL，结果不一样”。

反例九：把开发工具请求当成生产调用。ADT preview、浏览器、Postman、外部系统、communication user、principal propagation 的身份和 headers 可能不同。preview 通过不代表外部系统通过。

反例十：没有后验检查。接口返回成功，但数据库状态、application log、outbox、业务状态或外部副作用不一致。API 测试不能只停在 response，还要看后端事实。

反例十一：异常路径太笼统。只写“invalid data”没有价值；要明确是缺少必填、格式错误、业务状态不允许、主数据不存在、授权不足、并发冲突还是重复请求。

45.9 思考问题

  • 这个 API 的 contract evidence 是 $metadata、OpenAPI 文档，还是测试集合？
  • 哪些请求证明 happy path？哪些请求证明 negative path？
  • 权限测试覆盖了哪些用户/角色/technical user？
  • 错误 payload 是否能被 UI 或外部系统稳定解析？
  • 重复请求是否安全？如果不安全，是否需要 idempotency key/outbox/status check？
  • 移送到 QAS 后，哪些证据必须重新采集？
  • AC0567C 的 validate/execute/status 分别需要哪些测试用例？
  • 对同一个 API，开发者用户、业务用户、technical user 的期望差异是什么？
  • 如果 UI 只说“保存失败”，后端 API 测试证据要怎样帮助定位问题？
  • 哪些测试必须在 transport 后重新跑，哪些可以只保留 DEV 证据？
