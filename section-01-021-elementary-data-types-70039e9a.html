Chapter 01 - SAP 和 ABAP 基础

21. Elementary data types 基础技术类型

  • Chapter 01
  • Section 21
  • self-contained study note

本节结论

Elementary data types 基础技术类型 是 ABAP 里描述“一个值应该如何存放、显示、比较和转换”的最小技术类型单元。学这一节不是为了背类型表，而是为了在声明 ABAP data object 数据对象 时知道：金额不能随手写成字符串，日期不是普通文本，N 不是可计算的数字，P 的小数位必须说清楚，错误的类型选择会一路影响 SQL、接口、测试和上线缺陷。

正文

在上一节里，数据对象被解释为程序运行时内存中的一块有类型的值。现在要继续追问：这个“类型”具体可以是什么？在 ABAP 入门阶段，最常见的答案就是 built-in elementary data types，也就是 ABAP 语言本身预定义的基础技术类型。SAP Learning 当前把这些用于简单 numeric、character-like、binary data object 的 built-in 类型概括为 14 个预定义数据类型；初学阶段不需要一次记死所有边角差异，但必须先建立分类。

第一类是 numeric types 数值类型。常见的是 i、int8、p、decfloat16、decfloat34 和 f。i 适合普通整数，int8 适合更大的整数；p 是 packed number，项目里经常用于带小数位的金额、数量等十进制业务值；decfloat16 和 decfloat34 是十进制浮点；f 是二进制浮点，适合特定科学/近似计算，不适合作为金额字段的默认选择。Java/Python 背景的人常把 float 当成“有小数的数”，但 SAP 项目里的金额、税额、汇率、数量通常需要精确小数、货币或单位语义，不能只看“能不能存下小数”。

第二类是 character-like types 字符型。c 是定长字符，string 是变长字符串，n 是 numeric text。这里最容易误解的是 n：它看起来像数字，其实是只允许数字字符的文本，适合保存编号、补零代码、外部 key 的一部分，不应当被当成算术数字来做金额或数量计算。

第三类是 byte-like types 字节型。x 是定长 byte field，xstring 是变长 byte string。它们面向二进制内容、hash、raw key、技术协议字段等场景。初学者不需要马上写复杂二进制处理，但要知道 byte-like 和 character-like 不是一回事：一个字节序列不等于一段业务文本，不能因为“看起来都是一串东西”就混用。

第四类可以先理解为 date/time/timestamp types 日期、时间和时间戳类型。d 是日期，t 是时间，utclong 可用于 long UTC timestamp / タイムスタンプ型。d 和 t 不是随便写的显示文本；它们有 ABAP 内部格式，输出格式再由环境、转换例程或 UI 层处理。日期、时间、时区和时间戳的完整规则会在后面的 SQL、接口和业务对象章节继续展开。

还要特别理解 complete type 与 incomplete type 的区别。像 i、int8、d、t、string、xstring、utclong、decfloat16、decfloat34 这类类型在常见 DATA ... TYPE ... 声明中已经足够完整；而 c、n、x、p 常常需要补充长度，其中 p 还要补充 DECIMALS 小数位。比如 DATA amount TYPE p LENGTH 8 DECIMALS 2. 比只写 TYPE p 更接近项目代码标准；不写 DECIMALS 时默认 0 位小数，金额或数量可能被悄悄变成整数语义。某些版本资料还会提到 b、s 这类短整数类型；它们属于 SAP 文档中的内置类型知识，但不能直接写在普通 DATA ... TYPE ... 声明里。初学写项目代码时先按官方学习路径掌握常用类型即可，不要为了“类型表完整”过早引入低频细节。

最后要把技术类型和业务语义分开看。TYPE p LENGTH 8 DECIMALS 2 只能说明这是一个 packed decimal 技术字段，并不自动说明它是 JPY、EUR 还是 USD，也不说明数量单位是 EA、KG 还是 H。真正进入 Data element 数据元素、Domain 域对象、数据库表、CDS、OData 和 RAP 时，金额通常需要 currency key，数量通常需要 unit，日期时间还要考虑时区和业务含义。这就是为什么本节放在 Day 03：先把单个值的技术形状看清楚，后面再把它放进 Dictionary、SQL、服务接口和业务对象。

为什么重要

  • 类型决定了初始值、赋值、比较、排序、格式化和隐式转换的结果；同一个外观看起来像 123 的值，放在 i、n、c、string、p 里，项目意义完全不同。
  • 金额、数量、税率、日期、时间、编号这些字段一旦类型选错，问题不会只停在当前程序，后续 SQL 查询、接口 JSON、Fiori 显示、测试数据、迁移数据都可能被拖下水。
  • ABAP 允许不少隐式转换，这让简单 demo 很容易“看起来能跑”；但真实项目里隐式转换可能造成截断、舍入、runtime error、short dump 或难以解释的测试差异。
  • 对 Java/Python 背景学习者来说，本节的关键不是寻找一一对应的 int、str、float，而是学会 SAP 项目里“技术类型 + 业务语义 + Dictionary/接口约束”一起判断。

核心机制

  • Built-in elementary type 是 ABAP 语言提供的基础技术类型；它不同于 DDIC data element/domain，也不同于结构、内表、引用类型。
  • Numeric types 负责算术语义，但不同 numeric type 的范围、精度、舍入方式不同；业务十进制值优先考虑 p、十进制浮点或 DDIC 业务字段，而不是默认 f。
  • Character-like types 负责字符表示；n 是 numeric text，适合补零编号，不等同于可以安全计算的 number。
  • Byte-like types 负责二进制数据；x/xstring 不应和文本类型混用。
  • c、n、x、p 这类不完整类型在声明时通常要明确 length；p 还要明确 DECIMALS，否则默认 0 位小数，后续金额/数量缺陷很隐蔽。
  • VALUE 是起始值，CLEAR 回到类型特定 initial value；初始值和起始值不是同一个概念。
  • 隐式类型转换虽然方便，但要当作风险点审查：是否会截断、丢小数、补空格、去尾空格、转不成数字或输出格式变化。

日本项目现场表达

  • 常见日语：基本データ型、組込みデータ型、技術型、数値型、文字型、バイト型、日付型、時刻型、タイムスタンプ型。
  • 声明相关：完全型、不完全型、LENGTH指定、DECIMALS句、小数点以下桁数、初期値、暗黙型変換。
  • 业务字段相关：パック数値、通貨項目、通貨キー、数量項目、単位、丸め誤差、桁あふれ。
  • 说明句：金額項目は単なる文字列ではなく、通貨キーと小数点以下桁数を含めて型を確認する必要があります。
  • Review 句：この項目は N 型なので数値計算には使わず、コード値またはゼロ埋め文字列として扱う前提でよいか確認してください。
  • 使用场景：基本設計の項目定義、詳細設計の型確認、コード review、単体テストデータ作成、移送前の影響調査。

例子

先看一段很小的 ABAP 代码：

DATA order_id     TYPE n LENGTH 10 VALUE '0000004711'.
DATA quantity     TYPE p LENGTH 8 DECIMALS 3 VALUE '12.500'.
DATA net_amount   TYPE p LENGTH 11 DECIMALS 2 VALUE '1980.00'.
DATA currency     TYPE c LENGTH 3 VALUE 'JPY'.
DATA created_on   TYPE d VALUE '20260613'.
DATA message_text TYPE string.

message_text = |Order { order_id } amount { net_amount } { currency }|.

这里 order_id 用 n，是因为订单号需要保留前导零，重点是“编号显示和比较”，不是做加减乘除。quantity 和 net_amount 用 p 并写出 DECIMALS，是因为数量和金额通常要保留固定小数位。currency 这里只是一个演示字段，在真实项目中更常见的是 DDIC currency key 或 CDS/OData 语义关联。created_on 用 d，表示这是 ABAP 日期技术类型，而不是一段任意格式的文本。message_text 用 string，因为它只是输出消息，不承担金额、日期或数量的业务计算语义。

最小练习：

  1. 在 ADT 里创建一个实现 IF_OO_ADT_CLASSRUN 的练习类。
  2. 分别把同一个值 19891109 放进 string、d、c LENGTH 8、n LENGTH 8、p LENGTH 8 DECIMALS 2。
  3. 用 out->write( ) 观察输出差异。
  4. 思考：哪些类型适合显示，哪些适合计算，哪些只是技术上“能赋值”但项目上不合理。

进阶练习：给一个金额字段写出三种错误声明，例如 TYPE string、TYPE f、TYPE p 但不写 DECIMALS，再说明每种写法可能造成什么测试缺陷。

常见误区

  • 把金额写成 string：界面上可能能显示，但排序、加总、舍入、接口校验和货币小数位都会变成隐患。
  • 把编号写成 i：0000004711 会失去前导零，和外部系统、表 key、用户看到的编号不一致。
  • 把 n 当成普通 number：它是 numeric text，不是为了业务计算设计的算术类型。
  • 把所有小数都写成 f：二进制浮点可能带来舍入差异，金额/税额场景尤其危险。
  • TYPE p 不写 DECIMALS：代码短了，但默认 0 位小数，金额或数量可能被悄悄截成整数语义。
  • 只看本地变量，不看 DDIC/CDS/OData：同一个技术类型到了接口层还要考虑 label、conversion、currency/unit、precision/scale 和消费端显示。

思考题

  • 如果一个金额字段被声明为 string，单元测试、SQL 排序、接口输出分别可能出现什么问题？
  • n LENGTH 10 和 i 都能表达数字字符时，为什么订单号更常见地接近 n 或 DDIC 字符语义？
  • p LENGTH 8 DECIMALS 2 的 length 和 decimals 分别控制什么？不写 DECIMALS 会有什么 review 风险？
  • d/t 和 string 最大的区别是什么？为什么日期时间不应该只按显示格式理解？
  • 在日本项目里看到 暗黙型変換、丸め誤差、桁あふれ 时，应该回到哪几类类型风险？
