14. Security Audit Log 安全审计日志

  • Chapter 04
  • Section 14
  • self-contained study note

14.1 一句话结论

Security Audit Log 安全审计日志 是 SAP 系统层面记录安全相关事件的审计证据入口；它帮助团队回答“谁在什么时间、从哪里、以什么身份，触发了哪些安全相关行为”。它和 Application Log 应用日志 不同：Application Log 主要解释业务处理过程和应用错误，Security Audit Log 主要服务于安全、合规、审计和异常访问调查。

初学 ABAP 时很容易把所有“日志”混成一个词。项目现场实际会区分多类证据：业务处理失败先看 Application Log；敏感数据读取可能看 Read Access Logging 读取访问日志；授权失败先结合 SU53 and authorization trace workflow；系统安全事件、异常登录、可疑远程访问、用户相关变更或高风险操作，则可能进入 Security Audit Log 的调查范围。

这一节不要求 ABAP 开发者自己配置安全审计策略，也不要求替代 Basis/security 团队做审计判断。要学的是：当接口、后台 job、技术用户、RFC/OData 调用或权限异常牵涉到安全问题时，开发者应该知道哪些信息要保留，哪些证据不能随意复制，什么时候要请 Basis/security 团队确认安全审计日志。

14.2 到底学什么

第一，要学 Security Audit Log 的定位。它属于 SAP security / Basis / audit 领域，不是普通应用日志。ABAP 开发者通常不负责最终配置，但必须知道它可能记录登录、授权、安全相关配置或敏感操作等事件，并且知道这些证据通常由有权限的安全或 Basis 角色查询。

第二，要学它常见记录什么。不同 SAP release、部署模式和配置会影响可记录事件范围；常见关注点包括 logon failure、successful or failed RFC/HTTP access、authorization-related events、user master 或 role 相关变更、security configuration changes、critical transaction/program access、remote access 等。具体项目不能凭记忆判断，必须以目标系统配置和官方文档为准。

第三，要学它不记录什么。Security Audit Log 通常不会告诉你“第 27 行代码为什么校验失败”，也不会替代业务处理明细。批处理处理了几条、哪一行失败、业务对象状态是什么，这些应由 Application Log、job log、API response 或业务表状态说明。安全审计日志关注的是安全事件本身。

第四，要学项目调查方式。开发者发现安全相关问题时，不应只说“程序有问题”。更好的做法是提供 investigation packet：系统/client、用户或技术用户、时间窗口、入口、业务对象、request id、application log id、SU53/trace 时间点、可疑操作描述。安全团队才能根据这些线索查询 Security Audit Log 或相关系统日志。

第五，要学证据最小化。Security Audit Log 可能包含用户、来源、时间、访问对象、安全事件等敏感信息。开发者不应把完整审计日志随意贴到聊天工具、issue 或测试报告里。通常应由授权人员截取最小必要证据，并在项目安全规则允许的范围内共享。

14.3 为什么重要

Security Audit Log 重要，是因为生产系统里的安全问题往往不能只靠代码解释。异常登录、权限失败、可疑技术用户调用、敏感操作、角色变更、远程访问，都需要系统层面的证据链。没有安全审计证据，团队只能停留在“看起来像权限问题”或“可能有人操作过”的猜测。

对 SAP 项目交付来说，它直接影响上线前检查、内部审计、合规响应和故障调查。尤其在金融、制造、医药、公共事业等客户项目中，安全日志、访问证据、权限设计和运维追踪通常是正式 review 的一部分。开发者如果不了解边界，就可能把安全事件误判为普通 bug，或把普通应用错误误报成安全事件。

对 ABAP 开发者来说，它的意义不是“多写一行日志”，而是知道证据链如何组合。例如用户说某个 API 被非法调用，开发者应先提供 API request id、application log id、technical user、client 和时间窗口；Basis/security 再判断是否需要查 Security Audit Log。这样才能避免把安全调查变成口头争论。

在日本项目里，这类内容常和 証跡、監査、アクセスログ、権限、セキュリティ担当、Basis担当 一起出现。现场表达通常强调“调查对象期間”“閲覧権限”“必要最小限の証跡共有”，而不是把完整日志无控制地扩散。

14.4 Key mechanism / 核心机制

Security Audit Log 的核心机制可以理解为：系统根据安全审计配置，把特定安全事件记录成审计证据；之后由有权限的人员按用户、client、时间、来源、事件类型等条件检索分析。它不是每个 ABAP 程序自动生成的业务日志，而是系统安全审计能力。

项目中要特别区分三层证据。

第一层是权限检查证据。AUTHORITY-CHECK 经典授权检查、CDS DCL、RAP authorization 决定用户能否执行某类操作；SU53 或 authorization trace 帮助定位权限检查为什么失败。Security Audit Log 可能提供安全事件证据，但它不替代权限设计本身。

第二层是应用处理证据。Application Log 记录业务过程、接口处理、后台任务或错误明细。它可以回答“处理到哪一步、哪条业务数据失败、失败原因是什么”。Security Audit Log 不应被拿来替代这类业务处理证据。

第三层是安全审计证据。Security Audit Log、Read Access Logging、系统变更记录、传输记录和角色变更记录共同形成安全与合规证据体系。项目文档中要说明哪类问题看哪类证据来源，不能一句“看日志”带过。

开发者最实用的做法，是在接口、后台 job 和重要服务中保留 correlation id / request id / application log id，并记录技术用户、业务键和时间窗口。这样当安全团队查询 Security Audit Log 时，可以把系统层事件和应用层处理关联起来。

可以使用下面的调查请求模板：

Security Investigation Request

Reported issue:
  What happened:

Scope:
  System/client:
  User or technical user:
  Time window:
  Entry point:
    SAP GUI transaction / Fiori app / OData / RFC / background job

  Program/service/job:

14.7 Example / 学习例子

假设生产中出现一个问题：某技术用户在短时间内大量调用一个 OData API，部分请求返回 authorization failure，业务团队怀疑是不是有人异常使用了接口。

不要马上把它写成“API bug”。更好的调查步骤是：

  • 从 API 或应用层拿到 request id、application log id、HTTP endpoint、technical user、client、时间窗口和业务键。
  • 用 Application Log 判断业务处理是否真的失败、失败在业务校验还是权限/认证前后。
  • 用 SU53 或 authorization trace 判断是否存在明确的授权对象失败。
  • 若怀疑登录、远程访问、技术用户异常、可疑来源或安全配置问题，请 Basis/security 查询 Security Audit Log。
  • 若涉及敏感数据读取，再判断是否需要 Read Access Logging 的证据。

在 AC0567C 的 batch upload API 场景中，可以这样组织证据：

AC0567C Security Evidence Packet

API layer:
  Endpoint:
  Technical user:
  Request id:
  HTTP status:

Application layer:
  Application log id:
  Business key:
  Error category:

Authorization layer:
  Business role:
  Failed operation:
  SU53 / trace timestamp:

Security audit layer:
  System/client:
  Time window:
  Event to check:
  Basis/security owner:

Decision:
  Business error / authorization error / security event / unclear:

这个例子说明：Security Audit Log 不是孤立使用的。它要和 Application Log、authorization trace、API evidence、时间窗口和用户信息拼在一起，才能形成可调查的证据链。

14.8 Counterexample / Failure mode

常见失败模式如下。

第一，把所有日志都叫 監査ログ。业务日志、应用日志、安全审计日志、访问日志不是同一个东西。混用术语会让调查方向错误。

第二，用 Application Log 代替 Security Audit Log。Application Log 可以说明业务处理失败，但不能证明系统安全事件是否发生。

第三，把安全审计日志完整复制到普通文档或聊天工具。审计日志可能包含敏感用户、来源、事件和系统信息，应该最小范围共享。

第四，没有时间窗口和上下文。只说“这个用户好像做了奇怪操作”，但不给 client、时间、入口、业务键、request id，安全团队很难查询。

第五，要求开发者直接查看完整审计日志。很多系统中 Security Audit Log 查询权限应限制给 Basis/security 或审计角色；开发者应提供线索，而不是越权获取日志。

第六，把日志调查当成权限设计。Security Audit Log 可以帮助调查安全事件，但不能替代 authorization object、DCL、RAP authorization 和测试矩阵。

第七，缺少 correlation id。API response、Application Log 和 Security Audit Log 时间窗口无法关联时，事故调查会依赖猜测。接口和后台任务应尽量保留 request id / job id / log id。
