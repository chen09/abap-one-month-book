Chapter 03 - CDS、RAP、OData

26. RAP feature control

  • Chapter 03
  • Section 26
  • self-contained study note

26.1 一句话结论

RAP feature control 是 RAP 用来控制“某个操作或字段此刻是否可用”的机制：它可以让 update、delete、draft Edit、自定义 action、甚至部分字段，根据业务对象状态或全局条件在 Fiori / OData 消费中显示为 enabled、disabled、read-only 等行为。

一句话记忆：authorization 问“你有没有权限”，feature control 问“这个功能现在该不该可用”。两者经常一起出现，但不能互相替代。

26.2 概念说明

Feature control 解决的是动态可用性问题。比如 Travel 已经 Accepted，就不应该再允许编辑、删除或再次执行 Accept action；Travel 已经 Rejected，就不应该再执行 Reject action；某些字段在特定状态下只读，防止用户改坏业务状态。RAP feature control 让这些规则进入 BO behavior，而不是只靠前端隐藏按钮。

ABAP Keyword Documentation 对 RAP feature control 的定义很直接：它用于启用或禁用 RAP BO operation，并有不同变体。SAP-samples RAP110 / RAP100 练习也把它用于标准操作 update / delete、draft action Edit 和自定义 action 的动态启停。也就是说，feature control 不是 UI 小技巧，而是 BO 行为契约的一部分。

初学者要先分清两组概念：

  • Static / design-time control：在 BDEF 里声明字段 readonly、mandatory、operation 是否存在等，规则不依赖单条实例状态。
  • Dynamic feature control：在运行时根据实例数据或全局条件返回可用性结果。

动态 feature control 又分两类：

  • Instance feature control：按每条 BO instance 的状态判断，例如某个 Travel 是 Open / Accepted / Rejected。
  • Global feature control：不依赖单条实例，而是按全局条件判断，例如系统开关、日期窗口、维护模式、业务配置。

26.3 为什么重要

没有 feature control，Fiori UI 可能会显示一堆“理论上存在但当前不该执行”的按钮。用户点了以后，后端 validation 再报错，体验差，测试也难解释。更糟的是，如果只在前端隐藏按钮，其他 OData / EML consumer 仍可能调用同样 action，造成行为不一致。

Feature control 把按钮、字段和操作的可用性放回 RAP BO 语义中。UI 可以根据 OData metadata / runtime response 展示正确状态；后台 action 也能和 BO 状态保持一致。它不是最后一道安全门，但能让“当前状态不该做什么”提前反馈给用户。

对项目交付来说，feature control 直接影响设计说明、测试用例和缺陷分析。比如“审批后按钮还可点”“草稿状态下按钮不该显示”“已关闭单据仍可删除”“字段应该只读但 UI 可编辑”，这些问题不一定是 UI annotation 错，也不一定是 authorization 错，可能就是 feature control 没声明或实现不完整。

对 Java/Python 背景学习者，可以把 feature control 类比成 domain object 的 command availability / state machine guard，再加上 UI metadata 的协作。但 RAP 的关键差异是：这个可用性由 BDEF 声明、behavior pool 实现，并通过 RAP runtime 暴露给 Fiori/OData/EML 语义。

26.4 边界

第一条边界：feature control 不是 authorization。用户没有权限执行 action，应由 RAP authorization、PFCG/IAM、authorization object 或实例授权处理。Feature control 可以让 action 在某个状态下 disabled，但不能替代真正的权限检查。安全设计里，二者要同时考虑。

第二条边界：feature control 不是 validation。Validation 是“你提交的内容是否有效”；feature control 是“这个操作/字段当前是否应该可用”。如果用户绕过 UI 调用 action，后端仍要用 authorization、validation、precheck 或 action logic 防止非法状态改变。

第三条边界：feature control 不等于 UI annotation。@UI annotation 决定字段如何展示、按钮如何布局、对象页如何组织；feature control 决定运行时可用性。按钮位置不对可能是 UI annotation；按钮可点不可点则常要看 feature control / authorization。

第四条边界：instance feature control 不适合做昂贵查询。get_instance_features 可能被 UI 频繁请求；实现里应读取必要字段，不要做复杂跨系统调用或大量数据库扫描。SAP-samples 的练习也只读取 TravelID 和 OverallStatus 来判断操作状态。

第五条边界：disabled 不是 hidden。Feature control 常见结果是 operation enabled/disabled、field read-only 等。UI 是否完全隐藏、置灰、只读，取决于 RAP 语义、OData metadata、Fiori Elements 支持和具体前端行为。写书稿时用“不可用/置灰/只读”比简单说“隐藏”更稳。

26.5 核心机制

第一层机制是 BDEF 声明。只有在 behavior definition 中把操作声明为受 feature control 管理，RAP runtime 才会生成相应 handler 方法并在运行时请求结果。常见写法包括：

define behavior for ZI_Travel alias Travel
{
  update ( features : instance );
  delete ( features : instance );

  draft action Edit ( features : instance );
  action ( features : instance ) acceptTravel result [1] $self;
  action ( features : instance ) rejectTravel result [1] $self;
}

这段是教学示意。具体语法受 ABAP release、draft/non-draft、projection behavior 和 strict mode 影响，实际项目以 ADT syntax check 和 SAP 文档为准。

第二层机制是 handler method。ADT 通常会为 entity 生成或提示添加 get_instance_features / get_global_features 这类方法。SAP developer tutorial 对 get_instance_features 的接口说明包括 keys、requested_features、result，以及 handler class 语境中可用的 failed、reported。也就是说，实现方法不是“随便返回一个布尔值”，而是按 RAP method signature 和隐式响应参数返回结构化 feature result。

第三层机制是只计算 requested features。requested_features 告诉实现当前 consumer 请求了哪些字段、operation 或 action 的可用性。高质量实现应先看 requested_features，只计算需要的内容，避免每次都算所有规则。这对复杂对象和大列表页很重要。

第四层机制是读取实例状态。Instance feature control 常在 handler 中用 EML local read 读取当前 BO 的关键字段：

READ ENTITIES OF zi_travel IN LOCAL MODE
  ENTITY Travel
  FIELDS ( OverallStatus )
  WITH CORRESPONDING #( keys )
  RESULT DATA(lt_travel)
  FAILED failed.

这里用 IN LOCAL MODE 是因为 feature control 属于当前 behavior implementation 语境，需要看到 transactional buffer 中的当前状态。直接 SELECT active table 可能读不到用户刚在草稿或事务缓冲中改过的状态。

第五层机制是返回 %features 或 operation 状态。SAP-samples 中常见结果是按实例 append 到 result，使用 if_abap_behv=>fc-o-enabled / if_abap_behv=>fc-o-disabled 控制 operation 或 action：

LOOP AT lt_travel ASSIGNING FIELD-SYMBOL(<travel>).
  APPEND VALUE #(
    %tky = <travel>-%tky
    %update = COND #( WHEN <travel>-OverallStatus = 'A'
                      THEN if_abap_behv=>fc-o-disabled
                      ELSE if_abap_behv=>fc-o-enabled )
    %delete = COND #( WHEN <travel>-OverallStatus = 'A'
                      THEN if_abap_behv=>fc-o-disabled
                      ELSE if_abap_behv=>fc-o-enabled )
    %action-acceptTravel = COND #( WHEN <travel>-OverallStatus = 'A'
                                   THEN if_abap_behv=>fc-o-disabled
                                   ELSE if_abap_behv=>fc-o-enabled )
  ) TO result.
ENDLOOP.

字段级 feature control 则常通过 %field-<fieldname> 返回字段状态，例如 if_abap_behv=>fc-f-read_only；其他常见值还包括 fc-f-mandatory 和 fc-f-unrestricted。字段、operation、action 的 result 结构不同，初学者不要硬套同一个 %action-... 写法。

第六层机制是 global feature control。Global feature control 用于不依赖单条实例的条件，例如系统维护开关、全局配置、当前日期是否在允许窗口、后台任务是否正在运行。BDEF 中声明 features : global 后，实现 get_global_features，返回全局可用性。一个极简锚点是 create ( features : global );，表示 create 是否可用由全局 feature control 决定。它适合“整个服务当前都不能创建/执行某 action”的场景。

第七层机制是 projection、association 和 draft。Feature control 可能涉及 draft action Edit、普通 action、create-by-association、projection behavior 暴露范围和 Fiori Elements 按钮状态。更进阶的 association feature control 可能通过 %assoc-<association_name> 影响关联创建可用性，本节只先建立入口概念。上一节 RAP draft handling 讲过，draft actions 是草稿生命周期的一部分；本节要记住，draft action 也可能需要 feature control，例如某个状态下不允许继续编辑。

26.6 项目意义

设计评审时，feature control 要回答：“这个按钮/字段/操作在什么状态下应该可用？”不要只写“审批按钮”，要写清楚 open 时可用、approved 时不可用、rejected 时不可用、draft 时是否可用、锁定时如何处理、无权限时是隐藏还是报错。

代码 review 时，重点看四类问题。第一，BDEF 是否声明了 ( features : instance ) 或 ( features : global )。第二，handler 是否实现了对应 get_*_features 方法。第三，是否用 READ ENTITIES ... IN LOCAL MODE 读取必要状态，而不是直接查表。第四，是否把 feature control 和 authorization / validation 的责任混在一起。

测试时，要覆盖 UI 和 API 两条路径。UI 上看按钮是否置灰、字段是否只读；API/EML 上还要确认即使 action 被调用，后端授权、validation 或 action logic 也不会让非法状态改变成功。Feature control 提升体验，但不能是唯一防线。

日本项目中，feature control 常出现在 Fiori 缺陷单里：承認済みなのに承認ボタンが押せる、却下済みなのに編集できる、項目が読取専用にならない。分析时不要直接说“UI bug”，要先确认 BDEF、behavior implementation、projection、authorization 和 Fiori metadata 是否一致。

26.7 Example / 学习例子

假设 Travel 有三个状态：Open、Accepted、Rejected。业务规则是：

  • Open：允许 update、delete、Accept、Reject。
  • Accepted：不允许 update、delete、Accept；通常也不允许 Edit draft。
  • Rejected：不允许 Reject；是否允许重新编辑要看业务。

BDEF 中先声明哪些 operation/action 受 instance feature control 管理：

update ( features : instance );
delete ( features : instance );
draft action Edit ( features : instance );
action ( features : instance ) acceptTravel result [1] $self;
action ( features : instance ) rejectTravel result [1] $self;

Behavior pool 中的 get_instance_features 读取 OverallStatus，然后给每个 instance 返回不同 feature state。这样列表页选中不同 Travel 时，Fiori Elements 可以把不该执行的按钮置灰；对象页中字段或 action 也可以按实例状态变化。

这个例子有两个重点。第一，按钮可用性来自后端 BO 状态，而不是前端写死。第二，feature control 只是提前告诉 consumer “现在不该点”，真正修改状态的 action 仍应检查业务前提，防止非 UI consumer 绕过按钮状态。

26.8 Counterexample / Failure mode

第一个失败模式是只在 UI 隐藏按钮。Fiori 页面看起来不能点，但 OData action 仍能被外部 consumer 调用。修正：把业务可用性放入 feature control，并在 action / authorization / validation 中保留后端防线。

第二个失败模式是把 authorization 写进 feature control。比如用户没权限审批，就只让按钮 disabled，却没有 RAP authorization 或权限对象检查。这样换一个 consumer 可能仍能调用。修正：权限归 authorization，状态可用性归 feature control，二者协作。

第三个失败模式是用 SELECT active table 判断状态。Draft 或 transaction buffer 中状态已变，active table 还是旧值，按钮状态就会错。修正：在 behavior implementation 中优先使用 EML READ ENTITIES ... IN LOCAL MODE。

第四个失败模式是不看 requested_features。实现每次都算所有 action、所有字段、还做复杂查询，列表页性能变差。修正：只计算 consumer 请求的 feature，并把判断字段控制在最小范围。

第五个失败模式是只测 happy path。Open 状态按钮正常，但 Accepted / Rejected / draft / locked / unauthorized 状态没测，上线后缺陷集中爆发。修正：把 feature matrix 写进测试用例。

第六个失败模式是误以为 disabled 等于业务规则已完成。Feature control disabled 只是可用性提示；如果 action handler 不检查状态，仍可能在特殊调用路径中执行。修正：feature control、authorization、validation、action logic 分层防守。

26.9 日本项目现场说法

  • 常见日语：フィーチャ制御、機能制御、ボタン制御、項目制御、有効 / 無効、読取専用、インスタンス依存、グローバル制御。
  • 说明句：Feature control は、RAP BO の状態に応じて操作や項目を有効化・無効化するための仕組みです。
  • Review 句：承認済みステータスの場合、更新・削除・承認アクションが無効になるように get_instance_features で制御されていますか。
  • 调查句：ボタンが押せる/押せない問題は、UI アノテーションだけでなく BDEF、feature control、authorization、metadata 更新を確認します。
  • 区分句：権限チェックは authorization、状態によるボタン制御は feature control として分けて設計します。
