Chapter 01 - SAP 和 ABAP 基础

6. ABAP Cloud 现代开发模型

  • Chapter 01
  • Section 06
  • self-contained study note

本节结论

ABAP Cloud 是 SAP 面向 cloud-ready、upgrade-stable 扩展和应用开发提出的现代 ABAP 开发模型；它不是“ABAP 放到云上运行”的泛称，也不等于 SAP BTP, ABAP environment 这个运行环境本身。

正文

前一节讲的是 BTP 上的 ABAP 云开发环境，这一节讲的是更抽象的一层：开发模型。一个环境回答“代码跑在哪里”，一个开发模型回答“代码应该用什么语言版本、对象类型、API 边界和质量检查来写”。这就是为什么不能把 ABAP Cloud 简单翻译成“云上的 ABAP”。它是一套规则：哪些 ABAP 语言特性可以用，哪些 SAP 对象可以依赖，哪些接口被认为生命周期稳定，哪些开发方式能通过 Clean Core 检查。

SAP Learning 对 ABAP Cloud 的描述可以拆成几块：ABAP for Cloud Development 这样的语言版本，RAP 和 ABAP analytics 这样的 cloud-native programming models，reuse services、identity/access management、connectivity 这样的平台能力，SAP S/4HANA Cloud 的 business APIs、extension points、events，以及 ADT 作为主要开发工具。Day 01 不要求把这些都学会，但要先知道它们属于同一套现代 ABAP 方向。

最关键的是语言版本。开发者创建对象时，会遇到 Standard ABAP、ABAP for Cloud Development、ABAP for Key Users 这样的语言版本概念。Standard ABAP 对应传统/Classic ABAP 世界；ABAP for Cloud Development 是开发者做 ABAP Cloud pro-code 开发时最需要关注的版本；ABAP for Key Users 也属于 ABAP Cloud 语境，但更多出现在 key user extensibility 场景。语言版本不是装饰字段，它会影响语法检查、可用对象、可调用 API 和 ATC 检查结果。

另一个关键点是 released APIs。ABAP Cloud 要求开发者优先使用 SAP 释放给客户使用的接口、对象和扩展点，而不是直接依赖 SAP 内部实现。例如，过去开发者可能会直接 SELECT 某张 SAP 标准表；在 ABAP Cloud 语境里，如果这张表没有 release，正确方向可能是使用 released CDS view、business API、RAP BO interface、BAdI extension point 或事件。这不是为了增加麻烦，而是为了让客户扩展在升级后仍然稳定。

本节只建立边界，不展开实现细节。RAP business object、CDS view entity、service definition、service binding、DCL、EML、save sequence、authorization、connectivity、software component lifecycle、gCTS / cTMS 等内容都会在后续章节展开。Day 01 只要读者能说清：“ABAP Cloud 是现代 ABAP 的受控开发方式，核心是语言版本、released APIs、ADT/ATC 检查、RAP/CDS/service 化，以及 Clean Core。”

为什么重要

ABAP Cloud 是理解现代 SAP 开发方向的分水岭。传统 SAP 项目里，开发者经常会遇到 report、function module、BAPI、BAdI、enhancement、SAP GUI dynpro、直接读表、后台 job 等经典做法。它们在 on-premise 或 Private Edition 的 brownfield 项目中仍然重要，本教程 Week 4 也会专门讲 recognition-level 的经典项目现实。但如果一个学习者只用这些旧经验理解 SAP S/4HANA Cloud、Clean Core 和 RAP，他会很快踩到边界：某些对象不能用，某些 API 未释放，某些语句会被检查拦住，某些直接修改标准对象的方案在架构 review 中过不了。

对 Java/Python 背景的人来说，可以把 ABAP Cloud 粗略类比成“带强约束的框架 + 平台 API 契约 + 静态检查规则”。但 SAP 的差异在于：这些约束不是某个团队自定的 lint 规则，而是嵌入到 ABAP Platform、ADT、compiler/runtime、ATC、SAP released objects 和 S/4HANA 扩展模型里的交付边界。它影响的不只是代码风格，还影响升级、传输、权限、测试、运维和客户审计。

在项目里，ABAP Cloud 还承担一个沟通作用。当团队说“这个方案要符合 Clean Core”，不是说“代码写得漂亮一点”，而是要回答：是否使用 released APIs？是否避免修改 SAP standard objects？是否能经受升级？是否适合 Public Edition、Private Edition 或 on-premise 的部署形态？是否通过 Released API dependency check 和 ATC 检查？因此，Day 01 就要先把 ABAP Cloud 作为“现代 SAP 开发的边界词”讲清楚。

核心机制

第一层机制是 product / platform / model 分离。SAP S/4HANA 是业务套件，ABAP Platform 是技术基础，SAP BTP, ABAP environment 是 BTP 上的 ABAP PaaS 运行和开发环境，ADT 是开发工具，而 ABAP Cloud 是开发模型。可以在不同环境里使用同一个开发模型：SAP BTP, ABAP environment 和 SAP S/4HANA Cloud Public Edition 以 ABAP Cloud 为默认且唯一的 ABAP 开发模型；SAP S/4HANA Cloud Private Edition 和 on-premise 中，ABAP Cloud 可以与 Classic ABAP 并存。

第二层机制是 evolution not revolution。SAP 官方资料强调，ABAP Cloud 不是要求客户把所有既有 Classic ABAP 一夜之间重写。Private Edition / on-premise 的现实是：历史代码、经典增强、BAPI、IDoc、报表和事务会长期存在。正确理解是：既有 Classic ABAP 可以继续被治理和维护；新开发、重构和面向云迁移的扩展，应尽量使用 ABAP Cloud、released APIs 和 Clean Core 方式。它是向前演进，不是把过去项目经验全部作废。

第三层机制是 language version。Standard ABAP、ABAP for Cloud Development、ABAP for Key Users 代表不同开发上下文。选择 ABAP Cloud 语言版本后，开发对象会受到更严格的检查：过时语句、classic UI 技术、未释放 SAP 对象、直接依赖内部表或内部 function module 的做法，都可能被语法检查或 ATC 拦下。学习者不需要 Day 01 就记住全部规则，但必须知道“为什么同样是 ABAP，有的代码在一个系统能跑，在 ABAP Cloud 对象里却不能写”。

第四层机制是 released object / released API。ABAP Cloud 里的“能不能用”，重点不只是语法上能不能写，而是对象是否被 SAP 释放给客户使用。Released APIs 可以是本地 ABAP platform APIs、SAP S/4HANA Cloud business APIs、released CDS views、released classes、BAdI extension points、events 或其他 extension points。ADT 的 released objects view、SAP Business Accelerator Hub、SAP Help / SAP Learning 都是寻找证据的入口。后续看到 I_COUNTRY 这类 released CDS view 替代直接读标准表的例子时，要把它理解成 ABAP Cloud 的核心思想，而不是零散技巧。

第五层机制是 checks。ABAP Cloud 不只是倡议，也会通过 compiler、runtime 和 ATC 检查落地。不同系统和版本里会看到不同 ATC check variants，例如 ABAP_CLOUD_DEVELOPMENT_DEFAULT、ABAP_CLOUD_READINESS，以及面向 3-tier clean-core setup 的 ABAP_CLOUD_DEVELOPMENT_3TIER。粗略说，ABAP_CLOUD_DEVELOPMENT_DEFAULT / ABAP_CLOUD_DEVELOPMENT_3TIER 更接近新开发时的边界检查，ABAP_CLOUD_READINESS 更多用于扫描存量 Classic ABAP 代码的 cloud-readiness。Day 01 不配置 ATC，只要知道这些名字代表“系统会检查你的代码是否遵守 ABAP Cloud / Clean Core 边界”；具体检查集、豁免和质量门禁放到 Week 2 / Week 4。

第六层机制是 programming model。ABAP Cloud 的核心实现路径通常会连接到 CDS、RAP、service definition / service binding、OData、events、authorization 和 testability。它不是让开发者随便把旧 report 包一层 API，而是鼓励围绕业务对象、released interfaces、服务化暴露和可测试逻辑来组织开发。也正因为如此，本节后面会自然连接到 Week 3 的 RAP / OData 学习。

学习例子

假设你要做一个国家代码下拉框。Classic ABAP 经验可能会让人想到直接从 SAP 标准表 T005 读取国家主数据。这个想法在很多历史项目里并不陌生，但在 ABAP Cloud 语境里，问题不是“SQL 会不会写”，而是“这张表是否 released、直接依赖它是否 upgrade-stable”。如果对象未释放，系统可能会给出错误或 ATC finding，并提示使用 released CDS entity，例如 I_COUNTRY。

这个例子很小，但能说明 ABAP Cloud 的思维方式：开发者不应该把 SAP 标准表当成任意数据库表随便读；应该先找 released API / released CDS view / business API / extension point。这样写出来的扩展更有可能在升级后继续工作，也更容易通过 Clean Core review。

再看一个业务例子：客户想在销售订单审批时加入额外风控逻辑。Day 01 不需要决定完整实现，但要知道判断路径。Public Edition 中要优先看 key user extensibility、released BAdI、released APIs、developer extensibility 和 side-by-side extension；Private Edition / on-premise 中可能还会遇到 classic BAdI 或既有增强，但新开发仍要问是否能用 ABAP Cloud / released API 方式实现。这个判断决定后面是走 RAP、OData、event、BAdI，还是只能作为 brownfield awareness 处理。

最小练习：打开 SAP Learning 的 Exploring ABAP Cloud，用自己的话写出三句话：ABAP Cloud 不是什么；它要求什么；它和 Classic ABAP 如何并存。

常见误区

  • 误区 1：把 ABAP Cloud 理解成“部署在云上的 ABAP”。修正：它首先是开发模型，重点是语言版本、released APIs、ADT/ATC 检查和 Clean Core。
  • 误区 2：以为有了 ABAP Cloud，Classic ABAP 就全部过时。修正：Private Edition / on-premise 中二者会长期并存，SAP 的思路是 evolution not revolution；既有代码需要治理，新开发尽量向 ABAP Cloud 靠拢。
  • 误区 3：把“语法能写”当成“项目可用”。修正：在 SAP 项目里，released state、upgrade stability、authorization、transport、ATC、客户部署形态都可能影响可用性。
  • 误区 4：直接把任意 SAP 标准表、function module 或 class 当成可依赖接口。修正：先查 released objects / SAP Business Accelerator Hub / SAP Help，必要时使用 released CDS view、business API、RAP BO interface、event 或 extension point。
  • 误区 5：Day 01 过早陷入 RAP save sequence、EML、DCL、IAM、gCTS 配置细节。修正：先建立边界；实现细节在对应章节学。

思考题

  • 如果一个对象在 Classic ABAP 中能用，但在 ABAP Cloud 中不能用，可能是哪个层面的限制：语言版本、object type、released API，还是 ATC？
  • 为什么 SAP 强调 released APIs？它和 Java/Python 项目里的 public API / internal API 边界有什么相似点和差异？
  • Private Edition / on-premise 中“ABAP Cloud 与 Classic ABAP 并存”对项目评审有什么影响？
  • 在日本项目里说明“不能直接读 SAP 标准表”时，应该用 リリース済み API、利用可能なオブジェクト、クリーンコア 中的哪个词组合？
  • 如果后续在 AC0567C POC 中看到 RAP service binding 或 released object 检查，它如何证明本节的边界？
