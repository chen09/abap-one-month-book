9. Root view entity 根视图实体

  • Chapter 03
  • Section 09
  • self-contained study note

9.1 一句话结论

Root view entity 根视图实体，是用 define root view entity 把一个 CDS view entity 声明为 RAP business object 的根节点。它不是“字段更多的 CDS view”，而是告诉 RAP：这个实体是一个业务对象的入口，后续 behavior definition、lock、authorization、composition child、projection view 和 OData 服务都要围绕它展开。

对初学者来说，可以先把它理解成“业务对象的 header / aggregate root”。例如 Travel 是根，Booking 是子；Sales Order 是根，Sales Order Item 是子。根视图实体负责定义这个业务对象最上层的识别方式、核心字段、对外关联和下级组成关系。真正的 create / update / delete、validation、determination、action、save sequence 不写在 root view entity 里，而是在后续 Behavior definition 行为定义 和 behavior implementation 中定义。

9.2 到底学什么

学习 Root view entity，要抓住三个层次。

第一，它仍然是 CDS view entity。它有 DDL source，有 as select from ...，有 key element、字段别名、annotation、association，也要在 ADT 中保存、语法检查和 activation。前面学过的 CDS view entity 视图实体、CDS annotation 注解基础、Association 关联关系 和 Path expression 路径表达式 在这里都还有效。

第二，它多了 root 这个 RAP 语义。SAP ABAP Keyword Documentation 对 ROOT 的定义很直接：这个 addition 把相关 CDS entity 定义为 ABAP RESTful Application Programming Model 中 RAP business object 的 root entity。也就是说，root 不是 UI 装饰，也不是数据库约束；它是 RAP BO 建模的入口语义。

第三，它通常会成为后续 repository object 的起点。SAP Learning 的 RAP 课程说明，behavior definition 直接依赖 CDS root entity；一个 interface / data-model behavior definition 只对应一个 root entity，而这个 data-model root 通常也只有一个对应的 interface behavior definition。这里要注意层次：projection root view 还可以有单独的 projection behavior definition，用来把底层 BO 的行为暴露给特定 consumer。初学者在项目里看到两个 .bdef 文件时，不要误以为 root 概念冲突，而要先判断一个是不是 projection layer。

所以本节要学的不是“怎么多写一个关键字”，而是：为什么一个 RAP 应用要先选出根，哪些关系应该挂在根下面，哪些能力不属于根视图实体本身。

9.3 为什么重要

Root view entity 决定 RAP business object 的形状。这个形状一旦进入 behavior definition、projection view、service definition、service binding 和前端消费，就会变成项目中的接口契约。字段名、key、composition、authorization master、lock master、projection 暴露范围，都可能受到根的选择影响。

如果根选错，后面会出现很典型的项目问题：

  • 一个本来应该作为子对象存在的 item 被当成 root 暴露，导致 create / update / delete 边界混乱。
  • 一个 root 只适合查询，却被拿来做 transactional BO，后续 behavior definition 要补很多奇怪的限制。
  • composition 和 to-parent association 不成对，activation 或行为建模时反复报错。
  • OData 服务暴露后，UI 看到的导航结构和业务人员理解的业务对象结构不一致。
  • authorization、lock、draft、save sequence 的讨论没有共同锚点。

Java/Python 背景的读者可以把 root view entity 类比成 DDD 里的 aggregate root 加上 ORM/DTO 层的一部分影子，但不能完全等同。RAP 的 root 不是普通类，不拥有 ABAP 方法实现；它是 CDS 数据模型中的根节点，行为由 BDL 和 behavior pool 接上。它也不是数据库表本身；一个 root view entity 可以从表、其他 view entity 或更复杂的数据模型选取字段，并通过 association/composition 表达业务关系。

9.4 Key mechanism / 核心机制

9.4.1 define root view entity 表达的是 RAP 根

普通 CDS view entity 通常写成：

define view entity ZI_Booking
  as select from zbooking
{
  key booking_id as BookingID,
      travel_id  as TravelID,
      ...
}

Root view entity 多了 root：

define root view entity ZI_Travel
  as select from ztravel
{
  key travel_id as TravelID,
      agency_id as AgencyID,
      ...
}

这个 root 的意义不是“它一定在 SQL 上更快”，也不是“它自动变成 OData 服务”。它只是把 CDS entity 声明为 RAP BO 的根。要让它变成可写的业务对象，还需要 behavior definition；要让它变成服务，还需要 projection view、service definition 和 service binding。

9.4.2 Root、child、composition、to-parent 要一起看

RAP 业务对象常常不是单表模型。以 Travel / Booking 为例，Travel 可以是 root，Booking 是 child。根到子通常用 composition 表示，子到父要有 association to parent。

define root view entity ZI_Travel
  as select from ztravel
  composition [0..*] of ZI_Booking as _Booking
{
  key travel_id as TravelID,
      ...
      _Booking
}

子实体则不是 root：

define view entity ZI_Booking
  as select from zbooking
  association [1..1] to parent ZI_Travel as _Travel
    on $projection.TravelID = _Travel.TravelID
{
  key travel_id  as TravelID,
  key booking_id as BookingID,
      ...
      _Travel
}

SAP-samples 的 RAP openSAP 教材也用 Travel / Booking 展示这个模式：先创建 Travel root view entity，再创建 Booking view entity，然后给 Travel 加 composition [0..*] of ... as _Booking，给 Booking 加 association to parent ... as _Travel。教材还提醒，这类相互依赖的 CDS views 初次创建时可能需要一起 activation。

这正是 root view entity 的实际价值：它不是孤立的一张“主表视图”，而是 composition tree 的入口。

9.4.3 Root view entity 不等于 projection view

Root view entity 属于 data model / interface model 一侧，projection view 属于消费或服务裁剪一侧。一个常见命名习惯是：

  • ZI_Travel：interface / data model root view entity。
  • ZC_Travel：consumption / projection root view entity。

两者都可能带 root，但语义层不同。前者定义业务对象的核心模型，后者决定某个服务或消费场景能看到哪些字段、哪些 association、哪些 operation。SAP Learning 在 behavior projection 部分也说明，projection behavior 是基于 projection root view 创建的，它把底层 business object 的行为暴露给特定 consumer，而不是重新实现底层行为。标准 RAP 服务建模中，通常要先建立 projection root view，再通过 service definition / service binding 暴露；不要把 interface root view 直接当成最终服务契约。

因此，本节先讲 root view entity；下一节 Projection view 投影视图 再讲“服务暴露时如何裁剪”。

9.4.4 Root view entity 不负责行为实现

Root view entity 能定义字段、key、association、composition、annotation，但它不写业务行为。下面这些不是 root view entity 的职责：

  • create / update / delete 是否允许。
  • validation 和 determination。
  • action 的定义与实现。
  • lock master / authorization master 的声明。
  • draft、early numbering、late numbering。
  • save sequence 中何时 check、adjust、save。
  • reported / failed message 如何返回。

这些属于 Behavior definition 行为定义、behavior implementation、Save sequence 保存序列、authorization 和 service 层。SAP Learning 的 behavior definition 模板会为 managed scenario 要求 persistent table、lock type 和 authorization type，并说明 root entity 总是 lock master 和 authorization master。这说明 root 是行为定义的锚点，但不是行为本身。

还有两个项目风险要提前记住。第一，在 Clean Core、SAP S/4HANA Cloud 和 SAP BTP ABAP Environment 语境下，root view entity、composition target、association target 和 behavior implementation 需要优先使用 released API / released CDS view；引用未 released 对象可能在 activation、ATC 或升级审查中出问题。第二，draft 不是给 root view entity 加一个字段那么简单；启用 draft 会牵涉 behavior definition、draft table、root/child 结构和保存流程，不能在 root 这一层孤立决定。

9.5 日本项目现场说法

常见日语表达：

  • ルートビューエンティティ
  • RAP BO のルートエンティティ
  • 親子構成
  • コンポジション
  • 親へのアソシエーション
  • プロジェクションビュー
  • ビヘイビア定義

设计说明中可以这样说：

この CDS は RAP BO のルートビューエンティティとして定義し、明細側は composition と to-parent association で親子関係を表現します。

代码 review 中可以这样说：

この項目はルートで保持すべき業務キーなのか、子エンティティ側の属性なのかを確認してください。

移送前确认中可以这样说：

ルートビュー、子ビュー、ビヘイビア定義、プロジェクションビューを同じ移送単位で確認しないと、依存関係の activation エラーが出る可能性があります。

9.7 Example / 学习例子

9.例 1：Travel 作为 root
@EndUserText.label: 'Travel root'
define root view entity ZI_Travel
  as select from ztravel as Travel
  composition [0..*] of ZI_Booking as _Booking
  association [0..1] to ZI_Agency as _Agency
    on $projection.AgencyID = _Agency.AgencyID
{
  key travel_id  as TravelID,
      agency_id  as AgencyID,
      begin_date as BeginDate,
      end_date   as EndDate,
      status     as Status,

      _Booking,
      _Agency
}

读这段代码时，不要只看字段。要问五个问题：

  • TravelID 是否真的是 root 的识别字段？
  • Booking 是否依附于 Travel 生命周期？
  • _Booking 是否应该作为 composition 暴露？
  • _Agency 是普通 lookup association，还是子对象？
  • 后续 behavior definition 是否应该以 ZI_Travel 为 root？
9.例 2：Booking 作为 child
@EndUserText.label: 'Booking child'
define view entity ZI_Booking
  as select from zbooking as Booking
  association [1..1] to parent ZI_Travel as _Travel
    on $projection.TravelID = _Travel.TravelID
{
  key travel_id  as TravelID,
  key booking_id as BookingID,
      flight_no  as FlightNo,
      price      as Price,

      _Travel
}

Booking 不是 root，因为从业务对象角度看，它通常依附于 Travel。它需要回指 parent，才能和 root 的 composition tree 对上。这个结构也为后续 EML、draft、save sequence 和 OData navigation 打基础。

9.例 3：从 root 创建 behavior definition

在 ADT 中，基于 root view entity 创建 behavior definition 时，工具会把 root entity 作为行为定义的入口。一个极简 managed behavior definition 会长得像这样：

managed implementation in class zbp_i_travel unique;
strict ( 2 );

define behavior for ZI_Travel alias Travel
  persistent table ztravel
  lock master
  authorization master ( instance )
{
  create;
  update;
  delete;
}

本例只是为了说明依赖关系：behavior definition 依赖 root view entity。不要在 root view entity 里寻找 create、update、validation 这类行为语句。如果这个 BO 有 composition child，真实 .bdef 通常还要定义 child entity 的 behavior、composition 相关能力和必要的 mapping；只定义 root behavior 并不等于整个复合 BO 已经完整。

9.8 Counterexample / Failure mode

9.误区 1：把所有 CDS view 都写成 root

如果一个项目里每个查询视图都写成 define root view entity，通常说明团队没有区分“查询模型”和“业务对象根”。Root 应该服务 RAP BO 建模，不是普通报表视图的默认模板。

9.误区 2：root 和 child 关系只用普通 association

如果 Travel 到 Booking 只是普通 association，而项目又希望它们作为一个 transactional BO 一起被创建、修改、保存，就会缺少 composition tree 的语义。普通 association 表示可导航关系；composition 表示更强的父子组成关系和生命周期依赖。

9.误区 3：以为 root view entity 自动带授权

Root view entity 可以有 @AccessControl.authorizationCheck，也可以在 behavior definition 里进入 authorization master 的讨论，但它不等于完整权限设计。DCL、RAP instance authorization、PFCG role、service exposure 仍然要分别检查。

9.误区 4：在 root 层塞太多消费字段

如果某个字段只是某个 Fiori 列表页需要的派生显示字段，不一定应该放进 interface root。可以考虑放在 Projection view 投影视图 或消费层。Root 层越靠近核心业务模型，越应该稳定、清晰、少混入 UI 偏好。

9.误区 5：以为可以后补一个 extension 把普通 view 变成 root

Root 是 data definition 的核心语义，不是 later extension 的装饰。项目中如果已经有普通 define view entity，不要假设通过 EXTEND VIEW ENTITY 就能把它“升级”为 root。更稳妥的做法是重新确认 BO 建模入口、依赖对象、activation 顺序和影响范围。

9.9 思考问题

  • 当前业务对象的 root 是什么？是 header、master data、document，还是某个明细行？
  • 子对象是否依附 root 生命周期？如果 root 删除，child 应该怎样处理？
  • 哪些关系是 composition，哪些只是 lookup association？
  • behavior definition 应该基于哪个 root view entity 创建？
  • projection root view 是否只是服务裁剪，还是错误地承担了核心模型职责？
  • 是否处于 Clean Core / ABAP Cloud / SAP S/4HANA Cloud 约束下？root 和关联对象是否都是 released object？
  • 如果启用 draft，draft table、root/child 结构和 behavior definition 是否一起设计？
  • 在日本项目评审中，能否用 ルートビューエンティティ、コンポジション、親エンティティへのアソシエーション、プロジェクションビュー、ビヘイビア定義 说明建模选择？
