Chapter 03 - CDS、RAP、OData

1. Why CDS 为什么需要 CDS

  • Chapter 03
  • Section 01
  • self-contained study note

1.1 一句话结论

Why CDS 为什么需要 CDS，核心不是“SAP 又发明了一种 SQL 写法”，而是要理解 ABAP CDS 在 SAP 里承担的语义数据模型角色。传统表和 SQL 主要回答“数据从哪里取、怎样连接、怎样过滤”；CDS 进一步回答“这些数据在业务上是什么意思、怎样被关联、怎样被授权、怎样被 UI / API / RAP / OData 消费”。

在本教程里，CDS 是从 Week 2 的 ABAP SQL overview 进入现代 ABAP 应用开发的第一座桥。读完这一节，学习者应该能说清楚：为什么项目不能只靠散落在 class、report、function module 里的 SELECT；为什么现代 SAP 项目会把复用数据模型放进 CDS view entity 视图实体；以及为什么 CDS 仍然不能替代 RAP behavior、事务保存、业务动作和完整授权设计。

1.2 到底学什么

本节不是马上要求写出复杂 CDS，而是建立一张够稳定的地图。后续 Day 15-21 会展开 view entity、annotation、association、projection、DCL、RAP、OData 和 service exposure；这一节先回答“为什么这一层存在”。

需要掌握五个判断点。

  • CDS 的定位：ABAP CDS 是 AS ABAP 上定义语义数据模型的开发技术，不只是 SQL view 的新语法。
  • CDS 的对象形态：CDS 模型在 ADT 中用 DDL / DCL 源定义，是 repository development object，并和 ABAP Dictionary / ABAP runtime metadata 集成。
  • CDS 的消费链路：基础表或 DDIC persistence -> CDS view entity -> association / annotation / projection / access control -> service definition / service binding -> RAP、OData、Fiori、analytics 或 ABAP SQL 消费。
  • CDS 的边界：CDS 适合表达可复用数据模型、字段语义、关联导航、读权限过滤和消费元数据；不适合承包所有业务流程、save sequence、locking、update authorization、side effects 或复杂事务编排。
  • CDS 的项目价值：让数据模型从“每个程序各写各的 SELECT”变成“可 review、可复用、可暴露、可治理”的 SAP repository 资产。

1.3 为什么重要

如果一个 SAP 项目没有统一的数据模型，常见结果是：每个开发者都在自己的 class 或 report 里重新 join 同一批表，字段别名不一致，金额和币种关系不清，权限过滤有的地方做、有的地方漏，UI 需要的 label 和 value help 又在别处补。系统可能能跑，但 review、障害調査、接口复用和升级稳定性都会变差。

CDS 的价值在于把“业务数据的意义”提前放到模型层。字段不只是数据库列，还可以带上 label、amount/currency 或 quantity/unit 语义、关联对象、消费元数据和访问控制入口。这样 ABAP SQL、RAP、OData、Fiori Elements、analytics 或其他 API 消费同一套语义模型时，不必每次重新解释“这个字段代表什么、和哪个对象有关、能不能被当前用户读到”。

对 Java/Python 背景学习者来说，可以粗略把 CDS 理解成“SQL view + DTO projection + ORM relationship + metadata annotation + 部分 read authorization 入口”的组合，但这个类比只能帮助入门。SAP 的关键差异是：CDS 是 AS ABAP repository 世界中的开发对象，会被 transport、activation、where-used、ABAP Dictionary metadata、RAP / OData 暴露和 Clean Core 策略共同约束。它不是应用里随手写的查询 helper。

CDS 也是 Week 3 的入口。没有 CDS 这层，后面的 Behavior definition 行为定义、Projection view 投影视图、Service definition 服务定义、Service binding 服务绑定 和 OData V4 基础差异 会变成一堆孤立名词；有了 CDS 链路，读者就能看到现代 ABAP 应用从 persistence 到 API 的主干。

1.4 Key mechanism / 核心机制

1.4.1 4.1 从表到语义模型

DDIC database table 字典表 负责定义持久化结构：有哪些字段、主键是什么、数据类型是什么。ABAP SQL 负责读取这些数据。CDS 则在这两者之上声明一个更接近业务语言的模型：哪些字段对外有意义、哪些字段是 key、哪些对象之间存在关联、哪些字段有单位或币种、哪些 annotation 会影响 UI/API/analytics 消费、哪些访问规则应该在读模型时生效。

所以，“为什么需要 CDS”的第一层答案是：因为项目需要一个比单条 SELECT 更稳定的共享语义层。单条 SQL 只存在于某个调用点；CDS view entity 是 repository object，可以被多个调用点、多个服务、多个 UI 或多个测试共同引用。

1.4.2 4.2 CDS 不是“把 SELECT 搬到另一个文件”

初学者很容易把 CDS 看成“把 join 写到 .ddls 里”。这只看到了语法，没有看到模型边界。如果只是把一个报表里的 SQL 复制到 CDS 中，命名仍然含糊、字段仍然缺少语义、关联仍然乱写、授权仍然靠调用方补，那么项目收益很有限。

好的 CDS 设计会先问：这个 view 是基础复用模型、interface view、root view、projection view，还是消费/API 视图？它面向内部复用，还是面向服务暴露？它应该暴露关联，还是只暴露扁平字段？它需要 access control，还是由更上层处理？这些问题都比“能不能写出 join”更重要。

1.4.3 4.3 View entity 是现代 CDS 学习的起点

本教程从 CDS view entity 视图实体 开始，而不是从旧式 DEFINE VIEW 开始。原因很简单：新模型应优先学习 DEFINE VIEW ENTITY 这一现代形态，再回头理解 legacy DDIC-based CDS view 的历史和迁移场景。项目上当然会遇到旧对象，但初学路径应先建立现代 mental model。

这并不意味着“旧对象都可以无脑删除”。SAP 客户系统里常有多年积累的 DDIC view、classic CDS view、custom extraction、BW/analytics 依赖和接口依赖。学习顺序可以现代化，项目处理必须保守：先识别依赖，再评估迁移或包装策略。

1.4.4 4.4 Association、annotation 和字段语义

CDS 的一个关键能力是把关系和元数据放进模型。Association 关联关系 让模型表达“订单有客户”“订单有项目”等可导航关系，而不是让每个消费方都重新写 join。CDS annotation 注解基础 和 CDS metadata extension and UI annotations 则让字段、实体、UI、服务或分析消费拿到额外语义。

例如金额字段不能只看成 decimal。业务上通常还要知道它对应哪个 currency 字段；数量字段也通常需要 unit。没有这些语义，UI 显示、接口解释、测试断言和财务类数据 review 都可能出错。CDS 的意义之一，就是把这类“业务读法”尽量固定在模型层，而不是散落在各个 consumer 里。

1.4.5 4.5 DCL / access control 是重要边界，但不是全部授权

CDS access control 访问控制 和 DCL role 访问规则 可以为 CDS read access 提供访问控制规则，常用于行级过滤或基于授权对象的读限制。对初学者来说，要记住一句话：DCL 能帮助“读模型时能看到哪些行”，但它不是 SAP authorization 的全部。

完整项目里还要考虑 SAP Authorization Architecture Overview、RAP global / instance authorization、action-level authorization、update authorization、UI 按钮可见性、technical user、background job、接口调用者和 audit/logging。把 DCL 当成“有了它就不用管权限”是危险的；把 DCL 完全忽略也危险，因为读模型可能在多个服务和调用点复用。

1.4.6 4.6 CDS 和 RAP/OData 的关系

CDS 给 RAP/OData 提供数据模型骨架，但 RAP 的事务行为不写在普通 CDS view entity 里。validation、determination、action、lock、save sequence、draft、reported/failed、side effects 等行为，需要进入 Behavior definition 行为定义、behavior implementation 和 Save sequence 保存序列 的世界。

这条边界很重要。CDS 可以告诉系统“订单有哪些字段、怎样关联到客户、怎样被投影、哪些字段可用于 UI 元数据”；但“保存订单时先校验什么、什么时候派生字段、失败怎样返回、能不能执行 release action、是否需要 additional save”不是 CDS 单独解决的问题。后续学习 RAP 时，要把 CDS 当成业务对象模型的地基，而不是把所有逻辑都塞进 CDS。

1.4.7 4.7 CDS 和 Clean Core

在 S/4HANA 和 ABAP Cloud 语境下，CDS 还连接到 Clean Core extensibility 和 Released API dependency check。Clean Core 的方向不是鼓励直接读取任意底层表，而是优先使用 released API、released CDS view、稳定 extension point 或明确允许的扩展方式。CDS 因此经常成为“可复用、可发布、可治理”的数据接口之一。

但这里也要避免过度简化。不是所有 CDS 都自动是 released API；不是所有自建 CDS 都适合公开给外部服务；不是所有 direct table access 都能用一个自建 CDS 就变成 Clean Core。项目上要确认 release status、API contract、authorization、performance、upgrade impact 和 transport/package 边界。

1.4.8 4.8 初学者应该怎样读第一个 CDS

第一次打开 CDS view entity 时，不要急着逐行翻译语法。建议按这个顺序读：

  1. 看名字：它是 ZI_... interface/basic 模型、ZR_... root 模型、ZC_... consumption/projection 模型，还是项目自定义命名？
  2. 看 as select from：底层数据来自表、其他 CDS、还是 view composition？
  3. 看 key：业务主键是什么，是否真的能唯一识别实体？
  4. 看字段列表：字段别名是否表达业务语义，金额/币种、数量/单位是否成对出现？
  5. 看 association：关联的 cardinality、on condition 和使用场景是否清楚？
  6. 看 annotation：哪些是技术 annotation，哪些会影响 UI、API、authorization 或 analytics？
  7. 看 access control：有没有 @AccessControl.authorizationCheck 或对应 DCL 设计？
  8. 看 consumer：它被 ABAP SQL、RAP、service definition、Fiori 或 analytics 谁消费？

这个读法比“背语法关键字”更接近项目现场。现场 review 经常问的不是“你会不会写 define view entity”，而是“这个模型的公开范围、字段语义、关联、授权和消费方是否清楚”。

1.4.9 4.9 CDS suitability checklist：什么时候真的需要 CDS

决定是否创建或复用 CDS 时，可以先用下面的表判断。目标不是“所有 SELECT 都搬进 CDS”，而是把稳定、可复用、需要语义和治理的数据模型放到合适层级。

QuestionCDS is a good fit when...Prefer another design when...
Is the model reusable?多个 ABAP class、RAP BO、OData service、Fiori/analytics 需要同一语义模型。只是一个 method 内部的一次性查询。
Is business meaning needed?字段别名、label、amount/currency、quantity/unit、association、annotation 需要集中表达。只是临时筛选或简单 existence check。
Is read authorization relevant?读模型需要 DCL/access control 入口或至少明确授权边界。逻辑主要是 update authorization、action authorization 或 save-time check。
Is it part of API/UI exposure?后续要进入 projection、service definition、service binding、RAP/OData。只是后台内部 helper，不应形成稳定 contract。
Is Clean Core involved?需要优先复用 released CDS/API，或定义可治理 extension model。直接封装 unreleased table 并对外暴露，反而扩大风险。
Is transaction behavior needed?CDS 只负责读模型，行为交给 RAP behavior。需求核心是 save sequence、locking、action、side effect。
Is performance/reuse clearer?模型可被 review、where-used、transport 和优化。复杂聚合/特定报表更适合 SQL aggregate、AMDP 或专门 analytical design。

一个保守规则是：CDS 应该让数据模型更稳定、更有语义、更容易治理；如果创建 CDS 只是为了隐藏一段复杂查询，但字段语义、消费方、授权和发布边界都不清楚，那只是把复杂度换了文件位置。

1.5 日本项目现场说法

  • 常见日语：CDSビュー、CDSビューエンティティ、データモデル、セマンティックモデル、意味付け、アソシエーション、アノテーション、アクセス制御、DCL、射影ビュー、消費ビュー、サービス定義、サービスバインディング。
  • 说明句：CDS は単なる SQL ビューではなく、業務データの意味と公開範囲を定義するセマンティックモデルです。
  • Review 句：この項目の意味付け、通貨/数量単位、関連先、アクセス制御を CDS 側で確認してください。
  • 边界说明句：トランザクション処理や保存順序は CDS ではなく、RAP Behavior 側で設計します。
  • Clean Core 句：標準テーブルへの直接参照ではなく、公開済み CDS/API を利用できるか確認してください。

现场沟通时，CDSビュー 常被宽泛使用；如果需要精确区分，建议说 CDSビューエンティティ。意味付け 比“注解”更贴近日语项目里说明字段语义的表达；アノテーション 则用于具体技术对象。

1.7 Example / 学习例子

先看一个不使用 CDS 的常见思路：某个 report、某个 class、某个接口实现都各自写 SELECT，读取订单表、客户表、金额字段和状态字段。短期看速度很快；长期看，每个地方都会重新决定字段别名、join 条件、空值处理、权限过滤和 UI/API 输出含义。

CDS 的做法是把可复用模型先声明出来。下面是简化示意，不代表可直接激活的完整生产代码：

@EndUserText.label: 'Sales Order Basic View'
@AccessControl.authorizationCheck: #CHECK
define view entity ZI_SalesOrder
  as select from zsales_order
  association [0..1] to ZI_Customer as _Customer
    on _Customer.CustomerId = zsales_order.customer_id
{
  key zsales_order.order_id     as SalesOrder,
      zsales_order.customer_id  as Customer,
      _Customer,
      zsales_order.net_amount   as NetAmount,
      zsales_order.currency     as Currency
}

这段示意里，ZI_SalesOrder 不只是一个查询。它把订单主键、客户关联、金额字段、币种字段和访问控制入口放在一个可复用模型里。后续可以有 projection view 暴露给 UI，有 service definition 暴露给 OData，也可以被 ABAP SQL 作为数据源读取。

但这个 CDS 仍然没有回答所有问题。比如“订单能不能 release”“保存时是否需要写日志”“失败时如何返回 message”“是否需要 lock”“重复调用 action 是否幂等”，这些属于 RAP behavior、application service、authorization 或 transaction design 的范围。第一天学 CDS，就要同时看到它的力量和边界。

最小学习动作：画出一条链路：

table / DDIC persistence
  -> CDS view entity
  -> association / annotation / DCL
  -> projection view
  -> service definition
  -> service binding
  -> RAP / OData / Fiori / ABAP SQL consumer

能把这条链路用中文讲清楚，就已经超过“只知道 CDS 是新 SQL”的阶段。

1.8 Counterexample / Failure mode

  • 误区 1：把 CDS 当成 SQL 文件夹。修正：先问模型语义、复用边界、消费方和授权边界，再写语法。
  • 误区 2：把所有 join 都写进一个巨大的 CDS。修正：分清 basic/interface view、root view、projection/consumption view；不要让一个 view 承包全部场景。
  • 误区 3：认为有 DCL 就等于完成授权。修正：DCL 主要覆盖 CDS read access 相关过滤；RAP authorization、update/action authorization、UI authorization 和技术用户场景仍要单独设计。
  • 误区 4：把复杂业务流程写进 CDS。修正：transactional behavior、save sequence、validation、determination、action 和 side effect 进入 RAP / ABAP application layer。
  • 误区 5：把 annotation 当装饰。修正：annotation 可能影响 UI、服务元数据、分析、授权检查或运行时解释，需要 review。
  • 误区 6：以为自建 CDS 天然 Clean Core。修正：确认 released API / released CDS / extension point / package boundary，而不是只换一层包装。
  • 误区 7：只会中文标题，不会英文检索。修正：用 ABAP CDS, CDS view entity, CDS access control, DCL role 去查官方资料。

1.9 思考问题

  • 如果一个项目不用 CDS，只在 class/report 里散写 SELECT，会带来哪些复用、授权、UI/API 和维护风险？
  • 这个 CDS 是基础复用模型、root model、projection/consumption model，还是服务暴露模型？
  • 哪些字段需要金额/币种、数量/单位、文本、value help 或 UI annotation 语义？
  • 哪些逻辑应该放在 CDS，哪些应该放在 RAP behavior、ABAP class 或 service layer？
  • DCL 解决了哪一部分访问控制？还有哪些 authorization 不能靠 DCL 解决？
  • 在日本项目里向 reviewer 说明时，应说 CDSビュー、CDSビューエンティティ、意味付け、アクセス制御 中哪一个词？
  • 这个模型是否能映射到 AC0567C POC Reading Map 中的 root、behavior、service binding 或 API 证据链？
