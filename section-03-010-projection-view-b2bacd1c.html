10. Projection view 投影视图

  • Chapter 03
  • Section 10
  • self-contained study note

10.1 一句话结论

Projection view 投影视图，是用 define [root] view entity ... as projection on ... 在 CDS/RAP 模型上建立一层面向服务或消费场景的裁剪层。它不是复制一份 root view entity，也不是前端页面配置文件；它决定某个 service consumer 能看到哪些字段、association、composition child、semantic alias 和可暴露的行为。

如果上一节 Root view entity 根视图实体 回答的是“这个 RAP business object 的根是谁”，那么 projection view 回答的是“这个业务对象以什么服务视角给外部看”。同一个底层 BO 可以有不同 projection：一个给 Fiori clerk 使用，只开放 create/update；一个给管理人员使用，额外开放 delete 或 action；一个给 Web API 使用，裁剪 UI annotation，强调稳定字段和集成契约。

10.2 到底学什么

学习 projection view，要先把它放在 RAP 架构链路中：

database table / released API
  -> CDS data model view / root view entity
  -> behavior definition / behavior implementation
  -> projection view
  -> projection behavior definition
  -> service definition
  -> service binding
  -> OData / Fiori / Web API consumer

SAP Learning 的 RAP 架构课程说明，BO projection 是为了特定 business service 对 business object 做 project 和 alias；它由 CDS projection views、CDS behavior projection，以及必要时的消费特定行为实现组成。service definition 再引用 projection views，指定哪些 entity 对 service consumer 可见。

所以本节要学四件事：

  1. Projection view 是服务/消费视角，不是底层数据模型本身。
  2. Projection view 可以裁剪字段、alias、association、composition child 和 annotation 暴露。
  3. Projection behavior definition 决定底层 behavior 中哪些操作在该服务视角可见。
  4. Service definition / service binding 才把 projection 暴露成具体服务和协议。

10.3 为什么重要

Projection view 是从“可复用业务模型”走向“对外契约”的门。没有这层，初学者很容易把 interface/root view entity 直接当 API 用，结果把内部字段、技术字段、临时字段、过宽的 association 或不该公开的 operation 暴露给外部 consumer。

项目上，projection view 至少影响五件事：

  • API contract：哪些字段、entity、navigation 会进入 OData metadata。
  • UI contract：Fiori Elements 列表页、对象页、value help、字段标签和 UI annotation 看到什么。
  • Behavior contract：create/update/delete/action 是否在某个服务中可见。
  • Security contract：projection 不替代 DCL/authorization，但它能减少不必要暴露面。
  • Upgrade/Clean Core：在 SAP S/4HANA Cloud / BTP ABAP Environment 中，projection 应尽量基于 released APIs / released CDS views，避免服务契约依赖不稳定对象。

对 Java/Python 背景读者，可以把 projection view 粗略类比成 DTO / API view model / GraphQL schema selection 的组合：它不应该重新实现核心业务逻辑，但它决定 consumer 看到的形状。区别在于，SAP projection view 是 ABAP repository object，有 activation、transport、provider contract、annotation、service definition、behavior projection 和 OData metadata 影响。

10.4 Key mechanism / 核心机制

10.4.1 as projection on 表示基于已有模型裁剪

典型 projection view 长这样：

@EndUserText.label: 'Travel projection'
@AccessControl.authorizationCheck: #NOT_REQUIRED
define root view entity ZC_Travel
  provider contract transactional_query
  as projection on ZI_Travel
{
  key TravelID,
      AgencyID,
      CustomerID,
      BeginDate,
      EndDate,
      Status,

      _Booking : redirected to composition child ZC_Booking
}

这里的重点不是 select from，而是 as projection on ZI_Travel。它说明当前 view 是基于已有 CDS entity 的投影，不是重新从表出发建一套模型。对现代 RAP transactional service 来说，projection root view 应明确写 provider contract，例如 provider contract transactional_query，表示它是用于 RAP business object projection layer 的 transactional query。示例中 projection 层使用 @AccessControl.authorizationCheck: #NOT_REQUIRED，是因为访问控制通常由底层 interface/root view 的 DCL 继承；projection 不应该让读者误以为要在服务裁剪层重新定义一套行级授权。

SAP Keyword Documentation 对 projection view 的语法有专门页面，DEFINE VIEW ENTITY AS PROJECTION ON 定义 CDS transactional query；provider contract 用来指定 projection view 的用途。项目教材里不应该用“可以省略”作为默认叙述：在现代 ABAP Cloud / S/4HANA Cloud 语境下，新建 projection view 应把 provider contract 当成必须设计的语义。常见 provider contract 包括：

  • transactional_query：用于 RAP BO projection layer，面向 transactional service / OData UI service 的常见场景。
  • transactional_interface：用于稳定 BO interface / BO-to-BO 复用等接口层场景。
  • analytical_query：用于分析查询场景。

本教程当前主线是 RAP transactional service，所以例子用 transactional_query。但读者要知道：不是所有 projection view 都该无脑写 transactional_query，要看它服务 UI、Web API、BO interface 还是 analytics。语法和 release 细节可能存在 no-contract projection view，但项目教材应把 provider contract 当成必须明确设计的语义，而不是依赖默认或历史写法。

10.4.2 Projection view 可以裁剪，不应该乱改核心语义

Projection view 可以做的事情包括：

  • 只暴露部分字段。
  • 给字段或 association 使用更适合 consumer 的 alias。
  • 暴露或隐藏 association / composition child。
  • redirect composition child 到对应的 projection child。
  • 承载或配合服务特定 annotation / metadata extension。
  • 为 service definition 提供被 expose 的 entity。

但它不应该承担核心业务模型职责。业务 key、核心 association、生命周期关系、root/child composition 和持久化 mapping 应该在 data model / root / behavior 层设计清楚。Projection view 负责“给这个服务看什么”，不是“业务对象到底是什么”。

这里有两个容易踩的坑。第一，如果 root projection 暴露了 composition child，通常要有对应的 child projection view，并用 redirected to composition child 明确重定向；否则 service binding 或 activation 阶段可能出现很难懂的依赖错误。第二，projection view 中新定义的 association 有语法和消费限制，不要假设它和底层 interface view 中已有的 association 完全一样可用；特别是同一 projection 里新建 association 后立刻用 path expression 取 target 字段，容易触发语法限制。初学阶段先优先投影和重定向底层模型中已经设计好的关系。

10.4.3 Projection behavior 决定服务中可见的行为

如果底层 RAP BO 有 behavior definition，projection 层通常还要有 projection behavior definition。SAP Learning 说明，BO Projection defines only the behavior relevant for the specific consumer；projection behavior delegates to the underlying layer，具体实现仍在底层 general business object 中。

一个简化 projection behavior 可能长这样：

projection;
strict ( 2 );

define behavior for ZC_Travel alias Travel
{
  use create;
  use update;
  use action cancelTravel;
}

这段代码表达的是“这个服务视角允许使用哪些底层行为”。它不是重新实现 cancelTravel。底层 action 的实现仍在 behavior pool / implementation 中。也就是说，projection behavior 像服务闸门：底层 BO 有能力，不代表每个服务都应该开放。

10.4.4 Service definition / binding 才是真正暴露

Projection view 本身还不是 OData 服务。要进入服务，还需要 service definition：

define service ZUI_TRAVEL {
  expose ZC_Travel as Travel;
  expose ZC_Booking as Booking;
}

再通过 service binding 选择 OData V2 / OData V4、UI service / Web API 等消费方式。SAP Learning 对 RAP 开发流程的说明也把这几步分开：先 project RAP BO，再 define service，最后 bind and test service。项目上看到字段没进 OData metadata，不要只改 projection；还要检查 service definition 是否 expose、service binding 是否激活、metadata cache 是否刷新、authorization 是否拦截。

10.7 Example / 学习例子

10.例 1：底层 root 与 projection root 分层

底层 root 表达业务对象模型：

define root view entity ZI_Travel
  as select from ztravel
  composition [0..*] of ZI_Booking as _Booking
{
  key travel_id as TravelID,
      agency_id as AgencyID,
      status    as Status,
      _Booking
}

服务用 projection root 表达消费视角：

define root view entity ZC_Travel
  provider contract transactional_query
  as projection on ZI_Travel
{
  key TravelID,
      AgencyID,
      Status,
      _Booking : redirected to composition child ZC_Booking
}

读这段代码时，要问：为什么 ZC_Travel 没有暴露所有字段？为什么 _Booking 要 redirect 到 projection child？这个 projection 是 UI service、Web API，还是 BO interface？这些问题比“字段有没有复制全”更重要。

10.例 2：projection behavior 只开放部分能力

底层 BO 可能支持 create、update、delete 和 action：

define behavior for ZI_Travel alias Travel
{
  create;
  update;
  delete;
  action cancelTravel;
}

某个 UI projection 只开放 create/update/action，不开放 delete：

projection;
strict ( 2 );

define behavior for ZC_Travel alias Travel
{
  use create;
  use update;
  use action cancelTravel;
}

这就是 projection 层的服务契约价值：底层有的能力，不等于所有 consumer 都能使用。

10.例 3：service definition 暴露 projection，不直接暴露所有模型
define service ZUI_TRAVEL {
  expose ZC_Travel  as Travel;
  expose ZC_Booking as Booking;
}

服务定义引用的是 projection entities。这样 service consumer 面对的是经过裁剪和命名整理的契约，而不是底层 interface model 的全部内部结构。

10.8 Counterexample / Failure mode

10.误区 1：把 projection view 当成复制粘贴的 root view

如果 projection view 只是把 root view 所有字段原样复制一遍，说明团队可能没有认真设计服务契约。Projection 层要回答“这个 consumer 需要什么”，不是机械复制。

10.误区 2：把 UI annotation 全塞进底层 root

如果某个 annotation 只服务一个 Fiori 页面，优先考虑 projection layer 或 metadata extension，而不是污染底层 interface/root model。底层越靠近核心业务模型，越应该稳定。

10.误区 3：以为 projection view 自动变成 OData

Projection view 还需要 service definition expose、service binding activate 和 preview/test。OData metadata 没出现字段时，要沿链路检查，不要只盯着 projection view。

10.误区 4：以为 projection behavior 会重新实现逻辑

Projection behavior 主要是委托和选择暴露底层 behavior。真正的业务逻辑仍在底层 behavior definition / implementation。服务层不应该复制业务规则。

10.误区 5：忽略 provider contract

项目上如果 projection view 用于 RAP transactional service，却没有明确 provider contract，容易产生 activation、语法检查、语义不清或后续迁移问题。新开发应把 provider contract 当成设计必填项，并根据场景选择 transactional_query、transactional_interface 或 analytical_query。

10.误区 6：暴露 composition 却忘记 child projection / redirection

Root projection 如果暴露 _Booking 这类 composition child，就要确认 ZC_Booking 是否存在、是否是正确的 child projection、是否在 root projection 里使用 redirected to composition child。不要等到 service binding activation 才发现导航结构不完整。

10.9 思考问题

  • 这个 projection 面向 UI service、Web API、BO interface，还是内部复用？
  • 哪些字段必须成为服务契约？哪些字段只是内部实现细节？
  • 哪些 association / composition child 应该暴露为 navigation？
  • 暴露 composition child 时，是否已经有对应 child projection view 和 redirection？
  • projection behavior 是否只开放当前 consumer 需要的操作？
  • provider contract 是 transactional_query、transactional_interface 还是 analytical_query？为什么？
  • service definition 是否 expose 了正确的 projection entity？
  • metadata extension、UI annotation 和 projection view 的边界是否清楚？
  • 当前系统是否要求 released API / Clean Core？projection 是否依赖了不稳定对象？
  • 在日本项目评审中，能否用 プロジェクションビュー、プロジェクション層、公開対象、サービス契約 说明设计选择？

10.12 学习任务

  1. 画出 ZI_Travel -> ZC_Travel -> service definition -> service binding 的链路。
  2. 从一个 projection view 中找出：哪些字段被保留，哪些字段没有暴露。

  4. 找一个 projection behavior definition，说明哪些底层 behavior 被 use 出来。
