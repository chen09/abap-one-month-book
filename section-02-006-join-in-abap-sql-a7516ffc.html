Chapter 02 - Core ABAP Programming

6. JOIN in ABAP SQL

  • Chapter 02
  • Section 06
  • self-contained study note

6.1 一句话结论

JOIN 是 ABAP SQL 里把多个数据源按关系组合成一个结果集的机制。它的价值不是“写得更高级”，而是避免在 ABAP 应用层循环里手工拼表，让数据库按明确的 ON 条件完成关联读取。但 JOIN 也不是魔法：它会受数据模型 cardinality、外连接 null、权限边界、client handling、性能和 Clean Core 访问规则影响。

6.2 到底学什么

在业务代码里，很少只读一张表。订单列表可能要带客户名称，明细行可能要带物料描述，库存数据可能要带工厂信息。初学者最容易写出这种模式：

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id
       WHERE created_at >= @lv_from_date
       INTO TABLE @DATA(lt_orders).

LOOP AT lt_orders INTO DATA(ls_order).
  SELECT SINGLE FROM zcustomer
         FIELDS customer_name
         WHERE customer_id = @ls_order-customer_id
         INTO @DATA(lv_customer_name).
ENDLOOP.

这就是典型的应用层手工拼表：先读订单，再循环订单逐条读客户。数据少时能跑，数据多时就变成 N+1 查询、性能差、错误路径分散、逻辑难 review。

JOIN 的基本写法是把关系放进一条 ABAP SQL：

SELECT FROM zsales_order AS so
       INNER JOIN zcustomer AS cu
          ON cu~customer_id = so~customer_id
       FIELDS so~sales_order_id,
              so~customer_id,
              cu~customer_name,
              so~created_at
       WHERE so~created_at >= @lv_from_date
       ORDER BY so~created_at DESCENDING,
                so~sales_order_id DESCENDING
       INTO TABLE @DATA(lt_order_list).

这里 AS so、AS cu 是 alias；so~customer_id、cu~customer_id 用 column selector ~ 指定字段来自哪个数据源；ON 是 join condition，说明订单和客户如何匹配。读懂 JOIN，就是读懂这三个问题：从哪些数据源来、按什么条件匹配、匹配后的行数是否符合业务预期。

6.3 为什么重要

JOIN 重要，是因为 SAP 业务数据天然是关系型的。销售订单头、订单行、客户、物料、状态、配置、文本、组织结构，很少孤立存在。如果每次都在 ABAP 内表里循环查另一张表，代码会逐步变成“数据库只负责搬数据，业务关系全靠应用层拼”的样子。

它的第一层价值是性能和可读性。数据库擅长按条件连接数据源；一条清楚的 JOIN 往往比循环中多次 SELECT SINGLE 更容易优化，也更容易让 reviewer 看出业务关系。

第二层价值是减少隐藏 bug。手工拼表时，很容易忘记处理找不到右侧记录、重复匹配、多语言文本、有效期、组织范围、client 边界和权限边界。JOIN 把这些关系集中在 FROM ... JOIN ... ON ... 和 WHERE 附近，让问题更容易被发现。

第三层价值是为 CDS/RAP 做准备。现代 SAP 项目中，很多关联关系会进入 CDS view entity、CDS association 或 RAP BO 模型。学习 ABAP SQL JOIN，不是为了永远直接 join 底表，而是为了看懂数据模型如何把业务关系表达成可查询的结构。

6.4 Key mechanism / 核心机制

6.4.1 Inner join：只保留匹配组合

INNER JOIN 只保留两边都匹配的行：

SELECT FROM zsales_order AS so
       INNER JOIN zcustomer AS cu
          ON cu~customer_id = so~customer_id
       FIELDS so~sales_order_id,
              cu~customer_name
       WHERE so~status = @lc_status_open
       INTO TABLE @DATA(lt_orders).

如果某张订单的 customer_id 在 zcustomer 里找不到，这张订单不会出现在结果里。这个行为适合“没有右侧主数据就不应显示”的场景；但如果业务要显示订单并标记客户缺失，就不应该用 inner join。

6.4.2 Left outer join：保留左侧行

LEFT OUTER JOIN 会保留左侧数据源中的行，即使右侧没有匹配：

SELECT FROM zsales_order AS so
       LEFT OUTER JOIN zdelivery AS dlv
          ON dlv~sales_order_id = so~sales_order_id
       FIELDS so~sales_order_id,
              so~status,
              dlv~delivery_id
       WHERE so~customer_id = @lv_customer_id
       INTO TABLE @DATA(lt_order_delivery).

如果订单还没有交货单，左侧订单仍然会出现，右侧交货字段来自数据库 null。ABAP 目标字段接收 null 时要小心：有些场景会转成类型初始值，有些场景应使用 null indicator 或 coalesce( ) 明确处理。教学阶段先记住一句：outer join 会制造“右侧不存在”的业务路径，不能把初始值直接当成真实业务值。

6.4.3 ON 和 WHERE 的边界

ON 说明两边如何匹配；WHERE 说明最终结果集要过滤哪些行。inner join 场景下，两者的误用通常只是可读性和优化问题；left outer join 场景下，过滤右侧字段尤其危险：

SELECT FROM zsales_order AS so
       LEFT OUTER JOIN zdelivery AS dlv
          ON dlv~sales_order_id = so~sales_order_id
       FIELDS so~sales_order_id,
              dlv~delivery_id
       WHERE dlv~delivery_status = @lc_status_open
       INTO TABLE @DATA(lt_rows).

这段代码在 WHERE 里要求右侧交货状态等于 open。右侧没有交货单的订单，本来 left outer join 会保留；但 WHERE 条件会把它们过滤掉。结果看起来就像 inner join。更稳的做法是先问清楚业务：是只要 open 交货单的订单，还是要所有订单并附带 open 交货信息？不同答案决定条件放在 ON、WHERE，还是拆成更清楚的数据模型。

6.4.4 Cardinality：JOIN 会放大行数

JOIN 不是把两张表“横向粘在一起”这么简单。它会为所有满足 ON 条件的组合生成行。如果一个订单有 3 个联系人、一条物料有 5 条文本、一个业务对象有多条状态历史，JOIN 结果就会被放大。

SELECT FROM zsales_order AS so
       INNER JOIN zcustomer_contact AS cc
          ON cc~customer_id = so~customer_id
       FIELDS so~sales_order_id,
              cc~contact_person
       WHERE so~sales_order_id = @lv_sales_order_id
       INTO TABLE @DATA(lt_contacts).

如果客户有多名联系人，一张订单会出现多行。这不是 JOIN 错了，而是数据模型告诉你“订单到联系人不是 1:1”。项目 review 要问 多重度は1対1ですか、1対多ですか。如果后续逻辑只想要一个联系人，就必须说明优先级、有效期、角色或排序规则。

6.4.5 Alias 和字段前缀

多数据源 JOIN 很容易出现同名字段，例如 customer_id、created_at、status。使用 alias 和 ~ 是可读性基础：

SELECT FROM zsales_order AS so
       INNER JOIN zcustomer AS cu
          ON cu~customer_id = so~customer_id
       FIELDS so~sales_order_id AS order_id,
              cu~customer_id    AS customer_id,
              cu~customer_name  AS customer_name
       INTO TABLE @DATA(lt_result).

不要让读者猜字段来自哪张表。尤其是 join 三张以上数据源时，alias 命名和 FIELDS 列表就是代码的说明书。

6.4.6 JOIN 不是权限和 Clean Core 许可

能 join 两张表，不代表业务上允许直接读它们。直接 join SAP 标准底表，可能绕过 CDS DCL、released API、RAP authorization 或 Clean Core 限制。ABAP Cloud / S/4HANA Cloud 项目尤其要先确认 released CDS view、released API、RAP BO 或项目自有持久化对象。

如果业务关系已经由 CDS view entity 或 CDS association 建模，应用代码往往应该优先使用那个 released data model，而不是在 ABAP SQL 中重复拼底表。学习 JOIN 是为了理解关系，不是鼓励所有场景都直接 join table。

6.7 Example / 学习例子

6.7.1 例 1：订单带客户名称
SELECT FROM zsales_order AS so
       INNER JOIN zcustomer AS cu
          ON cu~customer_id = so~customer_id
       FIELDS so~sales_order_id,
              so~customer_id,
              cu~customer_name,
              so~created_at
       WHERE so~status = @lc_status_open
       ORDER BY so~created_at DESCENDING,
                so~sales_order_id DESCENDING
       INTO TABLE @DATA(lt_orders).

读这段代码时，先看 ON cu~customer_id = so~customer_id 是否表达了业务关系，再看 WHERE so~status = ... 是过滤订单状态而不是客户状态。最后仍然要看 ORDER BY，JOIN 不会自动给结果稳定顺序。

6.7.2 例 2：订单可选交货单
SELECT FROM zsales_order AS so
       LEFT OUTER JOIN zdelivery AS dlv
          ON dlv~sales_order_id = so~sales_order_id
       FIELDS so~sales_order_id,
              so~status,
              coalesce( dlv~delivery_id, '' ) AS delivery_id
       WHERE so~customer_id = @lv_customer_id
       INTO TABLE @DATA(lt_rows).

这里用 LEFT OUTER JOIN，是因为没有交货单的订单也要显示。coalesce( ) 把右侧缺失时的 null 变成明确的默认值，避免后续把“无交货单”和“交货单号初始值”混成一件事。

6.7.3 例 3：JOIN 结果行数变多
SELECT FROM zsales_order AS so
       INNER JOIN zsales_order_item AS item
          ON item~sales_order_id = so~sales_order_id
       FIELDS so~sales_order_id,
              item~item_no,
              item~material_id
       WHERE so~sales_order_id = @lv_sales_order_id
       INTO TABLE @DATA(lt_items).

一张订单有多行 item，因此结果多行是正常的。不要在后续逻辑里假设 sales_order_id 唯一；真正唯一的是订单号加行号。

6.8 Counterexample / Failure mode

失败模式一：循环里 SELECT。

LOOP AT lt_orders INTO DATA(ls_order).
  SELECT SINGLE FROM zcustomer
         FIELDS customer_name
         WHERE customer_id = @ls_order-customer_id
         INTO @DATA(lv_customer_name).
ENDLOOP.

如果这只是为了给订单列表补客户名称，通常应该考虑 JOIN 或 CDS 模型，而不是每行打一次数据库。

失败模式二：left outer join 被 WHERE 条件改坏。

SELECT FROM zsales_order AS so
       LEFT OUTER JOIN zdelivery AS dlv
          ON dlv~sales_order_id = so~sales_order_id
       FIELDS so~sales_order_id,
              dlv~delivery_id
       WHERE dlv~delivery_status = @lc_status_open
       INTO TABLE @DATA(lt_rows).

右侧没有交货单的订单会被 WHERE 过滤掉。review 时要确认这是有意的，还是应该把条件放入 ON，或者拆成更明确的查询。

失败模式三：忘记 cardinality。订单 join 行项目、客户 join 联系人、物料 join 文本，都可能从一行变多行。如果后续只取第一行，就是把数据模型问题藏起来。

失败模式四：直接 join 标准底表。传统 on-premise 项目里可能常见，但 ABAP Cloud、S/4HANA Cloud、Clean Core、升级稳定性和权限语境下都要先问 released API / CDS view / RAP BO 是否存在。

失败模式五：以为 JOIN 自动处理权限。JOIN 只是数据读取方式；权限仍然要看 CDS DCL、RAP authorization、AUTHORITY-CHECK、服务层和应用逻辑。直接 table join 不会自动继承某个 CDS view 的 DCL 语义。

6.9 思考问题

  • 这次 JOIN 的左侧和右侧数据源分别是什么？为什么这样放？
  • ON 条件是否表达真实业务关系，还是只是字段名相同？
  • 这个关系是 1:1、1:n，还是 n:m？结果行数会不会放大？
  • 用 inner join 时，右侧缺失的数据是否应该被丢弃？
  • 用 left outer join 时，右侧字段为 null/initial 的业务含义是什么？
  • 右侧过滤条件应该放在 ON 还是 WHERE？
  • 是否存在循环中 SELECT SINGLE 的 N+1 代码可以改成 JOIN、CDS association 或更好的数据模型？
  • 是否直接访问了 SAP 标准底表？有没有 released API、released CDS view 或 RAP BO？
  • 权限检查在哪里发生？是否误以为 JOIN 自动处理权限？
  • JOIN 后是否仍然需要 ORDER BY 才能保证结果顺序？
