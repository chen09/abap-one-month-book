3. CDS annotation 注解基础

  • Chapter 03
  • Section 03
  • self-contained study note

3.1 一句话结论

CDS annotation 注解基础，是理解“元数据如何进入 CDS 模型”的入口。annotation 不是普通 comment，它是写在 CDS DDL / metadata extension 中、可被不同 SAP 框架或工具解释的 metadata：有的影响字段说明，有的影响 OData metadata，有的影响 Fiori Elements UI，有的影响 analytics，有的影响 access-control 相关检查或运行时解释。

上一节 CDS view entity 视图实体 讲的是“模型对象本身”。本节讲的是“模型上附着的语义说明”。读者要先建立一个边界：annotation 可以告诉消费者“这个字段怎么解释、怎么展示、怎么暴露、是否需要某类检查”；但 annotation 不会自动替代 ABAP 代码、DCL 规则、RAP behavior、authorization、locking、save sequence 或性能设计。

3.2 到底学什么

本节不要求一次背完所有 annotation vocabulary。初学者真正需要的是判断：这个 annotation 加在哪里、谁会读取它、它影响哪个层面、有没有放错层、是否会给下游消费者造成契约变化。

需要掌握六个判断点。

  • Metadata role：annotation 是可被工具/框架读取的元数据，不是给人看的普通注释。
  • Target / scope：annotation 可以挂在 CDS object、view element、association、parameter、metadata extension 等不同位置。
  • Consumer：不同 annotation 有不同消费者；ABAP runtime、ADT、OData、Fiori Elements、analytics、RAP 或 access-control 相关机制并不会读取同一批 annotation。
  • Vocabulary：annotation 名称和结构要符合 SAP 支持的 vocabulary / annotation definition，不要自造“看起来像配置”的字段。
  • Layering：基础语义、技术约束、UI 表达、analytics 表达、服务元数据不要无脑堆在同一个底层 view entity。
  • Boundary：annotation 只能补充模型语义和消费元数据，不能替代代码逻辑、授权规则、事务行为或测试。

3.3 为什么重要

annotation 是 CDS 从“SQL 查询”变成“语义模型”的关键原因之一。没有 annotation，view entity 仍然能投影字段和定义 association，但很多消费者无法知道字段的业务含义：哪个字段是 label，哪个金额对应哪个 currency，哪个数量对应哪个 unit，哪个字段适合搜索，哪个字段应该出现在 UI 列表，哪个 view 需要 access-control check，哪些元数据应该出现在 OData service 中。

项目交付中，annotation 的影响常常比初学者想象得大。一个看似小的 @EndUserText.label 可能影响 UI / 文档可读性；一个 @Semantics.amount.currencyCode 可能影响金额显示和接口解释；一个 UI annotation 可能改变 Fiori Elements 页面；一个 access-control 相关 annotation 可能改变读模型时的检查策略。annotation 写错，不一定导致语法报错，但可能导致服务 metadata、画面、授权或分析结果与预期不一致。

对 Java/Python 背景学习者来说，可以把 annotation 类比为 decorator、metadata attribute、schema annotation 或 OpenAPI / ORM metadata 的一部分。但 SAP 的差异是：CDS annotation 处在 ABAP repository 和 SAP framework 之间，它的消费者可能是 ADT、ABAP compiler/runtime、Gateway/OData、RAP、Fiori Elements、analytics engine 或其他 SAP 工具。读 annotation 时不能只问“它写了什么”，还要问“谁读取它”。

3.4 Key mechanism / 核心机制

3.4.1 Annotation 不是 comment

普通 comment 只给人读，系统不会按业务规则解释它。CDS annotation 以 @... 形式出现在 DDL source 中，属于模型元数据。比如：

@EndUserText.label: 'Sales Order Basic View'
define view entity ZI_SalesOrder
  as select from zsales_order
{
  key zsales_order.order_id as SalesOrder
}

@EndUserText.label 不是注释行。它会成为该 CDS object 的描述性元数据，可被工具或消费者用于显示说明。不同 annotation 的影响不同，有的只帮助可读性，有的会影响运行时或服务元数据。

3.4.2 Annotation 有 target 和 scope

annotation 必须看它挂在哪里。挂在 view entity 顶部，通常描述整个对象或对象级行为；挂在字段前，通常描述该字段；挂在 association 上，可能描述关系或消费行为；放在 metadata extension 中，则可能是把某些消费元数据从核心模型拆到单独对象。

例如：

@EndUserText.label: 'Sales Order Basic View'
@AccessControl.authorizationCheck: #CHECK
define view entity ZI_SalesOrder
  as select from zsales_order
{
  key zsales_order.order_id as SalesOrder,

  @Semantics.amount.currencyCode: 'Currency'
  zsales_order.net_amount as NetAmount,

  @Semantics.currencyCode: true
  zsales_order.currency as Currency
}

这里 object-level annotation 和 element-level annotation 的作用范围不同。@AccessControl.authorizationCheck 放在 entity 上；amount/currency 语义放在字段上。初学者 review annotation 时，第一步就是问：这个 annotation 的 target 是否正确？

3.4.3 Consumer matters：不是所有消费者都读同一批 annotation

annotation 的最大陷阱是“写了就以为生效”。实际上，annotation 必须被某个消费者理解，才会产生效果。ADT 可能显示 label；OData 可能暴露 metadata；Fiori Elements 可能读取 UI annotation 生成列表或表单；analytics 相关框架读取 analytics annotation；access-control 相关机制读取对应策略；ABAP SQL 读取 CDS 数据时，不一定关心 UI annotation。

所以要避免这种说法：“这个 annotation 会让系统自动做 X”。更准确的说法是：“在某个框架/消费场景中，这个 annotation 会被解释为 X”。项目文档中应该写清楚消费方，例如 Fiori Elements list report、OData V4 service、RAP BO projection、analytics query，而不是笼统说“annotation 会生效”。

3.4.4 常见 annotation 类型

初学者可以先把 annotation 分成五类。

  • Descriptive / semantic：描述名称、label、amount/currency、quantity/unit、text association、value help 等业务语义。
  • Technical / runtime：影响 ABAP CDS 或框架处理的技术行为，例如 access-control check 相关标记。
  • UI：影响 Fiori Elements 或 UI metadata，例如列表字段、选择字段、字段组等。
  • Service / OData：影响服务元数据、暴露方式或消费模型。
  • Analytics：影响分析场景下的度量、维度、聚合或查询语义。

这个分类不是 SAP 官方唯一分类，而是本书给初学者的阅读框架。实际项目中要回到 SAP Help / annotation vocabulary / 项目规范确认每个 annotation 的含义和支持范围。

3.4.5 Inline annotation 和 metadata extension

annotation 可以直接写在 CDS DDL source 中。基础语义、技术意图和离模型很近的说明，常常适合 inline，因为读模型时就能看到。例如 label、amount/currency、authorization check 入口等。

UI-heavy annotation 则经常需要更谨慎。UI 消费和核心数据模型变化频率不同，多个 UI 场景也可能消费同一个基础模型。如果把太多 UI 细节塞进底层 view entity，模型会被某个画面绑死。后续的 CDS metadata extension and UI annotations 会专门讨论 metadata extension：什么时候把 UI annotation 拆出去，什么时候保留在核心模型附近。

metadata extension 也不是“随便把 annotation 搬出去”的万能抽屉。能否拆出去，要看 annotation 的 target、scope、annotation category、系统版本和消费方是否支持。越靠近 compiler/runtime、access-control 入口或基础业务语义的 annotation，越要谨慎确认是否应该留在 DDL source 中；越靠近 UI 展示和消费层布局的 annotation，越值得考虑放在 projection / consumption 层或 metadata extension 中。

本节只记住原则：inline annotation 方便阅读，但不要把所有 UI/API/analytics 需求都堆在最底层 view entity。

3.4.6 Access-control annotation 和 DCL 的边界

很多初学者第一次见到 @AccessControl.authorizationCheck: #CHECK，会误以为“这个 view 已经做完授权”。这是危险的。annotation 只表达 CDS access-control 检查策略或入口，真正的访问规则通常还要看 CDS access control 访问控制 和 DCL role 访问规则。

更重要的是，CDS access control 主要围绕读模型相关的访问控制。它不替代 RAP action authorization、update authorization、UI 按钮控制、后台任务技术用户、接口用户、authorization object 检查或 audit 设计。annotation 可以参与权限模型，但不是权限模型的全部。

3.4.7 Annotation 不替代行为和代码

annotation 可以减少样板代码，让模型携带更多语义，但它不是魔法。下面这些不能靠 annotation 一句话解决：

  • 保存前校验：例如订单金额不能为负，需要 RAP validation 或 ABAP logic。
  • 派生逻辑：例如根据项目自动计算状态，需要 determination 或应用逻辑。
  • 动作：例如 release / cancel，需要 RAP action 或服务层设计。
  • 事务边界：save sequence、lock、commit、rollback 不在普通 annotation 中完成。
  • 性能设计：annotation 不能替代 SQL trace、索引、数据量评估、association/cardinality review。
  • 完整授权：annotation 不能替代所有 authorization check 和 DCL 规则。

所以 annotation 的正确定位是“让模型更可解释、更可消费、更能被框架识别”，不是“把业务逻辑从 ABAP 全部搬到 metadata”。

3.4.8 Annotation 的 review 方法

项目 review CDS annotation 时，可以按这个顺序检查：

  1. 它挂在正确 target 上吗？
  2. 它的 consumer 是谁？ADT、OData、Fiori Elements、analytics、RAP 还是 access-control？
  3. 它是否属于基础模型语义，还是 UI/API/analytics 的消费层需求？
  4. 它是否和下游契约绑定，改动是否是 breaking change？
  5. 它是否有对应的 DCL、metadata extension、service definition 或 behavior 设计配合？
  6. 它是否只是“看起来有用”，但当前消费方根本不会读取？
  7. 它是否让底层 view entity 过度耦合到某个画面或服务？
  8. 它是否只是 activation 通过，但还没有经过对应消费方的 metadata / UI / service / authorization 测试？

这个 review 方法能帮助初学者把 annotation 从“装饰语法”提升为“模型契约的一部分”。

3.4.9 Annotation consumer matrix：先问谁会读取

同样写成 @...，annotation 的影响范围可能完全不同。读 CDS 时可以先用下面的 matrix 判断它大概服务谁：

Annotation areaTypical consumerReview focus
@EndUserText...ADT、文档、UI/metadata 展示label 是否清楚、可翻译、是否适合目标语言。
@Semantics...UI、OData metadata、analytics、消费框架amount/currency、quantity/unit、date/time 语义是否成对正确。
@UI...Fiori Elements / UI metadata consumer是否属于 projection/metadata extension 层，而不是污染底层复用模型。
@OData... / service-related metadataOData/service exposure 相关消费者是否改变服务 metadata 或外部 API contract。
@Analytics...analytics engine / analytical consumermeasure、dimension、aggregation 语义是否和业务报表一致。
@AccessControl...CDS access control / DCL 相关机制是否和 DCL role、RAP authorization、authorization object 边界一致。
framework / lifecycle annotationsRAP、compiler/runtime、ADT 或生成工具是否属于当前层级，改动是否影响 activation、behavior 或 generated metadata。

一个保守规则是：annotation 越靠近 UI/API/analytics 消费，越要小心把它放在 projection 或 metadata extension 层；annotation 越像基础业务语义，例如 label、amount/currency、unit，则越适合在稳定的复用模型中统一表达。

3.5 日本项目现场说法

  • 常见日语：アノテーション、メタデータ、意味付け、項目ラベル、通貨/数量単位、UIアノテーション、メタデータ拡張、アクセス制御、DCL、消費側、公開範囲。
  • 说明句：CDSアノテーションはコメントではなく、フレームワークや消費側が解釈するメタデータです。
  • Review 句：このアノテーションはどの消費側で利用される想定か、UI・OData・分析・権限制御のどれに効くのかを明確にしてください。
  • 分层句：UI依存の強いアノテーションは、基礎ビューではなくメタデータ拡張または消費ビュー側に分離できるか確認してください。
  • 权限句：AccessControl のアノテーションだけでは権限制御は完結しないため、DCLロールとRAP側の認可設計も確認してください。

现场沟通时，アノテーションを付ける 只是动作；更重要的说明是 誰がそのアノテーションを解釈するか。如果不能说明消费方，就很难判断 annotation 是否必要。

3.7 Example / 学习例子

下面是一个教学用片段，展示 annotation 如何给 view entity 补语义：

@EndUserText.label: 'Sales Order Basic View'
@AccessControl.authorizationCheck: #CHECK
define view entity ZI_SalesOrder
  as select from zsales_order
{
  key zsales_order.order_id as SalesOrder,

  @EndUserText.label: 'Net Amount'
  @Semantics.amount.currencyCode: 'Currency'
  zsales_order.net_amount as NetAmount,

  @Semantics.currencyCode: true
  zsales_order.currency as Currency,

  zsales_order.status as SalesOrderStatus
}

这段可以这样读：

  1. @EndUserText.label 给 view 或字段提供人类可读说明。
  2. @AccessControl.authorizationCheck: #CHECK 表示该 view entity 有 access-control 检查意图，但还要查 DCL。
  3. @Semantics.amount.currencyCode: 'Currency' 把金额字段和币种字段关联起来。
  4. @Semantics.currencyCode: true 说明 Currency 字段承担 currency code 语义。
  5. 这段没有 UI line item，也没有 analytics query 语义；不要想象它自动生成完整画面或报表。
  6. 这段没有 validation、action、lock 或 save sequence；那些属于 RAP / ABAP application layer。

再看一个反例：

@UI.lineItem: [{ position: 10 }]
@Analytics.query: true
@AccessControl.authorizationCheck: #CHECK
define view entity ZI_SalesOrderEverything
  as select from zsales_order
{
  key order_id,
      net_amount,
      currency,
      internal_flag,
      technical_note
}

这不一定语法错，但设计上值得警惕：基础 view 同时背负 UI、analytics、access-control 和内部字段暴露，模型层级可能混乱。更好的做法通常是把基础语义放在基础 view，把 UI/analytics 消费语义放到 projection/consumption view 或 metadata extension，再根据消费方设计 service。

3.8 Counterexample / Failure mode

  • 误区 1：把 annotation 当普通注释。修正：annotation 是可被工具/框架解释的 metadata。
  • 误区 2：以为写了 annotation 就一定生效。修正：先确认哪个 consumer 会读取它。
  • 误区 3：把 UI annotation 堆在基础 view entity。修正：区分基础语义、projection/consumption 和 metadata extension。
  • 误区 4：以为 @AccessControl.authorizationCheck 等于完成授权。修正：还要看 DCL、RAP authorization、authorization object 和业务权限设计。
  • 误区 5：annotation 替代业务逻辑。修正：validation、determination、action、locking、save sequence 进入 RAP/ABAP 逻辑层。
  • 误区 6：自造 annotation 名称。修正：使用 SAP 支持的 annotation vocabulary，并确认系统版本和消费者支持。
  • 误区 7：改 annotation 不做测试。修正：UI/OData/analytics metadata 可能变化，需要 service preview、metadata diff、UI 回归或 API 测试。
  • 误区 8：把 activation 成功当作设计正确。修正：activation 只能证明语法和部分静态检查通过，不能证明目标 consumer 一定按预期解释。

3.9 思考问题

  • 这个 annotation 挂在 object、field、association、parameter 还是 metadata extension 上？
  • 它的消费者是谁？ADT、OData、Fiori Elements、analytics、RAP 还是 access-control？
  • 它表达的是基础业务语义、技术行为、UI 需求、analytics 需求，还是服务元数据？
  • 它是否应该留在基础 view entity，还是应该放到 projection view / consumption view / metadata extension？
  • 改动这个 annotation 会不会改变 OData metadata、UI 页面、analytics 结果或授权行为？
  • 这个 annotation 是否需要 DCL、behavior definition、service definition 或测试证据配合？
  • 在日本项目里，如何用一句话说明这个 annotation 的消费方和目的？
