37. Change request and incident workflow

  • Chapter 04
  • Section 37
  • self-contained study note

37.1 一句话结论

Change request and incident workflow 是把“发现问题或提出变更”转化为“可追踪、可评审、可测试、可移送、可关闭”的项目流程；在本教程里要学的是：日本 SAP 项目里变更申请、障害票、課題票、审批、测试、移送和关闭之间如何连接 ABAP 开发工作。

上一节 Japanese SAP deliverable documents 讲了设计、测试、移送、运维文档；本节讲这些文档在真实流程中如何流动。一个 bug 不只是“修一下代码”，它可能从障害票开始，经过原因分析、影响范围、修正方针、详细设计更新、单体测试、结合测试、移送申请、リリース判定、客户确认，最后关闭。一个新需求也不只是“加个字段”，它可能需要变更申请、估算、承认、设计、开发、测试和上线。

对 ABAP 开发者来说，理解 workflow 的意义在于：知道什么时候该写调查结果，什么时候该更新设计，什么时候不能直接改生产，什么时候需要客户承认，什么时候要把 transport、ATC、test evidence、application log 和 SAP Note 结果连到票据上。

初学者要先把 ticket 当成“项目事实的容器”，而不是聊天工具。一个 ticket 里应该能看到问题或变更从哪里来、谁判断、谁批准、改了什么、测了什么、何时移送、谁确认关闭。ABAP 开发者写的代码只是其中一段，真正交付时要把代码变更放回这条证据链。

在日本 SAP 项目里，很多沟通会围绕 影響範囲、対応方針、承認、証跡、クローズ条件 展开。能听懂这些词，才能知道自己下一步是调查、估算、修正、测试、补文档、申请移送，还是等待客户确认。

37.2 到底学什么

第一，要学票据类型。常见说法包括変更依頼、変更要求 (CR)、障害票、インシデント、課題、問い合わせ、改善要望。不同公司工具不同，但核心问题相似：谁提出、影响什么、优先级如何、谁负责、何时修、如何验证、谁承认关闭。

可以先这样粗分：障害票 / incident 偏“已经发生的问题或中断”；defect 偏测试阶段发现的缺陷；課題 / issue 偏尚未解决的设计、业务或项目问题；変更依頼 / change request 偏范围、功能或规格变更；問い合わせ / inquiry 偏确认、咨询或调查请求；SAP incident 是向 SAP 官方支持提交的问题请求。problem management 则更偏“反复发生或根因层面的问题管理”，不等同于单个 incident。

第二，要学分类。不是所有票都等同。生产障害、测试缺陷、设计课题、规格变更、数据修正、权限问题、SAP 标准限制、性能问题、运维咨询，处理路径不同。分类错误会导致修正方针、审批和 SLA 都错。

第三，要学开发者证据。ABAP 开发者通常需要提供：再现条件、原因分析、影响范围、修正对象、transport、测试结果、ログ、画面/接口证据、残留风险、回退方式。票据不是聊天记录，而是交付证据容器。

第四，要学状态流转。典型状态可能是新規、受付、調査中、対応方針確認、修正中、単体テスト中、結合テスト中、移送待ち、リリース済み、確認待ち、完了、保留、却下。实际名称因工具而异，但读者要理解每个状态背后的责任。

第五，要学“类型决定流程”。同样是“API 返回错误”，如果是 bug，要走障害/defect；如果是客户希望追加字段，要走 change request；如果是权限角色缺失，可能由权限团队处理；如果是 SAP 标准限制，可能要查 SAP Note 或开 SAP incident；如果是数据不整合，可能需要数据修正申请。分类错，后续审批、测试范围和责任都会错。

第六，要学关闭条件。票据不是开发者说“修完了”就结束。关闭通常需要测试者或业务方确认，证据完整，transport 已移送，相关设计/测试/运维文档已更新，残留风险被记录。如果没有明确关闭条件，ticket 很容易反复 reopen。

37.3 为什么重要

SAP 项目的风险不只在代码里，也在流程里。没有变更申请就改范围，可能造成费用和责任争议；没有障害票就私下修 bug，可能没有原因分析和回归测试；没有审批就移送，可能破坏冻结窗口；没有关闭条件，问题会在项目末期堆积。

这个 workflow 让团队能够回答：这个问题是什么类型？影响范围多大？谁承认修正方针？修正了哪些对象？测了什么？什么时候移送？上线后谁确认？如果复发如何追踪？这些问题都不是代码能单独回答的。

对 Java/Python 背景学习者来说，可以类比 Jira ticket、change request、incident、problem management、release approval、runbook handoff。但 SAP 场景多了 transport、client/system landscape、业务流程、Basis 运维、客户承认、外部接口和生产数据修正。

SAP 项目里的 ticket 还承担追溯功能。几个月后如果生产出现同类问题，团队会回查：当时为什么这么改？改了哪些 repository object？transport 是哪个？QAS 测试证据在哪里？客户是否承认残留风险？如果 ticket 写得太随意，维护者就只能重新考古。

对 ABAP 开发者来说，好的 ticket workflow 会保护自己。它把“我按谁的要求改”“我改了什么”“我测了什么”“哪些风险已说明”记录下来。没有这些记录，后续需求变更、障害复发或审计追问时，责任边界会变得模糊。

37.4 Key mechanism / 核心机制

可以把 workflow 分成两条主线：change request 和 incident/defect。

Change request 通常从“想改变什么”开始。流程是：需求提出、背景说明、业务价值、影响范围、估算、优先级、承认、设计、开发、测试、移送、上线、关闭。关键是 scope control：新增字段、改接口、改报表、改权限、改 job，都要明确是否属于原范围，是否影响设计和测试。

Incident / defect 通常从“出了什么问题”开始。流程是：现象记录、优先级判断、再现、一次切り分け、原因分析、暫定対応、恒久対応、修正、测试、移送、客户确认、关闭。关键是 evidence：错误消息、日志、时间、用户、输入、系统、transport、修正对象、测试结果。

ABAP 开发者在流程中的核心动作包括：

1. Reproduce:
   再现步骤、输入条件、用户、系统、时间。

2. Diagnose:
   自开发 / 配置 / 权限 / 数据 / SAP标准 / 接口 / 性能 / 运维问题。

3. Propose:
   修正方针、影响范围、测试范围、风险、工时。

4. Implement:
   修改对象、transport、code review、ATC。

5. Verify:
   单体测试、结合测试、日志、截图、API payload、job log。

6. Release:
   移送申请、承认、release note、运维手顺更新。

7. Close:
   客户确认、原因/対応内容/再発防止记录。

如果涉及 SAP Note and OSS note workflow，票据中还要记录 Note 编号、适用性判断、SAP incident 编号、SAP 回答、workaround 和恒久対応。若涉及 Production hardening checklist，还要把未解决风险和上线判定连接起来。

可以把 ticket 信息拆成一个最小字段集：

字段应写清楚的内容
Typechange / incident / defect / issue / inquiry / problem
Environmentsystem、client、user、time、release、browser/API/tool
Business impact谁受影响、影响哪个流程、严重度和优先级
Reproduction输入、步骤、期待、实际、日志和截图/API payload
Root cause自开发代码、配置、权限、数据、SAP 标准、外部系统、运维
Fix plan暫定対応、恒久対応、影响范围、测试范围、风险
Delivery evidencetransport、ATC、unit/integration test、log、review
Closure谁确认、确认环境、关闭条件、残留风险

对 ABAP 变更来说，ticket 还应能链接到具体对象：package、class、method、CDS、DCL、service binding、job、message class、application log object、authorization object、transport request。这样 reviewer 才能从业务问题追到技术变更，再从技术变更追到测试和移送。

一个好的 workflow 还有“禁止动作”。例如生产障害调查中，不应未经承认直接改生产数据；变更冻结期间，不应把无审批 transport 送进 PRD；权限问题不应先给全权限绕过；性能问题不应只凭一次测试就承诺改善。把这些禁止事项写进流程，是为了避免二次事故。

37.5 日本项目现场说法

常见日语包括：変更依頼、変更要求 (CR)、障害票、インシデント、課題票、問い合わせ、一次切り分け、原因分析、暫定対応、恒久対応、影響範囲、対応方針、承認、移送依頼、クローズ条件。

障害票中可以这样写：

  • 再現手順、発生条件、対象ユーザ、エラーメッセージ、ログ証跡を整理しました。
  • 原因は権限設定ではなく、ABAP側の例外処理漏れであると判断しました。
  • 暫定対応として手動再処理を実施し、恒久対応として修正プログラムを移送します。

变更申请中可以这样写：

  • 本変更により、対象APIのレスポンス項目を追加します。既存項目の仕様変更はありません。
  • 影響範囲はサービス定義、実装クラス、単体テスト、結合テスト、運用手順書です。
  • 移送前にATC結果、テスト証跡、リリース判定の承認を確認します。

状态更新中可以这样写：

  • 現在は一次切り分け中です。再現手順、対象ユーザ、ログ証跡を確認しています。
  • 暫定対応は完了しましたが、恒久対応として設計修正とABAP改修が必要です。
  • 修正対象、影響範囲、テスト範囲を整理し、対応方針の承認待ちです。
  • DEVで修正完了、単体テスト証跡を添付済みです。QAS移送後に結合テストを実施します。
  • 顧客確認が完了し、クローズ条件を満たしたため、本票を完了とします。

37.7 Example / 学习例子

假设 AC0567C status API 在结合测试中返回了错误状态。一个差的障害票只写：

API报错，请确认。

一个可交付的障害票应该写：

Title:
  Status API returns 500 when batch id does not exist

Environment:
  QAS / client 100 / test user TESTUSER01

Steps:
  1. Call GET /status?batch_id=NO_DATA_001
  2. Check response

Expected:
  404 or business error payload with clear message

Actual:
  HTTP 500, generic technical error

Evidence:
  request/response payload
  application log id
  timestamp
  user

Initial diagnosis:
  service implementation does not handle empty SELECT result

Fix:
  add explicit not-found handling and message

Test:
  normal id / missing id / unauthorized user / repeated request

Transport:
  DEVK9xxxxx

Closure:
  tester confirms expected payload in QAS

这个例子说明：ticket 的质量直接影响修正速度和交付可信度。技术问题需要变成结构化证据，才能进入项目流程。

再看一个 change request 示例。客户希望 AC0567C status API 追加 processedItemCount 字段。

Change Request:
  Add processedItemCount to status API response

Background:
  Operations team needs to compare uploaded item count and processed item count.

Scope:
  Add new response field only. No change to existing field semantics.

Impact:
  CDS projection, service logic, API contract, unit test,
  integration test, external client mapping, documentation.

Out of scope:
  No change to upload execution logic.
  No change to authorization rule.

Approval:
  Business owner and interface owner approval required before implementation.

Test:
  Existing client compatibility, normal status, empty batch, failed batch.

Transport:
  linked after development.

Closure:
  QAS integration test passed and interface owner confirms response mapping.

这个例子说明 change request 最怕范围滑动。只是“加字段”也可能影响 CDS、projection、API 文档、外部系统 mapping、测试用例和运维说明。写清 out of scope，可以避免开发过程中被顺手塞入不受控变更。

37.8 Counterexample / Failure mode

常见失败模式如下（约十余类）。

第一，票据没有再现步骤。开发者无法确认现象，只能靠猜。

第二，缺少环境信息。DEV/QAS/PRD、client、用户、时间、输入数据不同，原因可能完全不同。

第三，把变更当 bug 修。需求范围变化如果不走 change request，会引起工时、责任和测试范围争议。

第四，只写修了什么，不写为什么。没有原因分析和影响范围，后续问题容易复发。

第五，修正后没有测试证据。票据关闭缺乏依据。

第六，transport 和票据断开。上线后不知道某个 transport 对应哪个需求或障害。

第七，暫定対応和恒久対応混淆。临时手工处理不能替代正式修正。

第八，没有关闭条件。客户、测试、开发、运维对“完成”的理解不同，票据会反复 reopen。

第九，票据里没有影响范围。只写“改 class”不够，要说明是否影响接口契约、权限、job、CDS、transport、测试数据和运维手顺。

第十，暫定対応长期化。手动重跑、临时数据修正、权限临时放开如果没有恒久対応计划，迟早会变成生产隐患。

第十一，关闭前不更新文档。代码和测试都完成，但设计书、運用手順、release note 没更新，后续维护者仍然会按旧流程操作。

第十二，把 SAP 标准限制当自开发 bug。没有查 SAP Note、版本限制或标准文档就直接改 workaround，可能引入更大风险。

第十三，票据和 transport 多对多关系失控。一个 ticket 多个 transport、一个 transport 包含多个 ticket 都可能发生，但必须清楚记录关系，否则上线回溯会困难。
