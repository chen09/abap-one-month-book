Chapter 02 - Core ABAP Programming

27. Method visibility 可见性

  • Chapter 02
  • Section 27
  • self-contained study note

27.1 一句话结论

Method visibility 可见性是 OO ABAP class definition 中用 PUBLIC SECTION、PROTECTED SECTION、PRIVATE SECTION 控制 method、attribute、type 等 class components 谁能访问的规则。它回答的不是“这个代码能不能运行”，而是“这个能力要不要成为外部 contract、子类 contract，还是只属于本类内部实现”。

初学者最重要的判断是：public 越多，外部依赖越多，后续改动成本越高；private 越清楚，内部重构越自由；protected 不是“比 private 稍微开放一点”这么简单，它会把子类也纳入 contract，因此要结合 Inheritance 继承 谨慎使用。

27.2 到底学什么

上一节 Method signature 方法签名 讲的是 method 对外承诺“怎么调用、传什么、返回什么、可能抛什么”。本节讲的是：这个 method 该不该被外部调用者看到。

本节要掌握五个边界：

  • PUBLIC SECTION：外部调用者、其他 repository object、测试代码或框架能依赖的入口。
  • PRIVATE SECTION：本类内部实现细节，适合 helper method、内部状态和局部计算步骤。
  • PROTECTED SECTION：本类和子类可见，通常服务于 inheritance 设计。
  • PUBLIC SECTION 不等于 SAP released API，也不等于 Clean Core 允许调用。
  • visibility 不负责权限控制；业务权限要靠 Authorization check 权限检查、RAP authorization、DCL 或项目权限设计。

一个典型 class definition 会像这样：

CLASS lcl_status_service DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS get_text
      IMPORTING iv_status      TYPE string
      RETURNING VALUE(rv_text) TYPE string.

  PRIVATE SECTION.
    CLASS-METHODS normalize_status
      CHANGING cv_status TYPE string.
ENDCLASS.

get_text 是调用者真正需要的入口，因此 public。normalize_status 只是 get_text 内部用来整理输入的小步骤，因此 private。这样主流程可以使用 class 的能力，却不会依赖内部实现步骤。

27.3 为什么重要

在项目里，visibility 是设计质量最早暴露的位置之一。很多“后期不好改”的问题，不是因为实现很复杂，而是因为一开始把太多东西放进了 public。

例如：

  • public method 太多，调用者不知道哪个才是稳定入口。
  • public attribute 直接暴露内部状态，任何调用者都能改，类的不变量失效。
  • 本来只是 private helper 的 method 被外部对象调用，后续想重构就会破坏调用方。
  • protected method 被子类依赖，父类内部改动会影响继承链。
  • 把 PUBLIC SECTION 当成“授权通过”，忽略了真正的 SAP authorization。

Java/Python 背景学习者可以把它类比为 access modifier，但 ABAP 项目还要多想一层：global class 是 repository object，会被 transport，会被其他对象 where-used，会进入项目 review；public surface 一旦被别人用上，就不只是“一个类里的小改动”。

27.4 Key mechanism / 核心机制

27.4.1 visibility section controls components

ABAP class definition 里，components 放在哪个 section 下，就决定了它们的可见性：

CLASS lcl_order_rule DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS is_valid
      IMPORTING is_order        TYPE zstr_order
      RETURNING VALUE(rv_valid) TYPE abap_bool.

  PROTECTED SECTION.
    " reserved for subclass extension when inheritance is intentionally designed

  PRIVATE SECTION.
    CLASS-METHODS has_items
      IMPORTING is_order        TYPE zstr_order
      RETURNING VALUE(rv_valid) TYPE abap_bool.

    CLASS-METHODS has_customer
      IMPORTING is_order        TYPE zstr_order
      RETURNING VALUE(rv_valid) TYPE abap_bool.
ENDCLASS.

外部调用者只需要知道：

IF lcl_order_rule=>is_valid( ls_order ) = abap_true.
  " ...
ENDIF.

外部不应该知道 has_items 和 has_customer 两个内部步骤。这样以后内部校验规则可以拆分、合并、改名，而 public contract 不变。

27.4.2 public is contract

PUBLIC SECTION 的 method 是外部可见入口。更精确地说：public component 对 class 自身可见，也对所有能够引用这个 class 的使用者可见。对 Global class 全局类 来说，这通常意味着其他 repository object 只要能引用该 class，就可能调用它；对 Local class 局部类 来说，这只发生在 local context 内，不会变成系统级 public API。无论是 global 还是 local，public surface 都应该小而稳定。

这里还要加一个 SAP 项目里的现实提醒：如果 global class 后续被 release 成稳定接口，public components 的变更会受到更强约束。因此 public method 不只是“今天能被调用”，也可能成为以后 transport、where-used、回归测试和升级兼容性都要保护的 contract。

好的 public method 通常具备这些特征：

  • 名字表达业务能力，而不是内部步骤。
  • Method signature 方法签名 清楚，输入输出少而明确。
  • 不暴露临时结构、内部缓存、技术中间状态。
  • 失败路径明确：返回 result object、RAISING class-based exception，或遵守 RAP framework 约定。
  • 可以被 ABAP Unit 单元测试基础 通过行为测试验证。

坏的 public surface 往往像这样：

CLASS lcl_price_calculator DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-DATA gv_discount TYPE decfloat34.

    CLASS-METHODS read_customizing.
    CLASS-METHODS calculate_step_1.
    CLASS-METHODS calculate_step_2.
    CLASS-METHODS round_internal_amount.
    CLASS-METHODS clear_cache.
ENDCLASS.

这会把内部步骤、缓存状态、计算顺序都暴露出去。调用方可能开始直接调用 calculate_step_1 或修改 gv_discount，之后 class 内部就很难重构。

更合理的 public surface 是：

CLASS lcl_price_calculator DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS calculate
      IMPORTING is_item         TYPE zstr_item
      RETURNING VALUE(rv_price) TYPE decfloat34
      RAISING   zcx_price_error.

  PRIVATE SECTION.
    CLASS-METHODS read_customizing
      RETURNING VALUE(rs_config) TYPE zstr_price_config.

    CLASS-METHODS round_amount
      CHANGING cv_amount TYPE decfloat34.
ENDCLASS.

调用者只依赖 calculate。内部怎样读 customizing、怎样 round，都留给 class 自己。

27.4.3 private is implementation freedom

PRIVATE SECTION 是本类内部的实现空间。private method 的价值不是“藏起来显得高级”，而是让 public method 更短、更清楚，让内部步骤可以命名、测试间接验证、必要时重构。

严格说，private component 也可能被声明为 friend 的 class 访问；FRIENDS 是 OO ABAP 中明确的语言级例外。初学阶段先把 private 理解为“外部普通调用者不应依赖的实现细节”，等学到 friend class、test seam 或框架扩展时，再补充这个例外即可。

private method 适合：

  • 输入规范化。
  • 小的计算步骤。
  • 格式转换。
  • 内部判断。
  • 只服务于当前 class 的 helper。
  • 不应该被外部调用者直接依赖的缓存或状态管理。

private 也不是万能。如果某个 private helper 开始被多个 class 复制，说明它可能已经不只是 private helper，而是应该抽成新的 Global class 全局类、Interface 接口定义 或稳定工具类。visibility 不能用来掩盖职责边界错误。

还有一个容易忽略的封装漏洞：private attribute 本身不可被普通外部调用者直接访问，但 class method 如果把指向内部 attribute 的 data reference 或 field symbol 传出去，外部仍可能间接看到甚至修改内部状态。因此 private 只是访问规则，不能替代谨慎的 API 设计。

27.4.4 protected is inheritance contract

PROTECTED SECTION 常被误解为“给未来留扩展点”。实际上 protected 是给子类看的 contract。只要子类开始依赖 protected method，父类就不能随便改它。

和 private 类似，protected 也有 friend class 例外：被声明为 friend 的 class 可以访问相应 component。另一个更细的 ABAP 规则是：子类并不是在任何情况下都能通过“父类静态类型的引用变量”随意访问父类 protected component；继承访问仍然受对象动态类型和引用静态类型约束。初学阶段不需要死记这条边界，但要知道 protected 不是“随便给继承树开后门”。

例如：

CLASS lcl_base_formatter DEFINITION.
  PUBLIC SECTION.
    METHODS format
      IMPORTING iv_text        TYPE string
      RETURNING VALUE(rv_text) TYPE string.

  PROTECTED SECTION.
    METHODS normalize
      IMPORTING iv_text        TYPE string
      RETURNING VALUE(rv_text) TYPE string.
ENDCLASS.

如果后续子类 redefine 或调用 normalize，父类内部调整就会影响子类。初学阶段不应把 protected 当作默认选择。除非正在明确设计 Inheritance 继承 或 framework extension point，否则先用 private；确实需要外部稳定调用时，再设计 public method 或 interface。

27.4.5 public section is not released API

这是 SAP 项目里非常重要的区别：ABAP 的 PUBLIC SECTION 只说明 ABAP 语言层面可以访问，并不自动说明它是 SAP released API，也不自动符合 Clean Core。

在 ABAP Cloud / Clean Core 语境中：

  • 一个 class 的 public method 可以是语言层面的 public，但仍然不是 released API。
  • 只有被 SAP 或项目治理明确释放、允许使用的 API 才能作为稳定依赖。
  • 自定义 global class 的 public method 也要遵守 package/use-access、released API、dependency boundary 等规则。
  • RAP handler、behavior implementation、saver、authorization 等 framework 方法的可见性和签名通常由框架或父类/接口约束，不应随意扩大 public surface。

所以，review 时不能只问“能不能调用”。还要问：“应不应该调用？这是不是稳定契约？是不是 Clean Core 允许的依赖？”

27.4.6 visibility is not authorization

visibility 不是权限检查。PRIVATE SECTION 防止外部代码直接访问内部 component；它不防止业务用户执行某个操作。PUBLIC SECTION 也不代表所有业务用户都有权限。

例如一个 public method：

METHODS approve_order
  IMPORTING iv_order_id TYPE zorder_id
  RAISING   zcx_authorization_error.

它是 public，因为外部应用层需要调用审批动作。但 method 内部或上游 RAP authorization 仍然必须检查用户是否有审批权限。visibility 管代码访问边界；authorization 管业务访问权限。

27.5 日本项目现场说法

常见日语：

  • 可視性
  • 公開範囲
  • PUBLIC SECTION
  • PROTECTED SECTION
  • PRIVATE SECTION
  • 公開メソッド
  • 内部実装
  • 外部契約
  • パブリックコントラクト
  • 副作用
  • 継承

现场说明句：

  • このメソッドは外部から呼び出す必要があるため、PUBLIC SECTION に置いています。
  • この処理は内部実装なので、PRIVATE SECTION に移して公開範囲を狭くしています。
  • PUBLIC SECTION が大きすぎるため、外部契約として必要なメソッドだけに絞ってください。
  • PROTECTED SECTION は継承前提の契約になるため、安易に増やさない方がよいです。
  • PUBLIC SECTION であっても released API とは限らないため、Clean Core の観点で利用可否を確認してください。
  • 可視性は権限制御ではないため、業務権限は別途 authorization で確認してください。

日本项目 review 中，公開範囲を絞る、外部契約を明確にする、内部実装を隠蔽する 这些表达很实用。它们通常是在要求把“能调用”改成“应该调用的入口才公开”。

27.7 Example / 学习例子

27.7.1 例 1：把内部步骤藏到 private

不好的版本：

CLASS lcl_status_service DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS get_text
      IMPORTING iv_status      TYPE string
      RETURNING VALUE(rv_text) TYPE string.

    CLASS-METHODS normalize_status
      CHANGING cv_status TYPE string.
ENDCLASS.

normalize_status 是内部步骤，不应该让调用者直接依赖。改成：

CLASS lcl_status_service DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS get_text
      IMPORTING iv_status      TYPE string
      RETURNING VALUE(rv_text) TYPE string.

  PRIVATE SECTION.
    CLASS-METHODS normalize_status
      CHANGING cv_status TYPE string.
ENDCLASS.

CLASS lcl_status_service IMPLEMENTATION.
  METHOD get_text.
    DATA(lv_status) = iv_status.

    normalize_status(
      CHANGING cv_status = lv_status ).

    CASE lv_status.
      WHEN 'O'.
        rv_text = 'Open'.
      WHEN 'C'.
        rv_text = 'Closed'.
      WHEN OTHERS.
        rv_text = 'Unknown'.
    ENDCASE.
  ENDMETHOD.

  METHOD normalize_status.
    CONDENSE cv_status.
    cv_status = to_upper( cv_status ).
  ENDMETHOD.
ENDCLASS.

调用者只知道 get_text。内部 CONDENSE、to_upper、case mapping 都可以改，不破坏 public contract。

27.7.2 例 2：public attribute 的风险

不建议：

CLASS lcl_counter DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-DATA gv_count TYPE i.
ENDCLASS.

任何调用者都可以改 gv_count，class 无法保证状态合法。更好的写法是：

CLASS lcl_counter DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS increment.
    CLASS-METHODS get_count
      RETURNING VALUE(rv_count) TYPE i.

  PRIVATE SECTION.
    CLASS-DATA gv_count TYPE i.
ENDCLASS.

这样外部只能通过 method 使用状态，class 可以控制如何变更。

如果确实需要让外部读取某个稳定状态，也可以在 PUBLIC SECTION 中使用 READ-ONLY attribute，让外部只能读、不能直接写。它不是 private + getter 的完全替代品，但比完全可写的 public attribute 更安全。

27.7.3 例 3：protected 不是默认扩展点

如果还没有明确的继承设计，不要为了“以后可能要扩展”先把 helper 放 protected：

PROTECTED SECTION.
  METHODS calculate_internal_score.

这会暗示子类可以依赖它。初学阶段更保守的做法：

PRIVATE SECTION.
  METHODS calculate_internal_score.

等真的出现继承需求，再设计 protected contract，并配合 Inheritance 继承 说明子类应该如何使用。

27.8 Counterexample / Failure mode

27.8.1 误区 1：先 public，之后再收回

public method 一旦被调用方使用，再改名、改参数、改返回值都会影响 where-used。项目里“先 public 以后再说”通常会变成长期技术债。

修正动作：默认 private；只有外部真正需要依赖的入口才 public。

27.8.2 误区 2：public attribute 暴露内部状态

public attribute 让外部绕过 class 的业务规则直接改状态。更准确地说，风险最大的是可被外部直接写入的 public instance attribute 或 public class attribute；除非是非常稳定、只读、常量式的 public constant，或者明确使用 READ-ONLY 控制写入，否则业务状态不应轻易 public。

修正动作：用 private attribute + public method 管理读取和变更；必要时返回只读结果或结构副本。

27.8.3 误区 3：protected 滥用造成继承耦合

protected method 被子类调用后，父类内部实现就变成子类 contract。后续重构需要考虑整个继承树。

修正动作：没有明确继承设计时用 private；需要多态扩展时，先设计清楚 abstract/final/redefinition 规则。

27.8.4 误区 4：把 visibility 当权限控制

PUBLIC SECTION 只是代码可访问，不代表业务用户有权限；PRIVATE SECTION 只是代码不可访问，不代表业务动作安全。

修正动作：权限逻辑回到 Authorization check 权限检查、RAP authorization、DCL、业务角色和审计要求。

27.8.5 误区 5：把 PUBLIC SECTION 当 released API

在 SAP 标准对象或 ABAP Cloud/Clean Core 场景中，能在 ABAP 语法上看到 public method，不等于它是 released API。

修正动作：查 released API 状态、package/use-access、SAP Help / SAP API Business Hub / ADT release contract；不要只凭 PUBLIC SECTION 判断能不能依赖。

27.9 思考问题

  • 这个 method 是否真的需要外部调用？如果不是，为什么不是 private？
  • 这个 public method 是否表达业务能力，而不是内部步骤？
  • 如果明天重命名 private helper，会影响哪些调用者？
  • 这个 protected method 是否已经承诺给子类使用？有没有对应继承设计？
  • 这个 public attribute 是否会让外部绕过业务规则改状态？
  • 如果必须暴露 attribute，它是否应该是 READ-ONLY，还是应该改成 private attribute + method？
  • PUBLIC SECTION 中的类型和方法是否会变成长期 contract？
  • 这个 public method 是否还需要 authorization、RAP authorization 或 DCL，而不是只靠 visibility？
  • 在 Clean Core 语境下，它是否是 released API 或项目允许依赖的 API？
