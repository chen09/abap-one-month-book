24. EML Entity Manipulation Language

  • Chapter 03
  • Section 24
  • self-contained study note

24.1 一句话结论

EML Entity Manipulation Language 是 ABAP 中用于访问 RAP business object 的专用语言子集；它让 ABAP 程序、测试代码和 behavior implementation 用 READ ENTITIES、MODIFY ENTITIES、COMMIT ENTITIES 等语句按 RAP BO 契约读取、修改和提交业务对象，而不是绕过 BO 直接操作底层表。

一句话记忆：SQL 面向表，EML 面向 RAP BO；SQL 问“表里有什么”，EML 问“这个业务对象允许我读、改、执行 action、提交吗”。

24.2 概念说明

ABAP Keyword Documentation 对 EML 的定义很清楚：ABAP Entity Manipulation Language 是 ABAP 的一个子集，用来访问 RAP BO 和 RAP BO interface。SAP Learning 也说明，EML statements 允许读取或修改 RAP BO 的 data content / transactional buffer，并触发 modified data 的持久化。

EML 不是 SQL 的新写法。SELECT 从数据库表或 CDS entity 读取数据；EML 则通过 RAP runtime 访问 BO。它会进入 behavior definition、behavior pool、handler、saver、authorization、validation、determination、action、draft、lock、save sequence 等 RAP 生命周期。用 EML 访问 BO 时，调用者面对的是 BO 契约，而不是裸表。

EML 可以在普通 ABAP 程序中作为 BO consumer 使用，也可以在 behavior pool 内部使用。SAP Learning 特别提醒：在 behavior implementation 中也可以使用 EML，而且有一些 special variants。最常见的就是 handler method 内部使用 IN LOCAL MODE，读取或修改当前 BO 的 transactional buffer，而不是绕过 RAP runtime。

初学者要先把 EML 的三类动作分清楚：

  • READ ENTITIES：读取 RAP BO entity 实例。
  • MODIFY ENTITIES：执行 create、update、delete、create by association、action 等修改型操作。
  • COMMIT ENTITIES：结束 RAP LUW，触发 save sequence，把 transactional buffer 中的修改提交到持久层。

24.3 为什么重要

EML 是 RAP 世界里的“内部 API”。如果只会 CDS 和 OData，不会 EML，就很难写 action、determination、unit test、report 调用 RAP BO、后台任务消费 BO、或者在 behavior implementation 中安全读写当前事务数据。

它也改变了 ABAP 开发者的思维。传统 ABAP 常见做法是 SELECT、UPDATE、CALL FUNCTION、COMMIT WORK。RAP 中，若业务对象已经建成 BO，很多操作应该优先通过 EML 走 BO 契约。这样 validation、authorization、lock、draft、numbering、action result、message 和 save sequence 都能保持一致。

对项目交付来说，EML 直接影响测试和集成。ABAP Unit 可以用 EML 构造 BO 数据并验证 action；migration / batch / report 可以用 EML 调用 BO 而不是复制保存逻辑；故障分析时也可以从 failed / reported / mapped 看出到底是授权、validation、编号、save sequence 还是 action 失败。

对 Java/Python 背景学习者，可以把 EML 类比成类型安全的 domain API / repository API，但它不是普通 library call。它会触发 RAP runtime，并且以 RAP LUW 和 COMMIT ENTITIES 为事务边界。

24.4 边界

第一条边界：EML 不是 SQL。READ ENTITIES 不是 SELECT 的替代拼写；它访问的是 BO 实例，返回的是 BO 契约下的数据和响应。若只需要分析报表或大批量读取，仍可能使用 CDS/ABAP SQL；若要遵守 BO 行为、draft、authorization、action 和 save sequence，就应考虑 EML。

第二条边界：MODIFY ENTITIES 不等于立即写库。它通常修改 RAP transactional buffer。只有 COMMIT ENTITIES 触发后，RAP save sequence 才会运行。初学者最常见错误是执行 MODIFY 后去数据库表查，发现没有数据，就误以为 EML 失败；其实只是还没有 commit RAP LUW。

第三条边界：COMMIT ENTITIES 不是 COMMIT WORK 的同义词。RAP consumer 使用 COMMIT ENTITIES 提交 RAP BO 修改；behavior implementation 内部不能随意 COMMIT WORK。是否允许以及何时提交，必须遵守 RAP transaction boundary。

第四条边界：behavior pool 内部使用 EML 时要理解 IN LOCAL MODE。Local mode 用于在当前 BO 实现中访问 transactional buffer，常见于 action / determination 中读取或修改同一个 BO 的实例。它不是给外部 consumer 绕过权限和校验的后门。

第五条边界：EML 不是 OData。OData 是远程服务协议，EML 是 ABAP 内部访问 RAP BO 的语句。Fiori / HTTP consumer 通过 service binding / OData 访问；ABAP 程序和 behavior implementation 则可以通过 EML 访问同一 BO 契约。

24.5 核心机制

第一层机制是 BDEF-derived types。EML 的输入输出结构不是随便声明的内表，而是从 CDS view 和 behavior definition 派生出来的类型，例如 TYPE TABLE FOR READ IMPORT、TYPE TABLE FOR CREATE、TYPE TABLE FOR UPDATE、TYPE TABLE FOR ACTION IMPORT、TYPE TABLE FOR EVENT 等。这样 EML 语句和 BO 契约保持类型一致。

第二层机制是 response tables。几乎所有重要 EML modify/read 操作都要看 FAILED、REPORTED，create / numbering 相关场景还要看 MAPPED。FAILED 告诉你哪些实例失败；REPORTED 返回消息；MAPPED 把临时 %cid 映射到最终 key。不要只看语句有没有语法错误，更要看 response。

第三层机制是 technical keys 和 content ID。RAP 中常见 %tky、%key、%cid、%cid_ref、%is_draft、%control 等特殊组件。%tky 可以先理解为 transactional key，即 key 与 draft 状态等事务上下文组合出来的技术标识；%key 是业务对象实例的 key；%cid 用于一次 EML create 请求中临时引用新对象；%cid_ref 常用于 create by association，把 child 创建挂到 parent；%control 指示 update 中哪些字段被请求修改。

第四层机制是 read。典型语法是：

READ ENTITIES OF zi_travel
  ENTITY Travel
  ALL FIELDS WITH lt_keys
  RESULT DATA(lt_travel)
  FAILED DATA(ls_failed)
  REPORTED DATA(ls_reported).

它读取的是 RAP BO entity，不是直接读表。是否读到 draft、是否含 transactional changes、是否受 authorization / DCL / behavior 影响，要结合具体 BO、调用位置和语句变体判断。

第五层机制是 modify。典型 create/update/delete/action 通过 MODIFY ENTITIES 发起：

MODIFY ENTITIES OF zi_travel
  ENTITY Travel
  CREATE FROM lt_create
  MAPPED DATA(ls_mapped)
  FAILED DATA(ls_failed)
  REPORTED DATA(ls_reported).

这一步把请求交给 RAP runtime。Managed BO 可能由 runtime 处理标准 operation；unmanaged BO 可能转发给 handler class；action 会调用对应 action handler。它不是裸 SQL insert。

第六层机制是 commit。外部 ABAP consumer 或 EML test scenario 通常需要：

COMMIT ENTITIES
  RESPONSE OF zi_travel
  FAILED DATA(ls_commit_failed)
  REPORTED DATA(ls_commit_reported).

COMMIT ENTITIES 触发 save sequence。没有这一步，前面的 modify 可能只停留在 RAP transactional buffer。行为实现内部是否需要或允许 commit，要看是否是 consumer 代码还是 behavior implementation；不要在 handler/saver 中随意提交。

第七层机制是 local mode。Behavior implementation 中经常写：

READ ENTITIES OF zi_travel IN LOCAL MODE
  ENTITY Travel
  FIELDS ( Status TotalPrice )
  WITH keys
  RESULT DATA(lt_travel)
  FAILED DATA(ls_failed)
  REPORTED DATA(ls_reported).

IN LOCAL MODE 表示在当前 BO 的本地实现上下文中访问 transactional buffer。它适合 determination/action 内部读取当前状态、更新派生字段、避免直接 SQL 绕过 buffer。因为 local mode 面向 BO 自身实现，可能跳过普通外部 consumer 路径中的 authorization control 或 feature control，所以它应该出现在 behavior implementation 的受控语境里，而不是被当作外部调用者绕过规则的手段。

第八层机制是 action 与 create by association。EML 不只 CRUD，还可以执行 action、create by association、deep create 等。Composite BO 中常见 parent 和 child 一起创建，使用 %cid / %cid_ref 建立临时关系，commit 后 runtime 再完成真实 key 和保存。

24.6 项目意义

设计评审时，EML 要回答：“这个 ABAP 程序为什么要通过 BO，而不是直接访问表？”如果需求涉及业务规则、状态转换、authorization、draft、action、validation、numbering、消息返回，EML 通常更合适。如果只是大批量分析读数，ABAP SQL/CDS 可能更合适。

代码 review 时，重点看四类问题。第一，modify 后是否处理 failed / reported / mapped。第二，是否忘记 COMMIT ENTITIES。第三，behavior implementation 内部是否错误使用外部 consumer 风格 EML。第四，是否用 SELECT / UPDATE 绕过 BO，导致 validation、authorization、draft、lock 全部失效。

测试时，EML 非常有价值。单体测试或集成测试可以通过 EML create 数据、执行 action、检查 reported message、commit 后验证结果。相比直接插表，EML 更接近真实 consumer 行为，也更能发现 BDEF / behavior pool / save sequence 问题。

日本项目里，EML 经常出现在“RAP BO を ABAP から呼び出す”“テストコードで BO を操作する”“action を EML で実行する”“COMMIT ENTITIES を忘れていないか”这类讨论中。它是 RAP 项目调试和测试的关键工具。

24.7 Example / 学习例子

假设要在 ABAP report 或 test 中创建一个 Travel。第一步准备 create table，包含临时 %cid：

DATA lt_create TYPE TABLE FOR CREATE zi_travel\Travel.

lt_create = VALUE #(
  ( %cid = 'CID_001'
    AgencyId = '070001'
    CustomerId = '000123'
    BeginDate = cl_abap_context_info=>get_system_date( )
    EndDate = cl_abap_context_info=>get_system_date( ) + 3
    %control-AgencyId = if_abap_behv=>mk-on
    %control-CustomerId = if_abap_behv=>mk-on
    %control-BeginDate = if_abap_behv=>mk-on
    %control-EndDate = if_abap_behv=>mk-on ) ).

第二步用 EML create：

MODIFY ENTITIES OF zi_travel
  ENTITY Travel
  CREATE FROM lt_create
  MAPPED DATA(ls_mapped)
  FAILED DATA(ls_failed)
  REPORTED DATA(ls_reported).

这时要先看 ls_failed 和 ls_reported。如果 create 被 validation、authorization、numbering 或 handler 拒绝，不能直接继续假设成功。若 successful create 使用临时 %cid，还要从 ls_mapped 取得最终 key。

第三步提交：

COMMIT ENTITIES
  RESPONSE OF zi_travel
  FAILED DATA(ls_commit_failed)
  REPORTED DATA(ls_commit_reported).

只有 commit 后，save sequence 才真正运行。若使用 managed save，runtime 写主表；若使用 unmanaged save，saver class save() 可能参与；若使用 adjusted saver / additional save，save_modified 或附加保存逻辑可能在保存阶段发生。这个例子把前面几节串起来：BDEF 定义契约，handler 处理行为，saver 处理保存，EML 作为 ABAP consumer 调用 BO。

24.8 Counterexample / Failure mode

第一个失败模式是把 EML 当 SQL。开发者用 MODIFY ENTITIES 之后马上查数据库，发现没有记录，就在旁边加 COMMIT WORK。正确做法是理解 RAP LUW：外部 consumer 用 COMMIT ENTITIES，不要用 COMMIT WORK 破坏 RAP 边界。

第二个失败模式是不看 response。FAILED / REPORTED 里已经有 validation 错误或 authorization 错误，代码却只判断内表是否 initial，继续后续逻辑。结果是后续 key 空、消息丢失、UI 或 report 只显示“保存失败”。

第三个失败模式是 update 忘记 %control。EML update 中没有正确标记字段控制，runtime 或 handler 可能不知道哪些字段要更新；反过来，把未修改字段也标成 changed，可能覆盖旧数据。

第四个失败模式是 %cid / %cid_ref 用错。Deep create 时 child 没有正确引用 parent 临时 ID，导致 child 创建失败或挂错父对象。Composite BO 中这是非常常见的初学错误。

第五个失败模式是在 behavior pool 内部不用 IN LOCAL MODE，而是直接 SELECT 表来读当前状态。这样读不到 transactional buffer 中尚未提交的变化，determination/action 会基于旧数据计算。

第六个失败模式是把 EML 当远程 API。EML 只能在 ABAP 内部使用；外部系统要通过 OData/service binding/API，而不是“调用 EML”。设计接口时要区分 ABAP 内部消费和外部消费。

24.9 日本项目现场说法

  • 常见日语：EML、エンティティ操作言語、READ ENTITIES、MODIFY ENTITIES、COMMIT ENTITIES、トランザクショナルバッファ、ローカルモード、failed / reported / mapped。
  • 说明句：EML は ABAP 内部から RAP BO を操作するための言語要素で、テーブルを直接更新するのではなく BO の振る舞い定義に従って処理します。
  • Review 句：MODIFY ENTITIES の後に FAILED、REPORTED、MAPPED を確認しているか、最後に COMMIT ENTITIES が必要かを確認してください。
  • 边界句：外部システムから EML を直接呼び出すことはできません。外部公開は OData / service binding 側で設計します。
  • 调查句：保存されていない場合、MODIFY ENTITIES だけで終わっていないか、COMMIT ENTITIES が実行されているかを確認します。
  • Local mode 句：behavior implementation 内では IN LOCAL MODE を使い、現在の transactional buffer を参照する必要があります。
