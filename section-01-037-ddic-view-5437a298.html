37. DDIC view 传统视图

  • Chapter 01
  • Section 37
  • self-contained study note

37.1 一句话结论

DDIC view 传统视图，是 ABAP Dictionary 里较早一代的 view 机制，用来把一个或多个 DDIC database tables 的列组合成面向应用的视图。它在遗留 ABAP、ECC、on-premise S/4HANA 和日本 brownfield 项目里仍然可能出现，但已经不是现代数据建模、RAP/Fiori/OData 服务设计的主路径。

本节要学的是：看懂 classic DDIC view 在旧系统中的位置，能区分它和 CDS view entity 视图实体、CDS DDIC-based view、database table、DDIC structure 字典结构 的边界，并形成一个实战判断：读旧系统时要认识 DDIC view，写新模型时要优先走 CDS view entity、service definition、service binding 和 RAP/CDS 元数据。

37.2 到底学什么

先把名字说清楚。这里的 DDIC view，主要指 classic ABAP Dictionary view，尤其是 database view。SAP Keyword Documentation 对它的基本定义是：DDIC view 是把一个或多个 DDIC database tables 的列分组形成 application-specific view。它通常用来表达一段 join 或一个特定用途的读模型。它不像 database table 那样保存自己的业务数据，数据仍然来自底层表；view 激活后可以在数据库层有对应的 generated SQL view artifact，但不要把它理解成“又多了一张保存数据的表”。

在 Standard ABAP 的 legacy 世界里，DDIC view 可以作为 ABAP SQL 的数据源，也可以像扁平结构一样作为 structured type 使用。也就是说，旧代码里可能看到类似：

SELECT * FROM zxx_legacy_v INTO TABLE @DATA(lt_result).

DATA ls_view_line TYPE zxx_legacy_v.

第一句把 view 当作读数据源，第二句把 view name 当作结构类型名。这个能力会让初学者误以为 DDIC view 和 DDIC structure 字典结构 很像。关键差异是：DDIC structure 只定义全局结构类型，不代表数据库读模型；DDIC database view 首要目的仍然是定义一个数据库视图式的数据访问对象，同时附带可作为 structured type 的效果。

再看现代边界。SAP Learning 在 ABAP Dictionary 入门课里明确说明：过去 database views 由 ABAP Dictionary 处理，但这些 ABAP dictionary views 已经过时，应该使用 ABAP Core Data Services views。SAP-samples ABAP SQL cheat sheet 也把 classic DDIC views 归为 oldest form of views，说明它们不适用于 ABAP Cloud，且只能在 ABAP Workbench 里创建。这意味着本教程不应把 DDIC view 教成“新开发的推荐方式”，而应把它教成 legacy awareness：你需要能读懂、能影响分析、能迁移判断，但不要在现代 RAP/Fiori 项目里优先新建它。

本节还要学会区分三组容易混淆的对象。

第一组是 classic DDIC view 和 CDS view entity。classic DDIC view 是旧式 ABAP Dictionary view；CDS view entity 是现代 ABAP CDS 数据模型对象，用 define view entity 定义，没有旧式 @AbapCatalog.sqlViewName 所代表的 generated SQL view name。现代 RAP/OData 服务的数据模型通常应围绕 CDS view entity、projection view、metadata extension、behavior definition、service definition 和 service binding 展开。

第二组是 CDS DDIC-based view 和 CDS view entity。CDS DDIC-based view 是旧 CDS V1 写法，通常用 define view，并通过 @AbapCatalog.sqlViewName 生成一个 CDS-managed DDIC view。SAP Help 把 CDS DDIC-based views 标成 obsolete，并说明 CDS view entities 是继任者。SAP Gateway 课程也说明，从 ABAP Platform 7.55 起，SAP 推荐使用 CDS view entities；view entity 不再有 SQL view 可以像旧 SEGW 流程那样导入。

第三组是 DDIC database view、maintenance view、help view、projection view 和 external view。真实旧系统里，DDIC views 不只 database view。maintenance view 常和表维护、SM30 类流程有关；help view 常和 F4/search help 场景有关；classic projection view 是旧式 DDIC 视图子类型，不要和现代 CDS projection view 混为一谈；external view 是把外部 SAP HANA view 暴露给 ABAP 的 DDIC proxy 机制，也属于需要 legacy 判断的对象。初学阶段不需要掌握每一种创建细节，但必须知道“DDIC view”不是一个现代开发万能词。

37.3 为什么重要

DDIC view 重要，不是因为新开发一定要写它，而是因为日本 SAP 项目经常不是一张白纸。很多项目是 ECC 维护、S/4HANA conversion、on-premise brownfield、旧 Gateway 服务改造或 Clean Core 评估。这样的系统里，旧 ABAP SQL、SE11/SE80 对象、SEGW import 记录、接口程序、报表程序、甚至设计书里都可能出现 DDIC view 或 generated SQL view name。

如果初学者只学 CDS/RAP，而完全不认识 DDIC view，读旧代码时会卡在两个问题上。第一个问题是“这个对象到底是什么”。它看起来像表名，可以 SELECT；又看起来像结构名，可以 TYPE；但它不是业务主表，也不是纯结构。第二个问题是“该不该继续用”。如果只是维护旧报表或做影响分析，DDIC view 是必须识别的既存资产；如果是做新的 RAP/Fiori 服务，把它继续当主模型就会把项目带回旧技术栈。

对 Java/Python 背景学习者，可以把 DDIC view 粗略类比成数据库 view 加一个全局结构类型名，但 SAP 里的差异更重。它受 ABAP Dictionary、activation、transport、client handling、ABAP SQL、where-used list、Cloud readiness、Clean Core 约束影响。你不能只按普通 SQL view 来处理，也不能绕过 AS ABAP 直接从数据库访问它。SAP 的设计原则是：ABAP-managed database objects 应通过 AS ABAP 的数据库接口访问，项目上也应优先通过 ABAP SQL、CDS、RAP 和官方 release contract 判断可用性。

DDIC view 还关系到迁移风险。旧 CDS DDIC-based view 可能生成 SQL view name，老代码可能直接 SELECT 这个 SQL view name。后续迁移到 CDS view entity、ABAP Cloud 或更严格的语法检查时，这类访问会被视为 obsolete 或不兼容。一个合格的初学者不需要当天就会重构所有 legacy views，但要能在 review 时说清楚：这是 legacy artifact，当前只是保守维护；新增能力应寻找 CDS view entity 和 released API 方向。

37.4 Key mechanism / 核心机制

第一层机制是 classic DDIC database view。它把底层 DDIC database table 的字段组合成应用视图，常见用途是读数据和表达 join。它不保存自己的业务数据；数据来自底层表。它的列形成一个 flat structure，所以在 classic ABAP 中也能作为 structured type 使用。写操作要特别谨慎：通过 classic DDIC view 写数据只有在 view 基于单个 database table 时才有支持空间，多表 view 应按 read-only legacy artifact 理解。

第二层机制是创建与工具边界。classic DDIC views 属于旧式 ABAP Workbench / SE11 世界，不是 ABAP Cloud 里通过 ADT 新建的现代对象。SAP-samples 文档明确说 classic DDIC views 不适用于 ABAP Cloud，也只能在 ABAP Workbench 创建。学习时不要把“ADT 里写 CDS”与“SE11 里创建 DDIC view”混成一个动作。现代新建数据模型应从 CDS view entity 开始。

第三层机制是 CDS V1 与 CDS view entity 的差异。旧 CDS DDIC-based view 通常长这样：

@AbapCatalog.sqlViewName: 'ZXXSQLV'
define view ZXX_CDS_V1
  as select from zxx_table
{
  key field1,
      field2
}

现代 CDS view entity 的方向是：

define view entity ZXX_I_Invoice
  as select from zxx_invoice
{
  key invoice_id,
      company_code,
      gross_amount
}

注意两个重点：define view entity 不再使用 @AbapCatalog.sqlViewName；ABAP SQL 应面向 CDS entity / view entity 名称，而不是旧 generated SQL view name。SAP Learning 的 Gateway 课程也提醒：view entity 没有可供旧 SEGW “Import DDIC Structure from SQL view”流程导入的 SQL view。也就是说，旧 Gateway 技术里的导入路径不能原样套到现代 CDS view entity。

第四层机制是 ABAP SQL access。classic DDIC view 可以被 ABAP SQL 读取，旧系统里也可能被 AMDP、Native SQL 或外部数据库接口看到。但教学上要强调推荐边界：在 SAP 项目里不要绕过 AS ABAP 数据库接口直接访问 ABAP-managed view，尤其不要把 direct database access 当成稳定集成方式。直接访问可能绕过 client handling、CDS access control、buffering 或 ABAP Dictionary 语义，给权限、数据一致性和升级带来风险。

第五层机制是 ABAP Cloud / RAP 边界。ABAP Cloud 不支持 classic DDIC views 作为新开发对象；SEGW-based OData V2 生成也不是 ABAP Cloud 新开发路径。RAP/Fiori 中的 read model、value help、UI metadata、service exposure，应来自 CDS view entity、CDS projection view、metadata extension、service definition、service binding 和 behavior definition。DDIC view 在这里最多是迁移对象、遗留依赖或影响分析线索。

第六层机制是 view subtype。database view 用来读数据；maintenance view 偏表维护场景；help view 偏 search help / F4 语境；classic projection view 是旧式 DDIC view 子类型；external view 偏旧 HANA view proxy。它们都可能在旧系统里叫 view，但 access rule、用途和迁移策略不同。项目调查时不要只看名字，要看对象类型、where-used list、调用程序、transport 历史和是否被服务或报表依赖。

第七层机制是 Clean Core 判断。看到 DDIC view，不要立刻删除，也不要盲目继续扩展。正确动作是分三步：先确认是否仍被报表、接口、SEGW 服务、CDS V1、AMDP 或外部接口使用；再判断是否属于只读 legacy asset、需修复的 SQL view name access，还是应迁移到 CDS view entity；最后把新开发和旧资产维护分开管理。

37.5 日本项目现场说法

  • 常见日语：DDICビュー、ディクショナリビュー、従来型DDICビュー、旧来ビュー、データベースビュー、メンテナンスビュー、ヘルプビュー、CDS DDICベースビュー、CDS V1ビュー、CDSビューエンティティ、SQLビュー名、生成ビュー、廃止済み、非推奨、移行対象、移行計画、既存資産、保守対象、使用箇所、影響調査。
  • 说明句：この DDIC ビューは従来型のディクショナリビューであり、新規の RAP/Fiori データモデルとしては使用しません。
  • 影响分析句：この SQL ビュー名が ABAP SQL、SEGW、外部連携で直接参照されていないか、使用箇所を確認してください。
  • 迁移句：移行対象は従来型 DDIC ビューから CDS ビューエンティティへの置き換えです。移行計画と影響範囲を設計書に記録します。
  • Review 句：CDSビューという表現だけでは曖昧です。CDS DDICベースビューなのか、CDSビューエンティティなのかを明記してください。
  • Cloud 句：ABAP Cloud では従来型 DDIC ビューを新規開発の前提にせず、リリース済み API と CDS ビューエンティティを使用してください。

廃止予定 和 非推奨 在日语项目文档里语气不同。对 classic DDIC view 这种已经属于旧式/obsolete 边界的对象，写成 廃止予定 可能会让读者以为只是未来某天才会过时；更稳妥的表达是 廃止済み、非推奨、旧来技術，并说明“既存資産として保守するが、新規開発では使用しない”。

37.6 English Terms

  • Official / search terms: DDIC view, ABAP Dictionary view, classic DDIC view, DDIC database view
  • Modern contrast terms: CDS view entity, CDS DDIC-based view, CDS-managed DDIC view, ABAP CDS, service definition, service binding
  • Legacy / migration terms: generated SQL view, SQL view name, SEGW, Gateway Service Builder, where-used list, ABAP Cloud readiness
  • Recommended search patterns:
  • DDIC view ABAP Keyword Documentation
  • classic DDIC views ABAP Cloud
  • CDS DDIC-based views obsolete SAP
  • CDS view entity no SQL view SAP Gateway

读官方资料时，要特别留意 define view 和 define view entity 的区别，也要留意 @AbapCatalog.sqlViewName 是否出现在示例里。出现 @AbapCatalog.sqlViewName 时，通常说明你正在读旧 CDS DDIC-based view / CDS V1 语境，不是现代 view entity 语境。

37.7 Example / 学习例子

假设旧系统里有一个对象 ZVSO_OLD_ORDER，它把销售订单抬头表和项目表 join 到一起。你在旧报表中看到：

SELECT vbeln, posnr, matnr, netwr
  FROM zvso_old_order
  WHERE vbeln = @lv_vbeln
  INTO TABLE @DATA(lt_order_items).

第一步，不要马上把 ZVSO_OLD_ORDER 当成普通 table。先在 ADT/SE11/where-used list 里确认它是 database table、DDIC database view、CDS-managed DDIC view，还是 CDS view entity。对象类型不同，迁移和 review 结论完全不同。

第二步，确认它的用途。如果它只是 legacy report 的只读 join view，可以记录为 既存資産 / 保守対象，在修 bug 时谨慎维护。如果它被新 OData 服务继续引用，或者被 ABAP SQL 通过 generated SQL view name 访问，就要标出 Clean Core / migration risk。

第三步，给新开发方向。新的 RAP/Fiori 功能不应继续扩展 ZVSO_OLD_ORDER。更合理的方向是建立 CDS view entity，例如 ZI_SalesOrderItem，再基于 projection view、metadata extension、behavior definition、service definition 和 service binding 暴露服务。旧 view 保持可读、可定位、可替换，新功能走现代模型。

最小练习：找一个旧 view 名称，写出三行判断：

对象类型：classic DDIC database view / CDS DDIC-based view / CDS view entity / 其他
当前用途：legacy report / SEGW import / ABAP SQL source / external access / unknown
建议动作：保守维护 / where-used 追加调查 / 迁移到 CDS view entity / 禁止新开发引用

37.8 Counterexample / Failure mode

第一类错误，是把 DDIC view 当成现代 CDS。看到“view”就以为可以直接用于 RAP service，是典型误判。classic DDIC view 不提供 CDS annotation、association、access control、metadata extension、behavior definition 等现代 RAP 需要的语义层。

第二类错误，是把 CDS DDIC-based view 和 CDS view entity 混成一个词。旧 CDS V1 有 generated SQL view name；view entity 没有旧式 SQL view name。项目文档只写 CDSビュー，却不说明是哪一种，会让后续开发、测试和迁移判断全部变模糊。

第三类错误，是直接使用 generated SQL view name。旧系统里这种写法可能存在，但在迁移和严格检查语境下会成为 obsolete usage。初学者应养成习惯：ABAP SQL 访问 CDS 时优先使用 CDS entity / view entity 名称，不要依赖 @AbapCatalog.sqlViewName 生成的 SQL view name。

第四类错误，是绕过 AS ABAP 直接从数据库访问 DDIC view。普通数据库思维会觉得“既然有 SQL view，就直接查数据库”。SAP 项目里这会绕开 client handling、authorization、CDS access control、buffering、Dictionary 语义和升级兼容性，通常不是安全的集成方式。

第五类错误，是把 legacy 识别等同于 legacy 推广。本节讲 DDIC view，不是为了鼓励新项目继续新建传统视图，而是为了让学习者能读懂旧资产、能做影响分析、能知道现代替代路径。
