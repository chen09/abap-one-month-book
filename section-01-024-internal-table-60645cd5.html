24. Internal table 内部表基础

  • Chapter 01
  • Section 24
  • self-contained study note

本节结论

Internal table 内部表 是 ABAP 程序运行时内存里的 typed row collection：它用同一个 row type 保存 0 到多行数据，适合接住 SQL 查询结果、临时整理业务记录、循环处理和传递结果集。它不是数据库表，不会自己持久化数据，也不是无类型 list/dict；学会 internal table，才真正进入 ABAP 的日常数据处理世界。

正文

上一节的 Structure 结构类型 解释了“一条记录”是什么。Internal table 则解释“多条记录”如何存在 ABAP 程序内存里。一个 internal table 是一个 data object，它的类型叫 table type；table type 至少要说明 row type，通常还要说明 table category 和 key。最常见的业务内表是“行类型为结构”的内表，也就是每一行都是一个结构，多个结构排成一张程序内存中的表。

先把边界画清楚。Database table 是数据库里的持久化表，internal table 是 ABAP 程序运行时的内存对象。SELECT ... INTO TABLE @lt_items 会把查询结果放进 internal table，得到的是程序内存里的结果集副本；SELECT 结束后，lt_items 不会自动和数据库保持连接。现代代码里还常见 inline form：SELECT ... INTO TABLE @DATA(lt_items)，这会根据 SELECT 结果在编译期推断 internal table 的 row type。若这个内表要跨多个步骤复用，预先显式声明 lt_items 也完全合理。

Internal table 的 row type 可以是 elementary、structured、reference，甚至可以更复杂；但初学阶段先把“多数业务内表的 row type 是 structure”学稳。比如订单行、航班连接、消息明细、RAP reported/failed 记录，通常都是结构行。只有一个整数列表或字符串列表才接近 simple internal table。后面学 ABAP SQL 时还要注意：deep row type 在某些 SQL 数据源/结果场景有额外限制，本节先不展开。

Table type 可以直接写在 DATA 上，也可以先用 TYPES 命名，还可以来自 DDIC/global table type：

TYPES tt_items TYPE STANDARD TABLE OF ty_order_item WITH EMPTY KEY.

DATA lt_items TYPE tt_items.
DATA lt_items_local TYPE STANDARD TABLE OF ty_order_item WITH EMPTY KEY.

局部 helper 变量用 TYPE STANDARD TABLE OF ... WITH EMPTY KEY 很常见；方法参数、返回值、跨对象复用的类型，则更适合放在 DDIC table type 或 global class/interface public type 里，便于 review、复用和接口稳定。

本节只预览 table category。STANDARD TABLE 是 Day 03 的默认心智模型：它适合小到中等结果集、顺序处理、先 append 再 loop 的代码；按主键查找时主要是线性搜索。SORTED TABLE 按 key 保持排序，适合有序和 key-based 访问；HASHED TABLE 依赖 hash key，适合按唯一 key 快速读取。sorted/hashed、secondary key 和性能选择在后面专门展开；今天只要知道：内表不是单一的“数组”，它有 table category，访问模式和数据量会影响选择。

再讲 key。每个 internal table 都有 primary key，但“有 key”不等于“有业务主键”。如果 standard table 没有显式写 key，ABAP 会使用 default key。对 structured row type 来说，可以先粗略理解为：standard table 的 default key 来自行类型中所有非数值的 key-eligible components，而不是你心里想的订单号、物料号或业务主键。sorted/hashed table 的 key 规则又不同，后面再展开，不要把 standard table 的 default key 说明泛化到所有 table category。这个暗黙デフォルトキー 很容易让 READ TABLE、DELETE ADJACENT DUPLICATES、去重和比较逻辑变得意外。现代代码如果不打算用 key 查找，常显式写 WITH EMPTY KEY；如果有明确业务 key，就写 WITH NON-UNIQUE KEY field1 field2 或其他明确 key 形式。完整 key 规则留到 Week 2。

常见操作可以按四类记。第一，构造：VALUE #( ( ... ) ( ... ) ) 一次构造多行。第二，追加：对 standard table，APPEND row TO lt_items 把一行加到末尾。第三，循环：LOOP AT lt_items INTO DATA(ls_item). 把当前行复制到 work area；在 loop 里修改 ls_item 不会自动改回 internal table。若要原地修改，后面要学 ASSIGNING FIELD-SYMBOL(<ls_item>) 或 MODIFY。第四，读取：READ TABLE lt_items INTO DATA(ls_found) WITH KEY field = value 或 table expression 可以取单行；这里的 WITH KEY field = value 是按 component 条件查找的 free-key search，不等于在 WITH EMPTY KEY 表上使用 primary key。使用 INDEX 访问时，ABAP internal table 的索引从 1 开始，不是 Python list 的 0 开始。

还要认识 header line。旧 ABAP 代码里可能出现 DATA itab TYPE ... WITH HEADER LINE 或没有显式 work area 的短写法。它会让表体和一行隐式 work area 共用同一个名字，读起来非常容易混乱。SAP 文档已把 internal tables with header lines 标为 obsolete，并且在 classes 里不允许使用。新代码不要写 header line。维护旧代码时还要特别小心：对有 header line 的旧式内表，CLEAR itab 清的是 header line，不是表体；要清表体要写 CLEAR itab[]。普通无 header line 的现代内表，CLEAR lt_items 会把它清成 0 行的空表；FREE lt_items 也删除行，并且在内存释放行为上和 CLEAR 有区别，细节以后再讲。

为什么重要

  • ABAP 的业务处理大量围绕 internal table 展开：SQL 结果、接口 payload、校验消息、ALV/Fiori 数据、RAP failed/reported、批量更新前的中间结果，都经常是 internal table。
  • Internal table 把 Structure 结构类型、循环、条件分支、SQL、性能和 key 连接起来；它不是单纯的“数组”。
  • LOOP ... INTO 是 copy 语义，ASSIGNING 是更接近原地访问的语义；如果混淆，修改代码会静默无效。
  • Default key 和 WITH EMPTY KEY 是现代 ABAP review 的高频点；不懂 key，后面 standard/sorted/hashed、READ、table expression 都会学歪。
  • 对 Java/Python 背景学习者来说，internal table 可以类比 typed list of rows，但不能照搬 0-based index、动态 key、无类型 dict 的习惯。

核心机制

  • Internal table 是 variable-size data object，初始值是 0 行的 empty table。
  • Row type 决定每一行长什么样；多数业务内表的 row type 是 structured type。
  • Table type 描述 row type、table category、primary key，并可通过 local TYPES、inline TYPE TABLE OF、DDIC/global table type 表达。
  • STANDARD TABLE 是基础入口；SORTED TABLE、HASHED TABLE 与 secondary key 先作为性能/访问模式 preview。
  • WITH EMPTY KEY 显式说明“不打算使用默认 key”；不写 key 可能触发 implicit default key，常被 ATC/code review 标出。
  • VALUE #( ) 构造多行，APPEND 追加行，LOOP AT ... INTO 复制行到 work area，READ TABLE / table expression 读取单行。
  • CLEAR lt_itab 对现代无 header line 内表会清空行；legacy header-line table 的 CLEAR itab 只清 header line，需识别但不要新写。

日本项目现场表达

  • 常见日语：内部テーブル、行型、テーブル型、テーブルカテゴリ、標準テーブル、ソートテーブル、ハッシュテーブル。
  • Key 相关：主キー、二次キー、デフォルトキー、暗黙デフォルトキー、明示的なキー、WITH EMPTY KEY、空テーブル（行数0）。
  • 操作相关：ワークエリア、APPEND、LOOP AT、READ TABLE、テーブル式、フィールドシンボル、ヘッダ行、ヘッダ行あり内部テーブル。
  • 说明句：内部テーブルはプログラム実行時のメモリ上で複数行を保持するデータオブジェクトであり、DBテーブルそのものではありません。
  • Review 句：この宣言は暗黙デフォルトキーになるため、キーを使わない前提であれば WITH EMPTY KEY を明示してください。
  • Legacy 句：ヘッダ行あり内部テーブルは廃止扱いのため、新規開発では使用しません。保守時は CLEAR itab がヘッダ行だけを初期化する点に注意してください。
  • 语义边界：ワークエリア は LOOP AT ... INTO wa のようなコピー先の1行バッファを指すことが多く、ASSIGNING のフィールドシンボルとは区別します。

例子

用上一节的结构思想，构造一个订单行内表：

TYPES: BEGIN OF ty_item,
         travel_id TYPE /dmo/travel_id,
         item_no   TYPE n LENGTH 6,
         amount    TYPE p LENGTH 11 DECIMALS 2,
         currency  TYPE c LENGTH 3,
       END OF ty_item.

TYPES tt_items TYPE STANDARD TABLE OF ty_item WITH EMPTY KEY.

DATA lt_items TYPE tt_items.

lt_items = VALUE #(
  ( travel_id = '0000004711' item_no = '000010' amount = '1980.00' currency = 'JPY' )
  ( travel_id = '0000004711' item_no = '000020' amount = '2500.00' currency = 'JPY' ) ).

APPEND VALUE #( travel_id = '0000004711'
                item_no   = '000030'
                amount    = '900.00'
                currency  = 'JPY' ) TO lt_items.

LOOP AT lt_items INTO DATA(ls_item).
  out->write( |{ ls_item-travel_id } { ls_item-item_no } { ls_item-amount } { ls_item-currency }| ).
ENDLOOP.

这里 ty_item 是 row type，tt_items 是 table type，lt_items 是 internal table data object，ls_item 是 work area copy。WITH EMPTY KEY 明确告诉读者：这张 standard table 当前只按顺序处理，不依赖默认 key。后面如果要按 travel_id、item_no 查找或去重，就应该重新设计显式 key，而不是指望 default key 刚好符合业务含义。

再看一个容易误解的 loop：

LOOP AT lt_items INTO DATA(ls_copy).
  ls_copy-amount = 0.
ENDLOOP.

这段代码只改了 work area copy，不会把 lt_items 里的金额自动改成 0。要原地修改内表行，后面会学 ASSIGNING、MODIFY 和 table expressions。本节先记住：INTO 是复制到 work area，不是引用到原行。

常见误区

  • 把 internal table 当数据库表：以为改了 lt_items 就会自动更新 DB，忘了它只是程序内存里的数据对象。
  • 把 internal table 当 Python list：写 INDEX 0 或以为 loop 变量修改会自动改回集合。
  • 不写 key：standard table 触发 default key，ATC 或 reviewer 要求改成 WITH EMPTY KEY 或显式 key。
  • 用 READ TABLE 做大量线性查找：小数据能跑，大数据性能突然变差；后面要学 sorted/hashed/key。
  • 看到 header line legacy 写法就模仿：新代码不应使用 WITH HEADER LINE，classes 里也不允许。
  • 把 空テーブル 和 初期行 混淆：空表是 0 行，不是一行所有字段都是 initial value。

思考题

  • 为什么说 internal table 不是 database table？SELECT ... INTO TABLE 后两者是什么关系？
  • ty_item、tt_items、lt_items、ls_item 分别是 row type、table type、internal table object、work area 中的哪一个？
  • LOOP AT lt_items INTO DATA(ls_item) 为什么不会让你直接修改原表行？
  • 如果 standard table 没写 key，default key 为什么可能不是业务主键？
  • 什么时候应该先写 WITH EMPTY KEY，什么时候要设计明确 key？
  • 在 legacy code 里看到 header line 时，CLEAR itab 和 CLEAR itab[] 的差异是什么？
