9. UPDATE statement 更新数据

  • Chapter 02
  • Section 09
  • self-contained study note

9.1 一句话结论

ABAP SQL 的 UPDATE 用来修改数据库中已经存在的行。学习重点不是“把字段改成新值”，而是确认要改哪些行、只改哪些字段、实际改了几行、并发下是否会覆盖别人更新、谁来提交或回滚，以及这次修改是否经过授权、锁、业务校验和 Clean Core 允许的写入路径。

9.2 到底学什么

UPDATE 和 INSERT 的语义不同：INSERT 是新增不存在的行，UPDATE 是修改已经存在的行。如果目标行不存在，UPDATE 不会帮你创建新行。这个差异在接口重试、批处理补偿、主数据维护、状态流转里非常重要。

最常见的安全写法，是用 SET 指定要改的字段，用 WHERE 精确限定要改的行：

UPDATE zorder_note
   SET note_text  = @lv_note_text,
       changed_by = @sy-uname,
       changed_at = @lv_changed_at
 WHERE note_id = @lv_note_id.

DATA(lv_update_subrc) = sy-subrc.
DATA(lv_updated)      = sy-dbcnt.

读这段代码时，先看三件事：WHERE 是否足够精确，SET 是否只改了应该改的字段，sy-dbcnt 是否等于业务预期。比如按主键更新一条备注，预期 sy-dbcnt 应该是 1；如果是 0，说明目标不存在或并发条件不满足；如果大于 1，说明 WHERE 过宽。

UPDATE 也可以从结构或内表更新：

UPDATE zorder_note FROM @ls_note.
UPDATE zorder_note FROM TABLE @lt_notes.

这类写法会按主键找到目标行，并用来源结构/内表里的行进行更新。它适合“我手里有完整且可信的数据库行内容”的场景；如果来源结构字段不完整、带旧值或初始值，就可能把不该改的字段覆盖掉。

9.3 为什么重要

UPDATE 是最容易造成生产事故的 ABAP SQL 语句之一。原因很简单：它修改的是既有业务事实。一个漏掉 WHERE 的更新，可以把整张表都改掉；一个过宽的 WHERE，可以把别的组织、别的客户、别的期间的数据一起改掉；一个没有并发保护的更新，可以把别人刚保存的内容覆盖回旧值。

它的第一层价值是表达变更意图。UPDATE ... SET status = ... WHERE order_id = ... 比“先读一整行、改一个字段、再整行 UPDATE 回去”更清楚，因为它明确说明只改 status。

第二层价值是验证影响范围。写入后马上看 sy-dbcnt，可以确认实际更新件数是否符合预期。项目 review 里常问：更新件数は想定通りですか。这不是形式问题，而是防止误更新的最后一道简单检查。

第三层价值是并发安全。用户 A 读出数据后，用户 B 可能已经更新了同一行。如果用户 A 直接整行 UPDATE FROM @ls_old_row，可能把 B 的更新覆盖掉。需要 lock object、RAP lock、ETag/last changed timestamp、版本号或其他并发控制策略。

第四层价值是事务边界。UPDATE 成功不等于已经提交。它会进入当前 database LUW / SAP LUW，最终是否永久生效取决于 commit/rollback。可复用方法里随手 COMMIT WORK，和 INSERT 一样危险。

9.4 Key mechanism / 核心机制

9.4.1 UPDATE ... SET ... WHERE：局部字段更新

局部更新最适合“只改少数字段”的场景：

UPDATE zsales_order
   SET status     = @lc_status_released,
       changed_by = @sy-uname,
       changed_at = @lv_changed_at
 WHERE sales_order_id = @lv_sales_order_id.

这里的核心是 WHERE。如果 sales_order_id 是唯一主键，预期更新一行。写完后应立即检查：

DATA(lv_updated) = sy-dbcnt.

IF lv_updated <> 1.
  " No row or too many rows were updated. Handle as a business error.
ENDIF.

不要只看 sy-subrc = 0 就放心。sy-subrc = 0 说明有行被更新；但项目里更关心“是不是恰好更新了预期数量的行”。

9.4.2 没有 WHERE 可能更新全表

下面这种写法非常危险：

UPDATE zsales_order
   SET changed_by = @sy-uname.

没有 WHERE，语义就是对目标对象的所有适用行执行更新。某些批处理确实会做全量更新，但必须有明确设计、日志、件数检查、运行窗口和回滚方案。业务代码里如果不是有意全量更新，缺失 WHERE 就是高危缺陷。

9.4.3 UPDATE ... FROM @wa：整行更新要小心

从结构更新通常按主键定位目标行：

SELECT SINGLE FROM zorder_note
       FIELDS *
       WHERE note_id = @lv_note_id
       INTO @DATA(ls_note).

ls_note-note_text  = lv_new_text.
ls_note-changed_by = sy-uname.
ls_note-changed_at = lv_changed_at.

UPDATE zorder_note FROM @ls_note.

这段代码看起来方便，但它更新的是来源结构承载的一整行语义。风险在于：SELECT 和 UPDATE 之间，别人可能已经改了同一行；或者 ls_note 某些字段不是最新值。局部字段更新时，优先考虑 UPDATE ... SET ... WHERE ...，不要为了省事整行覆盖。

9.4.4 UPDATE ... FROM TABLE @itab：批量更新可能部分成功

批量更新可以写：

UPDATE zorder_note FROM TABLE @lt_notes.

DATA(lv_update_subrc) = sy-subrc.
DATA(lv_updated)      = sy-dbcnt.

官方文档说明，UPDATE FROM TABLE 会尽可能更新能匹配的行；如果 sy-subrc = 4，并不代表一行都没改，而是至少有一行没有被处理到。此时必须看 sy-dbcnt，并把预期件数和实际更新件数对比。

IF lv_updated <> lines( lt_notes ).
  " Some source rows did not find matching database rows.
  " Decide whether this is acceptable or an error.
ENDIF.

这和 INSERT ... ACCEPTING DUPLICATE KEYS 的教训类似：部分成功不能静默当作全成功。

9.4.5 sy-subrc 与 sy-dbcnt

对 UPDATE，常见判断是：

  • sy-subrc = 0：至少一行被更新，或内表中的行都能用于更新。
  • sy-subrc = 4：没有符合条件的行，或内表批量更新时至少有一行找不到对应数据库行。
  • sy-dbcnt：实际更新的行数。

最安全的习惯，是 UPDATE 后马上复制系统字段：

UPDATE zsales_order
   SET status = @lc_status_closed
 WHERE sales_order_id = @lv_sales_order_id.

DATA(lv_update_subrc) = sy-subrc.
DATA(lv_updated)      = sy-dbcnt.

然后用业务预期判断，而不是只写：

IF sy-subrc = 0.
  " Success?
ENDIF.
9.4.6 并发控制：避免 lost update

典型 lost update 是这样的：

  1. 用户 A 读出订单备注。
  2. 用户 B 修改并保存同一备注。
  3. 用户 A 用旧结构整行 UPDATE，把用户 B 的修改覆盖掉。

一种常见防御方式，是在 WHERE 里加入版本字段、last changed timestamp 或 ETag 语义：

UPDATE zorder_note
   SET note_text  = @lv_new_text,
       changed_by = @sy-uname,
       changed_at = @lv_now
 WHERE note_id    = @lv_note_id
   AND changed_at = @lv_old_changed_at.

IF sy-dbcnt <> 1.
  " The row was changed or deleted by someone else.
  " Ask the caller to reload or handle a conflict.
ENDIF.

如果数据库中的 changed_at 已经不同，说明这行在用户 A 读取后被别人改过，本次更新不应静默覆盖。经典 SAP 还会使用 lock object 做悲观锁；RAP 中则要理解 lock、ETag、authorization、validation 和 save sequence。

9.4.7 UPDATE 成功不等于已经提交

官方文档说明，修改会在下一个 database commit 后永久生效，在 commit 前仍可能被 rollback 撤销。UPDATE 还会带来数据库锁，直到 commit 或 rollback 释放。写太多行、事务太长、锁范围太大，都可能造成等待、死锁或回滚区压力。

所以，和 INSERT 一样，不要在可复用方法里随手提交：

METHOD change_status.
  UPDATE zsales_order
     SET status = @iv_status
   WHERE sales_order_id = @iv_sales_order_id.

  " Do not COMMIT WORK here unless this method owns the full transaction boundary.
ENDMETHOD.

谁拥有事务边界，谁决定提交。批处理、dialog transaction、BAPI、RAP、OData 服务和单元测试的规则不同。

9.4.8 UPDATE 不是 SAP update task

ABAP 里还有一个容易混淆的词：update task。UPDATE ztable ... 是 ABAP SQL 的数据库修改语句；CALL FUNCTION ... IN UPDATE TASK 是 SAP 的异步/延迟更新机制。日语里都可能说 更新処理，但技术含义不同。本节讲的是 ABAP SQL UPDATE，不是 update task 框架。

9.4.9 权限、锁和 Clean Core

UPDATE 不会自动执行业务授权检查，不会自动调用变更文档，不会自动触发标准业务校验，也不会自动符合 Clean Core。尤其不要直接改 SAP 标准表来“修数据”。真实项目里应优先使用 released API、BAPI、RAP BO、业务事务、修复程序或正式扩展点。

对自定义表，也要确认当前用户是否有业务权限、组织范围是否正确、是否需要 lock object、是否要记录 application log 或 change document，以及测试是否会污染共享数据。

9.7 Example / 学习例子

9.7.1 例 1：按主键更新一个状态
UPDATE zsales_order
   SET status     = @lc_status_released,
       changed_by = @sy-uname,
       changed_at = @lv_changed_at
 WHERE sales_order_id = @lv_sales_order_id.

DATA(lv_updated) = sy-dbcnt.

IF lv_updated <> 1.
  " Expected exactly one row to be updated.
ENDIF.

这个例子最重要的是 WHERE sales_order_id = ... 和 lv_updated <> 1。如果业务预期是一张订单，更新件数就应该是一件。

9.7.2 例 2：防止 lost update
UPDATE zorder_note
   SET note_text  = @lv_new_text,
       changed_by = @sy-uname,
       changed_at = @lv_now
 WHERE note_id    = @lv_note_id
   AND changed_at = @lv_old_changed_at.

IF sy-dbcnt <> 1.
  " The record was changed by someone else or no longer exists.
ENDIF.

这个例子用 changed_at 做乐观锁条件。它不是唯一方案，但能让初学者理解：WHERE 不只是定位主键，也可以表达“我只接受基于旧版本的更新”。

9.7.3 例 3：整行更新的风险
SELECT SINGLE FROM zorder_note
       FIELDS *
       WHERE note_id = @lv_note_id
       INTO @DATA(ls_note).

ls_note-note_text = lv_new_text.

UPDATE zorder_note FROM @ls_note.

这段代码可能没错，但要问：ls_note 是否包含完整且最新的一行？有没有别人同时改了其他字段？如果只是改 note_text，UPDATE ... SET note_text = ... WHERE ... 往往更清楚。

9.7.4 例 4：批量更新后检查件数
UPDATE zorder_note FROM TABLE @lt_notes.

DATA(lv_updated) = sy-dbcnt.

IF lv_updated <> lines( lt_notes ).
  " Not all source rows found matching database rows.
ENDIF.

批量更新的 sy-subrc = 4 可能意味着部分成功。不要把它当作“全部失败”，也不要当作“全部成功”。看 sy-dbcnt，再按业务决定报错、补偿、日志或重试。

9.8 Counterexample / Failure mode

失败模式一：漏掉 WHERE，导致全表更新。除非这是明确设计的批处理，否则应视为高危缺陷。

失败模式二：WHERE 太宽。比如只按状态更新，而没有公司代码、销售组织、客户、期间或主键条件，会把别的业务范围一起更新。

失败模式三：只看 sy-subrc，不看 sy-dbcnt。项目里经常需要“恰好更新 1 行”或“恰好更新 N 行”，这必须用件数判断。

失败模式四：整行 UPDATE FROM @wa 覆盖别人刚修改的字段。局部字段更新、锁、版本字段或 ETag 可以降低 lost update 风险。

失败模式五：在可复用方法里随手 COMMIT WORK。这可能提前提交调用方尚未完成的业务事务。

失败模式六：直接更新 SAP 标准表。标准表背后可能有业务校验、派生字段、缓存、变更文档、状态机和权限逻辑，直接 SQL 更新会绕过它们。

失败模式七：把 ABAP SQL UPDATE 和 update task 混为一谈。一个是 SQL 修改语句，一个是 SAP 更新任务机制，问题定位时要分清。

失败模式八：测试后不回滚或清理。更新类测试很容易污染共享测试数据，尤其是状态字段和主数据字段。

9.9 思考问题

  • 这次更新的业务语义是什么？改状态、改文本、改金额，还是补数据？
  • WHERE 是否能保证只更新预期业务范围？
  • 预期更新件数是多少？代码是否检查 sy-dbcnt？
  • 如果目标行不存在，是错误、忽略，还是应该新增？
  • 是否可能发生 lost update？有没有 lock、timestamp、version 或 ETag？
  • 是否只需要 SET 局部字段，还是必须 UPDATE FROM @wa 整行更新？
  • 批量更新时，部分成功如何处理？
  • 谁负责 COMMIT WORK 或 rollback？
  • 权限、业务校验、日志和变更文档在哪里完成？
  • ABAP Cloud / RAP / Clean Core 项目中，是否应该改用 released API、BAPI、RAP BO 或 EML？
