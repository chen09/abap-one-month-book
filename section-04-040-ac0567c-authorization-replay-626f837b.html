40. AC0567C authorization replay

  • Chapter 04
  • Section 40
  • self-contained study note

40.1 一句话结论

AC0567C authorization replay 是把 AC0567C batch upload API POC 放回 SAP 授权体系中复盘：一个 API 不是只要能被调用就算完成，还要能说明谁能上传、谁能查询、无权限如何被拒绝、错误如何返回、证据如何保存，以及哪些授权缺口必须在生产化前补齐。

本节最重要的结论是：AC0567C POC 已经有一个真实的 ABAP 授权检查点，ExecuteImport 会在写入前执行 AUTHORITY-CHECK OBJECT 'F_BKPF_BUK'，检查 BUKRS 和 ACTVT = '01'；但它还不是完整的生产级授权设计。当前 root view 使用 @AccessControl.authorizationCheck: #NOT_REQUIRED，BDEF 虽声明 authorization master ( instance ) 和 lock master，但 behavior pool 中 get_instance_authorizations 与 lock 为空；三个 action 又都是 static action，没有 get_global_authorizations 时不会被 instance authorization 实际挡住。也就是说，当前真正生效的业务授权主要在 ExecuteImport 的 service class 内部，ValidateBatch 和 GetStatus 仍需要明确是否也要授权。

这不是要否定 POC。恰恰相反，这个 POC 很适合教学：它清楚展示了“有一个检查点”与“有一套可交付授权设计”之间的距离。

40.2 到底学什么

第一，要学会从调用入口看授权。AC0567C 不是传统 SAP GUI 事务，而是 RAP/OData API POC。调用者可能是 Swagger、PowerShell、浏览器模拟器、本地 proxy、外部 middleware、communication user 或后台自动化流程。API 场景不能只问“哪个按钮可见”，要问“谁以什么身份调用服务”。

第二，要学会分辨授权层次。传统 ABAP 可以有 AUTHORITY-CHECK；CDS 可以有 DCL/access control；RAP 可以有 global authorization、instance authorization、feature control、precheck 或 validation；服务层还可能有 authentication、communication arrangement、service binding activation。复盘时要说清每层负责什么，不要把它们混成一句“权限已做”。

第三，要学会读 AC0567C 的实际实现。当前 POC 的事实是：

层次当前证据交付判断
CDS/DCLZRPA01_I_AC0567C_IMPORT.ddls 标注 @AccessControl.authorizationCheck: #NOT_REQUIRED当前 root view 不依赖 DCL，生产化要重新判断是否需要 DCL 或其他数据范围控制

第四，要学会设计授权测试证据。授权测试至少要有：有权限用户成功、无权限用户失败、只读用户不能写、组织范围外数据被拒绝、失败时没有写 DB、错误响应合理、日志或 trace 可查。只测 happy path 不算 authorization replay。

第五，要学会把授权复盘映射到项目文档。授权方针应进入基本设计，具体检查点进入详细设计，测试用户和结果进入单体/结合测试，角色/权限对象和运维排查进入上线或運用手順。日本项目里还要能用日语说明 権限不足時の挙動確認 和 証跡。

40.3 为什么重要

授权问题在 SAP 项目里通常很敏感，因为它直接关系到业务职责、内部控制、审计和生产事故。一个 batch upload API 如果授权不足，可能允许错误用户上传、修改或查看不该看的业务数据；如果授权设计过粗，又会造成合法用户无法工作。

对交付来说，authorization replay 能把“接口能调通”升级为“接口能被正确的人、在正确范围内、以正确动作调用”。这比代码语法更接近真实项目验收。

对 Java/Python 背景学习者来说，可以类比 authentication/authorization middleware、RBAC、row-level security 和 permission decorator。但 SAP 里还要考虑 authorization object、role/profile、client、org-level field、CDS DCL、RAP authorization、SU53/trace、业务事务和 transport。

更重要的是，SAP 的授权常常和组织结构绑定。Java/Python 项目里常见的 RBAC 可能只回答“用户是不是 admin / operator / viewer”；SAP 项目还会继续问“这个用户对哪个 company code、sales organization、plant、purchasing organization 有权限”。因此 AC0567C 这种 batch upload API，不能只写“上传角色可以上传”，还要考虑“上传哪些业务范围的数据”。

授权复盘也能帮助团队避免两个极端。一个极端是权限过松：为了 POC 快速通过，所有接口都给同一个技术用户高权限。另一个极端是权限过细但没有证据：开发、Basis、权限团队、业务顾问各自理解不同，测试时才发现合法用户调不通。好的 authorization replay 会把角色、动作、数据范围、错误响应和测试证据提前对齐。

40.4 Key mechanism / 核心机制

AC0567C 授权复盘可以按七步做。

第一，列入口。确认有哪些入口需要保护：JSON validate API、JSON execute API、Base64 TSV upload validate/execute、status query、本地 proxy、Swagger、PowerShell test、模拟器、未来外部系统用户。不同入口可能对应不同角色。

第二，列动作。AC0567C 当前有三类 RAP static action：ValidateBatch、ExecuteImport、GetStatus。三者不应默认同权。ValidateBatch 可能只做格式和业务校验，ExecuteImport 会进入真实写入路径，GetStatus 会读取导入状态。是否允许同一个 technical user 调用三者，是否允许业务只读用户查询状态，必须写清楚。

第三，列数据范围。当前 ExecuteImport 用 F_BKPF_BUK 检查 company code 的创建权限：

AUTHORITY-CHECK OBJECT 'F_BKPF_BUK'
  ID 'BUKRS' FIELD rs_result-company_code
  ID 'ACTVT' FIELD '01'.

F_BKPF_BUK 是按 company code 控制会计凭证过账相关权限的标准授权对象；ACTVT = '01' 通常表示 Create。这里把它作为 company-code-level create 权限的 POC 代理检查，是可以解释的，但生产设计不能自动认为它就是最合适的对象。因为目标表是自定义表 ZCOM_B_ACTT397，生产化时要决定：继续复用标准 FI 授权对象、组合更多标准对象，还是设计专用 Z 授权对象。这是一条很适合教学的证据：它说明业务范围不是抽象的“有权限/无权限”，而是落在 BUKRS、ACTVT 这样的 object-field-value 上。生产化时还要确认 customer、upload owner、batch visibility、日志可见范围等是否也需要控制。

还要注意实际执行顺序：execute_import 先调用 validate_batch，如果结果不是 VALID 就返回；然后检查 AllowRealUpdate <> 'X' 时返回 REJECTED；只有 AllowRealUpdate = X 且校验通过时，才会走到 AUTHORITY-CHECK F_BKPF_BUK。所以授权负例测试必须设置 AllowRealUpdate = X，否则被测到的是 POC 安全开关失败，不是授权失败。

第四，确认 RAP 层的真实生效状态。ZRPA01_I_AC0567C_IMPORT.bdef 写了 authorization master ( instance )，但当前 action 都是 static action。instance authorization 通常用于具体实例，而 static action 如果需要授权，应考虑 RAP global authorization 或 action 内显式检查。当前没有 get_global_authorizations，get_instance_authorizations 也是空方法，所以不要把 BDEF 声明误读成“RAP 授权已完成”。

第五，设计错误与日志。无权限不应返回模糊技术错误，也不应泄漏内部数据。错误 payload、message、application log 和排查路径要能帮助运维定位。AC0567C 当前 service class 会把授权失败转为 Status = REJECTED、MessageField = Authorization、MessageText = Authorization check F_BKPF_BUK/01 failed for CompanyCode。这比 dump 好，但生产级 API 还要确认 error code、log id、message language、consumer contract 和敏感信息边界。

第六，固定测试证据。至少准备三类用户：有完整权限、只读权限、无权限。再准备边界数据：允许 company code、禁止 company code、不存在的 batch、重复请求。记录 request、response、当前 user、时间戳、日志、trace 和 DB 未写入证据。

第七，把授权和副作用绑定验证。无权限请求不应创建 batch header、不应写入 item、不应触发后续 job、不应留下可被后续处理的半成品。AC0567C 的 AUTHORITY-CHECK 发生在 action/interaction phase 的 service class 逻辑中、staging 之前；behavior saver 的 save 方法只消费 staged rows 并执行插入，本身没有再做授权检查。因此，授权必须在进入 save buffer 之前挡住不合法请求。AC0567C 的测试证据目前证明了 no-write 测试 8/8、真实写入 2 行、重复 ImportId 返回 CONFLICT，但还没有证明“无 F_BKPF_BUK 权限用户执行 ExecuteImport 时完全无副作用”。这正是 authorization replay 的生产化缺口。

可以把 AC0567C 的授权复盘拆成五层：

Caller layer
  -> communication user / dialog user / background user
Service layer
  -> OData/RAP service activation, authentication, service exposure
Business object layer
  -> RAP global authorization, instance authorization, feature control, precheck
ABAP check layer
  -> AUTHORITY-CHECK F_BKPF_BUK in ExecuteImport
Evidence layer
  -> response payload, DB row count, application log, SU53 / trace, test case result

设计时可以先画一张动作矩阵：

动作当前 AC0567C 入口授权重点无权限时应避免
Validate batchValidateBatch / /validate是否允许外部用户提交校验请求，是否泄漏业务规则写 DB、暴露过多校验细节
Execute importExecuteImport / /executeF_BKPF_BUK、组织范围、是否需要 RAP global authorization插入 ZCOM_B_ACTT397、留下半成品
Upload validate/upload/validate文件来源、格式、大小、业务范围解析危险文件、泄漏校验细节
Upload execute/upload/execute文件执行权限、AllowRealUpdate、组织范围写入错误数据、重复导入
Query statusGetStatus / /status上传者/运维/业务负责人能看哪些 batch泄漏他人 batch 或组织数据
View logs/errors后续 application log / report日志可见范围、敏感字段遮蔽暴露 payload、个人信息或技术细节

RAP 中的 global authorization 和 instance authorization 可以帮助表达“动作层”和“对象实例层”的差异。global authorization 更适合判断用户是否原则上可以执行某类操作；instance authorization 更适合判断用户是否可以对某个具体 batch 或业务对象执行操作。CDS DCL 更偏读取时的行级访问控制，但它不能替代写操作的动作授权，也不能替代 service/API 层的 authentication。

还要注意 feature control 和 authorization 的边界。feature control 可以让某些 action 或 field 在特定状态下不可用，但“按钮不可用”不等于“用户无权”。如果 API 被直接调用，仍要在 behavior implementation 或授权机制中保证无权限/非法状态不会产生副作用。

40.5 日本项目现场说法

常见日语包括：

  • 権限チェック
  • 認可
  • 権限オブジェクト
  • ロール
  • 行レベル制御
  • RAP認可
  • グローバル認可
  • インスタンス認可
  • 事前チェック
  • SU53確認
  • 権限トレース
  • 権限エラー時の挙動確認

设计说明中可以这样说：

アップロード権限と照会権限を分離し、利用者ロールごとに許可される操作を整理します。
データ参照については、必要に応じてCDS DCLまたはRAP認可で行レベル制御を検討します。
権限不足時は業務的に理解可能なエラーを返し、調査用のログ証跡を残します。

测试说明中可以这样说：

権限ありユーザ、照会のみユーザ、権限なしユーザでAPI応答差異を確認します。
権限エラー時のレスポンス、アプリケーションログ、SU53/トレース結果を証跡として保存します。

设计 review 里还可以这样问：

  • API呼出ユーザは業務ユーザに紐づきますか、それとも技術ユーザとして全体権限を持ちますか。
  • アップロード、照会、再処理、取消の権限は同一ロールでよいですか。
  • 組織単位によるデータ制御は、DCL、RAP認可、または明示的な権限チェックのどこで担保しますか。
  • 権限不足時に業務データや後続処理が作成されないことを、どのテスト証跡で確認しますか。

缺陷分析时可以这样说：

  • 本件は認証エラーではなく、認可条件または組織レベル権限の不足として切り分けます。
  • SU53 だけで判断せず、必要に応じて権限トレース、アプリケーションログ、APIレスポンスを合わせて確認します。
  • 技術ユーザに過大権限を付与する対応は暫定回避策であり、本番設計としてはロール分離を再確認します。

  • POC object names: F_BKPF_BUK, BUKRS, ACTVT, ExecuteImport, ValidateBatch, GetStatus, ZRPA01_I_AC0567C_IMPORT, ZCL_RPA01_AC0567C_IMP_SVC, ZCOM_B_ACTT397.

40.7 Example / 学习例子

可以为 AC0567C 准备这样一张授权复盘表：

API:
  Execute batch import

Action:
  ExecuteImport

Current implemented check:
  AUTHORITY-CHECK F_BKPF_BUK with BUKRS = CompanyCode and ACTVT = 01

Allowed user:
  User with create authorization for target company code

Denied user:
  User without F_BKPF_BUK/01 for target company code

Expected allowed result:
  IMPORTED or VALID business response depending on AllowRealUpdate

Expected denied result:
  REJECTED authorization message, no insert into ZCOM_B_ACTT397

Evidence:
  request, response, sy-uname/technical user, timestamp, DB row count,
  application log or trace, test case ID

再看一个更接近生产化的测试矩阵：

Case用户动作数据期待结果必要证据
A1upload roleExecuteImport允许 company code成功或 staged/importedrequest/response、DB row count、log
A2read-only roleExecuteImport with AllowRealUpdate = X允许 company code授权错误，无业务数据保存response、DB 件数、trace
A3no-role userGetStatus已存在 batch授权错误或受控 not foundresponse、SU53/trace
A4upload roleExecuteImport with AllowRealUpdate = X禁止 company codeF_BKPF_BUK 错误或组织范围错误response、无明细保存
A5technical user/upload/executeBase64 TSV只在允许范围内执行proxy log、SAP response
A6normal userview log他人 batch不泄漏敏感日志response、log visibility

其中 A2 和 A4 特别关键。很多项目只证明“有权限时能成功”，却没有证明“无权限时没有副作用”。对上传 API 来说，失败后仍然留下 batch header、partial item、后台 job 或 external call，都会让授权问题变成事务一致性问题。测试时应明确检查数据库记录数、状态变化、application log 类型和后续处理是否被触发。

还可以把错误响应设计成固定格式：

{
  "code": "AC0567C_AUTH_001",
  "message": "You are not authorized to upload data for this organization.",
  "target": "companyCode",
  "logId": "BAL-20260616-0001"
}

这只是教学示意，不是要求所有 SAP 服务都用同一 JSON 结构。它想表达的是：权限错误要能被调用方识别、被测试脚本断言、被运维用 log id 追踪，并且不能暴露内部授权对象名、角色名或用户不该知道的数据。

40.8 Counterexample / Failure mode

常见失败模式如下。

第一，只在 UI 层隐藏按钮。API 仍可能被直接调用，必须在服务、业务对象、数据访问层考虑授权。

第二，只测有权限用户。无权限、只读、组织范围外、边界数据才是真正暴露风险的场景。

第三，把 authentication 当 authorization。能登录或有技术用户，不代表可以执行所有业务动作。

第四，把 AllowRealUpdate = X 当成授权。它只是 POC 的安全开关，不是权限机制；授权负例测试若不设置它，测试到的是安全开关，而不是 F_BKPF_BUK 授权检查。

第五，把 BDEF 的 authorization master ( instance ) 声明当成授权已完成。当前 AC0567C 的 instance authorization 方法为空，static action 也没有 global authorization gate。

第六，把 DCL 当成写操作授权。DCL 主要影响 CDS 读取结果，不能自动证明 upload / execute / cancel 这类动作安全。

第七，授权失败后仍产生副作用。无权限请求不应保存半成品、写入错误业务数据或触发后续处理。

第八，错误消息过于技术化。用户和运维不知道是权限问题、数据问题还是程序错误。

第九，不保存证据。上线后权限争议无法说明当时测过什么。

第十，用一个高权限 technical user 解决所有问题。这样测试最容易通过，但生产最难审计。技术用户如果代表外部系统，应限制服务、动作和数据范围，并保留调用来源、业务责任人或 request trace。

第十一，只看 SU53。SU53 对很多经典授权排查很有用，但 RAP/CDS/API 场景还可能需要 authorization trace、application log、response payload、service binding/runtime 错误和业务对象状态一起判断。

第十二，错误信息泄漏过多。返回“你无权访问 batch B001，因为它属于 company code 1000”可能在某些场景泄漏对象存在性或组织信息。是否返回详细原因，要按安全方针和业务可用性平衡。

第十三，运输遗漏。授权相关对象可能分散在 DCL source、behavior definition、class、role/profile 配置、service binding、communication arrangement 和测试数据中。只移送代码、不移送访问控制对象或配置，会造成 QAS/PRD 行为不一致。

40.9 思考问题

  • AC0567C 的 ValidateBatch、ExecuteImport、GetStatus 是否应该使用同一权限？
  • 哪些检查适合放在 RAP global authorization，哪些适合 instance authorization 或 DCL？
  • 当前 AUTHORITY-CHECK F_BKPF_BUK 证明了什么？还没有证明什么？
  • 对自定义表 ZCOM_B_ACTT397 来说，生产设计应继续使用 F_BKPF_BUK，还是需要自定义授权对象？
  • 无权限请求应该返回什么错误，并且不应该产生什么副作用？
  • 生产化前需要准备哪些授权测试用户和证据？
  • 如果客户说“某用户不能调用 API”，应如何用 SU53/trace/log 做一次切り分け？
