Chapter 02 - Core ABAP Programming

31. Inheritance 继承

  • Chapter 02
  • Section 31
  • self-contained study note

31.1 一句话结论

Inheritance 继承是 OO ABAP 中表达“一个 class 是另一个 class 的特殊版本”的机制。subclass 通过 INHERITING FROM 继承 superclass 的 public / protected contract，可以 redefine 允许重定义的方法，也可以新增自己的行为。它适合表达清楚的 is-a / specialization 关系，不适合单纯为了复用几行 helper code。

初学者最重要的判断是：继承不是“代码复用快捷键”，而是长期 contract。父类一旦把 public / protected method 暴露给子类和调用者，后续修改就会影响整条继承链。多数业务开发里，优先考虑小 class、composition、Interface 接口定义 和 constructor injection；只有当“子类确实是父类的一种”并且需要共享稳定扩展点时，才考虑 inheritance。

31.2 到底学什么

上一节 Interface 接口定义 讲的是 capability contract：调用方只关心“这个对象能做什么”。本节讲的是 specialization contract：子类不仅能做父类能做的事，还代表父类概念下的一个更具体类型。

本节要掌握七个边界：

  • CLASS lcl_child DEFINITION INHERITING FROM lcl_parent. 表示 child class 继承 parent class。
  • 如果省略 INHERITING FROM，普通 class 仍隐式位于 ABAP Objects 的根类 object 之下；因此后续会看到 TYPE REF TO object 这种泛型 object reference。
  • ABAP class inheritance 是单继承：一个 class 只能有一个 direct superclass，但可以实现多个 interfaces。
  • subclass 可以访问 superclass 的 public / protected components；不能直接访问 superclass private components。FRIENDS 是高级例外，但不应成为普通继承设计的理由。
  • PROTECTED SECTION 是给子类看的 contract，不是“稍微开放一点的 private”。
  • superclass method 可以被 subclass 用 METHODS method_name REDEFINITION. 重写，前提是该 method 是 public/protected instance method、不是 private/static/final method，并且 signature 不改变。
  • ABSTRACT class / method 表达“不能直接实例化，必须由子类完成或使用”；FINAL class / method 表达“不允许继续继承或重定义”。ABSTRACT 和 FINAL 不能同时用于同一个 class 或 method。
  • 如果只是为了替换实现、做 ABAP Unit test double、隔离依赖，优先用 interface；如果只是为了共享实现细节，优先用 composition 或 private helper。

最小语法模型：

CLASS lcl_document DEFINITION ABSTRACT.
  PUBLIC SECTION.
    METHODS get_kind ABSTRACT
      RETURNING VALUE(rv_kind) TYPE string.

    METHODS describe
      RETURNING VALUE(rv_text) TYPE string.
ENDCLASS.

CLASS lcl_document IMPLEMENTATION.
  METHOD describe.
    rv_text = |Document kind: { get_kind( ) }|.
  ENDMETHOD.
ENDCLASS.

CLASS lcl_invoice DEFINITION
  INHERITING FROM lcl_document.
  PUBLIC SECTION.
    METHODS get_kind REDEFINITION.
ENDCLASS.

CLASS lcl_invoice IMPLEMENTATION.
  METHOD get_kind.
    rv_kind = 'Invoice'.
  ENDMETHOD.
ENDCLASS.

lcl_document 说“所有 document 都能说明自己的 kind”。它本身是 abstract，不能直接 NEW lcl_document( )。lcl_invoice 是 document 的一种，所以继承它，并提供 get_kind 的具体实现。

31.3 为什么重要

inheritance 是 ABAP Objects 里最容易被“写得出来但设计错”的主题之一。语法并不难，难的是判断它是不是合适的建模方式。

在项目里，继承通常影响三件事：

  • 扩展点：父类允许子类在哪些地方 redefine，哪些地方不能动。
  • 可见性：protected method / attribute 会成为子类 contract，父类不能随便改。
  • 多态：父类 reference 可以指向子类对象，运行时调用子类重定义后的 method。

如果继承关系清楚，它可以减少重复、统一流程、提供 framework extension point。例如 SAP framework 里会看到类似 INHERITING FROM cl_abap_behavior_saver 的类，用父类定义生命周期方法，再由具体 behavior pool 子类实现保存逻辑。项目里读到这种代码时，要先问：“父类规定了什么生命周期 contract？子类重定义了哪个 hook？”

如果继承关系不清楚，它会制造隐性耦合。一个看似小的父类 protected method 改名、改参数、改语义，可能让所有子类同时受影响。Java/Python 背景学习者容易把继承当作复用实现的第一选择；在 ABAP 项目里，更保守的选择通常是 interface + composition。

31.4 Key mechanism / 核心机制

31.4.1 inheritance expresses is-a

继承表达的是 is-a 关系，而不是 uses-a 关系。

  • Invoice is a Document：可能适合 inheritance。
  • PaymentService uses a TaxCalculator：更适合 composition 或 interface injection。
  • OrderValidator and DeliveryValidator share 20 lines of code：通常不够成为 inheritance 理由。

判断句很简单：如果不能自然说“子类是父类的一种”，就不要急着写 INHERITING FROM。

31.4.2 subclass inherits public and protected contract

subclass 继承 superclass 的可见 contract。调用者通过 superclass reference 能看到 superclass public components；subclass 内部能访问 superclass public / protected components。

CLASS lcl_rule DEFINITION ABSTRACT.
  PUBLIC SECTION.
    METHODS is_valid ABSTRACT
      RETURNING VALUE(rv_valid) TYPE abap_bool.

  PROTECTED SECTION.
    METHODS normalize
      IMPORTING iv_value        TYPE string
      RETURNING VALUE(rv_value) TYPE string.
ENDCLASS.

CLASS lcl_rule IMPLEMENTATION.
  METHOD normalize.
    rv_value = to_upper( val = iv_value ).
  ENDMETHOD.
ENDCLASS.

normalize 放在 PROTECTED SECTION，意思是父类承诺子类可以调用它。它不只是父类自己的 private helper，而是继承关系的一部分。以后如果父类改变 normalize 的 signature 或语义，子类都会受影响。

这就是为什么前面 Method visibility 可见性 一节强调：protected 要谨慎。private 是父类内部自由，protected 是父类对子类的承诺。

还有一个容易忽略的细节：subclass 访问 inherited protected components 时，通常使用 me->component 或直接写 component name。不要把“任何 superclass reference 都能访问 protected”理解成通用规则；通过 superclass 静态类型的任意 reference 去访问 protected component，会破坏其他 subclass 的封装边界。初学阶段把 protected 视为“本对象继承链内可见”，而不是“所有父类引用可见”。

31.4.3 abstract class and abstract method

ABSTRACT class 不能直接实例化。它通常用来提供 shared contract 或 shared flow，让子类补齐差异点。

CLASS lcl_message_formatter DEFINITION ABSTRACT.
  PUBLIC SECTION.
    METHODS format ABSTRACT
      IMPORTING iv_text          TYPE string
      RETURNING VALUE(rv_result) TYPE string.
ENDCLASS.

abstract method 没有 superclass implementation，必须由非 abstract subclass 提供实现。包含 abstract method 的 class 自身也必须声明为 ABSTRACT；如果某个 subclass 仍没有补齐所有 abstract methods，它也必须保持 abstract，不能直接被实例化。对初学者来说，可以把 abstract method 理解为“父类说这个能力必须存在，但不知道每个子类具体怎么做”。

31.4.4 final class and final method

FINAL 是 inheritance 的刹车。

CLASS lcl_status_text DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS get_text
      IMPORTING iv_status      TYPE string
      RETURNING VALUE(rv_text) TYPE string.
ENDCLASS.

如果一个 class 没有打算给别人继承，尤其是普通业务 service、formatter、validator、mapper，写成 FINAL 通常更清楚：它不是 extension point。其他 class 不能 INHERITING FROM 一个 final class；尝试继承 final class 会在语法检查阶段失败。后续要替换实现时，用 interface；要共享内部逻辑时，用 composition；不要默认让每个 class 都可继承。

同理，method 也可以被设计成不可重定义的 contract。final method 不能被 subclass redefine。初学阶段不必记住所有组合语法，但要理解意图：ABSTRACT 打开必须实现的扩展点，FINAL 关闭继承或重定义；两者互斥，不能同时表达“必须重写”和“不许重写”。

31.4.5 method redefinition changes runtime behavior

redefinition 是 inheritance 里最重要的行为变化。subclass 可以重新实现 superclass method：

CLASS lcl_base_pricing DEFINITION ABSTRACT.
  PUBLIC SECTION.
    METHODS calculate
      IMPORTING iv_amount     TYPE decfloat34
      RETURNING VALUE(rv_net) TYPE decfloat34.
ENDCLASS.

CLASS lcl_base_pricing IMPLEMENTATION.
  METHOD calculate.
    rv_net = iv_amount.
  ENDMETHOD.
ENDCLASS.

CLASS lcl_discount_pricing DEFINITION
  INHERITING FROM lcl_base_pricing.
  PUBLIC SECTION.
    METHODS calculate REDEFINITION.
ENDCLASS.

CLASS lcl_discount_pricing IMPLEMENTATION.
  METHOD calculate.
    rv_net = iv_amount * '0.90'.
  ENDMETHOD.
ENDCLASS.

当变量类型是 superclass reference、实际对象是 subclass 时，调用会走运行时对象的实现：

DATA lo_pricing TYPE REF TO lcl_base_pricing.
lo_pricing = NEW lcl_discount_pricing( ).

DATA(lv_net) = lo_pricing->calculate( iv_amount = '100.00' ).

这就是下一节 Polymorphism 多态 要展开的内容：同一个 call，根据实际对象不同，执行不同实现。

redefinition 有几个硬边界：可重定义的方法必须是 not private/static/final。不能 redefine private method，因为子类看不到它；不能 redefine static method，因为 static method 不参与对象实例的动态分派 dynamic dispatch；不能 redefine final method，因为父类明确关闭了这个扩展点。redefinition 也不能改变 method signature：参数方向、类型、返回值和异常 contract 都属于父类 method contract，subclass 只能改变 implementation。

31.4.6 super call preserves parent behavior

如果 subclass redefine method，但仍想复用 superclass 的原有逻辑，可以在 subclass method 中调用 super->method_name( )。

CLASS lcl_prefixed_document DEFINITION
  INHERITING FROM lcl_document.
  PUBLIC SECTION.
    METHODS describe REDEFINITION.
    METHODS get_kind REDEFINITION.
ENDCLASS.

CLASS lcl_prefixed_document IMPLEMENTATION.
  METHOD describe.
    rv_text = |[SAP] { super->describe( ) }|.
  ENDMETHOD.

  METHOD get_kind.
    rv_kind = 'Generic'.
  ENDMETHOD.
ENDCLASS.

super->describe( ) 表示调用 superclass 的 describe implementation。它适合“先执行父类标准流程，再补一点子类差异”的场景。但如果每个子类都要绕开父类大半逻辑，说明父类抽象可能设计错了。

super-> 只能在 subclass 的 instance method 中使用，指向 direct superclass 的 implementation；不能写 super->super->... 跳过中间层。constructor 场景还要更谨慎：如果 subclass 定义了自己的 constructor，而 superclass constructor 有 mandatory importing parameters，subclass constructor 需要显式调用 super->constructor( ... ) 来完成父类初始化。初学阶段不要在 constructor 链里放业务副作用，否则所有子类创建都会变重。

31.4.7 inheritance vs interface vs composition

三者经常混在一起，需要明确区分：

  • inheritance：lcl_invoice 是 lcl_document 的一种。
  • interface：lcl_invoice 能提供 lif_printable 的能力。
  • composition：lcl_invoice_service 使用 lif_tax_calculator 或 lcl_number_formatter。

项目里的保守判断：

  • 需要多实现、可替换、test double：优先 interface。
  • 只是把一个对象交给另一个对象使用：优先 composition。
  • 需要表达 is-a，且父类提供稳定扩展点：才使用 inheritance。
31.4.8 inheritance decision checklist

在 code review 里，继承设计可以用下面这张表收口。重点不是“ABAP 语法能不能继承”，而是“这个父子关系是否真的表达稳定的 is-a contract，父类是否愿意长期承诺 public / protected extension points”。一旦父类把大量内部状态暴露成 protected，后续每次修改父类都可能影响所有子类；这时继承就不再是复用捷径，而是长期耦合。

QuestionInheritance may fit when...Prefer interface / composition when...
Is there a true is-a relation?子类确实是父类的一种，例如 invoice is a document。只是想复用几行逻辑，或只是多个对象有相似动作。
Is the parent contract stable?public / protected methods 是长期扩展点，调用顺序和含义清楚。父类内部 helper 还经常变化，子类只是顺便调用它。
Do subclasses share lifecycle or flow?父类定义模板流程，子类只补差异点。每个实现只是同一 capability 的不同来源，例如不同税率计算器。
Will protected members stay small?protected API 少、稳定、命名清楚。需要把大量内部状态暴露给子类才能完成需求。
Is polymorphism needed?父类 reference 调用子类行为是设计目标。调用方只需要一个 interface contract，不关心具体父类。
Can tests stay simple?每个 subclass 可以复用共同 contract test。测试必须理解父类内部状态、hook 顺序和隐藏副作用。

普通业务代码里，很多 class 更适合先设计成 FINAL，再用 interface 表达可替换能力，用 composition 组合 helper 或 service。继承仍然有价值，尤其在 framework lifecycle、template method、RAP saver 这类由父类规定 extension point 的场景；但它的成本是更强的耦合、更难的测试边界和更谨慎的 protected contract 维护。

31.5 日本项目现场说法

  • 常见日语：継承、親クラス、子クラス、スーパークラス、サブクラス、抽象クラス、再定義、protectedメソッド、is-a関係、継承階層。
  • 设计说明句：このクラスは共通処理を親クラスに持たせ、差分処理をサブクラスで再定義する設計です。
  • Review 指摘句：この設計は is-a 関係ではなく処理共通化が目的に見えるため、継承ではなくインターフェースまたは委譲を検討してください。
  • 风险提示句：protectedメソッドは子クラス向けの契約になるため、後続変更の影響範囲を確認する必要があります。
  • 使用场景：class design review、framework extension point 分析、RAP behavior pool 读取、ABAP Unit 设计、重构讨论、影响调查。

日本项目里 継承 经常和 共通化 一起出现，但两者不能等同。共通化したいから親クラスを作る 不是充分理由；要进一步确认是否存在稳定的 親子関係 和 再定義ポイント。

31.6 English Terms

  • Official / search terms: inheritance, INHERITING FROM, superclass, subclass, method redefinition, ABSTRACT, FINAL, super->method
  • Related syntax: CLASS ... DEFINITION INHERITING FROM ..., METHODS ... REDEFINITION, METHODS ... ABSTRACT, CLASS ... DEFINITION ABSTRACT, CLASS ... DEFINITION FINAL
  • Design terms: is-a relationship, extension point, template method, composition, delegation, polymorphism, dynamic dispatch, protected contract, mutually exclusive
  • 读官方资料时注意：inheritance 也可能出现在 package inheritance、DDIC domain inheritance 等语境；本节讲的是 OO ABAP class inheritance。

31.7 Example / 学习例子

31.7.1 例 1：用 abstract superclass 统一流程
CLASS lcl_exporter DEFINITION ABSTRACT.
  PUBLIC SECTION.
    METHODS export
      RETURNING VALUE(rv_text) TYPE string.

  PROTECTED SECTION.
    METHODS get_header ABSTRACT
      RETURNING VALUE(rv_header) TYPE string.

    METHODS get_body ABSTRACT
      RETURNING VALUE(rv_body) TYPE string.
ENDCLASS.

CLASS lcl_exporter IMPLEMENTATION.
  METHOD export.
    rv_text = |{ get_header( ) }: { get_body( ) }|.
  ENDMETHOD.
ENDCLASS.

export 是父类统一流程：先 header，再 body。get_header 和 get_body 是子类必须补齐的差异点。

CLASS lcl_invoice_exporter DEFINITION
  INHERITING FROM lcl_exporter.
  PROTECTED SECTION.
    METHODS get_header REDEFINITION.
    METHODS get_body   REDEFINITION.
ENDCLASS.

CLASS lcl_invoice_exporter IMPLEMENTATION.
  METHOD get_header.
    rv_header = 'Invoice'.
  ENDMETHOD.

  METHOD get_body.
    rv_body = 'Amount: 100.00'.
  ENDMETHOD.
ENDCLASS.

调用方可以只知道它是 lcl_exporter：

DATA lo_exporter TYPE REF TO lcl_exporter.
lo_exporter = NEW lcl_invoice_exporter( ).

DATA(lv_text) = lo_exporter->export( ).

这个例子适合 inheritance，因为父类真的定义了一套稳定流程，子类只填补差异步骤。

31.7.2 例 2：不适合继承的复用
CLASS lcl_order_validator DEFINITION
  INHERITING FROM lcl_string_helper.
ENDCLASS.

如果 lcl_order_validator 并不是 lcl_string_helper 的一种，只是想复用字符串处理方法，这就是错误信号。更好的做法通常是 composition：

CLASS lcl_order_validator DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS constructor
      IMPORTING io_formatter TYPE REF TO lif_text_formatter.

  PRIVATE SECTION.
    DATA mo_formatter TYPE REF TO lif_text_formatter.
ENDCLASS.

这样 lcl_order_validator 仍然是 validator，formatter 只是它使用的协作者。

31.7.3 例 3：RAP saver 里的继承意识

后续 RAP 章节会看到类似：

CLASS lsc_zxxx DEFINITION
  INHERITING FROM cl_abap_behavior_saver.
ENDCLASS.

RAP 中还会看到 handler 相关基类，例如 CL_ABAP_BEHAVIOR_HANDLER，以及 saver 相关基类 CL_ABAP_BEHAVIOR_SAVER。前者偏向 transactional handler methods，后者偏向 save sequence methods。它们都不是为了复用普通 helper，而是 SAP framework 定义生命周期父类，子类按规定 method signature 实现业务逻辑。

读这种代码时，要连接到 RAP saver phase is where database writes should happen，理解“父类规定生命周期，子类实现扩展点”的模式。同时要保留 Clean Core / ABAP Cloud 意识：继承 SAP standard class 是否允许，不只看语法，还要看该 class 在当前系统和语言版本中是否 released / allowed。不要为了访问 protected area 去继承未释放的 SAP 标准类。

31.8 Counterexample / Failure mode

31.8.1 误区 1：为了复用 helper 就继承

两个 class 都需要 format_amount，不代表它们有父子关系。

修正动作：把 helper 抽成 private method、utility collaborator、formatter class，或通过 interface 注入。

31.8.2 误区 2：把 protected 当成默认可见性

protected 会被子类依赖，后续改动成本比 private 高。

修正动作：默认 private；只有明确设计 subclass extension point 时才 protected。

31.8.3 误区 3：重定义后破坏父类 contract

父类 method 说 calculate 返回 net amount，子类却返回 gross amount。语法可能能过，但语义坏了。

修正动作：redefinition 不能改变调用方对父类 method 的基本期待；必要时新建 method 或改抽象模型。

31.8.4 误区 4：继承层级太深

三四层以上的业务继承，很容易让缺陷调查变成“到底哪个 class 改了行为”的追踪游戏。

修正动作：压平设计；用 interface + composition 拆出变化点。

31.8.5 误区 5：把 inheritance 和 authorization / customizing 混在一起

不同公司代码、不同税码、不同权限，不一定需要不同 subclass。

修正动作：业务差异优先考虑配置、strategy object、rule class、BAdI / enhancement point 或 RAP behavior design；继承只用于稳定类型层级。

31.8.6 误区 6：父类 constructor 做太多事

如果父类 constructor 里做 DB 更新、remote call、authorization side effect，所有子类创建对象都会背上这个成本。

修正动作：constructor 只建立有效初始状态；业务动作放到显式 method；依赖通过 interface 注入。

31.8.7 误区 7：通过继承访问未释放 SAP 标准类

看到某个 SAP standard class 有 protected method，不代表项目可以通过 subclass 继承来使用它。

修正动作：优先使用 released API、documented extension point、BAdI、RAP framework contract 或 SAP Help 明确允许的方式；ABAP Cloud / Clean Core 场景下尤其要确认 release status。

31.9 思考问题

  • 这个 subclass 真的是 superclass 的一种吗？能自然说出 is-a 关系吗？
  • 这里需要的是继承、interface，还是 composition？
  • superclass 暴露了哪些 protected contract？未来改动会影响哪些子类？
  • 哪些 method 允许 redefinition？子类重定义后是否仍满足父类语义？
  • class 是否应该是 FINAL，避免无意成为 extension point？
  • 如果使用 abstract class，哪些步骤由父类固定，哪些步骤由子类补齐？
  • 日本项目 review 时，应说 継承関係、再定義，还是 委譲 / インターフェース化？
  • 在 RAP 或 framework code 中，是否能识别 INHERITING FROM 背后的生命周期 contract？
