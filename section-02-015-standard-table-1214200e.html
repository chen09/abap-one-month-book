15. Standard table 标准表

  • Chapter 02
  • Section 15
  • self-contained study note

15.1 一句话结论

Standard table 标准表，是 ABAP internal table 内部表的默认表类别：它用 primary table index 管理行，允许重复行，适合按顺序追加、遍历、临时承接查询结果和小规模处理；但如果主要需求是按 key 高频查找或保证唯一性，就要考虑 Sorted table 排序表、Hashed table 哈希表 或 Secondary table key 二级键。

15.2 到底学什么

先把 standard table 和数据库表分开。ABAP internal table 是运行在 ABAP 程序内存中的动态数据对象，用来临时保存多行同结构数据。它不是 DDIC database table，也不是 SAP HANA 上的真实持久化表。常见来源包括：

SELECT FROM zt_demo
  FIELDS key_field, text
  INTO TABLE @DATA(lt_demo).

这里的 lt_demo 是 ABAP 程序内存中的 internal table。它可能来自数据库查询结果，也可能来自接口输入、BAPI 返回表、ALV 数据、RAP EML response、手工构造的测试数据。学习 standard table，就是学习这类内存表的默认行为。

ABAP internal table 有三种基本 table category：

STANDARD TABLE
SORTED TABLE
HASHED TABLE

standard table 的特点可以先记成四句话：

  • 它有 primary table index，所以可以按行号访问，也可以按顺序 LOOP AT。
  • 它的 primary key 不唯一，允许重复行。
  • 它没有天然排序；行的顺序通常来自 append / insert / sort / select result 的处理过程。
  • 它用 primary key 读取时，对 standard table 通常是 linear search；大表高频 key lookup 可能会慢。

因此，standard table 是“最顺手的默认容器”，不是“所有场景的最佳容器”。它像 Java 的 ArrayList 或 Python 的 list：追加和顺序遍历很自然，但如果你总是按 key 找对象，就要考虑 Map / dict 风格的数据结构。在 ABAP 里，这个选择通常对应 sorted table、hashed table 或 secondary key。

15.3 为什么重要

ABAP 项目里，internal table 几乎无处不在。SELECT ... INTO TABLE、BAPI 的 TABLES 参数、ALV 输出、批处理临时数据、RAP response、CSV 导入、接口 payload 转换，都会让你面对一张或多张 internal table。

初学者常见写法是：

LOOP AT lt_header INTO DATA(ls_header).
  READ TABLE lt_item INTO DATA(ls_item)
    WITH KEY header_id = ls_header-header_id.
ENDLOOP.

如果 lt_item 是很小的 standard table，这可能没问题。但如果 lt_header 和 lt_item 都有几万行，而 READ TABLE ... WITH KEY 每次都对 standard table 做线性搜索，就会形成典型的 nested-loop performance issue。项目上常说“跑得很慢”，很多时候不是数据库慢，而是 ABAP 内存表选型和访问方式不对。

standard table 还影响可读性。TYPE TABLE OF ... 没写表类别时，通常就是 standard table；SELECT ... INTO TABLE @DATA(...) 的 inline declaration 也常产生 standard table，并且 key 可能是 empty key。代码如果没有明确写 WITH EMPTY KEY、WITH NON-UNIQUE KEY ... 或 secondary key，reviewer 很难判断作者是否真的考虑过 key 语义。

对交付来说，standard table 学不好，会带来三类问题：

  • 性能问题：大表反复 READ TABLE ... WITH KEY，没有 sorted/hashed/secondary key。
  • 语义问题：以为 key 唯一，但 standard table 的 primary key 不能保证唯一。
  • 维护问题：默认 key / standard key 不清晰，排序、查找、去重逻辑让后续维护者猜。

15.4 Key mechanism / 核心机制

15.4.1 table category、line type、table key

一张 internal table 至少要理解三个维度：

line type     每一行长什么样
table category 行如何被内部管理：standard / sorted / hashed
table key      用哪些字段作为查找语义

standard table 由 primary table index 管理。primary table index 给每一行一个顺序位置，所以 standard table 可以按 index 访问：

READ TABLE lt_demo INTO DATA(ls_demo) INDEX 1.

也可以顺序遍历：

LOOP AT lt_demo INTO DATA(ls_demo).
  " process line
ENDLOOP.

这也是 standard table 非常常见的原因：很多业务处理就是先取得一组结果，再按顺序处理、输出、转换或汇总。

15.4.2 standard table 的 key 不唯一

standard table 的 primary key 总是 non-unique。即使你写了：

TYPES ty_t_demo TYPE STANDARD TABLE OF ty_demo
  WITH NON-UNIQUE KEY id.

这只是告诉 ABAP “这些字段组成 key 语义”，不是告诉 ABAP “不允许重复”。如果你需要主键唯一并且按 key 快速读取，应该优先考虑 hashed table；如果你需要有序并且按 key 范围或前缀读取，可以考虑 sorted table。

15.4.3 empty key 与 default key

standard table 可以有 empty key：

DATA lt_demo TYPE STANDARD TABLE OF ty_demo WITH EMPTY KEY.

empty key 表示 primary table key 不包含任何 key fields。它很适合“这张表只是顺序容器，不按 key 查找”的场景。显式写 WITH EMPTY KEY 的好处是让 reviewer 明白：作者知道这张表没有 key 语义。

另一个常见写法是省略 key：

DATA lt_demo TYPE TABLE OF ty_demo.

省略不等于“没有 key”。在许多场景下，ABAP 会使用 standard key / default key。structured line type 的 standard key 通常由 character-like 和 byte-like 组件组成，这可能不是业务上真正想要的 key。项目里建议尽量显式写 key：要么 WITH EMPTY KEY，要么 WITH NON-UNIQUE KEY ...，不要让默认 key 替你做设计决定。

15.4.4 key 访问为什么可能慢

对 standard table 使用 primary key 访问时，关键点是：primary key lookup 不会像 hashed table 那样天然 O(1)，也不会像 sorted table 那样天然 binary search。标准表的 key 访问通常是 linear search，从前往后找。

READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY material = lv_matnr.

如果 lt_items 是 standard table，这句在小表中很自然；但在大表中、尤其在外层循环里反复执行时，就可能成为性能瓶颈。解决方向通常有三个：

只顺序处理
  -> standard table 足够

频繁按唯一 key 读取
  -> hashed table

频繁按有序 key / 范围读取
  -> sorted table

已有 standard table 但还需要某种快速 key 读取
  -> secondary table key
15.4.5 standard table 没有天然排序

standard table 的行顺序来自你如何填充它。APPEND 通常加到末尾；INSERT ... INDEX ... 可以插入指定位置；SORT 会重排行顺序。不要把 SELECT ... INTO TABLE 的返回顺序当成业务排序，除非 SQL 里有明确 ORDER BY，并且后续 internal table 处理没有再次改变顺序。

如果后续逻辑依赖顺序，要么明确 ORDER BY，要么在 internal table 上明确 SORT BY ...，并在注释或设计说明里写出排序目的。

15.4.6 standard table 与 sy-tabix

standard table 是 index table，很多 internal table 操作会让 sy-tabix 有意义。例如 LOOP AT 时，sy-tabix 通常表示当前行号；READ TABLE ... INDEX 读取成功时也能反映行号。后续学习 READ TABLE 和 LOOP AT 时会详细讲。

但不要把 sy-tabix 当成业务 key。行号只是当前内存表状态下的位置；一旦 SORT、DELETE、INSERT，行号就会变化。业务关联应使用业务字段或显式 key。

  • この重複は許容される仕様ですか。
  • ソート順に依存している場合、どこで SORT していますか。

15.7 Example / 学习例子

15.7.1 例 1：最常见的 standard table
TYPES: BEGIN OF ty_item,
         id     TYPE i,
         text   TYPE string,
         amount TYPE p LENGTH 8 DECIMALS 2,
       END OF ty_item.

DATA lt_items TYPE STANDARD TABLE OF ty_item WITH EMPTY KEY.

APPEND VALUE #( id = 1 text = `A` amount = '10.00' ) TO lt_items.
APPEND VALUE #( id = 2 text = `B` amount = '20.00' ) TO lt_items.

LOOP AT lt_items INTO DATA(ls_item).
  " 顺序处理
ENDLOOP.

这张表没有 key 语义，只是按顺序保存一批数据。WITH EMPTY KEY 是一个很好的信号：这不是忘了设计 key，而是明确不需要 key。

15.7.2 例 2：显式 non-unique key
DATA lt_items TYPE STANDARD TABLE OF ty_item
  WITH NON-UNIQUE KEY id.

APPEND VALUE #( id = 1 text = `A` amount = '10.00' ) TO lt_items.
APPEND VALUE #( id = 1 text = `A2` amount = '15.00' ) TO lt_items.

这段代码允许两个 id = 1 的行同时存在。WITH NON-UNIQUE KEY id 帮助读者知道 id 是查找语义的一部分，但它并不保证唯一。

15.7.3 例 3：标准表 key lookup 的风险
LOOP AT lt_headers INTO DATA(ls_header).
  READ TABLE lt_items INTO DATA(ls_item)
    WITH KEY header_id = ls_header-header_id.

  IF sy-subrc = 0.
    " process item
  ENDIF.
ENDLOOP.

如果 lt_items 是 standard table，READ TABLE ... WITH KEY 可能每次都线性搜索。小数据量可以接受；大数据量时，应考虑把 lt_items 改成 sorted/hashed table，或给 standard table 加合适的 secondary key。

15.7.4 例 4：用 secondary key 改善 standard table 查找
TYPES ty_t_item TYPE STANDARD TABLE OF ty_item
  WITH EMPTY KEY
  WITH NON-UNIQUE SORTED KEY by_header COMPONENTS header_id.

DATA lt_items TYPE ty_t_item.

READ TABLE lt_items INTO DATA(ls_item)
  WITH TABLE KEY by_header COMPONENTS header_id = lv_header_id.

这仍然是 standard table，但多了一个 secondary sorted key。它适合“主要还是按顺序处理，但某些场景需要按另一个 key 快速查找”的情况。代价是额外内存和 key 维护成本；小表或频繁修改的表不一定值得。

15.7.5 例 5：不要把 standard table 当数据库表
SELECT FROM zt_demo
  FIELDS *
  INTO TABLE @DATA(lt_demo).

READ TABLE lt_demo INTO DATA(ls_demo)
  WITH KEY key_field = lv_key.

zt_demo 是数据库表；lt_demo 是 internal table。READ TABLE 只能读取 internal table，不能直接读 DDIC database table。数据库读取用 ABAP SQL SELECT，内存表读取用 READ TABLE / table expression / loop。

15.8 Counterexample / Failure mode

15.8.1 误区 1：省略 key 就以为没有 key
DATA lt_items TYPE TABLE OF ty_item.

这通常会创建 standard table，但 key 语义可能来自 default / standard key。项目代码里最好显式写：

DATA lt_items TYPE STANDARD TABLE OF ty_item WITH EMPTY KEY.

或者：

DATA lt_items TYPE STANDARD TABLE OF ty_item
  WITH NON-UNIQUE KEY id.

这样 reviewer 能看出你的意图。

15.8.2 误区 2：认为 standard table 会自动防重复

standard table 的 primary key 不唯一。即使你声明了 key，也不会因为重复而拒绝插入。如果业务不允许重复，需要用 hashed/sorted unique key、secondary unique key，或者在插入前显式检查。

15.8.3 误区 3：在大表循环里反复读 standard table
LOOP AT lt_a INTO DATA(ls_a).
  READ TABLE lt_b INTO DATA(ls_b)
    WITH KEY id = ls_a-id.
ENDLOOP.

这类代码在测试数据很少时看不出问题，上生产后可能变成性能缺陷。先问：lt_b 有多少行？这个 read 会执行多少次？能否改成 hashed table、sorted table 或 secondary key？

15.8.4 误区 4：依赖偶然顺序

standard table 的顺序不是业务保证。只要代码中有 SORT、DELETE、INSERT INDEX 或不同的 SELECT 来源，顺序都可能变化。需要顺序时要明确 ORDER BY 或 SORT BY。

15.8.5 误区 5：把 internal table 性能问题归咎于数据库

很多“程序慢”并不是 SQL 慢，而是已经把数据取到内存后，在 standard table 上做了低效搜索。性能分析时要分清 database time 和 ABAP processing time。

15.9 思考问题

  • 当前 internal table 是顺序处理为主，还是 key lookup 为主？
  • 这张表是否允许重复？如果不允许，standard table 能保证吗？
  • 代码是否显式写了 WITH EMPTY KEY 或 WITH NON-UNIQUE KEY ...？
  • READ TABLE ... WITH KEY 是否在循环里反复执行？
  • 如果这张表超过 1 万行，当前 table category 还合适吗？
  • 这张表的顺序是否有业务意义？如果有，排序在哪里保证？
  • 是否应该改成 Sorted table 排序表、Hashed table 哈希表 或增加 Secondary table key 二级键？
  • 用日语 review 时，如何说明 標準テーブル、空キー、線形検索？
