39. AC0567C POC architecture replay

39.1 一句话结论

AC0567C POC architecture replay 是把本教程中的 AC0567C batch upload API POC 当作一个小型真实 SAP 项目来复盘：从业务场景、数据模型、RAP business object 业务对象、Service definition 服务定义、Service binding 服务绑定、OData API、authorization、idempotency、保存阶段、日志、测试证据、性能和上线准备一路串起来，确认读者不只是“看过代码”，而是能说明一个 SAP 交付方案为什么这样设计、怎样验证、还缺什么才能进入生产。

它不是单纯的代码导读，也不是把 POC 包装成已经可上线的产品。正确的读法是：AC0567C 是本工程里的可运行、可观察、可复盘练习项目；architecture replay 是把练习项目拆回 SAP 项目现场会问的几个层次：业务问题是什么，SAP repository object 如何组织，API contract 如何暴露，RAP 保存时序和事务边界由谁负责，失败时怎样留下证据，上线前还需要哪些 hardening。

本节会用本地 POC 目录作为证据入口：

39.2 到底学什么

第一，要学会把 POC 放回业务语境。一个 batch upload API 不是只有“上传文件”或“调用接口”这么简单。它通常意味着外部系统、RPA 替代方案、前端工具或中间件提交一批业务数据，SAP 侧需要校验、落库、返回状态、记录错误，并且允许后续查询或重试。读者要能说清楚：调用方是谁，业务成功的定义是什么，重复请求如何处理，失败记录保存在哪里。

第二，要学会画对象链。AC0567C 的 POC 不是抽象概念，它有实际对象名：

层次POC 证据
服务入口ZRPA01_SB_AC0567C_IMPORT，在 POC 架构总结和 runtime metadata 语境中作为 OData V4 service binding；当前包未复制 .srvb 源文件
Service definitionZRPA01_SD_AC0567C_IMPORT，暴露 ZRPA01_I_AC0567C_IMPORT as PriorAllocationImportsC
RAP root viewZRPA01_I_AC0567C_IMPORT，以 TADIR 中的服务类对象作为技术锚点
Behavior definitionZRPA01_I_AC0567C_IMPORT.bdef，声明 ValidateBatch、ExecuteImport、GetStatus 三个 static action
Behavior poolZBP_RPA01_I_AC0567C_IMPORT，把 action 请求转交给服务类，并在 saver 阶段执行写入
Service classZCL_RPA01_AC0567C_IMP_SVC，处理校验、授权、重复检查、key 派生和结果结构
Target tableZCOM_B_ACTT397，真实写入的目标表
API evidence03_api_docs、04_tests/TEST_EVIDENCE.md、04_tests/test-results/ac0567c-no-write-latest.json

第三，要学会把运行时机制和交付证据对应起来。RAP 的 interaction phase、validation、determination、action、save sequence、message reporting、authorization 和 commit/rollback 边界，最终都应该能在测试证据、错误响应、Application Log 应用日志、ATC 结果、性能观察或交付文档中找到对应说明。

第四，要学会区分 POC 可复用部分和生产缺口。AC0567C 已经证明 JSON Rows[] 多行校验、Base64 TSV validate path、真实插入、重复 ImportId 冲突和状态查询这些关键路径，但它仍然明确标注 production hardening 未完成，例如授权负例、lock/enqueue、persistent audit log、CSV/TSV parser 强化、容量测试和最终 Claude/Joule 复核。

39.3 为什么重要

很多初学者学完 RAP 或 OData 后会有一种错觉：只要 service preview 能打开，接口能返回 JSON，就已经完成了项目交付。AC0567C 的价值恰好在于打破这个错觉。真实 SAP 项目关心的不只是“能不能跑”，还关心“谁来用、错了怎么办、权限怎么挡、重复请求会不会产生脏数据、上线后怎么查、升级后会不会坏”。

对 ABAP 学习者来说，这一节是 Week 3/4 的总复盘。前面的 CDS、behavior、service binding、OData、authorization、application log、ATC、performance 和 production hardening，如果各自孤立学习，很容易变成术语清单。放到 AC0567C 里，它们会变成一条项目证据链：业务输入进入 API，经过 RAP 行为和保存逻辑，生成业务状态和日志，再由测试与交付材料证明它可以被团队 review。

对 Java/Python 背景读者来说，可以把它类比为一次从 demo 到 delivery checklist 的架构复盘：domain model、service layer、controller/API contract、transaction、test evidence 和 deployment readiness 都要讲清楚。但 SAP 还额外强调 repository object、activation、transport、client、system landscape、released API、authorization object、RAP save sequence 和 Clean Core 约束。

39.4 Key mechanism / 核心机制

复盘 AC0567C 时，可以按七步走。

第一步，确认业务问题。这个 POC 的业务问题是把 AC0567 风格的多行 prior allocation upload 从旧 SAP GUI transaction/report 思路，改造成 API-first 的 batch upload POC。它不复制旧事务，而是新建 API/service，让调用方通过 JSON 或 Base64 TSV 提交行项目，并通过 validate、execute、status 三类动作取得结果。

第二步，画 repository object chain。最小链路是：

Caller / Swagger / PowerShell / Simulator
  -> Local SAP proxy
  -> OData V4 service binding ZRPA01_SB_AC0567C_IMPORT
  -> Service definition ZRPA01_SD_AC0567C_IMPORT
  -> RAP entity ZRPA01_I_AC0567C_IMPORT
  -> Behavior definition ZRPA01_I_AC0567C_IMPORT.bdef
  -> Behavior pool ZBP_RPA01_I_AC0567C_IMPORT
  -> Service class ZCL_RPA01_AC0567C_IMP_SVC
  -> RAP saver
  -> ZCOM_B_ACTT397

这条链路告诉读者：API 不是某一个类，也不是某一个 URL；它是多个 SAP repository object、runtime binding、测试脚本和交付证据共同组成的方案。

第三步，读 API contract。ZRPA01_SD_AC0567C_IMPORT 只暴露一个公开实体名：

define service ZRPA01_SD_AC0567C_IMPORT {
  expose ZRPA01_I_AC0567C_IMPORT as PriorAllocationImportsC;
}

这意味着 PriorAllocationImportsC 会成为外部文档、proxy、测试脚本和 URL path 里的公开契约名。后续若重命名，不只是改 ABAP 对象，还会影响 consumer、OpenAPI 文档和测试证据。打开 BDEF 时还会看到 singular alias PriorAllocationImportC；它是 behavior 内部 alias，而 service 暴露给外部的是 plural name PriorAllocationImportsC，两者不要混读。

第四步，读 RAP action 设计。BDEF 声明了三个 static action：

ValidateBatch  - 校验批量输入，不写 DB
ExecuteImport  - 校验、授权、重复检查，通过后进入保存路径
GetStatus      - 按 ImportId 派生 key 查询插入结果

ValidateBatch 和 ExecuteImport 使用 deep parameter ZRPA01_A_AC0567C_BATCH_REQ，其中 _Rows 是多行输入。这个设计适合讲“API 输入结构”与“RAP action parameter”的连接。

第五步，读保存模型。POC 初版曾被 Joule 指出 action call chain 里直接写 ZCOM_B_ACTT397 的风险，最终版本把 ExecuteImport 改成 staging：先校验、授权、重复检查并生成待插入数据，再由 behavior saver 的 save 方法执行 INSERT zcom_b_actt397 FROM TABLE @lt_insert。这正好连接 AC0567C RAP saver replay 和 COMMIT WORK restriction in RAP：RAP 场景下不要把 COMMIT WORK 或数据库写入随意散落在 action handler 里。这里也要注意，POC 用 lcl_ac0567c_save_buffer 的 static class-data 作为 action 到 saver 的桥接，它比 action 中直接写 DB 更接近 RAP 保存阶段，但不是生产级 transaction-isolated buffer；并发、lock/enqueue 和清理策略仍是本番化课题。

第六步，读幂等和状态查询。ImportId 末尾的 14 位时间戳会派生 zzup_dat 和 zzup_tme。真实测试中的 POC-AC0567C-20260612115720 映射为：

zzup_dat = 20260612
zzup_tme = 115720

重复执行同一个 ImportId 会返回 CONFLICT，状态查询会通过 GetStatus 找到已插入行数。这里应连接 AC0567C idempotency replay。

第七步，读权限和生产缺口。源码里有三层值得分开看：root view 当前标注 @AccessControl.authorizationCheck: #NOT_REQUIRED，说明这条 CDS view 不依赖 DCL；BDEF 声明了 authorization master ( instance ) 和 lock master，但 behavior pool 里的 get_instance_authorizations 与 lock 目前是空 stub；三个 action 又都是 static action，没有 get_global_authorizations 时不会被 instance authorization 实际挡住。因此，当前真正执行的业务授权是 service class 在 ExecuteImport 路径中执行的 AUTHORITY-CHECK OBJECT 'F_BKPF_BUK'，检查 BUKRS 和 ACTVT = '01'。这证明 POC 已经有业务写入前的授权检查意识，但也说明生产化还要补：static action 的 RAP global authorization 设计、ValidateBatch / GetStatus 是否需要授权门、授权负例测试、DCL/RAP/ABAP authorization 边界说明和运维审计证据。这里应连接 AC0567C authorization replay。

  • 処理フロー
  • 証跡

设计 review 时可以这样说：

本POCは、CDS、RAPビヘイビア定義、サービス定義、サービスバインディング、

说明 POC 和生产化差距时可以这样说：

POCとして成立している範囲と、本番化に向けて追加確認が必要な範囲を

向客户或上级说明上线前提时可以这样说：

本番化する場合、権限、監視、性能、移送、運用手順について追加確認が必要です。

这些说法的重点是避免把 POC 说成“已经可以直接上线”。日本项目现场通常更看重边界表达：哪里已经验证，哪里只是设计候选，哪里必须在本番化前补证据。

39.7 Example / 学习例子

可以把 AC0567C 的架构复盘写成一页项目说明：

Business scenario:
  Replace a screen-style prior allocation upload flow with an API-first
  batch upload POC. Caller submits JSON Rows[] or Base64 TSV data.

Data/model layer:
  ZRPA01_I_AC0567C_IMPORT anchors the RAP BO on the service class object.
  ZRPA01_A_AC0567C_BATCH_REQ and ZRPA01_A_AC0567C_BATCH_ROW define deep
  input parameters for multi-row batch data.
  ZRPA01_A_AC0567C_RES defines the action result.

Behavior layer:
  ZRPA01_I_AC0567C_IMPORT.bdef declares static actions:
  ValidateBatch, ExecuteImport, GetStatus.
  ZBP_RPA01_I_AC0567C_IMPORT maps action parameters to service class calls.
  The saver inserts staged rows into ZCOM_B_ACTT397.

API layer:
  ZRPA01_SD_AC0567C_IMPORT exposes PriorAllocationImportsC.
  Local proxy exposes validate, execute, upload/validate, upload/execute,
  and status routes for Swagger/PowerShell/simulator testing.

Evidence:
  No-write test passed 8/8.
  Real insert test proved InsertedRows = 2.
  Repeated ImportId returned CONFLICT.
  GetStatus returned IMPORTED and InsertedRows = 2.

Production gaps:
  Authorization negative test, lock/enqueue, persistent audit log,
  parser hardening, larger-batch performance, transport/monitoring,
  and final project review are still required.

这个例子的价值不在于所有项目都要照抄这些对象名，而在于读者能把一个 POC 翻译成项目交付语言。只要能用这张表说明对象链、运行机制、证据和缺口，就已经从“看代码”进入“读架构”。

39.8 Counterexample / Failure mode

常见失败模式如下。

第一，只看代码，不看业务问题。这样会导致对象很多，但没有人知道接口成功的业务定义是什么。

第二，只看 API 成功返回，不看失败响应、权限失败、重复请求、application log 和生产化证据。这样的 POC 无法支撑 review。

第三，把 POC 当成架构定稿。POC 可以简化，但生产项目必须补测试、监控、transport、Clean Core、性能和运维。

第四，混淆 RAP 层次。CDS、behavior、handler、saver、service binding 各有责任；如果都混在一个“接口实现”里，设计会很难维护。

第五，忽略保存时序。在 RAP 中随意使用 COMMIT WORK 或绕开保存序列，容易破坏事务一致性。

第六，忽略项目证据。没有 request/response、错误 payload、日志、ATC、性能和 transport 证据，架构复盘只能停留在口头说明。

第七，把本地 POC 名称当成通用术语。AC0567C 是本工程案例名，不是 SAP 官方产品名；在客户说明中应换成具体业务场景。

第八，把 @AccessControl.authorizationCheck: #NOT_REQUIRED 误读成“不需要权限”。在本 POC 中，它只说明这个 CDS view 不走 DCL；BDEF 的 instance authorization 和 lock 目前也只是声明/空 stub，static actions 没有 global authorization gate；业务写入路径实际依赖 ExecuteImport 内的 AUTHORITY-CHECK F_BKPF_BUK。生产设计要把 DCL、RAP global/instance authorization 和 ABAP authorization 的职责写清楚。

39.9 思考问题

  • AC0567C 的业务目标是什么？它不是在展示哪一个孤立语法点？
  • 数据模型、RAP behavior、service binding、API evidence 分别位于哪一层？
  • PriorAllocationImportsC 为什么是 API contract 的一部分，而不只是一个内部别名？
  • ValidateBatch、ExecuteImport、GetStatus 三个 action 分别证明什么？
  • 哪些设计可以作为 reusable pattern，哪些必须按客户系统重新确认？
  • 如果要生产化，最先补哪三类证据？
  • 这份 POC 如何映射到基本设计、详细设计、测试规格和运维手顺？
  • 如果客户问“这个 POC 是否可以直接上线”，应该怎样回答？
