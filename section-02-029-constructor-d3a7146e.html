Chapter 02 - Core ABAP Programming

29. Constructor 构造方法

  • Chapter 02
  • Section 29
  • self-contained study note

29.1 一句话结论

Constructor 构造方法是 OO ABAP 中对象或 class 被初始化时由 runtime 自动调用的特殊 method。METHODS constructor 是 instance constructor，用来把一个 object instance 建成“可用且状态一致”的对象；CLASS-METHODS class_constructor 是 class constructor，用来初始化 class-level static state。

初学者最重要的判断是：constructor 应该建立对象的初始状态和必要依赖，不应该把主业务流程、数据库更新、远程调用、commit、权限判断或复杂副作用塞进去。一个好的 constructor 像“上车前检查座位和安全带”，不是“开完整趟车”。在 SAP 项目里，constructor 设计不好，常见后果是 ABAP Unit 难写、依赖难替换、RAP/服务逻辑生命周期不清、错误发生在对象创建阶段而不是业务动作阶段。

29.2 到底学什么

上一节 Static vs instance 静态与实例 已经区分了 object instance 与 class-level static state。本节继续问：对象刚创建出来时，哪些事情应该发生？哪些事情不应该发生？

本节要掌握七个边界：

  • METHODS constructor 声明 instance constructor；创建对象时由 runtime 自动调用。
  • instance constructor 的接口只能用于接收创建对象所需的输入和异常处理；不要期待它有 EXPORTING、CHANGING 或 RETURNING 结果。
  • constructor 可以接收必要输入或依赖，常见用途是初始化 instance attribute；用于 testability 时，依赖应优先声明成 interface reference，而不是具体 class。
  • 现代 ABAP 常用 NEW lcl_service( ... ) 创建对象；经典代码里也可见 CREATE OBJECT lo_service EXPORTING ...，但 ABAP Cloud / ABAP for Cloud Development 中应使用 NEW，不要把 CREATE OBJECT 当成新代码默认写法。
  • constructor 不是普通业务 method，不能把它当作 lo_obj->constructor( ) 这样的普通 instance method 手工调用。
  • CLASS-METHODS class_constructor 声明 class constructor；class 第一次被使用前由 runtime 自动调用，用来初始化 static state；它不能有参数，也不能声明异常。
  • constructor injection 是把外部依赖通过 constructor 传入对象，常用于让逻辑可测试、可替换、可解释。
  • constructor 和 class constructor 都不应承担业务副作用；主业务动作应放在明确命名的 public method 中。

最小例子：

CLASS lcl_price_service DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS constructor
      IMPORTING iv_currency TYPE string.

    METHODS get_currency
      RETURNING VALUE(rv_currency) TYPE string.

  PRIVATE SECTION.
    DATA mv_currency TYPE string.
ENDCLASS.

CLASS lcl_price_service IMPLEMENTATION.
  METHOD constructor.
    mv_currency = iv_currency.
  ENDMETHOD.

  METHOD get_currency.
    rv_currency = mv_currency.
  ENDMETHOD.
ENDCLASS.

调用者创建对象：

DATA(lo_service) = NEW lcl_price_service( iv_currency = 'JPY' ).

创建完成后，lo_service 代表一个已经知道 currency 的 price service。constructor 的职责是“初始化对象状态”，不是马上读数据库、生成会计凭证或提交事务。

29.3 为什么重要

constructor 是对象生命周期的入口。入口设计清楚，后面的 method 才容易读；入口设计混乱，对象会带着隐形状态和隐形副作用进入系统。

在项目里，constructor 常见问题包括：

  • constructor 里做数据库更新或远程调用，只是创建对象就产生业务副作用。
  • constructor 里读取大量 customizing 或缓存，导致 ABAP Unit 慢、测试结果受环境影响。
  • constructor 参数太多，说明 class 职责可能太大。
  • constructor 没有接收依赖，而是在内部直接调用 static helper、数据库、系统日期，导致 test double 很难注入。
  • class constructor 中初始化可变 static cache，测试顺序或运行上下文改变后结果不稳定。
  • 对象创建失败时抛出异常，但调用方没有意识到 NEW 本身可能失败。

Java/Python 背景学习者会觉得 constructor 很熟悉，但 ABAP 项目还要多想几层：class 是 repository object，global class 可能被很多对象通过 where-used 调用；对象创建可能发生在 dialog、background、RAP handler、unit test、service call 等不同 runtime context；constructor 的副作用会在这些上下文里一起出现。

29.4 Key mechanism / 核心机制

29.4.1 instance constructor initializes one object

instance constructor 用 METHODS constructor 声明，用 METHOD constructor. ... ENDMETHOD. 实现。它属于对象创建流程的一部分：

CLASS lcl_range DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS constructor
      IMPORTING iv_from TYPE i
                iv_to   TYPE i.

    METHODS contains
      IMPORTING iv_value         TYPE i
      RETURNING VALUE(rv_result) TYPE abap_bool.

  PRIVATE SECTION.
    DATA mv_from TYPE i.
    DATA mv_to   TYPE i.
ENDCLASS.

CLASS lcl_range IMPLEMENTATION.
  METHOD constructor.
    mv_from = iv_from.
    mv_to   = iv_to.
  ENDMETHOD.

  METHOD contains.
    rv_result = xsdbool( iv_value >= mv_from AND iv_value <= mv_to ).
  ENDMETHOD.
ENDCLASS.

调用：

DATA(lo_range) = NEW lcl_range(
  iv_from = 10
  iv_to   = 20 ).

IF lo_range->contains( 15 ) = abap_true.
  " inside the range
ENDIF.

lo_range 创建出来后，mv_from 和 mv_to 已经有值。contains 不需要每次都重新传 from/to，这就是 constructor 初始化 instance state 的价值。

29.4.2 constructor is a contract too

constructor 不是普通 implementation detail，它也是调用者必须理解的 contract。调用者看到 constructor 参数时，会知道“创建这个对象必须提供哪些东西”。

好的 constructor 参数通常具备这些特征：

  • 数量少，表达对象成立所必需的输入。
  • 类型清楚，最好用 DDIC type、interface reference 或明确 value object。
  • 不混入业务动作参数，例如 iv_do_posting、iv_commit_now、iv_send_mail。
  • 不把可以由普通 method 输入的临时值提前塞进对象状态。

如果 constructor 参数很多，经常说明 class 做了太多事。可以考虑拆 class、引入 Interface 接口定义、使用配置对象，或把一次性业务输入移到业务 method。

29.4.3 constructor injection

constructor injection 是把依赖通过 constructor 传进对象，而不是在对象内部硬编码创建依赖。它是让 ABAP 业务逻辑变得可测试的关键技巧之一。

例子：price service 需要一个 discount reader。不要在 service 内部直接 new 一个具体 reader，也不要直接 static call 某个全局工具；更好的做法是依赖 interface：

INTERFACE lif_discount_reader.
  METHODS get_rate
    IMPORTING iv_customer_id TYPE string
    RETURNING VALUE(rv_rate) TYPE decfloat34.
ENDINTERFACE.

CLASS lcl_price_service DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS constructor
      IMPORTING io_discount_reader TYPE REF TO lif_discount_reader.

    METHODS calculate
      IMPORTING iv_customer_id     TYPE string
                iv_amount          TYPE decfloat34
      RETURNING VALUE(rv_amount)   TYPE decfloat34.

  PRIVATE SECTION.
    DATA mo_discount_reader TYPE REF TO lif_discount_reader.
ENDCLASS.

CLASS lcl_price_service IMPLEMENTATION.
  METHOD constructor.
    mo_discount_reader = io_discount_reader.
  ENDMETHOD.

  METHOD calculate.
    DATA(lv_rate) = mo_discount_reader->get_rate( iv_customer_id ).
    rv_amount = iv_amount * ( 1 - lv_rate ).
  ENDMETHOD.
ENDCLASS.

这样，生产代码可以传真实 reader，ABAP Unit 可以传 test double。constructor 的职责只是接收 interface-typed dependency，并保存到 private instance attribute。真正的业务计算仍然在 calculate。

真实项目里还要检查 io_discount_reader 是否 bound；如果依赖缺失，应使用项目约定的 exception class 或 factory 规则处理。本节先看结构，不把异常设计提前展开。

还有一种常见变体是 fallback implementation：constructor 参数设为 optional，如果调用者没有注入依赖，就在 constructor 内部创建默认实现。这个模式能兼顾调用方便和测试替换，但要保证默认实现不隐藏重业务副作用。

29.4.4 NEW and CREATE OBJECT

现代 ABAP 中常见写法是 constructor expression：

DATA(lo_service) = NEW lcl_price_service(
  io_discount_reader = lo_reader ).

classic 旧代码或老项目里也常见：

DATA lo_service TYPE REF TO lcl_price_service.

CREATE OBJECT lo_service
  EXPORTING
    io_discount_reader = lo_reader.

两种写法都表达“创建对象并调用 constructor”。本教程优先使用 NEW，因为它更短、更接近表达式风格，也更符合现代 ABAP / ABAP Cloud 的方向；读旧代码时要能认出 CREATE OBJECT，但不要在新代码里优先模仿它。

29.4.5 class constructor initializes class-level state

class constructor 用 CLASS-METHODS class_constructor 声明，用 METHOD class_constructor. 实现。它不是创建某个对象时调用，而是在 class 第一次被使用前由 runtime 触发。更精确地说，它通常在当前 internal session 中每个 class 执行一次，触发点可能是第一次创建实例、第一次调用 static method、第一次访问 public static attribute 等。

class_constructor 是特殊 static constructor：不能有 importing/exporting/changing/returning 参数，也不能声明异常。初学者如果尝试给它加参数或 RAISING，应把它理解成语法层面的错误，而不是风格问题。

适合 class constructor 的场景通常很小，例如初始化不可变 lookup table：

CLASS lcl_status_text DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS get_text
      IMPORTING iv_status      TYPE string
      RETURNING VALUE(rv_text) TYPE string.

  PRIVATE SECTION.
    TYPES:
      BEGIN OF ty_status_text,
        status TYPE string,
        text   TYPE string,
      END OF ty_status_text,
      tt_status_text TYPE HASHED TABLE OF ty_status_text WITH UNIQUE KEY status.

    CLASS-DATA gt_status_text TYPE tt_status_text.

    CLASS-METHODS class_constructor.
ENDCLASS.

CLASS lcl_status_text IMPLEMENTATION.
  METHOD class_constructor.
    gt_status_text = VALUE #(
      ( status = 'O' text = 'Open' )
      ( status = 'C' text = 'Closed' ) ).
  ENDMETHOD.

  METHOD get_text.
    READ TABLE gt_status_text WITH TABLE KEY status = iv_status INTO DATA(ls_text).
    rv_text = COND #( WHEN sy-subrc = 0 THEN ls_text-text ELSE 'Unknown' ).
  ENDMETHOD.
ENDCLASS.

这个 class constructor 只做轻量、确定、可解释的 static 初始化。它不读业务数据库、不发 HTTP 请求、不提交事务、不发消息、不根据当前用户做复杂分支。

29.4.6 creation control belongs to class design

在 Global class 全局类 中已经提过，ABAP class 可以通过 class options 控制实例化，例如 CREATE PUBLIC、CREATE PROTECTED、CREATE PRIVATE。这和 constructor 一起决定“谁能创建对象”。

常见策略：

  • CREATE PUBLIC：外部可以直接 NEW zcl_service( )。
  • CREATE PROTECTED：只允许受保护范围内创建，通常和继承或受控扩展一起出现。
  • CREATE PRIVATE：外部不能随意创建，通常配合 factory method。
  • factory method：把复杂对象创建规则集中在一个清楚入口，例如 zcl_service=>create( )。

典型 factory method 长这样：

CLASS-METHODS create
  RETURNING VALUE(ro_instance) TYPE REF TO zcl_price_service.

初学阶段先掌握 CREATE PUBLIC + 简单 constructor；看到 CREATE PRIVATE、CREATE PROTECTED 或 factory pattern 时，理解为“项目想控制对象创建入口”，不要把它当成神秘技巧。

29.4.7 constructor and exceptions

constructor 如果发现对象无法被正确创建，可以通过项目约定的 class-based exception 报错。调用者要意识到：NEW 本身也可能进入 TRY...CATCH。

示意：

TRY.
    DATA(lo_service) = NEW lcl_price_service(
      io_discount_reader = lo_reader ).
  CATCH zcx_price_config_error INTO DATA(lx_error).
    " handle construction failure
ENDTRY.

但不要把 constructor 设计成“大量业务校验 + 复杂异常分支”的地方。对象创建失败应该是“依赖或初始配置不满足”，不是“业务流程执行失败”。业务流程失败更适合发生在 calculate、post、save、validate 等明确业务 method 中。

如果 constructor 抛出异常，对象创建不会完成，调用者也不应假设已经拿到了可用实例。这个语义很重要：constructor exception 是“对象没有被正确创建”的信号，不是“对象已经创建但业务结果失败”的返回方式。

29.4.8 constructor review checklist

在 code review 里，可以用下面这张表判断某段逻辑是否应该放在 constructor 中。核心原则是：constructor 建立“对象能成立”的前提，业务 method 执行“对象要做的动作”。

QuestionConstructor is acceptable when...Move to normal method when...
Is this required for object validity?没有它对象就无法保持一致状态只是某次业务动作才需要的输入
Does it have business side effects?只保存 dependency / initial value会写 DB、发消息、commit、调用外部系统
Can it fail?依赖缺失、配置缺失、输入不合法导致对象不能创建业务规则失败、审批失败、过账失败
Is it fast and deterministic?只做轻量初始化或不可变 lookup读取大量 customizing、远程调用、等待锁
Can tests replace dependencies?依赖通过 interface reference 注入constructor 内部硬编码 NEW 具体 class 或 static call
Is lifecycle clear?对象创建后可预测、可重复、无隐藏动作仅引用 class 或创建对象就触发业务流程

如果 review 中看到 NEW zcl_service( ) 就发送邮件、更新状态、写 application log、调用 BAPI 或执行 COMMIT WORK，通常应该要求重构。更好的设计是：constructor 接收必要依赖和初始配置，public method 例如 send( )、post( )、save( )、execute( ) 执行业务动作，并让调用者清楚看到副作用发生在哪里。

29.5 日本项目现场说法

  • 常见日语：コンストラクタ、インスタンスコンストラクタ、クラスコンストラクタ、初期化処理、依存性注入、副作用、生成制御。
  • 设计说明句：このクラスでは、必要な依存オブジェクトをコンストラクタで受け取り、業務処理は calculate メソッド側に分離しています。
  • Review 指摘句：コンストラクタ内でDB更新や外部API呼び出しを行うと、オブジェクト生成だけで副作用が発生するため避けるべきです。
  • Class constructor 指摘句：クラスコンストラクタは軽量な静的初期化に限定し、ユーザ依存・トランザクション依存の処理を入れない方針にします。
  • ABAP Cloud 指摘句：新規開発では CREATE OBJECT ではなく NEW を使い、クラウド開発で許可される構文に揃えます。
  • 使用场景：class 设计说明、ABAP Unit testability review、factory method 设计、RAP handler/service 依赖管理、障害调查中确认对象创建时点。

日本项目里常说 初期化，但要追问“初始化什么”。初始化对象自己的稳定状态是 constructor 的职责；执行审批、保存、发送、commit，不应伪装成初始化。

29.6 English Terms

  • Official / search terms: constructor, class constructor, METHODS constructor, CLASS-METHODS class_constructor, NEW, CREATE OBJECT, constructor injection, factory method
  • Related syntax: METHODS constructor IMPORTING ..., METHOD class_constructor, NEW lcl_class( ... ), CREATE OBJECT, CREATE PUBLIC, CREATE PROTECTED, CREATE PRIVATE, CLASS-METHODS create
  • Review terms: initialization, dependency injection, side effect, business side effects, object creation, factory, test double
  • 读官方资料时优先保留英文关键字：constructor 和 class_constructor 是特殊 method 名，不要翻译成普通函数名。

29.7 Example / 学习例子

29.7.1 例 1：constructor 初始化对象状态
CLASS lcl_user_context DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS constructor
      IMPORTING iv_user_id TYPE string.
    METHODS get_user_id
      RETURNING VALUE(rv_user_id) TYPE string.

  PRIVATE SECTION.
    DATA mv_user_id TYPE string.
ENDCLASS.

CLASS lcl_user_context IMPLEMENTATION.
  METHOD constructor.
    mv_user_id = iv_user_id.
  ENDMETHOD.

  METHOD get_user_id.
    rv_user_id = mv_user_id.
  ENDMETHOD.
ENDCLASS.

iv_user_id 是对象成立所需的初始信息，放进 constructor 合理。后续 method 不需要每次重复传 user id。

29.7.2 例 2：constructor injection 让测试更容易
DATA(lo_service) = NEW lcl_price_service(
  io_discount_reader = lo_test_reader ).

在 ABAP Unit 中，lo_test_reader 可以是一个实现 lif_discount_reader 的 local test double。这样测试不需要真实数据库、远程服务或复杂 customizing。constructor 的价值不是“多写一个 method”，而是把依赖关系显式放到对象创建入口。

29.7.3 例 3：不要把业务动作塞进 constructor

不建议：

METHOD constructor.
  mv_order_id = iv_order_id.
  update_order_status( iv_order_id ).
  COMMIT WORK.
ENDMETHOD.

这样只是 NEW lcl_order_service( ... ) 就会更新订单并提交事务。调用者很难预期，测试也很难控制。

更合理：

METHOD constructor.
  mv_order_id = iv_order_id.
ENDMETHOD.

METHOD approve.
  update_order_status( mv_order_id ).
ENDMETHOD.

对象创建和业务动作分开，调用者可以明确知道什么时候真正执行审批。

29.7.4 例 4：class constructor 只做轻量初始化

适合：

METHOD class_constructor.
  gt_status_text = VALUE #(
    ( status = 'O' text = 'Open' )
    ( status = 'C' text = 'Closed' ) ).
ENDMETHOD.

不适合：

METHOD class_constructor.
  call_remote_service( ).
  update_custom_table( ).
  COMMIT WORK.
ENDMETHOD.

class constructor 由 runtime 在 class 首次使用时触发，调用者很难从业务代码上看出它何时执行。因此越轻越好，越确定越好。

29.8 Counterexample / Failure mode

29.8.1 误区 1：constructor 做主业务流程

constructor 里直接审批、保存、发送、commit，会让对象创建和业务动作混在一起。

修正动作：constructor 只初始化状态和依赖；主业务动作放进清楚命名的 public method。

29.8.2 误区 2：constructor 参数无限膨胀

如果 constructor 有十几个参数，class 多半承担了太多职责，或缺少配置对象 / context object / interface。

修正动作：拆分 class，提取 value object，或用 interface 表达外部依赖。

29.8.3 误区 3：class constructor 读取环境相关业务数据

class constructor 中按当前用户、日期、client、customizing、远程系统状态做复杂初始化，会让首次使用 class 的行为变得不可预测。

修正动作：class constructor 只做轻量、稳定、幂等的 static 初始化；环境相关逻辑放在显式 method 中。

29.8.4 误区 4：不用 constructor injection，依赖全靠内部 new/static call

业务 service 内部自己创建所有依赖，测试时就很难替换数据库、远程 API、日期、用户或 customizing reader。

修正动作：对不稳定依赖使用 interface + constructor injection；简单纯函数 helper 可以保持 static。

29.8.5 误区 5：把 constructor exception 当普通业务失败

constructor exception 表示对象无法正确创建。不要把“订单审批失败”“保存失败”“业务校验失败”都塞进对象创建阶段。

修正动作：创建失败和业务执行失败分开；前者在 constructor / factory，后者在业务 method。

29.9 思考问题

  • 创建这个对象时，哪些信息是对象成立所必需的？
  • 哪些参数只是某次业务动作的输入，不应该放进 constructor？
  • 这个 constructor 是否有数据库更新、远程调用、commit、message、authorization 等副作用？
  • 依赖对象是否通过 interface 传入，还是在内部硬编码创建？
  • 如果写 ABAP Unit，能不能传入 test double？
  • class constructor 是否只做轻量静态初始化？是否依赖当前用户、日期、client 或数据库状态？
  • 对象创建失败和业务执行失败是否区分清楚？
  • 在日本项目里 review 时，应说 初期化処理，还是应指出 業務副作用がコンストラクタに入っている？
