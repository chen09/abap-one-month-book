Chapter 03 - CDS、RAP、OData

5. CDS metadata extension and UI annotations

  • Chapter 03
  • Section 05
  • self-contained study note

5.1 一句话结论

CDS metadata extension and UI annotations，是把“服务和画面消费需要的 UI metadata”从核心 CDS 数据模型中分离出来的关键机制。它让 CDS view entity 视图实体 和 projection view 能继续表达业务数据语义，同时把 Fiori Elements、OData UI service、字段位置、facet、line item、identification、search、value help 等消费层元数据放到更合适的层。

本节最重要的结论是：metadata extension 不是创建字段、不是改数据库、不是写业务逻辑；它主要是为已有 CDS entity 补 annotation。UI annotations 也不是“画面代码”，而是让 Fiori Elements / OData UI consumer 读取并生成或调整页面的一类 metadata。读者要学会判断：哪些 annotation 应该靠近核心模型，哪些应该放在 projection / metadata extension，哪些根本不该用 annotation 解决。

5.2 到底学什么

前面的 CDS annotation 注解基础 讲了 annotation 是 metadata，不是普通注释；CDS built-in functions 内置函数 讲了字段值如何在模型中被派生。本节进一步回答一个项目问题：annotation 应该写在哪里？

需要掌握六个判断点。

  • Metadata extension：一种 ABAP repository object，用来在主 CDS DDL source 之外给 CDS entity 添加 annotation。
  • Allow extension：被扩展的 CDS entity 通常需要允许 metadata extension，例如在合适位置使用 @Metadata.allowExtensions: true。
  • Annotation layer：metadata extension 会声明 layer，例如常见的 @Metadata.layer: #CORE；layer 表达 metadata 所属层级，不是随便装饰。
  • UI annotations：@UI... 相关 annotation 可为 Fiori Elements / OData UI service 提供列表、表单、facet、字段显示等 UI metadata，但只有相关消费者理解时才有实际效果。
  • Projection layer：SAP Learning 对 RAP/OData UI service 的建模强调 projection layer 是 service-specific metadata 的关键位置，UI annotation、search、text、value help 等常在 projection/metadata extension 层处理。
  • Boundary：metadata extension 和 UI annotation 不替代 behavior definition、DCL、authorization、save sequence、validation、determination、service definition 或 service binding。

5.3 为什么重要

如果把所有 annotation 都写进底层 view entity，初学时看起来方便：打开一个文件，什么都在里面。但项目越大，问题越明显。底层模型本来应该表达稳定业务语义，例如字段含义、金额/币种、数量/单位、关联关系、访问控制入口等；UI 页面却会根据用户角色、服务场景、Fiori app、项目阶段和客户需求不断调整。如果把大量 UI-specific annotation 放在底层模型里，底层模型就会被某个画面绑死。

metadata extension 的价值，是把“同一个数据模型如何被某个 UI/service 消费”抽出来。比如 ZI_Travel 或 root view 负责表达旅行申请的业务数据；ZC_Travel projection view 面向某个 OData UI service；metadata extension 再描述这个 projection 在 Fiori Elements 里哪些字段出现在 line item、哪些字段放进 identification、哪些 facet 分组、哪些字段用于 search 或 value help。这样，核心模型、服务视角和 UI metadata 就有了更清楚的边界。

SAP Learning 在 OData UI service 建模中把 projection layer 明确放在 service-specific metadata 的位置，并建议把 projection view 的 UI annotations 外包到 metadata extension，以提高可读性并便于后续调整。这里的重点不是“所有 annotation 都必须外包”，而是要把服务/UI 特有的 metadata 和通用业务模型语义分开。

对项目交付来说，这个边界非常实际。UI annotation 改动可能影响 Fiori Elements 画面、OData metadata、测试截图、用户验收、前端联调和 defect 分析；但它通常不应该改变底层业务对象的行为。把 UI metadata 分层，review 时就能分清：这是模型语义变化、服务契约变化、画面展示变化，还是 RAP behavior 变化。

对 Java/Python 背景学习者，可以把 metadata extension 粗略理解成“给 schema/model 追加外部 metadata 的配置对象”，有点像把 OpenAPI/serializer/UI schema 与 domain model 分开。但 SAP 的差异是：metadata extension 是 ABAP repository object，会被 activation、transport、where-used、ADT template、service binding preview 和 Fiori Elements 消费链路共同约束。

5.4 Key mechanism / 核心机制

5.4.1 Metadata extension 是 repository object

metadata extension 不是注释文件，也不是前端配置文件。它是 ABAP repository 中的开发对象，和 CDS data definition、behavior definition、service definition、service binding 一样，要创建、保存、激活、传输、review。ADT 中通常可以从 CDS data definition 或 projection view 的上下文菜单创建 metadata extension。

这意味着它有两个后果。

第一，它进入 transport 和 activation 流程。metadata extension 没激活，UI metadata 可能不会出现在预览或服务消费中。第二，它属于影响分析对象。改字段名、删 annotation、改 facet 或 line item 时，要检查对应 projection view、service binding、Fiori Elements preview、测试用例和用户验收文档。

5.4.2 主 CDS source 要允许 extension

metadata extension 不是想贴就贴。被扩展的 CDS entity 需要允许 metadata extension。SAP Learning 的 OData UI service 练习中，会先在 projection view 上加入类似 @Metadata.allowExtensions: true 的 view annotation，然后创建 metadata extension。

这条规则对初学者很重要：如果只创建 metadata extension，但目标 CDS entity 没允许扩展，ADT/activation 会给出问题，或者 metadata 不会按预期生效。读别人代码时，也可以把 @Metadata.allowExtensions: true 当成一个信号：这个 view 的部分 annotation 可能不在主 DDL source 里，而是在 metadata extension 里。

5.4.3 @Metadata.layer 不是装饰品

metadata extension 通常会包含 @Metadata.layer，例如：

@Metadata.layer: #CORE
annotate view ZC_Travel with
{
  ...
}

#CORE 表示当前 metadata 属于核心层。系统和工具可以基于 layer 理解不同来源的 metadata。初学者不需要一开始背完所有 layer，但要避免两个误解。

  • 误解 1：layer 是项目自定义标签。实际 layer 是 SAP 定义的 annotation 机制的一部分，不能随意创造。
  • 误解 2：layer 能替代包、命名空间或 transport 管理。layer 只是 metadata 层级，不是 transport 边界。

项目上如果遇到 customer / partner / localization 等扩展层，要先查项目规范和 SAP 文档，不要把 #CORE 机械复制到所有场景。

5.4.4 annotate view/entity ... with { ... }

metadata extension 的核心语义是“给已有 CDS entity 增加 annotation”。不同 ABAP release、对象类型和 ADT template 中可能看到 annotate view ... with 或 annotate entity ... with 形式；本教程不把某一个模板写死成唯一答案。真正项目里以当前系统的 ADT template、ABAP Keyword Documentation 和 syntax check 为准。

一个教学示意：

@Metadata.layer: #CORE
annotate view ZC_Travel with
{
  @UI.lineItem: [
    { position: 10, label: 'Travel' }
  ]
  @UI.identification: [
    { position: 10 }
  ]
  TravelID;

  @UI.lineItem: [
    { position: 20, label: 'Customer' }
  ]
  CustomerID;
}

这段示意的重点不是复制语法，而是阅读结构：metadata extension 没有重新定义 TravelID 和 CustomerID 字段；它只是对已经存在的元素附加 UI annotation。字段是否存在、类型是什么、是否 key、是否有 association，仍然来自原来的 CDS entity / projection view。

5.4.5 UI annotations 的消费者是 Fiori Elements / OData UI service 等

UI annotation 的典型用途，是告诉 Fiori Elements 或 OData UI consumer 如何理解一个实体的展示方式。例如：

  • @UI.lineItem：字段是否出现在列表行项目中，以及位置、label 等。
  • @UI.identification：字段是否出现在对象页识别区或详情展示中。
  • @UI.facet：对象页上如何分组、分区、导航到相关区块。
  • Selection/search/value-help 相关 annotation：帮助 UI 提供搜索、筛选、值帮助等体验。

但必须记住：annotation 只有在消费者读取它时才会产生效果。ABAP SQL 读取这个 CDS entity 时，不会因为 @UI.lineItem 就自动改变查询结果；RAP behavior 也不会因为一个 UI annotation 就自动校验字段；服务是否可访问，还要看 Service definition 服务定义 和 Service binding 服务绑定。

5.4.6 Projection layer 是 service-specific metadata 的自然位置

SAP Learning 的 RAP OData UI service 课程里有一个很重要的建模思想：projection layer 是 service-specific layer。也就是说，底层 data model view 不应该背负所有服务/UI 的细节；面向某个服务的 projection view 才更适合承载服务特有的 metadata。

可以这样理解：

  • Interface / root / data model view：表达稳定业务数据模型。
  • Projection view：表达某个服务或消费场景看到的数据模型。
  • Metadata extension：把 UI-heavy annotation 从 projection DDL source 中分离出来，提升可读性，也方便后续调整。
  • Service definition：决定哪些 CDS entities 被暴露成服务。
  • Service binding：决定协议和消费场景，例如 OData UI service 或 Web API。

这条链路一旦清楚，初学者就不会把 @UI.lineItem、service definition、service binding、Fiori preview 混成一件事。

5.4.7 哪些 annotation 留在主 DDL source，哪些放进 metadata extension

一个实用判断法如下。

适合留在主 DDL source 附近的，通常是和核心模型语义紧密相关的 annotation，例如：

  • entity / field 的基本 label 或说明。
  • amount/currency、quantity/unit 等语义关系。
  • access-control check 入口或其他核心技术语义。
  • 其他所有消费者都应理解的稳定语义。

适合放进 projection / metadata extension 的，通常是更接近服务或 UI 的 annotation，例如：

  • Fiori Elements 页面布局、line item、facet、field group、identification。
  • 某个 UI service 的 search、selection、text、value help 细节。
  • 只对一个 app 或一个 service 有意义的展示顺序和 label 调整。

这个划分不是绝对规则。项目上要看 SAP release、RAP provider contract、项目规范和复用范围。但初学阶段先按这个方向思考，能避免把底层模型做成“UI 装饰仓库”。

还有一个容易忽略的边界：projection view 可以为某个服务选择、改名或调整暴露元素，也可能包含 virtual element 等服务层设计；metadata extension 只是给已有 entity 或元素补 annotation。不要把“projection view 可以塑造服务视角”和“metadata extension 可以新增字段/逻辑”混在一起。

5.4.8 Metadata extension 不替代 RAP / authorization / service exposure

metadata extension 只能补 metadata。它不能做这些事：

  • 不能新增持久化字段，也不能改变数据库表结构。
  • 不能替代 Behavior definition 行为定义、validation、determination、action、draft、side effects。
  • 不能替代 CDS access control 访问控制、DCL role 访问规则、RAP authorization 或 authorization object。
  • 不能替代 Service definition 服务定义 或 Service binding 服务绑定；没有 service exposure，UI metadata 只是模型上的 metadata。
  • 不能保证性能；Fiori preview 慢时，还要看 CDS stack、association、DCL、query、service binding 和前端请求。

所以 defect 分析时要先分类。如果问题是“字段不显示”，metadata extension / UI annotation 可能相关；如果问题是“按钮不可用”，可能是 feature control、authorization 或 behavior；如果问题是“保存失败”，通常要看 RAP behavior、validation、save sequence 和 message handling，而不是盯着 UI annotation。

5.4.9 Metadata placement review checklist

设计或修改 UI annotation 前，可以用下面的表决定它应该放在哪里、该由谁负责验证。

QuestionPut near core model when...Put in projection / MDE when...Look elsewhere when...
Is it stable semantic metadata?label、amount/currency、quantity/unit 等所有 consumer 都应理解。某个 app 想调整字段显示名称或顺序。文本需要复杂多语言/i18n 运行时逻辑。
Is it UI layout?很少属于底层模型。line item、facet、field group、identification、selection fields。需要自定义 UI 行为或前端逻辑。
Is it service-specific?多个服务共享同一语义。只服务某个 OData UI service 或 projection。服务根本没有通过 service definition/binding 暴露。
Is it behavior-related?annotation 只能描述 metadata。UI metadata 可以辅助显示。action enablement、side effect、draft、save failure 要看 RAP behavior。
Is it authorization-related?access-control 入口可在模型层表达。projection 可限制暴露范围。实际读权限看 DCL/RAP authorization/authorization object。
Is the defect about preview?字段 metadata 缺失可能在 MDE。Fiori Elements layout/visibility 多半查 MDE/projection。保存、锁、消息、性能慢要查 RAP/CDS stack/service trace。

一个实用 review 句子是：UI annotation belongs close to the UI contract, not necessarily close to the persistence model. 如果 annotation 只为了一个 Fiori Elements 页面服务，优先考虑 projection / metadata extension，而不是把底层 interface view 绑死到某个画面。

5.5 日本项目现场说法

常见日语：

  • メタデータ拡張：metadata extension。
  • UIアノテーション：UI annotation。
  • CDSアノテーション：CDS annotation。
  • 投影ビュー、射影ビュー：projection view。
  • サービス固有メタデータ：service-specific metadata。
  • 一覧項目：line item。
  • 詳細表示、識別情報：identification / object detail display。
  • ファセット：facet。
  • 検索条件、値ヘルプ：search / value help。
  • サービス定義、サービスバインディング：service definition / service binding。

项目说明句：

  • UI固有のアノテーションは、コアデータモデルではなく投影ビューまたはメタデータ拡張に分離してください。
  • このメタデータ拡張は画面表示用の定義であり、業務ロジックや保存処理を実装する場所ではありません。
  • @Metadata.allowExtensions が設定されているため、UIアノテーションは別のメタデータ拡張オブジェクトで管理されています。
  • Fiori Elements の表示不備は、UIアノテーションだけでなくサービス定義、サービスバインディング、権限、feature control も確認してください。

Review 中常见指摘：

  • コアモデルにUI固有の定義が混在しているため、メタデータ拡張への分離を検討してください。
  • このUIアノテーションは対象サービス固有のため、投影ビュー側に配置してください。
  • 値ヘルプや検索条件の要件が未確定のため、annotation の確定前に画面仕様を確認してください。

5.6 English Terms

  • Official / search terms: CDS metadata extension, ABAP CDS metadata extension, ANNOTATE ENTITY, ANNOTATE VIEW, @Metadata.allowExtensions, @Metadata.layer, CDS UI annotations, Fiori Elements annotations, OData UI service.
  • UI/service terms: @UI.lineItem, @UI.identification, @UI.facet, field group, selection field, search metadata, value help, text element.
  • Architecture terms: projection layer, service-specific metadata, service definition, service binding, Fiori Elements Preview, OData metadata.

读官方资料时要特别区分：

  • annotation：metadata 本身。
  • metadata extension：存放 annotation 的扩展对象。
  • projection view：服务视角的数据模型层。
  • service definition：暴露哪些 entity。
  • service binding：绑定到哪种协议和消费场景。

5.7 Example / 学习例子

一个最小阅读链路可以这样走：

  1. 打开一个 projection view，例如 ZC_Travel。
  2. 确认它是否有 @Metadata.allowExtensions: true。
  3. 在 ADT 中找到对应 metadata extension，例如 ZC_TRAVEL。
  4. 读 @Metadata.layer 和 annotate view/entity ... with { ... }。
  5. 找到 @UI.lineItem、@UI.identification、@UI.facet 或 search/value-help annotation。
  6. 打开 service definition，确认 projection view 是否被 expose。
  7. 打开 service binding，确认它是 UI service 还是 Web API，并尝试 Fiori Elements Preview。

练习时不要急着记住所有 annotation 名称。先回答四个问题：

  • 这个 annotation 挂在哪个对象或字段上？
  • 它的消费者是谁：Fiori Elements、OData metadata、RAP、ADT、analytics，还是只是人读？
  • 它是核心语义，还是某个服务/UI 的特定需求？
  • 如果它不生效，应该检查 metadata extension、projection view、service definition、service binding、authorization 还是 behavior？

5.8 Counterexample / Failure mode

常见失败模式如下。

  • 把 @UI... 全部写在底层 interface view：底层模型被某个 app 绑定，后续复用困难。
  • 创建 metadata extension，但忘记在目标 CDS entity 上允许 extension：activation 或消费结果不符合预期。
  • 改了 metadata extension 后只看 CDS Data Preview：Data Preview 不等于 Fiori Elements UI 预览，要看对应 service binding。
  • 看到字段不显示就只改 @UI.lineItem：实际问题可能是 service definition 没 expose、field hidden、authorization、projection 没投影、metadata cache 或 Fiori preview 未刷新。
  • 把 UI annotation 当业务规则：按钮能不能点、字段能不能改、保存时是否校验，通常还要看 RAP feature control、authorization、validation 和 behavior。
  • 在 metadata extension 里硬编码大量 UI 文本，却没有考虑多语言、field label、text element、项目 UI 标准。
  • 把 UI service 和 Web API 的 metadata 混在一起：UI service 可以携带 UI annotations，Web API 更强调稳定 API contract，不应因为画面布局需求污染外部 API 语义。

修正动作：

  • 先确认对象链路：data model view -> projection view -> metadata extension -> service definition -> service binding -> preview/client。
  • 再确认 annotation target：entity、field、association、facet 还是 service-specific projection。
  • 最后确认问题类型：显示、搜索、值帮助、服务暴露、权限、行为、保存，分别找对应层。

5.9 思考问题

  • 当前 annotation 是核心语义，还是只服务某个 Fiori app？
  • 如果两个 UI service 复用同一个底层模型，它们的 UI annotation 是否应该分开？
  • @Metadata.allowExtensions: true 出现时，项目约定把 metadata extension 放在哪个 package / transport？
  • @UI.lineItem 不生效时，除了 metadata extension，还应该检查哪些对象？
  • UI annotation 是否可能影响外部 OData consumer 的 metadata 契约？
  • 日本项目 review 时，应说 メタデータ拡張、UIアノテーション、投影ビュー、サービス定義 还是 サービスバインディング？
