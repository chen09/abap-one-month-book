Chapter 04 - Project Delivery And Production Readiness

42. AC0567C RAP saver replay

  • Chapter 04
  • Section 42
  • self-contained study note

一句话结论

AC0567C RAP saver replay 是用本地 POC 复盘 RAP 保存阶段的责任边界：ExecuteImport action 负责校验、授权、重复检查和 staging；真正的 INSERT zcom_b_actt397 FROM TABLE @lt_insert 放在 behavior saver 的 save 方法中执行。这是 AC0567C 从“action handler 直接写表”修正为“RAP save phase 写表”的关键设计点。

一句话记忆：AC0567C 当前的保存链路是 static action ExecuteImport -> service class staging -> local save buffer -> behavior saver save -> ZCOM_B_ACTT397，不是“action 里直接 INSERT 再自己 COMMIT”。这符合 RAP 要求的方向，但它仍然是 POC 级实现：lcl_ac0567c_save_buffer 是 static class-data 桥接，不是完整生产级 request ledger / lock / log / rollback framework。

这一节要学的不是抽象背诵 save_modified。在 AC0567C 的实际源码里，saver class 是 lsc_ZRPA01_I_AC0567C_IMPORT，继承 cl_abap_behavior_saver，重定义的是 finalize、check_before_save、save、cleanup、cleanup_finalize。真正有落库动作的是 save，不是 finalize，也不是 action handler。

到底学什么

第一，要学 BDEF 先行。AC0567C 的 ZRPA01_I_AC0567C_IMPORT.bdef 明确是：

unmanaged implementation in class zbp_rpa01_i_ac0567c_import unique;
strict ( 2 );

define behavior for ZRPA01_I_AC0567C_IMPORT alias PriorAllocationImportC
authorization master ( instance )
lock master
{
  static action ValidateBatch ...
  static action ExecuteImport ...
  static action GetStatus ...
}

这告诉读者：它是 unmanaged implementation，有三个 static action；BDEF 写了 authorization master ( instance ) 和 lock master，ZBP_RPA01_I_AC0567C_IMPORT.localtypes.abap 的 behavior handler 中也确实有空的 get_instance_authorizations 与 lock 方法。但这三个业务入口都是 static action，因此不要误以为 instance authorization / instance lock 已经保护了这些 action。对当前 AC0567C POC 来说，真正生效的门禁是 service class 内部的 AUTHORITY-CHECK F_BKPF_BUK；生产化时还需要补 global authorization / get_global_authorizations 或等效的全局 action 授权设计。

第二，要学 action handler 的职责。ExecuteImport handler 负责把 deep parameter Rows[] 转成 service class 的 internal table，调用 ZCL_RPA01_AC0567C_IMP_SVC=>execute_import。service class 返回 Status、InsertedRows、MessageField 等结果，同时通过 et_insert 返回准备落库的 ZCOM_B_ACTT397 行。

第三，要学 staging 的边界。handler 只有在 ls_service_result-status = IMPORTED 且 lt_insert 非空时，才调用：

lcl_ac0567c_save_buffer=>add( it_insert = lt_insert ).

这一步不是数据库写入，只是把待插入行放进本地 save buffer。当前实现用 CLASS-DATA gt_insert 暂存这些行；这对 POC 可读、可演示，但生产化时要谨慎评估 request 隔离、并发、异常清理和多 action 调用场景。

第四，要学 saver 的实际落库点。lsc_ZRPA01_I_AC0567C_IMPORT->save 中的核心逻辑是：

DATA(lt_insert) = lcl_ac0567c_save_buffer=>consume( ).
IF lt_insert IS INITIAL.
  RETURN.
ENDIF.

INSERT zcom_b_actt397 FROM TABLE @lt_insert.

这正是 Joule 早前要求修正的点：真实 DB insert 不应该留在 action call chain 里，而应进入 RAP save phase。conclusion-and-architecture.md 也把它列为 Saver phase fix。

第五，要学空方法也是证据。当前 finalize 和 check_before_save 是空实现；cleanup 和 cleanup_finalize 清空 buffer；save 没有显式 application log、没有 catch insert error、没有把 DB insert failure 映射成业务消息的证据。这些不是批评，而是 POC 边界。

第六，要学保存证据的读法。AC0567C 已证明真实 insert InsertedRows = 2，重复同一 ImportId 返回 CONFLICT，GetStatus 返回 IMPORTED 和 count 2。但这些证据主要证明“最终表里有数据”和“顺序重复被拒绝”，并不能单独证明并发安全、rollback 语义、application log 完整性或 production-ready atomicity。

为什么重要

RAP 项目里最危险的误解之一，是把 action handler 当作普通 service/repository：收到请求就直接 INSERT、UPDATE、COMMIT WORK。在传统 ABAP report 或 function module 里，这种写法很常见；但在 RAP business object 中，runtime 管理 interaction phase、save sequence、消息返回和提交边界。把写表放错位置，可能导致回滚语义失真、response 与 DB 状态不一致、重复请求更难控制。

AC0567C 正好提供了一个真实修正过程。早期实现路径曾经把 INSERT zcom_b_actt397 放在 action call chain 中，Joule review 把这标成 write-before-execute blocker。最终实现改成：service class 只准备 lt_insert，handler 把它放入 save buffer，saver save 执行实际 insert。这个变化本身就是一本书里非常好的项目经验：不是所有“能写进去”的代码都应该放在最早能写的位置。

对批量上传 API 来说，保存阶段更关键。一次请求可能包含多行；一行失败时是整批失败还是部分成功；重复请求是在 action 阶段被拒绝还是 saver 阶段冲突；真实 insert 失败时 response 怎么表达；日志是否和业务数据一起提交或独立保留。这些问题不解决，POC 可以演示，生产却难以运维。

在日本项目里，review 常常会问得很直接：DB登録はどのフェーズで実行されますか、保存前チェックはどこで行いますか、COMMIT WORK は実行していませんか、保存失敗時のロールバック仕様はどうなっていますか。Section 42 的目标，就是让读者能用 AC0567C 的代码回答这些问题。

Key mechanism / 核心机制

AC0567C saver replay 可以按六层读。

42.1 BDEF 层：这是 unmanaged static action 模型

ZRPA01_I_AC0567C_IMPORT.bdef 的重点不是字段，而是模型边界：

  • unmanaged implementation in class zbp_rpa01_i_ac0567c_import unique
  • strict ( 2 )
  • authorization master ( instance )
  • lock master
  • static action ValidateBatch
  • static action ExecuteImport
  • static action GetStatus

这说明 AC0567C 不是 managed CRUD 教材例子，也不是 draft BO。它是一个以 static action 为主的 unmanaged POC。保存行为主要靠 behavior pool 和 saver class 自己实现，因此必须更严格地检查 action phase 与 save phase 的责任分工。

这里有一个容易误读的点：authorization master ( instance ) 和 lock master 出现在 BDEF 里，并不等于 static actions 已经被 instance authorization / lock 保护。ZBP_RPA01_I_AC0567C_IMPORT.localtypes.abap 中当前 get_instance_authorizations 和 lock 是空方法；更关键的是，它们不是这些 static action 的主要门禁。Section 40 已经说明过，AC0567C 当前真正执行的业务授权在 service class 的 ExecuteImport 路径中；Section 42 只补充保存阶段视角，不把 instance authorization 写成已完成的保护机制。

42.2 Handler 层：把 deep parameter 转成 service input

lhc_PriorAllocationImportC->ExecuteImport 遍历 action keys，把 Rows[] 转成 zcl_rpa01_ac0567c_imp_svc=>tt_row，再调用 service class：

zcl_rpa01_ac0567c_imp_svc=>execute_import(
  EXPORTING
    iv_company_code      = ...
    iv_customer_code     = ...
    iv_import_id         = ...
    iv_source_file_name  = ...
    iv_allow_real_update = ...
    it_rows              = lt_rows
  IMPORTING
    et_insert            = lt_insert ).

handler 随后把 service result 映射回 action result：Status、Simulation、ReceivedRows、ValidRows、InsertedRows、ImportId、MessageType、MessageField、MessageText 等。这些是 API consumer 看到的证据。

42.3 Service 层：校验、授权、重复检查、准备待插入行

ZCL_RPA01_AC0567C_IMP_SVC=>execute_import 的实际源码顺序是：

validate_batch
  -> if invalid: return
AllowRealUpdate = X
  -> if not X: REJECTED
AUTHORITY-CHECK F_BKPF_BUK / ACTVT = 01
  -> if failed: REJECTED
derive_upload_timestamp( ImportId )
loop Rows[]
  -> build zcom_b_actt397 row
  -> SELECT SINGLE duplicate check
  -> if duplicate: CONFLICT
append rows to lt_insert
return et_insert and Status = IMPORTED

注意这里的 Status = IMPORTED 是 service result 的业务语义，表示“本次 action 已经准备好导入并把待插入行交给 saver”。实际 DB insert 仍在 saver save 中发生。写书时不能把它说成 service class 已经落库。若 saver 在 save sequence 抛出异常，RAP runtime 应以错误结束并回滚当前 LUW；本 POC 的缺口不是“调用方一定会看到成功但 DB 为空”，而是缺少清晰的 insert-failure message mapping、application log 和可审计的失败证据。

42.4 Save buffer 层：POC 桥接，不是生产级 ledger

本地类 lcl_ac0567c_save_buffer 很短：

CLASS-DATA gt_insert TYPE zcl_rpa01_ac0567c_imp_svc=>tt_actt397.

METHOD add.
  APPEND LINES OF it_insert TO gt_insert.
ENDMETHOD.

METHOD consume.
  rt_insert = gt_insert.
  CLEAR gt_insert.
ENDMETHOD.

METHOD clear.
  CLEAR gt_insert.
ENDMETHOD.

它解决的是 POC 桥接问题：action handler 拿到了 lt_insert，但真实 insert 要留到 saver，所以用本地 buffer 暂存。这个设计能说明“写入移到了 save phase”，但不应被包装成完整 production pattern。生产环境还要考虑：

  • 同一 internal session 中多个 action 调用是否会混入同一 static buffer。
  • 并发请求是否隔离。
  • saver insert 失败后 buffer 是否已经被 consume 清掉。
  • 是否需要 request header/status table 替代临时 class-data buffer。
  • 是否需要 application log / outbox / retry table 记录 save attempt。

42.5 Saver 层：唯一实际 INSERT 点

lsc_ZRPA01_I_AC0567C_IMPORT 继承 cl_abap_behavior_saver，重定义：

finalize
check_before_save
save
cleanup
cleanup_finalize

当前实现中：

  • finalize 是空方法。
  • check_before_save 是空方法。
  • save 从 save buffer consume 待插入行，并执行 INSERT zcom_b_actt397 FROM TABLE @lt_insert。
  • cleanup 和 cleanup_finalize 调用 lcl_ac0567c_save_buffer=>clear()。

所以 AC0567C 的保存事实很清楚：只有 saver save 真正写 ZCOM_B_ACTT397；没有 save_modified 方法；没有 additional save；没有 application log 写入；没有在 saver 内做第二次 authorization check；没有显式处理 insert exception 的证据。

42.6 事务与回滚边界

当前 POC 的正确方向是：不在 action handler 里直接 COMMIT WORK，也不把真实 insert 提前放在 service class。RAP runtime 仍然是事务边界的 owner。这样至少避免了“action 中已经写表，后续 save/commit 失败却无法统一处理”的典型错误。

但 production review 还不能停在这里。需要继续确认：

  • INSERT FROM TABLE 如果因重复 key、DB error 或 authorization/lock 之外的问题失败，调用方会看到什么。
  • Status = IMPORTED 已经放进 action result 后，save failure 是否会被 OData/RAP runtime 正确返回。
  • 大批量插入时是整批 atomic 还是部分成功。
  • 是否需要 ACCEPTING DUPLICATE KEYS，通常不应随便加；如果要加，必须有明确业务语义。
  • 是否需要在 check_before_save 做最终 invariant check。
  • 是否需要 application log 在失败时也保留，还是随主事务 rollback。

42.7 与 AC0567C 已证据/未证据的关系

已证明：

  • ExecuteImport 可以生成待插入行。
  • handler 只在 Status = IMPORTED 且 lt_insert 非空时加入 save buffer。
  • saver save 中有实际 INSERT zcom_b_actt397 FROM TABLE @lt_insert。
  • 真实测试中 InsertedRows = 2。
  • 同一 ImportId 顺序重复返回 CONFLICT。

未证明：

  • 并发重复请求的锁或唯一约束处理。
  • saver insert failure 的用户可读消息。
  • application log / BAL / cl_bali_log 持久证据。
  • 大批量 partial success vs all-or-nothing 策略。
  • production-grade request ledger/status table。
  • save buffer 在复杂多 action / 多请求场景下的隔离性。

review 时可以这样问：

  • DB 登録は action handler ではなく save phase で実行されていますか。

  • 一時バッファはリクエスト単位で安全に分離されていますか。
  • 保存失敗時の application log は残す設計ですか、それとも主トランザクションと一緒に rollback しますか。

一个 AC0567C 保存 replay 可以这样写：

Scenario: Execute AC0567C import

Input:
  CompanyCode = 1000
  CustomerCode = 0000001000
  ImportId = POC-AC0567C-20260612115720
  AllowRealUpdate = X
  Rows = 2

Interaction/action phase:
  ValidateBatch passes
  AllowRealUpdate gate passes
  AUTHORITY-CHECK F_BKPF_BUK/01 passes
  duplicate check finds no existing rows
  service returns Status = IMPORTED and et_insert = 2 rows
  handler adds rows to lcl_ac0567c_save_buffer

Save phase:
  saver save consumes buffer
  INSERT zcom_b_actt397 FROM TABLE @lt_insert

Evidence:
  ExecuteImport Status = IMPORTED
  InsertedRows = 2
  GetStatus Status = IMPORTED
  GetStatus InsertedRows = 2

再看一个反向 replay：

Scenario: Duplicate same ImportId

Interaction/action phase:
  ValidateBatch passes
  AllowRealUpdate gate passes
  AUTHORITY-CHECK passes
  duplicate check finds existing ZCOM_B_ACTT397 row
  service returns Status = CONFLICT
  handler does not add rows to save buffer

Save phase:
  saver save has no new rows to insert

Evidence:
  Repeat same ImportId Status = CONFLICT

这一反向 replay 很重要：重复请求不是在 saver insert 时才发现的，而是在 service class 的 duplicate check 中发现；因此并发场景还要额外评估 SELECT 与 INSERT 之间的 race window。

Counterexample / Failure mode

常见失败模式如下。

第一，只说“用了 RAP saver”，不检查实际 INSERT 位置。必须在 behavior pool 中看到 INSERT zcom_b_actt397 FROM TABLE @lt_insert 位于 saver save。

第二，把 static save buffer 当成生产级事务 buffer。CLASS-DATA gt_insert 是 POC 桥接，不能自动证明请求隔离、并发安全或异常恢复。

第三，在 action handler 或 service class 里重新加入 INSERT。这会退回 write-before-execute 风险，破坏本 POC 的关键修正。

第四，认为 Status = IMPORTED 就等于 DB 已经提交。当前 service result 先返回 imported 语义，实际 insert 在 saver；如果 saver 抛错，运行时应以错误结束并回滚 LUW，但当前 POC 缺少清晰的 insert-failure message mapping 与 application log，诊断证据会不足。

第五，check_before_save 空着却在设计书里声称有最终保存前检查。当前源码没有这层检查，不能写成已实现。

第六，缺少 application log。测试脚本能证明 POC，但生产障害調査需要 request id、user、timestamp、insert count、error cause 的持久证据。

第七，没有并发测试。顺序重复 CONFLICT 不等于并发请求一定安全；SELECT-before-INSERT 仍需锁或唯一约束证据。

第八，cleanup 清掉 buffer 后没有审计。清理临时状态是必要的，但审计证据必须另有持久化位置。

第九，把带旧事务副作用的 legacy posting FM 直接塞进 RAP save sequence。比如某些历史路径可能内部到达 CALL TRANSACTION、隐式提交或 GUI 依赖逻辑；这不是 AC0567C saver 已证明的问题，而是从 RAP 保存阶段设计中必须提前排除的通用风险。这类旧事务副作用应拆到更合适的 ICF/job/classic adapter 或异步 worker 设计中，不能简单塞进 RAP saver。

思考问题

  • AC0567C 的真实 INSERT 位于哪个 class、哪个 method？
  • lcl_ac0567c_save_buffer 的 add、consume、clear 分别在哪个阶段被调用？
  • 为什么 Status = IMPORTED 不能简单理解为“DB 已经 commit 成功”？
  • 当前 check_before_save 是空方法，这对 production hardening 意味着什么？
  • 如果 INSERT FROM TABLE 在 saver 里失败，当前源码中哪里能看到错误映射或 application log？
  • 为什么顺序 duplicate test 不能证明并发 duplicate safety？
