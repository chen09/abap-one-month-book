26. Conditional logic 条件分支

  • Chapter 01
  • Section 26
  • self-contained study note

本节结论

Conditional logic 条件分支 是把业务规则写成可执行路径的基础：满足条件就走某段逻辑，不满足就走另一段逻辑。Day 04 先掌握 IF ... ELSEIF ... ELSE ... ENDIF 和 CASE ... WHEN ... ENDCASE，并理解 ABAP 的 logical expression、IS INITIAL、AND / OR / NOT、sy-subrc 判断和现代 COND / SWITCH 表达式的边界。

正文

条件分支回答的是一个朴素问题：当前数据、状态或返回结果不同，程序应该做什么不同的事。比如订单金额为 0 时跳过计算，币种不是 JPY 时输出提醒，READ TABLE 找不到行时组装错误消息，权限检查失败时停止后续处理。这些都不是 SAP 特有的想法；Java/Python 也有 if / elif / else。但 ABAP 的写法、类型转换、初始值判断、系统字段和项目现场审查点有自己的规则。

最常用的是 IF ... ENDIF：

IF ls_item-amount IS INITIAL.
  out->write( |Amount is initial| ).
ELSEIF ls_item-currency <> 'JPY'.
  out->write( |Foreign currency: { ls_item-currency }| ).
ELSE.
  out->write( |JPY amount: { ls_item-amount }| ).
ENDIF.

IF 后面写 logical expression。它可以是简单比较，例如 amount > 0；也可以组合多个条件，例如 currency = 'JPY' AND amount > 0；还可以使用 ABAP 常见谓词，例如 IS INITIAL、IS NOT INITIAL、BETWEEN ... AND ...、line_exists( ... )。SAP Learning 特别提醒，ABAP 语句用句号 . 结束，所以 IF ... .、ELSEIF ... .、ELSE.、ENDIF. 都要有 delimiter。注意 ELSEIF 是一个关键字；如果写成 ELSE. 再嵌套 IF ... ENDIF.，内层就需要自己的 ENDIF.。对 Java/Python 背景的人来说，这一点很机械，却很容易在刚开始写代码时漏掉。

ELSEIF 是从上往下依次判断；一旦某个条件成立，执行对应 block，然后跳到 ENDIF 后面。ELSE 可选。如果没有 ELSE，并且所有条件都不成立，就什么也不执行。业务代码里要有一个习惯：如果分支代表“状态分类”“处理结果”“审批路径”这类业务规则，通常要想清楚是否需要 ELSE 或兜底错误消息。没有 ELSE 本身不一定错，但如果它让“未知状态”静默通过，就会变成项目缺陷。

CASE ... ENDCASE 适合“围绕同一个值做多路分支”：

CASE ls_item-status.
  WHEN 'N'.
    out->write( |New item| ).
  WHEN 'A'.
    out->write( |Approved item| ).
  WHEN 'R'.
    out->write( |Rejected item| ).
  WHEN OTHERS.
    out->write( |Unknown status: { ls_item-status }| ).
ENDCASE.

可以把 CASE 理解成一种更易读的特殊 IF：它把一个 operand 依次和多个 WHEN 值做相等比较。这个限制必须说清楚：WHEN 后面是用于等值匹配的值，不能写成 WHEN > 0、WHEN BETWEEN ...、WHEN IS INITIAL 这种条件表达式。ABAP 允许在一个 WHEN 里列多个值，例如 WHEN 'N' OR 'A'.，但这仍然是多个等值匹配，不是任意 logical expression。若当前逻辑是“金额是否大于 0、币种是否为空、状态是否在某组值里”这种多条件组合，用 IF 更自然；若逻辑是“状态字段等于 N/A/R 时分别处理”，用 CASE 更清楚。WHEN OTHERS 是兜底分支，类似 ELSE。日本项目 review 里经常会问“这个状态值是否網羅できている”，也就是是否覆盖了所有可能状态。

logical expression 的细节也要提前建立边界。ABAP 用 = 做值比较，也用 = 做赋值，区别由语句位置决定：lv_amount = 100. 是赋值；IF lv_amount = 100. 是比较。比较运算包括 =、<>、>、<、>=、<=。多个条件可以用 AND、OR、NOT 组合；没有括号时，NOT 优先于 AND，AND 优先于 OR。初学者不要依赖记忆优先级，复杂条件尽量加括号：

IF ( ls_item-currency = 'JPY' AND ls_item-amount > 0 )
   OR ls_item-priority = 'H'.
  out->write( |Process item| ).
ENDIF.

IS INITIAL 是 ABAP 里非常重要的判断。它不是简单的“空字符串”，而是判断某个 ABAP data object 数据对象 是否等于它所属类型的 initial value。i 类型的 initial value 是 0，string 是空字符串，d 类型日期的 initial value 是 00000000。所以 IF lv_amount IS INITIAL. 对金额字段来说可能意味着“技术上是 0”，不一定等于业务上“未输入金额”。Day 03 已经学过 Elementary data types 基础技术类型，这里要把类型意识带回来：分支条件永远不是孤立语法，它依赖字段类型和业务语义。

还要认识 ABAP 的 truth value / Boolean 限制。ABAP 的 logical expression 会产生 true / false 判断结果，但传统 ABAP 没有像 Java boolean 或 Python bool 那样的内建、普适 Boolean 数据类型。项目代码里会看到几组容易混淆的名字：BOOLE_D 常作为 DDIC data element 表达两值 Boolean-like 字段；ABAP_TRUE 和 ABAP_FALSE 是推荐用于赋值和比较的常量；ABAP_BOOL 来自 type group ABAP，本质上是长度为 1 的字符型，在既存代码里很常见，但不应和 DDIC 的 BOOLE_D、S/4HANA 中的 ABAP_BOOLEAN 混成一回事。初学阶段的安全写法是：遵守项目开发标准，比较时优先使用 ABAP_TRUE / ABAP_FALSE，阅读旧代码时认得 'X' / space 和 ABAP_BOOL，但不要把任意 X / space、0 / 1、空字符串混在一起当“真假”。

在 ABAP 项目里，条件分支经常和 ABAP system fields sy fields 连在一起。比如很多语句执行后会设置 sy-subrc，项目里会看到：

READ TABLE lt_items INTO DATA(ls_found) WITH KEY item_no = lv_item_no.
IF sy-subrc = 0.
  out->write( |Found item { ls_found-item_no }| ).
ELSE.
  out->write( |Item not found: { lv_item_no }| ).
ENDIF.

这段代码的重点不是“背 sy-subrc”，而是理解：条件分支常常承接前一条语句的执行结果。一般约定是 sy-subrc = 0 表示前一操作成功或命中，非 0 表示没有成功、没有命中或出现某种业务/技术状态；具体非 0 数字的含义由前一条语句或接口定义决定。READ TABLE、AUTHORITY-CHECK、某些 function module、旧式接口调用都可能让后续代码根据返回状态分流。现代 ABAP 也有 table expression、exception、predicate function 等替代写法，但在日本 SAP 既存系统里，IF sy-subrc = 0. 和 IF sy-subrc <> 0. 仍然是必须能读懂的现场语言。

现代 ABAP 还有 COND 和 SWITCH 这样的 conditional expression。它们不是替代所有 IF / CASE 的银弹，而是让“根据条件决定一个值”可以写在 operand position：

DATA(status_text) = SWITCH string( ls_item-status
  WHEN 'N' THEN 'New'
  WHEN 'A' THEN 'Approved'
  WHEN 'R' THEN 'Rejected'
  ELSE 'Unknown' ).

DATA(message_type) = COND string(
  WHEN ls_item-amount IS INITIAL THEN 'E'
  WHEN ls_item-currency <> 'JPY' THEN 'W'
  ELSE 'S' ).

如果整段条件逻辑只是为了决定一个返回值，COND / SWITCH 会很干净；如果分支里有多步处理、数据库访问、消息组装、状态变更，传统 IF / CASE 更容易读、更容易 debug。初学者先把 statement form 和 expression form 分清：IF / CASE 是控制执行 block；COND / SWITCH 是产生一个值。过长的 COND 链和复杂 SWITCH 表达式在 review 中也可能被要求改回显式分支。

最后，条件分支的质量不只看“语法对不对”。项目 review 会看条件是否和业务规则一致、状态是否覆盖、错误路径是否明确、嵌套是否过深、是否把类型转换风险藏在比较里。比如 IF lv_amount = '0001'. 可能触发隐式转换；IF lv_date IS INITIAL. 可能把技术初始日期误当成业务空值；IF sy-subrc <> 0. 后面没有错误消息，测试人员就很难知道为什么数据没有被处理。许多项目也会用 ATC 或本地开发标准限制过深嵌套；看到三层以上 IF 时就应该主动考虑 guard clause、拆方法或表驱动映射，嵌套深度超过 5 往往会成为质量门风险。好的条件分支应该让读者看出“为什么走这条路”，而不是只让机器能执行。

为什么重要

  • 条件分支是业务规则落地的第一层：状态、金额、币种、权限、错误路径、例外处理都要靠它分流。
  • IF / CASE 写得清楚，单体测试和 defect 分析才知道每条路径怎么被触发。
  • Java/Python 背景的人容易低估 ABAP 的 = 双重角色、IS INITIAL、sy-subrc、隐式类型转换和 Boolean-like flag 约定。
  • 日本项目常把条件分支写进详细设计书、单体测试规格和 code review 指摘；分支漏写会直接变成测试ケース漏れ。
  • 后续 RAP validation、authorization、OData error response、runtime error 处理都建立在“条件判断后走正确路径”这个能力上。

核心机制

  • IF ... ELSEIF ... ELSE ... ENDIF：按 logical expression 从上到下判断；命中一个 block 后跳到 ENDIF 后继续。
  • CASE ... WHEN ... WHEN OTHERS ... ENDCASE：围绕同一个 operand 做等值比较，适合状态码、类别码、固定枚举值；范围、大小比较、IS INITIAL 等复杂条件应使用 IF ... ELSEIF。
  • logical expression：由比较、谓词、函数、AND / OR / NOT 和括号组合而成，结果用于控制执行路径。
  • IS INITIAL：判断类型相关 initial value，不等于所有业务意义上的“未输入”。
  • sy-subrc 判断：0 通常代表成功或命中，非 0 代表未成功或特定状态；具体含义看前一语句或接口定义。
  • Boolean-like flag：新代码遵守项目标准，DDIC 场景优先识别 BOOLE_D，比较/赋值用 ABAP_TRUE / ABAP_FALSE；旧代码里的 ABAP_BOOL、'X' / space 要能读懂但不要随意混用。
  • COND / SWITCH：现代 conditional expression，用于根据条件产生一个值；不要把复杂副作用或过长分支塞进表达式。
  • 质量判断：条件是否互斥、是否覆盖未知状态、错误路径是否可解释、嵌套是否可读、比较是否存在隐式转换风险。

日本项目现场表达

  • 常见日语：条件分岐、分岐条件、IF文、ELSEIF文、CASE文、WHEN OTHERS、判定条件、初期値判定。
  • 设计相关：業務ルール、ステータス判定、網羅性、排他条件、ガード句、条件漏れ、異常系、未入力チェック。
  • Review 相关：ネストが深い、可読性が低い、判定順序、想定外ステータス、テストケース漏れ。
  • 说明句：この処理はステータスごとの条件分岐であり、WHEN OTHERS で想定外ステータスも明示的に扱います。
  • 术语区分：条件分岐 は分岐構造そのもの、分岐条件 はどの分岐に入るかを決める判定条件です。
  • 初期值句：IS INITIAL は初期値判定であり、業務上の未入力チェックと同一とは限りません。
  • 覆盖性句：WHEN OTHERS がないため、想定外ステータスに対する網羅性が担保されていません。
  • Review 句：条件分岐のネストが深いため、ガード句または CASE文 に分割して可読性を上げてください。
  • 单体测试句：正常系だけでなく、初期値、想定外ステータス、権限エラーの異常系テストケースも追加してください。

学习例子

把 Day 04 的 Loop constructs 循环语句 和条件分支连起来，可以写一个订单行检查：

LOOP AT lt_items INTO DATA(ls_item).
  IF ls_item-amount IS INITIAL.
    out->write( |Item { ls_item-item_no }: amount is initial| ).
    CONTINUE.
  ENDIF.

  IF ls_item-currency <> 'JPY'.
    out->write( |Item { ls_item-item_no }: foreign currency { ls_item-currency }| ).
  ENDIF.

  CASE ls_item-status.
    WHEN 'N'.
      out->write( |Item { ls_item-item_no }: new| ).
    WHEN 'A'.
      out->write( |Item { ls_item-item_no }: approved| ).
    WHEN 'R'.
      out->write( |Item { ls_item-item_no }: rejected| ).
    WHEN OTHERS.
      out->write( |Item { ls_item-item_no }: unknown status { ls_item-status }| ).
  ENDCASE.
ENDLOOP.

这里第一段 IF 是 guard clause：金额是 initial value 时先输出信息并 CONTINUE，避免后面逻辑继续处理无效行。第二段 IF 是额外提醒：外币不是错误，但需要提示。第三段 CASE 是状态码分流；WHEN OTHERS 让未知状态不会静默漏掉。这个例子虽然简单，但已经覆盖了项目里常见的三种判断：异常先返回、警告路径、状态分类。

如果只是决定一个显示文本，可以改成 SWITCH：

DATA(status_text) = SWITCH string( ls_item-status
  WHEN 'N' THEN 'New'
  WHEN 'A' THEN 'Approved'
  WHEN 'R' THEN 'Rejected'
  ELSE 'Unknown' ).

对初学者来说，判断标准很简单：如果分支里要执行多条语句，用 IF / CASE；如果分支只是产生一个值，可以考虑 COND / SWITCH。

常见误区

  • 把 IS INITIAL 当成“空字符串”：金额 0、日期初始值、字符空白的业务含义可能完全不同。
  • IF a = b OR c = d AND e = f. 不加括号：读者很难确认你的真实意图，review 时容易被要求改写。
  • 用 CASE 处理复杂条件：当条件不是围绕同一个值做相等比较时，IF 往往更清楚。
  • 没有 ELSE / WHEN OTHERS：未知状态静默通过，单体测试也可能漏掉异常路径。
  • 把 abap_bool 当作语言内建 Boolean 类型：任意 X / space、0 / 1 混用会让条件含义变模糊。
  • 过度嵌套 IF：三四层以后，读者很难看清主路径；可以用 guard clause、拆方法、CASE 或表驱动映射降低复杂度。
  • 比较不同类型却不显式转换：隐式转换可能让小样本通过，但在真实数据里触发 runtime error 或错误分支。

思考题

  • IF 和 CASE 的使用场景差异是什么？什么时候 CASE 比连续 ELSEIF 更可读？
  • IS INITIAL 为什么不是简单的“空”？请分别用金额、日期、字符串举例。
  • 如果业务状态只定义了 N、A、R，为什么仍然建议处理 WHEN OTHERS？
  • sy-subrc = 0 这种判断在项目里解决什么问题？为什么后续要专门学习系统字段？
  • 复杂条件中为什么要主动加括号，而不是依赖 AND / OR 优先级？
  • COND / SWITCH 适合替代哪一类 IF / CASE？不适合替代哪一类？
