11. Transaction boundary 事务边界

  • Chapter 04
  • Section 11
  • self-contained study note

11.1 一句话结论

Transaction boundary 是一次业务动作从“开始收集变更”到“统一提交或回滚”的一致性边界。ABAP 项目里不能只把它理解成数据库事务；还要同时区分 SAP LUW 逻辑事务单元、Database LUW 数据库事务、update task、enqueue lock、BAPI commit pattern、RAP transactional buffer 和 Save sequence 保存序列。

一句话判断：谁拥有 transaction boundary，谁才有资格决定 COMMIT WORK、ROLLBACK WORK、BAPI_TRANSACTION_COMMIT 或 COMMIT ENTITIES。一个 helper method、BAdI、RAP handler、旧 function module wrapper 如果擅自提交，就可能破坏调用方的业务一致性。

11.2 到底学什么

本节学习“事务边界是谁的”。

第一，理解 business transaction 不等于单条 SQL。一次业务保存可能包含 header、item、status、log、number range、change document、workflow/event、外部接口和 application log。业务上必须一起成功或一起失败的范围，才是设计讨论的重点。

第二，区分 SAP LUW 和 Database LUW。Database LUW 是数据库层 commit/rollback 边界；SAP LUW 是 SAP 应用层组织业务一致性的逻辑边界。COMMIT WORK 不只是 database commit，还可能触发 update task、结束 SAP LUW、释放当前边界相关的锁或推进后续处理；如果锁或释放动作与 update task / save processing 绑定，则要等对应更新或保存处理完成，不能把 commit 写成“立刻无条件释放一切锁”。

第三，理解 RAP transaction owner。在 RAP 中，handler、validation、determination、saver 等生命周期不是随便 commit 的地方。RAP runtime 通过 transactional buffer 和 save sequence 管理保存，外部 consumer 使用 EML/OData 保存语义触发提交。

第四，识别内部 commit 风险。旧 BAPI、function module、日志组件、外部接口 wrapper 或 helper class 如果内部 COMMIT WORK，会让调用方以为还能 rollback，但数据已经部分提交。

第五，识别 point of no return。某些动作一旦发生就不能简单回滚，例如号码段消耗、邮件发送、外部系统过账、文件输出、business event 发布、远程系统确认。事务设计不是假设一切都能 rollback，而是把不可撤销动作放在明确位置，并为失败后的补偿、重试和人工处理留下证据。

第六，识别不同入口的 transaction owner。前台 report、background job、RAP OData action、BAPI caller、RFC consumer、外部调度器，谁拥有提交点可能不同。服务类本身通常不应擅自 commit，而应把成功/失败和待保存内容交回拥有事务边界的调用方。

11.3 为什么重要

事务边界错误比普通语法错误更危险，因为它常常只在异常、并发、重试或部分失败时暴露。单人测试里保存成功，看不出问题；上线后才出现“主数据保存失败但日志显示成功”“编号被消耗但单据不存在”“接口超时后重试生成两笔”“RAP action 报失败但 Z 表已经插入”的事故。

项目交付里，事务边界是设计评审常问的问题：哪个程序拥有提交点？旧 API 是否内部 commit？锁何时释放？如果中途失败，哪些东西能 rollback，哪些东西已经不可撤销？外部系统调用放在 commit 前还是 commit 后？这些问题回答不清，系统就很难说是 production-ready。

从 Java/Python 背景看，可以把 transaction boundary 类比成 service layer unit of work。但 SAP 的差异是：classic report、dialog step、update task、BAPI、RAP、OData、background job、RFC 都可能有不同的提交语义。不能简单套用“方法开始开事务、方法结束提交”的想象。

对生产项目来说，事务边界还决定“失败后怎么解释”。同一个错误，如果发生在提交前，可能全部回滚；如果发生在外部接口成功之后，可能需要补偿；如果发生在日志写入之后但业务数据回滚，日志要能说明这是失败日志而不是成功证据。事务边界写清楚，障害调查才知道看数据库、application log、job log、外部系统还是业务状态。

11.4 Key mechanism / 核心机制

可以用三层边界理解：

Business consistency boundary
  -> one business action must succeed/fail as a whole

SAP LUW
  -> ABAP/SAP application-level logical unit of work
  -> COMMIT WORK / ROLLBACK WORK / update task / enqueue relation

Database LUW
  -> database transaction
  -> database commit / rollback / cursor and database locks

RAP 再叠加一层框架语义：

Modify / Action / EML / OData request
  -> RAP transactional buffer
  -> authorization / validation / determination / lock
  -> COMMIT ENTITIES or OData save
  -> save sequence
  -> final persistence and cleanup

设计时要问五个问题：

  1. 这次业务动作的 owner 是谁：report、job、BAPI caller、RAP runtime，还是外部 consumer？
  2. 哪些变更必须一起成功或一起失败？
  3. 哪些操作可能已经不可回滚：number range、外部接口、邮件、workflow、日志、远程系统？
  4. 如果失败，错误消息和日志如何证明边界？
  5. 是否存在内部 commit / rollback 抢走了调用方的控制权？

在 code review 中，看到 COMMIT WORK、ROLLBACK WORK、BAPI_TRANSACTION_COMMIT、CALL FUNCTION ... IN UPDATE TASK、legacy API wrapper、RAP action handler、additional save、external call，都应该顺手问一句：它属于谁的事务边界？

可以用下面的表做设计审查：

入口常见 transaction owner主要风险
Classic reportreport 本身或外层调度helper 内部 commit、前台/后台差异
Background jobjob step 中的程序部分成功、重跑、日志与状态不一致
BAPI callercaller，随后调用 BAPI_TRANSACTION_COMMIT被调 BAPI 是否要求外部 commit
RAP OData actionRAP runtime / consumer save semanticshandler 内部 commit、save sequence 被破坏
EML batchEML consumer / COMMIT ENTITIES多实体一致性、failed/reported 处理
External integrationSAP wrapper 与外部系统之间的契约外部副作用不可回滚、重试重复

事务边界设计文档至少应回答：

Transaction boundary record
  - owner
  - start condition
  - objects changed
  - side effects
  - commit trigger
  - rollback trigger
  - point of no return
  - retry / compensation rule
  - evidence: application log, status, request id, job log, API response

对于 API，尤其要区分 validate、execute、status 三种语义。validate 应尽量 side-effect-free；execute 可能创建或改变业务状态，必须有明确提交边界；status 通常是只读查询，不应意外提交或改变状态。

11.5 日本项目现场说法

  • 常见日语：トランザクション境界、コミット制御、ロールバック、内部コミット、保存シーケンス、更新タスク、整合性境界。
  • 设计说明：本処理では RAP runtime がトランザクション境界を管理するため、handler 内では COMMIT WORK を実行しません。
  • Review 说法：この汎用メソッドの内部で commit していないか、呼び出し元の rollback を妨げないか確認してください。
  • 障害调查：主処理は失敗していますが、外部連携とアプリケーションログが既にコミット済みかを切り分けます。
  • 测试说法：途中エラー時に header、item、status、log がどの状態で残るかを確認します。
  • API 说明：validate は副作用なし、execute は保存境界あり、status は参照のみとしてトランザクション境界を分けます。
  • BAPI 说明：BAPI 呼び出し後の commit 責任が呼び出し元にあるか、BAPI 側で内部 commit するかを確認します。
  • 恢复说明：外部連携後に障害が発生した場合の補償・再実行方針を明記します。

11.7 Example / 学习例子

假设一个批量上传 API 执行 execute：

  • 先做 Precheck 预检查，不产生正式副作用。
  • 按 batch id 做 Enqueue locking 加锁设计。
  • 将数据写入 RAP transactional buffer 或准备保存结构。
  • 做 authorization、validation、状态检查。
  • 由 RAP runtime / caller 触发保存语义。
  • save sequence 中统一落库或失败返回。
  • 如果要发 business event、写外部系统或生成不可撤销输出，要明确 point-of-no-return。

错误设计是：action handler 里直接 INSERT ztable 后 COMMIT WORK，然后后续 validation 失败。这样用户看到失败，但数据库已经有数据，事务边界被打破。

更好的 AC0567C execute 设计可以这样描述：

Validate phase:
  Check input, authorization, duplicate request id, object state.
  Do not create final business result.

Execute phase:
  Acquire lock for batch id.
  Prepare changes in transactional context.
  Write application log entries as failure/success evidence with request id.
  Persist only through the owner-controlled save/commit path.

After point of no return:
  If external posting or irreversible output is triggered,
  record external reference and define compensation / retry rule.

Status phase:
  Read current batch state and application log.
  Do not commit or mutate state.

测试也要覆盖中途失败。比如 item 50/100 校验失败时，header 是否保留？成功 49 条是否回滚？application log 是记录整批失败，还是每条状态？如果外部系统已经成功 10 条，SAP 失败后如何避免重试重复？这些都是事务边界，不是简单语法。

11.8 Counterexample / Failure mode

常见失败模式有十类。

第一，helper method 内部 commit。调用方想 rollback，但 helper 已经提交。

第二，RAP handler 直接 COMMIT WORK。框架事务语义被破坏，save sequence 无法完整控制。

第三，外部调用放错位置。数据库最终失败，但外部系统已经收到成功请求。

第四，日志和业务数据边界不一致。业务 rollback 了，日志却写“成功”。

第五，锁释放和事务结束不一致。锁过早释放导致并发修改，或异常后锁残留。

第六，把 Database LUW 当作完整业务边界。数据库 commit 成功不代表整笔业务动作已经一致完成。

第七，validate 阶段产生副作用。预检查或 validate 本应安全可重复，如果它已经写正式数据、发事件或调用外部系统，用户反复点击“检查”也可能改变系统。

第八，status 查询修改状态。查询接口顺手更新“最后访问时间”或清理临时数据，会让只读调用变成隐式写入。

第九，BAPI commit 责任不清。调用标准 BAPI 后忘记 commit，或在 wrapper 中偷偷 commit，都会造成调用方误判。

第十，没有补偿策略。外部系统已成功、SAP 本地失败时，如果没有 external reference、request id 和补偿规则，只能人工猜。

11.9 思考问题

  • 这段代码是否拥有完整 transaction boundary？
  • 是否存在内部 COMMIT WORK / ROLLBACK WORK？
  • 哪些数据和副作用必须一起成功或一起失败？
  • 失败时 rollback 能撤回哪些内容？哪些已经不可撤回？
  • RAP 场景中是否应由 COMMIT ENTITIES / OData save 触发保存？
  • AC0567C POC 中 validate、execute、status 查询分别属于哪个事务边界？
  • 哪一步是 point of no return？它之后失败如何补偿？
  • Background job 重跑时，哪些数据可能已经提交？
  • 标准 BAPI 的 commit 责任在调用方还是被调用方？

11.10 参考资料入口

官方依据补强：SAP LUW 不是普通数据库事务

ABAP Keyword Documentation 对 SAP LUW 的说明非常适合初学者建立正确心智模型：在经典 dialog-oriented application 里，一个应用程序可能跨多个 work process 执行，每一次 work process switch 都可能伴随 implicit database commit。因此，不能把“一个 ABAP 程序从头跑到尾”自动理解成“一个 database LUW”。SAP 为了让业务动作保持一致性，提供了 update task、background unit、PERFORM ... ON COMMIT 等 bundling techniques，把多个变更登记起来，再由一个受控的工作进程在一个 database LUW 中执行。

这也解释了为什么授权、validation 和业务停止条件要尽量在进入 update task 之前完成。update task 更适合执行已经被主流程确认过的持久化变更，不适合承担“现在才判断用户能不能做、业务状态能不能做”的职责。否则失败点会被推迟到事务后段，错误证据也会变得更难解释。

这也是为什么项目里总会追问“谁拥有 transaction boundary”。如果是 classic report 或 background job，通常是最外层程序或 job step 负责决定何时 COMMIT WORK / ROLLBACK WORK；如果是 BAPI 调用，常见模式是 BAPI 返回 RETURN 消息后，由调用方根据结果调用 BAPI_TRANSACTION_COMMIT 或回滚；如果是 RAP/OData，则 behavior implementation 不应在 handler 里直接 COMMIT WORK，而应让 RAP runtime、COMMIT ENTITIES 或 OData save 语义控制 save sequence。底层 helper、BAdI、utility class 如果擅自提交，就会让外层调用者失去 rollback 能力。

可以用一个现场判断句来记忆：transaction boundary 不是“哪一行 SQL 改了数据库”，而是“哪一个业务入口有资格宣布这次业务动作整体成功或整体失败”。 例如批量导入订单时，header、item、status、application log、number range、external API callback 如果被拆在不同提交点，就可能出现“业务数据回滚了但日志说成功”“外部系统已收到请求但 SAP 本地失败”“重跑时重复创建”的事故。真正的设计文档必须写清楚 start condition、changed objects、side effects、commit trigger、rollback trigger、point of no return 和 retry / compensation rule。
