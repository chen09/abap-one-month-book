16. Error message design 错误消息设计

  • Chapter 04
  • Section 16
  • self-contained study note

16.1 一句话结论

Error message design 错误消息设计 是把错误信息设计成“可判断、可处理、可追踪”的项目交付能力；在本教程里要学的是：错误消息不只是给用户看的文字，而是调用方、测试、运维和开发共同使用的诊断接口。

一个坏错误消息只说 处理失败、System error、不正な値です。这种消息看似提醒了用户，实际没有告诉任何人下一步该做什么：是输入数据错了？没有权限？对象被锁？配置缺失？系统异常？网络超时？还是已经处理过不能重复执行？如果调用方无法判断问题类型，就无法决定是否重试、是否修正数据、是否联系业务、是否转给 Basis，或者是否必须开发修复。

好的错误消息应该回答四个问题：发生了什么、影响哪个对象、原因是什么、下一步怎么做。对接口和批处理来说，还要能回答：是否可重试、是否部分成功、失败明细在哪里、有没有 Application Log 应用日志 编号或 request id 可追踪。错误消息设计得越清楚，生产障碍调查越短；错误消息越含糊，项目就越依赖开发者临时读代码。

把这节放在日志和异常章节之间，是因为错误消息是“人和系统共同消费的接口”。用户看到的是一句话，外部系统看到的是 error code 和 payload，运维需要 log id，开发者需要 request id 和上下文，测试人员需要可断言的稳定字段。一个成熟的错误消息设计，必须同时服务这些角色，而不是只在屏幕上显示一段中文或日文。

16.2 到底学什么

第一，要学错误分类。SAP 项目里常见错误至少可以分为业务数据错误、权限错误、锁冲突、状态不允许、配置缺失、重复请求、外部系统错误、技术异常。不同错误类型对应不同处理动作：业务数据错误通常由业务或调用方修正；权限错误需要角色或授权分析；锁冲突可能稍后重试；配置缺失需要功能顾问或运维确认；技术异常需要开发或 Basis 调查。

第二，要学错误消息和异常的边界。Exception vs message 异常与消息 说明的是程序控制流和用户/调用方反馈之间的关系。异常用于中断不应继续的执行路径，消息用于传达可理解的结果。不要把所有异常原文直接抛给用户，也不要把所有业务校验都写成不可区分的技术异常。

第三，要学错误消息和日志的配合。前台用户需要简洁可行动的提示；接口调用方需要稳定字段，例如 error code、message type、message text、target field、retryable、log id；运维需要 request id、业务键、时间和应用日志入口。错误消息不必承载所有细节，但必须能把人带到正确证据。

第四，要学 SAP 的消息体系。经典 ABAP 常用 message class / T100 message 管理可翻译消息文本，参见 Message class and T100 messages；RAP、OData、Fiori 场景还涉及 reported messages、field messages、HTTP status、business error payload 等表现方式。RAP 中可以粗略记成：failed 表示某个实例或操作不能继续完成，reported 负责把消息带回 consumer；字段级消息应尽量绑定到对象/字段，而不是只给一个全局泛化错误。不同技术栈形式不同，但设计原则一致：稳定、具体、可翻译、可追踪、不过度暴露敏感信息。

第五，要学错误响应契约。对外 API 不应该让调用方解析自由文本来判断错误类型。自由文本可以翻译、可以优化表达，但 error code、target、retryable、log id、业务键这类字段应稳定。特别是在日本项目中，UI 文言可能随客户表达调整，但系统间集成需要稳定契约。

第六，要学多语言和安全。SAP 项目经常需要中文学习、日语现场、英语技术术语并存。错误消息对用户可以本地化，但日志和接口证据要保留稳定的技术字段。同时，安全敏感信息不应出现在用户可见消息里：比如 SQL、内部类名、token、密码、完整 payload、个人信息、授权配置细节。

接口或批处理场景可以固定使用一个最小 error response evidence packet：

Error Response Evidence Packet

Error category:
Stable error code / message id:
User-facing message:
Target object / field / row:

Retryable:

这个模板不是要求每个 UI 弹窗都显示所有字段，而是要求测试证据和接口契约中能找到这些信息。用户消息可以简洁，日志和 payload 要足够让支持团队定位。

16.3 为什么重要

错误消息是生产系统的第一层诊断界面。很多时候，用户不会打开调试器，也不会知道底层表名、类名、异常栈；他们只会看到一条消息，并据此决定下一步。如果消息不清楚，用户会截图发给支持；支持看不懂再转开发；开发再要求重现；时间就在来回确认中消耗掉。

对接口来说，错误消息更像契约。外部系统需要根据错误类型判断是否自动重试、是否丢入人工队列、是否回滚本地状态、是否通知业务用户。如果 SAP 只返回 500 Internal Server Error 或一段不稳定文本，外部系统就无法做可靠处理。特别是 AC0567C 这类 API/批处理场景，错误消息必须和幂等、事务边界、应用日志、状态查询一起设计。

对日本项目交付来说，错误消息还影响测试证据和客户说明。结合测试缺陷单里经常需要写：原因、再現手順、期待結果、実際結果、対応方針。如果消息本身就能区分 業務エラー、権限エラー、ロックエラー、システムエラー，缺陷分流会快很多。

错误消息设计也是生产成本控制。没有稳定错误码，外部系统只能人工判断；没有 target field，业务用户不知道改哪一列；没有 retryable，接口可能反复重试不可重试的业务错误；没有 log id，运维找不到证据。好的错误消息不是“写得漂亮”，而是能让正确的人采取正确动作。

16.4 Key mechanism / 核心机制

错误消息设计可以从“分类、定位、动作、追踪”四个维度做。

分类，是指给错误一个稳定类型。比如 BUSINESS_VALIDATION_FAILED、AUTHORIZATION_FAILED、LOCK_CONFLICT、CONFIGURATION_MISSING、DUPLICATE_REQUEST、TECHNICAL_ERROR。中文和日语消息可以面向用户，但内部 error code 或 message id 应稳定，方便接口、测试和日志检索。

定位，是指指出哪个对象、字段、行号或业务键出错。批量上传不能只说“数据错误”，而应告诉第几行、哪个字段、哪个业务键；OData/Fiori 场景应尽量把字段级消息挂到对应字段；后台任务应记录业务对象和应用日志编号。定位信息越精确，用户越能自己修正。

动作，是指告诉下一步。比如“请维护公司代码配置后重试”“请联系权限管理员补充角色”“对象正在被其他用户处理，请稍后重试”“该请求已处理过，请使用状态查询接口确认结果”。动作不需要写成长篇说明，但必须让用户知道问题属于谁、能不能重试。

追踪，是指把错误消息连接到证据。接口响应中可返回 request id、correlation id、application log id；前台消息可提示日志编号；缺陷单可记录时间、用户、输入条件。这样 Application Log 应用日志、Security Audit Log、trace、后台 job 和业务状态才能串起来。

还要控制信息暴露。不要在错误消息中显示 SQL 语句、内部表名、完整异常栈、token、密码、完整个人信息或过度敏感的权限配置。对用户可见的消息应可行动但不过度泄露；详细技术信息可写入受控日志，由有权限人员查看。

可以用下面的错误分类表：

Error categoryRetryableOwnerTypical evidence
BUSINESS_VALIDATION_FAILED通常修正数据后可重试Business / callertarget field、row、business key
AUTHORIZATION_FAILED否，除非角色修正Security / role owneruser、role、SU53/trace

这些 error category 是教程中的设计示例，不是 SAP 固定枚举。项目可以按接口、客户标准和框架约束收敛成自己的稳定 error code set，但不能让调用方解析自由文本来判断错误类型。

一个 API error payload 可以这样设计：

{
  "errorCode": "BUSINESS_VALIDATION_FAILED",
  "message": "Row 27: Company code 9999 is not valid for this process.",
  "target": "items[27].companyCode",
  "retryable": true,
  "nextAction": "Correct the company code and submit the request again.",
  "businessKey": "BATCH-2026-0001",
  "requestId": "AC0567C-20260615-0001",
  "applicationLogId": "ZUPLOAD/20260615/0001"
}

这不要求所有 SAP 标准 OData 响应都长这样，而是给自建 API、批处理状态 API 和错误设计提供思路。RAP/OData/Fiori 的具体消息结构要按框架实现，但稳定字段、可行动消息和可追踪证据这几个原则不变。

16.5 日本项目现场说法

常见日语包括：エラーメッセージ設計、業務エラー、権限エラー、ロックエラー、システムエラー、再実行可否、エラーコード、メッセージID、ログ番号、原因切り分け、障害調査。

设计说明中可以这样写：

  • エラー応答には、エラーコード、メッセージ、対象項目、再実行可否、アプリケーションログ番号を含めます。
  • 業務エラー、権限エラー、ロックエラー、システムエラーを区別し、呼び出し元が後続処理を判断できるようにします。
  • ユーザ向けメッセージには機微情報や内部例外スタックを出力しません。

缺陷分析时可以这样问：

  • このエラーは業務データ修正で解消できますか、それとも権限・設定・プログラム修正が必要ですか。
  • 再実行可能なエラーですか。二重登録のリスクはありませんか。
  • 該当エラーのログ番号またはリクエストIDはありますか。

更接近设计 review 的表达：

エラーコードは外部連携の契約項目として扱い、文言変更とは別に管理します。

含义：error code 作为外部集成契约字段管理，与文案变更分开。

業務ユーザ向けメッセージと運用調査用のログ情報を分離し、必要最小限の情報だけを画面に表示します。

含义：把业务用户消息和运维调查日志分离，画面只显示必要最小信息。

再実行可能なエラーかどうかを明示し、二重登録リスクがある場合は status API で確認させます。

含义：明确是否可重试；如果有重复登记风险，让调用方通过 status API 确认。

16.7 Example / 学习例子

假设 AC0567C 有一个 execute upload API。外部系统传入一个 batch id 和 100 行数据，SAP 校验并执行。坏设计可能只返回：

status = E
message = Process failed.

调用方看到这条消息后什么都不能判断。更好的设计可以返回：

status = E
errorCode = BUSINESS_VALIDATION_FAILED
message = Row 27: Company code 9999 is not valid for this process.
target = items[27].companyCode
retryable = true
requestId = AC0567C-20260615-0001
applicationLogId = ZUPLOAD/20260615/0001

如果是权限错误，应该不同：

errorCode = AUTHORIZATION_FAILED
message = User is not authorized to execute upload for company code 1000.
retryable = false
nextAction = Contact the role administrator or execute with an authorized user.

如果是锁冲突，又不同：

errorCode = LOCK_CONFLICT
message = Batch AC0567C-20260615-0001 is currently processed by another request.
retryable = true
nextAction = Retry after the current process finishes or check the status API.

这三个错误都叫“失败”，但处理动作完全不同。错误消息设计的目的，就是让系统和人都能做正确分流。

AC0567C 可以设计一张最小错误矩阵：

errorCodeHTTP/status idearetryablenext action
VALIDATION_FAILED400 / business errortrue after correction修正字段后重新 validate
AUTHORIZATION_FAILED403false申请角色或更换授权用户
JOB_ALREADY_RUNNING409true later稍后查询 status 或等待当前 job 完成
DUPLICATE_REQUEST200/409 by designfalse for execute使用 status API 查询既有结果
CONFIGURATION_MISSING500/422 by designtrue after config功能/Basis 维护配置后重试
TECHNICAL_FAILURE500do not blindly retry提供 request id / log id 给运维调查

注意 HTTP status 和业务 error code 是两个层次。HTTP status 表示协议层结果，errorCode 表示业务/技术错误类型。对于某些幂等场景，重复请求可能返回既有结果而不是错误；关键是契约要稳定，并在 API testing workflow 中记录。

16.8 Counterexample / Failure mode

常见失败模式有八类。

第一，只返回通用文本。処理に失敗しました、エラーが発生しました、System error 不能支持调用方判断，也不能支持测试验收。

第二，把技术异常直接暴露给用户。完整 class name、SQL 错误、stack trace、内部表名、服务器路径、token 或个人信息都不应出现在普通用户或外部系统可见的消息里。

第三，错误码不稳定。今天返回 E001 表示业务校验，明天同一个代码改成权限错误，外部系统就会误处理。对接口来说，error code 是契约，必须谨慎变更。

第四，不说明可重试性。网络超时、锁冲突、重复请求、业务校验失败、权限不足的重试策略完全不同。如果消息不说明 retryable 或状态查询方式，调用方可能反复提交，造成重复处理。

第五，消息和日志脱节。用户拿到错误消息，但没有 request id、log id、业务键或时间窗口，运维无法定位对应 Application Log 应用日志，障碍调查就会变成猜谜。

第六，错误分类过细或过粗。过粗会失去分流能力，过细会让调用方无法维护判断逻辑。项目中常用做法是保持少量稳定 category，再用 message id/target/log 细化。

第七，把 retryable 写错。把不可重试的业务错误标成可重试，会造成重复提交；把锁冲突或临时外部错误标成不可重试，会增加人工处理。

第八，没有字段级定位。批量上传只说“第 27 行有误”还不够，最好指出字段、值、原因和可修正动作。
