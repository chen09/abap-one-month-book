Chapter 02 - Core ABAP Programming

8. INSERT statement 新增数据

  • Chapter 02
  • Section 08
  • self-contained study note

8.1 一句话结论

ABAP SQL 的 INSERT 用来把新行写入可写数据库对象，例如项目自有 DDIC table、可写 CDS entity 或其他被允许写入的持久化对象。它的关键不是“把一行塞进表里”这么简单，而是确认写入对象是否允许、主键是否唯一、重复键如何处理、实际插入了几行、谁来提交或回滚事务，以及这次写入是否符合权限、锁和 Clean Core 边界。

8.2 到底学什么

先把边界说清楚：本节讲的是 ABAP SQL INSERT，也就是写数据库持久化对象；它不是内部表的 INSERT 语句。内部表插入只是改 ABAP 内存，ABAP SQL INSERT 会进入数据库修改路径，会影响事务、锁、提交、回滚、权限、测试数据和生产数据安全。

最小单行写入长这样：

DATA(ls_note) = VALUE zorder_note(
  note_id        = lv_note_id
  sales_order_id = lv_sales_order_id
  note_text      = lv_note_text
  created_by     = sy-uname ).

INSERT zorder_note FROM @ls_note.

IF sy-subrc <> 0.
  " Handle duplicate key or failed insert according to the business rule.
ENDIF.

这里 zorder_note 是示例自定义表，ls_note 是要插入的一行。INSERT 的意思是“这行应该是新记录”。如果同一主键已经存在，业务上必须回答：这是错误、重复请求、幂等成功，还是应该改用 UPDATE / MODIFY / RAP action？不要用一个语句把业务规则糊过去。

多行写入通常来自内表：

INSERT zorder_note FROM TABLE @lt_new_notes.

这比循环里逐行 INSERT 更接近数据库批量操作，但也更要求提前处理重复键和错误路径。项目里真正要学的是：写入前准备数据，写入时处理数据库结果，写入后由正确的事务边界提交或回滚。

8.3 为什么重要

SELECT 写错，通常是读错数据或性能差；INSERT 写错，可能直接制造脏数据、重复数据、锁等待、短 dump、接口重试不幂等、测试环境污染，甚至生产事故。读操作可以慢慢查，写操作一旦提交，就进入业务事实的一部分。

它的第一层价值是明确“新建”的语义。INSERT 表示目标行不应该已经存在；如果存在同主键行，就不是普通新增，而是冲突。初学者常把 INSERT、UPDATE、MODIFY 混在一起用，最后看不出业务到底是新增、修改还是 upsert。

第二层价值是事务意识。INSERT 执行成功不等于业务已经安全结束。数据库变更通常要到 database commit 后才永久生效；在 SAP 程序里又要放进更大的 SAP LUW 里理解。一个可复用方法里随手 COMMIT WORK，可能破坏上层事务；一个批处理里不处理失败路径，可能半路写入一批不一致数据。

第三层价值是并发和锁。数据库会有自己的锁，但 SAP 业务对象常常还需要 enqueue lock、RAP lock、号码段、唯一性检查或幂等键设计。比如“先 SELECT MAX( id ) 再 INSERT id + 1”在多人并发下很危险，因为两个用户可能算出同一个新 ID。

第四层价值是 Clean Core。传统项目里直接写自定义表很常见；但直接写 SAP 标准表通常是高风险动作。S/4HANA、ABAP Cloud、RAP 和 Clean Core 语境下，优先找 released API、BAPI、RAP BO、released CDS/write model 或项目认可的扩展点。会写 INSERT 不代表应该直接写任意表。

8.4 Key mechanism / 核心机制

8.4.1 INSERT 的目标必须是可写对象

官方文档把 INSERT、UPDATE、MODIFY、DELETE 的目标归为 writable database object。对初学者来说，先按三类理解：

  • 项目自有 DDIC database table：最常见的练习和自定义开发场景。
  • 可写 CDS / RAP 相关对象：要遵守对应框架的事务模型。
  • SAP 标准持久化对象：不能因为语法可写就直接写，必须有官方 API 或项目治理许可。

因此，写 INSERT mara ... 这类直接写标准表的想法，在真实项目里应先停下来做架构确认。多数业务写入应该通过事务、API、BAPI、RAP BO 或应用服务完成。

8.4.2 单行写入：FROM @wa 或 VALUES @wa

常见写法有两种：

INSERT zorder_note FROM @ls_note.

以及：

INSERT INTO zorder_note VALUES @ls_note.

两者都表达插入一行。现代 ABAP 里也经常用 host expression 直接构造一行：

INSERT zorder_note FROM @( VALUE #(
  note_id        = lv_note_id
  sales_order_id = lv_sales_order_id
  note_text      = lv_note_text
  created_by     = sy-uname ) ).

这适合简单场景；如果字段多、校验多、默认值多，先构造 ls_note 并写清楚每个字段来源通常更可读。

8.4.3 多行写入：FROM TABLE @itab

批量新增使用内表作为来源：

INSERT zorder_note FROM TABLE @lt_new_notes.

不要轻易写成：

LOOP AT lt_new_notes INTO DATA(ls_new_note).
  INSERT zorder_note FROM @ls_new_note.
ENDLOOP.

循环逐行写不仅慢，还会把错误处理拆散。批量写入更容易让 reviewer 看出“这是一批新增”，也更容易统一处理 sy-subrc、sy-dbcnt、异常和日志。

但批量写入有一个危险点：如果内表中某些行和数据库已有行主键重复，或者内表内部存在重复键，不能假设系统会自动帮你合并。SAP Samples 的 ABAP SQL cheat sheet 明确提醒，FROM TABLE @itab 的重复键可能导致运行时错误；如果使用 ACCEPTING DUPLICATE KEYS，重复行会被丢弃，sy-subrc 会反映并非全部插入成功。

8.4.4 ACCEPTING DUPLICATE KEYS 不是“忽略错误就好”

如果业务允许重复请求被跳过，可以写：

INSERT zorder_note FROM TABLE @lt_new_notes ACCEPTING DUPLICATE KEYS.

DATA(lv_inserted_count) = sy-dbcnt.

IF sy-subrc = 4.
  " Some rows were skipped because of duplicate primary/unique keys.
  " Log or report skipped rows according to the business rule.
ENDIF.

这里 sy-dbcnt 才是实际插入行数。sy-subrc = 4 不应被当作“没事”。它表示不是所有行都插入成功。项目里要说明被跳过的重复行是预期幂等、数据清洗结果，还是需要报错给调用方。

如果业务不允许重复键，通常不要加 ACCEPTING DUPLICATE KEYS，而是让错误路径明确暴露，并用 TRY ... CATCH cx_sy_open_sql_db 或上层异常策略处理：

TRY.
    INSERT zorder_note FROM TABLE @lt_new_notes.
  CATCH cx_sy_open_sql_db INTO DATA(lx_sql).
    " Convert the database failure into a business/application error.
ENDTRY.
8.4.5 sy-subrc 和 sy-dbcnt 要立刻读

ABAP SQL 写入语句会设置系统字段。对 INSERT 来说，最常用的是：

  • sy-subrc = 0：单行或全部指定行插入成功。
  • sy-subrc = 4：单行未插入，或使用 ACCEPTING DUPLICATE KEYS 时不是所有行插入成功。
  • sy-dbcnt：实际插入的行数。

这些值应该在 INSERT 之后马上读取。不要隔几行又执行了别的 SQL 或语句后再看，因为系统字段可能已经被其他操作覆盖。更稳的写法是立即复制到业务变量：

INSERT zorder_note FROM TABLE @lt_new_notes ACCEPTING DUPLICATE KEYS.

DATA(lv_insert_subrc) = sy-subrc.
DATA(lv_inserted)     = sy-dbcnt.
8.4.6 INSERT 成功不等于已经提交

INSERT 把变更登记到当前数据库事务里；真正永久生效要看 commit。官方文档也强调，插入的行会在下一个 database commit 后永久进入表，在此之前仍可被 database rollback 撤销。

这不代表每个 INSERT 后都要写：

COMMIT WORK.

恰好相反，可复用方法、工具类、RAP handler、BAPI 内部、框架回调里随手提交，是常见事故来源。谁负责 COMMIT WORK，要由应用事务边界决定。批处理 report 可能在一批业务对象成功后提交；RAP 应用通常由 RAP save sequence 和 COMMIT ENTITIES / 框架事务模型处理；被其他程序调用的方法通常不应擅自结束调用方的 LUW。

8.4.7 INSERT 不会自动处理授权和业务校验

ABAP SQL 的 INSERT 不等于业务权限检查。能写表，不代表当前用户被业务允许创建这类对象。常见检查包括：

  • 当前用户是否能创建该业务对象。
  • 组织范围、公司代码、工厂、销售组织是否允许。
  • 字段组合是否符合业务规则。
  • 是否已经存在相同业务键，重试是否幂等。
  • 是否需要应用锁或号码段。

在经典 ABAP 中可能需要 AUTHORITY-CHECK、enqueue lock、应用校验和日志；在 RAP 中要看 global authorization、instance authorization、validation、determination、lock 和 save sequence。

8.4.8 INSERT、UPDATE、MODIFY 的语义不要混用

三者最容易混：

  • INSERT：新增。已存在同 key 时是冲突。
  • UPDATE：修改已存在行。找不到目标行时通常不是新增。
  • MODIFY：在 ABAP SQL 语境中可以插入或覆盖，接近 upsert 语义。

初学者喜欢用 MODIFY 省事，因为“有就改，没有就插”。但真实项目里这可能隐藏重要业务差异：重复请求、并发冲突、数据补偿、重跑批处理、接口幂等，都需要明确语义。只有当业务确实需要 upsert，并且审计、权限和字段覆盖规则都清楚时，才应该使用这种语义。

8.7 Example / 学习例子

8.7.1 例 1：单行新增
DATA(ls_note) = VALUE zorder_note(
  note_id        = lv_note_id
  sales_order_id = lv_sales_order_id
  note_text      = lv_note_text
  created_by     = sy-uname ).

INSERT zorder_note FROM @ls_note.

DATA(lv_insert_subrc) = sy-subrc.
DATA(lv_inserted)     = sy-dbcnt.

读这段代码时，要确认 note_id 如何生成、是否可能重复、当前用户是否允许给该订单新增备注、以及谁负责提交事务。lv_inserted 预期应为 1；如果不是 1，就要走错误路径。

8.7.2 例 2：批量新增并明确重复键策略
INSERT zorder_note FROM TABLE @lt_new_notes ACCEPTING DUPLICATE KEYS.

DATA(lv_insert_subrc) = sy-subrc.
DATA(lv_inserted)     = sy-dbcnt.

IF lv_insert_subrc = 4.
  " Some rows were skipped due to duplicate keys.
  " Decide whether this is idempotent success or a business error.
ENDIF.

这个例子不是推荐“任何重复都忽略”，而是展示：如果选择忽略重复键，就必须把实际插入件数和跳过件数纳入业务说明、日志和测试断言。

8.7.3 例 3：不允许重复键时捕获数据库异常
TRY.
    INSERT zorder_note FROM TABLE @lt_new_notes.
  CATCH cx_sy_open_sql_db INTO DATA(lx_sql).
    " Raise or log an application-specific error.
ENDTRY.

在严格业务场景里，重复键可能说明上游幂等键设计错了、批处理重跑策略错了，或者并发控制不足。让错误暴露出来，比静默跳过更安全。

8.7.4 例 4：不要在可复用方法里擅自提交
METHOD save_notes.
  INSERT zorder_note FROM TABLE @it_notes.

  " Do not commit here unless this method owns the full transaction boundary.
  " COMMIT WORK.
ENDMETHOD.

这个例子故意把 COMMIT WORK 注释掉。判断能不能提交，不看“这里有没有 INSERT”，而看这个方法是否拥有完整业务事务边界。RAP handler / saver、BAPI 调用链、批处理框架、测试框架都有自己的提交规则。

8.8 Counterexample / Failure mode

失败模式一：把 INSERT 当成“有就覆盖”。如果业务是新增，重复 key 应该是冲突；如果业务是 upsert，应该明确使用 MODIFY、RAP action 或应用服务，并说明字段覆盖规则。

失败模式二：循环逐行插入，没有统一错误处理。第一百条失败时，前九十九条是否已经进入当前 LUW？是否要回滚？是否要重试？这些都不清楚。

失败模式三：加了 ACCEPTING DUPLICATE KEYS 但不看 sy-subrc 和 sy-dbcnt。这会把“部分插入成功”伪装成“全部成功”。

失败模式四：在工具方法里随手 COMMIT WORK。这可能提前提交调用方还没校验完的数据，也可能破坏 RAP/BAPI/批处理框架的事务设计。

失败模式五：直接写 SAP 标准表。即使技术上有权限，升级稳定性、数据一致性、业务校验、缓存、变更文档、派生逻辑都可能被绕过。先找 released API、BAPI、RAP BO 或正式扩展点。

失败模式六：用 SELECT MAX( id ) + 1 生成主键再 INSERT。并发下两个会话可能得到同一个新 ID。应使用号码段、UUID、数据库唯一约束、锁或框架提供的编号机制。

失败模式七：只做数据库写入，不做业务授权。INSERT 本身不是 AUTHORITY-CHECK，也不是 RAP authorization。

失败模式八：测试后不清理数据。写入类单元测试或手动验证如果直接写共享表，容易污染后续测试和他人环境。要使用测试隔离、回滚、测试专用表或清理策略。

8.9 思考问题

  • 这个 INSERT 的目标对象是谁？它是否允许被当前程序直接写入？
  • 这是真正新增，还是 upsert、重试、补偿或导入？
  • 主键和唯一键从哪里来？是否有并发冲突风险？
  • 重复键是业务错误、幂等成功，还是应该跳过并记录？
  • 是否需要 ACCEPTING DUPLICATE KEYS？如果需要，如何处理 sy-dbcnt 少于预期？
  • INSERT 后是否立刻读取了 sy-subrc 和 sy-dbcnt？
  • 事务边界在哪里？谁负责 COMMIT WORK 或 rollback？
  • 是否需要 lock object、RAP lock、号码段或 UUID？
  • 权限和业务校验在哪里发生？
  • ABAP Cloud / RAP / Clean Core 项目中，是否应该改用 released API、BAPI、RAP BO 或 EML？
