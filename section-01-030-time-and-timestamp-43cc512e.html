30. Time and timestamp 时间戳

  • Chapter 01
  • Section 30
  • self-contained study note

本节结论

Time and timestamp 时间戳 是把“几点几分几秒”和“某个全球唯一时刻”分清楚的基础。Day 04 要先掌握三条边界：TYPE t 只是一天内的 local time，sy-uzeit / sy-timlo 只适合读懂旧代码，跨时区、审计日志、接口时间戳和持续时间计算要用 UTC timestamp / utclong 思维。

正文

上一节 Date arithmetic 日期计算 讲的是日期：今天、到期日、有效期间。时间和 timestamp 解决的是另一类问题：程序什么时候开始、什么时候结束、接口消息产生在什么时刻、日志按什么时间排序、用户看到的是哪个时区的时间、跨午夜的处理是否正确。初学者最容易犯的错误，是把 date + time 当成一个自然可靠的时间点，或者把本地显示时间当成审计用 UTC 时间。

先看 ABAP 的 time type：TYPE t。SAP Learning 明确说明，ABAP built-in type T 表示 time，内部格式是 HHMMSS，24-hour clock notation，精度到 1 秒。它表示“一天中的时刻”，例如 083000 表示 08:30:00。它不包含日期，也不包含时区。

DATA system_time TYPE t.
system_time = cl_abap_context_info=>get_system_time( ).

out->write( |System time: { system_time }| ).

老代码里会看到 sy-uzeit，它是 system time；也会看到 sy-timlo，它是 user local time，和 sy-datlo 一起对应用户本地日期/时间；sy-zonlo 表示 user time zone。现代 ABAP，尤其 ABAP Cloud / BTP ABAP Environment，建议优先使用 cl_abap_context_info 或 XCO 这类上下文 API，而不是把 system field 当成新代码默认写法。在 ABAP for Cloud Development 中，sy-datum、sy-uzeit、sy-datlo、sy-timlo、sy-zonlo 这类 date/time system fields 不是推荐写法；SAP 文档也提醒，cl_abap_context_info=>get_system_date( ) / get_system_time( ) 返回的是当前 UTC date/time，而不是传统意义上的用户本地时刻。这里的纪律和 ABAP system fields sy fields 一样：读旧代码要认识 sy-uzeit / sy-timlo，写新代码要确认当前语义到底是 system time、user local time，还是 UTC timestamp。

TYPE t 可以做简单 time calculation。ABAP 在计算时会把 time 转成从午夜开始的秒数。比如两个 time 相减可以得到秒数差，time 加整数可以理解为加秒。但 Day 04 要强调边界：TYPE t 没有日期，所以它无法知道“23:55 到 00:05”是不是跨了午夜。只要需求牵涉跨天、运行时长、SLA、审计、接口顺序、跨时区，就不要只用 TYPE t 硬算。

DATA start_time TYPE t VALUE '083000'.
DATA end_time   TYPE t VALUE '084500'.
DATA seconds    TYPE i.

seconds = end_time - start_time.
out->write( |Duration in same day: { seconds } seconds| ).

这段例子只适合“同一天、同一时区、同一个上下文”的简单说明。真实项目里，如果 batch job 从 23:59 跑到 00:10，或者接口发送方在 UTC、接收方在 JST，只看 TYPE t 会丢掉关键信息。

timestamp 的目标，是表示一个明确的时刻。现代 ABAP 可以使用 built-in type utclong 表示 UTC timestamp。SAP Learning 说明 utclong 按 ISO-8601 表示 UTC 时间，格式类似 YYYY-MM-DDTHH:MM:SS.sssssssZ，精度可到 100 nanoseconds。它适合表达“全世界同一个 instant”，而不是某个用户本地看到的时钟值。

DATA started_at TYPE utclong.
DATA finished_at TYPE utclong.
DATA runtime_seconds TYPE decfloat34.

started_at = utclong_current( ).

" ... do something ...

finished_at = utclong_current( ).
runtime_seconds = utclong_diff( high = finished_at low = started_at ).

这里比 sy-uzeit 的好处很清楚：即使跨午夜、跨时区，utclong_diff( ) 仍然是在比较两个 UTC timestamp，返回值是以秒为单位的 decfloat34 差值。需要拆成 days / hours / minutes / seconds 时，可以再认识系统类 CL_ABAP_UTCLONG 的 DIFF 方法；但 Day 04 先掌握 built-in function utclong_diff( ) 就够了。utclong_diff( ) 和 CL_ABAP_UTCLONG=>DIFF 不是互相替代的错误/正确关系，而是“秒数差”和“分量差”的两种用法。对 Java/Python 背景的人，可以把它类比成 timezone-aware instant / UTC timestamp，而不是 naive local datetime。

timestamp 和本地日期/时间之间需要显式转换。用户在日本看到的 JST 时间，和数据库里用于审计的 UTC timestamp，可能不是同一个表达。ABAP 提供 CONVERT ... TIME ZONE ... 相关语句/函数来在 UTC timestamp 和 local date/time 之间转换。Day 04 不要求背完整语法，只要求记住：转换时必须说明 time zone，不能只拼 date + time 字符串。

DATA now_utc TYPE utclong.
DATA user_date TYPE d.
DATA user_time TYPE t.
DATA user_time_zone TYPE timezone.

now_utc = utclong_current( ).

TRY.
    user_time_zone = cl_abap_context_info=>get_user_time_zone( ).

    CONVERT UTCLONG now_utc
            INTO DATE user_date
                 TIME user_time
            TIME ZONE user_time_zone.
  CATCH cx_abap_context_info_error cx_sy_conversion_no_date_time INTO DATA(time_error).
    out->write( time_error->get_text( ) ).
ENDTRY.

这段代码有两个细节。第一，CONVERT UTCLONG 的语法是先 INTO DATE ... TIME ...，再 TIME ZONE ...；第二，CONVERT UTCLONG 遇到无效 timestamp、无效 time zone 或无法转换的本地时间时，会抛出可捕获异常 CX_SY_CONVERSION_NO_DATE_TIME。旧代码里还会看到 packed-number timestamp，也就是 DDIC 的 TIMESTAMP / TIMESTAMPL；对应语句 CONVERT TIME STAMP 不是抛同一个异常模型，而是会设置 sy-subrc，例如无效 time zone 为 8、无效 timestamp 为 12。初学阶段只要记住：utclong 和 legacy packed timestamp 是两套时间戳表达，不要混用错误的错误处理方式。

如果项目需要把本地日期/时间转换回 timestamp，也要提供 time zone。否则 2026-06-13 09:00:00 到底是 JST、UTC、CET 还是别的地区时间，系统无法可靠判断。夏令时 / daylight saving time 也是同类风险：某些地区一天不是严格 24 小时，某些本地时间甚至可能不存在或重复。

审计和日志要特别小心。技术日志里写 Started at { sy-uzeit } 对本地调试可能够用，但生产审计、接口追踪、异步处理、跨系统 correlation 更需要 UTC timestamp、request ID、message ID、user、client、time zone 等组合信息。不要把 sy-datum + sy-uzeit 当作长期审计字段，也不要只存用户本地显示时间。后面讲 application log、security audit log、OData error response 和生产排查时，会继续回到这个边界。

本节的最小心智模型是：TYPE t 用来表示一天内的时刻，适合同日简单展示和初级计算；sy-uzeit / sy-timlo 用来读懂旧代码和 system/user local time 差异；utclong 用来表达 UTC instant，适合运行时长、接口顺序、审计和跨时区。只要问题里出现跨天、时区、UTC、审计、SLA、接口，就从 TYPE t 升级到 timestamp。

为什么重要

  • 接口日志、batch 运行时间、错误追踪、审计记录、OData timestamp 都需要准确时间语义。
  • TYPE t 只有 HHMMSS，没有日期和时区；用它做跨天 duration 很危险。
  • sy-uzeit、sy-timlo、sy-zonlo 的差异会影响日本用户、本地服务器和海外系统之间的时间解释。
  • UTC timestamp 能让跨系统、跨时区、异步处理按同一时间轴排序。
  • Java/Python 背景学习者常熟悉 datetime，但 SAP 项目还要结合 system field、user time zone、ABAP Cloud API、OData 类型和审计要求。

核心机制

  • TYPE t：ABAP built-in time type，内部格式 HHMMSS，24-hour notation，精度到 1 秒。
  • time calculation：ABAP 可把 time 转成从午夜起算的秒数来做简单加减。
  • cl_abap_context_info=>get_system_time( )：现代 ABAP 中表达当前系统时间的上下文 API。
  • sy-uzeit：system time，老代码常见；不等于 user local time。
  • sy-timlo / sy-datlo：user local time / date，和 user time zone 相关。
  • sy-zonlo：user time zone system field；读旧代码要认识，ABAP Cloud 新代码优先用 released API 或 XCO。
  • utclong：UTC timestamp built-in type，表示全球唯一时刻，适合接口、审计和 duration。
  • utclong_current( )：取得当前 UTC timestamp。
  • utclong_add( ) / utclong_diff( )：对 UTC timestamp 加减和计算差值；utclong_diff( ) 返回秒数差。
  • CL_ABAP_UTCLONG=>DIFF：需要 days / hours / minutes / seconds 分量时可认识的系统类方法。
  • CONVERT UTCLONG ... INTO DATE ... TIME ... TIME ZONE ...：把 utclong 拆成 local date/time；异常处理使用 CX_SY_CONVERSION_NO_DATE_TIME。
  • CONVERT TIME STAMP ... TIME ZONE ... INTO DATE ... TIME ...：legacy packed-number timestamp 的转换语句；错误处理看 sy-subrc。
  • audit boundary：生产审计不应只存 local display time；应保留 UTC timestamp、user/time zone/request context 等证据。

日本项目现场表达

  • 常见日语：時刻、時間、タイムスタンプ、UTCタイムスタンプ、タイムゾーン、ローカル時刻。
  • 系统/用户相关：システム時刻、ユーザ時刻、ユーザタイムゾーン、JST、サーバ時刻。
  • 日志/审计相关：処理開始時刻、処理終了時刻、経過時間、監査ログ、インターフェース連携時刻。
  • 风险相关：日跨ぎ、時差、夏時間、時刻変換、タイムゾーン考慮漏れ。
  • 说明句：この項目はユーザ表示用のローカル時刻であり、監査用の UTC タイムスタンプとは別に管理します。
  • Review 句：日跨ぎやタイムゾーン差異があるため、sy-uzeit ではなく UTC タイムスタンプで経過時間を算出してください。
  • Review 句：インターフェース連携時刻は送信元と受信先のタイムゾーンを明示してください。
  • Review 句：ユーザタイムゾーンを考慮した変換が実装されているか確認してください。
  • Review 句：日跨ぎが発生する場合は TYPE t の減算ではなく、UTC タイムスタンプで差分を算出してください。
  • Review 句：夏時間の切り替わり時刻をまたぐ処理は、ローカル時刻ではなく UTC タイムスタンプを基準にしてください。

学习例子

先读系统时间：

DATA system_time TYPE t.
system_time = cl_abap_context_info=>get_system_time( ).

out->write( |System time: { system_time }| ).

再用 UTC timestamp 计算运行时间：

DATA started_at TYPE utclong.
DATA finished_at TYPE utclong.
DATA runtime_seconds TYPE decfloat34.

started_at = utclong_current( ).

" ... processing ...

finished_at = utclong_current( ).
runtime_seconds = utclong_diff( high = finished_at low = started_at ).

out->write( |Runtime seconds: { runtime_seconds }| ).

把 UTC timestamp 转成用户本地日期/时间：

DATA user_date TYPE d.
DATA user_time TYPE t.
DATA user_time_zone TYPE timezone.

TRY.
    user_time_zone = cl_abap_context_info=>get_user_time_zone( ).

    CONVERT UTCLONG finished_at
            INTO DATE user_date
                 TIME user_time
            TIME ZONE user_time_zone.
  CATCH cx_abap_context_info_error cx_sy_conversion_no_date_time INTO DATA(time_error).
    out->write( time_error->get_text( ) ).
ENDTRY.

这些例子只建立心智模型。真实项目中，日志字段、DB 字段、OData 类型、时区配置、用户主数据和前端显示都可能影响最终设计。

常见误区

  • 用 sy-uzeit 计算 batch 运行时长：跨午夜后差值可能变成负数或无意义。
  • 把 sy-datum + sy-uzeit 拼成审计 timestamp：缺少 time zone，跨系统无法可靠排序。
  • 把用户本地显示时间存进接口字段：接收系统不知道这是 JST、UTC 还是服务器时区。
  • 只存 local time，不存 UTC timestamp：障害調査时无法还原事件顺序。
  • 忽略 daylight saving time：某些本地时间不存在或重复，duration 计算出错。
  • 把 TYPE t 当作 duration：010000 是 01:00:00 时刻，不自然等于“1 小时持续时间”。
  • 把 CONVERT UTCLONG 和 CONVERT TIME STAMP 的错误处理混在一起：前者要处理 CX_SY_CONVERSION_NO_DATE_TIME，后者要检查 sy-subrc。
  • 对 legacy TIMESTAMP / TIMESTAMPL packed number 直接加 86400：这不是“加一天”的可靠写法，应使用对应时间戳 API 或转换语句。
  • 在单元测试里直接依赖当前时间：测试随运行时刻变化，应该注入 timestamp 或 clock provider。
