Chapter 01 - SAP 和 ABAP 基础

22. TYPE vs LIKE 类型声明方式

  • Chapter 01
  • Section 22
  • self-contained study note

本节结论

TYPE 是“用一个独立的类型来声明数据对象”，LIKE 是“照某个已经可见的数据对象或字段的类型形状来声明数据对象”。初学者最重要的判断规则是：新代码和接口边界优先用 TYPE 指向稳定类型；只有当一个局部变量确实依赖另一个数据对象的形状时，才考虑 LIKE，并且要让这种依赖在代码 review 中说得清楚。

正文

上一节已经解释了 Elementary data types 基础技术类型，但真实 ABAP 代码不会只写 TYPE i、TYPE string。项目里更常见的是：字段来自 DDIC data element，结构来自 Dictionary structure，内表来自 table type，方法参数来自全局 class/interface 的 public type，或者局部变量想跟某个已有参数保持同样类型。这时就会遇到 TYPE 和 LIKE。

先看 TYPE。TYPE 后面放的是 data type 类型名。这个类型可以是 ABAP built-in type，例如 i、string、p LENGTH ... DECIMALS ...；也可以是当前程序里用 TYPES 定义的本地类型；也可以是全局 class / interface 暴露出来的 public type；还可以是 ABAP Dictionary 中可作为类型使用的对象，例如 Data element 数据元素、DDIC structure、table type。CDS view type 等后面章节会继续展开，本节先把声明方式的判断规则学稳。SAP Learning 还提醒：database table 作为类型使用时，表示的是该表的行结构；这种写法在旧代码里常见，但新代码如果不是明确要模拟数据库表行，通常不应直接把数据库表名当作通用结构类型，因为它会让程序变量和物理表定义过度耦合。

再看 LIKE。LIKE 后面不是一个抽象类型名，而是一个在当前位置可见的数据对象或字段。DATA lv_save LIKE iv_amount. 的意思不是“lv_save 指向 iv_amount”，也不是 Java/Python 里的引用别名；它只是在声明时借用了 iv_amount 的类型属性，声明出来的 lv_save 仍然是独立的 ABAP data object 数据对象。这点非常重要：LIKE 是声明期的类型继承，不是运行时的数据引用。真正的数据引用、对象引用要到 REF TO、data reference、field symbol 等后面章节再学。

为什么 ABAP 还需要 LIKE？一个典型场景是 helper variable。假设方法参数 iv_amount 的类型以后可能从一个 data element 改成另一个更合适的 data element，那么局部备份变量 DATA lv_amount_save LIKE iv_amount. 会跟着参数变化。另一个常见场景是 work area：DATA ls_item LIKE LINE OF lt_items. 明确表达“这一行变量依赖这张内表的行类型”。这类 LIKE 是有意建立依赖，项目 review 时可以解释。

但是 LIKE 也很容易被滥用。旧 ABAP 代码里经常看到 DATA lv_matnr LIKE mara-matnr. 这种写法，它是有效且需要能读懂的 legacy form：mara-matnr 是 DDIC 表字段，LIKE 借用了该字段类型。可是在新代码中，如果真正想表达的是“物料号这个业务字段”，更清晰的写法通常是指向对应 data element，例如 DATA lv_matnr TYPE matnr.。这样读者看到的是稳定语义类型，而不是“这个变量碰巧像 MARA 的某个字段”。Joule 和 SAP 指南都强调：如果数据对象直接依赖另一个数据对象，用 LIKE；其他情况用 TYPE 指向 standalone data type。

还要特别区分 data element 和 domain。Domain 域对象 是 DDIC 中定义技术属性、取值范围等底层规则的对象，但它本身不是 ABAP 声明里的类型名。ABAP 编译器不会把 domain name 当作合法 TYPE 目标；正确路径是引用基于该 domain 的 data element。换句话说，domain 是 DDIC 的取值/格式基础，data element 才是带技术属性和语义含义的可用类型入口。

最后，把 inline declaration 放在正确位置理解。DATA(result) = method( ).、SELECT ... INTO TABLE @DATA(lt_result) 这类写法是在编译期从表达式、方法签名或 SQL 结果静态推断类型，不是运行时动态类型，也不是 LIKE 的另一种写法。它们能让代码更短，但如果推断结果不明显，项目 review 仍可能要求显式 TYPE 或补充变量命名，让类型来源更容易追踪。

为什么重要

  • TYPE / LIKE 决定了变量、参数、work area、内表行的类型来源，也决定了以后改 DDIC、改接口、改方法签名时，哪些变量会自动跟随、哪些变量需要手工调整。
  • 用 TYPE 指向 data element 或全局 type，读者能看到业务语义；滥用 LIKE dbtab-field 会让类型来源分散到数据库表字段上，影响 review、重构和 Clean Core 判断。
  • 用 LIKE 表达直接依赖时很有价值，例如 helper variable 跟随方法参数、work area 跟随内表行；但如果依赖关系不真实，它就会变成隐藏耦合。
  • 对 Java/Python 背景学习者来说，最大误区是把 LIKE 理解成 runtime reference。ABAP 的 LIKE 只是声明方式，不会让两个变量共享同一块内存。

核心机制

  • TYPE type：引用一个 standalone data type，适合新代码、接口、方法签名、全局可复用类型、DDIC data element / structure / table type。
  • LIKE dobj：引用一个当前位置可见的数据对象或字段的类型属性，适合局部 helper、work area、LIKE LINE OF 等“真实依赖另一个对象”的场景。
  • LIKE dbtab-field：旧代码和维护项目里常见且需要能读懂；新代码通常优先找对应 data element，用 TYPE 表达业务语义。
  • TYPE domain_name：不要这样写。domain 不是 ABAP 声明里的类型名；要引用基于 domain 的 data element。
  • DATA(var) / @DATA(var)：属于 inline declaration 和静态型推论，不等于 LIKE，也不表示 ABAP 变成动态类型语言。
  • Field symbol、generic formal parameter、data reference 与 LIKE 的交互有额外规则；Day 03 只需知道不要把它当作初学写法，后面动态编程章节再展开。

日本项目现场表达

  • 常见日语：TYPE指定、LIKE指定、型参照、データオブジェクトの型継承、スタンドアロン型、型依存性。
  • DDIC 相关：データ要素、ドメイン（DDICのドメイン）、構造型、テーブル型、項目定義、技術属性、意味情報。
  • 代码 review 相关：インライン宣言、静的型推論、暗黙型推論、開発標準、保守性、影響範囲。
  • 说明句：この変数は入力パラメータに直接依存するため、TYPE ではなく LIKE 指定で型依存性を明示します。
  • Review 句：この LIKE mara-matnr は保守性の観点で型依存性が高いため、可能であればデータ要素 MATNR を TYPE 指定してください。
  • 注意句：ここでいうドメインは DDIC のドメインであり、DDD のビジネスドメインとは意味が異なります。

English Terms

  • Official / search term: TYPE

  • Official / search term: bound data type

例子

先看一组可以在 ADT 练习类里阅读的声明：

DATA lv_travel_id TYPE /dmo/travel_id.
DATA lv_travel_id_copy LIKE lv_travel_id. "directly depends on lv_travel_id

DATA lt_travel_keys TYPE /dmo/t_travel.
DATA ls_travel_key  LIKE LINE OF lt_travel_keys.

DATA lv_carrier_id TYPE /dmo/carrier_id.

lv_travel_id 和 lv_carrier_id 用 TYPE，因为它们要表达稳定的业务语义：旅行 ID、航空公司 ID。lv_travel_id_copy 用 LIKE lv_travel_id，因为它只是一个局部 helper，明确依赖上面的变量。ls_travel_key 用 LIKE LINE OF lt_travel_keys，因为它是内表行 work area，真正依赖 lt_travel_keys 的行类型。

再看两个 review 中常见的对比：

"Legacy form: valid and commonly seen in old code
DATA lv_matnr_old LIKE mara-matnr.

"New-code preference when a semantic DDIC type exists
DATA lv_matnr TYPE matnr.

第一种写法需要能读懂，但它把变量类型绑到某个数据库表字段上。第二种写法直接告诉读者：这是 material number 的 data element 语义。日本项目中做保守改修时，遇到第一种不要急着全局替换；新写代码或重构时，优先问“有没有更稳定的 data element / class type / interface type 可以作为 TYPE 目标”。

错误练习：

"Wrong mental model: domain is not a DATA TYPE target
DATA lv_status TYPE some_domain_name.

如果 some_domain_name 是 DDIC domain，这不是“风格不好”，而是类型目标不成立。应使用基于该 domain 的 data element，或者使用项目定义的全局 type。

常见误区

  • 把 LIKE 理解成引用：以为 lv_b LIKE lv_a 会让两个变量共享值，结果误解赋值、清空和调试行为。
  • 旧代码里看到 LIKE mara-matnr 就盲目模仿：变量类型被绑到数据库表字段，业务语义和可维护性都变差。
  • 用 domain name 写 TYPE：把 DDIC domain 当作 ABAP 类型名，语法层面就不成立。
  • 在方法签名和公共 API 上大量使用偶然字段形状：调用方难以理解语义，后续 RAP/OData/CDS 映射也不清楚。
  • 过度使用 inline declaration：代码短了，但 review 时看不出变量应当承载的业务语义，尤其在金额、数量、日期、key 字段上风险更高。
  • 真实依赖关系不用 LIKE：helper variable 本来应跟随参数类型，却手工写了另一个 TYPE，以后参数类型变更时局部变量可能被遗漏。

思考题

  • DATA lv_a TYPE matnr. 和 DATA lv_a LIKE mara-matnr. 分别把类型来源交给了谁？
  • 如果一个局部变量只是保存输入参数的旧值，用 TYPE 和 LIKE 哪个更能表达依赖关系？为什么？
  • 为什么 domain 不是 TYPE 的直接目标？data element 在这里多提供了什么？
