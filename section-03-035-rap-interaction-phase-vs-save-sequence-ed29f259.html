35. RAP Interaction Phase vs Save Sequence

  • Chapter 03
  • Section 35
  • self-contained study note

35.1 一句话结论

RAP Interaction Phase vs Save Sequence 是 RAP 事务模型中最重要的职责分界：interaction phase 处理 consumer 请求、BO 行为、transactional buffer 和可提前反馈的业务规则；save sequence 在保存触发后处理保存前检查、late numbering、最终持久化、附加保存和 cleanup。不能把两者都笼统叫“save phase”，否则很容易把逻辑放错位置。

一句话记忆：interaction phase 回答“用户/接口想做什么，BO 当前允许怎么变”；save sequence 回答“这些变化通过检查后，如何可靠保存和清理”。

35.2 概念说明

RAP 把 BO 修改分成有节奏的生命周期。外部 consumer 通过 OData、Fiori、EML 或测试代码发起请求后，首先进入 interaction phase。这里通常发生：

  • create/update/delete/read/action/function 请求处理。
  • authorization、feature control、lock、instance feature 判断。
  • on modify determination。
  • action handler 修改 transactional buffer。
  • unmanaged BO 中的标准 operation buffer 处理。
  • 使用 READ ENTITIES ... IN LOCAL MODE / MODIFY ENTITIES ... IN LOCAL MODE 读取或修改当前 BO buffer。

这些逻辑大多在 Handler class 处理器类 中实现，目标是响应请求、调整 BO buffer、返回 failed / reported / mapped / result 等反馈。此时通常还没有真正落库。

当 consumer 触发 COMMIT ENTITIES、OData Save 或 draft Activate 后，RAP runtime 进入 Save sequence 保存序列。这里通常发生：

  • FOR VALIDATE ON SAVE / FOR DETERMINE ON SAVE 等保存前相关 handler method。
  • finalize() / check_before_save() 等保存前步骤。
  • point-of-no-return 前后的边界处理。
  • late numbering / adjust_numbers()。
  • managed save、unmanaged save()、managed with unmanaged save 的 save_modified()。
  • additional save。
  • cleanup() / cleanup_finalize()。

这不是两个互不相关的世界，而是一条生命周期。Interaction phase 产生“准备保存的 BO 状态”；save sequence 决定这个状态是否能最终持久化、编号、写附加数据并清理。

可以把 interaction phase 想成“把用户意图转成 BO 状态变化”的阶段。它更接近业务对话：用户按了哪个 action、传入了哪些字段、当前对象是否允许这个动作、应该返回什么消息、buffer 中应变成什么状态。Save sequence 则更接近“把已经形成的状态变化可靠落地”的阶段：保存前再检查、分配最终编号、执行持久化、追加保存成功相关证据、清理临时数据。一个好的 RAP 设计，会让这两段像接力一样衔接，而不是互相抢职责。

初学者读代码时，可以先问一句：这段逻辑是在“决定变化”，还是在“保存变化”？如果是在决定变化，通常更靠近 handler、action、determination、validation、local EML；如果是在保存变化，通常更靠近 saver、save_modified、additional save、outbox/event、cleanup。这个问题不能替代官方文档，但能帮助你在陌生 behavior pool 中快速定位责任。

35.3 为什么重要

这个分界会直接决定代码写在哪里。一个审批按钮的状态检查和状态变更，通常属于 action handler / interaction phase；审批保存后写审计日志，可能属于 additional save；保存前检查日期是否合法，通常是 validation；最终把 unmanaged buffer 写表，属于 save sequence；清理静态 buffer，则属于 cleanup。

如果分不清这两个阶段，代码会出现典型项目问题：action handler 直接写库；validation 没有返回字段级消息；保存失败后旧 buffer 残留；save_modified 被当成普通 helper；external API 在 handler 中被提前调用；业务错误拖到 point-of-no-return 后才发现。

对项目交付来说，这个分界能让设计评审变得清楚。评审不只问“有没有实现”，还要问：“这个逻辑为什么放在 handler，而不是 saver？”“为什么在 save sequence 才编号？”“这个错误能不能提前到 validation？”“cleanup 是否覆盖失败路径？”

对 Java/Python 背景学习者，可以把 interaction phase 类比成 application service / domain command handling，把 save sequence 类比成 framework-managed unit-of-work commit。但 RAP 的关键差异是：method signature、BDEF 声明、response table、OData/Fiori runtime 和 SAP LUW 都参与这个生命周期。

这节也解释了为什么前一节 Direct INSERT 反模式 危险。Direct INSERT 的本质不是“SQL 语句丑”，而是把 save sequence 的工作提前偷放到 interaction phase。这样一来，validation、late numbering、authorization、lock、reported message、draft activation、cleanup 还没有走完，数据库或外部副作用却已经发生。理解 interaction/save sequence 的分界，就能看出很多“能跑”的代码为什么不可交付。

在团队协作中，这个分界还能减少争论。开发者说“这个逻辑我放 handler 里了”，reviewer 应继续问：它只是改变 buffer 和返回消息，还是已经写了持久表/外部系统？开发者说“这个逻辑在 save 里”，reviewer 应继续问：它是不是太晚才发现可预见的业务错误？这些问题让 review 从风格争论变成生命周期判断。

35.4 边界

第一条边界：interaction phase 不等于“可以立即写库”。它可以修改 RAP transactional buffer，但不应直接 INSERT / UPDATE 持久表并 COMMIT WORK。真正持久化应由 save sequence 统一处理。

第二条边界：save sequence 不等于“所有业务逻辑都放最后”。可预见的业务错误越早反馈越好。UI 字段错误、权限不允许、状态不合法、主数据缺失，通常应在 precheck、authorization、validation 或 check-before-save 之前/期间处理，而不是拖到 save()。

第三条边界：handler method 不一定都只在普通 interaction 时触发。FOR VALIDATE ON SAVE 和 FOR DETERMINE ON SAVE 仍是 handler methods，但它们在 save sequence 的 check-before-save 相关阶段触发。不要简单用“handler = interaction only”概括所有情况。

第四条边界：saver class 不等于 transaction owner。即使进入 saver class，也不能随便 COMMIT WORK / ROLLBACK WORK。RAP runtime 仍然拥有事务边界。

第五条边界：additional save 不等于 action side effect。保存时写日志、业务事件、扩展表等应考虑 additional save、business event 或 outbox；不要因为 action 被点击就在 action handler 中提前发不可回滚副作用。

第六条边界：draft Save / Prepare / Activate 也要按生命周期看。Draft 允许用户分阶段编辑，但 active 数据最终仍要经过保存前检查、编号和保存序列。不要把 draft 当成跳过 validation 或保存序列的捷径。

第七条边界：interaction phase 可以返回业务消息，但不应该把所有最终证据都写死。比如 action handler 可以告诉用户“请求已被接受到 buffer”，但如果最终保存失败，就不应提前留下“业务已完成”的审计记录。

第八条边界：save sequence 可以写最终数据，但不应该承担所有用户体验。字段级错误、权限错误、明显状态错误如果拖到很晚才发现，Fiori/OData consumer 往往只能得到粗糙错误，测试也难以定位。

第九条边界：cleanup 不是“可有可无的扫尾”。如果 BO 实现中有静态 buffer、临时映射、旧 API wrapper 状态、outbox 暂存或批量处理上下文，成功和失败后的 cleanup 都影响下一次请求是否被污染。

35.5 Key mechanism / 核心机制

第一层机制是 lifecycle trigger。Interaction phase 由 consumer 的 request 触发，例如 MODIFY ENTITIES、OData PATCH、Fiori action button。Save sequence 由 commit/save 触发，例如 COMMIT ENTITIES、OData Save、draft Activate。

第二层机制是 buffer。Interaction phase 的主要成果是 transactional buffer 中的 BO 状态变化。Handler 可以通过 local EML 读取和修改这个 buffer。Save sequence 则消费这个 buffer，决定是否持久化。

第三层机制是 response。Interaction phase 常见响应包括 failed、reported、mapped、result；save sequence 也会影响这些响应，尤其 create/numbering、validation failure、reported messages 和 OData error payload。测试不能只看“方法执行完”，要看 response table。

第四层机制是 method placement。可以用一个简化判断表：

需求更可能的位置
action 改状态interaction phase / action handler
字段变化后自动重算on modify determination
保存前业务规则检查validation / check-before-save
创建时默认值interaction phase determination
生成 UUID technical keymanaged/early numbering
保存前分配业务单号late numbering / adjust_numbers()
unmanaged 主保存saver save()
managed 主保存改走旧 APIsave_modified() with unmanaged save
managed 主保存后追加日志additional save

第五层机制是 point-of-no-return。Interaction phase 和保存前阶段都适合发现并返回可理解业务错误；越过 point-of-no-return 后，错误处理和回滚解释会变难。因此项目应尽量“早检查、晚副作用”。

第六层机制是 review checklist。每个复杂逻辑都问四个问题：

  • 它依赖当前未保存的 buffer 状态吗？如果是，是否用 local EML？
  • 它会写数据库或外部系统吗？如果是，是否应该放到 save sequence / event / outbox？
  • 它失败时需要字段级消息吗？如果是，是否该放 validation / reported？
  • 它是否需要在保存成功后才发生？如果是，不应放 interaction phase。

这张 checklist 的用法是：不要只问“代码能不能跑”，而要问“失败时 runtime 还能不能解释”。如果一段 handler 代码同时修改业务状态、直接写持久表、发送外部请求、又试图返回字段消息，通常说明它把 interaction phase、save sequence 和 additional save 的职责混在一起了。项目 review 时应把这些动作拆开，分别放到 handler、validation、saver、additional save、business event 或 cleanup 中。

可以进一步把放置决策写成一个流程：

Does the logic only compute or validate user input?
  -> interaction phase determination / validation / feature control

Does it modify the current BO state?
  -> handler/local EML/transactional buffer

Does it persist unmanaged BO data?
  -> saver save() or save_modified(), depending on BDEF strategy

Does it write audit/log data that must be consistent with the main save?
  -> additional save or same save boundary

Does it call an external system or trigger an irreversible side effect?
  -> outbox/event/job after successful save, with idempotency

Does it only release memory or temporary state?
  -> cleanup / cleanup_finalize

这个流程的价值在于“先分类再写代码”。不要看到 action button 就把所有逻辑塞进 action handler，也不要看到 save sequence 就把所有规则拖到最后。RAP 的强项就是把不同类型的逻辑放到可理解的位置。

35.6 日本项目现场说法

常见日语：

  • インタラクションフェーズ：interaction phase。
  • 保存シーケンス：save sequence。
  • 保存前チェック：保存前检查。
  • トランザクショナルバッファ：transactional buffer。
  • セーバークラス：saver class。
  • ハンドラクラス：handler class。
  • ポイント・オブ・ノーリターン：point-of-no-return。
  • 追加保存：additional save。

说明句：

この処理はユーザ操作に対する BO の状態変更なので、interaction phase の handler method に実装します。

意思是：这个处理是对用户操作的 BO 状态变更，因此实现到 interaction phase 的 handler method。

DB への最終保存や追加ログ登録は save sequence 側で設計します。

意思是：数据库最终保存和附加日志登记在 save sequence 侧设计。

评审句：

このロジックは interaction phase と save sequence のどちらに置くべきか、失敗時の戻し方も含めて確認してください。

意思是：请确认这段逻辑应该放在 interaction phase 还是 save sequence，并包含失败时如何恢复。

35.8 Example / 学习例子

例子一：审批 action。

用户点击 Accept。Action handler 检查当前状态、authorization、feature control，使用 local EML 把状态改成 Accepted，并返回 action result。此时通常只是 buffer 变化。用户保存或 OData flow 提交后，save sequence 才验证、编号、落库和 cleanup。

例子二：保存时写日志。

需求是“Travel 保存成功后写一条审计日志”。这不应放在 action handler 中提前 INSERT zlog。如果主数据保存失败，日志不应独立存在。更合理的设计是 additional save、business event/outbox，或与 save sequence 同步的日志表写入，并设计幂等。

例子三：日期错误。

用户输入 BeginDate > EndDate。这个错误应在 validation / check-before-save 相关阶段返回字段级消息，而不是等到 saver save() 写库时报数据库错误。这样 Fiori/OData consumer 能清楚知道哪一行哪一字段错了。

例子四：AC0567C POC 上传。

上传 API 的“解析 Excel 行、检查字段、准备 BO 修改”类似 interaction phase 思维；“最终写入、生成编号、写日志、失败回滚、清理缓存”类似 save sequence 思维。即使不是 RAP BO，也可以借这个分界重新审视批量上传设计。

可以把 AC0567C 的上传处理拆成模板：

Interaction-like stage
  - parse input and normalize rows
  - cheap precheck: required fields, file format, caller authorization
  - create or update batch buffer/status intention
  - return row-level reported messages where possible

Check-before-save stage
  - validate business rules with current master data
  - check duplicate rows and status transition
  - reject invalid rows before irreversible effects

Save sequence-like stage
  - persist accepted batch/header/item changes
  - assign final business number if needed
  - write application log or summary that matches the final result

Post-save/outbox stage
  - call downstream system asynchronously if needed
  - record request id and idempotency key
  - retry or compensate without changing the original action result silently

Cleanup stage
  - clear temporary parse buffers and mapping tables
  - preserve audit evidence, remove only implementation scratch state

这份模板让读者看到：RAP 生命周期不是只服务 RAP 语法，也是一种项目设计思维。即使某个批处理暂时还是 classic ABAP，也可以用 interaction/save sequence 的分界减少“先写一点表、失败再说”的风险。

35.9 Counterexample / Failure mode

反例一：action handler 直接 UPDATE ztable。后续 save sequence validation 失败，数据库却已变更。修正方式是 handler 只改 BO buffer。

反例二：把附加日志写在 interaction phase。用户触发 action 但最后取消或保存失败，日志却存在。修正方式是把附加副作用移到 save sequence、business event 或 outbox。

反例三：把所有错误拖到 save sequence 的 save()。UI 得不到字段级消息，编号可能已消耗。修正方式是能早检查的规则提前。

反例四：以为 FOR VALIDATE ON SAVE 不是 handler。结果找不到实现位置。修正方式是记住：它是 handler method，但在保存序列的 check-before-save 相关阶段触发。

反例五：在 saver class 中 COMMIT WORK。开发者以为“都保存阶段了可以提交”，但 RAP runtime 才是 transaction owner。修正方式是移除内部 commit，使用 RAP 允许的保存 API 和 no-commit 旧 API。

反例六：cleanup 只考虑成功路径。失败路径 buffer 未清，下一次请求串数据。修正方式是同时考虑 cleanup() 与 cleanup_finalize()。

反例七：把 outbox/event 放在 interaction phase。用户 action 还没有最终保存成功，下游系统已经收到消息。修正方式是保存成功后再发布或由 outbox job 处理。

反例八：把 expensive validation 全部放在 action handler。用户每次输入都触发重检查，锁持有时间变长，体验变差。修正方式是区分 cheap precheck、on-save validation 和最终保存前检查。

反例九：在 additional save 中发现本可提前发现的业务错误。此时主保存语义已经很接近 point-of-no-return，错误处理复杂。修正方式是把可预见业务错误前移到 validation / check-before-save。

反例十：测试只覆盖 happy path。没有保存失败、重复点击、draft cancel、late numbering failure、external side effect failure、cleanup 后再执行等场景，就无法证明生命周期设计正确。

35.10 思考问题

  • 这个逻辑需要立刻反馈给用户，还是保存成功后才应该发生？
  • 它是修改 BO buffer，还是最终写数据库/外部系统？
  • 它失败时是否需要字段级消息？
  • 它是否依赖尚未保存的 transactional buffer？
  • 它是否越过 point-of-no-return 后才发现可预见错误？
  • 它是否有 COMMIT WORK / ROLLBACK WORK 或不可回滚副作用？
  • 在日本项目设计书中，这段逻辑应写在 handler method、save sequence、additional save 还是 business event？
