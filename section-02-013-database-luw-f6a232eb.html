Chapter 02 - Core ABAP Programming

13. Database LUW 数据库事务

  • Chapter 02
  • Section 13
  • self-contained study note

13.1 一句话结论

Database LUW 是数据库层的 logical unit of work，也就是 database transaction：从一次数据库事务开始，到 database commit 或 database rollback 结束，中间的数据库修改要么整体持久化，要么整体撤销。在 ABAP 里学习 Database LUW，重点不是把它背成“数据库事务”四个字，而是理解它和 SAP LUW 的边界：database commit/rollback 会关闭 database cursor、释放 database locks、结束当前 database LUW；而 SAP 业务一致性通常还需要 update task、enqueue lock、RAP save sequence 和正确的 transaction owner。

13.2 到底学什么

先把边界说清楚：Database LUW 是数据库系统层面的原子性边界，不是 SAP 应用层的完整业务边界。它回答的是：

哪些数据库修改已经在同一个 database transaction 里？
什么时候 database commit 让它们持久化？
什么时候 database rollback 撤回尚未提交的修改？
commit / rollback 对 cursor 和 lock 有什么影响？

而 SAP LUW 回答的是：

这次业务动作从哪里开始，到哪里结束？
谁负责 COMMIT WORK / ROLLBACK WORK？
update task、enqueue lock、BAPI、RAP buffer 如何一起保持业务一致性？

在 AS ABAP 中，work process 通过数据库接口访问数据库。官方文档强调：一个 work process 一次只能执行一个 Database LUW，不能干预其他 work process 的 Database LUW。ABAP 程序运行过程中可能切换 work process，因此 Database LUW 可能因为显式或隐式 database commit / rollback 被切开。初学者如果只用普通 Java/Python 的 ORM transaction 经验理解 SAP，很容易误判。

一个最小的 database commit / rollback 心智模型如下：

Database LUW starts
  -> INSERT / UPDATE / MODIFY / DELETE
  -> more database operations
  -> database commit: persist changes and end the Database LUW
or
  -> database rollback: undo uncommitted changes and end the Database LUW

但在 ABAP 项目里，不要把“我想结束 database transaction”直接等同于“我可以在这里 COMMIT WORK”。COMMIT WORK 不只是 database commit，它还会结束当前 SAP LUW 并触发 update task 等 SAP 语义。反过来，如果只想理解数据库层事实，就先看 Database LUW；如果要决定业务提交点，就回到 SAP LUW 和 transaction owner。

13.3 为什么重要

Database LUW 是所有写入安全的底座。没有它，INSERT、UPDATE、DELETE 都会变成不可控的半成品；但只懂它也不够，因为 SAP 业务事务往往比一个数据库 transaction 更长、更复杂。

第一层价值是理解“已提交”和“未提交”。如果一条数据已经 database commit，再执行 rollback 不会把它自动带回之前状态。事故排查里，经常有人说“程序后来 rollback 了，为什么数据还在？”答案通常是：它已经在更早的 Database LUW / SAP LUW / update task / BAPI commit / 外部系统调用中提交了。

第二层价值是理解 implicit commit。ABAP 文档说明，AS ABAP 中 database commits 可以显式触发，也可以隐式触发。典型例子包括 classic dynpro dialog step 完成导致 work process 切换、某些 RFC 调用、ICF 通信等。也就是说，程序不一定只有你写了 COMMIT WORK 才发生 database commit。

第三层价值是理解 cursor。database commit 和 database rollback 都会关闭打开的 database cursors。SELECT ... ENDSELECT 背后依赖 cursor；OPEN CURSOR / FETCH 更明显。如果在 SELECT loop 中间触发 commit 或 rollback，后续继续读取可能出错或行为不可预期。很多初学者第一次遇到这个坑，是在循环处理数据时每行 COMMIT WORK。

第四层价值是理解 database lock。数据库系统会为数据修改设置数据库锁，但这些锁最迟会在下一次 database commit 或 rollback 时释放。一个 SAP 业务动作如果跨多个 dialog steps 或多个 Database LUWs，只靠 database lock 保护不了完整业务对象。此时需要 SAP lock object / enqueue lock 或 RAP lock。

第五层价值是理解性能和批处理。批处理里每一行 commit 会造成大量 database commits，也会让失败后整体 rollback 不可能；一次提交太多又可能锁太久、日志太大、恢复困难。真正的设计要在业务一致性、错误恢复、锁持有时间、日志和性能之间平衡。

13.4 Key mechanism / 核心机制

13.4.1 Database LUW 是数据库 transaction

可以把 Database LUW 直接理解成数据库 transaction。SAP Samples 的 SAP LUW cheat sheet 把它描述为 SAP-independent 的数据库一致性机制：数据库操作组成一个不可分割的序列，最终由 database commit 持久化，或由 database rollback 撤回。

最小例子：

INSERT zrequest_state FROM @ls_state.

UPDATE zrequest_state
   SET status = @lc_status_finished
 WHERE request_id = @ls_state-request_id.

这两条语句在当前数据库连接和当前 Database LUW 中产生未提交修改。后续如果触发 database commit，它们被持久化；如果触发 database rollback，它们在这个 Database LUW 内尚未提交的部分被撤回。

但项目里要继续追问：谁触发 commit？是否同时结束 SAP LUW？是否有 update task？是否有 implicit commit？是否已经有外部副作用？这就是 Database LUW 和 SAP LUW 的分工。

13.4.2 Database commit：持久化并结束当前 Database LUW

database commit 的效果是把当前 Database LUW 中的修改写入数据库，并结束该 Database LUW。ABAP 里可以通过多种方式触发 database commit：

显式：COMMIT WORK、COMMIT CONNECTION、DB_COMMIT 等。
隐式：work process 切换、dialog step 完成、某些 RFC / ICF 场景等。

普通业务代码里最常见的是：

COMMIT WORK.

但这句不只是 database commit。它还会结束当前 SAP LUW，触发 update task 和相关 SAP 事务机制。所以本节的安全表述是：

COMMIT WORK includes a database commit effect,
but it is not only a database commit.

如果只是讨论数据库连接层，ABAP 文档还提到 COMMIT CONNECTION / DB_COMMIT 等入口；但初学者不要把它们当成业务代码里的常规提交方式。业务提交点通常要由外层 SAP LUW / framework / BAPI / RAP 事务模型决定。

13.4.3 Database rollback：撤回未提交修改并结束当前 Database LUW

database rollback 撤回当前 Database LUW 中尚未提交的数据库修改，并结束该 Database LUW。ABAP 中它也可能显式或隐式触发：

显式：ROLLBACK WORK、ROLLBACK CONNECTION、DB_ROLLBACK 等。
隐式：runtime error、termination message 等。

普通项目里常见的是：

ROLLBACK WORK.

但同样要小心：ROLLBACK WORK 不只是 database rollback。它会结束当前 SAP LUW，清理当前 SAP LUW 的相关 registrations，并产生 SAP 事务语义。官方文档也提示，如果只是为了单纯 database rollback，ROLLBACK WORK 不是唯一也不总是最合适的概念；而在业务代码里，真正的问题通常不是“怎样回滚数据库连接”，而是“这个业务事务应该由谁回滚”。

最重要的限制是：

rollback 只能撤回当前 Database LUW 中尚未提交的数据库修改。
它不能撤回之前已经 commit 的内容。
13.4.4 Implicit database commit：不是所有 commit 都写在代码里

AS ABAP 使用 work processes 连接数据库。一个 work process 一次只能执行一个 Database LUW；如果程序运行导致 work process 切换，当前 work process 的 Database LUW 需要结束，因此会出现 implicit database commit。

典型场景包括：

  • classic dynpro dialog step 完成，程序等待用户下一次动作。
  • 调用某些同步或异步 RFC，控制权交给其他 work process 或系统。
  • 某些 HTTP / ICF 通信边界。

这不是要求初学者背完所有隐式 commit 场景，而是建立警觉：如果代码运行在 classic dialog、RFC、ICF、update task、background job 或框架回调中，事务边界可能不只是你看到的几行 SQL。

13.4.5 Database cursor：commit / rollback 会关闭 cursor

database commit 会关闭打开的 database cursors；database rollback 也会关闭打开的 database cursors。官方文档特别提醒，不要在 SELECT loop 或 OPEN CURSOR 后意外触发 commit / rollback。

危险写法：

SELECT FROM zrequest_state
       FIELDS request_id, status
       WHERE status = @lc_status_open
       INTO @DATA(ls_state).

  UPDATE zrequest_state
     SET status = @lc_status_processing
   WHERE request_id = @ls_state-request_id.

  COMMIT WORK. " Dangerous inside a SELECT loop.

ENDSELECT.

问题不是“每行提交一定永远错”，而是这里的 commit 会关闭 SELECT loop 背后的 cursor，后续继续读可能产生 runtime error 或不可控行为；同时每行 commit 也会造成部分成功和恢复困难。

更稳的入门写法：

SELECT FROM zrequest_state
       FIELDS request_id, status
       WHERE status = @lc_status_open
       INTO TABLE @DATA(lt_state).

LOOP AT lt_state INTO DATA(ls_state).
  UPDATE zrequest_state
     SET status = @lc_status_processing
   WHERE request_id = @ls_state-request_id.
ENDLOOP.

COMMIT WORK.

真实批处理还要进一步设计包大小、日志、错误行、重试和锁。这里只先学一个底线：不要在依赖 database cursor 的读取循环中随手 commit / rollback。

13.4.6 Database locks：生命周期短，不是业务锁

数据库系统会在修改数据库表记录时设置 database locks。SELECT ... FOR UPDATE 这类语义也属于数据库层锁思路。Database lock 的特点是：

  • 通常由数据库系统自动管理。
  • 最迟在下一次 database commit 或 database rollback 时释放。
  • 主要保护当前 Database LUW 内的数据一致性。
  • 不能自动表达“这个 SAP 业务对象正在被用户编辑”。

所以，database lock 更像短时间的数据库一致性保护；SAP enqueue lock 更像业务对象级别的并发协议。一个用户在画面上编辑订单几分钟，不能靠 database lock 一直保护。要保护整个业务动作，需要 lock object / enqueue 或 RAP locking。

13.4.7 Database LUW 与 SAP LUW 的组合关系

上一节讲过：SAP LUW 是业务一致性边界，Database LUW 是数据库层提交/回滚边界。它们的关系可以这样记：

一个 Database LUW 通常属于某个数据库连接 / work process 的当前事务。
一个 SAP LUW 可能为了业务一致性组织多个数据库动作、update task 和 buffer。
SAP LUW 结束时通常会触发一个或多个 Database LUW 层面的提交/回滚效果。

不要反过来说“一个 SAP LUW 就是一个 Database LUW”。在 classic ABAP 和 framework 场景里，这个说法经常不成立。

13.4.8 Direct SQL、update task 与异步边界

直接 ABAP SQL 写入在当前 Database LUW 中产生数据库修改；update task 则先登记，后续由 COMMIT WORK 触发，可能在 update work process 中执行。异步 update 的 database rollback 不一定能撤回其他 work process / 其他 Database LUW 中已经提交的工作。

因此，事务设计要避免把多个互相依赖的变化拆到不同 Database LUWs 里提前提交。能否拆、何时拆、失败后如何补偿，都是设计问题，不是语法问题。

13.4.9 Database LUW 不是授权、不是 Clean Core

Database LUW 只保证数据库层的一致性边界，不会自动做业务授权、状态校验、审计、released API 检查或 Clean Core 判断。即使一次 database commit 很完美，也可能提交了一次业务上不允许的直接标准表修改。

所以，Database LUW 要和这些主题一起看：

  • AUTHORITY-CHECK / CDS DCL / RAP authorization
  • SAP enqueue lock / RAP lock
  • BAPI / released API / RAP BO
  • application log / change document
  • Clean Core / ABAP Cloud restrictions

13.7 Example / 学习例子

13.7.1 例 1：一次简单写入与提交
UPDATE zrequest_state
   SET status = @lc_status_finished
 WHERE request_id = @lv_request_id.

IF sy-dbcnt = 1.
  COMMIT WORK.
ELSE.
  ROLLBACK WORK.
ENDIF.

从 Database LUW 角度看，UPDATE 产生未提交修改；COMMIT WORK 触发 database commit 效果；ROLLBACK WORK 触发 database rollback 效果。从 SAP LUW 角度看，还要继续问：当前程序是否拥有事务边界？是否有 update task？是否有调用方还要继续保存别的对象？

13.7.2 例 2：不要在 SELECT loop 中 commit
SELECT FROM zrequest_state
       FIELDS request_id
       WHERE status = @lc_status_open
       INTO @DATA(lv_request_id).

  UPDATE zrequest_state
     SET status = @lc_status_processing
   WHERE request_id = @lv_request_id.

  COMMIT WORK. " Avoid this pattern.

ENDSELECT.

这里的核心错误有两个：一是 commit 会关闭 cursor；二是每行提交导致部分成功，后续失败时不能整体 rollback。

13.7.3 例 3：先取结果集，再批量处理
SELECT FROM zrequest_state
       FIELDS request_id
       WHERE status = @lc_status_open
       INTO TABLE @DATA(lt_request_id).

LOOP AT lt_request_id INTO DATA(lv_request_id).
  UPDATE zrequest_state
     SET status = @lc_status_processing
   WHERE request_id = @lv_request_id.
ENDLOOP.

COMMIT WORK.

这仍然只是入门示例。真实批处理还需要包大小、错误日志、重试、锁和事务 owner 设计。

13.7.4 例 4：database lock 不能跨完整业务动作
SELECT SINGLE FROM zflight_stock
       FIELDS seats_free
       WHERE flight_id = @lv_flight_id
       FOR UPDATE
       INTO @DATA(lv_seats_free).

" The database lock is tied to the current Database LUW.
" It is released at the latest by commit or rollback.

如果用户还要在后续画面输入乘客信息、确认价格、保存订单，单靠这个 database lock 不够。业务锁应使用 lock object / enqueue 或 RAP locking。

13.8 Counterexample / Failure mode

失败模式一：把 Database LUW 和 SAP LUW 当成同一个东西。结果是以为 database commit 就等于业务动作完整成功。

失败模式二：在 SELECT loop 中 COMMIT WORK。这会关闭 cursor，并把批处理切成很多不可整体回滚的小段。

失败模式三：以为 rollback 能撤回所有内容。rollback 只能撤回当前 Database LUW 中未提交的数据库修改；已经提交或外部化的副作用不自动撤回。

失败模式四：只靠 database lock 保护业务对象。database lock 最迟在 commit/rollback 时释放，不能覆盖用户编辑过程或跨 dialog steps 的业务事务。

失败模式五：忽略 implicit database commit。classic dialog、RFC、ICF、work process 切换等场景可能切开 Database LUW。

失败模式六：在 helper method 里为了“保存一下”而 commit。它不仅结束 Database LUW，还可能结束 SAP LUW，触发 update task 或破坏调用方事务。

失败模式七：把 DB commit 当成权限和合规证明。数据库提交成功不代表这次写入经过授权、业务校验、released API 或 Clean Core 检查。

失败模式八：批处理 commit 粒度没有设计。太频繁导致部分成功和性能差；太大导致锁时间长、日志膨胀和恢复困难。

13.9 思考问题

  • 当前代码讨论的是 Database LUW，还是 SAP LUW？
  • 哪一句代码或哪个框架动作会触发 database commit？
  • 是否存在 implicit database commit 的可能？
  • 如果现在发生 runtime error，哪些未提交数据库修改会 rollback？
  • 是否在 SELECT loop 或 OPEN CURSOR 后触发了 commit / rollback？
  • 当前 database locks 会在什么时候释放？
  • 业务对象是否需要 SAP enqueue lock，而不是只靠 database lock？
  • 已经 commit 的数据如果后续出错，需要通过补偿、冲销、日志还是人工修复处理？
  • 这个 commit 粒度适合批处理、接口重试和生产恢复吗？
