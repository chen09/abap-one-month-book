17. Hashed table 哈希表

  • Chapter 02
  • Section 17
  • self-contained study note

17.1 一句话结论

Hashed table 哈希表，是 ABAP internal table 内部表中面向 unique full-key lookup 的表类别：它用 hash algorithm 通过唯一 key 管理行，适合大表中反复按完整 key 精确读取；但它不适合重复 key、顺序输出、范围读取、前缀 key 查询或依赖行号的处理。

17.2 到底学什么

如果 Standard table 标准表 像 ArrayList / list，Sorted table 排序表 像“按 key 保持顺序的索引结构”，那么 hashed table 更像 Java HashMap / Python dict：你给它一个完整 key，它直接找对应那一行。

但这个类比只能帮助入门，不能替代 ABAP 语义。ABAP hashed table 有几个硬规则：

  • primary key 必须是 unique key。
  • 主要访问方式是 key access，而不是 index access。
  • hash 优势通常要求使用完整 key。
  • 行没有业务排序保证，不能当成按顺序输出的表。
  • 不能存放同一个 primary key 的多行。

最基本声明：

TYPES ty_t_material TYPE HASHED TABLE OF ty_material
  WITH UNIQUE KEY matnr.

或者多字段 key：

TYPES ty_t_price TYPE HASHED TABLE OF ty_price
  WITH UNIQUE KEY sales_org material valid_from.

本节要学的是：hashed table 不是“永远最快的内部表”，而是非常适合“完整唯一 key -> 一行”的查找模型。如果访问模式不是这个，就要考虑 standard table、sorted table 或 secondary key。

17.3 为什么重要

很多 ABAP 性能问题来自在 standard table 上反复线性查找。例如：

LOOP AT lt_items INTO DATA(ls_item).
  READ TABLE lt_materials INTO DATA(ls_material)
    WITH KEY matnr = ls_item-matnr.
ENDLOOP.

如果 lt_materials 是几万行 standard table，这段代码可能很慢。若 matnr 唯一，并且循环中总是按完整 matnr 查材料信息，hashed table 就非常合适：

DATA lt_materials TYPE HASHED TABLE OF ty_material
  WITH UNIQUE KEY matnr.

项目上常见使用场景：

  • 把配置表读入内存后，按配置 key 高频查找。
  • 把物料、客户、供应商等主数据做成 lookup table。
  • 去重：用 unique key 防止重复行进入表。
  • 接口处理中按外部 ID / SAP ID 快速定位一行。
  • 测试或转换逻辑中，用 hash table 模拟字典结构。

但 hashed table 也很容易被误用。如果需求是“同一个 header 下有多条 item”，hashed primary key 就不合适，因为 key 不能重复。如果需求是“按日期范围取一段数据”，hashed table 也不合适，因为 hash key 不支持自然范围顺序。此时 sorted table 的 non-unique key 往往更自然。

17.4 Key mechanism / 核心机制

17.4.1 hash key 必须唯一

hashed table 的 primary table key 是 hash key，而 hash key 必须 unique。因此声明时必须写 WITH UNIQUE KEY ...：

DATA lt_customer TYPE HASHED TABLE OF ty_customer
  WITH UNIQUE KEY customer_id.

下面这种“non-unique hashed table”不是合法设计方向：

" 错误方向：hashed primary key 不能 non-unique
" DATA lt_x TYPE HASHED TABLE OF ty_x WITH NON-UNIQUE KEY id.

如果业务上同一个 key 下可能有多行，应该考虑：

需要按 key 分组且保留多行
  -> sorted table WITH NON-UNIQUE KEY ...

只是顺序处理或小表
  -> standard table

需要一张表同时支持多种访问路径
  -> secondary table key
17.4.2 full key access 才是 hashed table 的主场

hashed table 的优势来自完整 unique key。示例：

READ TABLE lt_customer INTO DATA(ls_customer)
  WITH TABLE KEY customer_id = lv_customer_id.

或者 table expression：

DATA(ls_customer) = lt_customer[ customer_id = lv_customer_id ].

这里的 customer_id 是 hashed table 的完整 key，查找可以利用 hash algorithm。若 key 是多字段：

WITH UNIQUE KEY sales_org material valid_from

就应该用完整组合查找：

READ TABLE lt_price INTO DATA(ls_price)
  WITH TABLE KEY sales_org  = lv_sales_org
                 material   = lv_matnr
                 valid_from = lv_date.

如果只提供 sales_org 或 material，就不再是 hashed table 的理想访问方式。初学者要记住：hashed table 不擅长前缀 key、范围 key、模糊条件或排序输出。

17.4.3 没有 primary table index

standard table 和 sorted table 是 index table；hashed table 不是按 primary index 管理的表。不要用下面这种思路：

" 不要把 hashed table 当成第 N 行容器
" READ TABLE lt_customer INDEX 1 INTO DATA(ls_customer).

使用 hash key 读取时，sy-tabix 也不是可依赖的业务位置。在 ABAP 文档中，hash key 相关读取场景下 sy-tabix 可以是 0。项目代码里不要把 hashed table 的读取结果和行号绑定在一起。

如果确实需要有序遍历或 index-like 访问，就说明 hashed table 可能不是正确主表类别，或者需要 secondary sorted key。

17.4.4 填充：使用 INSERT ... INTO TABLE

hashed table 不适合 APPEND。常见写法是：

INSERT ls_customer INTO TABLE lt_customer.

如果插入重复 key，要立刻处理返回状态或异常语义。不要把重复 key 当成“自动覆盖”。如果业务需要覆盖旧值，要明确写出策略，例如先删除再插入，或使用适合的构造/赋值模式，并在 review 中说明。

17.4.5 table expression 的成功与失败

table expression 很适合 hashed table：

IF line_exists( lt_customer[ customer_id = lv_customer_id ] ).
  DATA(ls_customer) = lt_customer[ customer_id = lv_customer_id ].
ENDIF.

也可以使用 OPTIONAL 避免未找到时报异常：

DATA(ls_customer) = VALUE ty_customer(
  lt_customer[ customer_id = lv_customer_id ] OPTIONAL ).

下一节会继续讲 table expression；在本节先记住：hashed table 很适合这种“按 key 取一行”的写法，但要处理 not found。

17.4.6 管理成本与使用边界

hash algorithm 需要内部管理结构。SAP Help / SAP Samples 都提醒：hashed table 查找很强，但管理成本也高。小表、只遍历一次的表、频繁插入删除但很少 key lookup 的表，不一定适合 hash。

选择 hashed table 时，至少问三件事：

key 是否唯一？
是否总是按完整 key 精确读取？
这张表是否足够大、读取次数是否足够多，值得付出 hash 管理成本？

如果答案不是三次“是”，就不要因为“hash 听起来快”而盲目使用。

常见日语：

  • ハッシュテーブル

  • キーアクセス
  • ハッシュアルゴリズム

  • 検索性能
  • 管理コスト

自然说明句：

  • この内部テーブルは完全キーでの検索が中心なので、ハッシュテーブルを使用します。
  • ハッシュテーブルは一意キーが前提のため、同一キーで複数明細を保持する用途には適しません。
  • 範囲検索やキー順の処理が必要な場合は、ソート済みテーブルを検討します。
  • 重複キーが発生する可能性があるため、登録時のエラーハンドリングを明確にします。
  • 小規模で全件ループのみのテーブルであれば、ハッシュテーブル化しても効果は限定的です。

review 里常见问法：

  • このキーは本当に一意ですか。
  • 検索条件は完全キーになっていますか。
  • 範囲検索があるならソート済みテーブルの方が適切ではありませんか。
  • 重複キー発生時の処理方針はありますか。

17.7 Example / 学习例子

17.7.1 例 1：主数据 lookup table
TYPES: BEGIN OF ty_material,
         matnr TYPE c LENGTH 18,
         mtart TYPE c LENGTH 4,
         text  TYPE string,
       END OF ty_material.

DATA lt_material TYPE HASHED TABLE OF ty_material
  WITH UNIQUE KEY matnr.

INSERT VALUE #( matnr = 'MAT001' mtart = 'FERT' text = `Finished product` )
  INTO TABLE lt_material.

READ TABLE lt_material INTO DATA(ls_material)
  WITH TABLE KEY matnr = 'MAT001'.

IF sy-subrc = 0.
  " found
ENDIF.

这就是 hashed table 的主场：matnr 唯一，而且代码按完整 matnr 高频查找。

17.7.2 例 2：table expression
IF line_exists( lt_material[ matnr = lv_matnr ] ).
  DATA(ls_material) = lt_material[ matnr = lv_matnr ].
ENDIF.

这种写法简洁，但要注意 not found。没有 line_exists、OPTIONAL 或异常处理时，table expression 找不到行会抛出异常。

17.7.3 例 3：多字段完整 key
TYPES: BEGIN OF ty_price,
         sales_org  TYPE c LENGTH 4,
         material   TYPE c LENGTH 18,
         valid_from TYPE d,
         price      TYPE p LENGTH 8 DECIMALS 2,
       END OF ty_price.

DATA lt_price TYPE HASHED TABLE OF ty_price
  WITH UNIQUE KEY sales_org material valid_from.

READ TABLE lt_price INTO DATA(ls_price)
  WITH TABLE KEY sales_org  = lv_sales_org
                 material   = lv_matnr
                 valid_from = lv_date.

这里要用完整 key。如果你经常只按 sales_org material 找“所有价格日期”，hashed table 就不合适；sorted table 的 non-unique / range 访问会更自然。

17.7.4 例 4：不适合一对多明细
" 一个订单有多条 item
" order_id 单独不是 unique key

如果需求是：

order_id = 100
  -> item 10
  -> item 20
  -> item 30

那么 HASHED TABLE WITH UNIQUE KEY order_id 不成立。可以考虑：

DATA lt_items TYPE SORTED TABLE OF ty_item
  WITH NON-UNIQUE KEY order_id item_no.
17.7.5 例 5：去重集合
DATA lt_seen TYPE HASHED TABLE OF ty_key
  WITH UNIQUE KEY id.

LOOP AT lt_input INTO DATA(ls_input).
  INSERT VALUE #( id = ls_input-id ) INTO TABLE lt_seen.
  IF sy-subrc <> 0.
    " duplicate key
  ENDIF.
ENDLOOP.

hashed table 可以用于“已经见过哪些 key”的集合语义。实际项目中要根据插入语句和表类型确认重复 key 的返回/异常处理方式。

17.8 Counterexample / Failure mode

17.8.1 误区 1：把 hashed table 当成万能最快

如果只是全表 LOOP AT，没有 key filter，hashed table 的 hash 查找优势没有发挥，还可能增加管理成本。小表或顺序处理优先考虑 standard table。

17.8.2 误区 2：只给部分 key
DATA lt_price TYPE HASHED TABLE OF ty_price
  WITH UNIQUE KEY sales_org material valid_from.

READ TABLE lt_price INTO DATA(ls_price)
  WITH KEY sales_org = lv_sales_org.

这不是 full-key lookup。hashed table 的优势不能按这个方式发挥。要么提供完整 key，要么重新设计 table category / secondary key。

17.8.3 误区 3：需要重复 key 却选 hashed table

hashed primary key 必须 unique。如果业务是一对多，使用 hashed table 会把模型写错。不要为了速度牺牲业务语义。

17.8.4 误区 4：依赖输出顺序

hashed table 不提供按业务 key 的自然输出顺序。要稳定输出，先转换到 standard table 后 SORT，或直接选择 sorted table。

17.8.5 误区 5：忽略 duplicate key 处理

插入 hashed table 时，重复 key 是业务信号，不是小问题。它可能表示输入重复、主数据不一致、接口幂等键重复或程序逻辑错误。应明确处理，而不是让程序在生产中随机失败。

17.9 思考问题

  • 这张表的 key 是否真的唯一？
  • 读取时是否总是提供完整 key？
  • 是否需要按 key 前缀或范围读取？如果需要，为什么不用 sorted table？
  • 是否需要输出有序结果？如果需要，排序在哪里保证？
  • 这张表大小和读取次数是否足以抵消 hash 管理成本？
  • 重复 key 发生时，是忽略、报错、覆盖，还是记录审计日志？
  • 是否有 secondary key 可以更好支持另一个访问路径？
  • 用日语 review 时，能否说明 完全キー、一意キー、範囲検索不可？
