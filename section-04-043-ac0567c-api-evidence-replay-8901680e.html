43. AC0567C API evidence replay

  • Chapter 04
  • Section 43
  • self-contained study note

43.1 一句话结论

AC0567C API evidence replay 是把 AC0567C POC 的 API 行为整理成可复盘、可审查、可交付的证据包。它不是说“Swagger 能打开”或“我本地调通过”，而是把 endpoint、request、response、后台结果、重复调用、错误路径和剩余风险串起来，让 reviewer 能判断这个接口到底证明了什么、还没有证明什么。

当前本地 POC 的证据很清楚：03_api_docs/AC0567C_ENDPOINTS.md 定义了 5 个 AC0567C C 端点；OpenAPI 文件 rpa01-d00020-screen-to-api.openapi.json 也列出这些路径；04_tests/TEST_EVIDENCE.md 和 04_tests/test-results/ac0567c-no-write-latest.json 证明最新 no-write 测试 8/8 通过；README 记录了早先真实写入测试已经证明 ExecuteImport Status = IMPORTED、InsertedRows = 2、重复同一 ImportId 返回 CONFLICT、GetStatus 返回 IMPORTED / 2。

一句话记忆：API evidence 不是“截图”，而是“调用证据 + 返回证据 + 后台证据 + 风险边界”。AC0567C 目前已经有较好的 no-write 与早先 real-write 证据，但还不能宣称已经具备完整 production evidence，因为 authorization-negative、并发 duplicate、save failure mapping、application log 和 QAS transport 后证据仍然需要补强。

43.2 到底学什么

第一，要学 contract evidence。AC0567C 当前本地 proxy base URL 是：

http://127.0.0.1:4315/sap-proxy

这个 URL 是本机 proxy 入口，不是最终生产 SAP API 地址。证据里必须说明它代表本地可复现路径；如果将来进入 QAS/PRD，还要重新记录真实 system/client、service root、authentication/communication setting 和 transport 后可访问性。

第二，要学 endpoint evidence。AC0567C_ENDPOINTS.md 记录的 5 个 AC0567C C 端点是：

MethodPathPurposeDB update
POST/PriorAllocationImportsC/validate校验 JSON Rows[] batchNo
POST/PriorAllocationImportsC/execute执行 JSON Rows[] importYes, only with AllowRealUpdate = X
POST/PriorAllocationImportsC/upload/validate校验 Base64 CSV/TSV fileNo
POST/PriorAllocationImportsC/upload/execute执行 Base64 CSV/TSV importYes, only with AllowRealUpdate = X
POST/PriorAllocationImportsC/status按 ImportId-derived key 统计状态No write

第三，要学 request evidence。AC0567C 的 JSON batch payload 至少要记录 CompanyCode、CustomerCode、ImportId、SourceFileName、AllowRealUpdate 和 Rows[]。其中 Rows[] 包含 FileLineNo、GroupNo、DocumentAmount、OthersFlag 等字段。证据文档不需要贴完整生产 payload，但必须保留足够字段来解释业务键、错误行、重复检查和后台写入。

第四，要学 response evidence。latest no-write JSON 证明成功 validate 时返回 Status = VALID、Simulation = X、ReceivedRows = 2、ValidRows = 2、InsertedRows = 0、MessageText = AC0567C batch validation OK; no DB update executed。非法 batch 返回 Status = REJECTED、MessageField = DocumentAmount、MessageLineNo = 1、MessageText = DocumentAmount must be numeric。这些字段比 HTTP 200/400 更重要，因为它们告诉调用方应该怎么修正数据。

第五，要学 backend evidence 和边界。早先真实写入测试证明 ImportId = POC-AC0567C-20260612115720 映射到 zzup_dat = 20260612、zzup_tme = 115720，ExecuteImport 插入 2 行，重复同一 ImportId 返回 CONFLICT，GetStatus 看到 2 行。但最新 no-write 8/8 明确跳过 real execute 和 duplicate/idempotency 测试，除非执行脚本加 -AllowRealUpdate。读证据时必须区分“安全可重跑的 no-write evidence”和“会写入 ZCOM_B_ACTT397 的 real-write evidence”。

第六，要学交付证据的脱敏和结论写法。API evidence 可能包含 URL、header、CSRF token、cookie、user、payload、业务键和 response。文档中要遮蔽 token/cookie/password/个人敏感数据，只保留能复盘的业务键、状态、错误字段和结论。每条 evidence 最后都要写 conclusion，例如“validate path 可用，但没有证明真实保存”“real execute 已证明插入和重复拒绝，但没有证明并发安全”。

43.3 为什么重要

项目中最危险的 API 证据，是一句“我本地调用成功了”。这句话没有说明系统、client、用户、URL、payload、response、执行时间、后台状态、日志编号、是否真实写入、是否允许重放，也没有说明失败路径。这样的证据不能支撑测试验收、障害調査、上线判断或客户说明。

AC0567C 的价值在于它已经把 API evidence 做成了一套可以复盘的材料：API docs 可打开，OpenAPI 有路径，PowerShell 脚本可重跑 no-write 检查，latest JSON 保存了每个检查的 result，README 记录了真实写入与重复拒绝的事实，n8n-like flow simulator 也有 smoke 证据。这比“凭记忆说跑过”可靠得多。

但也正因为证据比较完整，读者更要学会边界判断。no-write 8/8 不能证明真实 insert；真实 insert 证明了顺序重复拒绝，不等于证明并发 duplicate safety；status route 能返回 NOT_FOUND 或 IMPORTED，不等于完整 status workflow；XLSX 返回 400 证明“当前不支持”，不等于未来业务不需要。成熟的 API evidence replay 必须同时写清楚 pass 和 not proven。

对初学者来说，这一节把 Week 4 的几个主题连起来：Service definition 服务定义 和 Service binding 服务绑定 决定服务入口；OData V4 基础差异 决定 response 结构；AC0567C RAP saver replay 决定写入阶段；AC0567C authorization replay 决定谁能执行；AC0567C idempotency replay 决定重复请求是否安全；Application Log 应用日志 决定失败后是否能调查。

43.4 Key mechanism / 核心机制

43.1 Contract evidence：证据包先证明“接口合同存在”

AC0567C 的 contract evidence 来自三个地方：

  • 03_api_docs/AC0567C_ENDPOINTS.md：人工可读的 endpoint 表和 JSON/Base64 upload contract。
  • 03_api_docs/rpa01-d00020-screen-to-api.openapi.json：OpenAPI 文档，其中 info.title = RPA01-D-00020 Screen-to-API POC、version = 0.1.8-poc。
  • 03_api_docs/swagger-ui.html、redoc.html、static-api-reference.html：文档页面，TEST_EVIDENCE.md 记录它们经本地 proxy 检查为 HTTP 200。

注意 OpenAPI 文件还包含同一 screen-to-API POC 包中的其他非 AC0567C 路径。本节只讨论 AC0567C C 路径，不能把整个 OpenAPI 都说成 AC0567C 已完成证据。

43.2 Request evidence：每次调用都要能还原上下文

AC0567C no-write 测试脚本 Invoke-Ac0567CBatchPoc.ps1 会生成形如 POC-AC0567C-yyyyMMddHHmmss 的 ImportId，并从其中解析最后 14 位 timestamp。latest no-write 证据的 ImportId 是：

POC-AC0567C-20260612134648

它映射为：

TimestampToken = 20260612134648
ExpectedZzupDat = 20260612
ExpectedZzupTme = 134648

这说明证据不是孤立 response，而是能反推出后台 key 的测试上下文。对 batch upload API，ImportId、source file、group no、sequence no、user 和 timestamp 都是 replay 证据的核心字段。

43.3 Response evidence：不要只看 HTTP status

latest no-write JSON 包含 8 个检查。最有教学价值的是三类 response：

Scenario关键返回说明
JSON multi-row validate succeedsStatus = VALID, Simulation = X, ReceivedRows = 2, ValidRows = 2, InsertedRows = 0validate path 可用且不写 DB
Invalid batch is rejected before insertStatus = REJECTED, MessageField = DocumentAmount, MessageLineNo = 1, MessageText = DocumentAmount must be numeric错误能定位到字段和行
Status lookup route respondsStatus = NOT_FOUND, MessageField = ImportId, MessageText = No ZCOM_B_ACTT397 rows found for this ImportId/user keyno-write 下 status route 可用，但没有真实数据

这比“HTTP 200”更接近项目需要。HTTP status 说明通信层结果，Status、MessageField、MessageLineNo、MessageText 才说明业务层结果。对 OData/RAP action 来说，response body 里的业务字段常常决定测试是否真的通过。

43.4 No-write evidence：安全可重跑，但证明范围有限

latest no-write 测试结果是：

Passed = true
Total = 8
Failed = 0
AllowRealUpdate = false

8 个检查分别是：

  1. ImportId timestamp token maps to expected key date/time
  2. JSON multi-row validate succeeds
  3. Invalid batch is rejected before insert
  4. Base64 TSV validate succeeds
  5. XLSX unsupported returns error
  6. Status lookup route responds
  7. Real execute skipped unless -AllowRealUpdate is set
  8. Duplicate/idempotency skipped unless -AllowRealUpdate is set

这一组检查的阅读重点，是先判断“安全可重跑的证据”到底覆盖了哪些入口。它适合每天、每次改动后重复执行，因为它不制造新数据；但正因为它不写 DB，它的结论也必须更克制。

这组证据证明 validate、invalid batch、Base64 TSV validate、XLSX unsupported、status route 和 safety switch 行为；它不证明真实 insert、真实 duplicate conflict、真实 rollback 或 application log。书稿中必须把这条边界写清楚。

43.5 Real-write evidence：证明写入和重复拒绝，但要谨慎重跑

README 和 TEST_EVIDENCE.md 记录了早先真实写入测试。这个 real-write 证据来自旧路径脚本：

deliverables\RPA01-D-00020_api_poc_20260530\rpa-playwright\Invoke-Ac0567CBatchPoc.ps1 -AllowRealUpdate

它不是 latest no-write JSON 里的 current package rerun 结果，因此引用时要写明“早先受控真实写入证据”，不要让读者误以为当前 04_tests/test-results/ac0567c-no-write-latest.json 已经包含真实写入。

ImportId: POC-AC0567C-20260612115720
zzup_dat = 20260612
zzup_tme = 115720
ExecuteImport Status = IMPORTED
InsertedRows = 2
Repeat same ImportId Status = CONFLICT
GetStatus Status = IMPORTED
GetStatus InsertedRows = 2

这组证据证明了 AC0567C 最关键的正向路径：真实 execute 能插入 2 行，重复同一 ImportId 能被拒绝，status 能查到已插入行。但它会向 ZCOM_B_ACTT397 增加测试数据，所以不能像 no-write 测试那样随意重跑。交付文档中应把它标为 controlled real-write evidence。

43.6 Document and simulator evidence：证明可读性和操作入口

TEST_EVIDENCE.md 记录本地 proxy 下这些文档页面均为 HTTP 200：

  • http://127.0.0.1:4315/docs/swagger-ui.html
  • http://127.0.0.1:4315/docs/redoc.html
  • http://127.0.0.1:4315/docs/static-api-reference.html
  • http://127.0.0.1:4315/docs/ac0567c-upload-simulator.html

n8n-like flow simulator smoke test 还记录了：

{
  "validateBadge": "OK",
  "executeBadge": "Idle",
  "status": "VALID",
  "receivedRows": "2",
  "messageText": "AC0567C batch validation OK; no DB update executed"
}

这类证据不能替代 API/backend evidence，但能证明文档入口和演示路径可用。对客户说明或团队培训，它们很有价值；对生产上线判断，还必须回到真实 SAP endpoint、权限、日志和后台数据。

43.7 Evidence package：推荐的交付结构

证据包不是为了把文件堆得更多，而是为了让下一位 reviewer 不用从零推理。下面这个结构把“合同、no-write、real-write、剩余风险、结论”分开，适合作为交付文档或 Obsidian 附录的骨架。

AC0567C 的证据包可以这样整理：

AC0567C API Evidence Package

01-contract.md
  Base URL / SAP service root
  OpenAPI file
  Swagger / Redoc / static docs URL
  AC0567C endpoint table

02-no-write-test.md
  Command
  ImportId
  8/8 results
  JSON validate response
  invalid batch response
  Base64 TSV validate response
  XLSX unsupported response
  status NOT_FOUND response

03-real-write-test.md
  Controlled execution command
  ImportId
  inserted rows
  duplicate CONFLICT
  status IMPORTED / inserted rows
  backend key derivation

04-negative-and-risk.md
  authorization-negative: not yet proven
  concurrency duplicate: not yet proven
  save failure mapping: not yet proven
  application log: not yet proven
  QAS transport smoke: not yet proven

05-decision.md
  reviewer
  pass / conditional pass / blocked
  remaining production hardening

这个结构的好处是：读者不用从散落的 JSON、截图、README 和聊天记录里重新推理；每条证据都有结论，剩余风险也被明确保留。

43.5 日本项目现场说法

常见日语：

  • APIテスト証跡
  • リクエスト
  • レスポンス
  • ペイロード
  • 実行ユーザ
  • 実行時刻
  • 正常系
  • 異常系
  • 再実行
  • 二重登録防止
  • 権限パターン
  • 疎通確認
  • 障害調査
  • 証跡として残す

review 要求可以这样说：

プレビューだけではなく、正常系、異常系、権限パターン、再実行時の
APIテスト証跡を残してください。

证据边界可以这样说：

今回の no-write テストでは実登録は行っていません。
実登録と二重登録防止は、別途実行済みの real-write 証跡で確認しています。

障害调查时可以这样说：

障害調査のため、リクエストURL、ペイロード、レスポンス、実行ユーザ、
実行時刻、バックエンド状態を確認します。

风险说明可以这样说：

並行実行時の二重登録防止、保存失敗時のエラーマッピング、
アプリケーションログは今後の本番化課題です。

43.8 Counterexample / Failure mode

常见失败模式如下。

第一，只有 preview 或 Swagger 截图。文档能打开只证明入口存在，不证明业务 action 成功。

第二，把 no-write 8/8 写成“真实导入已证明”。latest no-write 明确 AllowRealUpdate = false，真实 execute 和 duplicate 被 skipped。

第三，把早先 real-write evidence 当成可随时重放。真实 execute 会向 ZCOM_B_ACTT397 插入测试数据，必须受控执行。

第四，只看 HTTP status。RAP/OData response body 里的 Status、MessageField、MessageLineNo、MessageText 才能说明业务结果。

第五，错误证据不可定位。只写 Validation failed，没有行号、字段、业务键或修正方向，用户无法处理。

第六，宣称“幂等已完成”，但只证明了顺序重复同一 ImportId 的 CONFLICT，没有并发 duplicate safety 证据。

第七，没有 authorization-negative evidence。当前章节不能用 AUTHORITY-CHECK 源码存在来替代低权限用户调用失败的 API 证据。

第八，没有 application log。测试 JSON 能证明 POC 行为，但生产障害調査通常还需要持久日志、request id、log id 和错误原因。

第九，证据未脱敏。token、cookie、密码、个人敏感信息或完整生产 payload 不能原样放入文档。

第十，用 DEV/local proxy 证据代替 QAS/PRD 证据。transport、service binding、权限、communication setting 和 customizing 都可能变化。

43.9 思考问题

  • AC0567C 的 5 个 C endpoint 分别证明什么？哪些会写 DB，哪些不会？
  • latest no-write JSON 的 8/8 通过能证明哪些事实？不能证明哪些事实？
  • 为什么 Status = VALID 和 Status = IMPORTED 不能只看 HTTP status 判断？
  • 如果外部系统说“收到 success，但 SAP 没有数据”，需要哪些 request、response 和 backend evidence 排查？
  • 为什么真实 insert evidence 不能随便重跑？
  • 当前重复请求证据为什么只能证明 sequential duplicate rejection，而不能证明 concurrency safety？
  • 哪些 payload/header 字段必须脱敏？哪些字段必须保留以便复盘？
  • 日本项目 review 时，如何一句话说明“no-write 証跡と real-write 証跡を分けて確認します”？
