Chapter 02 - Core ABAP Programming

25. Local class 局部类

  • Chapter 02
  • Section 25
  • self-contained study note

25.1 一句话结论

Local class 局部类，是定义在某个 ABAP program、class pool 或当前开发对象局部上下文中的 class：它有 CLASS ... DEFINITION 和 CLASS ... IMPLEMENTATION，可以像普通 OO ABAP class 一样拥有 method、attribute、visibility section 和 test class，但它不是独立 ABAP repository object，不能被整个系统随意复用。它最适合放“只服务当前对象”的 helper、demo、局部封装和 ABAP Unit test class；一旦逻辑要成为跨对象复用的业务能力，就应该考虑 Global class 全局类。

25.2 到底学什么

上一节 Global class 全局类 讲的是项目级 class：有 package、activation、transport、public contract，可以被多个 repository object 使用。local class 则反过来回答一个更细的问题：如果某段逻辑只属于当前程序或当前 global class，是否有必要把它提升成全局对象？

例如，在一个实现 IF_OO_ADT_CLASSRUN 的练习 global class 的 class pool 里，只想把状态码转换成显示文本。严格说，下面的 lcl_status_text 通常写在该 global class 对应的 local definitions / local implementations 区域，而不是作为另一个独立的 ZCL_... 对象存在：

CLASS lcl_status_text DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS get_text
      IMPORTING iv_status      TYPE string
      RETURNING VALUE(rv_text) TYPE string.
ENDCLASS.

CLASS lcl_status_text IMPLEMENTATION.
  METHOD get_text.
    CASE iv_status.
      WHEN 'O'.
        rv_text = 'Open'.
      WHEN 'C'.
        rv_text = 'Closed'.
      WHEN OTHERS.
        rv_text = 'Unknown'.
    ENDCASE.
  ENDMETHOD.
ENDCLASS.

这个 lcl_status_text 就是 local class。它把“状态码如何显示”从主流程里拿出来，让主流程更清楚；但它并没有成为一个能在当前局部上下文之外作为 public API 调用的 ZCL_... class。

本节要学的核心问题：

  • local class 和 global class 的边界是什么？
  • local class 写在哪里，为什么不能把它当成独立 repository object？
  • local class 的 PUBLIC SECTION / PRIVATE SECTION 是否仍然有意义？
  • 为什么 ABAP Unit test class 常写成 local class？
  • 什么逻辑适合留在 local class，什么逻辑应该提升为 global class？
  • 在日本项目 review 中，怎样说明“ローカルクラスに閉じる”或“グローバルクラスへ切り出す”？

25.3 为什么重要

初学 OO ABAP 时，很容易走向两个极端。

第一个极端是“什么都写在一个大 method 里”。这样虽然短期能跑，但随着条件分支、internal table 处理、消息组装、格式转换越来越多，主流程会变成几百行，测试困难，review 也只能靠人脑追踪。

第二个极端是“任何小 helper 都建一个 global class”。这样 repository 里很快堆满 ZCL_...，每个对象都要考虑 package、transport、where-used、public contract 和命名治理，反而增加维护成本。

local class 的价值在这两个极端之间：它允许把当前对象内部的复杂逻辑结构化，又不把局部实现细节过早暴露为全局 contract。

对 Java 背景学习者，可以把 local class 粗略类比成“只在当前文件/当前外部类附近使用的 helper class”。但 ABAP 的差异是：global class 是 repository object / class pool；local class 随所在 program 或 class pool 一起激活、传输和生命周期管理。对 Python 背景学习者，可以把它理解成模块内私有 helper 的 OO 版本，但 ABAP 的可见性和编译/激活边界更正式。

在真实项目里，local class 经常出现在：

  1. global class 的 local definitions / local implementations 区域，用作 private helper，或当前对象 / 当前组件专用的 test double。
  2. ABAP Unit 的 local test class，例如 ltcl_... DEFINITION FOR TESTING。
  3. demo、training、report 或 POC 里，把主流程拆成几个小对象。
  4. 某个大型 repository object 内部，为了减少一个 method 的复杂度而引入的局部策略类、格式化类、计算类。

25.4 Key mechanism / 核心机制

25.4.1 local class is local to its context

local class 不是单独的 ZCL_... repository object。它属于当前 ABAP program、class pool 或对应的 local type/include 上下文。外部对象通常不能像调用 global class 那样直接依赖它。

这带来三个直接后果：

  • 它适合表达局部实现细节，不适合作为局部上下文之外的 public API。
  • 它作为宿主 repository object 的一部分一起 syntax check、activation 和 transport，不能像独立 global class 那样单独移送。
  • 它的 where-used、复用边界和变更影响都应从宿主对象角度理解。

在 ABAP Cloud / RAP 语境下，还要多记一层限制：classic report/program 不是主要扩展形态，local class 更常出现在 global class 的 local definitions、local implementations 或 local test classes 中。local class 本身不是 released API 的治理对象；真正需要遵守 Clean Core / released API 约束的是宿主 global class、RAP behavior、service 暴露和它调用的外部 API。

所以，local class 的名字通常使用 lcl_...；local test class 常见 ltcl_... 或 ltc_...；local interface 常见 lif_...。这些不是语法强制，但在项目 review 里很有帮助：reader 一眼能看出这是局部对象，不是可复用的全局类。

25.4.2 definition and implementation

local class 同样分成 definition 和 implementation：

CLASS lcl_quantity_rule DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS is_valid
      IMPORTING iv_quantity     TYPE i
      RETURNING VALUE(rv_valid) TYPE abap_bool.
ENDCLASS.

CLASS lcl_quantity_rule IMPLEMENTATION.
  METHOD is_valid.
    rv_valid = xsdbool( iv_quantity > 0 AND iv_quantity <= 999 ).
  ENDMETHOD.
ENDCLASS.

definition 是“这个局部 class 对当前上下文暴露什么能力”；implementation 是“具体怎么做”。即使 local class 不对系统外部开放，definition 仍然是阅读入口。review 时先看 local class 的 public method 名称和参数，判断它是否真的把主流程变清楚，而不是把复杂度从一个大 method 挪到另一个大 method。

25.4.3 visibility still matters

local class 也可以有：

PUBLIC SECTION.
PROTECTED SECTION.
PRIVATE SECTION.

这里的 PUBLIC SECTION 不等于“全系统公开”。它表示在 local class 的可见范围内，外部代码可以访问哪些 component。对一个 lcl_... helper 来说，public surface 仍然应该小而清楚；内部计算步骤、常量、临时状态应该放在 PRIVATE SECTION。

SAP Help 中的 local friends 属于进阶访问控制细节：在少数场景下，可以让同一 class pool 上下文中的 local/global class 建立 friend 关系并访问非公开 component。初学阶段只需知道它不是常规复用方式；真正稳定的公共契约仍应优先设计为 global class 或 interface。

常见判断：

  • 主流程需要调用的入口 method：放在 PUBLIC SECTION。
  • 拆分出来的内部步骤：放在 PRIVATE SECTION。
  • inheritance 相关的 PROTECTED SECTION：初学阶段少用，等 Inheritance 继承 再展开。
25.4.4 local class vs global class

可以用这张判断表：

场景更适合
只被当前 global class/report 使用的小 helperlocal class
ABAP Unit test class / 当前对象专用 test doublelocal class
demo、training、POC 中的局部封装local class
被多个 repository object 复用的业务规则global class
RAP handler / saver 之外复用的 service 或 validatorglobal class
需要成为稳定 public contract 的 API wrapperglobal class
需要单独 package、transport、where-used 管理的对象global class

不要把 local/global 理解成“哪个更高级”。它们解决的是不同范围的问题：local class 让当前对象内部更清楚；global class 让跨对象复用和交付治理更清楚。

25.4.5 local test class

ABAP Unit 里，test class 通常就是 local class。它服务于被测试对象，不应该成为生产代码可调用的业务对象。测试类声明除了 FOR TESTING，通常还会写清楚 DURATION 和 RISK LEVEL，让测试运行成本和风险级别更明确。

CLASS ltcl_status_text DEFINITION FINAL FOR TESTING
  DURATION SHORT
  RISK LEVEL HARMLESS.
  PRIVATE SECTION.
    METHODS open_status FOR TESTING.
ENDCLASS.

CLASS ltcl_status_text IMPLEMENTATION.
  METHOD open_status.
    cl_abap_unit_assert=>assert_equals(
      act = lcl_status_text=>get_text( 'O' )
      exp = 'Open' ).
  ENDMETHOD.
ENDCLASS.

这段代码表达三个信息：

  • FOR TESTING 说明这是 ABAP Unit test class。
  • test method 也标记 FOR TESTING。
  • test class 依附于当前对象，用来验证当前对象或局部 helper 的行为。

测试逻辑放在 local test class，是为了让测试与被测代码保持同一生命周期，同时不把测试 helper 暴露给生产调用者。后面 ABAP Unit 单元测试基础 会专门展开。

25.4.6 local class in global class source

在 ADT 打开 global class 时，除了主要 class definition / implementation，通常还能看到 local types、local definitions、local implementations、test classes 等区域或 include。local class 常被放在这些局部区域里，用来支持主 global class。

这时要记住：宿主 global class 仍然是 repository object；local class 只是它内部的实现工具。移送、激活、包归属都跟着宿主对象走。

25.5 日本项目现场说法

常见日语：

  • ローカルクラス
  • グローバルクラス
  • ヘルパークラス
  • テストクラス
  • ローカル定義
  • ローカル実装
  • 局所的な責務
  • 再利用範囲
  • 公開契約
  • 切り出し

自然说明句：

  • このクラスは当該オブジェクト内だけで使う補助ロジックなので、ローカルクラスとして閉じています。
  • 複数のクラスや RAP オブジェクトから再利用する可能性がある場合は、ローカルクラスではなくグローバルクラスとして切り出してください。
  • テスト用の補助ロジックは本番の公開契約にしないため、ローカルテストクラスに置きます。
  • ローカルクラスは特定のリポジトリオブジェクト内に閉じた実装なので、再利用範囲が広がる場合はグローバルクラスまたはインターフェースへの切り出しを検討してください。

review 里常见问法：

  • このローカルクラスは本当にこのクラス内だけで完結しますか。
  • 再利用範囲が広がるならグローバルクラス化を検討してください。
  • ローカルクラスの PUBLIC SECTION が大きすぎるため、責務を分割してください。
  • テスト用のクラスは本番コードから参照されないようにしてください。

使用场景：详细设计、代码 review、ABAP Unit review、RAP behavior implementation 拆分、transport 前影响范围确认、旧 report 重构。

25.7 Example / 学习例子

25.7.1 例 1：用 local class 拆出局部 helper

主流程里直接写状态码转换：

DATA: lv_status TYPE string VALUE 'O',
      lv_text   TYPE string.

CASE lv_status.
  WHEN 'O'.
    lv_text = 'Open'.
  WHEN 'C'.
    lv_text = 'Closed'.
  WHEN OTHERS.
    lv_text = 'Unknown'.
ENDCASE.

如果这段逻辑在当前对象中出现多次，但还没有跨对象复用价值，可以先放进 local class：

CLASS lcl_status_text DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS get_text
      IMPORTING iv_status      TYPE string
      RETURNING VALUE(rv_text) TYPE string.
ENDCLASS.

CLASS lcl_status_text IMPLEMENTATION.
  METHOD get_text.
    CASE iv_status.
      WHEN 'O'.
        rv_text = 'Open'.
      WHEN 'C'.
        rv_text = 'Closed'.
      WHEN OTHERS.
        rv_text = 'Unknown'.
    ENDCASE.
  ENDMETHOD.
ENDCLASS.

调用者只读一行：

DATA(lv_text) = lcl_status_text=>get_text( lv_status ).

这不是为了“多写一个类显得高级”，而是让主流程表达业务步骤，把局部细节收进一个命名清楚的小对象。

25.7.2 例 2：local test class
CLASS ltcl_status_text DEFINITION FINAL FOR TESTING
  DURATION SHORT
  RISK LEVEL HARMLESS.
  PRIVATE SECTION.
    METHODS unknown_status FOR TESTING.
ENDCLASS.

CLASS ltcl_status_text IMPLEMENTATION.
  METHOD unknown_status.
    cl_abap_unit_assert=>assert_equals(
      act = lcl_status_text=>get_text( 'X' )
      exp = 'Unknown' ).
  ENDMETHOD.
ENDCLASS.

测试类本身不是业务 API。它和被测代码放在同一开发对象附近，是为了让修改和测试一起走，避免测试代码散落成独立业务对象。

25.7.3 例 3：什么时候提升成 global class

如果状态码转换只在一个 demo 或一个 global class 内部使用，local class 可以接受。若它开始被多个对象需要，例如：

  • RAP handler 要用。
  • batch job 要用。
  • OData wrapper 要用。
  • ABAP Unit 多个测试对象都要用。
  • 项目文档把它定义成可复用业务规则。

这时继续藏在 local class 里就会造成重复实现或隐式依赖。更合理的做法是提升成命名清楚的 global class，例如 ZCL_STATUS_TEXT 或更贴近项目命名规则的 service / helper class，并设计 public method、package、transport 和测试策略。

25.7.4 例 4：不好的 local class
CLASS lcl_all_business_rules DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS validate_order.
    CLASS-METHODS calculate_price.
    CLASS-METHODS post_document.
    CLASS-METHODS call_external_api.
ENDCLASS.

这不是“局部封装”，而是把多个责任塞进一个局部黑箱。它既不利于复用，也不利于测试和 review。修正方向是：按责任拆分，能局部完成的留在 local class；跨对象复用或有稳定 contract 的部分提升为 global class 或 interface。

25.8 Counterexample / Failure mode

25.8.1 误区 1：把 local class 当成低成本全局复用

如果多个对象都需要同一段逻辑，却把它重复写成多个 lcl_...，后续规则变更就要改很多地方。项目风险是：一个地方改了，另一个地方漏改，测试覆盖也分散。

修正动作：确认复用范围。如果规则已经跨 repository object 使用，设计 global class 或 interface。

25.8.2 误区 2：local class 的 public surface 太大

local class 虽然是局部的，但 PUBLIC SECTION 太大仍然会制造混乱。当前对象内部的主流程也会依赖太多细节，后续重构成本升高。

修正动作：只暴露主流程需要的入口 method；中间步骤放 PRIVATE SECTION。

25.8.3 误区 3：把测试 helper 混进生产 contract

测试用的 fake、double、assertion helper 如果做成生产可调用的 global class，可能污染业务 API。

修正动作：优先使用 local test class 或测试专用 helper；需要跨测试对象复用时，再按项目测试策略设计公共 test utility。

25.8.4 误区 4：用 local class 掩盖大 method

如果 local class 里仍然只有一个几百行 run method，问题没有解决，只是换了位置。

修正动作：用 Small methods 小方法原则 拆分职责；让 method name 表达意图，让参数和返回值清楚。

25.9 思考问题

  • 这段逻辑只属于当前对象，还是可能被多个对象复用？
  • 如果明天另一个 RAP handler 也需要它，是重复写一个 local class，还是抽成 global class？
  • 这个 local class 的 PUBLIC SECTION 是否只暴露了必要入口？
  • 这个 local class 是否只是把一个大 method 换了地方？
  • 测试逻辑应该留在 local test class，还是已经需要项目级 test utility？
  • 在日本项目 review 中，应说 ローカルクラスに閉じる，还是 グローバルクラスへ切り出す？
