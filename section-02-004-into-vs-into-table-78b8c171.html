4. INTO vs INTO TABLE 结果接收

  • Chapter 02
  • Section 04
  • self-contained study note

4.1 一句话结论

INTO 和 INTO TABLE 决定 ABAP SQL 的查询结果进入 ABAP 内存时是什么形状：INTO @DATA(ls_order) 通常接收一行到结构或工作区，INTO TABLE @DATA(lt_orders) 接收多行到内表。初学者要先建立这个判断：SELECT 不是“查完就结束”，它一定要把结果交给一个 ABAP data object；这个对象是单行结构还是 internal table，会直接影响后续循环、排序、查找、空结果处理和性能。

4.2 到底学什么

上一节 Host variables 宿主变量 讲的是 ABAP 变量如何进入 SQL；本节讲 SQL 结果如何回到 ABAP。看同一个数据源，目标不同，代码的语义就不同：

SELECT SINGLE FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status
       WHERE sales_order_id = @lv_sales_order_id
       INTO @DATA(ls_order).

这段代码的目标是 ls_order，名字里的 ls_ 暗示 line structure。它适合“业务上只应该有一行”的读取，例如完整主键读取。

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status
       WHERE customer_id = @lv_customer_id
       ORDER BY sales_order_id
       INTO TABLE @DATA(lt_orders).

这段代码的目标是 lt_orders，名字里的 lt_ 暗示 internal table。它适合“业务上可能有多行”的读取，例如某个客户的订单列表。

本节要学四件事：一是 INTO 不带 TABLE 时面向单行结构或 SELECT loop；二是 INTO TABLE 面向内表结果集；三是 inline declaration 很方便，但生成的内表类型和 key 需要理解；四是 INTO TABLE 不是排序、去重、权限或事务控制，结果形状和业务语义要分开判断。

4.3 为什么重要

很多 ABAP 初学者会把 INTO 看成语法尾巴：前面 SELECT 写对了，最后随便接一个变量就行。项目代码里，这恰恰是容易出错的地方。

第一类错误是把多行当单行。业务上一个客户可能有多张订单，却用单行结构接收，或者用 SELECT SINGLE 没有完整唯一条件。代码能激活，但业务语义不稳：到底取哪一行、是否稳定、没有排序时是否可重复，都说不清。

第二类错误是把单行当多行。完整主键读取本来只需要一个结构，结果为了“统一写法”读进内表，再 READ TABLE INDEX 1。这会让代码变长、错误路径变模糊，也容易掩盖“唯一性到底由谁保证”的问题。

第三类错误是不了解 inline 内表的默认形态。INTO TABLE @DATA(lt_orders) 很舒服，但 SAP Samples 明确提醒：这种 inline declaration 得到的内表通常是 standard table with empty key。它适合演示和一次性循环处理；如果后续要频繁按 key 查找、去重、二分读取或表达唯一性，就要显式设计 table type，例如 Standard table 标准表、sorted table 或 hashed table，而不是默认拿来就用。

第四类错误是误会 INTO TABLE 会保留旧数据。普通 INTO TABLE 会初始化目标对象，再写入本次结果。要保留已有行并继续追加，需要 APPENDING TABLE。这在报表分段读取、分页、包处理和多次条件合并时尤其重要。

4.4 Key mechanism / 核心机制

先把一条 SELECT 的结果想成一个 result set。这个 result set 可能是空的、单行的、多行的；INTO 子句决定它如何被映射到 ABAP data object。

4.4.1 单行目标：INTO @DATA(ls_order)

初学阶段，把 INTO 不带 TABLE 的写法优先和 SELECT SINGLE 放在一起理解：

SELECT SINGLE FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status
       WHERE sales_order_id = @lv_sales_order_id
       INTO @DATA(ls_order).

IF sy-subrc <> 0.
  " The requested order does not exist.
  RETURN.
ENDIF.

这里 ls_order 是一个结构。FIELDS 里选了三个列，ls_order 就有对应的三个组件。后续代码可以写 ls_order-status、ls_order-customer_id。这类写法的重点是：你必须能说明“为什么只会有一行”。完整主键、唯一索引、业务唯一性、或明确的 ORDER BY ... UP TO 1 ROWS，都是不同层次的理由；不能用“我只想要一行”替代唯一性证明。

严格说，INTO @wa 也可以出现在 SELECT loop 里逐行接收结果，也就是 SELECT ... INTO @wa. ... ENDSELECT.。本教程初学阶段不把它作为默认推荐，因为循环中逐行访问数据库结果容易和性能、锁、异常路径混在一起。先掌握 SELECT SINGLE ... INTO @DATA(ls_...) 和 SELECT ... INTO TABLE @DATA(lt_...) 两条主线。

4.4.2 多行目标：INTO TABLE @DATA(lt_orders)

INTO TABLE 表示结果集被接收到 internal table：

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status
       WHERE customer_id = @lv_customer_id
       ORDER BY sales_order_id
       INTO TABLE @DATA(lt_orders).

IF lt_orders IS INITIAL.
  " No orders found for this customer.
  RETURN.
ENDIF.

这里 lt_orders 是一个内表，每一行的结构由 FIELDS 结果决定。多行结果后续通常会 LOOP AT lt_orders INTO DATA(ls_order).，或者用 table expression、READ TABLE、line_exists( ) 等内部表操作继续处理。

INTO TABLE 会初始化目标内表，再放入本次查询结果。也就是说，如果 lt_orders 之前已经有内容，普通 INTO TABLE @lt_orders 会覆盖它。需要保留旧内容时，用 APPENDING TABLE @lt_orders，但要非常清楚为什么要追加、是否会重复、是否需要排序或去重。

4.4.3 Inline declaration 的便利和代价

@DATA(ls_order) 和 @DATA(lt_orders) 是 inline declaration。优点是少写类型声明，结果对象的类型由 SELECT 列表推导，demo 和局部逻辑很清爽。

代价是你必须知道推导出来的对象长什么样。对于 INTO TABLE @DATA(lt_orders)，SAP Samples 提醒：内部表目标的 inline declaration 通常会得到 standard table with empty key。这个默认形态对“读出来循环一次”很好，但对“后面要按键高频查找”未必好。比如：

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status
       WHERE customer_id = @lv_customer_id
       INTO TABLE @DATA(lt_orders).

" Later:
READ TABLE lt_orders WITH KEY sales_order_id = lv_sales_order_id
     INTO DATA(ls_order).

这段代码能读，但 lt_orders 的 key 不是业务主键；如果数据量大、读取频繁，就应该显式声明合适的 table type，或者在前面就用更准确的查询条件避免读太多。inline declaration 是方便，不是表类型设计的替代品。

4.4.4 CORRESPONDING FIELDS 和位置赋值

如果把查询结果放入一个已经声明好的结构或内表，不写 CORRESPONDING FIELDS 时，ABAP SQL 会按结果列的位置向目标组件赋值；写 CORRESPONDING FIELDS OF 时，才按同名组件匹配。这个差异在维护旧代码时很重要：

TYPES: BEGIN OF ty_order_list,
         sales_order_id TYPE zsales_order-sales_order_id,
         status         TYPE zsales_order-status,
       END OF ty_order_list.
DATA lt_orders TYPE STANDARD TABLE OF ty_order_list WITH EMPTY KEY.

SELECT FROM zsales_order
       FIELDS sales_order_id,
              status
       WHERE customer_id = @lv_customer_id
       INTO CORRESPONDING FIELDS OF TABLE @lt_orders.

新代码里，最简单可靠的做法是让 FIELDS 列表、别名和目标结构一致，减少位置不一致造成的隐性错位。INTO CORRESPONDING FIELDS OF TABLE 可以帮助维护已有结构，但不应该成为“字段列表随便写”的借口。

4.4.5 空结果和系统字段

SELECT ... INTO TABLE 没有读到行时，目标内表会是空表，sy-subrc 和 sy-dbcnt 也会反映读取结果。学习阶段可以先养成两个动作：读取后立即处理空结果；如果业务关心行数，立即保存或使用 sy-dbcnt，不要等后续语句把系统字段覆盖。

SELECT FROM zsales_order
       FIELDS sales_order_id
       WHERE customer_id = @lv_customer_id
       INTO TABLE @DATA(lt_orders).

DATA(lv_selected_count) = sy-dbcnt.

IF lt_orders IS INITIAL.
  " Empty result is a valid business path here.
ENDIF.

这里还要连接 ABAP system fields sy fields：系统字段是易变的，后续很多语句都可能改变它们。不要在十几行之后才想起检查 sy-subrc。

4.5 日本项目现场说法

常见日语：

  • 取得結果
  • 結果受け取り
  • 単一行取得
  • 複数行取得
  • 構造
  • 作業領域
  • 内部テーブル
  • インライン宣言
  • 標準テーブル
  • 空キー
  • 上書き
  • 追記
  • APPENDING TABLE
  • CORRESPONDING FIELDS
  • 取得件数
  • 空結果

现场说明句：

  • 単一行取得なので、構造 ls_order に INTO で受け取ります。
  • 複数行取得なので、内部テーブル lt_orders に INTO TABLE で受け取ります。
  • INTO TABLE は既存内容を初期化して上書きします。既存行を残す場合は APPENDING TABLE の要否を確認してください。
  • @DATA(lt_orders) で宣言される内部テーブルは標準テーブル・空キーになるため、後続のキー検索が多い場合は明示的なテーブル型を検討します。
  • CORRESPONDING FIELDS を使わない場合、列名ではなく位置で代入される点に注意してください。
  • 空結果の場合の業務上の扱いを明確にしてください。

日本项目里，“受け皿” 这个口语也常见，意思是“接收结果的目标对象”。代码 review 时如果被问 受け皿は構造ですか、内部テーブルですか，实际是在问本节的核心：结果是一行还是多行，后续如何处理。

4.7 Example / 学习例子

4.7.1 例 1：完整主键读取到结构
SELECT SINGLE FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status
       WHERE sales_order_id = @lv_sales_order_id
       INTO @DATA(ls_order).

IF sy-subrc <> 0.
  " Order not found.
  RETURN.
ENDIF.

IF ls_order-status = lc_status_released.
  " Continue processing.
ENDIF.

读法：业务上要处理一个订单，所以目标是结构。review 问题是：lv_sales_order_id 是否完整唯一？没有读到订单时业务要报错、提示，还是正常跳过？

4.7.2 例 2：列表读取到内表
SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status,
              created_at
       WHERE customer_id = @lv_customer_id
       ORDER BY created_at DESCENDING,
                sales_order_id
       INTO TABLE @DATA(lt_orders).

LOOP AT lt_orders INTO DATA(ls_order).
  " Build a response line or perform read-only checks.
ENDLOOP.

读法：业务上要处理订单列表，所以目标是内表。ORDER BY 仍然要写；INTO TABLE 只决定结果形状，不保证业务顺序。

4.7.3 例 3：追加而不是覆盖
DATA lt_orders TYPE STANDARD TABLE OF zsales_order WITH EMPTY KEY.

SELECT FROM zsales_order
       FIELDS *
       WHERE status = @lc_status_open
       APPENDING TABLE @lt_orders.

SELECT FROM zsales_order
       FIELDS *
       WHERE status = @lc_status_blocked
       APPENDING TABLE @lt_orders.

这段代码用于说明 APPENDING TABLE，不代表推荐在业务代码里随便 FIELDS *。真实项目里要先问：两次追加是否会重复？是否应该用一个 WHERE status IN ... 的查询？是否需要显式排序？是否应该用更小的字段列表？

4.7.4 例 4：显式目标类型
TYPES: BEGIN OF ty_order_key,
         sales_order_id TYPE zsales_order-sales_order_id,
         customer_id    TYPE zsales_order-customer_id,
       END OF ty_order_key,
       tt_order_key TYPE SORTED TABLE OF ty_order_key
                    WITH UNIQUE KEY sales_order_id.

DATA lt_order_keys TYPE tt_order_key.

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id
       WHERE customer_id = @lv_customer_id
       INTO TABLE @lt_order_keys.

这里目标内表不是 inline 推导出来的 standard table，而是显式 sorted table。这样写更长，但表达了后续“按 sales_order_id 唯一查找”的设计意图。初学阶段不必马上掌握所有 table category，但要知道 inline table 不是唯一选择。

4.8 Counterexample / Failure mode

失败模式一：非唯一条件下用单行目标。

SELECT SINGLE FROM zsales_order
       FIELDS sales_order_id,
              status
       WHERE customer_id = @lv_customer_id
       INTO @DATA(ls_order).

如果一个客户可能有多张订单，这段代码没有说明取哪一张。问题不在 INTO 本身，而在“目标是单行，但业务条件不是单行”。要么补唯一条件，要么用 ORDER BY ... UP TO 1 ROWS 表达业务规则，要么改成 INTO TABLE 处理列表。

失败模式二：以为 INTO TABLE 会保留旧内容。

SELECT FROM zsales_order
       FIELDS sales_order_id
       WHERE status = @lc_status_open
       INTO TABLE @lt_orders.

SELECT FROM zsales_order
       FIELDS sales_order_id
       WHERE status = @lc_status_blocked
       INTO TABLE @lt_orders.

第二次 INTO TABLE 会覆盖第一次结果。要追加用 APPENDING TABLE，但追加前要重新审视是否应该合并条件或去重。

失败模式三：把 inline 内表当成设计好的 key table。

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id
       INTO TABLE @DATA(lt_orders).

lt_orders 很方便，但默认 key 不能替你表达业务唯一性。后续如果频繁 READ TABLE ... WITH KEY，应考虑显式 sorted/hashed table type，或者从 SQL 条件上减少数据量。

失败模式四：以为 CORRESPONDING FIELDS 总是无脑安全。它按同名组件赋值，确实能减少位置错位，但也可能悄悄忽略没有同名目标的列。review 时要确认目标结构是不是故意只收这些字段，而不是漏字段。

失败模式五：读出来不处理空结果。空内表可能是正常业务路径，也可能表示输入错误、配置缺失、权限过滤、数据同步延迟。好代码不会只“读了就往下走”，而会说明空结果的业务含义。

4.9 思考问题

  • 这条 SELECT 的业务语义是一行还是多行？
  • 如果是一行，唯一性由完整主键、唯一索引、业务规则，还是排序限制保证？
  • 如果是多行，目标内表后续只循环一次，还是要按 key 高频查找？
  • @DATA(lt_...) 推导出的 standard table with empty key 是否够用？
  • 普通 INTO TABLE 是否会覆盖已有内容？如果要追加，为什么不用一个更清楚的查询条件？
  • 是否需要 CORRESPONDING FIELDS？如果需要，是因为维护旧结构，还是目标结构本来就应该重写？
  • 读取为空时，业务上是正常、警告、错误，还是权限/配置问题？
  • sy-subrc、sy-dbcnt 是否在读取后立即处理，避免被后续语句覆盖？
  • INTO TABLE 后是否还依赖第一条/最后一条？如果依赖，ORDER BY 在哪里？
