Chapter 01 - SAP 和 ABAP 基础

29. Date arithmetic 日期计算

  • Chapter 01
  • Section 29
  • self-contained study note

本节结论

Date arithmetic 日期计算 是用 ABAP 正确处理“今天、到期日、有效期间、相差天数”的基础。Day 04 要先掌握 TYPE d 的内部格式、日期加减天数、两个日期相减得到天数、有效期判断，以及一个重要边界：月份、工作日、节假日、时区和 timestamp 不能只靠简单 + 1 或字符串截取解决。

正文

SAP 项目里日期无处不在：订单日期、交货日期、有效开始日、有效结束日、价格条件期间、合同到期日、批处理基准日、测试数据日期、接口文件名日期、OData filter 日期。初学者容易把日期当成字符串，例如看到 20260613 就想用 substring 改月份、拼字符串、或者把今天加 30 当作“一个月后”。但 ABAP 里的 date 是一个专门的 built-in type：d。它看起来像 8 位字符，内部格式是 YYYYMMDD，但在很多计算位置里，ABAP 会把合法日期转换成从 01.01.0001 起算的天数来做加减。

最小的日期加减先从“天”开始。SAP Learning 明确把 direct date calculation 讲成三类：给日期加天数、从日期减天数、两个日期相减得到相差天数。例子：

DATA today    TYPE d.
DATA due_date TYPE d.
DATA days_open TYPE i.

today    = cl_abap_context_info=>get_system_date( ).
due_date = today + 7.
days_open = today - ls_order-created_on.

这里 today + 7 表示 7 个 calendar days 之后，不是 7 个 working days。today - ls_order-created_on 得到的是相差天数，适合写“创建后经过几天”“是否超过 N 天未处理”这类逻辑。对 Java/Python 背景的人，可以把它类比成 date object 的 day-based arithmetic；但 ABAP 里 d 仍然是 character-like date field，不是 Python datetime.date 对象，输出格式、用户格式和 offset 访问都有 ABAP 规则。

当前日期的来源也要说清楚。老代码里常见 sy-datum，它是系统字段，表示当前 ABAP runtime 语境里的 system date。还会看到 sy-datlo、sy-timlo 这类 user local date / user local time 字段，用来区别系统时间和用户本地时间。现代 ABAP，尤其 ABAP Cloud / BTP ABAP Environment 教材里，更推荐用 cl_abap_context_info=>get_system_date( ) 这类上下文 API 来表达“我要当前系统日期”；注意它表达的仍然是 system date，不是 user-local date。这里和前一节 ABAP system fields sy fields 的纪律一致：sy-* 可以读，但不要把它当成业务时间设计的万能来源。只要需求牵涉“用户所在时区的今天”“过去 24 小时”“审计时间”，就不要只看 sy-datum。

日期字段虽然能用 offset/length 访问年月日，但这只是读取组成部分，不是业务计算方法：

DATA valid_from TYPE d VALUE '20260601'.

DATA(year)  = valid_from(4).
DATA(month) = valid_from+4(2).
DATA(day)   = valid_from+6(2).

这段代码可以用于显示、日志、简单检查或构造解释文本。不要据此写“月份 + 1”的业务逻辑。月末、闰年、不同月份天数、财政期间、客户定义日历都不是字符串截取能解决的。比如从 20240831 得到“下个月同一天”就不是简单把月份改成 9，因为 9 月没有 31 日。也不要用 lv_date + 30 假装“加一个月”：这是加 30 个 calendar days，不是 month arithmetic。遇到月份、年、月末、财务期间、账期、工厂日历，先找 SAP 提供的 date/time function、业务对象 API、配置日历或项目既有工具，而不是自己手写规则。

在 ABAP Cloud / 现代 ABAP 语境里，SAP 的 XCO date/time API 会把 month-end 规则显式化，例如 preserving mode 和 ultimo mode 对 “2024-08-31 + 1 month” 的处理不同：前者保留“31 日”语义，目标月份没有 31 日时应让调用者处理异常；后者按月末语义落到 2024-09-30。这一层不是 Day 04 要背的 API，但它证明了一件事：月份/年/月末不是 raw TYPE d 的整数加减问题。在 ECC / on-premise 既存代码里，也可能看到 CL_RECA_DATE 或 DATE_CONVERT_TO_FACTORYDATE 这类 classic 工具；新代码尤其 ABAP Cloud 要优先确认 released API 和项目标准。

有效期判断是 SAP 项目里最常见的 date arithmetic。典型逻辑是：某条记录从 valid_from 开始生效，到 valid_to 结束失效。初学者要先写清 inclusive / exclusive 边界。多数业务表字段名如果叫 “valid from / valid to”，现场常常按首尾都包含理解，但真实项目必须看字段定义和功能顾问说明：

DATA(today) = cl_abap_context_info=>get_system_date( ).

IF ls_price-valid_from <= today
   AND ls_price-valid_to   >= today.
  out->write( |Price is valid today| ).
ENDIF.

这段代码最重要的不是语法，而是边界说明：valid_to >= today 表示到当天仍有效。如果项目规则是“结束日当天 00:00 已失效”，写法就不同。日本项目设计书里经常会把这种条件写成 有効開始日 <= 基準日 <= 有効終了日，实现时要和设计书完全一致。

还要处理 initial date。TYPE d 的初始值是 00000000（8 个 0）。它可以表示“未设置”，但它不是一个有效的日历日期。日期计算前要先检查：

IF ls_order-requested_date IS INITIAL.
  out->write( |Requested date is missing| ).
ELSEIF ls_order-requested_date < today.
  out->write( |Requested date is already past| ).
ENDIF.

不要把 00000000 当成真实日期参与加减，也不要把“未设置日期”和“无限期有效”混成一个意思。很多 SAP 表会用特定高值日期表示开放结束，例如 99991231，但这属于具体业务字段约定，不能看到一个 date 字段就自动假设它代表无限期。教材里只建立纪律：initial date、open-ended validity、business default date 必须分开解释。

working day / business calendar 是另一个边界。today + 7 是 calendar days；如果需求写的是 “7 営業日後” 或 “next working day”，就需要 factory calendar / holiday calendar 语义。SAP 的 factory calendar 基于 holiday calendar 定义哪些天是 working days。日本项目里，这通常牵涉公司日历、工厂、国家/地区、客户自定义休假日。Classic ABAP 里可能会遇到 DATE_CONVERT_TO_FACTORYDATE 这类 function module，并且 calendar ID 往往来自 plant 等主数据；ABAP Cloud / S/4HANA Cloud 场景则要确认 released API 或业务对象提供的日历能力。Day 04 不要求掌握具体 calendar API，只要求知道：不要用 + 7 假装自己算了 7 个营业日，也不要自己写“周六周日跳过”就认为完整，因为日本节假日、公司假、工厂日历都会改变结果。

最后，把 date 和 timestamp 分开。TYPE d 只表示本地日期，没有时分秒，也没有时区。跨时区的起飞/到达、UTC 记录、审计时间、接口时间戳、OData timestamp、过去 24 小时统计，都要进入 Time and timestamp 时间戳。SAP Learning 里用 utclong_current( )、utclong_add( )、utclong_diff( ) 和 CONVERT UTCLONG ... INTO DATE ... TIME ... TIME ZONE ... 展示 timestamp 计算；这些不是 Day 04 日期加减的主战场，但要知道何时该升级到 timestamp。

本节的最小心智模型是：如果只是“加/减 N 天、算相差天数、判断是否在有效期间内”，TYPE d 和直接 date arithmetic 可以胜任；如果是“月份、月末、财务期间、工作日、节假日、跨时区、UTC timestamp”，就不要手写小聪明，要找 SAP 官方函数、业务 API、配置日历或项目标准。

为什么重要

  • 订单、价格、合同、交货、批处理、授权有效期、接口筛选都离不开日期。
  • YYYYMMDD 看起来像字符串，但 date arithmetic 不是普通字符串处理。
  • 有效期边界错误会导致价格误用、数据筛选漏取、测试用例不稳定、上线后按日期才暴露缺陷。
  • calendar days 和 working days 混用，会让日本节假日、公司日历、工厂日历相关逻辑出错。
  • Java/Python 背景学习者容易相信 date library 会自动处理所有语义；SAP 项目还要看 DDIC 字段、业务配置、factory calendar、time zone 和功能设计书。

核心机制

  • TYPE d：ABAP built-in date type，内部格式 YYYYMMDD，表示 local date。
  • 合法日期计算：ABAP 可把合法 date 转成从 01.01.0001 起算的天数后做加减。
  • date + n / date - n：按 calendar days 加减。
  • date1 - date2：得到两个日期之间的天数差。
  • cl_abap_context_info=>get_system_date( )：现代 ABAP 中表达当前系统日期的上下文 API。
  • sy-datum：老代码常见 current system date system field；能读，但不要滥用为业务时间设计。
  • sy-datlo / sy-timlo：老代码里可见的 user local date / time；如果服务器和业务用户时区不同，要先确认语义。
  • offset/length：date(4)、date+4(2)、date+6(2) 可取年月日片段，但不等于月/年业务计算。
  • month/year arithmetic：月份、年、月末要用 released API、XCO date/time API、业务 API 或项目标准，不用 + 30 或 offset 手写。
  • initial date：00000000 常表示未设置，不是有效日历日期；计算前要先检查。
  • validity interval：要明确 valid_from <= key_date <= valid_to 是否首尾包含。
  • factory calendar / holiday calendar：working day 逻辑需要日历配置、calendar ID、released API 或项目标准，不是简单跳过周末。
  • timestamp / timezone：跨时区、UTC、审计时间进入 timestamp 主题，不用 TYPE d 硬算。

日本项目现场表达

  • 常见日语：日付計算、基準日、処理日、システム日付、有効開始日、有効終了日、有効期間。
  • 期限相关：期限日、締切日、支払期日、納期、経過日数、日数差。
  • 日历相关：営業日、稼働日、休日カレンダー、ファクトリカレンダー、祝日考慮。
  • 边界相关：期間判定、日付範囲チェック、初期日付、月末日、タイムゾーン、UTCタイムスタンプ。
  • 说明句：この判定は 有効開始日 <= 基準日 <= 有効終了日 の条件で有効期間内かどうかを確認します。
  • Review 句：7営業日後 の要件であるため、単純な日付加算ではなくファクトリカレンダーを考慮してください。
  • Review 句：初期日付 00000000 を日付計算に使用する前に、未設定として扱うか業務ルールを確認してください。

学习例子

先算到期日和经过天数：

DATA today TYPE d.
DATA due_date TYPE d.
DATA days_open TYPE i.

today = cl_abap_context_info=>get_system_date( ).
due_date = today + 14.
days_open = today - ls_ticket-created_on.

out->write( |Due date: { due_date DATE = USER }| ).
out->write( |Open for { days_open } days| ).

再判断是否在有效期间：

IF ls_condition-valid_from <= today
   AND ls_condition-valid_to   >= today.
  out->write( |Condition is valid| ).
ELSE.
  out->write( |Condition is not valid on { today DATE = USER }| ).
ENDIF.

取年月日片段只用于观察，不用于月份计算：

DATA doc_date TYPE d VALUE '20260613'.

DATA(doc_year)  = doc_date(4).
DATA(doc_month) = doc_date+4(2).
DATA(doc_day)   = doc_date+6(2).

这些例子都故意只处理 calendar day。若需求出现 営業日、稼働日、月末、UTC、タイムゾーン，就要进入项目日历、timestamp 或业务 API，而不是继续堆简单加减。

常见误区

  • 用字符串拼接构造日期，例如把 '20260231' 当作“2 月 31 日”，直到运行或校验时才出错。
  • 把 today + 30 当作“下个月”：月长、月末、闰年、业务期间都会让结果不可靠。
  • 把 today + 7 当作“7 个营业日后”：没有考虑 factory calendar、holiday calendar、公司假日。
  • 把 initial date 00000000 当作真实日期参与计算：未设置日期和有效日期被混在一起。
  • 有效期判断不写清边界：valid_to >= today 和 valid_to > today 含义不同。
  • 用 offset 修改月份或日：看起来像在处理日期，实际是在改字符。
  • 把 sy-datum 当作审计时间或跨时区时间戳：它不是 UTC timestamp，也不解决 user time zone。
  • 在单元测试里直接依赖“今天”：测试跨日期运行时结果变动，应该隔离基准日或注入 key date。

思考题

  • TYPE d 为什么既像字符字段，又能参与日期加减？
  • today + 7 是 calendar days 还是 working days？
  • 两个 date 相减得到什么类型的结果？它适合哪些业务判断？
  • 00000000 可以直接参与日期计算吗？为什么？
  • 有效期判断里，valid_to >= today 和 valid_to > today 的业务差异是什么？
  • 为什么不能用 offset/length 手写“下个月”？
  • 什么时候必须从 date arithmetic 升级到 timestamp / time zone 处理？
