15. Behavior definition 行为定义

  • Chapter 03
  • Section 15
  • self-contained study note

15.1 一句话结论

Behavior definition 行为定义，通常简称 BDEF，是 RAP business object 的事务契约：它说明这个业务对象允许哪些标准操作和自定义操作、哪些字段可以被修改、何时做校验和派生处理、如何加锁、如何做授权、是否支持 draft，以及这些行为由哪个 behavior pool 实现。

如果说 Root view entity 根视图实体 和 Composition tree 组合树 负责描述“这个业务对象长什么样、由哪些实体组成”，那么 behavior definition 负责描述“这个业务对象能怎么被安全地改变”。它不是普通的 class，也不是 service controller，而是 RAP 框架读取的行为声明。

15.2 概念说明

在 RAP 中，业务对象不是简单把一张数据库表暴露成 API。一个业务对象通常有 root entity、child entity、projection view、service definition、service binding、behavior definition 和 behavior implementation 等多个对象共同组成。BDEF 位在其中的“事务行为层”：它告诉 RAP runtime，外部消费者能不能 create / update / delete，哪些字段只读或 mandatory，哪些 action 可以被调用，保存前要执行哪些 validation，字段变化后要执行哪些 determination，以及锁、授权、ETag、draft、numbering 和 save sequence 应该如何参与。

初学者可以把 BDEF 想成业务对象的“可操作说明书”。只有 CDS view entity，并不等于这个对象已经可以事务性地创建、修改、保存。CDS 定义数据模型和关联关系，BDEF 定义行为边界。比如一个 Travel BO 可以允许创建 travel header、通过 association 创建 booking item、禁止直接修改系统生成字段、保存时检查日期合法性、创建时自动生成编号、提供一个 acceptTravel action，并要求所有修改都走 RAP 的事务缓冲和 save sequence。

BDEF 的典型写法以 managed implementation in class ... unique; 或 unmanaged implementation in class ... unique; 开始，后面跟 define behavior for ... alias ... 的 entity behavior body。现代示例里还常见 strict ( 2 );，它让 BDEF 使用更严格的语法检查，避免继续写旧式或模糊写法。managed 场景中，RAP runtime 可以承担标准 CRUD 和持久化的一部分工作；unmanaged 场景中，开发者需要在 behavior implementation 中实现更多标准操作和保存逻辑。projection behavior definition 又不同：它通常基于 projection root view，决定消费层暴露哪些底层行为，而不是重新选择 managed / unmanaged。

这一节不要求一次记住 BDL 的所有语法。真正要建立的判断是：BDEF 是事务型 RAP BO 的中心入口。看一个 RAP 对象是否可修改、能执行哪些 action、保存前会不会校验、是否支持 draft、权限和锁怎么挂上去，第一反应应该是打开 behavior definition。

15.3 边界

BDEF 不定义数据库表结构。字段的数据类型、主键、外键、technical fields 和基础关联，主要来自 DDIC table 和 CDS view entity。BDEF 可以声明字段特性，例如 readonly、mandatory、numbering、readonly during update 或 suppress，但它不是表设计工具。

BDEF 也不直接承载业务代码。validation、determination、action、authorization、feature control、unmanaged CRUD 和 saver 逻辑通常在 behavior pool 的 local handler class 或 saver class 中实现。BDEF 的职责是声明“有哪些行为、什么时候触发、由哪个类实现”，behavior pool 才是写 ABAP 逻辑的地方。把复杂逻辑堆在 BDEF 里是不可能的；把行为声明散落在普通 class 里，则会让 RAP runtime 无法理解对象能力。

BDEF 不是 OData service definition。service definition / service binding 决定如何把 projection 暴露为 OData 或 UI 服务，BDEF 决定这些暴露出来的实体能否被事务性操作。一个服务可以只读暴露，也可以通过 projection behavior 暴露一部分 action 或 CRUD；但服务层不应该绕过 BDEF 自行决定保存语义。

BDEF 也不是权限角色本身。它可以声明 authorization master、authorization dependent、instance authorization 或 global authorization 等行为入口，但具体权限判断仍要结合 DCL、authorization object、handler 实现和项目权限设计。初学者容易误解为“写了 authorization 就自动安全”，实际还要看授权检查是否实现、是否覆盖 create/update/delete/action、是否在 projection/service 层正确暴露。

15.4 核心机制

第一层机制是标准操作。BDEF 可以声明 create;、update;、delete;，并通过 association 支持 create-by-association。managed BO 中，标准操作通常由 RAP runtime 协调持久化；unmanaged BO 中，开发者要实现相应 handler 方法。只读对象可以不开放这些修改操作。项目上判断 API 是否允许变更，不能只看 OData endpoint 是否存在，而要看 BDEF 是否开放相应 operation。

第二层机制是字段控制、编号和字段映射。BDEF 可以把字段标成 readonly、mandatory、numbering managed、readonly during update 等。它解决的问题是：哪些字段由用户输入，哪些字段由系统生成，哪些字段只在创建时可填，哪些字段在更新时不能改。比如 technical key、created by、created at、last changed at 通常不应由前端随意传入；业务必填字段则应该在行为层或 validation 中明确约束。创建场景还要注意 numbering：前端或 EML 调用方可能先用 %cid 标识临时实例，保存后再由 RAP 映射到最终 key。若 CDS element 名称和持久化表字段名不一致，managed BO 还要在 BDEF 中用 mapping for 明确字段映射，否则“看起来字段都有”，保存时却可能无法按预期写入。

第三层机制是 determination。determination 是由 RAP 框架按触发条件调用的派生处理，用来根据当前实例状态计算或补齐字段。SAP Learning 对 determination 的典型说明是：它可以在 on modify 或 on save 阶段执行，并由 create、update、delete 或字段变化触发。比如输入数量和单价后自动计算金额、创建时默认状态为 open、保存前补齐派生字段。determination 应该改变业务对象实例本身，而不是做不可回滚的外部副作用。

第四层机制是 validation。validation 用于检查一致性，在 on save 阶段执行。它不能只是弹出提示，而是要通过 failed 和 reported 告诉 RAP runtime 哪些实例不能保存、哪些消息要返回给消费者。比如日期不能早于今天、数量必须大于零、child item 必须属于有效 parent、状态转换必须合法。validation 是把“保存前必须满足的业务规则”放进 BO 生命周期的关键机制。

第五层机制是 action 和 function。action 是非标准的业务操作，可能改变对象状态；function 则用于返回信息且不应产生 side effect。比如 acceptTravel、rejectTravel、recalculatePrice 可以是 action；读取某个预测结果或模拟计算则更接近 function。初学者要避免把所有按钮都理解成 CRUD，也要避免把复杂状态变更藏在 update 字段里。RAP 里重要业务动作应该显式建模为 action。

第六层机制是事务缓冲和 save sequence。RAP 修改请求先进入 transactional buffer，最后由 COMMIT ENTITIES 触发 save sequence。保存过程中可能发生 finalize、check before save、adjust numbers、save、cleanup 等阶段，具体由 managed/unmanaged/custom save 场景决定。这里的项目规则非常重要：不要在普通 handler 里随意 COMMIT WORK，也不要在保存前做不可回滚的外部调用。RAP 需要统一管理成功、失败、回滚、消息和响应。

第七层机制是 lock、authorization、ETag 和 draft。BDEF 可以声明 lock master/dependent、authorization master/dependent、etag master，以及 draft 支持。它们不是装饰语法，而是多人并发、权限边界、乐观锁和草稿编辑的基础。比如 root 是 lock master 时，child 修改可能依赖 parent 的锁；draft-enabled BO 中，前端编辑的是 draft instance，激活时才进入 active data。BDEF 把这些事务治理能力放在一个可读入口里。

第八层机制是 managed / unmanaged / projection 的差异。managed 更适合标准持久化和框架托管；unmanaged 常用于遗留逻辑、复杂保存、BAPI/旧表封装或特殊事务控制；projection behavior 用于消费层裁剪和暴露行为。不要把 managed 当成“低级”、unmanaged 当成“高级”。项目选择取决于持久化模型、遗留资产、Clean Core 要求、测试能力和交付风险。

第九层机制是扩展性。S/4HANA Public Cloud、SAP BTP ABAP Environment 和 Clean Core 语境下，很多场景不希望直接修改原始对象，而是通过 released API、extension point 或 BDEF extension 做受控扩展。本节只需要知道：BDEF 不只是项目内部开发文件，它也会影响后续能否被安全扩展。真正的扩展语法和限制放到 Clean Core / extensibility 章节再展开。

15.5 项目意义

BDEF 的价值是把“代码能写”变成“对象能被稳定消费”。没有 BDEF，CDS 只是可查询模型；有了 BDEF，RAP runtime 才知道哪些实例可以被改、哪些字段可写、什么时候校验、如何返回消息、如何处理锁和事务、哪些 action 构成业务能力。它把业务规则从零散 controller / report / interface 代码中拉回到 business object 生命周期。

对项目交付来说，BDEF 是设计评审和缺陷分析的核心文件。出现“前端按钮点了没反应”“保存时报错但消息不清楚”“并发覆盖了别人修改”“child item 删除后 parent 状态没更新”“OData update 成功但数据库没变”等问题时，排查入口常常就是 behavior definition 和 behavior implementation 的对应关系。看 BDEF 可以快速确认：operation 有没有开放、validation 有没有定义、determination 触发点是否合适、action 是否暴露到 projection、锁和授权是否挂在正确实体上。

对 Java/Python 背景学习者来说，BDEF 有点像把 controller route、domain service capability、entity lifecycle callback、validation rule、transaction boundary、authorization hook 和 API mutability contract 集中声明在一个对象旁边。但它又不是普通框架配置文件，因为 ABAP compiler、RAP runtime、ADT quick fix、EML、OData 暴露和 Fiori Elements 都会围绕它工作。它是 SAP 现代开发中“模型驱动事务行为”的关键入口。

对日本 SAP 项目来说，BDEF 会进入设计说明、レビュー、単体テスト、結合テスト缺陷分析和移送前确认。常见问题不是“代码有没有写”，而是 振る舞い定義上、更新可能になっていますか、保存時検証はどこで実行されますか、アクションは projection 側に公開されていますか、ロックと権限の親子関係は正しいですか。能读懂 BDEF，才能把 RAP 技术对象翻译成项目可确认的交付边界。

15.6 Example / 学习例子

以 Travel / Booking 类业务对象为例，root view entity 是 Travel，child entity 是 Booking，二者通过 composition 组成同一个 BO。BDEF 中可以对 Travel 声明 create; update; delete;，并声明 _Booking { create; }，表示可以通过 Travel 创建 Booking。Booking 自身可能允许 update/delete，但不能脱离 Travel 独立创建。这样，API 消费者看到的不是两张孤立表，而是一个有生命周期关系的 composite business object。

再看字段控制。Travel 的 UUID、created at、created by、last changed at 通常是 system-managed 字段，BDEF 里可以声明为 readonly 或 managed numbering。BeginDate、EndDate、CustomerID 可能是业务输入字段，其中 CustomerID 可能 mandatory。这样前端即使传入 technical field，RAP 也不会把它当作普通业务输入；测试也能围绕“哪些字段能由用户改”来设计。

再看 validation。可以定义 validation validateDates on save { field BeginDate, EndDate; }，在保存时检查结束日期不能早于开始日期。实现方法在 behavior pool 中，读取 keys 对应的实例，发现错误就填充 failed 和 reported。消费者收到的是可定位到实例和字段的消息，而不是一个模糊的 ABAP dump 或后台日志。

再看 determination。创建 Travel 时，可以定义 determination setInitialStatus on modify { create; }，把状态设为 open；更新 Booking 数量后，可以触发 parent 金额重新计算。这里要注意：determination 是对象内部状态派生，不适合在里面直接发邮件、调用不可回滚外部 API 或提交数据库事务。

最后看 action。Travel 可以定义 action accept result [1] $self;，表示这是一个业务动作，不是普通 update。实现中检查状态、授权和必填信息后，把 Travel 状态改为 accepted，并通过 reported 返回消息。projection behavior 还要决定这个 action 是否暴露给消费层，否则底层 BDEF 有 action，服务消费者也未必能调用。

15.7 Counterexample / Failure mode

第一个失败模式是把 BDEF 当成“CRUD 开关”。只写 create; update; delete;，却不设计字段控制、validation、determination、lock、authorization 和 messages，短期能跑通，长期会变成“前端能保存任何东西”。这种 BO 很难通过项目 review，也很难在结合测试中解释错误。

第二个失败模式是在 handler 里直接 COMMIT WORK、直接更新别的表、直接调用不可回滚外部接口。这样会绕开 RAP 的事务缓冲和 save sequence，导致部分成功、回滚不一致、消息无法准确返回，甚至破坏 draft / EML / OData 的语义。RAP 里要把可回滚的数据修改交给 BO 生命周期管理，把外部副作用放在合适的保存后流程或集成设计中。

第三个失败模式是 managed / unmanaged 选择不清。为了“快”把复杂遗留保存硬塞进 managed，后面发现标准持久化无法表达；或者为了“灵活”全部做 unmanaged，结果自己承担 CRUD、锁、响应、测试和错误处理，成本激增。正确做法是先画清楚持久化对象、遗留接口、事务边界和 Clean Core 要求，再选择 implementation scenario。

第四个失败模式是只改底层 BDEF，忘记 projection behavior。开发者在 root BDEF 里新增 action 或开放 update，但消费服务用的是 projection view，如果 projection behavior 没有暴露对应行为，Fiori / OData 消费者仍然不能调用。排查时要同时看 interface/root BDEF、projection BDEF、service definition 和 service binding。

第五个失败模式是把 authorization 和 validation 混在一起。validation 检查业务数据是否一致，authorization 判断当前用户是否有权执行操作。用 validation 模拟权限会让错误语义混乱，也可能漏掉 read/update/action 等不同入口。项目上应把二者分开设计，并在测试中覆盖。

15.8 日本项目现场说法

  • 常见日语：ビヘイビア定義、振る舞い定義、ビジネスオブジェクト、保存シーケンス、検証、派生処理、アクション、ロック制御、権限制御、ドラフト対応。正式说明中优先使用 ビヘイビア定義；振る舞い定義 更适合作为中文读者理解含义时的解释性说法。
  • 说明句：この RAP BO では、更新可否・保存時検証・アクションの公開範囲をビヘイビア定義で確認してください。
  • 说明句：保存時のチェックは validation、項目の自動補完は determination、業務ボタンの処理は action として設計します。
  • 说明句：projection 側で公開されていないアクションは、サービス利用者から呼び出せません。
  • 使用场景：RAP 基本设计、详细设计、code review、単体テスト項目作成、Fiori/OData 结合测试缺陷分析、移送前确认、Clean Core 审查。
