5. RAP instance authorization 实例授权

  • Chapter 04
  • Section 05
  • self-contained study note

5.1 一句话结论

RAP instance authorization 是 RAP behavior 层的“对象实例级授权”：它回答的是“当前用户是否允许对这一条具体业务对象实例执行某个操作”，例如某张单据、某个上传批次、某条申请、某个状态下的对象是否允许当前用户 update、delete、approve 或 cancel。

它和 RAP global authorization 全局授权 的差异在于：global authorization 先判断用户是否总体具备某类操作资格；instance authorization 再结合具体实例的数据、状态、组织维度、责任人、锁定/生命周期等条件判断是否允许。一个用户可以总体拥有 approve 权限，但只能 approve 自己组织内、状态为 submitted、且尚未被其他人处理的单据。

5.2 到底学什么

本节学习 instance authorization 的设计判断，而不是背某个固定代码模板。

第一，instance authorization 依赖具体实例。它通常需要读取当前业务对象实例的字段，例如 company code、plant、status、created by、responsible user、approval group、lock status 或业务阶段。没有实例数据，就无法判断“这一个对象是否允许当前用户操作”。

第二，它控制的是行为许可，不是列表读取本身。读取列表时的行级可见性更常见于 CDS Access Control DCL 行级访问控制；instance authorization 更关注用户已经定位到某个对象后，是否允许对它执行更新、删除、action 或某些实例级行为。

第三，它不是普通 validation。validation 关心数据是否业务有效，例如金额是否为空、日期是否合理、状态流转是否合法；instance authorization 关心当前用户是否有资格对这个实例做这件事。两者经常一起出现，但错误原因、测试用户和修复团队不同。

第四，它需要和 feature control 区分。feature control 可以影响 UI 上按钮/字段是否可用，也可以表达某个实例当前是否处于可操作状态；但它不是用户授权边界。API 直接调用 action 时，后端 instance authorization 仍必须成立，不能用“按钮已经隐藏”来证明安全。

第五，它需要和 Precheck 预检查、Enqueue locking 加锁设计、validation 分工。precheck 可以早期拒绝明显非法请求；locking 防止并发处理同一对象；validation 判断业务状态和输入是否有效；instance authorization 判断“这个用户对这个实例有没有资格”。这些机制经常在同一个 action 里连续出现，但错误分类、测试证据和修复 owner 不同。

第六，它要考虑批量语义。RAP/EML/OData action 可能一次处理多条实例。instance authorization 不能只按“整个请求是否有权限”回答，还要能说明每一条实例是否允许、失败原因是什么、是否允许部分成功，以及这种部分成功是否符合事务边界和业务规则。

5.3 为什么重要

很多真实 SAP 授权问题不是“用户有没有某个菜单权限”，而是“用户对这条业务数据有没有操作资格”。例如：采购申请只能由当前审批人批准；销售订单只能由所属销售组织维护；上传批次只能由创建人取消；已完成状态的单据任何人都不能再修改；跨公司代码的数据即使在列表中可见，也不能由非负责公司用户执行 release。

如果只做 global authorization，这些差异会被抹平：有权限的人能操作所有实例，无权限的人一个也不能操作。实际项目往往不是这么简单。instance authorization 让 RAP business object 在行为层表达这种细粒度规则，使 UI、API、测试和错误反馈都围绕同一套业务对象逻辑。

从 Java/Python 背景看，可以把它类比成 domain object / aggregate 上的 per-record policy check。但 RAP 的差异在于：这个判断发生在 behavior 模型里，和 EML、OData action、failed/reported 消息、save sequence、draft/active 状态、feature control 和 transaction behavior 紧密相关。

instance authorization 的沟通价值偏“细节证据”。测试人员经常会问：“为什么同一个用户能取消 A 批次，不能取消 B 批次？”“为什么同一张列表里有些行按钮可用，有些不可用？”“API 一次传 10 条，为什么 6 条成功 4 条失败？”这些问题如果没有实例级授权矩阵，就会被误判成角色配置不稳定或系统随机失败。

5.4 Key mechanism / 核心机制

可以用一张判断表理解 global 和 instance 的分工：

Operation: approve purchase request

Global authorization:
  Does this user generally have approval authority?

Instance authorization:
  Is this purchase request in a state that can be approved?
  Does it belong to the user's company code / plant / approval group?
  Is the current user the responsible approver or a delegated approver?
  Has it already been approved, cancelled, locked, or completed?

instance authorization 的典型设计步骤如下：

  1. 明确哪些 operation/action 需要实例级判断。
  2. 找出判断所需的实例字段：状态、组织、责任人、创建人、金额、对象类型、生命周期阶段等。
  3. 在 RAP 授权入口或相关行为实现中读取必要实例数据。
  4. 对不允许的实例返回可理解的 failed/reported 信息，而不是静默跳过。
  5. 验证批量场景：同一次请求里部分实例允许、部分实例拒绝时，响应是否清楚。
  6. 记录测试矩阵：用户、实例状态、组织字段、预期结果、实际结果、错误消息。

读代码时，可以先找这些实现钩子：

这里不要求初学者马上写完整模板代码，但要知道这些词在 ADT 和 SAP Help 中意味着什么。instance authorization 的 review 不是只看“有没有写 AUTHORITY-CHECK”，而是看它是否拿到了正确实例字段、是否按实例返回结果、是否默认安全失败、是否覆盖 OData/EML/批量调用。

特别要注意批量请求。RAP/OData/EML 常常不是一次只处理一条数据。instance authorization 必须能说明多条实例混合时的行为：哪些实例成功、哪些失败、失败原因是什么、是否会产生部分成功、是否符合业务和事务边界设计。

还要区分普通外部入口和内部 EML。一般 UI/OData/EML 调用应经过 RAP 授权控制；但在授权 handler 内部，为了读取“用于判断权限的实例状态”，常会看到 READ ENTITIES ... IN LOCAL MODE。这是避免递归授权检查的内部实现模式，不是给外部调用方绕过授权的通道。任何在 authorization、validation、determination 等框架指定内部位置之外使用 IN LOCAL MODE 或 PRIVILEGED 的生产代码，都需要安全理由和 review 证据。

可以用下面的矩阵设计实例级规则：

维度示例字段典型判断不应混淆为
组织范围company code, plant, sales org用户是否负责该组织的数据只看 global action 权限
责任人created by, approver, owner当前用户是否是创建人/审批人/代理人UI 按钮控制
状态draft, submitted, executed, posted, cancelled当前状态是否允许此用户执行 action输入 validation 的全部
金额/风险等级amount, risk level超过阈值时需要更高角色或审批组普通字段校验
对象类型internal/external, standard/custom不同对象类型有不同授权路径单一 role 名称

一个实用的设计排查顺序是：

1. global authorization:
   user may generally call this operation?

2. locking / latest-state check:
   instance is locked and current enough for modify processing?

3. instance authorization:
   user may operate these specific instances?

4. precheck:
   request shape and cheap conditions are valid?

5. validation / save sequence:
   business data can be persisted consistently?

实际运行顺序由 RAP 框架和具体实现方式决定，项目中要以目标系统文档为准；但设计评审时至少要能解释这几层的责任。global authorization 是第一道总体 gate；global 通过后，instance authorization 才有机会按具体实例判断。若 instance authorization 拒绝某个实例，其他允许实例仍可能继续处理，具体是否 partial success 或 all-or-nothing 要由业务事务设计和 RAP 响应结构说明。

5.5 日本项目现场说法

  • 常见日语：インスタンス権限、伝票単位の権限、オブジェクト単位の権限、担当者チェック、ステータスチェック、承認権限、組織別権限。
  • 设计说明：操作自体の可否は global authorization で判定し、対象伝票の会社コード・ステータス・担当者条件は instance authorization で判定します。
  • Review 说法：同じ action でも、対象インスタンスの状態や担当者によって許可・拒否が分かれるケースをテストしてください。
  • 测试说法：同一ユーザで、承認可能な伝票と承認不可の伝票を同時に API で呼び出し、成功・失敗の返却が明確か確認します。
  • 缺陷调查：全件不可なら global authorization、特定伝票だけ不可なら instance authorization または業務ルールを優先して確認します。
  • 设计书条目：インスタンス単位の判定項目として、会社コード、ステータス、担当者、金額条件、代理承認者を一覧化します。
  • API 说明：複数インスタンスを一括処理する場合、許可された明細と拒否された明細の返却仕様を明確にします。
  • 边界说明：ボタンの活性制御は UX 上の補助であり、API 直接呼び出し時の後端権限制御を省略できる根拠にはなりません。
  • 证据要求：STAUTHTRACE、SU53、OData エラーログ、アプリケーションログ、テスト結果をそろえて、権限設計書とテスト仕様書に承認記録を残します。

5.7 Example / 学习例子

假设 AC0567C 类似的上传服务有一个 cancel action。业务规则如下：

  • 创建人可以取消自己的上传批次。
  • 管理员可以取消任何未执行批次。
  • 批次状态为 EXECUTED 或 POSTED 后不能取消。
  • 其他用户即使能看到批次，也不能取消。

这里 global authorization 只回答“用户是否总体允许执行 cancel action”。instance authorization 还要读取每个上传批次的创建人、状态和可能的组织字段，逐条判断是否允许。

测试时不要只测一条成功数据。应准备至少四条实例：自己的 draft 批次、别人的 draft 批次、自己的已执行批次、管理员操作别人的未执行批次。API 响应要能说明哪些被拒绝、为什么拒绝，并且不应因为一条无权限实例导致不该成功的实例被错误处理，除非业务事务设计明确要求 all-or-nothing。证据上要同时看 API response、failed / reported 语义、后台数据是否未误改、STAUTHTRACE / SU53 / /IWFND/ERROR_LOG / application log 是否能解释失败原因。

可以把测试矩阵写得更具体：

用户实例状态关系/组织action预期
Creator_ABatch ADRAFT创建人，company code 1000cancel允许
Creator_ABatch BDRAFT非创建人，company code 1000cancel拒绝，非 owner
Creator_ABatch CEXECUTED创建人，company code 1000cancel拒绝，状态不允许
Admin_UBatch BDRAFT管理员，company code 1000cancel允许
User_XBatch DDRAFTcompany code 2000 无授权cancel拒绝，组织权限不足

在 AC0567C 风格的上传 API 中，execute 也可能需要 instance authorization。比如用户总体允许 execute，但只允许执行自己负责 company code 的 batch；或者只能执行状态为 VALIDATED 的 batch；或者超过金额阈值的 batch 需要管理员角色。这些规则不是 DCL 能完整解决的，因为 DCL 主要控制读取结果；也不是 precheck 能最终保证的，因为状态和责任人可能在请求到达后变化。

批量请求的响应要设计清楚。若业务允许部分成功，返回中应能看到每个 batch id 的 allowed / rejected、原因和消息；若业务要求 all-or-nothing，也要明确“一条无权限导致整批拒绝”的规则，并说明不会产生部分副作用。无论哪种模式，测试都要验证数据库状态、application log 和 API response 是否一致。

5.8 Counterexample / Failure mode

常见失败模式有十类。

第一，只做 global authorization。只要用户拥有 action 权限，就能操作所有实例，导致跨组织或跨责任人越权。

第二，把业务 validation 当成授权。状态不允许操作和用户无权操作是不同错误；混在一起会让测试、审计和用户沟通都变混乱。

第三，只在 UI 上 disabled 按钮。API 直接调用仍然可能绕过实例级规则。

第四，批量请求处理不清。多条实例中一部分无权限时，代码要么全部失败但不说明原因，要么静默跳过失败项，造成对账困难。

第五，实例字段读取过多或过晚。授权判断需要必要字段，但不应为了授权把复杂业务计算提前塞进 handler；也不应在产生副作用后才判断实例权限。

第六，错误消息泄露过多。无权限时需要可测试、可支持，但不能把其他组织的敏感字段、审批人或内部规则暴露给不应知道的用户。

第七，只测一个实例。用一条自己有权限的数据测试成功，不能证明 instance authorization 有效；必须准备允许、拒绝、边界状态、跨组织、批量混合等实例。

第八，把 instance authorization 写成复杂业务计算。授权判断需要必要字段，但不应承担整套定价、库存、会计校验或外部系统检查。复杂业务有效性应回到 validation、determination、save sequence 或业务 API。

第九，失败原因没有分层。用户无权、状态不允许、对象被锁、幂等重复、输入错误如果都返回“action failed”，调用方无法修正，也无法判断是不是权限缺陷。

第十，读数据和执行 action 的权限不一致但未说明。用户可能能看到列表，但不能执行 action；这本身可以是正确设计，但必须在说明书和测试预期里写清楚。

修正动作：为每个敏感 action 建一张 instance authorization matrix，列出实例状态、组织字段、责任人、用户类型、预期允许/拒绝、错误消息、API 响应和日志证据。

5.9 思考问题

  • 这个规则需要看具体实例字段吗？如果需要，它可能是 instance authorization。
  • 需要哪些字段才能判断：状态、公司代码、工厂、创建人、审批人、金额、对象类型？
  • 同一个用户对不同实例的结果是否可能不同？
  • 批量请求里部分实例失败时，响应格式是否能解释清楚？
  • 错误消息是否区分“无权限”和“状态不允许”？
  • 如果用户能看到实例但不能执行 action，这是否是预期设计？
  • 同一批请求中允许和拒绝实例混合时，事务边界如何定义？
  • AC0567C POC 中哪些 action 需要按批次实例状态或创建人判断？
