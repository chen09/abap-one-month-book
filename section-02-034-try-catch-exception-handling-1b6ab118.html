Chapter 02 - Core ABAP Programming

34. TRY CATCH exception handling

  • Chapter 02
  • Section 34
  • self-contained study note

34.1 一句话结论

TRY CATCH exception handling 是 ABAP 处理 class-based exception 的控制结构。TRY 包住可能抛异常的代码，CATCH 按 exception class 类型处理错误，未被处理的异常会沿调用链继续传播；真正好的异常处理不是“把错误挡住”，而是把错误转换成调用方、用户、日志、测试或框架能理解的结果。

上一节 Exception class 异常类 讲的是“错误用什么类型表达”。本节讲的是“错误发生后控制流怎么走”。初学者要特别记住：TRY...CATCH 不是到处套一层 CATCH cx_root 让程序继续跑，而是把异常路径写成可读、可测试、可追踪的流程。能处理的地方处理，不能处理的地方保留上下文继续传播。

34.2 到底学什么

本节要掌握八个点：

  • TRY...ENDTRY 定义一段受异常处理保护的代码范围。
  • CATCH zcx_... INTO DATA(lx_...) 捕获指定 exception class，并拿到 exception object。
  • 多个 CATCH 要从具体到通用；先写 superclass 会遮蔽 subclass，相关语法检查会把这种顺序问题当作语法错误处理。
  • 一个 CATCH 可以列出多个 exception class，用同一段处理逻辑处理一组错误；同一条 CATCH 的 class list 里也要遵守“具体在前、通用在后”的顺序。
  • 没有被当前 TRY 的 CATCH 处理的异常，会向上层 method / caller 传播；如果最终无人处理，会变成 runtime error / short dump。
  • method 可以选择本地 CATCH，也可以在 signature 中 RAISING 继续传播，但这必须是有意识的 contract。
  • CLEANUP 是异常传播时的清理块，不是 Java 的 finally；不要把它当作“无论成功失败都执行”的通用收尾逻辑。
  • CATCH cx_root 可以做兜底或测试基础设施处理，但空 catch / 吞异常是反模式。

最小形态：

TRY.
    DATA(lv_price) = price_service->get_price( iv_product_id ).
  CATCH zcx_price_missing INTO DATA(lx_missing).
    DATA(lv_text) = lx_missing->get_text( ).
    " convert to user-facing message, log, or test assertion
ENDTRY.

这段代码表达的是：正常路径希望得到 price；如果 price 缺失，用 zcx_price_missing 这条异常路径处理。TRY...CATCH 的价值，不是让失败“消失”，而是让失败进入明确的处理分支。

34.3 为什么重要

SAP 项目里的异常处理通常不是一个纯语法问题，而是交付质量问题。

如果 TRY...CATCH 写得清楚，代码 review 可以直接看出：

  • 哪一段代码可能失败。
  • 哪些 exception class 被本地处理。
  • 哪些异常会继续传播给上层。
  • 用户消息、日志、RAP reported / failed、ABAP Unit assertion 或技术异常如何衔接。
  • 是否存在 例外握りつぶし，也就是 catch 了却什么都不做。

如果写得不清楚，问题会很隐蔽。比如更新逻辑失败但被空 CATCH cx_root 吞掉，前台用户看起来“保存成功”，后台数据却没有写入；或者 batch job 里所有异常都被转成同一个泛泛的 zcx_error，运维无法判断是权限、配置、数据还是外部系统问题。

对 Java/Python 背景学习者来说，TRY...CATCH 的概念很熟，但 ABAP 有自己的差异：异常类分 CX_STATIC_CHECK / CX_DYNAMIC_CHECK / CX_NO_CHECK；method signature 的 RAISING 是接口 contract；CATCH 可以按 ABAP exception class 继承树匹配；CLEANUP 不是 Java finally。这些差异会直接影响生产代码的可维护性。

34.4 Key mechanism / 核心机制

34.4.1 TRY defines the protected block

TRY 后面的语句块是“可能发生异常、并希望由后续 CATCH 处理”的范围。范围越大，越难判断到底哪一句抛了异常；范围越小，错误路径越清楚。TRY...ENDTRY 也必须完整落在同一个 processing block 里，例如同一个 procedure、event block 或 dialog module 内，不能跨 method 或跨事件块。

裸的 TRY 不是普通代码分组。初学阶段可以简单记：写 TRY 就要有明确的异常处理结构，通常是至少一个 CATCH；如果只是用 TRY...ENDTRY 包住代码、没有任何有意义的 CATCH / CLEANUP 处理分支，语法检查和代码 review 都会把它当作问题。CLEANUP 只在确实需要异常传播清理时使用，不要把 TRY 当成普通代码分组。

不推荐：

TRY.
    read_input( ).
    check_authorization( ).
    calculate_price( ).
    post_document( ).
    send_mail( ).
  CATCH cx_root INTO DATA(lx_root).
    log->error( lx_root->get_text( ) ).
ENDTRY.

这段代码虽然有 catch，但 catch 太宽、try 范围太大。发生错误时，读者很难判断是输入、权限、价格、posting 还是通知失败。

更好的方向是让 method 拆分清楚，让不同异常在合适的层级处理：

TRY.
    DATA(ls_document) = document_factory->create_from_input( is_input ).
  CATCH zcx_document_invalid INTO DATA(lx_invalid).
    message_mapper->add_validation_message( lx_invalid ).
    RETURN.
ENDTRY.

poster->post_document( ls_document ).

这里本地只处理“输入不合法”。真正 posting 的异常让上层处理，或者由 post_document 的调用方根据 RAISING contract 决定。

34.4.2 CATCH matches by exception class

CATCH 后面写 exception class。运行时如果抛出的异常是该 class 或其 subclass，对应的 CATCH block 会执行。

TRY.
    calculator->divide( ).
  CATCH cx_sy_zerodivide INTO DATA(lx_zero).
    out->write( lx_zero->get_text( ) ).
  CATCH cx_sy_arithmetic_error INTO DATA(lx_arithmetic).
    out->write( lx_arithmetic->get_text( ) ).
ENDTRY.

cx_sy_zerodivide 是更具体的异常，必须放在更通用的 cx_sy_arithmetic_error 前面。否则通用 superclass 先匹配，后面的具体 catch 永远到不了，相关语法检查会把这种顺序视为语法错误。

34.4.3 One CATCH can handle a group

如果多个异常的处理动作完全相同，可以写在同一个 CATCH：

TRY.
    result = number1 / number2.
  CATCH cx_sy_arithmetic_overflow cx_sy_zerodivide INTO DATA(lx_arithmetic).
    out->write( lx_arithmetic->get_text( ) ).
ENDTRY.

这适合“处理策略相同”的情况。同一条 CATCH 里列多个 class 时，也不要把 superclass 放在 subclass 前面。不要因为语法允许就把语义不同的错误强行合并。比如 zcx_document_invalid 和 zcx_posting_failed 通常不该合并：前者更像用户输入或业务校验，后者可能是技术失败、锁、配置或外部依赖。

34.4.4 INTO gives access to the exception object

INTO DATA(lx_error) 让 catch block 拿到 exception object。最常见动作是调用 get_text( )：

CATCH zcx_document_invalid INTO DATA(lx_invalid).
  DATA(lv_message) = lx_invalid->get_text( ).

更复杂时，还可以读取 exception class 的公开属性、previous、source position 或 long text。初学阶段不要把 catch 写成“只知道发生了异常”，而要习惯拿到对象，保留错误证据。

34.4.5 Propagation is also a decision

不是每个 method 都应该本地 catch。当前层不知道如何处理时，继续传播通常比乱处理更好。

METHOD post_order
  RAISING zcx_document_invalid
          zcx_posting_failed.

  validator->check( is_order ).
  posting_service->post( is_order ).

ENDMETHOD.

这里 post_order 没有本地 TRY...CATCH，而是把两个异常放进 RAISING。这不是偷懒，而是告诉调用方：本 method 负责执行业务动作，但最终错误反馈策略由更上层决定。

相反，如果代码只是 catch 以后立刻原样再抛、没有日志、没有转换、没有补上下文，也没有恢复动作，通常没有价值：

TRY.
    posting_service->post( is_order ).
  CATCH zcx_posting_failed INTO DATA(lx_failed).
    RAISE EXCEPTION lx_failed.
ENDTRY.

这种情况通常直接在 method signature 中声明 RAISING zcx_posting_failed 更清楚。只有当 catch block 会补充上下文、包装为更合适的上层异常、写必要日志或转换为 message 时，本地 catch 才有实际意义。

34.4.6 CATCH cx_root is a tool, not a design default

cx_root 是所有 class-based exception 的根，所以 CATCH cx_root 能捕获很宽的一类异常。它有合理场景：

TRY.
    test_environment->prepare( ).
  CATCH cx_root INTO DATA(lx_root).
    cl_abap_unit_assert=>fail( lx_root->get_text( ) ).
ENDTRY.

在 ABAP Unit setup 或测试辅助代码里，这种写法可以把任何准备阶段异常转成清楚的 test failure。问题不是 CATCH cx_root 这个语法，而是生产代码里空处理：

TRY.
    poster->post_document( document ).
  CATCH cx_root.
ENDTRY.

这段代码就是典型 例外握りつぶし。它把失败从用户、日志、测试和运维面前藏起来，后续问题会更难查。

极少数场景下，如果 catch 后确实故意不做任何动作，代码里也应该有注释解释“为什么可以忽略”。如果不需要 exception object，就不要写 INTO DATA(lx_...) 假装会使用它。

34.4.7 CLEANUP is not Java finally

ABAP 的 CLEANUP 用于异常离开当前 TRY 范围、并继续向外传播时做清理。它不是 Java / C# 里“无论成功、失败、被 catch 都执行”的 finally。如果异常已经被同一个 TRY 里的 CATCH 本地处理，CLEANUP 不是用来再跑一遍通用收尾逻辑的地方。

可以把它当作高级意识点：如果某段代码获取了临时资源，而异常不在本层处理、要继续往外抛，CLEANUP 可以用于释放或恢复本层资源。但初学者不要把所有收尾逻辑都塞进 CLEANUP。普通成功路径的收尾，应写在正常控制流里；明确需要异常传播清理时，再考虑 CLEANUP。Clean ABAP 风格下，很多场景更推荐把 method 做小、让资源生命周期清楚，从而根本不需要复杂 CLEANUP。

形态示意：

DATA(lo_resource) = resource_pool->acquire( ).

TRY.
    sender->send_with_resource( lo_resource ).
  CLEANUP.
    lo_resource->release( ).
ENDTRY.

这里表达的是：如果异常没有在这个 TRY 内被 CATCH 处理，而是向外传播，本层要释放资源。真实项目里还要看资源对象、framework 生命周期和是否已有自动释放机制，不要机械套模板。

34.4.8 Error mapping belongs at the boundary

底层异常不一定直接暴露给最终用户。常见做法是：

  • service/helper 层：抛清楚的 class-based exception。
  • application boundary：转换为 message、log、RAP reported / failed 或 HTTP/OData error response。
  • test code：用 ABAP Unit assertion 验证是否抛出预期异常。

也就是说，TRY...CATCH 常常出现在“边界层”：用户交互边界、接口边界、batch job 边界、RAP behavior 边界、测试边界。业务内部的小 helper 如果不知道如何恢复，不必每层都 catch。

ABAP Cloud / RAP 里还要再加一条边界：新代码应使用 class-based exception，不应继续扩散 classic / non-class-based exception；在 ABAP Cloud 约束下，客户代码还要确认要捕获或抛出的 SAP-delivered exception class 是否在当前系统里是 released API。业务 validation failure 通常进入 reported-<entity> / failed-<entity>，而不是被普通 TRY...CATCH 包成技术异常。框架级或基础设施级失败才可能使用 exception class，例如 save sequence 的框架失败；这类异常是否能在 cloud-compliant 代码中使用，要以当前系统的 released API 状态和官方说明为准。

34.4.9 catch action checklist

写完 CATCH ... INTO DATA(lx_...) 以后，review 时要立刻问：这一段 catch 到底做了什么？通常只有下面几类动作是有意义的：

ActionWhen it is appropriateWatch out
Handle locally当前层能给出默认值、替代路径或明确用户消息。不要为了“继续跑”而掩盖真实失败。
Translate at boundary把底层 exception 转换为 RAP message、HTTP error、job status 或上层业务 exception。转换时要保留 business key、field、original text 或 previous。
Log evidence后台 job、接口、批处理需要后续调查。log 不等于处理；记录后仍要决定是否中断、重试或传播。
Rethrow当前层不能恢复，但要保持错误语义继续向上。原样 catch 再原样 raise 通常没有价值，除非补充上下文。
Fail testABAP Unit setup / assertion helper 捕获任意异常后让测试失败。生产代码不要照搬测试基础设施的 CATCH cx_root 模式。
Cleanup during propagation异常向外传播时释放临时资源。CLEANUP 不是 Java finally，不要放普通成功路径也必须执行的业务动作。

一个简单的 code review 句子是：CATCH 后必须能回答“用户看到什么、日志留下什么、上层得到什么、事务是否继续”。如果四个问题都答不上来，这段异常处理很可能只是把错误藏起来。

34.5 日本项目现场说法

  • 常见日语：例外処理、TRY-CATCH、例外捕捉、例外伝播、例外送出、再送出、例外握りつぶし、異常系、異常系テスト、ログ出力、エラーメッセージ変換、後続処理中断。
  • 设计说明句：この処理では入力チェック例外のみを捕捉し、業務メッセージに変換します。技術例外は上位層へ伝播させます。
  • Review 指摘句：CATCH cx_root の空ブロックで例外を握りつぶしているため、具体的な例外クラスで捕捉し、ログ出力または再送出を行ってください。
  • Review 精确句：CATCH cx_root 自体が禁止ではありません。問題は、例外内容を記録・変換・再送出せずに握りつぶすことです。
  • Java 经验者提示句：ABAP の CLEANUP は Java の finally と同じではなく、例外が外側へ伝播する場合のクリーンアップとして説明してください。
  • 测试说明句：異常系テストでは、対象メソッドが想定どおり zcx_document_invalid を送出することを確認します。
  • 使用场景：详细设计、method signature review、错误消息设计、ABAP Unit、batch job 障害調査、RAP validation / action 错误返回、上线前异常路径检查。

34.6 English Terms

  • Official / search terms: TRY, CATCH, ENDTRY, CLEANUP, class-based exception, exception handling
  • Related syntax: CATCH cx_root INTO DATA(lx_root), RAISING zcx_..., RAISE EXCEPTION TYPE zcx_..., get_text( ), previous
  • Design terms: exception propagation, catch order, fallback handler, error mapping, exception swallowing, rethrow, cleanup block
  • SAP project terms: runtime error, short dump, ABAP Unit assertion, RAP reported, RAP failed, OData error response
  • Recommended search pattern: ABAP TRY CATCH CLEANUP SAP Help, ABAP CATCH exception class order SAP Learning
  • 读官方资料时注意：ABAP 的 CLEANUP 不等同于 Java finally；ABAP 的 CATCH 基于 exception class 继承树匹配。

34.7 Example / 学习例子

34.7.1 例 1：本地处理可恢复业务错误
TRY.
    order_checker->check_quantity( iv_quantity ).
  CATCH zcx_quantity_invalid INTO DATA(lx_quantity).
    message_collector->add_error( lx_quantity->get_text( ) ).
    RETURN.
ENDTRY.

这适合“当前层知道如何恢复或反馈”的场景。quantity 不合法，当前层可以给用户或调用方返回清楚消息，并停止后续处理。

34.7.2 例 2：让上层处理 posting 失败
METHOD create_order
  RAISING zcx_document_invalid
          zcx_posting_failed.

  order_checker->check( is_order ).
  order_poster->post( is_order ).

ENDMETHOD.

这里没有本地 CATCH，因为当前 method 只是业务动作编排，不决定最终 UI message、batch log 或 OData response。它通过 RAISING 把错误 contract 明确交给上层。

34.7.3 例 3：具体到通用的 catch 顺序
TRY.
    calculator->divide( ).
  CATCH cx_sy_zerodivide INTO DATA(lx_zero).
    out->write( lx_zero->get_text( ) ).
  CATCH cx_sy_arithmetic_error INTO DATA(lx_math).
    out->write( lx_math->get_text( ) ).
ENDTRY.

顺序不能反过来。通用 superclass 如果写在前面，会挡住后面的具体 subclass。

34.7.4 例 4：测试基础设施中的 CATCH cx_root
METHOD class_setup.
  TRY.
      test_environment = lcl_test_environment=>create( ).
    CATCH cx_root INTO DATA(lx_root).
      cl_abap_unit_assert=>fail( lx_root->get_text( ) ).
  ENDTRY.
ENDMETHOD.

这是合理用法：任何 setup 失败都应该让测试失败，并显示真实异常文本。它和生产代码里的空 CATCH cx_root 完全不同。

34.7.5 例 5：吞异常反例
TRY.
    sender->send( message ).
  CATCH cx_root.
ENDTRY.

这段代码没有 message、没有 log、没有重抛、没有返回状态。上线后如果接口失败，项目只能看到“好像没有报错，但结果不对”。这是异常处理里最危险的写法之一。

34.8 Counterexample / Failure mode

34.8.1 误区 1：TRY 范围过大

一个 TRY 包住几十行代码，最后 CATCH cx_root，读者无法判断哪一步失败。

修正动作：缩小 try 范围，按业务边界拆 method，让异常类型和处理动作对应起来。

34.8.2 误区 2：CATCH 顺序从通用到具体

先写 CATCH cx_root，再写具体 exception class，后面的 catch 永远不会执行。

修正动作：从最具体写到最通用；兜底 cx_root 放最后，而且必须处理证据。

34.8.3 误区 3：空 catch

CATCH zcx_error. 或 CATCH cx_root. 后面什么都不做，是 例外握りつぶし。

修正动作：至少做到四选一：转换为业务 message、写日志、重抛/包装为上层异常、让测试失败。

34.8.4 误区 4：每层都 catch 又重新包一层

每个 method 都 catch，然后抛一个新的泛泛异常，会丢失根因。

修正动作：不知道如何处理就传播；确实需要包装时，用 previous 保留底层异常。

34.8.5 误区 5：catch 以后立刻原样再抛

如果 catch block 不记录、不转换、不补上下文、不恢复，只是立刻把同一个异常再抛出，读者会多看一层无意义控制流。

修正动作：直接在 signature 中用 RAISING 传播；只有需要添加价值时才本地 catch。

34.8.6 误区 6：把异常处理当权限或事务控制

TRY...CATCH 不能替代 AUTHORITY-CHECK、RAP authorization、lock、LUW 或 commit/rollback 设计。

修正动作：异常处理只负责错误路径控制；权限、锁和事务边界要按各自机制设计。

34.8.7 误区 7：把 CLEANUP 当 finally

从 Java/C# 经验出发，把所有收尾逻辑都写进 CLEANUP，会误解 ABAP 语义。

修正动作：记住 CLEANUP 主要服务于异常向外传播时的清理；如果异常被同一 TRY 的 CATCH 本地处理，它不是 Java finally 那种通用收尾块。普通成功路径收尾写在正常流程里。

34.9 思考问题

  • 这段代码为什么需要本地 TRY...CATCH，而不是让异常继续传播？
  • 当前 TRY 范围是否太大？能否缩小到真正可能失败的调用？
  • CATCH 顺序是否从具体到通用？
  • CATCH 里是否拿到了 exception object？是否使用了 get_text( )、公开属性或 previous？
  • 如果写了 CATCH cx_root，它是合理兜底、测试失败转换，还是吞异常？
  • 当前层应该把异常转换成 message、log、RAP reported / failed、OData error response，还是继续 RAISING？
  • 是否错误地把 TRY...CATCH 当成权限检查、事务控制或业务流程分支？
  • 是否真的需要 CLEANUP？还是普通正常流程收尾即可？
