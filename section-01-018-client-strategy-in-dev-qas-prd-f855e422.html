Chapter 01 - SAP 和 ABAP 基础

18. Client strategy in DEV / QAS / PRD 客户端策略

  • Chapter 01
  • Section 18
  • self-contained study note

1) 一句话结论

Client strategy in DEV / QAS / PRD，是把 SAP system landscape 系统环境分层 中的系统层级，进一步落到 SAP client、client role、client-dependent data、cross-client objects 和 Transport request 传输请求 的目标边界上；本节要学的是：client 不是 DEV/QAS/PRD 的同义词，登录哪个 SID/client 会直接影响数据、配置、权限、测试证据和上线风险。

2) 到底学什么

先把两个词分开：system 和 client。DEV / QAS / PRD 通常指 SAP landscape 中不同的 SAP system：开发系统、质量/验证系统、生产系统。client 则是在一个 SAP system 内部的逻辑业务和数据分区，通常用三位数字表示，例如 100、200、300。登录 SAP 时看到的 SID + client + user + language，共同决定“你到底进了哪一个环境”。

SAP Learning 对 client 的标准描述是：client 是一个在商业、组织和技术层面都相对独立的单位，拥有自己的 user master data 和一组表键值范围。对初学者来说，可以先理解成：同一个 SAP system 里可以有多个 client，它们共享同一套技术平台，但很多业务数据、用户数据和 customizing 数据按 client 分开。比如 DEV 系统里通常会有 CUST 这样的 development/customizing client 作为基线，也可能按项目需要另设 unit test client；QAS 系统里可能有 test client；PRD 系统里有 production client。具体编号不是全球统一标准，而是项目定义，独立开发测试 client 也不是每个项目都必须配置。

这里要加一个项目治理 caveat：SAP 推荐的基线通常是把 customizing 和 development 放在单一 CUST client 中受控完成，而不是让开发者随意要求“再开一个开发 client”。如果项目使用 ChaRM 或严格 transport governance，通常只有指定 client 可以创建和释放 transport request。若确实存在 TEST client，它更常用于把 request 内容复制过去做单元/局部验证，而不是替代正式 QAS。

最容易误解的是：client 不是“小号系统”。在同一个 ABAP system 内，很多 client-dependent table 的数据会按 client 分开；但 Repository object 仓库对象、ABAP Dictionary 对象、类、程序、CDS DDL source、部分 cross-client customizing/control data 属于 cross-client 范围，会影响整个 system。也就是说，在 DEV 系统 client 100 改了一个 ABAP class，它不是只属于 client 100；同一系统内其他 client 看到的是同一套 active repository object。client 可以隔离很多数据，却不能替代 DEV/QAS/PRD 三系统分层。

这也是 transport 的意义。开发和 customizing 通常在指定的 DEV client 中完成，进入 Transport task request import queue governance 后，再按 transport route 送到 QAS/PRD。repository objects 和 cross-client customizing 通常进入 workbench request；client-specific customizing 通常进入 customizing request，并按目标 client 导入。这里要精确一点：customizing 不是天然全部 cross-client，绝大多数业务 customizing 是 client-specific；少数系统级或控制类 customizing 才是 cross-client。这里也不要把规则背死：具体 transport route、target client、client change option、extended transport control 由 Basis / 架构 / 项目治理配置；ABAP 开发者的责任，是知道自己的对象和配置属于哪类边界，并在测试证据里写清 SID 和 client。

生产 client 的原则是保护。PRD client 是真实业务运行位置，通常不允许直接修改 repository object 或 customizing；如果系统设置为不允许变更，用户可能会遇到类似 Changes to Repository and cross-client Customizing are not permitted 的英文提示。変更不可 更接近系统强制状态，表示系统设置或权限层面不允许变更；変更禁止 更接近项目治理规则，表示生产 client 原则上禁止直接修改。个别紧急修复、运营设置或 Basis 操作可能有受控例外，但不能把“我有权限进去”理解成“可以直接改”。

对 ABAP 学习者而言，本节不要求掌握 client copy、SCC4、SCCL/SCCLN、system copy、client role customizing 的管理员细节。你需要掌握的是读环境、报环境、解释环境：在截图、缺陷单、测试证据、移送沟通中写清楚 SID/client/user/language/date-time；复现问题时先确认是不是同一个 client；看到 QAS 和 PRD 现象不一致时，不要马上怀疑代码，先检查 customizing、master data、authorization 和导入状态是否一致。

如果项目文档里出现 SCC1 或 SCC1N，初学者只要先知道它们常用于按 transport request 把内容复制到同一系统内的 TEST client 做验证。它不是“上线移送”，也不是开发者随意复制生产数据的工具；是否允许执行、由谁执行、复制到哪个 client，仍由 Basis / 项目治理决定。

云环境还要加一层 caveat。SAP S/4HANA Cloud Public Edition 的 2-system / 3-system landscape、CBC/central business configuration、SAP Cloud ALM 等概念，和经典 on-premise 的 SCC4/STMS 细节不完全一样；很多客户不会直接操作传统 client 设置。尽管如此，开发、测试、生产分离，变更经过受控路径进入生产，测试证据要标明具体环境，这些原则仍然成立。

3) 为什么重要

client strategy 决定“你看到的现象是不是同一个现象”。很多初学者会说：“我在系统里能看到这条数据，为什么测试看不到？”真实原因可能不是代码，而是登录了不同 client；也可能是 QAS 的 customizing 没导入、production client 被锁、测试用户权限不同、master data 没同步。没有 client 概念，排查会从一开始就跑偏。

它也决定测试证据是否可信。日本项目现场的缺陷单、単体テスト evidence、結合テスト evidence、移送連絡，通常都要能回答：在哪个 開発機/検証機/本番機，哪个 client，哪个用户，哪个语言，什么时间，执行了什么对象。缺少 client 信息的截图，在复杂 landscape 里几乎不能直接作为证据使用。

对 Java/Python 背景学习者，可以把 DEV/QAS/PRD 类比为不同 deployment environment，把 client 类比为同一环境里的逻辑 tenant 或数据域；但这个类比只能帮助入门，不能替代 SAP 规则。SAP 的 repository object 是跨 client 的，customizing 既可能 client-specific 也可能 cross-client，authorization 和 master data 又跟 client 强相关。你写的 ABAP 代码虽然在 repository 层激活，但它运行时读到的数据，经常取决于当前 client。

错误的 client strategy 会直接影响上线。比如在 DEV 的 unit test client 测过，但没有在 QAS 的业务测试 client 复现；在 sandbox client 改了设置，却以为已经进入正式 transport；在 PRD client 直接修 customizing，没有 request、没有 approval、没有回滚证据。项目事故经常不是“不知道 ABAP 语法”，而是“不知道自己改的是哪个边界”。

4) Key mechanism / 核心机制

第一层机制是 data category。SAP 系统里的数据可以粗略分成 client-specific data 和 cross-client data。client-specific data 包括 application data、user master data、很多 customizing data；cross-client data 包括 repository objects，以及部分 cross-client customizing/control data。这里的关键是“customizing 有两类”：多数业务 customizing 是 client-specific，少数影响整个系统的设置属于 cross-client。判断一个问题属于哪一类，是定位缺陷的第一步。

第二层机制是 client-dependent table。很多业务表带有 client 字段，ABAP Open SQL 在通常情况下会按当前 client 处理数据访问。于是同一张物理表里可能有多个 client 的数据，但应用程序默认只看到当前登录 client 对应的数据。初学者不需要立刻研究所有 Open SQL 例外，但必须知道：sy-mandt / current client 会影响数据读取范围。

第三层机制是 client role。SAP Learning 中常见的逻辑 client roles 包括 CUST development/customizing、QTST quality assurance、PROD production，也可能有 pre-production、training、sandbox、unit test、golden client 等。SCC4 中也有 technical client role，例如 Customizing、Test、Production、Demo、Training/Education、SAP Reference。项目文档中的逻辑角色和系统设置中的技术角色不一定一一同名，所以要问“这个 client 的实际用途是什么”。

第四层机制是 transport boundary。repository objects / cross-client customizing 和 client-specific customizing 的 transport 类型不同，导入目标也不同。workbench request 常用于 repository objects 和 cross-client customizing；customizing request 常用于 client-specific customizing。开发者不需要配置 TMS，但必须在移送说明中写清对象、request、target system/client、测试 client 和依赖关系。

如果 DEV 系统里有单独 TEST client，常见流程可能是：在 CUST client 中创建并记录 request，在释放前用 SCC1/SCC1N 把 request 内容复制到 TEST client 验证。这个动作的意义是同系统内的局部验证，不等同于导入 QAS，也不能代替正式 transport route 和 QAS evidence。

第五层机制是 change protection。生产 client 通常设置为不允许直接变更，QAS 也通常不允许随意改 repository 或 customizing。変更不可 是系统层面的“不能改”，変更禁止 是项目治理层面的“禁止改”。两者都要尊重：前者由系统设置和权限强制，后者由项目规则、审计和上线流程约束。

第六层机制是 evidence discipline。每一份测试截图、障害調査记录、移送依赖说明，都应该包含系统和 client。最简格式可以是：System: DEV, Client: 100, User: ZDEV001, Language: JA, Time: 2026-06-13 20:30 JST。这不是形式主义，而是让别人能复现、审查和判断风险。

5) 日本项目现场说法

  • 常见日语：クライアント、クライアント番号、クライアント役割、開発クライアント、カスタマイジングクライアント、検証クライアント、本番クライアント、研修クライアント、サンドボックス。
  • 数据边界：クライアント依存、クライアント非依存、クロスクライアント、クライアント依存テーブル、マスタデータ、カスタマイジング。现场上常用 クライアント非依存 说明对象或设置“不依赖某个 client”，用 クロスクライアント 说明影响跨 client 的数据或设置范围。
  • 管理操作：クライアント設定、クライアントコピー、変更不可、変更禁止、本番クライアントロック、システムID、移送先クライアント。
  • 说明句：この不具合は、検証機 100 クライアントでは再現しますが、開発機 200 クライアントではマスタデータが異なるため再現していません。
  • 说明句：本番クライアントは変更禁止のため、直接修正ではなく移送依頼経由で反映します。
  • 提问句：このクライアントの役割は何ですか。カスタマイジング用ですか、検証用ですか、それとも研修用ですか。
  • 使用场景：环境说明、测试证据、缺陷复现、移送计划、权限问题、生产变更审批、上线后障害調査。

7) Example / 学习例子

  • 例子 1：同一个 ABAP class 在 DEV client 100 被修改并激活后，DEV client 200 看到的也是同一个 active repository object；但两边执行程序读到的业务数据可能不同，因为数据按 client 分开。
  • 例子 2：测试人员说 QAS 没有复现缺陷。不要只问“你点了哪个按钮”，还要问 SID/client/user/language、测试数据、customizing 是否导入、transport request 是否真的进入 QAS。
  • 例子 3：生产问题调查时，截图只写了“SAP 系统里报错”，这是无效证据。合格证据至少写清 PRD / client 100 / user / time / transaction or app / input data / error message。
  • 例子 4：DEV 有 CUST client 和 TEST client。开发者在 CUST client 创建 request 后，项目流程允许由指定人员用 SCC1/SCC1N 把 request 内容复制到 TEST client 做局部验证；验证通过仍然需要通过正式 transport route 进入 QAS，不能把 TEST client 结果当成 QAS evidence。
  • 最小动作：画一个小表，列出 DEV 100、DEV 200、QAS 100、PRD 100 各自可能承担的角色，并说明哪些内容跨 client，哪些内容按 client 分开。
  • 进阶动作：找一个现有 transport request，判断它是 workbench request 还是 customizing request，并说明它的目标 system/client 为什么重要。

8) Counterexample / Failure mode

  • 误区 1：把 client 当成 DEV/QAS/PRD。修正：DEV/QAS/PRD 是 system landscape 层级，client 是 system 内部的逻辑分区。
  • 误区 2：以为 client 隔离所有东西。修正：repository objects 和部分 cross-client customizing/control data 会影响整个 system。
  • 误区 3：看到 QAS 没有数据就说代码错了。修正：先确认当前 client、master data、customizing、authorization、transport import 状态。
  • 误区 4：在 sandbox client 验证通过，就认为可以上线。修正：sandbox 适合探索，不等于受控 QAS 测试证据。
  • 误区 5：认为 production client 只要有权限就能直接改。修正：生产变更应走受控 request、approval、log 和回滚路径；直接变更通常违反治理和审计要求。
  • 误区 6：把 client copy 当成开发者日常操作。修正：client copy / client refresh 是 Basis 或管理员控制的高影响操作，开发者只需要理解它会改变测试数据和环境一致性。
  • 误区 7：把 sandbox 当成 transport 起点。修正：sandbox 通常用于探索和验证想法，原则上不应把 sandbox 里的临时改动直接送往 QAS/PRD；正式变更要在受控 development/customizing client 中重新实现、记录 request、review 和测试。

9) 思考问题

  • DEV 100 和 DEV 200 在同一个 SAP system 中，为什么它们可能看到同一个 ABAP class，却读到不同业务数据？
  • 缺陷单里只写“QAS 报错”，缺少 client 信息，会给复现和判断带来哪些风险？
  • workbench request 和 customizing request 在 client strategy 上有什么差异？
  • 如果 DEV 有 CUST client 和 TEST client，SCC1/SCC1N 复制 request 内容的目的是什么？它为什么不能替代 QAS？
  • 日本项目里如何用一句话说明“本番 client 不能直接改，要走移送”？
  • SAP S/4HANA Cloud Public Edition 中即使不强调传统 SCC4/STMS，为什么仍然需要开发/测试/生产分离？
