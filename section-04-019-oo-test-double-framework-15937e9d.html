19. OO Test Double Framework

  • Chapter 04
  • Section 19
  • self-contained study note

19.1 一句话结论

OO Test Double Framework 是 ABAP Unit 中用来替换面向对象依赖的测试替身能力；在本教程里要学的是：当 service logic 依赖数据库、外部系统、权限检查、时间或其他类时，如何用可控替身让测试稳定、快速、可验证。

上一节 ABAP Unit for service logic 讲的是要把业务规则写成可回归测试。但现实服务逻辑很少完全纯粹：它可能要读数据库、调用远程接口、取得当前用户、检查权限、写应用日志、读取系统时间。如果每个单元测试都依赖这些真实对象，测试就会慢、不稳定，也难以覆盖异常路径。Test Double 的作用，就是在测试中提供“可控制的替代对象”。

可以把 OO Test Double 理解为：生产代码面对的是接口或可替换依赖；测试代码提供一个假的实现，让它返回指定结果、抛出指定异常，或者验证某个方法是否被调用。这样就能测试“服务逻辑如何反应”，而不是每次都真的访问外部系统。

19.2 到底学什么

第一，要学 test double 的边界。它不是为了伪造项目成果，也不是为了让测试脱离真实系统永远自嗨。它的目标是隔离不稳定、昂贵、难准备的依赖，让核心业务规则可以被快速验证。真实数据库、真实 OData、真实授权、真实外部接口仍然需要集成测试和 API evidence replay 覆盖。

第二，要学依赖反转的思路。被测类如果直接 CREATE OBJECT 某个具体依赖、直接读系统时间、直接调用静态工具、直接访问外部接口，就很难替换。更可测试的设计是依赖接口，例如 zif_exchange_rate_provider、zif_auth_checker、zif_clock、zif_upload_repository。生产环境注入真实实现，测试环境注入 test double。

第三，要学测试替身可以验证什么。它可以控制返回值，例如“权限检查返回失败”；可以控制异常，例如“外部系统超时”；可以验证交互，例如“保存方法只被调用一次”；也可以避免副作用，例如不真的发邮件、不真的提交数据、不真的锁对象。

第四，要学不要滥用。不是每个小方法都需要 test double，也不是所有依赖都必须抽接口。只有当依赖阻碍稳定测试、导致副作用、难以准备数据、难以覆盖错误路径时，替身才有价值。过度抽象会让代码变复杂。

第五，要学“测试边界”怎么选。一个好的单元测试不是把整个系统都 fake 掉，而是围绕一个 service class 或一段业务规则建立边界：边界内是真实逻辑，边界外是不稳定依赖。比如上传服务的状态判断、错误映射、幂等分支可以是真实代码；权限系统、远程 API、当前时间、持久化 repository 可以用替身控制。

第六，要学 test double 和交付证据的关系。单元测试能证明服务逻辑在指定依赖返回值下行为正确，但不能证明真实 PFCG role、真实 DCL、真实数据库性能、真实 OData contract 或真实外部系统可用。因此测试替身要和 AC0567C API evidence replay、SU53 and authorization trace workflow、SQL Trace performance analysis 等证据互补。

19.3 为什么重要

Test Double 重要，是因为真正有价值的单元测试往往不是 happy path，而是异常系和边界条件。外部系统成功时很好测；外部系统超时、返回格式错误、权限失败、重复请求、锁冲突、数据库无数据，这些才是生产问题常出现的地方。如果没有 test double，这些场景很难稳定制造。

对 SAP 项目而言，很多依赖本身很重：授权配置可能随环境变化，业务数据可能被其他测试改掉，外部系统可能不可用，系统时间不可控，数据库提交可能污染环境。Test Double 让开发者把注意力放回服务逻辑：当依赖返回某种结果时，业务代码是否做了正确分支。

对 Java/Python 背景学习者来说，概念类似 mock/stub/fake，但 ABAP 项目要结合 ABAP Unit、interface、global class、repository object、transport、ATC 和系统 release。不是照搬 Java mock 框架思维，而是在 SAP 约束下设计可测试边界。

项目里的真正收益通常体现在“异常系”。没有 test double，开发者很难稳定制造“权限失败”“外部系统 timeout”“应用日志写入失败”“当前日期跨期间”“repository 返回重复 request id”这些场景。于是测试只覆盖 happy path，上线后异常路径才第一次被真实用户触发。Test double 的意义，就是把这些异常路径提前变成可重复、可审查、可回归的用例。

19.4 Key mechanism / 核心机制

OO Test Double 的机制可以用三步理解。

第一步，定义可替换依赖。生产服务类不要直接绑死外部依赖，而是依赖接口。例如上传服务需要检查权限，就依赖 zif_auth_checker；需要读上传历史，就依赖 zif_upload_repository；需要当前时间，就依赖 zif_clock。

第二步，在生产代码中注入真实实现，在测试中注入替身。注入方式可以是 constructor 参数、setter、factory 或测试专用创建方法。选择哪一种要看项目风格，不要为了测试破坏封装，但也不要把依赖全部藏在类内部导致无法测试。

第三步，在测试中配置替身行为并断言结果。比如让权限检查返回 false，然后断言 service 返回 AUTHORIZATION_FAILED；让 repository 表示 request id 已存在，然后断言 service 不再创建新 batch；让外部系统 adapter 抛出异常，然后断言错误被安全映射为技术错误并记录日志线索。

需要区分几类替身：

  • Stub：提供固定返回值，帮助测试走到某个分支。
  • Mock：除了返回值，还验证某个方法是否被调用、调用几次、参数是什么。
  • Fake：一个轻量可运行实现，例如内存 repository。
  • Spy：记录调用情况，供测试断言。

ABAP 的具体框架和语法会随 release、对象类型和 ABAP Cloud 限制变化，但核心思想不变：让依赖可控，让断言有意义。

一个可操作的设计流程如下：

1. Identify behavior under test
   Example: upload service rejects unauthorized company code.

2. List external dependencies
   Auth checker, repository, clock, external API, application log.

3. Decide which dependencies to replace
   Replace only dependencies that are slow, unstable, hard to prepare, or side-effecting.

4. Introduce an interface or injection point
   Keep production code readable and avoid test-only architecture noise.

5. Configure the double
   Return value, exception, stored in-memory data, or expected call.

6. Assert business behavior
   Result code, state transition, no save, log call, retryability, message.

7. Cover the real integration elsewhere
   API test, authorization evidence, SQL trace, application log, integration test.

不同替身应服务于不同问题：

需求更合适的替身例子
固定返回一个值Stubauthorization checker returns denied
记录是否调用Spy / Mockrepository save must not be called
轻量模拟状态Fakein-memory upload repository
制造异常Stub / Mockexternal API raises timeout
验证复杂交互顺序Mock，谨慎使用retry adapter calls once then stops

断言优先级也很重要。优先断言业务结果，例如错误码、状态、是否保存、是否可重试；只有当交互本身就是业务契约时，才断言内部调用次数。否则测试会变成“锁死实现细节”，让合理重构也大量失败。

19.5 日本项目现场说法

常见日语包括：テストダブル、モック、スタブ、フェイク、依存関係、インターフェース化、単体テスト、外部依存、副作用、異常系テスト。

设计或 review 中可以这样说：

  • 外部API呼び出し部分はインターフェース化し、ABAP Unit ではテストダブルで応答を制御します。
  • 権限チェック、時刻取得、DBアクセスなどの外部依存を分離し、サービスロジックを安定してテストできるようにします。
  • テストダブルは単体テスト用であり、実環境での結合テストやAPI証跡確認を置き換えるものではありません。

代码 review 时可以这样问：

  • この依存関係はテストで差し替え可能ですか。
  • 異常系をテストするために、外部依存の戻り値や例外を制御できますか。
  • モックのためだけに設計が過剰になっていませんか。
  • テスト対象のビジネス判断と、外部依存の差し替え範囲を分けて確認します。
  • モックの呼び出し回数だけでなく、業務結果とエラーメッセージを検証してください。
  • テストダブルで確認した単体テスト結果とは別に、実API・実権限・実DBでの結合確認も必要です。

19.7 Example / 学习例子

假设 AC0567C 上传服务依赖一个权限检查器：

Upload service
  -> auth checker
  -> upload repository
  -> application log writer

如果测试直接使用真实权限检查，就需要准备真实用户、角色、组织值，测试会变慢且不稳定。更好的做法是让 service 依赖 zif_auth_checker，测试中替换为 test double。

测试场景可以这样表达：

Given: auth checker returns "not authorized" for company code 1000
When: upload service validates the request
Then: result errorCode is AUTHORIZATION_FAILED
And: repository save is not called
And: response is not retryable until authorization is fixed

另一个场景：

Given: upload repository says request id AC0567C-001 already exists
When: execute is called again with the same request id
Then: create batch is not called
And: existing status is returned

这些测试的重点不是证明权限系统本身正确，而是证明服务逻辑在“权限失败”“重复请求”等条件下做了正确反应。真实权限配置仍要由集成测试、SU53/trace 和项目权限测试覆盖。

再看一个 AC0567C 的完整 Given / When / Then：

Given:
  auth checker double returns allowed for BUKRS=1000
  repository fake contains no existing request id
  clock stub returns 2026-06-16T10:00:00+09:00
  external adapter double raises timeout

When:
  upload service executes request AC0567C-001 for company code 1000

Then:
  service returns TECHNICAL_ERROR
  repository records request as failed or retryable according to design
  no duplicate success document is created
  application log writer is called with request id and error category
  response does not expose raw external-system exception text

这个测试不需要真实外部系统，也不需要真实权限配置。它要证明的是：外部系统 timeout 时，服务逻辑是否安全地记录、分类、返回、允许或禁止重试。这里的 ISO-8601 时间只是伪代码；真实 ABAP 设计中，zif_clock 可以按项目约定返回 utclong、timestamp 或封装后的业务时间对象。真实外部系统连通性应放到 API/integration test 中验证。

另一个例子是时间依赖。月末处理、会计期间、到期日计算、job cutoff time 都不适合直接读系统当前时间。若代码依赖 zif_clock，测试可以固定时间，覆盖“月末最后一天”“跨日后台 job”“时区差异”这些难以手工稳定制造的场景。

19.8 Counterexample / Failure mode

常见失败模式有以下十类。

第一，类直接创建所有依赖。被测类内部到处 CREATE OBJECT 或调用静态方法，测试无法替换依赖，只能走真实系统。

第二，mock 过多导致测试只验证实现细节。测试过度关心内部调用顺序，稍微重构就失败，却没有真正保护业务行为。

第三，用 test double 替代所有真实测试。单元测试通过，不代表 OData 服务、权限、事务、数据库、锁和外部系统都正确。Test double 是单元层工具，不是交付证据的全部。

第四，替身行为和真实系统差距太大。测试 double 返回了现实中不会出现的组合，或者漏掉真实系统的重要限制，导致测试给出虚假安全感。

第五，为了测试过度设计。每个小函数都抽接口、每个值都注入，会让代码难读。抽象应服务于真实复杂度和测试价值。

第六，把 mock 调用次数当成业务正确性。测试只证明某方法调用了 1 次，却没有断言最终状态、错误码和日志，价值很低。

第七，替身隐藏了事务问题。单元测试里的 fake repository 很容易成功，但真实数据库提交、锁、rollback、update task 或 RAP save sequence 仍可能失败。

第八，测试数据过度理想化。double 返回的对象没有空值、边界状态、重复 key、权限差异，导致测试覆盖不到真实问题。

第九，为了方便测试而暴露过多 public 方法。可测试性不等于破坏封装；优先通过接口和构造注入表达真正依赖。

第十，单元测试和集成测试断裂。test double 场景没有对应的真实 API/权限/DB 验证，项目无法证明替身假设与现实一致。

19.9 思考问题

  • 哪些依赖最适合用 test double 替换？数据库、权限、时间、外部 API、应用日志分别如何判断？
  • mock 和 stub 的差异是什么？在 ABAP 项目里什么时候只需要 stub？
  • 为什么 test double 不能替代端到端 API 测试？
  • AC0567C POC 中哪个依赖最阻碍 ABAP Unit？怎样抽接口最自然？
  • 在日本项目里，如何说明“这里需要テストダブル”而不是“为了测试把设计复杂化”？
  • 这个 test double 是为了控制返回值、制造异常、记录调用，还是模拟轻量状态？
  • 当前断言是在保护业务行为，还是只是在锁死内部实现？
  • 哪些 test double 假设需要用集成测试或 API evidence 再验证一次？
