Chapter 02 - Core ABAP Programming

42. REDUCE expression 归约表达式

  • Chapter 02
  • Section 42
  • self-contained study note

42.1 一句话结论

REDUCE expression 是 ABAP 的 reduction operator，用来把一次或多次 FOR iteration expression 的结果“折叠”为一个最终值。它最常见的用途是对已经在 ABAP 内部表中的数据做小型、纯粹、可读的汇总，例如求和、计数、拼接文本、构造一个 summary structure，或者在 ABAP Unit 中从测试输入算出期望值。

DATA(total_amount) = REDUCE decfloat34(
  INIT total = CONV decfloat34( 0 )
  FOR item IN lt_items
  NEXT total = total + item-amount
).

可以把它读成：先把 accumulator total 初始化为 0，然后逐行遍历 lt_items，每一轮都用当前行的 amount 更新 total，最后整个 REDUCE 表达式返回最终的 total。它不是“更高级的 LOOP”，也不是数据库聚合的替代品。它适合把一个简单的内部表归约逻辑写成一个值表达式；如果逻辑需要消息、异常、数据库访问、复杂分支或大量调试，普通 LOOP 往往更清楚。

42.2 到底学什么

本节要掌握六件事。

  • REDUCE type( INIT ... FOR ... NEXT ... ) 的基本形状。
  • INIT 定义 accumulator，且第一个 INIT 变量决定整个 REDUCE 的最终结果。
  • FOR 在 REDUCE 中是核心和必需部分，可用于 table iteration，也可用于条件式迭代。
  • NEXT 描述每一轮如何更新 accumulator，并且多个 NEXT 赋值按顺序影响后续赋值。
  • REDUCE 和 LOOP、VALUE #( FOR ... )、ABAP SQL SUM / COUNT / GROUP BY 的边界。
  • 日本项目 review 中如何说明 REDUCE式、畳み込み、集約、累積変数、INIT指定、NEXT指定 和 LOOP文に戻す。

初学者不要先追求短写法。先问三个问题：我要归约成什么结果？初始值是什么？每一轮的更新规则是否一眼能看懂？只要这三个问题不清楚，REDUCE 就会比 LOOP 更难读。

42.3 为什么重要

ABAP 项目里常见这种内部表汇总：

DATA total_amount TYPE decfloat34.

LOOP AT lt_items INTO DATA(item).
  total_amount = total_amount + item-amount.
ENDLOOP.

如果逻辑只是“把内部表每一行的金额加起来”，REDUCE 可以把结果类型、初始值、迭代对象、更新公式集中在一个表达式中。

DATA(total_amount) = REDUCE decfloat34(
  INIT total = CONV decfloat34( 0 )
  FOR item IN lt_items
  NEXT total = total + item-amount
).

这对小型 helper method、ABAP Unit、DTO summary、测试数据构造很有价值。读者能看到：结果类型是 decfloat34，初始值是 0，遍历的是 lt_items，每轮只做一件事。

但 SAP 项目中也有一个重要边界：如果数据还在数据库里，不要为了使用 REDUCE 先把大量数据 SELECT 到 ABAP 再汇总。能在数据库端安全完成的聚合，应优先考虑 ABAP SQL 的 SUM、COUNT、GROUP BY。REDUCE 的主要位置是“数据已经在 ABAP internal table 中，并且需要本地小型归约”。

42.4 Key mechanism / 核心机制

Basic form sums a column

最小示例是内部表字段求和。

DATA(total_quantity) = REDUCE i(
  INIT total = 0
  FOR item IN lt_items
  NEXT total = total + item-quantity
).

逐段读：

  • REDUCE i( ... )：整个表达式的结果类型是 i。
  • INIT total = 0：定义 accumulator total，初始值是 0。
  • FOR item IN lt_items：逐行遍历内部表。
  • NEXT total = total + item-quantity：每一轮用旧的 total 加当前行数量，得到新的 total。

循环结束后，第一个 INIT 变量 total 的最终值就是 REDUCE 表达式的返回值。

First INIT variable is the result

INIT 可以定义多个临时变量，但第一个 INIT 变量决定最终返回值。后面的变量是辅助 accumulator。

DATA(text) = REDUCE string(
  INIT result = ``
       count  = 0
  FOR item IN lt_items
  NEXT count  = count + 1
       result = |{ result }{ count }: { item-name }; |
).

这里最终返回的是 result，不是 count。count 只是帮助拼接序号。review 时看到多个 INIT 变量，要先问：第一个变量真的是想返回的东西吗？如果不是，变量顺序就错了。

这一点对初学者很容易踩坑。因为最后一行 NEXT result = ... 看起来像“返回 result”，但真正的规则来自 INIT 的第一个变量，而不是 NEXT 中最后出现的赋值。

FOR is mandatory

REDUCE 必须包含至少一个 FOR iteration expression。没有迭代，就没有归约。

Table iteration：

DATA(total_amount) = REDUCE decfloat34(
  INIT total = CONV decfloat34( 0 )
  FOR item IN lt_items
  NEXT total = total + item-amount
).

Conditional iteration：

DATA(sum_1_to_10) = REDUCE i(
  INIT sum = 0
  FOR n = 1 UNTIL n > 10
  NEXT sum = sum + n
).

如果迭代变量不是默认加 1，也可以用 THEN 指定下一轮如何变化。

DATA(countdown) = REDUCE string(
  INIT text = ``
  FOR n = 10 THEN n - 1 WHILE n > 0
  NEXT text = |{ text } { n }|
).

一段 REDUCE 里也可以出现多个 FOR，形成类似嵌套循环的效果。但这通常会迅速降低可读性。书稿和项目代码里，多个 FOR 的 REDUCE 要非常克制，必要时拆成 LOOP 或 helper method。

NEXT updates the accumulator

NEXT 是每一轮迭代结束时如何更新 accumulator 的地方。

NEXT total = total + item-amount

这句可以读成：新的 total 等于旧的 total 加当前行金额。初学阶段建议先使用这种完整赋值写法，不急着用 += 或 &&=。

如果 NEXT 中有多个赋值，按照从左到右的顺序求值和更新。后一项可以读到前一项更新后的值。因此多个 accumulator 之间存在依赖时，顺序要显式 review。

DATA(text) = REDUCE string(
  INIT result = ``
       count  = 0
  FOR item IN lt_items
  NEXT count  = count + 1
       result = |{ result }{ count }: { item-name }; |
).

这里 result 使用的是已经更新后的 count。如果调换赋值顺序，结果就可能不同。

Joule review 还特别提醒：虽然 INIT 技术上可以定义 field symbol，但初学和普通项目代码中，不要把 field symbol 当成主 accumulator，尤其不要和 +=、&&= 这类 compound assignment 混用。field symbol 的赋值语义更接近重新绑定目标，容易导致读者误解。普通求和、拼接、计数时，用普通变量 accumulator。

Table iteration additions

FOR item IN itab 是 iteration expression，因此可以带一些表迭代相关 addition。

DATA(open_amount) = REDUCE decfloat34(
  INIT total = CONV decfloat34( 0 )
  FOR item IN lt_items
  WHERE ( status = 'OPEN' )
  NEXT total = total + item-amount
).

WHERE ( ... ) 表示只处理满足条件的行。还可能看到 INDEX INTO、USING KEY、FROM、TO、STEP 等 table iteration 相关 addition。它们不是 REDUCE 专有，而是 FOR iteration expression 的能力。

如果过滤逻辑很复杂，或者 key 选择影响性能，就不要把所有东西塞进一行 REDUCE。先用普通 LOOP 或拆出 helper method，确认行为后再考虑是否表达式化。

Building a table with REDUCE is possible but not always best

REDUCE 的结果可以是 internal table。例如把每一行转换成一行文本。

DATA(lines) = REDUCE string_table(
  INIT result = VALUE string_table( )
  FOR item IN lt_items
  NEXT result = VALUE #(
    BASE result
    ( |{ item-id }: { item-amount }| )
  )
).

这能工作，但不一定是最清楚的写法。SAP 文档和项目经验通常会把“从一张表构造另一张表”优先写成 VALUE #( FOR ... ) 的 table comprehension。

DATA(lines) = VALUE string_table(
  FOR item IN lt_items
  ( |{ item-id }: { item-amount }| )
).

可以这样区分：如果核心意图是“多行归约成一个标量、结构或小型 summary”，REDUCE 合适；如果核心意图是“从表生成表”，VALUE #( FOR ... ) 往往更直观。

Structured accumulator needs preservation

当 accumulator 是结构时，要小心 VALUE #( ... ) 重建结构时遗漏组件。

TYPES: BEGIN OF ty_summary,
         total TYPE decfloat34,
         count TYPE i,
       END OF ty_summary.

DATA(summary) = REDUCE ty_summary(
  INIT result = VALUE ty_summary( )
  FOR item IN lt_items
  NEXT result = VALUE #(
    BASE result
    total = result-total + item-amount
    count = result-count + 1
  )
).

这里使用 BASE result 是为了保留未显式赋值的组件。如果写成 VALUE #( total = ... )，没有列出的组件会回到初始值，容易造成“上一轮累计值突然消失”的问题。结构 accumulator 是 REDUCE 中比较容易出错的地方，初学阶段要优先写得清楚，而不是写得短。

REDUCE vs LOOP vs SQL aggregate

项目里可以按这个顺序判断。

  • 数据还在数据库里，要做总数、平均、最大最小、分组汇总：优先考虑 ABAP SQL SUM、COUNT、MIN、MAX、AVG、GROUP BY。
  • 数据已经在 internal table 中，逻辑是纯粹小型归约：可以考虑 REDUCE。
  • 需要消息、异常、日志、多字段状态更新、复杂判断、外部调用或逐步调试：使用 LOOP。
  • 从一张表构造另一张表：优先考虑 VALUE #( FOR ... )，不必硬用 REDUCE。
  • 逻辑看起来像一行数学公式，但 review 时解释不清：退回 helper method 或 LOOP。

不要把 REDUCE 当成性能优化。它首先是表达力工具。性能关键场景要看数据位置、表类型、key、算法复杂度、数据库下推可能性，而不是看代码是不是一行。

Debugging and readability

SAP-samples cheat sheet 提醒过一个很实用的点：constructor expressions 和 iteration expressions 可以让代码更短，但复杂时可读性、可维护性和可调试性会下降。

当 REDUCE 结果不对时，最稳的调试办法是临时展开成 LOOP。

DATA total TYPE decfloat34.

LOOP AT lt_items INTO DATA(item).
  total = total + item-amount.
ENDLOOP.

确认每一轮变化后，再决定是否恢复为 REDUCE。在日本项目 review 里，REDUCE式をLOOP文に戻す 不是“退步”，而是为了障害調査、レビュー、保守性做出的工程判断。

42.5 日本项目现场说法

常见日语：

  • REDUCE式

  • 内部テーブル集約
  • 累積変数
  • 集約結果
  • INIT指定
  • FOR反復式
  • REDUCE内のFOR反復式
  • NEXT指定
  • 条件付き反復
  • テーブル反復
  • 型推論
  • 可読性
  • 保守性
  • デバッグ容易性
  • LOOP文に戻す

现场说明句：

  • この REDUCE式 は、内部テーブルの複数行を一つの集約結果に畳み込むための式です。
  • INIT指定 の最初の変数が REDUCE式 の最終結果を決定します。
  • NEXT句 内の複数の代入は左から右の順に評価されるため、代入順序を確認してください。
  • テーブルからテーブルを構築する目的であれば、REDUCE式 より VALUE #( FOR ... ) の方が読みやすい可能性があります。
  • データがまだDB上にある大量データ集計であれば、ABAP側の REDUCE式 ではなく SQL の SUM / GROUP BY を検討してください。
  • 複雑な処理になっているため、可読性とデバッグ容易性を考慮して LOOP文 または helper method に戻してください。

术语注意：FOR式 这个说法在日语材料里可能同时指 REDUCE 内的迭代，也可能指 VALUE #( FOR ... ) 的 table comprehension。为了避免混淆，本节建议说 REDUCE内のFOR反復式。

42.6 English Terms

  • Official / search terms: REDUCE, REDUCE operator, reduction operator, iteration expression, FOR expression
  • ABAP syntax: REDUCE type( [ LET ... IN ] INIT result = start FOR item IN itab NEXT result = ... )
  • Core additions: INIT, FOR, NEXT, WHERE ( ... ), INDEX INTO, USING KEY, UNTIL, WHILE, THEN
  • Related terms: accumulator, temporary variable, field symbol, result type, table iteration, conditional iteration, aggregation, summary value
  • Comparison terms: LOOP AT, VALUE #( FOR ... ), SUM, COUNT, GROUP BY, internal table aggregation, debuggability
  • Recommended search pattern: ABAP REDUCE operator INIT FOR NEXT SAP Help, ABAP REDUCE expression SAP Samples, ABAP iteration expressions FOR REDUCE

读官方资料时要把 FOR 和 NEXT 一起看。只看 FOR 会以为它只是循环；只看 NEXT 会看不出每一轮从哪里来。REDUCE 的核心是：用 FOR 产生迭代，用 NEXT 更新 accumulator，用第一个 INIT 变量作为最终结果。

42.7 Example / 学习例子

例 1：金额求和
DATA(total_amount) = REDUCE decfloat34(
  INIT total = CONV decfloat34( 0 )
  FOR item IN lt_items
  NEXT total = total + item-amount
).

适合场景：数据已经在内部表中，数量不大，逻辑只有求和。金额字段还要注意货币语义；不同 currency 的金额不能随便直接相加。

例 2：只汇总 OPEN 行
DATA(open_amount) = REDUCE decfloat34(
  INIT total = CONV decfloat34( 0 )
  FOR item IN lt_items
  WHERE ( status = 'OPEN' )
  NEXT total = total + item-amount
).

这里 WHERE 是 FOR table iteration 的过滤条件。review 时要确认 status = 'OPEN' 是技术常量还是业务 code list，必要时用常量替代裸字面量。

例 3：计数加拼接
DATA(text) = REDUCE string(
  INIT result = ``
       count  = 0
  FOR item IN lt_items
  NEXT count  = count + 1
       result = |{ result }{ count }: { item-id }; |
).

这个例子展示多个 accumulator。最终返回 result，count 只是辅助变量。注意 count 在 result 前更新，所以 result 读到的是新计数。

例 4：结构型 summary
DATA(summary) = REDUCE ty_summary(
  INIT result = VALUE ty_summary( )
  FOR item IN lt_items
  NEXT result = VALUE #(
    BASE result
    total = result-total + item-amount
    count = result-count + 1
  )
).

重点是 BASE result。没有它，未列出的结构组件可能被初始化，造成累积结果丢失。

例 5：ABAP Unit 中固定期望值
DATA(actual) = REDUCE i(
  INIT total = 0
  FOR item IN lt_items
  NEXT total = total + item-quantity
).

cl_abap_unit_assert=>assert_equals(
  exp = 6
  act = actual
).

测试应覆盖 normal table、empty table、filtered-out table、zero / negative value 等边界。REDUCE 自身不是测试，测试要证明初始值和每轮更新符合业务规则。

42.8 Counterexample / Failure mode

误区 1：把所有 LOOP 都改成 REDUCE

错误心智：Modern ABAP 就是把循环变成表达式。

修正：REDUCE 适合产生一个结果值，不适合承载消息、日志、数据库更新、异常路径、多对象状态修改。复杂流程应留在 LOOP。

误区 2：忘记第一个 INIT 变量才是结果

错误心智：NEXT 最后一行赋值给谁，谁就是返回值。

修正：最终结果由第一个 INIT 变量决定。多个 accumulator 时，先确认第一个变量是不是想返回的结果。

误区 3：初始值随手写 0

错误心智：求最大值、最小值、金额汇总都可以从 0 开始。

修正：初始值必须符合业务语义。负数、空表、初始金额、不同货币、不同单位都可能让 0 成为错误默认值。

误区 4：用 ABAP REDUCE 替代数据库聚合

错误心智：先 SELECT 全部数据到内部表，再用 REDUCE 汇总。

修正：如果数据还在数据库中，优先让数据库做 SUM / COUNT / GROUP BY，同时处理 currency/unit、权限、released API / Clean Core 等边界。

误区 5：结构 accumulator 忘记 BASE

错误心智：VALUE #( total = ... ) 只会改 total。

修正：VALUE #( ... ) 重建结构时，没有列出的组件会变成初始值。需要保留旧组件时用 BASE result，或者改成更清楚的 LOOP。

误区 6：表达式太复杂难以 debug

错误心智：能写成一行就更好。

修正：调试困难时，先展开为 LOOP。通过测试和 review 后，再决定是否恢复为 REDUCE。

42.9 思考问题

  • 这个 REDUCE 的最终 result type 是什么？
  • 第一个 INIT 变量是谁？它真的是想返回的结果吗？
  • 初始值是否对 empty table、负数、初始金额、不同单位都合理？
  • FOR 遍历的是 internal table 还是条件式计数？是否有 WHERE、USING KEY、INDEX INTO？
  • NEXT 中多个赋值是否依赖顺序？后一项是否读取前一项更新后的值？
  • 数据是否还在数据库里？是否应该用 SQL aggregate 而不是 ABAP 侧 REDUCE？
  • 这段逻辑是否需要日志、消息、异常或逐步 debug？如果需要，是否应该改成 LOOP？
  • 如果结果是 internal table，是否用 VALUE #( FOR ... ) 更直观？
  • 结构 accumulator 是否需要 BASE result？
  • 在日本项目 review 中，应说 REDUCE式、内部テーブル集約、累積変数，还是建议 LOOP文に戻す？
