25. Loop constructs 循环语句

  • Chapter 01
  • Section 25
  • self-contained study note

本节结论

Loop constructs 循环语句 是 ABAP 控制流的基础，但 Day 04 最先要掌握的是 LOOP AT itab：它让程序逐行处理 Internal table 内部表基础。初学者要先分清两件事：INTO 是把当前行复制到 work area，改了 copy 不会自动改回表；ASSIGNING 是直接指向当前行，改字段会立刻影响表内容。这个差异比背所有 loop 语法更重要。

正文

ABAP 里不只有一种循环。官方资料把常见循环分成 DO、WHILE、LOOP、SELECT ... ENDSELECT 等形式；后面还会遇到 LOOP AT GROUP BY、FOR 表达式、REDUCE 等现代写法。但业务 ABAP 的第一入口通常是 LOOP AT internal_table，因为项目里大量数据先被放进 internal table，再逐行校验、转换、累计、组装消息或生成结果。

先看最普通的读循环：

LOOP AT lt_items INTO DATA(ls_item).
  out->write( |{ ls_item-travel_id } { ls_item-item_no } { ls_item-amount }| ).
ENDLOOP.

INTO DATA(ls_item) 的意思是：每一轮把当前行复制到一个 work area。适合只读输出、计算临时值、或者先改 work area 再用 MODIFY 写回。它不等于 Python/Java 里“拿到集合元素引用”。如果在 loop 里写 ls_item-amount = 0.，但没有 MODIFY lt_items FROM ls_item ...，表里的原行不会自动被改掉。SAP Learning 也明确提醒：改 work area 后不写回，下一轮 work area 被新行覆盖时，修改就丢了。

再看直接访问：

LOOP AT lt_items ASSIGNING FIELD-SYMBOL(<ls_item>).
  <ls_item>-amount = <ls_item>-amount * '1.10'.
ENDLOOP.

ASSIGNING FIELD-SYMBOL(<ls_item>) 让 field symbol 指向当前表行。改 <ls_item>-amount 就是在改 internal table 当前行。本节只建立“copy vs direct update”的判断；field symbol 的生命周期、typing、unassigned 风险会在后面专门学。还有第三种形式 REFERENCE INTO DATA(lr_item)，会得到当前行的数据引用；Day 04 只需要知道它存在，完整 data reference 放到动态编程章节。

可以把三种 row access 先整理成一张小表：

写法行为初学推荐理解
LOOP AT itab INTO DATA(row)copy 到 work area读为主；要改表必须显式 MODIFY
LOOP AT itab ASSIGNING FIELD-SYMBOL(<row>)直接指向当前行原地修改；细节后面学 field symbol
LOOP AT itab REFERENCE INTO DATA(ref)当前行 data reference先认识；完整引用语义后面学

过滤循环可以用 WHERE：

LOOP AT lt_items INTO DATA(ls_jpy) WHERE currency = 'JPY'.
  out->write( ls_jpy-amount ).
ENDLOOP.

WHERE 只让满足条件的行进入 loop body。对 standard table 来说，这通常仍然要顺序扫描；对 sorted table，如果条件能匹配排序 key 的左侧前缀，系统可以更好地利用 key。Day 04 不需要掌握优化细节，但要知道：WHERE 不是魔法性能开关，性能仍取决于 table category、key 和条件写法。FROM / TO 可以限制 index table 的行号范围，STEP 还可以改变步长或方向；这些偏 index-table 技巧先作为 awareness，后续再学。

再讲 sy-tabix。sy-tabix 是系统字段，表示当前 loop pass 处理到的 table index / row position，但它不是可靠的通用 counter。对 standard table 和 sorted table / sorted key，它通常可以反映当前行位置；对 hashed table，sy-tabix 在每一轮都是 0，因为 hashed table 没有行号索引。并且 loop body 里的 READ TABLE、嵌套 LOOP 等语句可能改写 sy-tabix，所以如果真的要保存当前行号，要在 loop body 一开始立刻写 DATA(current_tabix) = sy-tabix.。如果只是想要 1、2、3 这样的连续计数，自己声明 DATA(counter) = 0. 并在 loop 里加一，更清楚。

控制跳转有三类要先会读。EXIT 在 loop 内退出当前 loop，类似 Java 的 break；但如果写在 loop 外，也会退出当前处理块。CONTINUE 跳过本轮剩余代码，进入下一轮，类似 Java/Python 的 continue。CHECK condition 在 loop body 里如果条件为 false，会跳过当前 loop pass；但 CHECK 在方法或处理块层级也可能直接退出整个处理块，所以新手很容易读错作用范围。日本项目 review 中更偏向显式写法：IF NOT condition. CONTINUE. ENDIF.，比 CHECK condition. 更容易被团队读懂。

还要知道两个风险。第一，不要在遍历同一张 internal table 时随意 INSERT、APPEND、DELETE 这张表，否则可能漏处理、重复处理或触发难查的 runtime error。初学写法可以先收集要改/要删的 key 到另一张内表，loop 结束后统一处理；或者在明确安全的场景下使用 ASSIGNING / MODIFY 修改当前行的字段。第二，避免无意识的 nested loop：LOOP AT lt_a. LOOP AT lt_b. 在小数据里能跑，在真实项目里可能变成性能事故。重复查找时要考虑 key、sorted/hashed table 或预先构造 lookup table。ATC / Code Inspector 往往会把 nested loop 当作质量检查点。

最后给现代语法留一个位置。FOR 表达式、REDUCE、LOOP AT GROUP BY 可以让某些循环写得更函数式、更短；Java/Python 背景的人可能会联想到 list comprehension / reduce / grouping。但这些写法需要先理解 internal table、row type、constructor expression 和 grouping 语义，本书后面再展开。Day 04 的目标是：把 LOOP AT ... INTO、ASSIGNING、WHERE、EXIT、CONTINUE、sy-tabix 的基本边界讲清楚。

为什么重要

  • ABAP 项目里的校验、转换、消息组装、批量处理、RAP reported/failed 处理，经常都是 loop over internal table。
  • INTO copy 与 ASSIGNING direct update 的差异会决定修改是否真正写回内表；这是 Java/Python 学习者最容易误判的行为。
  • sy-tabix 被滥用会导致 hashed table、nested loop、READ TABLE 场景下的隐藏 bug。
  • EXIT、CONTINUE、CHECK 让流程变短，但也可能让代码变难读；日本项目 code review 很重视是否能解释跳转意图。
  • Nested loop 性能风险往往不是语法错，而是数据量变大后才爆出来；越早建立“访问模式和 key”的意识越好。

核心机制

  • LOOP AT itab INTO DATA(row)：逐行复制到 work area，适合只读或显式写回。
  • LOOP AT itab ASSIGNING FIELD-SYMBOL(<row>)：field symbol 直接指向当前行，修改字段会影响内表。
  • WHERE：按条件过滤 loop pass；性能取决于 table category、key 和条件能否被优化。
  • FROM / TO / STEP：按 index 范围、步长、方向控制 loop，先作为 index-table awareness。
  • sy-tabix：对 hashed table 始终为 0；不是通用 counter，且可能被嵌套语句覆盖。
  • EXIT 在 loop 内退出 loop，在 loop 外退出当前处理块；CONTINUE 跳到下一轮；CHECK 在 loop 内可跳过当前 pass，但在 loop 外语义不同，初学优先写显式 IF ... CONTINUE。
  • 不要在 loop 中随意插入/删除同一张内表；避免无意识 nested loop。

日本项目现场表达

  • 常见日语：LOOP処理、ループ処理、条件付きLOOP、ワークエリア、フィールドシンボル、コピー処理、直接更新。
  • 跳转相关：途中終了、次のループへスキップ、条件付き早期終了、EXIT文、CONTINUE文、CHECK文。
  • 性能相关：ネストループ、二重ループ、ネストループ問題、性能劣化、計算量、ループ内テーブル変更。
  • 说明句：LOOP AT ... INTO はワークエリアへのコピー処理であり、内部テーブル本体を更新するには MODIFY が必要です。
  • Review 句：CHECK 文はループ外では条件付き早期終了となるため、可読性の観点で IF ... CONTINUE に書き換えてください。
  • 性能 review 句：二重ループによる計算量増大が懸念されるため、キー付きテーブルまたは事前作成したルックアップテーブルを検討してください。

例子

从一个订单行内表开始：

DATA(counter) = 0.

LOOP AT lt_items INTO DATA(ls_item) WHERE currency = 'JPY'.
  counter += 1.
  DATA(current_tabix) = sy-tabix.

  IF ls_item-amount IS INITIAL.
    CONTINUE.
  ENDIF.

  out->write( |#{ counter } tabix={ current_tabix } { ls_item-travel_id } { ls_item-amount }| ).
ENDLOOP.

这段代码里，ls_item 是 copy，适合只读输出。counter 是人工计数器，比用 sy-tabix 当连续编号更稳定，因为 WHERE 过滤后，sy-tabix 代表当前表行位置，不代表第几次输出。current_tabix 在 loop body 一开始保存，是为了避免后面其他 internal table 操作改写 sy-tabix。

如果要原地修改金额，可以用 ASSIGNING：

LOOP AT lt_items ASSIGNING FIELD-SYMBOL(<ls_item>) WHERE currency = 'JPY'.
  <ls_item>-amount = <ls_item>-amount * '1.10'.
ENDLOOP.

这时 <ls_item> 指向 internal table 的当前行。修改 <ls_item>-amount 会直接修改表里的值，不需要 MODIFY。但这也意味着要更小心：不要在同一个 loop 里随意 DELETE lt_items 或 APPEND 到 lt_items，除非你明确知道当前语句对 loop cursor 的影响。

常见误区

  • 以为 LOOP AT ... INTO 给的是引用：改了 work area 却没有 MODIFY，结果表内容没变。
  • 把 sy-tabix 当连续 counter：WHERE 过滤、nested loop、hashed table 都可能让这个理解出错；hashed table loop 下它始终是 0。
  • 在 loop 里随意 DELETE / APPEND 同一张内表：小样本看不出问题，真实数据里漏行、重复、runtime error。
  • 用 CHECK 让流程过度隐式：loop 外的 CHECK 会变成处理块早退，review 时很容易被误读。
  • 双重 loop 做重复查找：数据量一上来就性能劣化，ATC 或 Code Inspector 可能直接提示 nested loop 风险。
  • Day 04 就急着使用 FOR/REDUCE：代码短了，但读者还没理解 row type、copy/direct update 和 key，后续难以维护。

思考题

  • LOOP AT lt_items INTO DATA(ls_item) 和 LOOP AT lt_items ASSIGNING FIELD-SYMBOL(<ls_item>) 最大的行为差异是什么？
  • 如果 loop 里修改 ls_item-amount，为什么原表不一定变？
  • 为什么 sy-tabix 不应该被当作通用 counter？hashed table 下它是什么值？
  • EXIT、CONTINUE、CHECK 三者分别会让程序流走向哪里？
  • 如果发现 LOOP AT lt_a 里又 LOOP AT lt_b，你会先问哪两个问题？
