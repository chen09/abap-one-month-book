3. Host variables 宿主变量

  • Chapter 02
  • Section 03
  • self-contained study note

3.1 一句话结论

Host variables 是 ABAP SQL 里连接“ABAP 程序变量”和“SQL 语句”的桥。现代 ABAP SQL 用官方称为 escape character 的 @ 前缀标出 ABAP data object，让读者和编译器都能看清：customer_id = @lv_customer_id 左边是数据库字段，右边是 ABAP 程序中的变量；INTO TABLE @DATA(lt_orders) 则表示数据库查询结果回到 ABAP 内表。学会 host variables，不是记一个符号，而是学会区分 SQL 世界、ABAP 内存世界、类型转换、动态 SQL 风险和代码 review 的边界。

3.2 到底学什么

上一节 SELECT statement 查询语句 讲了一条 SELECT 的骨架。本节只盯住一个小而关键的点：SQL 语句里的值从哪里来，又回到哪里去。

先看一条常见读取语句：

DATA lv_customer_id TYPE zsales_order-customer_id VALUE '1000001'.
CONSTANTS lc_status_open TYPE zsales_order-status VALUE 'O'.

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status
       WHERE customer_id = @lv_customer_id
         AND status      = @lc_status_open
       INTO TABLE @DATA(lt_orders).

这段代码里有三类容易混淆的东西。

第一类是 SQL 标识符，例如 zsales_order、sales_order_id、customer_id、status。它们属于 ABAP SQL 能识别的数据源和字段名。

第二类是 ABAP 程序里的数据对象，例如 lv_customer_id 和 lc_status_open。当它们出现在 ABAP SQL 的 operand position 时，要用 @ 前缀写成 host variables。这里的 @ 是语法上的 escape character，用来告诉 ABAP SQL parser “这是 ABAP data object，不是数据库字段名”。它不是字符串插值，也不是 SQL 注入意义上的 escape。

第三类是结果接收目标，例如 @DATA(lt_orders)。查询结果不是留在数据库里给 ABAP 自动读取，而是被放入 ABAP data object。多行结果进入 internal table，单行结果进入结构、工作区或 inline declaration。后面 INTO vs INTO TABLE 结果接收 会专讲这条线。

初学者要形成一个简单读法：看到 @，先问“这是 ABAP 程序里的哪个值？它的类型从哪里来？这个值进入 SQL 条件、SQL 表达式，还是作为结果目标？”这个问题问清楚，SELECT 才会从一串符号变成可解释的业务读取。

3.3 为什么重要

Host variables 重要，是因为 ABAP 项目里的 SQL 条件很少是死的。真实程序通常会根据用户输入、Fiori service 参数、BAPI 参数、配置表、前一步读取结果或业务常量来查询数据。没有 host variables，ABAP 程序无法清楚、安全、可检查地把这些运行时值交给 ABAP SQL。

它的第一层价值是可读性。WHERE customer_id = @lv_customer_id 很明确：数据库字段 customer_id 要等于 ABAP 变量 lv_customer_id。如果忘记 @，读者会疑惑右边到底是字段、SQL 名称，还是程序变量；在现代 strict syntax、尤其 ABAP for Cloud Development 语境里，这通常不是风格问题，而是语法检查会拦下来的问题。

第二层价值是类型和维护。lv_customer_id TYPE zsales_order-customer_id 比 lv_customer_id TYPE string 更有交付意义，因为它继承了业务字段的长度、技术类型和转换规则。项目里很多细小 bug 来自“看起来都是客户号，实际长度、前导零、大小写、conversion exit、domain fixed value 都不一致”。host variable 让 ABAP 类型系统参与 SQL 访问，但前提是变量本身要有合适的类型。

第三层价值是安全边界。静态 ABAP SQL 配合 host variables，通常比手工拼接动态 SQL 更清楚、更容易检查，也避免把用户输入直接拼成 SQL 片段。但是动态 token 是另一套机制：WHERE (lv_where) 里的 lv_where 不属于 host variable，不能通过在前面加 @ 变安全。只要外部输入进入动态条件、动态字段列表或动态排序 token，就必须按动态 SQL 的安全规则验证、白名单限制或转义；在 ABAP Cloud / Clean Core 审查中，还要确认该动态用法是否允许。

第四层价值是 review。日本 SAP 项目 review 经常会问：ホスト変数の型はDB項目と合っていますか、この値は外部入力ですか、動的SQLになっていませんか、リテラルではなく定数化した方がよいですか。能回答这些问题，才算真正掌握 host variables。

3.4 Key mechanism / 核心机制

Host variable 的核心机制可以拆成五件事。

第一，host variable 是 ABAP 程序中已经声明的数据对象。它可以是局部变量、全局变量、常量、字段符号指向的对象等；初学阶段先掌握普通 DATA 和 CONSTANTS 就够了。写进 ABAP SQL 时，用 escape character @ 作为前缀。这里的 escape character 是“给 ABAP SQL parser 区分名字空间的语法前缀”，不是“把危险字符串转义为安全 SQL”的安全动作。

DATA lv_sales_order_id TYPE zsales_order-sales_order_id.

SELECT SINGLE FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status
       WHERE sales_order_id = @lv_sales_order_id
       INTO @DATA(ls_order).

这里 sales_order_id 出现了两次：FIELDS sales_order_id 是读取数据库列；@lv_sales_order_id 是把 ABAP 变量的值传入 WHERE。@ 的存在让两者不会混在一起。

第二，host variable 既可以用于输入，也可以用于输出目标。WHERE status = @lc_status_open 是把 ABAP 值传给数据库查询；INTO @DATA(ls_order) 或 INTO TABLE @DATA(lt_orders) 是把数据库结果接收到 ABAP 对象。方向不同，但都属于 ABAP SQL 和 ABAP data object 之间的连接点。

第三，host expression 是比 host variable 更进一步的写法，用 @( ... ) 把一个 ABAP 表达式的结果传入 SQL。官方文档说明 host expressions 会从左到右求值，结果像 host variables 的内容一样传给数据库。它适合简单、清楚的计算，不适合把复杂业务逻辑塞进 SQL 行里。使用 host expressions 时也会进入更严格的语法检查语境；面向 ABAP Cloud 或 Clean Core 的代码，要按目标 ABAP language version 验证。

DATA lv_min_amount TYPE zsales_order-net_amount VALUE '1000'.
DATA lv_tolerance  TYPE zsales_order-net_amount VALUE '50'.

SELECT FROM zsales_order
       FIELDS sales_order_id,
              net_amount
       WHERE net_amount >= @( lv_min_amount + lv_tolerance )
       INTO TABLE @DATA(lt_large_orders).

这段代码的意思是：先在 ABAP 侧计算 lv_min_amount + lv_tolerance，再把结果作为 SQL 条件值。不是所有 host variable 位置都能无条件换成 host expression；动态 token 等位置还有额外限制。项目代码里，如果表达式开始影响可读性，就应先算成有名字的变量，再作为 @lv_threshold 传入。

第四，literal 不带 @。WHERE status = 'O' 中的 'O' 是 ABAP SQL literal，不是 ABAP 变量，所以不写 @。不过项目代码通常更推荐把有业务含义的值做成常量或 domain/fixed value 的清晰表达：

CONSTANTS lc_status_open TYPE zsales_order-status VALUE 'O'.

SELECT FROM zsales_order
       FIELDS sales_order_id,
              status
       WHERE status = @lc_status_open
       INTO TABLE @DATA(lt_open_orders).

这样 review 时能讨论 lc_status_open 的业务语义，而不是猜 'O' 到底代表 open、old、ok，还是某个项目自定义状态。

第五，ABAP SQL 的 @ 和 Native SQL 的 : 不要混淆。ABAP SQL 现代写法使用 @ 标记 host variables；Native SQL / embedded SQL 语境里常见冒号 :。本教程的主线是 ABAP SQL，不建议初学者把 Native SQL 当默认访问方式。

3.7 Example / 学习例子

3.7.1 条件值进入 SQL
DATA lv_customer_id TYPE zsales_order-customer_id VALUE '1000001'.

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              created_at
       WHERE customer_id = @lv_customer_id
       ORDER BY created_at DESCENDING,
                sales_order_id
       INTO TABLE @DATA(lt_orders).

这类写法最常见。用户、接口或前一步逻辑给出 lv_customer_id，ABAP SQL 用它过滤数据库记录。review 时要看三点：变量类型是否接近数据库字段，值是否已经过输入校验，条件是否足够限制数据量。

3.7.2 常量表达业务含义
CONSTANTS lc_status_released TYPE zsales_order-status VALUE 'R'.

SELECT FROM zsales_order
       FIELDS sales_order_id,
              status
       WHERE status = @lc_status_released
       INTO TABLE @DATA(lt_released_orders).

这里直接写 status = 'R' 也可能能跑，但可读性差。常量名能告诉读者 'R' 的业务含义，并且后续如果状态值由配置或 domain fixed values 管理，也更容易定位影响范围。

3.7.3 结果接收目标也是 ABAP 对象
SELECT SINGLE FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status
       WHERE sales_order_id = @lv_sales_order_id
       INTO @DATA(ls_order).

IF sy-subrc <> 0.

  RETURN.
ENDIF.

@DATA(ls_order) 是 inline declaration，表示在这里声明并接收一个 ABAP data object。它让 demo 和小范围逻辑更简洁；大型逻辑里，如果类型、生命周期或后续处理复杂，也可以先显式声明结构。

3.7.4 Host expression 适合简单计算
DATA lv_min_amount TYPE zsales_order-net_amount VALUE '1000'.
DATA lv_tolerance  TYPE zsales_order-net_amount VALUE '50'.

SELECT FROM zsales_order
       FIELDS sales_order_id,
              net_amount
       WHERE net_amount >= @( lv_min_amount + lv_tolerance )
       INTO TABLE @DATA(lt_large_orders).

这段代码适合展示 host expression，但项目里要控制复杂度。如果计算本身有业务含义，更推荐先写成有名字的变量：

DATA lv_minimum_required_amount TYPE zsales_order-net_amount.
lv_minimum_required_amount = lv_min_amount + lv_tolerance.

SELECT FROM zsales_order
       FIELDS sales_order_id,
              net_amount
       WHERE net_amount >= @lv_minimum_required_amount
       INTO TABLE @DATA(lt_large_orders).

后一种写法多一行，但 review 更容易说清楚。

3.8 Counterexample / Failure mode

失败模式一：忘记 @，把 ABAP 变量写成 SQL 名称。

SELECT FROM zsales_order
       FIELDS sales_order_id
       WHERE customer_id = lv_customer_id
       INTO TABLE @DATA(lt_orders).

在现代 ABAP SQL 语法下，这种写法很容易触发语法问题，或者让读者误解 lv_customer_id 是 SQL 侧标识符。新代码要坚持 @lv_customer_id。

失败模式二：变量类型随便用 string。

DATA lv_customer_id TYPE string.

客户号、物料号、公司代码、销售组织、状态值通常都有明确 DDIC 类型和 domain 语义。随便用 string 可能带来前导零、长度、大小写、转换例程和比较行为问题。更稳的写法是参考表字段、data element 或接口参数类型。

失败模式三：把 @ 当成动态 SQL 安全保证。

DATA lv_where TYPE string.
" lv_where is built from external input.

SELECT FROM zsales_order
       FIELDS sales_order_id
       WHERE (lv_where)
       INTO TABLE @DATA(lt_orders).

这里风险不在普通 host variable，而在动态 SQL token。WHERE (lv_where) 这种括号里的动态条件不属于 host variable，也不能写成 WHERE (@lv_where) 来解决。外部输入一旦进入动态条件、动态字段列表、动态排序或动态表名，就必须按 SAP 官方动态 SQL 安全规则检查、白名单限制或转义；在 ABAP Cloud / Clean Core 场景，还要先确认该动态模式是否被允许。@ 在这里不是“不够安全”，而是“不适用”。

失败模式四：过度使用 host expression。

WHERE net_amount >= @( get_base_amount( ) + get_risk_buffer( ) - get_discount( ) )

如果表达式里混入方法调用、隐式副作用或复杂业务公式，SQL 行会变得难审查。初学者应该优先写清楚的中间变量，让业务含义在 ABAP 代码里显性化。

失败模式五：把 literal 写满代码。

WHERE status = 'R'

孤立看这行没有语法问题，但业务含义隐藏了。项目里更推荐常量、枚举式建模、domain fixed values、CDS/RAP 语义或配置表，让状态值的来源可追踪。

3.9 思考问题

  • 这条 SQL 里的每个 @... 分别是输入值、输出目标，还是 inline declaration？
  • host variable 的 ABAP 类型是否来自相同 data element、相同表字段，或至少和数据库字段兼容？
  • 如果变量来自 UI、接口或文件上传，输入校验在哪里发生？
  • 这里该用 literal、常量、配置值，还是 domain fixed value？
  • 如果用了 @( ... )，表达式是否简单到 review 时一眼能看懂？
  • 有没有动态 SQL token？如果有，它不属于 host variable，是否做了白名单、转义或 CL_ABAP_DYN_PRG 类检查？在 ABAP Cloud / Clean Core 中是否允许？
  • @DATA(...) 声明的结果对象生命周期是否合适？后续是否需要明确表类型或 key？
  • 如果迁移到 ABAP Cloud / Clean Core 场景，这个数据源和访问方式是否允许？
