41. OData V2 基础差异

  • Chapter 03
  • Section 41
  • self-contained study note

41.1 一句话结论

OData V2 基础差异 是学习 SAP OData 时必须认识的旧生态基础：很多 ECC、SAP Gateway、早期 Fiori、既有接口和客户扩展仍在使用 OData V2。现代 RAP 新开发通常更偏向 OData V4，尤其是 OData V4 UI / Web API binding；但项目现场仍需要读懂 V2 的 URL、metadata、batch、CSRF token、Gateway error 和兼容限制。

一句话记忆：OData V2 不是“应该新学的唯一方向”，而是“很多旧系统和既有 Fiori/API 仍然存在的现实”。

41.2 到底学什么

本节不要求把 OData V2 规范全部背下来，而是要让初学者在 SAP 项目中看到 V2 服务时不慌。尤其是在日本 SAP 项目里，维护既有 Fiori app、Gateway service、/sap/opu/odata/sap/... 路径、$metadata、$batch、CSRF token、SEGW 生成类、IWFND 错误日志，仍然很常见。

要先分清三件事。第一，OData 是协议，不是 ABAP 语言，也不是 CDS 本身。第二，OData V2 是协议版本和 SAP Gateway 旧生态中的主力之一；OData V4 是后续更现代的协议方向，RAP 新服务常围绕 V4 展开。第三，协议版本不等于业务能力：一个 V2 服务也可以很复杂，一个 V4 服务也可能只暴露简单读取；关键是看服务模型、实现方式、权限和事务语义。

本节重点学习 V2 的项目识别能力：看到 URL 能判断它大概率是 V2；看到 $metadata 能知道 entity set 和 association；知道修改请求通常要带 CSRF token；知道批量请求可能通过 $batch；知道错误排查不只看浏览器，还要看 Gateway / backend log；知道从 RAP 新开发角度，V2 多数是兼容或历史原因，而不是默认首选。

还要学会 V2 的“维护者视角”。新开发者常问“既然有 V4，为什么还学 V2？”答案是：很多项目不是从空白开始。既有 UI5 可能写死了 V2 model；外部系统可能只认证过 V2 endpoint；客户网关、SICF、hub deployment、metadata cache 和错误日志流程都围绕 SAP Gateway 建好。读懂 V2，才能安全维护、联调和评估迁移，而不是盲目重写。

本节不会教读者从零手写 SEGW 服务，而是让读者掌握排查链：URL -> metadata -> request headers -> CSRF/cookie -> batch inner response -> Gateway log -> backend implementation -> authorization/transaction evidence。能走完这条链，就已经具备项目初级交付能力。

41.3 为什么重要

OData V2 重要，是因为项目不会因为新技术推荐 V4 就立刻没有 V2。很多生产系统运行多年，Fiori app、UI5 前端、外部接口、Excel/ETL 工具、middleware 都可能已经绑定 V2 endpoint。学习 ABAP/RAP 时，如果只知道“现在推荐 V4”，却读不懂旧 V2 服务，就很难参与真实项目维护。

对交付来说，V2 常出现在缺陷分析和接口联调中。用户说“Fiori 保存失败”，实际可能是 CSRF token 过期、Gateway metadata cache 未更新、$batch 中某个 change set 报错、DPC_EXT 实现抛异常、backend authorization 失败，或前端解析 V2 error payload 不完整。没有 V2 基础，就容易把所有问题都归咎于 CDS 或 RAP。

对 Java/Python 背景学习者，OData V2 看起来像 REST，但不完全等同普通 REST。它有 $metadata、entity set、navigation、query option、batch、CSRF token、SAP Gateway runtime 和 Fiori/UI5 消费约定。不能只用“GET/POST JSON”理解它。

同时也要避免另一个极端：不要把 V2 当成所有新项目的默认答案。若是在 ABAP Cloud / RAP 新开发语境下，通常应先看 OData V4 UI 或 Web API binding 是否更合适；只有在兼容既有 UI5、Gateway、外部系统约束或客户标准时，才有充分理由选择或维护 V2。

41.4 Key mechanism / 核心机制

第一层机制是 endpoint 形态。经典 SAP Gateway V2 服务经常能看到类似 /sap/opu/odata/sap/Z_SRV/ 的路径。服务名、entity set、key predicate、navigation、query option 都会进入 URL。看到这类路径时，要先想到 Gateway / OData V2 生态。

第二层机制是 $metadata。V2 metadata 描述 entity type、entity set、property、association、navigation property、function import 等对象。前端和工具通过 metadata 理解服务模型。调试时，先看 metadata 是否包含预期 entity/action/function，再看具体请求。

第三层机制是 query option。V2 常见 $filter、$select、$expand、$orderby、$top、$skip 等查询参数。它们很像数据库查询条件，但不是直接 SQL。后端实现需要把这些请求语义转换成合适的读取逻辑，并注意性能和权限。

第四层机制是 CSRF token。对修改类请求，SAP OData V2 场景常需要先获取 X-CSRF-Token，再在 POST/PUT/MERGE/DELETE 或 $batch 中带上。很多“POST 失败”其实不是业务逻辑错，而是 token、cookie、session 或 header 处理不对。

第五层机制是 $batch 和 change set。前端可能把多个读取或修改请求打包成 batch。错误分析时，要打开 batch response，找到具体哪一个 change set 或 operation 失败。只看最外层 HTTP status 往往不够。

第六层机制是 Gateway runtime。传统 V2 服务可能由 SEGW / MPC_EXT / DPC_EXT、CDS publish、Gateway hub/embedded deployment 等方式提供。错误排查常涉及 /IWFND/ERROR_LOG、/IWBEP/ERROR_LOG、metadata cache、service activation、ICF node、authorization、backend dump。

第七层机制是和 RAP 的关系。RAP 也可以暴露 OData 服务，但现代 RAP 教程通常优先讲 V4 binding。学习 V2 的目的，是识别旧服务、理解兼容场景、做联调和迁移评估，而不是把 RAP 写回旧 Gateway 手工实现模式。

第八层机制是服务实现位置。传统 V2 服务常见两类：一类是 SEGW 项目生成的 MPC/MPC_EXT、DPC/DPC_EXT 类，模型定义和数据提供逻辑分开；另一类是基于 CDS 或其他框架暴露的 Gateway 服务。项目排查时要先问“这个 service 是怎么实现的”：是手写 DPC_EXT、CDS exposure、BOPF/Fiori 旧栈、还是标准 SAP 服务。实现来源不同，能改的对象、测试方式和风险完全不同。

第九层机制是 Gateway hub vs embedded。某些系统是前端 Gateway hub 和后端业务系统分离；某些是 embedded Gateway。错误可能发生在前端服务器的 service activation、ICF、metadata cache、系统别名、destination，也可能发生在后端 ABAP 实现、权限、dump 或业务逻辑。只看一个系统的日志，可能看不到完整原因。

第十层机制是 V2/V4 迁移判断。迁移不是“把 /sap/opu/odata/sap/Z_SRV 改成一个 V4 URL”。要检查消费者代码、metadata、function import/action 映射、batch 行为、CSRF/session、错误 payload、authorization、ETag/concurrency、性能、监控和运维。若外部系统依赖 V2 payload 格式，迁移必须作为 API 变更管理。

可以用下面的排障矩阵：

项目现场真正困难的地方，往往不是“V2 和 V4 哪个更新”，而是“当前系统、当前消费者、当前交付风险允许哪一种”。可以用下面的表做初步判断：

场景倾向判断理由必须确认

如果只能记住一个原则：新开发先问 V4 是否合适，既有服务先问 V2 依赖在哪里。前者避免继续复制旧模式，后者避免为了“升级技术”破坏生产接口。日本项目中，这类判断通常要落到 既存互換性、影響調査、移行方針 和 回帰テスト範囲，不能只写“V2 は古いので V4 にする”。

41.7 Example / 学习例子

一个典型 V2 服务 URL 可能像这样：

/sap/opu/odata/sap/Z_TRAVEL_SRV/TravelSet('0000000123')

开发者首先要识别几个部分：

  • /sap/opu/odata/sap/：SAP Gateway OData 服务路径的常见形态。
  • Z_TRAVEL_SRV：服务技术名。
  • TravelSet：entity set。
  • ('0000000123')：key predicate。

查看 metadata 时，可能访问：

/sap/opu/odata/sap/Z_TRAVEL_SRV/$metadata

如果要执行修改请求，常见联调顺序是：

1. GET service root or metadata with header X-CSRF-Token: Fetch
2. 从 response header 取 X-CSRF-Token 和 cookie
3. POST / PUT / MERGE / DELETE 时带回 token 和 cookie
4. 检查 HTTP status、response payload、Gateway error log、backend log

如果前端使用 $batch，还要展开 batch response，确认是哪个 operation 失败。很多时候浏览器只显示“batch failed”，但真正的业务错误在 change set 内层 response。

把这个例子和 RAP V4 对比：V4 章节会更关注现代 RAP binding、action/function、metadata 和事务语义；V2 章节更强调既有 Gateway 服务的兼容、联调和故障分析。

可以把 V2 联调证据整理成一页：

V2 API Evidence Sheet

Service:
  Technical service name:
  Service root:
  System/client:
  Gateway deployment: hub / embedded / unknown

  Entity sets:
  Navigation:
  Function imports:
  Last cache refresh / activation evidence:

  Method:
  URL:
  Headers:
  CSRF token handling:
  Payload:

  HTTP status:
  OData error payload:

41.8 Counterexample / Failure mode

反例一：把 V2 当普通 REST。只会 Postman 发 JSON，不看 $metadata、CSRF token、batch 和 Gateway log。修正方式是按 OData/Gateway 证据链排查。

反例二：POST 失败就改后端业务代码。实际原因是缺少 X-CSRF-Token 或 cookie。修正方式是先验证 HTTP header/session/token。

反例三：只看最外层 batch status。实际失败在 batch 内某个 change set。修正方式是解析完整 batch response。

反例四：新 RAP 项目默认选 V2。没有兼容理由却沿用旧习惯，导致错过 V4/RAP 更自然的服务模型。修正方式是先评估 OData V4 UI / Web API binding。

反例五：忽略 metadata cache。后端模型已改，前端看到旧 metadata。修正方式是检查 Gateway metadata cache、服务注册和目标系统状态。

反例六：只在 DEV 测试。QAS/PRD 的 Gateway hub、ICF、role、service activation 不一致。修正方式是把系统环境分层和移送后验证纳入测试。

反例七：把 function import 当成 RAP action。V2 旧服务里的 function import 和 RAP/V4 bound action 在模型、事务语义、错误返回和 UI 消费上可能不同。修正方式是迁移时重新设计 operation contract。

反例八：metadata cache 没有作为交付步骤。开发系统模型已变，QAS 仍读旧 metadata，前端字段或 navigation 不出现。修正方式是在移送/部署 checklist 中加入 metadata/cache/activation 验证。

反例九：迁移 V2 到 V4 不做消费者影响分析。旧 UI5 或 middleware 依赖 V2 payload、batch 和错误格式，换成 V4 后解析失败。修正方式是保留兼容层、版本化接口或安排消费者改造。

41.9 思考问题

  • 当前服务是 OData V2 还是 V4？证据是什么？
  • URL 中的 service name、entity set、key predicate 分别是什么？
  • $metadata 中是否包含预期 entity set、association、function import？
  • 修改请求是否正确处理 CSRF token、cookie 和 header？
  • 如果使用 $batch，是否检查了内层 response？
  • 错误排查时应看前端 console、Gateway log、backend dump 还是 authorization trace？
  • 新开发时选择 V2 的理由是兼容、客户标准，还是只是习惯？
