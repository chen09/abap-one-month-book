17. Exception vs message 异常与消息

  • Chapter 04
  • Section 17
  • self-contained study note

17.1 一句话结论

Exception vs message 异常与消息 的核心，是区分“程序是否还能继续执行”和“人或调用方应该看到什么反馈”；在本教程里要学的是：异常用于控制错误路径，消息用于表达可理解的业务结果，两者要配合，而不是互相替代。

在 ABAP 项目里，很多代码坏味道都来自这个边界不清。有人把所有业务校验都写成技术异常，结果用户看到一堆内部错误；有人把严重技术错误吞掉，只返回一条 处理失败，结果开发者无法追踪；也有人在深层工具类里直接 MESSAGE e...，导致方法无法复用、无法测试、也无法被接口场景正确处理。

正确的思路是：底层逻辑发现无法继续的技术或契约错误时，可以抛出 exception；业务规则判断出的可预期结果，应转成明确 message 或 reported message；接口、批处理和 UI 层再把这些信息组织成 Error message design 错误消息设计 中讲过的 error code、message、target、retryable、log id。这样程序内部可维护，对外反馈也可理解。

17.2 到底学什么

第一，要学异常的角色。异常表示当前执行路径遇到了不能正常继续的情况，例如输入契约被破坏、依赖对象不可用、数据库访问失败、外部系统返回不可恢复错误、程序状态不一致等。异常的重点是控制流：抛出、捕获、转换、记录、终止或恢复。

第二，要学消息的角色。消息表示给用户、调用方、测试人员或业务流程看的反馈，例如“公司代码不存在”“该单据已审批，不能修改”“用户没有执行权限”“对象正在被锁定，请稍后重试”。消息的重点是沟通：让接收者理解问题和下一步动作。

第三，要学分层转换。底层类不应该随便决定 UI 怎么显示，接口层也不应该把所有内部异常原文暴露出去。更稳妥的设计是：底层抛出语义清楚的 exception，应用服务层捕获并转换成业务消息或 API error response，最外层再附上 request id、Application Log 应用日志 编号或字段定位。

第四，要学可测试性。异常可以用 ABAP Unit 验证“这个输入会抛出哪个异常”；消息可以用测试验证“这个业务场景会返回哪个 message id / error code / target field”。如果代码在任意深度直接弹消息或终止流程，就很难写测试，也很难在 RAP、OData、后台 job 和批处理之间复用。

第五，要学错误分类。项目里常见错误至少可以分为业务校验、权限、锁/并发、幂等/重复请求、技术依赖失败、系统程序错误。不同类别对外消息、重试建议、日志深度和负责人都不同。把它们都变成 SYSTEM_ERROR，会让用户、接口调用方和运维都失去判断依据。

第六，要学入口差异。同一个错误在 Fiori、OData/API、background job、ABAP Unit 中表现不同：Fiori 需要字段定位和用户文案；API 需要稳定 error code 和 payload；background job 需要 job log / spool / application log；ABAP Unit 需要可断言的结果对象或异常类型。因此深层逻辑不要直接绑定某一种入口。

17.3 为什么重要

这个边界影响项目的稳定性。用户看到的不是代码结构，而是消息；开发者维护的是异常路径和调用栈；运维需要的是日志和证据；接口调用方需要的是可解析的错误契约。如果 exception 和 message 混乱，四方都会痛苦。

在经典 SAP GUI 场景里，MESSAGE 语句可能直接控制画面流程；但在现代项目里，同一段业务逻辑可能被 Fiori、OData、RAP behavior、后台 job、测试类和接口复用。深层逻辑直接发 UI 消息，会把代码绑定到某个调用方式。把异常和消息分层处理，才能让业务规则在不同入口下保持一致。

在 RAP/OData/API 场景里，这个边界更关键。RAP handler 或 service 层需要把业务校验结果放到 reported messages 或 response payload；真正不可恢复的技术问题要以受控方式记录和返回。外部系统不应看到内部异常栈，但也不能只收到一个无法判断的 failed。在 RAP 里，failed 标记哪条 key / instance 失败，reported 携带要返回的消息文本，两者配合使用。

错误设计还会影响支持成本。一个好消息能告诉用户“改哪一项、能不能重试、联系谁”；一个好异常能告诉开发者“哪里坏了、上下文是什么、调用栈怎么走”；一个好日志能告诉运维“哪笔业务、哪个 request id、哪个技术依赖失败”。这三者不是互斥，而是同一个错误契约在不同受众面前的不同表达。

17.4 Key mechanism / 核心机制

可以用一个简单规则判断：如果问题表示“这条执行路径不能继续”，优先考虑 exception；如果问题表示“业务上不允许或需要提醒”，优先考虑 message；如果问题既中断流程又要告诉用户原因，就在上层捕获 exception 后转换成 message 或 API error。

例如，数据库连接异常、反序列化失败、必需依赖对象为空、外部服务协议不符合预期，这些更接近技术异常。公司代码不存在、审批状态不允许修改、输入日期超过允许范围、用户缺少某公司代码权限，这些更接近业务消息或可预期错误。锁冲突、重复请求、外部系统超时则需要结合上下文判断：它们可能以异常形式中断当前路径，但对外必须转成可行动消息，说明是否可重试。

异常类应尽量有语义，不要所有地方都抛同一个泛用异常。参见 Exception class 异常类 和 TRY CATCH exception handling。业务消息应尽量稳定、可翻译、可定位，参见 Message class and T100 messages。两者的共同点是都属于错误契约，不能随意变更。

还有一个重要机制是“不要吞异常”。捕获异常后如果只写一条含糊消息，不记录原始上下文，就会丢失诊断证据。比较好的做法是：用户可见消息保持清楚和安全；内部日志记录 exception type、上下文、request id、业务键和 application log id；敏感信息不要外泄。

可以用下面的错误分类表做设计：

错误类别内部表达对外消息是否可重试证据
业务校验validation result / business exception字段级 message、业务 error code修正数据后可重试target field、业务键
权限不足authorization result / exception无权限 message，不暴露内部对象过多细节授权修复后可重试SU53/trace、user、role
锁冲突lock exception/result对象被占用、稍后重试通常可重试lock key、holder/time
重复请求idempotency decision已处理/返回既有状态按状态判断request id、existing status
外部系统失败technical exception技术错误、稍后重试或联系支持视情况external ref、log id
程序错误unchecked/program exception安全泛化消息不建议用户盲目重试dump/log/correlation id

分层映射可以这样写：

Deep logic
  Raise semantic exception or return structured validation result.

Application service
  Map exception/result to error category, message id, target, retryable.

API/RAP/UI/job adapter
  Convert to reported message, OData error, job/application log, or UI message.

Operations evidence
  Store request id, business key, root cause, safe user message, log id.

这样做的关键是：内部 root cause 不丢，对外信息不泄露，调用方能判断下一步。

17.5 日本项目现场说法

常见日语包括：例外、例外クラス、メッセージ、業務エラー、技術エラー、エラー応答、メッセージID、呼び出し元、再実行可否、障害調査。

设计说明中可以这样写：

  • 下位クラスでは画面メッセージを直接出力せず、例外または結果オブジェクトとして上位層へ返却します。
  • 業務チェック結果は、呼び出し元でユーザ向けメッセージまたはAPIエラー応答に変換します。
  • 技術例外は内部ログに詳細を記録し、ユーザ向けには機微情報を含まないメッセージを返却します。

代码 review 时可以这样问：

  • この例外は呼び出し元で適切にメッセージへ変換されていますか。
  • 深い階層のクラスで MESSAGE を直接出していませんか。
  • 業務エラーと技術エラーの切り分けは明確ですか。
  • ユーザ向けメッセージには対応方法を示し、内部例外の詳細はアプリケーションログに記録します。
  • API 応答では安定したエラーコード、対象項目、再実行可否を返してください。
  • バックグラウンドジョブでは、ジョブログだけでなく業務キー付きのアプリケーションログを確認します。

17.7 Example / 学习例子

假设有一个校验公司代码的服务方法。坏设计是在工具方法里直接弹消息：

  IF lv_bukrs IS INITIAL.
    " 假设 zmsg 的 001 号消息文本为 "Company code is required"
    MESSAGE e001(zmsg).
  ENDIF.

这样做的问题是：这个方法一旦被后台 job、OData 服务或 ABAP Unit 调用，MESSAGE e 的行为可能不符合调用方需要。更好的设计是让底层表达业务校验结果或抛出有语义的异常，由上层决定怎么转成消息。

概念上可以这样分层：

Domain/service logic:
  detects missing company code
  returns validation result or raises business exception

Application/API layer:
  converts it to message ID / error code / target field
  adds request id and application log id

UI/API caller:
  displays or processes the message

在 AC0567C 的批量上传中，第 27 行公司代码无效，底层校验可以返回结构化 validation error；API 层把它转成 BUSINESS_VALIDATION_FAILED，target 为 items[27].companyCode；Application Log 记录完整处理明细。这样既没有把 UI 消息写死在底层，也没有丢失用户需要的信息。

再看一个完整映射例子：

Internal condition:
  Company code 9999 is not allowed for current user.

Internal result:
  zcx_authorization_denied
  context: user, company_code, operation=EXECUTE

Application mapping:
  errorCategory = AUTHORIZATION
  errorCode = AUTHORIZATION_FAILED
  target = header.companyCode
  retryable = false until authorization is changed
  logId = BAL-4711

API response:
  Do not expose raw authorization object values unless project policy allows it.
  Return stable error code and user-safe message.

Operations evidence:
  SU53/trace can be collected with user/time/operation.

同样地，外部系统 timeout 不应该直接把原始 HTTP stack trace 返回给调用方。服务层可以把它映射成 TECHNICAL_DEPENDENCY_FAILED，返回 request id 和 log id；Application Log 保存外部 endpoint、response code、timeout category 和是否可重试。

17.8 Counterexample / Failure mode

常见失败模式有以下十类。

第一，深层工具类直接 MESSAGE。这会让代码难以复用，也让测试和接口处理变得不可控。

第二，所有错误都抛技术异常。业务校验失败本来是可预期结果，却被包装成系统异常，用户和调用方无法判断是否修正数据即可重试。

第三，吞掉异常只返回泛用消息。开发者丢失 root cause，运维无法定位，用户只能反复截图。

第四，把内部异常细节暴露给外部系统。完整 stack trace、SQL、类名、服务器路径、token 和敏感字段都不应直接出现在外部 error response。

第五，没有统一映射规则。不同开发者各写各的异常和消息，导致同一种业务错误在不同 API、报表、Fiori app 中表现不同，测试和客户都无法形成稳定预期。

第六，业务消息没有 target。批量输入第 27 行错了，但消息只说“输入错误”，用户不知道改哪里。

第七，所有技术异常都返回可重试。某些程序错误或数据不一致不是用户重试能解决的，错误契约要区分 retryable。

第八，日志只记录用户可见文本。用户文本通常经过翻译和简化，不能替代 root cause、exception type、业务键和 request id。

第九，API、UI、job 各自定义一套错误码。同一业务错误在不同入口表现不同，回归测试和客户文档都会混乱。

第十，把安全敏感信息写进外部消息。SQL、token、内部类名、服务器路径、完整授权对象值都需要按项目安全规则控制。

  • 一个业务校验失败应该抛 exception、返回 message，还是两者都参与？在哪一层转换？
  • 为什么深层类直接 MESSAGE e... 会影响复用和测试？
  • 技术异常对用户可见时，哪些信息应该隐藏？哪些信息应写入受控日志？
  • 在 RAP/OData 场景里，reported message 和 exception 的职责有什么差异？
  • AC0567C POC 中哪一段逻辑最需要明确 exception-to-message mapping？

  • 用户可见消息和内部 root cause 是否分开保存？
