33. Table field 字段定义

  • Chapter 01
  • Section 33
  • self-contained study note

本节结论

Table field 字段定义，是 ABAP Dictionary database table 的列级定义。它不只是“字段名 + 数据类型”，而是同时决定字段是否属于 key、直接使用 built-in dictionary type 还是引用 data element、是否需要 not null、是否带 currency / quantity reference、会如何影响 ABAP SQL、Data Preview、RAP/CDS、测试数据和日本项目设计审查。

学习要点

上一节已经把 DDIC database table 字典表 看成一个 repository object。本节把镜头拉近到 field list。Database table 的 field list 是花括号 { ... } 里的字段清单；每一个 field definition 通常由 field name、冒号、field type、可选的 key / not null / annotation，以及结尾分号组成。SAP Developers 教程里先让读者理解 table fields，再添加 built-in type 字段、existing data element 字段和 new data element 字段，这个顺序很适合初学者。

先看最小形状：

define table zxx_order_i {
  key client   : abap.clnt not null;
  key order_id : zxx_order_id not null;
  key item_no  : abap.numc(6) not null;
  product_id   : zxx_product_id;
  created_by   : abap.char(12);
  created_at   : abap.utclong;
}

这里每一行都是 table field。client、order_id、item_no 前面有 key，所以它们组成 primary key。client 的 type 是 abap.clnt，表示 client 字段；item_no 和 created_by 直接使用 built-in dictionary type；order_id 和 product_id 使用自定义 data element，因此可以复用字段 label、documentation、可能的 search help 和背后的 domain 技术属性。created_at 使用 abap.utclong，表示这是 UTC timestamp 字段。

对 Java/Python 背景学习者来说，最重要的是不要把 field 看成 ORM class 的一个普通属性。ABAP Dictionary field 位于 repository 世界中，它的定义会被 ABAP SQL syntax check、Data Preview、CDS/RAP、UI metadata、foreign key / search help、currency / quantity 语义和 transport/activation 一起使用。字段定义一旦被多个程序、CDS view、OData 服务、RAP BO 或报表依赖，修改字段长度、类型、key 或 reference 关系就不是本地小改，而是项目影响分析。

字段类型有两条主路径。第一条是直接使用 built-in dictionary type，例如 abap.char(12)、abap.numc(6)、abap.int4、abap.utclong。这适合教学、技术字段或语义很局部的字段，但字段 label、documentation、复用语义较弱。第二条是使用 data element，例如 zxx_order_id、/dmo/customer_id。Data element 是更推荐的业务字段入口，因为它把技术属性、字段标签、说明和背后的 Domain 域对象 复用起来。上一节已经确认：domain 不能直接写成 database table field list 里的 field type；正确路径是 field -> data element -> domain。如果输入了一个不存在的 data element，ADT 通常会给出 quick fix，常见做法是在错误位置用 Ctrl+1 创建新的 data element，而不是把 domain 名硬塞进字段类型位置。

key 和 not null 要一起读，但不要混成业务必填。key 表示该字段属于 primary key，key fields 要放在 field list 开头，并共同唯一标识一行。not null 在 key field 上是 Dictionary DDL 的语法/工具层要求，ADT 可能在 activation 时自动补齐；它不应被讲成运行时业务必填校验，也不要手工加到 abap.string、abap.rawstring 这类没有普通初始值语义的字段上，否则编辑器会直接报语法错误。比如“用户必须输入有效产品号”“金额不能为负数”不是靠 not null 完成，而应通过输入检查、RAP validation、domain fixed values、foreign key、application logic 或 UI/service 校验完成。

金额和数量字段要特别小心。SAP Learning 明确说明，CURR amount field 需要 reference 到 CUKY currency code field，QUAN quantity field 需要 reference 到 UNIT unit field。在 Dictionary DDL 中常用 @Semantics.amount.currencyCode 和 @Semantics.quantity.unitOfMeasure 表达这种关系，并且 annotation 要放在对应 amount / quantity 字段的前一行，argument 通常写成 tablename.fieldname。初学时至少要形成条件反射：看到 amount / quantity，不要只问 length 和 decimals，还要问 currency/unit 在哪里。

下面的片段只用于读懂 field-level semantic reference，不要求立刻照抄到项目里；真实 table 还需要表头 annotation 等完整定义才能激活：

define table zxx_order_i {
  key client        : abap.clnt not null;
  key order_id      : zxx_order_id not null;
  key item_no       : abap.numc(6) not null;
  currency_code     : abap.cuky;
  @Semantics.amount.currencyCode : 'zxx_order_i.currency_code'
  net_amount        : abap.curr(15,2);
}

这段代码里，net_amount 本身只是金额数值，必须通过 currency_code 才知道是 JPY、USD 还是 EUR。没有这个 reference，系统和读者都缺少业务语义。数量字段同理，需要 unit-of-measure field。后面学习 Data element 数据元素、Domain 域对象、CDS annotation 和 RAP validation 时，还会继续回到这个“技术类型 + 业务语义”的分工。

Field definition 还要分清“能 activation”和“设计合理”。一个字段直接用 abap.char(80) 可能可以激活，但如果它其实是客户编号、物料编号、状态、币种、单位、语言、日期或金额，就应该优先找已有 SAP data element 或项目标准 data element。一个字段长度从 20 改成 10 可能在语法上只是改一行，但会影响已有数据、接口截断、selection condition、CDS projection、OData payload、ALV 显示、测试数据和报表 layout。字段设计是数据库、程序和交付文档的共同契约。

Key field 还会带来 primary index 的阅读方式。Primary key 会形成 primary index；查询如果能按 key 的前导字段组合筛选，通常更容易得到稳定的 key access。反过来，只用复合 key 的后半段字段做条件，往往会成为性能 review 的关注点。Day 05 不要求马上学 SQL optimizer，但要知道 key order 不是随便排版，它会影响后续 SELECT statement 查询语句、锁设计和 RAP key 讨论。

本节的最小心智模型是：table field 是 DDIC table 的列级契约。读字段时至少看六件事：字段名、是否 key、直接 type 或 data element、背后 domain、currency/quantity/reference、是否受 client/key/activation/transport 影响。能按这个顺序读表，后面学 data element、domain、foreign key、search help、lock object 才不会散。

为什么重要

  • ABAP SQL、CDS、RAP、OData、报表、接口和测试数据都直接依赖 table field 的名称、类型、长度和语义。
  • Field type 选错会造成截断、转换错误、排序/比较错误、金额小数位错误、日期时间误解和 UI label 混乱。
  • Key field 设计会影响唯一性、查询路径、锁对象、RAP BO key、Data Preview 和数据迁移。
  • Currency / quantity field 若缺少 reference，金额和数量会失去业务单位，后续 UI、报表、接口和审查都会出问题。
  • Java/Python 背景学习者容易只看数据库列；SAP 项目必须同时看 data element、domain、field label、fixed values、foreign key、search help 和 transport/activation。
  • 日本项目里，字段定义经常出现在基本设计、详细设计、テーブル定義書、レビュー指摘、移送前确认和障害調査中。

Key mechanism / 核心机制

  • Field list：define table 花括号中的字段清单，每个字段用分号结束。
  • Field name：字段名是程序、SQL、CDS、接口和文档共同引用的稳定标识，修改成本高。
  • Built-in dictionary type：例如 abap.char(12)、abap.numc(6)、abap.int4、abap.utclong，可直接用于 DDIC table field。
  • Data element：推荐的业务字段类型入口，承载字段语义、label、documentation，并可间接复用 domain。
  • Domain：只能通过 data element 等间接路径影响 database table field，不要直接写成 table field type。
  • Key field：用 key 标记，组成 primary key；key fields 应连续位于 field list 开头。
  • Primary index awareness：primary key 会形成 primary index，key order 会影响后续查询和性能审查。
  • Client field：client-dependent table 通常把 abap.clnt 字段放在第一个 key field。
  • not null：key field 的语法/工具层要求，不能替代业务必填、值域检查或 RAP validation，也不要随意加到 string/rawstring 等字段，否则会触发语法错误。
  • Currency reference：CURR amount field 需要 currency code field，常用 @Semantics.amount.currencyCode 表达。
  • Quantity reference：QUAN quantity field 需要 unit field，常用 @Semantics.quantity.unitOfMeasure 表达。
  • Technical vs semantic：length、decimals、data type 是技术形状；字段 label、业务含义、单位、值域和使用场景是语义。
  • Activation impact：字段变更可能触发数据库表调整、依赖对象语法检查、数据转换、transport 和测试影响分析。

日本项目现场表达

  • 常见日语：テーブル項目、項目定義、項目名、データ型、桁数、小数桁、キー項目、主キー項目。
  • 类型/语义：組込み型、データエレメント指定、ドメイン参照、項目ラベル、固定値、値域。
  • 金额/数量：通貨コード項目、金額項目、数量項目、単位項目、参照項目、参照テーブル。
  • 审查/错误：NOT NULL、有効化エラー、項目追加、項目変更、影響調査、データ移行影響。
  • 说明句：この項目はデータエレメントを指定しており、項目ラベルと技術属性は DDIC 側で再利用されます。
  • Review 句：金額項目の場合は、通貨コード項目との参照関係を確認してください。
  • Review 句：キー項目の追加・変更は、既存データ、検索条件、ロック設計、RAP BO キーへの影響を確認してください。

学习例子

观察一张 table 的字段时，可以逐行填这个表：

Field name:
Key field: yes / no
Client field: yes / no
Direct type or data element:
Built-in dictionary type:
Data element:
Domain behind data element:
Length / decimals:
Currency or quantity reference:
Field label / documentation:
Project risk if changed:

以一个订单行项目表为例：

key item_no   : abap.numc(6) not null;
product_id    : zxx_product_id;
created_by    : abap.char(12);
currency_code : abap.cuky;
@Semantics.amount.currencyCode : 'zxx_order_i.currency_code'
net_amount    : abap.curr(15,2);

读法如下：

  • item_no 是 key field，直接使用 abap.numc(6)，要检查排序、补零、唯一性和接口显示规则。
  • product_id 使用 data element，应该继续追踪到 data element 和 domain，看 label、长度、值域、search help 或 foreign key。
  • created_by 只是技术记录字段，直接用 abap.char(12) 可以作为教学例子，但真实项目可能更适合复用用户相关 data element。
  • net_amount 是 amount field，必须结合 currency_code 解读；没有 currency，就不能可靠解释金额。

练习时不要只看字段名。请在 ADT 或 SAP Developers tutorial 里找一张练习表，至少追踪三个字段：一个 key field、一个 data element field、一个 amount 或 quantity field。

常见误区

  • 只看字段名，不看 data element/domain：会漏掉 field label、长度、decimals、fixed values、documentation 和 search help。
  • 把 domain 直接写成 table field type：会混淆 DDIC 分层并导致难懂的 activation error；字段应直接使用 built-in dictionary type 或 data element。
  • 用 abap.char(40) 存所有业务 ID：短期容易，长期会丢失语义、label、值域、搜索帮助和复用能力。
  • 把 not null 当成业务必填校验：它不能替代输入检查、RAP validation、domain fixed values 或应用逻辑。
  • 把 annotation 放错位置：@Semantics.amount.currencyCode / @Semantics.quantity.unitOfMeasure 要放在对应 amount / quantity 字段前面，不能随便放在表头或字段后面。
  • 金额字段没有 currency reference：后续报表、UI、接口和测试都无法确定金额单位。
  • 数量字段没有 unit reference：后续换算、显示和业务解释都会出错。
  • 随意修改 key field：可能影响已有数据、唯一性、锁对象、RAP key、migration 和所有 SELECT 条件。
  • 改短字段长度或改变 decimals 前不做 where-used：可能造成数据截断、转换错误和接口不兼容。
  • 在生产障害调查时只说“字段不对”：应该指出 table、field、data element/domain、当前值、期望值、发生 client 和影响对象。
