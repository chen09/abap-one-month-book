Chapter 03 - CDS、RAP、OData

11. CDS access control 访问控制

  • Chapter 03
  • Section 11
  • self-contained study note

11.1 一句话结论

CDS access control 访问控制，是 ABAP CDS 用 DCL（Data Control Language）在读取 CDS entity 时自动追加访问条件的一套机制。它的目标是让同一个 CDS view entity 在被 ABAP SQL、SADL、RAP/OData 等消费时，只返回当前用户有权看到的数据行。

这句话有三个边界必须一起记住。第一，CDS access control 主要控制 read access，也就是“能看到哪些数据”。第二，它依赖 CDS role / DCL source 中的 access rule，而不是只靠 @AccessControl.authorizationCheck: #CHECK 这个 annotation。第三，它不是完整 SAP 授权设计：能不能启动应用、能不能执行 action、能不能修改数据、后台技术用户怎样访问、审计怎样留痕，仍然要看 PFCG role、authorization object、AUTHORITY-CHECK、RAP authorization、service 暴露和运维控制。

11.2 到底学什么

本节先学“CDS access control 是什么”，下一节 DCL role 访问规则 再展开 DCL role 的具体写法。初学者不应该一上来背复杂语法，而应该先建立五个判断。

  1. 哪个 CDS entity 需要 read access control？
  2. 这个 entity 上的 @AccessControl.authorizationCheck 是 #CHECK、#MANDATORY、#NOT_REQUIRED 还是 #NOT_ALLOWED？
  3. 是否真的存在对应 DCL source / CDS role？
  4. DCL 规则是 literal condition、PFCG condition，还是继承其他 access condition？
  5. 当前消费路径是否会触发隐式 CDS access control？

SAP Learning 对 ABAP CDS authorization 的说明很适合初学者：DCL 用来以 modeled、declarative 的方式定义 CDS entities 所需授权；CDS role 用 GRANT SELECT ON ... WHERE ... 把 access condition 关联到 CDS view；访问 CDS view 时，结果会按 access condition 自动过滤。SAP-samples 的 authorization cheat sheet 也把它和 AUTHORITY-CHECK 分开：显式授权检查用 ABAP 语句，读 CDS entity 时可以用 CDS access control 做隐式读过滤。这里的“隐式”不是神秘行为，而是说读 CDS entity 的 ABAP SQL / SADL 路径会把 DCL 条件纳入取数；应用代码通常不需要再手写同一段 WHERE SalesOrg IN ...。

11.3 为什么重要

没有 CDS access control，很多项目会把权限过滤散落在 ABAP 代码、UI filter、OData handler 或前端逻辑里。这样做短期能跑，长期会出现几个典型风险：

  • 同一个 CDS view 被多个应用复用时，每个消费方都要自己补过滤，容易漏。
  • ABAP SQL 直接读表或读无 DCL 的 CDS view 时，默认不会替开发者自动做业务授权判断。
  • Fiori / OData 列表看起来隐藏了数据，但后台 job、接口或另一个报表可能仍然能读到。
  • 项目 review 时无法明确说明“授权条件在哪里生效、谁维护、如何测试”。
  • 审计时只能看到零散代码，不能从模型层解释数据访问边界。

CDS access control 的价值是把一部分 read authorization 前移到模型层。它让数据模型不仅说明“数据是什么”，还说明“当前用户读这个模型时应该看到哪些行”。这对 RAP/OData 尤其重要，因为服务、Fiori 页面、ABAP SQL 消费可能共享同一套 CDS 模型。

对 Java/Python 背景读者，可以把它粗略类比成 ORM query filter、row-level security 或 repository 层自动加 tenant / organization 条件。但 SAP 的差异在于：它是 ABAP repository object，依赖 CDS DCL、PFCG authorization object、current user、SADL/ABAP SQL 消费路径和系统授权配置。

11.4 Key mechanism / 核心机制

11.4.1 annotation 只是入口，不是规则本身

一个需要访问控制的 CDS view entity 常见写法如下：

@EndUserText.label: 'Sales order secured view'
@AccessControl.authorizationCheck: #CHECK
define view entity ZI_SalesOrder
  as select from zsales_order
{
  key sales_order_id as SalesOrderID,
      sales_org      as SalesOrg,
      customer_id    as CustomerID,
      gross_amount   as GrossAmount
}

@AccessControl.authorizationCheck: #CHECK 的意思不是“授权已经完成”，而是告诉语法检查和运行时：这个 entity 预期会参与 CDS access control。SAP-samples 对 annotation 选项的说明很清楚：

  • #NOT_REQUIRED：声明这个 entity 不需要 CDS-level access control；如果没有其它限制，所有用户可读到没有 DCL 限制的数据。它不是“更安全”的选项，而是明确说这里不靠 DCL 做行级限制。
  • #CHECK：期望存在 access control；如果没有 access control object，开发时会看到 warning。但没有 DCL role 时，它不会凭空产生限制，结果仍然不会被 CDS DCL 过滤。
  • #MANDATORY：必须存在 access control object。和 #CHECK 的 warning 不同，缺少 access control 时会成为更强的错误/激活阻断信号。对生产中需要行级限制的 CDS entity，通常比 #CHECK 更能暴露“忘记创建或移送 DCL”的问题。
  • #NOT_ALLOWED：不允许 access control；即使有 DCL 也会被忽略或导致相应检查问题。它适合明确不允许 DCL 的技术对象，而不是业务敏感数据的默认选择。

所以 review 时要问：这里写的是哪一个值？为什么？如果是 #CHECK 或 #MANDATORY，DCL role 在哪里？如果是 #MANDATORY，DCL 是否和 CDS entity 一起 activation、一起进入 transport request？如果是 #NOT_REQUIRED，是否确认这个 view 不需要行级限制？如果是 #NOT_ALLOWED，是否有清楚的技术理由？

11.4.2 DCL role 才定义访问规则

真正的规则写在 DCL source 中，典型形式是：

@EndUserText.label: 'Sales order access'
@MappingRole: true
define role ZR_SO_ACCESS {
  grant select on ZI_SalesOrder
    where ( SalesOrg ) =
      aspect pfcg_auth( ZSO_AUTH, VKORG, ACTVT = '03' );
}

这段意思是：读取 ZI_SalesOrder 时，把 CDS 字段 SalesOrg 和 authorization object ZSO_AUTH 中的授权字段 VKORG 关联起来，并要求 display activity ACTVT = '03'。当前用户在 PFCG role 中拥有哪些授权值，就会影响这个 CDS entity 返回哪些行。

@MappingRole: true 也要看懂：它表示这个 CDS role 会作为映射角色参与隐式访问控制评估，通常不需要像业务角色那样把 DCL role 单独分配给某个用户。真正决定用户能看到哪些值的，仍然是 DCL 条件和当前用户的 PFCG / IAM 授权上下文。

下一节 DCL role 访问规则 会专门讲 define role、grant select on、where、aspect pfcg_auth、literal condition、继承规则和 full access rule。本节只要先抓住：CDS access control = CDS entity 上的 access-control 意图 + DCL role 中的 access rule + 当前用户授权上下文。

11.4.3 它通常在读 CDS entity 时隐式生效

ABAP Keyword Documentation 说明，如果一个非 abstract CDS entity 有 CDS role / access rule，access condition 会在 ABAP SQL read 中隐式评估；只有满足条件的数据会被读取。SAP Learning 也强调，如果没有 DCL role，查询结果不会因为 #CHECK 自动变安全。这个过滤是设计特性：它让数据库层面少读无权限数据，既保护数据，也避免应用层再拿全量数据做二次过滤。

这意味着：

SELECT *
  FROM ZI_SalesOrder
  INTO TABLE @DATA(orders).

如果 ZI_SalesOrder 有生效的 DCL role，上面的 SELECT 会拿到被 DCL 限制后的结果。ABAP 程序通常不会知道“数据库里本来有 100 行，但因为 DCL 只返回了 40 行”。对程序来说，它只是读到了 40 行。这一点对业务错误处理很重要：CDS access control 不会自动告诉用户“你没有权限看另外 60 行”。

11.4.4 它和 projection view / RAP authorization 有边界

上一节 Projection view 投影视图 里，projection example 使用了 @AccessControl.authorizationCheck: #NOT_REQUIRED，原因是 projection 层通常继承底层 interface/root view 的 DCL，不应该让读者误以为服务裁剪层要重新定义一套行级授权。

但这不是说 projection、RAP、service 层不用管授权。它们要管的是不同问题：

  • CDS access control / DCL：当前用户读取这个 CDS entity 时能看到哪些行。
  • Projection view：服务视角要不要暴露某些字段或 navigation。
  • Service definition / binding：哪些 projection entity 被服务暴露，用什么协议消费。
  • RAP authorization：当前用户能否执行 create/update/delete/action、是否通过 global / instance authorization。
  • Classic authorization：能否启动应用、进入事务、执行报表、使用后台 job、访问技术功能。

项目上不能把这些混成一句“权限已经做了”。正确说法应该是：DCL 负责某个 read model 的行级读过滤；RAP behavior / authorization 负责业务对象操作权限；PFCG / IAM / role design 负责用户授权分配；service 层负责外部契约暴露。

11.4.5 还有绕过和失效场景

CDS access control 很有用，但不是魔法。必须知道几个边界：

  • 如果没有 CDS role / access rule，#CHECK 只会帮助开发时发现 warning，不会自动生成规则。
  • 如果使用 #NOT_ALLOWED，access control 会被禁用。
  • ABAP SQL 可以在特定场景用 WITH PRIVILEGED ACCESS 绕过当前直接读取对象上的 CDS access control；这应该非常谨慎，并有 release/context、审计和设计理由。不要把它当成普通报表“看全量”的开关，也不要假设它会自动绕过所有 association target 上的访问控制。
  • CDS access control 主要在“被直接访问的 CDS entity”入口生效。一个 CDS entity 作为另一个 CDS entity 的数据源时，不能简单假设所有底层 DCL 都按预期自动传递到最外层服务；需要用 inherited condition 或在外层 entity 上定义清楚的 DCL。
  • 一个 CDS entity 可以有多个 DCL role，实际权限可能来自多个规则的组合。初学者不要假设“一个 view 只能有一个 DCL 文件”。
  • DCL source 本身也是 ABAP Repository object，移送时要和被保护的 CDS entity、authorization object / role design 一起检查。尤其使用 #MANDATORY 时，如果只移送 CDS entity 而漏掉 DCL，生产环境可能出现全员读不到数据的事故。
  • Cross-client access 与 CDS access control 有特殊限制，不能把 DCL 当成跨 client 数据隔离方案。

这些边界是 review 的重点。安全设计不是“写了 DCL 就结束”，而是确认它在哪条消费路径、哪个 user context、哪个 service contract 中生效。

11.7 Example / 学习例子

11.例 1：销售组织行级过滤

CDS view entity：

@EndUserText.label: 'Sales order secured view'
@AccessControl.authorizationCheck: #CHECK
define view entity ZI_SalesOrder
  as select from zsales_order
{
  key sales_order_id as SalesOrderID,
      sales_org      as SalesOrg,
      customer_id    as CustomerID,
      status         as Status
}

DCL role：

@EndUserText.label: 'Sales order access by sales org'
@MappingRole: true
define role ZR_SO_ACCESS {
  grant select on ZI_SalesOrder
    where ( SalesOrg ) =
      aspect pfcg_auth( ZSO_AUTH, VKORG, ACTVT = '03' );
}

读法：

  • ZI_SalesOrder 是被保护的 CDS entity。
  • ZR_SO_ACCESS 是 DCL role。
  • @MappingRole: true 表示这个 DCL role 参与隐式访问控制评估。
  • grant select on 说明这是读访问规则。
  • SalesOrg 是 CDS 字段。
  • ZSO_AUTH 是 authorization object。
  • VKORG 是 authorization field。
  • ACTVT = '03' 表示 display authorization。

如果用户只被授权 VKORG = 1000，读取这个 CDS view 时只应该看到 SalesOrg = 1000 的订单。

11.例 2：DCL 不等于修改权限

假设用户能通过 DCL 看到某些订单，不代表他能修改这些订单：

能看到订单行：由 CDS access control / DCL 控制。
能修改订单：由 RAP update authorization、behavior definition、业务规则、PFCG/IAM 等共同控制。
能执行 cancel action：由 RAP action authorization / instance authorization 控制。
能打开应用：由 launchpad / business catalog / role / start authorization 控制。

这就是为什么项目文档不能只写“已做 DCL”。更好的写法是：read access is controlled by DCL; modify/action authorization is controlled by RAP behavior authorization and role design。

11.例 3：排查数据少了

如果用户说“列表少了数据”，不要马上认为 SQL 错了。排查顺序可以是：

  1. 用相同条件确认基础表或底层 view 是否有数据。
  2. 检查 CDS entity 是否有 @AccessControl.authorizationCheck。
  3. 检查是否存在 DCL role。
  4. 检查 DCL 的 where 条件是否按预期过滤。
  5. 检查用户 PFCG role / business role 是否有对应授权值。
  6. 检查是否通过 projection / service / OData 或另一个 CDS entity 间接消费，外层 entity 是否也有必要的 DCL / inherited condition。
  7. 检查 DCL source 是否随 CDS entity 一起移送和激活，特别是 #MANDATORY 场景。
  8. 不要用 WITH PRIVILEGED ACCESS 作为日常修复；它只是排查或特殊技术场景的工具。

11.8 Counterexample / Failure mode

11.误区 1：以为 #CHECK 等于完成授权

#CHECK 只是检查策略和开发提示。没有 DCL role，就没有具体访问条件。项目上看到 #CHECK，必须继续找 DCL source。

11.误区 2：把 DCL 当成所有权限

DCL 控制 read result。它不控制按钮是否显示、不控制 action 是否能执行、不控制 update/delete 是否允许、不控制用户是否能启动应用。完整授权要结合 RAP authorization、PFCG/IAM、service、UI 和 audit。

11.误区 3：在 projection 层重复做底层 DCL

Projection view 主要是服务裁剪层。多数 RAP 场景下，read access 的核心 DCL 放在底层 interface/root view 更清晰。Projection 层要确认继承和服务暴露，不要随手复制一套不同的 DCL 逻辑。

11.误区 4：忘记用户看不到数据也可能是“被过滤”

ABAP 程序通常无法区分“数据不存在”和“被 DCL 过滤掉”。用户反馈数据缺失时，要把 DCL 纳入调查。

11.误区 5：滥用 privileged access

WITH PRIVILEGED ACCESS 会绕过 CDS access control。它不能作为普通报表“看全量数据”的捷径。需要设计理由、权限限制和审计说明。

11.误区 6：只保护底层 view，忘记外层 view

如果外层 CDS entity 直接被 ABAP SQL、SADL 或 OData 服务消费，只在底层 view 上写 DCL 未必足够。项目上要确认实际入口对象是否也有 access control 或 inherited condition，否则复用链路可能绕过原本以为存在的限制。

11.误区 7：漏移送 DCL source

DCL role 不是注释文字，而是独立的 repository object。CDS entity、DCL source、authorization object、PFCG/business role 的移送和配置顺序要一起检查。否则开发系统看起来正常，测试或生产系统可能因为缺 DCL 或缺授权值而读不到数据。

11.9 思考问题

  • 这个 CDS entity 是否需要行级读访问控制？
  • @AccessControl.authorizationCheck 的值为什么是当前选择？
  • 是否存在对应 DCL role？DCL role 是否已 activation？
  • DCL 条件是 literal、PFCG condition，还是继承？
  • PFCG role / IAM role 是否给用户分配了正确 authorization object value？
  • 当前消费路径是直接 ABAP SQL、SADL、RAP/OData，还是另一个 CDS view 间接消费？
  • 如果是 view-on-view，外层是否通过 DCL inheritance 或自己的 access condition 保留了授权语义？
  • 如果用户看不到数据，如何区分数据不存在和被 DCL 过滤？
  • DCL source 是否和 CDS entity 同步移送、同步 activation？
  • 哪些权限不应该交给 DCL，而应该交给 RAP authorization、PFCG、service 或 UI？
