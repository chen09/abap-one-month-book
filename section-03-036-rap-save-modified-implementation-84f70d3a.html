36. RAP save_modified implementation

  • Chapter 03
  • Section 36
  • self-contained study note

36.1 一句话结论

RAP save_modified implementation 是 RAP saver class 中最容易被误用的方法之一：在 with unmanaged save 中，save_modified 负责替代 managed runtime 的主保存；在 with additional save 中，save_modified 只负责在 managed runtime 主保存之外追加附加逻辑。方法名相同，职责完全不同，必须先看 BDEF 的保存选项再读代码。

一句话记忆：save_modified 不是“保存时随便写点代码”的入口，而是由 BDEF 保存策略决定语义的 save sequence 方法。

36.2 概念说明

save_modified 出现在 RAP behavior pool 的 local saver class 中，属于 Save sequence 保存序列。它不是 handler method，也不是 action method。只有在特定保存选项下，runtime 才会调用它。

第一种常见语境是 Managed Business Object with Unmanaged Save。BDEF 中声明 with unmanaged save 后，RAP managed runtime 仍负责 interaction phase、标准 operation buffer、authorization、validation 等很多框架工作，但不再执行默认 managed write。进入 save sequence 后，runtime 把本次 create/update/delete 的 modified content 交给 save_modified，由项目负责主保存。

第二种常见语境是 Additional Save 附加保存。BDEF 中声明 with additional save 后，RAP managed runtime 仍执行标准 managed write；save_modified 只是在 managed write access 之外追加逻辑，例如写审计表、change document、扩展表、business event/outbox 记录。它不是主保存的替代。

因此，读 save_modified 的第一步不是看 ABAP 代码，而是看 BDEF：

managed implementation in class zbp_i_travel unique;
with unmanaged save;

或：

managed implementation in class zbp_i_travel unique;
with additional save;

这两段都可能让你在 saver class 里看到 save_modified，但 review 关注点完全不同。前者要确认主数据是否完整保存；后者要确认主数据没有被重复保存，附加逻辑是否幂等、可回滚、可解释。

36.3 为什么重要

save_modified 一旦写错，往往不是小 bug，而是保存语义错位。最常见的事故有两类：把 unmanaged save 当成 additional save，导致主业务数据根本没保存；把 additional save 当成 unmanaged save，导致主数据被 runtime 和项目代码重复保存。

它也常常是旧系统迁移到 RAP 的关键点。项目想保留既有保存 API、BAPI、function module、多个 Z 表写入、复杂 mapping，就会考虑 managed with unmanaged save；项目想在标准 managed save 之外写审计、事件、扩展字段，就会考虑 additional save。两者都可能进 save_modified，但风险模型不同。

对项目交付来说，save_modified 影响测试边界。测试不能只看 UI “保存成功”。要确认 create/update/delete 是否都保存、delete 是否真的删除、字段变更是否尊重 %control、旧 API 是否内部 commit、失败时是否 rollback、附加日志是否重复、cleanup 后第二次请求是否干净。

对 Java/Python 背景学习者，可以把 save_modified 类比成框架 unit-of-work 的 flush hook，但 RAP 中这个 hook 的职责不是固定的。它由 BDEF 的 with unmanaged save 或 with additional save 决定，是替代主保存还是追加副保存。

36.4 边界

第一条边界：save_modified 不属于 interaction phase。不要在 action handler 里调用它，也不要从普通 ABAP 程序 new saver class 调它。它由 RAP runtime 在 save sequence 中调用。

第二条边界：save_modified 不等于 fully unmanaged BO 的 save()。Fully unmanaged BO 通常由项目实现 save() 保存 application buffer；managed with unmanaged save / additional save 常见入口是 save_modified()。不要把两个 saver method 混用。

第三条边界：with unmanaged save 中的 save_modified 是主保存。主数据不在这里完成，就不完整。不要以为 managed runtime 还会照常写 persistent table。

第四条边界：with additional save 中的 save_modified 是附加保存。主数据仍由 managed runtime 保存。不要在这里重复写主业务表。

第五条边界：save_modified 中不能随意 COMMIT WORK / ROLLBACK WORK。RAP runtime 仍是事务边界 owner；旧 API 如果内部 commit，必须重构、寻找 no-commit 版本，或重新评估设计。

第六条边界：可预见业务错误不要拖到 save_modified。金额不合法、主数据不存在、状态不允许、权限不足，应尽量在 validation、authorization、precheck 或 check-before-save 处理。save_modified 更适合处理最终持久化、legacy API 技术错误和附加保存。

36.5 Key mechanism / 核心机制

第一层机制是 saver class。save_modified 写在 local saver class 中，通常继承 CL_ABAP_BEHAVIOR_SAVER。读 behavior pool 时，先找 lsc_... 类，而不是 lhc_... handler 类。

第二层机制是 importing data。save_modified 接收本次 save 中发生变化的数据。具体签名随 BDEF、entity 和 release 示例不同，但通常会区分 create、update、delete 或 change/delete request。读者要关注“本次事务发生了哪些变更”，不是重新 SELECT 全数据库。

第三层机制是 BDEF-derived table types。在 managed with unmanaged save 示例中，常见理解方式是 TYPE TABLE FOR CREATE <entity>、TYPE TABLE FOR UPDATE <entity>、TYPE TABLE FOR DELETE <entity> 这类 operation-specific 派生类型。在 additional save 示例中，也可能看到 TYPE REQUEST FOR CHANGE / TYPE REQUEST FOR DELETE 等变化请求结构。不要把所有示例机械套成一个固定签名。

第四层机制是 %control。Update 场景必须知道哪些字段是 consumer 本次请求修改的。若把未修改字段当成初始值写给 legacy API，就会把旧数据清空。保存旧接口时，要把 %control 或 changed fields 语义翻译成旧 API 能理解的更新结构。

第五层机制是 create/update/delete 三类责任。主保存场景中，save_modified 至少要回答：

  • create：新实例写到哪里？key 和 mapping 是否完整？
  • update：只更新哪些字段？并发和 ETag 如何处理？
  • delete：是物理删除、逻辑删除，还是调用旧 API 的取消？

第六层机制是错误处理。save_modified 已处在保存阶段，错误消息和回滚语义比 interaction phase 更敏感。能提前校验的规则应前移；这里剩下的错误要尽量是持久化或 legacy API 层不可避免的错误，并以 RAP message / exception 语义返回。

第七层机制是幂等。用户可能重复点击保存，OData consumer 可能重试，additional save 可能写日志或事件。save_modified 中的附加记录、事件、旧接口调用都要考虑重复执行风险，必要时使用唯一键、业务 key、outbox 或 processed flag。

第八层机制是测试。save_modified 的测试至少覆盖：

  • create/update/delete 混合请求。
  • update 只改一个字段时不会覆盖其他字段。
  • delete 请求真的进入保存逻辑。
  • validation 失败时 save_modified 不应写主数据。
  • legacy API 抛错时整体保存失败且无部分提交。
  • additional save 重试不产生不可接受重复日志。
  • 旧 API 内部没有 COMMIT WORK。
save_modified review checklist

评审 save_modified 时，第一步永远是回到 BDEF。不要先看方法体。确认当前 BO 是 with unmanaged save 还是 with additional save：前者代表 save_modified 承担主保存，后者代表 managed runtime 已经负责主保存，save_modified 只做追加保存。这个判断错了，后面的 code review 都会偏。

第二步确认 modified content 的覆盖范围。Create、update、delete 是否都被处理？Child entity 是否被处理？Composition tree 中 parent 和 child 的保存顺序是否可解释？如果只处理了 update，没有处理 delete，UI 删除看似成功但旧表残留，就是典型缺陷。

第三步确认 update 语义。save_modified 不能把未修改字段当成“要清空”。特别是调用旧 BAPI / function module / Z API 时，要把 %control 或 changed-fields 语义翻译成旧接口的 update indicator。否则 consumer 只改一个状态字段，旧 API 却收到一堆 initial value，把其他字段覆盖掉。

第四步确认旧 API 的事务行为。任何被 save_modified 调用的 function module、class method、BAPI wrapper，都要确认是否内部 COMMIT WORK、ROLLBACK WORK、update task、background task 或外部不可回滚副作用。若旧 API 内部提交，RAP runtime 的回滚语义会被破坏，必须寻找 no-commit 版本或重新设计边界。

第五步确认错误前移。主数据不存在、金额非法、状态不允许、权限不足、字段格式错误这类可预见业务错误，不应该等到 save_modified 调旧 API 后才失败。能放 validation / authorization / precheck / check_before_save 的规则应前移，save_modified 里只留下真正保存层或遗留接口层才会暴露的问题。

第六步确认幂等和重复请求。Additional save 写日志、outbox、审计表、事件记录时，必须说明重复保存或 API retry 是否会重复写。Managed with unmanaged save 调旧保存 API 时，也要确认同一 request 重试是否会创建重复业务对象。必要时使用 request id、业务 key、唯一约束或状态机保护。

第七步确认测试证据。最低限度应有：create 保存成功、update 单字段不覆盖其他字段、delete 生效、validation 失败时不进入主保存、legacy API 抛错时整体失败、additional save 不重复写、旧 API 无内部 commit 证据。没有这些证据，不应把 save_modified 视为生产就绪。

36.6 日本项目现场说法

常见日语：

  • save_modified メソッド：save_modified 方法。
  • アンマネージド保存：unmanaged save。
  • 追加保存：additional save。
  • 主保存：主保存。
  • 追加処理：附加处理。
  • 保存処理の差し替え：替换保存处理。
  • 既存ロジックの再利用：复用既有逻辑。
  • コミット制御：commit 控制。
  • 二重登録：重复登记。

说明句：

with unmanaged save の save_modified は主保存を担当します。

意思是：with unmanaged save 中的 save_modified 负责主保存。

with additional save の save_modified は標準保存に対する追加処理であり、主保存を差し替えるものではありません。

意思是：with additional save 中的 save_modified 是对标准保存的附加处理，不是替换主保存。

评审句：

この save_modified は主保存なのか、追加保存なのかを BDEF の保存オプションで確認してください。

意思是：请根据 BDEF 的保存选项确认这个 save_modified 是主保存还是附加保存。

36.8 Example / 学习例子

例子一：managed BO with unmanaged save。

项目已有一个旧保存 API zcl_legacy_travel_save=>save( )，它一次性保存 Travel header、items、附件索引和若干 Z 表。标准 managed save 只能写一张 persistent table 或简单 mapping，不够表达旧逻辑。团队选择 with unmanaged save，让 RAP managed runtime 负责 interaction phase，进入 save sequence 后由 save_modified 把 create/update/delete 转换成旧 API 输入并完成主保存。

关键检查：

  • BDEF 是否没有错误配置 persistent table 自动保存。
  • save_modified 是否处理 create、update、delete。
  • 旧 API 是否无内部 commit。
  • %control 是否正确翻译。
  • 保存失败是否 rollback 且消息可理解。

例子二：additional save 写审计日志。

Travel 主数据完全可以由 managed runtime 保存，但保存成功时还要写审计表或 raise business event。团队使用 with additional save。此时 save_modified 不写主 Travel 表，只根据本次 modified content 写审计记录或事件 outbox。

关键检查：

  • 主数据是否仍由 runtime 保存。
  • 附加记录是否幂等。
  • 附加逻辑失败时是否应让整体保存失败。
  • 是否避免同步调用不可回滚外部系统。

例子三：AC0567C POC 迁移思考。

如果批量上传 POC 未来迁移成 RAP BO，而保存逻辑必须复用既有批量 API，可以考虑 managed with unmanaged save；如果主数据走标准 managed save，只需要额外写上传日志、行级审计、事件通知，则更像 additional save。关键不是“都叫 save_modified”，而是主保存责任在哪里。

36.9 Counterexample / Failure mode

反例一：把 additional save 当主保存。BDEF 写 with additional save，但开发者以为 runtime 不会写主数据，于是在 save_modified 里又写主业务表。结果可能重复写、锁冲突或覆盖 runtime 保存。

反例二：把 unmanaged save 当 additional save。BDEF 写 with unmanaged save，但 save_modified 只写日志，没有写主数据。结果 UI 显示保存流程走完，主业务数据却没落库。

反例三：旧 API 内部 commit。save_modified 调用 legacy function module，function module 内部 COMMIT WORK。后续 RAP 保存失败时，旧 API 已经提交。修正方式是 no-commit API、重构包装层或改变设计。

反例四：update 不看 %control。前端只改了状态，save_modified 把其他字段的初始值也传给旧 API，导致数据被清空。修正方式是保留 changed-fields 语义。

反例五：delete 被漏掉。代码只处理 create/update，忽略 delete importing table。用户在 UI 删除了 child item，保存后旧表仍有残留。修正方式是明确 delete 语义。

反例六：业务校验太晚。客户主数据不存在，本来 validation 可以提前报字段级错误，却等到 save_modified 调旧 API 才失败。修正方式是把可预见业务规则前移。

36.10 思考问题

  • BDEF 写的是 with unmanaged save 还是 with additional save？
  • 当前 save_modified 是主保存，还是附加保存？
  • create/update/delete 三类 modified content 是否都处理了？
  • update 是否尊重 %control 或 changed fields？
  • 是否调用了内部 commit 的旧 API？
  • 可预见业务错误是否已经前移到 validation / authorization / precheck？
  • additional save 里的附加记录是否幂等？
