Chapter 02 - Core ABAP Programming

43. sy-subrc discipline and silent failure patterns

  • Chapter 02
  • Section 43
  • self-contained study note

43.1 一句话结论

sy-subrc 是 ABAP system field 中最常见的返回码字段，用来表达“上一条会设置返回码的 ABAP 语句”执行后的语句级结果。所谓 sy-subrc discipline，不是要求到处写 IF sy-subrc = 0，而是要求先确认该语句是否真的设置 sy-subrc，再在 READ TABLE、SELECT、动态 ASSIGN、AUTHORITY-CHECK、经典 function module 调用等语句之后，立刻、按该语句的文档语义检查结果；否则程序可能不会 dump，却会沿着错误数据、旧数据、未授权路径或未处理的空结果继续运行，形成 silent failure。

43.2 到底学什么

本节接在 ABAP system fields sy fields、READ TABLE 读取内部表 和 Table expression 表表达式 之后。前面已经知道 sy-* 是 ABAP runtime 提供的一组系统字段，也知道 internal table 既可以用 READ TABLE，也可以用 itab[ ... ] 读取。本节要补上的，是项目里真正会被 code review 追问的一层：哪一类语句需要看 sy-subrc，什么时候要马上看，什么时候应该改用 table expression、line_exists、异常类或明确的返回对象。

先看一个经典例子：

READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY item_id = lv_item_id.

IF sy-subrc = 0.
  process_item( ls_item ).
ENDIF.

这里 sy-subrc = 0 的意思不是“程序没有错误”，而是“这次 READ TABLE 按当前写法找到了匹配行，并且该语句对应的成功条件成立”。如果 READ TABLE 之后先执行了另一个会设置 sy-subrc 的语句，再回来检查，检查到的就不一定是这次读取的结果。更稳的写法是把返回码立即判断，或者在必须隔开时保存下来：

READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY item_id = lv_item_id.

DATA(read_subrc) = sy-subrc.

IF read_subrc <> 0.
  RETURN.
ENDIF.

process_item( ls_item ).

本节的学习目标有四个：

  • 理解 sy-subrc 是 statement-specific return code，不是统一异常对象。
  • 记住 sy-subrc 要在相关语句之后立即处理，必要时保存到本地变量。
  • 能区分经典返回码风格、现代 table expression 风格和 class-based exception 风格。
  • 能在日本 SAP 项目 review 中说明“未检查戻りコード”为什么会造成 silent failure。

43.3 为什么重要

sy-subrc 的危险之处在于：很多错误不会立刻变成 runtime error。比如 READ TABLE 没找到行，程序却继续使用 ls_item；SELECT SINGLE 没有读到数据，后续逻辑却把目标变量当成有效业务数据；AUTHORITY-CHECK 没检查返回码，未授权用户可能继续进入处理；ASSIGN 没检查是否成功，后面使用 field symbol 时可能短 dump，也可能在某些动态写法中误用旧状态。

这类问题和 Java/Python 的异常不同。Java/Python 初学者习惯用 exception、None、Optional、返回对象表达失败；ABAP 的经典语句大量使用 sy-subrc 表达“语句执行后的技术状态”。它比 exception 更轻，但也更容易被忽略。项目代码里最糟糕的不是“失败时 dump”，而是“失败时继续跑，还看起来像成功了”。这就是 silent failure。

在 SAP 项目交付中，sy-subrc discipline 影响四件事：

  • 正确性：没有找到业务对象时，不能继续按“找到了”处理。
  • 安全性：权限检查失败不能被当成普通空结果。
  • 可维护性：reviewer 必须能看到失败分支的业务意图。
  • 可测试性：ABAP Unit 要能覆盖 found / not found / unauthorized / exception path。

43.4 Key mechanism / 核心机制

43.4.1 sy-subrc 是语句级返回码

sy-subrc 属于系统字段 sy 结构。它由特定 ABAP 语句设置，具体值必须按该语句的 ABAP Keyword Documentation 解释。常见约定是 0 表示成功或条件满足，非 0 表示失败、未找到、未完全处理或某种语句特定状态；但不能把所有非 0 都解释成同一种错误，也不能假设每一条 ABAP 语句都会刷新 sy-subrc。如果文档没有说明该语句设置返回码，就不要用残留的 sy-subrc 猜测执行结果。

READ TABLE 是最适合初学者理解的例子。常见情况下，0 表示找到匹配行，4 表示未找到；如果使用 COMPARING，2 可以表示“找到了行，但比较字段不同”；在某些读取变体中，sy-subrc = 8 与未找到相关，但 sy-tabix 的设置方式不同；如果使用 WHERE 条件，常见语义又会收敛到找到/未找到。旧式 BINARY SEARCH 还要求 standard table 已按搜索 key 正确排序，排序不满足时不能指望 sy-subrc 帮你识别错误。项目里不能只背“0 成功、4 失败”，而要知道当前语句和 additions 的含义。

43.4.2 必须在正确位置检查

返回码最常见的 bug，是检查位置错了：

READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY item_id = lv_item_id.

helper->write_trace( lv_item_id ).

IF sy-subrc = 0.
  process_item( ls_item ).
ENDIF.

这段代码的问题不在于 trace 一定会改写 sy-subrc，而在于读者已经无法一眼确认 IF sy-subrc = 0 检查的是哪条语句。一个普通方法调用、数据库语句、table 操作或动态处理都可能让返回码语义变得不可追踪。ABAP Keyword Documentation 还特别提醒：functional method call 成功时会把 sy-subrc 设为 0，NEW 创建实例也有类似影响。因此，在 READ TABLE 和检查之间插入一个看似无害的 helper method，也可能把原本的 not-found 返回码覆盖掉。项目标准里通常会要求“直後に確認”：在需要的语句之后立刻判断，或者保存为本地变量，并给变量起一个能说明来源的名字。

更清楚的写法是：

READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY item_id = lv_item_id.

IF sy-subrc <> 0.
  RETURN.
ENDIF.

helper->write_trace( lv_item_id ).
process_item( ls_item ).

如果确实要延迟判断：

READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY item_id = lv_item_id.

DATA(item_read_subrc) = sy-subrc.

helper->write_trace( lv_item_id ).

IF item_read_subrc = 0.
  process_item( ls_item ).
ENDIF.
43.4.3 READ TABLE 后不要信任目标变量

经典错误是：

READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY item_id = lv_item_id.

process_item( ls_item ).

读者看不出 ls_item 是否真的来自当前 key。安全的心智模型是：只有在 sy-subrc = 0 的分支里，才把 READ TABLE ... INTO 的目标变量当成“本次读取成功得到的数据”。如果未找到是正常业务场景，就写清楚 fallback；如果未找到是异常业务场景，就抛出 exception 或返回明确错误。

READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY item_id = lv_item_id.

IF sy-subrc <> 0.
  RAISE EXCEPTION TYPE zcx_item_not_found
    EXPORTING item_id = lv_item_id.
ENDIF.

process_item( ls_item ).
43.4.4 SELECT 与 sy-dbcnt

ABAP SQL 语句也会设置系统字段。对 SELECT ... INTO TABLE，常见判断是：sy-subrc = 0 表示有结果传入目标对象，sy-subrc = 4 表示没有读到行；sy-dbcnt 表示处理行数。写书时要避免一句“SELECT 成功了”带过，因为空结果在业务上可能是正常、警告或错误，必须按场景说明。

SELECT FROM zsales_order
       FIELDS sales_order_id, customer_id, status
       WHERE customer_id = @lv_customer_id
       INTO TABLE @DATA(lt_orders).

IF sy-subrc <> 0.
  " No rows found. Decide whether this is normal or an error.
  RETURN.
ENDIF.

DATA(selected_rows) = sy-dbcnt.

这里还要提醒初学者：SELECT 本身不是权限设计，也不是业务存在性的全部证明。读取不到数据可能是条件不匹配、权限/DCL 过滤、client、数据状态、测试数据缺失等原因；读到了数据也不代表后续 action 一定被授权。权限要连接 AUTHORITY-CHECK 经典授权检查、CDS Access Control DCL 行级访问控制 和 RAP global authorization 全局授权。

SELECT SINGLE 还要多一层警惕：如果 WHERE 条件不是唯一业务 key 或完整主 key，读到一行并设置 sy-subrc = 0 并不等于“唯一且稳定地找到了那条业务记录”。它只说明当前语句传回了一行。需要“最新一条”“最大一条”“唯一一条”时，应使用明确的 key、ORDER BY、聚合或业务约束，而不是把 SELECT SINGLE 当成稳定选择器。

ABAP SQL 写入语句也类似。INSERT、UPDATE、MODIFY、DELETE 会设置 sy-subrc / sy-dbcnt，sy-dbcnt 可以辅助判断影响行数或部分处理，但不能替代返回码语义。比如使用 ACCEPTING DUPLICATE KEYS 时，重复 key 可能不会抛出异常，而是通过返回码和处理行数体现；如果完全不看这些字段，写入失败或部分写入也会变成 silent failure。

43.4.5 ASSIGN 与 field symbol

这里必须非常精确。不是所有 ASSIGN 都用 sy-subrc 判断。静态形式：

ASSIGN ls_item TO FIELD-SYMBOL(<ls_item>).

这个 static ASSIGN / 静态 ASSIGN 不要写成：

IF sy-subrc = 0.
  ...
ENDIF.

这种静态 ASSIGN 不是通过 sy-subrc 表达成功与否；更合适的保护方式是使用 <ls_item> IS ASSIGNED，尤其是在 field symbol 可能来自不同代码路径时。

动态 ASSIGN、table expression 赋值给 field symbol 时，才需要结合具体语句语义看 sy-subrc：

ASSIGN lt_items[ item_id = lv_item_id ]
  TO FIELD-SYMBOL(<ls_item>).

IF sy-subrc = 0.
  <ls_item>-status = 'X'.
ENDIF.

这里的重点是：普通 table expression 找不到行时通常走 CX_SY_ITAB_LINE_NOT_FOUND，但 ASSIGN itab[ ... ] TO <fs> 是重要例外，可以用 sy-subrc 判断是否赋值成功。动态 ASSIGN 也可以通过 sy-subrc 表达赋值失败。除此之外，field symbol 的 assigned 状态仍要明确处理。复杂场景下可以搭配 ELSE UNASSIGN、UNASSIGN 或 <fs> IS ASSIGNED，让读者看得出失败时不会误用旧绑定。

43.4.6 AUTHORITY-CHECK 必须检查

权限检查是 sy-subrc discipline 的安全分水岭：

AUTHORITY-CHECK OBJECT 'Z_SALES'
  ID 'ACTVT' FIELD '03'
  ID 'VKORG' FIELD lv_sales_org.

IF sy-subrc <> 0.
  RAISE EXCEPTION TYPE zcx_not_authorized.
ENDIF.

这里不要写成“先检查一下权限，失败也继续跑”。AUTHORITY-CHECK 的非 0 返回值有具体语义，初学阶段不需要死背所有值，但必须知道：0 才是授权通过；非 0 必须进入拒绝、错误消息、日志或上层错误转换。未检查的 AUTHORITY-CHECK 不是普通 code smell，而是安全缺陷：代码表面上做了权限检查，实际上失败后仍然放行。日本项目里这常被 review 为 権限チェック後の戻りコード未確認 或 AUTHORITY-CHECK後のsy-subrcチェック漏れ。

43.4.7 经典 function module 与现代 exception

经典 function module 常见写法是：

CALL FUNCTION 'Z_READ_ITEM'
  EXPORTING
    iv_item_id = lv_item_id
  IMPORTING
    es_item    = ls_item
  EXCEPTIONS
    not_found  = 1
    OTHERS     = 2.

IF sy-subrc <> 0.
  RAISE EXCEPTION TYPE zcx_item_read_failed.
ENDIF.

在这种风格里，sy-subrc 和 EXCEPTIONS 的编号映射有关。OTHERS = n 会把未单独列出的异常折叠成同一个返回码；如果后面不检查 sy-subrc，就等于把 function module 的失败吞掉了。某些 function module 还会通过 error_message 或经典 exception 机制把消息类错误映射到返回码。维护 legacy ABAP 时，CALL FUNCTION ... EXCEPTIONS 后面的检查必须写清楚。

现代 class-based exception 则不同，TRY...CATCH 处理的是 exception object，不应强行把所有失败都压回 sy-subrc。也就是说，本节不是鼓励到处使用返回码，而是要求在遇到返回码式 API 时，把它处理干净；在新设计的方法里，优先考虑清晰的返回值、异常类或 RAP 的 failed / reported 结构。

43.4.8 table expression 不是 sy-subrc 写法

对 internal table 的现代读取，很多场景可以不用 sy-subrc：

IF line_exists( lt_items[ item_id = lv_item_id ] ).
  " exists
ENDIF.

或者：

DATA(ls_item) = VALUE ty_item(
  lt_items[ item_id = lv_item_id ] OPTIONAL ).

这类写法要小心两点。第一，普通 table expression 找不到行时不是设置 sy-subrc，而是抛出 CX_SY_ITAB_LINE_NOT_FOUND，除非使用 OPTIONAL、DEFAULT、line_exists 等显式处理。第二，OPTIONAL 会把 not-found 变成 initial value，不抛异常，也不设置返回码；如果 initial value 和真实业务值无法区分，就可能把 silent failure 换成另一种 silent failure。能用，但要有业务意图。

43.4.9 return-code handling checklist

Review 返回码式 ABAP 时，可以用下面的表快速判断是否存在 silent failure 风险：

Statement / APIWhat to check immediatelyFailure should become...
READ TABLEsy-subrc、目标变量是否来自本次读取。not-found 分支、line_exists、OPTIONAL with explicit intent，或 exception。
SELECT ... INTOsy-subrc、sy-dbcnt、目标变量是否保留旧值。empty result handling、message、或业务允许为空的明确分支。
ASSIGN ... TO <fs>static ASSIGN 用 <fs> IS ASSIGNED；dynamic ASSIGN / ASSIGN itab[ ... ] TO <fs> 结合 sy-subrc 与 assigned state。明确跳过、raise exception、UNASSIGN 或 fallback value。
AUTHORITY-CHECKsy-subrc = 0 才允许继续。authorization error message、RAP authorization failure、或安全日志证据。
CALL FUNCTION ... EXCEPTIONSsy-subrc 与 exception number mapping。转换为 class-based exception、return object、message 或终止流程。
table expression是否使用 OPTIONAL / DEFAULT / line_exists / TRY...CATCH。不要误以为它会设置 sy-subrc。

最重要的 review 问题是：失败路径有没有被命名？如果代码只是“找不到就继续、没权限也继续、ASSIGN 失败也继续”，那不是宽容设计，而是 silent failure。返回码检查后的动作应当让维护者看懂：继续、跳过、报错、记录、重试，还是把失败交给上层。

团队层面不要只靠人工记忆。ATC / Extended Program Check 可以帮助发现未检查返回码、在不设置 sy-subrc 的语句后检查返回码、以及一些容易误导维护者的写法。工具不能替代业务判断，但它能把“忘了检查”这类低级错误提前暴露出来。

43.5 日本项目现场说法

常见日语：

  • 戻りコード
  • リターンコード
  • sy-subrc確認
  • 直後に確認
  • 戻りコード未確認
  • 戻りコード確認漏れ
  • sy-subrcチェック漏れ
  • 未チェック
  • サイレント失敗
  • 暗黙的な失敗
  • READ TABLE後の判定
  • SELECT後の判定
  • ASSIGN後の判定
  • AUTHORITY-CHECK後の判定
  • 権限チェック
  • 存在チェック
  • 例外処理

自然说明句：

  • READ TABLEの直後にsy-subrcを確認しないと、未取得のワークエリアを後続処理で使用する可能性があります。
  • AUTHORITY-CHECK後の戻りコード未確認は、権限制御上の重大な不備になります。
  • AUTHORITY-CHECK後のsy-subrcチェック漏れは、コード品質ではなくセキュリティ上の脆弱性として扱います。
  • このケースではsy-subrcではなく、line_existsまたは例外処理で未存在ケースを明示した方が読みやすいです。
  • sy-subrcを確認する前に別処理が入っているため、戻りコードをローカル変数に退避してください。
  • メソッド呼び出しでsy-subrcが上書きされる可能性があるため、戻りコードは設定した文の直後で確認してください。
  • 静的なASSIGNではsy-subrcではなく、<fs> IS ASSIGNEDで判定してください。

项目 review 时，更自然的表达不是“请检查错误”，而是“请说明未找到、未授权、未赋值时业务上要怎么处理”。这能把 sy-subrc 从机械语法变成业务分支。

43.7 Example / 学习例子

43.7.1 例子一：存在检查

如果只是判断是否存在，不一定要读取整行：

IF line_exists( lt_items[ item_id = lv_item_id ] ).
  mark_item_seen( lv_item_id ).
ENDIF.

这比下面的旧式写法更表达意图：

READ TABLE lt_items TRANSPORTING NO FIELDS
  WITH KEY item_id = lv_item_id.

IF sy-subrc = 0.
  mark_item_seen( lv_item_id ).
ENDIF.

但旧式写法不是错。需要 sy-tabix、需要兼容旧代码风格、需要和周围 READ TABLE 代码保持一致时，READ TABLE ... TRANSPORTING NO FIELDS 仍然可读。关键是：不要读了以后不判断。

43.7.2 例子二：读取后立即失败
READ TABLE lt_items INTO DATA(ls_item)
  WITH TABLE KEY item_id = lv_item_id.

IF sy-subrc <> 0.
  APPEND VALUE #( item_id = lv_item_id
                  message = 'Item was not found' ) TO et_messages.
  RETURN.
ENDIF.

calculate_item_amount( ls_item ).

这段代码比“只在成功分支里处理，失败时什么也不做”更适合教程。因为读者能看到失败分支的业务含义。

43.7.3 例子三：权限检查
AUTHORITY-CHECK OBJECT 'Z_SALES'
  ID 'ACTVT' FIELD '02'
  ID 'VKORG' FIELD lv_sales_org.

IF sy-subrc <> 0.
  APPEND VALUE #( type = 'E'
                  text = |No authorization for sales org { lv_sales_org }| )
         TO et_messages.
  RETURN.
ENDIF.

update_sales_document( ).

这里不要把 sy-subrc <> 0 写成 CONTINUE 然后不留痕迹。权限失败必须能被用户、日志、测试或上层调用方看见。

43.8 Counterexample / Failure mode

常见 silent failure 模式：

  • READ TABLE ... INTO wa 后不检查 sy-subrc，继续使用 wa。
  • SELECT SINGLE ... INTO ls_data 后不检查空结果，后续把目标变量当成真实业务对象。
  • SELECT SINGLE 的 WHERE 不是唯一条件，却把 sy-subrc = 0 理解成唯一且稳定地找到了业务记录。
  • AUTHORITY-CHECK 后不检查返回码，权限失败没有阻断。
  • ASSIGN ... TO <fs> 后直接使用 <fs>，失败时可能未分配或语义不清。
  • 静态 ASSIGN dobj TO <fs> 后检查 sy-subrc，误以为该语句设置了返回码。
  • 先执行 A 语句，之后执行 B 语句，再用 sy-subrc 判断 A 的结果。
  • 把所有 sy-subrc <> 0 都写成 CONTINUE 或 RETURN，没有消息、日志、异常或业务说明。
  • 在新方法设计中仍然模仿 function module 风格，用裸 sy-subrc 表达复杂业务失败。
  • CALL FUNCTION ... EXCEPTIONS OTHERS = 1 后不检查 sy-subrc，把所有经典 function module 错误吞掉。
  • 用 OPTIONAL 吃掉 table expression 的 not-found，但没有区分“真 initial value”和“没找到”。

一个危险反例：

READ TABLE lt_price INTO DATA(ls_price)
  WITH KEY material = lv_material.

DATA(lv_amount) = ls_price-price * lv_quantity.

更好的写法：

READ TABLE lt_price INTO DATA(ls_price)
  WITH KEY material = lv_material.

IF sy-subrc <> 0.
  RAISE EXCEPTION TYPE zcx_price_missing
    EXPORTING material = lv_material.
ENDIF.

DATA(lv_amount) = ls_price-price * lv_quantity.

如果未维护价格是允许的业务场景，也要写出 fallback：

READ TABLE lt_price INTO DATA(ls_price)
  WITH KEY material = lv_material.

IF sy-subrc <> 0.
  DATA(lv_amount) = 0.
  RETURN.
ENDIF.

lv_amount = ls_price-price * lv_quantity.

重点不是每次都抛异常，而是失败路径必须显式。

43.9 思考问题

  • 当前语句是否真的会设置 sy-subrc？具体值由哪一页 ABAP Keyword Documentation 定义？
  • sy-subrc 检查是否紧跟在相关语句之后？如果没有，是否保存到了本地变量？
  • sy-subrc <> 0 在当前业务里代表未找到、未授权、未赋值、未更新，还是系统级失败？
  • 如果改用 table expression，not-found 是抛异常、OPTIONAL、DEFAULT，还是 line_exists？
