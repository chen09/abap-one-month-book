37. COMMIT WORK restriction in RAP

  • Chapter 03
  • Section 37
  • self-contained study note

37.1 一句话结论

COMMIT WORK restriction in RAP 指的是：在 RAP behavior implementation 的 handler、validation、determination、saver 等受控生命周期中，项目代码不能把 COMMIT WORK / ROLLBACK WORK 当成普通保存按钮来使用。RAP runtime 才是事务边界的 owner；外部 RAP BO consumer 应通过 EML Entity Manipulation Language 的 COMMIT ENTITIES，或通过 OData/Fiori 保存语义触发 Save sequence 保存序列。

一句话记忆：RAP 中的“保存”不是谁写到数据库谁就提交，而是由 RAP runtime 收集 transactional buffer、执行保存序列、统一提交或回滚。

37.2 到底学什么

本节要解决一个很实际的问题：为什么传统 ABAP 里常见的 CALL FUNCTION ...、UPDATE ztable、COMMIT WORK，到了 RAP 中会变成高风险写法？

在经典 ABAP report、dialog program 或 batch program 中，程序往往自己决定何时结束 SAP LUW。调用 BAPI 后写 BAPI_TRANSACTION_COMMIT，或更新若干表后执行 COMMIT WORK，是很多旧系统代码的常见形态。但 RAP 的目标不是让每个方法各自提交，而是让一个 business object 的 create / update / delete / action 在同一套事务模型里完成：先进入 transactional buffer，再由 validation、determination、authorization、lock、numbering、save sequence、message response 共同决定是否能保存。

因此，本节要学三件事。第一，区分 COMMIT ENTITIES 和 COMMIT WORK：前者是 RAP EML consumer 提交 RAP BO 修改的语义，后者是经典 ABAP SAP LUW 提交语句，不能在 RAP behavior implementation 里随意使用。第二，理解哪些位置容易误用提交：action handler、validation、determination、save_modified、legacy API wrapper、additional save。第三，学会在 code review 中识别“内部 commit”风险，尤其是旧 BAPI、function module、update task、外部接口和日志写入。

初学者不需要一开始背完所有底层限制，但必须形成一个判断习惯：只要代码处在 RAP BO 生命周期内，就先问“这次事务是谁拥有的”。如果答案是 RAP runtime，就不要自己抢着提交。

37.3 为什么重要

COMMIT WORK 限制之所以重要，是因为它关系到“这次业务操作能不能作为一个整体成功或失败”。RAP BO 不是一组孤立 SQL。一个保存请求可能包含 header、item、draft、late numbering、authorization、validation message、action result、additional save 和 cleanup。只要中间某个方法提前提交，后面的失败就无法完整回滚，用户看到的消息也可能和数据库状态不一致。

典型事故包括：

  • action handler 已经 INSERT 主表并 COMMIT WORK，后续 on-save validation 失败，画面显示保存失败但数据库已经有脏数据。
  • save_modified 调用旧 function module，旧 function module 内部 commit，RAP 主保存失败后旧接口数据却已经提交。
  • additional save 提前写审计日志，主数据回滚后日志仍显示成功。
  • 用户重复点击保存时，外部系统被调用两次，而 RAP 只返回一次失败消息。

对项目交付来说，这不是“代码风格问题”，而是上线风险。日本 SAP 项目在 review 中会非常关注 コミット制御、トランザクション境界、内部コミット。如果说明书只写“保存処理を実装する”，但没有说明谁负责 commit、失败时如何 rollback、旧 API 是否 no-commit，就很难通过设计评审。

对 Java/Python 背景学习者，可以把 RAP runtime 类比成 framework-managed unit of work。不同点在于：RAP 的 unit of work 和 BDEF、EML、OData、draft、SAP LUW、ABAP Cloud restricted syntax、save sequence 方法强绑定。开发者不是在任意 service method 里开关事务，而是在 RAP 允许的生命周期点提供逻辑。

37.4 Key mechanism / 核心机制

第一层机制是 transactional buffer。MODIFY ENTITIES、OData PATCH、Fiori action button 触发的修改，通常先进入 RAP transactional buffer。此时 handler 可以读写 BO 状态，validation 可以返回 failed / reported，determination 可以派生字段，但这不等于数据库已经提交。

第二层机制是 RAP LUW。外部 ABAP consumer 使用 COMMIT ENTITIES 结束 RAP LUW，触发 save sequence。Fiori Elements / OData consumer 则通过协议层的保存请求触发相同方向的保存语义。项目代码不应该在 behavior implementation 内部用 COMMIT WORK 直接结束 SAP LUW，因为那会绕过 RAP runtime 对 BO 生命周期的控制。

第三层机制是 save sequence。保存触发后，RAP runtime 负责进入保存序列：保存前检查、late numbering、managed save、unmanaged save、additional save、cleanup 等步骤按框架语义执行。即使代码已经进入 Saver class 保存器类，也不代表项目可以自己提交。Saver 是保存阶段的项目扩展点，不是 transaction owner。

第四层机制是 illegal statement 风险。不同 ABAP release、strict mode、ABAP Cloud 语境和 RAP 方法位置对语句限制会有差异，但原则很稳定：在 RAP behavior implementation 的受控方法中使用 COMMIT WORK / ROLLBACK WORK / commit-coupled legacy construct，通常会导致语法、激活、运行时或事务一致性问题。遇到系统实际报错时，应以当前系统的 ABAP Keyword Documentation 和 ADT 检查结果为准。

第五层机制是 legacy API。很多旧 BAPI 或 function module 有两种风险：一种是调用者需要显式 commit；另一种是 API 内部自己 commit。RAP 项目最怕第二种。内部 commit 会让 RAP runtime 无法回滚。迁移时应优先寻找 no-commit API、拆分保存逻辑、改造 wrapper、使用 outbox / business event，或重新评估是否适合放进 RAP save sequence。

第六层机制是副作用边界。发邮件、HTTP 调用、文件输出、外部系统通知、不可回滚日志都不是普通数据库 update。即使它们没有写 COMMIT WORK，也可能在业务回滚后留下外部副作用。RAP 中应优先把这类动作设计成保存成功后的事件、outbox、异步处理或可幂等重试机制。

一个简化流程如下：

Consumer request
  -> RAP handler / validation / determination
  -> RAP transactional buffer
  -> COMMIT ENTITIES / OData Save
  -> RAP save sequence
  -> framework-controlled database commit or rollback

错误流程如下：

Action handler
  -> direct INSERT / legacy API
  -> COMMIT WORK
  -> later validation fails
  -> RAP reports error, but database or external side effect already exists
Internal commit review checklist

把旧逻辑迁移到 RAP 时，最危险的不是明面上写了 COMMIT WORK，而是“看起来只是调用一个保存 API，里面其实已经 commit”。评审旧 function module、BAPI、class method 或 Z wrapper 时，可以按下面顺序查。

第一，全文搜索显式提交语句：COMMIT WORK、ROLLBACK WORK、BAPI_TRANSACTION_COMMIT、BAPI_TRANSACTION_ROLLBACK。如果出现在 behavior implementation 或被其同步调用的保存路径上，就要停下来确认事务边界。

第二，检查 API 文档和命名。很多旧 BAPI 设计为“调用 BAPI 后由 caller commit”，这种还可以通过外层控制；更危险的是 wrapper 名称叫 save_and_commit、post_document、execute_final，内部可能已经提交。不要只看方法名好听，要读实现或官方说明。

第三，检查 update task / background task / tRFC / qRFC。即使没有直接 COMMIT WORK，CALL FUNCTION ... IN UPDATE TASK、IN BACKGROUND TASK、外部队列写入、文件输出、HTTP 调用也可能把效果推到 RAP rollback 无法自然撤销的位置。它们需要单独设计 outbox、event、重试和补偿。

第四，确认失败路径。假设旧 API 成功写了一部分，随后 RAP validation 或 additional save 失败，旧 API 的结果能否一起回滚？如果不能，就不应直接放在 handler、validation、determination 或 save_modified 中同步调用。

第五，给出替代方案。优先找 no-commit API；没有时，考虑拆分旧 API、把外部副作用改成保存成功后的 business event/outbox、使用异步 job 加 idempotency，或重新评估这个 legacy save 是否适合 RAP BO。评审结论不能只写“内部 commit 有风险”，还要写清下一步可执行路径。

风险信号应问的问题常见处理
BAPI_TRANSACTION_COMMIT是否在 RAP behavior implementation 内部调用？移到外部 consumer，或改成 RAP COMMIT ENTITIES 语义
旧 FM 内部 COMMIT WORK是否有 no-commit 版本？替换 API / 重构 wrapper
同步 HTTP / 邮件SAP rollback 后外部能否撤销？outbox / event / async job
IN UPDATE TASK谁触发最终提交？避免在 RAP 内混用，或重新设计保存边界
直接写日志表主保存失败后日志是否仍存在？additional save / application log 策略 / 幂等设计

37.5 日本项目现场说法

  • コミット制御: commit control。
  • トランザクション境界: transaction boundary。
  • 内部コミット: internal commit。
  • COMMIT WORK 禁止: 禁止在该上下文使用 COMMIT WORK。
  • RAP LUW: RAP logical unit of work。
  • 保存シーケンス: save sequence。
  • 部分コミット: partial commit。
  • ロールバック不可: cannot be rolled back。

この処理は RAP のトランザクション境界内なので、handler や saver の中で COMMIT WORK を実行しないでください。

意思是：这段处理处在 RAP 的事务边界内，所以不要在 handler 或 saver 中执行 COMMIT WORK。

この汎用モジュールは内部で commit していないか確認が必要です。内部コミットがある場合、RAP のロールバックと整合しません。

意思是：需要确认这个 function module 内部是否 commit。如果有内部 commit，就不能和 RAP rollback 保持一致。

外部連携は保存成功後のイベントまたは outbox 方式で設計し、RAP 保存失敗時に部分実行が残らないようにしてください。

意思是：外部集成应设计为保存成功后的事件或 outbox 方式，避免 RAP 保存失败时留下部分执行结果。

37.7 Example / 学习例子

例子一：普通 ABAP consumer 调用 RAP BO。

MODIFY ENTITIES OF zi_travel
  ENTITY Travel
  CREATE FROM lt_create
  MAPPED   DATA(mapped)
  FAILED   DATA(failed)
  REPORTED DATA(reported).

IF failed IS INITIAL.
  COMMIT ENTITIES
    RESPONSE OF zi_travel
    FAILED   DATA(commit_failed)
    REPORTED DATA(commit_reported).
ENDIF.

这里的重点不是语法细节，而是提交语义。MODIFY ENTITIES 把创建请求交给 RAP BO；COMMIT ENTITIES 触发 RAP 保存。不要把它改成 COMMIT WORK，也不要在 BO 的 handler 方法内部自己提交。

例子二：action handler 改状态。

用户点击 Approve。正确思路是：action handler 检查当前状态，必要时用 local EML 修改 transactional buffer，把结果和 message 返回给 RAP runtime。真正落库等到保存触发后由 save sequence 处理。错误思路是：action handler 直接 UPDATE ztravel SET status = 'A'，然后 COMMIT WORK。这样后续 validation 失败、draft activate 失败或其他 entity 保存失败时，状态已经提交。

例子三：save_modified 调旧 API。

某项目在 with unmanaged save 中使用 save_modified 调用旧保存 class。review 时必须确认旧 class 没有内部 COMMIT WORK。如果旧 API 只能“保存并提交”，它就不适合直接放进 RAP save sequence。更安全的方向是拆出 no-commit 保存方法，或让 RAP 保存主数据后写 outbox，由异步 job 调用外部系统。

例子四：AC0567C POC 的迁移思考。

如果未来把批量上传 POC 改造成 RAP BO，上传行的解析、字段检查、状态派生属于 interaction / validation 思维；最终写入、上传日志、事件通知属于保存或保存后语义。不能为了“按钮点完立刻有数据”就在 action handler 里批量插表并 commit。

37.8 Counterexample / Failure mode

反例一：handler 中直接 COMMIT WORK。开发者以为 action 已经完成，所以立即提交。后续 save sequence validation 失败，RAP 返回错误，但数据库状态已经变化。修正方式是：handler 只改变 BO buffer，由 RAP runtime 统一保存。

反例二：用 COMMIT WORK 修复“MODIFY ENTITIES 后数据库查不到数据”。这是典型误解。MODIFY ENTITIES 后没有持久化，通常是因为还没有 COMMIT ENTITIES，不是因为需要经典 commit。修正方式是：外部 EML consumer 使用 COMMIT ENTITIES，behavior implementation 内部不要自提交。

反例三：旧 BAPI 内部 commit。某 function module 在成功时自己 COMMIT WORK。RAP 后续失败时，该 BAPI 的结果已经提交。修正方式是寻找 no-commit 版本、改造 legacy API、或用 outbox / event 进行事务后处理。

反例四：additional save 写外部系统。保存审计日志可以是数据库附加保存，但同步调用外部 HTTP API、发邮件、发文件并不自动受 SAP rollback 控制。修正方式是使用可重试、可去重、可补偿的设计。

反例五：saver class 中认为“已经是保存阶段，所以可以 commit”。即使进入 saver，RAP runtime 仍然掌握 transaction boundary。修正方式是把 saver 看成保存序列中的项目实现点，而不是独立事务程序。

反例六：把 RAP 和经典程序混在一个提交模型里。普通 report 里调用 BAPI 后可能需要 BAPI_TRANSACTION_COMMIT，但 RAP behavior implementation 里不能照搬。修正方式是先判断当前代码是 RAP BO consumer、behavior implementation，还是传统程序。

37.9 思考问题

  • 当前代码是在 RAP consumer 侧，还是 behavior implementation 内部？
  • 这段逻辑是否只修改 transactional buffer，还是直接修改数据库？
  • 如果后续 validation 失败，这段处理能否自动 rollback？
  • 调用的旧 function module、BAPI、class method 内部是否 commit？
  • 是否存在 COMMIT WORK、ROLLBACK WORK、update task、外部 HTTP、发邮件、文件输出等不可回滚副作用？
  • 外部 consumer 是否应该用 COMMIT ENTITIES？
  • 在日本项目设计书中，应如何说明 コミット制御 和 トランザクション境界？

  • 用自己的话区分 COMMIT ENTITIES 和 COMMIT WORK。
  • 找一段 behavior implementation，确认是否存在 COMMIT WORK / ROLLBACK WORK / 内部 commit API。
  • 写 3 句日语 review comment：一句说明禁止内部 commit，一句说明 transaction boundary，一句说明外部连携应考虑 outbox。
  • 设计一个测试：validation 失败时，handler 中的状态变更不应直接落库，外部副作用不应发生。
