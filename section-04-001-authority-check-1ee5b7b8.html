Chapter 04 - Project Delivery And Production Readiness

1. AUTHORITY-CHECK 经典授权检查

  • Chapter 04
  • Section 01
  • self-contained study note

1.1 一句话结论

AUTHORITY-CHECK 是经典 ABAP 里最直接的应用层授权检查语句：程序拿当前用户、授权对象和字段值去问 SAP 授权系统“这个用户是否允许做这件事”，然后用 sy-subrc 决定继续、报错或拒绝请求。

对初学者来说，本节先建立一个核心判断：授权不是“画面上有没有按钮”，而是“后端代码、数据访问和业务动作是否真的被保护”。如果一个用户不能修改某个公司代码的数据，ABAP 程序不能只依赖 Fiori / SAP GUI 的按钮隐藏，也不能只相信前端传来的参数；在进入敏感读取、创建、修改、删除、审批、提交等动作之前，后端必须有明确的授权判断。

1.2 到底学什么

本节学习的是传统 ABAP 权限入口，不是整套 SAP 权限架构。需要掌握六件事。

第一，知道 AUTHORITY-CHECK 检查的对象不是“用户是否登录”，而是用户角色中是否包含某个授权对象及其字段值。常见字段包括活动类型 ACTVT，例如显示、修改、创建、删除等；实际项目还会有组织维度字段，例如公司代码、销售组织、工厂、采购组织或自定义字段。

第二，知道检查结果必须被处理。AUTHORITY-CHECK 执行后会设置 sy-subrc。一般理解为：sy-subrc = 0 表示通过；非 0 表示不通过或检查条件不满足。项目代码最危险的写法不是检查失败，而是检查完以后忽略 sy-subrc，继续执行后续更新。

第三，知道它和 CDS Access Control DCL 行级访问控制、RAP global authorization 全局授权、RAP instance authorization 实例授权、Precheck 预检查 的边界。AUTHORITY-CHECK 是显式写在 ABAP 代码里的检查；DCL 偏向 CDS 数据读取层的访问控制；RAP authorization 偏向 RAP 行为模型中的操作授权；precheck 偏向在正式修改前尽早拒绝明显非法的请求。它们可以互补，但不能随便互相替代。

第四，知道如何在项目里验证。授权相关需求不能只用“有权限用户 happy path”验收，还要准备无权限用户、边界组织数据、不同操作类型、API 直接调用、错误消息和日志证据。

第五，知道它适合放在哪里。AUTHORITY-CHECK 常见于 classic report、function module、class method、BAdI、user exit、RFC/OData wrapper、后台 job 和旧业务 API 调用前后。它不应该散落在每个小 helper 里，也不应该只放在 UI 入口。更好的设计是把授权判断集中到 service/policy/checker 层，让调用路径清楚、测试可替换、错误处理一致。

第六，知道它不是“写了就安全”。真正安全取决于三件事：检查的 authorization object / field / value 是否正确；sy-subrc 或异常结果是否被正确处理；失败后是否真的停止敏感动作且没有副作用。缺任何一个，代码表面上有授权，实际上仍可能越权。

1.3 为什么重要

授权检查是生产就绪的底线之一。一个 ABAP 程序即使功能完全跑通，只要授权边界错误，就可能造成越权查看、越权修改、审计不通过、接口绕过 UI 限制、上线后紧急回退等问题。特别是在 SAP 项目里，权限不是单独属于 Basis 或安全团队的事情；ABAP 开发者必须知道自己的代码在哪里消耗权限、在哪里暴露业务动作、在哪里需要向权限设计提出字段和角色需求。

从 Java/Python 背景看，可以把 AUTHORITY-CHECK 类比成 service 层的 authorization guard，但 SAP 的差异更强：权限对象通常和业务组织维度绑定，角色由 PFCG 或业务角色维护，失败后还常常需要用 SU53、trace 或安全团队的角色配置来排查。也就是说，ABAP 代码里的一个检查语句，背后连接的是业务岗位、组织结构、授权对象、角色维护、测试用户和上线审计。

这一点在接口场景尤其重要。UI 层隐藏按钮只能改善用户体验，不能保护接口。只要 OData / RFC / Web API / batch job 入口仍能被调用，真正的保护必须落到服务实现、行为实现、CDS/DCL 或底层业务 API 的授权路径上。

在日本项目现场，AUTHORITY-CHECK 还常常是缺陷沟通的共同语言。开发者可以把“检查对象、字段和值”交给权限团队，权限团队可以通过 role/profile 配置和 SU53/trace 证据判断是否缺权限。没有这条共同语言，缺陷很容易变成“开发说不是代码问题，权限团队说 role 已给，测试说还是失败”。

1.4 Key mechanism / 核心机制

经典模式可以用一段伪代码理解：

AUTHORITY-CHECK OBJECT 'Z_TRAVEL'
  ID 'ACTVT'    FIELD '03'
  ID 'ZCOMPANY' FIELD lv_company_code.

IF sy-subrc <> 0.
  " Stop the read operation and return a clear authorization error.
ENDIF.

这段代码表达的不是“用户是否能进入系统”，而是“当前用户是否被授予 Z_TRAVEL 这个授权对象，并且其中 ACTVT = 03、ZCOMPANY = lv_company_code 的组合是允许的”。实际项目中对象名和字段名要来自系统里的授权对象设计，不能凭空写一个看起来合理的名字。

ACTVT 通常用于区分动作类型，例如显示、创建、修改、删除、执行等。组织字段用于限制数据范围，例如只允许访问某些公司代码或工厂。检查通过后程序才继续；检查失败时应返回清楚、可测试、不会泄露过多敏感信息的错误。错误处理可以是 message、exception、RAP reported/failed、HTTP 403 风格响应或项目框架规定的错误格式，但核心原则相同：不能继续执行敏感动作。

还要记住 classic authorization object 的字段完整性。一个授权对象通常有多个 authorization field；代码要么用 ID '...' FIELD ... 检查该字段，要么用 ID '...' DUMMY 明确表示这个字段本次不参与检查，不能无意漏掉。实际处理时可以把任何非 0 sy-subrc 都当作 deny 安全失败；其中常见的 sy-subrc = 4 更像普通“用户没有匹配授权”，而 8、12 这类结果往往提示对象名、字段名或检查写法/配置有问题，需要开发和权限团队修正，不能只当成普通缺权限糊弄过去。

还要注意三个边界。

第一，AUTHORITY-CHECK 不会自动给 SQL 加 WHERE 条件。它只告诉程序某个授权组合是否通过；如果需求是“只能看到自己公司代码的数据”，代码仍要在选择数据、DCL、RAP instance authorization 或业务 API 中保证数据范围正确。

第二，它不等于 SU53。SU53 是排查用户最近一次授权失败的常用入口，帮助定位缺了什么授权；AUTHORITY-CHECK 是程序中的检查动作。一个用于执行，一个用于诊断。

第三，现代 RAP / ABAP Cloud 场景要优先尊重框架推荐的授权模型和 released API 约束。学习 AUTHORITY-CHECK 仍然必要，因为大量既有代码、BAdI、exit、classic report、function module、utility class 和迁移项目都能看到它；但在新建 RAP 服务中，不能因为熟悉经典语句就跳过 behavior authorization、DCL 或服务层的设计。

可以用下面的表判断 classic check 的位置：

场景AUTHORITY-CHECK 的典型作用仍需配合
Classic report 执行前判断用户是否可执行功能或组织范围selection-screen 检查、日志、错误消息
Function module / class service在敏感业务动作前做后端授权调用方事务边界、异常处理
RFC / OData wrapper防止 API 绕过 UI 限制authentication、service exposure、RAP/OData 错误格式
批量上传行处理按 company code / plant / action 逐行或逐批检查批量错误返回、application log
RAP 新实现可作为被复用的底层授权逻辑RAP global/instance authorization、failed/reported
CDS 列表读取不自动过滤 SQL/CDS 结果DCL、query condition、service projection

sy-subrc 处理可以写成固定纪律：

AUTHORITY-CHECK OBJECT 'Z_UPLOAD'
  ID 'ACTVT' FIELD '02'
  ID 'BUKRS' FIELD lv_bukrs.

IF sy-subrc <> 0.
  " Convert the failed check into a controlled authorization error.
  " Do not update, enqueue, call external APIs, or write success logs.
  RAISE EXCEPTION TYPE zcx_no_authority.
ENDIF.

项目上不要把失败处理写成空分支，也不要只写注释。失败后应进入 message、exception、RAP failed/reported、HTTP 403 风格响应或项目统一错误结构。更重要的是：失败路径不能继续保存、不能拿锁长时间占用资源、不能写“成功”应用日志，也不能泄露其他组织的敏感数据。

1.4.1 Authorization evidence matrix

AUTHORITY-CHECK 的验收不能只靠“有权限用户能成功”。授权缺陷最常发生在边界路径：无权限用户、跨组织数据、API 直接调用、批量混合数据和错误处理继续执行。可以用下面的矩阵准备证据。

Test pattern要验证什么关键证据
Authorized happy path正确角色、正确组织值、正确 ACTVT 能通过用户、role、授权对象/字段值、成功结果
Unauthorized user没有对应授权对象或组织值时被拒绝sy-subrc 处理、错误消息、无 DB 更新
Wrong activity只有显示权限的用户不能执行修改/删除/提交ACTVT 值、操作类型、拒绝结果
Cross-organization允许公司代码 A 的用户不能处理公司代码 B请求数据、组织字段值、拒绝结果
API direct call绕过 UI 直接调用 OData/RFC/job 也被后端拦截HTTP/RFC response、后端日志、无副作用
Batch mixed data同一批次含有允许和不允许组织值时处理策略明确按行拒绝或整批拒绝的规则和证据
Failure path safety授权失败后不 enqueue、不 update、不 call external API、不写成功日志trace、application log、DB 后验
Troubleshooting缺权限时能用 SU53/trace/日志定位对象和值授权对象、字段、值、用户、时间

这张矩阵也能帮助和权限团队沟通。开发者不要只说“请给权限”，而要交付可维护的信息：检查哪个 authorization object、哪些 field、ACTVT 是什么、组织值来自哪里、失败后返回什么错误、哪些入口共用同一检查。权限团队才能把 PFCG role、测试用户和 SU53/trace 证据对上。

1.5 日本项目现场说法

  • 常见日语：権限チェック、権限オブジェクト、権限項目、ACTVT、PFCGロール、SU53、権限エラー、認可漏れ。
  • 设计说明：この処理では、更新前に AUTHORITY-CHECK で会社コード単位の権限を確認します。
  • Review 说法：sy-subrc の判定後に処理が継続していないか、権限エラー時のメッセージと戻り値を確認してください。
  • 测试说法：権限ありユーザと権限なしユーザの両方で、画面操作だけでなく API 直接呼び出しも確認します。
  • 障害调查说法：SU53 だけで結論を出さず、該当処理の AUTHORITY-CHECK 対象、項目値、ロール設定を突き合わせます。
  • 设计书条目：権限オブジェクト、権限項目、ACTVT、組織値、チェック箇所、エラー時の処理を明記します。
  • Review 追加句：AUTHORITY-CHECK 後に外部連携、DB更新、成功ログ出力へ進んでいないか確認してください。
  • API 说明：画面制御とは別に、API 直接呼び出し時にも同一のバックエンド権限チェックが実行されることを確認します。

1.7 Example / 学习例子

假设有一个批量上传 API，要把外部文件里的销售数据写入 SAP。需求说：“用户只能上传自己负责公司代码的数据。”这个需求不能只靠前端下拉框限制，因为 API 调用者可以直接构造请求。

一个可交付的设计通常要回答：

  • 在哪里检查：进入写入逻辑前，按请求中的公司代码执行授权检查。
  • 检查什么：动作是创建/修改还是显示，组织字段是公司代码、销售组织还是工厂。
  • 失败怎么办：拒绝本条或整批请求，返回统一错误，不执行后续写入。
  • 怎么测试：准备有权限和无权限用户，分别验证 UI 操作、API 直接调用、批量多公司代码混合数据、错误消息和日志。

最小学习动作：在一个示例报告或类方法里写出 AUTHORITY-CHECK OBJECT ... ID ... FIELD ... 的结构，然后在 sy-subrc <> 0 时停止处理。

进阶学习动作：在 AC0567C POC Reading Map 中找一个“上传、确认、更新、提交”类入口，标注它应该在哪里做 classic authorization、哪里适合 DCL、哪里适合 RAP authorization 或 precheck。

一个 AC0567C 风格的上传 API 可以这样拆：

入口classic AUTHORITY-CHECK 可能负责还要看
validate用户是否总体可使用上传检查功能、是否可检查某组织范围precheck、错误清单、是否暴露敏感数据
execute用户是否允许创建/修改业务数据，ACTVT 与组织值是否覆盖请求数据RAP global/instance authorization、locking、save sequence
cancel用户是否允许取消/删除上传批次instance authorization、状态规则、owner 判断
status用户是否可查看该批次状态或日志DCL/read authorization、日志访问控制

如果这套 API 封装旧 BAPI 或 function module，还要确认旧 API 内部是否已经做授权检查、检查的是哪个 object、是否内部 commit、错误如何返回。重复检查不一定错，但职责要清楚；完全依赖未知旧 API 也不安全，因为测试人员无法证明具体对象和组织字段是否被保护。

测试矩阵至少应包含：

Authorized user + allowed company code -> success
Authorized user + disallowed company code -> authorization error
Display-only user + execute action -> authorization error
No-authority user + API direct call -> authorization error before side effects
Technical/communication user -> only allowed scope succeeds

证据要包含 API response、业务数据未被修改、application log 没有误写成功、SU53 或 trace 可解释缺失对象/字段值，以及角色修复后有权限用户成功、无权限用户仍失败。

1.8 Counterexample / Failure mode

常见失败模式有十类。

第一，只隐藏按钮。用户在画面上看不到“修改”按钮，但接口、批处理或调试入口仍可能触发后端更新。

第二，只在程序入口检查一次。入口检查了“可以使用此功能”，但后续处理每一行数据时没有按公司代码、工厂或对象实例检查，导致用户能修改不属于自己的数据。

第三，检查后不处理 sy-subrc。代码里有 AUTHORITY-CHECK，review 时看起来像做了权限，但失败后仍继续执行。这比没有检查更危险，因为它会制造“已经做过权限”的错觉。

第四，把 DCL、RAP authorization、precheck 和 classic check 混为一谈。读取列表、显示详情、执行 action、保存变更、调用外部 API 是不同风险点，不能用一个泛泛的“授权已做”覆盖所有路径。

第五，测试只覆盖有权限用户。授权测试必须有失败场景，否则无法证明越权路径被关住。

第六，只检查功能权限，不检查组织值。用户有 ACTVT = 02，但不代表他能修改所有 company code / plant / sales org。

第七，把 AUTHORITY-CHECK 放在副作用之后。先写表、发接口、消耗编号，最后才发现无权限，已经太晚。

第八，错误消息过于技术或过度泄露。无权限要可测试、可支持，但不应把其他组织敏感字段、审批人、内部规则完整暴露给调用者。

第九，接口用户被赋予过大权限。Communication user 或后台用户也要遵守最小权限和审计原则，不能因为“系统调用”就绕过授权。

第十，迁移到 RAP 后旧检查失效。把 classic logic 包成 RAP service 时，如果只暴露 action 而没有把 classic check 接入 RAP authorization/failed/reported 路径，API 语义和错误处理可能变得不一致。

修正动作：对每个敏感用例画出“入口、读取、实例选择、动作执行、保存、错误返回”的路径，并在路径上标注使用 AUTHORITY-CHECK、DCL、RAP authorization、precheck 或业务 API 内置授权的具体位置。

1.9 思考问题

  • 这个检查是在“功能入口”层、数据行层、实例行为层，还是保存前校验层？
  • 当前需求里的授权维度是什么：功能、动作、公司代码、工厂、销售组织、用户归属，还是对象实例？
  • 如果只在 UI 隐藏按钮，不在后端检查，API 是否还能绕过？
  • sy-subrc <> 0 时，代码是否真的停止了敏感动作？
  • 检查失败时是否已经产生了 DB 更新、外部调用、编号消耗或成功日志？
  • 组织字段和 ACTVT 是否都被测试？
  • API 直接调用和后台 job 是否走同一授权路径？
  • 这个需求更适合 classic AUTHORITY-CHECK、CDS Access Control DCL 行级访问控制、RAP global authorization 全局授权、RAP instance authorization 实例授权，还是组合使用？
  • 在日本项目里向客户或权限团队说明时，应该说 権限オブジェクト、権限項目、ACTVT 还是 ロール設定？
