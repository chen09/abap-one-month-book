Chapter 02 - Core ABAP Programming

1. ABAP SQL overview

  • Chapter 02
  • Section 01
  • self-contained study note

1.1 一句话结论

ABAP SQL 是 ABAP 程序访问 SAP 业务数据的主要语言：它不是直接把数据库厂商 SQL 写进程序，而是通过 ABAP 数据库接口，以相对数据库无关的方式访问 ABAP Dictionary 表、视图和 CDS 数据模型。初学者先要建立一个判断：看到 SELECT / INSERT / UPDATE / DELETE 时，不只是“在查数据库”，而是在 SAP 应用服务器、ABAP 类型系统、客户端处理、权限、事务和后续 RAP / Clean Core 约束之间工作。

1.2 到底学什么

这一节不是要一次讲完 SELECT 的所有语法，而是建立 ABAP SQL 的总览地图。后面会分别展开 SELECT statement 查询语句、Host variables 宿主变量、INTO vs INTO TABLE 结果接收、ORDER BY 排序确定性 和 JOIN in ABAP SQL；本节先回答“ABAP 程序为什么需要 ABAP SQL、它和普通 SQL 有什么不同、项目里容易在哪里犯错”。

先看一个简化例子：

DATA lv_customer_id TYPE kunnr VALUE '1000001'.

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              created_at
       WHERE customer_id = @lv_customer_id
       ORDER BY created_at DESCENDING,
                sales_order_id
       INTO TABLE @DATA(lt_orders).

IF sy-subrc = 0.
  " Process the selected orders.
ENDIF.

这段代码里至少有六个 ABAP SQL 总览点。

第一，zsales_order 不是一个随便拼出来的物理表名，而应该是 ABAP repository 世界里可被 ABAP SQL 访问的数据源，例如 ABAP Dictionary database table、classic view、CDS view entity 或项目允许使用的接口视图。第二，lv_customer_id 是 ABAP 程序里的数据对象，写进 SQL 条件时要用 @ 标出 host variable。第三，FIELDS 只取需要的列，不把整行 * 当默认习惯。第四，WHERE 是性能、正确性和业务范围的第一道门。第五，ORDER BY 明确业务排序，不依赖数据库当前碰巧返回的顺序。第六，结果进入 ABAP 内表 lt_orders，后续由 ABAP 逻辑处理，同时还要检查 sy-subrc、sy-dbcnt 和异常路径。

1.3 为什么重要

ABAP 项目里大量业务逻辑都是“读出业务对象相关数据、判断、更新、记录消息、再交给事务边界提交”。ABAP SQL 是这个过程的入口。如果只把它当普通 SQL 学，很容易产生三类问题。

第一类是正确性问题。没有 ORDER BY 却取“第一条”，在 HANA、不同索引、不同执行计划或数据量变化后可能得到不同业务结果。SELECT SINGLE 如果不是完整主键读取，也不能自然等同于“取最新一条”或“取业务上唯一一条”。只要业务语义里出现“最新、最大、优先级最高、第一条”，就应该问：排序条件在哪里，唯一性由谁保证？

第二类是 SAP 边界问题。ABAP SQL 运行在 ABAP 运行时和数据库接口之上，有隐式客户端处理、ABAP 数据类型映射、Open SQL / ABAP SQL 语法演进、SAP LUW 与 database LUW 的边界。它不是 Lock object 锁对象，不是权限框架，也不是事务提交语句。读取或更新数据时，仍然要结合 SAP Transaction Boundary and LUW Model、锁、COMMIT WORK / ROLLBACK WORK、异常处理和消息设计理解。

第三类是交付风险。日本 SAP 项目里，经常会在代码 review、障害调查、性能调查、权限调查里追问：“このSELECTはどの条件で絞っていますか”“ソート順は保証されていますか”“権限チェックはどこで行っていますか”“本番データに直接アクセスしてよいオブジェクトですか”。学会 ABAP SQL overview，就是学会在写第一行 SELECT 之前先问这些问题。

1.4 Key mechanism / 核心机制

ABAP SQL 以前常被称为 Open SQL。现代官方文档更多使用 ABAP SQL 这个名称。它的核心价值是：ABAP 程序用统一的 ABAP SQL 语法写数据库访问，由 ABAP database interface 转换并发送到具体数据库。相比 Native SQL 或 SQLScript，ABAP SQL 追求的是 ABAP 集成、可移植性、DDIC/CDS 类型一致性和 SAP 平台约束下的可维护性。

它的运行路径可以这样理解：

  1. ABAP 程序写 SELECT / INSERT / UPDATE / MODIFY / DELETE 等 ABAP SQL 语句。
  2. 语句引用 ABAP Dictionary 或 CDS 世界中的数据源。
  3. 程序变量通过 host variable 进入 SQL 语句，现代写法用 @ 明确区分 ABAP 变量和 SQL 标识符。
  4. ABAP database interface 处理语法、类型映射、客户端处理、数据库连接和数据库相关转换。
  5. 数据库执行后，结果回到 ABAP data object、work area、internal table 或 inline declaration。
  6. ABAP 程序继续处理 sy-subrc、sy-dbcnt、异常、锁、事务提交和业务消息。

这里有几个初学者必须先放稳的边界。

ABAP SQL 不是 Native SQL。Native SQL 指通过 Native SQL interface 传递平台相关数据库语句，例如某些数据库专用 SQL；它会降低可移植性，也不适合作为初学者默认路径。SQLScript 是 SAP HANA 侧的数据库过程语言，也不是普通 ABAP 应用程序里写业务读取逻辑的默认工具。

ABAP SQL 不是权限。你能写 SELECT 访问某个数据源，不等于当前用户在业务上有权看这些数据。权限可能在 CDS DCL、RAP authorization、经典 AUTHORITY-CHECK、应用服务层逻辑或 UI/service 层共同实现。后续要连接 CDS Access Control DCL 行级访问控制、RAP global authorization 全局授权 和 AUTHORITY-CHECK 经典授权检查。

ABAP SQL 也不是事务边界。UPDATE 成功执行到数据库接口，不代表业务事务已经完整提交；同一个业务动作里还可能有 update task、BAPI、锁释放、应用日志和异常回滚。Week 2 会用 SAP Transaction Boundary and LUW Model 把这条线补齐。

1.7 Example / 学习例子

把 ABAP SQL 想成“ABAP 程序和 SAP 数据模型之间的受控通道”。如果 Java/Python 程序直接用数据库 driver 拼 SQL，关注点通常是连接串、SQL dialect、ORM、事务和索引；ABAP 项目还要多看一层：这个数据源是不是 SAP 允许访问的 repository object？是否是 client-dependent？是否有 CDS DCL？是否符合 ABAP Cloud 语言版本？是否应该通过 released API、CDS view entity 或 RAP BO，而不是直接查底表？

一个更接近 review 的例子：

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              created_at
       WHERE customer_id = @lv_customer_id
         AND status      = @lc_status_open
       ORDER BY created_at DESCENDING,
                sales_order_id
       INTO TABLE @DATA(lt_open_orders).

读这段代码时，reviewer 不只问“能不能激活”，还会问：

  • zsales_order 是项目自有表、CDS view entity，还是不该直接访问的 SAP 标准持久化对象？
  • customer_id、status 的类型是否来自 data element / domain，而不是随便用 string？
  • lv_customer_id、lc_status_open 是否用 @ 明确作为 host variable？
  • WHERE 是否足够限制数据量？
  • ORDER BY 是否表达了业务含义？
  • 读取结果为空时，sy-subrc / 空内表路径是否被处理？
  • 如果这段逻辑后面会更新数据，锁和事务边界在哪里？
  • 如果是服务/API 场景，权限在哪里执行？

最小学习动作：找一段项目或 POC 里的 SELECT，用上述八个问题逐行批注。不要先追求写复杂 SQL，先训练“看懂一条 SELECT 背后的 SAP 约束”。

1.8 Counterexample / Failure mode

失败模式一：把 SELECT SINGLE 当成“随便拿一条就行”。如果 WHERE 不是完整主键，或者业务上并不唯一，数据库可以返回任意符合条件的行。今天测试数据少时看起来正确，明天生产数据多起来就可能取错订单、取错价格、取错状态。

失败模式二：不写 ORDER BY 却依赖“当前看起来的顺序”。关系数据库不保证无排序结果集的业务顺序；执行计划、索引、并行度、数据库版本、过滤条件都可能改变返回顺序。在 ABAP SQL 里，只要后续逻辑依赖“第一条、最后一条、最新一条”，就必须把排序条件写出来。

失败模式三：把 ABAP SQL 当权限。SELECT 能读到数据，只能说明技术访问路径存在，不能说明用户有业务权限。特别是从 table 直接读数据时，不要以为 CDS DCL 一定自动生效；DCL 的前提是通过带 access control 的 CDS 数据源访问，并且设计正确。

失败模式四：在 ABAP Cloud / Clean Core 场景里直接查 SAP 标准底表。传统 on-premise 项目里能运行的表访问，不代表在 ABAP Cloud 或 S/4HANA Cloud extensibility 里允许。现代项目要优先确认 released API、released CDS view、RAP BO 或项目自有持久化对象。

失败模式五：只关注 SQL 性能，不关注 LUW。性能当然重要，但 ABAP SQL 写操作还牵涉锁、提交、回滚、update task、BAPI commit pattern 和消息处理。只看 SQL 一行，会漏掉真正的业务一致性问题。

1.9 思考问题

  • 这条 ABAP SQL 访问的数据源属于 DDIC table、classic view、CDS view entity，还是 released API 背后的接口视图？
  • 如果结果为空，业务上应该是正常情况、警告、错误，还是权限/配置问题？
  • 如果只取一行，唯一性由完整主键、唯一索引、业务规则，还是 ORDER BY 保证？
  • 如果读取多行，是否需要稳定排序？
  • 这条语句是否依赖 implicit client handling？有没有不该手写 MANDT 的地方？
  • 权限在哪里发生：DCL、RAP authorization、AUTHORITY-CHECK，还是完全缺失？
  • 后面是否有写操作？如果有，锁、SAP LUW、database LUW 和 commit/rollback 怎么处理？
  • 在 ABAP Cloud / Clean Core 审查下，这个数据源是否允许直接访问？
