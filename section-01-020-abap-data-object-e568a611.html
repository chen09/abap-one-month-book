20. ABAP data object 数据对象

  • Chapter 01
  • Section 20
  • self-contained study note

20.1 一句话结论

ABAP data object 数据对象，是 ABAP 程序运行时占用内存并保存值的对象；本节要学的是：它不是 Repository object 仓库对象，也不是 Data element 数据元素 或 Domain 域对象，而是一个带有 data type 的运行时值容器，例如 variable、constant、literal、system field 或后续会遇到的 anonymous data object。

20.2 到底学什么

先把“对象”这个词放回 ABAP 语境里。前面讲的 repository object 是系统里可创建、激活、传输的开发对象；data object 则是 ABAP program 执行时内存里的数据实例。一个 class、program、CDS DDL source 可以是 repository object；而 lv_amount、ls_item、lt_items、sy-subrc、'A'、` ABC ` 这类东西，是运行时参与语句计算的数据对象或字面值。

SAP Learning 的入门说法很直接：ABAP program 中的 data object 表示程序内存中保留的一段区域。ABAP 语句不是对“抽象变量名”做魔法，而是在读写这些内存区域里的值。变量、常量和 literal 是初学者最先要掌握的三类。variable 的内容可以在运行时变化；constant 有名字但值不能在运行时改变；literal 是写死在源码里的匿名值，没有可复用名字。

第二个边界是 data object 与 data type。data object 是“装值的实例”，data type 是“这个实例能装什么、如何表示”的规则。SAP 文档也用类似 class/instance 的类比解释：data object 是 data type 的实例，就像对象是 class 的实例。DATA lv_count TYPE i. 创建的是一个名叫 lv_count 的 data object；i 是它的数据类型。TYPE、LIKE、DDIC data element、domain、structure、table type 的细节后面展开，本节先记住：对象和值在运行时存在，类型决定它能以什么形态存在。

ABAP data object 总是 typed。对普通静态声明的 named data object 来说，类型在声明时确定，并在一次 program execution 中保持稳定。不要把 ABAP 变量理解成 Python 那种“名字先绑定一个整数，下一行又绑定一个字符串”的动态模型。ABAP 可以发生 implicit conversion，但这不是“变量变类型”，而是赋值、比较、表达式或参数传递时按规则转换值。

这里也要加一个早期 caveat：field symbol、data reference、generic type 会让“类型稳定”看起来变复杂。比如 field symbol 可以在运行时 assign 到不同对象；data reference variable 保存的是引用；generic type 可能到赋值或调用时才具体化。这些不是 Day 03 第一节要掌握的语法，但要知道它们是后续动态编程内容，不要拿它们反驳基础规则。

第三个边界是初始值。ABAP 变量不是“未初始化就是 null”。如果 DATA 没写 VALUE，系统会按类型给 type-specific initial value：数值型通常是 0，character type C 是空白，numeric text type N 等类型有自己的初始值规则，日期和时间也有各自初始表示，reference 初始时没有指向对象。VALUE 可以给 start value；对 constant，如果希望它就是类型对应的初始值，ABAP 也有 VALUE IS INITIAL 这种写法。项目 code review 里经常会问：这个 initial value 是业务上有意义，还是只是技术初始值？

第四个边界是 literal。ABAP 的引号很重要：单引号 'ABC' 是 text literal，类型行为更接近固定长度 character；反引号 ` ABC ` 是 string literal，更接近 variable-length string。数字 literal 也有自己的类型推断规则。Java/Python 背景的人如果把单引号和双引号经验直接带进 ABAP，很容易在字符串、尾随空格、比较和接口 payload 上踩坑。

本节先只讲 elementary data object、variable、constant、literal、initial value 和类型边界。structure、internal table、reference object、field symbol、inline declaration、FINAL、generic type、DDIC 绑定，会在后续小节逐步展开。Day 03 的目标不是一口气背完语法，而是先建立“值在内存中、类型决定行为、业务语义不能靠变量名自动保证”的基础模型。

20.3 为什么重要

data object 是后面所有 ABAP 语法的地基。SELECT ... INTO 要把数据库结果放进 data object；IF 判断读的是 data object 的当前值；LOOP 每次把 internal table 行放进 work area 或 field symbol；RAP handler、BAPI call、OData service、application log 最终也都在读写 typed data object。这个概念不稳，后面看代码会像看一堆关键字。

它也决定很多缺陷的根因。比如金额用 character 接收，比较时发生隐式转换；日期用字符串拼接，结果格式和时区被忽略；数量没有单位，后续计算看似正确但业务含义错；常量没有集中定义，多个 literal 分散在代码里，改规则时漏掉一处。这些问题不是“语法不会”，而是没有把 data object、type 和业务语义连起来。

对 Java/Python 背景的人，ABAP 的核心差异在于类型和业务字段更紧密。Python 变量名可以在运行时绑定不同类型对象；Java 变量有静态类型，但很多业务语义靠 class/model 层封装。ABAP 项目里，很多业务字段直接带着 DDIC data element、domain、currency/unit、date/time 等语义进入程序。你可以写出能跑的赋值语句，但如果类型选错，后面 SQL、接口、ALV、CDS、RAP、日志都会被影响。

日本 SAP 项目里，code review 经常会围绕 型、初期値、暗黙型変換、固定値、リテラル直書き 讨论。能说清 data object 是什么，才能解释为什么要定义 constant、为什么要避免 magic literal、为什么金额和数量要配单位、为什么某个变量初始值不是业务默认值。

20.4 Key mechanism / 核心机制

第一层机制是 declaration。DATA 声明 variable，CONSTANTS 声明 named constant，literal 直接出现在源码里。最小形态可以是 DATA lv_count TYPE i.、CONSTANTS lc_status_open TYPE c VALUE 'O'.。本教程会逐步把 DATA、TYPES、CONSTANTS、VALUE、TYPE、LIKE 拆开学。

第二层机制是 name and visibility。named data object 可以通过名字在其作用域内访问；literal 没有名字，所以不能被复用；system-defined data object 例如 sy-subrc、sy-datum 是系统提供的运行时字段。text symbol、anonymous data object、field symbol、data reference 都属于后续 awareness；现在先知道它们存在，避免看到项目代码时以为它们不是 data object。

第三层机制是 type-specific initial value。每种类型有自己的初始值，不要用 Java/Python 的 null 心智模型套 ABAP。IS INITIAL 判断的是“是否等于该类型的初始值”，不是“是否没有分配内存”。对业务代码来说，initial 有时表示“未输入”，有时只是技术空值，两者必须在设计和 precheck 中分清。

第四层机制是 assignment and conversion。赋值看似简单，但 ABAP 会在某些场景做隐式转换。'001'、1、'1 '、` 1 ` 在不同类型、比较和输出中可能表现不同。开发者要学会在语法检查、debugger、ABAP Unit 和边界测试里观察实际值，而不是只看源码表面。

第五层机制是 business meaning. ABAP data object 的类型既有技术层，也常常连接业务语义。金额字段需要 currency，数量字段需要 unit，日期和时间要关注格式和时区，状态值应尽量用 constant 或 domain fixed value 管理。变量名可以帮助阅读，但真正约束行为的是类型、检查、domain、precheck 和测试。

第六层机制是 scope of learning。Day 03 先从 elementary data object 开始，因为它是 structure、internal table、SQL result、RAP request/response 的组成单位。后面看到 ls_、lt_、lr_、<fs>、VALUE #( ) 时，都可以回到这一层：这是某个 type 的 data object，只是形态更复杂。

第七层机制是 runtime memory。Data object 只在程序执行期间存在；它的生命周期取决于声明位置和调用上下文。局部变量在方法或过程执行期间使用，全局数据在程序/类上下文中更长时间可见，internal table 可能承载多行运行时数据，reference 变量则保存指向对象的引用。理解生命周期，才能解释为什么某个值在下一次 dialog step、另一个 work process、另一个后台 job 中不一定还在。

第八层机制是 DDIC 语义。ABAP data object 可以直接用内置类型声明，也可以通过 DDIC data element、structure、table type 或 CDS/RAP 生成的类型承载业务语义。项目中更推荐让金额、数量、日期、状态、组织字段尽量贴近业务定义，而不是全部用 c 或 string 接收。这样 ALV、接口、日志、SQL、RAP response 和测试都会更稳定。

可以用下面的表快速区分几个容易混淆的概念：

概念存在位置作用例子
Repository objectSAP repository开发对象，可激活/传输class、program、CDS DDL source
Data type类型定义规定值的表示和操作规则i、string、DDIC data element
Data object程序运行时内存保存具体值lv_amount、ls_item、lt_items
Literal源码中的匿名值直接参与表达式'O'、` ABC 、123`
DomainDDIC 语义/取值约束字段技术属性和固定值范围状态域、公司代码域

接口和 RAP 场景中，data object 的选择会直接影响 payload。比如 OData JSON 里来的数字、日期、布尔状态、金额和文本，进入 ABAP 后都要映射到 typed data object。类型太宽，错误会被延后；类型太窄，合法输入可能被截断或转换失败；把业务字段全当字符串，短期省事，长期会让 validation、message、application log 和测试证据变差。

  • 值与类型：初期値、型依存初期値、実行時値、代入、暗黙変換、暗黙型変換、固定値。

  • 使用场景：code review、単体テスト、障害調査、型変換问题、接口字段映射、RAP precheck、日志输出。

20.7 Example / 学习例子

  • 例子 1：DATA lv_count TYPE i VALUE 0. 中，lv_count 是 variable data object，i 是 data type，0 是 VALUE addition 给出的 start value；如果不写 VALUE，系统才按 type-specific initial value 创建初始内容。运行时可以给 lv_count 赋新值，但它仍然是整数型变量。
  • 例子 2：CONSTANTS lc_status_open TYPE c VALUE 'O'. 中，lc_status_open 是 named constant。比起在多处直接写 'O'，constant 更容易 review、搜索和修改。
  • 例子 3：'ABC' 和 ` ABC ` 看起来都像字符串，但 ABAP 中前者是 text literal，后者是 string literal。后面学字符串比较和接口 payload 时，要特别注意这个差异。
  • 例子 4：sy-subrc 是系统字段，也是 system-defined data object。它不是你用 DATA 声明的变量，但它在运行时保存上一条语句的结果码。
  • 最小动作：写 3 行代码，分别声明一个 variable、一个 constant、一个 literal 使用场景，并说明 data object 与 data type 各是什么。
  • 进阶动作：在 debugger 中观察一个变量的当前值、type、initial value，以及赋值前后的变化。

调试时可以这样观察 data object：

观察项要看什么
Name变量名是否表达业务含义
Type是内置类型、DDIC 类型、structure 还是 internal table
Current value当前运行时值是什么
Initial valueIS INITIAL 是否成立，业务上是否也代表“未输入”
Conversion赋值或比较前后是否发生隐式转换
Unit / currency金额/数量是否有配套单位或币种

这张表能帮助初学者把 debugger 从“看值”升级为“看类型和值是否符合业务语义”。

20.8 Counterexample / Failure mode

  • 误区 1：把 data object 当 repository object。修正：前者是运行时内存里的值容器，后者是系统开发对象。
  • 误区 2：把 ABAP 变量当 Python 动态变量。修正：普通 ABAP data object 的 type 稳定，转换发生在赋值/表达式/参数传递中。
  • 误区 3：把 initial 当 null。修正：ABAP 有 type-specific initial value，业务“未输入”和技术初始值要分开判断。
  • 误区 4：到处写 literal。修正：业务状态、标志、固定码值应优先考虑 constant、domain fixed value 或集中定义。
  • 误区 5：忽略 quote 差异。修正：单引号 text literal 和反引号 string literal 在类型行为上不同。
  • 误区 6：金额、数量、日期都用 character 接。修正：先确认业务字段语义，再选 data type / DDIC data element / unit / currency。
  • 误区 7：把运行时 data object 当成跨请求缓存。修正：不同 dialog step、work process、job 或 API request 的生命周期不同，需要持久化就用数据库、application log、shared memory 或明确的缓存设计。
  • 误区 8：接口 payload 全部先接成 string。修正：输入解析可以有 string 阶段，但进入业务逻辑前应转换成有业务语义的 typed data object，并处理转换错误。
  • 误区 9：变量名好看但类型错误。修正：变量名只是阅读辅助，真正影响行为的是 type、domain、conversion、unit/currency 和 validation。

20.9 思考问题

  • DATA lv_amount TYPE p DECIMALS 2. 里，data object 和 data type 分别是什么？
  • 为什么 sy-subrc 可以被读，但不是你自己声明的 variable？
  • VALUE 没写时，ABAP 变量为什么仍然有 well-defined initial value？
  • 为什么业务状态值不建议到处写 literal？
