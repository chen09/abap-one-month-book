36. DDIC structure 字典结构

  • Chapter 01
  • Section 36
  • self-contained study note

36.1 一句话结论

DDIC structure 字典结构，是 ABAP Dictionary 中定义全局 structured type 的 repository object。它不会在数据库里创建物理表，也不能被 SELECT；它的价值是把一组属于同一业务或技术语义的 components 做成可复用的全局结构类型，用来给 ABAP 变量、method parameter、internal table line type、接口参数或其他 DDIC 对象提供一致的字段形状。

本节要学的是：把 Structure 结构类型 里学过的“结构”升级到全局 DDIC 层，理解 DDIC structure 和 database table、CDS view entity、local structure 的边界，并知道 flat / nested / deep、include structure、enhancement category、RAP/Fiori 元数据边界这些项目中会被 review 的点。

36.2 到底学什么

先区分三个经常混在一起的东西。

第一，local structure。它是在 ABAP class、method、program 里用 TYPES BEGIN OF ... END OF ... 或 DATA BEGIN OF ... END OF ... 定义的局部结构，只在当前作用域附近使用。Day 03 已经用它帮助理解 ABAP 的 structured data object。

第二，DDIC structure。它是 ABAP Dictionary 里的全局结构类型，存在于 repository 中，有 package、transport、activation、where-used list 和 enhancement category。它不保存数据；它只是定义一组 components 的类型。ABAP 程序可以写：

DATA ls_invoice_item TYPE zxx_invoice_item_s.
DATA lt_invoice_item TYPE STANDARD TABLE OF zxx_invoice_item_s WITH EMPTY KEY.

这里的 zxx_invoice_item_s 可以是一个 DDIC structure。第一行创建一个结构变量，第二行创建一张以这个结构为 line type 的 internal table。

第三，DDIC database table。database table 也有 field list，并且在 ABAP 里也常常能作为 structured type 使用，例如 DATA ls_mara TYPE mara 这种老代码并不少见。但 SAP Learning 明确提醒：database table definition 首要任务是定义数据库对象，虽然也能作为 ABAP 数据类型使用，通常不推荐把它当作“纯类型”复用；如果只是要一个可复用结构类型，应该定义 dedicated structure type。这个建议对初学者很重要：不要为了给 method parameter 一个结构，就随手拿物理表当类型。

第四，CDS view entity / CDS abstract entity 等 CDS artifacts。SAP-samples 提醒，很多 CDS entities 也可以在 ABAP 程序中作为全局 structured type 使用，例如 CDS view entity 可以用于 ABAP TYPES、DATA 或 internal table line type；但反过来，CDS view entity 不能作为 DDIC database table 的字段类型或 DDIC structure 的 component type。进入 RAP 服务层时，CDS view entity / metadata extension / behavior definition 往往比 DDIC structure 更接近服务模型本身。

在 ADT 中，DDIC structure 使用 Dictionary DDL 表示。当前 ABAP Keyword Documentation 和 SAP-samples cheat sheet 以 define structure 作为结构定义的核心形式；一些旧 SAP Developers tutorial 或旧系统模板可能出现 define type。学习时以目标系统 ADT 生成的模板和当前 SAP Help 为准，不要把两个写法在同一项目里混用。一个典型结构可以长这样：

@EndUserText.label : 'Invoice item transfer structure'
@AbapCatalog.enhancement.category : #NOT_EXTENSIBLE
define structure zxx_invoice_item_s {
  invoice_id    : zxx_invoice_id;
  item_no       : abap.numc(6);
  currency_code : abap.cuky;
  @Semantics.amount.currencyCode : 'zxx_invoice_item_s.currency_code'
  net_amount    : abap.curr(15,2);
  status        : zxx_payment_status;
}

这个例子不是数据库表，没有 key client，没有 delivery class，也不会被 SELECT。它只是一个全局 structured type。它可以作为方法输入输出结构、internal table 行类型、接口映射结构，或被其他 DDIC structure / database table 作为 include 的基础。

36.3 为什么重要

DDIC structure 重要，是因为真实项目里大量接口、参数、返回值、internal table line type 和 legacy integration 都不是一个个散落的字段，而是一组有名字、有顺序、有类型、有复用关系的结构。一个好的 DDIC structure 能让多处代码共享同一套字段定义，减少“这里叫 amount，那里叫 net_value，长度还不一样”的混乱。

对 Java/Python 背景学习者，可以把 DDIC structure 粗略类比为一个全局 DTO、record、dataclass 或 interface schema。但这个类比只能帮助入门。SAP 的 DDIC structure 是 repository object，会被 package、transport、activation、where-used list、enhancement category、ABAP Cloud release check 等项目机制管理。它也可以被 DDIC、ABAP、classic Gateway、dynpro 和部分 legacy 工具识别，不只是程序语言里的一个 class。

项目上常见的场景是：一个 BAPI、function module、class method、Gateway service 或文件接口需要传一行发票项目数据。开发者可以在每个方法里局部写一套结构，也可以用一个全局 DDIC structure 统一字段。前者短期快，后者更利于 review、测试、接口文档和影响分析。特别是在日本项目中，テーブル定義書、インターフェース定義書、項目一覧 常常要求明确字段名、data element、桁数、必須、備考、変換规则；DDIC structure 能成为这些文档和代码之间的桥。

它还重要在“不要拿错类型来源”。很多老代码会用 database table 当结构类型，这能工作，但会把 method parameter、internal table line type 和物理表耦合起来。物理表新增字段、调整 include、改变语义时，使用它作类型的代码也可能被牵动。新开发中，如果只是传输数据形状，应优先考虑 dedicated DDIC structure、class/interface 中公开的结构类型，或在 RAP 服务层使用 CDS view entity 的结构，而不是随手绑定到物理表。

36.4 Key mechanism / 核心机制

第一层机制是 global structured type。DDIC structure 定义的是类型，不是值。zxx_invoice_item_s 本身不装数据；DATA ls_item TYPE zxx_invoice_item_s 才是在运行时创建一个结构变量。STANDARD TABLE OF zxx_invoice_item_s 则是创建一张以该结构为行类型的 internal table。

第二层机制是 components。结构由 components 组成，components 可以是 data element、built-in dictionary type、reference type、另一个 structured type、include structure 或 table type。业务字段优先用 data element，这样 label、documentation、domain 等语义能复用；技术字段或教学例子可以用 abap.char(...)、abap.int4、abap.numc(...) 等 built-in dictionary type。

第三层机制是 flat / nested / deep。flat structure 只包含 elementary components，并且不包含 string-like deep components、references 或 table components。nested structure 是结构里包含子结构。deep structure 是包含 string/xstring、reference type、table type 等 deep components 的结构。reference type、object reference、interface reference 或 table type component 都会把结构推向 deep。deep structure 可以作为 ABAP 变量类型、method parameter 类型、internal table line type；但不能作为 database table field type，也不能作为 database table 的 include structure。数据库表里可 include 的结构必须是 flat、兼容的结构。classic RFC/OData serialization 或持久化上下文中也要谨慎使用 deep structure：尤其是 object/interface reference 通常不能作为跨系统数据传输内容，持久化对象也不能随便保存引用；string、xstring 或 table component 是否可用，要看具体接口技术和目标系统限制。

第四层机制是 include structure。DDIC structure 可以被其他 structure 或 database table include，用来复用一组字段。include 时要注意 component name conflict；如果包含的 substructure 里字段名会和外层冲突，可以使用 suffix。SAP-samples 的当前示例/说明展示了 with suffix _in 这种短 suffix，并提到可以指定 three-character suffix；学习时要把它理解成目标 release 语法的一部分，而不是业务别名。实际项目里应以当前系统 syntax check 和项目标准为准，核心目标是让展开后的 component name 唯一、可读、可维护。

第五层机制是 enhancement category。@AbapCatalog.enhancement.category : #NOT_EXTENSIBLE 不是装饰。它说明这个结构是否允许后续 append / extension。对纯技术内部结构，#NOT_EXTENSIBLE 可能合理；对业务字段结构，如果后面可能有客户扩展字段，过早写死不可扩展会给后续变更制造麻烦。项目 review 里要问：为什么选择这个 enhancement category？如果选择 #NOT_EXTENSIBLE，是否接受以后不能通过 append / extension 增加字段？这个理由是否写进设计说明或 transport 备注？

第六层机制是 ADT / syntax version。当前资料里，DDIC structure 常用 define structure 说明。部分旧 tutorial 或旧系统模板可能出现 define type。学习者不要只复制网上片段，而应以当前系统创建对象时 ADT 生成的 skeleton、SAP Help 的目标 release 文档和项目标准为准。遇到语法差异时，不要怀疑自己“SAP 都一样”，而要确认 ABAP release、language version 和工具模板。

第七层机制是 modern UI / RAP 边界。DDIC structure 在 classic ABAP / Gateway / dynpro 时代可能承载很多 UI 辅助语义，但 SAP Learning 对现代 RAP 说得很明确：ABAP Dictionary 的 classic UI 任务不再适用于 modern UI 技术，classical ABAP UI techniques 也不属于 ABAP Cloud 支持范围。RAP/Fiori 服务元数据主要来自 CDS view entity、metadata extension、service definition/binding 和 behavior definition。DDIC structure 可以作为全局类型或 legacy/Gateway import 来源被认识；但在新 RAP/Fiori / ABAP Cloud 开发里，不要把 DDIC structure 当成 UI metadata 的来源，也不要依赖 classic search help、foreign key UI semantics 或 SEGW/Gateway 生成习惯来建模现代服务。

36.5 日本项目现场说法

  • 常见日语：DDIC構造、ディクショナリ構造、構造型、構造コンポーネント、項目、インクルード構造、フラット構造、ネスト構造、深い構造、参照型、テーブル型、拡張カテゴリ、有効化、使用箇所、影響調査、移送対象。
  • 说明句：この DDIC 構造はデータベーステーブルではなく、ABAP で再利用するためのグローバル構造型です。
  • Review 句：物理テーブルを型として流用するのではなく、メソッド引数用の専用 DDIC 構造を定義してください。
  • Include 句：インクルード構造を使う場合、項目名の衝突とサフィックスの指定を確認してください。
  • Deep 句：深い構造は ABAP 変数やメソッド引数では利用できますが、データベーステーブルの項目型やインクルード構造としては利用できません。
  • 拡張句：#NOT_EXTENSIBLE を設定した構造は後から拡張できないため、ビジネス項目を含む構造では拡張カテゴリの選定理由を設計書または移送依頼に残してください。
  • Cloud/RAP 句：RAP/Fiori の UI メタデータは CDS と Behavior 定義を中心に確認します。DDIC 構造のクラシック UI 情報に依存しないでください。
  • 使用场景：接口结构设计、method parameter review、Gateway legacy 调查、RAP 服务建模边界说明、数据库表 include review、移送前影响调查。

36.6 English Terms

  • Official / search term: DDIC structure, ABAP Dictionary structure

  • Related terms: structured type, component, flat structure, nested structure, deep structure, include structure, table type, enhancement category

36.7 Example / 学习例子

例子 1：方法返回结构。

假设一个 class method 要返回“发票项目展示行”。如果直接使用数据库表类型：

METHODS get_item
  RETURNING VALUE(rs_item) TYPE zxx_invoice_i.

这表示 method 返回值和物理表 zxx_invoice_i 的结构强耦合。表里以后新增技术字段、client 字段、审计字段时，method 的语义也会被牵动。更清晰的设计是定义专用 DDIC structure：

METHODS get_item
  RETURNING VALUE(rs_item) TYPE zxx_invoice_item_s.

zxx_invoice_item_s 可以只包含调用方需要的字段，例如 invoice id、item no、amount、currency、status 和 display text。这样接口类型更稳定，也更容易写设计书。

例子 2：internal table line type。

DATA lt_items TYPE STANDARD TABLE OF zxx_invoice_item_s WITH EMPTY KEY.

lt_items = VALUE #(
  ( invoice_id = '1000000001'
    item_no = '000001'
    currency_code = 'JPY'
    net_amount = '1200.00'
    status = 'N' )
).

这里 zxx_invoice_item_s 是每一行的结构类型，internal table 只是“多行容器”。DDIC structure 和 internal table 不是同一个概念：一个定义行长什么样，一个保存多行数据。

例子 3：include structure。

@EndUserText.label : 'Invoice output structure'
@AbapCatalog.enhancement.category : #NOT_EXTENSIBLE
define structure zxx_invoice_output_s {
  header : include zxx_invoice_header_s with suffix _hd;
  item   : include zxx_invoice_item_s   with suffix _it;
}

这个例子说明 include 可以复用已有结构，但要小心字段名冲突。suffix 让包含进来的字段带上后缀，避免多个 include 都有 created_by、currency_code 之类同名字段。真实项目里是否应该 include，取决于结构复用是否清晰；不要为了少写几行字段，把语义不同的结构硬拼在一起。

最小练习：找一个已有 DDIC structure，写下：

  • 它是 flat、nested 还是 deep？
  • components 用了哪些 data element / built-in type？
  • 有没有 include structure？有没有 suffix？
  • enhancement category 是什么？为什么？
  • 它被哪些 class、method、table type、Gateway/RAP/CDS 对象使用？

36.8 Counterexample / Failure mode

  • 误区 1：把 DDIC structure 当成数据库表。
  • 错误理解：有 field list，就能 SELECT * FROM zxx_invoice_item_s。
  • 修正：DDIC structure 是类型，不是 persistence object；能不能 SELECT 看的是 database table、CDS view entity 等数据源。
  • 误区 2：随手用 database table 当 method parameter type。
  • 问题：这会把业务接口和物理表耦合，后续表调整可能影响接口语义。
  • 修正：只需要数据形状时，优先考虑 dedicated DDIC structure、class/interface 公开结构类型，或 RAP/CDS 场景中的 CDS view entity。
  • 误区 3：把 deep structure include 到 database table。
  • 问题：包含 string、reference、table type 等 deep components 的结构适合 ABAP 内存类型，不适合直接塞进数据库表 include。
  • 修正：数据库表 include 使用 flat、兼容的结构；deep structure 用于 method parameter、internal table line type 或非持久化场景。
  • 误区 4：忽略 enhancement category。
  • 问题：#NOT_EXTENSIBLE 可能阻止后续 append / extension；选择错误会变成项目变更问题。
  • 修正：创建结构时就说明是否允许扩展，并把理由写进设计说明或 review 记录。
  • 误区 5：把 DDIC structure 当成 RAP/Fiori UI metadata 主来源。
  • 问题：modern RAP/Fiori 的 UI/service metadata 主要来自 CDS view entity、metadata extension、service binding、behavior definition。
  • 修正：DDIC structure 作为全局类型理解即可；RAP 服务要回到 CDS/RAP 模型检查。

36.9 思考问题

  • DDIC structure、local structure、database table、CDS view entity 都能在某些场景表达“结构”，它们的主要差异是什么？
  • 为什么 SAP Learning 不推荐把 database table definition 当成普通 reusable type 使用？
  • 什么是 flat / nested / deep structure？哪些 components 会让结构变 deep？
  • 为什么 deep structure 不适合作为 database table include？
  • @AbapCatalog.enhancement.category : #NOT_EXTENSIBLE 对后续项目扩展有什么影响？
  • 在日本项目里，什么时候说 構造型，什么时候说 項目，什么时候说 構造コンポーネント 更自然？
  • 如果一个 DDIC structure 被多个接口、class method 和 table type 使用，修改字段前要做哪些 影響調査？
