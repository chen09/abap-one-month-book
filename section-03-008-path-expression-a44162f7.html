8. Path expression 路径表达式

  • Chapter 03
  • Section 08
  • self-contained study note

8.1 一句话结论

Path expression 路径表达式，是在 CDS 或 ABAP SQL 中沿着已暴露的 association 读取目标字段、继续导航到下一级 association，或在特定位置使用 association 作为数据源的语法。它不是普通的对象属性访问，也不是“免费取关联数据”；它是在消费 Association 关联关系，并且在数据库层通常会落实成由 association 的 ON condition 派生出来的 join。

本节最重要的结论是：path expression 是 association 从“模型关系”变成“当前消费”的那一刻。上一节 Join vs association 二者区别 讲过，association 可以先定义和暴露而不立即取 target 字段；一旦写出 _Customer.Name、_Department._Head.LastName 或 ABAP SQL 中的 \_Airline-Name，消费者就明确要求沿着关系读取数据。项目上 review path expression，不能只看写法短不短，还要看 cardinality、to-many 行数变化、filter 是否改变语义、DCL/authorization 是否仍然生效、projection/service 是否真的应该暴露这个导航。

8.2 到底学什么

前两节已经建立了三个概念：association 定义关系，join 立即组合数据源，exposed association 把关系交给消费者。本节只回答一个问题：消费者如何沿着这个关系取字段？

需要掌握八个判断点。

  • CDS path expression：在 CDS view entity 里用 _Association.Field 或 _Association._NextAssociation.Field 读取关联目标字段。
  • ABAP SQL path expression：在 ABAP SQL 里用反斜线 \ 标记每个 association，用最后一个连字符 - 选择目标字段，例如 \_Airline-Name 或 chained path \_Airline\_Currency-CurrencyISOCode。
  • Propagate association：只把 association 名称放进元素列表，让下游消费者继续看到这个关系。
  • Use association：在当前 view / query 里沿着 association 取字段，此时会触发关联访问。
  • Chained path：连续跨多个 association，例如 _Department._Head.LastName。
  • To-one vs to-many：to-one path 通常保持当前行粒度；to-many path 可能放大结果行数，甚至在 WHERE 中被禁止。
  • Filtered path：在 association 后用 [...] 给目标集合加过滤条件，过滤条件会影响生成 join 的 ON condition。
  • Verification：在 ADT 中用 Data Preview 和 Show SQL Create Statement 观察 path expression 对结果集与生成 SQL 的影响。

8.3 为什么重要

Path expression 重要，是因为它把“可导航关系”真正带入消费层。没有 path expression，association 只是模型中可见的一条关系；有了 path expression，当前 view、ABAP SQL、projection、service 或 OData consumer 才开始拿目标实体的字段。也就是说，path expression 是连接 CDS 语义模型和实际查询结果的桥。

对 SAP 项目交付来说，这一层很容易出问题。比如部门 view 中沿着 _Employee 读取员工姓名，看起来只是多写了一个字段，但如果 _Employee 是 [0..*]，一个部门可能突然变成多行。再比如为了取文本写 _Currency._Text.CurrencyName，如果没有过滤语言，结果可能按语言数放大。又比如在 projection view 或 service 暴露前没有想清楚，外部 OData consumer 可能依赖某个导航关系，后续删除 association 就变成 API breaking change。

对 Java/Python 背景学习者，可以把 path expression 粗略理解为“沿着 model relationship 取 related entity 的字段”，有点像 ORM 中的 order.customer.name 或 GraphQL selection path。但 SAP 里的差异更强：它不是内存对象导航，而是 CDS / ABAP SQL 语法；它受 association cardinality、DCL、activation、SQL generation、projection view、service definition、OData metadata 和数据库执行计划约束。写起来像点号访问，实际是在声明数据库访问和服务模型消费。

8.4 Key mechanism / 核心机制

8.4.1 Propagate association 和 use association 要分开

SAP Learning 对 path expression 的第一组区分很关键：当一个 CDS view entity 读取另一个带有 exposed association 的 entity 时，可以选择传播 association，也可以选择使用 association。

传播 association，是把 association 名称放进当前 view 的元素列表：

define view entity ZC_SalesOrder
  as select from ZI_SalesOrder
{
  key SalesOrder,
      CustomerId,
      _Customer
}

这里 _Customer 对 ZC_SalesOrder 的消费者可见，但当前 view 没有读取 customer 的字段。换句话说，它把导航关系继续交给下游。

使用 association，是在 association 名称后加目标字段：

define view entity ZC_SalesOrder
  as select from ZI_SalesOrder
{
  key SalesOrder,
      CustomerId,
      _Customer.CustomerName as CustomerName
}

这里 _Customer.CustomerName 就是 path expression。当前 view 明确要求读取 association target 的 CustomerName。SAP Learning 说明，在数据库层 path expression 会被翻译成 source 和 target 之间的 join，join 的 ON condition 来源于 association 的 ON condition。

所以读代码时，先问一句：这里是在 propagate association，还是 use association？只传播关系时，消费者看到的是“可以导航”；使用关系时，当前结果集已经包含目标字段。

8.4.2 Path expression 是 join on demand 的触发点

Association 的一个核心特性是 join on demand。只定义 association、暴露 association、甚至让上层 view 继续传播 association，都不等于立刻生成目标表 join。只有当 association 被 path expression 消费，或者在服务导航等消费场景中被真正使用时，底层才需要访问目标。

可以用三个小例子理解。

第一，只定义 association：

association [0..1] to ZI_Customer as _Customer
  on $projection.CustomerId = _Customer.CustomerId

这只是模型关系。

第二，暴露 association：

{
  key SalesOrder,
      CustomerId,
      _Customer
}

这让消费者看到 _Customer，但没有取 customer 字段。

第三，使用 path expression：

{
  key SalesOrder,
      CustomerId,
      _Customer.CustomerName as CustomerName
}

这时当前 view 要 customer name，数据库层需要相应访问。SAP-samples 的 CDS association 示例也用注释说明：只暴露 association 时，Data Preview 不显示 target 字段，Show SQL Create Statement 也不显示对应 join；当消费者通过 path expression 请求 target 字段时，才会执行 join。

8.4.3 CDS 中用点号，ABAP SQL 中用反斜线和连字符

CDS DDL 里的 path expression 使用点号：

_Customer.CustomerName
_Department._Head.LastName

第一个例子沿着 _Customer 取客户名。第二个例子是 chained path expression：先从员工到部门，再从部门到部门负责人，最后取负责人姓氏。SAP Learning 明确给出类似 _Department._Head.LastName 的例子，并说明复杂的多数据源 join 会自动生成。

ABAP SQL 里的写法不同，因为 ABAP 语句用句点 . 结束语句，不能把点号当路径分隔符。ABAP SQL 使用反斜线标记 association，用连字符选择最终字段：

SELECT FROM ZI_SalesOrder
       FIELDS SalesOrder,
              CustomerId,
              \_Customer-CustomerName AS CustomerName
       INTO TABLE @DATA(lt_orders).

如果要继续沿着下一层 association，仍然给每个 association 写反斜线前缀，association 名称直接相邻，最后才用连字符取最终字段：

SELECT FROM ZI_Employee
       FIELDS EmployeeId,
              \_Department\_Head-LastName AS DepartmentHeadName
       INTO TABLE @DATA(lt_employees).

注意这里不是 \_Department-\_Head-LastName。ABAP SQL chained path 的 association 之间没有专门的分隔符；每个 association 名称自己带 \，最后一个连字符才把路径和目标字段分开。

实际项目里常见误读是把 CDS 写法搬到 ABAP SQL，写成 _Customer.CustomerName。这在 ABAP SQL 中不是正确的 path expression 语法。教学时要让读者记住：CDS 是 .；ABAP SQL 中每个 association 都带自己的 \ 前缀，association 名称之间没有额外分隔符，只有最后的 -field 选择目标字段。

8.4.4 To-many path expression 会改变结果行数

Path expression 最大的风险不是语法，而是结果集粒度变化。To-one association 通常表示一条 source 记录最多对应一条 target；to-many association 则可能让一条 source 记录对应多条 target。把 to-many association 放进 path expression，当前结果集就可能被放大。

一个典型例子：

define view entity ZC_Department
  as select from ZI_Department
{
  key DepartmentId,
      DepartmentName,
      _Employee.LastName as EmployeeLastName
}

如果 _Employee 是 [0..*]，一个部门有 5 名员工，那么原本一行部门数据可能变成 5 行。SAP Learning 对这个点给得很明确：to-many association 在 CDS view definition 的 WHERE clause 中用作 path expression 不允许，会导致语法错误；在元素列表中允许，但编辑器会显示 warning，提醒这个 path expression 可能改变结果集数量。

初学者要把 warning 当成设计问题，而不是“能激活就算通过”。项目 review 时要追问：

  • 当前 view 的粒度还是“每个部门一行”吗？
  • 消费方是否知道这个字段会把行数放大？
  • 是否应该从 employee 侧出发，用 to-one _Department 读取部门？
  • 是否应该使用 aggregation 和 GROUP BY，而不是直接暴露 to-many 字段？
  • 是否需要 filter 把目标集合收窄到 one row？
8.4.5 To-many 可以在 aggregation 或带明确过滤时被控制

To-many association 不是完全不能用。关键是要让结果集语义可解释。

如果要从部门取员工数量，不应该直接把 _Employee.LastName 放进元素列表，而应该表达聚合：

define view entity ZC_DepartmentEmployeeCount
  as select from ZI_Department
{
  key DepartmentId,
      DepartmentName,
      count( _Employee.EmployeeId ) as EmployeeCount
}
group by
  DepartmentId,
  DepartmentName

这里重点不是具体函数写法，而是思路：to-many 被 aggregation 吸收，结果仍然可以保持“一部门一行”。

另一种方式是过滤目标集合。Path expression 可以在 association 后加方括号：

_Currency._Text[ 1: Language = 'E' ].CurrencyName as CurrencyNameEn

SAP Learning 说明，filter condition 放在 [...] 中，位置紧跟 association 名称；类似 WHERE 条件，可以使用比较运算和布尔运算。技术上，filter condition 会被加入生成 join 的 ON condition。对于 to-many association，如果 filter 实际上保证只剩一条记录，可以在 CDS path expression 中用 1: 记录这个“过滤后按 to-one 使用”的意图，减少或避免相关 warning。

需要注意：1: 是 CDS path expression 中的语法，用来声明过滤后按 to-one 使用并影响语法检查；它不是数据库唯一性约束，也不会替开发者验证数据真的唯一。ABAP SQL path expression 不使用 1:，而是使用 MANY TO ONE WHERE、ONE TO ONE WHERE 等关键字形式来表达相应语义。过滤条件、数据模型和测试证据必须真的支持“一条”；如果语言文本表里同一语言仍可能多条，1: 只是掩盖问题。

8.4.6 Filter condition 的左侧必须来自 association target

过滤 path expression 时，不要把它当作随意写的 WHERE。SAP Learning 与 SAP-samples 都强调，path filter 的条件被加到 generated join 的 ON condition 中；SAP-samples 示例还说明，条件左侧 operand 必须是 association target 的字段。

示意：

_Currency._Text[ 1: Language = 'E' ].CurrencyName

这里 Language 是 _Text 目标中的字段。右侧可以是 literal、目标字段、函数等更灵活的表达式，具体可用范围要以当前 ABAP release 的 ABAP Keyword Documentation 为准。

项目 review 时要看两个问题。

  • filter 是在限制 association target，还是把业务筛选偷偷塞进关系里？
  • filter 改变 cardinality 后，是否有测试数据证明它真的从 to-many 变成 to-one？

如果 filter 用来表达“只取当前登录语言文本”，这通常是合理的模型消费；如果 filter 用来隐藏不想处理的业务状态，就可能让 CDS 模型变得难以解释。

8.4.7 Path expression 可以出现在不同位置，但限制不同

Path expression 不只出现在元素列表。根据 ABAP CDS / ABAP SQL 的上下文，它可以出现在 SELECT list、WHERE、FROM、ORDER BY 或其他 operand position。不同位置限制不同，尤其是 to-many association。

初学阶段不需要背完所有位置规则，但要记住三条安全原则。

  • 元素列表中使用 to-many path，要预期结果行数变化，并处理 warning。
  • WHERE 中使用 to-many path，在 CDS view definition 场景不允许，会直接语法错误。
  • FROM 中使用 association 作为数据源时，已经不是“取一个字段”，而是把 association target 作为当前查询的数据源之一，要单独 review 结果粒度和 join 语义。

SAP-samples 的示例也提到，可以在 FROM clause 中通过 path expression 使用 association 作为 data source；这种用法适合后续进阶，不是本节初学主线。对初学者，本节重点是：先掌握元素列表和 ABAP SQL FIELDS 中最常见的字段读取，再理解 WHERE/to-many/filter 的风险。

8.4.8 Path expression 不替代授权、行为、锁和服务契约

Path expression 只是沿着 association 取数。它不会自动完成以下事情：

  • 不会替代 CDS access control 访问控制 或 DCL。
  • 不会替代 RAP global / instance authorization。
  • 不会替代 RAP behavior、validation、determination、locking、save sequence。
  • 不会自动把内部关系变成稳定 OData API。
  • 不会绕过 Clean Core / released API 要求。

尤其在服务暴露场景，path expression 可能让字段进入 projection view 或 OData entity；exposed association 可能形成 navigation property；这会变成外部消费契约的一部分。后续删除字段、改名、收回 navigation，都可能影响前端或外部 API。因此，path expression 不是“我在 CDS 里方便拿个字段”的小事，而是模型消费和服务契约的一部分。

8.4.9 Path expression evidence checklist

Review path expression 时，至少保留下面几类证据，避免“看起来只是点一下关联”却改变结果或服务契约：

EvidenceWhat it should prove
Data Preview samplepath 字段值来自预期 target，空目标和重复目标行为可解释。
Cardinality checkto-one / to-many 与真实数据一致，结果行数没有意外放大。
Generated SQL / tracepath 被消费后生成的 join / access 方式可解释。
Filter reviewfiltered path 的条件确实作用在 association target 上，语言/有效期等过滤完整。
DCL / authorization checksource 与 target 数据都没有绕开应有授权边界。
Service metadata / API test如果进入 OData/RAP service，字段或 navigation 是否成为外部 contract 已确认。

一个简单判断是：path expression 如果只用于内部派生字段，重点看结果和性能；如果进入 projection/service，重点还要看 metadata、authorization 和 consumer compatibility。

8.5 日本项目现场说法

常见日语：

  • パス式：path expression。
  • アソシエーションを利用する：使用 association。
  • アソシエーションを公開する / 公開アソシエーション：expose / exposed association。
  • 関連先項目：association target field / related target field。
  • 結合が生成される：join is generated。
  • 件数が増える / 結果件数が変わる：row count increases / result count changes。
  • to-many のアソシエーション：to-many association。
  • フィルタ条件：filter condition。
  • SQL CREATE Statement を確認する：check generated SQL create statement。
  • ナビゲーションプロパティ：navigation property。

项目说明句：

  • パス式は、公開されたアソシエーションを利用して関連先の項目を取得するための書き方です。
  • アソシエーションを公開しただけでは関連先項目は取得されません。パス式で項目を参照した時点で結合が生成されます。
  • to-many のアソシエーションを要素一覧のパス式で使用すると結果件数が変わる可能性があります。WHERE句では構文エラーになるため、配置場所を確認してください。
  • 1: はCDSパス式の構文チェック上の宣言であり、データベースレベルで一意性を保証するものではありません。データ上も一意になることを確認してください。
  • 外部 API に出す項目やナビゲーションはサービス契約になるため、projection view と service definition の範囲を確認してください。

使用场景：

  • 设计说明：说明为什么某个字段通过 association target 取得，而不是直接 join。
  • 代码 review：检查 to-many path、filter、cardinality、DCL 和 service boundary。
  • 单体测试：准备一对一、一对多、无 target、多语言文本等测试数据。
  • 性能调查：用 Show SQL Create Statement、SQL Trace 或 PlanViz 观察生成 join。
  • API 设计：判断 path expression 字段是否应该进入 projection / OData 服务契约。

8.7 Example / 学习例子

8.例 1：在 CDS 中读取客户名称

先有一个订单 interface view，暴露客户 association：

define view entity ZI_SalesOrder
  as select from zso_header as Header
  association [0..1] to ZI_Customer as _Customer
    on $projection.CustomerId = _Customer.CustomerId
{
  key Header.sales_order as SalesOrder,
      Header.customer_id as CustomerId,
      Header.created_at  as CreatedAt,

      _Customer
}

再在消费 view 中读取客户名称：

define view entity ZC_SalesOrderList
  as select from ZI_SalesOrder
{
  key SalesOrder,
      CustomerId,
      _Customer.CustomerName as CustomerName,
      CreatedAt
}

读者要能说清楚：ZI_SalesOrder 暴露 _Customer；ZC_SalesOrderList 使用 _Customer.CustomerName；生成 join 的位置是消费 target 字段的地方。

8.例 2：在 ABAP SQL 中读取 association target 字段
SELECT FROM ZI_SalesOrder
       FIELDS SalesOrder,
              CustomerId,
              \_Customer-CustomerName AS CustomerName
       WHERE CreatedAt >= @lv_from_date
       INTO TABLE @DATA(lt_orders).

这里的语法重点是：

  • \_Customer：ABAP SQL 中 association 前要用反斜线。
  • -CustomerName：ABAP SQL 中只在最终字段前用连字符；chained association 之间不要加连字符。
  • 这不是普通结构字段访问，而是 SQL path expression。
8.例 3：to-many path 的行数风险
define view entity ZC_DepartmentEmployee
  as select from ZI_Department
{
  key DepartmentId,
      DepartmentName,
      _Employee.LastName as EmployeeLastName
}

如果 _Employee 是 [0..*]，这个 view 的粒度可能从“每个部门一行”变成“每个部门员工一行”。这可能是需求，也可能是 bug。正确动作不是忽略 warning，而是把粒度写进设计说明，并用测试数据验证。

8.例 4：带语言过滤的文本读取
define view entity ZC_CurrencyText
  as select from ZI_Currency
{
  key Currency,
      _Text[ 1: Language = 'E' ].CurrencyName as CurrencyNameEn
}

这个例子体现两个点：_Text 原本可能是 to-many，因为同一币种有多语言文本；filter 让当前消费只取英语文本；1: 是 CDS path expression 中声明过滤后按 to-one 使用的写法。项目上还要确认语言字段和数据唯一性，不要只因为 warning 消失就认为模型正确。若同样思想放到 ABAP SQL path expression，要改用 MANY TO ONE WHERE / ONE TO ONE WHERE 等关键字形式，而不是写 1:。

8.8 Counterexample / Failure mode

  • 把 path expression 当普通属性访问：看到 _Customer.Name 就以为没有数据库代价，忽略生成 join。
  • 只暴露 association，却以为 target 字段已经被读取：Data Preview 中看不到 target 字段时误判为 association 失效。
  • 在 to-many association 上直接取字段：部门变多行、订单变多行、报表合计重复。
  • 在 WHERE 中使用 to-many path：在 CDS view definition 中不允许，会直接语法错误。
  • 用 1: 掩盖数据问题：filter 后并不唯一，却声明为 to-one。
  • 在 projection/service 中随手暴露 path 字段：外部 OData consumer 开始依赖该字段，后续变更困难。
  • 用 path expression 绕过模型边界：直接沿着内部 association 取未释放对象字段，违反 Clean Core / released API 约束。
  • 忘记 DCL/authorization：以为通过 association 取字段就自然有权限控制。

修正动作：

  • 回到 association 定义，检查 source、target、ON condition、cardinality。
  • 用 ADT Data Preview 看实际结果行数。
  • 用 Show SQL Create Statement 看是否生成 join。
  • 为 to-many 准备至少 0、1、多条 target 的测试数据。
  • 如果是服务暴露字段，检查 projection view、service definition、service binding 和 OData metadata。

8.9 思考问题

  • 当前 path expression 是在 propagate association，还是 use association？
