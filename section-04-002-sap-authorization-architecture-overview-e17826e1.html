Chapter 04 - Project Delivery And Production Readiness

2. SAP Authorization Architecture Overview

  • Chapter 04
  • Section 02
  • self-contained study note

2.1 一句话结论

SAP authorization architecture 不是一个单点开关，而是一组分层机制：用户通过 role 获得 authorization object / field value，ABAP 代码可以用 AUTHORITY-CHECK 经典授权检查 做显式检查，CDS 可以用 CDS Access Control DCL 行级访问控制 限制读取结果，RAP 可以用 RAP global authorization 全局授权 和 RAP instance authorization 实例授权 控制业务对象操作，而 SU53 / authorization trace 用来调查失败证据。

本节的目的不是把所有授权技术一次讲完，而是给 Week 4 建一张地图。读者看到“读不到数据”“按钮不可用”“API 返回 401 / 403”“保存时报无权限”“测试用户和接口用户表现不同”时，能先判断问题大概发生在哪一层，而不是只说“找 Basis 加权限”。

2.2 到底学什么

本节学习七个层次。

第一，用户与角色层。SAP 用户不是因为“用户名属于某部门”就自动能操作数据，而是通过角色、授权对象和字段值获得权限。角色维护通常由权限/Basis/安全团队负责，但开发者必须能说清楚自己的程序需要检查什么对象、什么字段、什么动作、什么组织维度。

第二，经典 ABAP 代码层。AUTHORITY-CHECK 是显式写在 ABAP 代码里的检查，常见于 classic report、function module、class、exit、BAdI、接口实现和一些复用工具类。它适合保护特定功能入口、动作或组织维度，但不自动过滤 SQL 结果。

第三，CDS 读取层。DCL 让 CDS view entity 的读取结果受到访问控制影响。它更接近“数据行可见性”的问题：同样的 CDS，用户 A 能看到某些数据，用户 B 看不到。它不是 UI 装饰，也不是 RAP action 的全部授权。

第四，RAP 行为层。RAP global authorization 判断某类操作对当前用户是否总体允许；instance authorization 判断某个对象实例上的操作是否允许。例如“用户是否可以执行 approve 动作”和“这张单据当前状态是否允许这个用户 approve”通常不是同一个问题。

第五，诊断与证据层。SU53、authorization trace、应用日志、API 测试结果、测试用户矩阵、错误消息和截图/请求响应证据，构成权限缺陷调查的材料。生产项目里，授权问题不能只靠口头描述。

第六，入口与用户类型层。同一个业务功能，可能被 SAP GUI、Fiori、OData/API、后台 job、外部系统或测试工具调用。对话用户、测试用户、communication user、background user、batch user 的授权策略不应混在一起。初学者要特别注意：UI 上按钮隐藏，并不等于 API 直接调用也被后端拒绝；接口用户能调用，也不等于普通业务用户应该拥有同样权限。

第七，设计与审计层。授权不是“开发完再让权限团队补角色”的收尾动作，而是设计阶段就要写清楚的交付物：哪些业务动作需要授权、哪些组织维度参与判断、哪些角色可以看到哪些数据、失败时返回什么消息、如何测试无权限路径、上线后如何用 trace 和日志证明行为正确。

2.3 为什么重要

授权问题之所以难，不是因为每个语句都复杂，而是因为它跨团队、跨层、跨环境。开发者写 ABAP，安全团队维护 role，测试团队准备用户，业务顾问定义岗位，Basis 处理系统参数和 trace，接口团队管理 communication user 或 technical user。任何一层说不清，缺陷就会在团队之间来回转。

典型例子是“用户读不到数据”。可能原因至少有六种：CDS DCL 过滤了结果；PFCG role 缺少授权对象字段值；RAP instance authorization 拒绝了实例；服务绑定或业务角色没有暴露对应应用；查询条件本身没有命中数据；接口用户和对话用户不是同一个授权模型。没有架构地图，就很容易把所有问题都归因于“权限不够”，然后补错地方。

从 Java/Python 背景看，可以把 SAP 授权粗略类比成 RBAC、policy check、row-level security 和 service guard 的组合。但 SAP 项目里，授权对象、角色字段值、业务组织、事务码/应用入口、CDS/RAP 框架、trace 证据和上线审计经常一起出现。因此学习 Week 4 时，要从“哪条语句怎么写”上升到“这个访问路径由哪几层共同保护”。

这也是 Clean Core 项目特别强调授权设计的原因之一。扩展代码可以调用 released API，可以用 RAP 暴露服务，可以通过 CDS 建模数据，但只要授权边界没有说明清楚，系统就会在上线前暴露出两个相反风险：该拦的不拦，造成过授权和审计风险；该放的放不过，造成业务中断和缺陷返工。一本面向项目实战的 ABAP 书，不能只教 AUTHORITY-CHECK 语法，还要教读者如何把授权从需求、设计、实现、测试、缺陷调查一直串起来。

2.4 Key mechanism / 核心机制

可以把授权路径想成一条链。

User
  -> Role / business role / profile
  -> Authorization object + field values
  -> Runtime check point
       - ABAP AUTHORITY-CHECK
       - CDS DCL
       - RAP global authorization
       - RAP instance authorization
       - Framework / app / service exposure checks
  -> Result
       - allowed
       - denied with error
       - data filtered
       - action hidden or disabled
       - no data because query/service path differs

这条链里最关键的是“runtime check point”。同一个 role 可能被不同层以不同方式消费：ABAP 代码用 AUTHORITY-CHECK 显式判断，CDS DCL 在读取时应用访问控制，RAP 在 behavior implementation 或框架授权入口中判断操作是否可用，Fiori / service exposure 还可能影响用户是否看得到应用或服务。

因此，设计授权时要先问四个问题。

  1. 保护对象是什么：功能、数据行、业务对象实例、action、保存动作、接口入口，还是后台作业？
  2. 保护维度是什么：动作类型、公司代码、工厂、销售组织、用户归属、状态、审批责任人，还是技术通信用户？
  3. 检查位置在哪里：classic ABAP、CDS DCL、RAP authorization、precheck、业务 API、服务配置，还是多层组合？
  4. 失败证据在哪里：sy-subrc、RAP failed/reported、OData response、SU53、authorization trace、application log，还是测试用例结果？

一个实用规则是：功能入口和动作权限常由 ABAP/RAP 检查；数据可见性常要看 DCL 或查询条件；对象状态相关的“某一笔能不能操作”常要看 RAP instance authorization 或业务规则；权限缺失诊断要看 SU53 / trace，但不能只看诊断工具就跳过代码路径分析。

2.4.1 Authorization hierarchy and review matrix

初学者还要避免把 role、profile、authorization object、authorization field 混成一个词。可以先记住这条层级：

Authorization Object
  -> Authorization Field
  -> Field Value
  -> Authorization maintained in a role/profile
  -> Role / business role assigned to user
  -> Runtime check consumes that authorization

例如，一个 FI 相关检查可能关心授权对象、ACTVT、公司代码 BUKRS 等字段值；一个接口入口还可能同时受到服务暴露、technical user、S_RFC / S_ICF、业务对象授权和 DCL 的影响。开发者不需要替代权限团队维护角色，但必须能把“代码检查什么、角色需要给什么、测试如何证明”说清楚。

权限矩阵还要显式点名两类横向风险。第一是 over-broad field values：例如 ACTVT=*、BUKRS=*、过宽的 S_RFC / S_ICF 或接近 SAP_ALL 的 profile，会让测试“通过”但无法证明最小权限。第二是 SoD / segregation of duties conflict：同一用户同时拥有申请、审批、过账、取消等互相牵制的权限时，代码层面可能没有报错，但审计层面仍然不可接受。

Joule review 特别提醒：授权架构要覆盖访问入口，而不是只覆盖一个 ABAP 语句。可以把项目 review 写成下面这张矩阵：

这张矩阵里既有 preventive control，也有 detective evidence。前者负责阻止不该发生的访问；后者负责在测试、审计和障害調査时证明发生了什么。日本项目里，権限設計書 不应只列 role 名称，还应能对应到测试用户、操作入口、授权对象/字段值、负向测试、trace 或日志证据，以及 DEV/QAS/PRD 移送后一致性确认。

还要特别强调三条红线。第一，Fiori tile、按钮隐藏、菜单不可见不是完整安全边界；OData/RFC/API 直接调用仍要被后端授权拦住。第二，CDS DCL 主要解决读取结果过滤，不替代写操作、action、BAPI/RFC 或保存阶段授权。第三，technical / communication user 不是“超级用户”；它也应遵守最小权限、可审计、可追踪、可回收的原则。

2.5 日本项目现场说法

  • 常见日语：権限設計、権限オブジェクト、権限項目、ロール、PFCGロール、SU53、権限トレース、行レベル制御、インスタンス単位の権限。
  • 设计说明：本機能の権限制御は、一覧取得は CDS DCL、更新処理は RAP の authorization、既存処理呼び出しは AUTHORITY-CHECK の結果を確認する方針です。
  • 缺陷调查：権限不足だけでなく、DCL によるデータ絞り込み、RAP の instance authorization、ロール設定、検索条件を切り分けて確認します。
  • 对权限团队说明：開発側では必要な権限オブジェクト、項目値、テストユーザ、再現手順、SU53 または trace 証跡を提示します。
  • Review 说法：UI の表示制御だけでなく、API 直接呼び出し時にも同じ権限制御が効くか確認してください。
  • 设计确认：誰が、どの業務オブジェクトに対して、どの操作を、どの組織条件で実行できるかを権限設計書に明記します。
  • 接口说明：技術ユーザの権限は最小権限とし、API 呼び出し時の認証・認可・監査ログを確認します。
  • 缺陷分派：本件はロール設定、DCL、RAP authorization、業務ステータスのどこで拒否されたかを切り分けます。

2.7 Example / 学习例子

场景一：用户能打开 Fiori 应用，但列表为空。

不要马上判断“没有数据”。先切分：

  • 服务是否发布、用户是否能访问应用或 catalog？
  • CDS 查询本身是否有数据？
  • DCL 是否把数据过滤掉？
  • 用户 role 是否缺少组织字段授权？
  • 如果通过 API 直接查，结果是否和 UI 一致？
  • 是否有 trace 或日志能证明授权检查点？

场景二：用户能看到列表，但点击 Approve 动作时报无权限。

这更可能不是 DCL 的问题，因为列表已经可见。需要看 RAP action 的 global authorization、instance authorization、对象状态、审批责任人、后端实现里的 classic check，以及错误消息是否把原因表达清楚。

场景三：接口用户能上传数据，但普通用户不能。

这时要区分 dialogue user、communication user、background user 或 technical user 的授权策略。服务账号不应该被理解为“无需授权检查”，而应有清楚的最小权限、调用范围、日志、监控和审计说明。

场景四：测试用户能执行 Validate，但执行 Post 或 Approve 失败。

这说明“能访问服务”和“能执行业务动作”不是同一层权限。Validate 可能只做输入检查、模拟校验或只读检查；Post/Approve 可能触发状态变更、会计影响、锁、保存和后续消息。此时要把 operation 拆开列矩阵：read、validate、execute、approve、cancel、status query 分别由哪些 role、哪些 authorization object、哪些 RAP authorization 或 classic check 控制。

场景五：客户要求“给项目组临时全权限，先测试完再收回”。

这是很多项目会遇到的现实压力，但它会让测试结果失真。全权限用户测试通过，不能证明真实业务角色可用；全权限用户没有失败路径，也不能证明限制有效。更好的做法是准备至少三类测试用户：有完整权限的正向用户、缺少关键组织值的边界用户、完全无权限或只读用户。API 场景还应准备 technical user 的最小权限测试。

2.8 Counterexample / Failure mode

常见失败模式有十类。

第一，只补 role，不看代码。权限团队补了 PFCG role，但 ABAP 代码的 AUTHORITY-CHECK 检查的是另一个对象或字段值，问题仍然存在。

第二，只看 SU53，不看调用路径。SU53 对定位最近一次授权失败有帮助，但它不能自动解释 CDS DCL、RAP authorization、查询条件、服务暴露和 UI feature control 的全部逻辑。

第三，把“读不到数据”和“无权限操作”混在一起。列表为空、按钮灰掉、API 401、API 403、保存失败、后台 job 报错，背后的检查点可能完全不同；401 通常先看认证和身份，403 通常再看授权、role、服务暴露和后端检查。

第四，用硬编码绕过权限。开发阶段为了测试方便跳过检查，上线前忘记恢复，会直接破坏审计和合规。

第五，接口账号权限过大。为了“先跑通”，给 communication user 或后台用户过宽权限，甚至使用 ACTVT=*、组织字段 * 或近似 SAP_ALL 的 profile，后续很难证明最小权限原则。

第六，只测有权限用户。授权设计必须包含无权限用户、边界组织、跨公司代码、不同动作类型和 API 直接调用，否则无法证明限制有效；涉及审批、过账、取消等互斥职责时，还要确认是否存在 SoD conflict。

第七，把 UI 控制当作后端授权。按钮隐藏、字段只读、菜单不可见可以改善体验，但不能作为唯一安全边界；真正重要的检查必须在后端服务、RAP behavior、DCL 或 ABAP 代码路径中生效。

第八，把通信用户当成超级用户。为了接口方便给 technical user 过宽权限，会让外部系统绕过业务角色边界，后续审计时很难解释。

第九，忽略负向测试。只证明“应该能做的人能做”，没有证明“不应该能做的人不能做”，授权测试就不完整。

第十，缺陷修复后不补证据。权限问题修复后要保存修复前后证据：角色变更、trace/SU53、API response、测试用户矩阵、回归结果。否则上线后同类问题很难追溯。

2.9 思考问题

  • 当前问题是“看不到数据”“不能执行动作”“不能保存”，还是“不能进入应用”？
  • 这个控制点应该在 classic ABAP、CDS DCL、RAP authorization、precheck、服务配置还是 role 维护？
  • 角色里需要哪些 authorization object、authorization field 和组织字段值？
  • SU53 或 trace 能证明什么？它不能证明什么？
  • 如果 API 绕过 UI 直接调用，授权是否仍然有效？
  • 在 AC0567C 这类上传 API 中，普通业务用户、测试用户和技术接口用户的权限边界是否清楚？
  • 当前设计有没有负向测试用户？有没有跨组织、跨状态、API 直调和后台 job 场景？
  • 如果客户要求临时全权限测试，哪些测试结论会因此失真？
