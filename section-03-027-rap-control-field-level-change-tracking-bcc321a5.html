27. RAP %control field-level change tracking

  • Chapter 03
  • Section 27
  • self-contained study note

27.1 一句话结论

RAP %control field-level change tracking 是 RAP EML / behavior implementation 中用来说明“本次修改请求到底要改哪些字段”的字段级标记机制；它避免 update/create 请求把未修改字段也当成修改内容，从而降低静默覆盖、错误派生和旧逻辑误判的风险。

一句话记忆：字段值告诉 RAP “新值是什么”，%control 告诉 RAP “这个字段是不是本次请求要处理的字段”。没有 %control，很多 update 看起来有数据，实际上语义不完整。

27.2 概念说明

在普通 ABAP 里，一条结构体传进方法后，很难区分“字段被调用方故意传成 initial”还是“字段根本没有被修改，只是结构体默认 initial”。RAP 的 update / create 请求需要更精细的语义：如果 consumer 只改了 Status，就不应该把 CustomerId、BeginDate、EndDate 的 initial 值误当成要覆盖数据库的值。

%control 就是为了解决这个问题。它是 RAP derived type 中的特殊控制结构，通常出现在 TYPE TABLE FOR CREATE、TYPE TABLE FOR UPDATE、EML MODIFY ENTITIES 的输入数据，以及 unmanaged handler 的 create/update/import 参数里。注意：reported message 中常见的 %element 虽然也带字段名，但它用于消息字段定位，不是 %control。ABAP Keyword Documentation 明确说明，可用 IF_ABAP_BEHV 的结构化常量 if_abap_behv=>mk-on 和 if_abap_behv=>mk-off 来指定字段控制值。

最常见的写法是：字段赋值后，同时把对应 %control-<field> 设为 if_abap_behv=>mk-on：

DATA lt_create TYPE TABLE FOR CREATE zi_travel\Travel.

APPEND VALUE #(
  %cid = 'Travel_001'
  AgencyId = '070001'
  CustomerId = '000123'
  %control-AgencyId   = if_abap_behv=>mk-on
  %control-CustomerId = if_abap_behv=>mk-on
) TO lt_create.

这表示：本次 create 请求确实提供了 AgencyId 和 CustomerId。如果某个字段没有设置 %control-... = mk-on，RAP runtime 或 unmanaged handler 不应把它当成明确修改请求。

27.3 为什么重要

%control 的重要性在 update 场景最容易看出来。假设前端只把 Travel 的 OverallStatus 从 Open 改成 Accepted。如果后端把整个结构体直接搬进旧 update function module，未修改字段里的 initial 值可能覆盖原数据。上线后表现可能是“审批后客户号没了”“日期被清空”“金额变 0”。这类问题很隐蔽，因为语法没错，数据库也确实更新了，只是更新语义错了。

对 unmanaged RAP 来说，%control 更关键。Managed runtime 通常能根据请求和 BDEF 处理较多标准 update 语义；unmanaged handler / saver 里，项目代码要自己决定哪些字段传给旧逻辑、哪些字段不碰。忽略 %control，就等于把 PATCH 语义退化成“全量结构覆盖”。

对项目交付来说，%control 影响四件事。第一，接口稳定性：OData PATCH / EML update 不应覆盖未请求字段。第二，审计和变更日志：只有被请求修改的字段才应进入字段级差异判断。第三，validation / determination：只对相关字段触发或解释规则，避免无关字段造成误报。第四，旧系统集成：调用 BAPI、function module 或 legacy class 时要把“要改哪些字段”翻译清楚。

对 Java/Python 背景学习者，可以把 %control 类比成 JSON PATCH / partial update 中的 field mask 或 update mask。字段值本身像 payload，%control 像 mask：没有 mask，就无法可靠区分“不传”和“传空值”。

27.4 边界

第一条边界：%control 不是 feature control。上一节 RAP feature control 讲的是“字段或操作此刻能不能用”；本节 %control 讲的是“本次请求中哪些字段被标记为要修改/处理”。一个字段可以在 UI 上可编辑，但如果本次请求没改它，%control 就不应误标为 on。

第二条边界：%control 不是 validation。它不判断字段值是否合法，只表达字段是否参与本次请求。字段值是否合法仍由 validation、precheck、authorization、action logic 或旧系统校验决定。

第三条边界：%control 不是 change log。它告诉你请求里标记了哪些字段，但不等于数据库旧值和新值一定不同。用户可能把字段改成同一个值，也可能框架或 handler 又派生了其它字段。真正审计仍要比较旧值、新值、请求语义和保存结果。

第四条边界：%control 不等于 %element。%element-<field> 常用于 reported message 中标记消息关联的字段，让 UI 高亮字段；%control-<field> 用于修改请求中标记字段是否被控制/请求处理。两者名字相似，但语义不同。

第五条边界：field list 与 %control 有交集但不完全相同。MODIFY ENTITIES ... UPDATE FIELDS ( ... ) WITH ... 可以用 field list 指定更新字段；显式 %control 则常见于 derived type 数据结构、create/update table 和 unmanaged 场景。项目中要按语句变体和生成类型理解，不要混用成一套口号。

27.5 核心机制

第一层机制是 derived type。RAP 的 TYPE TABLE FOR CREATE、TYPE TABLE FOR UPDATE 等派生类型，会包含业务字段和若干技术组件，例如 %cid、%key、%tky、%is_draft、%control。开发者不是手写一个普通结构体，而是使用 BDEF-derived type 让输入数据和 BO 契约一致。

第二层机制是 mark constants。SAP Help 对 %control 的说明指出，可用 if_abap_behv=>mk-on 和 if_abap_behv=>mk-off 设置字段控制值。mk-on 表示字段被标记为参与当前请求；mk-off 表示没有标记或不参与。实际代码通常只显式写 on，未标记字段保持 off / initial。

第三层机制是 create。Create 请求里，%control 表示哪些字段由 consumer 显式提供。即使 create 是新建对象，也不能简单认为所有字段都被提供了：有些字段可能由 numbering、determination、defaulting、managed runtime 或数据库默认值产生。Managed BO 中，标准 create/update 的持久化细节多由 runtime 按请求语义处理；unmanaged BO 中，项目代码更需要显式读取 %control 并把它翻译给旧逻辑。

DATA lt_create TYPE TABLE FOR CREATE zi_travel\Travel.

lt_create = VALUE #(
  ( %cid = 'T001'
    AgencyId = '070001'
    CustomerId = '000123'
    BeginDate = cl_abap_context_info=>get_system_date( )
    %control-AgencyId   = if_abap_behv=>mk-on
    %control-CustomerId = if_abap_behv=>mk-on
    %control-BeginDate  = if_abap_behv=>mk-on ) ).

第四层机制是 update。Update 请求更依赖 %control。如果只修改状态，只应标记状态字段：

DATA lt_update TYPE TABLE FOR UPDATE zi_travel\Travel.

lt_update = VALUE #(
  ( %tky = ls_travel-%tky
    OverallStatus = 'A'
    %control-OverallStatus = if_abap_behv=>mk-on ) ).

MODIFY ENTITIES OF zi_travel IN LOCAL MODE
  ENTITY Travel
  UPDATE FROM lt_update
  FAILED DATA(ls_failed)
  REPORTED DATA(ls_reported).

这段的重点不是 OverallStatus = 'A'，而是只有 OverallStatus 的 %control 被打开。其它字段即使在结构体里是 initial，也不应被当成要更新。

第五层机制是 unmanaged handler。Unmanaged update handler 中，输入参数里的实体结构通常带 %control。项目代码需要把 %control 翻译成旧逻辑能理解的 update mask、changed-field list 或逐字段赋值逻辑。安全写法是：

  • 先读取当前数据库或 application buffer 的旧值。
  • 对每个字段检查 %control-<field> = if_abap_behv=>mk-on。
  • 只把标记为 on 的字段写入目标结构或调用 legacy API。
  • 未标记字段保留旧值，不用输入结构的 initial 覆盖。

第六层机制是 EML field list。某些 EML 语句可以使用 UPDATE FIELDS ( Field1 Field2 ) WITH ... 这类 field list 变体，显式告诉 runtime 要尊重哪些字段。它和 %control 都服务于字段级修改语义。初学者可以先理解为：无论用 field list 还是 %control，目标都是避免“整行结构覆盖”的错误。

第七层机制是 failed/reported 与 %element。当某个字段校验失败时，reported 中常用 %element-Field = if_abap_behv=>mk-on 告诉 UI 这个消息关联哪个字段；failed 里也可能标记 %update / %action-... 等失败位置。这和 %control 不是同一个结构，但常一起出现在 handler 代码中。读代码时要分清：%control 是请求 mask，%element 是消息定位。

27.6 项目意义

设计评审时，%control 要回答：“这个接口是全量更新还是部分更新？”如果是部分更新，必须明确未传字段如何处理，字段被传成 initial 时是否表示清空，旧逻辑是否支持 field mask。没有这个说明，接口测试很容易只测 happy path，看不出静默覆盖。

代码 review 时，重点看三类代码。第一，EML create/update 的输入是否正确设置 %control 或使用 UPDATE FIELDS。第二，unmanaged handler 是否按 %control 逐字段处理，而不是 MOVE-CORRESPONDING 到持久化结构。第三，legacy API 调用是否能区分“不改”和“改为空”。

测试时，要专门设计 partial update 用例。比如先创建一条 Travel，填好 Customer、BeginDate、EndDate；然后只更新 Status；最后确认 Customer 和日期没有变化。还要测试“明确清空可选字段”的场景，确认 %control on 且值 initial 时，系统按业务规则清空，而不是被误判成没传。

日本项目中，%control 常出现在 RAP unmanaged、BAPI 封装、OData PATCH 缺陷调查里。现场会问：未変更項目が初期値で上書きされていませんか、%control を見て更新対象項目だけ処理していますか、空値に更新する場合と未指定の場合を区別していますか。这些问题都在确认字段级变更语义。

27.7 Example / 学习例子

以 Travel 的 Customer 和 Status 为例。用户只把 Status 从 Open 改成 Accepted，Customer 没动。正确的 update 请求应只标记 OverallStatus：

lt_update = VALUE #(
  ( %tky = ls_travel-%tky
    OverallStatus = 'A'
    %control-OverallStatus = if_abap_behv=>mk-on ) ).

错误写法是把整条结构都搬过去，并把所有字段 %control 打开：

lt_update = VALUE #(
  ( %tky = ls_travel-%tky
    OverallStatus = 'A'
    CustomerId = ''
    BeginDate = '00000000'
    %control-OverallStatus = if_abap_behv=>mk-on
    %control-CustomerId    = if_abap_behv=>mk-on
    %control-BeginDate     = if_abap_behv=>mk-on ) ).

这表示“我确实要把 CustomerId 清空、BeginDate 改成初始值”。如果这不是业务意图，就是严重 bug。最危险的是它看起来像普通结构默认值，不像明显错误。

在 unmanaged handler 里，伪代码应更像这样：

IF entity-%control-OverallStatus = if_abap_behv=>mk-on.
  ls_legacy-status = entity-OverallStatus.
ENDIF.

IF entity-%control-CustomerId = if_abap_behv=>mk-on.
  ls_legacy-customer = entity-CustomerId.
ENDIF.

这段不是完整可运行代码，而是强调“逐字段看 mask”。如果直接 MOVE-CORRESPONDING entity TO ls_legacy，就很可能把没有修改的字段也覆盖掉。

27.8 Counterexample / Failure mode

第一个失败模式是全量覆盖。开发者把 update input 结构直接 MOVE-CORRESPONDING 到持久化结构，没有检查 %control。结果未传字段被 initial 值覆盖，生产数据被静默清空。

第二个失败模式是无法清空字段。为了避免覆盖，开发者写了“initial 就不更新”。但如果业务允许用户把可选字段清空，initial 可能是合法新值。正确判断应看 %control：%control on 且值 initial，才表示明确清空。

第三个失败模式是 create 时忘记 %control。Consumer 明明填了字段，但没有打开 %control，导致 runtime、handler 或旧逻辑没有按预期处理该字段，出现 defaulting、mandatory check 或 mapping 异常。

第四个失败模式是把 %element 当 %control。开发者在 reported message 中设置了 %element-Date = mk-on，以为这能控制 update 字段。实际上 %element 只是消息字段定位，不是修改请求 mask。

第五个失败模式是变更日志只看 %control。%control on 表示字段参与请求，不代表旧值和新值不同。审计日志如果只按 %control 记变更，可能记录“伪变化”；如果完全不看 %control，又可能漏掉显式清空字段。正确做法是结合 %control、旧值、新值和保存结果。

第六个失败模式是 legacy API 不支持 partial update。即使 RAP handler 正确读取 %control，后面调用的 BAPI / function module 可能仍按全量结构覆盖。需要在适配层把 %control 转成旧 API 的 X-structure、change indicator 或逐字段更新参数。

27.9 日本项目现场说法

  • 常见日语：項目単位の変更管理、変更対象項目、未変更項目、初期値上書き、部分更新、更新マスク、X構造、PATCH相当。
  • 说明句：%control は、RAP の更新要求でどの項目が変更対象かを示すための制御情報です。
  • Review 句：unmanaged update では %control を確認し、変更対象項目だけ旧ロジックへ渡してください。
  • 質問句：managed の場合、%control の確認はフレームワーク任せで問題ないですか。
  • 调查句：未変更項目が初期値で上書きされた場合、%control の設定と legacy API へのマッピングを確認します。
  • 区分句：%element はメッセージの項目紐付け、%control は更新要求の項目マスクとして分けて考えます。
