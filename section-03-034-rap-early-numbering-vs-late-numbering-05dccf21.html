34. RAP early numbering vs late numbering

  • Chapter 03
  • Section 34
  • self-contained study note

34.1 一句话结论

RAP early numbering vs late numbering 讲的是 RAP create 场景中最终 key 的分配时机：early numbering 在 create/modify 阶段就给新实例分配 key，late numbering 则等到 save sequence 中、接近最终保存时才分配 key。这个差异会影响 %cid / mapped、草稿、action、validation、业务单号是否跳号、以及 consumer 什么时候能拿到正式编号。

一句话记忆：early numbering 让新对象“早点有正式 key”，late numbering 让新对象“检查通过后再拿正式 key”。

34.2 概念说明

上一节 Numbering 编号策略 已经讲过：RAP numbering 要回答“谁给 key、什么时候给 key、如何把临时实例映射到最终 key”。本节只放大其中一个问题：什么时候给 key。

Early numbering 的意思是：create 请求处理期间就分配最终 key，key 很早进入 RAP transactional buffer。SAP Learning 对 unmanaged early numbering 的说明是：handler method 用 FOR NUMBERING 实现，entities 里是新建实例，开发者必须通过 mapped 返回每个 %cid 对应的 key，或者通过 failed 表示失败。Managed internal numbering 生成 UUID key 也属于早期编号思想：新实例很早就有技术 key。

Late numbering 的意思是：新实例在 interaction phase 中还没有最终 key，保存阶段才分配。SAP Learning 对 late numbering 的说明是：在 saver class 的 adjust_numbers 中处理，最终 key 在即将保存到数据库前分配。这样可以把号码消耗推迟到业务检查通过之后，适合某些希望尽量减少跳号的业务编号场景。

两者的核心差异不是“谁更高级”，而是取舍：

  • early numbering：开发和关联处理更直接，因为 key 早就可用；但如果保存失败，已分配的业务编号可能被浪费。
  • late numbering：更适合把正式编号留到保存前后，但 interaction phase 不能提前依赖最终业务号；实现、mapping 和测试更复杂。

如果 key 是 technical UUID，通常不必为跳号焦虑，early/managed numbering 很自然。如果 key 是用户可见的申请单号、凭证号、年度连续编号，就必须认真讨论 late numbering、number range、业务是否允许缺号，以及 SAP 标准对象是否已经有编号机制。

更贴近项目的说法是：RAP numbering 不是“生成一个 ID”这么简单，而是在 create 请求、transactional buffer、save sequence、OData response 和用户可见业务编号之间建立契约。读者可以先把 key 分成两类。第一类是 technical key，它主要服务于系统内部识别、关联、更新和持久化，常见选择是 UUID，适合早期可用。第二类是 business number，它会被用户、客户、审计、接口伙伴当成业务凭证使用，通常要考虑可读性、连续性、年度规则、组织规则、是否允许缺号、以及失败后如何说明。

很多项目的最佳实践不是在 early 和 late 之间二选一，而是把 technical key 和 business number 分离：技术 key 早期分配，让 RAP 内部关联稳定；业务编号在保存成功附近或特定业务动作中分配，让用户可见编号更符合业务规则。这样做会多一个字段和一段映射说明，但能显著降低“为了显示编号而提前消耗正式号”或“为了不跳号而让 UI/child create 无法关联”的矛盾。

初学者还要知道，编号策略会进入 API 契约。一个 create API 如果返回的是 technical UUID，consumer 需要后续用 UUID 查询；如果返回的是 business number，consumer 可能会把它写入外部系统；如果保存前只有 %cid，consumer 不能把 %cid 当正式 key。接口文档里必须写清楚 create 成功后返回什么、失败时是否可能消耗编号、重复提交是否会生成新号、deep create 中 parent 和 child 的对应关系如何表达。

34.3 为什么重要

编号时机一旦选错，后面会连锁影响很多设计。Early numbering 用在业务单号上，可能出现用户创建失败但号码已经被消耗；late numbering 用在需要早期关联和显示的技术 key 上，又可能让 action、child create、UI feedback 和测试复杂化。

对项目交付来说，early/late numbering 会影响接口契约。OData consumer 什么时候拿到正式 key？deep create 中 child 如何关联 parent？用户保存前是否能看到业务单号？保存失败后是否要解释缺号？重复请求是否生成多个号码？这些问题都不是 ABAP 语法细节，而是接口和业务验收条件。

对 Java/Python 背景学习者，可以把 early numbering 类比成 ORM flush 前就拿到 ID，把 late numbering 类比成 commit 末尾才分配业务编号。但 RAP 中还有 %cid、mapped、saver class、adjust_numbers、point-of-no-return、draft 和 OData metadata，因此不能简单套数据库自增主键模型。

在日本项目现场，这类讨论通常会用 早期採番、後期採番、欠番許容、採番タイミング、番号範囲 来表达。设计书如果只写“自动编号”，是不够的。

编号问题还会影响测试证据。单体测试不能只验证“创建成功后数据库里有一条记录”，还要验证 mapped 是否完整、失败时 failed/reported 是否可解释、并发 create 是否不会重复号、保存前失败是否消耗业务编号、保存后返回给 OData consumer 的 key 是否和数据库一致。对日本项目交付来说，这些证据通常会进入 単体テスト仕様書、結合テストエビデンス 或 API 受入确认材料。

34.4 边界

第一条边界：early/late numbering 不是 external/internal numbering 的同义词。External/internal 回答“谁给 key”；early/late 回答“什么时候给 key”。外部编号通常由 consumer 早期提供；内部编号既可能 early，也可能 late。

第二条边界：managed internal numbering 通常是 early 思维，但不等于所有 early numbering。Managed runtime 自动生成 UUID key是一种早期可用 key；unmanaged early numbering 则由项目代码在 handler method 中实现 FOR NUMBERING。

第三条边界：late numbering 不等于绝对无缺号。它把编号推迟到保存阶段，减少保存前失败导致的号码浪费；但系统失败、并发、number range 机制、法律凭证规则仍要单独设计。不能承诺“late numbering 就一定无跳号”。

第四条边界：late numbering 不适合被前期业务逻辑强依赖。如果 action、validation、determination 或 UI 在保存前必须知道正式业务号，late numbering 会让设计复杂化。要么改成 early/managed technical key + later business number，要么调整业务流程。

第五条边界：early numbering 也不等于可以随便 SELECT MAX + 1。早期生成 key 仍要考虑并发、锁、number range 和唯一性。裸 MAX + 1 是高风险反例。

第六条边界：temporary ID 不等于 final key。%cid 和 late numbering 中的临时识别只是事务内锚点；保存成功后才通过 mapped 或 OData response 得到最终 key。

第七条边界：numbering 不等于 idempotency。编号可以生成唯一 key，但不能自动保证同一个外部请求重复提交时只创建一次。如果接口 consumer 可能超时重试，需要独立设计 external request id、idempotency key、唯一约束或状态检查。否则 early numbering 会让每次重试拿到不同号码，late numbering 也可能在第二次成功保存时生成另一个正式号。

第八条边界：numbering 不等于 authorization。某个用户能否创建对象，应由 RAP authorization、DCL、业务角色或服务层检查决定；不能用“拿不到号码”来间接拒绝。编号失败消息也不应该掩盖权限错误、主数据错误或锁冲突。

第九条边界：number range 不等于无风险。即使用 SAP number range 或标准编号机制，也要确认缓冲、并发、回滚、法律凭证规则、测试系统数据和生产系统区间。尤其是财务凭证、发票、法定连续编号相关场景，必须以 SAP 标准对象和客户合规要求为准，不能只从 RAP 示例推导。

34.5 Key mechanism / 核心机制

第一层机制是 early numbering 的位置。Early numbering 发生在 create/modify 阶段，结果进入 transactional buffer。典型 unmanaged early numbering handler 形态是：

METHODS earlynumbering_create FOR NUMBERING
  IMPORTING entities FOR CREATE Travel.

实现时要为每个 create request 产生 key，并把 %cid 到 key 的映射填入 mapped。如果某条不能编号，要返回 failed / reported。关键规则是：每个 %cid 都要有明确结果。

第二层机制是 late numbering 的位置。Late numbering 在 saver class 中实现，典型入口是 adjust_numbers()。它不在普通 handler interaction phase 执行，而是在 Save sequence 保存序列 中、接近最终保存时分配最终 key。这个位置让它可以等保存前检查通过后再消耗号码，但也意味着可预见的业务错误不应拖到这里才发现。SAP Learning 对 late numbering 的说明指出 adjust_numbers 很晚，且没有普通 validation 那样的 failed 参数；所以金额、主数据、权限、重复 key 等检查应尽量提前完成。

第三层机制是 %cid 与 mapped。无论 early 还是 late，consumer 往往用 %cid 表示“这次 create 请求里的临时实例”。保存或 create 成功后，runtime/implementation 通过 mapped 告诉 consumer：这个 %cid 对应哪个最终 key。读 EML create 测试时，一定要检查 MAPPED，不要只看没有报错。

第四层机制是 %pid / temporary key。Late numbering 场景中，最终 key 尚未分配时，runtime 需要内部临时识别来跟踪实例。ABAP Keyword Documentation 和 SAP Learning 的 late numbering 示例会出现 %pid / temporary key 相关结构。初学者不必死背所有字段，但要理解：保存前的临时识别不是最终业务 key。

第五层机制是 child entity。Parent 如果使用 late numbering，child 在保存前不能依赖最终 parent key。RAP 需要通过 composition、temporary ID 和 mapping 保持关系。SAP Learning 还提醒，在某些 late numbering 设计中 parent/child 的 numbering 策略要一致或受约束；实际以当前 release 文档和 ADT 检查为准。

第六层机制是测试差异。Early numbering 测试要确认 create 后是否立刻拿到 key、失败时是否返回 failed/reported、重复请求是否不冲突。Late numbering 测试要确认保存前正式号不可用、保存成功后 mapping 正确、保存前失败不会消耗业务号、并发保存唯一、错误发生在 point-of-no-return 后时有可解释策略。

第七层机制是业务编号与技术 key 分离。很多项目最稳的设计是：technical key 用 UUID managed numbering，业务编号作为普通业务字段在保存阶段或业务动作中分配。这样内部关联稳定，用户可见编号也可以按业务规则处理。是否这么做，要结合标准对象、接口契约和客户编号规则确认。

可以用下面的决策树做初步判断：

Is the key user-visible and business/audit-relevant?
  No  -> Prefer technical key, often UUID / managed internal numbering.
  Yes -> Ask whether gaps are acceptable and when the user must see the number.

Must the UI/API know the final number before save?
  Yes -> Early numbering may be necessary, but document gap behavior.
  No  -> Late numbering or save-time business number is often safer.

Does the object have parent-child create or actions before save?
  Yes -> Keep a stable technical key or temporary mapping; do not rely blindly on final business number.

Can the same external request be retried?
  Yes -> Add idempotency design; numbering alone is not enough.

在 RAP 代码阅读时，可以按这个顺序找证据：

证据位置要看什么
Behavior definition是否声明 managed/unmanaged、numbering、lock、authorization
Handler class是否有 FOR NUMBERING 或 create 逻辑
Saver class是否实现 adjust_numbers()
EML testMAPPED、FAILED、REPORTED 是否检查完整

这张表的目的不是让初学者一次写出完整 RAP 框架代码，而是训练阅读顺序：先看模型声明，再看 handler/saver，再看测试和 API response。否则很容易只盯着某个 number_get_next 或 UUID 生成语句，却没有理解它在 RAP lifecycle 的位置。

34.6 日本项目现场说法

常见日语：

  • 早期採番：early numbering。
  • 後期採番：late numbering。
  • 採番タイミング：编号时机。
  • 欠番許容：是否允许跳号。
  • 正式番号：正式编号。
  • 仮キー：临时 key。
  • 技術キー：technical key。
  • 業務番号：business number。
  • 番号範囲：number range。

说明句：

技術キーは早期採番、業務番号は保存時の後期採番として設計します。

意思是：技术 key 设计为 early numbering，业务编号设计为保存时 late numbering。

後期採番のため、保存前の画面処理では正式番号に依存しない設計にします。

意思是：因为使用 late numbering，保存前的画面处理不依赖正式编号。

评审时可以这样问：

この番号は欠番を許容しますか。許容しない場合、後期採番だけで要件を満たせますか。

意思是：这个编号允许跳号吗？如果不允许，仅靠 late numbering 能满足需求吗？

34.8 Example / 学习例子

例子一：early numbering with UUID。

Travel technical key 是 UUID。创建时 runtime 或项目代码马上给新实例分配 UUID。好处是 parent-child 关联、action result、测试和后续 EML 读取都很直接；UUID 不是用户可见连续单号，因此即使创建失败，也通常没有“缺号”业务争议。

例子二：unmanaged early numbering with %cid。

EML create 传入 %cid = 'create-1'。FOR NUMBERING 方法生成 TravelID = 4711，并在 mapped-travel 中返回 %cid = 'create-1' 对应 TravelID = 4711。如果编号失败，就要把该 %cid 对应实例放入 failed 并返回消息。

例子三：late numbering for business number。

申请单草稿创建时没有正式申请号。用户编辑、保存前检查都通过后，save sequence 的 adjust_numbers() 分配 RequestNo = 2026-000123。保存成功后，response 才返回正式号。这样可以避免用户编辑一半就消耗正式业务号。

例子四：错误选择。

如果业务要求保存前页面必须显示正式编号，并且用户会把编号告诉其他人，那么 late numbering 可能不合适。反过来，如果业务要求尽量不跳号，create 一开始就消耗业务编号也不合适。正确做法是回到业务规则，决定 technical key 和 business number 是否分离。

例子五：deep create 的 parent-child。

一个上传批次有 header 和 items。header 最终业务号如果采用 late numbering，items 在 interaction phase 里不能提前写死这个最终业务号。更稳的方式是让 header 有一个 early technical key 或 temporary mapping，items 通过 composition / %cid_ref / transactional buffer 关联到 parent；保存成功后再把正式 business number 返回给 consumer。测试时不仅要看 header 是否创建成功，还要看 items 是否关联到同一个最终 parent。

例子六：外部系统重试。

外部系统调用 Create Request，网络超时后不确定 SAP 是否保存成功，于是再次发送同一 payload。没有 idempotency key 时，early numbering 可能创建两个 technical key 和两个业务号；late numbering 也可能在第二次保存成功时分配另一个正式号。正确设计通常要让外部请求带 request id，SAP 端用唯一约束或状态表识别重复请求，并在重复时返回已有对象，而不是单靠 numbering 解决。

例子七：AC0567C 风格上传 API。

AC0567C 的上传批次可以设计为：batch_uuid 是 technical key，创建时 early/managed 分配；upload_no 是业务可见编号，只有当 batch 通过预检查并进入正式接收状态时才分配；外部 request_id 用于 idempotency。这样 UI、OData、后台 job 都可以用 batch_uuid 追踪对象，用户和交付文档可以用 upload_no 沟通，接口重试则由 request_id 控制。

34.9 Counterexample / Failure mode

反例一：为了“少跳号”，把所有 key 都做 late numbering，结果 UI、child create、action、测试都拿不到正式 key。修正方式是分离 technical key 和 business number。

反例二：用 early numbering 生成业务单号，保存前 validation 失败后号码被浪费。修正方式是确认是否允许跳号；若不允许，考虑 late numbering 或标准编号机制。

反例三：late numbering 后还有可预见业务校验。号码已分配，后续又因为金额或主数据错误失败。修正方式是把可预见错误提前到 validation / check-before-save。

反例四：FOR NUMBERING 没处理所有 %cid。有些 create 请求既不在 mapped，也不在 failed，consumer 无法知道结果，runtime 也可能报错。修正方式是每条 create 都要明确 mapping 或失败。

反例五：误把 %cid 当最终 key 写入外部系统。%cid 只是请求内临时标识，不应作为业务编号传播。修正方式是等待 mapped / response 中的最终 key。

反例六：裸 SELECT MAX + 1 做 early numbering。并发保存时重复号风险极高。修正方式是使用 SAP number range、锁、UUID、late numbering 或标准对象编号。

反例七：把 technical UUID 暴露成客户业务编号。UUID 适合作为技术 key，但不一定适合发给业务用户、客户或外部合作方。修正方式是明确 technical key 和 business number 的展示边界。

反例八：late numbering 后没有测试 mapped。保存成功了，但 consumer 拿不到最终 key，后续查询、导航或外部回写失败。修正方式是在 EML/API 测试中把 MAPPED 和 OData response 作为验收项。

反例九：把“无缺号”写成绝对承诺。即使 late numbering 能减少保存前失败导致的号码浪费，系统故障、合规编号机制、number range buffering 或标准对象规则仍可能产生缺口。修正方式是把 欠番許容 和异常说明写进业务规则，而不是在技术设计中承诺绝对连续。

反例十：外部重试导致重复创建。编号策略本身没有识别“这是不是同一外部请求”的能力。修正方式是引入 idempotency key，并把重复请求返回已有对象的规则写进 API 契约。

34.10 思考问题

  • 这个编号是 technical key 还是 business number？
