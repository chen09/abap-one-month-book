Chapter 02 - Core ABAP Programming

7. Aggregate functions 聚合函数

  • Chapter 02
  • Section 07
  • self-contained study note

7.1 一句话结论

Aggregate functions 是 ABAP SQL 里把多行数据汇总成统计值的机制，例如 COUNT( * )、SUM( ... )、MIN( ... )、MAX( ... )、AVG( ... )。它的重点不是少写几行 LOOP，而是把“按数据库条件筛选、分组、汇总”的工作交给数据库完成，并让代码清楚表达结果粒度：到底返回一条总计，还是按客户、币种、状态等维度返回多条汇总。

7.2 到底学什么

从业务角度看，聚合函数回答的是“多少、合计、平均、最大、最小”这类问题：

  • 这个客户有多少张订单？
  • 本月每个客户的订单金额合计是多少？
  • 每个币种的未清金额合计是多少？
  • 最近一次创建时间是什么？
  • 每个状态下有多少条记录？

如果不用聚合，初学者很容易写成“先把所有明细读到内表，再在 ABAP 里循环统计”：

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              net_amount,
              currency_code
       WHERE customer_id = @lv_customer_id
       INTO TABLE @DATA(lt_orders).

DATA(lv_total_amount) = 0.

LOOP AT lt_orders INTO DATA(ls_order).
  lv_total_amount += ls_order-net_amount.
ENDLOOP.

这种写法在几条测试数据上能跑，但它把数据库擅长的统计工作搬到了 ABAP 应用层。数据量变大时，会多读很多不需要传输到应用层的明细；逻辑也会散落在 SELECT 和 LOOP 两处，review 时不容易看出统计口径。

聚合函数的基本写法，是直接在 ABAP SQL 的 FIELDS 列表里声明要统计什么：

SELECT FROM zsales_order
       FIELDS COUNT( * )
       WHERE customer_id = @lv_customer_id
       INTO @DATA(lv_order_count).

这段代码不是“读订单列表”，而是“返回订单条数”。一旦用了聚合函数，结果的含义就从明细粒度变成统计粒度。学习本节时，要把注意力放在三个问题上：统计范围由 WHERE 决定，统计维度由 GROUP BY 决定，统计后的组过滤由 HAVING 决定。

7.3 为什么重要

聚合函数重要，是因为 SAP 项目里大量需求都不是简单明细查询，而是带统计口径的业务判断。报表、接口前置检查、批处理汇总、监控指标、错误件数、状态分布、金额合计，都会用到聚合。

它的第一层价值是性能。数据库可以在靠近数据的位置完成 COUNT、SUM、MAX 等计算，ABAP 程序只接收统计结果。和“读出所有明细再循环”相比，这通常减少网络传输、应用服务器内存和 ABAP 循环成本。

第二层价值是可解释性。SELECT ... GROUP BY customer_id, currency_code 明确告诉 reviewer：这个结果是按客户和币种分组的，不是一张明细表。如果代码里只有一个很长的 LOOP AT，统计口径就需要读完整段逻辑才能理解。

第三层价值是减少业务错误。金额和数量的统计不能只看数字字段。SAP 数据里常见金额字段会配套 currency key，数量字段会配套 unit key。把 JPY、USD、EUR 混在一起 SUM( net_amount )，语法上可能能执行，业务上却是错误结果。正确做法通常是按 currency_code 分组，或者在明确的转换规则下先做货币换算。

第四层价值是为 CDS/RAP/OData 做准备。后续进入 CDS view entity、analytical query、RAP 查询或 OData 服务时，统计口径会变成模型的一部分。现在先在 ABAP SQL 中理解 GROUP BY 和 HAVING，后面读 CDS 的 aggregation、annotation 和服务返回结果会轻松很多。

7.4 Key mechanism / 核心机制

7.4.1 聚合函数会改变结果粒度

普通 SELECT 读的是明细行：

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              net_amount
       WHERE customer_id = @lv_customer_id
       INTO TABLE @DATA(lt_order_items).

如果符合条件的订单有 100 条，结果通常就是 100 行。聚合 COUNT( * ) 则把这 100 行变成一个统计值：

SELECT FROM zsales_order
       FIELDS COUNT( * )
       WHERE customer_id = @lv_customer_id
       INTO @DATA(lv_order_count).

这时结果不再是订单列表，而是“订单条数”。项目里必须把变量名、目标结构和接口字段命名成统计语义，例如 order_count、total_amount、latest_created_at，不要继续叫 lt_orders 这种明细名称。

7.4.2 COUNT( * )：统计行数

COUNT( * ) 统计符合条件的行数。它常用于存在性检查、数据件数检查、批处理结果统计和测试断言：

SELECT FROM zsales_order
       FIELDS COUNT( * )
       WHERE status = @lc_status_error
       INTO @DATA(lv_error_count).

如果只是想知道“有没有错误数据”，项目里也要考虑性能和语义：有时 UP TO 1 ROWS 读一条即可，有时 COUNT( * ) 是为了准确显示件数。不要机械套用一个写法。

在 outer join 或可为 null 的表达式场景中，还要区分 COUNT( * ) 和 COUNT( sql_exp )。当前 ABAP SQL 文档说明，COUNT( sql_exp ) 统计的是表达式结果不是 null 的行；COUNT( * ) 则是行数。这在 left outer join 后尤其重要，因为右侧不存在的记录会带来 null 语义。

7.4.3 SUM、MIN、MAX、AVG：统计数值或可比较值

常见聚合函数可以这样理解：

  • SUM( net_amount )：合计金额或数量。
  • MIN( created_at )：最早创建时间，或最小值。
  • MAX( created_at )：最新创建时间，或最大值。
  • AVG( net_amount )：平均值。

例如，按币种统计已释放订单的合计金额：

SELECT FROM zsales_order
       FIELDS currency_code,
              COUNT( * )       AS order_count,
              SUM( net_amount ) AS total_amount,
              MAX( created_at ) AS latest_created_at
       WHERE status = @lc_status_released
       GROUP BY currency_code
       ORDER BY currency_code
       INTO TABLE @DATA(lt_totals_by_currency).

这里 currency_code 不是装饰字段，而是统计口径的一部分。没有 GROUP BY currency_code，SUM( net_amount ) 就会把不同币种混在一起；这在 SAP 项目里通常是严重业务错误。

AVG 要额外注意结果类型和小数精度。ABAP SQL 支持在 AVG 中指定结果数据类型，例如官方文档中的 AVG( col AS dtype ) 形式。项目中如果平均金额、平均数量要暴露到接口或报表，不能只看语法通过，还要确认字段类型、舍入规则、币种/单位语义和前端显示方式。

7.4.4 GROUP BY：定义统计维度

没有 GROUP BY 时，聚合通常对整个结果集计算一个统计结果。加上 GROUP BY 后，数据库会按分组字段生成多条汇总结果：

SELECT FROM zsales_order
       FIELDS customer_id,
              currency_code,
              COUNT( * )       AS order_count,
              SUM( net_amount ) AS total_amount
       WHERE created_at BETWEEN @lv_from_date AND @lv_to_date
       GROUP BY customer_id,
                currency_code
       ORDER BY customer_id,
                currency_code
       INTO TABLE @DATA(lt_customer_summary).

这段代码的结果粒度是“客户 + 币种”。如果同一个客户有 JPY 和 USD 两种订单，会返回两行，而不是一行。这样的行数变化不是 bug，而是分组维度决定的结果。

GROUP BY 还有一个基本规则：SELECT 列表里只要出现非聚合字段，这些字段就必须属于分组维度。下面的例子表达不清，也不应通过语法检查：

SELECT FROM zsales_order
       FIELDS customer_id,
              sales_order_id,
              SUM( net_amount ) AS total_amount
       GROUP BY customer_id
       INTO TABLE @DATA(lt_wrong).

sales_order_id 是明细字段，但结果只按 customer_id 分组。系统无法知道一个客户多张订单时应该返回哪个 sales_order_id。修正方式不是随便把 sales_order_id 加进 GROUP BY，因为那会把粒度变回“客户 + 订单”。真正要先问业务：需要客户汇总，还是订单明细？

7.4.5 WHERE 和 HAVING：先过滤明细，再过滤分组

WHERE 在聚合前过滤明细行，HAVING 在聚合后过滤分组。这个顺序非常重要。

SELECT FROM zsales_order
       FIELDS customer_id,
              currency_code,
              COUNT( * )       AS order_count,
              SUM( net_amount ) AS total_amount
       WHERE created_at BETWEEN @lv_from_date AND @lv_to_date
       GROUP BY customer_id,
                currency_code
       HAVING SUM( net_amount ) > @lv_min_total
       ORDER BY total_amount DESCENDING,
                customer_id,
                currency_code
       INTO TABLE @DATA(lt_large_customers).

这里 WHERE 先限定日期范围；GROUP BY 按客户和币种汇总；HAVING 再留下合计金额超过门槛的分组。初学者常犯的错，是把聚合条件写进 WHERE：

" 错误思路：WHERE 不能这样表达聚合后的合计条件
WHERE SUM( net_amount ) > @lv_min_total

现场说明时可以这样说：WHERE は明細行の抽出条件、HAVING は集計後のグループ条件です。

7.4.6 聚合后的排序仍然要显式写

聚合不会自动保证结果顺序。需要稳定显示或稳定测试时，仍然要写 ORDER BY。如果按聚合结果排序，通常给聚合表达式起 alias，再用 alias 排序：

SELECT FROM zsales_order
       FIELDS customer_id,
              SUM( net_amount ) AS total_amount
       WHERE currency_code = @lv_currency_code
       GROUP BY customer_id
       ORDER BY total_amount DESCENDING,
                customer_id
       INTO TABLE @DATA(lt_ranked_customers).

total_amount DESCENDING 负责按金额从大到小，customer_id 负责在金额相同的时候给出稳定次序。没有第二个排序键时，相同合计金额的客户顺序仍然可能不稳定。

7.4.7 聚合不是权限检查，也不是 Clean Core 许可

聚合函数只是 ABAP SQL 的读取和计算方式。它不会自动补上业务权限，也不会让直接访问标准底表变得符合 Clean Core。举例说，直接对标准底表做 COUNT( * ) 或 SUM( ... )，仍然可能绕开 CDS DCL、RAP authorization、服务层权限和 released API 边界。

在 ABAP Cloud / S/4HANA Cloud 或强调 Clean Core 的项目里，要优先确认是否有 released CDS view、released API、RAP BO 或项目认可的数据模型。统计逻辑如果要暴露给 OData 或 Fiori，也要确认权限是在 CDS DCL、RAP authorization、service implementation 还是应用代码中完成。

7.7 Example / 学习例子

7.7.1 例 1：统计错误订单件数
SELECT FROM zsales_order
       FIELDS COUNT( * )
       WHERE status = @lc_status_error
       INTO @DATA(lv_error_count).

这不是读错误订单明细，而是读错误件数。后续如果只是判断是否要写监控日志，可以用 lv_error_count；如果要显示错误列表，则还需要另一条明细查询。不要把“件数检查”和“明细取得”混在一个变量名里。

7.7.2 例 2：按客户和币种统计订单金额
SELECT FROM zsales_order
       FIELDS customer_id,
              currency_code,
              COUNT( * )       AS order_count,
              SUM( net_amount ) AS total_amount,
              MAX( created_at ) AS latest_created_at
       WHERE status = @lc_status_released
       GROUP BY customer_id,
                currency_code
       ORDER BY customer_id,
                currency_code
       INTO TABLE @DATA(lt_customer_totals).

读这段代码时，要先说出结果粒度：一行代表一个客户在一个币种下的已释放订单汇总。COUNT( * ) 是订单数，SUM( net_amount ) 是该币种下的合计金额，MAX( created_at ) 是最新创建时间。

7.7.3 例 3：用 HAVING 留下大额分组
SELECT FROM zsales_order
       FIELDS customer_id,
              currency_code,
              SUM( net_amount ) AS total_amount
       WHERE created_at BETWEEN @lv_from_date AND @lv_to_date
       GROUP BY customer_id,
                currency_code
       HAVING SUM( net_amount ) > @lv_min_total
       ORDER BY total_amount DESCENDING,
                customer_id,
                currency_code
       INTO TABLE @DATA(lt_large_totals).

WHERE 控制日期范围，HAVING 控制“汇总后金额超过门槛”的分组。这个区别在报表和接口筛选里很常见。

7.7.4 例 4：left outer join 后统计右侧存在件数
SELECT FROM zsales_order AS so
       LEFT OUTER JOIN zdelivery AS dlv
          ON dlv~sales_order_id = so~sales_order_id
       FIELDS so~customer_id,
              COUNT( * )               AS joined_row_count,
              COUNT( dlv~delivery_id ) AS delivery_count
       GROUP BY so~customer_id
       ORDER BY so~customer_id
       INTO TABLE @DATA(lt_delivery_summary).

COUNT( * ) 统计 join 后的行数；COUNT( dlv~delivery_id ) 只统计右侧交货单号不是 null 的行。这个例子用来提醒：outer join 后的 null 语义会影响统计结果，不能只看函数名。

7.8 Counterexample / Failure mode

失败模式一：明细全读出来再循环统计。数据量小时看不出问题，数据量大时会浪费数据库到应用服务器的传输和 ABAP 内存。能由数据库明确完成的 COUNT、SUM、MAX，优先考虑在 ABAP SQL 中表达。

失败模式二：混淆明细字段和汇总字段。customer_id + SUM( net_amount ) 是客户汇总；customer_id + sales_order_id + SUM( net_amount ) 就变成客户订单粒度。不要为了让语法通过，把不理解的字段全部塞进 GROUP BY。

失败模式三：金额跨币种相加。SUM( net_amount ) 没有自动理解 JPY、USD、EUR 的业务含义。要么按 currency_code 分组，要么在明确汇率、日期和目标币种规则后做转换。

失败模式四：数量跨单位相加。10 EA + 5 KG 没有业务意义。数量统计也要按 unit key 分组或转换。

失败模式五：把聚合后的条件写进 WHERE。明细过滤用 WHERE，分组后的过滤用 HAVING。如果需求说“只显示合计金额超过 100 万的客户”，那就是 HAVING SUM( ... ) > ...。

失败模式六：以为聚合结果不需要排序。聚合结果仍然是多行结果集；展示、分页、测试断言、接口稳定输出都需要明确 ORDER BY。

失败模式七：用聚合掩盖权限问题。即使只返回总数或合计，也可能泄露用户无权查看的数据规模或金额。统计接口同样要确认授权边界。

7.9 思考问题

  • 这条查询返回的是明细粒度，还是统计粒度？
  • 如果有 GROUP BY，一行结果代表什么业务维度？
  • WHERE 过滤的是明细行，还是误把聚合条件写到了明细过滤里？
  • 是否需要 HAVING 过滤聚合后的分组？
  • SUM 的字段是否有 currency key 或 unit key？是否按币种/单位分组或转换？
  • AVG 的结果类型、精度和舍入规则是否符合接口或报表要求？
  • left outer join 后使用 COUNT( * ) 还是 COUNT( sql_exp )？null 语义是否明确？
  • 聚合结果是否需要稳定排序？
  • 权限检查在哪里发生？是否直接访问了不应直接访问的 SAP 标准底表？
  • 这个统计逻辑未来是否应该上移到 CDS view entity、RAP query 或 released API？
