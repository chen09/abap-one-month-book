Chapter 02 - Core ABAP Programming

20. Table expression 表表达式

  • Chapter 02
  • Section 20
  • self-contained study note

20.1 一句话结论

Table expression 表表达式，是现代 ABAP 用 itab[ ... ] 直接访问 internal table 某一行的表达式写法：它可以替代很多 READ TABLE 读取场景，也可以出现在赋值、构造表达式、line_exists、line_index、写位置中；但普通 table expression 不设置 sy-subrc，未找到行时通常会抛出 CX_SY_ITAB_LINE_NOT_FOUND。如果把 table expression 放在 ASSIGN 的源位置，这是重要例外：not-found 通过 sy-subrc 表达。其余场景必须用 OPTIONAL、DEFAULT、line_exists 或异常处理明确处理 not-found。

20.2 到底学什么

上一节 READ TABLE 读取内部表 的经典写法是：

READ TABLE lt_material INTO DATA(ls_material)
  WITH TABLE KEY matnr = lv_matnr.

IF sy-subrc = 0.
  ...
ENDIF.

table expression 的现代写法是：

DATA(ls_material) = lt_material[ matnr = lv_matnr ].

看起来只是短了几行，但语义不同。READ TABLE 是 statement，结果状态主要通过 sy-subrc 表达；table expression 是 expression，可以嵌入更大的表达式里，但未找到行时通常抛异常，不会给你一个 sy-subrc 分支。

本节要学的核心问题：

  • itab[ index ]、itab[ key = value ]、itab[ KEY keyname COMPONENTS ... ] 分别怎么理解？
  • table expression 与 READ TABLE 的 not-found 处理有什么不同？
  • line_exists、line_index、OPTIONAL、DEFAULT 分别适合什么场景？
  • table expression 是否更快？答案是：不一定，性能仍取决于 table category 和 key。
  • table expression 能否修改表内行？可以用于 write position，但要理解 key 字段和 not-found 风险。

20.3 为什么重要

现代 ABAP 代码大量使用 table expression。RAP、CDS helper、unit test、mapping、value construction、inline declaration 中都常见：

DATA(lv_text) = lt_texts[ langu = sy-langu ]-text.

这种代码可读性很高，但如果没有处理 not-found，就可能在生产中抛 CX_SY_ITAB_LINE_NOT_FOUND。很多初学者把 table expression 当成“新语法 READ TABLE”，结果写出下面这种错误心智模型：

DATA(ls_item) = lt_items[ item_id = lv_item_id ].
IF sy-subrc = 0.
  ...
ENDIF.

table expression 不按这个方式工作。它让代码更短，也更要求作者明确“找不到行是否是异常”。如果找不到就是程序错误，用异常语义没问题；如果找不到是业务正常分支，就应该用安全写法。

对 Java/Python 背景学习者，可以类比：

dict[key]
  -> key 不存在可能报错

dict.get(key, default)
  -> key 不存在返回默认值

ABAP table expression 也要区分“必须存在”和“可能不存在”。

20.4 Key mechanism / 核心机制

20.4.1 基本 key access

最常见形式：

DATA(ls_material) = lt_material[ matnr = lv_matnr ].

这和按 free key / key condition 读取一行类似。表类型仍然重要：

standard table
  -> 如果没有合适 key，可能线性搜索

sorted table
  -> 如果使用 key 的 leading part，可利用排序结构

hashed table
  -> 完整 unique key 查找是主场

secondary key
  -> 应显式指定 KEY keyname

table expression 不是性能魔法。把 READ TABLE 改成 itab[ ... ]，如果 table category 和 key 没改，性能问题可能仍然存在。

20.4.2 by index

按 index 读取：

DATA(ls_first) = lt_items[ 1 ].

这适合 index table，例如 standard table 和 sorted table。不要对 hashed table 建立“第 N 行”的业务模型；对静态已知的 hashed table，itab[ 1 ] 这类 index 访问本身就不是可用语义，语法检查或运行时都会阻止你把 hash table 当成有稳定行号的表来读。

20.4.3 named key

如果 internal table 有 secondary key，建议显式写 key name：

DATA(ls_item) = lt_items[
  KEY by_guid
  COMPONENTS item_guid = lv_guid
].

这与 READ TABLE ... WITH TABLE KEY by_guid COMPONENTS ... 的意图一致：告诉 reader 和 syntax check，你想使用哪个访问路径。

20.4.4 not-found：异常是默认风险

如果 table expression 找不到行，通常会抛出 CX_SY_ITAB_LINE_NOT_FOUND：

DATA(ls_item) = lt_items[ item_id = lv_item_id ].

如果 item_id 不存在，这不是 sy-subrc = 4，而是异常。只有在特定上下文中，系统提供了安全语义，例如：

IF line_exists( lt_items[ item_id = lv_item_id ] ).
  ...
ENDIF.

或：

DATA(ls_item) = VALUE ty_item(
  lt_items[ item_id = lv_item_id ] OPTIONAL ).

或：

DATA(ls_item) = VALUE ty_item(
  lt_items[ item_id = lv_item_id ] DEFAULT VALUE #( item_id = lv_item_id status = 'NEW' ) ).
20.4.5 line_exists

line_exists 只判断是否存在：

IF line_exists( lt_items[ item_id = lv_item_id ] ).
  " exists
ENDIF.

它适合替代：

READ TABLE lt_items TRANSPORTING NO FIELDS
  WITH KEY item_id = lv_item_id.
IF sy-subrc = 0.
  ...
ENDIF.

但 line_exists 不返回行内容，也不设置 sy-tabix。如果后续需要行内容，避免写成“先 line_exists 再同样 key 读一次”的重复 lookup；可以用 VALUE ... OPTIONAL、ASSIGN、READ TABLE 或异常处理。

20.4.6 line_index

line_index 可以取得 table expression 匹配行的 index：

DATA(lv_index) = line_index( lt_items[ item_id = lv_item_id ] ).

line_index 不会在 not-found 时抛 CX_SY_ITAB_LINE_NOT_FOUND。如果找不到行，它返回 0；如果找到的行是通过 hashed table 或 hashed secondary key 访问的，因为没有可表达的表索引，返回 -1。因此，只有在 index table 或 sorted key 场景里，lv_index > 0 才能被理解为“拿到了可用行号”。业务逻辑里不要把 index 当成长期稳定 key。

20.4.7 OPTIONAL 和 DEFAULT

如果找不到行是正常业务分支，OPTIONAL 很有用：

DATA(ls_item) = VALUE ty_item(
  lt_items[ item_id = lv_item_id ] OPTIONAL ).

找不到时返回 ty_item 的 initial value。

如果需要业务默认值：

DATA(ls_item) = VALUE ty_item(
  lt_items[ item_id = lv_item_id ]
  DEFAULT VALUE #( item_id = lv_item_id status = 'NEW' ) ).

OPTIONAL 和 DEFAULT 让“找不到怎么办”写在表达式里，比吞掉异常或事后检查不可靠状态更清楚。

20.4.8 ASSIGN 与 field symbol

如果想安全地指向一行并修改它：

ASSIGN lt_items[ item_id = lv_item_id ]
  TO FIELD-SYMBOL(<ls_item>).

IF sy-subrc = 0.
  <ls_item>-status = 'X'.
ENDIF.

在 ASSIGN 语句中使用 table expression 时，未找到可以通过 sy-subrc 判断。这是 table expression 的重要例外之一：普通读取位置的 table expression 不设置 sy-subrc，但 ASSIGN itab[ ... ] TO <fs> 这个模式会把 not-found 转成可检查的 sy-subrc。经典 READ TABLE ... ASSIGNING FIELD-SYMBOL(<fs>) 是同一类“命中后直接指向行”的写法，下一节 field symbol 会继续展开。

20.4.9 write position

table expression 可以出现在写位置：

lt_items[ item_id = lv_item_id ]-status = 'X'.

这很简洁，但风险也明显：

  • 找不到行会抛异常。
  • 对 sorted/hashed table 的 key 字段不能随意修改。
  • 如果表达式太长，可读性会下降。

不要把 table expression 当成“就地修改 key 的通道”。如果业务上真的要改变 key 值，通常要先删除旧行、再插入新行，或重新构造目标 internal table；否则很容易破坏 sorted/hashed table 的内部一致性。

复杂修改时，ASSIGN ... TO FIELD-SYMBOL 往往更清楚：

ASSIGN lt_items[ item_id = lv_item_id ]
  TO FIELD-SYMBOL(<ls_item>).

IF sy-subrc = 0.
  <ls_item>-status = 'X'.
  <ls_item>-changed_by = sy-uname.
ENDIF.
20.4.10 chaining

table expression 可以链式访问深层数据：

DATA(lv_text) = lt_orders[ order_id = lv_order_id
                         ]-items[ item_no = lv_item_no
                         ]-text.

这种写法很强，但不要为了炫技牺牲可读性。链越长，not-found 位置越难定位，异常处理也更复杂；初学阶段不用深究求值顺序，只要记住任一层 table expression 找不到行，都可能让整个表达式中断。教程建议：如果初学者看不清，就拆成两步。

20.5 日本项目现场说法

常见日语：

  • テーブル式
  • 内部テーブルアクセス
  • キー指定
  • 存在チェック
  • 未検出時の例外
  • OPTIONAL
  • DEFAULT
  • line_exists
  • line_index
  • READ TABLE との違い
  • sy-subrc を設定しない
  • 例外処理
  • 書込位置
  • フィールドシンボル

自然说明句：

  • テーブル式は READ TABLE と異なり、通常は sy-subrc を設定せず、未検出時に例外が発生します。
  • 未検出が業務上あり得る場合は、OPTIONAL、DEFAULT、line_exists、または例外処理を明示します。
  • 二次キーを使う意図がある場合は、テーブル式でも KEY 指定を明確にします。
  • 変更処理が複数項目に及ぶ場合は、テーブル式の書込位置より ASSIGN の方が読みやすい場合があります。

review 里常见问法：

  • このテーブル式で該当行が存在しない場合、例外は問題ありませんか。
  • OPTIONAL または DEFAULT を使うべきケースではありませんか。
  • sy-subrc をチェックしていますが、この式では設定されません。
  • 該当行が無い場合に CX_SY_ITAB_LINE_NOT_FOUND が発生します。sy-subrc は設定されないため、業務上未検出があり得るなら OPTIONAL / DEFAULT / line_exists、または例外処理で明示してください。
  • 二次キーを使う意図なら KEY 名を指定してください。

20.7 Example / 学习例子

20.7.1 例 1：必须存在时直接读取
DATA(ls_material) = lt_material[ matnr = lv_matnr ].

如果业务语义是“这行必须存在，否则就是程序错误”，这种写法可以接受。但如果找不到是正常分支，就要换安全写法。

20.7.2 例 2：可能不存在，用 OPTIONAL
DATA(ls_material) = VALUE ty_material(
  lt_material[ matnr = lv_matnr ] OPTIONAL ).

IF ls_material IS INITIAL.
  " not found or initial line; be careful if initial line is meaningful
ENDIF.

OPTIONAL 避免异常，但返回 initial value。要小心 initial line 本身是否有业务意义。

20.7.3 例 3：可能不存在，用 DEFAULT
DATA(ls_material) = VALUE ty_material(
  lt_material[ matnr = lv_matnr ]
  DEFAULT VALUE #( matnr = lv_matnr mtart = 'UNKN' ) ).

DEFAULT 更适合有明确 fallback 值的场景。

20.7.4 例 4：只判断存在
IF line_exists( lt_material[ matnr = lv_matnr ] ).
  " exists
ENDIF.

不要再写 IF sy-subrc = 0，因为 line_exists 不设置 sy-subrc。

20.7.5 例 5：使用 secondary key
DATA(ls_item) = lt_items[
  KEY by_guid
  COMPONENTS item_guid = lv_guid
].

如果存在 named secondary key，显式写 key name 可以让访问路径清楚。

20.7.6 例 6：修改表内一行
lt_items[ item_id = lv_item_id ]-status = 'X'.

简单修改可以这样写；复杂修改建议：

ASSIGN lt_items[ item_id = lv_item_id ]
  TO FIELD-SYMBOL(<ls_item>).

IF sy-subrc = 0.
  <ls_item>-status = 'X'.
  <ls_item>-changed_at = cl_abap_context_info=>get_system_date( ).
ENDIF.
20.7.7 例 7：不要把表达式链写到不可读
DATA(lv_text) = lt_orders[ order_id = lv_order_id
                         ]-items[ item_no = lv_item_no
                         ]-texts[ langu = sy-langu ]-text.

这很短，但异常点可能有三个。初学和项目代码 review 中，必要时拆开并分别处理 not-found。

20.8 Counterexample / Failure mode

20.8.1 误区 1：table expression 后检查 sy-subrc
DATA(ls_item) = lt_items[ item_id = lv_item_id ].
IF sy-subrc = 0.
  ...
ENDIF.

这是错误心智模型。普通读取位置的 table expression 不按 READ TABLE 方式设置 sy-subrc；把 table expression 放进 ASSIGN 源位置时才是例外。

20.8.2 误区 2：忽略 not-found 异常
DATA(lv_status) = lt_items[ item_id = lv_item_id ]-status.

如果找不到行，会抛 CX_SY_ITAB_LINE_NOT_FOUND。如果这是正常业务分支，要使用 OPTIONAL、DEFAULT、line_exists 或异常处理。

20.8.3 误区 3：先 line_exists 再同样 key 读一次
IF line_exists( lt_items[ item_id = lv_item_id ] ).
  DATA(ls_item) = lt_items[ item_id = lv_item_id ].
ENDIF.

这会做两次 lookup。小表没关系，大表或复杂 key 下要考虑 ASSIGN、VALUE ... OPTIONAL 或一次性异常处理。

20.8.4 误区 4：隐藏 secondary key

如果 table type 已经有 by_guid，但表达式只写：

DATA(ls_item) = lt_items[ item_guid = lv_guid ].

reviewer 不容易确认访问路径。性能敏感处应显式写 KEY by_guid COMPONENTS ...。

20.8.5 误区 5：表达式过长

现代语法不是越短越好。如果一行 table expression 混合多层读取、fallback、转换、方法调用，读者很难判断 not-found 和异常边界。书写时优先可读性。

20.9 思考问题

  • 找不到行是程序错误，还是正常业务分支？
  • 如果找不到是正常分支，应该用 OPTIONAL、DEFAULT、line_exists 还是 READ TABLE？
  • 这个 table expression 是否依赖 secondary key？是否显式写了 KEY keyname？
  • 这个表达式是否会在循环中大量执行？table category 是否合适？
  • 是否有人在 table expression 后检查 sy-subrc？
  • 是否需要修改表内行？ASSIGN 是否更清楚？
  • expression chain 是否太长，是否应该拆成多个步骤？
  • 用日语 review 时，如何说明 未検出時の例外 和 OPTIONAL 的必要性？
