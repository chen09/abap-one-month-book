Chapter 01 - SAP 和 ABAP 基础

11. ABAP package 包与子包

  • Chapter 01
  • Section 11
  • self-contained study note

本节结论

ABAP package 包与子包，是 ABAP Repository 中组织开发对象、连接 transport/software logistics、限定开发边界的基本容器；本节要学的是：为什么 repository object 不能像本地文件一样随便放，为什么正式开发对象要放进合适的 package，而不是放在 $TMP 或错误的 package 层级里。

正文

前一节讲 Repository object 仓库对象 时，已经知道 ABAP class、CDS view entity、DDIC table、behavior definition、service definition、service binding 等都是 SAP 系统里的开发对象。package 要回答下一层问题：这些对象放在哪里、归谁组织、能不能被传输、属于哪个软件交付单元、在 ADT 里从哪里进入。

ABAP package 不是 Eclipse 里的普通文件夹，也不是 Java/Python 的 package / module 目录。Java/Python 项目里，一个目录常常只是文件系统结构；ABAP 里，package 是 SAP Repository 管理的开发组织单元。它可以包含多个 repository objects，也会带有 package type、software component、transport layer、application component、superpackage 等属性。开发者在 ADT 里看到的 package 树，背后不是本地目录，而是 ABAP 后端系统里的 package model。

最简单的理解是：package 是“正式开发对象的家”。创建一个 ABAP class 时，ADT 会问 package；创建 CDS、RAP behavior、service binding 时，也要放入 package。放进不同 package，不只是界面上位置不同，还会影响 transport request、transport layer、software component、权限和后续影响分析。日本项目里经常会问：このオブジェクトはどのパッケージに入っていますか。这个问题本质上是在确认对象是否放在正确的交付边界里。

还要区分 local package 与 transportable package。$TMP 是常见的 local / temporary package，适合临时试验，不适合交付到 QAS/PRD 的项目成果。正式项目对象如果放在 $TMP，后面通常会遇到“为什么不能随 transport 去测试系统”“为什么交付清单里没有这个对象”“为什么别人系统看不到”的问题。Day 02 不要求掌握完整 CTS/TMS，但必须知道：package 是 transport 之前的第一道组织边界。

最后，本节只浅讲 package hierarchy。package 可以有 superpackage / subpackage 父子结构。subpackage 帮助把一个较大的业务或技术领域拆成更小的开发单元；但它不是随便嵌套的本地目录。在很多环境中，subpackage 会继承 superpackage 的 software component，不能自己改成另一个 software component。你还会看到 Structure package、Main package、Development package 等 package type；Day 02 只要记住：直接创建 class/CDS/table 等 repository objects 的通常是 Development package，Structure/Main package 更偏组织层级，不要把它们当成随便放对象的普通目录。

为什么重要

package 是初学者最容易轻视、但真实项目最常卡住的概念之一。能写一段 ABAP 代码，不等于能把它交付到项目里。项目里要能说明：对象属于哪个 package，package 是否可传输，transport layer 是否正确，是否放在客户命名空间，是否和同一功能域的对象放在一起，是否会被 package interface / package check 限制。

对 Java/Python 背景学习者来说，最容易误判的是把 package 当成本地目录。于是会出现几类典型错误：随手把练习对象放到 $TMP，后来发现不能送到 QAS；在错误 superpackage 下创建 subpackage，后来发现 software component 不匹配；把对象放进结构层 package，后来创建 class/CDS 失败；只看对象名，不看 package，导致影响调查时漏掉相关对象。ABAP 的“对象位置”是一部分架构信息，不只是整理文件用的视觉习惯。

对日本 SAP 项目交付来说，package 还连接到设计书、移送清单和障害调查。设计说明里经常要列 object name、object type、package、transport request；移送前要确认对象都进入正确 request；故障调查时要从 package、where-used、transport history 追踪影响范围。package 放错，不一定马上编译失败，但会在 review、移送、本番反映或保守阶段变成成本。

核心机制

第一层机制是 container。ABAP package 把 repository objects / development objects 组织成一个 self-contained development unit。也就是说，它把逻辑上属于同一开发单元的对象聚在一起。一个业务功能可能有 class、interface、CDS view entity、behavior definition、service definition、service binding、message class、authorization 相关对象；它们不应该散落在无关 package 中。

第二层机制是 package attributes。常见属性包括 package name、short description、package type、superpackage、software component、transport layer、application component、person responsible 等。不同系统、release、ABAP Cloud / on-premise 形态下字段和可选项会不同，但思路相同：package 不只是名字，而是带着软件物流和项目组织信息。

第三层机制是 package type。初学者先记住三类常见说法：Structure package 用来搭顶层结构，Main package 用来组织较大的语义区域，Development package 才是直接承载 repository objects 的日常开发包。更准确地说：Structure package 与 Main package 本身不直接存放 class、CDS、table 等 repository object，它们主要承载下级 package 与自己的 package interface；只有 Development package 才能直接容纳日常开发对象。真实项目中不要把所有层级都当成“文件夹”；如果 ADT 不允许在某个 package 下创建对象，第一反应应当是确认 package type 和 superpackage，而不是怀疑代码语法。

第四层机制是 superpackage / subpackage。subpackage 是某个 superpackage 下的子包，用于细分功能、团队或技术层。它通常继承上层 package 的 software component 等上下文；尤其在 BTP ABAP Environment / ABAP Cloud 相关场景里，software component 是 package 上方的重要组织边界。需要特别注意：subpackage 不能随意换成另一个 software component，否则容易遇到软件组件不匹配类错误，例如 ADT 提示 Invalid software component。

第五层机制是 transport layer 与 $TMP。transport layer 决定 package 及其中对象的 transport behavior；local package / $TMP 不用于正式跨系统交付。把对象放进可传输 package 后，创建或修改对象通常会关联 Transport request 传输请求；把对象放在 $TMP，则更像本机临时草稿，不应作为项目 deliverable。

第六层机制是 namespace。客户自开发对象通常以 Z 或 Y 开头，package 也通常遵守项目命名规范。更完整的客户 namespace、partner namespace、命名规则会在 Japanese SAP naming convention 和项目标准里展开；Day 02 只要知道：在 ADT 创建 package 或 repository object 时，第一字符就可能决定它是不是合规客户对象。

第七层机制是 package interface / package check。package 不只是装对象，还可以表达可见性和依赖边界。package interface 用于公开某些对象给其他 package 使用；package check 用于检查 package 之间的使用关系是否违反规则。本节只建立 awareness，不要求设计复杂 package interface。后面讲架构、Clean Core、released APIs 和大型项目依赖治理时，再深入理解 package boundary。

学习例子

假设今天要创建一个练习 class ZCL_ABAP_LEARN_01。在 ABAP project in ADT 中打开 Project Explorer，先确认目标 package，例如 ZABAP_LEARN_##，已经加入 Favorite Packages。然后在 package 上选择 New -> ABAP Class，输入 class name、description，并在向导中确认 package、transport request 等信息。完成后，这个 class 就不是一个本地 .java 或 .py 文件，而是某个 package 里的 ABAP repository object。

再观察 package 本身。打开 Properties 或创建向导时，注意 package type、superpackage、software component、transport layer、application component。看到这些字段时，不要急着全部背下来，而是问三个问题：它是不是 Development package？它是不是 $TMP？它是否属于预期的 software component / transport layer？这三个问题足够支撑 Day 02 的判断。

如果练习环境是 BTP ABAP Environment 或 SAP S/4HANA Cloud Public Edition，package 字段可能比 on-premise 示例更受限制。不要把 SAP Learning 里某个截图当成所有系统都一样。关键判断仍然相同：对象必须在允许的 package / software component / customer namespace 范围内创建，不能随便放到 SAP standard package，也不能随便修改标准对象。另外，ADT 创建向导里的 classical transport request 心智模型主要来自 on-premise / Private Edition；BTP ABAP Environment 通常围绕 software component、Manage Software Components 和 Git-based 生命周期组织移送，不要照搬经典 TMS / SE09 的判断方式。

常见误区

  • 误区 1：把 package 当成本地 folder。修正：package 是 SAP Repository / software logistics 的对象组织边界，不只是 UI 树节点。
  • 误区 2：把正式项目对象放在 $TMP。修正：$TMP 适合临时练习，不适合作为要送到 QAS/PRD 的交付对象位置。
  • 误区 3：在错误 package type 下创建对象。修正：Structure/Main package 偏组织层，日常开发对象通常放在 Development package 下。
  • 误区 4：以为 subpackage 可以随便换 software component。修正：subpackage 通常继承 superpackage 的 software component，创建前先确认父包。
  • 误区 5：只记 package name，不看 transport layer / software component。修正：package name、transport layer、software component 一起决定对象在项目交付链中的位置。
  • 误区 6：Day 02 就深挖 package interface。修正：先知道 package interface / package check 是依赖边界工具；详细设计留到架构治理章节。
