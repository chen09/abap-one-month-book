12. DCL role 访问规则

  • Chapter 03
  • Section 12
  • self-contained study note

12.1 一句话结论

DCL role 访问规则，是 ABAP CDS 里用 define role / grant select on / where 把“谁能读到哪些行”写成声明式规则的对象。上一节 CDS access control 访问控制 讲的是机制和边界，本节要把它落到代码：DCL role 不是 PFCG role 本身，也不是 UI 权限；它是 CDS access control 的规则层，会在读取被保护 CDS entity 时参与结果过滤。

初学者要先记住一句项目话：@AccessControl.authorizationCheck 只说明 CDS entity 对 access control 的态度，真正的过滤条件写在 DCL source 的 access rule 里。没有 DCL role，#CHECK 不会自动保护数据；DCL role 写错字段、写错 authorization object、漏移送或漏授权值，用户看到的数据就会和预期不一致。

12.2 到底学什么

本节学习 DCL role 的最小可读结构，而不是一次吃完所有 DCL 语法。核心问题有六个：

  1. define role 定义的是什么对象？
  2. grant select on 保护哪个 CDS entity？
  3. where 后面的 access condition 怎么影响结果？
  4. literal condition、PFCG condition、user condition、inheritance condition 分别适合什么场景？
  5. 多个 access rule 或多个 DCL role 同时存在时，限制是变松还是变严？
  6. 项目上线时，DCL source、CDS entity、authorization object、PFCG/IAM role 之间如何一起检查？

SAP-samples 的 authorization cheat sheet 给了最适合入门的链路：先创建带 @AccessControl.authorizationCheck: #CHECK 的 CDS view entity，再创建 access control role，使用 @MappingRole: true、grant select on ... where ... aspect pfcg_auth(...)，最后通过 ADT data preview 或 ABAP SQL 观察返回结果是否只包含当前用户有权查看的数据。

12.3 为什么重要

DCL role 是“模型层授权”真正落地的地方。没有它，团队很容易停留在“我们在 CDS 上写了 #CHECK”这种半成品状态。真正 review 时，顾问、开发和 Basis/安全团队会问的是：

  • 保护的是哪个 CDS entity？
  • 用哪个 authorization object？
  • CDS 字段和 authorization field 怎么映射？
  • ACTVT = '03' 是否只代表显示/读取？
  • 多个规则叠加后是 OR 还是 AND？
  • 这个 DCL source 是否和 CDS view 一起进入 transport？
  • 用户业务角色里是否真的有对应授权值？

对 Java/Python 背景读者，可以把 DCL role 类比成“声明式 row-level filter + 当前用户权限上下文”的组合。但 SAP 的关键差异在于：DCL role 是 ABAP repository object，依赖 SAP 授权对象和 PFCG/IAM 分配，且由 ABAP SQL / SADL / RAP 读路径隐式使用。它不是应用代码里的 if user.org in ...，也不是数据库原生 row-level security 的简单替代。

12.4 Key mechanism / 核心机制

12.4.1 最小结构：role、entity、rule、condition

一个典型 DCL role 长这样：

@EndUserText.label: 'Sales order access by sales org'
@MappingRole: true
define role ZR_SO_ACCESS {
  grant select on ZI_SalesOrder
    where ( SalesOrg ) =
      aspect pfcg_auth( ZSO_AUTH, VKORG, ACTVT = '03' );
}

读这段时，不要从语法细节迷路。先看四层：

  • define role ZR_SO_ACCESS：定义一个 CDS role / DCL role。
  • grant select on ZI_SalesOrder：这条 access rule 作用在 ZI_SalesOrder 这个 CDS entity 上。
  • where ...：定义 access condition，也就是读这个 CDS entity 时额外追加的限制。
  • aspect pfcg_auth(...)：把 CDS 字段映射到 SAP authorization object / authorization field，并用当前用户授权值决定哪些行可读。

@MappingRole: true 表示这个 role 作为 mapping role 参与 CDS access control 评估。它不是把 DCL role 直接分配给用户；用户能看到什么，仍然取决于 where 条件和当前用户在 PFCG/IAM 中被授予的 authorization object value。

12.4.2 literal condition：固定条件

最容易理解的 access condition 是固定值过滤：

@MappingRole: true
define role ZR_SO_EUR_ONLY {
  grant select on ZI_SalesOrder
    where Currency = 'EUR';
}

这表示读 ZI_SalesOrder 时，只允许读 Currency = 'EUR' 的行。literal condition 适合表达稳定、业务上明确的固定限制，例如只开放某个数据区域的演示数据。但项目上不能滥用它替代授权设计：如果不同用户应看到不同组织、公司代码、销售组织或工厂，通常要用 PFCG condition，而不是写死一个值。

literal condition 可以和 PFCG condition 组合。SAP Keyword Documentation 的示例就把 aspect pfcg_auth 和 currcode = 'EUR' 放在同一个 where 里，表示既要符合当前用户授权，也要符合固定币种条件。这里要特别看清 AND 和 OR：AND 通常让条件更严格，用户既要有授权，又要满足固定过滤；OR 可能让条件变宽，如果另一侧条件过宽，就可能绕开原本期待的 PFCG 限制。DCL 里的 OR 不是“多加一层保险”，而是“多开一条可通过的路”。

12.4.3 PFCG condition：把 CDS 字段映射到授权对象

项目上最常见的是 PFCG condition：

@MappingRole: true
define role ZR_SO_BY_SALES_ORG {
  grant select on ZI_SalesOrder
    where ( SalesOrg ) =
      aspect pfcg_auth(
        ZSO_AUTH,
        VKORG,
        ACTVT = '03'
      );
}

这里的含义是：

  • SalesOrg 是 CDS entity 里的字段。
  • ZSO_AUTH 是 authorization object。
  • VKORG 是 authorization field。
  • ACTVT = '03' 表示只看 display/read 相关授权。

SAP Keyword Documentation 对 pfcg_auth 的要点是：左侧括号里的 CDS elements 会和右侧 authorization object 的 authorization fields 按顺序映射；当前用户的授权值会被评估，只有存在授权的行会被读出。多个 authorization values 会组合成逻辑条件，实际 SQL 形态可能随 release 和优化方式变化，因此项目上不要依赖某个固定 SQL 展开形式，而要从 DCL 语义和测试结果验证。

如果左侧有多个 CDS 字段，例如公司代码和工厂：

where ( CompanyCode, Plant ) =
  aspect pfcg_auth( ZMM_AUTH, BUKRS, WERKS, ACTVT = '03' );

字段数量和映射顺序就非常重要。CompanyCode 对 BUKRS，Plant 对 WERKS。写反了不会变成“差不多也能用”，而是授权语义直接错位。

12.4.4 空左侧括号：只检查有没有对象授权

DCL 也允许左侧括号为空：

@MappingRole: true
define role ZR_SO_DISPLAY_ALLOWED {
  grant select on ZI_SalesOrder
    where ( ) = aspect pfcg_auth( ZSO_AUTH, ACTVT = '03' );
}

这不是行级过滤，而是“当前用户是否至少有这个 authorization object / activity 的授权”。如果有，整个 CDS entity 可读；如果没有，就读不到数据。它适合非常粗粒度的 read gate，不适合公司代码、销售组织、工厂这种需要按字段值过滤的场景。

初学者常犯的错误是：看到 pfcg_auth 就以为一定是行级过滤。是否行级过滤，要看左侧是否把 CDS element 映射到了 authorization field。

12.4.5 user condition：按当前用户过滤

除了 literal condition 和 PFCG condition，DCL 还可以用 user condition 按当前登录用户过滤数据。它适合表达“只能看自己负责的数据”“只能看自己创建的数据”这类场景。典型建模方式是 CDS entity 里有 CreatedBy、ResponsibleUser、Owner 之类字段，然后 access condition 把这些字段和当前用户关联起来。

初学时不要把 user condition 当成 PFCG 的替代品。user condition 解决的是“数据行属于谁”的过滤；PFCG condition 解决的是“当前用户是否拥有某个授权对象和字段值”的检查。项目上经常两者一起出现：PFCG 决定组织、工厂、公司代码等授权范围，user condition 决定本人相关的数据边界。

12.4.6 多个规则：先分清两个层次

同一个 CDS entity 可能被多个 DCL role 或多个 access rule 影响。判断“限制变严还是变松”之前，先分清两个层次：

  • 第一层是同一个 where 条件里的 AND / OR。AND 通常收紧访问，OR 通常放宽访问。
  • 第二层是同一个 CDS entity 上多个 access rule / 多个 DCL role 的组合。很多情况下，多个可用规则会形成 OR 语义：用户只要满足其中一条可通过路径，就能读到相应数据。

这也是 review DCL 时最容易出错的地方。团队看到“又加了一个 DCL role”，直觉上以为更安全；但如果它给同一个 CDS entity 增加了一条更宽的 grant select on 路径，结果可能是访问范围变大。真正要 review 的不是“role 数量”，而是最终 access condition 的组合语义。

COMBINATION MODE AND|OR 是 DCL 里控制规则组合的高级机制，用于声明同一 CDS entity 上多个 access rule 的组合方式。它不适合作为初学者复制粘贴的模板，因为语法位置、可用场景和 release 细节要以当前 ABAP Keyword Documentation 为准。现阶段只要记住：遇到 COMBINATION MODE，必须停下来单独 review，确认它是在收紧还是放宽最终访问范围。

12.4.7 inheritance condition：继承已有条件

CDS 模型经常是 view-on-view：底层 interface view、root view、projection view、consumption view 一层层叠上去。上一节已经提醒过，底层 DCL 不一定自动保护外层入口。因此 DCL 提供 inheritance condition，把已有 role 或 entity 的条件继承到当前 entity 上。

典型语义是：当前 view 复用了某个底层 CDS entity，就把底层 access condition 也纳入当前 view 的访问规则。这样可以避免“底层看起来有 DCL，外层服务入口却没继承授权语义”的安全空洞。

本书这里先不要求背所有 inheritance 变体。先记住项目规则：

  • 如果一个外层 CDS entity 被直接暴露给 service 或 ABAP SQL，必须确认它自己的 access control 入口。
  • 如果外层语义应沿用底层 DCL，就要用 inheritance condition 或相应的 projection/inherited access-control 机制清楚表达。
  • 继承不是免费复制。底层字段、别名、关联变化后，继承条件可能失效，需要 activation 和测试一起验证。

不同 ABAP release 和不同语法位置中，inheritance 相关写法可能以 INHERITING CONDITIONS、INHERIT ... FOR GRANT SELECT ON ...、INHERITING CONDITIONS FROM SUPER 等形式出现。初学者不要死背某一个片段；真正要掌握的是：外层 view 是否显式继承或重新定义了底层 access condition，以及这个继承关系在 view-on-view 链路中是否仍然成立。

12.4.8 full access、REDEFINITION 和危险开关

DCL 里还存在 full access rule、REDEFINITION 等高级语法。初学者先不要把它们当成常规工具。它们通常用于明确放开、替换或覆盖既有规则，项目风险比普通 where 条件高。

在客户项目中，一看到下面几类写法，就应该提高 review 强度：

  • 规则等价于 full access。
  • 使用 REDEFINITION 覆盖原有规则。
  • 使用 COMBINATION MODE OR 新增很宽的访问路径。
  • 用 literal TRUE 或类似方式让 access condition 实际失效。
  • 为了让测试通过临时放宽 DCL，但没有记录回收计划。

这些写法不一定永远错误，但必须有明确设计理由、测试证据和审批记录。

12.4.9 activation、性能和共享系统影响

DCL source 激活后，会影响后续对相应 CDS entity 的访问结果。在共享 DEV / QAS 系统里，这不是一个只影响自己编辑器的本地改动；同一个 CDS entity 的 Data Preview、ABAP SQL 查询、Fiori list 或 OData 读取，都可能因为 DCL 变化而看到不同数据。因此 DCL 修改要和 CDS entity、authorization object、PFCG/IAM role、transport 一起检查。

复杂 DCL 条件还可能影响 SQL 展开和执行计划。SAP Help 把 DCL 描述为在读 CDS entity 时隐式加入的访问控制条件；从运行时看，它会参与最终 SQL 条件、join 或子查询的生成方式。项目上如果 DCL-protected CDS entity 变慢，不能只看 ABAP 代码，也要用 ST05 SQL trace / SQL Monitor 观察 DCL 条件是否让查询变复杂。

本节不要求初学者优化 DCL 性能，但要形成一个 review 习惯：任何包含 PFCG condition、inheritance condition、多层 association、多个 access rule 的 CDS entity，都应该至少做一次“不同用户、不同授权、不同入口”的读取验证。权限逻辑不是写完就结束，必须用系统行为证明。

12.7 Example / 学习例子

12.例 1：国家维度的入门例子

SAP-samples authorization cheat sheet 使用 countryfr 做入门示例。简化理解如下：

@AccessControl.authorizationCheck: #CHECK
define view entity ZDEMO_FLIGHT_AUTH
  as select from zdemo_abap_flsch
{
  key carrid,
  key connid,
      countryfr,
      cityfrom,
      countryto,
      cityto
}

对应 DCL：

@EndUserText.label: 'Flight access by country'
@MappingRole: true
define role ZR_FLIGHT_COUNTRY {
  grant select on ZDEMO_FLIGHT_AUTH
    where ( countryfr ) =
      aspect pfcg_auth( ZAUTH_OBJ, ZAUTH_CTRY, ACTVT = '03' );
}

如果当前用户的授权值只允许 ZAUTH_CTRY = 'US' 且 ACTVT = '03'，读这个 CDS entity 时就只应该看到 countryfr = 'US' 的记录。

12.例 2：销售组织 + 固定状态

项目中经常需要把用户授权和固定业务条件结合：

@MappingRole: true
define role ZR_SO_RELEASED_BY_ORG {
  grant select on ZI_SalesOrder
    where ( SalesOrg ) =
      aspect pfcg_auth( ZSO_AUTH, VKORG, ACTVT = '03' )
      and OverallStatus = 'RELEASED';
}

这表示用户不仅要有销售组织授权，还只能读已发布状态的订单。这里的 and 很关键：如果误写成另一个很宽的 OR 规则，可能把授权范围放大。

12.例 3：如何验证

最小验证流程：

  1. 在 ADT 打开 CDS entity，确认 @AccessControl.authorizationCheck。
  2. 找到 DCL source，确认 define role 和 grant select on 的目标 entity。
  3. 检查 where 条件里的 CDS element、authorization object、authorization field、ACTVT。
  4. 用测试用户打开 ADT Data Preview、Fiori list 或 ABAP SQL 测试程序。
  5. 换一个授权值不同的用户或测试角色，确认返回结果不同。
  6. 如果用户看不到数据，检查是没有底层数据、DCL 条件过滤、PFCG/IAM 角色缺值，还是外层 view 没有继承条件。

12.8 Counterexample / Failure mode

12.误区 1：把 DCL role 当成 PFCG role

DCL role 是 repository object，PFCG role / business role 是用户授权分配。DCL role 规定“如何按授权过滤 CDS 数据”，PFCG/IAM 规定“当前用户有哪些授权值”。两者缺一不可。

12.误区 2：字段映射顺序写错

where ( CompanyCode, Plant ) = aspect pfcg_auth( ZMM_AUTH, BUKRS, WERKS, ACTVT = '03' ) 依赖顺序。如果 CDS 字段和 authorization field 对不上，结果可能过严、过松或完全不符合业务。

12.误区 3：只看当前 DCL 文件

同一个 CDS entity 可能有多个 access rule 或多个 DCL role。只看一个文件就判断“权限没问题”，很容易漏掉其它 OR 规则带来的放宽，或 AND 规则带来的额外限制。

12.误区 4：忘记外层 view 的入口

底层 interface view 有 DCL，不等于外层 consumption/projection view 自动安全。真正被 ABAP SQL、SADL、OData 或 Fiori 消费的入口对象必须单独确认。

12.误区 5：上线漏移送

DCL source、CDS entity、authorization object、PFCG/IAM role 配置不在同一个文件里。开发系统能跑，不代表测试/生产也能跑。尤其 #MANDATORY 场景，漏掉 DCL 或授权配置会直接造成用户读不到数据。

12.误区 6：把 full access 当临时修复

为了赶测试临时放开 DCL，是很危险的项目行为。full access、REDEFINITION、宽 OR 规则都要留下设计理由、审批和回收计划。

12.9 思考问题

  • 这个 DCL role 保护的 CDS entity 是哪个？
  • grant select on 的 entity 是否就是 service / ABAP SQL 直接消费的入口？
  • where 条件是 literal、PFCG、user，还是 inheritance？
  • CDS element 和 authorization field 的映射顺序是否正确？
  • ACTVT = '03' 是否只代表读/display？修改权限在哪里控制？
  • 多个 DCL role / access rule 的组合会让限制变松还是变严？
  • 是否需要继承底层 view 的 access condition？
  • DCL source、authorization object、PFCG/IAM role 是否都进入移送/配置清单？
  • 如何用两个不同授权值的测试用户证明 DCL 生效？
