26. SQL Trace performance analysis

  • Chapter 04
  • Section 26
  • self-contained study note

26.1 一句话结论

SQL Trace performance analysis 是用运行时证据定位数据库访问瓶颈的方法；在本教程里要学的是：当 ABAP 程序、CDS、OData 或后台作业变慢时，不靠感觉猜 SQL 问题，而是用 trace 看实际执行了哪些 SQL、执行了多少次、花了多久、读了多少数据。

ATC check set 质量门禁 可以静态提示潜在性能风险，例如循环里 SELECT、缺少 WHERE、未使用 key 等；SQL Trace 关注实际运行时。它回答的是：这一段操作真实发出了哪些数据库访问？哪个 SQL 最慢？是不是同一条 SQL 被执行了上千次？是否读取了远超需要的数据？是否缺少选择条件？

SQL Trace 不是万能性能工具。它主要帮助分析数据库访问，不负责解释全部 ABAP CPU 时间、UI 渲染、网络、锁等待、外部系统调用或应用服务器内存问题。遇到整体性能问题时，它要和 ABAP Profiler performance analysis、SQL Monitor 生产级 SQL 观察、应用日志和业务输入条件一起使用。

26.2 到底学什么

第一，要学 trace 的范围。SQL Trace 应该尽量小范围、可复现、可比较：明确用户、时间窗口、事务/API/job、输入条件、数据量。范围太大，trace 会充满无关 SQL；范围太小，又可能漏掉真正慢的步骤。

第二，要学看结果。初学者不要只看“总时间最长”一列，还要看执行次数、单次耗时、总耗时、读取行数、数据库表/视图、调用位置、是否重复执行。一次慢 SQL 和一条快 SQL 被调用一万次，是两类不同问题。

第三，要学典型问题模式。常见模式包括：循环内 SELECT 造成 N+1；WHERE 条件不完整；没有限制数据范围；缺少明确排序导致后续逻辑不稳定；JOIN 条件不正确导致行数膨胀；读取了很多字段但只用少数字段；CDS association/path expression 在特定消费下展开成昂贵 SQL。

第四，要学 trace 证据如何进入项目沟通。性能缺陷单里最好记录：测试用户、输入条件、数据量、trace 时间、最耗时 SQL、调用对象、怀疑原因、修改前后对比。没有证据的“感觉很慢”很难推动正确优化。

第五，要学 trace 的边界。SQL Trace 能告诉你数据库访问发生了什么，但它不能单独说明为什么业务流程设计成这样，也不能直接证明 UI、网络、authorization、lock、RFC、外部 API 或 ABAP CPU 是瓶颈。看到 SQL 热点后，还要回到业务输入、代码路径、CDS 分层、权限过滤和事务边界一起判断。

第六，要学“可复现”的价值。性能分析最怕一次性偶然结果：缓存是否已热、数据量是否一致、用户权限是否不同、后台负载是否异常、网络是否波动，都会影响结论。因此书稿里要训练读者写清楚复现条件，而不是只保存一张 trace 截图。

26.3 为什么重要

性能优化最大的风险是猜错方向。开发者看到程序慢，可能先重构 ABAP 内表，实际瓶颈是数据库重复访问；也可能怀疑数据库，实际慢在外部 RFC 或应用层循环。SQL Trace 让团队先看证据，再决定是否改 SQL、加缓存、批量读取、调整 CDS、减少字段，或在性能/Basis/DB 专家判断后调整数据库对象。

对 SAP 项目交付来说，性能不是上线后再说的“优化项”。批量处理、报表、Fiori 列表、OData 查询、后台 job，如果在真实数据量下过慢，会直接影响业务窗口、夜间批处理、用户体验和系统资源。Trace 证据能让性能问题从争论变成可处理任务。

对 Java/Python 背景学习者来说，SQL Trace 类似数据库 query log / ORM SQL logging / APM database span，但 SAP 场景会跨 ABAP SQL、CDS、Open SQL、DB interface、应用服务器和业务事务。开发者要学会从 SAP 工具看到“ABAP 代码最终对数据库做了什么”。

在日本项目里，SQL Trace 还是沟通工具。性能问题很容易变成“开发说数据库慢、Basis 说代码慢、业务说系统慢”的循环。Trace 证据能把讨论压缩到具体对象：哪个用户、哪个输入、哪个 SQL、多少次、多少行、多少时间、从哪个 ABAP/CDS/OData 路径触发。它不是为了追责，而是为了让修正方向不漂。

26.4 Key mechanism / 核心机制

SQL Trace 的基本流程是：准备可复现场景，开启 trace，执行目标操作，停止 trace，过滤和分析结果，记录结论。常见 on-premise 语境中会提到 ST05，但具体工具入口随 release、权限和部署模式不同，项目中要以系统实际可用工具为准。

分析结果时先看几个指标：

  • Executions：SQL 执行次数。次数异常高常提示循环内访问或重复读取。
  • Duration / Total time：总耗时和单次耗时。总耗时高可能来自次数多或单次慢。
  • Records / Rows：读取行数。行数远超业务需要，常提示 WHERE 条件或 join/filter 问题。
  • Object：访问的表、view、CDS 相关对象。
  • Call position：从哪个 ABAP 程序、类或框架路径触发。

在 classic / on-premise 语境下，SQL Trace 还可能帮助观察 table buffer 相关现象，例如某些访问是否经过表缓冲、是否频繁绕过缓冲。初学阶段不需要把 table buffering 当成主要优化手段，但看到这类线索时，要知道它属于 SAP 数据访问行为的一部分，需要结合表设置、数据一致性要求和项目规范判断。

常见优化方向包括：把循环内 SELECT 改成批量读取；补充 WHERE 条件；减少读取字段；用合适的 key 和排序；检查 join 条件；把重复读取缓存到内表；重新设计 CDS 消费方式；对大数据量场景分页或分批处理。具体是否需要数据库索引或 HANA 层优化，应由有权限的性能/Basis/DB 专家结合系统证据判断。

还要保留修改前后对比。同样输入条件下，优化前 trace 和优化后 trace 对比最有说服力。只说“我改了，应该快了”不够；要能说明 SQL 次数减少、读取行数减少、总耗时下降或热点转移。

一个更完整的 SQL Trace 作业模板如下：

Before trace
  - Identify one scenario: UI action, API call, report, or background job.
  - Fix user, system/client, input parameters, data volume, and time window.
  - Decide what "slow" means: response time, job duration, row count, timeout.

During trace
  - Start trace only for the target user/process when possible.
  - Execute the smallest reproducible operation.
  - Stop trace immediately after the target operation.

After trace
  - Sort by total time and execution count.
  - Identify repeated SQL, large row reads, missing filters, expensive joins.
  - Map SQL back to ABAP class/report/CDS/service path.
  - Record the top findings and compare with business expectation.

After change
  - Rerun the same scenario with the same input and comparable data volume.
  - Compare executions, total time, rows, and result correctness.

判读时可以用下面的表：

Trace 现象常见含义下一步
Executions 很高，单次很快N+1 / loop SELECT / 重复读取批量读取、缓存、调整调用位置
单次 Duration 很高SQL 本身昂贵、join/filter/key 问题看 WHERE、join、索引/DB 计划、CDS 展开
Rows 远超业务需要条件过宽、filter 下推不足、权限/前端过滤后置收紧 WHERE、分页、CDS 分层
同一 SQL 在不同用户结果差异大DCL/authorization/filter 数据范围不同固定用户和权限条件，比较返回行数
SQL 不多但整体仍慢瓶颈可能在 ABAP CPU、RFC、外部 API、锁、UI转向 ABAP Profiler、日志、trace 其他层
优化后结果变快但数据不一致优化破坏语义先修正确性，再谈性能

注意：HANA 系统上“全表扫描”“索引”这类词不要机械套用传统数据库经验。是否需要新增或调整数据库对象，应由项目规范、性能专家和系统证据共同判断。ABAP 开发者首先要做到：减少不必要访问、让过滤尽早发生、避免 N+1、控制字段和行数、证明修改前后结果一致。

26.5 日本项目现场说法

常见日语包括：SQLトレース、パフォーマンス分析、性能調査、検索条件、実行回数、総実行時間、取得件数、ボトルネック、性能劣化、改善前後の比較、再現条件。

障碍调查或性能分析中可以这样说：

  • 推測ではなく、SQLトレースで実際に発行されたSQLと実行回数を確認します。
  • 同じ条件で改善前後のトレースを取得し、総実行時間と取得件数を比較します。
  • この性能問題はDBアクセス起因か、ABAP処理起因かを切り分ける必要があります。
  • 対象ユーザ、入力条件、データ件数、実行時間帯を固定して SQL トレースを取得します。
  • トレース結果から、実行回数、総実行時間、取得件数、呼び出し元を確認します。
  • SQL の改善後も、取得結果が業務的に同一であることを確認します。

review 时可以这样问：

  • ループ内 SELECT になっていませんか。
  • 検索条件と取得件数は業務要件に対して妥当ですか。
  • 性能改善の根拠となるトレース結果はありますか。
  • N+1 アクセスになっていないことをトレースで確認していますか。
  • 改善前後で同一条件の比較になっていますか。

26.6 English Terms

  • Official / search term: SQL Trace, performance trace, SQL performance analysis, ABAP SQL trace.
  • Related terms: execution count, duration, records, rows, database access, N+1 query, full table scan (traditional-DB term; see HANA caution), table buffer, SQL Monitor, ABAP Profiler.
  • Common SAP on-premise search term: ST05 SQL Trace; tool availability and recommended entry points depend on release/deployment.
  • Recommended search pattern: SAP SQL Trace ABAP, SAP ST05 SQL Trace, SAP ABAP SQL performance analysis.

26.7 Example / 学习例子

假设 AC0567C 的 status API 在查询 100 个 batch 时变慢。代码看起来只是循环调用 get_status( batch_id )，每次内部 SELECT 一次 header 和一次 item。手工测试 3 条数据时没问题，真实数据 1000 条时很慢。

SQL Trace 可能显示：

SQL A: SELECT header by batch_id
Executions: 1000
Total time: high

SQL B: SELECT items by batch_id
Executions: 1000
Total rows: very high

这说明问题可能不是单条 SQL 很慢，而是重复访问次数过多。优化方向可能是一次性批量读取 header 和 item，再在内表中关联，或调整 CDS/query 设计。优化后再 trace，如果执行次数从 2000 降到 2，总耗时明显下降，就有证据说明优化有效。

另一个例子：CDS view 使用 association 后，上层 path expression 导致展开了昂贵 join。Trace 能帮助确认最终 SQL 是否行数膨胀，是否需要调整 CDS 分层、过滤条件或消费方式。

再看一个缺陷票写法：

Symptom:
  Status API response time is 18 seconds for 1,000 upload batches.

Trace condition:
  System/client: QAS/100
  User: TEST_API_1000
  Endpoint: GET /.../UploadBatches?$filter=status eq 'OPEN'
  Data volume: 1,000 batch headers, 20,000 item rows
  Time window: 2026-06-16 14:00-14:05 JST

Trace finding:
  SELECT item by batch_id executed 1,000 times.
  Total time and rows are dominated by item lookup.

Hypothesis:
  Service logic performs per-batch item SELECT instead of bulk read.

Fix candidate:
  Bulk-read items for all batch ids, then aggregate in internal table or CDS.

Regression evidence:
  Same input after change: item SQL executions reduced from 1,000 to 1,
  response time reduced from 18s to 2.5s, returned status count unchanged.

这个例子强调两件事：性能结论必须和业务结果一起验证；优化后的“快”如果改变了返回状态、漏了权限过滤或丢了排序，就不是合格优化。

26.8 Counterexample / Failure mode

常见失败模式有十类。

第一，凭感觉优化。没有 trace 就改代码，可能花很多时间优化非瓶颈。

第二，trace 范围太大。开着 trace 跑一堆无关操作，结果无法分析。

第三，只看单次耗时，不看执行次数。N+1 问题常常单次很快、总次数很高。

第四，只看 SQL，不看业务输入。数据量、选择条件、用户权限、时间范围不同，性能结论可能完全不同。

第五，把 SQL Trace 当长期生产监控。Trace 适合短时间诊断，不是持续监控全部生产 SQL 的工具；长期观察应考虑 SQL Monitor 等。

第六，没有前后对比。优化是否有效，要用同条件证据比较。

第七，把 SQL Trace 当成唯一真相。Trace 显示数据库访问，但不显示完整业务语义；有时 SQL 多是因为权限过滤、业务拆分或框架行为，不一定都是 bug。

第八，优化后不跑功能回归。减少 SQL 次数可能改变排序、重复处理、权限过滤或锁时机，必须确认结果仍然正确。

第九，把开发机小数据结论搬到生产。DEV 里 20 条数据很快，不代表 QAS/PRD 里百万数据和真实权限条件也快。

第十，没有控制权限和 DCL 条件。同一个 API 在不同用户下可能触发不同数据范围；性能比较时如果用户权限不同，结论可能失真。

26.9 思考问题

  • SQL Trace 能证明什么？不能证明什么？
  • 如果一条 SQL 单次很快但执行一万次，问题可能在哪里？
  • 何时应使用 SQL Trace，何时更适合 ABAP Profiler 或 SQL Monitor？
  • AC0567C POC 中哪个列表、状态查询或批处理最可能出现 N+1 查询？
  • 性能缺陷单中至少应该记录哪些 trace 条件和结果？
  • 如果 trace 显示 SQL 很少但用户仍觉得慢，下一步应该看哪个工具或证据？
  • 优化后如何证明“结果正确性”和“性能改善”同时成立？
  • DCL、authorization 和输入数据量如何影响性能 trace 结论？
