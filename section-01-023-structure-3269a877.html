Chapter 01 - SAP 和 ABAP 基础

23. Structure 结构类型

  • Chapter 01
  • Section 23
  • self-contained study note

本节结论

Structure 结构类型 是 ABAP 里承载“一条有多个字段的业务记录”的基本形态：它不是数据库表，不是内表，也不是没有类型约束的 JSON/dict，而是一个有名字、有 structured type、由多个 named components 组成的 typed record。学会 structure，才看得懂后面的 Internal table 内部表基础、SQL INTO 结果、RAP payload、message/result 结构和项目里的字段设计。

正文

上一节讲了 TYPE vs LIKE 类型声明方式：变量可以按独立类型声明，也可以照已有对象形状声明。现在要把“一个值”提升为“一条记录”。在 ABAP 里，structure 是一个 structured data object，它有一个 structured type，里面由多个 component 组成。每个 component 自己也有名字和类型，可以像普通变量一样读取、赋值、传参、参与 SQL INTO 或业务判断。

最先掌握的是访问方式。ABAP 用 structure component selector - 访问结构组件，例如 ls_connection-airport_from_id。- 前后不能有空格。它也不是对象引用访问的 ->，也不是静态类成员访问的 =>。Java/Python 背景的人很容易写成 ls_connection->airport_from_id，但那会把普通结构误解成对象引用，语法层面就错了。

Structure 的来源有几类。最基础的是本地结构类型：

TYPES: BEGIN OF ty_connection,
         carrier_id      TYPE /dmo/carrier_id,
         connection_id   TYPE /dmo/connection_id,
         airport_from_id TYPE /dmo/airport_from_id,
         airport_to_id   TYPE /dmo/airport_to_id,
       END OF ty_connection.

DATA ls_connection TYPE ty_connection.

这里 ty_connection 是结构类型，ls_connection 是结构变量。类型定义描述“这条记录有哪些 component”；数据对象才保存运行时值。项目里还会看到 DDIC Structure repository object、database table / classic view 的行结构、CDS view entity 的行类型、global class/interface public type 等作为结构类型来源。新代码的判断顺序通常是：优先让 component 使用稳定业务类型，例如 Data element 数据元素 或经过设计的 public type；RAP/CDS 相关代码会逐步使用 CDS view entity 的 row type；如果只是普通程序内部组合字段，可用本地 TYPES BEGIN OF ... END OF；直接用数据库表名当结构类型虽然有效，但会让变量绑定到物理表定义，除非目标就是处理表行，否则新代码要谨慎。

这里要把 database table、CDS view entity 和 structure 分清。Database table 是数据库持久化对象；把它放在 TYPE 后面时，ABAP 使用的是该表的行结构。CDS view entity 是现代建模对象，也可以在 ABAP 程序中作为结构化类型来源；这不等于它能在 ABAP Dictionary 中被当作 DDIC structure 使用。并且在 ABAP Cloud 或 Clean Core 语境下，还要看它是否有合适的 release contract / API 状态，不能把所有 CDS 都当成随便可用的公共类型。Structure 本身只是一条记录的形状，不负责保存多行数据，也不等于数据库表。

结构组件应尽量带业务语义。carrier_id TYPE /dmo/carrier_id 比 carrier_id TYPE c LENGTH 3 更适合项目代码，因为 data element 不只带技术类型，还承载 label、语义和后续 UI/CDS/OData 复用可能。今天只是 Day 03，但要先养成习惯：如果一个结构以后可能被提升到 CDS projection、OData entity 或 RAP DTO，就不要把金额、数量、日期、key 都临时写成 string 或匿名 c 字段，否则后面重构成本很高。

Structure 可以整体处理，也可以按 component 处理。ls_connection = VALUE #( carrier_id = 'LH' connection_id = '0400' ). 是一种清晰的初始化方式；没有写到的 component 会保持类型特定 initial value，给已有结构重新赋 VALUE #( ... ) 时也要意识到旧值会被覆盖。直接 target = source 只有在结构类型兼容时才安全；如果结构不同但组件同名，可以考虑 CORRESPONDING #( source )，但这不是“万能安全复制”。CORRESPONDING #( ) 会按同名 component 复制，并在源/目标 component 类型不同时静默套用 ABAP 标准转换规则，可能发生截断、舍入或格式转换而没有明显警告。跨结构类型使用时，务必确认同名 component 的类型兼容；必要时用显式 component-by-component assignment，或者用 MAPPING / EXCEPT 把意图写清楚。

最后建立 flat / nested / deep 的意识。只包含固定长度 elementary components 的结构通常可以理解为 flat structure；结构里面再包含结构，就是 nested structure，但如果里面仍然只有固定长度 elementary components，嵌套本身不一定让它变成 deep。只要直接或间接包含 string、xstring、internal table、reference variable 等 deep component，就要按 deep structure 认识。Day 03 不需要展开深拷贝、序列化、比较和性能细节，只要知道：deep structure 后面会和内表、引用、接口 payload 一起成为重要话题。

为什么重要

  • 业务程序很少只处理一个标量值；订单行、航班连接、消息、校验结果、RAP failed/reported payload 都更接近 structure。
  • Structure 是 internal table 的前置概念。多数业务内表的 row type 是结构：structure 解释“一行长什么样”，internal table 解释“多行如何存、查、循环、排序”。
  • 结构组件的类型选择会影响 SQL INTO、UI label、接口字段、金额/数量/日期语义和后续 RAP/CDS 映射。
  • CORRESPONDING #( )、direct assignment、component access 都是项目高频代码；如果结构不兼容，bug 往往不是语法错误，而是字段放错、静默转换、值被 initial 覆盖。
  • 对 Java/Python 背景学习者来说，structure 可以类比 typed record / dataclass / DTO，但 ABAP 的每个 component 都有静态类型和 Dictionary 语义，不能当成随意增删 key 的 dict。

核心机制

  • TYPES BEGIN OF ... END OF 定义 structured type；DATA var TYPE struct_type 声明 structured data object。
  • DATA BEGIN OF ... END OF 可直接声明结构数据对象；入门先优先理解 type 与 object 的区别。
  • structure-component 用 - 访问组件；不要写空格，不要和 ->、=> 混淆。
  • VALUE #( component = value ... ) 适合初始化或重建结构；没有指定的 component 会按类型特定 initial value 处理。
  • CORRESPONDING #( ) 按同名 component 复制；跨结构类型时要警惕静默类型转换，必要时使用 MAPPING、EXCEPT 或显式赋值。
  • DDIC Structure、database table row type、CDS view entity row type、local structured type 都可能成为结构类型来源；它们的生命周期、可见性和 Clean Core 风险不同。
  • Flat / nested / deep 是后续内表、引用、序列化和接口处理的基础分类，本节先建立识别能力。

例子

最小结构例子：

TYPES: BEGIN OF ty_order_item,
         travel_id TYPE /dmo/travel_id,
         item_no     TYPE n LENGTH 6,
         amount      TYPE p LENGTH 11 DECIMALS 2,
         currency    TYPE c LENGTH 3,
       END OF ty_order_item.

DATA ls_item TYPE ty_order_item.

ls_item = VALUE #(
  travel_id = '0000004711'
  item_no     = '000010'
  amount      = '1980.00'
  currency    = 'JPY' ).

out->write( ls_item-travel_id ).
out->write( ls_item-amount ).

这里 ty_order_item 是结构类型，ls_item 是结构变量，travel_id、item_no、amount、currency 是 component。访问字段时使用 ls_item-amount，不能写 ls_item -> amount，也不能写 ls_item - amount。

再看 CORRESPONDING 的风险：

DATA ls_target TYPE ty_order_item.

ls_target = CORRESPONDING #( ls_item ).

如果 ls_item 和 ls_target 类型完全一致，这段代码可读性很好。如果两边结构不同，但有同名 component，ABAP 会尝试按同名 component 复制；如果同名 component 类型不同，可能发生隐式转换。金额、数量、日期、编号字段尤其要小心。项目代码里跨类型使用 CORRESPONDING #( ) 时，最好能解释为什么同名字段类型兼容，或者用显式赋值把关键字段写清楚。

练习动作：

  1. 定义一个 ty_connection，包含 departure airport、arrival airport、carrier、connection id。
  2. 用 VALUE #( ) 给结构变量赋值。
  3. 用 out->write( ) 分别输出整个结构和单个 component。
  4. 故意把 - 写成 -> 或在 - 两边加空格，观察语法错误。

常见误区

  • 把 structure 当数据库表：以为 TYPE /dmo/travel 就是在读表数据，忘了它只是表行结构，不包含任何数据库记录。
  • 把 structure 当 internal table：以为一条结构能装多行订单，结果后面无法循环、排序、按 key 查找。
  • 把 structure 当无类型 dict：临时用 string/c 拼字段，后面金额、日期、数量、key 的语义全部丢失。
  • 混淆 selector：把 ls_data-field 写成 ls_data->field，把普通结构误认为对象引用。
  • 过度相信 CORRESPONDING #( )：同名字段复制成功，但类型不同导致截断、舍入或格式变化。
  • 忽略 deep structure：结构里包含 string、内表或 reference 后，赋值、比较、序列化、接口输出会比 flat structure 更复杂。

思考题
