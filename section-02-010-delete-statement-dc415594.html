Chapter 02 - Core ABAP Programming

10. DELETE statement 删除数据

  • Chapter 02
  • Section 10
  • self-contained study note

10.1 一句话结论

ABAP SQL 的 DELETE 用来从可写数据库对象中删除行。它是写入类语句里风险最高的一类：少一个 WHERE 可能删除整张表，条件过宽可能删除别的业务范围，直接物理删除可能破坏业务历史、审计、关联数据和 Clean Core 边界。学习 DELETE 的重点不是语法，而是删除范围、实际删除件数、事务回滚能力、业务保留规则、授权和官方写入路径。

10.2 到底学什么

先把边界说清楚：本节讲的是 ABAP SQL DELETE，也就是删除数据库持久化对象里的行；它不是内部表的 DELETE。内部表删除只是改 ABAP 内存，ABAP SQL DELETE 会进入数据库修改路径，可能影响生产数据、事务、锁、审计和业务一致性。

最常见的安全写法，是用 WHERE 精确限定删除范围：

DELETE FROM zorder_note
 WHERE note_id = @lv_note_id.

DATA(lv_delete_subrc) = sy-subrc.
DATA(lv_deleted)      = sy-dbcnt.

如果 note_id 是唯一主键，业务预期通常是删除一行。sy-dbcnt 应该等于 1；如果是 0，说明目标不存在、已经被别人删除，或者条件不满足；如果大于 1，说明 WHERE 写得不够精确。

DELETE 也可以按结构或内表里的主键删除：

DELETE zorder_note FROM @ls_note_key.
DELETE zorder_note FROM TABLE @lt_note_keys.

这类写法的重点是“来源结构/内表必须正确承载目标表主键”。如果 key 不完整、字段对不齐、来源数据不是想删的对象，就可能删错。真实项目里，删除前通常还要先确认业务状态、关联数据、权限和是否应该做逻辑删除。

10.3 为什么重要

删除比新增和更新更敏感，因为它直接移除数据。很多 SAP 业务对象并不鼓励物理删除，而是使用状态、取消、冲销、归档、作废标志或删除标志。比如订单、会计凭证、主数据、接口日志、审批记录，往往不能随便 DELETE。能写 SQL 删除，不代表业务上允许删除。

它的第一层风险是范围失控。没有 WHERE 的 DELETE FROM ztable 会删除所有可见行；过宽的 WHERE 会删除别的客户、组织、期间或状态下的数据。项目里必须把删除范围写到 review 可以直接看懂。

第二层风险是关联完整性。SAP 业务数据之间有很多关系，未必都由数据库外键强制。删除 header 但保留 item，删除主数据但保留交易数据，删除日志但保留统计索引，都可能造成后续程序异常或业务解释困难。

第三层风险是审计和恢复。删除后的数据如果已经 commit，恢复可能依赖备份、变更文档、日志或人工重建。生产环境里，“删了再说”通常不是可接受策略。清理程序要有日志、模拟模式、件数确认和回滚/恢复预案。

第四层风险是 Clean Core。直接删除 SAP 标准表数据通常是高风险动作。S/4HANA、ABAP Cloud、RAP 或 Clean Core 项目里，要优先使用业务事务、released API、BAPI、RAP BO、归档对象、数据清理工具或正式扩展点。

10.4 Key mechanism / 核心机制

10.4.1 DELETE FROM ... WHERE ...：条件删除

条件删除是最常见、也最需要 review 的写法：

DELETE FROM zorder_note
 WHERE note_id        = @lv_note_id
   AND sales_order_id = @lv_sales_order_id.

这里的 WHERE 决定删除哪些行。官方文档说明，符合 sql_cond 的行会被删除；如果没有符合条件的行，不会删除任何行，sy-subrc 会设置为 4。

写完后要马上保存系统字段：

DATA(lv_delete_subrc) = sy-subrc.
DATA(lv_deleted)      = sy-dbcnt.

IF lv_deleted <> 1.
  " Expected exactly one row to be deleted.
ENDIF.

项目里真正判断成功的不是“语句执行了”，而是“删除件数符合业务预期”。

10.4.2 没有 WHERE 是全量删除

下面这句非常危险：

DELETE FROM zorder_note.

语义是删除目标对象中当前客户端可见的全部行，或在使用 UP TO 时删除指定数量的行。只有在明确设计的数据清理、测试数据初始化、临时表清空或批处理归档场景下，才可能允许这种写法。即便如此，也要有运行条件、日志、件数输出和回滚策略。

在书写习惯上，除非有明确的“清空表”设计，否则 DELETE FROM ... 后面应该马上出现可解释的 WHERE。

10.4.3 按主键删除：DELETE dbtab FROM @wa

按结构删除时，数据库会根据来源结构中的主键内容寻找目标行：

DATA(ls_note_key) = VALUE zorder_note(
  note_id = lv_note_id ).

DELETE zorder_note FROM @ls_note_key.

这种写法适合“我手里有明确主键”的场景。但初学者要注意：这不是按任意字段匹配，而是按目标对象的主键语义匹配。来源结构最好基于目标表类型或清楚的 key 结构，避免字段顺序、长度或初始值造成误解。

如果删除条件需要表达业务范围、状态、期间或组织约束，用 DELETE FROM ... WHERE ... 往往更可读：

DELETE FROM zorder_note
 WHERE note_id        = @lv_note_id
   AND sales_order_id = @lv_sales_order_id
   AND status         = @lc_status_draft.

这段代码明确说明：只删除某订单下仍处于 draft 的备注。

10.4.4 批量删除：DELETE ... FROM TABLE @itab

批量删除可以用内表提供要删除的 key：

DELETE zorder_note FROM TABLE @lt_note_keys.

DATA(lv_deleted) = sy-dbcnt.

官方文档说明，如果内表某一行在数据库里找不到相同主键，那一行会被忽略，sy-subrc 会设置为 4；如果内表为空，不会删除行，但 sy-subrc 仍可为 0；sy-dbcnt 是实际删除的行数。

所以批量删除后不能只写：

IF sy-subrc = 0.
  " All good?
ENDIF.

更稳的做法是对比预期件数：

DELETE zorder_note FROM TABLE @lt_note_keys.

DATA(lv_deleted) = sy-dbcnt.

IF lv_deleted <> lines( lt_note_keys ).
  " Some requested keys were not deleted or did not exist.
ENDIF.

当然，有些清理需求允许“要删的不存在也算成功”。但这必须是业务决策，而不是因为代码没检查。

10.4.5 大量删除：分批、排序和提交边界

大量删除会带来锁、回滚区、日志、数据库资源和运行时间问题。官方文档提到，UP TO n ROWS 可以用于把大删除拆成多次执行，避免一次删除过多行造成资源问题。

DO.
  DELETE FROM zapplication_log
   WHERE created_at < @lv_cutoff
   UP TO @lv_package_size ROWS.

  DATA(lv_deleted) = sy-dbcnt.

  IF lv_deleted = 0.
    EXIT.
  ENDIF.

  COMMIT WORK.
ENDDO.

这个例子只适合当前程序拥有事务边界的清理 report。可复用方法、RAP handler、BAPI 内部或框架回调里不能这样随手提交。

如果需求是“删除最旧的 N 条”，就不能只写 UP TO n ROWS。官方文档说明，如果 UP TO 没有 ORDER BY，无法定义具体删除哪几行；如果 ORDER BY 不是唯一排序，也无法定义并列行中哪几行会被删。需要稳定删除范围时，应使用业务排序字段和唯一 tie-breaker。

10.4.6 sy-subrc 和 sy-dbcnt

对 DELETE 来说，常见判断是：

  • sy-subrc = 0：至少一行被删除，或者批量来源行都被处理到；内表为空时也可能是 0。
  • sy-subrc = 4：没有符合条件的行，或批量删除时至少有一行没有找到对应数据库行。
  • sy-dbcnt：实际删除的行数。

这些系统字段应该在 DELETE 后立即复制出来：

DELETE FROM zorder_note
 WHERE note_id = @lv_note_id.

DATA(lv_delete_subrc) = sy-subrc.
DATA(lv_deleted)      = sy-dbcnt.

不要在后续又执行其他 SQL 后才回来看 sy-subrc。

10.4.7 物理删除、逻辑删除和归档

SAP 项目里并不是所有“不要显示”的数据都应该物理删除。常见替代方案包括：

  • 设置删除标志，例如 is_deleted = abap_true。
  • 改状态，例如 cancelled、rejected、obsolete。
  • 业务冲销或反向凭证。
  • 归档对象或数据生命周期管理。
  • 清理临时表、队列表、日志表中的过期数据。

物理删除适合临时数据、明确可清理日志、草稿数据、测试数据或项目自有表中已经满足保留策略的数据。对于核心业务对象，应该优先找业务 API 或归档机制。

10.4.8 删除不是权限、锁和业务校验

DELETE 不会自动检查当前用户是否业务上允许删除，也不会自动检查关联对象是否仍依赖它。经典 ABAP 可能需要 AUTHORITY-CHECK、lock object、业务状态检查、日志和变更记录；RAP 中要看 authorization、lock、validation、determination、delete action 和 save sequence。

如果直接删除 SAP 标准表数据，可能绕过变更文档、派生逻辑、缓存、状态机、归档策略和业务一致性检查。这类操作必须通过正式工具、released API、BAPI、RAP BO 或项目批准的修复方案。

10.7 Example / 学习例子

10.7.1 例 1：按主键删除草稿备注
DELETE FROM zorder_note
 WHERE note_id        = @lv_note_id
   AND sales_order_id = @lv_sales_order_id
   AND status         = @lc_status_draft.

DATA(lv_deleted) = sy-dbcnt.

IF lv_deleted <> 1.
  " Expected exactly one draft note to be deleted.
ENDIF.

这里不是只按 note_id 删除，而是附加了订单和草稿状态。这样能表达业务规则：只能删除仍未正式提交的草稿备注。

10.7.2 例 2：批量按 key 删除
DELETE zorder_note FROM TABLE @lt_note_keys.

DATA(lv_deleted) = sy-dbcnt.

IF lv_deleted <> lines( lt_note_keys ).
  " Some requested keys did not exist or were not deleted.
ENDIF.

如果业务允许重复删除请求，即“已经不存在也算成功”，可以不报错，但应在接口说明或日志中说明这种幂等语义。

10.7.3 例 3：分批清理过期日志
DO.
  DELETE FROM zapplication_log
   WHERE created_at < @lv_cutoff
   UP TO @lv_package_size ROWS.

  DATA(lv_deleted) = sy-dbcnt.

  IF lv_deleted = 0.
    EXIT.
  ENDIF.

  COMMIT WORK.
ENDDO.

这个例子用于清理项目自有日志表，且假设当前 report 拥有事务边界。真实项目还应加模拟模式、输出删除件数、异常处理和运行权限检查。

10.7.4 例 4：逻辑删除优先
UPDATE zorder_note
   SET is_deleted = @abap_true,
       changed_by = @sy-uname,
       changed_at = @lv_now
 WHERE note_id    = @lv_note_id.

如果业务要求保留历史、审计或可恢复，逻辑删除可能比物理 DELETE 更合适。读者要学会问“是否真的要物理删除”。

10.8 Counterexample / Failure mode

失败模式一：漏掉 WHERE，导致全表删除。除非这是明确设计的初始化或清理程序，否则应视为高危缺陷。

失败模式二：WHERE 太宽。只按状态、日期或用户删除，可能把别的组织、客户、期间、系统批次的数据一起删掉。

失败模式三：只看 sy-subrc，不看 sy-dbcnt。删除一条、删除 N 条、删除 0 条的业务意义完全不同。

失败模式四：批量 DELETE FROM TABLE 后把 sy-subrc = 4 当作全部失败，或把它忽略成成功。它可能表示部分 key 没找到，实际已有部分行被删除。

失败模式五：把业务“取消”做成物理删除。很多业务对象应该取消、冲销、失效或设置删除标志，而不是 SQL 删除。

失败模式六：直接删除 SAP 标准表。这样可能绕过标准业务逻辑、变更文档、归档和一致性检查。

失败模式七：大量删除不分批、不提交、不记录件数。容易造成长事务、锁等待、回滚区压力或难以恢复。

失败模式八：测试后不清理或清理过度。删除类测试既可能污染共享环境，也可能误删他人测试数据。

10.9 思考问题

  • 这个需求是真正物理删除，还是逻辑删除、取消、无效化、归档？
  • DELETE 的目标对象是否允许被当前程序直接删除？
  • WHERE 是否能保证只删除预期业务范围？
  • 预期删除件数是多少？代码是否检查 sy-dbcnt？
  • 如果目标行不存在，是错误、忽略，还是幂等成功？
  • 是否存在 header/item、主数据/交易数据、日志/统计数据等关联关系？
  • 是否需要先检查业务状态、权限、锁和保留期限？
  • 大量删除是否需要分批、模拟模式、日志、commit 计划和恢复方案？
  • 谁负责 COMMIT WORK 或 rollback？
  • ABAP Cloud / RAP / Clean Core 项目中，是否应该改用 released API、BAPI、RAP BO、EML、归档对象或业务事务？
