35. ABAP Unit 单元测试基础

  • Chapter 02
  • Section 35
  • self-contained study note

35.1 一句话结论

ABAP Unit 是 ABAP 语言内建的单元测试框架，用来把“这个 method 应该怎样工作”写成可自动执行的测试。它不是画面测试，也不是完整集成测试；它最适合保护小而明确的业务规则、计算逻辑、字段映射、异常路径和 helper method。

初学者要先掌握一个判断：ABAP Unit 的核心不是“多写一些测试代码”，而是把可重复验证的业务约定放到系统里。手工点一次画面只能证明“刚才这个操作看起来通过”；ABAP Unit 可以在每次改代码、激活对象、执行质量检查或准备传输前，重新验证关键逻辑是否仍然成立。

本节要学会读懂并写出最小测试类：用 CLASS ... DEFINITION FOR TESTING 声明 test class，用 DURATION 和 RISK LEVEL 描述运行时间与风险，用 METHODS ... FOR TESTING 声明 test method，用 CL_ABAP_UNIT_ASSERT 断言实际结果和期望结果，并在 ADT 的 ABAP Unit 结果里读懂失败原因。

测试类虽然不是生产业务逻辑的一部分，但它仍然属于 ABAP repository object 的交付内容，会跟随被测对象一起进入开发、传输、质量检查和维护流程。换句话说，ABAP Unit 不是临时脚本；它是项目质量资产。

35.2 到底学什么

本节不追求覆盖 ABAP Unit 的所有高级能力，而是建立“能看懂、能写、能 review”的基础。需要掌握十个点。

  • ABAP Unit 是 unit test / module test，主要测试可隔离的代码逻辑。
  • Test class 通常是 local class，常放在被测对象的 test include 或 ADT 的 Test Classes 区域。
  • FOR TESTING 把 class 或 method 标记为测试相关对象。
  • DURATION SHORT|MEDIUM|LONG 表示测试预期运行时间。
  • RISK LEVEL HARMLESS|DANGEROUS|CRITICAL 表示测试可能影响系统状态的风险级别。
  • 如果省略 RISK LEVEL，默认是 CRITICAL；普通单元测试应显式写 RISK LEVEL HARMLESS。
  • Test method 通常没有参数，放在 test class 的 PRIVATE SECTION。
  • 测试代码常用 Arrange / Act / Assert：准备数据、执行被测代码、断言结果。
  • CL_ABAP_UNIT_ASSERT 提供常见断言，例如 assert_equals、assert_true、assert_bound、assert_initial、fail。
  • setup、teardown、class_setup、class_teardown 用来准备和清理测试上下文。
  • 好的 ABAP Unit 测试不依赖执行顺序，不直接修改生产数据，不把外部系统、真实权限、系统日期、真实数据库状态当作隐式前提。

一个最小骨架如下：

CLASS ltcl_quantity_rule DEFINITION FINAL FOR TESTING
  DURATION SHORT
  RISK LEVEL HARMLESS.

  PRIVATE SECTION.
    METHODS accepts_positive_quantity FOR TESTING.
ENDCLASS.

CLASS ltcl_quantity_rule IMPLEMENTATION.
  METHOD accepts_positive_quantity.
    DATA(lo_rule) = NEW lcl_quantity_rule( ).

    DATA(lv_result) = lo_rule->is_valid( 10 ).

    cl_abap_unit_assert=>assert_equals(
      act = lv_result
      exp = abap_true
      msg = 'Positive quantity should be valid' ).
  ENDMETHOD.
ENDCLASS.

这段代码表达了一个很小但完整的测试：准备一个 quantity rule，调用它，断言正数数量应被视为有效。以后有人修改 is_valid，如果破坏了这个约定，ABAP Unit 会用失败结果提醒团队。

35.3 为什么重要

SAP 项目里，很多缺陷不是“代码不能运行”，而是“某个规则在改动后悄悄变了”。例如金额计算少考虑了舍入，状态判断漏掉了取消状态，错误消息从具体业务错误变成了通用错误，字段映射把同名但不同语义的字段复制错了。这些问题靠人工测试很难长期稳定覆盖。

ABAP Unit 的价值在于把回归保护前移到开发阶段。它能让团队在 DEV 系统中快速验证小逻辑，减少“等到结合测试才发现”的成本。对于需要反复修改的业务规则，ABAP Unit 是非常直接的安全网。

对 Java/Python 背景学习者来说，可以把 ABAP Unit 类比成 JUnit 或 pytest，但 SAP 项目有自己的语境：测试代码是 ABAP repository 世界的一部分，被测对象需要激活；test class 常作为 local class 与被测代码一起维护；测试结果可能参与 ATC / quality gate / transport 前检查；测试设计还要考虑 SAP 数据库、权限、锁、事务和外部接口的隔离。

更重要的是，ABAP Unit 会反过来推动代码设计变好。如果一个 method 只有清楚输入、一个明确返回值、没有隐藏数据库写入、没有弹窗消息、没有直接 COMMIT WORK，它通常很容易测试。反过来，如果一个 method 同时查数据库、检查权限、改状态、发消息、调 BAPI、写日志、提交事务，就很难用 ABAP Unit 稳定验证。这就是它和 Small methods 小方法原则、Method signature 方法签名、Exception class 异常类 之间的关系。

35.4 Key mechanism / 核心机制

35.4.1 Test class is marked with FOR TESTING

ABAP Unit test class 用 FOR TESTING 标记。常见写法如下：

CLASS ltcl_price_calculator DEFINITION FINAL FOR TESTING
  DURATION SHORT
  RISK LEVEL HARMLESS.

ltcl_... 是常见 local test class 命名习惯，不是强制语法。FINAL 表示不需要继承。FOR TESTING 告诉 ABAP runtime 这是测试类，不是生产代码直接调用的业务类。

DURATION 用来说明测试预期运行时间。初学者的单元测试通常应尽量是 SHORT：不跑长时间批处理，不等待真实外部系统，不读大量生产数据。MEDIUM 和 LONG 应该有明确理由。

RISK LEVEL 用来说明测试是否可能改变系统状态或配置。常见理解如下：

Risk level含义
HARMLESS不改变持久化业务数据或系统设置，适合普通单元测试
DANGEROUS可能改变持久化数据，需要更谨慎执行
CRITICAL可能改变系统设置或 customizing，通常受执行限制

项目上常要求普通单元测试显式写 RISK LEVEL HARMLESS。不要让读者猜“这个测试会不会改系统状态”。

更精确地说，省略 RISK LEVEL 不是“自动变安全”，而是默认视为 CRITICAL，也就是最受限制的风险级别。在通过客户端级设置限制高风险测试的系统里，CRITICAL 测试可能不会被执行。普通业务逻辑单元测试应主动声明 RISK LEVEL HARMLESS，让开发者、reviewer 和质量门禁都能看出它不会修改持久化业务数据或系统设置。

35.4.2 Test method is marked with FOR TESTING

Test method 也用 FOR TESTING 声明：

METHODS calculates_net_amount FOR TESTING.

测试方法通常没有输入参数。它由 ABAP Unit framework 调用，而不是由生产代码调用；即使 test method 放在 PRIVATE SECTION，framework 也能调用它，这正是推荐把 test method 设为 private 的原因。一个 test class 至少应有一个 test method。多个 test method 的执行顺序不应该成为测试前提；如果测试依赖顺序，通常说明测试共享了不该共享的状态。

35.4.3 Arrange / Act / Assert

测试代码最容易读的结构是 Arrange / Act / Assert：

METHOD calculates_net_amount.
  " Arrange
  DATA(lo_calculator) = NEW lcl_price_calculator( ).

  " Act
  DATA(lv_net) = lo_calculator->calculate_net(
    iv_gross    = '110.00'
    iv_tax_rate = '0.10' ).

  " Assert
  cl_abap_unit_assert=>assert_equals(
    act = lv_net
    exp = '100.00'
    msg = 'Net amount should exclude tax' ).
ENDMETHOD.

不要把测试写成一大段难读的脚本。失败时，维护者应该能马上看出：输入是什么、调用了什么、期望是什么、实际哪里不对。

35.4.4 CL_ABAP_UNIT_ASSERT reports test results

CL_ABAP_UNIT_ASSERT 是 ABAP Unit 常用断言类。最常见的是比较实际值和期望值：

cl_abap_unit_assert=>assert_equals(
  act = lv_actual
  exp = lv_expected
  msg = 'Unexpected calculation result' ).

对象引用可以用类似 assert_bound / assert_not_bound 的断言；布尔结果可以用 assert_true / assert_false；初始值可以用 assert_initial / assert_not_initial。具体可用方法应以当前系统的类定义和 SAP 文档为准。

fail( ) 用于明确说明“执行到这里就是失败”。它常用于测试异常路径：

METHOD rejects_zero_quantity.
  DATA(lo_rule) = NEW lcl_quantity_rule( ).

  TRY.
      lo_rule->check( 0 ).
      cl_abap_unit_assert=>fail( 'Expected zcx_quantity_invalid' ).
    CATCH zcx_quantity_invalid.
      " expected path
  ENDTRY.
ENDMETHOD.

这里的逻辑是：如果没有抛出预期异常，就失败；如果抛出了具体异常，则测试通过。不要用空的 CATCH cx_root 把所有异常都吞掉。异常路径测试应该 catch 具体 exception class。

如果测试前提条件不满足，例如测试环境没有必要的配置或测试数据，ABAP Unit 中应使用 cl_abap_unit_assert=>abort( ) 表达“当前测试无法继续”，而不是把它写成普通 assertion failure。不要把 JUnit / pytest 里的 skip 或 assume_* 直接当作 ABAP Unit 方法名写进教程；具体可用方法要以当前系统的 CL_ABAP_UNIT_ASSERT 类定义为准。

35.4.5 Fixture methods prepare test context

Fixture methods 用来准备或清理测试环境：

  • setup：每个 test method 之前执行。
  • teardown：每个 test method 之后执行。
  • class_setup：一个 test class 执行前执行一次。
  • class_teardown：一个 test class 执行后执行一次。

常见例子：

CLASS ltcl_quantity_rule DEFINITION FINAL FOR TESTING
  DURATION SHORT
  RISK LEVEL HARMLESS.

  PRIVATE SECTION.
    DATA mo_rule TYPE REF TO lcl_quantity_rule.
    METHODS setup.
    METHODS accepts_positive_quantity FOR TESTING.
ENDCLASS.

CLASS ltcl_quantity_rule IMPLEMENTATION.
  METHOD setup.
    mo_rule = NEW #( ).
  ENDMETHOD.

  METHOD accepts_positive_quantity.
    cl_abap_unit_assert=>assert_equals(
      act = mo_rule->is_valid( 10 )
      exp = abap_true ).
  ENDMETHOD.
ENDCLASS.

setup 的好处是让每个测试都有新的上下文，减少测试之间互相污染。class_setup 适合准备昂贵且只读的共享资源，但不要把会被测试修改的对象放进去共享。

生命周期也要精确理解：setup / teardown 围绕每一个 test method 执行；class_setup / class_teardown 围绕一个 test class 的一次执行只执行一次。如果 setup 失败或用 abort( ) 中断，对应 test method 不应再进入业务断言。class_setup 适合准备昂贵、只读、不会被各个测试修改的资源；会变化的对象应在 setup 中为每个测试重新准备。

35.4.6 Unit test should isolate business logic

ABAP Unit 最适合测试确定性逻辑：计算、状态判断、字段映射、异常路径、formatter、converter、small helper、RAP handler 中抽出的纯业务判断。它不适合直接依赖真实生产数据、真实 RFC、真实 OData endpoint、真实权限配置或真实系统时间。

如果业务逻辑需要依赖外部资源，应考虑隔离边界：

  • 数据读取放到 repository / gateway / adapter 后面。
  • 当前日期时间通过 clock interface 提供。
  • 外部接口通过 interface 包装。
  • OO 依赖用 OO Test Double Framework 或手写 test double 替换；ABAP Test Double Framework 的核心入口是 CL_ABAP_TESTDOUBLE。
  • Open SQL 相关场景在后续学习 ABAP SQL Test Double Framework；SAP 也提供 ABAP SQL replacement service，例如 CL_OSQL_REPLACE，用于在测试执行时把生产代码里的 ABAP SQL 访问重定向到受控替代对象。
  • 服务级逻辑在 Week 4 继续连接 ABAP Unit for service logic。

本节只要求建立原则：单元测试越小、越快、越确定，越适合频繁运行；真实系统行为则交给集成测试、API evidence replay、ATC、SQL Trace、授权测试和业务验收补充。

ABAP Cloud / RAP 项目还要注意两点。第一，测试代码也要遵守 released API / C1 release 约束，不能为了测试去引用 unreleased SAP standard class。第二，RAP behavior handler / saver 是 framework-managed 对象，不应把 RAP 框架 wiring 当作普通 local object 直接实例化测试；更好的做法是把业务判断抽到可测试的 helper / service class，用 ABAP Unit 测 helper，把 RAP reported / failed、save sequence 和 OData 行为放到更高层级的测试或 evidence replay 中验证。

35.4.7 Run and read results in ADT

在 ADT 中，通常可以对 class、package 或测试类运行 ABAP Unit。结果会显示哪些 test method 通过、失败、跳过或因为异常中止。读失败结果时要区分：

  • assertion failure：实际值和期望值不一致。
  • unexpected exception：测试没有处理到某个异常路径。
  • skipped / assumption not met：前提条件不满足。
  • aborted / prerequisite not met：前提条件不满足，测试被 abort( ) 或 framework 前提中断。
  • runtime error / dump：测试或被测代码发生了更底层的运行时错误。

失败消息要对维护者有帮助。msg = 'error' 或 msg = 'failed' 几乎没有信息量；msg = 'Net amount should exclude tax' 才能帮助下一个人理解业务期望。

ADT 里的单对象 ABAP Unit 执行适合开发阶段快速反馈；package / mass execution 和 ATC 更适合质量门禁、transport 前检查或 CI/CD 流程。ABAP Unit 还可以和 code coverage 视图结合，帮助团队看出哪些生产代码路径被测试覆盖、哪些还没有被触达。Coverage 不是质量本身，但它能帮助发现明显没被保护的分支。

35.4.8 minimal ABAP Unit design checklist

第一次给业务逻辑补 ABAP Unit 时，可以按下面的顺序设计。它比“先写一个 test class”更可靠，因为它先逼你说清楚要保护的业务约定。

StepQuestionExample
1. Name the behavior这个测试证明哪条业务规则？calculates_net_amount_without_tax
2. Arrange minimal input需要哪些最小输入和依赖？amount、tax rate、test double notifier
3. Act once被测 public method 是哪一个？lo_service->calculate( ... )
4. Assert one main outcome最关键的期望是什么？net amount equals expected value
5. Add helpful message失败时维护者要知道什么？Net amount should exclude tax
6. Cover one failure path预期异常或非法输入如何验证？TRY ... CATCH zcx_... or assert_bound
7. Check isolation测试是否依赖真实数据、顺序、日期或外部系统？如依赖外部系统，应拆到集成/API 测试。

一个 test method 尽量只证明一件事。多个业务规则可以写多个 test method，而不是把所有断言塞进一个很长的测试里。这样失败结果会直接告诉维护者是哪条规则破了；在 transport 前或 ATC 前运行时，也更容易判断是业务逻辑变化、测试数据变化，还是环境依赖导致失败。

35.5 日本项目现场说法

常见日语：

  • ABAP Unit
  • 単体テスト
  • テストクラス
  • テストメソッド
  • アサーション
  • 期待値
  • 実績値
  • テストフィクスチャ
  • 正常系
  • 異常系
  • 境界値
  • 回帰テスト
  • 品質ゲート

现场说明句：

  • この業務ロジックは ABAP Unit で正常系・異常系を確認します。
  • このメソッドは外部依存が強いため、単体テストしやすい helper に切り出します。
  • 期待値と実績値が一致しないため、assert_equals の失敗として表示されています。
  • 通常の単体テストでは DURATION SHORT、RISK LEVEL HARMLESS を明示します。
  • このテストは永続データを変更しないため、HARMLESS として定義できます。
  • RISK LEVEL を省略すると CRITICAL 扱いになるため、通常の単体テストでは HARMLESS を明示してください。
  • 前提条件不足でテストを継続できない場合は、FAIL ではなく ABORT として扱います。
  • 前提条件不足の場合は、業務ロジックの失敗ではなくテスト前提の不足として扱います。

使用场景：设计说明、代码 review、缺陷修复后的回归测试、transport 前确认、ATC / quality gate、RAP helper logic 验证。

35.7 Example / 学习例子

35.7.1 例 1：测试纯业务计算
CLASS lcl_discount DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS calculate
      IMPORTING iv_amount TYPE decfloat34
      RETURNING VALUE(rv_discount) TYPE decfloat34.
ENDCLASS.

CLASS lcl_discount IMPLEMENTATION.
  METHOD calculate.
    IF iv_amount >= 1000.
      rv_discount = iv_amount * '0.10'.
    ENDIF.
  ENDMETHOD.
ENDCLASS.

CLASS ltcl_discount DEFINITION FINAL FOR TESTING
  DURATION SHORT
  RISK LEVEL HARMLESS.
  PRIVATE SECTION.
    METHODS gives_ten_percent_for_large_amount FOR TESTING.
ENDCLASS.

CLASS ltcl_discount IMPLEMENTATION.
  METHOD gives_ten_percent_for_large_amount.
    DATA(lo_discount) = NEW lcl_discount( ).

    cl_abap_unit_assert=>assert_equals(
      act = lo_discount->calculate( 1000 )
      exp = CONV decfloat34( '100' )
      msg = 'Amount 1000 should receive 10 percent discount' ).
  ENDMETHOD.
ENDCLASS.

这个测试不访问数据库，不依赖当前用户，不修改系统状态，因此适合 SHORT / HARMLESS。

35.7.2 例 2：测试异常路径
METHOD rejects_negative_amount.
  DATA(lo_service) = NEW lcl_invoice_service( ).

  TRY.
      lo_service->check_amount( -1 ).
      cl_abap_unit_assert=>fail( 'Expected zcx_amount_invalid' ).
    CATCH zcx_amount_invalid.
      " expected exception
  ENDTRY.
ENDMETHOD.

异常路径测试不是“手工读一眼就好”。一旦 bug 修复后应该抛出具体异常，就要把这个行为固定成测试。

35.7.3 例 3：用 setup 准备对象
CLASS ltcl_invoice_service DEFINITION FINAL FOR TESTING
  DURATION SHORT
  RISK LEVEL HARMLESS.

  PRIVATE SECTION.
    DATA mo_service TYPE REF TO lcl_invoice_service.
    METHODS setup.
    METHODS accepts_valid_invoice FOR TESTING.
ENDCLASS.

CLASS ltcl_invoice_service IMPLEMENTATION.
  METHOD setup.
    mo_service = NEW #( ).
  ENDMETHOD.

  METHOD accepts_valid_invoice.
    cl_abap_unit_assert=>assert_true(
      act = mo_service->is_invoice_valid( iv_invoice_id = '4711' ) ).
  ENDMETHOD.
ENDCLASS.

setup 让每个测试方法从干净对象开始。不要让第一个测试修改对象后，第二个测试依赖这个修改结果。

35.8 Counterexample / Failure mode

35.8.1 误区 1：把 ABAP Unit 当成 UI 测试

如果测试需要人工打开 Fiori、输入数据、点保存按钮，它不是 ABAP Unit 的典型场景。ABAP Unit 应优先测试可直接调用的 ABAP 逻辑。

修正动作：把核心业务判断抽到 helper / service method，再用 ABAP Unit 验证；UI 行为用合适的 UI/API/集成测试补充。

35.8.2 误区 2：测试依赖真实生产数据

测试里直接 SELECT 某个真实订单，然后假设它永远存在，是很脆弱的做法。

修正动作：用受控测试数据、test double 或专门测试环境。测试输入要由测试自己准备，不能依赖“系统里刚好有这条数据”。

35.8.3 误区 3：测试之间依赖执行顺序

第一个测试创建状态，第二个测试依赖这个状态，第三个测试再清理。这会让测试变得不稳定。

修正动作：让每个 test method 独立；必要时用 setup 准备新的对象。

35.8.4 误区 4：把所有异常都吞掉
CATCH cx_root.
  " ignore

这种写法会让真正的错误也被当成测试通过。

修正动作：catch 具体 exception class；未预期异常应该让测试失败。

35.8.5 误区 5：失败消息没有信息

msg = 'error' 对排查没有帮助。

修正动作：写清楚业务预期，例如 Net amount should exclude tax。

35.8.6 误区 6：为了测试暴露 public method

把 private helper 改成 public 只为了测试，会污染生产接口。

修正动作：优先测试 public behavior；如果 private helper 复杂到必须单独测试，考虑是否应抽成独立 class 或 service。

35.8.7 误区 7：把 ABAP Unit 当万能测试

ABAP Unit 不替代 ATC、SQL Trace、集成测试、授权测试、RAP/OData contract test 或业务验收测试。

修正动作：用 ABAP Unit 保护可隔离的业务逻辑；其它质量活动放到对应工具和阶段。

35.9 思考问题

  • 这个 method 的核心业务规则是什么？能否用一个 assert_equals 表达？
  • 这个测试是 HARMLESS 吗？它是否写入了数据库、customizing 或外部系统？
  • 这个测试为什么应该是 SHORT？如果不短，是否还是 unit test？
  • 每个 test method 是否互相独立？是否依赖执行顺序？
  • setup、teardown、class_setup、class_teardown 分别应该放什么？
  • 失败时 ABAP Unit View 的 message 能否让下一个开发者快速定位？
  • 异常路径是否写了测试？是否 catch 了具体 exception class？
  • 这个测试有没有直接依赖系统日期、当前用户、授权、真实 DB 或外部接口？
  • 是否应该把依赖抽成 interface，再用 test double 替换？
  • 这个 ABAP Unit 测试和 ATC、review、transport release 前检查如何配合？
