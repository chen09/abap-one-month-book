31. Idempotent determination 幂等 determination

  • Chapter 03
  • Section 31
  • self-contained study note

31.1 一句话结论

Idempotent determination 幂等 determination 指的是：同一个 RAP determination 即使因为相同输入或相同业务状态被触发多次，也应得到同样的业务对象结果，不应该产生重复累计、重复插入、重复消息或不可解释的副作用。

好的 determination 像“根据当前状态重新计算正确结果”；坏的 determination 像“在旧结果上再加一次”。在 RAP 中，determination 可能在 create、update、draft、save 或字段变化时多次触发，所以幂等不是可选优化，而是稳定交付的基本要求。

31.2 到底学什么

第一，要学 determination 的触发不一定只发生一次。on modify determination 可能在同一个 RAP LUW 中因为 create、update 或字段变化被多次触发；on save determination 可能在保存序列中触发。开发者不能把“触发一次”当成设计前提。

第二，要学幂等计算方式。安全的写法通常是从当前 BO 状态重新计算目标字段，例如 total = sum(current_items)；危险写法是基于旧值累计，例如 total = total + changed_amount。

第三，要学补默认值和覆盖用户输入的边界。默认值 determination 应只在字段为空或符合条件时补值；如果用户或 consumer 已经明确传入字段，determination 不应无条件覆盖。

第四，要学 determination 和 validation 的边界。Determination 负责派生、补值、计算；validation 负责阻止不合法保存。不要用 determination 偷偷修正所有错误，也不要让 determination 隐式吞掉业务问题。

第五，要学 determination 和 API idempotency 的区别。Determination 幂等是 BO 内部派生逻辑稳定；API idempotency 是 request id / idempotency key 防止外部重复请求产生重复业务效果。两者相关，但不是同一个层次。

31.3 为什么重要

RAP determination 在 demo 中经常看起来很简单：填一个默认值，算一个总额，派生一个状态。但项目中同一对象可能被 Fiori draft、多次字段修改、OData consumer、EML 测试和后台批处理反复触发。如果 determination 不幂等，生产数据会出现“越保存越大”“重复生成消息”“用户输入被覆盖”“状态反复跳回默认值”等问题。

对 AC0567C 这种 batch upload API 来说，幂等 determination 尤其重要。批量接口可能被重试，同一 batch 可能被 validate 多次，同一行可能被修正后再次提交。派生字段、error count、warning count、total amount、status summary 如果用累计写法，就会在重复执行后失真。

对 Java/Python 背景读者来说，可以把幂等 determination 类比为纯函数式 derived calculation：total = sum(items) 比 total += delta 更安全。SAP/RAP 的特殊点是 transactional buffer、local EML、draft/save lifecycle 会让“同一逻辑被多次调用”变得很常见。

31.4 Key mechanism / 核心机制

幂等 determination 可以用五条规则检查。

第一，目标字段从当前事实重算。金额合计、行数、状态摘要、warning/error count，应尽量从当前 item、当前状态、当前 buffer 中重新计算，而不是在旧值上追加。

第二，只在空值或明确条件下补默认值。例如 status 为空时设为 New；如果 status 已经是 Submitted 或 Executed，就不要无条件改回 New。

第三，避免外部副作用。Determination 不适合发送 HTTP 请求、发邮件、写外部系统或制造不可回滚的副作用。副作用应放到明确的 save/additional save/outbox/business event 设计中。

第四，使用当前 BO 状态。RAP determination 应尽量读取 transactional buffer 或当前 BO 语义，而不是直接查旧数据库状态后忽略本次未保存修改。必要时使用 local EML 观察当前上下文。

第五，写重复触发测试。测试不只验证 happy path，还要验证同一字段修改两次、同一请求保存两次、draft activate、多 item 修改、重复 validate 后结果仍稳定。

还要补一层官方机制边界。SAP Learning 对 determination 的描述强调：它属于 RAP business object behavior，定义在 behavior definition 中，实现通常在 local handler class 中；触发条件可以来自 create、update、delete 或字段变化，触发时机可以是 on modify 或 on save。Determination 可以通过 local EML 修改 entity instance，并通过 reported 返回消息，但没有 failed 参数，因此不适合承担“阻止保存”的职责；阻止不合法保存应交给 validation。官方资料还提醒：如果多个 determination 被同一条件触发，执行顺序不是固定保证，所以幂等 determination 不能依赖“另一个 determination 一定先跑完”。SAP Learning 课程中还给出 unmanaged non-draft 场景的 restriction；项目上要以当前目标系统 release、BDEF activation 和 ABAP Keyword Documentation 为准确认可用性。

Idempotency evidence checklist

幂等 determination 不能只靠代码读起来“好像安全”，要有证据。最小证据可以从四类测试开始。

第一类是重复触发测试。准备同一个 BO 实例，让同一字段变化触发 determination 两次，确认目标字段第二次不会继续变化。例如 header total 第一次算出 1000，第二次仍然是 1000，而不是 2000。

第二类是用户输入保护测试。创建或更新时让 consumer 明确传入一个非空字段，确认默认值 determination 不会无条件覆盖它。这个测试可以防止“创建时默认状态”在后续 action 或 API 更新中把业务状态打回初始值。

第三类是多入口一致性测试。用 Fiori/service preview、OData/API 或 EML 走不同入口，确认派生结果一致。若 determination 实际只依赖前端字段事件或 UI 默认值，就不是 BO 层稳定逻辑。

第四类是 AC0567C / 批处理重放测试。对同一 batch 或同一行执行 validate / prepare / execute 两次，确认 error count、warning count、total amount、派生说明和 reported message 不重复累计。批量接口里，幂等 evidence 比单条 UI 保存更重要，因为真实调用方最常发生 timeout retry 和重复提交。

31.5 日本项目现场说法

常见日语：

  • 冪等性
  • べき等性
  • 再実行
  • 再実行しても結果が変わらない
  • 累積加算
  • 再計算
  • 実行順序に依存しない
  • 重複実行

review 时可以这样说：

この determination は再実行されても同じ結果になるように、
現在の明細からヘッダ金額を再計算します。

指出风险时可以这样说：

旧値への累積加算ではなく、現在状態からの再計算にしてください。

测试要求可以这样说：

この派生処理が重複実行された場合でも冪等であることをテストで確認します。

31.7 Example / 学习例子

例子一：header total。

安全写法：

Read all current items.
total = sum(item.amount)
header.TotalAmount = total

危险写法：

header.TotalAmount = header.TotalAmount + changed_item.amount

如果同一 item 修改两次、draft activate 重放一次、或 determination 被多个字段变化触发，危险写法会把金额越加越大。

例子二：默认状态。

安全写法：

If OverallStatus is initial:
  OverallStatus = 'New'

危险写法：

OverallStatus = 'New'

如果用户或 action 已经把状态改成 Submitted，危险写法会把状态覆盖回默认值。

例子三：AC0567C batch summary。

安全写法：

For each current upload line in transactional buffer:
  derived_company_name = lookup_name(line.company_code)
  derived_amount_group = classify(line.amount)
  line.company_name = derived_company_name
  line.amount_group = derived_amount_group

batch.error_count = count(lines where severity = E)
batch.warning_count = count(lines where severity = W)
batch.total_amount = sum(lines.amount)

危险写法：

batch.error_count = batch.error_count + new_errors
batch.total_amount = batch.total_amount + current_line.amount
append line message "auto completed"

第二种写法会在重复 validate 或重复保存时制造重复计数和重复消息。

31.8 Counterexample / Failure mode

第一，金额重复累计。用户修改 item 两次，header total 被加两次，最终金额大于真实 item 合计。

第二，默认值覆盖用户输入。consumer 明确传入状态或字段，determination 每次触发都覆盖成默认值。

第三，依赖执行顺序。determineDiscount 必须先于 determineTotal，但项目没有保证顺序；结果取决于触发时机。

第四，重复 reported message。每次 determination 都追加同一条“已自动补全”消息，用户看到多条重复提示。

第五，在 determination 中调用外部 API。重复触发导致外部系统重复通知或重复写入，RAP rollback 也无法撤销外部副作用。

第六，直接读取数据库旧状态。忽略 transactional buffer 中尚未保存的 item 修改，导致派生值和用户当前编辑内容不一致。

第七，把 API idempotency 放进 determination。request-level idempotency 应放在 action/service/save 语义更清楚的位置；determination 只应负责 BO 内部派生稳定。

31.9 思考问题

  • 这段 determination 重复执行两次，目标字段会不会变化？
  • 它是从当前状态重算，还是在旧值上累计？
  • 它会不会覆盖用户明确输入的字段？
  • 它是否依赖另一个 determination 必须先执行？
  • 它是否通过 local EML 或 BO 语义读取当前 buffer，而不是直接读取旧数据库状态？
  • 它是否制造外部副作用？
  • 如何用一句日语说明这段 determination 具有 冪等性？
