9. PFCG role basics for developers

  • Chapter 04
  • Section 09
  • self-contained study note

9.1 一句话结论

PFCG role basics for developers 是 ABAP 开发者理解 SAP 授权落地方式的基础：开发者通常不负责在生产系统维护角色，但必须读懂 PFCG role、authorization object、authorization field、field value、organizational value / organizational level、generated profile、role assignment 和 SU53 / trace 证据，才能把代码里的授权检查和项目里的角色配置对齐。

一句话记忆：开发者不一定维护 PFCG，但必须能说明“我的代码检查什么对象、什么字段、什么值；测试用户有没有这些值；失败证据在哪里”。

9.2 到底学什么

本节学习的是开发者视角下的 PFCG，不是安全管理员完整操作手册。PFCG 是 classic SAP role maintenance 的核心入口之一，安全/Basis/授权团队会用它维护角色菜单、授权数据、组织级字段、生成 profile，并把角色分配给用户。ABAP 开发者要理解这些结构，因为 AUTHORITY-CHECK、RAP authorization、CDS DCL、OData service 访问和测试用户行为，最终都会和角色中的授权值发生关系。

第一，要知道 role 不是一个简单标签。一个用户被分配 role 后，系统会通过 role 中的 authorization data / generated profile 判断他是否拥有某些 authorization object 的字段值。例如一个上传 API 可能需要 ACTVT = 02 表示修改，ACTVT = 16 或项目定义的 execute 值表示执行，并需要 company code 或 plant 等组织字段。role 名字听起来像 UPLOAD_ADMIN 不代表权限一定正确，必须看对象和字段值。

第二，要知道开发者的责任边界。开发者可以提出授权对象、字段和值的需求，可以提供代码检查点、测试用户矩阵、SU53 或 trace 证据；但生产角色维护、广域授权开放、职责分离、审批流程通常属于授权团队和项目治理范围。开发者不应为了“让测试通过”直接放大生产角色。

第三，要知道 PFCG 不是全部授权世界。Fiori business role、catalog、space/page、IAM app、communication arrangement、CDS DCL、RAP global/instance authorization、service binding 暴露、background job user 都可能参与最终访问结果。PFCG 很重要，但不是唯一入口；不同系统形态和项目架构会有不同角色治理方式。

第四，要知道“role 已经分配”不是排查终点。一个用户是否真正通过检查，至少取决于目标系统/client、role 是否分配、authorization data 是否维护、profile 是否生成、用户比较/用户缓冲是否生效、测试用户是否重新登录、以及代码实际检查的 object/field/value 是否与 role 一致。项目现场经常听到“ロールは付いています”，但开发者还要继续追问：哪个 role、哪个 object、哪个 field、哪个 value、在哪个系统、哪个用户、什么时间生效？

第五，要知道开发者交付给授权团队的不是“请加权限”四个字，而是一份可执行的授权需求。它应包含业务动作、技术入口、检查点、authorization object、fields、values、正向/反向测试用户、复现步骤、失败证据和最小权限边界。这样授权团队才能维护 role，测试团队才能准备用户，reviewer 才能判断是否过授权。

9.3 为什么重要

很多授权缺陷不是因为代码太复杂，而是开发者和授权团队说的不是同一种证据。开发者说“我检查了 Z_UPLOAD”，授权团队问“需要哪些 field value”；测试人员说“用户已经有 role”，开发者问“role 里有没有对应动作值和公司代码 1000”；客户说“某用户不能执行”，项目组要判断是 role 没分配、profile 没生成、字段值不对、DCL 过滤、RAP instance authorization，还是服务暴露/通信用户问题。

对 Java/Python 背景读者，可以把 PFCG role 暂时类比成企业权限系统里的 role/policy 配置，但 SAP 的粒度更细。一个 role 中可能包含多个 authorization objects；每个 object 又有多个 fields；字段值可能是 ACTVT、company code、plant、sales org、customer 自定义字段等。代码中的 AUTHORITY-CHECK OBJECT ... ID ... FIELD ... 必须和 role 里的对象/字段/值对齐。

对日本 SAP 项目来说，PFCG 相关问题经常出现在単体テスト、結合テスト、権限テスト、障害票和本番リリース前確認中。没有 PFCG 基础，开发者很容易把所有错误都推给“権限チーム”，或者反过来让授权团队盲目加大权限，最后造成审计风险。

PFCG 也是开发者理解“最小权限原则”的入口。最小权限不是一句安全口号，而是测试矩阵里的具体值：能 display 的用户不一定能 change；能 change 公司代码 1000 的用户不应该自动能 change 2000；能从 Fiori 应用进入页面的用户不一定能通过 API 执行批量上传；technical user 不应该因为是接口账号就拥有全组织范围。一本项目实战书必须让读者习惯这种粒度，否则授权设计只会停留在“有 role / 没 role”的粗糙层面。

9.4 Key mechanism / 核心机制

开发者可以用下面的链路理解 PFCG：

Business requirement
  -> operation and data scope
  -> code/DCL/RAP authorization check point
  -> authorization object
  -> authorization fields
  -> expected field values
  -> PFCG role authorization data
  -> generated profile
  -> user role assignment
  -> test evidence

第一层是 business requirement。先说明用户要做什么：display、create、change、delete、approve、execute、cancel、upload、status query。不要一开始就说“加一个 role”，而要先说业务动作和数据范围。

第二层是 check point。classic ABAP 可能是 AUTHORITY-CHECK；CDS 读取可能是 DCL；RAP action 可能涉及 global authorization / instance authorization / precheck；OData service 还可能受到 service exposure、communication user 或 business catalog 影响。PFCG role 要和这些检查点对齐。

第三层是 authorization object / field / value。开发者要能列出最小需求，例如：

Object: Z_UPLOAD
Fields:
  ACTVT = 16
  BUKRS = 1000
Operation:
  Execute upload batch
Test users:
  USER_OK_1000 should pass
  USER_NG_2000 should fail

第四层是 evidence。授权失败时，SU53 可以提供当前用户最近一次失败的授权检查线索；authorization trace 能提供更完整的检查过程。开发者要知道如何读这些证据，但不要把 SU53 当成万能答案：如果问题来自 DCL、RAP instance rule、service binding、communication arrangement 或 UI catalog，证据位置可能不同。

第五层是 role generation and assignment。PFCG 中修改 role 后，通常还要生成 authorization profile，并把 role 分配给用户。测试时要确认目标系统、目标 client、测试用户、role assignment、profile generation、user buffer/重新登录等因素。否则“role 已经改了”不一定等于“当前测试用户已经生效”。

在真实项目里，还要能读懂 PFCG authorization data 的维护状态。某些授权值来自事务、服务或 SU24 authorization proposal；某些值由权限团队手动维护；系统升级、菜单变更或 proposal 调整后，也可能出现旧值或未维护状态。开发者不需要替代权限团队操作这些状态，但要知道它们解释了为什么“同一个 role 名字”在不同系统或不同时间可能代表不同授权内容。和授权团队沟通时，最有用的问题不是“有没有 role”，而是“这个 role 的 authorization data 是否已维护、profile 是否已生成、user comparison / user buffer 是否已生效、目标用户是否重新登录、SU53/trace 看到的 object-field-value 是否与设计一致”。

SU24 也要放在正确位置理解。它通常用于维护或影响事务/服务相关的授权建议值，帮助 PFCG 生成或维护 role authorization data；它不是运行时替代 AUTHORITY-CHECK、DCL 或 RAP authorization 的开关。开发者如果发现某个事务、服务或自定义入口总是需要人工补同一组授权值，可以和权限团队讨论 proposal/default 维护，但最终仍要用正向用户、反向用户、SU53 / trace 和 API/业务结果证明授权边界有效。

开发者可以把 PFCG 相关排查整理成两张表。

第一张是“需求到 role 的映射表”：

项目说明
Business operation用户要做什么，例如 display、execute、approve、cancel
Entry pointSAP GUI、Fiori、OData/API、background job、external system
Runtime check pointAUTHORITY-CHECK、CDS DCL、RAP authorization、service access
Authorization object标准或自定义 object，例如 Z_UPLOAD
FieldsACTVT、company code、plant、sales org、custom responsibility field
Values03 display、02 change、16 execute 或项目定义值、组织值、允许范围
Positive user应该通过的测试用户
Negative user应该失败的测试用户
EvidenceSU53、trace、API response、application log、test result

第二张是“缺陷分派表”：

现象首先判断可能责任边界
用户无权进入 Fiori 应用business role / catalog / app assignmentFiori/security/team setup
列表为空DCL、role 组织值、query conditionCDS/security/test data
action forbiddenRAP authorization、classic check、role valuedevelopment/security
API 403authentication、service authorization、technical user roleintegration/security
role 已分配但仍失败profile generation、user buffer、client/system、field valuesecurity/Basis/test
只有某公司代码失败organization level / field mappingsecurity/development/business

还要注意 ACTVT 的语义。很多项目会把 03 当 display，把 01 当 create，把 02 当 change，把 06 当 delete，把 16 或客户定义值当 execute，但实际使用要以具体 authorization object 和项目定义为准。开发者不能只背数字，要在设计书里写出业务动作和技术字段值的对应关系。

对 custom authorization object，开发者还要和权限团队确认字段设计是否稳定。字段太粗会导致过授权，字段太细会导致 role 维护困难；字段名和业务字段映射不清，会让 SU53 / trace 证据难以解释。比如上传批次只按 company code 授权，还是还要按 plant、upload type、approval group 授权，这不是语法问题，而是业务权限模型问题。

说明句：

開発側では、チェック対象の権限オブジェクト、項目、想定値、テストユーザ、SU53 証跡を提示します。

含义：开发侧提供检查对象、字段、预期值、测试用户和失败证据。

本番ロールの変更は権限チームの管理対象のため、開発側では必要権限と再現手順を明確にします。

含义：生产角色变更由授权团队管理，开发侧负责说明需求和复现。

ロールが割当済みでも、対象の組織値や ACTVT が不足していないか確認します。

含义：有 role 不代表字段值完整，还要看组织值和 activity。

開発側の確認範囲は、権限チェックポイント、必要な権限オブジェクト、項目値、再現手順、証跡の提示までとし、本番ロール変更は権限チームの手順に従います。

含义：开发侧说明检查点、对象、字段值、复现和证据；生产角色变更按授权团队流程。

一時的な広範囲権限でテストした結果は、実運用ロールでの権限テスト結果としては扱えません。

含义：临时全权限测试结果不能当成真实运用角色的授权测试证据。

9.7 Example / 学习例子

假设 AC0567C 的 execute action 只允许公司代码 1000 的上传负责人执行。开发者不需要直接维护生产 role，但需要给授权团队和测试团队一份最小授权说明：

Business operation:
  Execute upload batch

Technical check:
  RAP global/instance authorization or classic authorization check

Authorization object:
  Z_UPLOAD

Fields:
  ACTVT = 16
  BUKRS = 1000

Positive test user:
  TEST_UPLOAD_1000

Negative test user:
  TEST_UPLOAD_2000 or user without Z_UPLOAD

Evidence:
  API request/response
  SU53 or authorization trace
  application log / backend status

这样写的好处是：开发者没有越权改角色，但把“代码检查什么”和“角色应该给什么”讲清楚了。授权团队可以据此维护 role，测试团队可以据此准备正反用户，reviewer 可以据此判断最小授权是否合理。

如果把同一个例子写进缺陷票，可以更具体：

Symptom:
  TEST_UPLOAD_1000 calls Execute Upload Batch and receives 403 / no authorization.

Expected:
  User with Z_UPLOAD ACTVT=16 or the project-defined execute value and BUKRS=1000 can execute batch 4711.

Evidence:
  Endpoint: POST /.../UploadBatches('4711')/Execute
  User: TEST_UPLOAD_1000
  System/client: QAS/100
  Time: 2026-06-16 10:30 JST
  SU53/trace: missing Z_UPLOAD BUKRS=1000 ACTVT=16 or the project-defined execute value
  Application log: no save executed after authorization failure

Request to authorization team:
  Please confirm assigned role contains Z_UPLOAD with ACTVT=16 or the project-defined execute value and BUKRS=1000,
  and that profile/user assignment is effective in QAS/100.

这个模板的重点是“可验证”。授权团队能查 role，开发者能查代码检查点，测试人员能复现，业务顾问能确认公司代码范围。它避免了“权限不够，请处理”这种无法分派的缺陷。

再看一个反向测试例子：TEST_UPLOAD_2000 只有 BUKRS=2000，却尝试执行 BUKRS=1000 的 batch。预期结果应该是失败，而且失败后不能进入保存阶段，不能产生部分更新，不能让 API 返回成功。这类负向测试比正向测试更能证明授权边界有效。

9.8 Counterexample / Failure mode

反例一：只说“缺权限”，不说明 object、field、value、用户和复现步骤。授权团队只能猜，最后要么加错权限，要么来回沟通。

反例二：测试用户有 role，但 role 中字段值不对。例如有 ACTVT = 03 显示权限，没有 ACTVT = 02 修改权限或 ACTVT = 16 / 项目定义 execute 值；或只允许 company code 2000，却拿 company code 1000 测试。

反例三：开发者直接扩大生产角色。短期看测试通过，长期会破坏职责分离、审计和最小权限原则。

反例四：只测 UI，不测 API。UI 按钮可能隐藏或灰掉，但外部 API 直接调用仍要被后端授权拒绝。

反例五：把所有访问问题都归因于 PFCG。实际问题可能来自 CDS DCL、RAP instance authorization、service binding、communication arrangement、catalog、user buffer 或目标 client 不一致。

反例六：只给“超级测试用户”跑通流程。全权限用户能通过，不代表真实业务角色能通过；全权限用户也无法证明无权限路径会被拒绝。

反例七：role 变更后不记录证据。没有 role 名、profile 生成时间、用户分配、SU53/trace、测试结果，后续很难证明修复范围，也很难排查回归。

反例八：接口 technical user 权限过宽。为了赶进度给接口账号全公司代码、全 activity，短期容易跑通，长期会造成审计和数据外泄风险。

反例九：把组织级字段当成纯技术字段。company code、plant、sales org 这类值往往对应客户组织和职责分工，不能由开发者单方面决定。

反例十：自定义 authorization object 没有命名和字段说明。后续维护者看到 Z_OBJ1、FIELD1 很难判断业务含义，trace 证据也难以沟通。

9.9 思考问题

  • 当前代码或服务检查了哪个 authorization object？
  • 需要哪些 authorization fields 和 field values？
  • ACTVT 的值是否和业务动作一致：display、change、execute、delete？
  • 是否需要组织级字段，例如 company code、plant、sales org？
  • 正向测试用户和反向测试用户分别是谁？
  • 授权失败时，SU53、trace、API error payload 和 application log 能否互相解释？
  • AC0567C POC 中，technical user、business user、test user 是否需要不同角色？
  • 如果只用全权限用户测试，本节哪些结论无法被证明？
  • role 分配后，如何确认 profile、user assignment、用户会话和目标 client 都已经正确？
  • 自定义 authorization object 的字段是否能被业务顾问、授权团队和开发者共同理解？
