33. Save sequence 保存序列

  • Chapter 03
  • Section 33
  • self-contained study note

33.1 一句话结论

Save sequence 保存序列 是 RAP runtime 在 COMMIT ENTITIES 或 OData 保存请求之后，把 transactional buffer 中的 BO 变更检查、编号、保存、清理的一串受控流程；它回答的是“哪些逻辑还只是缓冲区变化，哪些逻辑已经进入真正保存，什么时候不能再安全回头”。

一句话记忆：interaction phase 是“先把意图放进 RAP buffer”，save sequence 才是“检查通过后真正提交这次 BO 变更”。

33.2 概念说明

RAP 把一次业务对象修改大致分成两段。第一段是 interaction phase：UI、OData 或 EML Entity Manipulation Language 发送 create/update/delete/action 请求，RAP runtime 和 handler class 把这些意图放进 transactional buffer，执行 authorization、feature control、action、on-modify determination 等逻辑。第二段才是 save sequence：当 consumer 触发保存，RAP runtime 按固定生命周期做保存前处理、检查、late numbering、最终保存和 cleanup。

初学者最容易误解的一点是：action handler、determination、validation、EML modify 成功，不等于数据库已经落库。很多时候只是 RAP transactional buffer 中有了变化。真正持久化通常发生在 COMMIT ENTITIES、OData save flow 或 draft Activate 触发之后。

可以把流程简化成这条线：

Modify / Action / EML
  -> RAP transactional buffer
  -> COMMIT ENTITIES / OData Save / Draft Activate
  -> save sequence
  -> save succeeds or fails
  -> cleanup

在 managed BO 中，标准保存通常由 RAP managed runtime 执行；在 unmanaged BO 或 managed with unmanaged save / additional save 场景中，项目会在 behavior pool 的 handler/saver 方法中承担部分保存职责。无论哪种方式，核心原则都一样：RAP runtime 是事务边界的 owner，项目代码不能随意 COMMIT WORK 或 ROLLBACK WORK。

33.3 为什么重要

不理解 save sequence，就会把 RAP 写成“按钮里直接插表”的老式程序。短期看能跑，长期会出现事务边界混乱：handler 已经写库但后续 validation 失败；late numbering 消耗了号码但业务数据没保存；additional save 写了日志但主保存回滚；cleanup 没跑导致下一次请求读到旧 buffer。

对项目交付来说，save sequence 是解释“保存失败到底发生在哪一步”的主线。缺陷单里常见描述是：“画面显示保存成功但数据库没有数据”“action 执行成功但保存时报错”“编号被消耗但单据不存在”“第一次保存失败后第二次保存带出旧数据”。这些都需要沿着 save sequence 分析，而不是只看某个 handler 方法。

对 Java/Python 背景学习者，可以把 save sequence 类比成 framework-managed unit of work commit。但 RAP 的差异是：它和 BDEF、behavior pool、handler/saver class、authorization、validation、determination、draft、late numbering、OData/EML response、SAP LUW 绑定在一起。开发者不是随时自己 commit，而是在 RAP 允许的生命周期点实现逻辑。

在日本 SAP 项目中，保存シーケンス 常常是 code review 和障害調査的关键词。レビュー会问：“この処理は interaction phase ですか、save sequence ですか”“point-of-no-return 後に業務エラーを出していませんか”“cleanup は成功/失敗の両方で実行されますか”。

33.4 边界

第一条边界：save sequence 不等于 Handler class 处理器类 的普通 action/update 处理。Handler 可以处理 action、authorization、feature control、on-modify determination、validation/on-save handler 等；最终保存和 cleanup 边界要看 runtime 或 Saver class 保存器类。

第二条边界：save sequence 不等于 COMMIT WORK。在 RAP 中，外部 consumer 用 COMMIT ENTITIES 或 OData 保存语义触发 BO 保存；RAP runtime 决定如何进入 save sequence。项目代码里随意写 COMMIT WORK / ROLLBACK WORK 会破坏框架事务语义。

第三条边界：save sequence 不是“所有错误都拖到最后”。能在 precheck、authorization、validation、check_before_save() 发现的问题，应该尽量提前；越接近 point-of-no-return，越不适合发现可预见业务错误。

第四条边界：save sequence 不等于 additional save。Additional Save 附加保存 只是 managed save sequence 中用于追加保存逻辑的扩展点；它不是替代主保存，也不是让 handler 提前提交副作用。

第五条边界：save sequence 不等于 late numbering。Late numbering 是保存序列中的一个敏感步骤，通常在 point-of-no-return 附近分配最终 key；save sequence 还包括保存前检查、主保存、附加保存、cleanup 等。

第六条边界：save sequence 不等于外部集成万能入口。如果保存时要发邮件、调用 HTTP、通知外部系统，必须考虑事务一致性、幂等、outbox、business event 和失败恢复。直接在保存阶段同步调用不可回滚外部系统，常常会制造 SAP 回滚但外部已执行的问题。

33.5 Key mechanism / 核心机制

第一层机制是触发。Save sequence 通常由 COMMIT ENTITIES、OData 保存请求、draft Activate 等触发。触发之前，BO 修改多数还在 RAP transactional buffer 或 application buffer 中；触发之后，runtime 进入保存流程。

第二层机制是保存前处理。根据实现方式，RAP 可能执行 on-save determination、validation、finalize()、check_before_save() 等步骤。注意：FOR VALIDATE ON SAVE 和 FOR DETERMINE ON SAVE 仍是 handler method，但触发点在保存序列的 check-before-save 相关阶段。

第三层机制是 point-of-no-return。这个点之前，如果检查失败，runtime 还能比较安全地拒绝保存并清理 buffer；这个点之后，保存过程已经进入更难回头的阶段。越靠后越不适合发现业务可预见错误。

第四层机制是 late numbering。若 BO 使用 Numbering 编号策略 中的 late numbering，最终 key 可能在 point-of-no-return 后通过 saver class 的 adjust_numbers() 分配。因为号码消耗通常不能像普通内存值一样简单撤回，所以应在更早阶段完成业务校验。

第五层机制是最终保存。Managed save 场景中，RAP runtime 根据 BDEF 的 persistent table、mapping、numbering 和 buffer 内容执行标准保存；fully unmanaged 场景中，项目在 saver class save() 中保存；managed with unmanaged save 场景中，项目在 save_modified() 中接管主保存；additional save 场景中，项目用 save_modified() 追加附属保存。

第六层机制是 response 和异常。保存失败不能只看 ABAP exception。RAP 有 FAILED、REPORTED、MAPPED、message、draft 状态、OData response 等多个反馈通道。尤其 create/numbering 场景要看 MAPPED 是否把临时 %cid 映射到最终 key。

第七层机制是 cleanup。保存成功或失败后都要清理 buffer。Unmanaged BO 如果使用自有 application buffer，必须覆盖正常 cleanup() 和失败路径 cleanup_finalize()。否则下一次请求可能带着上一次的内存状态，形成非常难排查的串数据。

这里要保持一个谨慎边界：本节的 save sequence 是读代码和排查问题用的生命周期地图，不是承诺每个 release、每种 managed/unmanaged 组合、每个 draft 场景都逐字按同一顺序调用。真正评审时，要回到当前系统的 BDEF saving option、saver class method、SAP Help / ABAP Keyword Documentation，以及实际 behavior pool 代码确认。初学者先掌握“越早发现业务错误越好、越晚越接近 point-of-no-return、cleanup 必须覆盖成功和失败路径”这三个判断标准。

第八层机制是测试。Save sequence 测试不能只测“保存成功”。至少要覆盖：

  • validation 失败时不会写库。
  • save sequence 中旧 API 报错时消息可理解。
  • late numbering 不在可预见业务错误前消耗号码。
  • additional save 重试时不会重复写日志或事件。
  • cleanup 后第二次请求不受第一次 buffer 影响。
  • create 成功后 MAPPED / OData response 能返回最终 key。

33.6 日本项目现场说法

常见日语：

  • 保存シーケンス：save sequence。
  • 保存処理：保存处理。
  • 保存前チェック：保存前检查。
  • ポイント・オブ・ノーリターン：point-of-no-return。
  • 後期採番：late numbering。
  • 追加保存：additional save。
  • クリーンアップ：cleanup。
  • バッファクリア：buffer 清理。
  • 二重登録：重复登记 / 重复写入。

说明句：

この処理は handler ではなく、RAP の save sequence 内で実行されます。

意思是：这个处理不是在 handler 中执行，而是在 RAP save sequence 内执行。

業務エラーは point-of-no-return 前に検出できるよう、validation または check_before_save に寄せます。

意思是：业务错误应尽量在 point-of-no-return 之前检测，所以放到 validation 或 check_before_save。

缺陷分析时可以这样问：

保存失敗は validation、check_before_save、adjust_numbers、save、additional save、cleanup のどこで発生していますか。

意思是：保存失败发生在 validation、check_before_save、adjust_numbers、save、additional save、cleanup 的哪一步？

33.8 Example / 学习例子

例子一：managed save 的普通保存。

用户在 Fiori 页面修改 Travel 日期并点击 Save。Interaction phase 中，update 请求进入 RAP buffer；保存时触发 validation，确认日期合法；managed runtime 根据 BDEF 的 persistent table 和 mapping 把变更写入数据库；成功后清理 buffer。开发者通常不需要自己写 save()。

例子二：validation 失败。

用户把 BeginDate 改成晚于 EndDate。Update 请求可能已经进入 buffer，但 save sequence 中 FOR VALIDATE ON SAVE 发现错误，把实例放进 failed，消息放进 reported。数据库不应写入这条非法变化。用户看到字段级错误后修正再保存。

例子三：late numbering。

创建申请单时，草稿阶段没有正式单号。保存激活时，validation 和 check_before_save() 先通过，之后 adjust_numbers() 分配正式号，最终保存成功后 response 返回最终 key / business number。若业务错误能提前发现，就不要等到号码分配后才报错。

例子四：additional save 写审计日志。

Travel 主数据由 managed runtime 保存；项目需要在同一事务里写一条审计日志。这个逻辑适合放在 additional save，而不是 action handler 里直接 INSERT zlog。同时要考虑幂等：重复保存或重试不能产生不可接受的重复日志。

例子五：AC0567C POC 类似批量上传。

批量上传 API 如果未来迁移到 RAP BO，上传行解析和 validation 只是前半段；真正写入业务对象、返回最终 key、写日志、处理失败和 cleanup，仍要按保存序列设计。否则会出现“部分行写了、部分行失败、日志重复、重试不可解释”的问题。

33.9 Counterexample / Failure mode

反例一：在 action handler 中直接 INSERT 主业务表。后续 validation 失败，但数据库已经写入。修正方式是让 action 修改 BO buffer，由 save sequence 统一保存。

反例二：在 saver 或 handler 中调用会内部 COMMIT WORK 的 BAPI。RAP runtime 失去事务控制，可能出现部分提交。修正方式是寻找 no-commit API、改造包装层，或重新选择实现方式。

反例三：所有业务检查都放到 save()。用户保存时才得到笼统错误，字段无法定位，号码可能已消耗。修正方式是把可预见错误提前到 validation、authorization、precheck 或 check_before_save()。

反例四：late numbering 前没有完成重复检查。到 adjust_numbers() 后才发现业务数据不合法，号码处理变复杂。修正方式是早检查、晚编号。

反例五：additional save 写外部系统。SAP 保存失败但外部系统已经收到通知。修正方式是使用 business event、outbox、application log 或异步集成，并设计幂等。

反例六：cleanup 缺失。第一次保存失败后，application buffer 没清；第二次请求混入旧数据。修正方式是明确 cleanup() / cleanup_finalize() 责任，并写重复请求测试。

33.10 思考问题

  • 这个逻辑应该发生在 interaction phase，还是 save sequence？
  • 它是保存前检查、late numbering、主保存、additional save，还是 cleanup？
  • 这个错误能不能在 point-of-no-return 前发现？
  • 如果保存失败，哪些数据已经写入，哪些只是 buffer？
  • 是否存在内部 COMMIT WORK / ROLLBACK WORK 或不可回滚外部调用？
  • 重复保存、重试、并发保存时，save sequence 是否仍然幂等？
  • 在日本项目缺陷分析中，如何说明“保存失败发生在哪个保存序列步骤”？
