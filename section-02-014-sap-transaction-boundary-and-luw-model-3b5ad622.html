Chapter 02 - Core ABAP Programming

14. SAP Transaction Boundary and LUW Model

  • Chapter 02
  • Section 14
  • self-contained study note

14.1 一句话结论

SAP Transaction Boundary and LUW Model 是把 SAP LUW、Database LUW、COMMIT WORK、ROLLBACK WORK、BAPI commit pattern、update task、enqueue lock 和 RAP save sequence 放到同一张图里理解的事务边界模型。它回答的不是“commit 语句怎么写”，而是“谁拥有这次业务动作的提交权、失败时能回滚到哪里、哪些副作用已经无法撤回”。

14.2 到底学什么

初学 ABAP SQL 写入时，很容易把事务理解成下面这种普通 SQL 脚本模型：

INSERT / UPDATE / DELETE
COMMIT

这个模型在 SAP 项目里太薄。真实项目至少有四层边界：

用户/接口/批处理的一次业务动作
  -> SAP LUW：SAP 应用层的一致性单元
    -> update task / RAP transactional buffer / BAPI 调用约定
      -> Database LUW：数据库层 commit / rollback 的原子边界

本节要学的不是再背一次 SAP LUW 和 Database LUW，而是把它们组合成判断规则：

  • 当前代码是完整业务动作的 owner，还是被别人调用的 helper？
  • 当前修改是直接 ABAP SQL 写库、注册 update task、调用 BAPI，还是通过 RAP EML 修改业务对象？
  • COMMIT WORK / ROLLBACK WORK 会不会过早结束外层事务？
  • 如果出现异常，是可以整体回滚，还是已经发生了部分提交、外部调用、日志、邮件、RFC、job 等副作用？
  • 在日本项目 review 中，是否能清楚说明 コミット責任、トランザクション境界、再実行時の整合性？

一个实用记忆法是：谁发起完整业务动作，谁通常拥有 transaction boundary；越底层、越可复用的代码，越不应该擅自 commit。

14.2.1 三种常见路径

第一种是 classic ABAP report / transaction code / batch program。程序本身通常拥有事务边界，可以在完成一组一致性修改后执行 COMMIT WORK 或在失败时执行 ROLLBACK WORK。但即使在这种路径里，也要避免“每处理一行就 commit”的写法，除非需求明确允许部分成功并且有清楚的重跑、补偿和日志设计。

第二种是 BAPI / RFC / 外部接口调用。现代 BAPI transaction model 的核心思想是 caller 拥有显式 transaction control：调用方可以把多个 BAPI 组合在同一个 LUW 里，最后再决定 commit 或 rollback。因此 BAPI 本身一般不应该随便 COMMIT WORK，调用方通常通过 BAPI_TRANSACTION_COMMIT 或 BAPI_TRANSACTION_ROLLBACK 结束事务。少数历史 BAPI 如果内部 commit，必须查 BAPI 文档确认。

第三种是 RAP / OData / Fiori 路径。RAP 使用 transactional buffer 和 save sequence。业务对象的 modify 阶段先把修改放进 RAP 事务上下文，真正持久化发生在 save sequence。外部 EML consumer 可以使用 COMMIT ENTITIES，而 OData 等原生支持的 RAP 场景通常由 RAP runtime 自动处理 commit / rollback。RAP behavior implementation 内部不能把 classic COMMIT WORK 习惯直接搬进去，否则会破坏框架的事务契约。

14.3 为什么重要

事务边界错误很少表现成“语法报错”。它更常见的样子是：

  • 接口返回成功，但数据库没有最终落库。
  • 第一行 commit 成功，第二行失败，业务上却要求两行一起成功或一起失败。
  • helper method 内部 COMMIT WORK，导致外层调用方想 rollback 时已经来不及。
  • BAPI 调用后忘记 BAPI_TRANSACTION_COMMIT，导致“测试时似乎没报错，但数据没出现”。
  • RAP handler 里写 classic commit，触发框架限制、运行时错误或保存语义混乱。
  • database lock 在 database commit 后释放，但业务对象在多个 dialog step 中仍然需要 SAP enqueue lock。
  • 重试接口时没有 idempotency 设计，第一次部分提交后第二次重复写入。

对 Java/Python 背景学习者来说，可以先把它类比为 service transaction / unit of work / ORM transaction boundary，但要马上补上 SAP 的差异：SAP 有 dialog work process、implicit database commit、update task、SAP enqueue、BAPI transaction model、RAP save sequence 和 Clean Core 约束。仅仅会写 UPDATE 和 COMMIT，还不足以在 SAP 项目里交付可靠写入逻辑。

14.4 Key mechanism / 核心机制

14.4.1 先问 transaction owner

设计写入逻辑时，第一问不是“这里要不要 commit”，而是“谁是 transaction owner”。

完整 report / batch job / transaction code
  -> 可能拥有 commit / rollback 责任

可复用 method / validation / BAdI / user exit / enhancement
  -> 通常不应该擅自 commit

BAPI caller / RFC caller / interface orchestration
  -> caller 通常决定何时 BAPI_TRANSACTION_COMMIT / ROLLBACK

RAP OData / Fiori request
  -> RAP runtime / save sequence 通常管理事务

外部 EML consumer
  -> 使用 COMMIT ENTITIES / ROLLBACK ENTITIES，并处理 FAILED / REPORTED / sy-subrc

把 commit 放错位置，最危险的不是“写入失败”，而是“写入太早成功”。太早成功的修改可能已经越过 rollback 能力边界，后面的错误只能靠补偿逻辑、反向凭证、冲销、状态修正或人工处理。

14.4.2 SAP LUW 与 Database LUW 的组合

Database LUW 是数据库系统层面的 transaction。database commit 持久化当前 database LUW 的修改并开始新的 database LUW；database rollback 撤回当前 database LUW 中未提交的修改。COMMIT WORK 和 ROLLBACK WORK 不只是数据库动作，它们还结束当前 SAP LUW，并触发 SAP LUW 相关处理。

SAP LUW 是 SAP 应用层组织一致性的单位。官方 SAP LUW cheat sheet 强调，SAP LUW 的 all-or-nothing 目标通常要求把变化推迟到事务末尾，用一个最终 database commit 写入一致结果。为了做到这一点，classic ABAP 可能使用 update task、temporary table、transactional buffer 或其他 bundling technique。

所以，正确心智模型是：

SAP LUW
  - 负责业务一致性和 commit / rollback 语义
  - 可能跨多个 dialog step / method / BAPI call
  - 通过 COMMIT WORK / ROLLBACK WORK 或 RAP commit/rollback 结束

Database LUW
  - 负责数据库层原子提交/回滚
  - commit / rollback 会结束当前 database transaction
  - implicit commit 可能由 work process 切换等运行时边界触发

这也是为什么“数据库锁已经有了”不能替代 SAP lock。database locks 通常随着 database commit / rollback 释放；但 SAP 业务动作可能比一个 database LUW 更长，需要 lock object / enqueue lock 来保护业务对象。

14.4.3 COMMIT WORK 不等于“保存一下”

COMMIT WORK 在 classic ABAP 里是强语义语句。它会结束当前 SAP LUW，触发注册的 update function modules / commit 相关处理，并对当前打开的 database connections 触发 database commit。COMMIT WORK AND WAIT 还会等待 update task 完成，适合后续逻辑必须读取更新结果的场景。

因此，不要把 COMMIT WORK 当成“保存一下，保险一点”。如果一个 helper method、BAdI、enhancement 或 API wrapper 擅自 commit，它会切断外层调用方的回滚能力。

典型判断：

这个程序自己从选择屏幕开始，自己完成一批业务动作，自己负责日志和失败处理
  -> 可以在明确边界处 COMMIT WORK / ROLLBACK WORK

这个方法会被多个上层流程复用
  -> 不要 COMMIT WORK；返回结果、消息和错误，让 caller 决定

这个逻辑在 update task、RAP handler/saver、bgRFC/tRFC/qRFC 管理的 LUW 中
  -> 遵守框架/运行时的事务契约，不要插入任性的 commit
14.4.4 ROLLBACK WORK 也不是时光机

ROLLBACK WORK 可以取消当前 SAP LUW 中尚未提交的注册内容，并触发 database rollback。但它不能撤回已经 database commit 的修改，也不能自动撤回外部系统调用、已发送邮件、已启动 job、已写入不可回滚日志、已提交的 BAPI 结果或已经对外可见的业务副作用。

所以事务边界设计还要考虑副作用顺序：

先完成校验
  -> 再准备写入
    -> 再提交
      -> 最后发送不可回滚通知 / 外部副作用

如果业务必须先调用外部系统再写 SAP，或者外部系统没有 distributed transaction，设计上就要明确补偿策略和幂等键，而不是假设 ABAP rollback 能解决一切。

14.4.5 BAPI commit pattern

BAPI 场景最容易出现两类错误：

错误 1：调用写入 BAPI 后不 commit
  -> 返回消息可能看起来成功，但数据没有最终提交

错误 2：在每个 BAPI 后立刻 commit
  -> 多个 BAPI 本应作为一个业务动作，却被切成多个不可整体回滚的小事务

现代 BAPI transaction model 的设计意图是让调用方显式控制事务。调用方可以组合多个 BAPI，然后在统一边界调用 BAPI_TRANSACTION_COMMIT；失败时调用 BAPI_TRANSACTION_ROLLBACK 或走错误处理。少数老 BAPI 内部可能包含 commit，必须以 BAPI 官方文档为准，不能凭命名猜。

在接口设计书里，BAPI 相关章节至少要写清楚：

  • 哪些 BAPI 是写入型调用。
  • 成功条件看 RETURN / message / failed item，而不是只看 RFC 没异常。
  • commit 是每笔、每组、每个 request，还是整批结束时。
  • 失败时是否允许部分成功。
  • 重跑时用什么业务键防止重复。
14.4.6 RAP transaction boundary

RAP 的事务边界要用 RAP 语言理解。MODIFY ENTITIES 通常把变化放进 transactional buffer；COMMIT ENTITIES 触发 RAP save sequence，并与当前 SAP LUW 的提交处理联动。不要把它简化成 classic COMMIT WORK 的别名：在 OData / Fiori 这样的 natively supported RAP scenario 中，commit request 通常由 RAP runtime 自动执行；在外部 EML consumer 中，才需要按 EML 规则显式提交并处理 response。

这意味着：

  • 在 RAP behavior implementation 中，通常不要显式 COMMIT ENTITIES 或 classic COMMIT WORK。
  • 外部 EML consumer 需要显式 COMMIT ENTITIES，并检查 FAILED / REPORTED / sy-subrc。
  • classic COMMIT WORK 缺少 COMMIT ENTITIES 的 RAP-specific response、key conversion、simulation mode 等语义；因此教程中优先教读者在 RAP 上下文使用 RAP 的 commit/rollback 概念。
  • ROLLBACK ENTITIES 清理当前 RAP transaction 的 transactional buffer，并与 ROLLBACK WORK 发生联动。
14.4.7 事务边界和 Clean Core

事务边界正确，不代表写入对象就正确。即使 commit 位置没错，直接写 SAP standard table、绕过 released API、绕过 RAP behavior、绕过 authorization 和 validation，仍然可能违反 Clean Core 和升级稳定性原则。

事务边界解决的是“一组修改如何一起成功或一起失败”；Clean Core / released API 解决的是“这组修改是否应该通过这个入口发生”。两者都要成立。

14.7 Example / 学习例子

14.7.1 例 1：classic report 拥有事务边界
LOOP AT lt_items INTO DATA(ls_item).
  " 校验、锁定、准备写入
  MODIFY zt_demo FROM @ls_item.
  IF sy-subrc <> 0.
    lv_failed = abap_true.
    EXIT.
  ENDIF.
ENDLOOP.

IF lv_failed = abap_true.
  ROLLBACK WORK.
ELSE.
  COMMIT WORK AND WAIT.
ENDIF.

这段示意代码的重点不是 MODIFY，而是 commit 粒度：如果业务要求整批一起成功，那么不要在 LOOP 里每行 commit。COMMIT WORK AND WAIT 表示后续逻辑依赖更新已完成；如果不依赖，可以根据项目约定使用异步 update。但无论哪种，都要在设计里写清楚。

14.7.2 例 2：helper method 不拥有事务边界
METHOD update_item.
  " 做校验、准备数据、执行必要的写入或注册 update task
  " 不在这里 COMMIT WORK
  " 把结果和消息返回给 caller
ENDMETHOD.

这种方法可能被 report、BAPI wrapper、job、RAP additional save 或测试代码复用。它如果内部 commit，就会强迫所有 caller 接受它的事务边界。更好的做法是返回成功/失败、消息、changed keys，让 caller 统一决定 commit 或 rollback。

14.7.3 例 3：BAPI caller 统一提交
CALL FUNCTION 'BAPI_EXAMPLE_CHANGE'
  EXPORTING
    ...
  TABLES
    return = lt_return.

IF line_exists( lt_return[ type = 'E' ] )
   OR line_exists( lt_return[ type = 'A' ] ).
  CALL FUNCTION 'BAPI_TRANSACTION_ROLLBACK'.
ELSE.
  CALL FUNCTION 'BAPI_TRANSACTION_COMMIT'
    EXPORTING
      wait = abap_true.
ENDIF.

实际项目要看具体 BAPI 文档、RETURN 结构和业务消息。示意重点是：BAPI 调用成功不等于业务提交完成；commit/rollback 是 caller 明确负责的边界。

14.7.4 例 4：RAP EML consumer 使用 COMMIT ENTITIES
MODIFY ENTITIES OF zi_demo_bo
  ENTITY Demo
  CREATE FIELDS ( Name Status )
  WITH VALUE #( ( %cid = 'C1'
                  Name = 'Test'
                  Status = 'N' ) )
  MAPPED DATA(mapped)
  FAILED DATA(failed)
  REPORTED DATA(reported).

IF failed IS INITIAL.
  COMMIT ENTITIES
    RESPONSE OF zi_demo_bo
      FAILED DATA(commit_failed)
      REPORTED DATA(commit_reported).
ENDIF.

外部 EML consumer 可以显式 COMMIT ENTITIES。但在 RAP behavior implementation 内部，通常不应这样写；框架会按照 save sequence 管理事务。初学者要先记住：RAP 有自己的事务语言，不要把 classic report 的 COMMIT WORK 习惯原样搬进去。

14.7.5 例 5：一个文字版事务边界图
Fiori / OData request
  -> RAP interaction phase
  -> RAP early save
  -> RAP late save
  -> final database commit

ABAP report / batch job
  -> validate
  -> enqueue
  -> ABAP SQL / update task registration
  -> COMMIT WORK / ROLLBACK WORK

External interface calling BAPI
  -> call one or more BAPIs
  -> inspect RETURN messages
  -> BAPI_TRANSACTION_COMMIT / BAPI_TRANSACTION_ROLLBACK

14.8 Counterexample / Failure mode

14.8.1 误区 1：在循环里每行 commit
LOOP AT lt_items INTO DATA(ls_item).
  UPDATE zt_demo FROM @ls_item.
  COMMIT WORK.
ENDLOOP.

如果业务允许部分成功，这可能是有意设计；但如果业务要求整批一致，这就是缺陷。第三行失败时，前两行已经提交，ROLLBACK WORK 不能把它们撤回。

14.8.2 误区 2：BAdI / enhancement 里擅自 commit

BAdI 往往运行在标准事务或框架控制的 LUW 中。增强里插入 COMMIT WORK 可能破坏标准事务的保存顺序，导致锁、缓存、update task、错误处理和标准回滚逻辑出现不可预期行为。除非 SAP 文档或增强点文档明确允许，否则不要这样做。

14.8.3 误区 3：把 BAPI_TRANSACTION_COMMIT 当成固定模板

不是所有函数模块都需要 BAPI commit；不是所有 BAPI 都应该每调用一次就 commit。正确做法是查 BAPI 文档、看是否写入型、看 RETURN、看业务是否需要多个 BAPI 合并在同一个 LUW。

14.8.4 误区 4：把 RAP 的 save sequence 当成普通 SQL commit

RAP 的保存包含 interaction、early save、late save、transactional buffer、failed/reported response 等语义。用 COMMIT WORK 粗暴替代 COMMIT ENTITIES，可能丢失 RAP-specific response 和错误处理能力。

14.8.5 误区 5：认为 rollback 能撤回外部副作用

ABAP rollback 不能自动撤回外部系统已经处理的请求，也不能撤回已发送邮件或已触发 job。涉及外部副作用时，要设计 idempotency key、状态表、补偿逻辑和重试策略。

14.9 思考问题

  • 当前代码是 transaction owner，还是被外层事务调用的组件？
  • 这次业务动作的 commit 粒度是单笔、整批、每个 request，还是每个业务对象？
  • 失败时允许部分成功吗？如果允许，如何记录、重跑和对账？
  • 如果一个 helper method 里出现 COMMIT WORK，外层调用方还怎么 rollback？
  • 当前 BAPI 是否需要 BAPI_TRANSACTION_COMMIT？是否有文档说明内部 commit？
  • RAP 场景是 OData runtime 自动提交，还是外部 EML consumer 需要 COMMIT ENTITIES？
  • database lock 释放后，业务对象是否仍需要 enqueue lock？
  • 哪些外部副作用发生在 commit 之前？失败时如何补偿？
  • 用日语向 reviewer 说明时，能否说清楚 コミット責任 与 再実行時の整合性？
