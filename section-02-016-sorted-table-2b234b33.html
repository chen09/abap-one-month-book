16. Sorted table 排序表

  • Chapter 02
  • Section 16
  • self-contained study note

16.1 一句话结论

Sorted table 排序表，是 ABAP internal table 内部表的一种表类别：它按 primary table key 自动保持升序，支持 unique 或 non-unique key，适合按 key 有序读取、范围读取、前缀 key 读取和需要稳定 key 顺序的场景；它不是“手动 SORT 过的 standard table”，而是表类型本身就带有排序和查找语义。

16.2 到底学什么

上一节学的 standard table 像一个顺序容器：适合追加、遍历、小规模临时结果。如果要按 key 反复查找，standard table 可能要线性搜索。sorted table 的定位正好在这里：

standard table
  -> 顺序容器，key 查找可能线性搜索

sorted table
  -> 按 primary key 自动有序，key 查找可利用排序结构

hashed table
  -> 按 unique key 进行 hash 查找，适合完整 key 精确查找

sorted table 的关键不是“它看起来排过序”，而是：表的内部管理方式保证它始终按 primary table key 排序。插入新行时，ABAP 会把它放到合适位置，而不是简单放到末尾。读取时，如果搜索条件符合 sorted key 的使用规则，就可以避免 standard table 那种从头到尾的线性搜索。

最基本声明如下：

TYPES ty_t_flight TYPE SORTED TABLE OF ty_flight
  WITH NON-UNIQUE KEY carrier_id connection_id flight_date.

也可以声明 unique key：

TYPES ty_t_country TYPE SORTED TABLE OF ty_country
  WITH UNIQUE KEY country_id.

因此本节要学的是四件事：

  • sorted table 如何通过 key 保持顺序。
  • UNIQUE KEY 和 NON-UNIQUE KEY 如何决定是否允许重复 key。
  • 什么样的 READ TABLE / table expression / LOOP AT ... WHERE 能从排序结构受益。
  • 什么时候 sorted table 比 standard table 或 Hashed table 哈希表 更合适。

16.3 为什么重要

ABAP 性能问题经常出现在 internal table，不一定出现在数据库。典型模式是：

LOOP AT lt_headers INTO DATA(ls_header).
  READ TABLE lt_items INTO DATA(ls_item)
    WITH KEY header_id = ls_header-header_id.
ENDLOOP.

如果 lt_items 是 standard table，大量重复 key lookup 可能会慢。把它改成 sorted table 后，只要 key 设计符合访问方式，就能让查找利用排序结构。

sorted table 也很适合“一对多分组”场景。例如订单 header 对应多个 item，或航班 carrier / connection 下有多个 date。你可能不是只找一行，而是找某个 key 前缀下的一组行：

carrier_id = 'LH'
connection_id = '0400'
  -> 多个 flight_date

这种场景 hashed table 只能按完整 unique key 很快找到单行；如果需要按 key 前缀取得一段有序数据，sorted table 往往更自然。

对 Java/Python 背景学习者，可以把 sorted table 类比成按 key 维护顺序的集合或有序索引结构，而不是普通 list。它牺牲一部分插入/维护成本，换来更好的 key 访问和有序处理能力。项目上选择 table category 时，不要问“哪一种最快”，而要问“我的主要访问方式是什么”。

16.4 Key mechanism / 核心机制

16.4.1 sorted table 按 primary key 自动排序

sorted table 是 index table，也有 primary table index；但它的行顺序由 primary table key 决定。官方 glossary 对 sorted table 的核心定义就是：由 primary table index 管理，并始终按 primary table key 排序。

示例：

TYPES: BEGIN OF ty_item,
         order_id TYPE i,
         item_no  TYPE i,
         text     TYPE string,
       END OF ty_item.

DATA lt_items TYPE SORTED TABLE OF ty_item
  WITH NON-UNIQUE KEY order_id item_no.

INSERT VALUE #( order_id = 100 item_no = 20 text = `B` ) INTO TABLE lt_items.
INSERT VALUE #( order_id = 100 item_no = 10 text = `A` ) INTO TABLE lt_items.

虽然第二行插入较晚，但 internal table 会按 order_id item_no 的 key 顺序维护。遍历时会看到 item_no = 10 在 item_no = 20 前面。

16.4.2 UNIQUE KEY 与 NON-UNIQUE KEY

sorted table 可以定义 unique primary key：

DATA lt_country TYPE SORTED TABLE OF ty_country
  WITH UNIQUE KEY country_id.

这表示同一个 country_id 不能重复。如果插入重复 key，程序会得到相应错误或失败语义，具体取决于语句和上下文。

也可以定义 non-unique primary key：

DATA lt_flights TYPE SORTED TABLE OF ty_flight
  WITH NON-UNIQUE KEY carrier_id connection_id.

这表示相同 carrier_id connection_id 可以有多行。non-unique sorted table 很适合按分组读取：一组 key 下有多条明细，而且希望它们在表中相邻。

16.4.3 key 顺序很重要

sorted table 的 key 字段顺序不是装饰，它决定排序结构，也决定哪些读取能高效。

WITH NON-UNIQUE KEY carrier_id connection_id flight_date

这个 key 适合：

carrier_id
carrier_id + connection_id
carrier_id + connection_id + flight_date

因为这些是 key 的 left-justified / leading part。它不适合直接把 flight_date 单独作为主要搜索条件，因为 flight_date 不是 key 的前导字段。

如果项目里主要按 flight_date 查，就应该重新设计 key，或在 standard/sorted table 上增加 secondary sorted/hash key。

16.4.4 读取：不是所有 WHERE 都自动快

常见读取方式：

READ TABLE lt_flights INTO DATA(ls_flight)
  WITH TABLE KEY carrier_id    = lv_carrid
                 connection_id = lv_connid.

如果这两个字段是 sorted table primary key 的前导部分，读取可以利用排序结构。table expression 也遵循类似的 key 访问思想：

DATA(ls_flight) = lt_flights[
  carrier_id    = lv_carrid
  connection_id = lv_connid
].

但如果你写的是非 key 字段，或 key 顺序不匹配，ABAP 可能无法使用 sorted key 的优势。初学者要避免把“这张表是 sorted table”理解成“任何搜索都快”。

16.4.5 填充：优先 INSERT ... INTO TABLE

standard table 常用 APPEND，但 sorted table 的安全写法是：

INSERT ls_flight INTO TABLE lt_flights.

INSERT ... INTO TABLE 会按 sorted key 语义把行放到正确位置。技术上，在某些情况下对 sorted table APPEND 可能通过语法检查，但如果追加的行不符合排序位置，就有运行时错误风险。项目代码里不要把 sorted table 当成“末尾追加容器”。

16.4.6 修改 key 字段要小心

sorted table 的 key 是排序结构的一部分。修改 key 字段相当于要把行移到另一个位置。通过 field symbol 或引用原地修改 key 字段，可能被禁止或导致错误。安全思路是：

要改非 key 字段
  -> 可以按规则修改

要改 key 字段
  -> 通常删除旧行，再按新 key 插入新行

这和 hashed table 类似：key 是内部管理结构的一部分，不是普通字段。

16.4.7 sorted table、standard table、hashed table 如何选择

可以用这张判断表：

主要是 append + 顺序遍历 + 小表
  -> standard table

主要按 key 有序读取、范围读取、前缀 key 读取、允许 non-unique group
  -> sorted table

主要按完整 unique key 做精确读取
  -> hashed table

已有一张表，同时需要多个访问路径
  -> secondary table key

sorted table 的插入和删除会维护排序结构，有管理成本。不要为了“看起来高级”把所有 internal table 都改成 sorted table。只有访问模式能利用排序结构时，才值得使用。

16.7 Example / 学习例子

16.7.1 例 1：non-unique sorted table 处理一对多
TYPES: BEGIN OF ty_flight,
         carrier_id    TYPE c LENGTH 3,
         connection_id TYPE c LENGTH 4,
         flight_date   TYPE d,
         price         TYPE p LENGTH 8 DECIMALS 2,
       END OF ty_flight.

DATA lt_flights TYPE SORTED TABLE OF ty_flight
  WITH NON-UNIQUE KEY carrier_id connection_id flight_date.

INSERT VALUE #( carrier_id = 'LH' connection_id = '0400' flight_date = '20260614' price = '100.00' )
  INTO TABLE lt_flights.
INSERT VALUE #( carrier_id = 'LH' connection_id = '0400' flight_date = '20260613' price = '90.00' )
  INTO TABLE lt_flights.

遍历 lt_flights 时，同一个 carrier/connection 的航班日期会按 key 顺序排列。因为 key 是 non-unique，也允许同一 key 前缀下有多条记录。

16.7.2 例 2：unique sorted table 防止重复 key
TYPES: BEGIN OF ty_country,
         country_id TYPE c LENGTH 3,
         name       TYPE string,
       END OF ty_country.

DATA lt_country TYPE SORTED TABLE OF ty_country
  WITH UNIQUE KEY country_id.

INSERT VALUE #( country_id = 'JPN' name = `Japan` ) INTO TABLE lt_country.

如果业务语义要求 country_id 唯一，UNIQUE KEY 能把这个约束写进 table type。相比 standard table 的 non-unique primary key，这种代码更容易 review。

16.7.3 例 3：按 leading key 读取
READ TABLE lt_flights INTO DATA(ls_flight)
  WITH TABLE KEY carrier_id    = 'LH'
                 connection_id = '0400'.

IF sy-subrc = 0.
  " 找到第一条匹配记录
ENDIF.

如果 table key 是 carrier_id connection_id flight_date，这里使用的是 key 的前导部分。sorted table 可以利用排序结构。如果只按 flight_date 查，优势就不成立，因为它不是 leading key。

16.7.4 例 4：范围/分组处理
LOOP AT lt_flights INTO DATA(ls_flight)
  WHERE carrier_id = 'LH'
    AND connection_id = '0400'.
  " process all matching flights
ENDLOOP.

当 carrier_id connection_id 是 sorted key 的前导部分时，匹配行在内部表里是连续的一段。这是 sorted table 很适合一对多明细处理的原因。

16.7.5 例 5：从 standard table 改为 sorted table 后删除冗余 SORT
" Before
DATA lt_flights TYPE STANDARD TABLE OF ty_flight WITH EMPTY KEY.
SORT lt_flights BY carrier_id connection_id flight_date.

" After
DATA lt_flights TYPE SORTED TABLE OF ty_flight
  WITH NON-UNIQUE KEY carrier_id connection_id flight_date.

改成 sorted table 后，表本身会按 key 维护顺序。再写一条按同样 key 的 SORT 通常没有意义；如果代码依赖不同排序顺序，就说明这个 table category 可能不是唯一需要考虑的访问路径。

16.8 Counterexample / Failure mode

16.8.1 误区 1：把 sorted table 等同于 SORT standard table
SORT lt_standard BY carrier_id connection_id.

这只是把当前 standard table 排了一次序。之后如果又 APPEND 新行，排序可能被破坏。sorted table 则是在插入/删除过程中持续维护 key 顺序。

16.8.2 误区 2：key 顺序设计错
DATA lt_flights TYPE SORTED TABLE OF ty_flight
  WITH NON-UNIQUE KEY carrier_id connection_id flight_date.

READ TABLE lt_flights INTO DATA(ls_flight)
  WITH KEY flight_date = lv_date.

这张表按 carrier_id connection_id flight_date 排序，但读取只按 flight_date。这种访问不能很好利用 primary sorted key。应该考虑调整 key 顺序、增加 secondary key，或选择另一种 table category。

16.8.3 误区 3：用 APPEND 当作通用填充方式

standard table 常见 APPEND，但 sorted table 推荐 INSERT ... INTO TABLE。如果一定要 append，就必须保证新行属于当前排序末尾，否则有运行时风险。项目代码里为了可读性和稳定性，优先写 INSERT。

16.8.4 误区 4：以为 sorted table 一定比 hashed table 慢或快

hashed table 适合完整 unique key 精确读取；sorted table 适合有序、前缀、范围、non-unique group 的读取。两者不是简单的“谁更快”，而是适用访问模式不同。

16.8.5 误区 5：为了“性能”盲目改表类型

如果代码只是全表 LOOP AT，没有 key filter，sorted/hashed 的性能优化意义不大，还可能增加插入和维护成本。先用 where-used list、runtime trace 或代码阅读确认访问方式，再改 table category。

16.9 思考问题

  • 这张 internal table 的主要访问方式是全表遍历、完整 key 查找，还是 key 前缀/范围读取？
  • key 字段顺序是否和 READ TABLE / LOOP AT ... WHERE 的条件一致？
  • 这张表允许重复 key 吗？应该用 UNIQUE KEY 还是 NON-UNIQUE KEY？
  • 代码里是否还有对 sorted table 的冗余 SORT？
  • 是否有人用 APPEND 填充 sorted table？这是否安全？
  • 如果需要按两个不同 key 快速读取，是否应该用 Secondary table key 二级键？
  • 这个场景和 Hashed table 哈希表 的差异是什么？
  • 用日语 review 时，能否说明 先頭キー項目 和 範囲検索？
