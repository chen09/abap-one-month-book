16. SAP Locking Architecture Overview

  • Chapter 03
  • Section 16
  • self-contained study note

16.1 一句话结论

SAP Locking Architecture Overview 是本教程的锁机制总览图：它把 Lock object 锁对象、SAP enqueue、SELECT ... FOR UPDATE、RAP locking、ETag/乐观并发、幂等和事务边界放在同一张地图里，帮助读者判断“当前并发风险应该在哪一层控制”。

一句话记忆：SAP 锁不是一个按钮，而是一组分层机制。数据库锁保护数据库行，SAP enqueue 保护业务对象，RAP locking 保护 RAP business object 生命周期，幂等保护重复请求，事务边界决定失败时能否一起回滚。

16.2 到底学什么

本节不是再次详细讲 lock object，也不是把所有锁语法一次讲完。前面 Week 1 已经建立 Lock object 锁对象 的基础，Week 2 讲过 SELECT FOR UPDATE vs SAP enqueue，Week 4 还会从生产就绪角度讲 Enqueue locking 加锁设计。本节的作用是把这些知识合并成一张架构判断图。

第一，要区分锁的层级。数据库锁发生在数据库事务和 SQL 访问层；SAP enqueue 发生在 SAP 应用层，围绕业务 key 表达排他控制；RAP locking 发生在 behavior/business object 模型中，和 action、authorization、save sequence、transactional buffer 一起工作。

第二，要区分锁和其他一致性机制。锁用于避免并发同时修改，不等于授权；锁能阻止别人同时处理，不等于幂等；锁能减少覆盖更新，不等于 transaction rollback；乐观并发控制通过版本/ETag 发现冲突，也不等于 enqueue 的悲观锁。

第三，要学会项目判断。遇到“两个用户同时改同一对象”“后台 job 和 API 同时处理同一批次”“外部系统重试导致重复执行”“保存时发现对象已被别人改过”这些问题时，要先判断风险属于并发修改、重复请求、权限边界、状态机、事务边界还是错误消息设计，再选择机制。

第四，要学会用同一套语言连接 classic ABAP、RAP 和 OData。classic ABAP 常说 lock object、enqueue/dequeue、SM12；RAP 常说 lock master、lock dependent、managed/unmanaged locking、FOR LOCK；OData/API 场景常说 ETag、If-Match、412 Precondition Failed、409 conflict、retry 和 idempotency。它们不是三套互不相干的知识，而是在不同入口表达“同一业务对象不能被错误地并发处理”。

第五，要学会区分“提前拦住”和“保存时发现”。SAP enqueue / RAP lock 更偏 pessimistic locking：处理前先取得排他资格；ETag / optimistic concurrency 更偏 conflict detection：允许用户拿到数据，但保存时检查版本是否已变化。两种方案都可以合理，关键是设计书、API contract 和测试预期必须一致。

16.3 为什么重要

并发问题在单人学习环境里很少出现，但在真实 SAP 项目里非常常见。开发者自己测试时一切成功，上线后才发现两个用户同时保存覆盖、批量上传被执行两次、后台 job 和手工操作抢同一对象、API 超时重试造成重复单据、RAP action 失败但部分 Z 表已写入。这些问题往往不是某一行代码错，而是锁、事务和幂等边界没有设计清楚。

对 Java/Python 背景读者来说，最容易误判的是把 SAP enqueue 当成普通数据库 row lock，或者把 SELECT FOR UPDATE 当成所有并发问题的答案。SAP 的关键差异在于业务对象常跨多张表、多条记录、多个 dialog step、update task、SAP LUW、OData request、RAP save sequence 和后台 job。只锁一行数据库记录，未必能保护完整业务对象。

对项目交付来说，锁架构影响设计书、测试用例和障害調査。设计书不能只写“加锁”，要写清锁对象、锁 key、锁时机、锁失败返回、解锁路径、重复请求处理和 rollback 语义。测试用例不能只测单用户保存，要测两个用户、两个请求、后台 job/API 同时处理同一业务对象的场景。

16.4 Key mechanism / 核心机制

可以用四层地图来理解 SAP locking architecture：

Database level
  -> SELECT ... FOR UPDATE, database transaction, row-level lock
  -> protects selected database rows for a short database transaction

SAP application level
  -> DDIC lock object, ENQUEUE/DEQUEUE, enqueue server
  -> protects a business key / business object across SAP application processing

RAP business object level
  -> behavior definition locking, managed/unmanaged locking, RAP runtime
  -> protects RAP BO lifecycle together with actions, save sequence and messages

API/project level
  -> idempotency key, status machine, ETag/If-Match, retry policy, application log
  -> protects external consumption, retries, auditability and conflict explanation

数据库层适合短时间、明确数据库行的并发保护。它不能天然表达“这张业务单据由 header、item、status、log 多张表组成”，也不能跨用户编辑流程自然存在。

SAP enqueue 适合业务对象级排他控制。典型做法是通过 lock object 定义 lock key，再用 generated enqueue/dequeue API 或现代 released locking API 获取和释放锁。它表达的是“这个业务对象现在被某个会话/流程处理”，比单纯 SQL row lock 更贴近 SAP 业务语义。

RAP locking 要放回 behavior 模型中理解。RAP business object 可能由 framework 管理锁，也可能在 unmanaged 场景需要开发者实现锁逻辑。RAP 里的锁还要和 global/instance authorization、validation、action handler、save sequence、reported/failed message 结合，不能随意在 handler 里绕过框架直接改表。

API/project 层则关注消费者行为。外部系统重试、用户重复点击、middleware timeout、并发调用，不一定只靠锁解决。重复请求要看 AC0567C idempotency replay；保存阶段要看 AC0567C RAP saver replay；API 证据要看 AC0567C API evidence replay。

进一步，可以把锁架构判断压缩成一个项目决策树：

1. Is the risk "same business object processed at the same time"?
   -> yes: consider SAP enqueue / RAP locking / business-object lock.

2. Is the risk "same request arrives again after timeout or retry"?
   -> yes: add idempotency key, unique constraint, status replay, or request log.

3. Is the risk "user saved stale data after another user changed it"?
   -> yes: consider ETag / optimistic concurrency / version check.

4. Is the risk "user is not allowed to do this operation"?
   -> yes: authorization, not locking.

5. Is the risk "business state is invalid"?
   -> yes: validation/state machine, not locking.

6. Is the risk "partial side effect after failure"?
   -> yes: transaction boundary, RAP save sequence, application log, rollback design.

这个决策树看起来简单，但它能避免项目里最常见的一句话误判：“这个问题加锁就好了。”很多时候真正需要的是“锁 + 状态 + 幂等 + 错误响应 + 日志证据”的组合。

RAP 语境下还要补一层心智模型。Behavior definition 里的 locking 声明不是装饰语法，它告诉 RAP runtime 哪个 entity 是 lock owner，child entity 是否依附 root 锁，以及 unmanaged 场景是否要项目代码实现 lock operation。Managed BO 往往让 framework 处理更多锁细节；unmanaged BO 或 wrapper 场景则更需要开发者明确 FOR LOCK、lock key、reported/failed message 和 released locking API 的使用边界。初学者不必立刻背语法，但读 BDEF 时看到 lock master 或 lock dependent，要马上想到：这里定义的是业务对象并发控制边界。

OData 语境下，ETag 是另一条重要线索。ETag 常用于 optimistic concurrency：客户端读取对象时拿到版本标识，更新时通过 If-Match 说明“我基于这个版本修改”。如果服务器发现对象已被别人修改，就可以拒绝保存，让消费者重新读取最新数据。它和 enqueue 的差异在于：ETag 不一定阻止第二个人开始编辑，而是在提交时发现版本冲突；enqueue 更偏在关键处理开始前阻止并发进入。

16.Locking architecture review checklist

做 SAP 锁设计 review 时，不要只问“有没有加锁”，而要按下面的顺序确认。第一，业务对象边界是否清楚：被保护的是一张表的一行、一个 header、一个 header 加 item 的组合，还是一个跨 API / job / UI 的处理批次？如果边界说不清，锁 key 通常也会错。

第二，锁 key 是否贴近业务对象。锁 key 过粗会把无关业务串行化，造成性能和用户体验问题；锁 key 过细会保护不了真正会被同时修改的状态。项目上常见的好问题是：两个不同用户同时处理同一个 document number、batch id 或 master data key 时，系统应该让谁通过、让谁等待或失败？

第三，锁的取得时机和释放时机是否可解释。只在保存瞬间加锁，可能保护不了长时间编辑或长处理；过早加锁又可能造成锁占用时间过长。释放路径也要覆盖正常保存、取消、异常、rollback、session 断开和后台 job 失败。设计书里如果只有“enqueue を実施する”，但没有取得/解除タイミング，就还不是完整锁设计。

第四，锁失败时的响应是否面向消费者。UI 用户需要知道对象正在被谁或哪个处理占用，API consumer 需要稳定的 HTTP status、business error code、message 和可重试判断。不要把 lock conflict 写成普通 validation error，也不要把 authorization failure 伪装成 lock failure。

第五，是否同时需要 ETag、状态机或幂等。并发同时处理靠 lock，旧版本保存靠 ETag / optimistic concurrency，重复请求靠 idempotency，业务状态跳转靠状态机。真实项目经常需要组合拳：例如 action 执行前用业务锁互斥，保存时用状态/版本检查兜底，对外部重试用 request id 保证幂等。

第六，RAP 场景是否回到 BDEF 和 behavior implementation 验证。看到 lock master / lock dependent 后，要继续确认 child 的锁是否依附 root、unmanaged lock 是否实现、reported / failed 消息是否清楚、projection/service 是否暴露了相关操作，以及是否没有在 handler 中用直接提交破坏 save sequence。

第七，是否有可重复的并发测试证据。至少准备一个“两请求同时处理同一对象”的测试，一个“锁冲突后错误消息可读”的测试，以及一个“失败后锁释放或状态可恢复”的测试。没有证据的锁设计，很容易在上线后才暴露为偶发障害。

16.7 Example / 学习例子

假设 AC0567C 有一个上传批次 BATCH-001，外部系统可能重复调用 execute，同时业务用户也可能在 UI 中手工重新处理。

项目判断可以这样拆：

风险首先考虑的机制说明
两个执行请求同时处理同一批次SAP enqueue / RAP locking用 batch id 作为业务锁 key
同一个 request id 重试idempotency key / 状态机重试不应重复创建业务数据
用户无权执行该批次global / instance authorization锁不能替代权限
保存中途失败transaction boundary / RAP save sequence确认是否 rollback、日志是否一致
两次读取之间数据被改ETag / optimistic locking / 状态检查根据 UI/API 消费模式选择
后台 job 和 API 同时处理enqueue + job/API 测试证据覆盖跨入口并发

读者可以用这个表复盘任意 SAP 对象：先问“我要保护什么业务对象”，再问“风险是同时处理、重复请求、无权限、状态非法，还是保存失败”。机制选错层，后面代码再漂亮也救不回来。

如果把它落到 AC0567C，可以设计一组最小并发证据：

测试入口预期结果证据
同一 batch 两次 execute 同时到达OData action / API client只有一个请求进入关键处理，另一个得到 lock/conflict 响应request id、batch id、HTTP response、application log
同一 request id 超时后重试API client / middleware retry返回已处理、处理中或可重放结果，不重复创建业务数据idempotency record、status、最终表数据
UI 用户与后台 job 同时处理同一 batchFiori/UI + background job两个入口使用同一业务 key 互斥job log、UI error、lock/application log

这个测试矩阵的重点不是制造复杂自动化，而是让设计可以被验证。项目 review 时，如果只说“已经加锁”，但拿不出两个请求、两个入口、一个失败响应和一个 application log，就还不能算生产就绪。

16.8 Counterexample / Failure mode

反例一：用 SELECT ... FOR UPDATE 代替业务对象锁。单表短事务可能没问题，但如果业务对象跨 header/item/log/status 多张表，或用户编辑流程跨多个步骤，数据库行锁无法完整表达业务排他。

反例二：只在 UI 按钮上禁用并发，后台 job/API 没加锁。结果 UI 用户看似安全，但外部接口或 job 仍能同时处理同一对象。

反例三：锁 key 过粗。为了省事按公司代码加锁，导致同一公司所有批次串行，性能和用户体验崩溃。正确做法通常是贴近业务对象 key，例如 document number、batch id、master data key。

反例四：锁 key 过细。只锁一条 item，实际 header 状态会被多个 item 流程同时改，最终产生状态错乱。

反例五：把重复请求当成锁问题。锁只能阻止同时处理，不能证明网络超时后第二次请求不会重复落库。重试安全仍要靠幂等 key、唯一约束、状态机和证据。

反例六：把 ETag 当成 enqueue。ETag 可以发现 stale update，但它通常不会阻止第二个用户进入编辑或第二个 API request 开始准备数据。需要提前互斥的长处理，仍要考虑 SAP enqueue / RAP locking / 状态锁。

反例七：把 RAP locking 当成 classic function module 调用清单。RAP 的锁要和 BDEF、behavior pool、transactional buffer、save sequence、reported/failed 消息一起看；在 ABAP Cloud 或 Clean Core 语境下，还要确认是否使用 released API。

16.9 思考问题

  • 当前业务对象的锁 key 应该是什么：document number、batch id、company code，还是组合 key？
  • 这个问题是并发同时修改，还是重复请求重放？应该用锁、幂等，还是两者都要？
  • 锁失败时，用户/API consumer 应该看到什么错误消息？
  • 锁什么时候释放：正常保存、rollback、用户取消、session 断开、job 异常时分别如何处理？
  • RAP managed/unmanaged 场景中，锁由 framework 管理还是项目代码负责？
  • 设计书和测试证据里是否说明了 2 用户 / 2 请求同时处理同一对象的结果？
