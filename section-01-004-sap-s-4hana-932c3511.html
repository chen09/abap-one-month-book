Chapter 01 - SAP 和 ABAP 基础

4. SAP S/4HANA 业务套件定位

  • Chapter 01
  • Section 04
  • self-contained study note

本节结论

SAP S/4HANA 是 SAP 的下一代 ERP / business suite 产品；它运行在 SAP HANA 上，面向企业核心业务流程，不等于数据库、IDE、ABAP Platform 或 ABAP Cloud 开发模型。

正文

学习 SAP S/4HANA，第一步不是背模块名，而是把它放在正确的位置上。SAP 官方资料把 S/4HANA 的完整名称解释为 SAP Business Suite 4 (for) SAP HANA。这里的重点有两个：它是 Business Suite，也就是承载企业业务流程的一套应用产品；它又是 for SAP HANA，也就是以 SAP HANA 作为核心数据库和数据平台来设计的新一代产品。

因此，本节要先排除几个常见混淆。SAP HANA 是底层 in-memory database / data platform，不是业务套件本身。SAP Fiori 是用户体验、launchpad 和应用入口，不是 S/4HANA 本身。ABAP Platform 是 ABAP 技术基础，不是业务产品。ADT 是开发工具，不是运行环境。ABAP Cloud 是受 Clean Core 和 released APIs 约束的现代 ABAP 开发模型，也不是 S/4HANA 这个产品本身。

真正要记住的是：SAP S/4HANA 是企业业务系统的主体。财务过账、采购订单、销售订单、库存移动、生产计划、主数据维护、审批流程、分析报表等标准业务能力，都以 S/4HANA 的标准流程、数据模型、应用和扩展点为中心展开。ABAP 开发者写的 class、CDS、RAP BO、OData service、BAdI 或 API 调用，通常不是孤立技术练习，而是在这个业务套件周围补足差异化需求。

这里的 simplified data model 对开发者不是营销词。它意味着 S/4HANA 对传统 ERP 的部分聚合表、索引表和历史数据结构做了重构，例如财务里的 Universal Journal 表 ACDOCA、库存管理里的 material document 表 MATDOC 会在 Week 2 的 ABAP SQL / CDS 章节再展开。Day 01 只要先记住一点：S/4HANA 的表和视图不能完全按 ECC 旧经验推断。

本节不展开 FI / MM / SD / PP 等模块的细节。Day 01 的目标只是建立产品定位：以后看到一个需求时，先问“这是 S/4HANA 标准业务流程里的哪个对象、哪个应用、哪个版本、哪种部署形态”，再讨论 ABAP 怎么做。

为什么重要

如果把 SAP S/4HANA 当成“数据库加一点 ABAP”，学习方向会立刻偏掉。Java/Python 背景的人容易从代码仓库、服务、数据库表和 CI/CD 去理解系统；这套视角有用，但不足以解释 SAP 项目的真实工作方式。在 SAP 项目里，很多开发决策要先服从标准业务流程、产品版本、扩展方式、权限模型、传输路径和升级稳定性。

例如，一个客户说“采购订单画面上要加一个字段”。普通 Web 项目里，可能会直接讨论前端表单、后端 DTO 和数据库表；S/4HANA 项目里，则要先判断标准功能、扩展方式、部署形态和 Clean Core 约束。完整判断过程放到本节“学习例子”中展开。

对 ABAP 开发者来说，S/4HANA 的定位决定了后续学习的上下文。学 CDS view entity 视图实体 时，要理解它可能服务于 S/4HANA embedded analytics、RAP 或 OData API；学 RAP business object 业务对象 时，要理解它通常围绕业务对象生命周期建模；学 Service binding 服务绑定 或 OData 时，要知道服务暴露出去的是业务能力，而不是随便包装一张表。学 Released API dependency check 时，也要知道为什么不能随便调用未释放对象：因为项目最终要在 S/4HANA 生命周期里可升级、可维护。

核心机制

可以用五层关系来定位 SAP S/4HANA：

Business product
  SAP S/4HANA

Database / data platform
  SAP HANA

User experience
  SAP Fiori launchpad, SAP Fiori apps, role-based UX

Technology foundation
  ABAP Platform, AS ABAP, DDIC, ABAP SQL, CDS, RAP, Gateway/OData

Development and extension model
  Classic ABAP, key user extensibility, developer extensibility, ABAP Cloud, released APIs, Clean Core

第一层是 product boundary。S/4HANA 是业务产品，回答“企业用哪套核心业务系统运行流程”。ABAP Platform 回答“ABAP 应用靠什么技术基础运行和开发”。AS ABAP 回答“请求如何在应用服务器层被执行”。ADT 回答“开发者用什么工具编辑 repository object”。ABAP Cloud 回答“在现代扩展场景下，允许使用哪些 ABAP 语言特性和 released APIs”。把这几层分开，后面读官方文档才不会把产品、平台、运行时和开发规则混成一团。

第二层是 HANA relationship。SAP S/4HANA 是为 SAP HANA 设计的产品，S/4HANA 后端服务器的主数据库（primary database）要求是 SAP HANA。HANA 提供内存计算、实时分析和简化数据模型的基础，但 HANA 本身不是 ERP。可以把它粗略类比为“高性能数据库和数据平台”，但不要把它当成采购、销售、财务这些业务流程的承载者。

第三层是 Fiori relationship。SAP Fiori 是 S/4HANA 的重要用户体验入口，SAP Fiori launchpad 让用户按角色访问应用。很多现代 S/4HANA 业务操作通过 Fiori apps 完成，背后可能调用 OData、CDS、RAP 或传统后端逻辑。但 Fiori 是“界面和体验层”，不是整套 S/4HANA 产品。看到一个 Fiori app，不等于后端没有 ABAP，也不等于所有扩展都在前端完成。还要注意部署差异：on-premise / Private Edition 项目里，SAP GUI 和传统界面仍然可能与 Fiori 并存；Public Edition 面向业务用户的入口则以浏览器 (HTTP/S) / SAP Fiori launchpad 为中心。

第四层是 deployment and extensibility。SAP S/4HANA 有不同产品/部署形态：on-premise、SAP S/4HANA Cloud Private Edition、SAP S/4HANA Cloud Public Edition。Public Edition 更标准化，强调 Fit-to-Standard、key user extensibility、developer extensibility、released APIs 和 ABAP Cloud；Private Edition / on-premise 通常会遇到更多 classic ABAP、BAdI、BAPI、ALE/IDoc 等 brownfield 现实，但这些不能被理解为“默认可以随便用”。正确理解是：Clean Core 治理适用于所有形态；能用 released APIs、RAP、kernel-based BAdI 等更稳定方式时，不应优先走 classic pattern。不同形态的可用工具、访问层级、传输治理、扩展边界和升级责任也不同。

第五层是 version evidence。项目里不要只说“我们是 S/4HANA”。还要确认具体 product version、ABAP Platform release、S4CORE、SAP Fiori 组件、support package 和部署形态。SAP Learning 的系统检查示例会让学习者到 System -> Status -> Installed Product Versions 查看 S4HANA ON PREMISE、ABAP PLATFORM、SAP FIORI FOR S/4HANA 等信息。对开发者来说，这些版本信息会影响 CDS/RAP/OData 能力、released APIs 可用性、ADT 检查结果和交付方式。

日本项目现场表达

  • 常见日语：次世代ERP、基幹システム（正式文档里也会见到 基幹業務システム）、業務アプリケーション、標準機能、標準プロセス、業務プロセス、Fit-to-Standard / フィット・トゥ・スタンダード、Public Edition、Private Edition、オンプレミス、アドオン開発、拡張、クリーンコア、リリース済みAPI。
  • 说明句：SAP S/4HANA は次世代ERP／基幹業務アプリケーションであり、SAP HANA や ABAP Platform そのものではありません。
  • 说明句：この要件は標準機能で対応できるのか、アドオン開発や拡張が必要なのかを確認します。
  • 说明句：Public Edition では拡張方式と移送方式に制約があるため、Fit-to-Standard とクリーンコアを前提に検討します。
  • 注意点：S/4HANA では ECC 時代の資料と日本語用語が変わることがあります。日本語の機能仕様書を読むときは、旧称と新称を Glossary で確認します。
  • 使用场景：需求确认、Fit-to-Standard 讨论、技术方案说明、Clean Core review、升级影响调查、客户/上级解释“为什么不能直接改标准对象”。

学习例子

假设项目里有一个需求：“采购订单审批时，需要按供应商风险等级显示额外提示。”初学者如果只从编程出发，可能会马上想建表、写接口、改画面。但在 S/4HANA 项目里，第一步应该是确认业务对象和标准能力：它是不是采购订单相关流程？对应的标准 Fiori app 或 SAP GUI 事务是什么？标准字段、标准配置、BAdI、released API、CDS view 或 extension point 是否已经能支持？当前系统是 Public Edition、Private Edition 还是 on-premise？

如果是 Public Edition，团队通常会先考虑 Fit-to-Standard、key user extensibility、developer extensibility 和 released APIs，不能假设可以随便修改标准对象。如果是 Private Edition 或 on-premise，可能存在 classic ABAP、BAdI、enhancement、custom table、RAP side-by-side/on-stack 扩展等选择，但也要检查 Clean Core 风险。这个例子说明：同样是“写 ABAP”，真正的起点不是语法，而是 S/4HANA 业务产品和部署形态。

最小练习：用 8-10 行中文解释下面五个词的关系：SAP S/4HANA、SAP HANA、SAP Fiori、ABAP Platform、ABAP Cloud。进阶练习：打开 SAP Fiori apps reference library，搜索一个和采购、销售或财务相关的 app，观察它支持的 product version 和 implementation information。

常见误区

  • 误区一：把 SAP S/4HANA 当成 SAP HANA 数据库。修正：HANA 是数据平台；S/4HANA 是运行企业业务流程的 ERP / business suite。
  • 误区二：把 SAP Fiori app 当成全部系统。修正：Fiori 是 UX 和应用入口，后端仍然可能是 ABAP、CDS、RAP、OData、authorization、workflow 和标准业务逻辑。
  • 误区三：以为会 ABAP 语法就能开发 S/4HANA。修正：S/4HANA 开发还要理解标准流程、版本、部署形态、released APIs、Clean Core、权限、传输和测试。
  • 误区四：以为 Private Edition / on-premise 可以随便改标准对象。修正：技术上可能更开放，但升级稳定性、审计、运维和 Clean Core 仍然约束开发方式。
  • 误区五：以为 Public Edition 不能做任何扩展。修正：Public Edition 有 key user extensibility、developer extensibility、released APIs 和 side-by-side 扩展路径，只是边界更清晰、更受管控。

思考题

  • 为什么 SAP S/4HANA 不是“ABAP Platform 的另一个名字”？
  • 为什么 SAP HANA 对 S/4HANA 很关键，但仍然不能把两者等同？
  • 一个 Fiori app 背后可能有哪些 ABAP / OData / CDS / authorization 组件？
  • 同一个扩展需求，在 Public Edition、Private Edition、on-premise 下会有哪些决策差异？
  • 在日本项目里，什么时候该说 標準機能で対応，什么时候该说 アドオン開発 或 拡張？
