7. Join vs association 二者区别

  • Chapter 03
  • Section 07
  • self-contained study note

7.1 一句话结论

Join 和 association 都能表达“两个数据源之间有关联”，但它们解决的是不同层级的问题：join 是当前查询里立即组合数据源的 SQL 操作；association 是 CDS view entity 视图实体 中声明可复用、可导航关系的建模机制。Join 回答“这一次查询现在要把哪些行和字段合在一起”；association 回答“这个实体和另一个实体之间有什么关系，后续消费者是否可以沿着这个关系导航”。

本节最重要的结论是：不要用“性能好不好”或“写法新不新”来简单选择。需要当前结果集马上合并字段时，用明确的 join；需要在 CDS/RAP/OData 模型中表达稳定业务关系、让 projection/service/Fiori/API 后续按需消费时，用 association。Association 最后也可能生成 join 或导航访问，但它的价值不是隐藏 join，而是把关系放到模型层，让复用、cardinality、服务暴露和 review 有一个稳定入口。

7.2 到底学什么

上一节 Association 关联关系 已经说明 association source、target、alias、ON condition、cardinality 和 exposed association。本节换一个角度：当读者看到两种写法时，如何判断它们不是同一件事？

需要掌握七个判断点。

  • 层级不同：join 属于当前 ABAP SQL / CDS SELECT 的查询结构；association 属于 CDS data model 的关系声明。
  • 发生时机不同：join 是当前查询的一部分；association 可以先定义和暴露，之后才通过 path expression、Data Preview、ABAP SQL 或 OData navigation 被消费。
  • 复用方式不同：join 通常只服务当前查询；association 可以被 projection view、service definition、OData metadata、Fiori Elements 或其他 CDS view 复用。
  • 结果集风险不同：join 的行数变化直接发生在当前结果集；association 的行数风险通常在 path expression 或 to-many navigation 被消费时显现。
  • 语义信息不同：join 本身不声明 cardinality；association 用 cardinality 表达 source -> target 的语义契约。
  • API 边界不同：join 不会自然形成 OData navigation property；exposed association 才可能进入服务导航链路。
  • Review 重点不同：join review 看 ON/WHERE、字段来源、结果行数、null、排序和性能；association review 看关系是否真实、cardinality 是否可信、是否应暴露、是否符合 service/Clean Core 边界。

7.3 为什么重要

在初学阶段，把所有关系都写成 join，看起来最直观：订单要客户名，就 join 客户；订单要项目，就 join 项目；页面要供应商文本，就 join 供应商文本。问题是，这种写法把关系逻辑散落在每个查询里。项目一大，ON condition、字段别名、语言文本、cardinality、权限、服务暴露和性能约束会被复制很多次，后续变更很难评估。

但反过来，把所有 join 都改成 association 也不对。某些查询就是要一次性拿到一个报表结果集，字段来源固定，消费方单一，后面也不需要 OData navigation 或 CDS 模型复用。此时一个清楚的 ABAP SQL join 或 CDS join 可能比为了“现代化”强行建一层 association 更简单。

对 SAP 项目交付来说，本节决定的是建模边界。底层 interface view / root view 里是否应该建 association？projection view 是否应该暴露这个 association？ABAP class 里是否应该直接 join SAP 标准底表？Fiori/OData 是否需要 navigation property？这些问题会影响设计说明、代码 review、单体测试、性能测试、权限测试、API 契约和上线后的变更成本。

对 Java/Python 背景学习者，可以把 join 理解成 SQL query 的组合操作，把 association 粗略理解成 schema/model relationship。但 SAP 的差异在于：association 是 ABAP repository object 的 CDS DDL 语义，进入 activation、transport、where-used、DCL、projection、service binding 和 OData metadata 链路；join 则更像某个查询语句里的执行结构。二者会在数据库层相遇，但在项目设计层不是同一个东西。

7.4 Key mechanism / 核心机制

7.4.1 Join 是当前查询立即组合数据源

ABAP SQL join 的核心，是在一个 SELECT 中把两个或多个数据源按 ON 条件组合成一个结果集。SAP-samples 的 ABAP SQL cheat sheet 也把 join 放在“Combining Data of Multiple Data Sources”下：inner join 只保留满足 join condition 的组合；outer join 会保留一侧数据源的行，并在另一侧没有匹配时产生 null。

一个典型 ABAP SQL join：

SELECT FROM zso_header AS so
       INNER JOIN zcustomer AS cu
          ON cu~customer_id = so~customer_id
       FIELDS so~sales_order,
              so~customer_id,
              cu~customer_name,
              so~created_at
       WHERE so~status = @lc_status_open
       ORDER BY so~created_at DESCENDING,
                so~sales_order DESCENDING
       INTO TABLE @DATA(lt_orders).

这段代码的问题空间很明确：

  • ON 是否表达真实业务关系？
  • inner join 是否允许右侧缺失时丢掉左侧记录？
  • WHERE 是过滤订单，还是过滤客户？
  • 结果是否可能因为 1:n 关系被放大？
  • 没有 ORDER BY 时，多行结果顺序是否不稳定？

Join 的优点是直接、明确、局部可读。缺点是复用性弱：如果十个消费方都需要“订单 -> 客户”，十个查询都可能复制这段关系。

7.4.2 Association 是模型层声明关系

CDS association 把关系放到 model 中：

define view entity ZI_SalesOrder
  as select from zso_header as Header
  association [0..1] to ZI_Customer as _Customer
    on $projection.CustomerId = _Customer.CustomerId
{
  key Header.sales_order as SalesOrder,
      Header.customer_id as CustomerId,
      Header.created_at  as CreatedAt,

      _Customer
}

这段代码的重点不是“比 join 短”，而是把 SalesOrder -> Customer 这个关系命名为 _Customer，并声明 cardinality。后续 projection view、另一个 CDS view、Data Preview、ABAP SQL path expression 或 OData service 可以按需要消费这个关系。

Association 的优点是关系可复用、可导航、可被服务层理解。缺点是它更依赖模型设计：target 是否合适，cardinality 是否真实，是否应该暴露，是否会泄漏内部模型细节，都需要 review。

7.4.3 一个简单判断：字段现在要不要进入当前结果集

如果当前查询马上需要 target 字段，而且这个查询就是最终消费形态，join 往往更清楚：

SELECT FROM zso_header AS so
       LEFT OUTER JOIN zdelivery AS dlv
          ON dlv~sales_order = so~sales_order
       FIELDS so~sales_order,
              dlv~delivery_id
       INTO TABLE @DATA(lt_rows).

如果当前 view 只是要表达“订单可以导航到客户”，但不一定每个消费方都要客户字段，association 更自然：

association [0..1] to ZI_Customer as _Customer
  on $projection.CustomerId = _Customer.CustomerId

判断句可以这样写：

  • 这一次查询一定要这些字段：优先考虑 join 或当前 view 的 path expression。
  • 这是一个稳定业务关系，多个消费者可能按需使用：优先考虑 association。
  • 这是外部 API 的导航契约：优先考虑 exposed association + projection/service 设计，而不是把 target 字段全部 join 进一个扁平列表。
7.4.4 Join 的行数变化立即发生；association 的行数风险在消费时显现

Join 的 cardinality 风险很直观。订单 header join item，如果一张订单有 5 行 item，结果就会有 5 行。后续如果还把 sales_order 当唯一 key，就会出错。

Association 把这个风险推迟到消费时。定义一个 _Item association 不会马上把 header 行变成 item 行；但如果后续通过 path expression 取 _Item.Material，或者 OData consumer expand 到 items，to-many 关系就会影响结果结构、行数、分页或 UI 展示。

所以不要说 association “不会放大行数”。更准确的说法是：

  • 定义或暴露 association 本身不是立即展开 target。
  • 一旦用 path expression 或 navigation 消费 to-many target，结果行数或响应结构就可能变化。
  • Cardinality 必须由真实业务关系和测试数据支撑。
7.4.5 Join 不声明 cardinality；association 必须面对 cardinality

Join 的 ON condition 能让数据库组合行，但它不会告诉 reader 这是 1:1、0:1、1:n 还是 n:m。reviewer 必须从表 key、唯一索引、业务规则和测试数据推断。

Association 则把 cardinality 写进模型：

association [0..1] to ZI_Customer as _Customer ...
association [0..*] to ZI_SalesOrderItem as _Item ...

这是一种语义承诺。写 [0..1]，就是告诉消费者“一条 source 记录最多一个 target”；写 [0..*]，就是承认可能多条。这个承诺不会替代数据库约束，但它会影响 syntax warning、reader 预期、OData navigation、Fiori 展示和性能分析。

项目上常见问题是：开发者为了消除 warning 或让代码好看，随手写 [1..1]。这是危险的。正确做法是反过来：先确认 target key、ON condition 和真实数据，再写 cardinality；如果确实一对多，就要让消费者知道它是一对多，而不是假装一对一。

7.4.6 Join 的 null 与 association 的“没有 target”要分清

Left outer join 中，右侧没有匹配行时，右侧列会进入 null 语义。ABAP 处理结果时，常常需要 coalesce( ) 或明确的业务默认值，避免把“没有右侧记录”和“右侧字段初始值”混在一起。

Association 中也会遇到 target 不存在的场景，尤其是 [0..1]。但读者要分清：association 层说的是关系可能没有 target；当它被 path expression 消费时，底层如何表现为 null、默认值或导航结果，要看使用位置、join 类型、SQL/path 表达式和服务协议。不要在模型层写了 [0..1]，就在应用层假设 target 字段永远有值。

7.4.7 Join 不会自然产生 OData navigation；association 才进入导航链路

显式 join 通常把结果做成一个扁平结构：

SalesOrder | CustomerId | CustomerName

这对报表或列表很方便，但它不表达“SalesOrder entity 可以导航到 Customer entity”。OData/Fiori/RAP 想要对象间导航，通常需要 CDS association、projection view 暴露、service definition 暴露和 service binding 激活。换句话说，join 更像“把客户名带到订单列表里”；association 更像“订单对象有一个可导航的客户关系”。

这也是 API 设计中的关键差异：

  • 如果外部 consumer 只需要订单列表中的客户名称，可以用 projection 字段或 join 派生一个扁平字段。
  • 如果外部 consumer 需要从订单导航到客户详情，或者用 $expand 取相关 entity，就要认真设计 association / navigation property。
  • 如果不希望暴露内部关系，projection/service 层不要随手公开 association。
7.4.8 Join 和 association 都不自动解决权限

Join 不是 authorization。直接 join 两张数据库表，不会自动执行某个 CDS DCL，也不会自动等同于业务授权。

Association 也不是 authorization。它只是关系声明。即使 association target 是一个带 DCL 的 CDS entity，也要结合当前消费路径、access control annotation、RAP authorization、service exposure 和运行时场景来验证。安全的写法不是“用了 CDS association 就有权限”，而是明确说明权限在哪里生效：

  • 直接 ABAP SQL table join：需要自己确认 AUTHORITY-CHECK、业务服务层或其他授权机制。
  • 选择带 DCL 的 CDS entity：要确认 access-control check 是否启用、DCL 是否覆盖当前消费。
  • RAP/OData 服务：还要确认 RAP authorization、feature control、service binding 和用户角色。
7.4.9 Clean Core 视角：不要用 join 绕开 released model

在 S/4HANA / ABAP Cloud / Clean Core 语境下，一个常见风险是：为了快，直接 join SAP 标准底表。传统 on-premise 项目中这种写法可能存在很多，但新开发尤其要先问：

  • 是否有 released CDS view / released API / RAP BO？
  • 是否允许直接访问该表？
  • 该表的字段语义、client handling、authorization、extension 和升级兼容性是否清楚？
  • 如果标准模型已经提供 association，为什么还要在应用代码里重写 join？

这不是说 join 不能用，而是说 join 的数据源要合规。Join 是工具，不能绕开平台契约。

7.4.11 Review evidence checklist

选择 join 或 association 后，review 不应只停在“设计上应该这样”。至少要准备能证明结果正确的证据：

EvidenceJoin choice should prove...Association choice should prove...
Sample data result行数、null、重复行、outer join 保留行符合预期。path expression / Follow Association 能找到正确 target。
Generated SQL / tracejoin 条件、filter、排序和性能可解释。association 被消费时实际生成的访问可解释。
Cardinality checkjoin 没有意外放大结果集。[0..1] / [0..*] 与真实数据一致。
Authorization evidencejoin 没绕开 DCL/released model/authorization boundary。source 和 target navigation 都有合适授权考虑。
OData metadata通常不产生 navigation property。exposed association 是否进入 service metadata，是否允许 $expand / navigation。
Consumer contract字段组合只服务当前查询。association alias、target、cardinality 是否已成为服务/API contract。
Regression testABAP Unit / integration evidence 覆盖代表性组合。navigation、projection、service binding preview 或 API test 覆盖消费链路。

一个实用判断是：join 的证据重点是“当前结果集是否正确”；association 的证据重点是“模型关系和后续消费是否正确”。两者都要看数据，但看数据的角度不同。

7.5 日本项目现场说法

常见日语：

  • 結合：join。
  • 内部結合：inner join。
  • 左外部結合：left outer join。
  • 結合条件、ON条件：join condition / ON condition。
  • アソシエーション：association。
  • 公開アソシエーション：exposed association。
  • パス式：path expression。
  • ナビゲーションプロパティ：navigation property。
  • カーディナリティ：cardinality。
  • 件数増加、重複行：row multiplication / duplicate rows。
  • N+1問題：loop 中重复查询导致的 N+1 problem。
  • Released API、Released CDS View：Clean Core / ABAP Cloud review 中常用。

项目说明句：

  • この処理は一回限りの一覧取得なので、明示的なJOINで十分です。
  • この関連は複数サービスで再利用されるため、CDSアソシエーションとしてモデル化してください。
  • ODataのナビゲーションとして公開する必要がある場合は、JOINではなくアソシエーションとprojection/service側の公開を確認してください。
  • このJOINは1対多の関係なので、結果件数が増える前提で後続処理を確認してください。
  • SAP標準テーブルを直接JOINする前に、Released CDS View または Released API の有無を確認してください。

Review 中常见指摘：

  • このJOIN条件は業務キーを正しく表しているか確認してください。
  • アソシエーションのカーディナリティと実データの一意性が一致していません。
  • 外部API契約にしたくない関連は、projection view で公開しないでください。
  • 権限制御はJOINやアソシエーションでは代替できないため、DCL / RAP authorization / AUTHORITY-CHECK を確認してください。

7.7 Example / 学习例子

7.例 1：当前列表只需要客户名称

如果需求是“显示订单列表和客户名称”，并且这个列表就是最终消费形态，一个清楚的 join 可以很合适：

SELECT FROM zso_header AS so
       INNER JOIN zcustomer AS cu
          ON cu~customer_id = so~customer_id
       FIELDS so~sales_order,
              so~customer_id,
              cu~customer_name
       WHERE so~status = @lc_status_open
       INTO TABLE @DATA(lt_rows).

Review 重点是：ON 是否正确、客户是否必须存在、是否会因为 1:n 文本/联系人而放大行数、是否需要 ORDER BY、是否有权限和 released model 问题。

7.例 2：订单对象需要可导航到客户

如果需求是“订单对象在 RAP/OData/Fiori 中可以导航到客户对象”，association 更合适：

define view entity ZI_SalesOrder
  as select from zso_header as Header
  association [0..1] to ZI_Customer as _Customer
    on $projection.CustomerId = _Customer.CustomerId
{
  key Header.sales_order as SalesOrder,
      Header.customer_id as CustomerId,

      _Customer
}

Review 重点变成：ZI_Customer 是否是合适 target、[0..1] 是否真实、是否应该在 projection view 暴露、是否会形成 OData navigation property、target 数据是否有 DCL/authorization 保护。

7.例 3：错误选择会带来不同问题

把该建 association 的关系写成到处 join，会导致关系分散、API 导航缺失、复用困难。把一次性报表强行拆成很多 association，也可能让读者看不出当前结果集到底从哪里来。

一个判断练习：

  1. 当前需求是不是“这次查询必须立刻拿 target 字段”？
  2. 这个关系以后是否会被多个 view / service / API 复用？
  3. 外部 consumer 是否需要 navigation / $expand？
  4. 关系 cardinality 是否足够稳定，可以写进模型？
  5. 数据源是否合规，尤其是否有 released CDS/API？

7.8 Counterexample / Failure mode

常见失败模式如下。

  • 所有关联都写成 join：导致 CDS/RAP/OData 模型没有导航关系，后续 Fiori/API 只能拿扁平字段。
  • 所有 join 都改成 association：简单报表变成过度建模，当前结果集来源反而不清楚。
  • 用 join 绕开 CDS model：标准对象已有 released CDS/API，却直接 join 底表，埋下 Clean Core 和升级风险。
  • 误以为 association 不会执行 join：定义时不执行，不代表消费时免费；path expression 或 OData navigation 仍然会触发访问。
  • 误以为 join 更快或 association 更快：性能取决于数据量、条件、索引、cardinality、生成 SQL、DCL、缓存和数据库优化器。
  • 误以为 join/association 处理权限：两者都不是授权机制，要看 DCL、RAP authorization、AUTHORITY-CHECK 或服务层。
  • 忽略 outer join 的 null：right side 没匹配时，字段语义要明确，必要时使用 coalesce( ) 或业务默认值。
  • 忽略 to-many：join 或 association path 都可能导致行数增加，影响分页、聚合、后续 READ TABLE 和 UI 展示。

修正动作：

  • 先写一句业务意图：这次是“取字段”，还是“建关系”？
  • 再画出 source -> target 的 cardinality。
  • 然后决定当前层：ABAP SQL、CDS interface view、projection view、service definition、OData。
  • 最后用 ADT Data Preview、Show SQL Create Statement、SQL trace、OData metadata / $expand 测试验证。

7.9 思考问题

  • 当前代码里的 join 是否只是为了补一个文本字段？这个文本关系是否应该进入 CDS model？
  • 当前 association 是否真的需要暴露到 projection/service？
  • 这段逻辑需要扁平结果，还是需要对象导航？
  • 如果关系是一对多，当前结果集或 OData response 会如何变化？
  • 如果右侧 target 不存在，业务上要显示空、默认值，还是不返回该行？
  • 权限在哪里生效？是否误以为 join/association 自动处理？
  • Clean Core 场景下，是否存在 released CDS view / released API 可以替代直接 join？
