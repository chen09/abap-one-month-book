32. Polymorphism 多态

  • Chapter 02
  • Section 32
  • self-contained study note

32.1 一句话结论

Polymorphism 多态是 OO ABAP 中“同一个调用形式，根据运行时对象不同而执行不同实现”的机制。调用方可以持有 superclass reference 或 interface reference，只按父类或接口 contract 调用 method；实际执行哪个 class 的 implementation，由 runtime object 决定。

初学者最重要的判断是：多态不是“写很多 IF 判断 class 名称”，而是让调用方依赖稳定 contract，把具体实现推迟到运行时对象。reference type 决定调用方能看见哪些 components；runtime object 决定真正执行哪个 method body。

32.2 到底学什么

前两节分别讲了 Interface 接口定义 和 Inheritance 继承。本节把它们连起来：interface 和 inheritance 都能形成多态，但项目里最常用、最安全的业务设计往往是 interface-based polymorphism。

本节要掌握七个边界：

  • superclass reference 可以指向 subclass object。
  • interface reference 可以指向任何 implementing class object。
  • static reference type 控制可见范围：interface reference 只能访问 interface components；superclass reference 只能访问 superclass public components。
  • dynamic type / runtime object 控制实际执行：redefined instance method 会按实际对象动态分派 dynamic dispatch。
  • 在合法对象引用中，static type 一定比 dynamic type 更泛化或相同；dynamic type 一定比 static type 更具体或相同。
  • static method 不参与对象多态；不要把 CLASS-METHODS 当成可替换业务行为。
  • upcast 通常安全，普通赋值 = 就可以完成；downcast 需要 CAST，通常先用 IS INSTANCE OF 防止 CX_SY_MOVE_CAST_ERROR。
  • 多态的项目价值在于减少条件分支、支持 test double、让实现可替换，而不是炫技。

最小图像：

TYPE REF TO lif_notifier  ---->  lcl_mail_notifier object
                            \
                             ->  lcl_log_notifier object
                            \
                             ->  lcl_test_notifier object

调用方只知道 lif_notifier->send( )。至于是发邮件、写日志、还是测试替身记录消息，由实际对象决定。

32.3 为什么重要

没有多态时，业务代码经常长这样：

IF iv_channel = 'MAIL'.
  " send mail
ELSEIF iv_channel = 'LOG'.
  " write log
ELSEIF iv_channel = 'TEST'.
  " collect test message
ENDIF.

这种代码的问题不是 IF 本身，而是调用方开始知道太多实现细节。新增一个 channel，就要改调用方；测试时也要绕过真实副作用。

有多态后，调用方只依赖 contract：

DATA lo_notifier TYPE REF TO lif_notifier.
lo_notifier = NEW lcl_mail_notifier( ).

lo_notifier->send( iv_text = 'Order released' ).

以后换成 lcl_log_notifier 或 lcl_test_notifier，调用方仍然只调用 send。这就是 ABAP Unit、constructor injection、Clean Core dependency boundary 里反复强调 interface 的原因：多态让业务代码从“判断具体实现”转向“依赖抽象 contract”。

对日本 SAP 项目来说，多态还影响 review 和测试策略。review 时常见问题不是“你有没有用 interface”，而是“调用方是否仍然直接 NEW zcl_concrete，导致差し替えできない”。多态用得好，单体测试、障害切り分け、影響調査都会更轻。

32.4 Key mechanism / 核心机制

32.4.1 reference type controls visible contract

先看 interface polymorphism：

INTERFACE lif_notifier.
  METHODS send
    IMPORTING iv_text TYPE string.
ENDINTERFACE.

CLASS lcl_mail_notifier DEFINITION FINAL.
  PUBLIC SECTION.
    INTERFACES lif_notifier.
ENDCLASS.

CLASS lcl_mail_notifier IMPLEMENTATION.
  METHOD lif_notifier~send.
    " production mail sending would be here
  ENDMETHOD.
ENDCLASS.

调用方变量写成 interface reference：

DATA lo_notifier TYPE REF TO lif_notifier.
lo_notifier = NEW lcl_mail_notifier( ).

lo_notifier->send( iv_text = 'Released' ).

这里 lo_notifier 的静态类型是 lif_notifier，所以调用方只能看见 lif_notifier 上声明的 components。即使 lcl_mail_notifier 有其他 public method，也不能通过 lo_notifier->... 直接调用。这是多态的保护：调用方被限制在 contract 里。

32.4.2 runtime object controls implementation

再看两个实现：

CLASS lcl_log_notifier DEFINITION FINAL.
  PUBLIC SECTION.
    INTERFACES lif_notifier.
ENDCLASS.

CLASS lcl_log_notifier IMPLEMENTATION.
  METHOD lif_notifier~send.
    " write application log would be here
  ENDMETHOD.
ENDCLASS.

同一个变量、同一个 call，可以执行不同 class 的实现：

DATA lo_notifier TYPE REF TO lif_notifier.

lo_notifier = NEW lcl_mail_notifier( ).
lo_notifier->send( iv_text = 'A' ).

lo_notifier = NEW lcl_log_notifier( ).
lo_notifier->send( iv_text = 'B' ).

第一段调用 mail implementation，第二段调用 log implementation。调用语法没有变，runtime object 变了，执行体就变了。

32.4.3 inheritance polymorphism uses redefinition

继承也能形成多态。上一节 Inheritance 继承 的例子里，superclass reference 指向 subclass object：

DATA lo_pricing TYPE REF TO lcl_base_pricing.
lo_pricing = NEW lcl_discount_pricing( ).

DATA(lv_net) = lo_pricing->calculate( iv_amount = '100.00' ).

lo_pricing 的 static type 是 lcl_base_pricing，但 dynamic type / runtime object 是 lcl_discount_pricing。static type 决定调用方能写出哪些 method call；dynamic type 决定 redefined instance method 最终执行哪个 implementation。如果 calculate 在 subclass 里 REDEFINITION，调用会动态分派到 subclass implementation。这就是 dynamic dispatch。

项目判断上要保守：framework extension point 常用 inheritance polymorphism；普通业务可替换实现更常用 interface polymorphism。不要为了多态而强行建父类。

32.4.4 table of interface references

多态的价值在集合处理中很明显。一个 internal table 可以保存同一 interface 的不同实现：

DATA lt_notifiers TYPE STANDARD TABLE OF REF TO lif_notifier
                  WITH EMPTY KEY.
DATA lo_notifier  TYPE REF TO lif_notifier.

lo_notifier = NEW lcl_mail_notifier( ).
APPEND lo_notifier TO lt_notifiers.

lo_notifier = NEW lcl_log_notifier( ).
APPEND lo_notifier TO lt_notifiers.

LOOP AT lt_notifiers INTO lo_notifier.
  lo_notifier->send( iv_text = 'Batch finished' ).
ENDLOOP.

loop 里没有 CASE class_name，也没有 IF channel = ...。每一行都是 lif_notifier，但每个 runtime object 可以不同。

32.4.5 upcast and downcast

把 concrete object reference 放进 superclass/interface reference，通常是 upcast：

DATA lo_notifier TYPE REF TO lif_notifier.
lo_notifier = NEW lcl_mail_notifier( ).

upcast 是多态的正常入口，赋值运算符 = 就能完成，不需要特殊 cast operator。反过来，如果想从 interface reference 拿回具体 class reference，就是 downcast：

IF lo_notifier IS INSTANCE OF lcl_mail_notifier.
  DATA(lo_mail) = CAST lcl_mail_notifier( lo_notifier ).
ENDIF.

downcast 不是禁止使用，但它是设计味道信号。如果调用方经常需要 downcast 才能继续工作，说明 interface contract 可能不完整，或者调用方不应该依赖这个抽象。

downcast 还有运行时风险：如果实际对象不是目标 class，CAST 会失败并触发 CX_SY_MOVE_CAST_ERROR。所以 beginner code 应先写 IS INSTANCE OF guard，再 CAST。旧代码里可能看到 ?= downcast operator；新写法优先用 CAST，ABAP Cloud / modern ABAP 场景下不要把 ?= 当成新代码模板。

32.4.6 polymorphism is not method overloading

Java 背景学习者容易把 polymorphism 和 method overloading 混在一起。本节讲的不是“同名 method 用不同参数列表区分”。ABAP Objects 不支持 Java-style method overloading：同一个 class 里不能声明两个同名 method 只靠参数列表不同来区分；method name 必须唯一，否则是 syntax error。这里的多态重点是 static reference type 与 dynamic type：同一个 contract call，根据实际对象执行不同 implementation。

如果确实需要根据对象类型做少量类型分支，现代 ABAP 还可以看到 CASE TYPE OF ... WHEN TYPE ... 这类 type case 写法；但初学阶段不要把它当成多态的主要模式。多态的目标是减少到处散落的类型判断，而不是把 IF ... IS INSTANCE OF ... 换成另一种更漂亮的类型分支。

32.4.7 ABAP Unit and test doubles

多态是 ABAP Unit 可测试性的基础。生产代码依赖 interface：

CLASS lcl_order_service DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS constructor
      IMPORTING io_notifier TYPE REF TO lif_notifier.

    METHODS release.

  PRIVATE SECTION.
    DATA mo_notifier TYPE REF TO lif_notifier.
ENDCLASS.

CLASS lcl_order_service IMPLEMENTATION.
  METHOD constructor.
    mo_notifier = io_notifier.
  ENDMETHOD.

  METHOD release.
    mo_notifier->send( iv_text = 'Order released' ).
  ENDMETHOD.
ENDCLASS.

测试时传入另一个 implementing class：

CLASS lcl_notifier_double DEFINITION FINAL.
  PUBLIC SECTION.
    INTERFACES lif_notifier.
    METHODS was_called RETURNING VALUE(rv_called) TYPE abap_bool.

  PRIVATE SECTION.
    DATA mv_called TYPE abap_bool.
ENDCLASS.

CLASS lcl_notifier_double IMPLEMENTATION.
  METHOD lif_notifier~send.
    mv_called = abap_true.
  ENDMETHOD.

  METHOD was_called.
    rv_called = mv_called.
  ENDMETHOD.
ENDCLASS.

这就是“同一个 contract，不同 runtime object”的可测试价值。生产实现可以发通知，测试实现只记录是否被调用。

32.4.8 polymorphism design checklist

多态设计可以用下面的检查表快速判断。重点不是“有没有 interface 或 superclass”，而是调用方是否真的只依赖稳定 contract，具体实现是否可以在不改调用方的情况下替换。

QuestionGood polymorphism signRisk signal
What contract does caller need?调用方只需要 send、calculate、validate 这类小而清楚的能力。interface 包含太多 method，调用方只用其中一两个。
Can implementation be replaced?生产实现、测试替身、不同渠道实现都能通过同一 reference 调用。调用方马上 CAST 回具体 class 才能继续工作。
Where is object creation?具体 class 在 factory、constructor injection、framework runtime 中决定。业务流程内部到处 NEW lcl_specific_xxx( )。
Is dynamic dispatch actually used?instance method 通过 interface/superclass reference 调用。使用 CLASS-METHODS，却期待它像对象多态一样可替换。
Is type checking rare?少量 type check 只用于边界、兼容或诊断。主流程大量 IS INSTANCE OF / CASE TYPE OF 分支。
Are tests simpler?ABAP Unit 可以注入 test double，验证 contract 行为。测试必须连接外部系统或依赖具体实现副作用。

在项目代码里，一个很实用的判断句是：如果新增一个实现 class 时，主业务流程不需要改，只需要在装配处选择新对象，这通常是健康的多态；如果每新增一个 class 都要去主流程里加 IF type = ...，那还只是“手写分派”，没有真正把变化隔离到 contract 后面。

32.5 日本项目现场说法

  • 常见日语：多態性、ポリモーフィズム、動的ディスパッチ、実行時型、参照型、インターフェース参照、具象クラス、実装クラス、差し替え可能性、アップキャスト、ダウンキャスト。
  • 设计说明句：呼び出し側はインターフェース参照に依存し、実行時に渡された実装クラスに応じて処理が切り替わります。
  • Review 指摘句：呼び出し側が具象クラスを直接生成しているため、多態性を活かせず、テストダブルに差し替えにくくなっています。
  • Downcast 指摘句：ダウンキャストが多い場合、インターフェース契約が不足している可能性があります。
  • 使用场景：constructor injection 设计、ABAP Unit、test double、framework hook、RAP handler/saver 阅读、Clean Core dependency review。

日本项目里常说 差し替え可能。这不是抽象口号，而是能不能把 production class 换成 test double、mock adapter、另一种 implementation，而调用方不用改。多态就是支撑 差し替え可能性 的 OO 机制之一。

32.6 English Terms

  • Official / search terms: polymorphism, dynamic dispatch, runtime type, static reference type, interface reference, superclass reference
  • Related syntax: TYPE REF TO lif_xxx, TYPE REF TO lcl_super, IS INSTANCE OF, CAST #( ... ), CAST lcl_xxx( ... ), METHODS ... REDEFINITION, CASE TYPE OF, legacy ?=
  • Design terms: contract, implementation class, test double, dependency inversion, constructor injection, upcast, downcast, replaceability
  • Runtime terms: static type, dynamic type, CX_SY_MOVE_CAST_ERROR, method overloading is not supported
  • 读官方资料时注意：SAP 文档可能从 object reference、interface reference、inheritance、method redefinition 等角度讲多态，不一定只用 polymorphism 这个标题。

32.7 Example / 学习例子

32.7.1 例 1：通知渠道的 interface polymorphism
INTERFACE lif_notifier.
  METHODS send
    IMPORTING iv_text TYPE string.
ENDINTERFACE.

CLASS lcl_mail_notifier DEFINITION FINAL.
  PUBLIC SECTION.
    INTERFACES lif_notifier.
ENDCLASS.

CLASS lcl_log_notifier DEFINITION FINAL.
  PUBLIC SECTION.
    INTERFACES lif_notifier.
ENDCLASS.

调用方只依赖 lif_notifier：

CLASS lcl_release_service DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS constructor
      IMPORTING io_notifier TYPE REF TO lif_notifier.

    METHODS release.

  PRIVATE SECTION.
    DATA mo_notifier TYPE REF TO lif_notifier.
ENDCLASS.

这样 release service 不知道“通知是邮件还是日志”，只知道“有一个对象能 send”。这就是面向 interface 的多态。

32.7.2 例 2：错误的类型判断分支
IF lo_notifier IS INSTANCE OF lcl_mail_notifier.
  " special mail handling
ELSEIF lo_notifier IS INSTANCE OF lcl_log_notifier.
  " special log handling
ENDIF.

如果这种代码到处出现，多态就被破坏了。调用方又开始知道所有 concrete class。

修正方向：把共同需要的行为放进 lif_notifier contract；或者把差异封装在各自 implementation 里。

32.7.3 例 3：framework polymorphism

RAP、BAdI、UI framework、conversion framework 等 SAP 场景经常通过父类或 interface 定义 hook。项目代码只实现规定 method，runtime 在合适时机调用。

读这类代码时，不要只看“某个 method 被谁调用”。要问：

  • 这个对象是通过 superclass reference 还是 interface reference 被框架持有？
  • 哪个 method 是 framework contract？
  • 当前 class 是 production implementation、test double，还是 framework subclass？
  • 这个 API / base class / interface 是否 released / allowed？

32.8 Counterexample / Failure mode

32.8.1 误区 1：直接 new 具体 class

service 里直接 NEW lcl_mail_notifier( )，调用方就被 concrete implementation 绑住。

修正动作：把 dependency 通过 constructor 传入，类型写成 interface reference。

32.8.2 误区 2：interface reference 后马上 downcast

拿到 TYPE REF TO lif_xxx 后立刻 CAST lcl_concrete( ... )，说明调用方并没有真正依赖 contract。

修正动作：检查 interface 是否缺 method；或者让具体行为留在 implementation 内部。

32.8.3 误区 3：用 static method 期待多态

CLASS-METHODS 绑定在 class 上，不是对象实例的动态分派点。

修正动作：可替换业务行为用 instance method + interface reference。

32.8.4 误区 4：多态 contract 太大

interface 上塞太多 method，所有 implementation 都要被迫实现一堆不相关能力。

修正动作：按 capability 拆小 interface，例如 reader、writer、notifier、calculator、validator。

32.8.5 误区 5：把多态当成不用读实现

多态降低调用方耦合，但不代表实现可以随便写。每个 implementation 仍要满足同一 contract 的语义。

修正动作：在 interface 注释、ABAP Unit、review checklist 中明确 contract；不同实现必须通过同一组行为测试。

32.9 思考问题

  • 当前调用方依赖的是 interface/superclass contract，还是 concrete class？
  • reference type 让调用方能看见哪些 components？哪些被刻意隐藏？
  • runtime object 可能有哪些 implementation？它们是否都满足同一 contract？
  • 如果新增一个 implementation，调用方是否需要改 IF/CASE？
  • 是否出现频繁 downcast？这说明 contract 缺失还是抽象用错？
  • 这里应使用 interface polymorphism，还是 inheritance polymorphism？
  • ABAP Unit 能否用 test double 替换 production implementation？
  • 日本项目 review 时，应说 多態性、差し替え可能性、还是 ダウンキャストが多い？
