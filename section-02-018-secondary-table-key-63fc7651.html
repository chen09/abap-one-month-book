18. Secondary table key 二级键

  • Chapter 02
  • Section 18
  • self-contained study note

18.1 一句话结论

Secondary table key 二级键，是 internal table 内部表上的额外命名访问路径：在 primary table key 之外，再为同一张内存表声明 sorted key 或 hashed key，让代码可以按另一组字段高效 READ TABLE、table expression 或 LOOP AT ... USING KEY。它像“内存表索引”，但不是数据库索引；它能改善访问性能，也会带来内存和维护成本。

18.2 到底学什么

前面三节讲的是 primary table category：

Standard table 标准表
Sorted table 排序表
Hashed table 哈希表

它们决定 internal table 的 primary key 和主要管理方式。但项目里常遇到一张表需要多个访问路径：

主要用途：按顺序输出 ALV
额外用途：按 material 快速读取
额外用途：按 customer + date 范围处理

如果只用 standard table，每次 key lookup 可能线性搜索；如果改成 hashed table，又失去顺序/范围处理的自然语义；如果改成 sorted table，可能只优化了其中一个 key。secondary table key 就是为这种“一个表，多种访问路径”准备的。

示意声明：

TYPES ty_t_item TYPE STANDARD TABLE OF ty_item
  WITH EMPTY KEY
  WITH NON-UNIQUE SORTED KEY by_order COMPONENTS order_id item_no
  WITH UNIQUE HASHED KEY by_guid COMPONENTS item_guid.

这表示：

primary key
  -> empty key，主用途是顺序容器

secondary sorted key by_order
  -> 按 order_id item_no 排序/分组读取

secondary hashed key by_guid
  -> 按 item_guid 唯一完整 key 快速读取

secondary key 的核心不是“写了就一定更快”，而是：当同一张 internal table 的主要表类型和实际访问方式不完全一致时，用命名 key 明确另一个访问路径。

18.3 为什么重要

在 ABAP 项目里，性能优化不只发生在 SQL。很多程序已经把数据从数据库读到内存，然后在 internal table 中反复查找、过滤、合并、对账。此时数据库 SQL 已经结束，性能瓶颈可能在 ABAP 层。

常见场景：

LOOP AT lt_orders INTO DATA(ls_order).
  READ TABLE lt_items INTO DATA(ls_item)
    WITH KEY order_id = ls_order-order_id.
ENDLOOP.

如果 lt_items 是 standard table，这可能是线性搜索。如果不能把 lt_items 的 primary category 改成 sorted table，因为它还要按原始顺序输出，就可以考虑给 lt_items 加 secondary sorted key：

WITH NON-UNIQUE SORTED KEY by_order COMPONENTS order_id

然后读取时显式使用 key：

READ TABLE lt_items INTO DATA(ls_item)
  WITH TABLE KEY by_order COMPONENTS order_id = ls_order-order_id.

这样代码意图更清楚：这不是随便 READ TABLE，而是作者明确希望使用 by_order 这个访问路径。

对 Java/Python 背景学习者，可以把 secondary key 类比成“同一组对象上的额外索引”。一个 list 本身按插入顺序保存，但你额外维护一个 Map<guid, item> 或 TreeMap<order_id, items> 来支持快速访问。ABAP 把这个能力做进 internal table 类型声明中。

18.4 Key mechanism / 核心机制

18.4.1 secondary key 是命名 key

secondary table key 必须有名字。常见命名如：

by_order
by_material
by_guid
sk_order
hk_guid

这个名字不是注释，而是后续读取时要用的 key name：

READ TABLE lt_items INTO DATA(ls_item)
  WITH TABLE KEY by_guid COMPONENTS item_guid = lv_guid.

或在 loop 中使用：

LOOP AT lt_items INTO DATA(ls_item)
  USING KEY by_order
  WHERE order_id = lv_order_id.
ENDLOOP.

table expression 也可以选择 key：

DATA(ls_item) = lt_items[
  KEY by_guid
  COMPONENTS item_guid = lv_guid
].

不同 ABAP release 的语法细节可能略有差异，实际写法以系统中的 ABAP Keyword Documentation 和 ADT syntax check 为准；本节重点是理解“显式指定 key name”。

18.4.2 secondary sorted key 与 secondary hashed key

secondary keys 主要有两类：

secondary sorted key
  -> 可以 UNIQUE 或 NON-UNIQUE
  -> 适合有序、范围、前缀、分组读取

secondary hashed key
  -> 必须 UNIQUE
  -> 适合完整 key 精确读取

示例：

DATA lt_items TYPE STANDARD TABLE OF ty_item
  WITH EMPTY KEY
  WITH NON-UNIQUE SORTED KEY by_order COMPONENTS order_id item_no
  WITH UNIQUE HASHED KEY by_guid COMPONENTS item_guid.

by_order 适合：

LOOP AT lt_items INTO DATA(ls_item)
  USING KEY by_order
  WHERE order_id = lv_order_id.
ENDLOOP.

by_guid 适合：

READ TABLE lt_items INTO DATA(ls_item)
  WITH TABLE KEY by_guid COMPONENTS item_guid = lv_guid.
18.4.3 secondary key 不改变 primary table category

如果 primary table 是 standard table，加了 secondary key 后，它仍然是 standard table。primary 的顺序、empty key、append 行为并不会消失。secondary key 是额外访问路径，不是把表“变成” sorted table 或 hashed table。

这点很重要，因为项目上常见需求是：

保留原始输入顺序
  -> primary: standard table WITH EMPTY KEY

按 order_id 查明细
  -> secondary sorted key by_order

按 guid 精确找一行
  -> secondary hashed key by_guid

这比创建三张独立 internal table 更集中，也比在循环里反复线性搜索更可解释。

18.4.4 使用 secondary key 必须显式

声明 secondary key 不代表所有读取都会自动使用它。代码最好显式写出：

WITH TABLE KEY by_order COMPONENTS ...
USING KEY by_order
KEY by_guid COMPONENTS ...

如果只写：

READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY order_id = lv_order_id.

ABAP 是否选择某个 key，取决于语法、key 条件和优化规则；对初学者和 code review 来说，显式 key name 更清楚。

18.4.5 维护成本与 lazy update

secondary key 不是免费优化。每个 secondary sorted key 都要维护额外 sorted index；secondary hashed key 要维护 hash administration。插入、删除、修改行时，这些 key 也可能需要更新。

SAP Help 特别提醒：secondary key 的更新可能是 delayed / lazy 的，某些更新成本可能在后续 read access 时才发生。也就是说，给一张频繁变化的表加很多 secondary key，可能把成本推迟到你以为要优化的读取点上。

因此，使用 secondary key 之前要问：

  • 表是否足够大？
  • 按 secondary key 读取是否频繁？
  • 表是否主要一次填充、多次读取？
  • 是否频繁修改 key 字段？
  • 是否真的需要多个访问路径？

如果表只有几十行，或者只读一次、loop 一次，secondary key 可能让代码更复杂而不是更快。

18.4.6 unique secondary key 的重复风险

unique secondary key 会带来约束：

WITH UNIQUE HASHED KEY by_guid COMPONENTS item_guid

如果插入或修改导致 item_guid 重复，secondary key 的 uniqueness 会被破坏，可能出现可捕获异常或运行时错误。这个错误不是“secondary key 小问题”，而是业务数据或设计假设冲突。

如果业务允许重复，就不要声明 unique secondary key；可以改用 non-unique sorted key，并在读取时处理多行结果。

18.7 Example / 学习例子

18.7.1 例 1：standard table + secondary sorted key
TYPES: BEGIN OF ty_item,
         order_id  TYPE i,
         item_no   TYPE i,
         item_guid TYPE sysuuid_x16,
         text      TYPE string,
       END OF ty_item.

DATA lt_items TYPE STANDARD TABLE OF ty_item
  WITH EMPTY KEY
  WITH NON-UNIQUE SORTED KEY by_order COMPONENTS order_id item_no.

LOOP AT lt_items INTO DATA(ls_item)
  USING KEY by_order
  WHERE order_id = lv_order_id.
  " process all items of this order
ENDLOOP.

primary table 仍然是 standard table，用来保留输入顺序；by_order 是额外 sorted access path，用来按 order 分组处理。

18.7.2 例 2：secondary hashed key 做 GUID lookup
DATA lt_items TYPE STANDARD TABLE OF ty_item
  WITH EMPTY KEY
  WITH UNIQUE HASHED KEY by_guid COMPONENTS item_guid.

READ TABLE lt_items INTO DATA(ls_item)
  WITH TABLE KEY by_guid COMPONENTS item_guid = lv_guid.

如果 item_guid 业务上唯一，这种 secondary hashed key 很适合快速定位一行。它比把 primary table 改成 hashed table 更灵活，因为 primary 顺序仍可以保留。

18.7.3 例 3：同时有两个 secondary keys
DATA lt_items TYPE STANDARD TABLE OF ty_item
  WITH EMPTY KEY
  WITH NON-UNIQUE SORTED KEY by_order COMPONENTS order_id item_no
  WITH UNIQUE HASHED KEY by_guid COMPONENTS item_guid.

同一张表可以支持两种访问方式：按订单读取多行，或按 GUID 读取单行。不要滥用很多 key；每个 key 都要有真实读取场景。

18.7.4 例 4：table expression 使用 named key
DATA(ls_item) = lt_items[
  KEY by_guid
  COMPONENTS item_guid = lv_guid
].

table expression 找不到行时可能抛出异常。实际项目可以配合 line_exists、OPTIONAL 或异常处理。

18.7.5 例 5：unique secondary key 违反示意
DATA lt_items TYPE STANDARD TABLE OF ty_item
  WITH EMPTY KEY
  WITH UNIQUE HASHED KEY by_guid COMPONENTS item_guid.

" 如果两个输入行具有同一个 item_guid，by_guid 的唯一性就会被破坏。

这不是性能问题，而是业务 key 假设失败。要么数据应该报错，要么 key 不应该声明为 unique hashed key。

18.8 Counterexample / Failure mode

18.8.1 误区 1：声明了 secondary key 但读取不用它
READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY order_id = lv_order_id.

如果代码没有显式 WITH TABLE KEY by_order COMPONENTS ...，reviewer 很难确定是否真的使用了你设计的 secondary key。教程中建议初学者先写清楚 key name。

18.8.2 误区 2：给小表加很多 key

一张几十行的小表，如果只 loop 一次，加 secondary key 可能没有收益。它增加类型复杂度、内存、维护成本和认知负担。

18.8.3 误区 3：频繁修改 key 字段

如果一张表在循环中不断插入、删除、修改 secondary key 字段，secondary key 的维护成本会变高。此时要重新评估数据结构和处理流程。

18.8.4 误区 4：unique secondary key 与业务重复冲突

如果业务上同一个 order_id 有多行，不能把 order_id 声明成 unique hashed secondary key。应该用 non-unique sorted key，或重新定义业务唯一 key。

18.8.5 误区 5：把 secondary key 当数据库索引

secondary table key 只存在于 ABAP internal table 的内存结构中。它不会优化数据库 SELECT。数据库读取慢，要看 SQL、数据库索引、CDS、where 条件和执行计划；内存表查找慢，才考虑 internal table key。

18.9 思考问题

  • 这张 internal table 的 primary category 为什么不能直接改成 sorted/hashed？
  • secondary key 对应的真实读取代码在哪里？
  • 读取时是否显式写了 WITH TABLE KEY keyname COMPONENTS ... 或 USING KEY keyname？
  • 这个 secondary key 是 sorted 还是 hashed？为什么？
  • key 是否需要 unique？业务上真的唯一吗？
  • 表是否一次填充、多次读取？还是频繁修改？
  • secondary key 维护成本是否比线性搜索成本更低？
  • 这个性能问题发生在数据库层，还是 ABAP internal table 层？
  • 用日语 review 时，能否说明 二次キーの保守コスト 和 アクセスパス？
