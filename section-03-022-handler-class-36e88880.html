22. Handler class 处理器类

  • Chapter 03
  • Section 22
  • self-contained study note

22.1 一句话结论

Handler class 处理器类 是 RAP Behavior pool 行为池 中承载 BO 行为逻辑的 local class，通常继承 CL_ABAP_BEHAVIOR_HANDLER，负责 action、determination、validation、authorization、feature control，以及在 unmanaged 场景中实现标准 operation；它不负责 save sequence 中的最终落库，那是 Saver class 保存器类 的边界。

一句话记忆：handler class 处理 BO 行为规则和 interaction/check-before-save 逻辑；saver class 处理保存序列中真正写库或附加保存的逻辑。

22.2 概念说明

在 RAP 中，behavior definition 只是声明 BO 能做什么，例如 create、update、delete、action、validation、determination、authorization。真正的 ABAP 实现放在 behavior pool 中。为了分清 interaction phase 和 save sequence，SAP Learning 明确说明：local handler classes 放 interaction phase 代码，local saver class 放 save sequence 代码。

Handler class 是 behavior pool 内部的 local class，不是一个普通全局 class。ABAP Keyword Documentation 说明，handler class 继承 CL_ABAP_BEHAVIOR_HANDLER；一个 behavior pool 可以为了 modularization 定义多个 handler classes。对于 composite BO，通常每个 entity 可以有自己的 handler class；默认命名常见为 lhc_<entity-or-alias>。

例如 root entity alias 为 Travel 时，ADT quick fix 可能生成 lhc_travel。如果有 child entity Booking，可能再有 lhc_booking。这些 handler class 都在同一个 global behavior pool class 的 local implementation 区域里工作，而不是各自独立的 repository object。

初学者要注意：handler class 不是“所有 RAP 代码的地方”。它适合写 interaction phase 逻辑，例如检查当前用户是否允许执行 action、根据输入字段派生默认值、为 UI 控制字段是否可编辑、处理 action result、在 unmanaged BO 中把 create/update/delete/read 请求放入 application buffer。它不适合直接 COMMIT WORK、随意写数据库、发送不可回滚外部请求，或者承担 save_modified / save() 的职责。

22.3 为什么重要

Handler class 是读懂 RAP 代码的入口。很多初学者打开 behavior pool 后，会看到 global class 是 ABSTRACT FINAL FOR BEHAVIOR OF ...，真正逻辑却藏在 local class 里。如果不知道 handler class 和 saver class 的分工，就会误以为 RAP 代码“找不到实现”，或者把 save sequence 的逻辑写到 action handler 里。

在项目交付中，handler class 直接影响 review 和测试。Action 是否返回正确 result，validation 是否给出行级消息，authorization 是否只依赖前端控制，feature control 是否和业务状态一致，unmanaged create 是否只写 buffer 而不是直接写表，这些都要从 handler class 看起。

对 Java/Python 背景学习者，可以把 handler class 类比成 domain service / application service 中处理命令和业务规则的部分，但 RAP 有更严格的生命周期：handler 不拥有 transaction commit；handler 方法签名由 BDEF 派生；输入输出结构如 keys、entities、failed、reported、mapped、result 都受 RAP runtime 契约约束。

22.4 边界

第一条边界：handler class 主要服务 interaction phase 和保存前检查，不属于最终持久化的 saver 边界。它可以响应 create/update/delete/read/action 等请求，也可以执行 validation、determination、authorization、feature control。注意 FOR VALIDATE ON SAVE 和 FOR DETERMINE ON SAVE 仍是 handler methods，但它们在 save sequence 的 check-before-save 相关阶段被触发；最终持久化仍属于 managed runtime 或 saver class。

第二条边界：managed 与 unmanaged 中 handler 责任不同。在 Managed RAP 管理式实现 中，标准 create/update/delete/read 通常由 RAP runtime 处理，handler class 主要实现 non-standard operations、validation、determination、action、authorization、feature control 等。在 Unmanaged RAP 非管理式实现 中，handler class 还要实现标准 operations，把请求写入项目自有 application buffer，并返回 failed / reported / mapped。

第三条边界：handler class 不等于 UI controller。它不直接操纵 Fiori 控件，也不决定 OData metadata 的全部外观；它只实现 BO 行为。UI button 是否出现可能来自 annotation、projection behavior、feature control 和 authorization 的共同结果。

第四条边界：handler class 不应绕过 RAP transaction。ABAP Keyword Documentation 和 SAP Learning 都反复强调 RAP transaction boundary。COMMIT WORK、ROLLBACK WORK、直接 DB write、update task、不可回滚副作用，都需要非常谨慎。尤其在 interaction phase，数据可能只是 transactional buffer 中的暂态内容，直接写库会破坏一致性。

22.5 核心机制

第一层机制是 class 继承。Handler class 通常写成：

CLASS lhc_travel DEFINITION INHERITING FROM cl_abap_behavior_handler.
  PRIVATE SECTION.
    METHODS validate_customer FOR VALIDATE ON SAVE
      IMPORTING keys FOR Travel~validateCustomer.
ENDCLASS.

CL_ABAP_BEHAVIOR_HANDLER 给 RAP runtime 一个统一的调用契约。开发者不直接 new 这个 handler，也不从普通程序调用它；runtime 根据 EML / OData / UI 请求和 BDEF 声明，在合适时点调用对应方法。

第二层机制是 handler method 的 METHODS ... FOR ... 语法。ABAP Keyword Documentation 对 RAP handler methods 有专门语法：方法必须定义并实现在 ABAP behavior pool 的 handler class 中，方法名可自选，但 FOR 后面的 handler method type 决定它服务哪个 BDEF 行为。常见类型包括 FOR MODIFY、FOR READ、FOR LOCK、FOR VALIDATE ON SAVE、FOR DETERMINE ON MODIFY、FOR DETERMINE ON SAVE、FOR INSTANCE AUTHORIZATION、FOR GLOBAL AUTHORIZATION、FOR INSTANCE FEATURES、FOR GLOBAL FEATURES、action handler 等。

第三层机制是方法签名由 BDEF 派生。Action 方法可能有 keys 和 result；validation 通常接收 keys，并通过 failed / reported 返回失败实例和消息；unmanaged create/update 接收 entities，read/delete/lock 多以 keys 标识实例；authorization 和 feature control 返回对应授权或可用性结果。不要把这些结构当成普通自定义参数，它们是 RAP 契约的一部分。

第四层机制是 handler 与 saver 的分工。Handler class 做 interaction phase 的业务判断和 buffer 操作；saver class 做 save sequence 的最终保存、unmanaged save、additional save 或 late-save 事件。上一节 Additional Save 附加保存 中的 save_modified 就不属于 handler class。

第五层机制是多 entity handler。Composite BO 中 root 和 child 可以有不同 handler class。SAP Learning 指出，local handler class 默认命名前缀通常是 lhc_，后接 entity 名或 alias；saver class 则通常是 lsc_，且一个 BO 只有一个 saver class。这个命名差异非常适合初学者在 ADT 中定位代码。

第六层机制是 local EML。Handler 方法内部可以使用 EML 的 local variants，例如 READ ENTITIES ... IN LOCAL MODE 或 MODIFY ENTITIES ... IN LOCAL MODE，读取或修改当前 BO 的 transactional buffer。这样可以避免绕过 RAP runtime，同时在 determination/action 中操作 BO 数据。

第七层机制是消息。Handler 方法常通过 reported 返回消息，通过 failed 标识失败实例。消息是否能正确落到 UI 行、字段或实例上，取决于 %tky、%msg、%element-<field> 等结构是否填对。错误处理不是简单 RAISE EXCEPTION；RAP 更强调结构化响应。

22.6 项目意义

设计评审时，handler class 要回答的问题是：“这个业务行为应该在 interaction phase 发生吗？”如果是字段默认值、业务动作、授权判断、UI feature control、保存前 validation，handler class 通常是合理位置。如果是最终写库、事件发出、审计表写入、调用旧保存 API，则要看 saver / additional save / unmanaged save。

代码 review 时，重点看四类风险。第一，handler 是否直接写数据库。第二，是否出现 COMMIT WORK / ROLLBACK WORK。第三，failed / reported / result 是否完整。第四，方法里是否混入大量查表、格式转换、外部调用，导致难以测试和维护。

测试时，handler class 的价值在于它把业务行为集中到明确方法中。Action 可以通过 EML 调用测试；validation 可以构造非法输入检查 failed / reported；feature control 可以用不同状态实例验证按钮是否可用；authorization 可以用权限对象或测试 double 验证结果。

日本项目里，handler class 经常出现在 code review、障害調査、単体テスト仕様書中。比如“このエラーは handler method の validation で返すべきか、それとも save sequence で検出すべきか”就是典型讨论。

22.7 Example / 学习例子

假设 Travel BO 有一个 action acceptTravel，用户点击后把 Travel 状态从 Open 改成 Accepted。BDEF 中声明 action：

action acceptTravel result [1] $self;

ADT quick fix 会在 lhc_travel 中生成一个 handler method。示意代码可以这样读：

CLASS lhc_travel DEFINITION INHERITING FROM cl_abap_behavior_handler.
  PRIVATE SECTION.
    METHODS accept_travel FOR MODIFY
      IMPORTING keys FOR ACTION Travel~acceptTravel RESULT result.
ENDCLASS.

实现中，handler 通常会先用 EML local mode 读取当前实例状态，判断是否允许从 Open 转到 Accepted。如果不允许，就填 failed / reported；如果允许，就用 MODIFY ENTITIES ... IN LOCAL MODE 修改 transactional buffer，并按 action result 要求返回 $self。

关键点是：这里仍然没有 COMMIT WORK。Action handler 只是改变 transactional buffer 中的 BO 状态，真正保存由后续 COMMIT ENTITIES 触发的 save sequence 处理。这样前端、OData consumer、draft、validation、authorization 和保存逻辑才能保持一致。

再看 unmanaged create。若 BDEF 是 unmanaged，lhc_travel 的 create method 可能要把 entities 转换成 application buffer，并为成功创建的临时 %cid 填 mapped。但即使在 unmanaged create 中，也不应该在 handler 里直接 INSERT 数据库；最终写库应交给 saver class 的 save()。

22.8 Counterexample / Failure mode

第一个失败模式是把 handler 当成 save。Action handler 里直接 UPDATE ztable 并 COMMIT WORK，短期看似能跑，长期会破坏 RAP LUW、draft、回滚、测试和 OData 错误反馈。

第二个失败模式是 response 不完整。Validation 发现错误只抛异常，不填 failed / reported；action 有 result 却不返回；unmanaged create 分配 key 却没有填 mapped。UI 会无法定位错误行，consumer 拿不到新 key，或者 action 按钮执行后没有刷新对象。

第三个失败模式是 handler 太大。一个 method 中混合权限、查表、格式化、远程调用、保存、日志，导致单体测试无法隔离。更好的做法是 handler 保持薄，把纯计算、格式化、策略判断提取到可测试 helper class 或 local helper method，但仍遵守 RAP 调用边界。

第四个失败模式是混淆 handler class 和 global behavior pool class。初学者看到 global class 空空如也，以为没有实现；实际上实现通常在 local classes / CCIMP 中。ADT quick fix 和 behavior pool 的 local class 区域是查找入口。

第五个失败模式是把 UI 控制全放前端。比如按钮是否可点只由 Fiori 控制，后端 action handler 不再检查状态和权限。正确做法是：UI 可以提示，handler / authorization / feature control 必须守住后端契约。

22.9 日本项目现场说法

  • 常见日语：ハンドラクラス、ハンドラメソッド、ビヘイビアプール、インタラクションフェーズ、アクション実装、バリデーション、デターミネーション、権限チェック、フィーチャ制御。
  • 说明句：handler class は RAP BO の interaction phase のロジックを実装する local class で、通常 CL_ABAP_BEHAVIOR_HANDLER を継承します。
  • 分工句：handler class は action、validation、determination などを担当し、saver class は save sequence の保存処理を担当します。
  • Review 句：この処理は handler method に置くべきか、save sequence / additional save に置くべきかを設計時に確認してください。
  • 风险句：handler method 内で COMMIT WORK や直接 DB 更新を行うと、RAP のトランザクション境界を壊す可能性があります。
  • 调查句：該当 action の実装は behavior pool の lhc_<entity> クラスを確認してください。

  • 用三句话说明 handler class 和 saver class 的职责差异。
  • 在一个 behavior pool 中找出 lhc_ 和 lsc_ 开头的 local class，并写出它们分别服务哪个阶段。
  • 用日语写一句 code review 问题：这个逻辑应该放 handler method 还是 save sequence？

  • 设计一个测试：action 状态不允许时，handler 正确返回 failed / reported，且没有直接写数据库。
