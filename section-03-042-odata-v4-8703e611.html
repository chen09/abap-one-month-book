42. OData V4 基础差异

  • Chapter 03
  • Section 42
  • self-contained study note

42.1 一句话结论

OData V4 基础差异 是现代 RAP 服务暴露中最重要的协议方向之一。和 OData V2 基础差异 相比，V4 更适合新的 RAP business service、Fiori Elements UI、Web API、action/function、metadata-driven consumption 和事务型 BO 暴露；但它不是“更高版本所以所有旧服务都自动替换”，仍要看消费者、系统版本、binding type 和项目兼容性。

一句话记忆：OData V4 是新 RAP 服务的优先理解对象，OData V2 是旧生态和兼容现实。

42.2 到底学什么

本节要把 OData V4 放在 RAP 学习路径里理解，而不是单独背协议。前面已经学过 service definition 和 service binding：当 binding type 选择 OData V4 UI 或 OData V4 Web API 后，RAP BO 就会通过 OData V4 形态被消费。消费者可能是 Fiori Elements，也可能是外部系统或测试工具。

OData V4 不是 ABAP 语法，也不是 CDS 本身。它是服务协议和 metadata contract。RAP 内部的 behavior definition、projection behavior、authorization、validation、action、draft、save sequence，会通过 service binding 暴露成可调用的 entity set、navigation、bound action/function、错误响应和事务语义。

初学者要学会三层判断。第一，这个服务是 V4 UI 还是 V4 Web API？第二，metadata 中是否包含预期 entity、navigation、action/function 和 annotation？第三，请求/响应能否证明业务语义可交付，而不仅是 preview 页面能打开。

还要补一层项目判断：V4 服务上线后由谁消费、谁维护、谁承担兼容责任。Fiori Elements 消费的 V4 UI service，通常由前端/功能团队和 ABAP/RAP 团队一起调整；外部系统消费的 V4 Web API，一旦进入接口契约，就不能随意改 entity name、字段类型、action 名称、key 结构或错误格式。对初学者来说，OData V4 的难点不是“URL 长什么样”，而是“服务模型一旦被消费者依赖，就变成了交付承诺”。

42.3 为什么重要

OData V4 重要，是因为它和现代 RAP 的学习目标高度重合。RAP 不是只生成一个 CRUD 接口，而是把业务对象的行为、权限、消息、draft、action、validation、事务边界通过 service contract 暴露出去。OData V4 能更自然地承载这些服务语义。

对 Fiori Elements 来说，V4 UI service 依赖 metadata 和 annotation 来生成页面、字段、action button、navigation、draft 行为和消息显示。一个 action 是否出现在 UI、错误消息是否定位到字段、draft 是否能 activate，往往都要回到 metadata、binding type 和 behavior/projection 设计中分析。

对外部 API 来说，V4 Web API 更关注稳定的 service contract。外部系统需要知道 entity set、key、payload、action/function、error format、authorization、ETag/concurrency、重复请求行为。只给一张 service preview 截图，不足以证明 API 可交付。

对 Java/Python 背景学习者，OData V4 可以类比成强 metadata 的 REST-like API，但它比普通 REST 更“模型驱动”。消费者会读取 $metadata，理解类型、导航和操作；RAP runtime 则把请求接到 BO lifecycle，而不是简单映射到 controller method。

42.4 Key mechanism / 核心机制

第一层机制是 OData V4 service binding。Service definition 选定暴露对象后，binding type 决定是否以 OData V4 UI / OData V4 Web API 等方式发布。Binding type 不只是标签；它影响消费场景、preview、metadata、UI 生成和交付检查。

第二层机制是 $metadata。V4 metadata 是服务契约的核心证据。它描述 entity type、entity set、navigation property、action/function、annotation、类型信息。测试时不能只调用一个 URL 成功，还要确认 metadata 是否符合前端/外部系统预期。

第三层机制是 bound action / function。RAP action 在服务暴露后可能成为绑定到 entity 的 operation。比如对某个 Travel 调用 accept 或 validate，不是普通表更新，而是进入 RAP BO action handler 和事务生命周期。要测试 action，就要记录 URL、method、payload、response、failed/reported/error 行为。

第四层机制是 navigation 和 $expand。V4 服务可以通过 association/composition 暴露导航关系。Travel 到 Booking 的 navigation 是否可见，$expand 是否可用，取决于 CDS/projection/service 暴露和 authorization 等因素。前端列表-明细、对象页、子表通常依赖这些关系。

第五层机制是 concurrency / ETag。事务型服务常需要处理并发修改。V4 API 测试不能只测单用户 happy path；要考虑 ETag、If-Match、重复提交、并发保存和错误响应。具体支持程度取决于 BO 设计和 binding/runtime。

第六层机制是 UI service vs Web API。UI service 面向 Fiori Elements，关注 annotation、draft、side effect、value help、message mapping；Web API 面向外部系统，关注 contract 稳定、payload、HTTP status、authentication、versioning。两者都可能是 V4，但测试重点不同。

第七层机制是与 V2 的差异。V4 不是简单“V2 URL 换个版本”。metadata 格式、operation 建模、UI framework 支持、binding 类型、错误处理和客户端库都可能不同。迁移 V2 到 V4 应作为接口重设计评估，而不是机械改 URL。

第八层机制是服务发布链路。一个可消费的 V4 服务通常不是单个对象，而是一条链：CDS root/projection entity 决定数据模型，behavior definition / projection behavior 决定行为，service definition 决定暴露哪些对象，service binding 决定 OData V4 UI 或 OData V4 Web API 的发布方式，publish/activation 决定 runtime 入口是否可用，authorization / DCL / behavior authorization 决定谁能访问。排查问题时要按这条链往回走，而不是只盯着 HTTP 404 或 preview 页面。

第九层机制是请求证据。OData V4 的学习不能只停在 $metadata。一个可交付的服务至少要保存四类证据：metadata 证据、happy path 请求证据、negative path 请求证据、authorization/concurrency 证据。特别是 action、PATCH、draft activate、validation failure、authorization failure、ETag conflict 这些路径，才真正证明 RAP 行为和服务协议连接起来了。

42.5 日本项目现场说法

  • OData V4：OData version 4。
  • V4 UI サービス：V4 UI service。
  • V4 Web API：V4 Web API。
  • メタデータ：metadata。
  • バウンドアクション：bound action。
  • ナビゲーションプロパティ：navigation property。
  • ETag / 楽観ロック：ETag / optimistic locking。
  • エラー応答：error response。

新規 RAP サービスでは、まず OData V4 UI または OData V4 Web API のどちらで公開するかを確認します。

意思是：新 RAP 服务首先要确认用 OData V4 UI 还是 OData V4 Web API 发布。

メタデータに entity set、navigation、action が期待通り出ているかを確認してください。

意思是：请确认 metadata 中 entity set、navigation、action 是否按预期出现。

V2 から V4 への移行は URL の置き換えではなく、API 契約とクライアント影響の再確認が必要です。

意思是：从 V2 迁移到 V4 不是替换 URL，而是需要重新确认 API contract 和客户端影响。

外部連携向けの Web API として公開する場合は、HTTP ステータス、エラー形式、認証方式、互換性方針を設計書に明記します。

意思是：作为外部集成 Web API 发布时，要在设计书中明确 HTTP status、错误格式、认证方式和兼容性策略。

42.7 Example / 学习例子

假设有一个 RAP Travel service，通过 OData V4 UI binding 发布。最小检查不是“页面能打开”一句话，而是：

1. Service binding: OData V4 UI
2. Published: yes
3. Metadata: Travel entity set exists
4. Navigation: Travel -> Booking appears if expected
5. Action: AcceptTravel appears as bound action if exposed
6. Draft: draft-related operations appear if BO is draft-enabled
7. Error: validation failure returns understandable error payload
8. Authorization: unauthorized user gets expected denial

如果是 Web API，需要换成接口视角：

GET    /Travel?$top=10
GET    /Travel('<key>')
PATCH  /Travel('<key>')
POST   /Travel('<key>')/<bound-action>
GET    /$metadata

每个请求都应保存 method、URL、headers、payload、status code、response body 和业务解释。对 action 或 update，还要验证 RAP behavior 中的 validation、authorization、save sequence 是否按预期运行。

和 V2 对比：V2 章节强调既有 Gateway、CSRF、batch、metadata cache；V4 章节强调 RAP service binding、metadata contract、bound operation、UI/Web API 选择和现代 BO 事务语义。

再看一个 AC0567C 风格的接口例子。假设要把一个“上传批次 validate / execute / status”的服务暴露给外部自动化程序。这个服务更像 V4 Web API，而不是 V4 UI service。读者应该写出这样的证据表：

Service purpose:
  Validate uploaded allocation data, execute posting, query status.

Consumer:
  External automation / middleware / technical client.

Service type:
  OData V4 Web API candidate.

Contract evidence:
  - $metadata contains Batch entity or action target.
  - validate action is visible as a bound/unbound operation as designed.
  - execute action requires authorization and returns stable business status.
  - status read can be repeated safely.

Negative tests:
  - invalid payload returns business validation message.
  - unauthorized user cannot execute.
  - duplicate request returns existing status or duplicate warning.
  - concurrent execute does not create duplicate postings.

Operation evidence:
  - request id / batch id / application log id can connect API response to SAP log.

如果同一个业务对象未来还要做 Fiori correction screen，可以另外建立 V4 UI service 或单独的 projection/binding，让 UI annotation、draft、side effect 不污染外部 Web API contract。这也是 UI service vs Web API 后续要重点展开的原因。

最后看一个迁移例子。一个老系统已有 OData V2 Gateway service，被多个外部系统调用。项目计划迁移到 RAP OData V4 时，不能只说“V4 更现代”。需要逐项确认：

Consumer inventory:
  谁在调用 V2？UI5 app、middleware、外部 vendor、batch script？

Contract delta:
  entity/key/path 是否变化？
  action/function import 调用方式是否变化？
  error payload 是否变化？
  batch/CSRF/ETag 行为是否变化？

Transition plan:
  是否并行保留 V2？
  是否提供 mapping document？
  是否有 consumer regression test？
  是否有 cutover 和 rollback 方案？

这段例子要训练的不是协议细节，而是项目判断：V4 是新 RAP 服务的重要方向，但已有 V2 consumer 的迁移是接口治理，不是语法升级。

42.8 Counterexample / Failure mode

反例一：把 V4 当 V2 的 URL 新版本。结果客户端库、metadata、operation、batch/error 处理都没评估。修正方式是把 V4 当 API contract 重新验证。

反例二：UI service 当外部 Web API。UI service 的 annotation、draft、side effect 适合 Fiori，但外部系统需要稳定 payload 和版本策略。修正方式是明确 UI service / Web API 分工。

反例三：只测 list page。列表能打开，但 action、navigation、draft activate、validation failure 都没测。修正方式是按 metadata 和 BO 行为列测试矩阵。

反例四：忽略 ETag / 并发。两个用户同时修改，后保存者覆盖前保存者。修正方式是确认 BO 的 concurrency 设计和 HTTP conditional request 行为。

反例五：metadata 中暴露了不该暴露的字段或 action。前端/外部系统开始依赖内部字段。修正方式是通过 projection 和 service definition 控制公开面。

反例六：迁移 V2 到 V4 只改 endpoint。外部系统解析错误 payload、batch 格式或 action 调用方式失败。修正方式是做消费者影响分析和并行测试。

反例七：把 metadata 当内部文档，随意改 entity alias、字段名或 action 名称。外部系统已经依赖这些名字后，改名就等于 breaking change。修正方式是把 $metadata 当 contract 管理，必要时做版本策略或新 service。

反例八：只在开发者自己的用户下测试。开发者有宽权限，preview 和 API 都成功；QAS 的 communication user 却没有业务权限、DCL 权限或 catalog/role。修正方式是用目标用户、目标 role、目标系统执行 API testing。

反例九：错误响应只给人看，不给系统处理。外部系统收到自然语言错误，却没有稳定 error code、business key 或 correlation id。修正方式是设计 API error payload 和 application log 关联方式。

42.9 思考问题

  • 当前 V4 binding 是 UI service 还是 Web API？
  • $metadata 中哪些 entity set、navigation、action/function 是 API contract 的一部分？
  • 如果 action 没出现在 metadata，应该检查 service definition、projection behavior、behavior definition 还是 binding？
  • 是否测试了 validation failure、authorization failure、并发冲突、重复请求？
  • V4 服务是否需要 ETag / optimistic locking？
  • 从 V2 迁移到 V4 时，哪些客户端代码会受影响？
  • 在日本项目设计书中，如何说明 V4 UI サービス 和 V4 Web API 的区别？
