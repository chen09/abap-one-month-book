Chapter 03 - CDS、RAP、OData

13. RAP business object 业务对象

  • Chapter 03
  • Section 13
  • self-contained study note

13.1 一句话结论

RAP business object（RAP BO，业务对象）是 ABAP RESTful Application Programming Model 里表达“一个可读、可改、可校验、可保存、可通过服务暴露的业务单元”的核心模型。它不是一张表，也不是一个 OData endpoint，而是由 CDS 数据模型、behavior definition、behavior implementation、service 暴露和运行时事务语义共同组成的业务能力边界。

本节要掌握的不是“怎么点向导生成 RAP 对象”，而是先能回答一个项目问题：为什么现代 ABAP 不应只把表包装成接口，而要把 Travel、Sales Order、Import Batch 这类业务能力建成一个有行为、有事务、有校验、有授权边界的 business object。

13.2 到底学什么

读完本节后，至少要能分清六件事：

  1. RAP business object 和普通 CDS view entity 的区别。
  2. Root entity、child entity、composition tree 为什么属于同一个业务对象。
  3. Behavior definition（BDEF）定义的是能力边界，不是实现细节。
  4. Behavior pool / handler / saver 承担的是实现和事务阶段职责。
  5. Service definition / service binding 暴露 BO，但不等于 BO 本身。
  6. AC0567C POC 这种 action-oriented API 为什么仍然要尊重 RAP save phase，而不是在 action handler 里随便写数据库。

这一节仍然是总体地图。后面的 Composition tree 组合树、Behavior definition 行为定义、Behavior pool 行为池、Managed RAP 管理式实现、Unmanaged RAP 非管理式实现、Saver class 保存器类、RAP Interaction Phase vs Save Sequence 会逐层展开。

13.3 为什么重要

RAP business object 是现代 ABAP 项目里“事务型应用”的中心词。前面 Week 3 已经学了 CDS view entity 视图实体、Root view entity 根视图实体、Projection view 投影视图、CDS access control 访问控制 和 DCL role 访问规则。这些对象各自重要，但如果不把它们放进 RAP BO 里，读者很容易误以为项目只是“建几个 view，再暴露 OData”。

真实项目的问题通常不是“接口能不能返回 JSON”，而是：

  • 谁可以创建、修改、删除这类业务数据？
  • 哪些字段只读、必填、自动派生？
  • 保存前要做哪些 validation？
  • 创建时编号何时生成？
  • 多个 child 记录和 root 记录是否一起提交或一起回滚？
  • action 执行时能不能直接 INSERT 数据库？
  • 错误消息如何返回给 Fiori / OData caller？
  • draft、lock、ETag、authorization、save sequence 是否都在正确边界内？

这些问题如果分散在 controller、utility class、SQL 片段里，代码可能短期能跑，但 review、测试、上线和升级都会变难。RAP BO 的价值，就是把“数据结构”和“可对它做什么”放在同一个模型边界里理解。

对 Java/Python 背景学习者，可以暂时把 RAP BO 类比成“领域对象 + repository + service contract + transaction framework”的组合，但这个类比不完全准确。RAP BO 不是只靠普通类组织代码；它依赖 ABAP repository objects、CDS hierarchy、BDL、EML、RAP runtime、service binding 和 SAP 权限/事务机制共同工作。

13.4 Key mechanism / 核心机制

13.4.1 BO 的第一层：CDS 数据模型

SAP 官方资料把 RAP BO 描述为围绕一组特殊的层级 CDS entities 建模的业务对象。最小情况可以只有一个 root entity；复杂情况会有 root、child、grandchild，沿着 composition path 组成 composition tree。

例如 Travel 业务对象可能有：

  • root：Travel
  • child：Booking
  • grandchild：Booking Supplement

这不是普通外键列表，而是“一个 Travel 业务对象由哪些节点组成”的事务边界。root 决定业务对象入口，child 随 root 一起被创建、读取、修改、删除或校验。前面学过的 Root view entity 根视图实体 和 Projection view 投影视图，在这里就有了位置：root view entity 是 BO 数据模型核心；projection view 通常面向 service / consumer 裁剪暴露。

13.4.2 BO 的第二层：Behavior definition 定义能做什么

Behavior definition（BDEF）是 ABAP repository object，用 Behavior Definition Language（BDL）描述 RAP BO 的事务行为。它回答的问题不是“字段有哪些”，而是“这些字段和实例能被怎样操作”。

常见声明包括：

  • implementation type：managed、unmanaged、managed with unmanaged save 等。
  • standard operations：create、update、delete。
  • field properties：mandatory、readonly、numbering、features。
  • association / composition behavior：root 和 child 之间能否通过 association 创建或读取。
  • action：业务动作，例如 approve、reject、execute import。
  • validation：保存前检查业务输入。
  • determination：在 modify 或 save 时派生字段。
  • authorization / lock / ETag / draft 等事务相关能力。

初学者要先建立一个判断：BDEF 是业务对象能力清单，不是“自动生成 OData 的装饰文件”。如果 BDEF 里没有声明某个 action，服务消费者就不应该假设可以调用它；如果字段声明 readonly，消费方就不应该把它当作可输入字段。

13.4.3 BO 的第三层：Behavior implementation 承担实现

BDEF 描述能力，但复杂业务逻辑通常还需要 behavior implementation。SAP Learning 里常说，RAP BO 的行为实现代码会放在 behavior pool 这种全局 ABAP class 中。为了分离职责，behavior pool 里通常会有 handler class 等 local classes；只有在 managed with additional save、managed with unmanaged save、unmanaged save 等需要自定义保存职责的场景，才会出现对应 saver / save-sequence 实现。标准 managed BO 的持久化通常由 RAP framework 处理，初学者不要期待每个 behavior pool 里都一定有 saver class。

可以先粗略这样理解：

  • handler：处理 interaction phase 的读取、检查、action、determination、validation 等逻辑。
  • saver：只在特定实现类型中处理 save sequence 中的真正保存、附加保存或 unmanaged save 逻辑。
  • mapped / failed / reported：把 key 映射、失败实例、消息返回给 RAP runtime，再由 OData / Fiori 等消费方看到。

这里最重要的边界是：RAP runtime 有自己的事务模型。在 RAP handler 方法中直接使用数据库修改语句（如 INSERT、UPDATE、DELETE）或直接 COMMIT WORK，不是“风格不好”，而是会绕开 RAP transactional buffer 和 save sequence 的架构禁忌。数据变更应该通过 RAP 事务缓冲和框架定义的保存阶段完成。后面 RAP Interaction Phase vs Save Sequence 会专门展开。

13.4.4 BO 的第四层：Service 暴露不是 BO 本身

Service definition 决定暴露哪些 CDS projection / entity；service binding 决定用什么协议和 runtime 暴露，例如 OData V4 UI service。它们让 BO 被外部访问，但它们不是 BO 的全部。

一个常见误区是：看到 OData endpoint，就以为业务对象已经设计好了。其实 endpoint 只是入口。真正决定事务语义的，是背后的 CDS root/projection、BDEF、behavior implementation、DCL、authorization、lock、save sequence 和 message handling。

因此，设计 review 不应只问“service binding 能不能打开”，还要问：

  • service definition 暴露的是 projection view 还是底层 interface view？
  • projection behavior 是否正确投影了 base BO behavior？
  • service 暴露的 action 是否真的在 BDEF 中定义并实现？
  • DCL / authorization 是否覆盖到被暴露的读取入口？
  • 发生错误时是否通过 RAP message/reporting 机制返回？
13.4.5 Managed、unmanaged 和责任分配

RAP BO 有不同 implementation type。初学阶段先不用背完所有语法，但要知道它们本质上是责任分配不同。

  • managed：框架承担较多标准持久化、locking、save sequence 细节，开发者主要补充行为逻辑。标准 managed BO 不要求开发者写 saver class。
  • unmanaged：开发者需要自己实现更多读取、修改、保存逻辑，适合包 legacy API / BAPI / 特殊表结构。
  • managed with unmanaged save / additional save：介于两者之间，用于框架管理大部分行为，但保存阶段需要自定义。

判断标准不是“哪个更高级”，而是：数据模型能否被 RAP framework 直接管理？是否要调用已有 BAPI？是否要写 legacy table？是否需要特殊 commit/rollback 语义？是否符合 ABAP Cloud / Clean Core 限制？

13.4.6 AC0567C POC：为什么 action 不能乱写数据库

AC0567C POC 是本教程里很好的反例和正例。它不是复制旧 SAPGUI transaction，而是做了一个新的 RAP/OData V4 API：输入多行 prior allocation 数据，经过 validate / execute / status action，最终写入 ZCOM_B_ACTT397。注意：这个 POC 使用了需要自定义保存职责的 saver-phase 设计；它不能被泛化成“所有 managed BO 都要自己写 saver”。

POC 架构链条可以简化为：

OData V4 service binding
  -> service definition
  -> behavior definition
  -> behavior pool
  -> service class
  -> RAP saver save
  -> ZCOM_B_ACTT397

Joule review 曾指出一个关键风险：如果在 action handler 调用链中直接 INSERT zcom_b_actt397，就会绕开 RAP transactional buffer / save sequence 的设计边界。最终 POC 改成：service class 负责 validate、authorization、duplicate check、staging；真正的数据库 INSERT 放到 behavior saver 的 save 阶段。这个例子说明，RAP BO 的重点不是“能不能找到一个地方写 SQL”，而是“写入是否在框架预期的事务阶段发生”。

13.4.7 BO 的边界：读、改、校验、保存、消息

一个可交付的 RAP BO 至少要从五个角度 review：

  • Read boundary：谁能读？通过哪个 CDS entity / service 读？DCL 是否生效？
  • Change boundary：哪些 operation / action 允许修改？字段是否 readonly / mandatory？
  • Validation boundary：保存前要检查哪些业务规则？错误如何返回？
  • Save boundary：真实数据库写入在哪个阶段发生？是否允许 rollback？
  • Message boundary：错误、警告、成功信息是否能通过 mapped / failed / reported 等 RAP response parameters 被 UI / API caller 理解？

如果只看 CDS 字段，不看 behavior，就会漏掉修改语义；如果只看 behavior，不看 service，就会漏掉暴露范围；如果只看 service，不看 DCL / authorization，就会漏掉安全边界；如果只看 action 代码，不看 save sequence，就会漏掉事务一致性。

13.4.8 和传统 ABAP 的差异

传统 ABAP 报表或 function module 常见写法是：读取参数、查表、校验、更新、COMMIT WORK、返回消息。RAP BO 不是简单把这段逻辑搬到 class 里。RAP runtime 会管理 request、interaction phase、save sequence、lock、draft、message、authorization 和 service response。

因此，迁移旧功能时要先做业务对象识别：

  • 旧 transaction 操作的是哪个业务对象？
  • 哪些输入属于 root，哪些属于 child table？
  • 哪些按钮其实是 action？
  • 哪些检查应该变成 validation？
  • 哪些自动填值应该变成 determination？
  • 哪些数据库写入必须推迟到 save sequence？
  • create 场景是否正确处理 %cid、mapped、failed、reported？

这也是 RPA-to-API migration 的核心：不是模拟旧屏幕点击，而是把旧屏幕背后的业务能力重新建模为 API-first 的 BO。

13.4.9 初学者的最小观察路径

在 ADT 里观察一个 RAP BO，可以按这个顺序：

  1. 找 root view entity，看业务对象根节点。
  2. 找 child / composition，看是否是 composite BO。
  3. 找 behavior definition，看允许哪些 operation、action、validation、determination。
  4. 找 behavior implementation class / behavior pool，看逻辑在哪里实现。
  5. 找 projection view 和 behavior projection，看服务层裁剪了什么。
  6. 找 service definition / service binding，看外部入口是什么。
  7. 用 Data Preview / service preview / EML test / HTTP client 验证读写行为。

这个顺序比“先打开 OData URL 看 JSON”更适合学习，因为它能让读者把对象链路和事务边界连起来。

13.7 Example / 学习例子

13.例 1：Travel BO 的教学模型

SAP Learning 和 SAP-samples 经常使用 Travel / Booking 作为 RAP 示例。它很适合初学者，因为业务对象层级直观：

Travel root
  -> Booking child
      -> Booking Supplement child

Travel 是 root，因为它代表一个旅行申请或旅行订单的入口。Booking 是 child，因为它依附于某个 Travel。Behavior definition 可以声明 Travel 能 create/update/delete，Booking 能通过 composition 创建，Travel 上有 accept / reject action，保存时执行 validateDates、validateCustomer 等 validation。

读这个例子时，不要只看字段。要问：

  • root 是谁？
  • child 如何属于 root？
  • BDEF 允许哪些 operation？
  • action 是对 root 还是 child？
  • validation 在 modify 还是 save 时触发？
  • service projection 暴露了哪些能力？
13.例 2：AC0567C POC 的 action-oriented BO

AC0567C POC 不是 Travel 这种典型维护 UI，而是批量上传 API。它仍然能帮助理解 RAP BO：

  • service binding：ZRPA01_SB_AC0567C_IMPORT
  • service definition：ZRPA01_SD_AC0567C_IMPORT
  • behavior definition：ZRPA01_I_AC0567C_IMPORT
  • behavior pool：ZBP_RPA01_I_AC0567C_IMPORT
  • service class：ZCL_RPA01_AC0567C_IMP_SVC
  • target table：ZCOM_B_ACTT397

其中 validate / execute / status 更像 action/API 能力，而不是普通 CRUD。关键设计点是：ExecuteImport 做 validation、authorization、duplicate check 和 staging；真实 DB insert 要进入这个 POC 的 RAP saver save。这让 POC 从“能写入数据库”提升为“尊重 RAP 事务模型的 API 设计”，同时也提醒读者：是否需要 saver，取决于 BO 的 implementation type 和保存责任分配。

13.例 3：一个最小判断练习

看到一个新需求“做销售订单批量导入 API”，不要先问“建哪个 endpoint”。先写出：

BO：SalesOrderImport
Root：ImportHeader
Child：ImportItem
Action：Validate, Execute, GetStatus
Validation：company code、customer、amount、duplicate import id
Save boundary：execute action stages rows; saver writes persistent rows
Service：OData V4 API for external caller

这个练习能让读者从业务对象开始，而不是从技术对象碎片开始。

13.8 Counterexample / Failure mode

  • 误区 1：把 RAP BO 当成 OData endpoint。修正：endpoint 是暴露入口，BO 是数据模型、行为、实现和事务语义的组合。
  • 误区 2：把 root view entity 当成 BO 全部。修正：root view entity 只是数据模型核心；没有 BDEF / behavior implementation，就没有完整事务行为。
  • 误区 3：在 action handler 中直接做数据库修改，或随意 COMMIT WORK。修正：RAP handler 中禁止绕开 RAP transactional buffer；修改要通过 RAP 事务语义，并在 save sequence 中持久化。
  • 误区 4：把 validation 写成普通 utility check，却没有通过 RAP mapped / failed / reported 返回实例级消息。修正：错误反馈要能被 UI / API caller 对应到对象和字段；create 场景还要处理 %cid 到真实 key 的映射。
  • 误区 5：只建 managed BO，不理解 unmanaged / unmanaged save。修正：implementation type 是责任分配，要结合 legacy API、BAPI、表结构和 Clean Core 约束。
  • 误区 6：把 projection / service 当成安全边界。修正：service 裁剪暴露范围，但权限还要看 DCL、authorization、behavior authorization 和 IAM/PFCG。
  • 误区 7：只看生成对象是否 active，不看 transport 是否完整。修正：CDS、BDEF、behavior implementation、service definition、service binding、authorization artifacts 要一起检查。
  • 误区 8：以为任何 behavior pool 都有 saver class。修正：标准 managed BO 通常由框架保存；只有 additional save、unmanaged save 等场景才需要对应 saver/save 方法。
  • 误区 9：启用 draft 却忘记 draft table。修正：draft-enabled BO 需要相应 draft persistence 设计和对象一起激活、传输、测试。
  • 误区 10：在 EML 测试或调用代码中把 COMMIT ENTITIES 和 COMMIT WORK 混用。修正：EML 场景要理解 COMMIT ENTITIES 如何触发 RAP save sequence。

13.9 思考问题

  • 这个需求的 root business object 是什么？它是否真的有 root / child 层级？
  • 哪些能力是 standard operation，哪些应该建成 action？
  • 哪些检查是 validation？哪些自动填值是 determination？
  • 数据库写入发生在 interaction phase 还是 save sequence？
  • 是否有代码在 handler / utility class 中直接 COMMIT WORK 或直接写数据库？
  • create / action / validation 的 mapped、failed、reported 是否完整？
  • BDEF 是否使用项目要求的 strict mode，并在目标系统 release 中可用？
  • 如果启用 draft，draft table 和 draft 行为是否完整？
  • service definition / binding 暴露的是 projection view 还是底层 view？
  • DCL、authorization、lock、ETag、draft 是否和 BO 生命周期一致？
  • AC0567C POC 中，为什么真实 insert 要放到 saver save？
  • 如果把这个 BO 改成 unmanaged，开发团队需要额外负责哪些事情？
  • 日本项目 review 时，应该说 RAPビジネスオブジェクト、ビヘイビア定義 还是 セーブシーケンス？
