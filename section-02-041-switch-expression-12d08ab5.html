Chapter 02 - Core ABAP Programming

41. SWITCH expression 多分支表达式

  • Chapter 02
  • Section 41
  • self-contained study note

41.1 一句话结论

SWITCH expression 是 ABAP 的 conditional expression / conditional operator，用来根据“同一个 operand 的不同取值”构造一个结果值。它最适合小而稳定的 code-to-value mapping，例如把状态码、消息类型、操作符、枚举式代码转换成显示文本、分类值或测试期望值。

DATA(status_text) = SWITCH string( lv_status
  WHEN 'A' THEN `Active`
  WHEN 'B' THEN `Blocked`
  ELSE `Unknown`
).

这段代码的核心不是“执行多个分支动作”，而是“产生一个值”。lv_status 是被比较的 operand；每个 WHEN 都拿自己的候选值和这个 operand 比较；第一个匹配的 WHEN 决定结果。如果没有匹配且没有 ELSE，结果不是“未定义行为”，而是指定或推断出来的 result type 的 type-specific initial value。学习 SWITCH 时，最重要的是把它和 COND expression 条件表达式、CASE statement 分清楚：SWITCH 适合单一 operand 的值映射，COND 适合任意 logical expression，CASE 适合需要执行多条 statement 的流程分支。

41.2 到底学什么

本节要掌握五件事。

  • SWITCH type( operand WHEN value THEN result ... ELSE result ) 的基本形状。
  • SWITCH #( ... ) 的类型推断边界，以及什么时候应该显式写 string、abap_bool、日期类型或自定义类型。
  • WHEN 的判断模型：比较的是同一个 operand，不是每个分支写一套任意条件。
  • ELSE 省略后的初始值行为，以及为什么业务代码通常要显式写 ELSE 或 ELSE THROW。
  • 在 SAP 项目里如何判断一段映射应该留在 SWITCH，还是移动到 domain fixed values、customizing table、mapping table、message class、value help 或 helper method。

初学者可以把 SWITCH 先理解成“表达式版本的 CASE”，但这个类比只能帮助入门。严格地说，CASE 是 statement，可以在每个分支里执行多步逻辑；SWITCH 是 expression，必须整体产生一个结果值。只要分支里开始出现日志、数据库写入、消息收集、多行处理，通常就已经越过了 SWITCH 的舒适区。

41.3 为什么重要

SAP 项目中有大量“代码值转换”：订单状态到显示文本、消息类型到 severity、接口字段代码到内部枚举、测试数据状态到期望结果、ALV/报表显示值到说明文本。传统写法常用 CASE 或一串 IF / ELSEIF：

CASE lv_status.
  WHEN 'A'.
    lv_status_text = `Active`.
  WHEN 'B'.
    lv_status_text = `Blocked`.
  WHEN OTHERS.
    lv_status_text = `Unknown`.
ENDCASE.

如果目标只是给一个变量赋值，SWITCH 可以把“判断”和“结果”收束到同一处：

DATA(lv_status_text) = SWITCH string( lv_status
  WHEN 'A' THEN `Active`
  WHEN 'B' THEN `Blocked`
  ELSE `Unknown`
).

这对代码 review 很有帮助。reviewer 一眼可以看到：比较对象是什么、有哪些已知取值、未知值如何处理、结果类型是什么。尤其在 ABAP Unit、RAP validation 前置转换、UI 显示字段、接口 code list 适配里，SWITCH 能让小型映射更集中。

但它也有风险。SAP 项目的 code list 往往来自业务配置、domain fixed values、customizing、外部接口规范或消息文本。如果这些值会由业务或配置维护人员改变，就不应该把越来越多的 WHEN 硬编码散落在代码中。SWITCH 适合“小而稳定的技术映射”，不适合承载可配置业务规则。

41.4 Key mechanism / 核心机制

Basic form is CASE-like value selection

基本形状如下。

DATA(text) = SWITCH string( lv_status
  WHEN 'A' THEN `Active`
  WHEN 'B' THEN `Blocked`
  ELSE `Unknown`
).

可以读成：构造一个 string 类型的结果，检查 lv_status，如果等于 'A' 就返回 Active，如果等于 'B' 就返回 Blocked，否则返回 Unknown。它的 mental model 接近 CASE lv_status. WHEN ...，但 SWITCH 本身是表达式，所以它出现在 operand position，并且最终只返回一个值。

Operand is the center

SWITCH 的第一个核心参数是 operand。

SWITCH string( lv_status ... )

后面的每个 WHEN 都围绕同一个 lv_status 比较。不要把 SWITCH 写成“多条件判断工具”。下面这种需求不适合 SWITCH：

" 这种判断更适合 COND 或 IF
lv_status = 'A' AND lv_amount > 0

判断标准很简单：

  • 同一个字段或表达式的不同取值决定一个结果：优先考虑 SWITCH。
  • 每个分支条件都不同，包含 AND、OR、范围判断、方法调用组合：优先考虑 COND 或 IF。
  • 每个分支要执行多条 statement：优先考虑 CASE 或 IF。
# needs result type context

SWITCH #( ... ) 依赖上下文推断 result type。

DATA lv_text TYPE string.

lv_text = SWITCH #( lv_status
  WHEN 'A' THEN `Active`
  ELSE `Unknown`
).

这里左侧 lv_text TYPE string 给了 # 足够的上下文。如果是 inline declaration、generic formal parameter、复杂 method actual parameter，类型推断可能不够清楚。面向初学者和项目 review，业务字段建议显式写出类型。

DATA(lv_text) = SWITCH string( lv_status
  WHEN 'A' THEN `Active`
  ELSE `Unknown`
).

这不是因为 # 不好，而是因为书稿、review、交付文档需要读者快速知道：结果到底是 string、abap_bool、金额、日期、还是某个 DDIC 类型。

WHEN is checked in order

多个 WHEN 按顺序检查，第一个匹配项决定结果。对普通、互斥的代码值来说，这个顺序通常不敏感；但如果候选值重复、常量定义混乱、或者分支含义重叠，就会出现“后面的分支永远不会到达”的阅读风险。

DATA(text) = SWITCH string( lv_status
  WHEN lc_status_open THEN `Open`
  WHEN lc_status_closed THEN `Closed`
  ELSE `Unknown`
).

项目中更推荐用常量命名业务含义，而不是到处写裸字面量。字面量越多，reviewer 越难判断它们是否来自同一个 domain、同一份接口规范或同一张 customizing 表。

ELSE omission means type-specific initial value

如果没有任何 WHEN 匹配，并且没有写 ELSE，SWITCH 的结果是 result type 的 type-specific initial value。

DATA(lv_text) = SWITCH string( lv_status
  WHEN 'A' THEN `Active`
).

如果 lv_status 不是 'A'，lv_text 会是 string 的初始值，也就是空字符串。对数字可能是 0，对 abap_bool 可能是 abap_false，对日期可能是初始日期。这在语法上成立，但在业务上经常危险，因为 unknown value 被悄悄吞掉了。

面向接口字段、状态字段、消息类型、权限判断结果时，通常应显式处理未知值。

DATA(lv_text) = SWITCH string( lv_status
  WHEN 'A' THEN `Active`
  WHEN 'B' THEN `Blocked`
  ELSE `Unknown`
).

如果未知值表示数据不合法，可以考虑 ELSE THROW。不过在 RAP 场景中，是否抛异常、填 reported / failed，或返回业务消息，要遵守当前行为实现和错误处理设计，不要为了“看起来现代”随意抛异常。

Result values share one result type

THEN 和 ELSE 后面的结果要能转换到同一个 result type。

DATA(category) = SWITCH string( lv_type
  WHEN 'C' THEN `Customer`
  WHEN 'V' THEN `Vendor`
  ELSE `Other`
).

不要让一个分支返回文本、另一个分支返回数字、第三个分支返回日期，然后依赖隐式转换“碰巧能过”。在 SAP 项目中，代码值、金额、日期、布尔值、消息 severity 要尽量保持类型清晰。类型清晰，后续 OData 暴露、RAP behavior、ALV 显示、ABAP Unit 断言都会更稳定。

THROW is a special result position

THROW 可以出现在 THEN 或 ELSE 的 result position，用来抛出 class-based exception。Joule review 的重要提醒是：THROW 不是普通“产生值的 operand”，它不需要和 result type 做类型兼容；它是特殊的异常路径。

DATA(status_text) = SWITCH string( lv_status
  WHEN 'A' THEN `Active`
  WHEN 'B' THEN `Blocked`
  ELSE THROW zcx_invalid_status( )
).

示例里的 zcx_invalid_status 代表项目自定义 exception class。真实项目里要使用已经定义好的异常类型，并考虑调用方是否会 TRY...CATCH。THROW SHORTDUMP 也存在，但它会触发 runtime error，不应该在普通业务代码里轻易使用。对初学者来说，优先理解 class-based exception 和 RAP 消息返回模型，比追求短写法更重要。

LET can reduce repeated helper values

SWITCH 可以配合 LET ... IN 定义表达式内部的临时辅助值。

DATA(result) = SWITCH string(
  LET lv_code = to_upper( lv_status ) IN lv_code
  WHEN 'A' THEN `Active`
  WHEN 'B' THEN `Blocked`
  ELSE `Unknown`
).

这里先把 lv_status 转成大写，再用 lv_code 做 operand。LET 适合“一次性、局部、简单”的预处理。只要预处理本身开始有多步逻辑、异常处理、数据库访问或业务规则，就应抽到 helper method 或普通 statement 中。

SWITCH vs COND vs CASE

三者的区别可以这样记。

选择适合什么不适合什么
SWITCH一个 operand 的多个固定取值映射成一个结果任意复杂条件、多步动作、大型业务规则
COND多个 logical expression 中第一个成立者产生结果简单 code list 映射时可能不如 SWITCH 直观
CASE一个 operand 的分支控制，每个分支可执行多条 statement只为构造一个小结果值时略显冗长

对比示例：

" SWITCH: one operand
DATA(text) = SWITCH string( lv_status
  WHEN 'A' THEN `Active`
  ELSE `Unknown`
).

" COND: arbitrary logical expression
DATA(flag) = COND abap_bool(
  WHEN lv_status = 'A' AND lv_amount > 0 THEN abap_true
  ELSE abap_false
).

" CASE: multiple actions
CASE lv_status.
  WHEN 'A'.
    append_message( ).
    update_display( ).
  WHEN OTHERS.
    log_unknown_status( ).
ENDCASE.

不要把 Modern ABAP 理解成“所有 CASE 都要改成 SWITCH”。正确的判断是：值映射用 expression，流程控制用 statement，复杂业务规则用清晰的 method、配置或表驱动设计。

SWITCH mapping review checklist

在项目 review 时，可以用这张小表判断是否适合 SWITCH。

QuestionSWITCH usually fits when...Prefer another design when...
Is there one operand?所有分支都围绕同一个 status/type/operator 比较每个分支条件不同，需要组合条件
Is the mapping small and stable?分支少，值集合稳定，接近技术枚举值来自客户配置、主数据、外部系统或频繁变更
Is unknown value handled?有明确 ELSE 或 ELSE THROW省略 ELSE 导致 unknown value 静默变成初始值
Is result type explicit enough?result type 清楚，# 不影响阅读类型推断依赖复杂上下文，reviewer 需要猜

41.5 日本项目现场说法

常见日语：

  • SWITCH式
  • CASE相当
  • 単一オペランド
  • 値マッピング
  • コード値変換
  • WHEN句、THEN句、ELSE句
  • 型推論
  • 型固有の初期値
  • 未想定値 / 未定義値
  • 可読性
  • 保守性
  • CASE文に戻す

现场说明句：

  • この SWITCH式 は、単一オペランドのコード値を結果値に変換するための式です。
  • 複雑な論理条件が必要な場合は、SWITCH式 ではなく COND式 または IF文 を検討してください。
  • ELSE句 がない場合、未想定値ではなく型固有の初期値が返るため、業務上正しいか確認してください。
  • このコード値変換は複数箇所に散在させず、domain fixed values、customizing、mapping table、または共通変換メソッドに寄せるべきではないでしょうか。
  • 分岐が長くなっているため、可読性と保守性を考慮して CASE文 または helper method に戻してください。

这里要特别说明“未想定値 / 未定義値”。在本节语境里，它不是 ABAP 的 undefined behavior，而是“WHEN 中没有列出的 operand 值”或“业务上没有预期到的代码值”。如果没有 ELSE，ABAP 返回的是 type-specific initial value；如果业务上不能接受，就应显式 ELSE 或异常处理。

41.6 English Terms

  • Official / search terms: SWITCH, SWITCH, Conditional Operator, conditional expression, conditional operator
  • ABAP syntax: SWITCH type( operand WHEN value THEN result ... ELSE result ), SWITCH #( ... ), WHEN, THEN, ELSE, LET ... IN, THROW, THROW SHORTDUMP
  • Type terms: operand, result type, operand type, type inference, type-specific initial value
  • Comparison terms: CASE statement, COND expression, IF statement, logical expression, value mapping, code list
  • SAP project terms: domain fixed values, customizing, mapping table, message class, value help
  • Recommended search pattern: ABAP SWITCH conditional operator SAP Help, ABAP SWITCH WHEN THEN ELSE, ABAP SWITCH THROW SAP Help

读官方资料时要注意：COND 和 SWITCH 都属于 conditional expressions / conditional operators，但 COND 的 WHEN 后面是 logical expression，SWITCH 是围绕 operand 的取值选择结果。

41.7 Example / 学习例子

例 1：状态码转显示文本
DATA(status_text) = SWITCH string( lv_status
  WHEN 'A' THEN `Active`
  WHEN 'B' THEN `Blocked`
  WHEN 'C' THEN `Closed`
  ELSE `Unknown`
).

适合场景：小型、稳定、技术性的状态码显示。review 时要问：这些状态码是否来自同一个 domain 或接口规范？Unknown 是否满足业务要求？是否需要 ABAP Unit 覆盖未知值？

例 2：消息类型转 severity
DATA(severity) = SWITCH string( lv_msgty
  WHEN 'E' THEN `Error`
  WHEN 'W' THEN `Warning`
  WHEN 'I' THEN `Information`
  WHEN 'S' THEN `Success`
  ELSE `Unknown`
).

如果这个映射用于 UI 或 OData 响应，后续可能需要和前端规范、Fiori message handling、RAP reported messages 对齐。SWITCH 只解决本地值映射，不等于完整错误处理设计。

例 3：未知操作符抛异常
DATA(operator_text) = SWITCH string( lv_operator
  WHEN '+' THEN `Addition`
  WHEN '-' THEN `Subtraction`
  WHEN '*' THEN `Multiplication`
  WHEN '/' THEN `Division`
  ELSE THROW zcx_invalid_operator( )
).

这里 unknown value 被视为非法输入，所以使用异常路径。真实项目中要确认调用方能捕获该异常，或者该异常会按项目规范转换成消息。

例 4：ABAP Unit 固定期望值
DATA(actual) = SWITCH string( lv_status
  WHEN 'A' THEN `Active`
  WHEN 'B' THEN `Blocked`
  ELSE `Unknown`
).

cl_abap_unit_assert=>assert_equals(
  exp = `Blocked`
  act = actual
).

测试代码里使用 SWITCH 很常见，因为测试经常需要把输入状态映射成期望值。但测试里的映射也不能太聪明；如果测试逻辑复制了生产逻辑，测试就失去验证价值。

41.8 Counterexample / Failure mode

误区 1：把 SWITCH 当成 COND

错误心智：只要有多个分支，就用 SWITCH。

修正：如果分支条件是 lv_amount > 0 AND lv_status = 'A' 这类 logical expression，应使用 COND 或 IF。

误区 2：省略 ELSE 让 unknown value 静默通过

错误心智：没有匹配就“不返回值”。

修正：ABAP 会返回 result type 的 type-specific initial value。业务字段通常要显式写 ELSE，至少让 unknown value 在代码中可见。

误区 3：把配置型业务规则写死在 SWITCH

错误心智：映射多一点也没关系，继续加 WHEN。

修正：如果值来自客户配置、主数据、外部接口、权限矩阵、多语言文本或经常变化的业务规则，就应考虑 customizing table、mapping table、domain fixed values、message class 或 helper method。

误区 4：过度相信 #

错误心智：SWITCH #( ... ) 越短越现代。

修正：在书稿、交付代码、review 截图、接口字段中，显式 result type 往往更可维护。短写法不是目标，可读性才是目标。

误区 5：把 THROW 当成普通返回值

错误心智：THROW 也必须能转换成 string 或 abap_bool。

修正：THROW 是特殊 result position，用于异常路径，不是 value-producing operand。真正要 review 的是：异常类型是否合适、调用方是否处理、RAP 或 OData 场景是否应该返回业务消息。

41.9 思考问题

  • 这段 SWITCH 的 operand 是什么？所有分支是否都围绕同一个 operand？
  • result type 是什么？使用 # 时，读者是否能看懂推断来源？
  • 如果没有任何 WHEN 匹配，当前代码返回 ELSE、抛异常，还是返回 type-specific initial value？
  • 这个 mapping 是技术常量、domain fixed value、customizing、外部接口 code list，还是业务可配置规则？
  • 如果新增一个 code value，需要改几处代码？是否应该集中到 helper method 或 mapping table？
  • 在日本项目 review 中，应该说 SWITCH式、値マッピング、コード値変換，还是 CASE文に戻す？
  • 是否需要 ABAP Unit 覆盖 normal value、unknown value、initial value 三条路径？
  • 这段逻辑在 RAP validation、OData 响应或 UI 显示中是否会影响错误处理和消息语义？
