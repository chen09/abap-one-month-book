Chapter 01 - SAP 和 ABAP 基础

28. String processing 字符串处理

  • Chapter 01
  • Section 28
  • self-contained study note

本节结论

String processing 字符串处理 是把业务数据变成可读消息、接口字段、日志文本和检索条件的基础。Day 04 不追求背完所有语句，而是先掌握四件事：string 与 fixed-length character 的差异、&& 和 string template 的组合方式、常用搜索/替换/分割/去空格操作，以及不要把字符串处理当成随意改业务字段格式的工具。

正文

ABAP 项目里到处都有字符串：错误消息、日志、OData response message、ALV 显示文本、文件名、接口字段、ID、日期展示、拼接给外部系统的 label。很多初学者觉得字符串只是“把几个值连起来”，但 SAP 项目里的字符串常常带有业务语义：客户号前导零不能随便删，金额要和币种一起显示，日期内部格式和画面格式不同，用户语言会影响可读文本，日本项目还会要求 message class、text symbol 或翻译资源，而不是把日语硬编码到代码里。

先区分两类常见字符数据。string 是变长字符串，适合消息、日志、临时拼接和长文本处理。c 这类 fixed-length character 类型有固定长度，短内容会被空格填充，常见于 DDIC 字段、旧接口、ID、状态码、语言键等。Day 03 学过 Elementary data types 基础技术类型，这里要把类型意识带到文本处理里：lv_code TYPE c LENGTH 10 和 lv_text TYPE string 都能放字符，但它们在 trailing blanks、长度、比较和输出上不完全一样。

最现代、最适合初学者读懂的拼接方式是 string template：

DATA(message) = |Item { ls_item-item_no } has amount { ls_item-amount } { ls_item-currency }|.
out->write( message ).

string template 用 |...| 包住文本，用 { ... } 嵌入表达式。它的优点是消息看起来接近最终输出，比一串 CONCATENATE 或 && 更容易 review。对 Java/Python 背景的人，可以把它类比成 f-string / template literal，但要记住 ABAP 的格式选项、类型转换和语言环境仍然按 ABAP 规则来。

&& 是 string expression 里的连接运算符，适合简单拼接：

DATA(full_name) = ls_customer-first_name && ` ` && ls_customer-last_name.

&& 不会自动给你加分隔符。中间的空格、逗号、斜线都要自己写清楚。如果拼接很多字段，string template 往往更易读。旧代码里也会看到 CONCATENATE ... INTO ... SEPARATED BY ...：

CONCATENATE ls_customer-street ls_customer-city
       INTO DATA(address)
       SEPARATED BY `, `.

这不是“错”，而且在日本 ECC 维护、既存改修、老接口程序里仍然很常见。但新代码里经常可以用 && 或 string template 写得更清楚。读旧代码时要能认出 CONCATENATE；写新代码时优先选择读起来像最终文本的表达方式。

空格处理是 ABAP 字符串里很容易出错的部分。这里要先区分 statement 和 function。CONDENSE lv_text. 这个语句会去掉前后空格，并把中间连续空格压缩为一个空格；CONDENSE lv_text NO-GAPS. 会去掉所有空格，包括中间空格。现代表达式里还有 condense( val = lv_text ) 这个函数，但它的行为对应不带 NO-GAPS 的 CONDENSE，没有一个等价的 NO-GAPS 参数；如果真的要删除所有普通空格，通常要写成 `replace( val = lv_text sub = with = occ = 0 )`，并清楚说明这是“删除空格”，不是普通整理。

CONDENSE 适合整理用户输入或显示文本，但不能随便用于业务 key。比如 material number、customer number、外部接口 code 里的空格、前导零、固定长度都可能有业务意义。写 CONDENSE lv_partner NO-GAPS. 或 `replace( ... sub = with = occ = 0 )` 之前，必须先问：这真的是显示文本清理，还是把业务字段改坏了？

搜索、替换、分割是第二组基础操作。FIND 用来找子串或模式；REPLACE 用来替换；SPLIT 用分隔符把字符串拆成多个部分；substring( ) 或 offset/length 用来取一段字符。例子：

DATA(raw_text) = `  PO-2026-0001  `.
CONDENSE raw_text.

DATA(clean_id) = replace( val = raw_text sub = `-` with = `` occ = 0 ).
DATA(prefix)   = substring( val = raw_text off = 0 len = 2 ).

这里 occ = 0 表示替换所有出现位置；如果省略 occ，很多替换场景默认只处理第一处，这是初学者很容易误会的地方。off = 0 表示从第一个字符开始。ABAP 的 offset/length 边界要小心：越界会触发异常或 runtime error，不要像处理 Python slice 那样期待它自动“宽容”。如果只想查某个子串是否存在，现代 ABAP 可以用 contains( val = text sub = ABC ) 这类 predicate function；如果需要找到位置，可以用 find( ) 或 FIND statement。

大小写和正则表达式也要有 awareness。contains( )、find( )、replace( ) 这类 ABAP string functions 的搜索和比较默认区分大小写，很多函数有 case = abap_false 之类参数来关闭大小写敏感。旧项目代码里还会看到 character-like relational operators，例如 IF lv_text CS 'ERROR'.、IF lv_text NS 'OK'.、IF lv_code CP 'A*'.、IF lv_code NP 'Z+'.。这里 Day 04 不要求背完全部规则，但要知道：CS 是 contains string，NS 是 contains no string，CP 是 covers pattern，NP 是 no pattern；这些表达式会影响 sy-fdpos，大小写、通配符 * / +、转义字符 # 和 trailing blanks 的规则要回到 ABAP Keyword Documentation 查证，不能按 Java/Python 的 contains 或 regex 直觉解释。

正则表达式在 ABAP 中常以 PCRE 形式出现，可以用于复杂搜索、校验和替换。但 Day 04 不鼓励初学者一上来就用复杂 regex 解决业务规则；regex 很强，也很难维护。日本项目 code review 里，如果一个正则表达式没有说明、没有测试、没有边界例子，很容易被要求拆开或补文档。

字符串处理和国际化要分开。技术日志里临时拼接 |item { lv_id } not found| 可以帮助学习和 debug；但用户可见消息、长期维护的错误文本、日本客户现场说明，应该优先考虑 message class、text element、translation 或框架自带消息机制。硬编码日语、中文或英文文本，会让多语言环境、测试截图、客户验收和后续维护都变麻烦。本书后续讲 message handling、RAP reported/failed、OData error response 时，会把“字符串拼接”和“正式消息设计”分开。

还有一个接口字段风险：字符串看起来像普通文本，但很多 SAP 字段其实是 code。客户号、物料号、company code、document number、language key、currency code、unit code、external ID 都可能有固定长度、前导零、大小写、domain value 或 conversion exit。不要因为“显示时不好看”就 CONDENSE NO-GAPS、SHIFT LEFT DELETING LEADING '0'、TRANSLATE TO UPPER CASE。字符串处理解决文本问题，不自动解决业务格式问题。

把它们放在一起，初学者的最小心智模型是：想输出给人看，优先 string template；想简单连接，用 &&；想清理显示文本，用 CONDENSE，但别动业务 key；想查找替换，先用清楚的 built-in function，再考虑 regex；想给用户正式报错，用 message/text 机制，而不是到处硬编码文本。

为什么重要

  • ABAP 的错误消息、日志、接口字段、OData response、测试输出都离不开字符串处理。
  • string 和 fixed-length c 的 trailing blanks 差异，会影响比较、拼接和输出。
  • 错误的 CONDENSE、去零、大小写转换会破坏客户号、物料号、单据号等业务 key。
  • string template 能让消息拼接更可读，是现代 ABAP 初学者应优先掌握的写法。
  • 日本项目重视多语言、消息 class、测试截图和障害調査日志；字符串不是只给程序员临时看的内容。

核心机制

  • string：变长文本，适合消息、日志、临时拼接和长文本。
  • fixed-length character (c)：固定长度、可能带 trailing blanks，常见于 DDIC 字段、code、ID、状态。
  • string template：|... { expr } ...|，适合构造可读消息和调试输出。
  • &&：连接运算符，简单拼接；不会自动添加分隔符。
  • CONCATENATE：旧式连接语句，常见于 legacy code；SEPARATED BY 可显式指定分隔符。
  • CONDENSE statement：整理空格；NO-GAPS 删除所有普通空格，风险更高，不能随便用于业务 key。
  • condense( ) function：表达式写法，对应不带 NO-GAPS 的整理逻辑；不能把它当作 CONDENSE ... NO-GAPS 的直接替代。
  • replace( )：occ = 0 表示替换所有出现位置；省略 occ 时通常只替换第一处。
  • FIND / find( )、REPLACE / replace( )、SPLIT、substring( )：搜索、替换、分割、取子串。
  • CS / NS / CP / NP：旧项目常见的 character-like comparison operators；识别它们，并用官方文档确认大小写、通配符和 sy-fdpos 规则。
  • offset/length：offset 从 0 开始，越界不是安全 slice，要谨慎检查长度。
  • regex / PCRE：适合复杂模式，但必须配测试和说明。
  • i18n / message design：用户可见文本优先 message class、text element、translation 或框架消息机制。

日本项目现场表达

  • 常见日语：文字列処理、文字列結合、文字列テンプレート、連結、検索、置換、分割。
  • 空格/长度相关：前後スペース、末尾ブランク、固定長文字列、可変長文字列、桁数、ゼロ埋め。
  • 质量相关：文字化け、大文字小文字、正規表現、パターン照合、メッセージ組立、ハードコード文言、翻訳対象。
  • 业务 key 相关：先頭ゼロ削除禁止、変換出口確認、文字列操作、文字列加工。
  • 说明句：この処理は表示用メッセージの文字列結合であり、業務キーの桁数やゼロ埋めは変更しません。
  • Review 句：CONDENSE NO-GAPS により業務キーの桁数が変わる可能性があるため、表示項目とキー項目を分けてください。
  • Review 句：ユーザー向け文言を直接ハードコードせず、メッセージクラスまたはテキストエレメントを使用してください。
  • Review 句：文字列加工ロジックに変換出口の考慮が不足しているため、DDIC の変換出口を確認してください。

学习例子

先写一个清楚的消息：

DATA(message) = |Item { ls_item-item_no } for customer { ls_item-customer_id } was not found|.
out->write( message ).

再把名字拼起来：

DATA(full_name) = |{ ls_customer-first_name } { ls_customer-last_name }|.

如果要整理用户输入的显示文本：

DATA(user_text) = `   ABAP     string   processing   `.
CONDENSE user_text.
" result: `ABAP string processing`

如果是业务 key，则不要直接照搬：

DATA(customer_id) = `0000123456`.
" 不要为了显示好看就随意去掉前导零；这可能是正式 customer number 格式。

搜索和替换：

DATA(raw_id) = `PO-2026-0001`.

IF contains( val = raw_id sub = `PO-` ).
  DATA(numeric_part) = replace( val = raw_id sub = `PO-` with = `` ).
ENDIF.

如果要删除所有连字符，要明确写出 occ = 0：

DATA(compact_id) = replace( val = raw_id sub = `-` with = `` occ = 0 ).

分割字符串时，可以先从 SPLIT 读起：

DATA(raw_csv) = `A,B,C`.
DATA parts TYPE STANDARD TABLE OF string WITH EMPTY KEY.

SPLIT raw_csv AT `,` INTO TABLE parts.

读旧代码时，可能会看到这种写法：

IF raw_id CS `PO-`.
  " contains string
ENDIF.

IF raw_id CP `PO-*`.
  " pattern check
ENDIF.

Day 04 只要求能认出它们：CS 是 contains string，CP 是 pattern check。遇到 NS、NP、sy-fdpos、通配符或大小写规则时，不凭直觉改代码，回到 ABAP Keyword Documentation 确认。

这些例子都故意很小。真实项目里，下一步要问的是：这个字符串是给人看的 message，还是要传给外部系统的 code？如果是后者，必须回到 DDIC、domain、conversion exit 或接口规格，而不是只靠字符串函数猜。

常见误区

  • 把 customer number 当普通数字字符串，删除前导零，导致接口或查询查不到数据。
  • 对业务 key 使用 CONDENSE NO-GAPS，把有意义的空格或固定长度破坏掉。
  • 用 && 拼接地址却忘记空格、逗号、换行，输出看起来连在一起。
  • 到处硬编码日语/英文用户消息，导致翻译、测试截图、客户验收和维护困难。
  • 用复杂 regex 做简单条件判断，缺少注释和测试，后续维护者不敢改。
  • 把 offset/length 当 Python slice，越界后才在 runtime error / short dump 中发现。
  • 忽略 fixed-length c 的 trailing blanks，比较或输出结果和预期不同。

思考题

  • string 和 fixed-length c 的差异，会怎样影响拼接、比较和输出？
  • 什么时候 string template 比 && 更可读？什么时候 CONCATENATE 仍然需要能读懂？
  • CONDENSE 和 CONDENSE NO-GAPS 的风险差异是什么？
  • 为什么不能随便去掉客户号、物料号、单据号里的前导零？
  • 用户可见错误消息为什么不应该长期硬编码在 ABAP 代码里？
  • offset/length 和 regex 分别适合什么场景？为什么都需要边界测试？

学习任务

  • 用 string template 写 3 条错误/警告消息，至少包含一个金额、一个 ID、一个状态。
  • 用 && 拼接姓名或地址，并手动加入空格/逗号，观察可读性。
  • 对一个带多余空格的显示文本分别使用 CONDENSE 和 `replace( ... sub = with = occ = 0 )`，说明两者差异。
  • 解释为什么业务 key 不应随便 CONDENSE NO-GAPS 或删除前导零。
  • 用 contains( ) 或 find( ) 判断一个文本是否包含指定前缀；再读一段使用 CS 或 CP 的旧代码，写出它在做什么。
