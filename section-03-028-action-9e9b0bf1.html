28. Action 业务动作

  • Chapter 03
  • Section 28
  • self-contained study note

28.1 一句话结论

Action 业务动作 是 RAP business object 中用于表达非标准 CRUD 的业务操作，例如审批、拒绝、取消、复制、重新计算、提交、生成后续对象等；它把“用户点按钮后发生的业务动作”显式建模在 behavior definition 和 behavior pool 中，而不是塞进普通 update、CDS 计算字段或前端按钮脚本里。

一句话记忆：CRUD 处理“增删改查”，Action 处理“业务动词”。如果需求说的是 accept、reject、cancel、submit、release、recalculate、copy，通常就要先想到 RAP action。

28.2 概念说明

SAP Developer Tutorial 对 RAP action 的入门解释很直接：action 是 non-standard operation，默认和 RAP BO entity instance 相关，并且会改变 instance 的状态；如果使用 static，则 action 不绑定某个实例，而是关联整个 entity。SAP-samples RAP100 练习也说明，action 在 behavior definition 中声明，在 ABAP behavior pool 中实现。

这和普通 update 有明显区别。Update 通常是“把某些字段改成 consumer 提供的新值”；Action 则是“执行一个业务动作，并由后端决定状态如何变化、消息如何返回、是否产生 result”。比如审批 Travel 时，前端不应该直接把 OverallStatus = 'A' 当成普通 update；更好的设计通常是 acceptTravel action：后端检查状态、授权、必填字段、锁和业务规则，再把状态改成 Accepted。

Action 也不是单纯的 UI button。Fiori Elements 可以把 action 渲染成按钮，OData consumer 可以调用 action，ABAP 代码可以通过 EML Entity Manipulation Language 执行 action。真正的契约在 BDEF 和 behavior implementation 中，而不是按钮本身。

初学者先记住四种常见形态：

  • Instance action：默认形态，作用在一个或多个具体 BO instance 上，例如 acceptTravel。
  • Static action：不绑定具体 instance，作用于 entity 级别，例如“批量导入”“全局重新计算”。
  • Action with parameter：调用时需要输入参数，例如取消原因、审批备注、模拟选项。
  • Factory action：action 产生新的 BO instance，例如复制一个 Travel，或从模板生成一个新对象。

28.3 为什么重要

没有 action，很多项目会把业务动作硬塞进 update。比如把状态字段从 Open 改成 Approved，就假设审批完成。这会让业务语义变得模糊：谁检查了权限？谁判断当前状态能不能审批？谁写审批消息？谁决定审批后哪些字段只读？谁保证重复点击不会产生副作用？

Action 把这些问题集中到 BO 行为里。设计文档可以明确写：acceptTravel 检查权限和状态，成功后把状态改为 Accepted，并返回更新后的 $self；rejectTravel 需要输入拒绝原因；copyTravel 创建一个新 Travel 并返回新实例。这样 UI、OData、EML、测试和 review 都围绕同一个业务能力讨论。

对项目交付来说，action 直接影响接口可读性和缺陷定位。看到 OData metadata 中的 action，读者就知道这是业务动作而不是普通字段修改；看到 behavior pool 的 FOR MODIFY action method，reviewer 就知道要检查 interaction phase、failed/reported/result、authorization、feature control、幂等和 save sequence 边界。

对 Java/Python 背景学习者，可以把 RAP action 类比成 domain command 或 application service command，例如 approve(orderId)。但 RAP 的差异是：action 是 BDEF 契约的一部分，会被 RAP runtime、EML、OData metadata、Fiori Elements、authorization、draft、feature control 和事务缓冲共同解释。

28.4 边界

第一条边界：action 不是普通 update 的漂亮名字。把状态字段改成 Approved 是 update；执行 approve 这个业务过程是 action。Action 可以内部修改字段，但它表达的是业务意图，而不是字段赋值请求。

第二条边界：action 不等于 validation。Validation 负责检查数据是否有效；action 负责执行业务动作。Action 里可以调用或触发检查，但不要把“保存时字段是否合法”都包装成 action。

第三条边界：action 不等于 determination。Determination 通常用于派生或自动补全，例如价格重算、默认值、总额更新；action 通常由 consumer 显式触发，例如用户点击 Accept。若某个逻辑应在字段变化时自动执行，更像 determination；若它代表业务按钮或命令，更像 action。

第四条边界：action handler 不应随意直接持久化数据库或 COMMIT WORK。Action handler 属于 interaction phase。它可以改变 RAP transactional buffer、返回消息和 result；真正持久化通常由后续 COMMIT ENTITIES / OData save flow 触发 save sequence。上一节 %control 和前面 Saver class 保存器类 都在强调：不要绕开 RAP 事务模型。

第五条边界：action 不是权限本身。Action 能不能被调用，要结合 RAP authorization、instance authorization、feature control、PFCG/IAM、业务状态检查。只靠按钮 disabled 或前端隐藏，不是完整安全设计。

28.5 核心机制

第一层机制是 BDEF 声明。最简单的 instance action 可以这样写：

define behavior for ZI_Travel alias Travel
{
  action acceptTravel result [1] $self;
  action rejectTravel parameter ZS_RejectReason result [1] $self;
}

result [1] $self 表示 action 返回一个当前实体类型的结果，通常用于把修改后的 instance 返回给 consumer。SAP Learning 说明，action 的 result 参数是 optional；但如果在 action definition 中声明了 result，就必须在 implementation 中填充它。这个点很容易漏，导致 UI 或 EML consumer 拿不到预期返回。

第二层机制是 static action。Static action 不绑定某个具体实例：

static action recalculateAllOpenItems result [*] $self;

它适合全局或集合级业务动作。这里的 result [*] $self 只是示意；实际 static action 也可以没有 result，或返回更适合业务的参数/抽象结构。但初学者不要把 static action 当成“随便写一个后台 job 入口”。它仍然是 RAP BO 契约的一部分，仍然要考虑权限、事务、性能和可测试性。

第三层机制是 factory action。Factory action 用 action 创建新实例，例如从已有 Travel 复制一个新 Travel。它通常会和 %cid、mapped、result、late/early numbering 相关。Factory action 不是普通 create 的替代品，而是表达“根据某个业务动作生成新对象”。

第四层机制是 behavior pool 实现。Action 的实现通常在 handler class 中，方法签名由 BDEF 生成，常见形式类似：

METHOD acceptTravel.
  READ ENTITIES OF zi_travel IN LOCAL MODE
    ENTITY Travel
    FIELDS ( OverallStatus )
    WITH CORRESPONDING #( keys )
    RESULT DATA(lt_travel)
    FAILED failed.

  " Check state, authorization-related preconditions, and business rules.

  MODIFY ENTITIES OF zi_travel IN LOCAL MODE
    ENTITY Travel
    UPDATE FIELDS ( OverallStatus )
    WITH VALUE #(
      FOR travel IN lt_travel
      ( %tky = travel-%tky
        OverallStatus = 'A' ) )
    FAILED failed
    REPORTED reported.

  result = VALUE #(
    FOR travel IN lt_travel
    ( %tky = travel-%tky
      %param = CORRESPONDING #( travel ) ) ).
ENDMETHOD.

这段是教学示意。真实 action method 的生成签名会带有 FOR MODIFY ... FOR ACTION ... RESULT result 这类 RAP action 语义，且 result 应在状态修改后重新读取或构造为最新实例数据。重点是：action handler 用 EML local mode 改变 BO transactional buffer，而不是直接 UPDATE ztable。

第五层机制是 failed / reported / result。Action 失败时，不应只 raise 一个模糊 exception，而应把失败实例放进 failed，把可读消息放进 reported。如果 action 声明了 result，就要填充 result。SAP Learning 对 result 的要求很明确：声明了就必须填。

第六层机制是 feature control。上一节 RAP feature control 已经说明，按钮是否可用通常由 feature control 控制；action 内部仍要检查业务前提。比如 Accepted 状态下 acceptTravel 应该 disabled，但 action method 也要防止重复审批，因为非 UI consumer 可能直接调用 action。

第七层机制是 projection behavior。底层 BDEF 有 action，不代表服务消费者一定能调用。Projection behavior 可能需要 use action acceptTravel; 才把它暴露给特定 service。这个边界在 Projection view 投影视图 中已经出现：projection 是消费层闸门，不能默认开放所有底层能力。

28.6 项目意义

设计评审时，action 要回答五个问题。第一，这个业务动作叫什么，是否比直接 update 字段更清楚。第二，它是 instance、static，还是 factory action。第三，它是否需要参数、result 和消息。第四，它在什么状态下可用，谁有权执行。第五，它是否只修改 transactional buffer，真正持久化是否交给 save sequence。

代码 review 时，重点看 action handler 是否遵守 RAP 生命周期。不要在 action 里直接 INSERT / UPDATE 数据库并提交；不要吞掉 failed/reported；不要忘记 result；不要只靠 feature control 防非法调用；不要把复杂保存后副作用塞进 interaction phase。

测试时，要覆盖 UI、OData、EML 三个视角。UI 上看按钮是否出现/置灰；OData 上看 action 是否暴露、参数是否正确、消息是否可读；EML 上看 action 能否通过 MODIFY ENTITIES ... EXECUTE 被调用，并且 failed / reported / result 是否符合预期。

日本项目中，action 常出现在 Fiori 按钮、审批流程、状态转换和 API 设计里。现场会问：この処理は更新ではなく action として定義すべきではないですか、action 実行時の権限チェックはどこですか、result と reported message は返していますか。这些问题都在确认业务动作是否被显式、可测试、可交付地建模。

28.7 Example / 学习例子

以 Travel 审批为例。业务需求是：Open 状态的 Travel 可以被审批为 Accepted；Accepted 或 Rejected 状态不能再次审批；审批时要返回成功消息，并把更新后的 Travel 返回给前端。

BDEF 可以声明：

action ( features : instance ) acceptTravel result [1] $self;

这里 ( features : instance ) 表示按钮可用性按每个 Travel instance 判断；result [1] $self 表示返回更新后的当前实例。

Behavior pool 中，acceptTravel method 通常做几件事：

  1. 用 READ ENTITIES ... IN LOCAL MODE 读取当前状态。
  2. 对不是 Open 的实例填充 failed / reported。
  3. 对可以审批的实例用 MODIFY ENTITIES ... UPDATE FIELDS ( OverallStatus ) 把状态改成 Accepted。
  4. 读取或构造 result，返回给 consumer。
  5. 不执行 COMMIT WORK，不直接写持久化表。

如果是 Fiori Elements UI，projection behavior 还要暴露这个 action，UI annotation / metadata 才能把它作为按钮呈现。若按钮不可用，还要检查 feature control 是否正确返回 disabled。

这个例子把 Day 19/20 的几节串起来：action 是业务动词，feature control 控制按钮可用性，%control / UPDATE FIELDS 控制字段修改语义，validation/authorization 控制可执行前提，save sequence 负责最终保存。

28.8 Counterexample / Failure mode

第一个失败模式是把 action 写成直接 SQL。开发者在 acceptTravel 中直接 UPDATE ztravel SET status = 'A'，再 COMMIT WORK。这会绕过 RAP transactional buffer、draft、failed/reported、save sequence 和回滚语义，是 RAP 项目里非常危险的反模式。

第二个失败模式是用 update 代替 action。前端直接 PATCH OverallStatus = 'A'，没有审批动作、没有权限语义、没有状态检查、没有审批消息。短期能改字段，长期会让流程不可审计、不可测试。

第三个失败模式是声明了 result 却不填 result。UI 或 EML consumer 期待 action 后返回更新后的 $self，结果拿到空 result 或旧数据。修正：如果 BDEF 声明 result，implementation 必须填充；如果不需要返回，就不要随便声明。

第四个失败模式是只靠 feature control。按钮在 UI 上 disabled，但 action method 不检查状态。外部 consumer 直接调用 action 时仍能重复审批。修正：feature control 是体验层和契约提示，action handler 仍要做后端业务检查。

第五个失败模式是 projection 忘记 use action。底层 BDEF 和 handler 都写好了，服务里却不能调用。排查时不要只看 behavior pool，还要看 projection behavior、service definition、service binding 和 metadata 刷新。

第六个失败模式是把长期外部副作用塞进 action。比如 action 中直接调用外部 HTTP 服务、发送不可回滚通知、写审计日志并立即提交。若后续 save 失败，就会出现外部世界已经变化而 BO 未保存的裂缝。需要根据场景把副作用放到 additional save、outbox、event 或异步集成设计中。

28.9 日本项目现场说法

  • 常见日语：アクション、業務アクション、承認アクション、却下アクション、取消アクション、インスタンスアクション、スタティックアクション、ファクトリアクション。
  • 说明句：Action は、RAP BO に対する標準 CRUD 以外の業務操作を明示的に定義する仕組みです。
  • Review 句：このステータス変更は単なる update ではなく、承認 action として定義した方が業務意図が明確です。
  • 调查句：ボタンが表示されない場合、BDEF の action 定義、projection behavior の use action、feature control、metadata 更新を確認します。
  • 事务句：action handler 内で直接 DB 更新や COMMIT WORK を行わず、RAP の transactional buffer と save sequence に従ってください。
