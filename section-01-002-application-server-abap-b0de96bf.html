Chapter 01 - SAP 和 ABAP 基础

2. Application Server ABAP 运行时基础

  • Chapter 01
  • Section 02
  • self-contained study note

2.1 一句话结论

Application Server ABAP（AS ABAP）是 ABAP 程序真正被调度、执行、访问数据库、返回结果的应用层运行环境。理解它，才能把前一节的 SAP system landscape 系统环境分层 连接到“ABAP 代码到底在哪里跑、为什么会有 work process、short dump、background job 和性能分析”。

2.2 到底学什么

先把 Application Server ABAP 理解成 SAP 系统中的“ABAP 应用运行层”，不是某一台物理服务器，也不是 ADT 或 SAP GUI。SAP Learning 明确提醒：硬件厂商说 application server 时可能指机器，SAP 语境里更常指一个软件上的 SAP instance / application server。它有自己的 instance profile、自己的内存 buffer，一组一起启动和停止的进程，并运行在某个 host 上；一个 SAP system 可以有一个或多个 application server。旧项目资料里还可能看到 central instance / dialog instance，新术语里通常对应 Primary Application Server（PAS）和 Additional Application Server（AAS），Day 01 先做到能识别即可。

对 ABAP 开发者来说，AS ABAP 解决的是“请求进来之后谁来处理”的问题。用户可能从 SAP GUI 操作事务，也可能从浏览器访问 SAP Fiori / Web 应用，外部系统也可能通过 RFC 或 HTTP 调用接口。请求不会直接跳进某个 ABAP class；它先进入 SAP 运行时，再由 dispatcher、work process、ICM、Gateway、Message Server 等组件配合处理。Day 01 不要求掌握 Basis 配置，但要知道这些词不是装饰，它们解释了为什么同一段代码在不同负载、不同系统、不同调用入口下表现不同。

本节先抓四个问题。第一，AS ABAP 在 presentation layer、application layer、database layer 之间处在哪一层。第二，dispatcher 和 work process 为什么是理解 ABAP 运行时的核心。第三，database interface 为什么让 ABAP SQL（7.52 之前常称 Open SQL）能相对独立于具体数据库，同时又要意识到 SAP S/4HANA 以 SAP HANA 为数据库。第四，开发者能在哪里观察运行时现象，例如 System Status、SM51、SM50/SM66、ST22、SM37，而不是只在代码编辑器里看源码。

2.3 为什么重要

如果只把 SAP 当成数据库或接口平台，就会误解很多 ABAP 项目问题。一个程序在 DEV 已经 activation，只说明对象在当前系统有 active version；真正运行时，还要看请求进入哪个 application server、由哪个 work process 处理、是否访问数据库、是否触发 update task、是否进入 background job、是否发生 runtime error。也就是说，activation 解释“对象版本是否生效”，AS ABAP 解释“对象如何被执行”。

对 Java/Python 背景的人来说，可以把 AS ABAP 类比成应用服务器运行时，但不能完全等同于 Spring Boot、Django、Flask 或 Node.js 进程。普通 Web 项目常常是某个服务进程加载本地代码文件，对外暴露 HTTP API；ABAP 项目里，代码主要以 Repository object 仓库对象 形式存在于 SAP repository 中，运行时由 SAP 系统管理。开发者通常不直接启动一个自己的 ABAP 进程，而是把对象激活到系统里，再由 SAP dispatcher 和 work process 根据请求来执行；这个调度过程会在下方核心机制里展开。

这也是交付排查的基础。性能慢，可能要看 SQL、buffer、work process、background job；短 dump，要看 ST22 和调用栈；接口不通，可能涉及 ICM、Gateway、RFC、OData 服务；用户说“我这里卡住了”，可能要知道他连到哪个 application server、是否有负载均衡、是否只有某个实例异常。初学者不需要成为 Basis 管理员，但必须知道 ABAP 代码不是在真空里运行。

2.4 Key mechanism / 核心机制

可以把一次典型请求想成下面这条路径：

SAP GUI / Browser / RFC caller
  -> dispatcher / ICM / Gateway
  -> request queue
  -> available work process
  -> ABAP interpreter + database interface
  -> database / buffer / update / spool / background handling
  -> result returned to caller

第一层机制是 dispatcher。SAP Learning 把 ABAP dispatcher 描述为 AS ABAP 的中心进程：它把请求先放入 request queue，再分发给可用 work process。重点不是背定义，而是理解“用户和 work process 没有固定一对一绑定”。一个用户的不同 dialog step 可能由不同 work process 处理，所以 SAP 运行时需要 user context 的 roll-in / roll-out 机制。

第二层机制是 work process。真正执行应用程序逻辑的是 work process。当前项目里最该先认识的 dispatcher-managed work process 类型，是 dialog、background、update 和 spool：dialog 处理交互请求，background 处理后台任务，update 处理延迟提交的更新逻辑，spool 处理打印/输出队列。逻辑锁由 Central Enqueue Service / enqueue server 负责，在当前 S/4HANA 语境下不要把它简单当成普通 work process 类型来教。数量、参数、实例分布属于 Basis 细节，后续遇到性能或生产问题时再深入。

第三层机制是 ABAP interpreter、task handler 与 database interface。经典 Dynpro 场景里会出现 screen / dynpro processor；work process 内部的 task handler 协调这些处理器之间的动作，实际 ABAP 处理逻辑由 ABAP interpreter（有些资料也会泛称 ABAP processor）执行；访问数据库时，每个 work process 通过 database interface 把 ABAP SQL（旧称 Open SQL）转成目标数据库能理解的 SQL。这里要加一个现实 caveat：database interface 提供了相对数据库独立的编程模型，但 SAP S/4HANA 只支持 SAP HANA，实际项目会越来越多地结合 HANA 能力、CDS 或数据库过程思考性能；所以它不是“以后不用理解数据库”的理由。

第四层机制是通信组件。ICM 处理 HTTP/HTTPS/SMTP 等 web 协议入口，Gateway 支持 RFC 通信，Message Server 为多个 application server 的 dispatcher 提供中央消息服务，并在登录负载均衡中帮助选择合适的实例；用户登录并连接到某个实例后，SAP GUI 通常和该实例的 dispatcher 通信，而不是每个业务请求都先由 Message Server 路由。这里先记住名字和责任边界：Fiori/OData/API 问题经常会碰到 ICM，RFC/BAPI 集成会碰到 Gateway，多实例登录和可用性会碰到 Message Server。

第五层机制是运行时观察入口。开发者不需要一开始就会 Basis 管理，但要知道问题发生时去哪看。ST22 看 short dump，SM37 看 background job，SM50 看当前实例 work process，SM66 看全系统 work process，SM51 看 application server 列表，系统状态里能看到当前 system/client/user/application server。把这些入口记住，后续看到“程序慢”“接口超时”“后台失败”时，就不会只盯着源码。

第六层机制是开发者与 Basis 的分工。ABAP 开发者负责理解业务逻辑、SQL、异常、权限、日志和可复现证据；Basis/运维更关注实例、work process 数量、profile parameter、ICM/Gateway 配置、系统日志、负载和可用性。好的项目沟通不是“你帮我看一下 SAP”，而是能说清楚：哪个系统、哪个 client、哪个用户、哪个时间、哪个 application server、哪个程序/API、发生了什么错误、已有哪类证据。

运行时对象初学者理解常见排查入口
Dispatcher请求排队和分发中心SM50/SM66 观察 work process 状态
Dialog work process处理交互式 dialog step前台卡顿、事务响应慢

把 AC0567C POC 放进这个视角，可以这样读运行路径：

External caller / Browser
  -> HTTP/OData entry through ICM
  -> OData/RAP runtime and service binding
  -> ABAP handler / behavior implementation in a work process
  -> local EML / database interface / application log
  -> response and backend evidence returned

2.7 Example / 学习例子

假设用户在 SAP GUI 里执行一个自定义事务，点击保存后系统很慢。初学者可能只盯着 ABAP 代码看，但 AS ABAP 视角会多问几层：这个请求是 dialog 处理还是触发了 update task？是否有慢 SQL？有没有等待锁？有没有 background job 在处理后续任务？是否只有某个 application server 慢？如果发生 runtime error，ST22 的 short dump 记录了哪个程序、哪一行、哪个用户、哪个 application server、哪个时间点。

再看一个接口例子。外部系统调用一个 SAP API，如果是 HTTP/OData 入口，问题可能涉及 ICM、service binding、authorization、OData runtime 和 ABAP handler；如果是 RFC/BAPI 入口，问题可能涉及 Gateway、RFC destination、BAPI 调用模式和 commit 处理。代码本身当然重要，但运行时入口决定了你从哪里查日志、和谁协作、用什么事务或工具定位。

最小练习：在可用 SAP 系统中打开 System -> Status，记录当前 SAP system、client、user、application server；再由有权限的人演示或截图 SM51 / SM50，观察 application server 和 work process 列表。没有系统权限时，至少画出“用户请求 -> dispatcher -> work process -> database interface -> database”的路径。

再做一个生产问题复盘例子。用户报告：“昨天晚上 AC0567C 上传 job 失败了。”一个只看代码的回答可能是“我本地没复现”。AS ABAP 视角下，应该先收集：

System / Client: QAS or PRD?
Time: when did the job fail?
User: dialog user or technical user?
Entry: background job, OData API, RFC, or manual transaction?
Runtime evidence: SM37 job log, ST22 dump, application log, work process/runtime info
Business evidence: request id, batch id, processed count, failed row

这份清单不要求初学者马上会操作所有事务，但能建立正确排查姿势：先定位运行环境，再定位代码。

2.8 Counterexample / Failure mode

  • 把 AS ABAP 当成“装 SAP 的机器”，会漏掉 instance、dispatcher、work process、buffer、ICM、Gateway 这些真正解释运行行为的概念。
  • 把 ADT 当成运行时，会误以为代码在本地 Eclipse 里执行；实际上 ADT 是编辑和开发入口，代码运行在 SAP 系统的 AS ABAP 中。
  • 把数据库当成一切，会忽略 ABAP interpreter、database interface、authorization、lock、update task、background job 对行为的影响。
  • 只会说“程序慢”，却不知道区分 SQL 慢、锁等待、后台 job、dialog work process、ICM/Gateway、short dump，会导致障害调查无法推进。
  • 修正动作：回到 MOC SAP System Foundations，把 Repository object 仓库对象、activation、transport、AS ABAP runtime 和 database 画在同一张图上。
  • 把所有运行问题都推给 Basis。很多性能、dump、权限、日志问题需要 ABAP 开发先提供程序名、时间、用户、输入、SQL 或日志证据；没有这些证据，Basis 只能看系统层症状。
  • 只在 DEV 观察运行时。QAS/PRD 的实例、负载、权限、数据量和接口配置可能完全不同。修正方式是在对应系统收集证据，并遵守生产访问和审计规则。
  • 混淆 ICM 和 Gateway。HTTP/OData 问题优先想到 ICM / HTTP runtime / OData/RAP；RFC/BAPI 远程调用优先想到 Gateway / RFC destination / commit pattern。

2.9 思考问题

  • 为什么说 AS ABAP 是应用层运行时，而不是物理服务器或 ADT？
  • dispatcher 和 work process 分别解决什么问题？
  • Java/Python 的 Web server / application server / worker process，分别能类比 AS ABAP 的哪一部分？不能类比哪一部分？
  • 如果用户报告“保存很慢”或“接口超时”，你会从哪些运行时入口开始排查？
  • 为什么 ABAP SQL 不是简单等同于直接写数据库 SQL？
  • AC0567C 的 OData API 进入 AS ABAP 后，大致经过哪些运行时组件？
  • 哪些问题应由 ABAP 开发先收集证据，哪些需要 Basis/运维一起看？
  • 为什么同一段代码在 DEV 快，在 QAS/PRD 慢？
