26. Method signature 方法签名

  • Chapter 02
  • Section 26
  • self-contained study note

26.1 一句话结论

Method signature 方法签名是一个 method 对外表达“你可以怎样调用我、要给我什么、我会还给你什么、我可能抛出什么错误”的契约。在 OO ABAP 里，它主要写在 METHODS 或 CLASS-METHODS 声明中，由 method name、parameter interface、parameter typing、OPTIONAL / DEFAULT、RETURNING、RAISING 等部分组成。

初学者不要把 method signature 理解成“方法里面的代码”。方法体在 METHOD ... ENDMETHOD 里；signature 在 class definition 的可见性区段里。读 signature 时，应先判断这个 method 是查询、计算、修改状态，还是调用外部动作，再看它是否把输入、输出和异常说清楚。一个好的 signature 可以让 Global class 全局类、Local class 局部类、Interface 接口定义 和 ABAP Unit 单元测试基础 都更容易读、更容易测。

26.2 到底学什么

上一节讲 Local class 局部类 时，已经看到 local helper 可以把一段逻辑收进小对象。现在要进一步看：这个对象暴露出来的 method 到底应该长什么样。

本节要掌握六件事：

  • METHODS 与 CLASS-METHODS 如何声明 instance method 和 static method。
  • IMPORTING、EXPORTING、CHANGING、RETURNING 分别代表什么调用契约。
  • VALUE(...)、REFERENCE(...)、省略写法对 pass by value / pass by reference 的影响。
  • OPTIONAL、DEFAULT、PREFERRED PARAMETER 如何影响调用者。
  • RAISING 与旧式 EXCEPTIONS 的区别，以及为什么新开发优先 class-based exceptions。
  • 如何用小而清楚的 method signature 支撑 Small methods 小方法原则、测试和日本项目 review。

例如下面这段声明已经能告诉读者很多信息：

METHODS get_status_text
  IMPORTING iv_status      TYPE string
            iv_language    TYPE sylangu DEFAULT 'E'
  RETURNING VALUE(rv_text) TYPE string
  RAISING   cx_sy_conversion_error.

这段 signature 表示：调用者必须提供一个 status；language 可不传，默认按英文处理；method 返回一个文本；如果转换失败，可能向外传播 class-based exception。读者即使还没看到实现，也能大致理解它的用途和风险。

26.3 为什么重要

在 Java/Python 背景里，method signature 常被理解为“函数名、参数、返回类型”。ABAP 的差异在于：signature 不只是类型说明，还强烈影响调用语法、异常处理、数据是否被原地改写，以及这个 method 能不能作为 functional method 写在表达式位置。

项目里最常见的问题不是“不知道怎么写 method”，而是 signature 暗示了错误的责任边界：

  • 一个 method 有五六个 EXPORTING 参数，调用者不知道哪个才是主结果。
  • 一个 method 用 CHANGING 改了调用者的数据，但名字看起来像只读查询。
  • 一个本来应该明确失败的业务检查，只用 abap_bool 悄悄返回 false，调用方漏判也不会明显报错。
  • 一个本来只需要一个结果的计算，却不用 RETURNING，导致无法写成表达式，也不方便在 ABAP Unit 单元测试基础 中断言。
  • 一个 helper 的参数全是 string，没有 DDIC type、domain、枚举语义，后续 review 很难判断合法值。

所以 method signature 是“设计质量”最早暴露出来的地方。代码实现再漂亮，如果 signature 已经把副作用、异常和数据边界说乱了，后面的测试、复用、接口封装都会变重。

26.4 Key mechanism / 核心机制

26.4.1 signature lives in method declaration

在 ABAP class 中，signature 写在 class definition 里：

CLASS lcl_status_text DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS get_text
      IMPORTING iv_status      TYPE string
      RETURNING VALUE(rv_text) TYPE string.
ENDCLASS.

CLASS lcl_status_text IMPLEMENTATION.
  METHOD get_text.
    CASE iv_status.
      WHEN 'O'.
        rv_text = 'Open'.
      WHEN 'C'.
        rv_text = 'Closed'.
      WHEN OTHERS.
        rv_text = 'Unknown'.
    ENDCASE.
  ENDMETHOD.
ENDCLASS.

CLASS-METHODS get_text ... 是 signature；METHOD get_text. ... ENDMETHOD. 是 implementation。review 时先看 signature，因为它决定调用者看到的 contract。implementation 当然重要，但它是“如何实现”；signature 是“对外承诺”。

这里也顺便连接上一节：CLASS-METHODS 声明 static method，METHODS 声明 instance method。static / instance 的细节会在 Static vs instance 静态与实例 展开；本节先把注意力放在 parameter interface。

26.4.2 four parameter directions

ABAP method declaration 中最常见的方向有四类：

方向中文理解典型用途设计提醒
IMPORTING调用者输入给 method查询条件、计算输入、上下文参数非 optional 的 input 调用时必须提供
EXPORTINGmethod 输出给调用者多个输出值、旧式 API 风格输出太多时考虑返回结构
CHANGING调用者传入，method 修改后带回明确的原地规范化、累积、转换副作用很强，名字必须表达“会修改”
RETURNINGmethod 的主返回值单一结果、可表达式化的计算只能有一个 return value，声明为 VALUE(...)

如果 method 的主语义是“给一个输入，算一个结果”，优先考虑 RETURNING：

CLASS-METHODS is_quantity_valid
  IMPORTING iv_quantity     TYPE i
  RETURNING VALUE(rv_valid) TYPE abap_bool.

这样调用很直观：

IF lcl_quantity_rule=>is_quantity_valid( 10 ) = abap_true.
  " ...
ENDIF.

如果 method 的主语义是“把调用者传进来的值规范化”，CHANGING 可以成立，但名字要明确：

CLASS-METHODS normalize_status
  CHANGING cv_status TYPE string.

调用者看到 CHANGING 就应当知道：cv_status 对应的实际变量会被改掉。

RETURNING 并不等于“这个 method 永远不能有其他形式参数”。SAP Help 的 METHODS / CLASS-METHODS 语法允许 method 在 RETURNING VALUE(r) 之外仍声明 IMPORTING、EXPORTING、CHANGING 等 formal parameters，所以这里讲的是设计准则，不是语法禁令。对初学者和项目 helper 来说，最容易读、最容易测的设计通常是 IMPORTING + RETURNING：输入从 IMPORTING 来，主结果从 RETURNING 回去。若一个 method 既有 RETURNING 又有多个 EXPORTING 或 CHANGING，虽然可能遇到合法场景，但 review 时应特别确认它是否把多个责任塞进了一个 method。

26.4.3 declaration direction and call-site direction are easy to confuse

ABAP 初学者很容易被调用语法绕住。声明端和调用端是镜像关系：method 声明为 IMPORTING 的参数，调用者在 call site 用 EXPORTING 传进去；method 声明为 EXPORTING 的参数，调用者在 call site 用 IMPORTING 接出来。换句话说，declaration 是 method 的视角，call site 是 caller 的视角。

声明端的 IMPORTING，在调用端通常写在 EXPORTING 下面，因为调用者是在“export 参数给 method”：

DATA lv_text TYPE string.

lcl_status_text=>get_text(
  EXPORTING iv_status = 'O'
  RECEIVING rv_text   = lv_text ).

同一个 method 也可以写成 functional method 调用：

DATA(lv_text) = lcl_status_text=>get_text( iv_status = 'O' ).

记忆方式是：declaration 看 method 自己的视角；call site 看调用语句的视角。method 的 RETURNING 参数在 standalone call 里可以用 RECEIVING 接住；在表达式调用里直接成为表达式结果。

再看一个有 EXPORTING 参数的旧式或工具类 API 风格：

lcl_reader=>read_status(
  EXPORTING iv_id      = lv_id
  IMPORTING ev_status  = lv_status
            ev_message = lv_message ).

这里 ev_status 和 ev_message 在 method declaration 里是 EXPORTING，但调用者接收它们时写在 IMPORTING 下。这个镜像关系是 ABAP method call 最容易让 Java/Python 背景学习者困惑的地方。

26.4.4 pass by value and pass by reference

SAP Help 对 method parameters 的规则很明确：除 RETURNING 参数外，method 参数可以按值传递，也可以按引用传递；如果只写参数名，默认是 pass by reference。显式写 VALUE(p) 表示 pass by value；显式写 REFERENCE(p) 表示 pass by reference。RETURNING 参数只能按值传递。

在初学阶段可以先这样理解：

METHODS calculate_text
  IMPORTING VALUE(iv_status) TYPE string
  RETURNING VALUE(rv_text)   TYPE string.

VALUE(iv_status) 表示 method 收到的是输入值的副本。对 elementary input 来说，这常常让 signature 更清楚：method 只是使用这个输入，不依赖调用者变量本身。

如果写成：

METHODS calculate_text
  IMPORTING iv_status      TYPE string
  RETURNING VALUE(rv_text) TYPE string.

iv_status 默认按引用传递，但因为它是 IMPORTING 参数，method 内不能修改这个 importing parameter；试图修改会被语法检查挡住。CHANGING 参数即使按引用传递，也正是为了让 method 修改并带回值而设计。也就是说，“按引用传递”和“可以修改调用者变量”不是一回事；是否能修改还要看 IMPORTING、EXPORTING、CHANGING 的方向。

EXPORTING 或 CHANGING 按值传递时，还有一个项目上容易忽略的点：如果 method 因异常没有正常结束，按值传递的输出通常不会写回实际参数。设计失败路径时，不要让调用者依赖半写回的输出值。

26.4.5 OPTIONAL and DEFAULT are not decoration

OPTIONAL 和 DEFAULT 可以让 input parameter 或 input/output parameter 在调用时不必传：

CLASS-METHODS get_text
  IMPORTING iv_status      TYPE string
            iv_language    TYPE sylangu DEFAULT 'E'
            iv_fallback    TYPE string OPTIONAL
  RETURNING VALUE(rv_text) TYPE string.

区别是：

  • OPTIONAL：没有传时，formal parameter 按类型得到 initial value。
  • DEFAULT：没有传时，formal parameter 使用指定的 replacement value。

这不是“少写参数”的装饰语法，而是 contract 的一部分。iv_language DEFAULT 'E' 表示默认语言就是设计决定；iv_fallback OPTIONAL 表示 fallback 是可选行为。不要把业务必需字段做成 optional，否则调用者漏传也不会在 signature 层暴露。

PREFERRED PARAMETER 允许在所有 input / input-output 参数都 optional 的情况下指定一个首选参数，使 meth( a ) 这样的短写能映射到该参数。初学阶段知道有这个语法即可，不要为了追求短写牺牲可读性。项目代码里，清楚写出 iv_status = ... 往往比短写更容易 review。

26.4.6 exceptions are also part of the signature

method signature 还可以声明异常。现代 OO ABAP 优先使用 class-based exceptions：

CLASS-METHODS divide
  IMPORTING iv_numerator     TYPE decfloat34
            iv_denominator   TYPE decfloat34
  RETURNING VALUE(rv_result) TYPE decfloat34
  RAISING   cx_sy_arithmetic_error.

实现里如果发生除零等算术错误，可以向调用者传播 cx_sy_arithmetic_error 或其子类。调用者就必须考虑：

TRY.
    DATA(lv_result) = lcl_math=>divide(
      iv_numerator   = 1
      iv_denominator = 0 ).
  CATCH cx_sy_arithmetic_error INTO DATA(lx_error).
    " error handling
ENDTRY.

旧式 EXCEPTIONS 是 non-class-based exceptions，会让调用者通过 sy-subrc 判断结果，也会限制 functional method 的使用。SAP Help 对新开发给出的方向是：推荐使用独立于单个 method 的 class-based exceptions。对本教程来说，除非在读旧代码或旧 API wrapper，否则不要把 EXCEPTIONS 当成新代码首选。并且在 ABAP for Cloud Development / ABAP Cloud 语境中，legacy EXCEPTIONS 不是“风格上不推荐”这么简单，而是应视为不可用于新开发的硬限制；应使用 class-based exceptions 或框架规定的错误返回方式。

26.4.7 signature review checklist

在日本项目 code review 中，看 method signature 时可以按六个问题快速判断。第一，这个 method 的主动作能不能从名字看出来？get_*、calculate_*、check_*、normalize_*、save_* 暗示的副作用不同。第二，主输入是否都在 IMPORTING，而且类型足够具体？如果所有参数都是 string，reviewer 很难判断合法值。第三，主结果是否清楚？一个单一结果优先 RETURNING；多个结果应考虑返回 structure，而不是堆很多 EXPORTING。第四，是否有隐藏副作用？只读名字却有 CHANGING、数据库更新、message 输出或 commit 边界，通常需要拆分或改名。第五，失败路径是否在 signature 中可见？业务错误、技术错误、not-found、invalid-input 不应只靠注释或调用方猜测。第六，能否写 ABAP Unit？如果测试必须准备真实 DB、当前用户、外部接口和系统时间，signature 可能已经把依赖藏得太深。

可以把这张表放进 review 记录：

Review questionGood signalRisk signal
What is the main intent?method name + signature 能说明一件事名字很泛，例如 process_data、do_check
Where is the input?少量 IMPORTING，类型具体过多 optional、全部 string、依赖全局变量
Where is the result?单一 RETURNING 或清楚 result structure多个 EXPORTING，调用者不知道主结果
Where is the side effect?CHANGING / save / update 在名字中可见查询式名字却修改传入对象或系统状态
How can it fail?RAISING 或框架错误返回明确abap_bool 悄悄 false，调用者容易漏判
Can it be tested?IMPORTING + RETURNING，依赖可替换直接碰 DB、权限、系统日期、外部接口

这张表不是额外格式要求，而是帮助初学者把 SAP Help 的语法规则转成项目判断。语法允许的 signature 不一定是好 signature；好 signature 应该让调用者、测试者和 reviewer 都能快速理解 contract。

26.4.8 RAP and ABAP Cloud caveat

在 RAP / ABAP Cloud 风格代码里，很多 framework method 的 signature 由 RAP behavior、接口或父类规定，开发者不能随意改。例如 handler、saver、authorization、validation、determination 等方法通常要遵守框架定义的参数形状。自定义 helper method 仍然要保持小而清楚：输入来自 RAP context，输出是明确结果、result object、reported/failed 等框架规定结构，或 class-based exception；不要用宽泛 CHANGING 去隐藏副作用。

Clean Core 角度看，method signature 也应避免泄露不稳定内部实现。对外可复用的 Global class 全局类 或 Interface 接口定义，参数类型应尽量稳定、可解释，优先使用明确的 DDIC type、released API 暴露的 type 或项目内稳定 type，而不是把临时结构和任意 string 当成长期 contract。

26.5 日本项目现场说法

常见日语：

  • メソッドシグネチャ
  • 引数
  • 戻り値
  • IMPORTINGパラメータ
  • EXPORTINGパラメータ
  • CHANGINGパラメータ
  • RETURNINGパラメータ
  • 任意パラメータ
  • デフォルト値
  • 例外クラス
  • 副作用

现场说明句：

  • このメソッドは RETURNING で単一の結果を返す形にして、テストしやすいシグネチャにしています。
  • この CHANGING パラメータは呼び出し元の値を更新するため、副作用がレビューで分かる名前にしています。
  • 必須項目なので OPTIONAL にはせず、呼び出し側で明示的に渡す設計にしています。
  • 旧式の EXCEPTIONS ではなく、例外クラスを RAISING に定義する方針です。
  • ABAP Cloud では旧式の EXCEPTIONS は新規実装の選択肢ではなく、例外クラスまたはフレームワークで定義されたエラー返却方式を使用します。
  • RAP 側のシグネチャはフレームワークで決まっているため、補助メソッド側で責務を小さく分けています。

日本项目 review 里，引数が多すぎます、副作用が分かりにくいです、戻り値で返せませんか 这类评论很常见。它们不是“代码风格洁癖”，而是在问：这个 method 的 contract 是否足够清楚。

26.7 Example / 学习例子

26.7.1 例 1：把多个输出收成一个 returning structure

初学者常写成这样：

CLASS-METHODS validate_quantity
  IMPORTING iv_quantity TYPE i
  EXPORTING ev_valid    TYPE abap_bool
            ev_message  TYPE string.

这能工作，但调用者要准备两个输出变量，且主结果不够集中。更适合教学和业务规则封装的写法，是定义一个小结果类型：

TYPES:
  BEGIN OF ty_validation_result,
    is_valid TYPE abap_bool,
    message  TYPE string,
  END OF ty_validation_result.

CLASS-METHODS validate_quantity
  IMPORTING iv_quantity      TYPE i
  RETURNING VALUE(rs_result) TYPE ty_validation_result.

调用者就可以直接断言：

DATA(ls_result) = lcl_quantity_rule=>validate_quantity( 10 ).

cl_abap_unit_assert=>assert_equals(
  act = ls_result-is_valid
  exp = abap_true ).

这类 signature 与 ABAP Unit 单元测试基础 配合很好：输入清楚、输出集中、没有隐藏副作用。

26.7.2 例 2：只有明确要原地修改时才用 CHANGING

如果 method 只是把 status 字符串规范化，CHANGING 可以接受：

CLASS-METHODS normalize_status
  CHANGING cv_status TYPE string.

METHOD normalize_status.
  CONDENSE cv_status.
  cv_status = to_upper( cv_status ).
ENDMETHOD.

调用者会看到变量被原地改写：

DATA(lv_status) = ` open `.

lcl_status_helper=>normalize_status(
  CHANGING cv_status = lv_status ).

但如果 method 名叫 get_status_text，里面却用 CHANGING 改调用者传入的 status，就会造成副作用误导。名字、方向和行为要一致。

26.7.3 例 3：异常属于 contract

一个除法 helper 可以用 RAISING 明确失败路径：

CLASS-METHODS divide
  IMPORTING iv_numerator     TYPE decfloat34
            iv_denominator   TYPE decfloat34
  RETURNING VALUE(rv_result) TYPE decfloat34
  RAISING   cx_sy_arithmetic_error.

调用者看到 signature 后就知道这不是单纯返回 0 或 abap_false 的方法，而是可能需要 TRY...CATCH：

TRY.
    DATA(lv_result) = lcl_math=>divide(
      iv_numerator   = 1
      iv_denominator = 0 ).
  CATCH cx_sy_arithmetic_error INTO DATA(lx_error).
    DATA(lv_message) = lx_error->get_text( ).
ENDTRY.

后续学习 Exception class 异常类 和 TRY CATCH exception handling 时，会继续展开异常类设计。

26.8 Counterexample / Failure mode

26.8.1 误区 1：参数越多越“通用”
CLASS-METHODS process_order
  IMPORTING iv_mode       TYPE string
            iv_status     TYPE string
            iv_user       TYPE string
            iv_date       TYPE d
            iv_flag       TYPE abap_bool
  EXPORTING ev_message    TYPE string
            ev_code       TYPE string
  CHANGING  cs_order      TYPE zorder.

这类 signature 看起来“什么都能处理”，实际很难测试，也很难 review。iv_mode、iv_flag 这种参数经常暗示 method 里藏了多个分支责任。修正方向是按业务动作拆分 method，必要时把相关输入收成结构，把主结果设计成返回结构。

26.8.2 误区 2：用 CHANGING 隐藏业务更新

如果 method 名称像查询，signature 却使用 CHANGING 修改调用者数据，日本项目 review 很容易被指出副作用不清楚。

修正方向：

  • 只读查询：使用 IMPORTING + RETURNING。
  • 明确更新：method name 写成 normalize_...、fill_...、apply_...，并在 signature 中保留 CHANGING。
  • 复杂业务更新：考虑让 method 返回结果对象或变更记录，不要让调用者猜测哪些字段被改。
26.8.3 误区 3：把 OPTIONAL 当成“以后再说”

业务必需参数不应该因为“调用方便”就写成 OPTIONAL。例如订单号、公司代码、语言、日期范围等参数，如果缺失会改变业务含义，应当在 signature 或专门校验逻辑中明确。

修正方向：只有真实可缺省的参数才用 OPTIONAL 或 DEFAULT。默认值要能被项目解释，例如默认语言、默认显示格式、默认 simulation flag。

26.8.4 误区 4：新代码继续设计 classic EXCEPTIONS

旧式 EXCEPTIONS 通过 sy-subrc 处理，读旧 SAP API 时会遇到；但新 OO ABAP helper 优先 class-based exceptions。否则 method 很难作为表达式调用，也容易让错误处理退回“忘记检查 sy-subrc”。在 ABAP Cloud 语境中，还要把它看成平台限制问题，而不只是 Clean ABAP 风格问题。

修正方向：新开发用 RAISING zcx_... 或合适的 SAP exception class；读旧 API 时，把 EXCEPTIONS 转换为明确的异常类或结果对象边界。

26.9 思考问题

  • 这个 method 的主结果应该是 RETURNING，还是确实需要多个 EXPORTING 参数？
  • CHANGING 是否真的必要？调用者是否能从 method name 看出它会修改数据？
  • 哪些参数是业务必需，哪些才适合 OPTIONAL 或 DEFAULT？
  • 这个 signature 是否暴露了不稳定的内部结构，导致以后难以重构？
  • 如果这个 method 失败，应该返回 result object、抛出 class-based exception，还是交给 RAP framework 处理？
  • 用 ABAP Unit 写测试时，是否能用一两行准备输入，并用一个清楚的返回值断言？
  • 在日本项目 review 中，如果被问 引数が多すぎませんか，应该怎样解释或拆分？
