  • Chapter 04

36.1 一句话结论

Japanese SAP deliverable documents 是「测试、质量与上线准备」中的一个核心知识点；在本教程里要学的是：理解基本設計書、詳細設計書、単体テスト仕様書、結合テスト仕様書与 ABAP 对象的关系。

在日本 SAP 项目里，代码只是交付物的一部分。一个 ABAP 对象要进入生产，通常还要能对应到設計書、テスト仕様書、移送申請（现场也常说移送依頼）、障害票、課題管理表、運用手順、リリース判定資料等文档。个别问题单可以被称为課題票，汇总管理表通常叫課題管理表。读者需要知道这些文档在项目里各自回答什么问题，以及 ABAP 开发者应该提供哪些内容和证据。

本节不是教你写所有日文文档模板，而是建立“代码和文档的映射感”：一个 CDS view、RAP behavior、OData service、background job、BAPI 调用、ATC 结果、SAP Note 处理，分别应该出现在什么设计/测试/上线资料里。这样你在日本现场就不会只说“実装完了”，而能说“設計、テスト、移送、運用引継ぎの証跡も揃っています”。

初学者最容易误解的一点是：文档不是“额外行政工作”，而是 SAP 项目中把责任、范围、验收和运维连接起来的介质。ABAP 代码回答“系统如何执行”，但项目还要回答“为什么这样设计、谁确认过、测了什么、上线后谁维护、出问题时看哪里”。这些问题如果没有落到文档，就会在测试、移送、上线判定或障害调查时反复被追问。

日本项目尤其重视 証跡。証跡 不只是截图，也包括测试数据、request/response、job log、application log、ATC result、transport request、review 记录、承认邮件、障害票处理记录。它的意义是让第三方在以后也能复现判断：当时测了什么、结果是什么、谁确认、残留风险是什么。

36.2 到底学什么

第一，要学主要文档类型。常见文档包括：要件定義書、基本設計書、詳細設計書、単体テスト仕様書、単体テスト結果、結合テスト仕様書、障害票、課題管理表、移送申請、リリース判定資料、運用手順書、引継ぎ資料。不同客户模板不同，但问题域相似。

第二，要学文档与 ABAP 对象的对应关系。比如：基本設計書说明业务流程、对象范围、外部接口、权限和异常方针；詳細設計書说明 class/method/CDS/table/API/job 的具体设计；単体テスト仕様書说明输入、前提、期待结果和 evidence；結合テスト仕様書说明跨模块、跨系统、权限和真实流程。

第三，要学证据语言。日本项目强调“証跡”。测试截图、日志、request/response、job log、application log、ATC 结果、transport request、SQL trace、SAP Note 编号、承认记录都可能成为证据。文档不是文学，而是可追溯的交付证据。

第四，要学边界沟通。ABAP 开发者不一定负责所有文档，但通常要提供技术内容、测试结果、影响说明、移送对象、障害调查结果和运维手顺。知道自己该交什么，比“我只写代码”更接近项目就绪。

第五，要学文档之间的流转。基本設計不是孤立文件，它会变成详细设计、测试规格、移送清单和运维手顺的输入。详细设计中写了 authorization check，单体测试就要有权限不足 case；设计中写了 application log，运维手顺就要说明日志对象、查询事务或监控路径；设计中写了 background job，リリース資料就要说明 job 登録、実行タイミング、失敗時対応。

第六，要学“模板不同，问题相同”。不同客户、SIer、PMO 的文档名称和列名可能不同：有的叫外部設計/内部設計，有的叫基本設計/詳細設計，有的把単体テスト写成 UT，有的把結合テスト写成 IT 或 SIT。学习时不要死背模板，而要记住每类文档回答的问题。

36.3 为什么重要

日本 SAP 项目经常通过文档来管理责任、范围和验收。代码写完但设计书没更新，测试结果没有证据，移送对象没列清，运维不知道怎么查日志，这些都会被视为交付不完整。

文档还能保护团队。出现障害时，如果有设计说明、测试证据、transport 记录和运用手顺，团队能快速判断问题是设计遗漏、实现缺陷、配置错误、数据问题、权限问题还是标准限制。没有文档，所有人只能靠记忆和口头说明。

对 Java/Python 背景学习者来说，这类似 design doc、test plan、runbook、release note、incident report、change request 的组合。但 SAP 项目的特殊点是：这些文档会和系统 landscape、transport、client、business process、Basis 运维、客户承认和外部审计强绑定。

这也解释了为什么“代码已经 merge”在 SAP 项目里常常不够。ABAP 对象还要被激活、打包、移送，从 DEV 到 QAS 再到 PRD；测试数据可能受 client、company code、role 和 customizing 影响；上线可能需要 Basis 安排窗口、导入顺序、job 停止/再開、外部系统切换和回退判断。文档把这些非代码条件显式化。

对学习者来说，掌握文档并不是为了变成 PMO，而是为了减少现场沟通摩擦。能够把自己的 ABAP 变更翻译成设计影响、测试证据、移送对象和运维手顺，说明已经从“写程序的人”接近“能参与交付的人”。

36.4 Key mechanism / 核心机制

可以把常见文档和 ABAP 证据这样对应。

基本設計書回答“做什么、为什么、影响哪里”。它通常包含业务流程、对象范围、接口方针、权限方针、错误处理方针、性能/数据量假设、Clean Core 方针、外部系统和运维要求。ABAP 开发者要能把 RAP/OData/job/report/interface 的设计意图讲清楚。

詳細設計書回答“怎么做”。它会落到 package、class、method、CDS、table、behavior、service definition/binding、BAPI/RFC、job、application log、message、authorization check、commit/rollback、exception handling 等。这里要避免只贴代码，要说明输入输出、处理顺序、异常路径和依赖对象。

単体テスト仕様書回答“一个对象自己是否正确”。它通常写前提条件、输入、步骤、期待结果、实际结果、证据截图/日志、测试数据。对 ABAP 来说，既可以是手动测试，也可以结合 ABAP Unit、service preview、HTTP 请求、job log 和 application log。

結合テスト仕様書回答“跨对象/跨系统/跨模块是否正确”。例如 OData service 被 Fiori 调用、background job 更新后 API 可查、BAPI 调用后外部系统收到结果、权限角色切换后行为正确。这里要特别关注接口、权限、数据量、错误处理和重试。

移送申請/リリース判定資料回答“能不能上线”。它要列 transport request、对象清单、依赖顺序、前置/后置作业、ATC 结果、测试结果、未解決課題、残リスク、回退方案和承认者。

運用手順書/引継ぎ資料回答“上线后怎么维护”。它要说明 job 监控、application log 查询、接口错误重试、SAP Note/incident 处理、常见错误、联系人、SLA 和升级路径。

把这些文档串起来，可以形成一条交付链：

要件定義
  -> 基本設計
  -> 詳細設計
  -> 実装
  -> 単体テスト
  -> 結合テスト
  -> 移送申請 / リリース判定
  -> 本番反映
  -> 運用引継ぎ
  -> 障害 / 変更管理

ABAP 开发者在这条链里的责任，通常不是“全部文档自己写完”，而是确保技术事实准确进入相应文档。例如：

ABAP 事实应进入的文档为什么
新增 CDS view / DCL詳細設計、単体テスト、移送申請影响数据模型、权限、移送对象
新增 RAP action基本設計、詳細設計、結合テスト影响业务动作、API 契约、权限
新增 background job基本設計、詳細設計、運用手順影响运行时间、失败处理、监控
使用 BAPI / RFC / 外部 API基本設計、結合テスト、運用手順影响接口边界、重试、错误调查
ATC exemptionリリース判定、品質管理資料影响上线风险和承认
SAP Note 对应障害票、変更記録、引継ぎ資料说明原因、修正依据、后续维护

还有一个很实用的判断：如果某个技术点上线后可能被问“为什么这样做”“怎么证明测过”“出错看哪里”，它就不应该只留在代码里。它至少要进入设计、测试或运维材料中的某一处。

交付文档也有粒度问题。不是所有小 method 都值得写一页详细设计，但所有对业务、接口、权限、数据、性能、上线、运维有影响的点，都应被可追溯地记录。好的文档不是越厚越好，而是让读者能从业务要求一路追到对象、测试、移送和运维证据。

36.5 日本项目现场说法

常见日语包括：要件定義書、基本設計書、詳細設計書、単体テスト仕様書、単体テスト結果、結合テスト仕様書、障害票、課題管理表、移送申請、リリース判定、運用手順書、引継ぎ資料、証跡。

设计交付中可以这样说：

  • 基本設計では業務フロー、権限、エラー処理、外部IF、運用方針を整理します。
  • 詳細設計ではABAPオブジェクト、処理順序、例外処理、ログ出力、移送対象を明確にします。
  • 単体テスト結果には、入力データ、期待結果、実行結果、ログまたは画面証跡を残します。

上线判定中可以这样说：

  • 移送対象、テスト結果、未解決課題、残リスク、切り戻し手順を確認したうえでリリース可否を判断します。
  • 運用引継ぎとして、ログ確認手順、ジョブ監視、障害時の一次切り分け手順を共有します。

日常沟通还会经常用到这些表达：

  • 詳細設計書の該当箇所を更新し、レビュー指摘を反映済みです。
  • 単体テスト結果には、正常系・異常系・権限エラーの証跡を添付しています。
  • 移送依頼には、関連する DDIC、クラス、サービス定義、ロール関連オブジェクトを含めています。
  • 運用手順書には、ジョブログ、アプリケーションログ、再実行可否の確認手順を追記しています。
  • 残課題と残リスクはリリース判定資料に明記し、承認者に確認済みです。

现场语感上，エビデンス 和 証跡 都常见。証跡 更偏正式交付/审计语境，エビデンス 更偏测试截图、日志附件、结果证明。写文档时可以按客户习惯使用，但要避免只说“エビデンスあります”而不说明具体文件、路径、case ID 和确认者。

36.6 English Terms

  • Japanese project terms: 基本設計書, 詳細設計書, 単体テスト仕様書, 結合テスト仕様書, 移送申請, リリース判定, 運用手順書, 障害票, 課題管理表, 証跡.
  • English equivalents: requirements specification, basic design, detailed design, unit test specification, integration test specification, release readiness, runbook, incident report, issue log, evidence.
  • SAP-related terms: transport request, ATC result, application log, test evidence, go-live readiness, cutover, production readiness.
  • Recommended search pattern: SAP Activate deliverables, SAP project test evidence, SAP go-live readiness, SAP transport release documentation.

36.7 Example / 学习例子

以 AC0567C batch upload API 为例，同一个开发对象可以映射到不同文档：

基本設計書:
  - 业务目标：批量上传并查询处理状态
  - 相关角色：上传用户、查询用户、运维用户
  - 外部接口：HTTP API / 未来 S/4HANA API
  - 错误方针：业务错误返回 payload，技术错误记录 application log

詳細設計書:
  - package / class / method / table / service definition
  - request/response structure
  - transaction boundary
  - authorization check
  - application log object/subobject
  - exception handling

単体テスト仕様書:
  - 正常上传
  - 空文件
  - 重复请求
  - 权限不足
  - 部分失败
  - application log evidence

結合テスト仕様書:
  - 前端或外部工具调用 API
  - 后台 job 更新状态
  - 不同角色查询
  - 大数据量场景

リリース判定資料:
  - transport request
  - ATC result
  - unresolved issues
  - rollback / stop job procedure
  - operation handover

这样写，读者就能看到“代码对象”和“项目文档”不是两套世界，而是一组互相证明的交付证据。

再看一个开发者真正要提交的“最小交付包”示意：

Feature:
  AC0567C batch upload API - duplicate request handling

Design:
  基本設計書: retry / duplicate prevention policy
  詳細設計書: idempotency key, status table, RAP save boundary

Build:
  Transport request: CDS, class, behavior, service, message/log objects
  ATC result: no critical findings or approved exemption

Test:
  単体テスト: same request twice, failed retry, no partial save
  結合テスト: external client timeout and resend
  Evidence: request/response, DB count, application log

Release:
  リリース判定: remaining risks, rollback/stop procedure
  運用手順: how to check batch status and log

这个示意的重点是“同一事实在不同文档中的形态不同”。幂等设计在基本設計里是方针，在详细设计里是字段和处理顺序，在测试规格里是 case，在 evidence 里是 request/response 和 DB 件数，在运维手顺里是重试与日志确认方法。

如果换成一个 CDS view，映射也会变化：

基本設計書:
  说明它支持哪个业务报表或 API 查询。

詳細設計書:
  写 base table/view、association、field alias、DCL、annotation。

単体テスト:
  验证选择条件、字段含义、权限过滤、边界数据。

結合テスト:
  验证它被 RAP/OData/Fiori/外部查询消费时是否符合预期。

移送申請:
  列出 DDL source、DCL source、metadata extension、依赖对象。

这样读者就能把本教程前面学过的 CDS view entity 视图实体、CDS Access Control (DCL) 行级访问控制、Service binding 服务绑定 放到交付语境中。

36.8 Counterexample / Failure mode

常见失败模式如下（约十余类）。

第一，文档只复制代码。设计书应该解释业务、边界、异常、依赖和证据，不是代码粘贴板。

第二，详细设计没有异常路径。正常路径写得很清楚，但权限不足、重复请求、rollback、job 失败、接口超时没有说明。

第三，测试规格没有期待结果。只有“执行某功能”，没有输入、前提、期待值和证据，无法验收。

第四，测试结果没有証跡。上线后无法证明当时测过什么、谁确认、结果如何。

第五，移送资料没有依赖顺序。DDIC、class、service、role、job、configuration、number range、communication setting 可能互相依赖。

第六，运维手顺缺失。上线后出错，运维不知道看 application log、job log、接口 monitor 还是 SAP Note。

第七，障害票只写现象不写原因。缺少再现条件、影响范围、一次切り分け、暫定対応、恒久対応。

第八，不更新文档。代码变了，设计/测试/运维资料没变，后续维护者会被旧信息误导。

第九，文档之间互相矛盾。基本設計写“失败可重试”，详细设计没有状态字段，测试规格没有重试 case，运维手顺又说“失败后联系开发”。这种不一致比文档缺失更危险。

第十，只保留个人本地文件。证据如果不在项目约定路径、测试管理工具、ticket 或共享目录中，别人就无法复核。

第十一，截图很多但没有结论。截图需要配 case ID、输入、期待结果、实际结果、判定、执行者和日期，否则只是图片堆积。

第十二，移送资料只写 request number，不写对象和依赖。上线窗口里一旦导入失败，团队很难判断缺哪个对象、是否能跳过、是否要回退。

第十三，运维手顺没有“不能做什么”。例如错误重跑、手动改表、跳过 authorization、重复导入、直接删除队列，都可能造成二次事故。手顺不仅要写操作，也要写禁止事项和升级路径。

  • 用中文说明基本設計書、詳細設計書、単体テスト仕様書、結合テスト仕様書的区别。
  • 为一个 ABAP service 写一张“代码对象到文档章节”的映射表。
