Chapter 02 - Core ABAP Programming

33. Exception class 异常类

  • Chapter 02
  • Section 33
  • self-contained study note

33.1 一句话结论

Exception class 异常类是 ABAP class-based exception 的类型。它不是单纯的“报错文本”，而是一个 exception object：一边作为控制流信号告诉运行时“正常路径中断了”，一边作为错误 payload 携带 exception type、message text、业务 key、字段名、上一层异常等上下文。一个好的异常类会把“可能失败”变成 method signature 和调用链上的显式 contract：哪里会 raise、谁应该 catch、错误里带什么上下文、是否必须由调用者处理。

现代 ABAP 新开发应优先使用 class-based exceptions，而不是旧式 function module EXCEPTIONS / sy-subrc 风格来表达复杂错误。实际项目里仍会遇到 classic / non-class-based exception，例如 function module 的 EXCEPTIONS、早期 OO method 的 classic exception、以及依赖 sy-subrc 的调用模式；学习者要能读懂和迁移这些旧代码。但在 ABAP Cloud / Clean Core 语境下，新代码应按 class-based exception 设计，不要把旧式 exception 继续扩散到新的接口里。

初学者最重要的判断是：异常类不是“让程序报错更高级”，而是让可预见错误有类型、有边界、有处理责任。业务上可恢复、调用方应该处理的错误，适合清楚的 checked exception；程序设计错误、不可恢复状态、框架级异常，则要谨慎选择 unchecked 或系统 exception。不是所有业务失败都应该抛 exception，尤其在 RAP validation/message 场景里，常常应返回 reported/failed message，而不是随手 dump。

33.2 到底学什么

上一节 Polymorphism 多态 讲了不同对象可以通过同一 contract 执行不同实现。本节讲另一个 contract：错误路径。一个 method 不只要说明“正常返回什么”，也要说明“失败时如何表达”。

本节要掌握八个边界：

  • class-based exception 都是 exception object，继承树的根类是 CX_ROOT。
  • 自定义异常类不能直接继承 CX_ROOT，必须继承 CX_STATIC_CHECK、CX_DYNAMIC_CHECK 或 CX_NO_CHECK 这三类之一。
  • customer global exception class 通常使用 ZCX_... / YCX_... 命名；SAP 交付的异常类常见 CX_...、系统异常常见 CX_SY_...。
  • CX_STATIC_CHECK 适合调用方应该显式处理或声明传播的可恢复错误；调用链要 catch 或继续在 RAISING 中声明。
  • CX_DYNAMIC_CHECK 可以写在 RAISING 中，但声明不是强制的；如果未声明却传播出 procedure，运行时可能以错误方式暴露。
  • CX_NO_CHECK 不能写进 RAISING，不会被静态/运行时声明检查强制要求；未捕获时仍会一路传播，最终可能 short dump，所以不是“可以忽略”。
  • method signature 里的 RAISING zcx_... 是错误 contract，不是装饰；改变它等于改变调用接口。
  • RAISE EXCEPTION TYPE zcx_... EXPORTING ... 会创建并抛出 exception object，EXPORTING 传入 constructor 参数、业务上下文或 textid。
  • TRY ... CATCH ... ENDTRY 是处理 class-based exception 的基本控制结构；CATCH 顺序、异常传播和 CLEANUP 放到下一节 TRY CATCH exception handling。

最小例子：

CLASS zcx_order_invalid DEFINITION
  INHERITING FROM cx_static_check.
ENDCLASS.

CLASS lcl_order_checker DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS check_quantity
      IMPORTING iv_quantity TYPE i
      RAISING   zcx_order_invalid.
ENDCLASS.

CLASS lcl_order_checker IMPLEMENTATION.
  METHOD check_quantity.
    IF iv_quantity <= 0.
      RAISE EXCEPTION TYPE zcx_order_invalid.
    ENDIF.
  ENDMETHOD.
ENDCLASS.

check_quantity 的 signature 告诉调用方：这个 method 可能因为 order quantity 不合法而失败，而且这个失败是 zcx_order_invalid。调用方不能只看 RETURNING；也要看 RAISING。

换句话说，RAISING zcx_order_invalid 是可读、可检查、可测试的错误 contract。

33.3 为什么重要

没有清楚异常类时，ABAP 代码常见三种问题。

第一种是错误靠 sy-subrc 或空返回值暗示。调用方忘记判断，程序继续跑，后面才出现更难查的错误。

第二种是到处 CATCH cx_root 然后什么都不做。表面上程序“不报错”，实际把问题藏起来，形成日本项目 review 常说的 例外握りつぶし。

第三种是异常类太泛，例如所有错误都抛 zcx_general_error。调用方无法区分权限问题、数据不存在、配置缺失、外部系统失败，只能写大而全兜底。

好的 exception class 让项目多三件东西：

  • 可读性：method signature 明确说出错误类型。
  • 可测试性：ABAP Unit 可以验证指定输入是否 raise 指定 exception。
  • 可交付性：日志、message、T100、RAP reported message 或上层错误处理可以按 exception type 做清楚转换。

对 Java/Python 背景学习者来说，ABAP exception class 有一点像 Java checked/unchecked exception，但不要直接套 Java 经验。ABAP 的 CX_STATIC_CHECK / CX_DYNAMIC_CHECK / CX_NO_CHECK 是 SAP 自己的分类；是否写在 RAISING、是否强制调用方处理、是否允许传播，都要按 ABAP 规则理解。尤其要注意：CX_NO_CHECK 不是“没有错误”，而是“语言不要求每层声明”；真正没有被捕获时，风险仍会在运行时集中爆发。

33.4 Key mechanism / 核心机制

33.4.1 CX_ROOT is the root

class-based exception 的根类是 CX_ROOT。CX_ROOT 提供所有异常对象共有的基础能力，例如通过 get_text( ) 取得异常文本、通过 get_longtext( ) 取得长文本、通过 get_source_position( ) 取得源代码位置，以及通过 previous 连接被包装的上一层异常。

但自定义 exception class 不能直接 INHERITING FROM cx_root。项目里创建自己的异常类时，要选择下面三条分类之一：

  • CX_STATIC_CHECK
  • CX_DYNAMIC_CHECK
  • CX_NO_CHECK

初学阶段不用记住所有 SAP 标准异常类，但要认识这些根类。看到 ZCX_... / YCX_... 时，通常是在 customer namespace 里定义的全局异常类；看到 CX_SY_... 时，常常是 SAP 系统运行时异常，例如类型转换、算术、内表访问等问题。

看到 CATCH cx_root 时，要立刻问：这里是不是过宽？有没有吞掉具体原因？是否记录、重抛或转换了上下文？CATCH cx_root 本身不是非法语法，但在生产业务代码里空 catch 是高风险反模式。

33.4.2 static check exception is explicit contract

CX_STATIC_CHECK 适合“调用方应该知道并处理”的可恢复错误。例如输入业务对象不合法、必须由上层给用户可理解反馈的校验错误。

CLASS zcx_price_missing DEFINITION
  INHERITING FROM cx_static_check.
ENDCLASS.

CLASS lcl_price_service DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS get_price
      IMPORTING iv_product_id TYPE string
      RETURNING VALUE(rv_price) TYPE decfloat34
      RAISING   zcx_price_missing.
ENDCLASS.

这里 RAISING zcx_price_missing 是 contract。调用方读 signature 就知道：不是所有 product 都一定能拿到 price；如果缺 price，上层要处理。这个处理可能是 TRY...CATCH，也可能是继续往上 RAISING 传播。对 CX_STATIC_CHECK 而言，这种“处理或声明传播”的意识是它的核心价值；ADT、语法检查和 ATC 会把这类 contract 暴露给开发者。

33.4.3 dynamic and no-check exceptions are not shortcuts

CX_DYNAMIC_CHECK 和 CX_NO_CHECK 容易被误用成“少写 signature 的方便工具”。这不对。

  • CX_DYNAMIC_CHECK 可以在 RAISING 中声明，也可以不声明；但如果没有声明、没有捕获、又传播出当前 procedure，运行时层面仍可能暴露为错误。
  • CX_NO_CHECK 不能写在 RAISING 中，调用方也不会被迫逐层声明；但如果最终没有任何地方捕获，仍会导致 runtime error / short dump。

它们都不是“偷懒不写错误处理”。选择这两类时，要能解释为什么调用方不应该被每一层强制处理，以及最终在哪里会记录、转换或终止。写书稿时可以把它们理解成“检查强度不同”，不要理解成“严重程度不同”：轻微错误也可能需要 CX_STATIC_CHECK，严重框架错误也可能是 unchecked。

33.4.4 RAISING belongs to method signature

前面 Method signature 方法签名 讲过，method signature 不只是参数列表。RAISING 是 signature 的一部分，并且只用于 class-based exceptions 中需要声明的异常类型。通常应在这里声明 CX_STATIC_CHECK 和需要显式告知调用方的 CX_DYNAMIC_CHECK 子类；CX_NO_CHECK 子类不能声明在 RAISING 中。

METHODS post_document
  IMPORTING is_document TYPE zdoc_header
  RAISING   zcx_document_invalid
            zcx_posting_failed.

这段签名表达两个不同错误：

  • zcx_document_invalid：输入 document 不满足业务校验。
  • zcx_posting_failed：真正 posting 动作失败，可能是数据库、锁、配置或外部依赖问题。

不要把它们都压成一个 abap_bool 返回值，也不要都塞进一个 zcx_error。错误类型越清楚，上层处理越稳定。对外暴露的 method 一旦改变 RAISING 列表，调用方、测试、接口说明和 review 都要重新确认；这和改变 IMPORTING / RETURNING 参数一样，是 contract 变化。

33.4.5 exception object can carry context

exception class 是 class，所以可以有 attributes、constructor、text/message 绑定。项目里常见做法是让异常携带业务 key、字段名、外部返回码或 message context。抛出时使用 RAISE EXCEPTION TYPE ... EXPORTING ...，这些 EXPORTING 参数会传给 exception class 的 constructor。

初学阶段先记一个原则：exception 里要带“上层处理需要知道的上下文”，不要带大对象、数据库连接、临时内部表等实现细节。比如：

  • 合适：sales_order_id、item_no、field_name、api_status_code
  • 不合适：完整订单大对象、数据库游标、未脱敏 payload、临时 debug text

捕获时常见写法是 CATCH zcx_... INTO DATA(lx_error)，然后调用 lx_error->get_text( ) 取得可读错误文本。需要保留底层异常时，使用 previous 把原始 exception object 接到新异常上，避免“包装后丢失根因”。

33.4.6 exception class and T100 message

ABAP 项目常把 exception class 和 message class / T100 message 结合，让异常既有类型，也有可本地化 message text。这个主题会在 Message class and T100 messages 继续展开。

本节先建立边界：exception class 是错误类型和传播机制；T100/message 是用户或日志可读文本。二者可以结合，但不是同一个概念。常见机制是 exception class 实现 IF_T100_MESSAGE 这类 message interface，通过 textid 选择具体 message text，再用 constructor 参数填充变量；包装 classic message 时也可能看到 IF_T100_DYN_MSG 这类把 SY / SYST message 字段转成异常文本的机制。也就是说，exception class hierarchy 决定“谁能 catch 它”，textid / T100 决定“它显示什么文本”，二者是两个不同维度。

33.4.7 catch specific, not only cx_root

最小处理形式：

TRY.
    DATA(lv_price) = price_service->get_price( iv_product_id ).
  CATCH zcx_price_missing INTO DATA(lx_missing).
    DATA(lv_text) = lx_missing->get_text( ).
    " create user-facing message, log entry, or test assertion
ENDTRY.

先 catch 具体 exception class，能处理就处理。多个 exception class 可以放在同一个 CATCH 中；如果有多条 CATCH，顺序必须从具体到通用。把 superclass 放在 subclass 前面会遮蔽后面的具体处理，在相关语法检查中会被指出。

CATCH cx_root 可以作为兜底，但必须记录足够上下文、转换为上层可理解错误、或重抛。它在 ABAP Unit setup 等测试基础设施里也可能是合理写法，例如捕获任意异常后 cl_abap_unit_assert=>fail( lx_root->get_text( ) )。真正要避免的是生产代码里只写 CATCH cx_root. 后空处理，这才是典型反模式。

33.4.8 exception vs normal business result

不是所有“业务上不成立”都应该变成 exception。

  • 用户输入非法、必须中断当前动作：可以考虑 exception 或 RAP validation message。
  • 查找结果为空且业务允许为空：通常返回 empty table / initial value / optional result 更自然。
  • 外部系统暂时不可用：可能是 exception，并且要带外部状态和重试策略。
  • RAP validation 发现字段错误：通常应通过 reported-<entity> 和 failed-<entity> 返回给框架，而不是直接抛出会变成 dump 的异常。

判断标准是：调用方是否应该把它当作异常路径处理？用户或上层框架是否需要结构化错误？如果只是普通分支，不要用 exception 代替业务流程设计。

在 RAP 里也不是“完全不用 exception”。业务 validation 的结果通常走 reported / failed，而框架级、基础设施级、保存序列相关的失败仍可能由 class-based exception 表达。写 Clean Core / ABAP Cloud 代码时，要确认使用的异常类是否是当前系统允许的 released API，并避免把未释放实现细节暴露到自建接口。

33.4.9 error expression decision checklist

设计 method 或 RAP handler 时，可以先决定“这个失败应该怎么表达”，再决定要不要创建异常类。下面的表可以作为 review 时的最小判断：

SituationPreferWhy
调用方必须处理或声明传播的可恢复错误CX_STATIC_CHECK 子类，写入 RAISING错误是接口 contract 的一部分，调用链不能假装不知道。
框架或基础设施层可能抛出的运行时异常具体 class-based exception，必要时包装并保留 previous保留底层原因，同时给上层更清楚的业务语义。
普通“没有找到”且业务允许为空empty table、initial value、optional result这是正常分支，不应让调用方用异常控制流程。
RAP 字段校验或业务规则不满足reported / failed messageFiori / OData 调用方需要结构化消息、字段定位和实例定位。
后台处理需要运维排查exception + Application Log 应用日志 evidenceexception 负责中断或传播，Application Log 负责可追踪证据。
程序设计错误、不可恢复状态谨慎使用 unchecked exception 或让系统异常暴露这类错误通常不是业务调用方能恢复的场景。

一个实用的命名和责任原则是：异常类名应该说明“失败的业务含义”，而不是只说明“哪里抛出”。例如 zcx_order_invalid 比 zcx_error_in_class_a 更容易被调用方理解；constructor 参数也应携带 order id、field name、external status 这类可诊断上下文，而不是把一整段调试文本塞进 message。

33.5 日本项目现场说法

  • 常见日语：例外クラス、クラスベース例外、非クラスベース例外、例外送出、例外捕捉、例外伝播、再送出、例外握りつぶし、異常系、エラー契約、メッセージクラス。
  • 设计说明句：このメソッドは入力チェックエラーを zcx_document_invalid として送出し、呼び出し側でメッセージに変換します。
  • 命名说明句：顧客作成のグローバル例外クラスは ZCX_ または YCX_ プレフィックスで定義します。
  • Review 指摘句：CATCH cx_root で例外を握りつぶしているため、具体的な例外クラスで捕捉し、ログ出力または再送出を行ってください。
  • Review 精确句：CATCH cx_root 自体が問題ではなく、空の CATCH ブロックで例外を握りつぶすことが問題です。
  • Signature 指摘句：このエラーは呼び出し側で処理すべきため、RAISING 句に明示してください。
  • RAP 指摘句：RAP の入力チェック結果は例外ではなく reported / failed によるメッセージ返却として扱うべきです。
  • 使用场景：method signature review、单体测试、错误消息设计、RAP validation、障害調査、日志设计、上线前异常路径检查。

日本项目里 異常系 不只是测试用语，也影响详细设计。异常类设计清楚时，異常系テストケース 可以直接对应 exception type、message、日志和恢复动作。

33.6 English Terms

  • Official / search terms: class-based exception, non-class-based exception, exception class, CX_ROOT, CX_STATIC_CHECK, CX_DYNAMIC_CHECK, CX_NO_CHECK
  • Related syntax: RAISING zcx_..., RAISE EXCEPTION TYPE zcx_..., TRY, CATCH, ENDTRY, CLEANUP
  • Design terms: checked exception, unchecked exception, exception contract, error payload, error propagation, rethrow, previous, message class, T100 message, exception swallowing
  • SAP project terms: reported, failed, RAP validation, business error, technical error
  • 读官方资料时注意：exception 可能指 class-based exception、classic non-class-based exception、runtime error、short dump、message type 等不同层级。本节聚焦 class-based exception classes。

33.7 Example / 学习例子

33.7.1 例 1：校验错误异常
CLASS zcx_quantity_invalid DEFINITION
  INHERITING FROM cx_static_check.
ENDCLASS.

CLASS lcl_quantity_rule DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS check
      IMPORTING iv_quantity TYPE i
      RAISING   zcx_quantity_invalid.
ENDCLASS.

CLASS lcl_quantity_rule IMPLEMENTATION.
  METHOD check.
    IF iv_quantity <= 0.
      RAISE EXCEPTION TYPE zcx_quantity_invalid.
    ENDIF.
  ENDMETHOD.
ENDCLASS.

这是适合 CX_STATIC_CHECK 的例子：调用方应该知道 quantity 可能不合法，并决定如何反馈给用户或上层流程。

33.7.2 例 2：调用方处理
TRY.
    quantity_rule->check( iv_quantity = -1 ).
  CATCH zcx_quantity_invalid INTO DATA(lx_quantity).
    DATA(lv_text) = lx_quantity->get_text( ).
    " convert to message, log, or test assertion
ENDTRY.

下一节 TRY CATCH exception handling 会展开 catch 顺序、异常传播和兜底处理。本节只要理解：异常类给错误路径命名，CATCH 按类型处理。

33.7.3 例 3：抛出时携带上下文

实际项目中的 exception class 往往有 constructor 参数，用来填充 attribute 或 message 变量。写法形态如下：

RAISE EXCEPTION TYPE zcx_quantity_invalid
  EXPORTING
    iv_quantity = iv_quantity.

如果异常类绑定了 T100/message，常见形态还会通过 textid 选择具体文本：

RAISE EXCEPTION TYPE zcx_quantity_invalid
  EXPORTING
    textid      = zcx_quantity_invalid=>quantity_must_be_positive
    iv_quantity = iv_quantity.

这里的重点不是背代码模板，而是理解：异常对象应该带足够上下文，让上层能生成 message、日志或测试断言。捕获者再通过 lx_error->get_text( ) 或公开属性读取这些信息。

33.7.4 例 4：包装底层异常时保留 previous
TRY.
    external_gateway->send( is_payload ).
  CATCH cx_root INTO DATA(lx_root).
    RAISE EXCEPTION TYPE zcx_posting_failed
      EXPORTING
        previous = lx_root.
ENDTRY.

这类写法适合把技术异常转换为业务层可理解的异常，同时保留原始根因。不要把底层异常吞掉后只抛一个没有上下文的新异常。

33.7.5 例 5：不要吞掉异常
TRY.
    poster->post_document( document ).
  CATCH cx_root.
ENDTRY.

这段代码危险，因为它把所有 class-based exception 都吞掉了。用户看不到错误，日志也可能没有证据，后续数据状态更难解释。

更好的方向：

TRY.
    poster->post_document( document ).
  CATCH zcx_document_invalid INTO DATA(lx_invalid).
    " convert to user-facing validation message
  CATCH zcx_posting_failed INTO DATA(lx_failed).
    " log technical context and stop the posting flow
ENDTRY.

33.8 Counterexample / Failure mode

33.8.1 误区 1：所有错误都返回 abap_false

返回 abap_false 只说明失败，不说明为什么失败、谁该处理、是否可恢复。

修正动作：对需要上层处理的明确错误，设计 exception class 或结果对象；不要只靠布尔值隐藏错误原因。

33.8.2 误区 2：自定义异常直接继承 CX_ROOT

CX_ROOT 是 class-based exception 继承树的根，但不是自定义异常类的直接父类。

修正动作：按处理策略继承 CX_STATIC_CHECK、CX_DYNAMIC_CHECK 或 CX_NO_CHECK。

33.8.3 误区 3：异常类太泛

zcx_general_error 到处抛，上层只能 CATCH zcx_general_error，无法分支处理。

修正动作：按业务语义和处理策略拆分，例如 zcx_document_invalid、zcx_price_missing、zcx_posting_failed。

33.8.4 误区 4：滥用 CX_NO_CHECK

为了少写 RAISING 全部继承 CX_NO_CHECK，会让错误 contract 从 signature 中消失。

修正动作：默认先问“调用方是否应该知道并处理”；如果是，优先考虑 CX_STATIC_CHECK。

33.8.5 误区 5：CATCH cx_root 后不处理

这是典型 例外握りつぶし。它可能让生产问题失去证据。

修正动作：catch 具体异常；兜底时记录上下文、转换或重抛。测试代码里可以用 CATCH cx_root INTO DATA(lx_root) 后立即 cl_abap_unit_assert=>fail( lx_root->get_text( ) )，这和生产代码空 catch 不是一回事。

33.8.6 误区 6：把 exception 当普通业务分支

如果“没找到数据”是业务允许的正常结果，直接抛 exception 可能让流程过重。

修正动作：区分 normal absence、validation error、technical failure；不同路径用不同表达。

33.8.7 误区 7：异常里放敏感或过大的上下文

异常对象可能进入日志、dump、trace 或消息转换。不要随便塞完整 payload 或个人信息。

修正动作：只放处理需要的业务 key、字段名、状态码；敏感信息脱敏。

33.8.8 误区 8：在新接口里继续扩散 classic exception

function module EXCEPTIONS 和 sy-subrc 风格在老代码中常见，但不适合作为新 OO / RAP / Cloud-ready 接口的主要错误 contract。

修正动作：读旧代码时识别 classic exception；写新代码时用 class-based exception 或 RAP structured message contract。

33.9 思考问题

  • 这个错误是调用方必须处理的可恢复错误吗？
  • 应该继承 CX_STATIC_CHECK、CX_DYNAMIC_CHECK，还是 CX_NO_CHECK？
  • 这个自定义异常类是否错误地想直接继承 CX_ROOT？
  • method signature 是否用 RAISING 清楚表达错误 contract？
  • 这个异常能否写进 RAISING？如果是 CX_NO_CHECK，为什么不应该写？
  • 上层应该 catch 具体 exception，还是继续传播？
  • CATCH 顺序是否从具体到通用？
  • 这个 exception 需要携带哪些上下文？哪些不应该放进去？
  • 是否需要用 previous 保留底层异常？
  • 是否需要转换为 T100/message class、RAP reported message 或日志？
  • 有没有 CATCH cx_root 后空处理的 例外握りつぶし？
  • 这个业务失败是 exception，还是普通返回结果/validation message 更合适？
