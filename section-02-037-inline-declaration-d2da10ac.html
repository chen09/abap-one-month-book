37. Inline declaration 内联声明

  • Chapter 02
  • Section 37
  • self-contained study note

37.1 一句话结论

Inline declaration 内联声明 是 Modern ABAP 里把变量声明放在“第一次需要它的 operand position / declaration position”上的写法。它的目标不是让 ABAP 变成动态类型语言，而是在编译期能静态推断类型的地方，用 DATA(...)、@DATA(...)、FIELD-SYMBOL(...) 或 FINAL(...) 减少样板代码，让变量更靠近使用点。

最常见的例子是：

DATA(total_amount) = price * quantity.

READ TABLE lt_items INTO DATA(ls_item)
     WITH KEY item_no = item_no.

LOOP AT lt_items ASSIGNING FIELD-SYMBOL(<ls_item>).
  <ls_item>-status = 'DONE'.
ENDLOOP.

SELECT FROM zorder
       FIELDS order_id, status
       WHERE status = @open_status
       INTO TABLE @DATA(lt_open_orders).

这些写法看起来像“临时变量随手创建”，但本质仍是静态类型 ABAP：编译器必须能从右侧表达式、method signature、table row type、SQL result 或语句规则推断出类型。推断不清楚、业务语义重要、金额/数量/日期/key 类型需要明确时，显式 DATA ... TYPE ... 仍然是更好的写法。

37.2 到底学什么

本节要掌握八个点：

  • inline declaration 的关键词是 declaration position：只有语法规则允许、且类型能静态推断的位置，才能写内联声明。
  • DATA(var) 声明普通 data object，常见于赋值、method call、READ TABLE ... INTO DATA(...)、LOOP AT ... INTO DATA(...)、CATCH ... INTO DATA(...) 等位置。
  • @DATA(var) 是 ABAP SQL 中常见写法；@ 是 host variable / host expression 语境的标记，DATA(...) 仍然是 inline declaration。
  • FIELD-SYMBOL(<fs>) 声明 field symbol，常见于 ASSIGNING FIELD-SYMBOL(<row>)、ASSIGN ... TO FIELD-SYMBOL(<fs>)，语义是引用/别名式访问，不是普通复制。
  • FINAL(var) 是 immutable variable 的 inline declaration；初始化后再次写入会在语法检查阶段被拦下。如果当前上下文后续不应再改这个变量，FINAL(...) 往往比 DATA(...) 更能表达 intent。
  • inline declaration 不等于动态类型，也不等于省略业务建模。变量仍有静态类型，只是类型由编译器从上下文推断。
  • inline declaration 能减少模板代码，但过度使用会让类型来源、生命周期和业务语义变得不清楚。
  • 日本项目 review 里，重点不是“能不能写短”，而是“型推論是否清楚、可読性是否高、障害調査时是否容易追踪”。

本节的最低目标是：看到 DATA(...) / @DATA(...) / FIELD-SYMBOL(...) 时，能说出它从哪里推断类型；自己写代码时，能判断内联声明和显式声明哪个更适合当前场景。

37.3 为什么重要

传统 ABAP 代码经常先在 method 或 form 开头写一大段变量声明：

DATA lv_total TYPE p LENGTH 16 DECIMALS 2.
DATA ls_item  TYPE zorder_item.
DATA lt_items TYPE STANDARD TABLE OF zorder_item WITH EMPTY KEY.

然后几十行之后才看到这些变量第一次被使用。对初学者来说，这种写法的问题不是“旧”，而是读者要在声明区和使用区之间来回跳：这个变量从哪里来、是什么类型、为什么需要、是否只用一次，都不够直观。

inline declaration 把变量放到使用点附近。比如：

DATA(total_amount) = price * quantity.

读者立刻知道：这里要算一个 total amount，它来自 price * quantity。如果右侧表达式和变量名都清楚，代码更短，也更容易 review。

在 ABAP SQL 中，inline declaration 也特别常用：

SELECT FROM zorder
       FIELDS order_id, status
       WHERE customer_id = @customer_id
       INTO TABLE @DATA(lt_orders).

这里 lt_orders 的 row type 来自 SELECT list，读者不需要提前写一个专门的结构或表类型。对于一次性查询结果、短 method、示例代码、ABAP Unit test fixture，这非常有效。

但 inline declaration 也容易被误用。金额、数量、汇率、日期、业务 key、跨多步复用的 internal table，如果只靠 DATA(result) 这样的名字，reviewer 可能看不出业务含义和技术类型。SAP 项目不是代码越短越好，而是长期维护、障害調査、升级和交接时能读懂。Modern ABAP 的目标是减少无意义样板，不是把类型信息全部藏起来。

37.4 Key mechanism / 核心机制

37.4.1 Inline declaration needs a declaration position

ABAP 不是任意位置都能写 DATA(x)。必须是在语法定义为 declaration position 的地方，也就是编译器知道“这里需要一个变量，并且能从上下文确定它的类型”的位置。

典型位置包括：

DATA(text) = |Order { order_id }|.

method_call(
  EXPORTING input  = source
  IMPORTING result = DATA(result)
).

READ TABLE lt_items INTO DATA(ls_item)
     WITH KEY item_no = item_no.

CATCH zcx_order_error INTO DATA(order_error).

这些例子里，类型来源分别来自右侧表达式、method signature、internal table row type、exception class。

反过来，如果上下文没有足够类型信息，inline declaration 就不合适，甚至语法上不成立。还有一个初学者常犯的错误：inline declaration 不能出现在 assignment 的右侧 / right-hand side 或普通表达式内部，例如不要写成 result = some_method( DATA(temp) ).。它必须出现在语句允许的写入/目标位置。不要把它理解成 JavaScript/Python 那种运行时动态变量创建；ABAP 仍然要求编译期类型可判定。

37.4.2 DATA(...) declares a normal data object

DATA(var) 声明的是普通 ABAP data object。它后续可以像普通变量一样被读取、传递和修改：

DATA(counter) = 0.

LOOP AT lt_items INTO DATA(ls_item).
  counter += 1.
ENDLOOP.

这里 counter 从 literal 0 推断出技术类型，ls_item 从 lt_items 的 line type 推断出结构类型。初学者要注意三件事。

第一，LOOP AT ... INTO DATA(ls_item) 是 value copy。修改 ls_item 不会自动修改 internal table 当前行；要原地修改，应使用 ASSIGNING FIELD-SYMBOL(<row>) 或在合适场景下使用 REFERENCE INTO DATA(ref)。

第二，DATA(ls_item) 不是每一轮循环重新声明一个新变量。inline variable 按 ABAP 的 declaration context / processing block 规则声明；它不会像 Java/C# 的块级局部变量那样在 ENDLOOP、ENDIF、ENDTRY 后自动消失。也就是说，在同一处理块后续代码里仍可能访问到它。这个规则很容易造成误解：如果 loop 一次都没有执行，后续变量可能只是初始值，而不是“不存在”。

第三，同一 context 中同名 inline variable 只能声明一次。在同一个 processing block 中第二次写 DATA(ls_item) 通常会造成语法错误。代码写长了以后，如果发现反复想声明同名临时变量，往往说明 method 应该拆小，或变量名应更具体。

37.4.3 @DATA(...) in ABAP SQL

ABAP SQL 里常见 @DATA(...)：

SELECT FROM zorder
       FIELDS order_id, status
       WHERE status = @open_status
       INTO TABLE @DATA(lt_orders).

这里有两层含义。@ 属于 ABAP SQL 的 host variable / host expression 标记，用来说明后面是 ABAP 程序世界里的对象或表达式。DATA(lt_orders) 是 inline declaration。两者组合起来，就是“把 SELECT 结果接收到一个在此处声明的 ABAP internal table”。

如果是单行结果，可以写：

SELECT SINGLE FROM zorder
       FIELDS order_id, status
       WHERE order_id = @order_id
       INTO @DATA(ls_order).

如果是多行 INTO TABLE @DATA(lt_orders)，结果是 standard internal table，不会自动带上你为业务访问设计的 sorted/hashed key 或 secondary key。前面学习 internal table 时已经讲过：inline 接收很方便，但如果后续要频繁按 key 查找、在 nested loop 里反复读取、需要 sorted/hashed table、或需要明确 table key，就不要只依赖默认推断，而应显式设计 table type。

37.4.4 FIELD-SYMBOL(...) declares a field symbol

FIELD-SYMBOL(<fs>) 常见于直接处理 internal table 行：

LOOP AT lt_items ASSIGNING FIELD-SYMBOL(<ls_item>).
  <ls_item>-status = 'DONE'.
ENDLOOP.

这和 LOOP AT ... INTO DATA(ls_item) 的语义不同。INTO DATA(ls_item) 是复制当前行到 work area；ASSIGNING FIELD-SYMBOL(<ls_item>) 是让 field symbol 指向当前行。修改 <ls_item>-status 会改 internal table 里的当前行。

也可以用在动态或条件 assign 中：

ASSIGN COMPONENT 'STATUS' OF STRUCTURE ls_order
       TO FIELD-SYMBOL(<status>).

IF <status> IS ASSIGNED.
  <status> = 'CLOSED'.
ENDIF.

这里必须理解上一组 field symbol 章节的规则：field symbol 可能未 assigned，访问前要确认；不要把 field symbol 当成普通变量；在 sorted/hashed table 的 key 字段、生命周期和别名修改上要非常小心。

37.4.5 FINAL(...) is inline declaration with immutability intent

现代 ABAP 中还可以看到：

FINAL(today) = cl_abap_context_info=>get_system_date( ).

FINAL(today) 表达的是：这个局部值初始化后不应该再被修改。再次写入 today 会在语法检查阶段成为错误，而不是等到运行时才失败。它对读者有一个很好的信号：后面看到 today，不用担心它在 method 中途被重新赋值。

因此，如果一个局部值初始化后就不应该再被改，FINAL(...) 比 DATA(...) 更能表达 intent。是否在项目中大量使用 FINAL(...)，还要看目标 ABAP release、团队 Clean ABAP 约定和代码风格；但它不只是“语法小知识”，而是现代 ABAP 里表达不变性的工具。

37.4.6 Type inference is compile-time, not runtime magic

inline declaration 的类型推断是静态的。常见类型来源如下：

  • literal 或表达式：DATA(count) = 0.
  • method signature：lo_service->read( IMPORTING result = DATA(result) ).
  • internal table line type：LOOP AT lt_items INTO DATA(ls_item).
  • field symbol assignment source：ASSIGN ls_order-status TO FIELD-SYMBOL(<status>).
  • ABAP SQL select list：INTO TABLE @DATA(lt_orders)
  • exception class：CATCH zcx_order_error INTO DATA(order_error).

还可以写对象引用创建：

DATA(pricing_service) = NEW zcl_pricing_service( ).

这里变量类型来自 NEW zcl_pricing_service( ) 的静态类型，适合短小、清楚的对象创建场景。

如果 operand type 来自 ABAP Dictionary / DDIC 类型，inline variable 可以继承相应的语义属性，例如 conversion exit、input help、field help 等。这是内联声明的优点之一：它不是“丢掉所有类型信息”，而是在上下文足够强时复用已有类型信息。

这也是 inline declaration 的判断标准：如果读者看上下文就能知道类型，内联通常很好；如果必须跳到多个地方才能猜出类型，或者推断结果不符合业务预期，就要显式声明。

反过来，如果类型来源过于 generic，例如 ANY、ANY TABLE、generic field symbol 或 generic interface parameter，编译器可能无法从 declaration position 完全静态地推出你想要的业务类型，或者只能得到过宽的技术类型。遇到这类边界时，不要硬写 DATA(result)；显式声明目标类型，或者先把 generic 边界收窄，会比让 reviewer 猜推断结果更安全。

37.4.7 Inline declaration does not remove naming discipline

DATA(result)、DATA(data)、DATA(temp) 这样的名字很容易写，但不一定适合项目代码。变量名仍然要表达业务语义：

" Weak
DATA(result) = calculate( ).

" Better
DATA(total_net_amount) = calculate_total_net_amount( ).

对一次性临时值，短名字可以接受；对业务 key、金额、状态、日期、跨多行使用的对象，变量名要让 reviewer 不需要猜。内联声明越短，命名越重要。

37.4.8 When explicit declaration is better

下面这些场景，显式声明常常更好：

  • 金额、数量、汇率、百分比等需要明确 length / decimals / currency / unit 语义。
  • internal table 需要明确 sorted / hashed / key。
  • 变量跨多个步骤复用，生命周期很长。
  • method signature 或 SQL result 太复杂，推断类型不直观。
  • 代码用于接口边界、公共 API、复杂 mapping，读者需要清楚类型 contract。
  • 团队 review 明确要求某类变量显式 TYPE。

比如：

DATA total_amount TYPE zamount.
total_amount = price * quantity.

这不“落后”。它是在业务语义重要时，把类型 contract 直接写出来。

金额和数量还要特别注意 DDIC 语义。SAP 业务字段往往有 currency / unit reference field。若 inline declaration 的来源不能清楚保留这些语义，后续显示、换算、ALV 输出或接口 mapping 都可能变得模糊。此时显式 TYPE 到合适的 data element、structure component 或专门类型，是更稳的项目写法。

37.4.9 inline declaration review checklist

Review 一段 Modern ABAP 时，可以用下面的表判断 inline declaration 是否真的提高了可读性。

QuestionInline is usually fine when...Prefer explicit declaration when...
Is the inferred type obvious?右侧是简单表达式、method return、table row 或清楚的 SQL result。类型来自复杂表达式、deep structure、generic parameter 或难懂 SQL。
Is the variable short-lived?变量只在邻近几行使用。变量跨多个逻辑步骤、多个分支或整个 method 使用。
Is business meaning encoded?变量名已经清楚表达语义，例如 net_amount。金额、数量、日期、key、currency/unit 需要明确 DDIC 类型。
Should the value change?后续确实会重新赋值，用 DATA(...)。初始化后不应改变，用 FINAL(...) 表达 intent。
Is alias semantics intended?FIELD-SYMBOL(...) 用于直接修改 table row 或避免复制。只是读取值，或者不希望修改原表；此时普通 INTO DATA(...) 更安全。
Will debugging be clear?变量名、位置和类型来源都容易追踪。过度嵌套表达式让调试时很难看出中间值。

一个实用规则是：inline declaration 应该缩短“声明和使用之间的距离”，而不是隐藏类型 contract。若 reviewer 需要一路追溯 method signature、DDIC structure 和 SQL result 才能理解变量类型，就说明这处 inline declaration 可能已经越界。

37.5 日本项目现场说法

  • 常见日语：インライン宣言、DATA(...)、FIELD-SYMBOL(...)、FINAL(...)、型推論、コンパイル時型推論、静的型付け、宣言位置、処理ブロックのスコープ、明示的な型宣言、明示的な型指定、可読性、保守性、正確性、命名規約。
  • 说明句：この変数はインライン宣言で問題ありません。右辺の式から型が明確に推論でき、利用箇所も限定されています。
  • Review 指摘句：この DATA(result) は型と業務意味が分かりにくいため、変数名を具体化するか、明示的な型宣言に変更してください。
  • Review 指摘句：金額項目の型推論が読み手に伝わりにくいため、TYPE を明示して桁数・小数点・通貨単位の意図を確認してください。
  • Review 指摘句：この変数は LOOP 内でインライン宣言されていますが、ABAP では処理ブロックのスコープに残るため、後続処理で誤用されないように変数名または処理分割を見直してください。
  • 障害調査场景：障害調査時に型の由来を追いやすいよう、長い処理ではインライン宣言を使いすぎず、必要に応じて明示的な宣言にしてください。
  • 使用场景：code review、Clean ABAP review、性能/障害调查、ABAP Unit test fixture、ABAP SQL 查询结果接收、field symbol 修改范围确认。

37.6 English Terms

  • Official / search terms: Inline Declarations, DATA, Inline Declaration for Variables, FIELD-SYMBOL, Inline Declaration for Field Symbols
  • ABAP syntax: DATA(var), @DATA(var), FIELD-SYMBOL(<fs>), FINAL(var), INTO DATA(...), INTO TABLE @DATA(...), ASSIGNING FIELD-SYMBOL(...)
  • Type terms: declaration position, operand position, statically identifiable type, type inference, data object, field symbol, declaration context
  • Chinese terms: 内联声明, 字段符号, 编译期型推断, 静态类型, 处理块作用域, 显式类型声明
  • Style terms: readability, explicit declaration, naming convention, scope, business semantics
  • Recommended search pattern: ABAP inline declarations SAP Help DATA FIELD-SYMBOL, ABAP DATA inline declaration ABAP Keyword Documentation, ABAP FIELD-SYMBOL inline declaration SAP Help

读官方资料时注意：Inline Declarations 是总入口；DATA, Inline Declaration for Variables 和 FIELD-SYMBOL, Inline Declaration for Field Symbols 分别说明两类最常用形式。ABAP SQL 里的 @DATA(...) 同时涉及 host variable escape @，不要把 @ 误解成 inline declaration 的一部分。

37.7 Example / 学习例子

37.7.1 例 1：一次性局部值
DATA(discounted_amount) = gross_amount - discount_amount.

这个例子适合 inline declaration，因为右侧表达式简单，变量名表达了业务意义，变量很可能只在附近使用。如果后续发现 amount 类型需要明确 currency / decimal 语义，再改成显式类型。

37.7.2 例 2：method call 的 importing result
lo_pricing_service->calculate(
  EXPORTING order_id = order_id
  IMPORTING amount   = DATA(net_amount)
).

这里 net_amount 的类型来自 calculate method signature。读者可以通过 method contract 追溯类型。这个写法比在 method 开头提前声明一个只用于这一句的变量更清楚。

37.7.3 例 3：READ TABLE result
READ TABLE lt_items INTO DATA(ls_item)
     WITH KEY item_no = item_no.

IF sy-subrc = 0.
  " use ls_item
ENDIF.

ls_item 的类型来自 lt_items 的 line type。这里要把 inline declaration 和 sy-subrc 处理一起看：变量声明成功不代表读取成功，仍要判断读取结果。

37.7.4 例 4：ABAP SQL multi-row result
SELECT FROM zorder
       FIELDS order_id, customer_id, status
       WHERE status = @open_status
       INTO TABLE @DATA(lt_open_orders).

这适合短小查询和局部使用。如果 lt_open_orders 后面要被频繁按 customer_id 查找，就要考虑显式定义 sorted/hashed table 或 secondary key，而不是满足于默认 inline 结果。

37.7.5 例 5：field symbol inline declaration
LOOP AT lt_items ASSIGNING FIELD-SYMBOL(<ls_item>).
  IF <ls_item>-quantity = 0.
    <ls_item>-status = 'INVALID'.
  ENDIF.
ENDLOOP.

这个例子里 <ls_item> 不是复制出来的 work area，而是指向当前 internal table 行。修改它会影响原表。初学者必须把它和 LOOP AT ... INTO DATA(ls_item) 区分开。

37.7.6 例 6：显式声明更好
DATA exchange_rate TYPE p LENGTH 9 DECIMALS 5.

exchange_rate = read_exchange_rate(
  source_currency = source_currency
  target_currency = target_currency
).

汇率的小数位、舍入和业务含义很重要。即使可以内联，显式类型也可能更适合 review 和交付文档。

37.8 Counterexample / Failure mode

37.8.1 误区 1：把 inline declaration 当成动态类型

DATA(x) = ... 不是运行时动态类型。ABAP 编译器仍然在编译期决定类型。

修正动作：每次看到 inline declaration，都问“类型从哪里推断出来？”

37.8.2 误区 2：变量名太泛
DATA(result) = get_data( ).
DATA(value)  = calculate( ).

这种代码短，但读者不知道业务含义。

修正动作：把名字改成业务语义，例如 open_order_items、net_amount、approval_status。

37.8.3 误区 3：用 inline declaration 掩盖复杂类型

复杂 SQL result、复杂 mapping、金额/数量/日期 key 如果全靠 DATA(...) 推断，review 时可能看不清类型 contract。

修正动作：显式声明结构、表类型或 domain/data element 绑定。

37.8.4 误区 4：忘记 field symbol 语义

FIELD-SYMBOL(<row>) 指向对象，不是复制。未 assigned 访问、修改 key 字段、生命周期混乱都可能造成问题。

修正动作：回到 Field symbol 字段符号 与 LOOP AT ASSIGNING vs LOOP AT INTO，确认 copy 和 direct update 的差异。

37.8.5 误区 5：过度嵌套现代表达式

inline declaration 常和 VALUE、COND、SWITCH、FOR 等表达式一起出现。嵌套过深会让代码难读。

修正动作：如果一行表达式需要读三遍才懂，拆成有名字的中间变量或小 method。

37.9 思考问题

  • 这条 inline declaration 的类型从哪里推断出来？
  • 如果把它改成显式 DATA ... TYPE ...，可读性会更好还是更差？
  • 变量名是否表达业务语义，还是只有 result、data、temp？
  • 这是普通 data object、field symbol，还是 final variable？
  • 在 ABAP SQL 中，@DATA(...) 里的 @ 和 DATA(...) 分别表示什么？
  • 这个 inline internal table 是否需要明确 sorted/hashed/key？
  • 日本项目 review 中，应该用 型推論が明確 还是 明示的な型宣言が必要 来说明？
