Chapter 02 - Core ABAP Programming

39. CORRESPONDING expression 字段映射

  • Chapter 02
  • Section 39
  • self-contained study note

39.1 一句话结论

CORRESPONDING expression 字段映射 是 ABAP constructor expression 中用于“按 component 名称把一个结构/表行映射到另一个结构/表行”的写法。最基本的形式是 CORRESPONDING target_type( source ) 或 CORRESPONDING #( source )：ABAP 会把 source 和 target 中同名 component 的值复制过去，目标中没有被填的 component 变成 initial value，除非使用 BASE 保留旧值。

最常见的例子是把数据库结构、接口结构、RAP helper structure、测试输入结构之间做转换：

DATA(api_order) = CORRESPONDING ty_api_order( db_order ).

如果两边字段名不同，可以用 MAPPING 显式说明：

DATA(api_order) = CORRESPONDING ty_api_order(
  db_order
  MAPPING order_id    = vbeln
          customer_id = kunnr
  EXCEPT  internal_note
).

这段代码的意思是：目标 order_id 来自源字段 vbeln，目标 customer_id 来自源字段 kunnr，目标 internal_note 不复制。初学者要特别注意：CORRESPONDING 只懂“字段名和映射规则”，不懂业务语义。它不会自动知道 amount_net 和 net_value 是同一件事，也不会自动判断货币、单位、日期格式或 key 是否语义兼容。

39.2 到底学什么

本节要掌握九个点：

  • CORRESPONDING 是 constructor operator / constructor expression，用于结构或内表行的 component mapping。
  • 基本规则是同名 component 复制；源有但目标没有的 component 被忽略，目标有但源没有的 component 会是 initial value，除非使用 BASE。
  • CORRESPONDING #( source ) 的 # 需要目标类型上下文；不清楚时写显式 target type。
  • MAPPING target_component = source_component 用于不同字段名之间的映射，方向不要写反。
  • EXCEPT target_component 用于排除某些目标 component，让它不从源复制。
  • BASE ( base ) 可在已有目标值基础上做映射，避免未映射 component 被 initial 覆盖。
  • DEFAULT 可和 MAPPING 配合，但要区分 target = DEFAULT expr 与 target = source DEFAULT expr 两种语义。
  • 内表映射要关注 unique key；必要时会看到 DISCARDING DUPLICATES、DEEP、APPENDING、FROM ... USING 等进阶形式。
  • component 类型不完全一致时会按 ABAP assignment / conversion rules 转换，可能截断、舍入、格式变化或运行时失败。
  • CORRESPONDING 可以让结构转换更简洁，但它不是业务 mapping 规则引擎；复杂接口、金额/数量/单位/日期/key 字段要显式 review。
  • 日本项目中常见 review 重点是 同名項目、MAPPING指定、EXCEPT指定、暗黙変換、未設定項目の初期値、項目マッピング表。

最小判断标准：看到 CORRESPONDING #( source ) 时，能回答“目标类型是什么、哪些同名字段会被复制、哪些字段会变 initial、有没有隐式转换、是否需要 MAPPING / EXCEPT / BASE”。

39.3 为什么重要

SAP 项目里经常要在不同结构之间移动数据：数据库表/视图结构、BAPI structure、RAP behavior helper、OData DTO、ALV output、CSV upload staging、ABAP Unit test data。字段很多时，逐个 assignment 很啰嗦：

api_order-order_id    = db_order-vbeln.
api_order-customer_id = db_order-kunnr.
api_order-status      = db_order-status.
api_order-currency    = db_order-waerk.

如果字段名一致，CORRESPONDING 可以极大减少样板：

api_order = CORRESPONDING #( db_order ).

如果字段名不一致，MAPPING 可以把意图写在一个地方：

api_order = CORRESPONDING #(
  db_order
  MAPPING order_id    = vbeln
          customer_id = kunnr
          currency    = waerk
).

这比“几十行 assignment 混在业务逻辑里”更像一张字段映射表。对代码 review、接口交接、数据迁移、RAP / OData 边界都很有价值。

但风险也很现实。CORRESPONDING #( source ) 容易让人以为“字段映射已经安全完成”。实际上它只按名称和规则复制，并按 ABAP 类型转换规则赋值。目标字段被省略时会是 initial value；同名字段类型不兼容时可能发生截断、舍入、日期/字符格式变化；业务上同名但含义不同的字段也会被复制。比如两个结构都有 status，一个表示销售订单状态，一个表示接口处理状态，直接同名复制就可能是 bug。

所以本节的核心是：CORRESPONDING 是强大的结构映射工具，不是免 review 的魔法。

39.4 Key mechanism / 核心机制

39.4.1 Basic form maps same-name components

最简单形式：

DATA(target) = CORRESPONDING ty_target( source ).

ABAP 会看 source 和 ty_target 中的 component 名称，把同名 component 的值复制到目标中。源结构中多出来的字段会被忽略；目标结构中没有被填的字段会保持 initial value。

这里的“同名”只看 component name，不看字段位置。也就是说，source 的第 1 个字段不会因为位置相同就自动复制到目标第 1 个字段；只有名称一致，才进入自动对应。对有 Java/Python 背景的读者来说，这一点要先刻进脑子：CORRESPONDING 是 name-based mapping，不是 position-based mapping。

例如：

TYPES BEGIN OF ty_db_order.
TYPES order_id TYPE string.
TYPES status   TYPE string.
TYPES changed_by TYPE string.
TYPES END OF ty_db_order.

TYPES BEGIN OF ty_api_order.
TYPES order_id TYPE string.
TYPES status   TYPE string.
TYPES text     TYPE string.
TYPES END OF ty_api_order.

DATA(api_order) = CORRESPONDING ty_api_order( db_order ).

结果是：order_id 和 status 被复制；changed_by 因为目标没有而被忽略；text 因为源没有而是 initial value。

39.4.2 # needs target type context

和 VALUE #( ... ) 一样，CORRESPONDING #( source ) 的 # 需要上下文能推断目标类型：

DATA api_order TYPE ty_api_order.

api_order = CORRESPONDING #( db_order ).

这里 # 从左侧 api_order TYPE ty_api_order 推断。下面这种写法通常上下文不足：

" Target type is unclear
DATA(api_order) = CORRESPONDING #( db_order ).

要么给左侧显式类型，要么在 CORRESPONDING 后写 target type：

DATA(api_order) = CORRESPONDING ty_api_order( db_order ).

初学阶段的纪律很简单：如果 reviewer 看不出目标类型，就不要用 #。

39.4.3 MAPPING handles different names

如果源和目标字段名不同，用 MAPPING：

DATA(api_order) = CORRESPONDING ty_api_order(
  db_order
  MAPPING order_id    = vbeln
          customer_id = kunnr
          currency    = waerk
).

方向要记清楚：左边是 target component，右边是 source component。可以读成“目标 order_id 取自源 vbeln”。

这点非常重要。方向写反时，可能语法就不通过；更糟的是，某些名字刚好都存在，代码能过但语义错。

另外，MAPPING 里不要用 component selector - 去直接写嵌套 component 路径。遇到 nested structure、nested table 或深层 DTO 时，应查对应语法形式，必要时拆成局部结构变量或显式 mapping，不要把深层业务转换压进一行看似漂亮的 CORRESPONDING。

39.4.4 EXCEPT excludes target components

EXCEPT 用于排除目标 component：

DATA(api_order) = CORRESPONDING ty_api_order(
  db_order
  EXCEPT internal_note
).

这表示不要从源复制到目标 internal_note。通常用于技术字段、内部备注、审计字段、接口不应暴露字段等。

如果字段敏感，建议显式 EXCEPT，或者干脆显式逐字段 mapping。不要依赖“源结构刚好没有这个字段”来保证安全，因为源结构未来可能扩展。

如果和 MAPPING 一起使用，还可以看到 EXCEPT *。它的意思不是“什么都不做”，而是“抑制所有自动同名 mapping，只保留显式 MAPPING 指定的字段”。这在接口 DTO、公开 API response、CSV upload staging 等场景很有价值，因为它把字段白名单写进了代码。

可以把 mental model 写成三步：先按同名 component 建立自动对应，再由 EXCEPT 排除不想复制的目标 component，最后让 MAPPING 指定的显式对应生效。EXCEPT * 的白名单效果来自“关闭所有自动同名对应”。如果没有任何 MAPPING，单独使用 EXCEPT * 会让目标 component 都保持 initial，这通常不是初学者想要的结果。

39.4.5 BASE preserves existing target values

默认情况下，CORRESPONDING 构造的是一个新的目标值；没有被填的 component 是 initial value。如果要在已有目标值基础上映射源字段，可以使用 BASE：

api_order = CORRESPONDING #(
  BASE ( api_order )
  db_order
  MAPPING order_id = vbeln
).

这表示：先以现有 api_order 为基础，再把 db_order 中映射到的字段覆盖上去。未映射字段继续保留 api_order 里的旧值。

和上一节 VALUE 一样，BASE 不是装饰。它决定未映射字段是保留旧值，还是变成 initial value。项目 review 中要问：这里想要“重建目标结构”，还是“在已有目标上补字段”？

注意语法细节：CORRESPONDING 的 BASE 要写成 BASE ( base )，括号是必须的；这和 VALUE 一节里看到的某些 BASE 写法不完全一样。常见、合理的更新模式是：

api_order = CORRESPONDING #(
  BASE ( api_order )
  db_order
  MAPPING order_id = vbeln
).

这表示先把原来的 api_order 作为 base value，再覆盖来自 db_order 的 mapped fields。

39.4.6 DEFAULT has two mapping fallback forms

DEFAULT 可和 MAPPING 一起使用，但要分清两种语义。第一种是 MAPPING target = DEFAULT expr，目标字段总是取 expr 的结果；第二种是 MAPPING target = source DEFAULT expr，只有当 source 的值是 initial 时，才改用 expr。它不是“源字段不存在就兜底”，也不是“类型不兼容就兜底”。

DATA(api_order) = CORRESPONDING ty_api_order(
  db_order
  MAPPING order_id = vbeln
          status   = status DEFAULT 'NEW'
).

这段例子的意思是：如果源字段 status 不是 initial，就把源 status 给目标 status；如果源 status 是 initial，就把 'NEW' 给目标 status。这里仍然要保持克制。DEFAULT 只是在 constructor expression 中提供一个值，不会自动检查 customizing、不会做合法性校验，也不会证明 'NEW' 是业务上正确的状态。项目里如果默认值有业务含义，应在字段设计、mapping 表或转换方法中说明。

39.4.7 Type conversion still happens

CORRESPONDING 复制 component 时，仍遵循 ABAP assignment / conversion rules。如果同名字段或 MAPPING 字段类型不同，可能发生：

  • 字符串/定长字符截断。
  • packed number 小数位舍入。
  • 日期/时间字符格式转换问题。
  • NUMC / CHAR 前导零变化。
  • 金额/数量字段失去 currency/unit 上下文。
  • conversion exit 没有按业务期望执行。
  • 无法转换时运行时异常。
  • source 和 target 是同一变量、又用 MAPPING 做 component 交换时，存在 CORRESPONDING_SELF 这类运行时风险，复杂场景不要炫技。

因此，跨结构类型使用 CORRESPONDING 时，不能只看字段名相同。金额、数量、日期、语言、单位、状态、业务 key 字段尤其要显式 review。

39.4.8 Internal table mapping

CORRESPONDING 也可以用于内表，把每一行按 line type 做对应映射：

DATA(api_orders) = CORRESPONDING ty_api_orders( db_orders ).

这里的含义是：对 db_orders 的每一行，构造一行 ty_api_orders 的 row type。具体能否这样写、目标 table key 如何处理、重复 key 如何处理，要看目标 table type 和 ABAP release / syntax form。初学阶段先掌握结构映射，再把 table-level mapping 当作同一机制在内表上的延伸。

如果目标 table 有 unique key，映射后的重复 key 可能导致运行时问题；如果目标是 standard table with empty key，则重复行会被接受。和 VALUE 一样，constructor expression 不会绕过 table key 设计。

当目标内表有 unique key，而源数据可能映射出重复 key 时，相关语法中可以看到 DISCARDING DUPLICATES：它用于丢弃会造成重复 key 的行，避免 duplicate key 异常。SAP cheat sheet 的示例体现了一个重要行为：重复 key 中通常保留先出现的行，后续重复行被静默丢弃。是否使用它不是纯技术选择，而是业务选择：重复订单行、重复客户编号、重复 condition record 是要报错、合并、保留第一条，还是丢弃后继续处理，必须由需求决定。把它写进代码前，要先确认业务允许这种 silent data loss。

更进阶的 DEEP / APPENDING 用于深层或嵌套 tabular component。初学者先记住三点：第一，浅层 CORRESPONDING 主要处理当前结构层级的 component，不等于递归业务 mapping；第二，DEEP 表示对深层 tabular component 继续按层级处理同名 component；第三，APPENDING 的重点是不要删除已有 nested target table，而是在其中追加对应行。遇到 nested table、深层结构、RAP payload 或复杂接口 DTO 时，不要想当然以为一行 CORRESPONDING #( ... ) 会得到业务期望的深层结果，应查语法文档并写 ABAP Unit 覆盖。

另一个进阶形态是 CORRESPONDING #( itab FROM lookup_tab USING ... )。它可用于把一个内表和 lookup table 按 key 对应起来，类似把 lookup 数据补进主表。这个语法依赖 sorted/hash key 等约束，不适合作为本节第一层掌握点；看到它时，应重点 review lookup key、唯一性、未命中时行为和目标表类型。

39.4.9 Relation to MOVE-CORRESPONDING and INTO CORRESPONDING FIELDS

旧代码里会看到：

MOVE-CORRESPONDING source TO target.

也会在 ABAP SQL 中看到：

SELECT ...
  INTO CORRESPONDING FIELDS OF TABLE @lt_result.

它们都属于“按对应字段名移动”的家族，但语境不同。CORRESPONDING #( ... ) 是现代 constructor expression，可以嵌在赋值、method call、VALUE、ABAP Unit test data 中；MOVE-CORRESPONDING 是 statement；INTO CORRESPONDING FIELDS 是 SQL result 接收语境。风险类似：方便，但可能静默丢字段、静默转换、让字段语义问题隐藏起来。

一个容易忽略的差异是初始化行为：没有 BASE 的 CORRESPONDING #( source ) 会构造新的目标值，未匹配字段通常变为 initial；而 MOVE-CORRESPONDING source TO target 是把 source 中对应字段移动到已有 target，未匹配字段不会因此被清空。把旧代码迁移成 expression 写法时，必须确认这种差异是否会改变业务结果。

旧代码里还可能看到 MOVE-CORRESPONDING ... KEEPING TARGET LINES 或 EXPANDING NESTED TABLES。前者和“保留目标内表既有行”有关，思路上接近 CORRESPONDING #( BASE ( itab ) source )；后者和 nested table 展开有关，思路上接近 CORRESPONDING ... DEEP。初学者不需要先掌握所有 legacy addition，但要知道：把老 MOVE-CORRESPONDING 改成现代 expression 之前，必须读清这些 addition 的行为，不能只做关键词替换。

INTO CORRESPONDING FIELDS 则属于 ABAP SQL 的接收语境。它按 SELECT list 的列名和目标 component 名称对应，常用于避免手工排列字段顺序，但它不是 CORRESPONDING #( ... ) 的替代品。遇到预先声明的 work area、内表接收、APPENDING CORRESPONDING FIELDS 等不同形式时，要查 SQL 文档确认未选字段、已有目标内容和追加行为，不要把 constructor expression 的 BASE 直觉套过去。

39.4.10 Not a business mapping engine

CORRESPONDING 只处理 ABAP component mapping。它不会：

  • 查 customizing。
  • 做 currency / unit conversion。
  • 翻译 status code。
  • 校验必填字段。
  • 判断业务语义是否等价。
  • 自动处理外部接口 version 差异。

这些仍然要靠显式业务逻辑、mapping table、conversion method、validation 或接口契约来完成。项目里如果字段映射有业务含义，不要只把它藏在一行 CORRESPONDING #( ... ) 里。

39.4.11 CORRESPONDING mapping review checklist

Review CORRESPONDING 时，重点不是“有没有少写代码”，而是映射结果是否可解释、可测试、可交付。下面的检查表可以直接用于接口字段 mapping review：

QuestionWhat to verifyRisk if ignored
What is the target type?# 是否能从左侧、参数或上下文清楚推断。读者不知道最终会构造什么结构或表。
Are same-name fields semantically same?同名 component 的业务含义、单位、币种、日期格式是否一致。名字相同但语义不同，导致静默错误。
Which fields become initial?未映射 target component 是否真的应该 initial。旧值被清空，接口必填字段漏值。
Is BASE needed?是否要保留目标已有字段、默认值或人工补充字段。误以为 CORRESPONDING 会自动保留旧目标内容。
Is MAPPING direction correct?target = source 是否写对，并和接口 mapping 表一致。方向写反或字段来源错误，review 很难一眼发现。
Are conversions safe?length、decimals、currency/unit、date/time、domain value 是否兼容。截断、舍入、格式变化或运行时转换错误。
Is table key behavior clear?unique key、duplicate row、DISCARDING DUPLICATES 是否符合业务。行丢失、重复键异常或结果顺序误解。
Is there test evidence?ABAP Unit 是否覆盖同名、MAPPING、EXCEPT、BASE、异常边界。mapping 被改坏时没有自动报警。

如果字段映射属于外部 API、IDoc/BAPI wrapper、RAP projection 或跨系统接口，最好保留一张可 review 的 mapping 表，并让代码里的 MAPPING 与表格保持一致。CORRESPONDING 可以减少机械赋值，但不能替代接口契约本身。

39.5 日本项目现场说法

  • 常见日语：CORRESPONDING式、項目マッピング、同名項目マッピング、同名項目、位置ではなく名前による対応、MAPPING指定、EXCEPT指定、EXCEPT *指定、BASE指定、暗黙変換、暗黙型変換、型変換、未設定項目、型固有の初期値、重複キー、深い構造、DEEP指定、項目対応表、インターフェース項目、内部項目。
  • 说明句：この CORRESPONDING #( ... ) は同名項目のコピーであり、業務上の変換やコード変換は別途実装が必要です。
  • 说明句：同名項目は位置ではなく名前による対応であり、同名でも業務上の意味が同じとは限りません。
  • Review 指摘句：MAPPING 指定では左辺がターゲット項目、右辺がソース項目です。方向を逆に読まないようにしてください。
  • Review 指摘句：EXCEPT * は同名自動マッピングを抑制し、MAPPING 指定された項目だけを対応付ける意図か確認してください。
  • Review 指摘句：DISCARDING DUPLICATES は後続の重複行を警告なしに破棄するため、業務要件上許容されるか確認してください。
  • Review 指摘句：同名項目ですが意味が異なる可能性があるため、status のマッピング意図を明示してください。
  • Review 指摘句：対象外にすべき内部項目があるため、EXCEPT を指定するか、明示的な項目代入に変更してください。
  • Review 指摘句：金額・数量・日付項目で暗黙変換が発生する可能性があるため、型互換性と単位/通貨の扱いを確認してください。
  • 使用场景：接口设计、RAP DTO/helper structure、OData response mapping、Migration/staging mapping、ALV output、ABAP Unit expected data、code review。

39.7 Example / 学习例子

39.7.1 例 1：同名字段复制
DATA(api_order) = CORRESPONDING ty_api_order( db_order ).

适合源和目标字段名大部分一致，且同名字段语义和类型都明确兼容的场景。

39.7.2 例 2：字段名不同，用 MAPPING
DATA(api_order) = CORRESPONDING ty_api_order(
  db_order
  MAPPING order_id    = vbeln
          customer_id = kunnr
          currency    = waerk
).

读法：目标 order_id 来自源 vbeln。

39.7.3 例 3：排除内部字段
DATA(public_order) = CORRESPONDING ty_public_order(
  internal_order
  EXCEPT internal_note changed_by changed_at
).

这比“希望目标结构没有这些字段”更稳，因为代码明确表达了不复制这些内部字段。

39.7.4 例 4：保留已有目标字段
api_order = CORRESPONDING #(
  BASE ( api_order )
  db_order
  MAPPING order_id = vbeln
).

如果 api_order 里已有 display_text、source_system 等字段，这种写法能保留它们。没有 BASE 时，未映射字段会变成 initial value。

39.7.5 例 5：类型转换风险
DATA(target) = CORRESPONDING ty_target( source ).

如果 source 里的 amount 是 character，target 里的 amount 是 packed number，ABAP 会尝试转换。能转换不代表业务正确；小数位、货币、空值、异常格式都要检查。

39.7.6 例 6：ABAP Unit 固定 mapping 行为
DATA(source) = VALUE ty_db_order(
  vbeln  = '4500000001'
  kunnr  = '0000100001'
  status = 'OPEN'
).

DATA(actual) = CORRESPONDING ty_api_order(
  source
  MAPPING order_id    = vbeln
          customer_id = kunnr
).

cl_abap_unit_assert=>assert_equals(
  exp = '4500000001'
  act = actual-order_id
).

mapping 逻辑重要时，用 ABAP Unit 固定它，比只靠肉眼 review 更可靠。

39.8 Counterexample / Failure mode

39.8.1 误区 1：以为字段名相同就语义相同

两个结构都有 status，但一个是订单状态，一个是接口处理状态。

修正动作：对状态、类型、分类、标志字段做业务语义 review；必要时显式 mapping 或显式赋值。

39.8.2 误区 2：忘记未映射字段会 initial

CORRESPONDING #( source ) 构造新目标值时，目标中没有被复制的字段不会保留旧值。

修正动作：需要保留旧值时使用 BASE ( target )，或者显式赋值。

39.8.3 误区 3：MAPPING 方向写反

MAPPING target = source。方向错了，代码要么不通过，要么语义错。

修正动作：读成“目标字段 = 源字段”，并在 review 中确认。

39.8.4 误区 4：隐式转换被忽略

同名字段复制时，类型不同可能发生截断、舍入、前导零变化或格式转换。

修正动作：金额、数量、日期、NUMC、语言、单位、货币字段优先显式 review。

39.8.5 误区 5：把 CORRESPONDING 当业务转换

字段映射不等于业务转换。currency conversion、unit conversion、status translation、code list mapping 都不是 CORRESPONDING 自动完成的。

修正动作：把业务转换写进专门 method、mapping table 或 conversion layer。

39.9 思考问题

  • 这段 CORRESPONDING 的目标类型是什么？
  • 哪些 component 是同名复制？哪些通过 MAPPING？哪些通过 EXCEPT 排除？
  • 未映射目标字段是 initial value，还是通过 BASE 保留旧值？
  • 是否存在同名但业务语义不同的字段？
  • 是否存在隐式类型转换、截断、舍入、前导零或日期格式风险？
  • 这段 mapping 是否值得用 ABAP Unit 固定？
  • 日本项目的字段映射表是否和代码里的 MAPPING 保持一致？
