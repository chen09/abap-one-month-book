Chapter 02 - Core ABAP Programming

5. ORDER BY 排序确定性

  • Chapter 02
  • Section 05
  • self-contained study note

5.1 一句话结论

ORDER BY 是 ABAP SQL 里把“数据库返回顺序”变成“业务可解释顺序”的关键子句。不写 ORDER BY，多行结果的顺序就是未定义的；写了 ORDER BY 但排序字段不唯一，也只能保证这些字段的排序，不能保证 tie 之间谁先谁后。只要业务语义里出现“第一条、最后一条、最新一条、前 10 条、分页、优先级最高”，就必须把排序字段和 tie-breaker 写清楚。

5.2 到底学什么

前面 SELECT statement 查询语句 讲了如何读取，INTO vs INTO TABLE 结果接收 讲了结果接收到哪里。本节专门讲结果的顺序。

先看一个容易误导初学者的例子：

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              created_at
       WHERE customer_id = @lv_customer_id
       INTO TABLE @DATA(lt_orders).

这段代码能读到订单，但没有说明订单顺序。今天开发机里看起来像按创建时间排列，不代表明天 QAS、PRD、不同数据量、不同索引或不同执行计划下仍然这样。关系数据库不会因为“上次看起来是这个顺序”就承诺业务顺序。

如果业务需要“最新订单优先”，就要写出排序：

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              created_at
       WHERE customer_id = @lv_customer_id
       ORDER BY created_at DESCENDING,
                sales_order_id DESCENDING
       INTO TABLE @DATA(lt_orders).

这里 created_at DESCENDING 表达“新到旧”；sales_order_id DESCENDING 是 tie-breaker，避免两个订单创建时间相同的时候顺序仍然不稳定。ORDER BY 的本质不是让列表“看起来整齐”，而是把业务规则写进数据库结果集。

5.3 为什么重要

ORDER BY 重要，是因为没有排序的 bug 往往不在开发当天出现。开发环境数据少、插入顺序稳定、索引碰巧合适，代码看起来一直正确；到生产环境，执行计划变化或数据量变大，第一条记录就可能变成另一条。

第一类风险是“取第一条”。很多代码先 SELECT ... INTO TABLE，再 READ TABLE lt_orders INDEX 1。如果前面的 SQL 没有 ORDER BY，这个“第一条”没有业务含义。它只是当前数据库执行结果里的某一条。

第二类风险是“取最新一条”。SELECT SINGLE 加非唯一条件并不等于最新。数据库不知道你的“最新”是按创建日、更新时间、凭证号、版本号、状态优先级，还是组合规则判断。最新必须由 ORDER BY 说明。

第三类风险是“取前 n 条”。SAP Help 对 UP TO n ROWS 的说明非常关键：如果没有 ORDER BY，传入结果集的是任意符合条件的 n 行；如果 ORDER BY 不能唯一排序，也不能定义 tie 中哪些行进入结果。换句话说，先随便取 10 行再在 ABAP 里 SORT，并不能变成“真正最新 10 行”。

第四类风险是交付解释。日本 SAP 项目 review 经常会问：並び順は保証されていますか、最新1件の根拠は何ですか、同一日時の場合のタイブレークはありますか。如果只能回答“测试时就是这个顺序”，代码很难通过严肃 review。

5.4 Key mechanism / 核心机制

ORDER BY 对一个 multirow result set 排序。常见写法有三类。

第一，按业务字段排序：

ORDER BY created_at DESCENDING,
         sales_order_id DESCENDING

这是最常见的业务排序。ASCENDING 表示升序，DESCENDING 表示降序；不写方向时通常按升序理解。业务上说“最新、金额最高、优先级最高”时，应该能在这里看到对应字段。

第二，按 primary key 排序：

ORDER BY PRIMARY KEY

这表示按单一数据源的主键升序排序，适合需要稳定主键顺序的读取。它不是万能语义排序：主键顺序未必等于业务最新、显示顺序或优先级顺序。对 join、union、某些 view/CDS 场景和 FOR ALL ENTRIES 还存在限制，遇到复杂查询要回到 ABAP Keyword Documentation 确认。

第三，按多个字段组成稳定顺序：

ORDER BY priority ASCENDING,
         created_at DESCENDING,
         sales_order_id ASCENDING

这类排序先按优先级，再按创建时间，再按唯一键 tie-break。判断一个排序是否足够稳定，可以问一句：如果前面的排序字段相同，下一步靠什么决定顺序？如果一直问到最后都没有唯一字段，结果中 tie 的相对顺序就仍然可能变化。

ORDER BY 和 UP TO n ROWS 的关系尤其重要。数据库先根据 WHERE 等条件形成 hit list，再排序，再截取 n 行。没有 ORDER BY 时，n 行是任意行；排序不唯一时，tie 中进入 n 行的具体记录也可能不确定。

SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              created_at
       WHERE customer_id = @lv_customer_id
       ORDER BY created_at DESCENDING,
                sales_order_id DESCENDING
       INTO TABLE @DATA(lt_latest_orders)
       UP TO 10 ROWS.

这段代码表达的是“该客户最新 10 张订单”。如果去掉 ORDER BY，它就只剩“任意 10 张订单”。如果只按 created_at DESCENDING 排序，而很多订单同一秒创建，那么第 10 条附近的 tie 仍然可能不稳定。

还要区分数据库排序和 ABAP 内表排序。ORDER BY 是数据库结果集进入 ABAP 之前的排序；SORT lt_orders BY ... 是结果已经进入 ABAP 内表之后的排序。两者都可能有用，但不能互相替代。尤其是 UP TO n ROWS、分页、offset 或“只取第一条”场景，必须先在数据库侧把顺序定义好，再截取结果。

5.5 日本项目现场说法

常见日语：

  • ORDER BY句
  • 並び順
  • ソート順
  • 昇順
  • 降順
  • 主キー順
  • 最新順
  • 優先順位順
  • タイブレーク
  • 先頭1件
  • 最新1件
  • 上位10件
  • 順序保証
  • 順序未定
  • 任意の1件

现场说明句：

  • ORDER BY がないため、取得結果の並び順は保証されません。
  • 最新1件を取得する場合は、更新日時の降順だけでなく、一意になるタイブレーク項目も指定してください。
  • UP TO 1 ROWS だけでは最新1件にはなりません。業務上のソート順を明示する必要があります。
  • 同一日時のデータが存在する場合、主キーなどで順序を確定させます。
  • 表示順だけであれば ABAP 側の SORT でもよい場合がありますが、上位N件を取得する場合はDB側の ORDER BY が必要です。

日本项目里，“順序保証” 是非常好用的 review 词。不要只说 ORDER BY を付けます，要能说 業務上必要な順序を保証するために、作成日時降順と伝票番号降順でソートします。

5.7 Example / 学习例子

5.7.1 例 1：列表显示的稳定顺序
SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status,
              created_at
       WHERE customer_id = @lv_customer_id
       ORDER BY created_at DESCENDING,
                sales_order_id DESCENDING
       INTO TABLE @DATA(lt_orders).

这段代码适合 Fiori 列表、报表输出或 API response。它不仅让结果“按时间倒序”，还在创建时间相同的情况下用 sales_order_id 固定 tie。

5.7.2 例 2：取最新一条
SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              status,
              created_at
       WHERE customer_id = @lv_customer_id
       ORDER BY created_at DESCENDING,
                sales_order_id DESCENDING
       INTO TABLE @DATA(lt_latest_order)
       UP TO 1 ROWS.

IF lt_latest_order IS NOT INITIAL.
  DATA(ls_latest_order) = lt_latest_order[ 1 ].
ENDIF.

这里不用非唯一条件下的 SELECT SINGLE。业务语义是“最新一条”，所以代码要显式说明“按什么算最新”。注意：如果 INTO 放在最后，SAP Samples 的现代语法示例中 UP TO n ROWS 位于 INTO TABLE ... 后面。

5.7.3 例 3：只按时间排序还不够
SELECT FROM zsales_order
       FIELDS sales_order_id,
              created_at
       WHERE customer_id = @lv_customer_id
       ORDER BY created_at DESCENDING
       INTO TABLE @DATA(lt_orders)
       UP TO 10 ROWS.

如果很多记录有相同 created_at，第 10 条附近的 tie 仍然不稳定。更稳的写法是加上能唯一化的字段：

ORDER BY created_at DESCENDING,
         sales_order_id DESCENDING
5.7.4 例 4：主键顺序不是业务顺序
SELECT FROM zsales_order
       FIELDS sales_order_id,
              customer_id,
              created_at
       WHERE customer_id = @lv_customer_id
       ORDER BY PRIMARY KEY
       INTO TABLE @DATA(lt_orders).

这能给出稳定的主键顺序，但不一定是最新顺、金额顺或状态优先级顺。项目 review 时要问：业务真的需要主键顺，还是开发者只是想让顺序稳定？

5.8 Counterexample / Failure mode

失败模式一：没有 ORDER BY 却读第一条。

SELECT FROM zsales_order
       FIELDS sales_order_id,
              created_at
       WHERE customer_id = @lv_customer_id
       INTO TABLE @DATA(lt_orders).

DATA(ls_first_order) = lt_orders[ 1 ].

lt_orders[ 1 ] 是第一行，但不是业务第一条。没有 ORDER BY，这里的第一行没有可解释含义。

失败模式二：UP TO 1 ROWS 假装最新。

SELECT FROM zsales_order
       FIELDS sales_order_id,
              created_at
       WHERE customer_id = @lv_customer_id
       INTO TABLE @DATA(lt_order)
       UP TO 1 ROWS.

这只表示“取任意一行”。如果业务是最新一行，必须补 ORDER BY created_at DESCENDING, sales_order_id DESCENDING。

失败模式三：排序字段不唯一。

ORDER BY created_at DESCENDING

如果 created_at 粒度只有秒，或者数据迁移时大量记录同一时间戳，tie 之间顺序仍然不确定。业务要稳定，就补 tie-breaker。

失败模式四：先截取再排序。

SELECT FROM zsales_order
       FIELDS sales_order_id,
              created_at
       WHERE customer_id = @lv_customer_id
       INTO TABLE @DATA(lt_orders)
       UP TO 10 ROWS.

SORT lt_orders BY created_at DESCENDING.

这段代码只是把任意 10 行在 ABAP 里排序，不是“最新 10 行”。要取最新 10 行，ORDER BY 必须在 SQL 里。

失败模式五：把 ORDER BY 当性能优化万能药。排序可以让业务结果稳定，但也可能增加数据库排序成本。大数据量、非索引字段、多字段排序和表达式排序都要结合索引、WHERE 条件和数据量评估。性能 review 不能只说“加 ORDER BY”，还要问排序字段是否合理。

5.9 思考问题

  • 这条查询后续是否依赖第一条、最后一条、前 n 条或分页？
  • 如果依赖，ORDER BY 是否写出了业务排序字段？
  • 排序字段是否能唯一决定顺序？如果不能，tie-breaker 是什么？
  • ORDER BY PRIMARY KEY 表达的是业务顺序，还是只是技术稳定顺序？
  • UP TO n ROWS 是否在排序之后才截取？没有排序时是否只是任意 n 行？
  • 如果已经用 ABAP SORT，它是在截取前还是截取后？能否替代数据库排序？
  • 是否存在 GROUP BY、aggregate function、FOR ALL ENTRIES、join、union 等会影响 ORDER BY 可用性的语法限制？
  • 排序字段是否在生产数据量上有性能风险？是否有合适索引或更小的 WHERE 范围？
  • 日本项目 review 中，能否用一句话说明 並び順を保証する根拠？
