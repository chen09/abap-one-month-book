6. Precheck 预检查

  • Chapter 04
  • Section 06
  • self-contained study note

6.1 一句话结论

Precheck 是“尽早拒绝明显不应继续的请求”的设计动作：在进入昂贵计算、锁定、编号、保存、外部调用或不可回滚副作用之前，先检查请求是否缺少必要数据、是否明显违反状态规则、是否没有基本操作资格，或者是否会造成后续流程必然失败。

它不是 RAP global authorization 全局授权、RAP instance authorization 实例授权、validation 或 lock 的替代品。更准确地说，precheck 是把“越早知道一定失败的事情，越早返回清楚错误”作为交付原则。做得好，系统更快、更可预测，错误更容易测试；做得差，问题会拖到 save sequence、外部接口或生产数据之后才暴露。

6.2 到底学什么

本节学习 precheck 的边界和设计时机。

第一，precheck 关注“早拒绝”。例如必填字段缺失、文件格式明显错误、action 不适用于当前状态、用户根本没有执行此类动作的资格、请求参数组合互相矛盾、重复请求已能被识别，这些都适合在进入正式处理前尽早挡住。

第二，precheck 不应承担所有最终一致性责任。某些业务规则必须在读取最新实例状态、加锁、保存前 validation、save sequence 或底层业务 API 中再次确认。precheck 可以降低失败成本，但不能假装后续世界不会变化。

第三，precheck 应避免副作用。预检查阶段如果已经创建正式数据、提交事务、调用不可回滚外部服务或消耗正式编号，就违背了“预”的意义。必要的轻量读取、参数检查、授权粗筛和幂等键检查可以做；真正的写入和不可回滚动作应留到正确阶段。

第四，precheck 要服务于错误体验。用户或 API 调用者应该尽早拿到明确错误：缺字段、无权限、状态不允许、重复请求、对象不存在、格式不合法。错误越晚、越模糊，测试和运维成本越高。

第五，precheck 要能形成交付清单。设计书不能只写“执行前做检查”，而要列出检查项、检查时机、是否有副作用、失败消息、是否按行返回、是否需要最终阶段再次确认。测试书也不能只测“validate 成功”，还要覆盖缺字段、重复 key、无权限、状态不允许、幂等重复、文件格式错误、对象不存在等负例。

6.3 为什么重要

生产系统最怕“明明一开始就知道会失败，却拖到最后才失败”。在 SAP 项目里，后续阶段往往涉及 enqueue lock、number range、update task、RAP save sequence、BAPI 调用、application log、外部接口、后台 job 或人工重试。一旦错误晚发生，就可能带来锁等待、编号空洞、部分处理、重复提交、难以解释的日志和用户投诉。

对 API 场景，precheck 尤其重要。调用者希望快速知道请求是否可执行，而不是等待系统处理半天后返回一个泛泛的失败。对批量上传场景，precheck 可以把文件结构、必填列、重复键、基本授权、明显状态冲突先报告出来，让用户在真正执行前修正。

从 Java/Python 背景看，可以把 precheck 类比成 controller/service 边界上的 request guard、dry-run check 或 early validation。但 SAP/RAP 的差异是：precheck 必须和授权、锁、事务边界、save sequence、reported/failed 消息和业务对象状态配合，不能只是一个孤立的 if。

precheck 的价值还在于保护后续资源。一个错误请求如果很早被拒绝，就不会占用 enqueue lock，不会消耗正式编号，不会把 update task 排进去，不会写“成功处理中”的 application log，也不会调用外部系统。它不是为了让代码多几层判断，而是为了把失败控制在最便宜、最可解释、最容易回滚的位置。

6.4 Key mechanism / 核心机制

precheck 可以按“越早越轻、越晚越权威”来理解：

Request received
  -> cheap syntactic checks
  -> basic authorization / operation eligibility
  -> idempotency or duplicate-key precheck
  -> state plausibility check
  -> lock / transaction / save sequence
  -> final validation and persistence

在 RAP 语境里，precheck 也不是随便起的名字，而是 behavior definition / projection 中可以声明的具体机制。读代码时可能会看到类似 create ( precheck )、update ( precheck )、delete ( precheck )，以及 ADT 生成的 precheck_create、precheck_update、precheck_delete、FOR PRECHECK 相关 handler 入口。不同 ABAP Platform / ABAP Cloud 版本、managed/unmanaged 场景和 projection 设置会影响实际签名，所以本节不背模板；但读者必须知道：RAP precheck 是“modify 请求进入后续处理前的早期检查钩子”，不是普通注释里的一个泛称。

它和 validation on save 的边界尤其重要。precheck 偏向在 create/update/delete 等请求进入更重处理前给出早期反馈；validation on save 偏向保存前的一致性闸门。项目里可以两者都存在：precheck 提前挡住明显错误，validation on save 兜住最终一致性。draft/update 场景还要注意 %control 和 delta payload：更新请求可能只带修改字段，不能把未传字段当成空值；必要时要通过 READ ENTITIES ... IN LOCAL MODE 读取当前实例状态，再按 changed fields 合并后判断。

早期检查适合做低成本、低副作用、高确定性的判断。例如字段是否为空、文件头是否正确、action 名称是否支持、用户是否完全没有此类操作权限、请求中同一个 key 是否重复、对象 ID 格式是否明显不合法。

后期检查则更权威。例如实例状态可能在 precheck 后被其他用户改变，所以保存前仍需要 validation 或 instance authorization；并发冲突需要 enqueue 或 optimistic locking；数据库唯一约束和业务 API 也可能在最终保存时发现问题。

一个实用判断是：如果某个检查失败时一定不能继续，且检查不需要产生副作用，就尽量放早；如果检查依赖最新数据、锁或正式保存上下文，就不能只依赖 precheck。

可以用下面的判断表区分：

检查项适合 precheck 吗为什么
必填字段缺失适合不需要锁和保存上下文，失败确定
文件格式明显错误适合解析失败时不应进入业务处理
请求内重复 key适合可以在内存中发现，避免后续重复处理
已处理过的幂等 key适合做早期提示可提前读取已处理记录，但最终仍要在锁内或保存前权威确认

precheck 的输出也要设计。批量场景里，错误通常应该能定位到 header、row、field 或 business key；RAP 场景里要通过 failed / reported、%tky、message class、状态区域或项目日志表达具体实例和字段的失败原因；API 场景里，应能区分 400 Bad Request 风格的输入错误、403 Forbidden 风格的权限错误、409 Conflict 风格的状态/并发/幂等冲突，以及真正的系统错误。SAP/RAP 项目不一定直接手写这些 HTTP status code，但这种分类有助于设计消息和测试期望。

6.5 日本项目现场说法

  • 常见日语：事前チェック、早期エラー返却、入力チェック、権限事前確認、重複チェック、ステータス確認、副作用を起こさない確認。
  • 设计说明：本 API では、保存処理に入る前に必須項目、重複キー、基本権限、対象ステータスを事前チェックします。
  • Review 说法：事前チェックで外部連携や正式更新などの副作用が発生していないか確認してください。
  • 测试说法：正常系だけでなく、必須項目不足、権限不足、重複リクエスト、ステータス不正の早期エラーを確認します。
  • 缺陷调查：エラーが保存後に発生している場合、事前チェックで検出できる条件か、最終 validation でしか検出できない条件かを切り分けます。
  • 设计书条目：チェック項目、チェックタイミング、副作用有無、エラーメッセージ、再チェック要否を一覧化します。
  • API 说明：validate は正式更新ではなく、入力・権限・重複・ステータスの早期確認を目的とします。
  • 边界说明：事前チェックで OK になっても、保存直前の validation、authorization、locking を省略してよい意味ではありません。
  • 证据要求：プリチェック設計書には、対象項目、%control 確認、ドラフト差分の扱い、エラーメッセージ、性能影響、テスト結果、承認記録を残します。

6.7 Example / 学习例子

以 AC0567C 类似的批量上传 API 为例，可以把流程分成 validate 和 execute。

validate 阶段适合做 precheck：

  • 文件是否能解析。
  • 必填列是否存在。
  • 每行 key 是否重复。
  • company code / plant / material 等字段格式是否明显错误。
  • 当前用户是否具备基本上传或执行资格。
  • 目标对象是否处于明显不能处理的状态。

execute 阶段仍要再次确认关键条件：

  • 实例状态是否在 validate 后被改变。
  • 是否需要锁定对象。
  • 是否通过最终 authorization / validation。
  • 是否已经处理过相同幂等键。
  • 保存失败时如何返回 failed/reported 和 application log。

这样设计后，用户可以先用 validate 得到可修正的问题清单，再用 execute 执行真正写入。precheck 不是“假执行”，也不是“提前保存”；它是降低失败成本的正式设计。

一个更接近项目设计书的 precheck 表可以这样写：

阶段检查内容失败响应是否产生正式副作用后续是否再确认
validate文件结构、必填列、行内重复 key返回行级错误清单否execute 前仍可重复检查关键字段
validate用户是否具备基本上传资格返回权限错误否execute 时仍做 operation/instance authorization
validatebatch 是否已处理完成返回重复/状态错误否execute 拿锁后再读最新状态
execute before lock请求格式、幂等键、cheap authorization早期拒绝否进入关键区前完成
execute after lock最新状态、实例授权、业务规则failed/reported 或业务错误否/受控save sequence 前最终确认
save sequence数据库约束、保存 API、不可避免的运行时错误保存失败并记录日志可能有受控日志由事务边界决定

以 AC0567C 上传为例，validate 可以告诉用户：“第 12 行缺少 material，第 18 行 company code 不在允许范围，第 21 行和第 25 行 key 重复。”这类错误不需要锁，也不应该写正式表。execute 则要面对更接近生产的事实：validate 之后别人可能已经处理了同一个 batch，用户授权可能变了，状态可能更新了，幂等键可能已被使用。因此 execute 不能只相信 validate 的结果，而要在关键点重新确认。

一个好的 API 响应还应避免把所有错误混在一起：

Input error:
  missing field, invalid format, duplicate row key

Authorization error:
  user cannot execute this operation or organization value

Business-state error:
  batch already processed, object not in executable status

Conflict / retry error:
  same idempotency key already processed, object locked by another process

System error:
  unexpected save failure, downstream unavailable

这种分类能帮助调用方决定下一步：修文件、换用户、等待重试、放弃重复请求，还是联系运维。

6.8 Counterexample / Failure mode

常见失败模式有十类。

第一，没有 precheck。所有错误都到保存阶段才出现，用户只能拿到晚而模糊的失败。

第二，precheck 做了副作用。预检查阶段就写正式表、调用外部系统、消耗编号或提交事务，失败后很难清理。

第三，把 precheck 当最终保证。预检查通过后不再做最终 validation、authorization 或 locking，导致并发变更后仍错误保存。

第四，错误分类混乱。权限不足、输入错误、状态不允许、重复请求都返回同一种消息，测试和调用方无法处理。

第五，只测 happy path。没有验证无权限、重复请求、边界状态、批量部分失败和 API 直调。

第六，precheck 太重。为了“提前发现所有问题”读取大量数据、执行复杂计算或锁定资源，反而拖慢接口并制造竞争。

第七，validate 和 execute 语义混乱。validate 里已经写正式表，execute 只是补提交；或者 execute 完全相信 validate 的旧结果，不再做最新状态检查。

第八，批量错误粒度不清。一个文件里只有一行错，却返回整批“失败”；或者整批失败但没有指出行号、字段和业务 key。

第九，把 precheck 放在锁内做太久。先拿锁，再做大量文件解析、权限粗筛和格式检查，会让其他用户长时间等待。一般应把 cheap、side-effect-free 检查放在锁前，把依赖最新状态的确认放在锁后。

第十，日志写反。precheck 失败却写成业务处理失败或成功日志，后续审计会误以为正式执行过。precheck 日志应明确是 early rejection / validation result，而不是 final posting result。

修正动作：把这些失败模式归入 early check、authorization、instance rule、locking、final validation、save error 等责任桶，并为每一类指定错误消息、测试用例和证据位置。

6.9 思考问题

  • 这个错误是否可以在进入保存/外部调用/锁定前发现？
  • precheck 是否产生了任何正式副作用？
  • precheck 通过后，哪些条件仍需要在最终保存前重新确认？
  • 错误消息是否区分输入错误、权限错误、状态错误、重复请求和系统错误？
  • 批量请求中是否能返回每行/每实例的预检查结果？
  • precheck 的错误分类是否能指导调用方下一步行动？
  • 哪些检查应该放在锁前，哪些必须放在锁后重新确认？
  • AC0567C POC 中 validate 与 execute 的职责是否清楚分开？

6.10 参考资料入口

书稿补强：Precheck 是早拒绝，不是提前保存

Precheck 的核心价值是把“明显不能继续的请求”尽早挡住，而且尽量不产生副作用。它适合检查输入格式、必填字段、重复 key、基本权限、对象是否存在、当前状态是否显然不允许执行等问题。它不适合偷偷创建正式数据、发送外部请求、消耗不可回滚的 number range、写成功日志或替代最终保存阶段的 validation。换句话说，precheck 的理想结果是：失败时给出清楚、可修正的原因；成功时只说明“可以进入下一阶段”，而不是承诺“最终一定会保存成功”。

读者可以把 precheck、validation、authorization、locking 和 save sequence 分成一条时间线：precheck 负责便宜、早期、可重复的检查；authorization 负责谁能做；locking 负责并发期间谁能改；validation on save 负责保存前基于最新状态的业务判断；save sequence 负责真正持久化和失败回滚。这样就不会把“前面检查通过”误解成“后面不会失败”。在 RAP 或批处理 API 中，即使 validate/precheck 通过，execute 时仍然要重新读取关键状态、重新确认锁或实例授权，并在失败时返回清楚的 failed / reported 或业务错误结构。

项目里最容易出事故的是“看起来像检查，实际上已经做了事”。例如 validate 接口顺手写了 application log 的成功记录、预检查中调用外部系统确认额度、或者为了判断是否重复而先插入一条正式记录。这些设计会让用户以为只是检查，系统却已经改变状态。更稳的做法是把 precheck 输出设计成一个 evidence object：包含 check item、target field、business key、error category、retryable flag、message 和下一步建议；真正的保存、外部发送、不可回滚副作用留给明确的 execute/save 阶段。

日本项目现场可以这样说：事前チェックは副作用を起こさず、早期に修正可能なエラーを返すための処理です。保存可否の最終判断は、最新状態を再確認したうえで保存処理側で行います。也就是：预检查负责早期发现，最终保存判断仍要在真正保存前重新确认。
