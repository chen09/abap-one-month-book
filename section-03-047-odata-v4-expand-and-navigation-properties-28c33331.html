47. OData V4 expand and navigation properties

  • Chapter 03
  • Section 47
  • self-contained study note

47.1 一句话结论

OData V4 expand and navigation properties 讲的是：CDS/RAP 中的 association 或 composition 经过 projection、service definition、service binding 暴露后，会在 OData V4 metadata 中形成 navigation property；消费者可以用 $expand 在一次请求中带出关联对象。它不是“随便把 join 结果塞进 JSON”，而是 metadata-driven 的导航契约。

一句话记忆：association 是模型里的关系，navigation property 是 OData metadata 里的关系入口，$expand 是消费者沿着这个入口取关联数据的请求方式。

47.2 到底学什么

前面学过 CDS association、path expression、projection view、service definition 和 OData V4。现在要把它们连起来。一个 Travel root entity 可能关联 Booking item。CDS 里写 _Booking association；projection view 决定是否暴露这条关系；service definition / binding 决定它是否进入服务；OData metadata 中出现 navigation property；消费者才能请求 Travel?$expand=_Booking。

这不是单纯 SQL join。Join 是服务端建模时把表按条件合并；navigation property 是服务 contract 中公开“可沿着关系导航”的能力。$expand 是消费者选择在这次请求里把关联对象一起取回。它们背后的 SQL 可能由 runtime 优化生成，但设计语义不同。

本节要学三个判断。第一，为什么 metadata 中有无 navigation property，取决于 CDS/projection/service 暴露。第二，$expand 很方便，但可能带来性能、权限和 payload 变大的风险。第三，Fiori object page、list/detail、外部 API 读取父子数据，常常都依赖 navigation 和 expand，因此要把它纳入 API 测试。

第四，要学会把 $expand 当成“消费者选择读取关系”的语法，而不是服务端默认行为。一个服务可以在 metadata 中声明 navigation property，但消费者每次请求是否展开、展开到哪一层、是否配合 $select / $filter，仍然是 API contract 和调用方责任。项目上不能只说“有 association”，还要说“哪些 navigation 支持 expand、推荐怎样调用、哪些场景不允许深度展开”。

第五，要学会从 $metadata 反推问题。遇到 expand 失败，不要马上改 ABAP handler。先看 metadata：entity type 里有没有 NavigationProperty，entity set 是否存在，目标 entity 是否被 service definition 暴露，navigation 名称是否和请求 URL 完全一致，cardinality 是否符合预期。很多问题其实是暴露链断了，而不是 runtime 出错。

47.3 为什么重要

这个知识点重要，是因为很多 Fiori 和 API 问题表面看是“页面没有子表”“接口读不到明细”“expand 失败”，实际根因在 CDS association、projection 暴露、service metadata、DCL/RAP authorization 或性能边界。

如果 navigation 没有进 metadata，前端就没有稳定方式从 header 导航到 item。开发者可能用手写 URL 或额外接口绕开，结果接口契约混乱。反过来，如果把所有 association 都暴露，外部消费者可能读到不该公开的关系，payload 变大，权限边界模糊，性能也可能变差。

对交付来说，navigation/expand 是 API contract 的一部分。设计书应该说明哪些 entity 之间允许导航、cardinality 是什么、是否支持 expand、是否受权限过滤、是否有 payload 限制、是否需要分页或限制层级。测试证据也要包括 $metadata 中的 navigation property 和实际 $expand 响应。

对 Java/Python 背景学习者，可以把 navigation property 类比成资源关系或 GraphQL field selection，但 OData 的不同是它由 metadata 明确声明，且和 CDS association、RAP BO composition、Fiori metadata 和 authorization runtime 绑定。

47.4 Key mechanism / 核心机制

第一层机制是 CDS association/composition。CDS view entity 中定义 association，RAP BO 中可能还有 composition parent-child 关系。它们表达模型关系，不等于消费者一定能看到。

第二层机制是 projection exposure。Projection view 可以暴露、重命名或隐藏 association。如果 projection 层没有公开这条关系，service metadata 中通常也不会出现对应 navigation。

第三层机制是 service metadata。Service binding 发布后，OData V4 $metadata 中会出现 entity type、entity set、navigation property。是否能 $expand，首先看 metadata 中有没有这个 navigation property。

第四层机制是 $expand 请求。消费者可以请求类似：

GET /Travel?$select=TravelID,Status&$expand=_Booking

这表示读取 Travel 时一并展开 _Booking。也可以结合 $select 限制字段，避免 payload 过大。

第五层机制是权限过滤。Expand 出来的 child data 仍然要符合授权设计。DCL、RAP authorization、service 层权限可能影响 parent 或 child。不能因为 parent 可见，就默认所有 child 都可见。

第六层机制是性能。Expand 可能触发更多 join、更多查询或更大响应。对一对多关系，尤其要注意数据量、分页、过滤、嵌套 expand 和网络 payload。API 设计不能把无限层级关系全部开放给消费者。

第七层机制是 Fiori 消费。Object page 的子表、facet、navigation、line item 展开常依赖 navigation property 和 annotation。页面缺子对象时，要同时检查 CDS association、projection、metadata extension、service definition、binding 和 authorization。

第八层机制是 URL 组合。OData V4 的 $expand 通常和 $select、$filter、$orderby、$top、$skip 等 system query option 一起设计。初学者最容易写成：

GET /Travel?$expand=_Booking

这能说明方向，但生产 API 更常需要限制字段和数据量：

GET /Travel?$select=TravelID,Status&$expand=_Booking($select=BookingID,Status)

含义是：Travel 只返回 TravelID、Status，展开的 Booking 也只返回 BookingID、Status。这样既保留 parent-child 关系，又避免把所有字段和所有明细无节制地传给消费者。

第九层机制是 expand depth。一级 expand 已经可能带来很大 payload；多层 expand 例如 header -> item -> schedule line -> text，会把 API 从“读取一个对象”变成“读取一棵对象树”。有些服务或网关会限制 expand 层级、结果大小或查询复杂度。设计书中应写清推荐层级和不支持场景，而不是让消费者自由试探。

第十层机制是 parent-child 语义。RAP composition 里的 child entity 往往天然适合 object page 子表或 header-item API，但“适合导航”不等于“适合一次 expand 全取”。如果 child 数据量大、权限复杂、或需要独立分页，最好设计单独的 child entity set 查询或带过滤的 navigation，而不是强迫所有消费者用一个巨大 expand。

可以把技术链路写成下面的调试顺序：

CDS association/composition
  -> projection exposes association
  -> service definition exposes source and target
  -> service binding is published/refreshed
  -> $metadata contains NavigationProperty
  -> consumer sends $expand with the exact navigation name
  -> authorization and data volume checks pass
  -> response contains expected child collection/entity

其中任意一环断掉，最终都会表现为“expand 不工作”。这就是为什么 OData 问题不能只看 URL，也不能只看 CDS。

$expand contract checklist

把 navigation property 暴露给消费者之前，建议把 $expand 当成 API contract 逐项确认。尤其是外部 Web API，一旦消费者开始依赖 navigation 名称和返回结构，后续重命名、隐藏、改变 cardinality 或改变 child 字段都会变成 breaking change。

Check要确认什么风险
Navigation namemetadata 中的 NavigationProperty 名称是否稳定改名会直接破坏消费者 URL
Source / targetsource entity 和 target entity 是否都在 service definition 中按预期暴露target 未暴露时 expand 不可用或 metadata 不完整
Cardinality[0..1]、[0..*] 等语义是否真实错误 cardinality 会误导 UI/API consumer
Field scopechild entity 是否需要 $select 限制字段payload 过大或泄漏内部字段
Expand depth是否只支持一级 expand，是否禁止深层对象树深层 expand 会导致性能和响应大小失控
Authorizationparent 和 child 权限是否分别验证parent 可见不代表 child 可见
Empty result无 child 时返回空集合还是按约定返回 null/缺省consumer 解析逻辑不一致
Performance大 parent、大 child、多 child 场景是否测过测试数据少时正常，上线后超时
Versioningnavigation 改名、隐藏、字段变化是否有版本策略外部系统无预警失败

这张 checklist 也适合调试。遇到 $expand 失败，不要只看 HTTP status；先保存 $metadata、请求 URL、当前用户、parent key、child 数据量、权限结果和响应 payload，再判断是暴露链、权限、数据、性能还是消费者 URL 写错。

47.7 Example / 学习例子

假设 Travel 有多个 Booking：

CDS root/projection:
  Travel
    association/composition _Booking

OData metadata:
  EntitySet Travel
  NavigationProperty _Booking

HTTP request:
  GET /Travel?$select=TravelID,Status&$expand=_Booking

一个最小测试证据要包含：

1. `$metadata` 中有 Travel entity set。
2. Travel entity type 中有 `_Booking` navigation property。
3. `$expand=_Booking` 返回 booking 明细。
4. 无权限用户不能通过 expand 看到不该看的 child data。
5. 大数据量 parent 不会返回不可控 payload。

如果 metadata 中没有 _Booking，不要先怪前端。先检查 CDS association 是否存在，projection 是否暴露，service definition 是否暴露相关 entity，binding 是否重新 publish，metadata cache 是否更新。

更完整的请求/响应设计可以这样写：

Request:
  GET /Travel?$select=TravelID,Status
      &$expand=_Booking($select=BookingID,BookingDate,Status)

Expected response shape:
  TravelID
  Status
  _Booking: [
    { BookingID, BookingDate, Status },
    ...
  ]

Checks:
  Parent key is stable.
  Child collection name matches metadata.
  Unauthorized child rows are not leaked.
  Empty child collection returns clearly.
  Large child collections are limited or paged by design.

如果 AC0567C 有 upload batch header 和 upload item，可以把 header -> item 关系作为 navigation 候选。API 设计时要问：状态查询是否需要一次带出全部 item？错误明细是否应该用 $expand 返回？大批量文件的 item 数量是否太多，应该改成单独查询 Items?$filter=batch_id eq ...？这类问题比“能不能 expand”更接近项目交付。

47.8 Counterexample / Failure mode

反例一：把 $expand 当 join。开发者以为 expand 只是后端 join，于是无节制开放所有关系。修正方式是把 navigation 当 API contract 管理。

反例二：projection 没暴露 association。CDS interface 有 _Item，但 projection/service metadata 没有，前端无法 expand。修正方式是检查 projection 和 service 暴露链。

反例三：expand 泄漏 child 数据。Parent 可见，child 应该按权限过滤却全部返回。修正方式是确认 DCL/RAP authorization/service 层权限。

反例四：payload 过大。一个 header expand 出几千条 item，页面和网络都慢。修正方式是限制 expand、分页、过滤或设计专用查询。

反例五：metadata 变了但消费者没同步。外部系统仍按旧 navigation 名称调用。修正方式是把 navigation 名称作为 API contract 管理。

反例六：只测 expand 成功，不测无权限、空 child、多 child、性能。修正方式是补 API testing workflow。

反例七：把 deep expand 当成免费功能。header 一次展开 item、schedule、log、message、text，测试数据少时很快，上线后 payload 巨大。修正方式是限制层级、字段和结果数量，必要时拆成专用查询。

反例八：navigation 名称随手改。外部系统已经依赖 _Items，projection 重命名或隐藏后没有版本策略，导致消费者失败。修正方式是把 navigation property 名称纳入 API contract。

反例九：只在开发者权限下测试。开发者能 expand 出 child，不代表业务用户、技术用户、外部 OAuth client 也应该看到同样数据。修正方式是用目标角色测试 parent 和 child 授权。
