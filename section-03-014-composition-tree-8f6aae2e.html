14. Composition tree 组合树

  • Chapter 03
  • Section 14
  • self-contained study note

14.1 一句话结论

Composition tree（组合树）是 RAP business object 里表达 root entity、child entity、grandchild entity 生命周期依赖关系的结构。它告诉 RAP runtime：哪些节点共同组成一个业务对象，谁是入口，谁依附于谁，哪些数据应该作为一个事务边界一起创建、校验、保存、锁定和暴露。

初学者要先把它和普通 association 分开。Association 表达“可以导航到另一个实体”；composition 表达“这个 child 是 parent 的组成部分，生命周期依赖 parent”。在 RAP 里，composition tree 不是画图装饰，而是 BO 数据模型和后续 behavior definition 的骨架。

14.2 到底学什么

本节学五件事：

  1. root entity 为什么只能有一个，为什么它代表整个 RAP BO 的入口。
  2. child entity / grandchild entity 为什么通过 composition 而不是普通 association 进入 BO。
  3. composition [0..*] of ... 和 association to parent ... 各自写在哪里。
  4. composition tree 如何影响 create-by-association、delete cascade、lock master / lock dependent、authorization、save sequence 和 service exposure。
  5. 什么时候不该建 composition，而只该建 ordinary association。

14.3 为什么重要

上一节 RAP business object 业务对象 讲的是 BO 总体边界。本节进一步回答：BO 内部如果不止一个实体，结构应该怎么建。

真实业务很少只有一张表。一个采购申请有 header 和 item；一个 Travel 有 Booking 和 Booking Supplement；一个导入批次有 import header 和 rows；一个订单有 header、items、partners、texts、pricing conditions。问题不只是“这些表能不能 join”，而是：

  • item 是否能脱离 header 单独存在？
  • 删除 header 时 item 是否应该一起消失或一起失效？
  • 创建 item 是否必须通过 header？
  • 保存 header 和 item 是否属于同一个事务？
  • 锁 header 时是否也要限制 item 修改？
  • UI / OData 是否需要从 root 导航到 child？
  • child 的 authorization 是自己判断，还是依赖 root / lock master？

Composition tree 把这些问题提前放进模型层，而不是等到 handler 里用一堆 if / SQL 临时补。对 Java/Python 背景学习者，可以类比 aggregate root 和 aggregate members，但 SAP RAP 的 composition tree 还会影响 CDS syntax、BDL、service exposure、draft、lock、authorization 和 save sequence，因此不能只按普通 ORM 关系理解。

14.4 Key mechanism / 核心机制

14.4.1 Composition tree 有且只有一个 root

SAP Learning 对 RAP composite BO 的限制讲得很明确：composition tree 中有且只有一个 root entity。root 是业务对象的入口和顶层节点，可以作为 composition 的 source，但不能作为 composition 的 target。

这句话对初学者很重要。一个 BO 不能同时有两个互不相属的 root。若需求看起来有两个 root，例如 Customer 和 Sales Order，那通常说明它们是两个业务对象，通过普通 association 或 service 关系协作，而不是同一棵 composition tree。

在 CDS 中，root entity 通常用 define root view entity 表达：

define root view entity ZI_Travel
  as select from ztravel
  composition [0..*] of ZI_Booking as _Booking
{
  key travel_id as TravelID,
      agency_id as AgencyID,
      customer_id as CustomerID,
      _Booking
}

这里 ZI_Travel 是 root，_Booking 是从 root 指向 child 的 to-child composition。

14.4.2 Child 必须有 to-parent association

Composition 是双向建模的一组特殊 CDS association。父实体定义 to-child composition；子实体定义 association to parent 指回父实体。ABAP Keyword Documentation 明确说明，to-child associations 和 to-parent associations 一起定义 RAP BO 的 CDS composition tree。

这里有一个很容易在 ADT 里立刻踩到的顺序问题：先在 child entity 中定义并激活 association to parent ... on ...，再回到 parent entity 中定义 composition ... of Child as _Child。ON condition 写在 child 的 to-parent association 上；parent 侧的 composition 不再写 ON condition，composition 的连接条件来自 to-parent association。如果反过来先写 parent composition，而 child 还没有激活 to-parent association，通常会遇到 syntax check / activation 问题。

一个 child view entity 通常长这样：

define view entity ZI_Booking
  as select from zbooking
  association to parent ZI_Travel as _Travel
    on $projection.TravelID = _Travel.TravelID
{
  key travel_id  as TravelID,
  key booking_id as BookingID,
      flight_no  as FlightNo,
      _Travel
}

这里的 $projection.TravelID 指当前 child view entity 已经暴露出来的 TravelID 元素。初学阶段可以先把它理解成“用本视图投影后的字段去和 parent 的 key 对齐”。

注意方向：

  • parent 里写 composition ... of Child as _Child。
  • child 里写 association to parent Parent as _Parent。
  • ON condition 只写在 child 的 association to parent 里。
  • child 的 to-parent cardinality 在 RAP BO 中通常必须是 [1..1] 语义：一个 child 实例属于一个明确 parent。

如果只在 parent 里写了普通 association，RAP 不会把它理解成 child 生命周期依赖。普通 association 可以导航；composition 才表达“组成部分”。

14.4.3 Composition vs association：不是强弱 join 的区别

很多初学者把 composition 理解成“更强的 join”。这个理解不对。Composition 和 association 不是性能语法的强弱版本，而是业务生命周期语义不同。

可以这样判断：

  • 如果目标实体可以独立存在、独立维护、独立授权，只是当前页面需要读它的文本或状态，用 ordinary association。
  • 如果目标实体必须依附 parent 存在，通常通过 parent 创建、修改、删除、保存和锁定，用 composition。
  • 如果目标实体是主数据，例如 Customer、Material、Company Code，即使当前 BO 经常引用它，也通常不是 child；它更适合 association。
  • 如果目标实体是明细行，例如 Sales Order Item、Travel Booking、Import Row，就更可能是 child。

错误使用 composition 会让 BO 边界变得过大；错误使用 association 会让 child 生命周期散落在代码里，导致保存、删除、锁、授权和 UI 导航都变复杂。

14.4.4 Cardinality 是模型契约

Composition 也有 cardinality。SAP Learning 提到，RAP BO 中 composition 的 cardinality 可以是 0..1 或 0..n；to-parent association 的 cardinality 必须是 1..1。初学时先用业务语言理解：

  • [0..1]：一个 parent 最多有一个 child，例如一个 header 最多有一个 extension detail。
  • [0..*]：一个 parent 可以有多条 child，例如一个 order 有多条 item。
  • to-parent [1..1]：每个 child 必须归属一个 parent。

Cardinality 不只是给 UI 看。它会影响路径导航、join 语义、结果行数、service metadata 和 review 判断。写错 cardinality，可能短期还能激活，但后续 Fiori、OData navigation、EML create-by-association、性能分析都会变难。

14.4.5 Composition tree 会进入 behavior definition

Composition tree 是 CDS 层的结构骨架；behavior definition 会在这个骨架上定义每个节点的行为。一个 composite BO 的 BDEF 可能会包含：

define behavior for ZI_Travel alias Travel
{
  create;
  update;
  delete;
  association _Booking { create; }
}

define behavior for ZI_Booking alias Booking
{
  update;
  delete;
  association _Travel;
}

这段只是教学示意，不是完整模板。它表达的意思是：Travel root 可以创建、修改、删除；Booking child 可以通过 _Booking composition 从 Travel 创建。真正项目中还要看 managed / unmanaged、lock、authorization、etag、draft、field properties、validation、determination、action 等定义。

14.4.6 Composition tree 影响 save、lock 和 authorization

Composition tree 不只是读模型。它会影响事务型行为：

  • Save：root 和 child 的修改通常在同一个 BO 事务中提交或回滚。
  • Create by association：child 往往通过 parent 的 composition association 创建。
  • Delete：删除 root 时，child 生命周期如何处理必须明确，不能只靠数据库外键想象。
  • Lock：child 可以依赖 root 或 ancestor 的 lock master，避免 header/item 被不同用户分别改坏。
  • Authorization：root-level authorization、instance authorization、child-level checks 要和 BO 层级一致。多层 composition 中，lock、authorization、ETag master 不一定都要通过直接 ordinary association 连到 root；较新的 RAP/BDL 语法还支持 ancestor association 等机制表达跨层主从关系。初学阶段先记住：不要只看相邻 child，要确认整棵树的 master/dependent 关系。
  • Draft：draft-enabled composite BO 还要考虑每个节点的 draft persistence。

这就是为什么 composition tree 不是“画一棵树给 UI 看”，而是事务模型的主干。

14.4.7 Projection layer 也要处理 composition

前面 Projection view 投影视图 讲过，projection view 是 service/consumer layer。Composite BO 暴露服务时，projection root、projection child、projection behavior 也要正确处理 composition 和 redirection。不能只投影 root，却忘记 child；也不能在 service definition 中暴露了 child，却没有清楚的 navigation / behavior projection。

在 Fiori Elements 或 OData V4 UI service 中，root 和 child 的 composition 会影响对象页、子表、navigation、create subnode 等体验。读者不需要在本节掌握所有 annotation，但要知道：composition tree 设计错，后面 UI 和 API 暴露都会跟着别扭。

14.4.8 AC0567C POC 如何借鉴 composition thinking

AC0567C POC 当前更像 action-oriented API，并不一定是完整 header/item composite BO 教学样板。但它的业务输入天然有“批次 + 多行”的结构：

PriorAllocationImport root
  -> ImportRow child

如果未来把 AC0567C 从 POC hardening 成生产级 RAP BO，可以考虑是否把 upload request/header 作为 root，把 rows 作为 child，借助 composition tree 表达一批导入数据的生命周期。这样 validate / execute / status action、duplicate check、authorization、save sequence、error messages 都可以围绕同一个 BO 结构设计。

本节不要求立即重构 AC0567C。它只是提醒：看到“一个请求带多行明细”时，不要只想到 JSON array 和 internal table，也要想到 RAP composite BO 是否更适合表达这个业务对象。

14.4.9 观察 composition tree 的最小路径

在 ADT 里观察一个 composite BO，可以按这个顺序：

  1. 打开 root view entity，看是否有 define root view entity。
  2. 在 root 中找 composition ... of Child as _Child。
  3. 打开 child view entity，看是否有 association to parent Root as _Parent。
  4. 检查 cardinality：to-child 是 [0..1] 还是 [0..*]，to-parent 是否是 [1..1] 语义。
  5. 打开 behavior definition，看 child 是否通过 composition association 支持 create/update/delete。
  6. 打开 projection layer，看 composition 是否被正确投影和重定向。
  7. 打开 service definition / binding，看 root-child navigation 是否按预期暴露。
  8. 如果有多层 child，检查 lock master、authorization master、ETag master 是否通过相邻 parent 或 ancestor association 表达清楚。

14.7 Example / 学习例子

14.例 1：Travel / Booking / Booking Supplement

经典 RAP 教学例子是 Travel composite BO：

Travel root
  -> Booking child
      -> Booking Supplement grandchild

Travel 是 root。Booking 不能脱离 Travel 成为同一个 BO 中的独立入口。Booking Supplement 又依附 Booking。这个结构适合演示 root、child、grandchild、composition 和 to-parent association。

读这个例子时，不要只背名字。要检查每一层：

  • parent 是否声明了 composition？
  • child 是否声明了 association to parent？
  • behavior definition 是否允许通过 parent 创建 child？
  • service projection 是否暴露了 root-child navigation？
  • lock / authorization / draft 是否和层级一致？
14.例 2：Sales Order Header / Item

销售订单也很适合类比：

SalesOrder root
  -> SalesOrderItem child

Item 通常不是独立业务对象。它依附订单存在，订单删除、锁定、状态变更、保存失败时，item 也必须一起考虑。若把 item 只做成普通 association，开发者可能会在 handler 里手工补一堆生命周期逻辑，最后造成不同入口行为不一致。

14.例 3：AC0567C 的导入批次思考

AC0567C POC 已经证明 action-oriented API 能工作。若未来把它建成更典型 composite BO，可以考虑：

ImportBatch root
  -> ImportRow child

这样 validate action 可以针对整个 batch，row-level validation 可以返回到具体 child 行，execute action 可以按 batch 控制 save sequence，status 可以按 root 查询。是否采用这种设计，要结合项目复杂度、UI/API 需求、持久化表结构和 Clean Core 约束判断。

14.8 Counterexample / Failure mode

  • 误区 1：把 composition 当成 stronger association。修正：composition 表达生命周期依赖，不是更强 join。
  • 误区 2：把主数据做成 child。修正：Customer、Material、Company Code 等通常独立存在，应优先考虑 association。
  • 误区 3：只有 parent 写 composition，child 没有 association to parent。修正：composition tree 需要 to-child 和 to-parent 一起建模。
  • 误区 3a：先写 parent composition，再补 child to-parent association。修正：先定义并激活 child 的 association to parent ... on ...，再定义 parent 侧 composition；ON condition 不写在 composition 侧。
  • 误区 4：cardinality 随便写 [0..*]。修正：cardinality 是业务契约，会影响 navigation、结果行数、service metadata 和 review。
  • 误区 5：root / child 都能独立创建。修正：如果 child 生命周期依赖 root，通常应通过 composition association 创建。
  • 误区 6：只暴露 root service，忘记 child projection / behavior projection。修正：service layer 要和 BO 层级一致。
  • 误区 7：删除、锁、授权只按单表考虑。修正：composition tree 要和 delete、lock master / dependent、authorization、save sequence 一起 review。
  • 误区 8：把 JSON array 等同于 composition tree。修正：API 输入结构像树，不代表 RAP BO 已经建成树；要看 CDS composition 和 BDEF。

14.9 思考问题

  • 这个 BO 是否只有一个 root entity？
  • 哪些实体是 child？它们能不能脱离 parent 独立存在？
  • parent 是否用 composition ... of 指向 child？
  • child 是否用 association to parent 指回 parent？
  • to-child cardinality 是 [0..1] 还是 [0..*]？为什么？
  • child 是否需要通过 parent 的 association 创建？
  • lock master / lock dependent 是否和组合树一致？
  • 多层 child 的 lock / authorization / ETag master 是直接 parent，还是 ancestor？
  • authorization 是 root 统一控制，还是 child 也需要 instance authorization？
  • service projection 是否暴露了正确的 root-child navigation？
  • AC0567C 的 batch / row 结构是否适合未来建成 composite BO？
