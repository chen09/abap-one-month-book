20. ABAP SQL Test Double Framework

  • Chapter 04
  • Section 20
  • self-contained study note

20.1 一句话结论

ABAP SQL Test Double Framework 是在 ABAP Unit 中隔离 ABAP SQL（旧资料常称 Open SQL，类名中也常出现 OSQL）数据库依赖的测试能力。本教程里要学的是：让读取数据库的业务逻辑可以用受控测试数据验证，而不是依赖当前系统里“刚好存在”的真实业务数据。

上一节 OO Test Double Framework 讲的是替换面向对象依赖，比如权限检查器、外部 API adapter、clock provider。ABAP SQL Test Double 关注的是另一类依赖：代码里直接或间接执行的 ABAP SQL。很多服务逻辑的判断都来自数据库，例如“这个 request id 是否已存在”“公司代码是否有效”“某个状态是否允许执行”。如果测试依赖真实表数据，测试结果会随环境变化。

SQL Test Double 的目标不是模拟整个数据库，也不是验证 HANA 性能，而是在单元测试里准备一小组可控数据，让被测代码执行 Open SQL 时读到这些测试数据。这样就能稳定验证 WHERE 条件、JOIN 逻辑、空结果、重复结果、排序假设、状态判断和错误分支。

20.2 到底学什么

第一，要学它解决的问题。服务逻辑常常需要查表：主数据是否存在、配置是否维护、历史请求是否重复、状态是否允许、关联明细是否完整。如果测试直接读开发机或测试机的真实表，就会出现“今天通过、明天失败”“别人改了数据我测试坏了”“为了测试必须手工维护很多数据”的问题。

第二，要学它和 OO Test Double 的区别。OO Test Double 替换的是对象依赖；ABAP SQL Test Double 替换的是 Open SQL 访问的数据来源或表内容。两者经常配合使用：repository class 可以用 OO double 替换；如果要测试 repository 里的 SELECT 逻辑，则可以用 SQL test double 准备表数据。

第三，要学适合测试什么。它适合验证查询逻辑和基于查询结果的业务判断，例如 WHERE 是否漏条件、JOIN 是否选错键、空结果是否返回正确业务错误、重复记录是否被正确处理、ORDER BY 是否明确、聚合结果是否正确。它不适合证明生产数据库性能、数据库索引选择、真实权限、真实锁或真实事务提交行为。

第四，要学数据最小化。测试数据应小而清楚：一条成功数据、一条缺失数据、一条重复数据、一条边界数据。测试不是把生产表复制一份，而是用最少数据证明规则。

第五，要学测试 repository/query，而不是把所有业务流程塞进 SQL double。SQL Test Double 最适合验证 repository class、query helper、read model adapter 这类“把输入条件翻译成 SQL 结果”的代码。更高层 service logic 可以用 OO Test Double 替换 repository；更低层性能、锁和真实授权要交给 trace、集成测试和系统证据。

第六，要学 fixture 可读性。测试数据不是越真实越好，而是越能说明意图越好。每一行测试数据都应回答一个问题：匹配记录、干扰记录、边界记录、重复记录、空结果记录。读者看到数据就知道它在验证哪个 WHERE/JOIN/ORDER BY 约定。

20.3 为什么重要

数据库依赖是 ABAP 单元测试中最常见的障碍之一。很多 ABAP 代码一写就 SELECT，导致业务逻辑和系统状态绑在一起。测试时如果没有隔离，开发者会倾向于放弃单元测试，只做手工测试或集成测试，回归保护就变弱。

对项目交付来说，SQL Test Double 可以保护很多“看起来简单、实际容易错”的逻辑。比如 request id 幂等判断少了 client 或 company code 条件，测试数据一构造就能暴露；JOIN 键错了，测试可以用两条相似数据抓出来；空结果应该返回 CONFIGURATION_MISSING，却被写成技术异常，测试可以固定这个契约。

对 AC0567C 这类 API/批处理场景，SQL Test Double 特别适合验证 status/read/execute 背后的读取逻辑。端到端 API 测试证明服务整体能跑；SQL Test Double 证明某个 repository 或查询规则在小数据集下是正确的。两者结合，才能既快又有证据。

这类测试还有一个现实价值：防止“数据巧合”。真实测试环境里，某些字段值刚好唯一，JOIN 键写错也可能返回正确结果；某个 request id 刚好不存在，漏掉 company code 条件也看不出来。SQL Test Double 可以故意准备相似数据和干扰数据，让错误条件一定暴露。

20.4 Key mechanism / 核心机制

ABAP SQL Test Double 的核心机制可以概括为：在测试中创建受控的 SQL 测试环境，向相关表或 CDS 依赖准备测试数据，让被测代码执行 ABAP SQL 时读取这些测试数据，再通过 ABAP Unit 断言结果。具体类名和 API 会随 SAP release、ABAP Platform、ABAP Cloud 支持范围变化，项目中必须查官方文档确认。

设计时要先确定测试边界。若目标是测试 service class 的业务规则，可以把 repository 用 OO Test Double 替换；若目标是测试 repository 的 SELECT/JOIN/聚合逻辑，就需要 SQL Test Double 或受控测试数据。不要在一个测试里同时覆盖太多层，否则失败原因会变得模糊。

测试数据要围绕业务问题设计。例如测试“重复 request id”时，需要准备一条已存在记录；测试“只查当前 company code”时，需要准备一条同 request id 但不同 company code 的干扰数据；测试“无配置”时，需要明确不插入配置记录；测试排序时，需要准备乱序数据，并断言结果顺序。

还要注意事务和副作用。SQL Test Double 更适合读取逻辑和局部写入隔离测试，不代表真实 SAP LUW、enqueue lock、commit/rollback、database trigger 或生产性能都被验证。涉及真实事务边界的内容仍要结合 Transaction boundary 事务边界、集成测试和系统级证据。

一个实用的 SQL test design record 可以这样写：

Query under test
  Method: zcl_upload_repository=>exists_request
  Inputs: request_id, company_code
  Expected contract: same request id in another company code is not a duplicate.

Tables / dependencies
  zupload_header

Fixture rows
  matching row
  same request id, different company code
  different request id, same company code

Assertions
  exact match -> exists
  cross-company match -> not exists
  missing request -> not exists

测试数据设计可以用下面的表：

数据类型目的示例
Matching row验证 happy pathrequest A / company 1000
Distractor row防止漏 WHERE 条件request A / company 2000
Boundary row验证边界日期/状态status CLOSED / old date
Duplicate row验证重复处理two items with same key
Empty fixture验证无数据分支no config row
Out-of-order row验证 ORDER BYsequence 2 before sequence 1

断言不要只写“有结果”。应该写清楚 result count、关键字段、排序、错误分类、是否走空结果分支。例如 status 查询应该断言只返回目标 company code 的 batch，且返回顺序由明确业务字段决定，而不是依赖数据库自然顺序。

20.5 日本项目现场说法

常见日语包括：SQLテストダブル、DB依存、テストデータ、検索条件、結合条件、空結果、重複データ、単体テスト、リポジトリクラス、回帰確認。

设计或 review 中可以这样说：

  • DB依存のある検索ロジックは、SQLテストダブルまたは制御されたテストデータで回帰確認します。
  • 実データに依存した単体テストは不安定になるため、必要最小限のテストデータを明示的に準備します。
  • SQLテストダブルは性能検証や結合テストを置き換えるものではありません。

代码 review 时可以这样问：

  • この SELECT の検索条件はテストデータで検証されていますか。
  • 空結果、重複結果、境界値のケースはありますか。
  • 本番DBの性能やロックまで単体テストで証明しようとしていませんか。
  • テストデータには、正常データだけでなく検索条件漏れを検出するための妨害データも含めます。
  • SQL テストダブルでは検索ロジックを確認し、性能・ロック・権限は別の証跡で確認します。
  • ORDER BY の前提がある場合は、順序を検証できるテストデータを用意します。

20.7 Example / 学习例子

假设 AC0567C 有一个 repository 方法：

exists_request( request_id, company_code ) -> abap_bool

这个方法用于幂等判断。如果真实数据库里刚好没有数据，测试可能通过；但这并不能证明 WHERE 条件正确。更好的测试数据设计是：

row 1: request_id = AC0567C-001, company_code = 1000
row 2: request_id = AC0567C-001, company_code = 2000
row 3: request_id = AC0567C-002, company_code = 1000

然后测试：

  • 输入 AC0567C-001 / 1000 应返回存在。
  • 输入 AC0567C-001 / 3000 应返回不存在。
  • 输入 AC0567C-002 / 1000 应返回存在。

如果代码只按 request id 查询，第二个场景会误判。SQL Test Double 让这种错误在单元测试里暴露出来，而不是等到结合测试或生产数据碰巧触发。

再看 JOIN 场景。上传明细表和状态表通过 batch id 关联，如果 JOIN 键写错，真实系统可能因为数据巧合暂时看不出来。测试中准备两组相似 batch id 和状态，就能验证 JOIN 是否真的按正确键关联。

再给一个 AC0567C status query 的 fixture：

Header rows:
  BATCH-001 / company 1000 / status OPEN
  BATCH-001 / company 2000 / status CLOSED   "distractor
  BATCH-002 / company 1000 / status ERROR

Item rows:
  BATCH-001 / company 1000 / item 10
  BATCH-001 / company 2000 / item 10          "wrong company distractor
  BATCH-002 / company 1000 / item 20

Expected:
  query_status(company=1000, batch=BATCH-001)
  returns only company 1000 header and item.

如果 JOIN 只按 batch_id，就会把 company 2000 的 item 混进来。这样的测试数据比“只有一条干净数据”更能保护生产逻辑。

对于 ORDER BY，也要故意准备乱序数据。如果业务要求按 line number 输出错误，就准备 line 20、line 10、line 30，然后断言返回顺序是 10、20、30。没有 ORDER BY 的代码可能在某个数据库上看似稳定，在另一个环境或优化后顺序就变了。

20.8 Counterexample / Failure mode

常见失败模式有以下十类。

第一，测试依赖真实业务数据。开发机、测试机、客户机数据不同，测试结果不稳定，也无法说明测试覆盖了哪个业务规则。

第二，测试数据太大。复制一堆生产样本进测试，会让测试难读、难维护。单元测试数据应最小化、意图明确。

第三，只测有数据的情况。空结果、重复结果、边界日期、不同 client/company code 的干扰数据，往往更容易暴露 SQL 条件问题。

第四，把 SQL Test Double 当性能测试。它不能证明真实 HANA 执行计划、索引、buffer、SQL Monitor 结果或生产性能。

第五，忽略事务和锁。SQL 读取逻辑测试通过，不代表真实并发、enqueue lock、commit/rollback 和 RAP save sequence 都正确。

第六，只准备 happy path 数据。测试数据太干净，漏 WHERE、错 JOIN、缺 ORDER BY 都可能看不出来。

第七，把 fixture 写得像生产数据转储。维护者看不懂每一行数据的目的，测试失败时也不知道业务含义。

第八，在同一个测试里塞太多表和太多断言。失败后无法判断是查询条件、JOIN、排序还是业务判断出了问题。

第九，忘记 client / tenant / organization 维度。SAP 数据经常有 client、company code、plant、sales org 等边界；测试若不准备跨边界干扰数据，就不能证明条件完整。

第十，SQL double 假设和真实 CDS/DCL 不一致。若生产路径通过 CDS DCL 或 RAP projection 消费，而测试只覆盖底层表 SELECT，仍需额外测试 CDS/service 路径。
