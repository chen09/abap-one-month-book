Chapter 02 - Core ABAP Programming

22. FIELD-SYMBOL vs DATA reference

  • Chapter 02
  • Section 22
  • self-contained study note

22.1 一句话结论

FIELD-SYMBOL 和 DATA reference 都能让 ABAP 代码“不是复制值，而是指向已有数据”，但它们不是同一个东西：field symbol 是运行时临时别名，常用于局部循环、动态字段和直接修改当前 table row；data reference 是真正的数据引用变量，可以保存、传递、返回，也可以通过 CREATE DATA 指向匿名数据对象。初学者要学会用一句话区分：field symbol 更像“当前这段代码里的别名”，data reference 更像“一个变量里装着指向数据对象的引用”。

22.2 到底学什么

上一节 Field symbol 字段符号 已经说明，field symbol 本身不保存业务值，而是在 ASSIGN 或 LOOP AT ... ASSIGNING 成功后指向某个数据对象或 internal table 行。

data reference 则是另一种机制：

DATA lr_amount TYPE REF TO i.

GET REFERENCE OF lv_amount INTO lr_amount.

IF lr_amount IS BOUND.
  lr_amount->* = lr_amount->* + 1.
ENDIF.

这里 lr_amount 是一个变量，它的内容是“指向 lv_amount 的引用”。通过 lr_amount->* dereference 后，才访问到被引用的数据对象。

本节要学的核心问题：

  • field symbol 和 data reference 都能避免复制，但语义差异是什么？
  • <fs> IS ASSIGNED 和 lr_data IS BOUND 分别检查什么？
  • ASSIGN ... TO <fs>、GET REFERENCE OF ... INTO lr_ref、REF #( ... )、CREATE DATA lr_ref 分别解决什么问题？
  • LOOP AT ... ASSIGNING 和 LOOP AT ... REFERENCE INTO 怎么选？
  • 为什么 data reference 更适合跨方法传递、动态对象和引用型 API，而 field symbol 更适合局部处理？
  • 为什么两者都可能直接修改原始数据，review 时不能只看语法短不短？

22.3 为什么重要

在 ABAP 项目里，这一节解决的是一个非常实际的阅读障碍：看到 <ls_item>、lr_item->*、REFERENCE INTO DATA(lr_item) 时，初学者必须马上判断“这是复制、别名，还是引用变量”。

如果分不清，常见后果是：

  • 以为 LOOP AT ... ASSIGNING 只是更短的 INTO，结果不小心直接改了 internal table。
  • 以为 data reference 和 field symbol 一样只能在局部用，结果错过了需要传递引用的设计。
  • 以为 reference variable 一定安全，忘了检查 IS BOUND 或误用已经失效的 stack/table-row reference。
  • 为了“性能”把简单代码改成引用式写法，但没有改善真正瓶颈，反而降低可读性。

对 Java/Python 背景学习者，可以这样类比：field symbol 像一段代码里临时绑定到某个对象/字段的别名；data reference 更接近一个可以赋值、传参、放到结构或 internal table 里的 reference value。但 ABAP 有自己的 ASSIGN、IS ASSIGNED、IS BOUND、->*、stack/heap reference 规则，不能完全照搬 Java/Python。

22.4 Key mechanism / 核心机制

22.4.1 field symbol 是 alias，不是 reference variable
FIELD-SYMBOLS <amount> TYPE i.

ASSIGN lv_amount TO <amount>.

IF <amount> IS ASSIGNED.
  <amount> = <amount> + 1.
ENDIF.

<amount> 指向 lv_amount，但 <amount> 不是一个装着 reference value 的普通变量。它不能像 lr_amount 那样赋值给另一个 reference variable，也不能作为 structure/internal table 的组件保存或跨调用栈返回。它是当前程序位置上的运行时别名。

22.4.2 data reference 是变量，里面的值是 reference
DATA lr_amount TYPE REF TO i.

GET REFERENCE OF lv_amount INTO lr_amount.

IF lr_amount IS BOUND.
  lr_amount->* = lr_amount->* + 1.
ENDIF.

lr_amount 是一个 data reference variable。它可以是 initial，也可以 bound 到某个数据对象。访问被引用对象时要 dereference，最基本形式是 lr_amount->*。

22.4.3 CREATE DATA 创建匿名数据对象

field symbol 只能指向已经存在或已被创建的数据对象。data reference 可以配合 CREATE DATA 在 heap 上创建匿名数据对象：

DATA lr_item TYPE REF TO ty_item.

CREATE DATA lr_item.

lr_item->*-status = 'N'.
" If the referenced type is known and structured, lr_item->status is also common.

这里新建的数据对象没有普通变量名，只有 lr_item 指向它。动态框架、通用容器、运行时类型处理经常需要这种能力。

22.4.4 GET REFERENCE / REF 取得已有对象的引用
DATA lr_item TYPE REF TO ty_item.

GET REFERENCE OF ls_item INTO lr_item.

现代写法也可以用 REF #( ... )：

DATA(lr_item) = REF #( ls_item ).

这类 reference 指向已有数据对象。review 时要问：这个 reference 会不会被保存到当前对象或返回给调用方？被引用数据对象的生命周期是否足够长？

22.4.5 LOOP AT ASSIGNING vs REFERENCE INTO

field symbol 版：

LOOP AT lt_items ASSIGNING FIELD-SYMBOL(<ls_item>).
  <ls_item>-status = 'C'.
ENDLOOP.

reference 版：

LOOP AT lt_items REFERENCE INTO DATA(lr_item).
  lr_item->status = 'C'.
ENDLOOP.

两者都不是复制当前行。ASSIGNING 直接给你一个 field symbol；REFERENCE INTO 给你一个 data reference。局部直接更新一行时，ASSIGNING 往往更直接；如果后续 API 需要 reference variable，或你要把引用传给方法，REFERENCE INTO 才有意义。

22.4.6 ASSIGNED vs BOUND

field symbol 用：

IF <ls_item> IS ASSIGNED.
  ...
ENDIF.

data reference 用：

IF lr_item IS BOUND.
  ...
ENDIF.

不要混用概念。IS ASSIGNED 检查 field symbol 当前是否指向某个内存区域；IS BOUND 检查 reference variable 是否非空、已经绑定。它不保证被引用对象当前仍然有效：某些失效的 stack reference 或 table-row reference 仍可能 IS BOUND 为真，但 dereference 时出错。

22.4.7 生命周期和保存范围

field symbol 通常是局部处理工具。data reference 可以被保存、传递、返回，但这并不表示所有 reference 都适合长期保存。GET REFERENCE OF 局部变量取得的是 stack 相关引用，过程结束后就有生命周期风险；CREATE DATA 创建的是 heap 上的数据对象，只要还有 reference 持有它就可以继续存在。指向 internal table 当前行的 reference，如果 table 后续被删除、重排、清空，也可能变成失效设计。业务上需要长期定位一行时，很多时候更稳的是保存 business key，再按 key 重新读取。

22.5 日本项目现场说法

  • 常见日语：フィールドシンボル、データ参照、参照変数、別名、参照渡し、逆参照、IS BOUND、IS ASSIGNED。
  • 说明句：フィールドシンボルは一時的な別名として元データを直接参照しますが、データ参照は参照値を保持する変数として受け渡しできます。
  • Review 句：この参照をメソッド外へ渡す場合、参照先データのライフタイムと IS BOUND チェックを必ず確認してください。
  • 行参照の注意：内部テーブルの行参照を長期保持すると、後続の DELETE / SORT / CLEAR で無効になる可能性があるため、業務キーを保存して都度読み直す方式を優先します。
  • 直接更新说明：ASSIGNING も REFERENCE INTO もコピーではないため、変更が元の内部テーブル行に反映される点をレビューで確認します。
  • 使用场景：内部表更新、动态数据对象、通用框架、接口参数设计、性能 review、缺陷调查中确认“是否保存了不该保存的引用”。

22.7 Example / 学习例子

22.7.1 例 1：同一个变量，用 field symbol 修改
DATA lv_amount TYPE i VALUE 10.

ASSIGN lv_amount TO FIELD-SYMBOL(<amount>).

IF <amount> IS ASSIGNED.
  <amount> += 5.
ENDIF.

" lv_amount is now 15

<amount> 是 lv_amount 的别名。它不是副本。

22.7.2 例 2：同一个变量，用 data reference 修改
DATA lv_amount TYPE i VALUE 10.
DATA lr_amount TYPE REF TO i.

GET REFERENCE OF lv_amount INTO lr_amount.

IF lr_amount IS BOUND.
  lr_amount->* += 5.
ENDIF.

" lv_amount is now 15

lr_amount 是 reference variable，必须通过 ->* dereference 才能访问 lv_amount。

22.7.3 例 3：更新 internal table 当前行

局部更新时，field symbol 更直接：

LOOP AT lt_items ASSIGNING FIELD-SYMBOL(<ls_item>).
  <ls_item>-status = 'C'.
ENDLOOP.

如果 API 或后续设计需要 reference variable，可以使用：

LOOP AT lt_items REFERENCE INTO DATA(lr_item).
  lr_item->status = 'C'.
ENDLOOP.

后者不要只是为了“看起来现代”而使用。它多了一层 reference 语义，适合需要 reference variable 的场景。

22.7.4 例 4：动态创建数据
DATA lr_text TYPE REF TO string.

CREATE DATA lr_text.

lr_text->* = |Created at runtime|.

这是 data reference 的典型能力：创建 heap 上、没有普通变量名的匿名数据对象。field symbol 本身不能单独创建数据，它只能指向已有或已创建的数据。

22.7.5 例 5：动态引用再转 field symbol
DATA lr_data TYPE REF TO data.

CREATE DATA lr_data TYPE string.

ASSIGN lr_data->* TO FIELD-SYMBOL(<value>).

IF sy-subrc = 0.
  <value> = |dynamic value|.
ENDIF.

ASSIGN dref->* TO <fs> 和普通 ASSIGN 一样会设置 sy-subrc。很多动态编程里，两者会一起出现：data reference 负责“拥有/保存引用”，field symbol 负责“用一个可操作的别名访问实际数据”。

22.8 Counterexample / Failure mode

22.8.1 误区 1：把 field symbol 当成可保存 reference

field symbol 适合局部处理，不能作为长期对象关系保存到结构、internal table 或对象属性里，也不能跨调用栈返回。需要跨方法传递、保存到对象属性、放到 internal table 中时，应该考虑 data reference、object reference 或直接保存业务 key。

22.8.2 误区 2：忘记检查不同状态
IF <fs> IS BOUND.      " wrong concept
ENDIF.

IF lr_data IS ASSIGNED. " wrong concept
ENDIF.

field symbol 检查 IS ASSIGNED，data reference 检查 IS BOUND。概念混乱通常说明作者没有真正区分两种机制。

22.8.3 误区 3：保存 table row reference 太久

REFERENCE INTO 可以取得 internal table 行的 reference，但如果 table 后续会 delete、sort、clear 或重新构造，长期保存 row reference 会让代码变脆。跨步骤、跨方法、跨 LUW 的业务定位，通常保存 key 更稳。

22.8.4 误区 4：认为 data reference 一定比 field symbol 更现代

data reference 是更适合传递/保存的引用变量，但不是所有局部循环都应该改成 REFERENCE INTO。如果只是当前 loop 内直接更新行，ASSIGNING FIELD-SYMBOL(<line>) 往往更清楚。

22.8.5 误区 5：为了性能隐藏副作用

无论 field symbol 还是 data reference，只要指向原数据，修改后都可能影响原对象。性能优化不能替代可读性和副作用说明。review 时要看：这次修改是业务上有意的，还是误把副本当原件？

22.9 思考问题

  • 这段代码使用的是 field symbol、data reference，还是普通 work area？
  • <fs> IS ASSIGNED 和 lr_data IS BOUND 分别检查什么？
  • 这段代码修改的是副本，还是原数据对象？
  • 这个 reference 是否会被保存到方法外、对象属性或 internal table 里？
  • 被引用的数据对象生命周期是否足够长？
  • 如果只是局部循环更新，ASSIGNING 是否比 REFERENCE INTO 更清楚？
  • 如果需要动态创建数据，是否必须使用 data reference / CREATE DATA？
  • 在日本项目 review 中，应该提醒 ライフタイム、直接更新，还是 IS BOUND 检查？
