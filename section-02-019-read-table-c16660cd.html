Chapter 02 - Core ABAP Programming

19. READ TABLE 读取内部表

  • Chapter 02
  • Section 19
  • self-contained study note

19.1 一句话结论

READ TABLE 是 ABAP 读取 internal table 内部表单行的经典语句：可以按 index、primary key、secondary key 或 free key 读取，并通过 INTO、ASSIGNING、REFERENCE INTO、TRANSPORTING NO FIELDS 决定读取结果；真正要学的是“用哪个 key 读、是否只读一行、如何处理 sy-subrc / sy-tabix、何时改用 table expression 或 LOOP AT”。

19.2 到底学什么

READ TABLE 读取的是 ABAP 程序内存中的 internal table，不是数据库表。数据库读取用 ABAP SQL SELECT；内存表读取才用 READ TABLE、table expression 或 LOOP AT。

最小例子：

READ TABLE lt_material INTO DATA(ls_material)
  WITH TABLE KEY matnr = lv_matnr.

IF sy-subrc = 0.
  " found
ENDIF.

这段代码包含四个需要理解的问题：

lt_material 是什么 table category？
  standard / sorted / hashed？

使用了哪个 key？
  primary key / secondary key / free key？

读取结果放到哪里？
  INTO copy / ASSIGNING field symbol / REFERENCE INTO reference / TRANSPORTING NO FIELDS？

未找到怎么处理？
  sy-subrc / exception / OPTIONAL / line_exists？

初学者不要只背 READ TABLE ... WITH KEY ...。这句在小 standard table 上可能没问题，但在大表、循环、secondary key、hashed table、sorted table 中，差异很大。READ TABLE 是把前面几节 Standard table 标准表、Sorted table 排序表、Hashed table 哈希表、Secondary table key 二级键 串起来的实际操作语句。

19.3 为什么重要

项目里大量缺陷都藏在 READ TABLE 周围：

  • 没检查 sy-subrc，未找到时继续使用旧 work area。
  • 在循环里对 standard table 反复 READ TABLE ... WITH KEY，形成性能瓶颈。
  • 有 secondary key 却没有显式 WITH TABLE KEY keyname COMPONENTS ...。
  • table expression 未找到时抛异常，却以为会像 READ TABLE 一样设置 sy-subrc。
  • non-unique key 只读到第一行，却误以为读到了所有匹配行。
  • hashed table 读取后依赖 sy-tabix，把 0 当成业务位置。
  • BINARY SEARCH 用在没有正确排序的 standard table 上，结果不可靠。

对 Java/Python 背景学习者来说，READ TABLE 类似“从 list/map 中取一个元素”，但 ABAP 的关键在于 table category 和 key。你不能只看语句形式，必须看 lt_... 的声明。

19.4 Key mechanism / 核心机制

19.4.1 READ TABLE ... INDEX

按 index 读取适合 index table：

READ TABLE lt_items INTO DATA(ls_item) INDEX 1.

standard table 和 sorted table 是 index table；hashed table 不应当被当作 index/position container 使用。按 index 读的语义很直接：我要第 N 行。但第 N 行通常不是业务 key，只是当前 internal table 状态下的位置。只要 SORT、DELETE、INSERT，行号就可能变化。

适用场景：

  • 读取第一行、最后一行、当前 sy-tabix 附近的行。
  • 已经明确知道 table 的顺序就是业务处理顺序。
  • 对 ALV / list / sequential buffer 做位置处理。

不适用场景：

  • 按业务 ID 找数据。
  • hashed table。
  • 表顺序不稳定却依赖 index。
19.4.2 READ TABLE ... WITH TABLE KEY

按 table key 读取是项目里更重要的模式：

READ TABLE lt_items INTO DATA(ls_item)
  WITH TABLE KEY order_id = lv_order_id
                 item_no  = lv_item_no.

如果有 named secondary key，建议显式写 key name：

READ TABLE lt_items INTO DATA(ls_item)
  WITH TABLE KEY by_guid COMPONENTS item_guid = lv_guid.

这样 reviewer 能一眼看出你想使用哪个 access path。特别是 table 声明中有 secondary keys 时，不要写得含糊。

19.4.3 READ TABLE ... WITH KEY

WITH KEY 是 free key 风格，常见于旧代码或简单读取：

READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY order_id = lv_order_id.

它并不自动等于“最优 key”。ABAP 会根据语法和可用 key 判断，但初学者在性能敏感代码中更应该显式使用 WITH TABLE KEY keyname COMPONENTS ...。如果系统或 syntax check 提示有合适 secondary key 没用，应优先修正代码而不是忽略 warning。

19.4.4 result：INTO、ASSIGNING、REFERENCE INTO

INTO 把行复制到 work area：

READ TABLE lt_items INTO DATA(ls_item)
  WITH TABLE KEY item_id = lv_item_id.

适合只读值，或者后续修改不希望影响原 internal table。

ASSIGNING 把 field symbol 指向表内实际行：

READ TABLE lt_items ASSIGNING FIELD-SYMBOL(<ls_item>)
  WITH TABLE KEY item_id = lv_item_id.

IF sy-subrc = 0.
  <ls_item>-status = 'X'.
ENDIF.

适合要直接修改表内行，避免 copy。但要注意 sorted/hashed table 的 key 字段不能随便原地修改。

REFERENCE INTO 取得行引用：

READ TABLE lt_items REFERENCE INTO DATA(lr_item)
  WITH TABLE KEY item_id = lv_item_id.

适合需要把行引用传给其他逻辑，但要小心生命周期和可变性。

19.4.5 TRANSPORTING NO FIELDS

如果只想判断存在，不需要复制行：

READ TABLE lt_items TRANSPORTING NO FIELDS
  WITH TABLE KEY item_id = lv_item_id.

IF sy-subrc = 0.
  " exists
ENDIF.

这比读入 work area 更表达意图。现代 ABAP 里也常用：

IF line_exists( lt_items[ item_id = lv_item_id ] ).
  " exists
ENDIF.

line_exists 可以看作 existence check 的现代写法；READ TABLE ... TRANSPORTING NO FIELDS 在需要 sy-tabix 或兼容旧代码时仍然常见。

19.4.6 sy-subrc

READ TABLE 会设置 sy-subrc。常用值：

0  找到行
2  找到行，但 COMPARING 的比较结果不同
4  未找到
8  未找到，并且在某些 binary search 场景中表示搜索到表尾之后

日常项目最常用的是 0 和 4。读取后必须立即处理 sy-subrc，不要在中间插入其他会改变系统字段的语句。

错误写法：

READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY item_id = lv_item_id.

" 中间做了别的语句
IF sy-subrc = 0.
  ...
ENDIF.

更清楚：

READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY item_id = lv_item_id.

IF sy-subrc <> 0.
  RETURN.
ENDIF.
19.4.7 sy-tabix

sy-tabix 的含义取决于读取方式：

  • 对 standard/sorted primary index 或 sorted secondary key，找到行时通常是相应 table index 中的行号。
  • 对 hash key，sy-tabix 可能是 0，不应作为业务位置。
  • binary search 未找到时，sy-tabix 可能表示插入位置或表尾之后的位置。
  • 普通未找到时，sy-tabix 可能未定义。

因此：sy-tabix 可以用于紧接着的位置处理，但不要把它当成业务 key，也不要跨很多语句长期保存当作稳定标识。

19.4.8 BINARY SEARCH

旧代码常见：

SORT lt_items BY item_id.

READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY item_id = lv_item_id
  BINARY SEARCH.

BINARY SEARCH 的前提是 standard table 已经按相同 key 正确排序。如果排序字段、升降序、读取 key 不一致，结果就不可靠。现代 ABAP 更推荐：

需要按 key 有序读取
  -> sorted table

standard table 还要保留顺序，但需要额外 key 读取
  -> secondary sorted key

完整唯一 key 精确读取
  -> hashed table / secondary hashed key

所以，不是说 BINARY SEARCH 永远不能用，而是初学者不要把它当成性能魔法。先设计 table category 和 key。

19.4.9 non-unique key 只读第一行

如果 key 不唯一：

READ TABLE lt_items INTO DATA(ls_item)
  WITH KEY order_id = lv_order_id.

READ TABLE 读取的是第一条合适的行，不是所有行。要处理全部匹配项，应使用：

LOOP AT lt_items INTO DATA(ls_item)
  WHERE order_id = lv_order_id.
ENDLOOP.

如果有 sorted key，可以配合 USING KEY：

LOOP AT lt_items INTO DATA(ls_item)
  USING KEY by_order
  WHERE order_id = lv_order_id.
ENDLOOP.
19.4.10 table expression 与 READ TABLE

现代 ABAP 常用 table expression：

DATA(ls_item) = lt_items[ item_id = lv_item_id ].

它更短，但语义不同：table expression 不像 READ TABLE 那样设置 sy-subrc；找不到时通常会抛出 CX_SY_ITAB_LINE_NOT_FOUND，除非使用 OPTIONAL、DEFAULT、line_exists 或其他安全写法。

因此：

想要经典 sy-subrc 分支
  -> READ TABLE

想要表达式风格，且已处理 not-found
  -> table expression

只判断是否存在
  -> line_exists(...) 或 READ TABLE ... TRANSPORTING NO FIELDS

要读取多行
  -> LOOP AT ... WHERE

19.5 日本项目现场说法

常见日语：

  • READ TABLE
  • 内部テーブル読込

  • 存在チェック

  • sy-subrc チェック

  • 線形検索
  • 二分探索
  • ハッシュキー
  • テーブル式
  • 未検出時の例外

自然说明句：

  • この READ TABLE は二次キーを明示していないため、想定したアクセスパスを使用しているか確認が必要です。
  • 存在チェックだけであれば、TRANSPORTING NO FIELDS または line_exists を使用します。
  • 非一意キーの場合、READ TABLE では最初の1件のみ取得されるため、複数件処理なら LOOP AT ... WHERE を使用します。
  • ハッシュキーで読み取る場合、sy-tabix を業務上の位置として扱わないでください。
  • BINARY SEARCH を使う場合は、事前のソート順と検索キーが一致していることを確認してください。

review 里常见问法：

  • sy-subrc は直後にチェックされていますか。
  • この WITH KEY は標準テーブルの線形検索になっていませんか。
  • 二次キーを定義しているのに、WITH TABLE KEY で指定していない理由はありますか。
  • 複数件取得が必要なのに READ TABLE で1件だけ読んでいませんか。

19.7 Example / 学习例子

19.7.1 例 1：按 primary key 读一行
READ TABLE lt_material INTO DATA(ls_material)
  WITH TABLE KEY matnr = lv_matnr.

IF sy-subrc <> 0.
  " not found
  RETURN.
ENDIF.

这是经典读取模式。关键是：读完立刻判断 sy-subrc。

19.7.2 例 2：只检查存在
READ TABLE lt_material TRANSPORTING NO FIELDS
  WITH TABLE KEY matnr = lv_matnr.

IF sy-subrc = 0.
  " material exists in the internal table
ENDIF.

如果不需要行内容，不要无意义复制到 work area。

19.7.3 例 3：直接修改找到的行
READ TABLE lt_items ASSIGNING FIELD-SYMBOL(<ls_item>)
  WITH TABLE KEY item_id = lv_item_id.

IF sy-subrc = 0.
  <ls_item>-status = 'X'.
ENDIF.

ASSIGNING 会指向表内实际行，适合修改非 key 字段。对 sorted/hashed table 的 key 字段要特别小心。

19.7.4 例 4：显式使用 secondary key
READ TABLE lt_items INTO DATA(ls_item)
  WITH TABLE KEY by_guid COMPONENTS item_guid = lv_guid.

如果 table type 声明了 by_guid，读代码时就能看出作者明确使用这个 access path。

19.7.5 例 5：non-unique key 要用 loop 读多行
LOOP AT lt_items INTO DATA(ls_item)
  USING KEY by_order
  WHERE order_id = lv_order_id.
  " process all matching items
ENDLOOP.

READ TABLE 只读一行；一对多明细处理通常要 LOOP AT ... WHERE。

19.7.6 例 6：table expression 的安全写法
IF line_exists( lt_material[ matnr = lv_matnr ] ).
  DATA(ls_material) = lt_material[ matnr = lv_matnr ].
ENDIF.

或者：

DATA(ls_material) = VALUE ty_material(
  lt_material[ matnr = lv_matnr ] OPTIONAL ).

table expression 简洁，但不要误以为它会设置 sy-subrc。

19.8 Counterexample / Failure mode

19.8.1 误区 1：未找到后继续使用旧值
READ TABLE lt_items INTO ls_item WITH KEY item_id = lv_item_id.
" 没检查 sy-subrc
lv_status = ls_item-status.

如果未找到，ls_item 可能保留旧值或初始值，导致非常隐蔽的 bug。读取后要立即处理 not found。

19.8.2 误区 2：在大循环中线性搜索 standard table
LOOP AT lt_a INTO DATA(ls_a).
  READ TABLE lt_b INTO DATA(ls_b)
    WITH KEY id = ls_a-id.
ENDLOOP.

如果 lt_b 是大 standard table，这可能很慢。应考虑 sorted/hashed table 或 secondary key。

19.8.3 误区 3：需要多行却用 READ TABLE

READ TABLE 是单行读取。non-unique key 下它只拿第一条合适行。要所有匹配行，用 LOOP AT ... WHERE 或过滤/分组逻辑。

19.8.4 误区 4：滥用 BINARY SEARCH

BINARY SEARCH 必须建立在正确排序上。只要排序字段和读取 key 不一致，结果就可能错。现代代码优先考虑 sorted table 或 secondary sorted key。

19.8.5 误区 5：table expression 后检查 sy-subrc
DATA(ls_item) = lt_items[ item_id = lv_item_id ].
IF sy-subrc = 0.
  ...
ENDIF.

这是错误心智模型。table expression 不按 READ TABLE 的方式设置 sy-subrc；要处理 not found，请用 line_exists、OPTIONAL、DEFAULT 或异常处理。

19.9 思考问题

  • 当前读取是按 index，还是按 business key？
  • 这张 internal table 是 standard、sorted、hashed，还是带 secondary key？
  • 这个 READ TABLE 是否在循环中大量执行？
  • 是否显式使用了正确的 key name？
  • 读取后是否立即检查 sy-subrc？
  • 如果 key 不唯一，是否只需要第一行？
  • 是否只是 existence check？能否使用 TRANSPORTING NO FIELDS 或 line_exists？
  • sy-tabix 是否真的可靠？hash key 读取时它有什么意义？
  • 这个场景是否更适合 table expression？
  • 用日语 review 时，如何说明 二次キー指定 和 sy-subrc チェック？
