Chapter 02 - Core ABAP Programming

46. Clean ABAP naming 命名规则

  • Chapter 02
  • Section 46
  • self-contained study note

46.1 一句话结论

Clean ABAP naming 命名规则，是在 ABAP 代码内部使用能表达意图、可搜索、可沟通、可维护的名字，让读代码的人不必靠猜。它不是“把名字写短一点”，也不是“看到旧代码里的 lv_、lt_ 就马上全部改掉”。本节要学的是：新写代码时尽量让名字说明业务或技术含义；维护遗留对象时尊重项目既有规范，避免在同一个 development object 里混用两套风格。

这里要先分清两类命名。第一类是 repository object 命名，例如 ZCL_SD_ORDER_VALIDATOR、ZIF_SD_ORDER_CHECK、Z... package、CDS view entity、message class 等，它关系到 Repository object 仓库对象、ABAP package 包与子包、Transport request 传输请求 和项目命名規約，已经在 Japanese SAP naming convention 里展开。客户开发对象使用 Z / Y 或注册 namespace /ABC/... 这类规则，往往是系统约束和项目约束，不是单纯风格偏好。第二类是代码内部命名，例如变量、method、parameter、local class、helper method、test method 的名字。本节主要讲第二类，同时提醒你不要把 Clean ABAP 对局部代码的建议误用到客户 namespace 或项目对象命名规则上。

46.2 到底学什么

Clean ABAP naming 的核心问题不是“哪个名字更漂亮”，而是“读者能不能从名字直接看出它负责什么”。在 SAP 项目里，读者可能是开发者、reviewer、测试者、运维、离岸团队、客户侧 IT 或未来接手的人。一个名字如果只能靠上下文猜，短期看只是风格问题，长期看会影响 code review、debug、障害調査、where-used 分析和交接。

本节需要掌握八个判断点。

  • Descriptive name：名字要表达内容和含义，例如 sales_orders、blocked_order_ids、read_user_preferences，不要只写 data、list、flag、get_data。
  • Problem domain / solution domain：业务层优先用业务词，例如 sales_order、business_partner、ledger；技术层可以用技术词，例如 queue、mapper、factory，但前提是它真的承担这个职责。
  • snake_case：ABAP program-internal identifier 不区分大小写；underscore 能保留词边界，即使 formatter 改变大小写，sales_order 也比 salesOrder 更稳定。这个建议主要针对局部变量、parameter、method 等代码内部名字，不覆盖 ZCL_...、ZIF_... 这类 repository object 命名。
  • Abbreviation：能写全就写全；必须缩写时，缩写要统一，例如同一项目里不要同时用 bp、bupa、partner 指同一个概念。
  • Noun / verb：class、interface、object 通常用名词或名词短语；method 和 function 通常用动词或动词短语。boolean method 用 is_、has_ 这类开头会更自然，但要知道旧 SAP 前缀规则里 is_ 也可能表示 importing structure parameter，含义要靠上下文区分。
  • Noise word：data、info、object、manager、helper 这些词如果没有增加含义，就只是噪音。
  • One word per concept：read、get、fetch、select 不要随意混用；如果项目里规定它们有不同含义，就写进规范并坚持使用。
  • Legacy boundary：Clean ABAP 倾向避免 Hungarian notation 和类型前缀，例如 lv_、lt_、rv_；但很多 SAP 项目仍有既有前缀规则。维护旧对象时，统一性比“局部改成新风格”更重要。

46.3 为什么重要

命名是最便宜也最容易被低估的设计。ABAP 项目里很多对象不是一次性脚本，而是会被长期 transport、修正、扩展、审计和交接的 repository object。一个 method 叫 process_data，reviewer 需要继续读几十行才知道它是检查销售订单状态、转换单位、写应用日志，还是调用 BAPI。一个 internal table 叫 lt_data，debug 时 watch 里只会出现“数据”，不会告诉你它是 open items、blocked orders 还是 customizing entries。

对初学者来说，命名最大的价值是降低认知负担。刚开始学 ABAP 时，你已经要同时理解 Global class 全局类、Interface 接口定义、Inline declaration 内联声明、Method signature 方法签名、Open SQL、DDIC、transport 和调试。如果变量名和 method 名还不表达意图，就会把简单代码变成猜谜。好的名字等于把一部分注释写进代码本身。

对项目交付来说，命名影响可搜索性和影响分析。read_open_sales_orders 比 get_data 更容易被 where-used、全文搜索和障害调查人员找到。has_missing_authorization 比 lv_flag 更容易在 review 中讨论。zcl_sd_order_validator 这种对象名不一定完美，但至少能让人从 namespace、模块和用途上初步定位；而代码内部如果全是 lt_data、ls_wa、lv_result，对象名再规范也救不了实现的可读性。

对 Java/Python 背景学习者来说，ABAP 命名还有一个特殊点：大量真实项目长期使用 Hungarian notation 或项目 prefix。Clean ABAP 的明确立场是，类型前缀会增加噪音，类型变化后还可能变成误导；现代 ABAP 的 inline declaration 和 IDE 支持也降低了靠前缀读类型的必要性。项目现实则是，许多客户代码库和開発標準仍要求 lv_、lt_、iv_、rv_。不要把这理解成“谁对谁错”，而要学会区分三层：SAP/styleguides 的 Clean ABAP 建议、客户项目的命名規約、当前对象的历史风格。新建练习对象时可以写得更 clean；改旧对象时不要制造同一文件内的风格断层。

46.4 Key mechanism / 核心机制

46.4.1 4.1 Descriptive names：让名字说出意图

一个好名字要回答“这是什么”或“它做什么”。变量名要能说明内容，method 名要能说明动作。

" Hard to review
DATA lt_data TYPE STANDARD TABLE OF zsales_order.
DATA lv_flag TYPE abap_bool.
METHOD get_data.

" Easier to review
DATA sales_orders TYPE STANDARD TABLE OF zsales_order.
DATA has_blocked_items TYPE abap_bool.
METHOD read_open_sales_orders.

这里的差异不在长度，而在信息密度。sales_orders 告诉读者这张 internal table 装的是销售订单；has_blocked_items 告诉读者它是一个正向 boolean；read_open_sales_orders 让人期待 method 内部主要负责读取 open 状态订单，而不是混入校验、保存和 UI 输出。

46.4.2 4.2 Domain terms：业务词和技术词不要乱混

业务层的名字应该靠近业务语言。订单、客户、供应商、会计凭证、库存、价格、税、交货、请购、审批，这些是 problem domain terms。技术层的名字可以使用 solution domain terms，例如 cache、queue、factory、mapper、adapter、repository。但不要为了显得“架构化”随便给 class 加 factory、manager 或 service。如果一个 class 并不创建对象，就不要叫 factory；如果只是校验订单状态，就不要叫 manager。

在 SAP 项目里，业务词还要和 glossary、数据模型、CDS view、接口字段、设计书保持一致。比如项目已经统一使用 business_partner，就不要在代码里一会儿写 bp，一会儿写 bupa，一会儿写 partner。如果这些词确实有差异，要用名字表达差异，而不是让读者猜。

46.4.3 4.3 snake_case：用 underscore 保留词边界

ABAP program-internal object name 不区分大小写。也就是说，不能靠 salesOrder、SalesOrder、SALESORDER 来表达不同含义。更精确地说，snake_case 的价值不是“看起来小写”，而是 underscore 提供了不会被大小写格式化破坏的词边界。即使 ABAP Pretty Printer 或 ADT formatter 把大小写改掉，sales_order 仍然能读出两个词，而 salesOrder 可能变成 salesorder 或 SALESORDER。

Clean ABAP 因此推荐在代码内部稳定使用 snake_case：

DATA max_wait_time_in_seconds TYPE i.
DATA user_preferences TYPE ztt_user_preference.

repository object 名称在 ADT、SE80、transport 和项目规范里常会以大写出现，例如 ZCL_SD_ORDER_VALIDATOR。这不和本节矛盾：对象名遵守客户 namespace、系统命名限制和项目命名規約；代码内部的变量、method、attribute、parameter 才是 Clean ABAP naming 更直接发挥作用的地方。初学者不要把 snake_case 建议套到 ZCL_... / ZIF_... 这类全局对象名上。

46.4.4 4.4 Abbreviation：缩写必须可读、统一、可搜索

ABAP 有些名字长度限制，SAP 领域也有大量固定缩写，例如 CDS、RAP、API、UUID、LUW、BAPI、BP。缩写不是禁止项，但不能随手发明。一个典型坏例子是 cust，它可能表示 customer、customizing 或 custom。如果项目里确实需要缩写，就把它写进 glossary 或命名規約，并在所有对象里使用同一个缩写。

Clean ABAP 还提醒：遇到名字长度限制时，不要退回到难读的 flatcase、全大写缩写串或临时暗号。正确做法是保留最能表达意图的 domain term，必要时拆小 method、抽 helper，或让上下文承担一部分说明，而不是把名字压缩到只有作者自己看得懂。

初学者可以用一个简单检查：这个缩写是否能被团队里另一个模块的人看懂？是否能被搜索？是否和 SAP 标准术语冲突？如果答案不确定，先写全。

46.4.5 4.5 Noun / verb：对象是名词，动作是动词

Clean ABAP 建议 class、interface、object 用名词或名词短语，method 和 function 用动词或动词短语。

CLASS zcl_sd_order_validator DEFINITION.
  PUBLIC SECTION.
    METHODS validate_order
      IMPORTING order_id TYPE zorder_id
      RETURNING VALUE(result) TYPE abap_bool.

    METHODS has_blocked_items
      IMPORTING order_id TYPE zorder_id
      RETURNING VALUE(result) TYPE abap_bool.
ENDCLASS.

zcl_sd_order_validator 表示一个校验者；validate_order 表示执行校验动作；has_blocked_items 读起来像一个 predicate。相比之下，check、execute、do_it、handle 这类名字太宽，除非上下文非常小，否则会让读者继续追源码。

注意 is_ / has_ 的语境。is_valid( ) 作为 boolean method 名，表示“是否有效”；但在许多旧 SAP 命名规范里，is_order 可能表示 importing structure parameter。两者前缀一样，含义完全不同。读旧代码时先判断它是 method、parameter 还是变量；写新代码时不要让这两套约定在同一对象里互相打架。

46.4.6 4.6 Avoid noise words：不要用空泛词掩盖职责

data、info、object、manager、helper、processor 这些词经常出现在 AI 生成或赶工代码里。它们的问题不是绝对不能用，而是太容易什么都不说明。

" Weak
DATA customer_info TYPE zcustomer.
DATA response_time_variable TYPE i.
CLASS zcl_sd_order_manager DEFINITION.

" Better
DATA customer TYPE zcustomer.
DATA response_time_in_seconds TYPE i.
CLASS zcl_sd_order_validator DEFINITION.

如果 manager 真的是协调多个 collaborator 的对象，可以保留；如果它只是因为想不出名字而写上的后缀，就应该拆成更准确的职责名。

46.4.7 4.7 Avoid encodings：Clean ABAP 理想与 SAP 现场遗留

Clean ABAP 倾向避免 Hungarian notation 和类型前缀。它背后的理由很直接：前缀通常只告诉你技术类型，不告诉你业务含义；类型一旦变化，前缀还可能变成假信息。下面两组名字表达的是同一个数据，但阅读体验不同：

" Legacy-style prefix
DATA lv_order_id TYPE zorder_id.
DATA lt_orders TYPE STANDARD TABLE OF zsales_order.

" Clean ABAP-style intent
DATA order_id TYPE zorder_id.
DATA sales_orders TYPE STANDARD TABLE OF zsales_order.

但很多 SAP 项目，尤其是历史较长的日本项目，会在開発標準里要求 lv_、ls_、lt_、iv_、ev_、rv_。初学者要学会“双语阅读”：看到旧代码前缀时能理解它们；新写 ABAP Cloud / RAP 练习对象时，如果项目没有相反规定，可以从一开始使用更接近 Clean ABAP 的意图命名；维护旧对象时不要在同一 method 或同一 class 里一半 lt_orders、一半 sales_orders，造成更大的不一致。

还有一个容易混淆的点：is_ 在旧式 parameter 命名里可能表示 importing structure，在 Clean ABAP boolean method 里可能表示 predicate，例如 is_empty。这就是为什么项目规则和上下文必须清楚。

46.4.8 4.8 Built-in functions：不要遮蔽 ABAP 内置函数

Clean ABAP 特别提醒，不要在 class 里定义和 ABAP built-in function 同名的 method，例如 lines、line_exists、strlen、condense。这样会让读者分不清你是在调用内置函数还是类自己的 method，也可能让语义变得混乱。

" Avoid
METHODS lines RETURNING VALUE(result) TYPE i.
METHODS strlen RETURNING VALUE(result) TYPE i.

如果确实需要表达“计算行数”，可以用更具体的业务名字，例如 count_open_items、count_failed_messages。

46.4.9 4.9 Naming review checklist：把风格争论变成可执行标准

命名 review 不应该停留在“我觉得这个名字不好”。可以用下面的表把意见落到可执行标准上：

QuestionGood signRisk signal
Does the name reveal intent?blocked_sales_orders、calculate_net_amount。data、result、flag、process、handle。
Is the concept consistent?同一概念始终用 business_partner 或项目规定缩写。同一对象在不同 method 里叫 bp、partner、customer。
Is it searchable?关键词能被 where-used / text search 找到。太短、太泛，或者只靠注释解释。
Is it at the right domain level?业务类用业务词，技术 helper 用技术词。业务规则 method 叫 convert_data，技术类却混入客户业务词。
Is boolean meaning clear?is_released、has_errors、can_post。flag、ok、check，读者不知道 true 代表什么。
Does it respect project context?新对象遵守当前 naming convention，旧对象保持内部一致。在同一 class 中混用 Clean ABAP 风格和旧前缀风格。
Does it avoid misleading type encodings?名字表达含义，类型由 declaration / ADT / signature 看。lt_ 变量后来变成 hashed table 或 structure，前缀误导。
Does it avoid built-in shadowing?method 名避开 lines、strlen、line_exists。自定义 method 和 ABAP built-in function 同名。

一个实用原则是：重命名前先判断范围。局部变量可以小步改；public method、interface、repository object、API 字段名和外部 contract 改名会影响 where-used、测试、文档、接口和 transport，应当作为设计变更处理，而不是普通清理。

46.5 日本项目现场说法

  • 常见日语：命名規約、命名ルール、可読性、検索性、保守性、意図が分かる名前、ドメイン用語、略語、接頭辞、ハンガリアン記法、既存規約、チーム規約、命名揺れ、表記ゆれ。
  • review 说法：この名前では処理意図が読み取りにくいです。、命名規約に合わせてください。、この略語はプロジェクト標準ですか。、read / get / fetch の使い分けを統一してください。
  • 说明句：既存オブジェクトでは接頭辞付きの命名に合わせますが、新規クラスでは Clean ABAP の方針に沿って意図が分かる名前を優先します。
  • 交接句：このメソッド名は業務意図を表しているため、障害調査時に where-used と全文検索で追いやすくなります。

日本 SAP 项目里，命名讨论经常不是纯代码风格讨论，而是 開発標準、レビュー指摘、保守性、影響調査 的一部分。表达时不要说“这个名字不好看”，而要说“意図が分かりにくい”“検索性が低い”“命名規約と揺れています”。

46.6 English Terms

  • Official / search terms: Clean ABAP naming, Use descriptive names, Use snake_case, Avoid abbreviations, Avoid Hungarian notation, ABAP built-in functions obscuring.
  • Related terms: problem domain terms, solution domain terms, noise words, one word per concept, nouns for classes, verbs for methods, boolean methods, repository object naming.
  • Recommended search pattern: Clean ABAP Names Use descriptive names snake_case Hungarian notation SAP.
  • 读官方或一手资料时，保留 Clean ABAP、snake_case、Hungarian notation、built-in functions、development object、namespace 等英文术语。

46.7 Example / 学习例子

先看一个常见初学者版本：

METHOD get_data.
  DATA lt_data TYPE STANDARD TABLE OF zsales_order.
  DATA lv_flag TYPE abap_bool.

  SELECT * FROM zsales_order
    INTO TABLE @lt_data.

  LOOP AT lt_data INTO DATA(ls_data).
    IF ls_data-status = 'B'.
      lv_flag = abap_true.
    ENDIF.
  ENDLOOP.
ENDMETHOD.

这段代码不是语法错误，但读者要不断猜：data 是什么数据？flag 表示什么？B 是什么状态？get_data 是读取、过滤、检查，还是准备输出？

更适合教程的写法是：

METHOD has_blocked_sales_orders.
  DATA blocked_status TYPE zorder_status VALUE 'B'.
  DATA sales_orders TYPE STANDARD TABLE OF zsales_order.

  SELECT FROM zsales_order
    FIELDS order_id, status
    WHERE status = @blocked_status
    INTO TABLE @sales_orders.

  result = xsdbool( sales_orders IS NOT INITIAL ).
ENDMETHOD.

这里仍然只是简化示例，但名字已经提供了阅读路径：method 回答“是否存在 blocked sales orders”；blocked_status 表示业务状态；sales_orders 表示 internal table 内容。后续如果进入 Small methods 小方法原则 和 ABAP Unit 单元测试基础，这个 method 也更容易被测试。

46.8 Counterexample / Failure mode

  • 误区 1：把所有变量都叫 lt_data、ls_data、lv_value。修正：用名字表达集合内容和业务含义，例如 open_items、failed_messages、customer_numbers。
  • 误区 2：method 全叫 get_data、process、execute。修正：写出动作和对象，例如 read_open_sales_orders、validate_posting_period、map_bapi_messages。
  • 误区 3：为了显得面向对象，随便使用 manager、factory、service、helper。修正：只有真的符合职责或设计模式时再用。
  • 误区 4：在一个旧对象里局部改成 Clean ABAP 风格，导致前半段 lt_orders，后半段 sales_orders。修正：维护旧对象时优先保持对象内一致；重构要有范围和 review。
  • 误区 5：混淆 Clean ABAP naming 和项目 repository 命名。修正：ZCL_、ZIF_、/ABC/... 是客户 namespace / object naming 规则；本节的 sales_orders、read_entries 是代码内部可读性规则。
  • 误区 6：在 class 里定义 lines、strlen 这类 method 名。修正：避免遮蔽 built-in functions，改成具体业务动作。

46.9 思考问题

  • 你现在看到 lt_data、lv_flag、get_data 时，能不能说出它们缺少什么信息？
  • 什么时候应该遵守旧项目的 lv_ / lt_ 前缀，什么时候可以在新对象里使用 Clean ABAP 风格？
  • read、get、fetch、select 在你的项目里是否有约定？如果没有，应该如何统一？
  • 一个 class 叫 ZCL_SD_ORDER_MANAGER 时，你能不能从名字判断它到底负责什么？如果不能，应该如何改名？
  • 为什么命名会影响 Where-used list 影响分析、review 和障害調査？

  • 找一段练习代码，把 lt_data、lv_flag、get_data 改成能表达业务含义的名字。
  • 写 3 个 boolean method 名称，要求能自然放进 IF ... . 里阅读，例如 has_missing_items( )。
