Chapter 02 - Core ABAP Programming

11. MODIFY statement 修改语义

  • Chapter 02
  • Section 11
  • self-contained study note

11.1 一句话结论

ABAP SQL 的 MODIFY 用来把来源行写入可写数据库对象：目标主键不存在时插入，目标主键已存在时覆盖既有行。它接近 upsert 语义，但不能把它理解成“省事版 UPDATE”。学习重点是确认业务是否真的允许“有则覆盖、无则新增”，来源结构是否完整可信，实际处理了几行，事务、锁、权限、Clean Core 和 RAP save sequence 是否都在正确边界内。

11.2 到底学什么

先把边界说清楚：本节讲的是 ABAP SQL MODIFY，也就是对数据库持久化对象执行插入或覆盖；它不是内部表 MODIFY，也不是 RAP EML 的 MODIFY ENTITIES。内部表 MODIFY 只是改 ABAP 内存；ABAP SQL MODIFY 会进入数据库写入路径；MODIFY ENTITIES 则属于 EML 和 RAP business object 的事务模型。

一个最小的 ABAP SQL MODIFY 示例长这样：

DATA(ls_state) = VALUE zrequest_state(
  request_id = lv_request_id
  status     = lc_status_processed
  changed_by = sy-uname
  changed_at = lv_changed_at ).

MODIFY zrequest_state FROM @ls_state.

DATA(lv_modify_subrc) = sy-subrc.
DATA(lv_processed)    = sy-dbcnt.

读这段代码时，不要只看“执行成功没有”。要先问：request_id 是否就是唯一业务 key？如果这行已经存在，是否允许用 ls_state 的内容覆盖它？如果这行不存在，是否允许创建新行？如果 ls_state 里某些非 key 字段是 initial，会不会把数据库里原本有值的字段覆盖成空？这几个问题回答不清楚，就不应该轻易使用 MODIFY。

多行写入使用内表：

MODIFY zrequest_state FROM TABLE @lt_state.

这行看起来简洁，但风险更集中：lt_state 每一行都会按 key 被插入或覆盖；sy-dbcnt 反映处理件数；如果唯一索引冲突、来源内表里有重复 key、或部分行无法处理，就必须有日志和错误策略。批量 MODIFY 不应该只是“把内表丢给数据库”，而应该是一个经过校验的 upsert 批处理。

11.3 为什么重要

INSERT、UPDATE、DELETE 的意图相对清楚：新增、更新、删除。MODIFY 最危险的地方，正是它把“新增”和“更新”合在一个语句里。这个合并在接口幂等、状态同步、缓存刷新、主数据导入、日志 upsert 等场景里很有价值；但在订单创建、财务过账、审批流、库存移动、标准业务数据修正等场景里，也可能隐藏严重业务差异。

它的第一层价值是表达 upsert。比如外部系统重复推送同一个 request id，本系统希望记录“最后一次处理状态”。如果同一 request id 不存在就创建，存在就覆盖状态，这就是合理的 MODIFY 候选场景。

第二层价值是防止误用。初学者常因为不想处理 INSERT 的重复 key，也不想处理 UPDATE 的 0 件，于是直接换成 MODIFY。这会把“重复创建”“目标不存在”“并发覆盖”这些本该暴露出来的问题变成静默覆盖。代码看似更短，业务语义反而更模糊。

第三层价值是字段覆盖意识。MODIFY ... FROM @wa 不是“只改 wa 里我关心的字段”。如果目标行已存在，它会用来源行覆盖既有行的非 key 字段。来源结构不完整、字段默认 initial、从旧数据读出后又被别人修改过，都可能导致 lost update。

第四层价值是事务边界。MODIFY 成功不等于业务已经提交。它仍然属于当前 Database LUW / SAP LUW 的一部分，最终是否永久生效取决于 COMMIT WORK、ROLLBACK WORK 或框架事务。可复用方法里随手提交，会破坏调用方的事务设计。

第五层价值是 Clean Core。能写 MODIFY mara ... 这类语法，不代表项目允许直接覆盖 SAP 标准表。SAP S/4HANA、ABAP Cloud、RAP 和 Clean Core 语境下，要优先使用 released API、BAPI、RAP BO、业务事务、扩展点或项目认可的持久化对象。

11.4 Key mechanism / 核心机制

11.4.1 MODIFY 的语义：insert or overwrite

SAP Help 对 ABAP SQL MODIFY 的核心定义是：把来源行插入目标可写数据库对象，或覆盖已经存在的行。换成项目语言，就是：

如果 key 不存在：insert
如果 key 已存在：overwrite / update existing row

所以 MODIFY 不是普通 UPDATE。UPDATE 在目标行不存在时不会创建新行；MODIFY 会。MODIFY 也不是普通 INSERT。INSERT 在同主键行已存在时应暴露冲突；MODIFY 会尝试覆盖既有行。

学习时可以把三者放在一张小表里：

INSERT  : 期望目标行不存在；已存在通常是冲突。
UPDATE  : 期望目标行已存在；不存在通常是异常。
MODIFY  : 明确允许不存在则插入、存在则覆盖。

这张表比语法更重要。写入语句首先表达业务意图，然后才是数据库操作。

11.4.2 MODIFY ... FROM @wa：按 key 处理单行

单行 MODIFY 的来源通常是一个结构：

DATA(ls_pref) = VALUE zuser_pref(
  user_id    = sy-uname
  pref_key   = 'DEFAULT_PLANT'
  pref_value = lv_plant
  changed_at = lv_changed_at ).

MODIFY zuser_pref FROM @ls_pref.

IF sy-subrc <> 0 OR sy-dbcnt <> 1.
  " Handle unexpected persistence result.
ENDIF.

这个例子里，user_id 和 pref_key 可以共同构成偏好设置的 key。如果记录不存在，创建一条偏好；如果已经存在，覆盖偏好值。这种语义通常是合理的，因为“同一个用户同一个偏好键只有一条当前值”。

但如果业务是“创建订单备注”，就不一定合理。创建备注时，同一个 note_id 已经存在往往是冲突，不应该静默覆盖原备注。此时 INSERT 更清楚：

INSERT zorder_note FROM @ls_note.

IF sy-subrc = 4.
  " Duplicate key: this is a business conflict, not a silent upsert.
ENDIF.

如果业务是“只允许修改已存在备注”，则 UPDATE 更清楚：

UPDATE zorder_note
   SET note_text  = @lv_note_text,
       changed_by = @sy-uname,
       changed_at = @lv_changed_at
 WHERE note_id = @lv_note_id.

IF sy-dbcnt <> 1.
  " Expected exactly one existing row.
ENDIF.
11.4.3 MODIFY ... FROM TABLE @itab：批量 upsert

批量 MODIFY 看起来像批量同步：

MODIFY zrequest_state FROM TABLE @lt_state.

DATA(lv_processed) = sy-dbcnt.

IF sy-subrc <> 0 OR lv_processed <> lines( lt_state ).
  " Log the failed or unexpected rows according to the batch design.
ENDIF.

这种写法适合“来源系统给出一批当前状态，本系统按 key 做覆盖同步”的场景。比如接口状态表、临时统计表、用户偏好、可重算缓存、幂等请求状态等。

批量场景要多问三件事：

  • lt_state 里是否有重复 key？如果有，哪一条应该获胜？
  • 覆盖既有行时，来源行是否包含所有必须保留的非 key 字段？
  • 如果只有部分行处理成功，日志、重试、回滚和告警怎么做？

如果答案是“失败一行就整批回滚”，要确保当前事务边界能做到；如果答案是“成功多少算多少”，就必须把失败行记录到 application log 或接口返回里。不要只靠一行 MODIFY ... FROM TABLE 结束设计。

11.4.4 sy-subrc 与 sy-dbcnt

MODIFY 会设置 sy-subrc 和 sy-dbcnt。初学者可以先记住三个常用判断：

sy-subrc = 0 : 单行已插入/修改；多行全部处理；或者来源内表为空。
sy-subrc = 4 : 行未处理，或多行中并非全部处理。
sy-dbcnt    : 已处理的行数。

来源内表为空时，sy-subrc 仍可能是 0，sy-dbcnt 是 0。因此批量处理不能只写：

IF sy-subrc = 0.
  " All good?
ENDIF.

更好的写法是把件数和业务预期一起检查：

DATA(lv_expected)  = lines( lt_state ).
DATA(lv_processed) = sy-dbcnt.

IF lv_expected = 0.
  " Nothing to persist. Decide whether this is allowed.
ELSEIF sy-subrc <> 0 OR lv_processed <> lv_expected.
  " Not all rows were processed.
ENDIF.

另外，如果变更会造成 unique secondary index 的重复，目标行可能不会被处理，sy-subrc 会反映失败。也就是说，主键不是唯一性规则的全部；表上定义的唯一索引、数据模型约束和业务 key 都要一起看。

11.4.5 MODIFY 没有业务条件版的 WHERE

这是 MODIFY 和 UPDATE ... SET ... WHERE ... 的一个关键差异。UPDATE 可以把业务条件写进 WHERE：

UPDATE zsales_order
   SET status = @lc_status_released
 WHERE sales_order_id = @lv_sales_order_id
   AND status         = @lc_status_checked
   AND changed_at     = @lv_old_changed_at.

这段代码表达的是：只有订单仍处于预期状态、并且没有被别人改过，才允许更新。如果条件不满足，sy-dbcnt 会是 0，调用方可以报告并发冲突或状态不合法。

MODIFY 则主要靠来源行的 key 决定目标行。它适合明确 upsert，不适合表达“只有当前状态仍为 X 才覆盖”的业务规则。遇到状态流转、乐观锁、ETag、版本号、审批状态、库存扣减等场景，通常要优先考虑 UPDATE ... WHERE ...、lock object、RAP lock / ETag 或业务 API，而不是直接 MODIFY。

11.4.6 覆盖不是局部更新

下面这个例子很危险：

DATA(ls_note) = VALUE zorder_note(
  note_id   = lv_note_id
  note_text = lv_note_text ).

MODIFY zorder_note FROM @ls_note.

如果 zorder_note 还有 sales_order_id、created_by、created_at、changed_by、status 等字段，而 ls_note 使用的是完整表类型，那么未赋值字段可能是 initial。目标行已存在时，这些字段可能被覆盖成空或默认值。项目 review 里看到这种写法，要立刻问：这是完整行覆盖，还是误把 upsert 当成局部字段更新？

如果只是改一个字段，用 UPDATE ... SET ... WHERE ... 更清楚。如果确实要覆盖整行，先从可信来源构造完整行，并说明字段来源和覆盖规则。

11.4.7 数据库锁、SAP lock 与提交边界

ABAP SQL 写入会在数据库层参与锁和事务，官方文档也提示 MODIFY 会设置数据库锁直到下一次 database commit 或 rollback。但数据库锁不是 SAP 业务锁。它通常只覆盖数据库操作和当前 database LUW，不能自动表示“这个业务对象正在被用户编辑”。

SAP 项目里保护业务对象通常还需要 SAP logical lock、RAP lock、号码段、状态检查、版本字段或 ETag。比如两个用户都读出同一行，然后分别构造完整结构执行 MODIFY，后保存的人可能覆盖先保存人的字段。MODIFY 自身不会解决 lost update。

11.4.8 Clean Core 与标准表边界

MODIFY 能操作 writable database object，但“技术上可写”不等于“项目上可写”。尤其是 SAP 标准表、标准业务对象底表、缓存表、聚合表、状态表，直接覆盖可能绕过：

  • 标准业务校验
  • 授权检查
  • 变更文档
  • 派生逻辑
  • 缓存刷新
  • 锁和状态机
  • 升级兼容性

因此，真实项目里要优先确认 released API、BAPI、RAP BO、业务事务、扩展点或项目自有表。对 SAP 标准表直接 MODIFY，通常应视为高风险修复动作，而不是普通开发习惯。

11.4.9 RAP 中不要混淆 MODIFY 与 MODIFY ENTITIES

RAP 里常见的修改入口是 EML：

MODIFY ENTITIES OF zi_sales_order
  ENTITY SalesOrder
  UPDATE FIELDS ( Note )
  WITH lt_update
  FAILED   DATA(lt_failed)
  REPORTED DATA(lt_reported).

这和 ABAP SQL MODIFY ztable FROM @wa 不是一回事。MODIFY ENTITIES 进入 RAP business object 的 behavior、authorization、validation、determination、lock、save sequence 和消息返回模型；直接 ABAP SQL MODIFY 底表则可能绕过这些机制。

在 RAP handler、validation、determination 或 action 里，如果为了“赶快落库”直接 MODIFY ztable，再配合随手 COMMIT WORK，通常是高风险设计。相关边界要和 COMMIT WORK restriction in RAP、RAP action handler 直写反模式、RAP saver 和 EML 一起学习。

  • 登録または更新

  • Upsert

  • 主キー
  • 重複キー
  • 更新件数
  • 冪等性
  • トランザクション境界
  • 直テーブル更新

设计说明里可以这样说：

この処理では、同一リクエストIDの状態を登録または更新するため、MODIFY文を使用します。

代码 review 里可以这样问：

MODIFY文は既存行を上書きする可能性があるため、業務上 upsert でよいか確認してください。

如果担心字段覆盖，可以说：

入力構造に未設定の項目がある場合、既存行の値を初期値で上書きしないか確認してください。

如果担心标准表直写，可以说：

SAP標準テーブルを直接 MODIFY せず、Released API / BAPI / RAP BO / 業務トランザクションの利用可否を確認します。

11.7 Example / 学习例子

11.7.1 例 1：合理的 upsert - 请求状态表

接口处理里经常会有一个请求状态表。外部系统可能重复发送同一个 request id，本系统希望记录最后处理状态：

DATA(ls_state) = VALUE zrequest_state(
  request_id   = lv_request_id
  status       = lc_status_processed
  payload_hash = lv_payload_hash
  changed_by   = sy-uname
  changed_at   = lv_changed_at ).

MODIFY zrequest_state FROM @ls_state.

IF sy-subrc <> 0 OR sy-dbcnt <> 1.
  " Persisting the request state did not match the expected single-row upsert.
ENDIF.

这里使用 MODIFY 的理由可以写清楚：同一个 request_id 只保留一条当前状态；重复请求可以覆盖状态；调用方关心最终状态，而不是每次重复请求都生成新记录。

11.7.2 例 2：不该用 MODIFY 的新增场景

如果业务是创建订单备注，重复 key 应该暴露为冲突：

INSERT zorder_note FROM @ls_note.

IF sy-subrc = 4.
  " Duplicate note id. Report a conflict instead of overwriting silently.
ENDIF.

这里改成 MODIFY zorder_note FROM @ls_note 会隐藏重复创建的问题。短期看少写了错误处理，长期看会让审计和问题追踪变困难。

11.7.3 例 3：不该用 MODIFY 的状态流转

如果业务要求“只有 checked 状态才能 release”，要把当前状态写进条件：

UPDATE zsales_order
   SET status     = @lc_status_released,
       changed_by = @sy-uname,
       changed_at = @lv_changed_at
 WHERE sales_order_id = @lv_sales_order_id
   AND status         = @lc_status_checked.

IF sy-dbcnt <> 1.
  " The order is missing or not in the expected status.
ENDIF.

这种语义不适合直接 MODIFY，因为 MODIFY 主要按 key 插入或覆盖，不能自然表达“只有旧状态为 checked 才允许变更”。

11.7.4 例 4：批量 MODIFY 前先定义重复 key 策略
" lt_state_to_upsert must already be validated:
" - one row per request_id
" - complete non-key fields
" - explicit rule for retry and overwrite
MODIFY zrequest_state FROM TABLE @lt_state_to_upsert.

DATA(lv_expected)  = lines( lt_state_to_upsert ).
DATA(lv_processed) = sy-dbcnt.

IF sy-subrc <> 0 OR lv_processed <> lv_expected.
  " Write an application log entry and let the transaction owner decide rollback.
ENDIF.

这段代码的重点不在语法，而在注释里的前提。批量 upsert 必须先决定重复 key、覆盖字段、失败行、日志和事务策略。

11.8 Counterexample / Failure mode

失败模式一：为了躲开 INSERT 重复 key，直接换成 MODIFY。这会把“重复创建”静默变成“覆盖既有数据”。

失败模式二：本来只想更新已存在行，却用 MODIFY。目标行不存在时，它可能创建新行，导致系统出现不该存在的业务数据。

失败模式三：用不完整结构执行 MODIFY ... FROM @wa。既有行被覆盖时，未赋值字段可能变成 initial 或旧值，造成数据丢失。

失败模式四：用 MODIFY 表达状态流转。状态流转通常需要 WHERE status = ...、版本号、ETag、lock 或 RAP validation，而不是简单 key upsert。

失败模式五：批量 MODIFY 前不处理重复 key。来源内表里同一业务 key 出现多次时，最终结果可能无法从业务上解释。

失败模式六：只看 sy-subrc 不看 sy-dbcnt。来源内表为空、部分处理、或处理件数不符合预期时，业务仍可能需要报错。

失败模式七：混淆 ABAP SQL MODIFY、内部表 MODIFY 和 RAP MODIFY ENTITIES。三者语法相似，语义和事务边界完全不同。

失败模式八：直接 MODIFY SAP 标准表。即使技术上可执行，也可能绕过标准业务逻辑、授权、变更文档、缓存、锁和升级稳定性。

11.9 思考问题

  • 这段代码需要的是 INSERT、UPDATE，还是明确的 upsert？
  • 如果目标 key 已存在，覆盖既有行是否符合业务规则？
  • 如果目标 key 不存在，创建新行是否符合业务规则？
  • 来源结构是否包含完整且可信的非 key 字段？
  • sy-dbcnt 应该是多少？0、1、大于 1 分别代表什么业务含义？
  • 当前方法是否拥有事务边界？谁负责 COMMIT WORK 或 rollback？
  • 是否需要 lock object、RAP lock、ETag、版本号或状态条件来防止 lost update？
  • 是否直接写了 SAP 标准表？有没有 released API、BAPI、RAP BO 或正式扩展点？
  • 在 RAP 中，这里应该是 ABAP SQL MODIFY，还是 EML MODIFY ENTITIES / saver 逻辑？
