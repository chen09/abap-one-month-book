45. Message class and T100 messages

  • Chapter 02
  • Section 45
  • self-contained study note

45.1 一句话结论

Message class and T100 messages 是 ABAP 里管理可复用、可翻译业务消息的基础机制。message class 提供 message ID 和 message number 对应的文本目录，T100 是这个经典消息表背后的关键词；MESSAGE statement 负责把这些消息按类型、占位符和运行时语境发出或格式化。初学者要学会区分三件事：消息文本从哪里来、消息类型如何影响控制流和 UI、消息如何与 exception class、BAPI/API 返回、RAP reported / failed 连接。

本节不是重新讲 Exception class 异常类 或 TRY CATCH exception handling。异常类回答“错误路径用什么对象表达”，TRY...CATCH 回答“异常发生后控制流如何走”；本节回答“给用户、日志、接口或上层框架看的业务消息如何标准化、翻译和传递”。

45.2 到底学什么

在 SAP 项目里，错误和提示不是随手拼一个字符串。一个可交付的系统通常需要：

  • 中文、日语、英语等多语言文本可以翻译。
  • 同一个业务问题在画面、batch、接口、日志中有一致的 ID / number。
  • message text 可以带订单号、客户号、字段名等动态值。
  • 用户看到的文本和开发者排查用的 message key 能对应起来。
  • 低层业务类不要随便弹出 dialog message，避免破坏调用方的 UI、事务边界或 API contract。

所以，本节要掌握七个问题：

  • message class 是什么，和 T100 message 有什么关系。
  • MESSAGE e001(zmsg) WITH ... 与 MESSAGE ID ... TYPE ... NUMBER ... WITH ... 如何读。
  • WITH 的四个 message variables 如何填入 &1、&2、&3、&4。
  • S / I / W / E / A / X message type 为什么不是普通字符串级别。
  • MESSAGE ... INTO lv_text 何时用于“格式化文本而不直接显示”。
  • sy-msgid、sy-msgno、sy-msgty、sy-msgv1 到 sy-msgv4 为什么常在日志和错误返回里出现。
  • T100 message 如何和 class-based exception、BAPIRET2、RAP message 对象衔接。

一个最小例子：

MESSAGE e001(zorder) WITH lv_order_id.

这行代码可以读成：从 message class ZORDER 取 message number 001，以 error message 的方式发出，并把 lv_order_id 作为第一个占位符值。它不是简单 WRITE，也不是普通字符串拼接；它有 message key、message type、placeholder、语言翻译和运行时行为。

45.3 为什么重要

message class 和 T100 message 是 SAP 项目里“用户可理解反馈”和“技术可追踪证据”的交叉点。一个保存失败、校验失败、权限不足、外部系统调用失败、数据缺失、配置缺失，都可能最终需要变成一条消息。消息写得好，用户能知道要修什么，测试人员能截图或记录 message ID，开发者能按 ID / number 找到代码和文本，运维能把日志和接口返回对应起来。

写得不好时，问题会很难查：

  • 代码里到处硬编码中文、日文或英文字符串，无法统一翻译。
  • 同一个业务错误在不同程序里用不同文本，测试无法判断是否同一问题。
  • message type 写成 E、A 或 X 后，运行时中断行为超出预期。
  • 低层 reusable class 里直接 MESSAGE e...，调用方本来想收集错误，却被画面逻辑打断。
  • 接口只返回本地化后的 message text，外部系统无法稳定识别错误类型。
  • RAP validation 里直接沿用传统 dialog MESSAGE 思维，导致消息没有正确进入 reported / failed。

对 Java/Python 背景学习者，可以把 message class 类比成“可翻译的错误码目录 + 格式化模板”。但 SAP 的差异是：message type 会影响 ABAP runtime / SAP GUI / dynpro / list / batch / RAP 等不同语境中的行为；同一个 message number 不是普通 exception type；message text 也不应该成为程序判断条件。项目里真正稳定的判断依据是 message ID、number、type、变量和业务上下文，而不是翻译后的自然语言文本。

45.4 Key mechanism / 核心机制

45.4.1 Message class 是消息目录

message class 可以理解成一组编号消息的目录。每条消息通常由以下信息识别：

  • language，例如当前登录语言。
  • message class / message ID，例如 ZORDER。
  • message number，例如 001。
  • message text，例如 Order &1 is blocked for customer &2。

这里有一个术语细节必须讲清楚：很多文档和代码里说 message ID，很多项目现场说 message class，它们在这个语境下指向同一个标识。MESSAGE e001(zorder) 里的 zorder 就是 message class 名，也是运行时 message ID。初学者不要误以为 message ID 是另一个单独对象。

经典表名 T100 经常出现在文档、调试、接口返回和老项目说明里。T100 的 key 思维可以简单理解为：language + message class / ID + 3 位 message number。另一个技术表 T100A 保存 message class 定义本身；如果有人绕过工具直接维护 T100，而没有正确的 message class 定义，就可能出现 SE91 / ADT 看不到或一致性异常的问题。

因此，初学者不需要、也不应该直接维护数据库表 T100。实际开发通常通过 ADT / ABAP Workbench 的 message class 工具维护消息类、消息号、短文本和长文本。message class 是 repository object，通常作为开发对象随 transport 迁移，而不是当成 client-dependent Customizing 随便在本番手改。项目里说 T100メッセージ，通常就是指这类可翻译的 SAP 标准消息机制，而不是让你直接操作 T100 表。

命名上，customer message class 常见 Z... / Y...，也会按模块起名，例如 ZSD_ORDER、ZMM_PURCHASE。不要把所有项目错误都塞进一个 ZMSG，否则后期查找、翻译和责任归属都会变差。

45.4.2 Message type 是运行时行为，不是消息目录的一部分

message class 里保存的是 message ID / number / text。message type 通常在 MESSAGE statement 发出消息时指定：

MESSAGE e001(zorder) WITH lv_order_id.

这里 e 表示 error message。相同 message number 在不同语境下也可能用不同 type 发出，例如某些检查在输入阶段是 warning，在保存阶段是 error。这个选择属于运行时控制和业务流程设计，不是单纯文本维护。

常见 type 可以先这样理解：

Type常见含义初学者注意点
SSuccess通常表示成功或状态提示；也可能配合 DISPLAY LIKE 改变显示样式
IInformation通常要求用户确认或阅读信息，可能打断交互流程
WWarning表示可以继续但需要注意；在不同 UI 阶段行为可能不同
EError通常阻止当前步骤继续，常见于输入校验
AAbend / termination高风险，会终止当前处理；不要用于普通业务校验
XExit / runtime error通常导致 short dump；普通业务错误不应该用它

message type 的实际效果依赖运行语境。dialog screen、selection screen、list processing、background job、update task、RAP behavior、OData/API 入口都可能不同。书写代码时不能只问“文本对不对”，还要问“这个 type 在当前上下文会不会中断不该中断的流程”。

45.4.3 两种常见 MESSAGE 写法

静态 message class / number 常写成：

MESSAGE e001(zorder) WITH lv_order_id.

动态指定 message ID、type、number 时常写成：

MESSAGE ID 'ZORDER' TYPE 'E' NUMBER '001'
        WITH lv_order_id.

第一种适合 message class、type 和 number 在代码里明确固定的场景；这些信息在语法检查时就写死了。第二种允许 message ID、type、number 在运行时由变量提供，因此常见于把 sy-msgid、sy-msgty、sy-msgno 和 sy-msgv1 等字段重新格式化、转发、记录日志，或者从配置 / 返回结构中接收消息 key 的场景。

例如从 BAPI return table 转发消息时，通常不能写静态 MESSAGE e001(zorder)，因为 ID、number、type 来自返回结构：

MESSAGE ID ls_return-id
        TYPE ls_return-type
        NUMBER ls_return-number
        WITH ls_return-message_v1
             ls_return-message_v2
             ls_return-message_v3
             ls_return-message_v4
        INTO DATA(lv_message_text).

这就是长写法的重要价值：它能处理动态决定的消息。相应地，review 也要更严格：ls_return-id、ls_return-number 是否可信，变量是否按正确顺序传入，是否保留了原始 key，而不是只保留最终文本。

不建议初学者一开始就大量动态拼 message ID / number。动态消息更灵活，也更容易隐藏错误：拼错 ID 或 number 时，问题可能到运行时才暴露，翻译和测试也更难覆盖。

45.4.4 WITH 提供 message variables

T100 message text 里常见 &1、&2、&3、&4 这样的 placeholder。MESSAGE ... WITH 会把最多四个值填进去：

" Message 001 in ZORDER:
" Order &1 is blocked for customer &2.

MESSAGE e001(zorder) WITH lv_order_id lv_customer_id.

不要用字符串拼接替代 placeholder：

" Bad for translation and diagnostics
MESSAGE |Order { lv_order_id } is blocked for customer { lv_customer_id }| TYPE 'E'.

硬拼字符串看起来更快，但会损失 message ID / number，也让翻译、长文本、测试、接口返回和日志关联变差。正确习惯是：稳定的业务文本进入 message class，动态值用 placeholders。

placeholder 顺序也很重要。英语可能写 Order &1 is blocked for customer &2，日语可能更自然地把 customer 放前面。翻译时可以调整文本顺序，但 WITH 传入的变量含义必须稳定。不要把 &1 今天当订单号、明天当客户号。

45.4.5 sy-msg* 是消息诊断线索

很多传统 ABAP 代码会在发出或捕获消息后使用这些 system fields：

  • sy-msgid
  • sy-msgty
  • sy-msgno
  • sy-msgv1
  • sy-msgv2
  • sy-msgv3
  • sy-msgv4

它们常用于把一条消息转成日志、BAPI return、异常文本或调试证据。例如某个 function module 失败后，调用方可能读取 sy-msgid / sy-msgno / sy-msgv1 到 sy-msgv4，再放进 BAPIRET2 或 application log。

但不要把 sy-msg* 当成全局长期状态使用。它们是 system fields，容易被后续语句覆盖。和 sy-subrc discipline and silent failure patterns 一样，读取这类运行时结果要靠近产生它的语句，并尽快转成局部变量、日志结构或返回对象。

45.4.6 MESSAGE ... INTO 用于格式化文本

有时你不想直接弹出消息，只想把 T100 message 格式化成一段文本，例如写日志、拼接测试断言、把消息放进接口返回：

MESSAGE ID 'ZORDER' TYPE 'E' NUMBER '001'
        WITH lv_order_id lv_customer_id
        INTO DATA(lv_message_text).

MESSAGE ... INTO 的价值是复用 message class 的翻译和 placeholder 替换，同时避免直接触发 UI message 行为。它适合“需要文本，但不希望这个语句控制画面或事务”的场景。

不过，INTO 只是格式化文本，不等于完整错误处理。日志、API return 或 exception object 仍应保留 message ID、number、type 和变量。只保存最终文本会丢失可稳定检索的错误码，也会让多语言系统里的外部判断变脆。

45.4.7 DISPLAY LIKE 只改变显示，不改变语义

传统 UI 代码里可能看到：

MESSAGE s001(zorder) WITH lv_order_id DISPLAY LIKE 'E'.

这表示消息类型本身仍是 S，但显示样式像 error。它可能用于“处理已完成，但要用红色样式提示用户注意”的场景。初学阶段不需要滥用它，只要记住：显示样式不等于 message type 的语义；review 时要问为什么不用真正的 E，也要问为什么要把 S 显示成 error。

45.4.8 Message class 与 exception class 的关系

message class 是文本目录；exception class 是错误对象和控制流信号。两者可以连接，但不能互相替代。

class-based exception 通过 get_text( ) 提供可读文本。更进一步，exception class 可以实现 IF_T100_MESSAGE 或 IF_T100_DYN_MSG，把 exception object 连接到 T100 message。这样做的好处是：exception 保持类型和上下文，message class 提供可翻译文本。

概念上可以这样理解：

RAISE EXCEPTION TYPE zcx_order_blocked
  EXPORTING
    textid   = zcx_order_blocked=>order_blocked
    order_id = lv_order_id.

这里 exception type 表达“订单被阻止”这个错误类别；textid 和 order_id 让 get_text( ) 能生成可翻译文本。实际项目中，exception class 的 T100 key、属性名和 constructor 由 class 定义决定，不要只复制示例代码。重点是边界：异常对象负责错误路径，T100 message 负责可翻译文本。

45.4.9 MESSAGE ... RAISING 是旧式连接点

你还可能在旧代码里看到 MESSAGE ... RAISING ...。它通常和 classic / non-class-based exception 相关，尤其是 function module 或早期过程式接口。典型含义是：function module 通过 MESSAGE ... RAISING exception_name 抛出 classic exception，同时填充 sy-msgid、sy-msgty、sy-msgno 和 sy-msgv1 到 sy-msgv4，让调用方能在 CALL FUNCTION ... EXCEPTIONS 后读取这些 system fields 或转成返回结构。

现代新代码不应该把它当成首选错误设计；前面 Exception class 异常类 已经讲过，新开发应优先使用 class-based exceptions 或框架规定的 message return 机制。读旧代码时，要能识别它的含义：不是单纯显示消息，而是把 message 和旧式 exception path 绑在一起。迁移时要判断：应该转成 class-based exception、BAPIRET2 / return table、application log，还是 RAP reported / failed message。

在 ABAP Cloud / RAP 项目里，更要把 MESSAGE ... RAISING 看成 legacy 识别点，而不是新设计模板。Cloud-ready helper 应优先使用清晰的 return value、class-based exception、RAP message protocol 或 released framework API；如果保留旧式 function module 调用，也要在边界层立刻把 sy-msg* 固定成结构化返回或日志证据。

45.4.10 RAP 中不要照搬 dialog MESSAGE 思维

RAP behavior implementation 里，业务校验反馈通常通过 reported 和 failed 返回给框架，而不是在 handler 里随手写传统 dialog MESSAGE e...。一个典型方向是创建 message object，并附着到具体 entity / key / field：

APPEND VALUE #(
  %tky = ls_order-%tky
  %msg = new_message(
           id       = 'ZORDER'
           number   = '001'
           severity = if_abap_behv_message=>severity-error
           v1       = ls_order-order_id ) )
  TO reported-order.

这段代码的重点不是让初学者马上背 RAP 语法，而是建立边界意识：在 RAP / OData / API 世界里，消息要进入框架的 response model。用户界面、Fiori elements、OData client、test tool 都需要结构化 message，而不是一个突然弹出的 SAP GUI dialog message。

45.4.11 BAPIRET2 与接口返回也常携带 T100 key

许多传统 BAPI / function module / API 返回结构会使用类似 BAPIRET2 的字段：

  • TYPE
  • ID
  • NUMBER
  • MESSAGE
  • MESSAGE_V1 到 MESSAGE_V4

这正是 message class/T100 思维在接口返回中的体现：既给人类可读 message，也保留机器和开发者可识别的 ID / number / variables。接口设计里不要只返回最终 message text；外部系统通常需要稳定错误码来做判断、重试、映射或翻译。

换句话说，T100 text 是给人看的翻译文本，不应单独承担 API contract。接口契约要保留稳定的 ID、number、type、variables，以及必要时的业务错误码；文本可以变得更自然，但机器判断不应依赖一句可能被翻译或改写的短文本。

45.4.12 它不是业务校验本身

message class 只是表达结果，不是校验逻辑。下面这些事情不能交给 T100 message 自动完成：

  • 权限检查。
  • 数据存在性和一致性校验。
  • lock / enqueue。
  • transaction boundary / commit / rollback。
  • RAP validation / determination / save sequence。
  • API error contract 和 HTTP status 设计。
  • application log / audit log 的持久化策略。

好的代码通常先做业务判断，再把判断结果转换成 message、exception、return table 或 RAP reported message。不要反过来：因为有了 message class，就把业务规则散落在各处 MESSAGE e... 语句里。

45.4.13 message design checklist

设计一条 T100/message class 消息时，可以先用下面的表做 review。重点是让消息同时满足用户可理解、测试可记录、开发可追踪、接口可稳定处理。

QuestionGood message designRisk signal
Who consumes it?明确是 dialog user、batch log、API caller、RAP/Fiori message，还是 developer trace。同一条消息同时想服务所有场景，文本又长又模糊。
Is the key stable?message class + number 可以长期追踪和测试。只返回拼接后的文本，外部系统无法稳定判断。
Are placeholders meaningful?&1-&4 放 order id、field name、status、external id 等关键上下文。占位符顺序混乱，或放入过长/敏感文本。
Is message type intentional?E/W/I/S/A/X 的控制流影响被调用方理解。低层 service 直接 MESSAGE e... 破坏 API/RAP/事务边界。
Is translation possible?固定文本在 message class 中，动态值走 WITH。代码里硬编码中文/日文/英文字符串。
Does API need structure?返回 ID/number/type/variables/text，必要时带 HTTP/RAP field target。只返回最终 text，调用方无法重试、映射或定位字段。
Is evidence captured?对 batch/interface，message key 可进入 Application Log 或 return table。用户看到错误，但运维和开发找不到对应证据。

一个实用原则是：面向用户的 message text 要短而可行动；面向接口和日志的 message payload 要稳定而可追踪。两者可以来自同一条 T100 key，但不要把所有诊断信息都塞进用户文本里。

45.7 Example / 学习例子

45.7.1 例子一：用 message class 表达业务校验

假设 message class ZORDER 里有 message number 001：

Order &1 is blocked for customer &2.

在传统 dialog 程序里可能写：

IF lv_is_blocked = abap_true.
  MESSAGE e001(zorder) WITH lv_order_id lv_customer_id.
ENDIF.

读者要能看出四件事：

  • ZORDER 是 message class / message ID。
  • 001 是 message number。
  • E 是运行时 message type。
  • lv_order_id 和 lv_customer_id 分别进入 &1 和 &2。
45.7.2 例子二：格式化文本但不直接显示

如果这段逻辑运行在 batch、接口或 reusable class 里，直接发 dialog message 可能不合适。可以先格式化成文本，并把 key 一起放进返回结构或日志：

MESSAGE ID 'ZORDER' TYPE 'E' NUMBER '001'
        WITH lv_order_id lv_customer_id
        INTO DATA(lv_message_text).

APPEND VALUE bapiret2(
  type       = 'E'
  id         = 'ZORDER'
  number     = '001'
  message    = lv_message_text
  message_v1 = lv_order_id
  message_v2 = lv_customer_id ) TO lt_return.

这里 lv_message_text 给人读，id / number / message_v1 / message_v2 给系统和开发者追踪。不要只保留 message 文本。

45.7.3 例子三：不要在低层服务类里直接 MESSAGE

不推荐：

METHOD check_order.
  IF lv_is_blocked = abap_true.
    MESSAGE e001(zorder) WITH iv_order_id.
  ENDIF.
ENDMETHOD.

这段代码把低层 method 和当前 UI 行为绑死了。调用者如果是 batch job、OData service、RAP validation 或 ABAP Unit，就很难控制这个错误。

更好的方向是返回结构化结果、抛 class-based exception，或在 RAP 中填充 reported / failed。例如普通业务 service 可以选择：

IF lv_is_blocked = abap_true.
  RAISE EXCEPTION TYPE zcx_order_blocked
    EXPORTING
      order_id = iv_order_id.
ENDIF.

然后在边界层把 exception 转成 message、日志或 API response。这样 message class 仍然可以用于文本，但控制流不被低层 MESSAGE 偷走。

45.7.4 例子四：RAP validation 的消息思维

RAP 中业务反馈要进入 framework message model。初学者可以先读懂这个方向：

APPEND VALUE #(
  %tky = ls_order-%tky
  %msg = new_message(
           id       = 'ZORDER'
           number   = '001'
           severity = if_abap_behv_message=>severity-error
           v1       = ls_order-order_id ) )
  TO reported-order.

APPEND VALUE #( %tky = ls_order-%tky ) TO failed-order.

这不是普通 MESSAGE e001(zorder)。它告诉 RAP：哪个 entity 失败了、失败严重度是什么、要返回哪条 message。Week 3 的 RAP validation 验证 会继续展开。

45.8 Counterexample / Failure mode

常见失败模式：

  • 在代码里硬编码 MESSAGE 'Order is blocked' TYPE 'E'，导致不可翻译、不可追踪。
  • message text 只有“发生错误”，没有业务对象、字段、原因或下一步。
  • 把 message text 当成程序判断条件，遇到翻译就失效。
  • 低层 service / utility class 里直接 MESSAGE e...，破坏调用方的 UI / batch / API 边界。
  • 普通业务校验使用 MESSAGE x...，造成 short dump。
  • 该返回 warning 的地方发 error，导致用户无法继续；该阻止保存的地方只发 success/info，导致错误数据进入后续流程。
  • 使用 MESSAGE ... INTO 后只保存文本，不保存 ID / number / variables。
  • placeholder 顺序不稳定，翻译后变量含义错位。
  • 把同一个 message class 做成垃圾桶，所有模块都往里塞泛用消息。
  • RAP behavior 里沿用 SAP GUI dialog MESSAGE 思维，没有进入 reported / failed。

一个典型反例：

IF lv_is_blocked = abap_true.
  MESSAGE 'Order is blocked' TYPE 'E'.
ENDIF.

它的问题不是“显示不出来”，而是太不可交付：

  • 没有 message ID / number。
  • 没有翻译入口。
  • 没有变量位置约定。
  • 日志和接口无法稳定识别。
  • 低层代码很可能直接控制 UI。

更好的方向：

MESSAGE e001(zorder) WITH lv_order_id lv_customer_id.

或者在非 UI 边界：

MESSAGE ID 'ZORDER' TYPE 'E' NUMBER '001'
        WITH lv_order_id lv_customer_id
        INTO DATA(lv_message_text).

然后把 ID、number、variables 和 text 一起交给日志、返回结构或框架。

45.9 思考问题

  • 这条消息是给终端用户、测试人员、运维、外部系统，还是开发者看的？
  • message class / number 是否能稳定定位业务问题？
  • message type 会不会中断当前流程？这个中断是需求要求的吗？
  • 变量 &1 到 &4 的含义是否稳定，翻译后是否仍然清楚？
  • 这段代码运行在 dialog、batch、RFC/BAPI、OData、RAP，还是 ABAP Unit？
  • 低层方法是否应该直接 MESSAGE，还是应该返回结构化结果或抛 exception？
  • 日志/API 返回是否保留了 ID、number、type 和 variables，而不是只有最终文本？
  • 这条消息能不能指导业务用户下一步该怎么做？
