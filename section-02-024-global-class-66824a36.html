24. Global class 全局类

  • Chapter 02
  • Section 24
  • self-contained study note

24.1 一句话结论

Global class 全局类，是 ABAP repository 中可被多个程序、服务、RAP 对象、测试类复用的 OO ABAP 主力组织单元：它不是“写在某个 report 里的临时 class”，而是一个有包、激活、传输、可见性、public contract 和生命周期管理的 repository object。学会 global class，就是开始用 ABAP 的项目级方式组织业务逻辑、技术边界和可测试代码。

24.2 到底学什么

Day 10 已经讲完 internal table 的读取、更新和引用式访问。Day 11 开始进入 OO ABAP。第一件事不是背“类是对象模板”，而是理解 ABAP 项目里为什么常用 global class。

一个典型 global class 的骨架类似：

CLASS zcl_order_status DEFINITION
  PUBLIC
  FINAL
  CREATE PUBLIC.

  PUBLIC SECTION.
    METHODS close_item
      IMPORTING iv_item_id TYPE zitem_id
      CHANGING  ct_items   TYPE ztt_item.

  PRIVATE SECTION.
    METHODS set_closed
      CHANGING cs_item TYPE zstr_item.
ENDCLASS.

CLASS zcl_order_status IMPLEMENTATION.
  METHOD close_item.
    LOOP AT ct_items ASSIGNING FIELD-SYMBOL(<ls_item>)
         WHERE item_id = iv_item_id.
      set_closed( CHANGING cs_item = <ls_item> ).
    ENDLOOP.
  ENDMETHOD.

  METHOD set_closed.
    cs_item-status = 'C'.
  ENDMETHOD.
ENDCLASS.

这段代码要看懂三层含义：

  • CLASS ... DEFINITION PUBLIC ... 让它成为 class pool 中的 global class，属于 ABAP repository / class library。
  • PUBLIC SECTION 是外部调用者能依赖的 contract。
  • PRIVATE SECTION 和 IMPLEMENTATION 把内部细节藏起来，让代码可维护、可测试、可替换。

本节要学的核心问题：

  • global class 和 local class 有什么区别？
  • 为什么 ABAP 项目里的主要业务逻辑常放在 global class，而不是 report 顶层或一个巨大 method？
  • PUBLIC、FINAL、CREATE PUBLIC、PUBLIC SECTION、PRIVATE SECTION 分别表达什么？
  • global class 为什么和 package、transport、activation、syntax check、ABAP Unit 有关系？
  • 初学者应该怎样设计一个小而清楚的 global class？

24.3 为什么重要

在真实 SAP 项目里，global class 往往是交付代码的核心单元。RAP behavior implementation class、service/helper class、校验类、转换类、API wrapper、batch processing helper、authorization helper、ABAP Unit test target，很多都会以 global class 的形式出现。

如果不会 global class，代码很容易退回到：

  • 一个 report 顶层塞满全局变量和 FORM。
  • 一个方法几百行，没人知道 public contract 是什么。
  • 所有字段都 public，测试和调用者随便改内部状态。
  • 逻辑无法被 ABAP Unit 测试，只能靠手工跑画面或接口。
  • 项目 review 只能看“能不能跑”，不能看“边界是否清楚”。

对 Java 背景学习者，可以把 global class 类比成一个可复用 class，但 ABAP 的关键差异是：它同时是 SAP repository object，属于 package，需要激活，会进入 transport request，并受 ABAP language version、release/use access、Cloud/Clean Core 规则影响。对 Python 背景学习者，可以把它类比成模块中的 class，但 ABAP 的激活、包、传输和 repository 依赖关系更显式。

24.4 Key mechanism / 核心机制

24.4.1 global class 是 repository object

SAP Learning 对 local/global class 的区分很直接：global class centrally stored，位于自己的 repository object，也就是 class pool；local class 则定义在某个 ABAP program 或 global class 的 local types 中，只能在局部使用。

这意味着 global class 有项目交付属性：

  • 属于一个 package。
  • 必须通过 syntax check 并完成 activation。
  • 修改时通常进入 transport request。
  • 可被其他 repository object 调用。
  • 可被 where-used / dependency analysis 追踪。
  • 可作为 ABAP Unit 测试对象或被 test class 覆盖。
24.4.2 definition 和 implementation

ABAP class source 通常分成 definition 和 implementation：

CLASS zcl_demo DEFINITION PUBLIC FINAL CREATE PUBLIC.
  PUBLIC SECTION.
    METHODS run.
ENDCLASS.

CLASS zcl_demo IMPLEMENTATION.
  METHOD run.
    " executable logic
  ENDMETHOD.
ENDCLASS.

definition 说明“这个 class 暴露什么能力、有哪些属性/方法/类型”；implementation 才写 method 的执行逻辑。review 时先看 definition，确认 public contract，再看 implementation 是否履行 contract。

24.4.3 visibility sections

一个 class definition 最常见有三段：

PUBLIC SECTION.
PROTECTED SECTION.
PRIVATE SECTION.
  • PUBLIC SECTION：外部调用者可以访问，应该少而稳定。
  • PROTECTED SECTION：本类和子类可访问，涉及 inheritance 设计。
  • PRIVATE SECTION：只有本类内部访问，适合 helper method、内部状态和实现细节。

本节例子主要使用 PUBLIC SECTION 和 PRIVATE SECTION。PROTECTED SECTION 留到 inheritance 主题再展开，初学阶段先把 public contract 和 private implementation 分清楚。

SAP Help 对 visibility 的建议很重要：public components 要尽量少，并经过仔细设计。因为 public method 一旦被其他对象依赖，就变成项目 contract，后续改动成本会很高。

24.4.4 class options

常见 global class 头部：

CLASS zcl_demo DEFINITION
  PUBLIC
  FINAL
  CREATE PUBLIC.

初学阶段这样理解：

  • PUBLIC：使 class 在 class library 中作为 global class 可见。
  • FINAL：不允许被继承，适合不准备开放 inheritance contract 的业务类。
  • CREATE PUBLIC：允许外部创建实例，例如 NEW zcl_demo( )。这是默认实例化行为；如果要限制外部创建，才显式设计 CREATE PRIVATE、CREATE PROTECTED、factory method 或其他受控取得实例的方式。

如果 class 不应该被外部自由创建，还要同时考虑调用方如何取得实例、测试如何注入依赖；这些属于后续 OO 设计主题。

24.4.5 package / transport / activation

在 ADT 中创建 global class 时，通常要选择 package、填写 description、选择 transport request。这个动作不是形式主义。它决定：

  • 代码归属哪个开发包。
  • 是否可传输到 QAS/PRD。
  • 与其他 repository object 的依赖边界。
  • 是否符合项目命名规则和 Clean Core 约束。

本教程中不要把 global class 当成“本地脚本文件”。它是 SAP 系统里的 repository object。

24.4.6 global class vs local class

global class：

  • 适合被多个对象复用。
  • 有独立 repository object / class pool。
  • 可以被其他程序、RAP 类、测试类引用。
  • 适合业务 service、utility、adapter、validator、application service。

local class：

  • 定义在某个 program/global class 的 local types 中。
  • 只服务当前上下文。
  • 适合 demo、局部 helper、ABAP Unit test double、非常局部的结构化代码。

不要因为“local class 写起来快”就把项目级业务能力藏在 local class；也不要因为“global class 更正式”就把所有微小 helper 都提升成全局对象。

24.5 日本项目现场说法

  • 常见日语：グローバルクラス、ローカルクラス、クラスプール、パッケージ、移送依頼、有効化、公開メソッド、privateメソッド、責務分割。
  • 说明句：このロジックは複数箇所から再利用されるため、レポート内のローカル実装ではなくグローバルクラスとして切り出します。
  • Review 句：PUBLIC SECTION は外部契約になるため、公開メソッドを最小限にして責務と引数を明確にしてください。
  • 交付句：このクラスはパッケージと移送依頼に紐づくリポジトリオブジェクトなので、有効化と依存関係を確認します。
  • 使用场景：基本设计、详细设计、代码 review、ABAP Unit 设计、RAP behavior implementation 拆分、transport 前检查。

24.7 Example / 学习例子

24.7.1 例 1：最小可执行 global class

在学习环境里，常用 IF_OO_ADT_CLASSRUN 创建可以直接 Run As ABAP Application 的 class：

CLASS zcl_demo_hello DEFINITION
  PUBLIC
  FINAL
  CREATE PUBLIC.

  PUBLIC SECTION.
    INTERFACES if_oo_adt_classrun.
ENDCLASS.

CLASS zcl_demo_hello IMPLEMENTATION.
  METHOD if_oo_adt_classrun~main.
    out->write( |Hello, OO ABAP| ).
  ENDMETHOD.
ENDCLASS.

这类 class 适合练习和 demo。项目业务类不一定实现 IF_OO_ADT_CLASSRUN，但这个例子能让初学者理解 global class 的 definition/implementation 结构。

24.7.2 例 2：业务校验类
CLASS zcl_sales_item_validator DEFINITION
  PUBLIC
  FINAL
  CREATE PUBLIC.

  PUBLIC SECTION.
    METHODS is_valid_quantity
      IMPORTING iv_quantity TYPE i
      RETURNING VALUE(rv_valid) TYPE abap_bool.

  PRIVATE SECTION.
    CONSTANTS c_max_quantity TYPE i VALUE 999.
ENDCLASS.

CLASS zcl_sales_item_validator IMPLEMENTATION.
  METHOD is_valid_quantity.
    rv_valid = xsdbool(
      iv_quantity > 0 AND iv_quantity <= c_max_quantity ).
  ENDMETHOD.
ENDCLASS.

这里 public method 是 contract：外部只需要知道“传入数量，返回是否有效”。最大数量常量藏在 private section，后续调整实现时不影响调用者。

24.7.3 例 3：不好的 global class
CLASS zcl_everything DEFINITION PUBLIC FINAL CREATE PUBLIC.
  PUBLIC SECTION.
    DATA gt_items TYPE ztt_item.
    METHODS run_everything.
    METHODS update_database.
    METHODS call_api.
    METHODS format_text.
ENDCLASS.

这个 class 暴露了太多内部状态和职责。它看起来是 OO，实际只是把过程式代码搬进 class。review 时要拆职责：校验、格式化、持久化、外部 API、事务边界不要混在一个巨大 public surface 中。

24.7.4 例 4：global class 与 transport

在 ADT 中创建 ZCL_SALES_ITEM_VALIDATOR 时，选择 package 和 transport request。这个选择会影响对象能否随项目从 DEV 移送到 QAS/PRD。global class 名称、package、依赖对象、activated state 都是交付检查的一部分。

24.8 Counterexample / Failure mode

24.8.1 误区 1：把 global class 当成 Java 文件

ABAP global class 不是单纯文件。它是 SAP repository object/class pool，有 package、activation、transport、where-used、runtime load 等项目属性。

24.8.2 误区 2：public section 过大

PUBLIC SECTION 不是“先全 public，之后再改”。public method 和 public attribute 会被外部依赖，改动成本高。初学阶段应默认 private，必要时再公开。

24.8.3 误区 3：类名是 OO，代码仍是过程式

一个 run_everything 方法 500 行，不会因为放进 global class 就变成好设计。global class 的价值在于清楚的职责、边界、方法和测试入口。

24.8.4 误区 4：忽略 package/transport

把 class 建在 $TMP 或错误 package，后续无法正常移送或被项目对象复用。练习可以用 local package，项目代码必须按团队包结构创建。

24.8.5 误区 5：local class 和 global class 乱用

只在一个 demo 或 test 中使用的 helper，可以是 local class；被多个对象调用的业务能力，应考虑 global class。判断标准是复用范围、transport、where-used、测试和可维护性，而不是“哪个写起来快”。

24.9 思考问题

  • 这个 class 是 repository object 吗？属于哪个 package？
  • 它的 PUBLIC SECTION 是否真的是外部 contract，还是暴露了内部状态？
  • 这个 class 的职责能用一句话说明吗？
  • 方法是否太大，是否应该拆 private helper？
  • 这个逻辑应该是 global class，还是 local class 就够？
  • 创建/修改 class 时是否需要 transport request？
  • 在 ABAP Cloud / Clean Core 语境下，它是否依赖 unreleased API 或直接访问不该访问的对象？
  • 在日本项目 review 中，应该讨论 責務分割、公開メソッド、移送依頼，还是 有効化？
