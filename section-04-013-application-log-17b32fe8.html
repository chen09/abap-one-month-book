13. Application Log 应用日志

  • Chapter 04
  • Section 13
  • self-contained study note

13.1 一句话结论

Application Log 应用日志 是 SAP 应用程序为业务处理过程留下的可查询、可追踪、可交付的处理记录。它回答的问题不是“程序有没有写过日志”，而是“某一次接口、批处理、后台 job 或业务动作到底处理了什么、在哪一步失败、失败后能不能重跑、运维和开发如何定位”。

在 SAP 项目中，很多问题不是在前台按钮点击的一瞬间解决的，而是在后台作业、接口调用、批量导入、异步处理和上线后故障调查中解决的。Application Log 的价值就在这里：它让一次业务处理从“口头说失败了”变成一组可以被用户、开发、测试、Basis 和运维共同确认的证据。

它和 Security Audit Log 安全审计日志、Read Access Logging 读取访问日志 不同。Application Log 关注业务处理本身，例如一批数据导入了多少条、哪几条失败、失败原因是什么、是否可重跑；Security Audit Log 关注安全事件；Read Access Logging 关注敏感数据读取。三者都叫 log，但项目责任和使用场景完全不同。

13.2 到底学什么

第一，要学 Application Log 的边界。它不是 WRITE 输出，也不是临时 debug 文字。一个合格的应用日志应该服务于生产调查：能按对象、子对象、外部请求号、业务键、时间、用户、处理阶段和消息严重度定位一次处理。

第二，要学一条日志应包含什么。最低限度应有 log object / subobject、external ID 或 request ID、业务键、处理阶段、消息严重度、可理解的消息文本、处理数量、失败行号或失败对象、是否可重试、下一步建议。对接口和批处理来说，request id、batch id、application log id 尤其关键。

第三，要学日志和事务边界的关系。日志记录不能制造假的成功感。比如数据还没有提交，却已经写出“保存成功”；或者事务回滚了，但日志说“处理完成”。Application Log 的设计必须和 Transaction boundary 事务边界、RAP save sequence、后台 job 提交点一起考虑。

第四，要学它在项目交付中的用途。Application Log 不是装饰功能，而是测试证据、障害調査、运维手顺、客户说明和生产监控的一部分。日本项目现场常会问：処理ログはどこで確認できますか、再実行できますか、失敗明細を特定できますか。

第五，要学不要把敏感信息写进日志。Application Log 可以记录业务键、行号、错误分类和必要上下文，但不应记录完整密码、token、个人敏感信息、完整 payload 或未经脱敏的数据。敏感访问证据应根据安全设计转向 Read Access Logging 或安全审计流程。

13.3 为什么重要

没有 Application Log 的系统，通常只能靠用户截图、后台 job 状态或开发者 debug 来调查问题。这在开发环境还能勉强凑合，在生产系统里很快会失控。尤其是批处理和接口场景，一次失败可能影响几十、几百、几千条业务数据；如果没有日志，就无法判断哪些成功、哪些失败、哪些可以重跑、哪些必须人工修正。

对项目交付来说，Application Log 是“可运维”的证明。它帮助测试人员验证异常路径，帮助运维人员按 runbook 定位问题，帮助开发者缩小修正范围，帮助客户确认失败原因不是凭猜测得出的。一个好的日志设计，会把故障调查从“请开发看一下”推进到“根据 log id 和业务键已经定位到第 27 行公司代码无权限”。

对 Java/Python 背景读者来说，可以把它类比为 structured application logging，但 SAP 场景更强调业务对象、后台作业、接口证据、可查询性、客户运维和审计协作。它不是控制台输出，而是 SAP 项目交付物的一部分。

13.4 Key mechanism / 核心机制

Application Log 的核心可以拆成两层：日志头和消息明细。

日志头描述“一次处理”。例如 log object、subobject、external ID、request id、batch id、执行用户、技术用户、程序或服务名、开始时间、结束时间、总数、成功数、失败数。它让人能快速找到“是哪一次处理”。

消息明细描述“处理过程中发生了什么”。例如 success / information / warning / error / abort 等严重度，具体业务键、行号、字段、错误原因和建议动作。它让人能知道“为什么失败”和“下一步该做什么”。

log object 和 subobject 的设计很重要。对象太粗，例如所有 Z 开发都用一个对象，运维查询会混乱；对象太细，例如每个小方法一个对象，也会难以维护。比较实用的做法是：按业务能力或接口域划分 object，按处理阶段划分 subobject，例如 UPLOAD、VALIDATE、EXECUTE、RETRY。

消息严重度也要统一。S 表示成功步骤，I 表示上下文信息，W 表示可继续但需关注，E 表示某个对象或步骤失败，A 或 abort 类消息表示整体处理无法继续。通常可以把 E 理解成“某条对象或某个步骤失败”，把 A 理解成“整体运行必须终止”，这样监控和运维才能判断优先级。项目中最糟糕的做法，是所有问题都写成同一种严重度，导致监控和运维无法判断优先级。

Application Log 应与 Error message design 错误消息设计 配合。错误消息要让人知道业务规则、失败对象和下一步动作；Application Log 则负责把这些消息组织成一次可追踪的处理记录。对于 OData API 或后台 job，response 或 job log 可以只返回 log id / request id，详细明细保存在 Application Log 中。

在经典 ABAP / SAP GUI 项目里，Application Log 常用 SLG1 查询，开发和运维会用 log object、subobject、external ID、时间、用户等条件定位一次处理。技术实现上常见 BAL / Business Application Log 相关函数或类；在 ABAP Cloud / Clean Core 语境下，要优先确认目标 release 中可用的 released API，而不是直接沿用老系统里的未释放对象。

13.5 日本项目现场说法

常见日语：

  • アプリケーションログ
  • 業務ログ
  • 処理ログ
  • バッチログ
  • ログ番号
  • 外部ID
  • リクエストID
  • 再実行
  • 障害調査
  • 証跡

设计说明时可以这样说：

本機能では、バッチ処理単位でアプリケーションログを出力し、
処理件数、成功件数、警告件数、エラー明細を確認できるようにします。

接口证据说明可以这样说：

外部システムからのリクエストIDをログの外部IDとして保持し、
障害調査時にAPIレスポンスとSAP側の処理ログを突合できるようにします。

障害调查时可以这样问：

該当処理のアプリケーションログ番号は分かりますか。
失敗した明細行とエラーメッセージを確認できますか。

13.7 Example / 学习例子

假设 AC0567C batch upload API 收到 100 行数据。系统先校验，再执行保存，最后返回处理结果。没有 Application Log 时，接口可能只返回 failed，用户不知道哪一行错了；运维不知道是否可以重跑；开发不知道问题发生在哪个阶段。

较好的 Application Log 可以这样设计：

Log object: ZAC0567C_UPLOAD
Subobject: EXECUTE
External ID: REQ-20260616-00042
Batch ID: BATCH-4711

S  Received 100 rows for execution.
I  Validation completed: success 96, warning 1, error 3.
W  Row 15 / Key A10015: duplicate business key; skipped according to rule.
E  Row 27 / Key A10027: company code 1000 is not authorized for this user.
E  Row 41 / Key A10041: required field PostingDate is missing.
I  No business update was committed. Correct errors and retry with the same request id.

如果业务允许部分成功，日志必须明确说明部分提交和重试规则：

W  Partial success mode is active.
S  96 rows were committed.
E  3 rows failed and can be corrected in a new retry file.
I  Retry must not resend already committed rows unless the same idempotency key is used.

这个例子说明：Application Log 不只是“报错文字”。它必须和事务策略、幂等策略、API response、后台 job 和运维手顺一起设计。

13.8 Counterexample / Failure mode

常见失败模式如下。

第一，只写泛化错误，例如 System error、Failed、Exception occurred。这样的日志对用户、测试和运维几乎没有帮助。

第二，把敏感数据写进日志。完整 payload、token、密码、个人敏感信息、银行账号等不应直接进入 Application Log。

第三，日志和实际事务结果不一致。数据回滚了但日志写“成功”，或部分提交了但日志只写“失败”，都会破坏信任。

第四，没有 correlation id。API response、job log、Application Log 和后端处理无法关联时，生产调查会依赖人工猜测。

第五，把 Application Log 当成万能工具。它不能替代 Security Audit Log、Read Access Logging、ST22、SAT/ST05、job log 或系统监控。

第六，没有保留策略。日志永久不清理会带来容量和隐私风险；清理太快又会影响运维和审计。项目应定义保留期限和归档/删除策略。

第七，消息没有重跑语义。日志只说失败，不说是否可重跑、是否部分成功、是否需要人工修正，运维无法行动。
