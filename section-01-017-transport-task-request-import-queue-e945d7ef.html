Chapter 01 - SAP 和 ABAP 基础

17. Transport task / request / import queue 治理

  • Chapter 01
  • Section 17
  • self-contained study note

1) 一句话结论

Transport task / request / import queue 治理，是把 Transport request 传输请求 从“一个变更容器”推进到“谁 release、何时 export、如何进入 QAS/PRD import queue、哪些 approval 和 log 证明可上线”的项目管理链路；本节要学的是：task release、request release、QAS import、QA approval、PRD import 是不同关口。

2) 到底学什么

前面已经知道 transport request 不是 Git commit，也不是 deployment。现在要继续追问：一个 request 里为什么有 task？开发者 release task 后发生了什么？request owner release request 后是否已经进 QAS？QAS import 后为什么 PRD import queue 里还显示 approval / inactive？STMS 里的 RC 4 到底能不能签收？这些问题不只是 Basis 的事，ABAP 开发者至少要能读懂状态、说清责任边界。

最基本的层级是 request / task。transport request 是上层容器，task 是开发者记录自己对象变更的下层单位。团队开发时，系统会给参与者各自的 task；开发者完成自己的对象后，先 release task。所有相关 task release 后，request owner / project lead 才能 release transport request。这个顺序不能反过来：未 release 的 task 会阻止 request 正式释放。

接下来要分清三个动作。task release 是开发者动作，表示自己的变更记录完成；request release 是负责人动作，通常触发 export，把 transportable request 从 DEV 导出到 transport directory / transport buffer；import 是把 request 导入 QAS/PRD 等目标系统的动作，由 TMS/cTMS、transport admin、项目流程或相应 cloud transport 工具控制。看到 Released，不代表 QAS 已经变了，更不代表 PRD 已经上线。

import queue 是目标系统侧的关键证据。按照 SAP transport route，DEV release 后，request 会进入后续系统的 import buffer / import queue。import queue 不只是“待导入列表”，还保留 import sequence。SAP Learning 明确说明 import queue 包含已释放的 transport requests，并记住它们 release 的顺序；TMS import all 的目的之一，就是避免缺少依赖 request 或旧对象版本覆盖新对象版本。开发者可以读 import queue 状态，但不应该擅自 reorder 或手动 import。

QAS 不是装饰层。标准流程是：request import 到 QAS 后，团队做测试；若启用 QA approval procedure，request 进入后续 delivery system / PRD import queue 时可能先是 inactive / approval pending，只有 approval steps 全部完成后才变成可导入状态。若客户系统没有配置 STMS_QA，也不代表可以省略测试签收；流程上仍应有 test sign-off、approval evidence 和本番リリース计划。

本节只讲开发者必须懂的 governance awareness：task / request / release / import queue / return code / QA approval / log / Japanese reporting。STMS 路由配置、transport domain controller、tp command、client import、ChaRM release cycle、Cloud ALM deployment setup 等，是 Basis / project governance 深水区，后面或附录再讲。

3) 为什么重要

这个小节的核心价值，是让初学者不要把 SAP 移送看成“我点了 release，就等于上线”。真实日本 SAP 项目里，一次变更从开发到本番反映，至少会经过开发者 task release、request owner release、QAS import、QAS test、approval、PRD import、return code/log check、post-import verification。每个关口都可能卡住，也都有不同责任人。

它也解释了项目沟通中为什么总在问 移送依頼番号。别人问这个编号，不只是为了查对象列表，而是为了追踪：task 是否 release？request 是否 release？是否进入 QAS import queue？QAS import RC 是多少？是否已测试？是否 QA承認済み？PRD import queue 中状态是 active 还是 承認待ち？本番リリース窗口是哪天？本番反映是否完成？

对 Java/Python 背景学习者来说，这类似 CI/CD pipeline 的 stage，但不要直接等同。GitHub Actions 里一个 job status 可能就是当前流水线状态；SAP transport 里有 source system、target system、transport buffer、import queue、return code、QA approval、client、Basis 操作窗口、change request 审批。开发者不一定执行 import，但必须能读懂 evidence，不能只说“我这边开发完了”。

错误理解会带来高风险事故。比如开发者为了让自己的 request 快点进 PRD，要求 Basis 调整 import queue 顺序，可能导致旧版本覆盖新版本；看到 RC 4 就以为“只是 warning”直接签收，可能漏掉详细 log 中的 object-level error；看到 request 已在 PRD queue 里就催 import，却不知道 QA approval 尚未完成。上线周末最怕的，往往不是不会写 ABAP，而是看不懂移送状态。

4) Key mechanism / 核心机制

第一层机制是 task / request ownership。task 通常归开发者，request 通常归 request owner / project lead。开发者 release task 前，要确认自己的对象已经激活、where-used / impact check 已完成、ATC / unit test / 手动测试证据齐全、request description 有意义。request owner release request 前，要确认所有相关 task 都已 release，object list 属于同一个 testable unit。

第二层机制是 release sequence。release task 不会把对象送到 QAS；release request 才把 transportable request 导出并放入后续系统 import queue。request release 也不是 import；import 是目标系统侧动作。把这三步混在一起，会造成沟通错位。正确说法是：タスクリリース済み、移送依頼リリース済み、QASインポート済み、本番インポート待ち、本番反映済み，每个状态都不同。

第三层机制是 import queue sequence。import queue 会保留 released requests 和 import sequence。不要把它想成可以任意拖动的个人优先级列表。SAP 文档强调要确保 request 以正确顺序导入；若多个 request 包含同一个对象，后导入的版本会成为目标系统 active version。擅自调整顺序可能让旧对象覆盖新对象，尤其在 PRD release weekend 是严重事故。SAP 建议尽量整体导入完整的 CTS project / 相关 request 组，避免漏掉依赖 request，这也是不要单独抽取或插队的原因之一。开发者应报告依赖和优先级，由 transport admin / project lead 按批准的 移送シーケンス 处理。

第四层机制是 return code / log。常见 awareness：RC 0 通常表示成功；RC 4 是 warning，但不能自动签收，必须看 detailed log；RC 8 表示 error；RC 12 或更高通常是严重错误或中止。request-level RC 4 里也可能有 object-level 错误或未导入对象，必须打开 transport log / import history / detailed step log 判断。日本项目里 RC4なので問題ありません 这种说法不够严谨，至少要补一句 詳細ログ確認済み。

第五层机制是 QA approval。启用 QA approval procedure 时，QAS import 完成后，request 可能进入 PRD import queue 但处于 inactive / 承認待ち 状态；只有配置的 approval steps 完成后才变成 active / importable。看到 PRD queue 中有 request，不等于可以立刻 import。若没有系统级 QA approval，也要用项目流程保证测试完成、approval evidence 可追踪。

第六层机制是 developer role boundary。开发者要会读 Transport Organizer、object list、task/request status、QAS/PRD queue status、return code、import log、approval status，并能把 blocker 写清楚。开发者通常不应配置 STMS、调整 transport route、手动 reorder queue、执行 PRD import、使用 tp command，除非项目明确授权。边界清楚，才不会把技术问题变成治理事故。

第七层机制是 cloud caveat。SAP S/4HANA Cloud Public Edition、SAP BTP ABAP Environment、ABAP Cloud、cTMS、Cloud ALM、Manage Software Components、software collections 的 UI 和机制可能不同。BTP ABAP Environment 里可能通过 Manage Software Components / Git-backed lifecycle 推进，而不是 classic STMS import queue；Public Edition 也可能使用 ATO / Export Software Collection / Import Collection 等 Fiori apps。Day 02 只要记住：工具不同，但 task/request/import/approval evidence 的治理思维仍然要问清。

5) 日本项目现场说法

  • 常见日语：タスク、タスクリリース、移送依頼、移送依頼番号、移送依頼リリース、依頼リリース、インポートキュー、移送ログ、インポートログ、戻りコード、RC、QA承認、品質承認、承認待ち、品質承認済み、移送シーケンス、移送前確認、リリース前確認、本番リリース、本番反映。
  • 用语区别：本番リリース 是计划中的生产导入事件，常出现在 release 计划和作业窗口说明；本番反映 是 import 完成后确认已经反映到生产的状态，不要把“计划 release”误读成“已经反映完毕”。
  • 用语区别：品質承認 / 品質承認済み 常用于移送承认证据和测试完成确认；QA承認 更常用于 STMS_QA 或系统级 approval 讨论，承認待ち 是状态表达。
  • 说明句：タスクリリース、移送依頼リリース、QASインポート、本番インポートはそれぞれ別の状態として確認してください。
  • 说明句：RC4 の場合でも、詳細ログを確認したうえで品質承認可否を判断します。
  • 使用场景：移送前确认、QAS 导入后测试、QA approval、release weekend 本番リリース、import error triage、变更审批和上线报告。

7) Example / 学习例子

  • 在 ADT Transport Organizer 或 SE09/SE10 中找一个 request，观察它下面的 tasks、owner、status、object list。说明谁 release task，谁 release request。
  • 读一个 request 的生命周期：modifiable -> task released -> request released -> QAS import queue -> QAS import history/log -> QA approval -> PRD import queue -> PRD import log。
  • 看一次 STMS import history 或项目移送记录，记录 RC 值。若是 RC 4，写出详细 log 里具体 warning 内容，而不是只写“warning”。
  • 用日语写一段汇报：移送依頼番号、QASインポート状況、戻りコード、品質承認状況、本番リリース予定。
  • Week 3/4 后，把一个 RAP/OData 修正映射到 transport request、service binding activation、QAS test evidence、API test evidence、PRD import evidence。

8) Counterexample / Failure mode

  • 误区一：开发者 release task 后就告诉团队“已经移送”。task release 只是开发者动作，不是 request release，也不是 import。
  • 误区二：request release 后以为 QAS/PRD 已经改变。request release 通常只是 export / 进入 import queue，目标系统 import 是单独动作。
  • 误区三：看到 RC 4 就直接签收。RC 4 也必须查详细 log，确认是否有 object-level error、生成 warning 或后续测试风险。
  • 误区四：为了赶进度随意调整 import queue 顺序。可能导致依赖缺失或旧版本覆盖新版本。
  • 误区五：看到 PRD import queue 里有 request，就以为可以立刻 import。若 QA approval 未完成，状态可能是 inactive / 承認待ち。
  • 误区六：把 classical STMS 经验直接套到 Public Edition / BTP ABAP Environment。cloud 场景工具不同，必须按客户系统和项目手顺确认。

9) 思考问题

  • タスクリリース済み、移送依頼リリース済み、QASインポート済み、本番反映済み 分别意味着什么？
  • 为什么 transport request 应该表达一个 testable unit，而不是每个对象一个 request？
  • import queue 的顺序为什么不能随便改？
  • RC 4 和 RC 8 的项目处理方式有什么不同？为什么 RC 4 也要看 detailed log？
  • 如果 PRD import queue 里 request 是 承認待ち，应该找开发者、Basis、QA approver 还是 project lead？
  • 在 cloud / BTP 环境下，如果看不到 STMS import queue，应该问哪些替代工具和 evidence？
