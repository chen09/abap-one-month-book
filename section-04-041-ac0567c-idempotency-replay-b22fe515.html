41. AC0567C idempotency replay

  • Chapter 04
  • Section 41
  • self-contained study note

41.1 一句话结论

AC0567C idempotency replay 不是泛泛讲“接口要防重复”，而是用本地 POC 证据复盘：同一个 AC0567C batch upload 请求被重复执行时，当前实现如何用 ImportId 派生业务 key、如何在写入前查 ZCOM_B_ACTT397、如何返回 CONFLICT，以及这个 POC 设计距离生产级幂等还差哪些锁、唯一约束、request log 和 application log。

一句话记忆：AC0567C 当前已经证明了“重复同一 ImportId 的真实执行不会再次插入同一组已存在行，而是返回 CONFLICT”；但它还不是完整的生产级 idempotency framework。它没有独立的 request header/status table，没有持久化 idempotency ledger，没有并发重复压测证据，也没有 application log 级别的全链路审计。

为什么要这么谨慎？因为幂等性很容易被写成一句漂亮口号：重复请求不会造成重复业务结果。真正项目里，必须追问三个问题：

  • “同一个请求”到底由什么 key 定义。
  • 重复命中时返回已有结果、返回 conflict，还是允许重试。
  • 并发、网络超时、失败后重跑、不同用户重发时，证据是否仍然成立。

AC0567C 的学习价值就在这里。它是一个真实 POC，不是教材里凭空画的理想架构。它已经有可引用的 ABAP class、RAP behavior pool、PowerShell 测试脚本和 test evidence；同时也清楚暴露出 POC 边界：当前重复检查是 row-key duplicate check，不是完整的 API-level idempotency store。读这一节时，要同时学会肯定已经证明的部分，也要能指出不能夸大成生产完成的部分。

41.2 到底学什么

第一，要学 idempotency key 的边界。AC0567C API 参数里有 ImportId，但当前实现并不是把完整 ImportId 原样保存成 request ledger。服务类 ZCL_RPA01_AC0567C_IMP_SVC 会从 ImportId 里寻找最后一段连续 14 位数字，把它解释成 YYYYMMDDHHMMSS，再派生为目标表 ZCOM_B_ACTT397 的 zzup_dat 和 zzup_tme。

第二，要学当前 POC 的重复判断字段。ExecuteImport 构造 ZCOM_B_ACTT397 行时，重复检查使用这些字段：

bukrs      = CompanyCode
kunnr      = CustomerCode
zzup_dat   = ImportId 派生日期
zzup_tme   = ImportId 派生时间
zzup_name  = sy-uname
zzgroup_no = Rows[] 中的 GroupNo
zzseq_no   = 当前行序号

也就是说，当前 AC0567C 的幂等证据更准确地说是“基于 ImportId 派生时间戳 + 当前 SAP 用户 + group/sequence 的目标表重复保护”。它不是“只要 ImportId 相同，任何 payload 变化都被 request log 精确识别”的完整设计。

第三，要学重复请求的当前响应契约。真实执行第一次通过时，测试证据显示 ExecuteImport Status = IMPORTED 且 InsertedRows = 2；同一个 ImportId 再执行时，状态为 CONFLICT，消息字段为 Idempotency，文本是 Duplicate ZCOM_B_ACTT397 key found for this ImportId/user/group/sequence。

第四，要学 AllowRealUpdate 和幂等测试的关系。当前测试脚本中，没有 -AllowRealUpdate 时，真实写入和 duplicate/idempotency 检查都会被跳过；只有设置 AllowRealUpdate = X，才会真正执行插入和重复检查。这个开关是 POC 安全门，不是幂等键，也不是授权。

第五，要学 RAP save sequence 边界。当前 POC 的服务类在 action path 中做 validation、authorization、duplicate check 和 staging；真正 INSERT zcom_b_actt397 FROM TABLE @lt_insert 在 behavior saver 的 save 方法中执行。这比直接在 action handler 里 INSERT 更接近 RAP 事务模型，但它也带来一个复盘重点：duplicate check 发生在 staging 前，最终插入发生在 saver，中间仍要考虑并发和 save failure。

第六，要学证据和缺口并列写。AC0567C 已有真实 insert、重复 CONFLICT、status lookup 证据；但 artifact quality gate 仍把 production lock/enqueue、persistent audit log、authorization-negative test、CSV/TSV parser hardening 列为未完成。一本面向项目交付的书，不能只写“通过”，也要写“没证明什么”。

41.3 为什么重要

上传类 API 最怕“调用方不知道刚才到底成功了没有”。外部系统发送 batch 后，如果网络超时，调用方可能按 retry policy 再发一次；用户在画面上看不到返回，可能再点一次；job 失败后可能重新跑；middleware 也可能因为 at-least-once delivery 重发消息。没有幂等设计时，这些重复动作可能变成重复创建、重复过账、重复日志、重复状态推进。

对 AC0567C 来说，业务对象是批量上传到 ZCOM_B_ACTT397。如果同一个上传批次重复插入，后续 AC0568、对账、账务调整或人工确认都会被污染。重复数据不是“多了一行”这么简单，它会让客户/公司/组号/序号维度上的业务事实变得不可信。

对 Java/Python 背景学习者来说，可以把 AC0567C 类比成一个 HTTP API 使用 Idempotency-Key 的场景，但 SAP 项目有自己的约束：ABAP 运行在 AS ABAP work process 中，保存受 SAP LUW 逻辑事务单元 和 RAP save sequence 影响，重复保护经常落在 DDIC table key、enqueue lock、application log、business object 状态和测试证据之间，不只是 web controller 里查一下 Redis。

在日本 SAP 项目里，这类问题通常会以更业务化的语言出现：二重登録防止、再送時の扱い、リトライ仕様、排他制御、証跡。客户不一定会问“idempotency key 有没有设计好”，但会问“同じファイルを再送した場合、二重登録されませんか”。本节就是训练读者把这个现场问题翻译成可检查的 SAP/ABAP 设计。

41.4 Key mechanism / 核心机制

AC0567C 当前实现可以按一条实际链路理解：

POST /PriorAllocationImportsC/execute
  -> RAP static action ExecuteImport
  -> ZCL_RPA01_AC0567C_IMP_SVC=>execute_import
  -> validate_batch
  -> AllowRealUpdate = X check
  -> AUTHORITY-CHECK F_BKPF_BUK / ACTVT = 01
  -> derive_upload_timestamp( ImportId )
  -> SELECT SINGLE zcom_b_actt397 by derived row key
  -> if found: Status = CONFLICT, MessageField = Idempotency
  -> if not found: stage rows into save buffer
  -> RAP saver save
  -> INSERT zcom_b_actt397 FROM TABLE @lt_insert
41.4.1 ImportId 到日期时间 key 的派生

derive_upload_timestamp 的实现先把 ev_date / ev_time 默认设成当前 sy-datum / sy-uzeit，然后扫描 ImportId 中的数字串。如果找到长度至少 14 的连续数字串，就取最后 14 位作为 timestamp。例子：

ImportId = POC-AC0567C-20260612115720
timestamp token = 20260612115720
zzup_dat = 20260612
zzup_tme = 115720

这个实现修正了一个很重要的 POC 问题：AC0567C 本身也包含数字，如果简单“收集全部数字”，会把业务代号里的 0567 混进时间戳。最终版使用最后一段连续 14 位 timestamp token，测试证据也记录了 POC-AC0567C-20260612115720 映射到 20260612 / 115720。

41.4.2 重复检查不是完整 request ledger

当前 duplicate check 是对目标表 ZCOM_B_ACTT397 做 SELECT SINGLE mandt：

SELECT SINGLE mandt FROM zcom_b_actt397
  WHERE bukrs = @ls_target-bukrs
    AND kunnr = @ls_target-kunnr
    AND zzup_dat = @ls_target-zzup_dat
    AND zzup_tme = @ls_target-zzup_tme
    AND zzup_name = @ls_target-zzup_name
    AND zzgroup_no = @ls_target-zzgroup_no
    AND zzseq_no = @ls_target-zzseq_no
  INTO @DATA(lv_duplicate_mandt).

如果查到已有行，服务返回：

Status = CONFLICT
MessageField = Idempotency
MessageText = Duplicate ZCOM_B_ACTT397 key found for this ImportId/user/group/sequence

这说明当前幂等保护的核心是“目标业务表中是否已经存在同一派生 row key”。它能证明重复执行同一测试 payload 不会再插入同一组行；但它不能证明系统有一张完整 request log，可以回答“这个 request 第一次何时收到、payload hash 是什么、第一次返回给调用方的 response 是什么、是否可以把既有 result 原样返回”。

41.4.3 真实测试证据

04_tests/TEST_EVIDENCE.md 记录了两类证据。

无写入测试：

ImportId = POC-AC0567C-20260612134648
8/8 checks passed
Real execute skipped unless -AllowRealUpdate is set
Duplicate/idempotency skipped unless -AllowRealUpdate is set

真实写入测试：

ImportId = POC-AC0567C-20260612115720
ExecuteImport Status = IMPORTED
InsertedRows = 2
Repeat same ImportId Status = CONFLICT
GetStatus Status = IMPORTED
GetStatus InsertedRows = 2

这个证据很有价值，因为它不是“设计上应该如此”，而是脚本确实调用了 /PriorAllocationImportsC/execute 两次，并记录了第一次插入和第二次 CONFLICT。同时也要注意：这组真实 insert / CONFLICT 证据来自 TEST_EVIDENCE.md 中标注为 Real Insert Test Already Proven 的早期实写/手动验证运行；最新 no-write 8/8 回归测试是安全回归，明确跳过 real execute 和 duplicate/idempotency。它证明的是顺序重复测试，不是并发重复测试。

41.4.4 GetStatus 的边界

GetStatus 会根据 CompanyCode、CustomerCode、ImportId 派生日期时间和 sy-uname 去 ZCOM_B_ACTT397 计数。查到行时返回 IMPORTED 和 InsertedRows；查不到时返回 NOT_FOUND 或带信息消息。

这对 POC 足够：测试可以证明“刚才那个 ImportId 派生的 key 已经有两行”。但生产级接口通常还需要独立状态模型，例如 RECEIVED、VALIDATED、PROCESSING、IMPORTED、FAILED_RETRYABLE、FAILED_FINAL。当前 AC0567C 没有这种 request header/status table，所以不要把 GetStatus 误读成完整工作流状态机。

41.4.5 并发风险

当前 duplicate check 是“先 SELECT，再 stage，最后 saver INSERT”。如果两个完全相同请求几乎同时到达，理论上可能都先 SELECT 到“没有记录”，然后都进入 staging。最终是否重复写入，要看 ZCOM_B_ACTT397 的真实主键/唯一约束、RAP lock/enqueue、数据库冲突处理以及 saver 错误处理。当前证据没有证明并发重复安全。

因此，本节对 AC0567C 的准确评价是：

  • 已证明：顺序重复同一真实 ImportId 会返回 CONFLICT，不会再次插入同一组已存在记录。
  • 未证明：并发重复、跨用户重复、失败后重试、payload 改写后重试、完整 request-response replay、application log 全链路审计。
41.4.6 生产级补强方向

如果要把 AC0567C 从 POC 推到生产设计，至少要补四类对象：

41.5 日本项目现场说法

常见日语包括：冪等性、重複リクエスト、二重登録防止、再送、リトライ、リクエストID、一意キー、ステータス管理、再処理、排他制御、証跡。

描述当前 AC0567C POC 时，可以这样说：

  • AC0567C POC では、同一 ImportId から派生される日付・時刻、会社コード、得意先、ユーザ、グループ番号、行番号を用いて、ZCOM_B_ACTT397 の重複登録を検知します。
  • 同一 ImportId で再実行した場合、初回は IMPORTED、二回目は CONFLICT を返すことをテスト証跡で確認済みです。
  • ただし、現時点では request header table や application log による完全な冪等性管理までは実装していません。

review 时可以这样问：

  • 重複判定のキーは「ImportId」そのものですか、それとも ImportId から派生した登録日付・時刻と業務キーですか。
  • 並行リクエストが来た場合、DB一意制約または排他制御で二重登録を防げますか。
  • 再送時に CONFLICT を返す設計でよいですか。それとも既存の処理結果を返すべきですか。
  • エラー後の再処理可否は、ステータス管理または運用手順書に明記されていますか。

交付文档中可以使用更稳妥的表达：

  • 本 POC は、同一 ImportId の順次再実行に対して二重登録を防止することを確認済みです。
  • 本番化にあたっては、同時再送時の排他制御、request ID の永続管理、application log による証跡管理を追加検討します。
  • AllowRealUpdate は実更新を許可するセーフティスイッチであり、冪等性キーまたは権限制御ではありません。

41.7 Example / 学习例子

先看 AC0567C 已验证的真实顺序重复场景：

First call:
  POST /PriorAllocationImportsC/execute
  ImportId = POC-AC0567C-20260612115720
  AllowRealUpdate = X
  Rows = 2

Result:
  Status = IMPORTED
  InsertedRows = 2

Second call:
  same endpoint
  same ImportId
  same payload
  AllowRealUpdate = X

Result:
  Status = CONFLICT
  MessageField = Idempotency

这说明：在同一用户、同一 company/customer、同一 ImportId 派生日期时间、同一 group/sequence 的顺序重复请求下，当前 POC 不会再次把同一批目标行插入 ZCOM_B_ACTT397。

再看一个生产化设计表。当前 POC 已覆盖第一行和第三行的一部分，其他行需要补强：

场景当前 POC 证据生产建议
首次真实上传IMPORTED, InsertedRows = 2保留 request log 和 response trace
无写入验证no-write 8/8 passed作为 CI/smoke test 保留
成功后顺序重试same ImportId returns CONFLICT明确是否返回 409 conflict 或既有 result
成功后并发重试未证明增加 enqueue/unique constraint/并发测试
失败后重试未证明定义 retryable / final failure 状态
授权失败后重试未证明区分 authorization denied 与 idempotency conflict
payload 改写但 ImportId 相同未证明用 payload hash 或 request ledger 判断非法重放
跨用户同 ImportId当前 key 包含 sy-uname判断是否应按 technical user / business user 归属

最后看一个缺陷票式例子：

Incident:
  外部系统因 HTTP timeout 重发同一上传请求。

Observed:
  SAP 返回 CONFLICT，没有新增第二组 ZCOM_B_ACTT397 行。

Evidence:
  ImportId = POC-AC0567C-20260612115720
  first ExecuteImport = IMPORTED / InsertedRows 2
  repeat ExecuteImport = CONFLICT
  GetStatus = IMPORTED / InsertedRows 2

Remaining question:
  这是顺序重试证据；并发重试还需要锁或唯一约束证据。

这样的例子能帮助初学者把“幂等”从抽象名词落到可审计证据：请求参数、返回状态、目标表行数、重复消息、测试脚本、未覆盖场景。

41.8 Counterexample / Failure mode

常见失败模式如下。

第一，把 ImportId 当成已经完整持久化的 idempotency ledger。当前 POC 只是从 ImportId 派生 zzup_dat/zzup_tme 并检查目标表行 key，没有保存完整 request / payload / response 历史。

第二，把 AllowRealUpdate 当成幂等控制。它只是实更新安全开关；不开 AllowRealUpdate = X 时，真实 insert 和 duplicate test 都不会执行。

第三，只看 CONFLICT，不看 key 组成。如果重复检查 key 包含 sy-uname，那么 technical user、business user、代理执行或 batch user 的设计都会影响重复判断范围。

第四，只证明顺序重复，不证明并发重复。两个同时到达的请求可能都在 INSERT 前 SELECT 到“未存在”。生产要补锁、唯一约束或冲突处理。

第五，把 GetStatus 当成完整状态机。当前 GetStatus 是按 ImportId-derived key 计数目标表行，不是 request workflow header。

第六，重复请求返回技术异常。重复是可预期业务场景，应明确是 CONFLICT、existing result、processing 还是 retry-after。

第七，不区分授权失败、业务校验失败和幂等冲突。F_BKPF_BUK 授权失败应是 authorization denial；金额格式错误应是 validation error；已有目标 key 才是 idempotency conflict。

第八，没有 application log。测试脚本能证明 POC，但生产运维需要 BAL / cl_bali_log 或等价 request log 来追踪首次处理、重复命中、失败原因和操作用户。

第九，payload 改了但 request id 相同。当前 row-key duplicate check 可能只发现已经存在的 group/sequence；如果调用方用同一 ImportId 发送不同 payload，生产设计应决定是拒绝非法重放、返回原结果，还是允许新批次。

第十，只在设计书写“防止重复”，没有把 BDEF、service class、saver、test script、test result 和 transport evidence 串起来。幂等设计必须能被 review、测试和运维证明。

41.9 思考问题

  • AC0567C 当前重复检查的 key 是否等同于完整 idempotency key？差异在哪里？
  • ImportId = POC-AC0567C-20260612115720 为什么能派生出 20260612 / 115720？
  • 如果同一 ImportId、同一用户、同一 group/sequence 顺序重试，当前 POC 返回什么？
  • 如果两个同样请求并发到达，当前证据能否证明不会重复插入？还缺什么？
  • 生产设计应选择“重复返回既有成功结果”还是“重复返回 CONFLICT”？这会影响调用方 retry 逻辑吗？
  • GetStatus 当前证明了什么，又没有证明什么？
