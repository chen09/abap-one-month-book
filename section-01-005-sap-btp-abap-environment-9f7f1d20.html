Chapter 01 - SAP 和 ABAP 基础

5. SAP BTP ABAP environment 云开发环境

  • Chapter 01
  • Section 05
  • self-contained study note

本节结论

SAP BTP, ABAP environment 是 SAP BTP 上用于 ABAP Cloud 开发的云端 ABAP 环境，常用于 side-by-side extension 和新云应用开发；它不是 SAP S/4HANA 本体，也不是传统 S/4HANA DEV 系统搬到云上。

正文

学到这里，前四节已经把 SAP S/4HANA、AS ABAP、ABAP Platform 和用户界面/开发模型的边界分开了。本节再加一个位置：SAP BTP。SAP BTP 是 SAP 的业务技术平台，里面有多种运行和开发选择，例如 Cloud Foundry、Kyma、Integration Suite、Build、以及本节的 ABAP environment。它不是 ERP 产品本身，而是承载扩展、集成、自动化、数据和应用开发的云平台。

SAP BTP, ABAP environment 可以理解为“在 SAP BTP 上运行的、受 SAP 管理的 ABAP 云开发/运行环境”。SAP Learning 对它的定位是：在 SAP BTP 的 Cloud Foundry environment 中，可以为 ABAP development 创建 ABAP environment；它可以为 ABAP-based products（例如 SAP S/4HANA Cloud）创建扩展，也可以开发新的 cloud applications，还可以把部分既有 ABAP custom code 或 extension 转向云端。

它和传统 ABAP 系统的最大差异，不在于“写不写 ABAP”，而在于“写哪种 ABAP、运行在哪里、如何连接核心系统”。在 BTP ABAP environment 中，开发者主要使用 ABAP Cloud、ABAP for Cloud Development 语言版本、ADT、CDS、RAP、service binding、released APIs、software components 和云端生命周期工具。它不是让开发者随意使用 classic ABAP、SAP GUI dynpro、未释放 API、直接改标准表或直接读任意 S/4HANA 表的地方。

本节还要区分两个容易混的场景：SAP BTP, ABAP environment 和 S/4HANA Cloud tenant 内的 embedded ABAP Cloud developer extensibility。两者都和 ABAP Cloud 有关，但代码运行位置和架构角色不同。前者是在 SAP BTP 上单独 provision 的 PaaS 服务，适合 side-by-side；后者是在 S/4HANA Cloud 租户上下文中做开发者扩展。Day 01 不要求记住所有产品细节，只要先会问：“这段 ABAP 代码到底跑在 S/4HANA 核心/租户上下文里，还是跑在 BTP 上？”

为什么重要

SAP 项目进入云时代后，扩展方式的核心问题变成：怎样满足客户差异化需求，同时不破坏标准核心。过去很多 on-premise 项目习惯直接在核心系统里做增强、表、报表、接口和批处理。到了 SAP S/4HANA Cloud、Clean Core 和持续升级的场景里，直接改核心对象会带来升级、测试、合规和运维风险。BTP ABAP environment 的价值，就是给 ABAP 团队一个“离开核心系统，但仍然使用 ABAP/RAP/CDS 能力”的扩展位置。

对 Java/Python 背景的学习者，可以把它粗略类比成“云平台上的应用运行环境”，但要马上补上 SAP 差异。普通云应用可能直接连数据库、自己定义 API、自己控制部署流水线；BTP ABAP environment 面向 SAP 生态，强调 ABAP Cloud、released APIs、软件组件、SAP 身份/连接服务、S/4HANA 标准 API 和 Clean Core 约束。它不是自由云主机，也不是可以安装任意传统 ABAP 事务的服务器。

对项目交付来说，BTP ABAP environment 会影响架构判断。一个需求如果要保存独立扩展数据、组合多个系统、通过 released API 调用 S/4HANA、以松耦合方式扩展标准流程，就可能适合 side-by-side。一个需求如果必须和 S/4HANA 标准事务在同一个 LUW（逻辑工作单元，一次性提交的事务边界）中强一致执行，或者必须直接使用未释放的内部表/函数，则往往不适合简单放到 BTP side-by-side，需要回到 S/4HANA extensibility model、released API 和 Clean Core 规则重新判断。

核心机制

第一层是 runtime location。BTP ABAP environment 的代码运行在 SAP BTP 上，不运行在 SAP S/4HANA core 里。它可以通过 released APIs、OData/SOAP services、events、destinations、communication arrangements 等方式和 S/4HANA 通信，但这是一种通过受控接口协作的关系，不是“云端直接打开 S/4HANA 后台表随便改”。

第二层是 programming model。BTP ABAP environment 使用 ABAP Cloud，并通过 ABAP for Cloud Development 语言版本来限制对象可用范围。也就是说，开发者在 ADT 创建对象时要面对 cloud-ready 的语言版本、compiler / runtime / ATC checks、released objects、RAP、CDS、service definition / service binding 等约束。classic ABAP 里的很多历史习惯不能照搬，比如 SAP GUI dynpro、直接依赖未释放 SAP 对象、随意调用内部 function module、直接改标准表等。

第三层是 side-by-side extensibility。Side-by-side 的意思是扩展应用运行在 SAP S/4HANA 之外，但通过标准接口和核心业务系统交互。它适合做松耦合扩展：独立 UI、独立扩展数据、跨系统流程、补充审批、外部系统整合、报表/分析辅助、或不应该写进 S/4HANA 核心的应用逻辑。它不适合被理解成“把所有客户开发都搬到 BTP”，因为有些需求仍然需要 key user extensibility、developer extensibility、classic extensibility 或标准配置。

第四层是 lifecycle。传统 ABAP 项目常围绕 package、transport request、CTS/TMS 去理解交付。BTP ABAP environment 也有软件生命周期，但 Day 01 先抓几个入口：software component、Manage Software Components、Git-backed repository / commit、SAP Cloud Transport Management、CI/CD 或三方 pipeline。这里既不是纯 Java/Python 式的 Git branch/merge，也不是完全照搬 on-premise 的 TMS export/import；gCTS、abapGit 和更完整的 ABAP lifecycle 会在后面的 transport / DevOps 章节再细讲。

第五层是 runtime choice。SAP BTP 不只有 ABAP environment。Cloud Foundry 可以跑 Java、Node.js、CAP 等应用；Kyma 面向 Kubernetes 场景；Build / Integration Suite / Event Mesh 等服务也各有用途。选择 BTP ABAP environment 的理由通常是：团队有 ABAP/RAP/CDS 能力，需求和 SAP 业务语义贴得很近，希望用 ABAP Cloud 和 released SAP APIs 做生命周期稳定的扩展。它不是所有 BTP 扩展的默认答案。

学习例子

假设客户需要一个“客户发票投诉管理”小应用。这个应用要记录投诉原因、处理状态、负责人和备注，还要从 S/4HANA 读取发票/客户信息作为参考。直接改 S/4HANA 标准发票流程可能风险高，也未必符合 Clean Core。一个 side-by-side 方案是：在 SAP BTP, ABAP environment 中建立扩展应用，自己的扩展数据存在 BTP ABAP environment 的表里；需要发票信息时，通过 S/4HANA 暴露的 released OData service 或 API 读取，不直接改 S/4HANA 标准表。

从开发者视角看，这个例子会包含几类对象：在 ADT 中创建 software component / package / table / CDS / RAP business object / service binding；通过 destination 或 communication arrangement 连接 S/4HANA；通过 released API 读取核心数据；把 UI 或 OData service 暴露给前端或其他系统。Day 01 不要求做出这个应用，只要求知道它为什么是“BTP ABAP environment 里的 side-by-side 扩展”，而不是“在 S/4HANA DEV 里随便加一堆 Z 表和 Z 程序”。

最小练习：用 8-10 行中文解释 “S/4HANA core 中的开发” 和 “SAP BTP side-by-side extension” 的区别。进阶练习：找一个 SAP 官方页面，确认 SAP BTP, ABAP environment 可以用于 ABAP-based products 的扩展和新 cloud applications。

常见误区

  • 误区一：把 BTP ABAP environment 当成 S/4HANA 的 DEV 系统。修正：它运行在 SAP BTP 上，是 side-by-side / cloud application 的运行位置，不是 S/4HANA core 的开发客户端。
  • 误区二：以为它支持所有 classic ABAP。修正：BTP ABAP environment 使用 ABAP Cloud，只能使用允许的语言范围和 released objects。
  • 误区三：以为 side-by-side 可以直接读写任意 S/4HANA 表。修正：应通过 released APIs、services、events 和受控 connectivity 与核心系统交互。
  • 误区四：以为所有扩展都应该放到 BTP。修正：简单字段/逻辑可能适合 key user extensibility；需要贴近 S/4HANA 业务对象生命周期的场景可能适合 developer extensibility；必须同一 LUW（逻辑工作单元）的需求不能简单外置。
  • 误区五：把 BTP ABAP environment、S/4HANA Cloud tenant 内的 embedded ABAP Cloud developer extensibility、ABAP Cloud 混成一个词。修正：ABAP Cloud 是开发模型；BTP ABAP environment 是 BTP 上单独 provision 的运行/开发环境；embedded developer extensibility 是 S/4HANA Cloud 租户上下文中的扩展场景。
