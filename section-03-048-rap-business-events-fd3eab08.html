48. RAP business events

  • Chapter 03
  • Section 48
  • self-contained study note

48.1 一句话结论

RAP business events 是 RAP business object 在业务状态发生重要变化时，对外发布“发生了什么”的事件机制。它适合把“主业务保存成功后需要通知、集成、异步处理”的场景从 action handler 和 save logic 中解耦出来；但它不是 validation、determination、action、additional save 或接口调用的替代品，也不是初学 RAP 必须第一天掌握的核心语法。

一句话记忆：business event 是“保存成功后的业务事实通知”，不是“在 handler 里顺手做外部副作用”。

把这节放在 API testing workflow 接口测试流 和 RAP extensibility 之间，是因为事件是现代 SAP 扩展设计里非常容易被误用的一环。很多需求听起来像“保存后通知一下外部系统”，但如果直接把 HTTP 调用、邮件发送、外部写入塞进 handler 或 saver，就会把 SAP LUW、外部系统可用性、重试、超时和回滚混在一起。Business event 的意义，是把“主业务事实已经成立”与“下游如何反应”分开。

初学者要先记住一个边界：event 描述已经发生的事实，不负责决定这个事实能不能发生。能不能保存、能不能批准、字段是否合法、用户有没有权限，这些仍然属于 validation、precheck、authorization、action result 或 save sequence。Event 只适合在事实成立之后，通知下游系统、workflow、集成层或扩展逻辑去处理后续动作。

48.2 到底学什么

本节放在 Day 21 的末尾，是因为读者已经学过 action、save sequence、additional save、OData/API testing 和 direct side-effect 反模式。现在可以理解一个更成熟的设计问题：当 Travel 被批准、上传任务完成、采购申请状态改变时，外部系统、workflow、notification、analytics 或 extension 要知道这件事，应该怎么设计？

坏做法是在 action handler 或 save_modified 中直接发 HTTP、发邮件、写外部系统，然后希望它和 SAP 保存一起“自然一致”。好做法通常是把主业务保存和外部响应解耦：保存成功后产生 business event 或 outbox 记录，由事件消费者、integration flow、background job 或后续处理接走。

本节不追求让初学者立刻实现完整 event framework，而是建立边界意识：event 表达业务事实；event consumer 是异步/解耦的一方；event payload 要稳定；event 可能重复或延迟；消费者必须考虑幂等；不能用 event 来替代同步业务校验和事务内保存。

学习本节时，可以把 business event 拆成五个问题：

问题要回答什么
What happened?事件名称是否是业务事实，例如 TravelApproved
When is it true?它是否只在保存成功后才成立
Who consumes it?下游系统、workflow、BTP integration、后台任务还是审计流程
What is the contract?payload 是否稳定、最小、版本化
How is it operated?重复、延迟、失败、监控和重试如何处理

如果这些问题回答不出来，就不应该急着说“用 event”。Event-driven design 不是把同步代码换个地方，而是承认下游处理是独立生命周期，需要明确 contract 和运维机制。

48.3 为什么重要

Business events 重要，是因为它们解决的是项目里最容易出事故的一类需求：主业务保存成功后，还要通知别人。比如订单批准后通知外部仓储，上传批次处理完成后通知前端轮询，主数据变更后触发 BTP integration flow。若这些动作直接塞进 handler，保存失败、重复点击、超时、外部系统不可用都会让事务边界变得混乱。

事件思维让系统更可维护。主系统只负责说明“业务事实已发生”；消费者负责对事实做反应。这样可以降低耦合，也更符合 Clean Core 和扩展架构：核心 BO 不必知道所有下游系统细节，下游系统也不必通过轮询或直接读表理解状态变化。

但 event 也不是银弹。事件通常是异步的，可能有延迟、重复、失败重试和消费顺序问题。项目必须设计 event payload、event version、idempotency key、monitoring、dead-letter/retry、消费者权限和审计。对于必须同步阻断的规则，仍然要用 validation、authorization、precheck 或 action result，而不是等 event consumer 再失败。

对 Java/Python 背景学习者，可以类比 domain event / integration event / outbox pattern。RAP business event 的特殊点在于它和 RAP BO、BDEF、save sequence、SAP eventing/integration 能力以及 Clean Core 扩展策略相关。

对 SAP 项目交付来说，business events 常常是 Clean Core 的关键工具。核心系统不应该知道每个外部系统的调用细节；它只发布“订单已批准”“上传已完成”“状态已变更”这样的业务事实。下游通过 SAP eventing、integration flow、side-by-side service、background job 或订阅机制处理自己的事情。这样升级标准 BO、调整外部系统或增加新消费者时，核心逻辑不必被每个下游需求拖着改。

但这也带来新的责任：事件一旦成为 contract，就要像 API 一样管理。事件名称、payload 字段、语义、版本、兼容策略、监控方式，都要进入设计和测试。否则事件只是“更难排查的隐形接口”。

48.4 Key mechanism / 核心机制

第一层机制是业务事实。Event 名称应该表达已经发生的事实，例如 TravelApproved、UploadCompleted、SalesOrderChanged，而不是命令式的 ApproveTravel。命令是 action，事实是 event。

第二层机制是触发时机。Business event 应该和保存成功后的业务状态一致。不要在 interaction phase 里过早发事件；否则后续 validation/save 失败时，下游已经收到错误事实。

第三层机制是 payload。Event payload 应足够让消费者定位业务对象和理解变化，但不应把内部实现全部暴露。常见 payload 包括 BO key、event type、timestamp、changed semantic fields、correlation id。过大 payload 会增加兼容和隐私风险。

第四层机制是幂等。Event consumer 必须假设事件可能重复到达。消费者应使用业务 key、event id、version、processed table 或 outbox 状态避免重复处理。

第五层机制是监控和重试。事件发布或消费失败不能靠“用户再点一次”解决。项目需要监控、重试、错误队列、application log 或 integration monitoring。

第六层机制是与 outbox 的关系。某些场景中，项目可能不直接使用 RAP business event，而是保存一条 outbox 记录，再由后台任务或 integration layer 发布。两者目标相近：避免在业务事务中直接调用不可回滚外部系统。

第七层机制是边界。Event 不替代 Validation 校验、Action 业务动作、Save sequence 保存序列、Additional Save 附加保存 或 API testing workflow 接口测试流。它是解耦通知机制，不是同步业务流程的全部。

一个更完整的设计流程如下：

1. Identify the business fact
2. Confirm the fact is true only after successful save
3. Define event name and version
4. Define minimal stable payload
5. Decide publisher responsibility
6. Decide consumer responsibility
7. Design idempotency and retry
8. Design monitoring and evidence
9. Test duplicate, delay, failure, and replay

事件 payload 可以用这个模板设计：

Business Event Payload

eventType:
  TravelApproved

eventVersion:
  1

eventId:
  globally unique id

businessKey:
  TravelUUID / UploadJobId / SalesOrder

occurredAt:
  timestamp

correlationId:
  request id / job id / application log id

semanticFields:
  only stable fields needed by consumers

最小 payload 不等于信息越少越好。它应该足够让消费者重新读取必要数据、做幂等判断、关联日志和解释业务语义；同时避免暴露内部表字段、临时状态、敏感信息或 UI-only 字段。对于可能变化的字段，最好让 consumer 通过 released API 读取最新状态，而不是把所有内部数据塞进 event。

还要在设计书里写清楚“event 不保证什么”。例如 TravelApproved 可以表示生产者系统中 Travel 已进入 approved 业务事实，但它不保证仓储系统已处理、邮件已发送、外部系统已同步成功。后者是 consumer 侧处理结果，需要自己的 monitoring、retry、application log 和验收证据。把 producer event 与 consumer side effect 混为一谈，是事件驱动项目最常见的误解。

事件测试也要像 API 测试一样具体：

测试目的

Delayed deliveryconsumer 晚到仍能读到一致状态

Business event 的设计审查要同时看 producer 和 consumer。只讨论“什么时候发事件”是不够的；还要确认消费者如何处理重复、延迟、失败和版本变化。

Review areaProducer sideConsumer sideEvidence
Business fact事件名是否表达已成立事实，而不是命令是否把事件当事实处理，而不是反向调用生产系统完成命令event name / semantic definition
Save boundary是否只在保存成功后产生事件或 outbox 记录是否假设事件到达时主业务状态已可查询save-success / save-failure test

Idempotency是否提供 event id、business key、correlation id是否记录已处理 event，避免重复副作用duplicate delivery test

这张矩阵也能帮助判断“是否应该用 event”。如果需求要求同步阻断保存、立即返回外部系统结果、或把外部失败作为当前事务失败条件，那么它更像 action/validation/API 编排问题，不应该简单丢给 business event。相反，如果主系统只需要在事实成立后通知下游，让下游异步响应，那么 business event 或 outbox 就是更健康的方向。

意思是：这个处理是保存成功后的通知，因此不应在 handler 中直接 HTTP 调用，而应考虑 business event 或 outbox 方式。

イベント消費側は重複受信を前提に、冪等に処理できる設計にしてください。

意思是：事件消费方要以前提“可能重复收到”为基础，设计成幂等处理。

このイベントは同期チェックではないため、保存を止める業務ルールは validation 側で実装します。

意思是：这个 event 不是同步校验；需要阻断保存的业务规则要在 validation 侧实现。

设计审查常用表达：

イベント名は命令形ではなく、保存後に成立した業務事実として定義します。

含义：事件名不要用命令式，而要定义成保存后成立的业务事实。

イベントペイロードは最小限かつ安定した項目に限定し、内部実装項目を公開しない方針にします。

含义：event payload 限定为最小且稳定的字段，不公开内部实现字段。

イベント失敗時の監視、再処理、デッドレター相当の運用手順を設計に含めます。

含义：把事件失败时的监控、再处理、类似 dead-letter 的运维流程纳入设计。

外部システム側は event id または業務キーで重複処理を防止します。

含义：外部系统侧用 event id 或业务 key 防止重复处理。

48.7 Example / 学习例子

假设 Travel 被批准后要通知外部系统。不要这样设计：

Action handler AcceptTravel
  -> update status in buffer
  -> directly call external HTTP API
  -> save later fails

这样会出现 SAP 保存失败但外部系统已收到批准通知的问题。

更合理的思路是：

Action handler
  -> change BO status to Approved in transactional buffer

Save sequence
  -> validate and persist the approved state

After successful save
  -> publish TravelApproved business event
  -> consumer/integration flow processes notification

如果项目暂时没有完整 event framework，也可以用 outbox：

Additional save / save sequence
  -> write outbox row with event type and key

Background job / integration flow
  -> reads outbox
  -> calls external system
  -> marks processed / retries on failure

对 AC0567C POC，可以把上传任务完成看成事件候选：UploadValidationCompleted、UploadExecutionCompleted、UploadFailed。但要注意，初学阶段不必马上实现 event；先把状态、日志、API testing workflow 做稳定，再考虑事件解耦。

AC0567C 的 event candidate 可以这样分析：

Event candidate何时成立可能消费者Payload 最小字段
UploadValidationCompletedvalidate 阶段完成且结果已保存/可查询UI、监控、后续 execute 编排job id、validation status、summary count、log id
UploadExecutionCompletedexecute 阶段完成且状态已提交外部通知、报表、后续流程job id、execution status、success count、error count、log id
UploadFailedvalidate 或 execute 最终失败且错误可查询监控、运维通知job id、failure stage、error category、log id

这些事件不应该替代 /status API。Event 通知“状态变了”，/status API 仍然是消费者查询稳定状态的入口。这样 payload 可以保持小，消费者也能在收到事件后用 released/stable API 拉取详细结果。

如果项目暂时不接入 SAP eventing，也可以先做 outbox：

Outbox row:
  event_type
  event_version
  business_key
  payload_json
  status = NEW / SENT / FAILED
  retry_count
  last_error
  created_at
  application_log_id

后台 job 或 integration flow 负责读取 outbox、发送、重试和标记状态。这样比在 handler 里直接 HTTP 调用更容易监控和重跑。

48.8 Counterexample / Failure mode

反例一：把 event 当 action。ApproveTravel 是命令，TravelApproved 才是事实事件。修正方式是区分 action 与 event。

反例二：保存前就发 event。后续 validation 失败，下游已经处理。修正方式是把事件放到保存成功后的语义中。

反例三：event payload 包含过多内部字段。消费者开始依赖内部实现，后续无法重构。修正方式是定义稳定、最小、语义化 payload。

反例四：consumer 不幂等。重复事件导致重复发货、重复通知、重复日志。修正方式是使用 event id / business key / processed flag。

反例五：没有监控。事件消费失败后无人知道。修正方式是设计 monitoring、retry、application log。

反例六：把 event 当同步校验。外部系统拒绝后才发现 SAP 已保存。修正方式是同步阻断规则前移到 validation/precheck/authorization。

反例七：payload 暴露敏感字段。为了方便下游，把个人信息、权限敏感字段或内部诊断字段全部塞进 event。修正方式是做数据最小化和授权/隐私审查。

反例八：没有版本策略。payload 改名或删除字段后，旧 consumer 直接失败。修正方式是维护 event version，新增字段保持向后兼容。

反例九：事件顺序假设过强。consumer 假设一定先收到 A 再收到 B，但异步系统可能延迟或重试。修正方式是用业务状态查询和幂等设计，而不是只靠顺序。

反例十：没有事件证据。上线后外部系统说没收到，SAP 侧没有 event id、outbox 状态、application log 或 integration monitoring。修正方式是把 event evidence 纳入测试和运维。

48.9 思考问题

  • 这个需求是同步阻断，还是保存成功后的通知？
  • Event 名称是业务事实，还是命令？
  • Event payload 最小需要哪些字段？
  • 消费方如何处理重复事件？
  • 事件失败如何监控和重试？
  • 是否需要 outbox，而不是直接外部调用？
  • AC0567C 的 validate/execute/status 中，哪些状态变化适合作为 event？
  • payload 中哪些字段是稳定 contract，哪些只是内部实现？
  • consumer 如果重复收到同一个 event，会发生什么？
  • event 失败后，谁负责监控、重试和人工恢复？
