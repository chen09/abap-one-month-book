32. DDIC database table 字典表

  • Chapter 01
  • Section 32
  • self-contained study note

32.1 一句话结论

DDIC database table 字典表 是 ABAP Dictionary 中对持久化数据库表的统一元数据定义。它不是“随便建一张 SQL 表”，而是把表名、字段、主键、client dependency、data element、背后的 domain、数据维护方式、delivery class、activation 和 transport 管理串在一起，成为 ABAP 程序、ABAP SQL、CDS/RAP、数据浏览和项目交付共同依赖的基础对象。

32.2 到底学什么

Day 05 开始进入 ABAP Dictionary。前四天主要在看 ABAP 代码如何运行，今天开始看“代码操作的数据结构从哪里来”。在 Java/Python 项目里，初学者常把 database table 理解成数据库 schema 或 ORM entity；在 SAP 项目里，database table 首先是一个 ABAP repository object。它有包、传输请求、激活状态、字段元数据、语义标签、client 规则和工具入口，最终才映射到物理数据库中的表。

SAP Learning 对 ABAP Dictionary 的说法很关键：ABAP Dictionary 中的 database table definition 是数据库中表的 representation，并且独立于具体数据库细节。它使用 ABAP Dictionary 自己的 built-in dictionary types，而不是直接使用某个数据库厂商的类型；定义里包含 field list 和额外 table properties。对初学者来说，这句话意味着：SAP 表不是先从 HANA/Oracle/SQL Server 的 DDL 角度理解，而是先从 ABAP Dictionary 元数据角度理解。

一个 DDIC database table 至少要看四层。

第一层是 table 本身：表名、description、package、transport request、table category、delivery class、data maintenance、activation。现代 ADT 里可以通过 New -> Other ABAP Repository Object -> Database Table 创建源代码式 Dictionary object；经典系统里也常见 SE11。无论入口是 ADT 还是 SE11，核心都是同一件事：定义一个全局可复用的持久化表对象。

第二层是 field list：哪些字段组成主键，哪些字段存业务数据，字段是否 not null，类型来自 built-in dictionary type 还是 data element。字段不是只有“列名 + 类型”。字段顺序、key 设计、client 字段、金额/数量的参考语义、时间戳、状态字段、外键和搜索帮助，都会影响后续读取、维护和排查。

第三层是 data element 和 domain。Data element 偏语义层，常用于 field label、documentation、默认 component name 等；domain 偏技术和值域层，定义 data type、length、decimal places、fixed values 等。这里有一个初学者很容易误会的边界：在 database table 的 field list 里，字段可以直接使用 built-in dictionary type 或 data element；domain 通常不能直接写成字段类型，而是通过 data element 间接发挥作用。后面几节会分别展开 Table field 字段定义、Data element 数据元素 和 Domain 域对象，本节先建立总图：table field 通常通过 data element 取得可复用的业务语义，再通过 data element 关联到 domain 的技术属性和值域。

第四层是 client dependency。很多业务表是 client-dependent table，第一 key field 通常是 type CLNT 的 client 字段，常见字段名是 MANDT 或 client。这表示同一张表的定义在系统中是全局的，但数据按 client 隔离。ABAP SQL 对 client-dependent table 有 implicit client handling，初学者不要一上来手写 MANDT = sy-mandt；但读表结构时必须知道 MANDT 为什么在主键最前面。这个自动处理只适用于 ABAP SQL 的正常访问路径；Native SQL、AMDP 或直接数据库访问不会自动继承同一层 SAP 语义，跨 client 风险要单独审查。Client-independent table 则跨 client，共享范围更大，项目审查更谨慎。

下面是一个简化的 ADT Dictionary DDL 风格片段，用来读懂结构，不要求照抄到项目里：

@EndUserText.label : 'Training order header'
@AbapCatalog.enhancement.category : #NOT_EXTENSIBLE
@AbapCatalog.tableCategory : #TRANSPARENT
@AbapCatalog.deliveryClass : #A
@AbapCatalog.dataMaintenance : #RESTRICTED
define table zxx_order_h {
  key client   : abap.clnt not null;
  key order_id : zxx_order_id not null;
  created_by   : abap.char(12);
  created_at   : abap.utclong;
  status       : zxx_order_status;
}

这段代码要读出几件事。define table 声明一个 DDIC database table。client 是 client-dependent table 的 client key。order_id 和 status 使用自定义 data element，因此字段语义可以复用；这些 data element 背后可以再关联 domain，但 domain 本身不要直接拿来写成表字段类型。created_by 则直接使用 built-in dictionary type，说明练习表里既可能直接写内置字典类型，也可能引用 data element。created_at 使用 utclong，说明它是 UTC timestamp。deliveryClass 和 dataMaintenance 不是装饰，它们会影响安装、升级、client copy、数据维护入口和项目治理。@AbapCatalog.deliveryClass : #A 通常表示应用表，也就是主数据或交易数据一类的业务数据；@AbapCatalog.dataMaintenance : #RESTRICTED 表示显示/维护受限制，不能理解成“生产可以随便改表内容”。真正项目里还会继续考虑 currency/quantity reference、foreign key、search help、authorization、buffering、data aging、change document、table maintenance generator 等，但 Day 05 第一节先不要全吃下去。

再补两条语法层面的安全线。第一，primary key fields 要连续放在 field list 开头，且 key field 数量有上限；这和很多 Java/Python 开发者“字段随便排列，最后再声明 primary key”的习惯不同。第二，not null 在 key field 上是 Dictionary DDL 里的标准写法，ADT 编辑器也可能自动补齐；初学时先把它当作“key field 定义的一部分”来读，不要把它误解成所有非 key 字段都能靠这一句获得额外业务校验。

从交付角度看，字典表设计直接决定很多后续问题。字段长度太短，接口后面会截断；client dependency 选错，测试数据和生产数据隔离会出问题；key 设计不合理，ABAP SQL 查询、锁对象和 RAP BO key 都会受影响；data element 不复用、domain 设计不一致，UI label、翻译、值域和文档会散落；delivery class 和 maintenance 设置不清，移送和本番反映时就会争论“这到底是业务数据、customizing，还是技术控制表”。

本节的最小心智模型是：DDIC database table 是“SAP repository 中的数据库表定义”，不是单纯物理表。看一张表时，至少要看 table properties、field list、key/client、data element 到 domain 的间接关系，以及 activation/transport 状态。等这张图稳定后，再进入字段、数据元素、域、结构、视图、外键、搜索帮助和锁对象。

32.3 为什么重要

  • ABAP 程序、ABAP SQL、CDS、RAP、OData、报表和接口都经常围绕 DDIC table 读写数据。
  • 字典表不只是存储结构，还承载字段语义、文本标签、值域、client 规则和维护治理。
  • 表定义错误会扩散到代码、测试数据、权限、性能、移送、数据迁移和生产障害调查。
  • Java/Python 背景学习者容易只看 SQL schema；SAP 项目必须同时看 repository object、activation、transport、client、data element 和 domain。
  • 日本项目里，表设计经常出现在基本设计、详细设计、レビュー、移送前确认和障害調査中。

32.4 Key mechanism / 核心机制

  • ABAP Dictionary：集中管理表、结构、数据元素、域、视图、搜索帮助、锁对象等数据定义的 repository 世界。
  • DDIC database table：ABAP Dictionary 中对数据库表的定义，独立于具体数据库平台细节。
  • Table field：表中的列，必须有名称、类型、key/null 等属性；字段类型可以直接使用 built-in dictionary type 或 data element。
  • Data element：字段的语义定义，常承载 label、documentation、默认 component name。
  • Domain：字段的技术和值域定义，常承载 data type、length、decimal places、fixed values；domain 通常通过 data element 间接用于表字段，不要直接写成 field type。
  • Primary key：至少要有 key field；key fields 要连续位于 field list 开头，复合主键不能无限扩张。
  • Client-dependent table：通常以 type CLNT 的 client 字段作为第一 key field，常见字段名 MANDT 或 client。
  • Client-independent table：没有 client 字段，数据跨 client，需要更谨慎的治理。
  • Table category：例如普通 transparent table 和较少见的 global temporary table；初学阶段重点认识 transparent table。
  • Delivery class：影响表数据在安装、升级、client copy、transport、table maintenance 中的处理语义；不要把它简化成“对象能不能移送”。
  • Data maintenance：定义表内容能否通过工具显示/维护，以及显示/维护是否受限制。
  • ABAP SQL client handling：ABAP SQL 正常访问 client-dependent table 时会处理当前 client；Native SQL、AMDP、直接数据库访问需要额外审查 client 条件。
  • Activation：表定义只有激活后才成为可用的 repository object，并可能影响底层数据库对象。
  • Data preview / table contents：用于观察数据，但受权限、client、data maintenance 和系统规则限制。

  • 字段语义：データエレメント、ドメイン、技術属性、項目ラベル、値域、固定値。
  • client/治理：クライアント依存、クライアント非依存、MANDT、デリバリクラス、テーブルメンテナンス。

32.7 Example / 学习例子

观察一张表时，不要只记表名。按这个顺序读：

  1. Table properties：table category、delivery class、data maintenance、description、package。
  2. Field list：key fields、client field、business fields、technical fields。
  3. Field typing：每个字段直接使用 built-in dictionary type 还是 data element；domain 是否通过 data element 间接参与。
  4. Client：是否有 MANDT / client，是否 client-dependent。
  5. Activation / transport：对象是否激活，是否挂在正确 transport request。
  6. Data Preview：能否查看数据，查看的是哪个 client，是否有权限。

可以在 ADT 中找一张练习表或项目表，写出类似摘要：

Table:
Purpose:
Client-dependent:
Primary key:
Main business fields:
Important data elements:
Important domains / fixed values:
Data maintenance:
Project risk if changed:

32.8 Counterexample / Failure mode

  • 只看表名和字段名，不看 data element/domain：会漏掉字段语义、长度、取值限制、UI 标签和翻译。
  • 把 DDIC table 当成普通 SQL table：会忽略 package、transport、activation、client handling 和 SAP 工具链。
  • 在表字段里直接写 domain 名称：会把“domain 是技术和值域层”误读成“domain 可以直接当字段类型”，实际应通过 data element 间接使用。
  • 手写 MANDT = sy-mandt 当成默认习惯：可能和 ABAP SQL client handling 重复或制造错误假设；如果是 Native SQL、AMDP 或直接数据库访问，则又必须另行确认 client 条件。
  • 不区分业务数据表和 customizing / 控制表：delivery class、维护方式、移送策略会被误判。
  • 不看 key 设计就写 SELECT：性能、唯一性、锁对象和后续 RAP key 都可能出问题。
  • 在生产系统用 Data Preview 随意查看敏感数据：权限、审计和数据保护都可能有问题。
  • 修改表结构前不做 where-used / 影响分析：Open SQL、CDS、RAP、接口、报表和迁移程序都可能受影响。
