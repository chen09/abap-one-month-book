Chapter 01 - SAP 和 ABAP 基础

34. Data element 数据元素

  • Chapter 01
  • Section 34
  • self-contained study note

34.1 一句话结论

Data element 数据元素，是 ABAP Dictionary 中“给字段和程序变量复用的语义类型入口”。它不只是字段的中文名，也不只是 domain 的外壳；它把技术类型、字段标签、说明、翻译、可能的 search help / 参数语义，以及背后的 Domain 域对象 连接起来，让 table field、structure component、ABAP 变量、CDS/RAP/OData 元数据和项目文档能说同一种业务语言。

34.2 到底学什么

上一节已经看到 Table field 字段定义 可以直接使用 built-in dictionary type，也可以引用 data element。本节要回答的是：为什么真实项目里很多业务字段更推荐引用 data element，而不是每个字段都写 abap.char(10)、abap.numc(6) 或 abap.curr(15,2)。

先看一个简单例子：

define table zxx_invoice_i {
  key client     : abap.clnt not null;
  key invoice_id : zxx_invoice_id not null;
  item_no        : abap.numc(6);
  payment_status : zxx_payment_status;
}

item_no 直接使用 built-in dictionary type，读者只能看到它是 6 位 numeric text。payment_status 使用 zxx_payment_status 这个 data element，读者就可以继续追踪：它的技术类型是什么、字段标签怎么显示、说明文档写了什么、背后是否有 domain、domain 是否限制固定值、是否有搜索帮助、是否已经被其他表字段或程序复用。一个字段从“技术列”变成“业务概念”，主要靠这层元数据。

SAP Learning 对 data element 的定位很清楚：data element specifies the technical and semantic properties of a table field。这里要特别注意“technical and semantic”这两个词。初学者常说 data element 只管语义、domain 只管技术，这句话有帮助但不完整。更精确的说法是：data element 是可被字段和程序引用的类型对象；它可以直接基于 built-in dictionary type，也可以基于 domain。若它基于 domain，data type、length、decimals、fixed values 等底层技术和值域信息主要由 domain 提供；data element 自己则承担字段语义、label、说明、翻译和复用入口。对 database table field 来说，SAP Learning 更推荐通过 domain 定义技术类型；直接基于 predefined / built-in dictionary type 的 data element 技术上可以存在，但不应成为业务表字段设计的首选。

Data element 最容易被读者看见的是 field labels。ADT / Dictionary editor 通常会维护 short、medium、long、heading/title 等不同长度的字段标签。它们主要在 dynpro、Web Dynpro、ALV 等经典 UI 和部分 metadata 消费场景中发挥作用；SAP Samples 也提醒，Medium Text 这类 label 可能被暴露到 OData metadata。比如 zxx_payment_status 的短标签可以是 Pay.stat，中标签可以是 Payment status，长标签和 heading 可以是 Invoice payment status；Paid、Open、Cancelled 则应留给 domain fixed values 或值文本，而不是作为字段标签。字段标签不是为了好看，而是为了让同一个业务字段在不同宽度、不同语言、不同界面里有稳定表达。

Data element 还有 description / documentation 的作用。SAP Learning 特别提醒：data element 的 description 主要不是给终端用户看的，而是给分析既存代码、判断是否复用该 data element 的开发者看的。字段标签和文本还涉及 original language 与 translation。日本项目里如果对象原语是日语，就要确认日语 field label 是否能用于设计书、画面项目名、ALV 列标题和障害调查；如果对象原语是英语，则要看是否已有日语翻译，而不是在每个报表里随手写一套不同文言。

Data element 也能作为 ABAP 程序里的类型使用：

DATA lv_payment_status TYPE zxx_payment_status.
TYPES ty_invoice_id TYPE zxx_invoice_id.

这和 DATA lv_payment_status TYPE c LENGTH 1. 的含义不同。后者只说明变量是一位字符；前者告诉读者“这是发票支付状态”，并让变量继承 data element 背后的技术属性。这里也要补一条安全线：ABAP data object 用 data element 声明时，运行时主要继承类型，不会自动把 field label、documentation、search help 带进普通变量输出。也就是说，DATA lv_payment_status TYPE zxx_payment_status 不会让 WRITE lv_payment_status 自动显示“Invoice payment status”列标题；label 和 documentation 是 DDIC metadata，需要 UI、ALV、screen、CDS/OData metadata 或工具消费。后面写 method signature、internal table line type、structure component、test data helper 时，如果有合适的 data element，就能把业务语义带进 ABAP 代码，而不是把含义藏在变量名和注释里。

创建流程也要能读懂。SAP Developers tutorial 展示了一个典型 ADT 工作流：把结构字段改成一个尚不存在的 data element 名后，系统会出现 syntax error；用 Ctrl+1 Quick Fix 创建 data element；输入 description、type name 和 field labels；保存并 activation；再回到引用它的 structure 或 table 检查并激活。这个流程说明 data element 是 repository object，不是本地变量别名。它要命名、描述、保存、激活、传输，也会被 where-used list 找到。

还要分清 classic DDIC 语义和现代 ABAP Cloud / RAP 语义。Data element 的 field labels、search help、parameter ID 等语义在经典 ABAP UI 中非常重要；但不能把它们直接搬到 ABAP Cloud / RAP。parameter ID 在 ABAP Cloud 中不受支持，会成为 cloud readiness / ATC 类检查问题。Classic data element search help 适合 on-premise dynpro 等场景；新 RAP/Fiori UI 的 value help 通常应通过 CDS view entity field 上的 @Consumption.valueHelpDefinition、metadata extension 或相关 value help 定义表达。Field labels 和 documentation 仍然是有用的元数据，不是被废弃。SAP Samples 的 ABAP Dictionary Cheat Sheet 还提醒：一些 semantic properties 主要属于 classic ABAP，不能在 ABAP for Cloud Development 中指定；同时 CDS simple types 可以作为 DDIC data elements 的后继概念之一。但这只是在 CDS 层的后继/相关概念，不能理解成 DDIC data element 已经过时，也不能把 CDS simple type 当成 define table 的 table field type；DDIC database table field 仍要使用 built-in dictionary type 或 data element。

本节的最小心智模型是：data element 是“字段可复用语义类型”。读一个 table field 时，如果看到 data element，就顺手追踪四层：field 使用哪个 data element；data element 的 labels / documentation 是否表达业务含义；data element 的技术类型是直接 built-in dictionary type 还是通过 domain；它被哪些表、结构、程序、CDS/RAP 对象复用。能按这个顺序读，下一节 Domain 域对象 才会自然：domain 负责更底层的技术形状和值域，data element 负责把这些形状变成可被项目复用的业务字段语言。

34.3 为什么重要

  • Data element 让多个 table field、structure component、ABAP 变量和接口字段共享同一个业务类型入口，减少“同义字段各自定义”的混乱。
  • Field label、description、translation 会影响 UI、ALV、OData metadata、设计书、测试证据和日本项目沟通。
  • Data element 到 domain 的关系决定字段的底层 type、length、decimals、fixed values 或 value range，改动时必须做 where-used 和影响分析。
  • 新代码里优先使用已有合适 data element，通常比随手写 abap.char(...) 更可维护；但纯技术字段仍可直接用 built-in dictionary type。
  • Java/Python 背景学习者容易把它看成 type alias；SAP 项目里它更像“全局类型 + 字段标签 + 文档 + 翻译 + 元数据复用点”的组合。
  • 日本项目里，data element 经常出现在テーブル定義書、項目定義、画面項目名、レビュー指摘、翻訳確認、使用箇所調査和移送影响分析中。

34.4 Key mechanism / 核心机制

  • Repository object：data element 是 ABAP Dictionary repository object，需要 package、activation、transport 和 where-used 管理。
  • Global type：data element 可以作为全局类型，被 database table field、DDIC structure component、ABAP DATA / TYPES、方法签名等引用。
  • Technical properties：data element 可以直接基于 built-in dictionary type，也可以引用 domain 来继承 data type、length、decimals 等属性；database table field 的业务 data element 通常优先采用 domain 作为技术基础。
  • Semantic properties：data element 承载 field labels、description / documentation、translation、可能的 search help 或 classic UI 相关语义；ABAP DATA ... TYPE data_element 继承的是类型，不会自动继承 label 到普通变量输出。
  • Field labels：常见 label 有 short、medium、long、heading/title 等长度变体，经典最大长度通常是 10 / 15 / 40 / 55，用于不同 UI 或 metadata 场景。
  • Original language / translation：field labels 和文本可翻译；创建时的 logon language 会影响 original language，需要在日本项目里特别确认。
  • Domain link：推荐的可复用路径通常是 table field -> data element -> domain；domain 不能直接作为 table field type 或 ABAP TYPE 目标，这不是风格偏好，而是语法/激活边界。
  • Direct built-in type path：对于 client、纯技术字段、教学例子或语义非常局部的字段，可以直接使用 built-in dictionary type，但要知道失去了 data element 复用语义。
  • Where-used impact：改 data element 的 label、type、domain 或 documentation 可能影响所有引用它的 table field、structure、program、CDS/RAP/service 消费。
  • Classic vs modern metadata：classic search help / parameter ID / field label 等语义在传统 UI 很重要；ABAP Cloud 不支持 parameter ID，新 RAP/Fiori value help 通常看 CDS @Consumption.valueHelpDefinition、metadata extension 和 value help 定义。
  • CDS simple type awareness：在现代 ABAP 中，CDS simple type 可作为 DDIC data element / domain 的后继概念之一，但它不能替代 DDIC database table field 的 data element 用法，Day 05 先学稳 DDIC 基础。

34.5 日本项目现场说法

  • 常见日语：データエレメント、項目定義、項目ラベル、短テキスト、中テキスト、長テキスト、見出し。
  • 技术/语义：技術属性、意味属性、ドメイン参照、値域、固定値、検索ヘルプ、パラメータID。
  • 交付/审查：原語、翻訳、使用箇所、影響調査、影響範囲、有効化、移送対象、テーブル定義書。
  • 说明句：この項目はデータエレメントを参照しているため、項目ラベルと説明文を共通で利用できます。
  • Review 句：既存の標準データエレメントを利用できる場合は、新規作成せず再利用可否を確認してください。
  • Review 句：データエレメントの型やドメインを変更する場合は、使用箇所と画面表示、インタフェース影響を確認してください。
  • Cloud 句：ABAP Cloud ではパラメータIDはサポート対象外のため、ATCチェック対象になります。
  • RAP 句：RAPベースのUIでは、検索ヘルプではなく @Consumption.valueHelpDefinition などの value help 定義を確認してください。

34.7 Example / 学习例子

观察一个 data element 时，可以按这个表记录：

Data element:
Purpose:
Used by fields / structures:
Based on built-in dictionary type or domain:
Domain:
Data type / length / decimals:
Short label:
Medium label:
Long label:
Heading / title:
Description / documentation:
Search help / classic semantic property:
Translation needed:
Project risk if changed:

以 zxx_payment_status 为例，可以这样读：

Data element: zxx_payment_status
Purpose: invoice payment status
Based on: domain zxx_d_payment_status
Technical shape: CHAR 1, inherited from domain
Possible values: P = Paid, O = Open, C = Cancelled
Short label: Pay.stat
Medium label: Payment status
Long label: Invoice payment status
Heading: Invoice payment status
Used by: zxx_invoice_i-payment_status, report output, RAP projection later
Risk if changed: UI label, fixed values, existing data, tests, interface mapping

再把它放回 table field：

payment_status : zxx_payment_status;

这行不是只把字段长度设成 1，而是把“发票支付状态”这个业务概念挂到字段上。读者看到 zxx_payment_status，应继续追踪 data element 和 domain，而不是停在字段名。

34.8 Counterexample / Failure mode

  • 每个字段都直接写 abap.char(...)：短期能激活，长期会丢失 label、translation、documentation、domain value range 和复用语义。
  • 把 data element 当成 domain：会误以为 fixed values、length、decimals 都一定写在 data element 自己身上；实际常由 domain 提供。
  • 把 domain 当成 data element：domain 不能直接作为 table field type 或 ABAP TYPE 目标；这会导致语法错误或 activation error，应通过 data element 间接使用。
  • 看到标准 data element 不复用，直接新建 Z 对象：会制造重复语义，增加翻译、测试和接口 mapping 成本。
  • 随意改 data element length / domain：可能影响所有引用字段、结构、程序、CDS/RAP 对象和外部接口。
  • 只改中文/日文 label，不看 original language 和翻译：会导致不同登录语言下 UI 或 metadata 显示不一致。
  • 把 field label 当成唯一业务说明：label 只是短文本，复杂规则应写在设计书、documentation、domain/value help、validation 或业务代码里。
  • 在 ABAP Cloud / RAP 场景中只依赖 classic search help：现代服务还要检查 CDS value help、annotation 和 metadata extension。
  • 以为 DATA lv_x TYPE zxx_data_element 会自动带出 field label：ABAP 变量继承类型，不会自动继承 UI label 到普通输出。
  • 把 CDS simple type 当成 DDIC table field type：CDS simple type 属于 CDS 层后继/相关概念，不等于可以替代 data element 来定义 DDIC database table field。
  • 以为 where-used 为空就一定可删：动态引用、接口约定、文档约定、外部 consumer 仍需项目证据确认。
