23. LOOP AT ASSIGNING vs LOOP AT INTO

  • Chapter 02
  • Section 23
  • self-contained study note

23.1 一句话结论

LOOP AT ... INTO 和 LOOP AT ... ASSIGNING 的差别，是 internal table 循环里“复制当前行到 work area”与“用 field symbol 直接指向当前行”的差别：INTO 更像拿一份副本来读写，修改后需要明确写回；ASSIGNING 直接操作原 table row，修改会立即反映到 internal table。选择哪一个，不是语法偏好，而是要看本次循环是只读、临时计算，还是要原地更新大量行。

23.2 到底学什么

前三节已经铺好了背景：

  • READ TABLE 读取内部表 讲单行读取。
  • Table expression 表表达式 讲表达式式单行访问。
  • Field symbol 字段符号 和 FIELD-SYMBOL vs DATA reference 讲 alias / reference 语义。

本节把这些概念落到最常见的 ABAP 循环：

LOOP AT lt_items INTO DATA(ls_item).
  ...
ENDLOOP.

和：

LOOP AT lt_items ASSIGNING FIELD-SYMBOL(<ls_item>).
  ...
ENDLOOP.

二者看起来只是 INTO 与 ASSIGNING 的差别，实际影响很大：

  • INTO：当前行内容复制到 work area ls_item。
  • ASSIGNING：field symbol <ls_item> 指向当前 internal table 行。
  • INTO 中修改 ls_item 不会自动改回表。
  • ASSIGNING 中修改 <ls_item> 会直接改原 table row。
  • INTO 更适合只读、临时计算、避免副作用。
  • ASSIGNING 更适合大量行更新、大结构处理、明确需要原地修改。

23.3 为什么重要

ABAP 初学者最常见的 internal table bug 之一，就是在 LOOP AT ... INTO 里改了 work area，以为表已经变了：

LOOP AT lt_items INTO DATA(ls_item).
  ls_item-status = 'C'.
ENDLOOP.

这段代码循环结束后，lt_items 里的 status 没有因此改变。因为 ls_item 是副本。除非再 MODIFY 回表，否则改动只停留在 work area。

另一类 bug 刚好相反：代码用了 ASSIGNING，作者以为只是临时处理，实际上直接改了原表：

LOOP AT lt_items ASSIGNING FIELD-SYMBOL(<ls_item>).
  <ls_item>-status = 'C'.
ENDLOOP.

这段代码会直接修改 lt_items 当前行。它通常更直接；在大表更新时也可能更快，但也更有副作用。项目 review 要看清楚：这是有意更新，还是误改原数据？

对 Java/Python 背景学习者，可以这样理解：INTO 像把当前元素拷贝到一个局部变量，ASSIGNING 像拿到当前元素的可写别名。但 ABAP 的 internal table 有 standard/sorted/hashed、primary/secondary key、MODIFY、sy-tabix、key 字段限制等规则，不能只用 Python list 或 Java collection 的直觉判断。

23.4 Key mechanism / 核心机制

23.4.1 LOOP AT ... INTO：复制到 work area
LOOP AT lt_items INTO DATA(ls_item).
  IF ls_item-status = 'O'.
    ls_item-status = 'C'.
  ENDIF.
ENDLOOP.

这段代码只改了 ls_item。如果要把修改写回 internal table，必须明确写回：

LOOP AT lt_items INTO DATA(ls_item).
  IF ls_item-status = 'O'.
    ls_item-status = 'C'.

    MODIFY lt_items FROM ls_item INDEX sy-tabix
      TRANSPORTING status.
  ENDIF.
ENDLOOP.

上面示例假定 lt_items 是可以按 index 修改的 index table，例如 standard table。对 sorted/hashed table，不能机械套用 INDEX sy-tabix，要结合 table category、key 和 MODIFY TABLE 等语义设计。

23.4.2 LOOP AT ... ASSIGNING：直接指向当前行
LOOP AT lt_items ASSIGNING FIELD-SYMBOL(<ls_item>).
  IF <ls_item>-status = 'O'.
    <ls_item>-status = 'C'.
  ENDIF.
ENDLOOP.

这里不需要 MODIFY lt_items FROM ...。<ls_item> 指向当前 internal table 行，赋值就是直接修改该行。SAP Learning 对 field symbol 处理 internal table 的解释也强调：使用 ASSIGNING 时，程序直接处理 table line，而不是先复制到 work area 再写回。

23.4.3 read-only loop 选择

如果只是读数据并做统计，INTO 往往更容易理解：

DATA lv_total TYPE p LENGTH 16 DECIMALS 2.

LOOP AT lt_items INTO DATA(ls_item)
     WHERE status = 'C'.
  lv_total += ls_item-amount.
ENDLOOP.

这段代码没有修改表。对小表或简单结构，INTO 的清晰性可能比少量 copy 成本更重要。对大结构、大表、性能敏感场景，可以评估 ASSIGNING，但不要把所有 loop 自动改成 field symbol。

23.4.4 update-heavy loop 选择

如果本来就是要批量更新 table rows，ASSIGNING 通常更直接：

LOOP AT lt_items ASSIGNING FIELD-SYMBOL(<ls_item>)
     WHERE status = 'O'.
  <ls_item>-status = 'C'.
  <ls_item>-changed_by = sy-uname.
ENDLOOP.

它避免了每行 copy out 和 MODIFY back。SAP Learning 的练习里展示了 work area + MODIFY 与 field symbol 更新 internal table 的 runtime 对比，并说明更新场景通常能减少复制成本；但实际项目仍应结合数据量、结构大小和 ABAP Profiler/SAT 结果判断。

23.4.5 REFERENCE INTO

本节主要对比 INTO 和 ASSIGNING，但上一节提过第三种：

LOOP AT lt_items REFERENCE INTO DATA(lr_item).
  ASSIGN lr_item->* TO FIELD-SYMBOL(<ls_item>).
  <ls_item>-status = 'C'.
ENDLOOP.

REFERENCE INTO 给的是 data reference。只有当后续 API 或设计确实需要 reference variable 时才使用；局部直接更新当前行时，ASSIGNING 往往更清楚。

23.4.6 key 字段和 table category

ASSIGNING 不是绕过 key 规则的后门。对 sorted table、hashed table 的 key 字段，通过 field symbol 赋值会触发运行时错误；不要试图用 <ls_item>-key_field = ... 直接改变 internal table 的排序/哈希一致性。需要改变 key 时，通常要删除旧行并插入新行，或重新构造目标表。

23.4.7 删除、插入、排序中的 caution

在 LOOP AT 中边循环边 DELETE、INSERT、SORT、CLEAR 同一张 internal table，会让当前行、sy-tabix、field symbol/reference 的有效性变复杂。初学阶段先采用清楚的两阶段写法：先收集要变更的 key 或结果，再统一处理；确实需要边循环边改表时，查 LOOP AT、DELETE itab、MODIFY itab 的官方语义并写出测试。

23.5 日本项目现场说法

  • 常见日语：ワークエリア、コピー、ASSIGNING、フィールドシンボル、直接更新、MODIFY不要、副作用、性能改善。
  • 说明句：INTO は内部テーブル行をワークエリアへコピーするため、変更を反映するには MODIFY が必要です。
  • 说明句：ASSIGNING は現在行をフィールドシンボルで直接参照するため、<ls_item> の変更は内部テーブルへ即時反映されます。
  • Review 句：このループは ASSIGNING で元データを直接更新しているため、副作用が意図通りか確認してください。
  • 性能说明：大量データ更新ではコピー削減により性能改善が期待できますが、読み取りだけの小さなループでは可読性も考慮します。

23.7 Example / 学习例子

23.7.1 例 1：只读统计，INTO 很自然
DATA lv_total TYPE p LENGTH 16 DECIMALS 2.

LOOP AT lt_items INTO DATA(ls_item)
     WHERE status = 'C'.
  lv_total += ls_item-amount.
ENDLOOP.

这里只读，不改表。INTO 的副本语义反而让意图清楚：本循环不会改变 lt_items。

23.7.2 例 2：用 INTO 更新，需要 MODIFY
LOOP AT lt_items INTO DATA(ls_item)
     WHERE status = 'O'.
  ls_item-status = 'C'.

  MODIFY lt_items FROM ls_item INDEX sy-tabix
    TRANSPORTING status.
ENDLOOP.

如果忘记 MODIFY，lt_items 不会被更新。TRANSPORTING status 可以减少无关字段覆盖风险。

23.7.3 例 3：用 ASSIGNING 更新，不需要 MODIFY
LOOP AT lt_items ASSIGNING FIELD-SYMBOL(<ls_item>)
     WHERE status = 'O'.
  <ls_item>-status = 'C'.
ENDLOOP.

这段是原地更新。代码短，但必须确认直接修改原表符合业务意图。

23.7.4 例 4：大结构更新
LOOP AT lt_sales ASSIGNING FIELD-SYMBOL(<ls_sale>)
     WHERE recalculation_required = abap_true.
  <ls_sale>-net_amount = <ls_sale>-quantity * <ls_sale>-unit_price.
  <ls_sale>-changed_on = cl_abap_context_info=>get_system_date( ).
ENDLOOP.

如果 lt_sales 行结构很宽、数据量大、循环本来就是更新行，ASSIGNING 能减少每行复制和写回成本。

23.7.5 例 5：需要保留原表不变时，不要用 ASSIGNING
LOOP AT lt_items INTO DATA(ls_item).
  ls_item-status = 'SIMULATED'.
  APPEND ls_item TO lt_preview.
ENDLOOP.

这里业务意图是生成预览表，不改变原表。INTO 更安全。

23.8 Counterexample / Failure mode

23.8.1 误区 1：INTO 中改了 work area，以为原表已变
LOOP AT lt_items INTO DATA(ls_item).
  ls_item-status = 'C'.
ENDLOOP.

这是初学者高频 bug。修正：要么 MODIFY 回表，要么改用 ASSIGNING。

23.8.2 误区 2：ASSIGNING 中做临时计算，误改原表
LOOP AT lt_items ASSIGNING FIELD-SYMBOL(<ls_item>).
  <ls_item>-amount = <ls_item>-amount * '1.10'.
  APPEND <ls_item> TO lt_preview.
ENDLOOP.

这会改原表金额。若只是生成预览，应该用 INTO 或显式复制到另一个结构。

23.8.3 误区 3：盲目用 INDEX sy-tabix

sy-tabix 和 index-based modification 依赖 table category 和访问方式。对 hashed table 没有业务位置概念；对 sorted table 也不能随便改 key。不要把 MODIFY ... INDEX sy-tabix 当成所有 internal table 的通用答案。

23.8.4 误区 4：为了性能牺牲可读性

ASSIGNING 对大表更新很有意义，但小表只读循环不一定值得。性能优化要看数据量、结构大小和 SAT/Profiler 证据。

23.8.5 误区 5：循环中边改结构边改集合

在同一个 loop 中同时使用 ASSIGNING 修改当前行、再 DELETE / INSERT / SORT 同一张表，容易制造难查的缺陷。先写清楚 intent，再考虑是否拆成两阶段处理。

23.9 思考问题

  • 这段 loop 是只读、生成副本，还是更新原 internal table？
  • 如果使用 INTO，修改 work area 后是否需要 MODIFY？
  • 如果使用 ASSIGNING，直接修改原 table row 是否符合业务意图？
  • 当前 internal table 是 standard、sorted，还是 hashed？是否依赖 sy-tabix？
  • 是否修改了 key 字段？这对 sorted/hashed table 是否安全？
  • 这段 loop 的数据量和 line width 是否真的需要 performance optimization？
  • 在日本项目 review 中，应该强调 コピー、直接更新、MODIFY不要，还是 副作用？
