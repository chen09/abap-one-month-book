Chapter 02 - Core ABAP Programming

28. Static vs instance 静态与实例

  • Chapter 02
  • Section 28
  • self-contained study note

28.1 一句话结论

Static vs instance 静态与实例是 OO ABAP 里区分“属于 class 本身”与“属于某个 object instance”的基础规则。CLASS-METHODS、CLASS-DATA 是 static components，使用 class selector => 调用；METHODS、DATA 是 instance components，通常要先创建 object reference，再用 instance selector -> 调用。

初学者最重要的判断不是“static 写起来更短”，而是“这个能力是否需要对象自己的状态”。如果只是纯计算、稳定常量、无副作用 helper，static 可以很清楚；如果逻辑依赖订单、客户、缓存、测试替身、事务上下文或可变状态，instance 往往更安全。滥用 static mutable state 会把 class 变成隐形全局变量，让 ABAP Unit 单元测试基础、并发、后台作业和 RAP/服务逻辑都更难解释。

28.2 到底学什么

上一节 Method visibility 可见性 讲的是 method 谁能访问。本节讲的是：method 和 attribute 归谁所有，是归 class 这个类型所有，还是归某个 object instance 所有。

本节要掌握七个边界：

  • CLASS-METHODS 声明 static method，用 zcl_xxx=>method( ) 或 lcl_xxx=>method( ) 调用。
  • METHODS 声明 instance method，通常先 NEW #( ) 创建对象，再 lo_obj->method( ) 调用。
  • CLASS-DATA 声明 class attribute，属于 class，在当前 ABAP runtime context 中被该 class 的使用者共享。
  • DATA 在 class definition 中声明 instance attribute，每个 object instance 都有自己的值。
  • static method 没有某个对象的 me，不能直接访问 instance attribute 或 instance method；要访问实例状态，必须通过 object reference。
  • instance method 可以访问自己的 instance state，也可以不通过 object reference 访问 class-level static attribute / method；这个方向是允许的，但它会把对象逻辑和共享 class state 绑在一起。
  • static 不等于数据库持久化、不等于跨系统共享、不等于线程安全或事务安全。

一个最小对比：

CLASS lcl_status_text DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS get_text
      IMPORTING iv_status      TYPE string
      RETURNING VALUE(rv_text) TYPE string.
ENDCLASS.

CLASS lcl_counter DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS increment.
    METHODS get_count
      RETURNING VALUE(rv_count) TYPE i.

  PRIVATE SECTION.
    DATA mv_count TYPE i.
ENDCLASS.

lcl_status_text=>get_text( 'O' ) 不需要对象状态，只是把状态码转成文字，因此可以是 static helper。lcl_counter 的 mv_count 是每个 counter 自己的状态，因此应该是 instance attribute；两个 counter object 的计数不应互相污染。

28.3 为什么重要

static / instance 的选择会直接影响代码能不能测试、能不能解释、能不能在项目里安全维护。

常见问题是：初学者觉得 static method 不用 NEW，调用很方便，于是把所有 service、cache、当前用户、处理结果、错误列表都塞进 CLASS-DATA。短期看代码少了几行，长期看问题会变多：

  • 单体测试顺序不同，static cache 没清，结果互相污染。
  • dialog 执行和 background job 执行看到的 class state 不一致，排查困难。
  • RAP handler 或 service logic 被 framework 多次调用时，上一轮请求留下的 static 状态影响下一轮。
  • public static attribute 变成隐形全局变量，任何地方都能改。
  • 想给逻辑注入 test double 时，因为全是 static call，很难替换依赖。

Java/Python 背景学习者可以把 static method 类比为 Java static method 或 Python class-level helper，但 ABAP 项目还要多想运行环境：ABAP class 是 repository object，代码运行在 ABAP internal session 中，业务事务还受到 SAP LUW 逻辑事务单元、authorization、lock、RAP save sequence 和 background processing 影响。static state 不是企业级共享缓存，只是当前运行上下文里的一份 class-level state。

28.4 Key mechanism / 核心机制

28.4.1 class component selector and instance component selector

ABAP 用两个 selector 帮你区分归属：

DATA(lv_text) = lcl_status_text=>get_text( 'O' ).

DATA(lo_counter) = NEW lcl_counter( ).
lo_counter->increment( ).
DATA(lv_count) = lo_counter->get_count( ).

=> 表示访问 class component。看到它时先问：这个 method / attribute 是否真的不需要某个对象自己的状态？

-> 表示通过 object reference 访问 instance component。看到它时先问：这个对象代表什么业务对象、服务对象或测试对象？它内部有哪些状态？生命周期由谁管理？

28.4.2 static method is not object behavior

static method 属于 class，不属于某个 object instance。它适合表达不依赖对象状态的能力，例如：

  • 单纯格式转换：status code 到显示文本。
  • 稳定的计算规则：金额四舍五入、字符串规范化。
  • 工厂入口：根据输入创建某类对象或返回接口引用。
  • 常量或枚举式查询：返回稳定 code list。

示例：

CLASS lcl_status_text IMPLEMENTATION.
  METHOD get_text.
    CASE iv_status.
      WHEN 'O'.
        rv_text = 'Open'.
      WHEN 'C'.
        rv_text = 'Closed'.
      WHEN OTHERS.
        rv_text = 'Unknown'.
    ENDCASE.
  ENDMETHOD.
ENDCLASS.

这个 method 不读取订单对象内部状态，也不修改 shared cache，作为 static helper 是合理的。

但如果 method 需要当前订单的 items、customer、pricing context、错误列表或 test double，它就不是“无状态 helper”，更适合 instance method 或通过 Interface 接口定义 传入依赖。

28.4.3 instance method owns object state

instance method 属于一个具体 object。每次 NEW lcl_counter( ) 都会得到一个新的 object instance，每个 instance 有自己的 mv_count。

CLASS lcl_counter IMPLEMENTATION.
  METHOD increment.
    mv_count = mv_count + 1.
  ENDMETHOD.

  METHOD get_count.
    rv_count = mv_count.
  ENDMETHOD.
ENDCLASS.

如果创建两个对象：

DATA(lo_a) = NEW lcl_counter( ).
DATA(lo_b) = NEW lcl_counter( ).

lo_a->increment( ).
lo_a->increment( ).
lo_b->increment( ).

lo_a 的 count 是 2，lo_b 的 count 是 1。这个差异就是 instance state 的意义：对象把自己的数据和行为封装在一起。

28.4.4 CLASS-DATA is shared class state

CLASS-DATA 是 static attribute。它不属于某个对象，而属于 class：

CLASS lcl_counter DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS constructor.
    CLASS-METHODS get_created_count
      RETURNING VALUE(rv_count) TYPE i.

  PRIVATE SECTION.
    CLASS-DATA gv_created_count TYPE i.
    DATA mv_count TYPE i.
ENDCLASS.

CLASS lcl_counter IMPLEMENTATION.
  METHOD constructor.
    gv_created_count = gv_created_count + 1.
  ENDMETHOD.

  METHOD get_created_count.
    rv_count = gv_created_count.
  ENDMETHOD.
ENDCLASS.

这里 gv_created_count 记录当前运行上下文中创建过多少个 counter。它不是每个对象自己的 count，而是 class-level count。

这类设计要非常谨慎。CLASS-DATA 适合稳定常量、只读配置、一次性初始化后的不可变表，或者明确的 class-level 计数/缓存。它不适合保存“当前订单”“当前用户处理结果”“本次请求错误列表”这类请求级或业务对象级状态。

28.4.5 static method and instance state

static method 没有当前对象，因此不能直接写：

CLASS-METHODS reset_count.

DATA mv_count TYPE i.

然后在 static method 里直接访问 mv_count。因为 mv_count 属于哪个对象？static method 没有答案。

如果确实要让 static method 操作某个对象，应显式传入 object reference：

CLASS-METHODS reset
  IMPORTING io_counter TYPE REF TO lcl_counter.

这里有一个不对称关系：static method 不能在没有 object reference 的情况下直接访问 instance component；但 instance method 可以访问 static component。后者语法上允许，却不一定是好设计。只要 instance method 读写了可变 CLASS-DATA，它的行为就不再只由当前 object state 决定，测试和排查都会变难。

因此 review 时要小心：如果 static method 总是需要传入对象并修改对象状态，可能说明它本来就应该是 instance method；如果 instance method 频繁读写 static attribute，可能说明 class-level mutable state 被滥用了。

28.4.6 constructor and class_constructor

Constructor 构造方法 会在下一节展开，本节先建立最小边界：

  • METHODS constructor 是 instance constructor，创建 object instance 时执行，用来初始化这个对象自己的状态。
  • CLASS-METHODS class_constructor 是 class constructor，class 第一次被使用前由 runtime 触发，用来初始化 class-level static state。

class constructor 不应承担业务副作用。不要把数据库更新、authorization、远程调用、消息提交、全局状态变更或复杂业务流程放进 class constructor；否则只是引用一个 class，就可能触发难以预期的动作。class constructor 最适合做轻量、可解释、幂等的 static 初始化。

28.4.7 static is not persistence

static attribute 看起来“放在 class 上”，但它不是数据库表，不是跨用户共享内存，也不是跨 application server 的稳定缓存。不同用户、不同 internal session、不同 background job、不同 application server 之间不能把普通 CLASS-DATA 当成可靠共享状态；用英文检索时可以记作：ordinary CLASS-DATA is not cross-user, cross-job, or cross-application-server persistence.

如果项目真的需要跨请求、跨会话、跨系统稳定保存状态，应使用数据库表、lock object、shared memory object、enqueue、配置表或 RAP persistence 等明确机制，而不是靠普通 static attribute。

28.4.8 static or instance decision checklist

在 code review 里，static / instance 的判断可以用一张小表收口。不要只问“能不能写成 static”，要问“写成 static 后谁拥有状态、谁负责清理、谁能替换依赖、测试之间会不会互相污染”。

QuestionStatic is usually acceptable when...Prefer instance when...
Does it need object state?只依赖输入参数和稳定常量需要订单、批次、用户上下文、错误列表、缓存对象
Is the result deterministic?同样输入总是同样输出结果依赖当前用户、日期时间、DB、锁、外部系统
Is there mutable shared state?没有，或只读初始化后不再变需要保存 request/session/business-object state
Can tests isolate it?ABAP Unit 不需要清理 static cache测试需要不同 fake dependency 或不同对象状态
Can dependencies be replaced?没有外部依赖，或依赖通过参数传入需要 interface / test double / constructor injection
Is it framework-owned?纯 helper 或 factory 入口RAP handler、service object、repository、adapter 等有生命周期

一个实用规则是：static helper 可以小而纯；一旦 method 开始记住东西、访问外部资源、依赖当前用户、或需要在 ABAP Unit 中替换依赖，就应该认真考虑 instance object。尤其不要把 CLASS-DATA 当成“方便放临时结果的地方”，也不要留下 hidden CLASS-DATA cache 让测试和运行时行为互相污染。如果一定要使用 static cache，也要写清楚生命周期、清理方式、失效条件、测试隔离方式和并发/后台 job 风险。

28.5 日本项目现场说法

  • 常见日语：静的メソッド、インスタンスメソッド、クラス属性、インスタンス属性、状態管理、副作用、テスト容易性。
  • 设计说明句：この処理はインスタンス状態を持たない純粋な変換なので、静的メソッドとして定義しています。
  • Review 指摘句：この CLASS-DATA はリクエスト単位の状態を保持しているため、テストや並行実行で副作用が出る可能性があります。
  • 改善说明句：注文ごとの状態はインスタンス属性に持たせ、外部依存はインターフェース経由で注入する方針にします。
  • Review 补充句：クラスコンストラクタでは、DB更新や外部呼び出しなどの業務副作用を避け、軽量な初期化に限定します。
  • 使用场景：class 设计、代码 review、ABAP Unit 失败分析、RAP handler 设计、cache 设计、后台 job 障害调查。

在日本 SAP 项目现场，static 经常被简单说成 静的，但 review 里真正关心的是 状態を持つかどうか。也就是：这个处理是否有状态？状态是否跨对象共享？是否会产生副作用？是否影响单体测试？

28.7 Example / 学习例子

28.7.1 例 1：无状态转换适合 static

状态码转文本，不需要对象状态：

CLASS lcl_status_text DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS get_text
      IMPORTING iv_status      TYPE string
      RETURNING VALUE(rv_text) TYPE string.
ENDCLASS.

调用：

DATA(lv_text) = lcl_status_text=>get_text( 'O' ).

这个例子适合 static，因为输入完整、输出明确、没有共享可变状态。

28.7.2 例 2：每个对象自己的状态适合 instance

购物车、订单草稿、计算器上下文、校验结果收集器，都可能需要对象自己的状态：

CLASS lcl_message_collector DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS add_message
      IMPORTING iv_text TYPE string.
    METHODS get_count
      RETURNING VALUE(rv_count) TYPE i.

  PRIVATE SECTION.
    DATA mt_messages TYPE STANDARD TABLE OF string WITH EMPTY KEY.
ENDCLASS.

CLASS lcl_message_collector IMPLEMENTATION.
  METHOD add_message.
    APPEND iv_text TO mt_messages.
  ENDMETHOD.

  METHOD get_count.
    rv_count = lines( mt_messages ).
  ENDMETHOD.
ENDCLASS.

每个 lcl_message_collector object 都有自己的 mt_messages。单元测试可以为每个 test method 创建新对象，状态清楚，副作用少。

28.7.3 例 3：static mutable state 的风险

不建议：

CLASS lcl_order_context DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-DATA gv_current_order_id TYPE string.
    CLASS-DATA gt_messages TYPE STANDARD TABLE OF string WITH EMPTY KEY.
ENDCLASS.

这等于把“当前订单”和“当前消息”放进 class-level 全局状态。任何调用者都可能改它；测试之间、处理步骤之间、异常恢复之后都可能留下旧值。

更好的方向是：

CLASS lcl_order_context DEFINITION FINAL.
  PUBLIC SECTION.
    METHODS constructor
      IMPORTING iv_order_id TYPE string.
    METHODS add_message
      IMPORTING iv_text TYPE string.

  PRIVATE SECTION.
    DATA mv_order_id TYPE string.
    DATA mt_messages TYPE STANDARD TABLE OF string WITH EMPTY KEY.
ENDCLASS.

现在状态属于某个 order context object。生命周期由创建者控制，测试也可以创建干净的新对象。

28.7.4 例 4：static facade 与 instance implementation

有时项目为了调用方便，会保留一个很薄的 static facade，但真正逻辑仍然放在 instance service 中：

CLASS lcl_quantity_check DEFINITION FINAL.
  PUBLIC SECTION.
    CLASS-METHODS is_valid
      IMPORTING iv_quantity     TYPE i
      RETURNING VALUE(rv_valid) TYPE abap_bool.

    METHODS check
      IMPORTING iv_quantity     TYPE i
      RETURNING VALUE(rv_valid) TYPE abap_bool.
ENDCLASS.

CLASS lcl_quantity_check IMPLEMENTATION.
  METHOD is_valid.
    DATA(lo_check) = NEW lcl_quantity_check( ).
    rv_valid = lo_check->check( iv_quantity ).
  ENDMETHOD.

  METHOD check.
    rv_valid = xsdbool( iv_quantity > 0 ).
  ENDMETHOD.
ENDCLASS.

这不是每个场景都需要的模式，但它展示了一个思路：外部可以有简单入口，内部仍然保留可测试的 instance 逻辑。如果逻辑以后需要依赖 currency service、customizing reader 或 test double，就可以继续把依赖放到 constructor 或 interface 中。

28.8 Counterexample / Failure mode

28.8.1 误区 1：为了少写 NEW 全部改 static

static call 很短，但短不等于设计正确。只要逻辑需要对象状态、依赖注入或测试替身，全部 static 会让后续维护更难。

修正动作：先判断是否需要状态；无状态 helper 可以 static，有状态业务服务优先 instance。

28.8.2 误区 2：把 CLASS-DATA 当跨请求缓存

普通 CLASS-DATA 不是跨用户、跨 job、跨 application server 的可靠缓存。把它当企业级缓存会导致“我这里有值、别人那里没有值”的问题。

修正动作：需要持久化就用数据库；需要并发控制就用 lock/enqueue；需要共享缓存就使用项目认可的缓存机制。

28.8.3 误区 3：static method 偷偷改 mutable shared state

一个看起来像纯函数的 static method，如果内部改 CLASS-DATA，调用者很难预期副作用。

修正动作：static helper 尽量保持输入输出清楚，避免修改 mutable static state；若项目确实需要 cache，明确命名、明确生命周期、提供清理入口，并写测试覆盖清理。不要把“用得小心”当成充分理由，默认应避免 public/writable static attribute 和隐藏 CLASS-DATA cache。

28.8.4 误区 4：instance method 滥用 static dependencies

instance service 虽然有对象状态，但如果内部到处直接调用 static dependency，就仍然很难测试。

修正动作：对外部系统、时间、用户、数据库访问、远程调用等不稳定依赖，优先考虑 interface + constructor injection，或至少在小范围内隔离 static call。

28.8.5 误区 5：把 static 当权限或事务边界

static / instance 只是 OO 归属边界，不是 authorization、lock 或 SAP LUW 逻辑事务单元 边界。static method 仍然要遵守权限检查、事务提交、错误处理和 Clean Core 依赖规则。

修正动作：权限逻辑回到 Authorization check 权限检查 或 RAP authorization；事务逻辑回到 LUW、commit、rollback、save sequence；依赖逻辑回到 Clean Core 核心干净。

28.9 思考问题

  • 这个 method 是否依赖某个对象自己的状态？如果依赖，为什么是 static？
  • 这个 CLASS-DATA 保存的是稳定配置，还是本次请求的临时状态？
  • 两个 object instance 同时存在时，它们的状态是否应该互相独立？
  • 这个 static method 是否会偷偷修改 shared state？
  • 如果把这个逻辑放进 ABAP Unit，测试之间是否会互相污染？
  • 如果后续要替换数据库访问、当前日期、当前用户或外部 API，static call 是否会妨碍 test double？
  • 在日本项目里 review 时，应说 静的メソッドでよい，还是应指出 状態管理上、インスタンス化した方が安全です？
  • 哪一个 SAP Help 页面可以证明 METHODS / CLASS-METHODS / CLASS-DATA 的语法区别？
