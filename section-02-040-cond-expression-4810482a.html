Chapter 02 - Core ABAP Programming

40. COND expression 条件表达式

  • Chapter 02
  • Section 40
  • self-contained study note

40.1 一句话结论

COND expression 条件表达式 是 ABAP conditional expression / conditional operator 中用于“根据条件产生一个值”的写法。它常和 VALUE、CORRESPONDING、SWITCH 等现代表达式写法一起学习，但在 ABAP Keyword Documentation 的分类里，COND 属于 conditional expressions，而不是 VALUE 那类 constructor expression。它把简单的 IF ... ELSEIF ... ELSE ... ENDIF 赋值逻辑压缩成一个表达式，形式类似：

DATA(result) = COND string(
  WHEN lv_status = 'A' THEN `Active`
  WHEN lv_status = 'B' THEN `Blocked`
  ELSE `Unknown`
).

这段代码的核心不是“少写几行”，而是表达清楚：result 是一个值，这个值由一组条件决定。COND 适合小而清楚的 value selection，例如状态显示文本、布尔标志、简单分类、默认值选择、ABAP Unit expected value。它不适合承载多步业务流程、数据库更新、消息组装、权限判断链或复杂副作用逻辑。

初学者要先抓住四件事：第一，COND 返回一个结果值；第二，WHEN 后面是 logical expression，按顺序判断；第三，所有 THEN / ELSE 的结果必须能转换到同一个 result type；第四，如果没有任何 WHEN 成立且没有写 ELSE，结果会是 result type 的 initial value。

40.2 到底学什么

本节要掌握十个点：

  • COND 是 conditional operator / conditional expression，用来构造“条件决定的结果值”。
  • 基本语法是 COND type( WHEN condition THEN result ... ELSE result )，也可以在有明确上下文时写 COND #( ... )。
  • WHEN 后面写 ABAP logical expression，例如 =, <>, BETWEEN, IS INITIAL, AND, OR。
  • 多个 WHEN 按顺序检查；第一个为 true 的分支决定结果。
  • ELSE 不是必须写，但没有 ELSE 时，所有条件都不成立会得到 result type 的 initial value。
  • THEN / ELSE 后面的结果必须能转换为 COND 的 result type。
  • THROW 可以作为条件表达式结果，用于在非法输入时直接抛出 class-based exception；它不返回值，因此不参与 result type conversion。
  • THROW SHORTDUMP 是更强烈的 runtime error 语境，初学阶段只要知道它存在，并且业务代码不要轻易使用。
  • LET 可在 COND 内定义局部 helper value；它可以出现在外层，也可以出现在 THEN / ELSE 的 result 位置前，作用域只在对应表达式内。
  • COND 和 IF 的差异是：COND 产生一个值，IF 执行一组语句；不要为了“现代写法”把多步骤流程硬塞进表达式。
  • 日本项目 review 重点是 COND式、条件分岐、初期値、型推論、可読性、THROW指定、IF文に戻すべきか。

最小判断标准：看到 COND #( ... ) 时，能回答“最终 result type 是什么、每个 WHEN 条件是否互斥/有顺序依赖、没有 ELSE 时会得到什么、是否应该改回 IF 或抽 helper method”。

40.3 为什么重要

SAP 项目里常见大量“根据状态给一个值”的小逻辑：

IF lv_status = 'A'.
  lv_text = `Active`.
ELSEIF lv_status = 'B'.
  lv_text = `Blocked`.
ELSE.
  lv_text = `Unknown`.
ENDIF.

如果这段逻辑只是为了产生 lv_text，COND 会更贴近意图：

DATA(lv_text) = COND string(
  WHEN lv_status = 'A' THEN `Active`
  WHEN lv_status = 'B' THEN `Blocked`
  ELSE `Unknown`
).

这在 ABAP Unit、RAP helper method、OData DTO 组装、ALV output 字段、message severity 分类、简单 release flag 判断中都很实用。读者一眼能看出“这里不是一个业务流程，只是在选一个值”。

但 COND 也容易被滥用。过深嵌套、过长条件、带副作用 method call、混合多种业务规则的 COND，会比 IF 更难 debug。项目里发生障害调查时，reviewer 需要逐个看 WHEN 条件、result type、initial value、异常路径。如果表达式变成一屏还看不完，就已经失去价值。

40.4 Key mechanism / 核心机制

40.4.1 Basic form returns one value

最基础形式：

DATA(priority_text) = COND string(
  WHEN lv_priority = '1' THEN `High`
  WHEN lv_priority = '2' THEN `Middle`
  ELSE `Normal`
).

COND string( ... ) 明确说明结果是 string。ABAP 会按顺序检查每个 WHEN，第一个成立的分支返回 THEN 后的值。如果没有任何 WHEN 成立，就返回 ELSE 后面的值。

可以把它读成：“构造一个 string，这个 string 的值由条件决定。”

40.4.2 # needs result type context

和 VALUE #( ... )、CORRESPONDING #( ... ) 一样，COND #( ... ) 的 # 依赖静态可推断的 operand type：

DATA lv_text TYPE string.

lv_text = COND #(
  WHEN lv_status = 'A' THEN `Active`
  ELSE `Unknown`
).

这里左侧 lv_text TYPE string 提供了 result type。如果用 inline declaration：

DATA(lv_text) = COND #(
  WHEN lv_status = 'A' THEN `Active`
  ELSE `Unknown`
).

reviewer 就要确认 ABAP 能否按预期推断类型，以及推断出来的类型是否足够稳定。把 COND #( ... ) 直接传给 generic formal parameter（例如 ANY）或上下文不够明确的位置时，可能出现语法错误或推断结果不符合意图。初学阶段建议在业务字段、金额、数量、日期、状态代码、message text 这类场景中优先写显式类型，例如 COND string( ... )、COND abap_bool( ... )、COND decfloat34( ... )，不要把业务类型交给 # 猜。

40.4.3 WHEN is a logical expression

WHEN 后面不是一个固定值，而是完整 ABAP logical expression：

DATA(can_release) = COND abap_bool(
  WHEN lv_status = 'OPEN'
   AND lv_amount <= lv_limit
   AND lv_blocked = abap_false
  THEN abap_true
  ELSE abap_false
).

这和下一节 SWITCH 的差异很重要。COND 适合“每个分支都有自己的判断条件”；SWITCH 更像 CASE，适合“对一个 operand 的不同取值做映射”。如果只是把一个状态码映射成文本，SWITCH 可能更清楚；如果判断条件包含金额、日期、权限、状态组合，COND 更合适。

多个 WHEN 分支按从上到下的顺序检查，第一条 true 的分支决定结果。因此条件顺序有业务含义。更具体的条件应该放在更泛化的条件之前，否则后面的具体条件可能永远到不了。

40.4.4 ELSE omission means initial value

官方资料明确提醒：如果没有写 ELSE，并且没有任何 WHEN 条件成立，结果是指定或推断出的 data type 的 initial value。

DATA(lv_flag) = COND abap_bool(
  WHEN lv_status = 'OPEN' THEN abap_true
).

如果 lv_status 不是 OPEN，lv_flag 会是 abap_bool 的 initial value，通常等价于 abap_false。这可能正是你要的，也可能是隐藏 bug。比如金额、日期、状态文本、message severity 没写 ELSE，就可能静默得到 0、空日期、空字符串或空状态。

初学阶段的纪律：业务结果建议写 ELSE，除非 initial value 本来就是明确的业务语义，并且用变量名或注释让 reviewer 看得出来。

对业务关键字段，ELSE THROW cx_...( ) 往往比静默返回 initial value 更安全。空字符串、0、00000000 这类 initial value 不会自动报错，却可能一路传到接口输出、会计金额、状态判断或 message 组装中。

40.4.5 Result type conversion

COND 的所有分支结果最终都要进入同一个 result type：

DATA(lv_amount_text) = COND string(
  WHEN lv_amount IS INITIAL THEN `0`
  ELSE |{ lv_amount }|
).

这里两个分支都是可转换为 string 的值。相反，如果写：

DATA(lv_result) = COND string(
  WHEN lv_ok = abap_true THEN `OK`
  ELSE 1
).

语法或转换层面可能可以处理，但语义上很差：一个分支像文本，一个分支像数字。项目里不要只问“能不能编译”，还要问“这个 result type 是否表达业务含义”。

如果某个 THEN / ELSE 结果无法转换为 COND 的 result type，通常会在语法检查或 activation 时暴露，而不是等到生产环境才发现。THROW 是特殊情况：它不是一个要转换成 result type 的值，而是直接抛出异常。

40.4.6 THROW in COND

COND 的结果位置可以使用 THROW 抛出 class-based exception。典型场景是“非法输入时没有合法结果”：

DATA(lv_rate) = COND decfloat34(
  WHEN lv_quantity <> 0 THEN lv_amount / lv_quantity
  ELSE THROW cx_sy_zerodivide( )
).

这和先写 IF lv_quantity = 0. RAISE EXCEPTION ... ENDIF. 再计算的意图类似，但表达得更紧凑。初学者要注意：只有当异常路径确实是“无法产生结果”时，THROW 才适合放在 COND 里。不要把复杂消息组装、日志记录、状态更新一起塞进去。

THROW 在这里和 RAISE EXCEPTION TYPE ... 的性质一致：它会抛出 class-based exception，调用方需要通过 RAISING 契约或外层 TRY...CATCH 处理。它不会因为写在 COND 里面就被“吃掉”。THROW SHORTDUMP 也可作为更强的 runtime error 形式，但业务应用代码通常应优先使用 class-based exception、RAP reported/failed 或清晰的错误处理，不要轻易制造 short dump。

40.4.7 LET avoids repeated helper logic

LET 可在 COND 内声明局部 helper value。官方语法允许在 COND 外层使用 LET，也允许在分支 result 位置前使用 LET。本书先讲最常见、最容易读的外层形式：

DATA(time_text) = COND string(
  LET lv_time = cl_abap_context_info=>get_system_time( ) IN
  WHEN lv_time BETWEEN '050001' AND '120000' THEN |Good morning { lv_time TIME = ISO }|
  WHEN lv_time BETWEEN '120001' AND '180000' THEN |Good afternoon { lv_time TIME = ISO }|
  ELSE |Good evening { lv_time TIME = ISO }|
).

这样可以避免重复调用方法或重复写同一段表达式。LET 中声明的 helper value 只在这个表达式内可见，不像 DATA(var) inline declaration 那样在整个 processing block 中可见。

但 LET 不是炫技工具。如果 LET、COND、VALUE、FOR、THROW 嵌套到读者需要来回数括号，就应该抽 helper method 或改回多行 IF。

40.4.8 COND vs IF

COND 和 IF 的区别可以这样判断：

  • 如果目标是“计算一个值”，且每个分支只给出一个简单结果，优先考虑 COND。
  • 如果目标是“执行一组动作”，例如更新数据库、调用多个 service、组装多条 message、写日志、填多个字段，优先用 IF。
  • 如果分支条件超过三四个，或者每个 THEN 后面都是复杂表达式，优先考虑 helper method。
  • 如果嵌套过深，ATC / code review 很可能会把它当作可维护性问题。
  • 如果线上障害调查时需要逐步观察每个中间值，IF 通常比深层 COND 更容易 debug。复杂 COND 往往是单步执行的表达式，调试时不容易逐个观察每个 WHEN 分支。

Modern ABAP 的目标不是“所有 IF 都消灭”，而是让值选择更贴近值选择，让流程控制仍然像流程控制。

40.4.9 Operand-position caveats

COND 是表达式，但这不等于它可以无脑放进任何看起来像“表达式位置”的地方。项目里只需要记住两个 awareness 级别的注意点。

第一，某些 string template formatting option 位置不接受直接嵌入的 COND，例如把 COND i( ... ) 直接写进 ALIGN = ( ... ) 这类格式选项时，具体 release 可能在语法检查阶段拒绝。遇到这种情况，先把 COND 结果赋给一个 helper variable，再把变量放进 formatting option，通常更清楚也更稳定。

第二，在旧的 on-premise release 中，把 COND #( ... ) 直接作为 predicative method call 的实参使用，可能因为类型推断或 logical expression 语境而出现 release-dependent 的语法/activation 问题。更保守的写法是让方法调用结果显式比较为布尔值，例如 IF method( param = COND abap_bool( ... ) ) = abap_true.，或者提前用局部变量保存 COND 的结果。初学阶段不用背这些边角限制，但要建立一个习惯：如果 COND #( ... ) 的上下文不够明确，就写显式 result type 或拆成变量。

40.4.10 COND review checklist

Review COND 表达式时，可以用下面的表快速判断它是否仍然保持“值选择”的边界：

QuestionGood COND signPrefer IF / helper method when...
Is it producing one value?每个分支只返回一个简单值或 THROW。分支里实际想做多步动作、日志、消息或数据库更新。
Is the result type clear?COND string( ... ) 或上下文能清楚推断 #。# 需要读很多层 expression 才能知道类型。
Is ELSE intentional?有明确 ELSE，或没有 ELSE 时 initial value 正是业务默认。漏写 ELSE 会让 false path 静默变 initial。
Are conditions ordered deliberately?顺序代表优先级，且互斥/覆盖关系清楚。条件重叠，后续 WHEN 可能永远不会执行。
Is THROW appropriate?非法输入应中断并由调用方处理。只是普通业务分支，应该返回 value、message 或 validation result。
Is debugging still readable?分支少、表达式浅，出错时容易定位。COND、LET、VALUE 嵌套太深，难以逐步观察。
Is it tested?ABAP Unit 覆盖主要分支、默认分支和异常分支。分支条件多，但没有测试固定行为。

一句项目判断：COND 适合把“小型条件赋值”写得更靠近结果值；如果它开始承担流程编排，就应该退回 IF 或抽出一个有名字的 method。可读性比“看起来现代”更重要。

40.5 日本项目现场说法

  • 常见日语：COND式、COND条件式、条件式、論理式、論理条件、条件分岐、条件値選択、条件による値構築、WHEN句、THEN句、ELSE句、コンパイル時型推論、初期値、型固有の初期値、THROW指定、例外送出、実行時エラー、可読性、保守性、デバッグ容易性、IF文に戻す。
  • 说明句：この COND #( ... ) は条件に応じて一つの値を返す式であり、複数の処理を実行するためのものではありません。
  • 说明句：WHEN 句の中の条件は 論理式 であり、COND式 全体が返す値とは区別して説明してください。
  • Review 指摘句：ELSE がない場合、どの条件にも該当しないと型固有の初期値になります。この初期値が業務上正しいか確認してください。
  • Review 指摘句：COND #( ... ) の結果型が読み取りにくいため、COND string( ... ) のように明示型にしてください。
  • Review 指摘句：条件分岐が長くなっているため、可読性とデバッグ性を考慮して IF文 または helper method に戻してください。
  • Review 指摘句：THROW指定 を使用していますが、この分岐は本当に値を返せない異常系か確認してください。
  • Review 指摘句：複数の WHEN 句は上から順に評価されるため、より具体的な条件を先に置いているか確認してください。
  • Review 指摘句：COND式 を IF文に戻す ことは、デバッグ時や分岐が複雑になった場合の標準的なリファクタリング手法であり、スタイル上の後退ではありません。
  • 使用场景：状态文本转换、release flag 判断、message severity 分类、ALV 显示字段、ABAP Unit expected value、RAP helper method 的小型值选择。

40.6 English Terms

  • Official / search terms: COND, conditional expression, COND operator, logical expression, conditional operator, COND, Conditional Operator
  • ABAP syntax: COND type( WHEN log_exp THEN result ... ELSE result ), COND #( ... ), WHEN, THEN, ELSE, LET ... IN, THROW, THROW SHORTDUMP
  • Related terms: result type, operand type, type inference, initial value, class-based exception, runtime error
  • Comparison terms: IF statement, SWITCH expression, CASE statement, value selection, control flow, side effect
  • Recommended search pattern: ABAP COND conditional expression SAP Help, ABAP COND THROW SAP Help, ABAP COND SWITCH result SAP Help

读官方资料时注意：COND 的 WHEN 是 logical expression；SWITCH 是另一种条件表达式，更接近 CASE。不要把二者混成一个语法。

40.7 Example / 学习例子

40.7.1 例 1：状态码转显示文本
DATA(status_text) = COND string(
  WHEN lv_status = 'A' THEN `Active`
  WHEN lv_status = 'B' THEN `Blocked`
  ELSE `Unknown`
).

适合 ALV、日志、DTO 输出中的小型 display value。注意：如果这个映射是正式业务 code list，项目里通常还要有 customizing、domain fixed values、mapping table 或统一转换方法，不能散落在各处。

40.7.2 例 2：布尔标志
DATA(can_release) = COND abap_bool(
  WHEN lv_status = 'OPEN'
   AND lv_amount <= lv_limit
  THEN abap_true
  ELSE abap_false
).

这比先声明变量再写四行 IF 更像“计算一个 flag”。但如果 release 条件来自复杂权限、审批流、锁对象或外部状态，就应该抽成 is_release_allowed( ) 之类的方法。

40.7.3 例 3：无 ELSE 的 initial value
DATA(error_text) = COND string(
  WHEN lv_has_error = abap_true THEN `Error`
).

如果 lv_has_error 不是 true，error_text 是 string 的 initial value，也就是空字符串。这个例子可用于显示层；但如果空字符串会让调用方误以为“已经有正常文本”，就应写明确 ELSE。

40.7.4 例 4：THROW 表达非法输入
DATA(unit_price) = COND decfloat34(
  WHEN lv_quantity > 0 THEN lv_amount / lv_quantity
  ELSE THROW cx_sy_zerodivide( )
).

这表示：数量有效时产生单价；数量无效时没有合法单价，直接进入 exception path。

40.7.5 例 5：用 ABAP Unit 固定分支行为
DATA(result) = COND string(
  WHEN lv_status = 'A' THEN `Active`
  WHEN lv_status = 'B' THEN `Blocked`
  ELSE `Unknown`
).

cl_abap_unit_assert=>assert_equals(
  exp = `Blocked`
  act = result
).

项目里如果 COND 体现的是接口字段、状态规则、消息 severity 或金额分类，就值得用 ABAP Unit 固定几个边界条件，特别是 unknown / initial / ELSE path。

40.8 Counterexample / Failure mode

40.8.1 误区 1：把 COND 当成 IF 的万能替代

COND 适合产生一个值，不适合执行多步流程。把数据库更新、message append、日志、状态更新塞进 THEN 表达式，会让代码难读也难 debug。

修正动作：多动作流程用 IF；复杂值选择抽 helper method。

40.8.2 误区 2：漏写 ELSE，却不知道 initial value 是什么

没有 ELSE 时，所有条件都不成立会得到 result type 的 initial value。对 string 是空字符串，对 numeric 是 0，对 date/time 是初始值，对 abap_bool 通常相当于 false。

修正动作：业务语义上有“兜底值”时写 ELSE；只有 initial value 本身就是明确语义时才省略。

40.8.3 误区 3：过度相信 #

COND #( ... ) 在上下文清晰时很方便，但在 inline declaration、混合 literal、金额/日期/状态字段中可能让 result type 不够明显。

修正动作：业务字段优先写显式 type，如 COND string( ... )、COND abap_bool( ... )、COND decfloat34( ... )。

40.8.4 误区 4：结果分支类型混乱

一个分支返回文本，另一个分支返回数字，虽然可能有转换规则，但读者会困惑。

修正动作：先决定 result type，再让每个 THEN / ELSE 都表达同一类业务含义。

40.8.5 误区 5：嵌套表达式过深

COND 里面再嵌 VALUE、FOR、REDUCE、另一个 COND，短期看很“现代”，长期维护很痛苦。

修正动作：超过一屏、超过三四个分支、或需要调试中间值时，抽 method 或改回 IF。

40.9 思考问题

  • 这段 COND 产生的 result type 是什么？是否需要显式写出来？
  • 所有 WHEN 条件是否有顺序依赖？第一个满足条件是否就是业务期望？
  • 没有 ELSE 时，initial value 是否是正确业务结果？
  • 每个 THEN / ELSE 后面的值是否属于同一种业务含义？
  • 这里用 COND 是否比 IF 更清楚？如果不是，为什么还要用？
  • 是否存在异常路径？如果用 THROW，调用方是否能处理？
  • 这段条件是否需要 ABAP Unit 覆盖边界值？
  • 在日本项目 review 中，应如何说明 COND式 和 IF文 的取舍？
